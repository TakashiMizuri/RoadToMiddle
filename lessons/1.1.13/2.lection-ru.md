# ref, out, in, params — семантика

> Roadmap: `1.1.13` · Node: `1.1` — C#: основы языка · Depth: **глубоко**

## Learning Objectives

После этой лекции ты сможешь:

- Объяснить разницу между **pass by value** и **pass by reference** в C# на уровне языка и IL.
- Корректно использовать **`ref`**, **`out`**, **`in`** и **`params`** и предсказывать поведение caller/callee.
- Сформулировать правила **definite assignment** для каждого модификатора.
- Выбирать между **`ref`**, **`out`** и возвратом tuple/value при проектировании API.
- Понимать, когда **`in`** избегает копирования без разрешения мутации.
- Описать, как компилятор понижает **`params`** и какие аллокации это может вызвать.

## Why This Matters

В уроке `1.1.12` ты разобрал параметры и возвращаемые значения методов. По умолчанию C# передаёт аргументы **by value**: callee получает копию битов, представляющих аргумент. Для маленьких value types вроде `int` копирование дёшево. Для крупных struct — `Matrix4x4`, кастомный `Point3D` с десятком полей — копия на каждый вызов в горячем цикле заметна. Для reference types копируется только указатель (8 байт на 64-bit), но callee не может **заменить** переменную caller'а, пока не получит **адрес** этой переменной.

C# даёт четыре связанных инструмента — **`ref`**, **`out`**, **`in`**, **`params`** — которые меняют способ передачи аргументов через границу метода. Это не «мелкие ключевые слова», а **контракты** между вызывающим и вызываемым кодом. Неправильное использование даёт compile error (хорошо) или тонкие баги (плохо): чтение неинициализированного `out`, мутация через `in`, уверенность что `params` не аллоцирует, когда аллоцирует.

Middle-разработчик ежедневно видит эти модификаторы в API фреймворка: `int.TryParse(..., out int result)`, `Interlocked.CompareExchange(ref location, ...)`, numerics с `in T` для крупных readonly struct, logging с `params object[]`. Понимание семантики — не заучивание сигнатур — позволяет читать BCL и проектировать методы с ясным intent.

## Core Concepts

### Pass by Value: базовая модель

Когда ты пишешь `void Increment(int x)` и вызываешь `Increment(n)`, runtime копирует значение `n` в `x`. Изменения `x` внутри метода не затрагивают `n`. Для reference types копируется сам reference: caller и callee указывают на один объект в heap, поэтому **мутация состояния объекта** (`list.Add(1)`) видна caller'у, а **переприсваивание параметра** (`list = new List<int>()`) — нет.

Это различие — **изменить объект** vs **переприсвоить переменную** — фундамент для понимания, зачем нужен `ref`. Без `ref` метод не может заставить переменную caller'а ссылаться на другой объект или хранить другое value.

### ref: alias на storage caller'а

Модификатор **`ref`** означает «передай alias переменной caller'а». Параметр и аргумент — **одна и та же ячейка памяти**. Присвоение параметру в callee немедленно видно caller'у.

```csharp
static void Swap(ref int a, ref int b)
{
    int temp = a;
    a = b;
    b = temp;
}

int x = 1, y = 2;
Swap(ref x, ref y);
// x == 2, y == 1
```

**Правило definite assignment для `ref`:** аргумент должен быть **definitely assigned** до вызова. Нельзя передать `ref` неинициализированной локали. Нельзя передать `ref` property или большинству выражений — только **переменным** (на стороне return `ref` из `1.1.11` — отдельная история).

Используй `ref`, когда метод должен **читать и писать** то же storage, что у caller: swap, interlocked, ref struct returns, или осознанная мутация крупного struct без копии.

### out: выходной слот без инициализации у caller

**`out`** тоже pass-by-reference, но контракт другой. Caller может передать **неинициализированную** переменную; callee **обязан присвоить** параметр перед каждым return (компилятор проверяет на нормальных путях). Callee не должен **читать** значение до записи — входящее состояние считается мусором.

```csharp
static bool TryDivide(int numerator, int denominator, out int quotient)
{
    if (denominator == 0)
    {
        quotient = 0;
        return false;
    }
    quotient = numerator / denominator;
    return true;
}
```

С C# 7 **`out`-переменные** объявляются inline: `TryDivide(10, 2, out int result)` — идиоматично для `Try*`.

**`ref` vs `out`:** на IL оба byref; различие — **definite assignment analysis**. Выбирай **`out`**, когда параметр только **возвращает** значение вместе со status flag. Выбирай **`ref`**, когда метод **читает** текущее значение и может обновить его.

### in: read-only alias для эффективности

**`in`** передаёт by reference, но запрещает callee менять storage (кроме unsafe). Для **крупных readonly struct** `in T` избегает копии без риска случайной мутации.

```csharp
readonly struct LargePayload
{
    public readonly double A, B, C, D, E, F;
}

static double Sum(in LargePayload p) =>
    p.A + p.B + p.C + p.D + p.E + p.F;
```

На call site **`in` опционален** для переменных: `Sum(payload)` неявно передаёт `in payload`. Для **`readonly struct`** компилятор может автоматически применять `in`, чтобы избежать defensive copy.

Используй **`in`** для read-only доступа к крупным struct в performance-sensitive коде. Не используй для `Point` из четырёх байт — копия часто дешевле indirection.

**Ограничение:** `in` нельзя модифицировать; **`ref struct`** (`Span<T>`) имеют дополнительные ограничения по heap и generics.

### params: переменное число аргументов

**`params`** позволяет последнему параметру принимать ноль и более аргументов element type. Компилятор собирает их в **один массив**.

```csharp
static int Sum(params int[] values) { /* ... */ }

Sum();              // пустой массив
Sum(1, 2, 3);       // new int[] { 1, 2, 3 }
Sum(new[] { 4, 5 }); // явный массив — без двойной обёртки
```

**Правила:** только **последний** параметр; один `params` на метод; тип — **одномерный массив**. **`params ReadOnlySpan<T>`** (C# 13 / .NET 9+) снижает аллокации — проверяй target framework.

## Under the Hood

### IL и managed pointer

Обычные value parameters приходят на evaluation stack by value. **`ref`**, **`out`**, **`in`** в сигнатуре помечены **`byref`**. Caller генерирует **`ldloca`**; callee использует **`ldind`/`stind`**.

Это не C++ pointers: GC отслеживает managed pointers при compaction. Raw pointer arithmetic запрещён в safe code.

**`out`** и **`ref`** делят IL convention; различие — **dataflow analysis** компилятора. **`in`** добавляет **`readonly`** в metadata (modreq), JIT это уважает.

### Definite assignment analysis

Компилятор строит **dataflow graph**. Для **`out`** каждый путь к `return` требует assignment. Для **`ref`** входное значение считается valid для read. Смена `TryParse` с `out` на `ref` сломала бы контракт — caller мог бы передать мусор.

### params и компилятор

`Log("a", "b", "c")` против `void Log(params string[] args)` компилятор превращает в `Log(new string[] { "a", "b", "c" })` — **heap allocation** массива. Hot paths часто используют overloads или `ReadOnlySpan<T>`.

### Связь с ref returns (1.1.11)

**`ref` return** отдаёт alias storage caller'а. Lifetime не может выходить за storage — `ref` на локальную переменную в return запрещён.

## Examples

### Try pattern с out

```csharp
if (int.TryParse(userInput, out int parsed))
    ApplyDiscount(parsed);
```

`parsed` definitely assigned только когда `TryParse` вернул `true` — flow analysis это понимает.

### ref для in-place mutation

```csharp
static void Normalize(ref Vector3 v) { /* делит компоненты на длину */ }
```

Return нового struct работал бы, но копирует; **`ref`** документирует мутацию на месте.

## Common Mistakes & Anti-patterns

**Чтение `out` до assignment** — logic bug. **`ref` везде «для скорости»** — шум в API. **`params` в tight loop** — скрытые аллокации. **`in` ≠ const`** — не deep-freeze для mutable reference внутри struct.

---

## Production & Real-World Notes

BCL: **`out`** для `Try*`, **`ref`** в `Interlocked` и pooling, **`in`** в numerics/game math. Публичные API: tuple `(bool, T)` часто читабельнее нескольких `out`. Nullable + `[NotNullWhen(true)]` для custom Try methods.

---

## Comparison / Trade-offs

Return vs **`out`**: tuple composable; **`out`** — стиль BCL без wrapper для primitives. **`ref` vs return struct`**: для маленьких struct return проще; для slot в массиве — **`ref`**. **`in` vs by value`**: измеряй; порог ~24–32 байта эвристичен. **`params` vs overloads`**: surface area vs allocations.

## Key Takeaways

- По умолчанию **pass by value**; `ref`/`out`/`in` — **managed pointer** с разными контрактами.
- **`ref`** = read+write; **`out`** = write-only output; **`in`** = read-only; **`params`** = array wrapper.
- IL: `ref` и `out` одинаковы; семантика — compile-time analysis.
- API clarity важнее micro-optimization.
