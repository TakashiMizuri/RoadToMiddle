# HTTP Headers — краткая выжимка

> `0.3.7` · ~10 мин на повторение

## Главная идея

**Headers** — metadata HTTP-сообщения (`0.3.3`): format body, negotiation, credentials, caching. Status code (`0.3.6`) говорит *исход*; headers — *как читать*, *кому отдавать*, *сколько хранить*.

## Ключевые пункты

- **Content-Type** — media type body (request/response); **Accept** — preferred response format (не путать).
- **Authorization: Bearer** — explicit token per request; **Cookie** — browser echo stored session (`Set-Cookie` в `0.3.9`).
- **Cache-Control**: `no-store` (private data), `private`/`public`, `max-age`, `immutable` для hashed static.
- **ETag** + **If-None-Match** → **304** без body при unchanged representation.
- **`Vary: Authorization`** — если кэшируешь auth responses (лучше не кэшировать `/api/me`).
- Не логировать **Authorization** / **Cookie**.

## Самые важные примеры

**JSON POST:**
```
Content-Type: application/json
Authorization: Bearer <token>
```

**Conditional GET:**
```
Response: ETag: "v47"
Request:  If-None-Match: "v47"
→ 304 Not Modified
```

**Private API response:**
```
Cache-Control: no-store
```

## «Под капотом» в одном абзаце

Kestrel парсит headers до action. ASP.NET input/output formatters используют Content-Type/Accept. Auth middleware читает Authorization или Cookie. Cache layers (browser → CDN → origin) интерпретируют Cache-Control и ETag; revalidation генерирует 304. За reverse proxy нужны forwarded headers для correct HTTPS redirects.

## Частые ошибки

- Accept и Content-Type swapped.
- `Cache-Control: public` на personalized JSON.
- JSON POST без Content-Type.
- Ignore ETag/304 — лишний bandwidth.

## Мнемоника

> **Content-Type** = «что внутри посылки»; **Accept** = «что хочу получить в ответ»; **Authorization** = «вот мой пропуск на этот запрос».
