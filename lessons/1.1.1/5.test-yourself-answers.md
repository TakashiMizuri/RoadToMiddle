## Answers

### 1. [Recall]
**Answer:** Версии установленных **SDK**, **runtimes**, информация о **host** (`dotnet` executable), paths к SDK, OS/architecture, и т.д.

**Why:** `dotnet --info` — диагностика toolchain; первая команда после install.

---

### 2. [Recall]
**Answer:** **`bin/`** и **`obj/`** — генерируются при build, зависят от машины/конфигурации, раздувают repo и вызывают merge conflicts.

**Why:** Артефакты воспроизводимы через `dotnet build`; в Git хранят исходники и `.csproj`.

---

### 3. [Recall]
**Answer:** **Target Framework Moniker** — проект компилируется под .NET 8 API и ожидает соответствующий runtime.

**Why:** TFM определяет доступные BCL APIs и версию runtime при FDD.

---

### 4. [Understanding]
**Answer:** **SDK** включает compiler, MSBuild, templates, NuGet — нужен для **create/build/publish/test**. **Runtime** — только CLR + libraries для **запуска** уже собранных apps.

**Why:** Dev machine = SDK; production server часто = runtime-only (меньше footprint).

---

### 5. [Understanding]
**Answer:** SDK-style project **импортирует** defaults через `<Project Sdk="Microsoft.NET.Sdk">` — сотни targets вынесены в SDK, не в ваш файл.

**Why:** DRY для MSBuild; override через `<PropertyGroup>` в `.csproj`.

---

### 6. [Understanding]
**Answer:** **`--`** отделяет опции **`dotnet run`** от **аргументов приложения**, передаваемых в `Main(string[] args)` — здесь `"Alice"`.

**Why:** Без `--` CLI может интерпретировать аргументы как свои флаги.

---

### 7. [Application]
**Answer:**
```bash
dotnet new console -n Greeter -o Greeter
dotnet run --project Greeter
```
(или `cd Greeter` затем `dotnet run`)

**Why:** `-n`/`-o` создают project; run требует путь к `.csproj` или cwd внутри project.

---

### 8. [Application]
**Answer:** `dotnet add package Serilog`

**Why:** Добавляет `<PackageReference>` в `.csproj` и triggers restore on build.

---

### 9. [Application]
**Answer:** `dotnet build -c Release` (или `dotnet run -c Release`)

**Why:** `-c` / `--configuration` выбирает Debug vs Release.

---

### 10. [Analysis]
**Answer:** **Неверно.** `dotnet run` **сначала build** (если нужно), затем запускает **скомпилированную assembly** через host, не интерпретирует `.cs` напрямую.

**Why:** C# — compiled language; IL в `bin/`.

**Common mistake:** Путать `dotnet run` с script runner.

---

### 11. [Analysis]
**Answer:** **C** — финальный deployable output для production получают через **`dotnet publish`**, не из `obj/`; `obj/` — intermediate (obj files, caches).

**Why:** `bin/` — build output; `obj/` — промежуточные файлы компилятора/MSBuild.

---

### 12. [Understanding]
**Answer:** **Фиксирует версию SDK** для repo, чтобы local и CI использовали один toolchain; избегает «works on my machine» из-за разных SDK.

**Why:** `global.json` в корне; `rollForward` управляет допустимым drift.

---

### 13. [Application]
**Answer:** `bin/Debug/net8.0/<ProjectName>.dll` (и host `.exe` на Windows).

**Why:** Path pattern: `bin/<Configuration>/<TargetFramework>/`.

---

### 14. [Analysis]
**Answer:** **Нет** (для framework-dependent). Runtime 6 не выполняет app, собранную под **net8.0** — нужен matching **.NET 8 runtime**.

**Why:** TFM привязан к major версии shared framework (для типичного FDD).

---

### 15. [Understanding]
**Answer:** **Project** — одна compilable unit (`.csproj`). **Solution** — контейнер, группирующий **несколько projects** и их relationships для IDE/CI, сам по себе не компилируется в assembly.

**Why:** Real apps = API project + class library + test projects в одном `.sln`.
