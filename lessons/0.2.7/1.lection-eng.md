# Upstream Tracking, push -u, pull --rebase

> Roadmap: `0.2.7` · Node: `0.2` — Git: branches and collaboration · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Explain **upstream** (tracking) branch configuration and where Git stores it.
- Use **`git push -u`** to publish a branch and set its upstream in one step.
- Run **`git pull --rebase`** to integrate remote changes with a linear history.
- Interpret **`git status`** output that references upstream ahead/behind counts.
- Configure **`branch.*.merge`** and **`branch.*.remote`** (conceptually) via `-u` and `git branch -u`.
- Decide when **rebase-on-pull** is appropriate vs merge-on-pull, connecting to `0.2.4`.

---

## Why This Matters

In `0.2.6` you learned fetch, pull, and push — the verbs that move commits between your laptop and `origin`. Every day you type `git pull` and `git push` on a feature branch without specifying `origin feature/login` each time. That convenience depends on **upstream tracking**: a local branch remembering which remote branch it corresponds to.

Without upstream, Git cannot tell you **"Your branch is ahead of 'origin/feature/x' by 2 commits"** — invaluable before opening a pull request. Without upstream, bare `git pull` and `git push` fail or require extra arguments. Without a team convention for **`pull --rebase`**, integrating latest `main` into your feature branch sprinkles merge commits through history, making `git log --oneline` noisy and `git bisect` harder — topics you touched in `0.2.3` and `0.2.4`.

Setting upstream with **`git push -u`** on first push is a one-second habit that prevents weeks of friction. Choosing **rebase when pulling** (or when updating your branch onto `main`) keeps feature history readable for reviewers in `0.2.8`. Middle developers configure defaults once and understand what happens when rebase rewrites commits you already pushed.

## Core Concepts

### What Is Upstream Tracking?

An **upstream branch** is the **default remote branch** paired with your local branch. Git stores this in config:

```ini
[branch "feature/login"]
    remote = origin
    merge = refs/heads/feature/login
```

When tracking is set, **`origin/feature/login`** (remote-tracking) is the upstream of local **`feature/login`**. Commands can use short forms:

```bash
git pull          # pull from upstream of current branch
git push          # push to upstream of current branch
git status        # ahead/behind vs upstream
```

View tracking:

```bash
git branch -vv
# * feature/login  abc1234 [origin/feature/login: ahead 2] Add JWT middleware
```

The bracket notation `[origin/feature/login: ahead 2]` is upstream relationship plus divergence summary.

### git push -u — Publish and Link

**`-u`** (or **`--set-upstream`**) on first push does two things: uploads your commits **and** writes the tracking config so future `git push` / `git pull` know where to go.

```bash
git push -u origin feature/login
```

Equivalent to push plus:

```bash
git branch -u origin/feature/login
```

If the remote branch did not exist, `-u` creates it (server permitting) and links. This is the standard first-push pattern after `0.2.6`.

**Already pushed without `-u`?** Fix later:

```bash
git branch -u origin/feature/login
# or
git push -u origin feature/login   # if upstream not set but branch exists on remote
```

### git pull --rebase — Integrate Without Merge Commits

Default **`git pull`** merges remote changes into your current branch — fine on `main` in some teams, often noisy on **feature branches**. **`git pull --rebase`** fetches, then **rebases** your local commits on top of the updated upstream tip — replaying your work as if you started from the latest remote state.

```bash
git checkout feature/login
git pull --rebase origin feature/login
# or, with upstream set:
git pull --rebase
```

Mechanically (from `0.2.3`): Git temporarily removes your unique commits, fast-forwards to match upstream, reapplies your commits one by one. Conflicts may appear per commit — resolve using `0.2.5`, then `git rebase --continue`.

**Updating feature branch with latest main** is a related pattern:

```bash
git fetch origin
git rebase origin/main        # while on feature/login
# or: git pull --rebase origin main  (only if upstream of feature is main — uncommon)
```

Most teams **rebase feature onto `origin/main`** before PR, not change feature's upstream to `main`.

### Ahead and Behind

After `git fetch`, status compares local branch tip to upstream:

- **Ahead** — you have commits not on remote (need `git push`).
- **Behind** — remote has commits you lack (need `git pull` or `pull --rebase`).
- **Diverged** — both (need integrate, then push).

```
        origin/feature ────●────●────●  (remote)
                           \
        local (diverged)    ●────●      (your commits)
```

Diverged often follows amending or rebasing **after** a previous push — rewrite rules from `0.2.4` and `0.1.10` apply; may need `push --force-with-lease` on **your** feature branch only.

### Configuring Defaults

Set global pull behavior (team policy varies):

```bash
git config --global pull.rebase true       # all pulls rebase by default
git config --global pull.rebase merges   # only rebase when merging remote-tracking into topic branch
```

Many teams use **`pull.rebase true`** for developers and protect `main` with required linear history or squash merges (`0.2.8`). Know your team's rule before assuming.

## Under the Hood

Upstream lives in **`.git/config`** under `[branch "name"]`. `git push -u` updates this after a successful push. `git branch -vv` reads `branch.*.merge` and resolves remote-tracking ref via `remote.*.fetch` refspec — typically `+refs/heads/*:refs/remotes/origin/*`.

`git pull --rebase` invokes `git fetch` then `git rebase` onto `@{upstream}` — the upstream of HEAD. The rebase machinery creates new commit objects (new SHAs) for replayed commits; old tips remain in reflog until garbage-collected.

`git status` ahead/behind counts commits reachable from one ref but not the other — purely graph comparison, no network.

## Syntax / Commands / API

```bash
# Set upstream
git push -u origin feature/x
git branch -u origin/feature/x
git branch --unset-upstream

# Pull with rebase
git pull --rebase
git pull --rebase origin feature/x
git fetch origin && git rebase origin/main    # update feature onto main

# Inspect
git branch -vv
git status
git rev-parse --abbrev-ref @{upstream}

# Rebase recovery
git rebase --abort
git rebase --continue
git rebase --skip

# Safer force after rebase (your feature branch only)
git push --force-with-lease
```

## Examples

### Example 1: First push on new feature

```bash
git switch -c feature/health
# ... commits ...
git push -u origin feature/health
git status
# "Your branch is up to date with 'origin/feature/health'."
```

### Example 2: Daily sync on tracked feature

Colleague pushed to same branch:

```bash
git fetch origin
git status    # behind 1
git pull --rebase
git push
```

Linear history: your commits sit on top of colleague's.

### Example 3: Rebase onto main before PR

```bash
git fetch origin
git rebase origin/main          # on feature branch
# fix conflicts if any
git push --force-with-lease       # if you had pushed feature before rebase
```

`--force-with-lease` refuses to overwrite remote if someone else pushed — safer than `--force`.

### Example 4: Forgot -u

```bash
git push origin feature/x    # works once
git push                     # fatal: no upstream branch
git branch -u origin/feature/x
git push                     # works
```

## Common Mistakes & Anti-patterns

**Never setting upstream** — endless `git push origin long/branch/name` copy-paste.

** `pull --rebase` on shared main** after local merge commits others expect — can confuse; follow team policy.

**Rebase after others pulled your branch** — rewrites public commits; coordinate or use merge instead (`0.2.4` golden rules).

**Blind `push --force`** after rebase — use **`--force-with-lease`** on solo feature branches only.

**Assuming `git pull` without args pulls `main`** — it pulls **current branch's upstream**, not always `main`.

**Global `pull.rebase true` without team agreement** — surprises colleagues expecting merge commits.

## Production & Real-World Notes

Hosting UIs show ahead/behind on PR branches relative to target branch. CI may require branch **up to date with `main`** before merge — triggers rebase or merge main into feature locally.

**GitHub "Update branch"** button merges or rebases via API — understand what your platform does. **Azure DevOps**, **GitLab** similar.

Document in team **CONTRIBUTING.md**: first push `-u`, feature sync `pull --rebase`, pre-PR `rebase origin/main`.

## Comparison / Trade-offs

| Approach | History shape | When |
|----------|---------------|------|
| `git pull` (merge) | Merge commits | Shared branches, beginners, explicit merge policy |
| `git pull --rebase` | Linear | Feature branches, pre-PR cleanup |
| `fetch` + manual rebase | Full control | Before risky integration |

Upstream tracking vs explicit `origin branch` every time: ergonomics vs explicitness in scripts (CI uses explicit refs).

## Quick Reference

```
UPSTREAM     = default remote branch for local branch (config)
git push -u  = push + set upstream
git pull --rebase = fetch + rebase onto upstream

git branch -vv     → tracking + ahead/behind
ahead  → push needed
behind → pull / pull --rebase needed

Feature onto main before PR:
  git fetch && git rebase origin/main
After rebase of pushed branch: push --force-with-lease
```

## Key Takeaways

1. **Upstream** links local branch to `origin/same-name` (usually).
2. **`git push -u`** on first push is standard professional habit.
3. **`git pull --rebase`** keeps feature history linear when integrating remote work.
4. **`git status` / `git branch -vv`** show ahead/behind vs upstream.
5. Rebase before PR onto **`origin/main`** is common; conflicts use `0.2.5` skills.
6. After rebasing pushed branches, prefer **`--force-with-lease`**, not bare `--force`.
7. Align with team policy on `pull.rebase` and shared branch rules (`0.2.4`).
