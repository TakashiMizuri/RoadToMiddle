# Exercises: Pull Request / Merge Request workflow

> Complete after lectures and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Open a draft PR
**Goal:** Walk through PR creation without merging.

**Steps:**
1. Branch `exercise/pr-draft` from current `main`, one small commit.
2. `git push -u origin exercise/pr-draft`.
3. Open **Draft PR** to `main` on your hosting platform.
4. Note source/target and files changed tab.

**Success criteria:** PR visible as draft; you can explain source vs target.

---

### E2. PR description template
**Goal:** Write reviewer-friendly metadata.

**Tasks:**
Write a PR body with sections:
- **Summary** (1–2 sentences)
- **How to test** (steps)
- **Related ticket** (fake ID ok)

**Success criteria:** Another developer could test without asking you in chat.

---

## Medium

### M1. Update PR with new commit
**Goal:** See live PR updates.

**Tasks:**
1. On open PR branch, add second commit addressing «review comment» (e.g. rename variable).
2. Push.
3. Observe PR commits list and CI re-run (if configured).

**Success criteria:** Both commits visible; you explain auto-update behavior.

---

### M2. Resolve conflict locally
**Goal:** Simulate «main moved» scenario.

**Setup:** PR branch behind `main` with conflicting edit (same line) in another clone on `main`.

**Tasks:**
1. UI or `git` shows conflict.
2. `git fetch && git rebase origin/main`, resolve, push.
3. Confirm PR mergeable.

**Success criteria:** Conflict resolved using `0.2.5` skills; no merge until you fixed locally.

---

### M3. Post-merge cleanup
**Goal:** Hygiene habit.

**Tasks:**
After PR merged (or simulate):
```bash
git switch main && git pull origin main
git branch -d exercise/pr-draft
git push origin --delete exercise/pr-draft
```

**Success criteria:** Local and remote feature branch removed; `main` current.

---

## Hard (stretch)

### H1. Compare merge strategies
**Goal:** Understand team policy trade-off.

**Tasks:**
1. Read your org/repo settings: allowed merge methods.
2. Write 5 sentences: which strategy your team uses and why (squash vs merge commit).

**Success criteria:** Connect to `0.2.2` fast-forward / merge commit concepts.

---

## Submission checklist

- [ ] Created draft or real PR
- [ ] Wrote description with how-to-test
- [ ] Pushed update commit to PR branch
- [ ] Performed post-merge cleanup (or documented steps)
