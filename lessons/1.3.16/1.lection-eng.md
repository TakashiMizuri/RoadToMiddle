# IDisposable, using, await using, Finalizers

> Roadmap: `1.3.16` · Node: `1.3` — C# Advanced Topics · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Explain **unmanaged vs managed** resources and why GC alone is insufficient.
- Implement **`IDisposable.Dispose`** correctly, including **`GC.SuppressFinalize`**.
- Use **`using`** and **`await using`** for deterministic cleanup.
- Implement **`IAsyncDisposable`** for async resource release (streams, DB).
- Describe **finalizers** (`~ClassName`), **finalization queue**, and why to avoid them.
- Apply the **dispose pattern** (base + derived) for classes owning handles.

---

## Why This Matters

File handles, socket connections, database connections, `MemoryPool` rentals, `HttpClient` streams — all hold resources **outside** the GC heap's simple «free when unreachable» model. Waiting for GC means **handle leaks**, **connection pool exhaustion**, and **locked files** on Windows.

`IDisposable` is the contract for **deterministic cleanup**: release now, not «eventually maybe.» `using` guarantees `Dispose` on success **and** exception paths. In ASP.NET Core, `DbContext`, streams, and locks rely on this daily.

Misusing disposal causes production incidents: `SqlConnection` leaks under load, temp files never deleted, **double dispose** crashes, or **never dispose** → native OOM. Middle developers must know sync vs async disposal and when finalizers still matter (rare).

## Core Concepts

### Managed vs Unmanaged Resources

**Managed** — memory tracked by GC (`new List<int>()`). **Unmanaged** — OS handles the CLR does not fully own: file descriptors, socket handles, GDI handles, raw `IntPtr` from P/Invoke.

Managed wrappers (`FileStream`, `SqlConnection`) hold unmanaged handles internally — they implement `IDisposable` to release handles **promptly**.

GC reclaims **managed object memory** when unreachable; it does **not** call `Dispose` automatically (unless finalizer path — see below).

### IDisposable

```csharp
public interface IDisposable
{
    void Dispose();
}
```

`Dispose` should:

- Release unmanaged resources.
- Optionally release expensive managed objects (e.g., dispose owned `HttpClient` if policy says so).
- Be **idempotent** — safe to call twice; use guard flag `_disposed`.

```csharp
public sealed class LogWriter : IDisposable
{
    private readonly Stream _stream;
    private bool _disposed;

    public void Dispose()
    {
        if (_disposed) return;
        _stream.Dispose();
        _disposed = true;
        GC.SuppressFinalize(this);
    }
}
```

### using Statement

```csharp
using (var writer = new LogWriter())
{
    writer.Write("...");
} // Dispose called here — even on exception
```

Compiler lowers to try/finally:

```csharp
var writer = new LogWriter();
try { ... }
finally { writer?.Dispose(); }
```

C# 8 **using declaration**:

```csharp
using var writer = new LogWriter();
// Dispose at end of enclosing block
```

Order: reverse of declaration at scope exit.

### IAsyncDisposable and await using

Async cleanup — flush network buffer, close DB async:

```csharp
public interface IAsyncDisposable
{
    ValueTask DisposeAsync();
}
```

```csharp
await using var conn = new AsyncDbConnection();
await conn.ExecuteAsync(...);
// await DisposeAsync at end
```

`await using` generates `ConfigureAwait(true)` by default in async methods — same finally semantics as sync `using`.

Types may implement **both** `IDisposable` and `IAsyncDisposable`. Prefer **`await using`** when `DisposeAsync` does I/O; sync `using` calls `Dispose()` which may block on async resources if implemented poorly — implement both correctly in BCL style.

### Finalizers (Destructors)

```csharp
~LogWriter()
{
    // cleanup as last resort
}
```

Syntax `~ClassName()` defines a **finalizer**. Object with finalizer gets **extra GC cycle**: when unreachable → **finalization queue** → `Finalize` runs on background thread → then memory reclaimed.

Problems:

- **Non-deterministic** delay (GC pressure dependent).
- **Expensive** — promotes objects, blocks generations.
- **No order** guarantees between finalizers.

**Rule:** avoid finalizers unless wrapping **raw unmanaged pointer** you must free if user forgets `Dispose`. If you implement finalizer, **must** implement full dispose pattern and `SuppressFinalize` in `Dispose`.

### Dispose Pattern (SafeHandle era)

For inheritance:

```csharp
public void Dispose()
{
    Dispose(disposing: true);
    GC.SuppressFinalize(this);
}

protected virtual void Dispose(bool disposing)
{
    if (_disposed) return;
    if (disposing)
    {
        // dispose managed IDisposable fields
    }
    // free unmanaged always
    _disposed = true;
}
```

Modern code often uses **`SafeHandle`** for unmanaged memory — reduces need for custom finalizers.

### What Not to Dispose

- **`HttpClient`** — typically **do not** dispose if shared/singleton (socket exhaustion if new per request); **do** dispose short-lived instances in isolated scenarios per Microsoft guidance evolution — know your DI lifetime.
- **`Task`** — not disposable.
- **`string`**, **`int`** — no handles.

**DbContext** — always dispose (scoped lifetime in ASP.NET).

## Under the Hood

### using IL

`using` → `try` + `finally` calling `Dispose`. Compiler may optimize if struct enumerator implements `Dispose`.

### Finalization pipeline

Object with finalizer: allocated → if no `SuppressFinalize`, registration with **FCALL** to GC — object **survives** one collection, promoted, finalized, then collected. This **extends lifetime** — the « resurrection » problem if `Finalize` stores `this` somewhere.

`GC.SuppressFinalize(this)` tells GC **skip finalizer** when user disposed correctly.

### Threading

`Dispose` should be **idempotent** but not necessarily thread-safe unless documented. BCL often documents «not thread-safe dispose.»

`DisposeAsync` — same; may use `SemaphoreSlim` internally in complex types.

### CA1816 / analyzers

FxCop/roslyn analyzers flag missing `SuppressFinalize`, public disposable fields, etc.

## Examples

### File read with using

```csharp
using var stream = File.OpenRead(path);
using var reader = new StreamReader(stream);
return await reader.ReadToEndAsync();
```

Nested usings — inner disposed first on exit.

### Custom wrapper

```csharp
public sealed class TempFile : IDisposable
{
    public string Path { get; }
    public TempFile() { Path = System.IO.Path.GetTempFileName(); }
    public void Dispose() => File.Delete(Path);
}
```

### Async SQL (conceptual)

```csharp
await using var cmd = connection.CreateCommand();
await cmd.ExecuteNonQueryAsync();
```

## Common Mistakes & Anti-patterns

**Omitting using on DbContext/Stream** — leaks until GC (handles may linger).

**Disposing shared singleton** — e.g., static `HttpClient` in wrong pattern.

**Throwing from Dispose** — masks original exception; swallow or log secondary.

**Finalizer calling `Dispose(true)` without suppress** — double free risk if not idempotent.

**Blocking in `Dispose()`** — call async path via `DisposeAsync` or `GetAwaiter().GetResult()` only as last resort (sync-over-async smell).

**Implementing empty Dispose «for convention»** on pure managed class — noise unless base/type requires it.

## Production & Real-World Notes

ASP.NET Core **DI scoped** disposes at end of request — `DbContext`, `IMemoryCache` scoped services. **Singleton** must not depend on scoped disposable without factory.

`IHostApplicationLifetime` shutdown — flush disposables on app stop.

Observability: connection pool metrics spike when `SqlConnection` not disposed — watch `pg_stat_activity`, SQL Server DMVs.

Libraries: **`Microsoft.Extensions.ObjectPool`** for reuse vs dispose churn.

## Comparison / Trade-offs

| Mechanism | Deterministic | Cost |
|-----------|---------------|------|
| `using` / `Dispose` | Yes | Developer discipline |
| Finalizer | No | GC overhead |
| `GC.Collect()` | Force full GC — avoid in app code | STW pauses |

## Key Takeaways

- GC frees **managed memory**, not OS handles — **`Dispose`** releases handles promptly.
- **`using`** / **`await using`** = try/finally + Dispose/DisposeAsync.
- **Finalizers** are slow, non-deterministic — last resort + dispose pattern.
- **`SuppressFinalize`** when user disposes correctly.
- **Idempotent** `Dispose`; don't dispose shared long-lived clients blindly.
- **DbContext**, streams, connections — always dispose in scoped lifetimes.
