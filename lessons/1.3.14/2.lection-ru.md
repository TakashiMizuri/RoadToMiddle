## Learning Objectives

## Why This Matters

Ты каждый день вызываешь `.Where()`, `.Select()`, `.Any()` — этих instance methods на `IEnumerable<T>` нет. Это **extension methods** в `System.Linq`. Без понимания compile-time rewrite они выглядят как «магия», добавленная к типу post factum.

Extension methods дают **additive API** для типов, которыми ты не владеешь (`string`, BCL interfaces, third-party DTO), без наследования и wrappers. Они же лежат в основе **fluent interfaces** и многих helper-API в ASP.NET.

Middle-разработчик должен знать: это **syntactic sugar над static calls** — это влияет на overload resolution, null behavior, тестируемость, stack traces и IL.

## Core Concepts

### Синтаксис и ограничения

Extension method — **static method** в **static class**, первый параметр с `this`:

```csharp
public static class StringExtensions
{
    public static bool IsNullOrEmpty(this string? value) =>
        string.IsNullOrEmpty(value);
}
```

Call site:

```csharp
"hello".IsNullOrEmpty();
```

Требования:

- Static class, static method.
- Только первый параметр с `this` — **extended type**.
- Нельзя заменить существующий instance method с той же сигнатурой (instance побеждает).
- Нет доступа к **private/protected** extended type (кроме `internal` в той же assembly).

### Compile-time lowering

Компилятор **не** внедряет методы в metadata целевого типа. Он переписывает вызов:

```csharp
name.IsNullOrEmpty();
// становится примерно:
StringExtensions.IsNullOrEmpty(name);
```

Если левая часть **null**, extension **всё равно вызывается** — **`this` может быть null**. В отличие от instance method, нет автоматического `NullReferenceException` до входа в метод (если внутри не разыменовать `this`).

```csharp
string? s = null;
var ok = s.IsNullOrEmpty();  // OK
```

Это отличается от `s.Length`, который бросает.

### Discovery: usings и namespaces

Extensions находятся через **using**, импортирующие namespace static class. Без `using System.Linq` — `items.Where(...)` не скомпилируется, хотя `IEnumerable<T>` в scope.

Extensions **не** наследуются между assemblies без явного import.

### Resolution: instance beats extension

Если у `Customer` есть `void Save()` и ты определил `Save(this Customer c)`, **instance method wins**. Extensions не **override** и не **hide** instance methods.

При нескольких подходящих extensions — правила **closer namespace** и **better conversion**; ambiguity → **CS0121**.

Generic extensions:

```csharp
public static T Id<T>(this T value) where T : struct => value;
```

`this` может быть generic; constraints работают как обычно.

### LINQ как канонический пример

```csharp
public static IEnumerable<TSource> Where<TSource>(
    this IEnumerable<TSource> source,
    Func<TSource, bool> predicate)
{
    foreach (var item in source)
        if (predicate(item)) yield return item;
}
```

`source.Where(p)` → `Enumerable.Where(source, p)`. Query syntax тоже desugar в extensions:

```csharp
from x in items where x > 0 select x * 2
// ≈ items.Where(x => x > 0).Select(x => x * 2)
```

`IQueryable<T>` extensions в `Queryable` с `Expression<Func<...>>` — тот же синтаксис, другой static class (см. `1.3.11`, `1.3.12`).

### Chaining и fluent APIs

Extension возвращает `this` или новый объект для цепочки:

```csharp
services.AddControllers()
        .AddJsonOptions(...)
        .AddApplicationPart(...);
```

Return type каждого extension определяет следующие доступные extensions — **discoverable API** без правки `IServiceCollection`.

## Under the Hood

### IL и metadata

Extension methods помечены в IL **`ExtensionAttribute`** (на method). IDE и analyzers читают это для instance-style IntelliSense. На runtime **нет special dispatch** — обычный `call` static method, receiver первым аргументом.

Decompile:

```csharp
// C#
list.Where(x => x > 1);

// IL (conceptual)
ldarg.0          // load list
ldftn <> lambda
call Enumerable.Where
```

Нет vtable slot на `List<T>` для `Where`.

### Overload resolution (упрощённо)

1. Собрать подходящие **instance** methods по имени.
2. Если не подошли — **extension** methods из usings (с generic inference).
3. Instance предпочтительнее extension при равной применимости.
4. Обычные правила betterness (лучшее преобразование типов).

Extensions участвуют в **lambda conversion** и **params** как static methods.

### Performance

Extension call = static call. **Нет allocation** на dispatch. LINQ operators могут аллоцировать (delegates, iterators) — это operator, не механизм extension.

JIT может **inline** маленькие static extension bodies.

```csharp
public static class Extensions
{
    public static ReturnType MethodName(this ExtendedType receiver, OtherArgs...)
}
```

| Rule | Detail |
|------|--------|
| Class | `static` |
| Method | `static`, first param `this T` |
| Visibility | `public` / `internal` |
| Null receiver | Allowed; check inside |

## Examples

### API на DTO, который нельзя менять

```csharp
public static class OrderDtoExtensions
{
    public static decimal TotalWithTax(this OrderDto dto, decimal rate) =>
        dto.Subtotal * (1 + rate);
}

var total = order.TotalWithTax(0.2m);
```

### Null-safe string helper

```csharp
public static string OrEmpty(this string? s) => s ?? string.Empty;
```

Работает при `s == null`.

### Разрешение ambiguity

```csharp
NamespaceA.IntExtensions.Format(42);
```

## Common Mistakes & Anti-patterns

**Extensions как наследование.** В reflection они не instance methods (кроме scan по ExtensionAttribute). Нужен правильный `using`.

**Тяжёлая логика в extensions.** Business rules — в services; extensions для мелких transforms.

**Extension на `this object`.** Засоряет IntelliSense — дисциплина namespaces.

**Ожидание throw на null.** Документируй null-tolerant behavior.

**Domain methods на своих типах** — лучше instance method; extensions для cross-cutting и third-party.

## Production & Real-World Notes

Папки `Extensions/`, **один static class на extended type**. `internal` extensions в той же assembly, что extended type.

Stack traces показывают **static method name** — полезно при debugging.

Versioning: добавление extension **binary compatible**; удаление/смена signature ломает как public API.

## Comparison / Trade-offs

| Mechanism | When | Limitation |
|-----------|------|------------|
| Instance method | Свой тип | Нельзя добавить к sealed/third-party |
| Extension method | Additive API, LINQ | No private; instance preferred |
| Wrapper | Нужно state | Extra type |
| Partial class | Тот же assembly | Не для `string`, interfaces |

```
static class X { static T M(this T self, ...) { } }

Instance > extension

this может быть null
```

## Key Takeaways

- Extensions — **static methods** с `this` sugar; rewrite только на compile time.
- **LINQ** = extensions на `IEnumerable` / `IQueryable`.
- **Instance methods win** при той же сигнатуре.
- **Null receiver** allowed — проектируй явно.
- **`using`** импортирует namespaces с extensions.
- IL — обычные static calls; **ExtensionAttribute** для tooling.
- Core behavior на типе; extensions — additive слой.
