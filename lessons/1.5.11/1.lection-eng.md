# Memory Leaks in Managed Code — Events and Static Roots

> Roadmap: `1.5.11` · Node: `1.5` — C#: .NET Runtime and Memory · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Explain how **managed memory leaks** occur despite GC (unreachable vs **reachable forever**).
- Trace **GC roots**: static fields, events, caches, timers, and **DI singletons** holding graphs alive.
- Diagnose **event handler leaks** (`+=` without `-=`) in WinForms, WPF, and custom pub/sub.
- Identify **static collections** and **AsyncLocal**/thread-static pitfalls that pin large object graphs.
- Use **dotnet-gcdump**, **dotMemory**, or **Visual Studio heap snapshots** to find retaining paths.
- Apply fixes: **weak events**, **unsubscribe**, **scoped lifetime**, **`WeakReference`**, cache eviction.

---

## Why This Matters

"C# has GC, so I cannot leak memory" is false. A **leak** in managed code means objects remain **reachable** from a **GC root** when you logically discarded them — the GC correctly keeps them alive. Classic symptoms: **RAM grows** over hours/days, **Gen2** never shrinks, **event handlers** fire on destroyed UI windows, **static caches** hold last million users' DTOs.

ASP.NET Core services registered as **singleton** that capture **scoped** dependencies cause subtle leaks and stale state. Desktop apps leak when **`button.Click += Handler`** outlives the form. Background **`Timer`** callbacks hold **`this`**. Middle developers must read **retention paths**, not only dispose handles.

This connects to **`1.5.6` GC roots** and **`1.5.10` disposal** — undisposed events are not handle leaks but **reference leaks** with similar production impact.

## Core Concepts

### Reachability Is the Whole Game

GC collects when no chain of strong references exists from **roots** (stack locals, statics, CPU registers, GC handles, finalizer queue, etc.) to the object. If **one** path remains, memory stays.

**Logical leak**: you believe nothing should reference `UserSession`, but **`EventHub.OnUpdate += session.Handle`** on a **static** event keeps `session` alive until unsubscribed.

### Event Handler Leaks

Delegates are **objects** holding **target reference** (`Target`) and **method**. Instance method handler:

```csharp
publisher.SomethingHappened += this.OnSomething;
```

`publisher` (directly or via multicast delegate chain) references **`this`** until **`-=`**. If `publisher` is **long-lived** (static, singleton, `AppDomain`-lifetime) and subscriber is **short-lived** (form, request-scoped object incorrectly wired), subscriber **never collects**.

**Fix:**

```csharp
publisher.SomethingHappened -= this.OnSomething;
// or in Dispose:
```

For UI, prefer **weak event pattern** or framework mechanisms (WPF weak events in some cases). **Lambda** handlers make unsubscribe harder unless you store the delegate:

```csharp
EventHandler handler = (_, _) => DoWork();
obj.Event += handler;
// later:
obj.Event -= handler;
```

Anonymous lambda without stored reference → **cannot unsubscribe** → leak.

### Static Roots

**Static fields** live for **AppDomain/process lifetime**:

```csharp
static readonly List<byte[]> _cache = new();
```

Every added array stays until removed. **Static event** is worst of both worlds:

```csharp
public static event Action<string>? GlobalNotify;
```

All subscribers pinned until `-=` or process exit.

**Lazy static singletons** holding **`HttpClient`**, **`MemoryCache`**, **`ConcurrentDictionary`** — fine if bounded; **unbounded** → leak by design.

### Timers, Threads, and Background Services

**`System.Threading.Timer`** and **`Task.Run`** closures capture **`this`**:

```csharp
_timer = new Timer(_ => Process(), null, 0, 1000);
```

Timer itself must be **`Dispose`d**; callback keeps target alive until timer disposed.

**`BackgroundService`** with channel subscription — ensure **CancellationToken** stops and handlers detach on shutdown.

### ASP.NET Core Lifetime Leaks

| Pattern | Problem |
|---------|---------|
| Singleton captures **DbContext** | Context never disposed; connection leak |
| Singleton **list** of per-request items | Grows forever |
| Static **ConcurrentBag** of logs | Unbounded |
| **IHostedService** event never unsubscribed | Handlers on dead scoped objects |

**Captive dependency** (singleton → scoped) fails **scope validation** at startup in Development — fix with **`IServiceScopeFactory`**:

```csharp
using var scope = _scopeFactory.CreateScope();
var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
```

### Caches and WeakReference

**MemoryCache** with **size limit** and **expiration** is bounded. **`Dictionary` as cache** without eviction leaks keys/values.

**`WeakReference<T>`** allows GC to collect target when memory pressure — good for **optional** caches; must handle **`TryGetTarget`** returning false.

**ConditionalWeakTable** — keys don't keep values alive (used internally for metadata).

### IDisposable vs Reference Leaks

Undisposed **`SqlConnection`** is handle + object leak until GC/finalizer. **Event subscription** keeps **whole object graph** reachable **immediately** — often worse because Gen2 holds megabytes of related data.

Fix both: **dispose resources**, **unsubscribe events**, **bound caches**.

## Under the Hood

**MulticastDelegate** stores **invocation list** — each entry holds strong ref to target. **Static event** field in publisher type is a **GC root**.

Diagnostic tools build **dominator tree**: "Who keeps 90% of `UserDto` instances alive?" → path **`StaticCache._items` → List → UserDto**.

**dotnet-gcdump** diff two snapshots after repro — objects that **should** be gone but grew in count.

**Finalizers** don't fix reference leaks — object is still **reachable** via event.

## Examples

### Leak

```csharp
public static class AppEvents
{
    public static event Action? UserLoggedOut;
}

public class SessionCleanup
{
    public SessionCleanup() => AppEvents.UserLoggedOut += OnLogout;
    void OnLogout() { /* ... */ }
    // never -= → every SessionCleanup instance stays alive
}
```

### Fix

```csharp
public void Detach() => AppEvents.UserLoggedOut -= OnLogout;
// call from Dispose or IRequestEnd
```

## Common Mistakes & Anti-patterns

**Static events for "global messaging"** without unsubscribe contract.

**Lambdas in `+=`** without stored delegate for `-=`.

**Singleton service** storing per-user state in **`List<T>`**.

**Believing ` = null` on local field** helps if static event still references `this`.

**WeakReference everywhere** — complexity; prefer clear ownership.

## Production & Real-World Notes

Long-running APIs: monitor **working set** and **Gen2 size** in production. **Load test** 24h — memory should plateau. **Ops** alerts on monotonic growth.

Code review: every **`+=`** on long-lived source needs **lifetime owner** documented.

Blazor Server circuits, SignalR hubs — watch **connection** lifetime vs **handler** registration.

## Comparison / Trade-offs

| Fix | When |
|-----|------|
| `-=` unsubscribe | Known pairing, short subscriber |
| `IDisposable` detach | Components with clear end |
| Weak events | Many short subscribers, UI frameworks |
| Scoped not singleton | Per-request state |
| Bounded cache | True need for retention |

## Quick Reference

- Leak = **reachable** but **unwanted** strong chain from root.
- **Events** = delegate holds **target**.
- **Static** = process lifetime root.
- Fix: **unsubscribe**, **scope**, **bound**, **weak** (last resort).

---

## Key Takeaways

- Managed leaks are **reference retention**, not GC failure.
- **Static events + instance handlers** = classic desktop leak.
- **Singleton capturing scoped** = classic ASP.NET leak.
- **Diagnostics**: heap diff, retention paths.
- Every **`+=`** needs a **lifecycle plan** for **`-=`**.
