# stackalloc — аллокация на stack

> Roadmap: `1.5.13` · Node: `1.5` — C#: .NET Runtime и память · Depth: понимание

## Learning Objectives

- Объяснить **`stackalloc`** как allocation из **current stack frame** → **`Span<T>`**.
- Сравнить с heap и **ArrayPool** для короткоживущих буферов.
- Знать лимиты stack и риск **StackOverflowException**.
- Безопасно использовать в hot path без return Span наружу.
- **CA2014** — stackalloc в loops.

---

## Why This Matters

Малые буферы (parse header, format key) — **zero GC** через stack. Ошибка: **stackalloc в loop** или огромный size → crash. Threshold: stack для tiny, **ArrayPool** для large (`1.5.14`).

---

## Core Concepts

```csharp
Span<byte> buffer = stackalloc byte[128];
```

Память умирает с **return** метода — **не return Span**.

**Когда да:** fixed small bound, один method.**Когда нет:** user size, loops, outlive method.

Stack ~**1 MB**/thread (OS-dependent). **`stackalloc byte[512]`** ok; **100_000** — почти всегда нет.

vs **ArrayPool**: stack = method lifetime; pool = shared, Return обязателен.

---

## Common Mistakes

Return Span. User-controlled size. stackalloc в цикле. Premature optimization.

---

## Key Takeaways

- stackalloc + Span = method-scoped, no GC.
- Cap size; ArrayPool для большого.
- Связь **`1.5.12`** Span.
