# Safe vs Idempotent Methods — The Matrix

> Roadmap: `0.3.5` · Node: `0.3` — HTTP and the Web · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Define **safe** and **idempotent** methods in HTTP semantics precisely, not colloquially.
- Build and interpret the **safe × idempotent matrix** for all common methods.
- Explain why **safe** matters to caches, crawlers, and prefetchers.
- Explain why **idempotent** matters to **retries**, load balancers, and client libraries.
- Distinguish HTTP idempotency from **application-level** guarantees (Idempotency-Key).
- Predict behavior of intermediaries (CDN, proxy) based on method properties.
- Apply the matrix when designing ASP.NET Core APIs and outbound `HttpClient` retry policies.

---

## Why This Matters

In `0.3.4` you learned what each HTTP method is *for*. Two cross-cutting properties — **safe** and **idempotent** — govern how the *entire ecosystem* treats requests: browsers, CDNs, reverse proxies, retry middleware, and your own `HttpClient` Polly policies. They are not interchangeable words. **Safe** means “read-only intent” — intermediaries may replay without asking. **Idempotent** means “same effect if applied once or many times” — clients may retry after ambiguous network failure.

Confusing them causes real incidents. A team retries POST on timeout and doubles payments. A CDN caches a response to a misimplemented GET that mutates inventory. A load balancer replays a non-idempotent request to a second server during failover. Middle developers must internalize the matrix below the way they internalize `async/await` rules: not as trivia, but as **contracts** between client, server, and infrastructure.

This lesson formalizes the matrix referenced throughout node `0.3` and AGENTS.md HTTP section. It connects forward to status codes (`0.3.6` — when to retry 503), headers (`Cache-Control` on GET), and API design (`Idempotency-Key` on POST).

## Core Concepts

### Safe Methods

**Definition (RFC 9110):** A method is **safe** if its **intended** semantics are read-only — the request does not alter server state relevant to the user. Safe requests are **information retrieval** only.

Safe methods: **GET, HEAD, OPTIONS, TRACE** (TRACE rare/disabled in production).

**Important nuance:** “Safe” describes **intent**, not a guarantee your buggy server won't write logs or update `Last-Accessed`. It also does not mean “no side effects anywhere” — logging and analytics are gray areas, but **business state** must not change. A safe GET must not charge a credit card or delete a row.

**Why infrastructure cares:**

- **Browser prefetch** and **link preview** bots may GET URLs without explicit user click.
- **CDNs and shared caches** may store GET/HEAD responses and serve them to other users when headers allow.
- **Crawlers** follow GET links recursively.

If GET mutates state, you get CSRF via `<img src="...">`, accidental double actions, and stale cached wrong data.

### Idempotent Methods

**Definition:** A method is **idempotent** if the **intended effect** on the server is the same whether you perform it **once or many times** with the same request. Repeated identical idempotent requests should not multiply outcome (second DELETE does not delete “more”; second PUT with same body leaves resource in same state).

Idempotent methods: **GET, HEAD, PUT, DELETE, OPTIONS, TRACE**.

**Not idempotent by default:** **POST**, **PATCH** (often idempotent in practice if patch is absolute set, but not guaranteed by spec for all patch types).

**Why clients care:**

Network fails mid-request — client cannot know if server processed the request (**ambiguous outcome**). For **idempotent** methods, automatic **retry** is safe: send again. For **non-idempotent** POST, blind retry may duplicate — need **Idempotency-Key**, deduplication store, or user confirmation.

**Idempotent ≠ safe:** PUT and DELETE are idempotent but **not safe** — they change state. You can retry them safely regarding *duplication*, not regarding *read-only*.

**Safe ⇒ idempotent:** All safe methods are idempotent in HTTP design (reading twice doesn't change more than reading once). The converse is false.

### The Matrix

| Method | Safe | Idempotent | Typical retry without extra keys | Cached by shared cache (if headers allow) |
|--------|------|------------|----------------------------------|-------------------------------------------|
| GET | Yes | Yes | Yes | Yes |
| HEAD | Yes | Yes | Yes | Yes |
| OPTIONS | Yes | Yes | Yes | Rarely |
| POST | No | No | **No** — risk duplicate | No |
| PUT | No | Yes | Yes | No |
| PATCH | No | Often* | Caution | No |
| DELETE | No | Yes | Yes | No |

*PATCH: applying identical JSON Patch twice may be idempotent; incremental counters may not be.

Visual grouping:

```
                    IDEMPOTENT
                 Yes          No
            ┌─────────┬─────────┐
      Yes   │ GET     │ (none)  │  SAFE
            │ HEAD    │         │
            │ OPTIONS │         │
            ├─────────┼─────────┤
      No    │ PUT     │ POST    │  NOT SAFE
            │ DELETE  │ PATCH   │
            └─────────┴─────────┘
```

### POST and Application-Level Idempotency

HTTP says POST is **not** idempotent. Payment APIs still need retries. Pattern:

1. Client generates unique **`Idempotency-Key`** header (UUID).
2. Server stores `(key → response)` for 24h.
3. Duplicate POST with same key returns **same** stored response without re-executing charge.

This is **application** idempotency, not changing HTTP's classification of POST. Stripe, many internal gateways use this.

Similarly, **at-least-once delivery** from message queues into your API requires deduplication keys or natural idempotency (UPSERT by business id).

### PUT and DELETE Idempotency Details

**PUT** `/users/42` with body `{name:"Ann"}` sent twice → user name still Ann. First PUT creates or replaces; second identical PUT same result.

**DELETE** `/users/42` twice → first deletes; second typically **404 Not Found** or **204** — resource still absent; **not** “double delete harm.” Server should not error in way that implies duplicate damage.

**GET** twice → two reads; idempotent because server state intended unchanged (aside from benign logging).

### PATCH Edge Cases

PATCH `{ "loginCount": "+1" }` is **not** idempotent — two requests increment twice. PATCH `{ "status": "active" }` setting absolute value **is** idempotent if already active.

Design PATCH as **absolute field replacement** when possible; avoid relative increments in PATCH without version tokens.

### Intermediaries and Method Properties

**Shared caches** (RFC 9111): primarily cache **GET** and **HEAD** responses with valid cache headers. Must not cache POST responses as generic representations (exceptions rare).

**Proxies** may retry **idempotent** requests on connection reset to origin — another reason POST retry is dangerous.

**Browser back/forward** may resubmit POST with confirmation prompt — not idempotent UX.

**Load balancer retry:** Some configure retry only for GET/HEAD/PUT/DELETE on idempotent failure — never blind POST retry.

## Under the Hood

### How ASP.NET Core and HttpClient Treat Retries

Default `HttpClient` **does not** auto-retry POST. Libraries like Polly require explicit policy: `WaitAndRetry` on `GET` safe; for POST wrap with idempotency key check.

`[HttpGet]` endpoints accidentally performing `SaveChanges()` on every hit break safe semantics — cache or prefetch triggers DB writes.

### Conditional Requests and Safe Methods

GET with **`If-None-Match: "etag"`** → **304 Not Modified** — still safe, still idempotent; reduces body transfer (`0.3.7`).

### Idempotency in Distributed Systems

Two app servers behind LB: client retries PUT during failover — both may process; idempotent PUT converges to same resource state. Retried POST without key → two rows in `Orders` table.

**Database unique constraints** (e.g. on `Idempotency-Key`) enforce deduplication at persistence layer.

### TRACE and Safe Methods

TRACE included in matrix for completeness; disabled on most production servers (XST security). OPTIONS remains for CORS.

## Syntax / Commands / API

**Retry policy sketch (conceptual Polly):**

```csharp
// Safe for idempotent GET only — illustrative
Policy.Handle<HttpRequestException>()
    .RetryAsync(3)
    .ExecuteAsync(() => httpClient.GetAsync(url));

// POST — do NOT blindly retry; use idempotency key + server dedup
```

**Idempotency-Key header:**

```http
POST /api/payments HTTP/1.1
Idempotency-Key: 7c9e6679-7425-40de-944b-e07fc1f90ae7
Content-Type: application/json

{"amount":100,"currency":"USD"}
```

ASP.NET Core filter reads key, checks store, short-circuits duplicate.

## Examples

**Example 1 — Safe violation.** `GET /api/cart/add?sku=99` adds to cart. Googlebot crawls link → phantom cart lines. Fix: POST `/api/cart/items` with body.

**Example 2 — Idempotent retry.** Mobile app PUT profile, connection drops after server saved but before response. Client retries same PUT → OK, same profile, no duplicate users.

**Example 3 — POST duplicate.** Client POST order, timeout, retry without key → two orders. Fix: Idempotency-Key header stored server-side.

## Common Mistakes & Anti-patterns

**“GET is safe enough for our admin panel”** — crawlers and CSRF don't know admin context in URL alone; use POST with antiforgery.

**Retry POST in default HttpClient wrapper** — duplicate side effects.

**Assuming PATCH idempotent** without analyzing patch document semantics.

**Caching POST responses** in custom CDN rules — breaks semantics.

**Confusing safe with authenticated** — safe does not mean “public”; auth still required.

## Production & Real-World Notes

Document method properties in API guidelines: public APIs often state “GET/PUT/DELETE idempotent; POST requires Idempotency-Key.”

**Monitoring:** spike in duplicate order IDs correlates with client retry after gateway timeout — fix idempotency, not just increase timeout.

**Testing:** integration tests send same DELETE twice expect 404/204 second time; same POST twice without key expect two resources (document as bug if business forbids).

**GraphQL** mutations via POST — all non-idempotent at HTTP layer; idempotency inside mutation design.

## Comparison / Trade-offs

**Idempotent PUT create vs POST create** — PUT idempotent when client owns URI (`PUT /files/{clientGeneratedId}`); POST when server assigns.

**At-least-once vs exactly-once** — HTTP gives at-least-once on network; exactly-once needs application dedup (keys, transactions).

**Safe read via POST** — complex queries sometimes POST `/search` — **not safe**, not cacheable; acceptable trade for body size/privacy.

## Quick Reference

| Property | Question it answers |
|----------|---------------------|
| Safe | Will replay change server state? (should be No) |
| Idempotent | Will retry duplicate effect? (should be No for duplicates) |

**Safe:** GET, HEAD, OPTIONS  
**Idempotent not safe:** PUT, DELETE  
**Neither:** POST (PATCH: caution)

---

## Key Takeaways

- **Safe** = read-only intent; governs cache, prefetch, CSRF surface.
- **Idempotent** = same effect once or N times; governs **retry** after network failure.
- All **safe** methods are **idempotent**; PUT/DELETE idempotent but not safe.
- **POST** not idempotent — use **Idempotency-Key** for retry-safe commands.
- **PATCH** idempotency depends on patch semantics — prefer absolute updates.
- Intermediaries (CDN, proxy, LB) rely on these properties — broken semantics → production bugs.
- Matrix is a **contract** with the whole HTTP ecosystem, not just your controller code.
