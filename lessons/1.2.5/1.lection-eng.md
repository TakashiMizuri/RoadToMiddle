# Encapsulation — Why Public Fields Are Bad

> Roadmap: `1.2.5` · Node: `1.2` — C#: OOP · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Define **encapsulation** as bundling data with behavior and hiding implementation.
- Explain why **`public` fields** break invariants and couple callers to layout.
- Contrast **properties**, **methods**, and **factories** as controlled access mechanisms.
- Identify **leaky encapsulation** through mutable collections and `protected` fields.
- Relate encapsulation to **maintainability**, **testing**, and **API versioning**.
- Apply practical patterns: **`private set`**, **`init`**, **guard clauses**, **Tell, Don't Ask**.

---

## Why This Matters

You now know classes (`1.2.1`), properties (`1.2.2`), constructors (`1.2.3`), and visibility (`1.2.4`). Encapsulation is the **design principle** tying them together: objects enforce their own rules; callers use a **stable surface**.

Public fields look convenient — fewer lines, JSON "just works" on simple DTOs in tutorials. In production they cause **silent corruption**: negative balances, empty emails, orders with zero lines shipped, shared list mutation from unexpected callers. Once a field is public, **every external assignment** becomes part of your semver contract. Renaming or splitting storage becomes a breaking change.

Middle developers defend invariants at **object boundaries**, not in every controller action. That is how domain models survive multiple entry points — REST, background jobs, message consumers — without duplicating validation logic.

## Core Concepts

### What Encapsulation Means

**Encapsulation** hides **how** state is stored and updated while exposing **what** the object can do. External code depends on **operations**, not memory layout.

A well-encapsulated `BankAccount` does not expose `_balance` for arbitrary writes. It exposes **`Deposit`**, **`Withdraw`**, and a read-only **`Balance`**. The object **owns** its rules: no overdraft, audit trail, currency rounding — all centralized.

This pairs with **access modifiers**: `private` storage, selective `public`/`internal` behavior (`1.2.4`).

### Public Fields: Direct Exposure

```csharp
// Problem shape
public class Account
{
    public decimal Balance;
}
```

Any caller:

```csharp
account.Balance = -1_000_000m;
```

No hook runs — no validation, logging, or domain event. The type **cannot protect itself**. Refactoring to computed balance from transactions requires changing **every assignment site** in the codebase — a combinatorial search.

Compare:

```csharp
public class Account
{
    private decimal _balance;
    public decimal Balance => _balance;

    public void Deposit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentOutOfRangeException();
        _balance += amount;
    }
}
```

Callers keep compiling while internals evolve — add fees, concurrency checks, events — behind the same methods.

### Properties Are Not Magic — They Are a Seam

Auto-properties `{ get; set; }` are **better syntax** than fields but **still expose unrestricted mutation** if public set:

```csharp
public string Email { get; set; }   // still leaky if no validation
```

Encapsulation requires **deciding who may change state and when**:

- **`{ get; private set; }`** — external read, internal write paths.
- **`{ get; init; }`** — immutable after construction (`1.2.2`).
- **Full property** — validation in setter.
- **No setter** — mutation only via methods expressing domain verbs.

The seam lets you add validation **once** instead of trusting every caller.

### Tell, Don't Ask

**Tell, Don't Ask** avoids external code pulling state, deciding, pushing back:

```csharp
// Ask-heavy (fragile)
if (order.Total > 1000 && order.Customer.IsVip)
    order.ApplyDiscount(0.1m);

// Tell (encapsulated)
order.ApplyBestDiscount();
```

Logic migrates **into** the type that owns the data. Fewer bugs when new discount rules appear — one place to change.

Not every line must be a method — but **invariants** should not depend on callers doing the right sequence of reads/writes to public fields.

### Leaky Encapsulation Beyond Public Fields

**Mutable collection exposed:**

```csharp
public List<OrderLine> Lines { get; set; } = new();
```

Callers can `Lines.Clear()` or replace the list, bypassing `AddLine` pricing rules.

Mitigations: private list, expose `IReadOnlyList<OrderLine>`, methods `AddLine`, or init-only collection with controlled mutation.

**`protected` fields** in base classes let subclasses tweak base state directly — fragile base class (`1.2.11`). Prefer **`protected` methods** or controlled hooks.

**Anemic domain model** — classes with only public data bags, all logic in services — is organizational encapsulation failure; objects become DTOs with no behavior.

### Encapsulation vs Performance Myth

"Properties are slower than fields" — JIT often inlines simple auto-properties to field access. Even when slightly slower, **correctness and maintainability** dominate business apps. Hot paths can be measured; default is **encapsulate first**.

### DTOs and Boundaries

At **system boundaries** (HTTP JSON, database rows), anemic data shapes are acceptable — **DTOs** are not domain entities. The mistake is treating **every** class as a DTO with public fields/properties and keeping **domain rules** scattered in controllers.

Map: `CreateOrderDto` → construct **`Order`** via factory/constructor enforcing invariants → persist.

## Under the Hood

### Compiler and Metadata View

Fields and properties compile differently. A **field** is a simple metadata slot with direct IL **`ldfld`/`stfld`**. A **property** emits **`get_X`/`set_X` methods** (or `.get`/`.set` accessors) — invocation goes through methods even when source looks like field access.

That indirection is what enables **breakpoints in setters**, **logging**, **INotifyPropertyChanged**, and **EF Core** change tracking hooks. Public fields skip that seam — frameworks cannot intercept uniformly.

### Serialization and Reflection

System.Text.Json by default binds **public properties** with public getters (and setters/init). Public fields require **`[JsonInclude]`** — a signal they are uncommon in idiomatic C# models.

Reflection can read private fields anyway — encapsulation is **not security** against malicious code in the same process. It is **engineering discipline** against accidental coupling across teams and time.

### Versioning and Binary Compatibility

Changing a **public field** type or name breaks **all compiled callers** at metadata level. Changing private field while preserving property signature is **source-compatible** for consumers of your library. Encapsulation protects **evolution** of types shipped in NuGet packages.

### Identity and Encapsulation

For entities (`1.2.1`), encapsulation preserves **identity invariants** — an `OrderId` set once, status transitions only through defined methods (`Submit`, `Ship`), preventing "teleport" to Shipped without payment. EF Core tracks property changes; bypassing properties with public fields can **break change tracking** expectations.

## Syntax / API

```csharp
// Prefer
private decimal _balance;
public decimal Balance => _balance;
public void Deposit(decimal amount) { ... }

// Immutable surface
public required Guid Id { get; init; }
public IReadOnlyList<Line> Lines => _lines.AsReadOnly();

// Avoid in domain core
public decimal Balance;           // public field
public List<Line> Lines { get; set; } = new();  // leaky collection
```

## Examples

**Example 1 — invariant in one place:**

```csharp
public sealed class Email
{
    public string Value { get; }
    public Email(string value)
    {
        if (!value.Contains('@')) throw new ArgumentException();
        Value = value.Trim().ToLowerInvariant();
    }
}
```

No setter — invalid email cannot exist in instance.

**Example 2 — collection encapsulation:**

```csharp
public class ShoppingCart
{
    private readonly List<Item> _items = new();
    public IReadOnlyList<Item> Items => _items;

    public void Add(Item item)
    {
        if (item is null) throw new ArgumentNullException();
        _items.Add(item);
    }
}
```

External code cannot replace `_items` reference without reflection.

**Example 3 — state transition methods:**

```csharp
public class Job
{
    public JobStatus Status { get; private set; } = JobStatus.Pending;

    public void Start()
    {
        if (Status != JobStatus.Pending)
            throw new InvalidOperationException();
        Status = JobStatus.Running;
    }
}
```

Status machine rules live inside `Job`, not at every caller.

## Common Mistakes & Anti-patterns

**Public fields "for JSON".** Use properties, `[JsonConstructor]`, records, or explicit DTO types.

**Auto-property `{ get; set; }` on entity** without rules — same leak as field.

**Getter returns mutable internal collection** — caller mutates backing store.

**God service class** mutating public DTO graphs — no object owns invariants.

**Over-encapsulation** — private everything with 40 trivial getters; balance behavior and data.

## Production & Real-World Notes

Code reviews flag public fields in domain layers — StyleCop/analyzers CA1051 (do not declare visible instance fields).

Legacy interop (P/Invoke structs) sometimes requires public fields — isolate at boundary, map inward.

Immutability (`init`, records) strengthens encapsulation for commands/events in CQRS pipelines.

## Comparison / Trade-offs

| Approach | Coupling | Invariant enforcement | Refactor freedom |
|----------|----------|----------------------|------------------|
| Public fields | Highest | None | Lowest |
| Public `{ get; set; }` | High | Optional in setter | Medium |
| Methods + private state | Lower | Strong | High |
| Immutable (`init`/ctor) | Lower | Strong at creation | High |


## Key Takeaways

- **Encapsulation** protects invariants by controlling mutation paths.
- **Public fields** couple callers to layout and skip validation hooks.
- **Properties and methods** provide a stable, evolvable surface.
- **Mutable collection exposure** is a common leak — guard with private storage.
- Encapsulation is **maintainability**, not runtime security against reflection.
- Use **DTOs at boundaries**, **rich types** in domain core.
