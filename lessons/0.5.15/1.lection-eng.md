# data-* Attributes and Custom Elements (Concept)

> Roadmap: `0.5.15` · Node: `0.5` — HTML · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Use **`data-*` attributes** to store custom metadata in valid HTML.
- Read and write `data-*` values via **`element.dataset`** in JavaScript.
- Explain why `data-*` is preferred over non-standard arbitrary attributes.
- Describe **Web Components** at a high level: Custom Elements, Shadow DOM, HTML templates.
- Distinguish **semantic HTML + data-*** from **custom element tags** and know when each fits.

---

## Why This Matters

HTML gives you rich semantic elements — `<article>`, `<button>`, `<form>` — but applications also need to attach **private metadata** to nodes: product SKU on a row, tab panel ID, JSON config for a widget, or Cypress test hooks. Before `data-*`, developers abused `class="sku-12345"` or invalid attributes like `sku="12345"`, breaking validation and confusing CSS selectors.

The **`data-*` naming convention** is the standards-track way to extend markup without polluting the global attribute namespace. In React, you will use `data-testid`, `data-active`, or pass server values into components — under the hood many patterns mirror `data-*` or props that become attributes on DOM nodes. Understanding `dataset` bridges static HTML (`0.5.x`) to DOM scripting and later **Custom Elements**, the browser-native path to reusable encapsulated widgets.

Custom Elements let you register `<user-card>` as a first-class tag with lifecycle callbacks. You may not author Web Components daily in a React codebase, but you will **consume** them (design systems, video players, chart libraries) and debug them in DevTools. Conceptual clarity prevents fighting the platform.

## Core Concepts

### The data-* Attribute Pattern

Any attribute starting with **`data-`** (followed by at least one character, no uppercase in HTML) is a **custom data attribute**. The HTML spec guarantees browsers will not conflict with future standard attributes. Names are **case-insensitive in HTML** but **`dataset` exposes them in camelCase**.

```html
<article
  id="product-42"
  data-sku="WDG-42"
  data-stock="12"
  data-featured="true">
  <h2>Widget Pro</h2>
</article>
```

Rules worth memorizing:

- Prefix **`data-`** is mandatory.
- No uppercase letters in the HTML attribute name (`data-productId` is invalid; use `data-product-id`).
- Values are always **strings** in the DOM — convert numbers and booleans in JavaScript.
- Attributes are meant for **metadata**, not hidden JSON stores for huge configs (size and maintainability limits).

### Reading Data with dataset

The **`HTMLElement.dataset`** property provides a **`DOMStringMap`**: each `data-x-y` becomes `element.dataset.xY`.

```javascript
const article = document.querySelector('#product-42');
console.log(article.dataset.sku);      // "WDG-42"
console.log(article.dataset.stock);    // "12" — string!
console.log(article.dataset.featured); // "true"

const stock = Number(article.dataset.stock);
article.dataset.featured = 'false'; // updates attribute data-featured="false"
```

Setting `dataset` keys creates or updates corresponding attributes. Removing a property removes the attribute. For complex nested data, prefer **`JSON.stringify`** in one attribute sparingly or fetch from API — and parse with try/catch.

```html
<button data-action="add-to-cart" data-product-id="42">Add</button>
```

```javascript
document.body.addEventListener('click', (e) => {
  const btn = e.target.closest('[data-action]');
  if (!btn) return;
  if (btn.dataset.action === 'add-to-cart') {
    const id = btn.dataset.productId;
    // call cart API
  }
});
```

This **event delegation** pattern scales well: one listener, behavior driven by `data-action` markers — common in vanilla JS and sometimes replicated in React with `data-*` on native elements.

### data-* vs id, class, and ARIA

| Mechanism | Best for |
|---|---|
| `id` | Unique identifier, fragment links, label associations |
| `class` | CSS styling hooks, multiple elements |
| `data-*` | JS behavior config, test hooks, non-visible metadata |
| ARIA attributes | Accessibility semantics when native roles insufficient |

Do not store styling-only information in `data-*` if `class` suffices. Do not use `data-*` to replicate ARIA roles (`data-role="button"` on `<div>`) — use `<button>` (`0.5.11`).

### Validation and Selectors

CSS can target data attributes:

```css
tr[data-status="shipped"] { opacity: 0.7; }
```

```javascript
document.querySelectorAll('[data-featured="true"]');
```

Invalid non-`data` custom attributes fail **HTML validation** (`0.5.16`) and may be stripped by parsers. Stick to `data-*` for custom keys.

### Custom Elements — Conceptual Overview

**Custom Elements** are part of **Web Components**, a suite of standards:

1. **Custom Elements** — define new tag names and lifecycle
2. **Shadow DOM** — encapsulated subtree and styles
3. **HTML Template** — `<template>` inert markup cloned into DOM

Register a custom element:

```javascript
class UserCard extends HTMLElement {
  connectedCallback() {
    const name = this.getAttribute('name') ?? 'Guest';
    this.textContent = `Hello, ${name}`;
  }
}
customElements.define('user-card', UserCard);
```

```html
<user-card name="Ada"></user-card>
```

**Autonomous custom elements** extend `HTMLElement` directly (new tag). **Customized built-in elements** extend specific classes (`class ExpButton extends HTMLButtonElement`) — less common, Safari historically limited support for customized built-ins.

Lifecycle hooks (conceptual):

- **`connectedCallback`** — element inserted into document
- **`disconnectedCallback`** — removed
- **`attributeChangedCallback`** — observed attributes change
- **`adoptedCallback`** — moved to new document (rare)

Custom elements work with **`observedAttributes`** static getter to react to attribute updates — similar mental model to React props changing on a component.

### Shadow DOM (Brief)

Shadow DOM attaches a hidden subtree with **scoped CSS**. External page styles do not leak in; internal styles do not leak out. Design systems ship `<ds-modal>` with Shadow root so `.button` inside does not fight Bootstrap `.button` globally.

You inspect shadow roots in DevTools (flattened view). React 19+ and other frameworks may coexist; many teams still choose React components over hand-rolled Web Components for app UI — but platform components remain valuable for **cross-framework widgets**.

### When data-* vs Custom Elements

**Use `data-*`** when:

- Enhancing existing semantic HTML with JS sprinkles
- Metadata for CSS/JS/tests on standard elements
- React/Vue app where components are framework-level

**Consider Custom Elements** when:

- Publishing a **framework-agnostic** widget consumed by multiple stacks
- Encapsulation via Shadow DOM is a hard requirement
- Browser-native lazy-defined elements fit marketing embed script

Many codebases combine both: `<tabs-widget data-default="settings">` before `customElements.define` upgrades behavior.

## Under the Hood

`data-*` attributes live in the DOM as **`Attr` nodes** on the element. `dataset` is a live view — mutating one side updates the other. The HTML parser lowercases attribute names; `data-product-id` maps to `dataset.productId` via ASCII lowercase and hyphen-to-camel algorithm defined in the spec.

Custom Elements registry is global per **window/realm**. `customElements.define` throws if name already registered or contains uppercase. Names must contain a **hyphen** (e.g. `x-`) to avoid collision with future HTML tags. Parser sees unknown tag `user-card` as **`HTMLUnknownElement`** until definition runs — scripts should defer interaction until `customElements.whenDefined('user-card')`.

Upgrade timing: **autonomous** elements defined before DOM parse upgrade immediately; deferred definitions upgrade nodes already in tree when `define` runs. This matters for FOUC and hydration mismatches — a preview of challenges when mixing Web Components with SSR React.

## Syntax / Commands / API

HTML:

```html
<div data-user-id="7" data-role="admin"></div>
<template id="row-tpl">
  <tr data-editable="true"><td></td></tr>
</template>
<product-badge data-sku="ABC" data-qty="3"></product-badge>
```

JavaScript:

```javascript
el.dataset.userId = '8';
delete el.dataset.role;
const id = el.getAttribute('data-user-id'); // alternative explicit read
await customElements.whenDefined('product-badge');
```

Custom element skeleton:

```javascript
class ProductBadge extends HTMLElement {
  static observedAttributes = ['data-sku'];
  attributeChangedCallback(name, oldVal, newVal) { /* update UI */ }
}
customElements.define('product-badge', ProductBadge);
```

Note: reflecting `data-sku` as observed attribute is valid; some teams prefer unprefixed observed attrs (`sku`) on custom tags while keeping `data-*` on native elements — consistency matters within a project.

## Examples

### Tabs with data-* Only

```html
<div class="tabs" data-active-tab="profile">
  <button type="button" data-tab-target="profile" aria-selected="true">Profile</button>
  <button type="button" data-tab-target="billing">Billing</button>
  <section data-tab-panel="profile">...</section>
  <section data-tab-panel="billing" hidden>...</section>
</div>
```

JS reads `data-tab-target` clicks, toggles `hidden` and `aria-selected`. No custom tag required — progressive enhancement friendly.

### Minimal Custom Element

```javascript
class FormattedDate extends HTMLElement {
  connectedCallback() {
    const iso = this.getAttribute('datetime');
    if (iso) {
      this.textContent = new Date(iso).toLocaleDateString();
    }
  }
}
customElements.define('formatted-date', FormattedDate);
```

```html
<p>Ships on <formatted-date datetime="2026-07-15"></formatted-date>.</p>

```

Works without JavaScript framework on static pages; enhances in place.

## Common Mistakes & Anti-patterns

**Storing large JSON blobs in data attributes** — hard to maintain, escapes painful, blows HTML size. Fetch or use `<script type="application/json">` pattern instead.

**Using camelCase in HTML** (`data-productId`) — invalid; use kebab-case.

**Treating dataset values as numbers/booleans without parsing** — logic bugs (`if (dataset.enabled)` true for `"false"` string).

**Custom element name without hyphen** — `define('usercard')` throws.

**Div button with data-click** instead of `<button>` — accessibility regression.

**Assuming Custom Elements replace React** in large SPAs — usually complementary, not either/or.

## Production & Real-World Notes

Testing libraries use **`data-testid`** (React Testing Library convention) to decouple tests from CSS classes that change with redesigns. Design tokens rarely go in `data-*`; classes and CSS variables preferred.

Micro-frontends and embeddable chips (chat bubble, consent banner) often ship as Custom Elements + Shadow DOM for style isolation. Document **`observedAttributes`** and public API attributes for consumers.

CSP and Custom Elements: inline scripts defining elements need **`unsafe-inline`** avoidance — bundle definitions in external JS files.

When migrating to React, **`data-*` on native DOM** inside components is normal; ref + dataset for imperative bridges to third-party libraries.

---

## Comparison / Trade-offs

**data-* vs hidden inputs** — hidden inputs participate in form submit; data-* does not. Choose based on form integration.

**Custom Elements vs React components** — CE: native, portable, Shadow encapsulation; React: richer ecosystem, SSR tooling, team familiarity.

**Attributes vs properties** — JS property `input.value` vs attribute `value`; custom elements must sync if both matter. React abstracts much of this.

---

## Key Takeaways

- **`data-*`** is the valid, future-proof way to attach custom metadata to HTML elements.
- **`dataset`** maps kebab-case attributes to camelCase; values are always strings.
- Use **semantic elements first**; data-* configures behavior — not replacement for buttons/links.
- **Custom Elements** register new tags with lifecycle hooks; names must contain a hyphen.
- **Web Components** = Custom Elements + Shadow DOM + `<template>`; often used for portable widgets.
- React apps still rely on DOM **`data-*`** and will encounter Custom Elements in the wild — know how to inspect and integrate.
