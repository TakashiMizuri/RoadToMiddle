# Exercises: git cherry-pick and git revert for Hotfixes

> Complete after reading `1.lection-eng.md`, `2.lection-ru.md` and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Cherry-pick creates new hash
**Goal:** See that picked commit ‚Č† original hash.

**Steps:**
1. Create repo with `main` and `feature` branches; one commit only on `feature` (e.g. add line to `hotfix.txt`).
2. Note commit hash on `feature`.
3. Checkout `main`, `git cherry-pick <hash>`.
4. Compare hashes and `git log --oneline` on both branches.

**Success criteria:** Written explanation: same file content possible, different commit IDs, two branch tips.

---

### E2. Revert on a shared-style branch
**Goal:** Roll forward to undo.

**Steps:**
1. On branch `demo-main`: commit `Good` ‚Üí `Bad` (introduce typo) ‚Üí `AlsoGood`.
2. `git revert <Bad-hash> --no-edit`
3. `git log --oneline -5` ‚ÄĒ is `Bad` still listed?

**Success criteria:** Three sentences contrasting revert vs `reset --hard` if branch were pushed.

## Medium

### M1. Hotfix with cherry-pick -x
**Goal:** Full hotfix path.

**Steps:**
1. On `feature/api`, commit fix message `FIX: null check in Auth`.
2. On `main`, create `hotfix/auth-null`, cherry-pick with `-x`.
3. Show commit message contains `(cherry picked from commit ...)`.
4. Create second branch `release/1.0` from older `main` point; cherry-pick same fix.

**Success criteria:** Screenshot or paste of both cherry-pick messages with `-x` trailer.

---

### M2. Cherry-pick conflict lab
**Goal:** `--continue` vs `--abort`.

**Steps:**
1. Same file edited differently on `main` and `feature/fix` commits.
2. Attempt cherry-pick; resolve conflict manually, `--continue`.
3. Reset branch; repeat pick and use `--abort`.

**Success criteria:** Table: action | HEAD position | uncommitted changes after.

---

### M3. Revert then forward fix
**Goal:** Incident timeline in history.

**Steps:**
1. Commit breaking change on `main-line`.
2. Revert it.
3. New branch with proper fix; merge or cherry-pick to `main-line`.
4. Draw ASCII log: bad ‚Üí revert ‚Üí fix.

**Success criteria:** Log shows all three; explain why this is acceptable for audit.

## Hard (stretch)

### H1. LTS backport checklist
**Goal:** Production mindset.

Write a checklist (8+ items) for shipping SEC-401 fix when `main` is v3.0 and customers still run v2.2 and v2.3. Include: branches, cherry-pick, tags, QA, push targets.

**Success criteria:** Mentions `-x`, no force push to shared main, semver tag reference.

---

### H2. git cherry -v
**Goal:** Detect unpicked patches.

**Steps:**
1. Two branches with partially overlapping fixes (same line change, different commits).
2. Run `git cherry -v main feature`.
3. Explain output vs plain `git log main..feature`.

**Success criteria:** One paragraph on patch-id comparison.

---

## Submission checklist

- [ ] Used `-x` at least once
- [ ] Resolved or aborted at least one conflict
- [ ] No force push to any shared remote
