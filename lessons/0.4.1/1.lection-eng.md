# OSI Model vs TCP/IP Model — Four Layers

> Roadmap: `0.4.1` · Node: `0.4` — Networking basics · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Explain why layered network models exist and what problem they solve for developers.
- Compare the seven-layer OSI model with the four-layer TCP/IP model and map each layer to real protocols.
- Trace how an HTTP request from node `0.3` travels down the stack and back up on the receiving side.
- Name the responsibility of each TCP/IP layer without confusing layers (e.g., routing vs delivery vs application semantics).
- Identify which layer handles a given networking symptom (wrong IP, connection refused, garbled HTTP).
- Use the models as a mental map when reading logs, packet captures, or error messages in production.

---

## Why This Matters

In node `0.3` you studied HTTP: methods, status codes, headers, TLS, and REST. HTTP is an **application-layer protocol** — it defines what a *request* and *response* look like as text (or binary in HTTP/2). But HTTP does not know how to find a server on the internet, how to survive packet loss on Wi‑Fi, or how to turn bits into electrical signals on a cable. Those jobs belong to **lower layers**.

When your ASP.NET Core API returns `502 Bad Gateway`, the problem might be HTTP (upstream sent malformed response), transport (TCP connection reset), or network (wrong route to the pod). Without a layered mental model, debugging becomes guesswork. With it, you ask: *Which layer failed, and what does that layer guarantee?*

Network engineers and textbooks present two overlapping maps: the **OSI model** (seven layers, pedagogical) and the **TCP/IP model** (four layers, closer to how the internet actually grew). As a fullstack developer you do not need to memorize every OSI acronym, but you **do** need a reliable stack picture — because every tool you use (browser, Kestrel, PostgreSQL client, Docker bridge network) sits at a specific layer and talks to the layer below through a well-defined interface.

This lesson builds the foundation for subtopics `0.4.2`–`0.4.6`: IP addresses, TCP, UDP, DNS, and localhost binding all live at particular layers. HTTP from `0.3.14` (Keep-Alive, connection pooling) assumes a **reliable byte stream** from TCP at the layer below. Understanding where HTTP ends and TCP begins is what separates "the API is slow" from "the TCP handshake is timing out."

## Core Concepts

### Why Layer at All?

Imagine writing a web server that must simultaneously understand Wi‑Fi frame formats, fiber optic timing, and JSON request bodies. Every change to Wi‑Fi hardware would break your JSON parser. **Layering** solves this by splitting responsibilities: each layer offers a **service** to the layer above and uses the **service** of the layer below, without knowing internal details.

The interface between layers is a contract. The transport layer promises the application layer: "Give me a byte stream to host X port Y; I will deliver it reliably (TCP) or quickly without guarantees (UDP)." The application layer does not need to know whether packets traverse Ethernet, LTE, or a VPN tunnel. This ** abstraction** is why you can run HTTPS over Wi‑Fi at home and over LTE on the train without changing your React fetch call.

Layers also enable **independent evolution**. IPv6 replaced IPv4 at the network layer without redesigning HTTP. TLS moved from application-adjacent to a de-facto layer between application and transport. The models are maps, not physical boxes — but the map is essential for navigation.

### The OSI Model — Seven Layers (Reference)

The **Open Systems Interconnection (OSI)** model, standardized in the 1980s, divides networking into seven layers. You will hear these names in interviews and documentation:

| Layer | Name | Typical protocols / artifacts |
|-------|------|-------------------------------|
| 7 | Application | HTTP, DNS, SMTP, FTP |
| 6 | Presentation | TLS (often cited here), encoding, compression |
| 5 | Session | Logical session management (rarely explicit today) |
| 4 | Transport | TCP, UDP |
| 3 | Network | IP, ICMP, routing |
| 2 | Data Link | Ethernet, Wi‑Fi (802.11), MAC addresses |
| 1 | Physical | Cables, radio, voltages |

In practice, modern stacks **collapse** layers 5–7 into "application" and often treat TLS as sitting between application and transport. OSI remains useful vocabulary — "that's a Layer 4 problem" means transport — even when your daily work follows TCP/IP's four layers.

### The TCP/IP Model — Four Layers (Working Model)

The **TCP/IP model** (also called the **Internet model**) reflects how the ARPANET and internet were built:

```
┌─────────────────────────────────────────┐
│  Application  │  HTTP, DNS, TLS, SSH    │
├─────────────────────────────────────────┤
│  Transport    │  TCP, UDP               │
├─────────────────────────────────────────┤
│  Internet     │  IP (v4/v6), ICMP       │
├─────────────────────────────────────────┤
│  Link         │  Ethernet, Wi‑Fi, ARP    │
└─────────────────────────────────────────┘
```

**Link layer** — moves frames between nodes on the *same* local network segment. Your laptop's Wi‑Fi chip speaks 802.11; a switch forwards Ethernet frames by MAC address. When the destination is not on the local subnet, the frame goes to the **default gateway** (your router).

**Internet layer** — routes **packets** across networks using **IP addresses**. IP is *best-effort*: packets may arrive out of order, duplicate, or not at all. Routing tables and protocols like BGP decide the path from your ISP to a cloud region.

**Transport layer** — multiplexes many applications on one host using **ports**. TCP provides reliable, ordered delivery and congestion control; UDP provides minimal datagram service. This is where `0.4.3` and `0.4.4` dive deep.

**Application layer** — protocols meaningful to programs and users: HTTP, DNS queries, PostgreSQL wire protocol, gRPC. Your Kestrel server and browser live here.

### Mapping OSI to TCP/IP

Rough correspondence:

| TCP/IP (4) | OSI (7) |
|------------|---------|
| Application | Application + Presentation + Session (7, 6, 5) |
| Transport | Transport (4) |
| Internet | Network (3) |
| Link | Data Link + Physical (2, 1) |

When someone says "Layer 3 switch," they mean OSI network layer (IP routing). "Layer 7 firewall" inspects HTTP hostnames and paths — application semantics.

### Encapsulation — Data Flow Down the Stack

Sending an HTTP GET illustrates **encapsulation**. Each layer adds its own header (and sometimes trailer) around the payload from above:

```
Browser (Application):  GET /api/users HTTP/1.1\r\nHost: example.com\r\n...
        ↓ wrapped by TLS (if HTTPS)
Transport (TCP):        [TCP header | encrypted HTTP bytes]
Internet (IP):          [IP header | TCP segment]
Link (Ethernet/Wi‑Fi):  [Frame header | IP packet | Frame trailer]
        ↓ physical bits on wire/air
```

On the server, the process **decapsulates** upward: network card receives frame → kernel strips link header → IP module checks destination address → TCP reassembles segments → TLS decrypts → Kestrel parses HTTP. Each layer only reads its header; the rest is opaque payload.

This matches what you learned in `0.3.3` about HTTP message structure: the HTTP start line and headers are the application payload from TCP's point of view. TCP sees a byte stream; it does not parse `Content-Type`.

### Where HTTP and TLS Sit Relative to 0.3

From node `0.3`:

- **HTTP/1.1** — application layer text protocol; `0.3.14` Keep-Alive reuses one **TCP connection** for multiple HTTP requests (transport concern exposed to app tuning).
- **HTTPS** — HTTP inside **TLS**, which encrypts before handing bytes to TCP. TLS handshake (`0.3.8`) happens after TCP connects but before the first HTTP byte.
- **DNS** (`0.4.5`) — resolves `example.com` to an IP at application layer, but uses **UDP** (usually) at transport — a cross-layer story.

Sequence for `https://api.example.com/users`:

1. **Application:** DNS lookup → IP address.
2. **Transport:** TCP three-way handshake to IP:443 (`0.4.3`).
3. **Application:** TLS handshake → encrypted channel.
4. **Application:** HTTP request/response (`0.3`).

Failure at step 2 surfaces as `connection timed out` or `ECONNREFUSED` — not an HTTP status code, because HTTP never started.

## Syntax / Commands / API

Layer models are conceptual, but you observe them via tools:

```bash
# Application layer — HTTP (from 0.3.12)
curl -v https://example.com/api/health

# Transport — show TCP sockets on Windows
netstat -an | findstr :443

# Internet — route to destination
tracert example.com

# Link / local — ARP cache (same subnet)
arp -a
```

In `-v` curl output, `* Connected to ... port 443` is transport; `> GET / ...` is application; TLS details appear between them.

## Examples

### Example 1: Diagnosing "Connection Refused"

Your React app calls `http://localhost:5000/api/items`. Browser shows `ERR_CONNECTION_REFUSED`.

- **Application (HTTP):** Not reached — no HTTP response exists.
- **Transport (TCP):** SYN sent to port 5000; host responds with RST (nothing listening) or SYN never completes.
- **Internet (IP):** `127.0.0.1` routes locally (`0.4.6`).
- **Fix:** Start Kestrel on port 5000 — an application/host binding issue masquerading as transport.

Layering tells you not to debug JSON serializers first.

### Example 2: 404 vs Unreachable Host

| Symptom | Likely layer | Meaning |
|---------|--------------|---------|
| HTTP 404 Not Found | Application | TCP+TLS OK; server has no matching route |
| `Could not resolve host` | Application (DNS) | Name never became IP |
| `Connection timed out` | Transport / Network | Packets dropped or firewall silent drop |
| `SSL certificate problem` | TLS (app/presentation) | TCP OK; identity/trust failed |

## Common Mistakes & Anti-patterns

**Treating the models as literal hardware stacks.** Layers are logical. TLS might be implemented in kernel or user space; the model still helps you reason about responsibilities.

**Blaming "the network" for application bugs.** A valid HTTP 500 is application layer — the network delivered bytes perfectly.

**Confusing IP address with port.** IP identifies the **host** (internet layer); port identifies the **process** (transport). `203.0.113.10:5432` is PostgreSQL on a specific machine — topics of `0.4.2`.

**Memorizing OSI without TCP/IP mapping.** In daily work and cloud docs, four layers plus "TLS wraps HTTP" is enough. Know OSI numbers for interview vocabulary.

## Production & Real-World Notes

Load balancers often operate at **Layer 4** (forward TCP by IP:port) or **Layer 7** (route by HTTP path, Host header, cookies). An L4 balancer is faster and simpler; L7 enables canary releases and path-based routing but must parse HTTP.

Kubernetes **Services** map cluster IPs (network) to pod endpoints; **Ingress** is Layer 7 HTTP routing. Security groups and firewalls are often described by layer: "allow TCP 443" (L4) vs "block `/admin`" (L7 WAF).

When correlating logs, align layers: APM traces show HTTP spans; `tcpdump` shows packets; `EXPLAIN` is database — don't mix tools from different layers without translating.

## Comparison / Trade-offs

**OSI seven layers** — precise teaching vocabulary, certification exams, vendor docs ("L7 DDoS"). Can feel over-segmented (session vs presentation rarely matter separately).

**TCP/IP four layers** — matches Linux kernel structure (`socket` API → TCP/IP → netdevice), internet history, and most developer docs. Less granular for presentation/session.

For fullstack work, prefer **TCP/IP four layers + explicit TLS** as your working picture; use OSI numbers when reading legacy networking material.

## Quick Reference

| TCP/IP layer | Unit | Key identifiers | Your stack |
|--------------|------|-------------------|------------|
| Application | Message | URL, DNS name, HTTP method | React fetch, Kestrel, Npgsql |
| Transport | Segment/Datagram | IP + port, TCP/UDP | Connection pooling `0.3.14` |
| Internet | Packet | IP address | Routing, VPN, cloud VPC |
| Link | Frame | MAC address | Wi‑Fi, Ethernet, Docker bridge |

**Encapsulation mnemonic:** Data flows **down** adding headers; arrives **up** stripping headers.

---

## Key Takeaways

- Layered models separate concerns so HTTP does not depend on Wi‑Fi hardware details.
- TCP/IP's four layers (Link, Internet, Transport, Application) are the practical map for developers.
- OSI's seven layers provide vocabulary; layers 5–7 often collapse into "application" in real systems.
- HTTP from node `0.3` sits at application layer; it requires TCP (usually) and IP below.
- Encapsulation wraps upper-layer data in lower-layer headers at each step.
- Debug by identifying which layer's guarantees failed before changing application code.
