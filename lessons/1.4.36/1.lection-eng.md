# TimeProvider (.NET 8+): Testable Time and Delays

> Roadmap: `1.4.36` · Node: `1.4` — C# async · Depth: **практика**

## Learning Objectives

After this lesson you will be able to:

- Explain why **`DateTime.UtcNow`** and **`Task.Delay`** make unit tests **slow and flaky**.
- Use **`TimeProvider`** and **`ITimeProvider`** (via abstraction) to inject **virtual time** into services.
- Register **`TimeProvider.System`** in DI and replace it with **`FakeTimeProvider`** in tests.
- Advance virtual time with **`Advance`** / **`SetUtcNow`** and observe **`Task.Delay`** complete without real waiting.
- Apply **`TimeProvider`** to **timeouts**, **retry backoff**, and **token expiry** logic.
- Avoid common mistakes: static time calls, mixing real and fake clocks, forgetting periodic timers.

---

## Why This Matters

Production code constantly depends on **time**: session expiry, rate limiting windows, retry delays, cache TTL checks, scheduled jobs. The natural implementation uses `DateTime.UtcNow` and `Task.Delay(500)`. It works in development. It becomes painful in tests.

A test that calls `await Task.Delay(TimeSpan.FromMinutes(5))` literally waits five minutes — unacceptable in CI. Developers sometimes shorten delays with `#if DEBUG` flags, which means **tests do not exercise production timing logic**. Others mock `Task.Delay` with brittle reflection or wrapper libraries that fight the framework.

**.NET 8** introduced **`TimeProvider`** — an abstract clock and timer factory. **`TimeProvider.System`** delegates to real OS time. **`FakeTimeProvider`** (in `Microsoft.Extensions.TimeProvider.Testing`) lets tests **advance time instantly** while async code using `TimeProvider`-based delays behaves as if real time passed. ASP.NET Core registers `TimeProvider.System` in the default host. Your services take `TimeProvider` (or a thin `IClock` wrapper) and become **deterministic under test** without changing business logic.

This is not academic. Middle developers ship retry policies, Polly timeouts, and background sweepers. Interviewers ask how you test "wait 30 seconds then fail." **`TimeProvider`** is the modern, framework-aligned answer.

## Core Concepts

### The Problem with Direct Time APIs

`DateTime.UtcNow`, `DateTimeOffset.UtcNow`, `Stopwatch`, and `Task.Delay` are **static** or **framework-global**. They bind your logic to the machine clock. Consequences:

- **Flaky tests** — assertions near midnight, DST transitions, or slow CI agents fail intermittently.
- **Slow tests** — real delays accumulate across hundreds of tests.
- **Untestable branches** — "if token expires in 15 minutes" requires time travel or unsafe hacks.

The fix is **dependency inversion**: code that needs time receives an abstraction. Before .NET 8, teams invented `IClock`, `ISystemClock`, or `IDateTimeProvider`. .NET 8 standardized on **`TimeProvider`**.

### What TimeProvider Provides

`TimeProvider` is an abstract class in `System` (namespace root). Key members:

- **`GetUtcNow()`** — current UTC instant as `DateTimeOffset`.
- **`GetLocalNow()`** — local time (uses system timezone unless overridden).
- **`CreateTimer(callback, state, dueTime, period)`** — periodic/one-shot timers compatible with the provider's clock.
- **`Delay(delay, cancellationToken)`** — async delay tied to this provider's timeline.
- **`TimeProvider.System`** — singleton using real time (wraps `Timer`, `Task.Delay`, etc.).

When you call `await timeProvider.Delay(TimeSpan.FromSeconds(30), ct)`, a **`FakeTimeProvider`** does not block for 30 seconds. The delay registers a continuation; when the test calls **`fake.Advance(TimeSpan.FromSeconds(30))`**, pending delays complete — **synchronously from the test's perspective**, while still exercising real async state machines.

### FakeTimeProvider in Tests

Package: **`Microsoft.Extensions.TimeProvider.Testing`** (or included in recent test SDK bundles). Core workflow:

```csharp
var fake = new FakeTimeProvider(startDateTime: new DateTimeOffset(2026, 1, 1, 12, 0, 0, TimeSpan.Zero));
var service = new TokenService(fake);

var refreshTask = service.RefreshAfterDelayAsync(TimeSpan.FromMinutes(5));
fake.Advance(TimeSpan.FromMinutes(5));
await refreshTask; // completes without wall-clock wait
```

`SetUtcNow` jumps the clock to an absolute instant. `Advance` moves forward relatively. Both trigger timer and delay callbacks that are due.

### DI in ASP.NET Core

From .NET 8, `WebApplication.CreateBuilder` registers:

```csharp
builder.Services.AddSingleton(TimeProvider.System);
```

Inject into services:

```csharp
public sealed class RateLimiterService(TimeProvider timeProvider)
{
    public async Task<bool> TryAcquireAsync(CancellationToken ct)
    {
        await timeProvider.Delay(TimeSpan.FromMilliseconds(100), ct);
        // ...
    }
}
```

In tests, replace registration:

```csharp
services.AddSingleton<TimeProvider>(new FakeTimeProvider());
```

Or use `WebApplicationFactory` service override. Production keeps real time; tests swap the provider.

## Under the Hood

`FakeTimeProvider` maintains an internal **priority queue** of scheduled work: delays, timers, and `PeriodicTimer` equivalents keyed by **due time** on the fake timeline. `Advance` drains all entries with `due <= newNow`, invoking callbacks and completing `Task` objects from `Delay`. This mirrors how the thread pool would fire timer callbacks — but under test control.

`TimeProvider` does **not** replace `CancellationTokenSource.CancelAfter` automatically — that still uses real timers unless you use **`CancellationTokenSource.CreateLinkedTokenSource`** with delays created from your `TimeProvider`. For full determinism, prefer `await timeProvider.Delay(timeout, ct)` in your own timeout wrappers.

`PeriodicTimer` in .NET 6+ can be constructed with a `TimeProvider` overload in .NET 8 — periodic polling loops become testable the same way.

## Syntax / Commands / API

| API | Purpose |
|-----|---------|
| `TimeProvider.System` | Production real-time provider |
| `timeProvider.GetUtcNow()` | Injectable "now" |
| `await timeProvider.Delay(ts, ct)` | Injectable async delay |
| `timeProvider.CreateTimer(...)` | Injectable timer |
| `new FakeTimeProvider(start)` | Test double |
| `fake.Advance(delta)` | Move fake clock forward |
| `fake.SetUtcNow(instant)` | Jump to absolute time |
| `services.AddSingleton(TimeProvider.System)` | Default host registration |

## Examples

### Retry with backoff

```csharp
public sealed class ApiClient(TimeProvider time)
{
    public async Task<string> GetWithRetryAsync(Func<Task<string>> call, CancellationToken ct)
    {
        var delay = TimeSpan.FromSeconds(1);
        for (var attempt = 0; attempt < 5; attempt++)
        {
            try { return await call(); }
            catch (HttpRequestException) when (attempt < 4)
            {
                await time.Delay(delay, ct);
                delay += delay; // exponential backoff
            }
        }
        throw new InvalidOperationException("Unreachable");
    }
}
```

Test: use `FakeTimeProvider`, start retry loop, `Advance` by 1s, then 2s, then 4s — verify attempt count without sleeping.

### Token expiry check

```csharp
public sealed class SessionValidator(TimeProvider time)
{
    public bool IsExpired(DateTimeOffset expiresAt) =>
        time.GetUtcNow() >= expiresAt;
}
```

Test: `fake.SetUtcNow(expiresAt.AddSeconds(1))` → `IsExpired` returns true.

## Common Mistakes & Anti-patterns

**Still calling `DateTime.UtcNow` inside a `TimeProvider`-injected class** — only the injected path is testable; mixed clocks cause confusing failures.

**Advancing fake time before starting the async operation** — delays scheduled after an advance at T=0 need a later `Advance` to fire; order matters.

**Using `Task.Delay` directly "just once"** — breaks determinism; use `timeProvider.Delay` consistently in the type under test.

**Assuming `CancelAfter` respects `FakeTimeProvider`** — it does not; wrap timeouts explicitly.

**Sharing one `FakeTimeProvider` across parallel tests without isolation** — use per-test instances.

## Production & Real-World Notes

Teams often wrap `TimeProvider` in a domain-facing `IClock` with only `GetUtcNow()` if timers are rare — but prefer raw `TimeProvider` for new .NET 8+ code to align with framework types (`PeriodicTimer`, `RateLimiter` integrations emerging in extensions).

Libraries like **Polly** v8 expose `TimeProvider` on resilience strategies. **Entity Framework** does not use `TimeProvider` for DB timeouts — connection/command timeouts remain driver-level.

Log timestamps should still use real time in production logging infrastructure; `TimeProvider` targets **application logic**, not Serilog sinks.

---

## Comparison / Trade-offs

| Approach | Pros | Cons |
|----------|------|------|
| `TimeProvider` + `FakeTimeProvider` | Official, works with `Delay`/timers | .NET 8+, extra package in tests |
| Custom `IClock` | Full control, works on older TFMs | Duplicates framework; migration cost |
| `Task.Delay` in prod only | Simple | Untestable delays |
| Short delays in `#if DEBUG` | Quick hack | Tests ≠ production |

## Key Takeaways

- Direct `DateTime` and `Task.Delay` couple logic to wall clock — bad for tests.
- **`TimeProvider`** is the .NET 8+ abstraction for now, delays, and timers.
- **`FakeTimeProvider.Advance`** completes pending delays without real waiting.
- Register in DI; override in `WebApplicationFactory` tests.
- Use consistently — one stray `Task.Delay` undermines the pattern.
- `CancelAfter` still uses real time unless you design around it.
