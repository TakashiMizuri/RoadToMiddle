# Типы контента: application/json, form-urlencoded, multipart

> Roadmap: `0.3.11` · Node: `0.3` — HTTP и Web · Depth: понимание

## Learning Objectives

После этого урока ты сможешь:

- Объяснить, что объявляет заголовок **`Content-Type`** и почему от него зависят client и server.
- Описать, как **`application/json`** кодирует структурированные данные и когда это стандарт для REST API.
- Объяснить **`application/x-www-form-urlencoded`** — правила encoding, наследие HTML-форм и OAuth token endpoints.
- Различить **`multipart/form-data`** и urlencoded-формы; понять, зачем upload файлов требует multipart.
- Выбрать правильный content type для payload (JSON body, простая форма, upload файла).
- Предсказать поведение model binding в ASP.NET Core и browser forms для каждого типа.

## Why This Matters

В предыдущих уроках по HTTP ты узнал, что request несёт method, URL, headers и иногда **body**. Body на wire — просто байты; без договорённости об интерпретации они бессмысленны. Эта договорённость — заголовок **`Content-Type`** (а на стороне client часто зеркалится **`Accept`**). Когда React-приложение шлёт credentials, когда ASP.NET Core controller принимает DTO, когда Postman отправляет файл — каждая сторона должна знать: это JSON, плоская строка key-value или последовательность MIME-частей с boundary.

Неверный тип даёт тонкие production-баги. Frontend шлёт JSON, а API ждёт form-urlencoded — model binding падает или DTO пустой. API принимает multipart для крошечного JSON — лишний overhead и сложнее logging. OAuth client POSTит token как JSON вместо form-urlencoded — spec-compliant server отклонит. Middle-разработчик не зубрит MIME-строки — он понимает **что кодирует каждый формат** и **какой слой его парсит**.

## Core Concepts

### Content-Type как контракт

Каждое HTTP-сообщение с body должно объявлять media type в **`Content-Type`**. Для request server выбирает parser: JSON deserializer, form parser или multipart reader. Для response client решает, вызывать ли `response.json()`, читать text или сохранять bytes как download.

Заголовок может включать **`charset`**, обычно `utf-8`. Для JSON charset часто опускают (JSON = UTF-8 по определению), но у form types исторически были неоднозначные encodings — современные API явно указывают UTF-8.

```
POST /api/users HTTP/1.1
Host: api.example.com
Content-Type: application/json; charset=utf-8
Content-Length: 47

{"email":"dev@example.com","name":"Alex"}
```

Контракт отделён от **HTTP method**. JSON можно слать POST, PUT или PATCH; method говорит *какая операция*, content type — *как устроены байты*.

### application/json

**`application/json`** кодирует данные как **JSON-текст** — objects, arrays, strings, numbers, booleans, `null`. Это lingua franca REST и GraphQL API: хорошо ложится на JavaScript objects и C# classes через `System.Text.Json`.

JSON силён в **вложенной структуре**: user с address object, список order lines, optional fields. Читаем в logs (с лимитами в production), валидируется JSON Schema, поддерживается всеми HTTP clients. Минусы: нет нативного binary inline (Base64 или отдельный upload endpoint), нет streaming form fields без framing, чуть больше размер для numeric-heavy payload.

ASP.NET Core с `[FromBody]` по умолчанию ждёт JSON при `Content-Type: application/json`. Input formatter десериализует в DTO; затем validation attributes.

```json
{
  "title": "Fix login bug",
  "priority": 2,
  "tags": ["auth", "frontend"]
}
```

### application/x-www-form-urlencoded

**`application/x-www-form-urlencoded`** — default encoding **классических HTML forms** при `method="POST"` без override `enctype`. Body — одна строка **`key=value` через `&`**, с **percent-encoding** для reserved characters (пробел → `+` или `%20`, `&` → `%26`, non-ASCII → UTF-8 bytes как `%HH`).

```
username=alex%40example.com&password=Secret%21123&remember=true
```

Формат **плоский**: один уровень keys. Повторяющиеся keys или bracket notation (`items[0].id=1`) — conventions отдельных server, стандартной вложенности как в JSON нет. Компактен для login/search forms и **обязателен в OAuth 2.0** для token requests (`grant_type`, `client_id`, …) в body `application/x-www-form-urlencoded`.

ASP.NET Core bind'ит urlencoded через `[FromForm]`. `[FromBody]` **не** читает form-urlencoded — частая ошибка интеграции SPA с legacy endpoints.

### multipart/form-data

**`multipart/form-data`** оборачивает каждое field (и каждый file) в **отдельную MIME part**, разделённые **boundary** в header:

```
Content-Type: multipart/form-data; boundary=----WebKitFormBoundary7MA4YWxkTrZu0gW
```

У каждой part свои mini-headers (`Content-Disposition: form-data; name="field"` или `filename="photo.jpg"`) и raw body bytes. Binary file content **не** раздувается Base64 внутри JSON — идёт as-is в part, upload эффективен.

Browser автоматически использует multipart при `<form enctype="multipart/form-data">` с `<input type="file">`. API принимают multipart для паттерна «post с картинкой». Parsing тяжелее urlencoded/JSON: scan boundaries, size limits per part, защита от malicious filenames.

```
------WebKitFormBoundary7MA4YWxkTrZu0gW
Content-Disposition: form-data; name="title"

Summer vacation
------WebKitFormBoundary7MA4YWxkTrZu0gW
Content-Disposition: form-data; name="photo"; filename="beach.jpg"
Content-Type: image/jpeg

<binary JPEG bytes>
------WebKitFormBoundary7MA4YWxkTrZu0gW--
```

Closing boundary заканчивается на `--` — маркер EOF.

### Как три типа связаны

Все три решают **«как положить structured user input в HTTP body?»**, но с разными trade-offs. JSON — programmatic clients и rich structure. Urlencoded — простые forms и OAuth. Multipart — **file bytes** без encoding overhead. Одно приложение часто использует все три: JSON для `/api/*`, urlencoded для `/connect/token`, multipart для `/api/documents/upload`.

## Under the Hood

Kestrel принимает request и stream'ит body bytes в middleware. Для API controllers **input formatters** идут по порядку: `Content-Type` сопоставляется с supported formatters. Wrong или missing `Content-Type` → **415 Unsupported Media Type** или silent mis-binding, если formatter ошибочно «забрал» request.

Percent-encoding в urlencoded: `+` → space; `%XX` — hex bytes. JSON parsing через `Utf8JsonReader` — invalid JSON → 400 с problem details.

Multipart parsing в хороших server **streaming**: parts выше порога пишутся на disk/cloud, а не в RAM целиком. Boundary не должен встречаться в content part; clients генерируют random boundary. **Sniffing** `Content-Type` внутри part ненадёжен — проверяй magic bytes server-side.

На response `Content-Type: application/json` → browser `fetch().json()`; `text/html` → document. **MIME type** не определяет method или caching — это отдельные headers.

## Syntax / Commands / API

**Request headers (client):**

| Content-Type | Typical use |
|--------------|-------------|
| `application/json` | REST API bodies, webhooks |
| `application/x-www-form-urlencoded` | HTML forms, OAuth token POST |
| `multipart/form-data; boundary=...` | Forms with files |

**curl:**

```bash
curl -X POST https://api.example.com/tasks \
  -H "Content-Type: application/json" \
  -d '{"title":"Review PR"}'

curl -X POST https://api.example.com/login \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "email=user%40example.com&password=secret"

curl -X POST https://api.example.com/upload \
  -F "title=Report" \
  -F "file=@./report.pdf"
```

**ASP.NET Core:**

```csharp
[HttpPost("login")]
public IActionResult Login([FromForm] LoginRequest request) { ... }

[HttpPost("items")]
public IActionResult Create([FromBody] CreateItemDto dto) { ... }

[HttpPost("upload")]
public IActionResult Upload([FromForm] IFormFile file, [FromForm] string title) { ... }
```

## Examples

### Example 1: SPA login к JSON API

React шлёт credentials как JSON — API документирует `[FromBody] LoginDto`. `fetch` с `Content-Type: application/json` и `JSON.stringify`. Если frontend переключить на urlencoded без смены controller — properties `null`; fix: **один** тип на обеих сторонах.

### Example 2: OAuth token endpoint

OAuth 2.0 требует token requests **form-urlencoded**, не JSON. Client POSTит `grant_type=authorization_code&code=...`. JSON → `400 invalid_request` — не из-за данных, а из-за **media type contract**.

### Example 3: Profile photo upload

Endpoint принимает `multipart/form-data`: `displayName` (text) и `avatar` (file). `IFormFile` даёт stream, name, length. JSON+Base64 для 5 MB раздувает payload ~33% и парсит всё в memory — multipart stream'ит file part.

## Common Mistakes & Anti-patterns

**JSON body с `[FromForm]` или urlencoded с `[FromBody]`** — binding и content type должны совпадать.

**Без `Content-Type` на POST с body** — reject или wrong guess.

**Multipart для чистого JSON API** — лишний overhead.

**Trust client file `Content-Type` в multipart** — validate server-side.

**`+` и space в JSON** — правила только urlencoded; в JSON — UTF-8 или `\u`.

## Production & Real-World Notes

API gateway и WAF смотрят `Content-Type`. OpenAPI — **`requestBody.content`** per media type. Большие files — **presigned S3 URLs** с PUT и своим `Content-Type`.

Logging redact'ит bodies; multipart усложняет redaction — отдельные upload URL с short-lived tokens.

## Comparison / Trade-offs

| Aspect | JSON | Urlencoded | Multipart |
|--------|------|------------|-----------|
| Structure | Nested | Flat keys | Flat + files |
| Binary | Poor | No | Native |
| Browser default | fetch only | Yes | With file input |
| Typical use | REST | OAuth, forms | Uploads |

**Machine-to-machine:** JSON. **Spec says form:** urlencoded. **File bytes:** multipart или отдельный binary upload.

```
Content-Type = как интерпретировать body bytes

application/json           → [FromBody]

multipart/form-data        → [FromForm] + IFormFile

415 = unsupported media type для endpoint
```

## Key Takeaways

1. **`Content-Type`** — контракт интерпретации body.
2. **JSON** — structured API data, nested objects.
3. **Urlencoded** — flat forms, OAuth tokens.
4. **Multipart** — fields + files через boundaries.
5. **`[FromBody]` vs `[FromForm]`** — must match client.
6. Wrong type → empty models, 415, spec rejections.
7. Не доверяй client file metadata.
