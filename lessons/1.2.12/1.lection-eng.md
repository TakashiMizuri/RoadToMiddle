# Static Members, Static Classes, and Static Constructors

> Roadmap: `1.2.12` · Node: `1.2` — C# OOP · Depth: понимание

## Learning Objectives

After this lesson you will be able to:

- Explain what **`static`** means at the **type level** vs **instance level**.
- Use **static fields**, **static properties**, and **static methods** correctly — and know when they harm testability.
- Describe **static classes** (`static class`) and their restrictions.
- Understand **static constructors** (type initializers): when they run, thread safety, and ordering rules.
- Distinguish **static** utility code from **singleton services** in ASP.NET Core DI.
- Avoid common pitfalls: mutable static state, hidden dependencies, and static in generic types.

---

## Why This Matters

After `1.2.1` you know objects live on the heap with reference semantics; after `1.2.11` you compose instance services via DI. **`static`** is the other axis: members belong to the **type itself**, not to any particular object. There is exactly one storage slot for a static field per **app domain** (per loaded type), regardless of how many instances you create.

You see `static` everywhere: `Math.Max`, `string.IsNullOrEmpty`, `File.ReadAllText`, `DateTime.UtcNow`, `Program.Main`. Misusing it — mutable static caches without synchronization, `static` service locators holding `DbContext` — causes bugs that reproduce only under load and evade unit tests. A Middle developer knows when `static` is idiomatic utility, when it should be an **instance + interface** in DI, and when the CLR runs static constructors behind the scenes.

## Core Concepts

### Instance vs Type Members

An **instance member** operates on a specific object: `order.Total` reads that order's total. A **static member** belongs to the declaring type: `decimal.Max(a, b)` needs no `Math` instance because there is no per-call state on a `Math` object — in fact you cannot create `Math` at all.

```csharp
public class Counter
{
    public int InstanceCount { get; private set; }

    public void Increment() => InstanceCount++;

    public static int GlobalCount;
    public static void IncrementGlobal() => GlobalCount++;
}
```

Two `Counter` instances have separate `InstanceCount` values. `GlobalCount` is shared across **all** instances and even accessible without an instance: `Counter.IncrementGlobal()`.

Static methods are **not** virtual in the polymorphic sense you learned in `1.2.7` — there is no `this` pointer. They cannot access instance members unless you pass an instance explicitly. That is why `static` methods cannot implement interface instance methods directly (unless the interface defines a static abstract member in C# 11+ — advanced; default pattern remains instance methods on implementations).

### Static Classes

A **`static class`** is a class that cannot be instantiated and can contain only static members. The compiler marks it `abstract sealed`, so inheritance is forbidden.

```csharp
public static class StringValidation
{
    public static bool IsValidEmail(string email)
        => email.Contains('@') && email.Contains('.');
}
```

Use a static class when the type is a **pure namespace for utilities** with no instance identity — like `System.Math`. If you need polymorphism or state per consumer, use a normal class + interface and register in DI instead.

**Restrictions:** no instance constructors (only static constructor), no instance fields or methods, cannot implement interfaces as instance types (C# 11+ static interface members are a separate feature).

### Static Fields and Properties

A **static field** allocates once per type load. Initialization order matters: static fields initialize before the static constructor runs; explicit field initializers run in declaration order.

```csharp
public class AppConfig
{
    public static string EnvironmentName { get; set; } = "Development";
    private static readonly Lazy<IReadOnlyList<string>> _features = new(LoadFeatures);

    public static IReadOnlyList<string> Features => _features.Value;

    private static IReadOnlyList<string> LoadFeatures()
        => File.ReadAllLines("features.txt");
}
```

**Mutable static state** is a global variable in disguise. In ASP.NET Core, avoid `public static DbContext` or `public static IServiceProvider` except in tightly controlled infrastructure (sometimes `Program` bootstrap). Concurrent requests mutate shared static fields → race conditions.

**Readonly static** (`static readonly`) is safer for constants computed once. **`const`** is compile-time constant for primitives/strings — even more restricted, inlined by compiler.

### Static Constructors (Type Initializers)

A **static constructor** (also **type constructor**, `.cctor` in IL) runs **once per type** before any static or instance member of that type is first accessed, in a **CLR-guaranteed thread-safe** manner.

```csharp
public class DatabaseBootstrap
{
    public static string ConnectionString { get; }

    static DatabaseBootstrap()
    {
        ConnectionString = Environment.GetEnvironmentVariable("DB_CONN")
            ?? throw new InvalidOperationException("DB_CONN missing");
    }
}
```

Rules:

- No access modifiers (`static DatabaseBootstrap()` — implicitly `private`).
- No parameters; cannot be called explicitly.
- Runs at most once per app domain per type.
- If static constructor throws, type becomes **faulted** — subsequent uses throw `TypeInitializationException`.

**Order:** CLR initializes base type static fields/constructors before derived type. Across unrelated types, order is **not** guaranteed unless you create explicit dependencies (one type's static ctor touches another type).

### Static in Generics

Each **closed constructed type** gets its own static fields:

```csharp
public class TypeCache<T>
{
    public static int AccessCount;
}
```

`TypeCache<int>.AccessCount` and `TypeCache<string>.AccessCount` are **separate** storage. This is how some frameworks store per-`T` metadata without dictionaries. It surprises developers who expect one global counter for all `T`.

### Static Local Functions and Static Lambdas

C# supports `static` on **local functions** and lambdas (when they capture nothing) to prevent accidental closure allocations:

```csharp
public int Sum(int[] values)
{
    static int Add(int a, int b) => a + b;
    return values.Aggregate(0, Add);
}
```

This is about **capture semantics**, not type-level static — but the keyword reconnects to "no hidden instance state."

## Under the Hood

### Where Static Data Lives

Instance fields live inside each object on the heap. **Static fields** belong to the **type object** (method table / type handle metadata) associated with the loaded type in the app domain. The JIT resolves static field access through a fixed offset on that type storage — fast, but globally shared.

Because there is one copy, the GC tracks static fields as **GC roots** — they keep referenced objects alive for the process lifetime unless you null them out.

### When the CLR Runs `.cctor`

The runtime triggers type initialization on first touch: `new Derived()`, access to `Derived.StaticField`, or `typeof(Derived)` in some cases. The CLR uses a lock per type so two threads initializing `DatabaseBootstrap` do not run `.cctor` twice.

**BeforeFieldInit** flag (C# classes default): CLR may run field initializers slightly lazily within rules — rarely bites in application code, but explains subtle ordering bugs in complex static graphs.

### Static vs Singleton Instance

A **singleton** (one instance of a class) still has **instance methods** and can implement interfaces — testable via DI replacement. A **static class** is hard-coded to the type — no mock without wrappers.

```csharp
// Hard to test — direct static call
public static class Clock
{
    public static DateTime Now => DateTime.UtcNow;
}

// Testable — inject IClock
public interface IClock { DateTime UtcNow { get; } }
public sealed class SystemClock : IClock { public DateTime UtcNow => DateTime.UtcNow; }
```

Production ASP.NET registers `IClock` as singleton **instance** — behaviorally one object, but replaceable in tests.

## Syntax / API

| Construct | Syntax | Notes |
|-----------|--------|-------|
| Static field | `public static int Count;` | Shared; thread-safety is your job |
| Static readonly | `private static readonly Regex Rx = new(...);` | Init once |
| Static property | `public static string Name { get; set; }` | Avoid mutable in web apps |
| Static method | `public static int Parse(string s)` | No `this` |
| Static class | `public static class Util { }` | No instance members |
| Static constructor | `static MyClass() { }` | Parameterless, private |
| const | `public const int Max = 100;` | Compile-time, value types/strings |

**Cannot:** override static methods (hiding with `new` possible but confusing — avoid).

## Examples

### Example 1: Idiomatic Utility

```csharp
public static class Guard
{
    public static void AgainstNull(object? value, string paramName)
    {
        if (value is null)
            throw new ArgumentNullException(paramName);
    }
}
```

Pure functions, no mutable static state — fine everywhere.

### Example 2: Lazy Static Initialization

```csharp
public sealed class FeatureFlags
{
    private static readonly Lazy<HashSet<string>> _enabled = new(Parse);

    public static bool IsEnabled(string flag) => _enabled.Value.Contains(flag);

    private static HashSet<string> Parse()
        => File.ReadAllLines("flags.txt").ToHashSet(StringComparer.OrdinalIgnoreCase);
}
```

File read happens once, thread-safe via `Lazy<T>`.

### Example 3: Anti-pattern in Web App

```csharp
public class UserContext
{
    public static int CurrentUserId; // BAD in ASP.NET Core
}
```

Two concurrent HTTP requests overwrite each other's user id. **Fix:** `AsyncLocal<int>`, `HttpContext.User`, or scoped service.

## Common Mistakes & Anti-patterns

**Mutable static in multi-threaded apps** — race conditions without `lock` or concurrent collections.

**Static service locator** — `MyApp.Services.GetRequiredService<T>()` hides dependencies; use constructor injection from `1.2.11`.

**Static `DbContext` or connections** — lifetime and thread-safety violate EF Core design.

**Assuming static ctor order across types** — fragile bootstrapping; prefer explicit `IHostedService` startup in ASP.NET.

**Testing static methods that call IO** — tests hit real filesystem unless you wrap behind instance interfaces.

## Production & Real-World Notes

Framework code uses static extensively (`EntityFrameworkQueryableExtensions`, `Enumerable`). Application code leans **instance + DI** for anything with side effects or external IO.

**Configuration:** prefer `IOptions<T>` over `public static AppSettings` — reloadable, testable.

**Analyzers:** CA1810 suggests performance tweaks on static constructors; Sonar flags singletons implemented as static mutable state.

**Source generators** sometimes emit `static partial` helpers — still stateless utilities.

## Comparison / Trade-offs

| Approach | When | Risk |
|----------|------|------|
| `static` utility | Pure helpers, no IO or fixed one-time init | Low if immutable |
| `static` mutable | Legacy globals | High in web/server |
| Singleton instance (DI) | Shared service, testable | Lifetime must match scope |
| `const` | True constants | Must recompile to change |

## Quick Reference

| Need | Use |
|------|-----|
| Math/string helpers | `static` methods on static class |
| One-time type setup | `static constructor` + `static readonly` |
| Shared per-request state | **Not** static — scoped service |
| Per-generic-type cache | `static` field on `Cache<T>` |
| Testable time/API | `IClock` instance, not `DateTime.UtcNow` in business logic |

## Key Takeaways

- **`static`** = one member per **type**, not per object.
- **Static classes** are sealed utility holders — no instances, no inheritance.
- **Static constructors** run once, thread-safe, before first type use.
- **Generic static fields** are per closed type `Foo<int>` vs `Foo<string>`.
- **Mutable static** in ASP.NET Core is almost always wrong — use DI scopes.
- **Singleton service** ≠ static class: singleton is replaceable instance with one lifetime.
