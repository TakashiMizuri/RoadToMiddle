# CDN — Edge Caching and Static Assets

> Roadmap: `0.4.9` · Node: `0.4` — Networking · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Explain what a **CDN (Content Delivery Network)** does and why it reduces latency for static assets.
- Distinguish **origin server** from **edge PoP (Point of Presence)** and describe cache hit vs miss flow.
- Identify which parts of a React + ASP.NET stack typically belong on a CDN vs the API origin.
- Describe **TTL**, **cache invalidation**, and **cache busting** via hashed filenames.
- Relate CDN behavior to DNS (`0.4.5`) and reverse proxies (`0.4.8`).
- Recognize when a CDN is unnecessary overhead for a small application.

---

## Why This Matters

When you run `npm run build` on a React project, Vite or Create React App emits static files: `index.html`, JavaScript bundles, CSS, fonts, and images. In production these files must reach users' browsers quickly worldwide. Serving every `main.a3f2b1.js` request from a single server in Frankfurt works for European users; a user in Singapore pays the full round-trip time on every asset — multiplying latency from lesson `0.4.11` across dozens of files.

A **CDN** places copies of cacheable content at **edge locations** close to users. The first request to a region fetches from your **origin** (S3 bucket, Azure Blob, nginx static folder); subsequent requests in that region are served from the edge cache until the content expires. Your API on ASP.NET Core usually stays on the origin or app servers — dynamic JSON is not blindly cached at the edge without explicit design.

Middle developers configure CDN URLs in deployment pipelines, debug "users see old UI after deploy," and choose cache headers. Confusing CDN with load balancer (`0.4.8`) leads to wrong expectations: CDN optimizes **delivery of cacheable content**, not **compute scaling** of your API.

## Core Concepts

### Origin and Edge

The **origin** is the authoritative source: an S3 bucket with your React build, a storage account, or your web server. **Edge servers** (PoPs) are distributed globally — CloudFront has 400+ locations, Cloudflare similar. Each PoP runs caching software that stores responses keyed by URL (and sometimes headers like `Accept-Encoding`).

When a user requests `https://cdn.example.com/assets/logo.png`, DNS resolves to the **nearest PoP**. If the object is cached and fresh, the PoP returns it immediately — **cache hit**. If not — **cache miss** — the PoP fetches from origin (possibly through a shield region), stores a copy, returns to user. The user might never touch your origin directly for static assets.

### What Belongs on a CDN

**Good CDN candidates:** hashed JS/CSS bundles (`app.[hash].js`), images, fonts, video, downloadable files, public `favicon.ico`. Content that changes rarely or where the URL changes on every deploy.

**Usually not on CDN (or short TTL only):** `index.html` (often low TTL or instant invalidation — it references hashed bundles), API responses with user-specific data, authenticated endpoints, WebSocket connections to your API.

Typical fullstack split:

```
https://app.example.com/          → CDN → S3 (React static)
https://api.example.com/orders    → ALB → Kestrel (dynamic API)
```

The React app loaded from CDN then calls the API origin via `fetch` — crossing origins triggers CORS rules previewed in `0.4.10`.

### Caching Mechanics: TTL and Headers

CDNs respect HTTP caching headers from origin unless overridden by CDN policy. **`Cache-Control: max-age=31536000, immutable`** on hashed assets tells browsers and CDN to cache for a year — safe because a new deploy produces new filenames. **`Cache-Control: no-cache`** on `index.html` forces revalidation so users pick up new bundle references.

**TTL (Time To Live)** is how long an edge node serves a cached copy without checking origin. Long TTL reduces origin load and latency; stale TTL causes users to see old content after deploy unless you **invalidate** (purge) CDN cache or change URLs.

**Cache busting** without purge: Vite/Webpack content hashes in filenames — deploy is automatically a new URL. This is preferred over manual purge in mature pipelines.

### Invalidation and Deploy Workflow

After deploy, teams often:

1. Upload new build to origin (S3 sync).
2. Invalidate `/index.html` or `/*` on CDN (costs money on some providers per path).
3. Rely on hash filenames so only `index.html` needs short TTL.

CloudFront **invalidation** propagates in minutes; during propagation some edges may still serve old `index.html`. Planning deploy communication around this prevents "I refreshed and still see the bug" reports when the bug was already fixed in JS bundle but HTML was stale.

### CDN vs Other Layers

From `0.4.8`, a **reverse proxy/load balancer** sits in front of **your application servers** for dynamic requests. A **CDN** sits in front of **cacheable content** — often object storage, not Kestrel. Many setups use **both**: CDN for static, ALB for API. DNS (`0.4.5`) maps `app.example.com` to CDN distribution, `api.example.com` to ALB.

Some CDNs also offer **edge compute** (Cloudflare Workers, Lambda@Edge) — run small JavaScript at PoP for redirects or A/B tests. That is beyond basic static hosting but shows CDN as an programmable edge, not just a file cache.

## Under the Hood

### Cache Hit Path (Simplified)

```
User (Tokyo)
   │ GET /assets/app.abc123.js
   ▼
Edge PoP (Tokyo) ── hit? ──yes──► 200 + body (few ms)
   │
   miss
   ▼
Origin fetch (e.g. S3 in eu-west-1)
   │ store at edge, forward to user
   ▼
User receives file; next Tokyo user hits cache
```

**Latency:** Hit = RTT to nearby city. Miss = RTT to edge + edge-to-origin (can cross continents once per TTL period per PoP).

### How DNS Ties In

`app.example.com` CNAME to `d111111abcdef8.cloudfront.net`. GeoDNS or anycast IP routes client to nearest PoP. TTL on DNS record affects how fast a CDN endpoint change propagates — separate from asset cache TTL.

## Syntax / Commands / API

### Example cache headers (nginx origin)

```nginx
location /assets/ {
    add_header Cache-Control "public, max-age=31536000, immutable";
}
location = /index.html {
    add_header Cache-Control "no-cache";
}
```

### AWS S3 + CloudFront (conceptual)

- S3 bucket private; CloudFront **OAC/OAI** reads objects.
- Default behavior: cache based on headers; compress objects automatically.

### Checking cache status (curl)

```bash
curl -I https://cdn.example.com/assets/app.js
# Look for: Cache-Control, Age, X-Cache: Hit from cloudfront, CF-Cache-Status
```

## Examples

### Vite React Deploy

Build output:

```
dist/index.html
dist/assets/index-B4k7x2.js
dist/assets/index-D8m3.css
```

Upload `dist/` to S3; CloudFront origin path `/`. Users request `index.html` (short cache), browser parses script src with hash, loads JS with long cache. Second visit: only `index.html` might revalidate; JS/CSS from browser or CDN cache.

### When Skip CDN

Internal admin tool with five users in one office — CDN adds complexity, DNS, cost, and cache debugging for negligible gain. Single-region VPS serving static files via nginx is fine until latency or bandwidth metrics justify CDN.

## Common Mistakes & Anti-patterns

**Caching `index.html` for a year** — users stuck on old bundle references after deploy.

**Putting API on CDN without understanding cache rules** — accidental caching of personalized JSON.

**Forgetting HTTPS on CDN** — mixed content or cert mismatch; use CDN-managed certs.

**Purge as only deploy strategy** — expensive and slow; prefer hashed assets.

---

## Production & Real-World Notes

**Versioned deploys:** Some teams use `/v/20250703/` path prefix for major releases. **Brotli/gzip** at CDN reduces bytes — ties to bandwidth vs latency (`0.4.11`).

**DDoS absorption:** CDN spreads traffic across edge, hiding origin IP if configured correctly (only CDN can reach origin).

**Cost model:** egress from origin to CDN vs edge to user — usually CDN egress cheaper at scale.

## Comparison / Trade-offs

| Approach | Pros | Cons |
|----------|------|------|
| CDN + object storage | Global low latency, scalable static | Config complexity, invalidation |
| nginx on same VM as API | Simple | Single region latency, VM bandwidth limit |
| SSR from Kestrel | Dynamic HTML | Not a CDN replacement for static bundles |

## Quick Reference

- **CDN** = geographically distributed cache for static/cacheable content.
- **Origin** = source of truth (S3, blob, server).
- **Hit** = served from edge; **Miss** = fetched from origin.
- **Hash filenames** + long `max-age` = best practice for JS/CSS.
- **`index.html`** = short TTL or purge on deploy.
- **API** = typically not CDN-cached (unless explicitly designed).

---

## Key Takeaways

- CDN reduces **latency** and **origin load** for static React build assets.
- Edge PoPs cache by URL/TTL; **cache busting** via content hashes avoids stale JS.
- Split **app.example.com** (CDN static) and **api.example.com** (dynamic API).
- Coordinate with **DNS** and distinguish CDN from **load balancer**.
- Small internal apps may not need a CDN — trade simplicity vs global performance.
