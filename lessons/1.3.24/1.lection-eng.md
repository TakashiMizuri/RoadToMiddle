# LINQ Any, All, Contains — Existence Checks

> Roadmap: `1.3.24` · Node: `1.3` — C#: Advanced Topics · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Use **`Any()`** and **`Any(predicate)`** for existence checks with **short-circuit** semantics.
- Use **`All(predicate)`** for universal quantification and express negated conditions safely.
- Choose **`Contains(item)`** vs **`Any(x => x == item)`** with correct **equality comparer**.
- Translate boolean intent to LINQ without **`Count() > 0`** anti-patterns.
- Apply operators on **`IEnumerable`** vs **`IQueryable`** (SQL **`EXISTS`**, **`IN`**, **`ALL`**) consciously.
- Combine with **nullable**, **empty sequences**, and **validation** pipelines in APIs.

---

## Why This Matters

Business rules constantly ask: "Is there any unpaid invoice?", "Are all lines valid?", "Does this list contain role Admin?" These are **existential** and **universal** queries — logically **`∃`** and **`∀`**. LINQ expresses them as **`Any`**, **`All`**, and **`Contains`** instead of counting entire collections or writing manual loops with flags.

Using **`list.Count() > 0`** or **`list.Where(p).Count() > 0`** materializes or walks the full sequence when **`Any(p)`** stops at the **first match**. On **`IQueryable`**, **`Any`** translates to SQL **`EXISTS`** — often index-friendly — while **`Count() > 0`** may generate **`SELECT COUNT(*)`** scanning more data than needed. **`All`** short-circuits on first false; negation pitfalls (**`!All`** vs **`Any(!)`**) cause authorization bugs.

This is practical, daily LINQ — readable intent, correct performance, safe logic in validation and auth.

## Core Concepts

### Any — Is There at Least One?

```csharp
if (orders.Any())
    Console.WriteLine("Has orders");

if (orders.Any(o => o.Total > 10_000))
    Console.WriteLine("Large order exists");
```

**`Any()`** without predicate: sequence **non-empty**. **`Any(predicate)`**: at least one element satisfies predicate.

**Short-circuit:** stops at first match — O(1) best case, O(n) worst.

**Empty sequence:** **`Any()`** → **`false`**; **`Any(predicate)`** → **`false`**.

Do **not** write **`Count() > 0`** when **`Any()`** communicates intent and may short-circuit.

### All — Do All Elements Satisfy?

```csharp
if (order.Lines.All(l => l.Qty > 0))
    Submit(order);
```

**Vacuous truth:** **`All`** on **empty sequence** returns **`true`** — mathematically ∀x∈∅, P(x) is true. Surprises developers validating "at least one line required":

```csharp
if (lines.All(l => l.Qty > 0)) // true when lines empty — bug!
```

Combine with **`Any()`** first or check **`Count`**/non-empty explicitly for business rules.

**Short-circuit:** stops at first **false**.

### Contains — Membership by Value

```csharp
var allowed = new[] { "Admin", "Editor" };
if (allowed.Contains(user.Role))
    Grant();
```

**`Contains`** on **`IEnumerable<T>`** extension (LINQ) walks sequence with default equality comparer unless overload with **`IEqualityComparer<T>`** is used.

For **`List<T>`**, **`Contains`** may use **`List<T>.Contains`** directly — O(n) linear search; **`HashSet<T>`** for repeated lookups.

```csharp
roles.Contains("admin", StringComparer.OrdinalIgnoreCase);
```

**Reference types** without overridden **`Equals`**: reference equality. **Value types**: value equality.

### Any vs Contains

| Intent | Expression |
|---|---|
| Value in collection | **`Contains(value)`** or **`HashSet`** |
| Value match with custom rule | **`Any(x => x.Id == id)`** |
| Case-insensitive string | **`Contains`** with comparer or **`Any`** with **`Equals`** |

**`Contains`** in EF on **`IQueryable`** may translate to SQL **`IN`** or column equality; **`Any`** with predicate translates to **`EXISTS`** subquery — not always interchangeable depending on navigation shape.

### Negation and De Morgan

| Intent | LINQ |
|---|---|
| None satisfy | **`!Any(predicate)`** |
| Not all satisfy | **`!All(predicate)`** or **`Any(x => !predicate(x))`** |
| At least one fails | **`Any(x => !valid(x))`** |

**Authorization:** "user lacks Admin" → **`!roles.Contains("Admin")`**, not **`roles.All(r => r != "Admin")`** (vacuous true on empty roles — **security bug**).

```csharp
// Wrong for empty roles — grants access
if (roles.All(r => r != "Admin")) GrantAdminFeatures();

// Safer
if (!roles.Contains("Admin")) Deny();
```

## Under the Hood

**`Enumerable.Any`**: iterate until match or end. **`All`**: iterate until false or end. No allocations beyond enumerator.

**`IQueryable.Any`/`All`**: expression tree nodes translated by provider:

```sql
-- Any(predicate) often becomes
EXISTS (SELECT 1 FROM ... WHERE predicate)

-- Contains on column vs constant list
WHERE Role IN ('Admin', 'Editor')
```

**`Count() > 0`**: may generate **`SELECT COUNT(*)`** — full count or index scan vs **`EXISTS`** stop-at-first — provider dependent but **`Any`** expresses intent.

**Deferred:** **`Any`/`All`/`Contains`** are **immediate** — they execute enumeration/query when called, not when building expression (for **`IQueryable`**, execution at **`AnyAsync`** etc.).

## Syntax / API

```csharp
sequence.Any();
sequence.Any(x => condition);
sequence.All(x => condition);
sequence.Contains(value);
sequence.Contains(value, comparer);

// EF async
await db.Orders.AnyAsync(o => o.Total > 100);
await db.Users.AnyAsync(u => u.Email == email);
```

## Examples

### API validation

```csharp
public static bool IsValidOrder(Order o) =>
    o.Lines.Any() &&
    o.Lines.All(l => l.Qty > 0 && l.UnitPrice >= 0);
```

Explicit **`Lines.Any()`** rejects empty orders before **`All`** vacuous pass.

### Permission check

```csharp
public bool CanModerate(IReadOnlyList<string> roles) =>
    roles.Contains("Moderator", StringComparer.OrdinalIgnoreCase)
    || roles.Contains("Admin", StringComparer.OrdinalIgnoreCase);
```

For hot paths, **`HashSet<string>`** with comparer once — **`Contains`** O(1).

### Database existence before insert

```csharp
if (await db.Customers.AnyAsync(c => c.Email == dto.Email))
    return Conflict("Email taken");
```

Prefer **`AnyAsync`** over **`CountAsync() > 0`**.

### Filtering negation

```csharp
var hasInvalid = batch.Any(item => !validator(item));
if (hasInvalid)
    return BadRequest();
```

Clearer than **`!batch.All(validator)`** when validator named **`IsValid`** — use **`Any(x => !IsValid(x))`**.

## Common Mistakes & Anti-patterns

**`Count() > 0`** instead of **`Any()`**.

**`All` on possibly empty** without guard — vacuous true.

**`roles.All(r => r != "Admin")`** for security on empty roles.

**Reference equality **`Contains`** on strings interned vs not — use explicit comparer.

**Multiple enumeration** of expensive **`IEnumerable`** calling **`Any`** then **`All`** separately — consider one pass or materialize.

**`Contains` on infinite sequence** — hangs if value never found and sequence never ends.

**Confusing **`List.Contains`** item search with **`Contains` extension** on **`IEnumerable`** — both exist; extension walks lazy source.

## Production & Real-World Notes

**EF Core:** **`AnyAsync`** for existence checks in handlers — minimal data transfer. **`AllAsync`** rare in SQL — sometimes translated, sometimes client eval; verify SQL.

**Validation attributes / FluentValidation:** **`RuleForEach`** + **`Must`** often clearer than LINQ for DTO rules; LINQ fine for in-memory collections on domain objects.

**Unit tests:** table-driven tests for empty/single/all-match/none-match edge cases for **`All`/`Any`**.

**Performance:** large in-memory collections checked repeatedly → **`HashSet`**. Distributed cache membership → not LINQ **`Contains`** on list — use cache API.

## Comparison / Trade-offs

| Check | Operator | Empty seq |
|---|---|---|
| Non-empty | **`Any()`** | false |
| Some match | **`Any(p)`** | false |
| Every match | **`All(p)`** | **true** (vacuous) |
| Value member | **`Contains(v)`** | false |

**`Exists` in SQL** ≈ **`Any`**. **`IN`** ≈ **`Contains`** on constant set or **`Any`** with equality.

## Quick Reference

```csharp
items.Any()                 // has elements
items.Any(x => f(x))        // some satisfy
items.All(x => f(x))        // all satisfy (true if empty!)
items.Contains(key)         // value present
!items.Any(x => f(x))       // none satisfy
```

---

## Key Takeaways

- **`Any`** short-circuits; prefer over **`Count() > 0`**.
- **`All`** on empty → **true** — guard when business requires elements.
- **`Contains`** for membership; pass **comparer** when needed.
- Security: never rely on **`All(r => r != forbidden)`** alone on empty collections.
- **`IQueryable`**: **`AnyAsync`** → **`EXISTS`**-style queries.
- **`Any`/`All`/`Contains`** are immediate terminal operators.
