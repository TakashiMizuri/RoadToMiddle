# Interface: Implicit vs Explicit Implementation

> Roadmap: `1.2.9` · Node: `1.2` — C# OOP · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Implement an **interface implicitly** — public members visible on the class and through the interface.
- Implement an **interface explicitly** — members callable only when reference is typed as that interface.
- Resolve **name collisions** when a class implements two interfaces with identical member signatures.
- Explain **dispatch** and metadata differences between implicit and explicit implementations.
- Choose explicit implementation to **hide** implementation detail or separate conflicting APIs.
- Read BCL patterns (**`IEnumerable`**, **`IComparable`**) that rely on explicit implementation.

---

## Why This Matters

Lesson `1.2.8` positioned **interfaces** as contracts decoupled from class hierarchies. When a class **`implements`** an interface, C# offers two syntaxes that look like bookkeeping but change **who can call what** and **how members appear in IntelliSense**.

Implicit implementation exposes interface methods as **public class members** — convenient when the interface *is* the class's primary surface. Explicit implementation **hides** members from direct class access — essential when **`Draw()`** means different things for **`IDrawable`** and **`IPrinter`**, or when **`CompareTo`** should not clutter **`Widget`'s** public API.

Middle developers hit explicit implementation in **`IComparable<T>`**, **`IEnumerable<T>`**, and adapter classes wrapping legacy APIs. Misusing explicit hides legitimate public API; avoiding explicit when two interfaces collide causes **CS0539** / ambiguous call errors. This lesson makes the choice deliberate.

## Core Concepts

### Implicit Interface Implementation

Default style: class declares **`public`** member matching interface signature. The member is **both** a class member and interface fulfillment.

```csharp
public interface ILogger
{
    void Log(string message);
}

public class ConsoleLogger : ILogger
{
    public void Log(string message) =>
        Console.WriteLine(message);
}

// Usage:
ConsoleLogger c = new();
c.Log("hi");                    // OK — public on class

ILogger l = c;
l.Log("hi");                    // OK — via interface
```

**IntelliSense** on **`ConsoleLogger`** shows **`Log`**. External callers treating object as **`ConsoleLogger`** see logging as part of its public contract. Appropriate when interface methods **are** the class purpose (**`SmtpEmailSender : IEmailSender`**).

Rules:

- Member must be **`public`** (unless interface member is protected in C# 8+ rare cases).
- One method satisfies interface and class API simultaneously.
- **Virtual** on implicit impl: **`public virtual void Log`** can be overridden in derived class; interface mapping follows override chain.

### Explicit Interface Implementation

Prefix member with **interface name**:

```csharp
public class QuietLogger : ILogger
{
    void ILogger.Log(string message) =>
        Console.WriteLine($"[quiet] {message}");
}

QuietLogger q = new();
// q.Log("hi");           // CS1061 — Log not on QuietLogger

ILogger l = q;
l.Log("hi");                // OK
```

Explicit members are **not public** on the class — they are accessible **only** through a reference typed as that interface (or **`dynamic`** / reflection). Metadata marks them **`private`** at IL level with **`InterfaceImpl`** mapping — **callable through interface mapping**, not as **`public`** methods on class.

Use explicit when:

1. **Hide helper-like interface members** from class public API (**`IComparable<T>.CompareTo`**).
2. **Resolve signature collision** — two interfaces define **`void Reset()`**.
3. **Separate semantic duplicates** — class already has **`Draw()`** for screen; **`IPrinter.Draw()`** explicit.
4. **Legacy adapter** — implement foreign interface without polluting surface.

### Name Collisions and the Diamond of Names

C# forbids **multiple class inheritance**, but a class may implement **`IA`** and **`IB`** both with **`void Save()`**. Implicit public **`Save`** cannot satisfy both with different semantics. Explicit disambiguates:

```csharp
public interface IA { void Save(); }
public interface IB { void Save(); }

public class Document : IA, IB
{
    public void Save() => SaveForDatabase();       // implicit for IA if only one public Save — PROBLEM

    void IA.Save() => SaveForDatabase();
    void IB.Save() => SaveForBackup();
}
```

Only **one** implicit **`Save`** if signatures match — implements **at most one** interface member visibly; second requires explicit. Pattern: **one implicit OR both explicit** for clarity when behaviors differ.

### Combining Implicit and Explicit

Common BCL-style:

```csharp
public class Customer : IEquatable<Customer>, IComparable<Customer>
{
    public int CompareTo(Customer? other) => Id.CompareTo(other?.Id);

    int IComparable<Customer>.CompareTo(Customer? other) => CompareTo(other);
}
```

Above redundant — real code picks **explicit for `IComparable`** only:

```csharp
public class Customer : IComparable<Customer>
{
    public int Id { get; init; }

    int IComparable<Customer>.CompareTo(Customer? other) =>
        Id.CompareTo(other?.Id);
}
```

Public API stays clean; sorting via **`((IComparable<Customer)c)`** or **`OrderBy`** using comparer that calls interface method.

### Properties and Indexers

Explicit works for **properties** and **indexers**:

```csharp
public interface IReadOnlyById
{
    string this[int id] { get; }
}

public class Registry : IReadOnlyById
{
    private readonly Dictionary<int, string> _map = new();

    string IReadOnlyById.this[int id] => _map[id];
}
```

No public indexer on **`Registry`** except through **`IReadOnlyById`**.

## Under the Hood

### Metadata and Interface Mapping

Each implementing class has **InterfaceImpl** table: pairs **(InterfaceMethod, ClassMethod)**. Implicit **`public void Log`** maps **`ILogger.Log` → ConsoleLogger.Log** — same MethodDef. Explicit **`void ILogger.Log`** generates **separate MethodDef** (name mangled **`ILogger.Log`** in IL/debug) marked **private** visibility on class, only invoked through **interface call stub**.

Call **`((ILogger)obj).Log()`** compiles to **`callvirt`** on **interface slot**, targeting explicit or implicit method as mapped.

### Virtual Explicit Implementation

C# allows **`public virtual`** implicit member overridden in derived; explicit re-implementation in derived class **re-maps** interface for that derived type:

```csharp
public class Base : IRunner
{
    public virtual void Run() { }
}

public class Derived : Base, IRunner
{
    void IRunner.Run() { /* explicit only when cast to IRunner on Derived */ }
}
```

Edge cases confuse — prefer **consistent pattern** in hierarchies: document if derived re-implements interface explicitly.

### Default Interface Methods Interaction

When interface provides **default method** (`1.2.10`) and class does not implement, class inherits default via mapping. Class **explicit** implementation overrides default for that interface slot.

## Syntax / API

```csharp
// Implicit
public void Member() { }

// Explicit
ReturnType IInterfaceName.Member(params) { }
ReturnType IInterfaceName.Property { get; set; }
ReturnType IInterfaceName.this[int i] { get; }
```

Cannot combine **`public`** with explicit prefix — compile error.

## Examples

### Example 1: IEnumerable Pattern

```csharp
public class NumberSequence : IEnumerable<int>
{
    private readonly int[] _items;

    public NumberSequence(params int[] items) => _items = items;

    public IEnumerator<int> GetEnumerator()
    {
        foreach (var n in _items) yield return n;
    }

    IEnumerator IEnumerable.GetEnumerator() => GetEnumerator();
}
```

**`IEnumerable<int>`** public **`GetEnumerator()`**; non-generic **`IEnumerable.GetEnumerator()`** often **explicit** to avoid two public **`GetEnumerator`** on class (older pattern; modern code may use **`yield return`** with single public generic only).

### Example 2: Collision Resolution

```csharp
public class Storage : IPersistent, ICache
{
    void IPersistent.Flush() => WriteToDisk();
    void ICache.Flush() => ClearMemory();

    private void WriteToDisk() { /* ... */ }
    private void ClearMemory() { /* ... */ }
}
```

Callers use **`((IPersistent)storage).Flush()`** vs **`((ICache)storage).Flush()`** — no ambiguous public **`Flush`**.

### Example 3: When Implicit Is Correct

```csharp
public sealed class StripePaymentGateway : IPaymentGateway
{
    public Task<PaymentResult> ChargeAsync(Money amount, Card card, CancellationToken ct)
    {
        // Stripe SDK...
    }
}
```

**`IPaymentGateway`** *is* the service's identity — public implicit **`ChargeAsync`** expected on **`StripePaymentGateway`**.

## Common Mistakes & Anti-patterns

**Explicit everything "for encapsulation"** — hurts discoverability; use when hiding is real requirement.

**Public method + explicit duplicate** for same interface — redundant, confusing which maps.

**Forgetting cast** — need **`((IMyInterface)obj).M()`** when explicit; **`obj.M()`** fails.

**Two implicit methods same signature** — impossible; second interface unimplemented → CS0535.

**Explicit impl trying to be virtual public** — wrong tool; use protected virtual helper called from explicit member.

## Production & Real-World Notes

Framework types use explicit for **non-generic IEnumerable**, **ICloneable**, **IComparable** — keep public API minimal. Application services rarely need explicit unless integrating two third-party interfaces with same names.

Analyzers rarely flag choice; code review should ask: *"Should this member appear on class public surface?"*

## Comparison / Trade-offs

| | Implicit | Explicit |
|---|----------|----------|
| Visible on class | Yes (public) | No |
| Call syntax | `obj.M()` | `((I)obj).M()` |
| Collision handling | One signature only | Per-interface methods |
| Use case | Primary API | Hidden/auxiliary/conflict |

---

## Quick Reference

- **Implicit** = public, dual access class + interface.
- **Explicit** = **`InterfaceName.Member`**, interface-only access.
- **Collisions** → explicit per interface.
- **BCL pattern** — **`IComparable`**, **`IEnumerable`** non-generic often explicit.

---

## Key Takeaways

- Implicit implementation merges interface and class public API.
- Explicit implementation hides members from class-typed references.
- Explicit resolves **same-signature** conflicts across multiple interfaces.
- CLR maps both styles through **InterfaceImpl**; explicit methods are separate MethodDefs.
- Choose implicit for primary service contracts; explicit for auxiliary or colliding semantics.
- Default interface methods participate in same mapping rules (`1.2.10`).
