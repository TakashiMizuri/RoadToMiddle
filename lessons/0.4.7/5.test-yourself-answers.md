## Answers

### 1. [Recall]
**Answer:** Какие **сетевые пакеты** разрешено пропускать/блокировать — обычно по source/destination IP, port, protocol (TCP/UDP).

**Why:** Firewall = network reachability policy, не application logic.

---

### 2. [Recall]
**Answer:** **443** (HTTPS over TLS).

**Why:** IANA well-known port для https.

---

### 3. [Recall]
**Answer:** **5432**.

**Why:** Default PostgreSQL listener port.

---

### 4. [Understanding]
**Answer:** Firewall только **доставляет** HTTPS-трафик до сервера. **Authorization** (JWT, roles, policies) проверяется в приложении. Открытый 443 = «дверь в здание открыта», не «каждый может в сейф».

**Why:** Разные слои defense in depth.

**Common mistake:** Путать transport security с access control.

---

### 5. [Understanding]
**Answer:** **Refused** — на порту никто не accept'ит (service down) или явный reject. **Timeout** — пакеты **drop'ятся** без ответа (typical firewall/SG deny), клиент ждёт до лимита.

**Why:** Drop vs RST/ICMP unreachable дают разные симптомы.

---

### 6. [Understanding]
**Answer:** БД содержит sensitive data; bots **сканируют** open 5432 постоянно; brute force / misconfig / missing patches → быстрый compromise. БД должна быть в private network, доступ только с app tier.

**Why:** Network exposure умножает impact любой app-уязвимости.

---

### 7. [Application]
**Answer:** Публично: **80** (redirect), **443** (HTTPS для SPA и API через gateway). Между серверами: **5432** (API → PostgreSQL), internal app port (ALB → Kestrel). SSH **22** — только bastion/office IPs, не весь internet.

**Why:** Users need 443; database never directly.

---

### 8. [Application]
**Answer:** Binding на `127.0.0.1:5432:5432` вместо `0.0.0.0`; host firewall блокирует inbound 5432; не deploy на public IP с таким mapping. Убедиться, что в офисной сети нет route к dev machine как к server.

**Why:** `0.0.0.0:5432` слушает все interfaces.

---

### 9. [Understanding]
**Answer:** **Stateful** помнит established sessions и **автоматически** пропускает related return packets. **Stateless** match'ит каждый packet изолированно — нужны explicit inbound rules для responses.

**Why:** Практически все modern SG/host firewalls stateful.

---

### 10. [Analysis]
**Answer:** HTTPS защищает **in transit** browser↔server, но не: (1) **authorization** в API, (2) **exposed internal ports** (5432, Redis), (3) **app vulnerabilities** (SQLi, IDOR), (4) **misconfiguration** (debug endpoints). Firewall всё ещё нужен для least-privilege exposure.

**Why:** TLS ≠ complete security.

---

### 11. [Analysis]
**Answer:** Любой в internet может **attempt connect** к PostgreSQL; brute force credentials; known CVE exploits; data exfiltration; ransomware. Даже strong password — unnecessary attack surface.

**Why:** Security through obscurity и password alone insufficient на public DB port.

---

### 12. [Recall]
**Answer:** **Redirect HTTP→HTTPS**; **Let's Encrypt HTTP-01** challenge; legacy clients/bookmarks с `http://`.

**Why:** 80 не для sensitive data, а для upgrade path.

---

### 13. [Application]
**Answer:** (1) SG/NACL: allow 5432 **from API SG**? (2) RDS в **correct subnet/VPC**? (3) **Route tables** / peering между app и DB tier? Также: wrong host in connection string, но timeout → network first.

**Why:** Timeout = packets не доходят или drop.

---

### 14. [Understanding]
**Answer:** Открывать **только** необходимые ports для **минимального** набора source IP/security groups, нужных для работы — не «wide open just in case».

**Why:** Каждый open port = enlarged attack surface.

---

### 15. [Analysis]
**Answer:** **SG** — на уровне cloud network interface (до VM), managed в AWS/Azure/GCP. **Host firewall (ufw)** — на OS внутри VM. Оба: SG блокирует traffic до instance; host firewall — второй барьер если другой compromised instance в VPC или misconfigured SG.

**Why:** Defense in depth at different layers.
