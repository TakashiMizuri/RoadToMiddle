# localhost, 127.0.0.1, and 0.0.0.0 Binding

> Roadmap: `0.4.6` · Node: `0.4` — Networking basics · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Explain loopback addressing (`127.0.0.1`, `::1`) and why traffic never leaves the host.
- Distinguish **`localhost` as hostname** from numeric loopback addresses and when each resolves.
- Describe what **`0.0.0.0` and `[::]` mean in bind/listen** vs using them as a client destination.
- Configure Kestrel and Docker publish scenarios for local-only vs LAN-accessible dev servers.
- Relate loopback to DNS (`0.4.5`), sockets/ports (`0.4.2`), and HTTP dev URLs from `0.3.2`.
- Avoid common production mistakes: binding only loopback behind reverse proxy, confusing `0.0.0.0` with "public URL."

---

## Why This Matters

Every ASP.NET developer runs Kestrel at `https://localhost:7150` daily. React `fetch('http://localhost:5000/api')` assumes the API listens where the name resolves — usually **`127.0.0.1`**, a **loopback** address handled inside the OS network stack (`0.4.1` internet layer special range). Packets to loopback never hit Wi‑Fi or Ethernet; the kernel loops them to a local socket (`0.4.2`). That is why `localhost` works offline and why binding matters for **who can connect**.

**`0.0.0.0`** appears in `launchSettings.json`, Docker, and nginx `listen 0.0.0.0:80` — meaning **bind all IPv4 interfaces**, not "address zero on the internet." Misreading it causes "works on my machine" when teammates cannot reach your dev server from another device, or production containers only listening on loopback inside the pod.

Connecting HTTP/TCP from node `0.3`–`0.4.3` always targets a concrete `(IP, port)`. Choosing loopback vs `0.0.0.0` bind vs machine LAN IP determines whether Docker port publish, mobile device on same Wi‑Fi, or Kubernetes probe can succeed. Middle developers debug `ERR_CONNECTION_REFUSED` daily — often bind address mismatch, not broken controllers.

## Core Concepts

### Loopback — Talk to Yourself

**Loopback** is a virtual network interface implemented in software. IPv4 **`127.0.0.0/8`** is reserved for loopback; **`127.0.0.1`** is the conventional single-host address. IPv6 equivalent **`::1`**.

When you TCP connect to `127.0.0.1:5000` (`0.4.3`), IP layer delivers to local stack; if a process bound that port on loopback, kernel connects them. No physical NIC, no DNS required (though `localhost` usually resolves via hosts/DNS to loopback — `0.4.5`).

**Use cases:** local dev, inter-process communication on same machine, health checks scoped to container self (sometimes).

**Limitation:** Remote machines cannot reach your `127.0.0.1` — it identifies "this host" from every host's perspective.

### localhost — A Name, Not an Address

**`localhost`** is a hostname convention (RFC 6761 special-use). OS **hosts file** maps:

```
127.0.0.1       localhost
::1             localhost
```

Resolver (`getaddrinfo`) returns loopback addresses. Browsers and `HttpClient` use whichever family they prefer (often IPv6 `::1` first on modern Windows — can cause "works with 127.0.0.1 but not localhost" if server binds IPv4 only).

**`localhost` is not a routable public name** — no registrar DNS for your laptop's loopback.

### Binding vs Connecting

**Bind (listen)** — server chooses which local IP:port accepts inbound connections.

| Bind address | Meaning |
|--------------|---------|
| `127.0.0.1:5000` | Only local processes/clients on same machine |
| `0.0.0.0:5000` | All IPv4 interfaces (LAN IP, loopback, Docker bridges) |
| `[::]:5000` | All IPv6 interfaces (often dual-stack with IPv4 mapped) |
| Specific `192.168.1.50:5000` | Only that interface |

**Connect (client)** — must target a **specific reachable** address: `127.0.0.1`, LAN IP, or public IP — **never** `0.0.0.0` as destination (undefined/wrong on most OSes).

From `0.4.2`: `bind()` attaches socket to interface; `connect()` targets remote IP:port.

### Kestrel and launchSettings

```json
"applicationUrl": "https://localhost:7150;http://localhost:5150"
```

Default templates bind via `localhost` → loopback. Teammate on another PC must use your **LAN IP** (e.g. `http://192.168.1.50:5150`) **and** you must listen on `0.0.0.0` or that specific IP — not `127.0.0.1` only.

**ASPNETCORE_URLS** environment variable same semantics.

### Docker and Port Publishing

Container nginx `listen 80` often binds `0.0.0.0:80` inside container network namespace. `docker run -p 8080:80` maps **host** `0.0.0.0:8080` (by default all interfaces) to container. From host browser `localhost:8080` works via loopback to published port.

If Kestrel inside container binds **only** `127.0.0.1:80`, other containers and host publish may fail — bind `0.0.0.0` for services meant to be reached externally.

### Link to HTTP Dev Workflow (0.3)

React dev server `http://localhost:5173` calling API `http://localhost:5000` — both loopback, same machine, no CORS cross-origin if same site policy differs by port (**different origins** — port matters for CORS `0.4.10` preview). Phone testing same API requires LAN IP + firewall allow + bind all interfaces.

HTTPS dev cert for `localhost` — TLS validates name `localhost` (`0.3.8`), not `127.0.0.1` unless cert includes IP SAN (uncommon dev).

## Syntax / Commands / API

```bash
# Ping loopback
ping 127.0.0.1
ping localhost

# See what listens where (Windows)
netstat -ano | findstr :5000

# Kestrel all interfaces (dev only — security!)
set ASPNETCORE_URLS=http://0.0.0.0:5000
dotnet run
```

**Program.cs / Kestrel ( .NET 6+ ):**

```csharp
builder.WebHost.ConfigureKestrel(o =>
    o.ListenAnyIP(5000)); // 0.0.0.0
// vs ListenLocalhost(5000) → 127.0.0.1
```

## Examples

### Example 1: Refused from Phone, Works Locally

Kestrel `http://localhost:5000` → binds loopback only. Phone uses `http://192.168.1.50:5000` → SYN to LAN interface, nothing listening → **connection refused** (`0.4.3`). Fix: `ListenAnyIP` + allow firewall.

### Example 2: 0.0.0.0 in logs

Log "Now listening on: http://0.0.0.0:8080" means **all interfaces**, not "URL is http://0.0.0.0:8080". Users still browse `http://localhost:8080` or machine hostname.

## Common Mistakes & Anti-patterns

**Using `0.0.0.0` in browser URL** — invalid destination for clients.

**Binding loopback in Docker/K8s** expecting cluster traffic — probes fail.

**Assuming localhost ≡ 127.0.0.1 always** — IPv6-only bind breaks `localhost` resolution order.

**Exposing `0.0.0.0` on untrusted networks** without auth — dev convenience becomes security hole.

**Confusing loopback with "no network"** — loopback is network stack; still TCP/IP (`0.4.3`).

## Production & Real-World Notes

Production Kestrel often listens **`127.0.0.1` or Unix socket** behind nginx/IIS reverse proxy on same host — only proxy faces public. Containers in K8s listen `0.0.0.0:8080` inside pod; Service routes cluster traffic.

Health checks: liveness probe hits pod IP, not loopback of node (unless hostNetwork).

Document which bind pattern each environment uses in runbooks.

## Comparison / Trade-offs

| Bind | Pros | Cons |
|------|------|------|
| 127.0.0.1 | Secure local only | Not reachable remotely |
| 0.0.0.0 | Dev friendly, containers | Exposed on all NICs if firewall open |
| Specific NIC IP | Controlled exposure | Breaks if IP changes (DHCP) |

## Quick Reference

| Term | Role |
|------|------|
| `127.0.0.1` / `::1` | Loopback address — same machine |
| `localhost` | Hostname → usually loopback |
| `0.0.0.0` | Bind all IPv4 interfaces (listen only) |
| `[::]` | Bind all IPv6 |

**Client connects to specific IP or resolvable name — never `0.0.0.0`.**

---

## Key Takeaways

- Loopback (`127.0.0.1`, `::1`) keeps traffic inside the host — ideal for local dev defaults.
- `localhost` resolves to loopback via hosts/DNS; IPv4/IPv6 bind mismatch causes subtle failures.
- `0.0.0.0` in listen means all interfaces; it is not a valid client target URL.
- Remote access requires binding non-loopback (or `0.0.0.0`) plus firewall and correct LAN/public IP in URL.
- Docker/K8s services should bind broadly inside namespace; loopback-only breaks publishing.
- HTTP/TCP from `0.3`–`0.4.3` still applies — loopback is just another IP destination.
