# Ковariance и contravariance: `in`, `out` и `IEnumerable

> Roadmap: `1.3.2` · Node: `1.3` — C# продвинутые темы · Depth: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Объяснить **variance** через **substitutability** (можно ли подставить более конкретный тип вместо общего).
- Различать **covariance** (`out T`), **contravariance** (`in T`) и **invariance** (по умолчанию).
- Предсказывать assignability для **`IEnumerable<T>`**, **`List<T>`**, **`Action<T>`**, **`Func<T, TResult>`**.
- Объявлять **variant generic interfaces** с модификаторами `in`/`out`.
- Описать **CLR enforcement** variance и почему mutable collections не covariant.
- Избегать ловушек: arrays, value types, nested generics.

## Why This Matters

Этот код компилируется:

```csharp
IEnumerable<string> names = new List<string> { "Ada" };
IEnumerable<object> objects = names; // covariance
```

А этот — нет:

```csharp
List<string> strings = new();
List<object> objects = strings; // error — invariant
```

Понимание **почему** экономит часы рефакторинга и объясняет дизайн .NET: **`IEnumerable<out T>`** covariant; **`List<T>`** invariant, потому что в него **пишут**. Variance нужна при проектировании read-only API, LINQ, **`Action<object>`** вместо **`Action<string>`** (contravariance).

## Core Concepts

### Subtyping и substitution

Если **`Dog` : `Animal`**, то `Animal a = new Dog()` — OK. Вопрос generics: **`List<string>`** — это **`List<object>`**? Тогда можно **`Add(new object())`** в «список строк» — safety ломается. **`List<T>` invariant**.

Variance applies к **generic type parameters** в interfaces и delegates, не к arbitrary classes вроде `List<T>`.

### Covariance: `out T` — producers

Если `Derived` : `Base`, то **`IProducer<Derived>`** → **`IProducer<Base>`** — данные **выходят** из API (**out**).

```csharp
IEnumerable<string> strings = new[] { "a" };
IEnumerable<object> objects = strings; // OK
```

Interface с `out T` не может иметь methods, принимающие `T` как input (кроме special cases).

**Arrays** для reference types — **covariant** (legacy):

```csharp
object[] arr = new string[3]; // compiles — опасно!
// arr[1] = 42; // ArrayTypeMismatchException
```

### Contravariance: `in T` — consumers

**`IConsumer<Base>`** substitutes **`IConsumer<Derived>`** — **consume** менее specific types.

```csharp
Action<object> printObject = obj => Console.WriteLine(obj);
Action<string> printString = printObject; // OK — contravariance
```

Handler принимает `object` — значит справится со `string`.

**`Func<in T, out TResult>`** — input contravariant, return covariant.

### Invariance

**`List<T>`**, **`Dictionary<,>`** — invariant: read **and** write `T`. Generic **classes** не могут объявлять variant parameters — только **interfaces** и **delegates**.

### Declaring variant interfaces

```csharp
public interface IProducer<out T> { T GetNext(); }
public interface IConsumer<in T> { void Accept(T item); }
```

**`IReadOnlyList<out T>`** — covariant (getters only). **`IComparer<in T>`** — contravariant.

## Under the Hood

### CLR rules

CLI variance flags (`+`/`-`). Verifier rejects implementations, нарушающие variance. **`out T`** — `T` не в parameter positions (кроме return-only).

### Почему List invariant

Hypothetical covariant `List<out T>` → запись `int` в `List<string>` через `List<object>`. Compile-time block.

### Delegates

**`Action<in T>`**, **`Func<in T, out TResult>`** — variance metadata как у interfaces.

### Limitations

- **Value types** — `IEnumerable<int>` → `IEnumerable<object>` **illegal** (boxing меняет semantics).
- **Nested generics** — outer `List<>` invariant; inner covaries только на interface assign.
- Variance только для **reference type** substitutions at conversion site.

## Common Mistakes & Anti-patterns

- Думать, что **все** generics covariant как arrays.
- Cast `List<string>` к `List<object>` — unsound.
- Covariance для **value types** — не работает.
- `out T` с `void Add(T)` — compile error.
- Путать направление contravariance в памяти.

## Production & Real-World Notes

Public API — **`IReadOnlyList<T>`**, **`IEnumerable<T>`** вместо `List<T>`.

LINQ returns **`IEnumerable<T>`** для composability.

**`IComparer<in T>`** — contravariance в sorting APIs.

## Comparison / Trade-offs

| Type | Variance |
|---|---|
| `List<T>` | Invariant |
| `IEnumerable<T>` | Covariant |
| `Action<T>` | Contravariant |
| `Func<T,R>` | in T, out R |

---

## Quick Reference

| Keyword | Variance | Mnemonic |
|---|---|---|
| `out T` | Covariant | T **out** to caller |
| `in T` | Contravariant | T **in** from caller |

**PECS**: Producer **out**, Consumer **in**.

---

## Key Takeaways

- Variance — следует ли generic subtyping за subtyping type arguments.
- **`out`** producer, **`in`** consumer, default **invariant**.
- **`List<T>`** invariant из-за mutation.
- Arrays covariant — legacy footgun.
- Value types — отдельная история с boxing.
