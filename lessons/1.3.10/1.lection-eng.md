# Immediate Execution — ToList, Count, First

> Roadmap: `1.3.10` · Node: `1.3` — C#: advanced topics · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Classify LINQ operators as **deferred** vs **immediate** (terminal).
- Use **`ToList`**, **`ToArray`**, **`ToDictionary`** to **materialize** sequences.
- Apply **`Count`**, **`LongCount`**, **`Any`**, **`All`** for existence/quantity without always loading full lists.
- Choose **`First`**, **`FirstOrDefault`**, **`Single`**, **`SingleOrDefault`** by semantic intent (`1.3.27` deep dive later).
- Explain when **`Count()`** on **`ICollection<T>`** is **O(1)** vs full scan through **`Where`**.
- Materialize at the **right boundary** in ASP.NET Core pipelines (before JSON serialization, after query complete).
- Connect materialization to EF **`ToListAsync()`** / **`CountAsync()`** (preview).

---

## Why This Matters

Lesson `1.3.9` established deferral: pipelines wait. Production code must **decide when to pull the trigger**. Too early — load millions of rows into RAM. Too late — double database round-trips, disposed **`DbContext`**, or unstable iterators.

**Immediate execution** operators are that trigger. They force the enumerator to run (or use fast paths) and often produce a **concrete data structure** or **scalar**. API actions typically end with **`ToListAsync()`** on a shaped **`Select`** projection, not raw **`IQueryable`** passed to the serializer.

This lesson is **understanding** depth: know the categories, defaults, and pitfalls — not every overload edge case (those spread across `1.3.27`, EF chapters).

## Core Concepts

### Terminal Operators Force Execution

Any call that must **produce a final answer** consumes the deferred chain:

| Category | Examples | Result |
|----------|----------|--------|
| Materialize | `ToList`, `ToArray`, `ToDictionary`, `ToHashSet` | In-memory collection |
| Scalar aggregate | `Count`, `Sum`, `Max`, `Min`, `Average` | Number |
| Element pick | `First`, `Single`, `Last`, `ElementAt` | One item or throw |
| Predicate test | `Any`, `All` | `bool` |

Until one of these (or **`foreach`**) runs, **`Where`/`Select`** remain plans.

### Materialization — Snapshot

```csharp
List<User> snapshot = users
    .Where(u => u.IsActive)
    .OrderBy(u => u.Name)
    .ToList();
```

**`ToList()`** allocates a **`List<T>`**, fills it by enumerating the entire upstream pipeline, and returns. Further changes to **`users`** do not affect **`snapshot`**. Re-iterating **`snapshot`** is cheap and consistent.

**`ToArray()`** — similar but fixed-size array; slightly less overhead in some hot paths.

**`ToDictionary(key)`** — materialize with key uniqueness enforced; duplicate keys throw.

Choose **`ToList`** when you need growable collection or will pass to APIs expecting **`List<T>`**; **`ToArray`** for immutable-size interop.

### Count vs Any — Do You Need the Number?

```csharp
if (query.Any()) { ... }           // stops at first match (usually)
int n = query.Count();             // full scan unless optimized
bool empty = query.Count() == 0; // worse than !Any()
```

**`Any()`** without predicate tests whether enumeration produces at least one element — can short-circuit.

**`Count()`** on **`ICollection<T>`** or **`List<T>`** through **`Enumerable.Count`** extension may use **`collection.Count`** in **O(1)** — but **`users.Where(...).Count()`** wraps with **`Where`** iterator → **full scan**.

**`LongCount`** for sequences longer than **`int.MaxValue`** (rare in business apps).

### First and FirstOrDefault

```csharp
var user = users.First(u => u.Id == id);           // throws if none
var user = users.FirstOrDefault(u => u.Id == id); // null if none (reference type)
```

**`First`** — exactly one expected; throws **`InvalidOperationException`** if empty (or **`ArgumentNullException`** patterns with OrDefault variants on value types default).

Use **`First`** when absence is exceptional; **`FirstOrDefault`** when absence is normal. **`Single`**/`Single`** enforce uniqueness — later `1.3.27`.

### Foreach Is Immediate (Per Iteration)

```csharp
foreach (var x in query)  // executes pipeline
    Process(x);
```

Each **`foreach`** is a full enumeration unless **`query`** is already materialized.

### Multiple Terminal Calls — Multiple Executions

```csharp
var q = orders.Where(o => o.Open);
var count = q.Count();      // scan 1
var list = q.ToList();      // scan 2
```

Unless **`orders`** is trivial, prefer:

```csharp
var list = orders.Where(o => o.Open).ToList();
var count = list.Count;     // O(1) on List
```

### Where to Materialize in a Web Request

Typical ASP.NET Core handler:

```csharp
var dtos = await _db.Orders
    .Where(o => o.CustomerId == id)
    .OrderByDescending(o => o.Date)
    .Select(o => new OrderDto(o.Id, o.Total))
    .ToListAsync(cancellationToken);

return Ok(dtos);
```

Materialize **after** filtering, sorting, and projection — **before** JSON serializer. Do not return **`IQueryable`** from controllers (execution undefined/disposed context).

In pure memory LINQ, same rule: materialize before crossing module boundaries (cache, parallel **`Parallel.ForEach`**, multiple consumers).

## Under the Hood

### `ToList()` Implementation Sketch

LINQ **`ToList`** roughly:

1. Allocate empty **`List<T>`**.
2. **`foreach`** over source, **`Add`** each element.
3. Return list (may resize internal array as grows).

Single allocation pattern; no magic sharing with source.

### Short-Circuiting

**`Any(predicate)`** can stop at first matching element — **`MoveNext`** stops early.

**`First`** with predicate — stops at first match.

**`Count`** — must visit every element unless source implements **`ICollection<T>`** without wrapper operators.

## Examples

### Example 1 — Pagination materialization

```csharp
var page = allUsers
    .OrderBy(u => u.Id)
    .Skip(pageIndex * pageSize)
    .Take(pageSize)
    .ToList();
```

One enumeration produces exactly **`pageSize`** items (after sort buffer inside **`OrderBy`**).

### Example 2 — Avoid double DB hit (EF preview)

```csharp
// Bad
if (query.Any())
    return await query.ToListAsync();

// Better
var list = await query.ToListAsync();
if (list.Count == 0) return NotFound();
```

Or use **`AnyAsync`** only when you truly do not need data afterward.

## Common Mistakes & Anti-patterns

**`Count() == 0`** instead of **`!Any()`** on large filtered sequences.

**Materializing before final `Where`/`Select`** — pulls unnecessary data.

**Assuming `Count()` is always fast** — not through **`Where`**.

**Returning deferred `IEnumerable` from API** — serializer may enumerate multiple times or after request scope ends.

## Production & Real-World Notes

Log query + row counts in EF with **`ToListAsync`** result **`.Count`**, not separate **`CountAsync`** + **`ToListAsync`** unless needed.

For large collections, consider **`IAsyncEnumerable`** streaming instead of **`ToList`** — different pattern for export endpoints.

## Quick Reference

| Need | Operator |
|------|----------|
| Snapshot list | `ToList()` / `ToListAsync()` |
| Fixed array | `ToArray()` |
| How many (full scan OK) | `Count()` |
| Any exists? | `Any()` |
| One item or throw | `First()` |
| One item or default | `FirstOrDefault()` |

**Rule:** one terminal materialization per logical operation when possible.

---

## Key Takeaways

- Terminal operators **end deferral** and run the pipeline.
- **`ToList`** = snapshot, safe reuse, decoupled from source mutations.
- **`Any`** often cheaper than **`Count()`** for emptiness checks.
- **`Count()`** on filtered sequences scans all matches.
- Materialize **once** at API/repository boundary before multiple consumers.
- EF: **`ToListAsync`**, **`CountAsync`**, **`FirstOrDefaultAsync`** — same immediate semantics async.
