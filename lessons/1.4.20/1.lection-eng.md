# Async All the Way — No Sync-over-Async

> Roadmap: `1.4.20` · Node: `1.4` — C# async · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Define **sync-over-async** and **async-over-sync** and why both harm ASP.NET Core.
- Trace a request through **Kestrel → middleware → controller → service → EF** as fully async.
- Refactor **`Task.Result`**, **`.Wait()`**, **`GetAwaiter().GetResult()`** call sites to **`await`**.
- Explain **thread pool starvation** when sync blocks wait for async work (**`1.4.10`**, **`1.4.14`**).
- Apply **async all the way** in DI, filters, middleware, and library boundaries.
- Recognize **legacy API bridges** and isolate them at the edges with documented risk.

---

## Why This Matters

ASP.NET Core has **no `SynchronizationContext`** that marshals back to a request thread — but it **does** use **thread pool threads** for request processing. Calling **`.Result`** on an incomplete task **blocks** that thread while the async work needs **another** pool thread to complete → under load, **pool exhaustion**: requests queue, latency spikes, health checks fail.

The mantra **"async all the way"** means: from the first **`await`** in the request delegate down to **`DbContext`**, **`HttpClient`**, and **`Stream`**, use **`async Task`** and **`await`** — no blocking waits in between. Sync-over-async is a **scalability bug**, not a style preference.

Junior codebases often "fix" CS4033 by wrapping in **`Task.Run`** — hides deadlock locally but **doubles thread usage** and masks architectural debt. Middle developers refactor call chains and push blocking legacy SDKs to **process edges** (background workers, dedicated threads), not request paths.

## Core Concepts

### Sync-over-async defined

**Sync-over-async:** synchronous code **blocks** waiting for asynchronous work:

```csharp
// BAD in ASP.NET request
public IActionResult Get(int id)
{
    var order = _service.GetOrderAsync(id).Result; // blocks thread
    return Ok(order);
}
```

The blocked thread cannot serve other work efficiently; the async continuation may need a **free** thread pool thread — classic **deadlock** with UI **`SynchronizationContext`**, **starvation** in ASP.NET under concurrency.

**Async-over-sync:** **`async`** method that only calls sync CPU/IO via **`Task.Run`** to fake asynchrony:

```csharp
public async Task<byte[]> ReadFileAsync(string path)
{
    return await Task.Run(() => File.ReadAllBytes(path)); // wastes thread
}
```

Use **`await File.ReadAllBytesAsync`** instead — true async I/O where available.

### The request pipeline

```
Client
  → Kestrel (async IO)
  → Middleware (InvokeAsync)
  → Routing
  → Controller action (async Task<IActionResult>)
  → Service (async)
  → EF Core / HttpClient (async)
  → await response
```

Any **sync block** in this chain reduces throughput at that tier. **Middleware** must be **`Task InvokeAsync`** — not sync **`Invoke`** wrapping **`task.Wait()`**.

### Refactoring patterns

**Controller:**

```csharp
// Before
public IActionResult Get(int id) => Ok(_svc.GetAsync(id).Result);

// After
public async Task<IActionResult> Get(int id) => Ok(await _svc.GetAsync(id));
```

**Interface cascade:** if **`GetAsync`** exists, **every caller up the stack** must become async — one sync signature forces **`.Result`** somewhere.

**Library without async API:** prefer **`IAsyncEnumerable`**, add **`FooAsync`**, or isolate:

```csharp
// Edge isolation — NOT in per-request hot path if avoidable
await Task.Run(() => LegacySdk.SyncCall()); // last resort + document
```

### DbContext and HttpClient

**EF Core:** **`ToListAsync`**, **`SaveChangesAsync`**, **`ExecuteSqlAsync`** — never **`ToList()`** on request path for large queries.

**HttpClient:** **`GetStringAsync`**, **`SendAsync`** — **`Send`** + **`.Result`** exhausts sockets and threads.

### Task.Run is not the fix

**`Task.Run(() => api.GetAsync().Result)`** — still blocking inside pool thread + extra scheduling overhead. Fixes UI deadlock sometimes; **wrong** for ASP.NET scalability.

When CPU-bound work is required, **`Task.Run`** with ** throttling** (**`1.4.18`**) is intentional offload — distinct from masking sync-over-async on I/O.

### Testing and constructors

**Constructors cannot be async** — use **`IHostedService`**, factory pattern, or sync init only for cheap work. Don't **`.Result`** in ctor (**`1.4.38`** interview trap).

Integration tests: **`WebApplicationFactory`** supports **`async Task`** test methods — keep tests async too.

## Under the Hood

Blocking **`Wait()`** on incomplete task:

1. Current thread enters wait state.
2. Async continuation queues to thread pool when I/O completes.
3. If pool is starved (all threads blocked in **`.Wait()`**), continuation never runs → **self-deadlock**.

ASP.NET Core **doesn't** capture request context on **`SynchronizationContext`** (null by default since .NET Core) — UI-style deadlock less common, **starvation** remains.

**`ConfigureAwait(false)`** in library code avoids unnecessary context captures where context exists (legacy ASP.NET, UI) — doesn't replace async all the way.

## Syntax / API

| Anti-pattern | Replacement |
|--------------|-------------|
| `.Result` | `await` |
| `.Wait()` | `await` |
| `GetAwaiter().GetResult()` | `await` |
| Sync action `IActionResult` | `async Task<IActionResult>` |
| `Task.Run` for I/O | Native `*Async` API |
| Sync middleware | `InvokeAsync` + await |

## Examples

### Service layer propagation

```csharp
public interface IOrderService
{
    Task<OrderDto> GetOrderAsync(int id, CancellationToken ct);
}

public class OrderController : ControllerBase
{
    [HttpGet("{id}")]
    public async Task<ActionResult<OrderDto>> Get(int id, CancellationToken ct)
    {
        var order = await _orders.GetOrderAsync(id, ct);
        return order is null ? NotFound() : Ok(order);
    }
}
```

Changing controller to async **forced** async service — intentional cascade.

### Legacy SDK bridge (edge only)

```csharp
public class LegacyWrapper
{
    private readonly Channel<LegacyJob> _queue = Channel.CreateBounded<LegacyJob>(100);

    // Background hosted service drains queue calling sync SDK
    // HTTP handler: await _queue.Writer.WriteAsync(job); return 202 Accepted
}
```

Sync SDK runs off request thread — async all the way **on the web path**.

## Common Mistakes & Anti-patterns

**Partial async** — async controller calling sync repository with **`GetById`** blocking on DB.

**`.GetAwaiter().GetResult()` "because Result is banned"** — same blocking semantics.

**Blocking in **`lock`** then calling async** — deadlock + pool issues.

**Sync **`IEnumerable`** return loading EF with lazy eval during serialization** — use **`ToListAsync`** before return or async stream.

**Assuming **`Task.Run`** makes I/O scalable** — it doesn't.

## Production & Real-World Notes

Enable **sync-over-async detection** in code review and analyzers where possible.

Monitor **thread pool queue length** and **active threads** during load tests — spike correlates with blocking.

Refactor priority: **middleware**, **auth handlers**, **EF queries**, **outbound HTTP** — highest traffic first.

Document **only** approved sync islands (e.g., startup **`Main`**, legacy worker).

## Comparison / Trade-offs

| Strategy | Throughput | Complexity |
|----------|------------|------------|
| Async all the way | Best for I/O | Requires async signatures throughout |
| `.Result` in controller | Poor under load | "Simple" locally |
| `Task.Run` bridge | Mediocre | Hides debt |
| Dedicated sync worker | Good for legacy | Architecture split |

## Key Takeaways

- **Sync-over-async** blocks pool threads → starvation at scale.
- **Async all the way** = async signatures + await through the request chain.
- **`.Result`/`.Wait()`** are equivalent hazards — eliminate, don't rename.
- **`Task.Run`** for I/O is a code smell; use native async APIs.
- Interface changes **cascade** — refactor callers together.
- Isolate unavoidable sync legacy **off** the request path.
