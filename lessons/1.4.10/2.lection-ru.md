# Thread pool starvation — симптомы и исправления

> Roadmap: `1.4.10` · Node: `1.4` — Async/await · Depth: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Описать, как **.NET thread pool** планирует work и почему **blocking** на pool threads вызывает **starvation**.
- Распознать **симптомы**: latency spikes, «лечение» через `SetMinThreads`, hung requests под нагрузкой.
- Отличить starvation от **SyncContext deadlock** (`1.4.9`) и **sync-over-async** (`1.4.14`).
- Применять fixes: **async all the way**, bounded **`Task.Run`** для CPU, не **`Thread.Sleep`** в request path.
- Связать starvation с **continuations state machine** на pool (`1.4.7`).

---

## Why This Matters

ASP.NET Core убрал SyncContext — teams думают, deadlock невозможен. Под load apps **замирают**: health checks timeout, очередь Kestrel растёт. Часто **thread pool starvation** — все workers blocked synchronously, тысячи continuations ждут thread для `MoveNext`.

Middle-разработчик читает **dotnet-counters** и ищет blocking в async controllers. Starvation — **production scalability** failure, не trivia.

---

## Core Concepts

### Thread pool basics

CLR **thread pool** — worker threads + **global queue** (+ local queues). `Task.Run`, timers, ASP.NET processing, **async continuations** без SyncContext — всё pool.

**`.Join()`**, **`.Wait()`**, **`.Result`**, **`Thread.Sleep`**, sync **`HttpClient`**, long **CPU** без offload — thread **не** берёт новую work. Under burst **all workers block** — **starvation**.

Async I/O не держит thread на kernel wait. Проблема — **sync blocking + async** на том же pool, который resume'ит state machines.

### Симптомы

- P99 spikes, **CPU moderate** (threads blocked).
- **`PendingWorkItemCount`** растёт.
- **`SetMinThreads`** временно помогает — smell.
- **`async` never completes** — continuations без thread.
- Hang после **`Task.WhenAll`** с hidden sync внутри «async».

State machine: await completion → **`MoveNext`** в queue pool. Нет free thread → задержка.

### Причины в ASP.NET Core

1. Sync-over-async — `.Result`, `.Wait()`.
2. **`Task.Run().Wait()`** при exhausted pool.
3. **`Thread.Sleep`**, long **`lock`** на request path.
4. Sync repo в **`Task.Run`** на каждый request.
5. Cold start + slow injection (реже на modern .NET).

### Fixes

**Primary:** **`await`** end-to-end. Legacy SDK — isolate blocking с **limited parallelism**.

**CPU:** **`await Task.Run(...)`** + **`SemaphoreSlim`** throttle.

**ConfigureAwait** не добавляет threads — не fix starvation.

**Tuning:** **`SetMinThreads`** для cold burst — last resort с metrics.

---

## Under the Hood

Pool **inject'ит** threads slowly (hill-climbing). Blocking на всех workers опережает injection.

**IOCP** для async I/O; continuations часто на pool для `MoveNext`.

**Lock contention** на singletons — тот же symptom class.

## Examples

```csharp
// BUG under load
public async Task<Order> GetOrderAsync(int id) =>
    await Task.Run(() => _legacySyncRepo.GetOrder(id));

// Middleware BUG
_auth.Validate(ctx).Wait();
```

Fix: async API, **`await ValidateAsync`**.

---

## Key Takeaways

- Pool **finite**; block на pool → все `MoveNext` страдают.
- Starvation ≠ SyncContext (`1.4.9`).
- Fix: async all the way, убрать `.Wait()` (`1.4.14`).
- **`SetMinThreads`** — mitigation.
- Diagnose: **dotnet-counters**.
