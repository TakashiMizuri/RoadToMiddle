# DNS: Resolver, A/AAAA/CNAME Records, and TTL

> Roadmap: `0.4.5` · Node: `0.4` — Networking basics · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Explain A, AAAA, and CNAME record types and when each is appropriate.
- Describe TTL (time to live) caching behavior and its impact on deployments and incident response.

## Why This Matters

Every HTTPS call to `https://api.contoso.com` begins with a question: **what IP address is `api.contoso.com`?** Until DNS answers, TCP cannot connect (`0.4.3`), TLS cannot start (`0.3.8`), and HTTP never runs (`0.3`). DNS is the **phonebook of the internet** — application layer protocol (`0.4.1`) typically carried over **UDP** port 53 (`0.4.4`).

Deploying a new API behind a load balancer, migrating cloud regions, or rotating disaster-recovery endpoints all require DNS updates. Set TTL too high and clients hit old IPs for hours after cutover. Set TTL too low and you increase query load and latency. CNAME to CDN (`api` → `d123.cloudfront.net`) simplifies ops but adds lookup steps and constraints at apex domains.

Middle developers configure `ConnectionStrings`, CORS origins, cookie domains, and TLS certificates using **hostnames**, not IPs — DNS correctness is production availability. "Site works on my machine" often differs because `/etc/hosts` or corporate resolver cache bypasses public DNS.

This lesson connects naming (`0.3.2`) to addressing (`0.4.2`) and explains caching mechanics under the hood.

## Core Concepts

### What DNS Solves

Humans remember **names**; IP networks route **addresses**. DNS maps hierarchical names like `www.example.com` to records (IPv4, IPv6, aliases, mail servers, etc.). The namespace is a **tree**: root `.` → TLD `com` → `example.com` → `www.example.com`.

Resolution is **distributed** — no single global database. Each zone owner publishes records on **authoritative nameservers**; resolvers cache answers worldwide.

### Resolution Actors

**Stub resolver (client)** — library in OS or browser (`getaddrinfo`, Chrome async DNS). Sends query to configured **recursive resolver** (ISP, `8.8.8.8`, `1.1.1.1`, corporate AD DNS).

**Recursive resolver** — walks the tree: root hints → TLD servers → authoritative NS for zone → final answer. Caches results per **TTL**. Returns A/AAAA to client.

**Authoritative nameserver** — holds zone file / DNS provider API for `example.com`. Answers authoritatively; sets TTL on records.

**Registrar** — where you register domain; points to nameserver hostnames (glue records).

Flow for browser → `api.shop.com`:

```
Browser stub → Recursive resolver (cache?) 
  → if miss: root → .com TLD → shop.com NS → authoritative 
  → A 203.0.113.55 → cached → returned to stub → TCP connect
```

### Record Types You Must Know

**A (Address)** — IPv4 address for name. `api.shop.com. IN A 203.0.113.55`

**AAAA** — IPv6 address. Dual-stack hosts often publish both; client chooses per Happy Eyeballs algorithm.

**CNAME (Canonical Name)** — alias: name points to another hostname, not directly to IP. `www.shop.com CNAME shop.com.` Final A/AAAA fetched for target. **Cannot** CNAME at zone apex (`shop.com` bare) in classic DNS — use ALIAS/ANAME at provider or A record at apex.

Other records (preview): **MX** mail, **TXT** verification/SPF, **NS** delegation, **SRV** service location.

### TTL — Cache Duration

Each DNS record includes **TTL** in seconds. Resolvers and stub caches store answer until TTL expires. `TTL=300` → 5 minutes cached.

**Low TTL (60–300s)** before migration — faster cutover; more queries, slightly higher latency variance.

**High TTL (3600+)** — stable CDN front; slow propagation on misconfiguration.

**Negative caching (NXDOMAIN)** also has TTL — "name does not exist" cached briefly.

After deploy, "some users still on old IP" is often **TTL + resolver hierarchy**, not app bug.

### DNS Transport: UDP and TCP (`0.4.4`)

Most queries: single UDP datagram to port 53. Large responses (DNSSEC, many records) or **truncation (TC bit)** → retry over **TCP**. Firewalls must allow both for robust resolution.

### Linking to HTTP Stack (0.3)

From `0.3.2` URL `https://api.shop.com/v1/items`:

1. **Resolve** `api.shop.com` → A/AAAA (this lesson).
2. **TCP connect** to IP:443 (`0.4.3`).
3. **TLS** SNI sends hostname `api.shop.com` (`0.3.8`) — certificate must match name even if IP changes.
4. **HTTP** Host header often same name (`0.3.7`).

Changing DNS without updating TLS cert or CORS allowed origins breaks step 3–4 while step 1 "succeeds."

## Under the Hood

### Recursive Resolution Step-by-Step

Resolver starts with **root server hints** (hardcoded or configured). For `api.shop.com`:

1. Query root: "where is `.com`?" → referral to `.com` TLD NS.
2. Query `.com` TLD: "where is `shop.com`?" → referral to `shop.com` authoritative NS (e.g., `ns1.dnsprovider.net`).
3. Query authoritative: "A for `api.shop.com`?" → answer section with A record + TTL.

**CNAME chain:** if `api.shop.com CNAME lb.cloudprovider.com`, resolver continues until A/AAAA (**CNAME chaining limit** ~10 hops enforced).

**DNSSEC (concept):** cryptographic chain of trust; larger responses → more TCP.

### Cache Layers

| Layer | Location | Effect |
|-------|----------|--------|
| Browser cache | Chrome in-memory | Ignores low TTL sometimes; chrome://net-internals |
| OS stub cache | Windows DNS client | getaddrinfo hits cache |
| Recursive resolver | ISP / 8.8.8.8 | Shared by many users |
| Authoritative | None (source of truth) | Defines TTL |

**Flush** on dev machine: `ipconfig /flushdns`. Production incident: reduce TTL **before** change (DNS pre-staging).

### Query Message Format (Conceptual)

Header: ID, flags (QR query/response, RD recursion desired), counts. Question section: QNAME, QTYPE (A=1, AAAA=28, CNAME=5). Answer section: RRs with NAME, TYPE, TTL, RDATA.

Tools decode this: `dig +trace api.shop.com`, Wireshark filter `dns`.

### Split-Horizon and Private Zones

Corporate **internal DNS** returns private IP for `db.corp.local`; public resolver returns NXDOMAIN or different IP. Kubernetes **CoreDNS** serves cluster.local. Same name different answers by resolver context — explains "works in VPN, fails at home."

### Failure Modes

- **NXDOMAIN** — name does not exist (typo, deleted record).
- **SERVFAIL** — resolver/authoritative error ( lame delegation, DNSSEC fail).
- **Stale cache** — old TTL after emergency IP change.
- **CNAME + other records** — RFC violation at same name (MX + CNAME apex issues).
- **Circular CNAME** — resolution fails.

## Examples

### Example 1: Blue-Green via DNS

`api.example.com A` switches from `198.51.100.10` (blue) to `198.51.100.20` (green). TTL was 3600 — worst case 1 hour stragglers. Pre-lowering TTL to 60 one day before reduces window.

### Example 2: CDN CNAME

`www.example.com CNAME d111111abcdef8.cloudfront.net` — IPv4 changes at CDN; you update provider, not A record. Extra lookup latency negligible vs CDN benefit.

## Common Mistakes & Anti-patterns

**Hard-coding IP instead of hostname** in config — breaks CDN failover and cert validation.

**CNAME at apex** (`example.com` bare) — many providers forbid; use ALIAS.

**Forgetting AAAA** — IPv6-only clients fail if only A published.

**Assuming instant global DNS change** — respect TTL at every cache tier.

**Ignoring DNS in CI** — integration tests using real DNS flaky; use hosts or test containers.

## Production & Real-World Notes

Use managed DNS (Route53, Cloudflare, Azure DNS) with IaC (Terraform). **Health checks** + weighted routing for failover. Monitor resolution latency from multiple regions (synthetic checks).

**Certificate issuance** (Let's Encrypt DNS-01) requires TXT records — DNS as security control.

Kubernetes **Ingress** host rules assume cluster DNS or external resolver reaches Service via published records.

## Comparison / Trade-offs

**A vs CNAME at subdomain:** A → direct IP control; CNAME → indirection, easier CDN swap.

**Low vs high TTL:** agility vs performance/load.

**Public recursive vs corporate:** split-horizon complexity.

## Key Takeaways

- DNS maps hostnames to addresses before TCP/HTTP/TLS can run.
- Recursive resolvers cache answers until TTL expires; stale cache causes partial outages.
- A/AAAA return addresses; CNAME aliases to another name (no CNAME at classic apex).
- UDP 53 primary; TCP fallback for large/truncated responses (`0.4.4`).
- TLS SNI and HTTP Host depend on correct name resolution (`0.3.8`, `0.3.2`).
- Plan TTL ahead of migrations; flush only fixes local stub cache.
