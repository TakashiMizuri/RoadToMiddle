# Composition over Inheritance

> Roadmap: `1.2.11` · Node: `1.2` — C# OOP · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Explain **why inheritance is not the default reuse mechanism** in professional C# design.
- Distinguish **"is-a"** relationships (inheritance) from **"has-a"** relationships (composition).
- Recognize the **fragile base class problem** and how subclass overrides break base-class assumptions.
- Refactor an inheritance hierarchy into **composition + delegation** without losing polymorphic behavior where needed.
- Choose between **inheritance**, **composition**, and **interfaces** using a practical decision guide.
- Apply composition in ASP.NET Core patterns: **middleware pipeline**, **decorator-style services**, **strategy injection**.

---

## Why This Matters

In `1.2.6`–`1.2.7` you learned inheritance and polymorphism: `base`, `override`, virtual dispatch at runtime. Those tools are powerful — and dangerous when used as the first answer to every design question. Junior codebases often grow deep inheritance trees: `Animal` → `Mammal` → `Dog` → `GoldenRetriever`, or worse, `BaseController` → `CrudController` → `ProductController` with business logic creeping into abstract parents.

The industry mantra **"favor composition over inheritance"** does not mean "never inherit." It means: **inheritance couples you to implementation details of ancestors**; **composition couples you to behavior you explicitly choose**. When Microsoft ships ASP.NET Core, most extensibility is composition (inject `IMiddleware`, wrap `HttpClient` with `DelegatingHandler`, compose `IAuthorizationHandler` lists). Inheritance appears where the framework truly models substitutability (`Exception`, `Stream`, `DbContext`).

A Middle developer must refactor a fragile hierarchy without panic, explain *why* a `new` keyword or `override` broke production, and know when a sealed base class plus interface is the right trade-off.

## Core Concepts

### Inheritance Recap: What You Actually Buy

When class `Dog : Animal`, the compiler and runtime establish a permanent relationship. `Dog` inherits fields, methods, and contracts from `Animal`. Polymorphism from `1.2.7` lets you write `Animal a = new Dog(); a.Speak();` and get the dog's override. That is **substitutability**: code written for `Animal` can run with any derived type.

The price is **tight coupling**. `Dog` depends on every protected member, every virtual method's semantics, and every future change in `Animal`. The base class author and every subclass author are now on the same team forever. Change `Animal.Eat()` to validate calories before swallowing, and a subclass that overrode `Eat()` without calling `base.Eat()` may silently violate the new invariant. Change a `protected` field to `private` and subclasses break at compile time. Add a new abstract method and **every** concrete descendant must implement it immediately.

Inheritance is a **compile-time contract about an entire family of types**. That is appropriate when the domain truly has a stable taxonomy and shared implementation — but rare in application code that changes every sprint.

### Composition: Building Behavior from Parts

**Composition** means an object **has** other objects and **delegates** work to them instead of inheriting implementation. A `EmailNotificationService` does not inherit from `SmsNotificationService`; it holds an `INotificationSender` (or several) and calls them. The outer object controls which parts exist and how they interact.

```csharp
public sealed class OrderService
{
    private readonly IOrderRepository _repository;
    private readonly IPaymentGateway _payments;
    private readonly IEmailSender _email;

    public OrderService(
        IOrderRepository repository,
        IPaymentGateway payments,
        IEmailSender email)
    {
        _repository = repository;
        _payments = payments;
        _email = email;
    }

    public async Task PlaceOrderAsync(Order order, CancellationToken ct)
    {
        await _payments.ChargeAsync(order.Total, ct);
        await _repository.SaveAsync(order, ct);
        await _email.SendAsync(order.CustomerEmail, "Order placed", ct);
    }
}
```

`OrderService` is not a "kind of" repository or payment gateway. It **uses** them. You can swap `IPaymentGateway` for a test fake without touching `OrderService`'s inheritance chain — because there is none. This is the daily shape of ASP.NET Core DI: constructors receive capabilities; classes compose behavior.

### "Is-a" vs "Has-a"

The classic heuristic: inheritance models **is-a** (a `GoldenRetriever` **is a** `Dog`), composition models **has-a** (a `Car` **has an** `Engine`). The heuristic fails when people force is-a for code reuse only.

Ask: **Would it be nonsense in the real domain to substitute the parent for the child everywhere?** If `Manager` inherits `Employee` for shared `Name` and `Id`, substitution often works. If `Stack` inherits `List<T>` to reuse storage — as early Java did — you get `stack.Push(1); stack.Insert(0, 2);` destroying stack semantics. That is **not** a true is-a; it was inheritance for convenience.

A sharper test from *Effective Java* (applicable to C#): **Is the subclass a strict subset of the parent's behavior contract, or does it add constraints?** Subtypes must not weaken preconditions or strengthen postconditions in ways callers do not expect. If the child forbids what the parent allows, inheritance is the wrong tool.

### The Fragile Base Class Problem

The **fragile base class problem** names a specific failure mode. The base class maintainer ships a "safe" internal change. Subclasses, written years ago, assumed old behavior. Tests pass on the base library; applications break in production.

Consider a logging framework:

```csharp
public class Logger
{
    public virtual void Log(string message)
    {
        var formatted = Format(message);
        WriteToSink(formatted);
    }

    protected virtual string Format(string message) => $"[{DateTime.UtcNow:O}] {message}";
    protected void WriteToSink(string text) => Console.WriteLine(text);
}

public class JsonLogger : Logger
{
    protected override string Format(string message)
        => $"{{\"ts\":\"{DateTime.UtcNow:O}\",\"msg\":\"{message}\"}}";
}
```

Later, `Logger.Log` is updated to call `Validate(message)` before formatting — except `JsonLogger` never knew validation existed and still overrides only `Format`. Or `Log` begins batching writes and expects `Format` to be pure; `JsonLogger` side-effects. The base class cannot evolve without auditing every subclass in every consumer assembly.

With composition:

```csharp
public interface ILogFormatter { string Format(string message); }
public interface ILogSink { void Write(string text); }

public sealed class Logger
{
    private readonly ILogFormatter _formatter;
    private readonly ILogSink _sink;

    public Logger(ILogFormatter formatter, ILogSink sink)
    {
        _formatter = formatter;
        _sink = sink;
    }

    public void Log(string message)
    {
        var formatted = _formatter.Format(message);
        _sink.Write(formatted);
    }
}
```

`Logger` owns its algorithm. Swap formatters without subclassing. New validation lives in `Logger.Log` once; no hidden overrides.

### Delegation vs Inheritance for Reuse

**Implementation inheritance** reuses code by merging vtables and protected members. **Delegation** reuses code by holding an instance and forwarding calls.

The **Decorator pattern** is composition in disguise:

```csharp
public interface IOrderValidator
{
    ValidationResult Validate(Order order);
}

public sealed class OrderValidator : IOrderValidator
{
    public ValidationResult Validate(Order order)
    {
        if (order.Lines.Count == 0)
            return ValidationResult.Fail("Empty order");
        return ValidationResult.Ok();
    }
}

public sealed class FraudCheckingValidator : IOrderValidator
{
    private readonly IOrderValidator _inner;

    public FraudCheckingValidator(IOrderValidator inner) => _inner = inner;

    public ValidationResult Validate(Order order)
    {
        if (order.Total > 10_000 && order.IsNewCustomer)
            return ValidationResult.Fail("Manual review required");

        return _inner.Validate(order);
    }
}
```

Register in DI: `services.AddSingleton<IOrderValidator>(sp => new FraudCheckingValidator(new OrderValidator()));`. You stacked behavior without a `FraudCheckingValidator : OrderValidator` hierarchy. Each decorator is small, testable, and optional.

### When Inheritance Remains Correct

Inheritance is justified when:

1. **Framework extension points** require it (`ControllerBase`, `BackgroundService`, `DbContext`, `Exception`).
2. **True substitutability** holds: any code using the base type is correct with any derived type (Liskov Substitution Principle).
3. **Shared algorithm skeleton** with controlled extension points — **Template Method** — and the team owns the whole hierarchy.

```csharp
public abstract class CsvExporterBase
{
    public async Task ExportAsync<T>(IEnumerable<T> rows, Stream output, CancellationToken ct)
    {
        await WriteHeaderAsync(output, ct);
        foreach (var row in rows)
            await WriteRowAsync(row, output, ct);
        await output.FlushAsync(ct);
    }

    protected abstract Task WriteHeaderAsync(Stream output, CancellationToken ct);
    protected abstract Task WriteRowAsync<T>(T row, Stream output, CancellationToken ct);
}
```

Here the **sequence** is fixed; subclasses customize steps. Risk is contained if the base is `abstract`, methods are few, and the hierarchy is shallow and owned by one team.

**Interfaces** (`1.2.8`–`1.2.9`) decouple contract from implementation. Prefer `INotificationSender` + multiple implementations over an `abstract NotificationSender` with deep children unless Template Method genuinely shares non-trivial code.

## Under the Hood

### How Virtual Dispatch Encourages the Wrong Hierarchy

From `1.2.7`, `callvirt` on a virtual method walks the method table at runtime. Inheritance merges those tables down the chain. Each override is a permanent hook in the dispatch path. Tooling (IDE "go to implementation") lists every subclass when you rename a virtual method — cross-assembly ripples.

Composition replaces one vtable indirection with **interface call** on a private field — often devirtualizable by the JIT when the field type is sealed. More importantly, the **graph of dependencies** becomes explicit in the constructor: you can static-analyze who depends on `IPaymentGateway`; inheritance hides dependencies in `protected` members.

### Liskov Substitution Principle (LSP)

**LSP:** if `S` is a subtype of `T`, objects of type `T` should be replaceable with objects of type `S` without breaking correctness.

Violations are subtle:

```csharp
public class Rectangle
{
    public virtual int Width { get; set; }
    public virtual int Height { get; set; }
    public int Area => Width * Height;
}

public class Square : Rectangle
{
    public override int Width
    {
        set { base.Width = value; base.Height = value; }
    }

    public override int Height
    {
        set { base.Width = value; base.Height = value; }
    }
}
```

A function `void Resize(Rectangle r) { r.Width = 5; r.Height = 10; }` expects area 50. Pass a `Square` and area becomes 100. The subtype changed invariants the base promised. **Fix:** do not inherit; use separate types or a `IShape` interface without shared mutable width/height.

LSP is the formal reason composition wins for many "reuse" hierarchies: composition does not promise substitutability you cannot honor.

### Memory and Lifetime Coupling

A subclass instance **is** a base instance: one object header, base fields laid out before derived fields. The subclass cannot discard base state. Composition allows **multiple implementations** swapped at runtime and **smaller** objects when you only need a facet.

In DI scoped services, composing `DbContext` + `IClock` + `IEmailSender` keeps each lifetime explicit. Inheriting `BaseServiceWithDbAndEmail` tends toward **god base classes** accumulating dependencies every feature requests.

## Syntax / API

No new C# keywords — this lesson is design. Relevant mechanisms:

| Mechanism | Role in composition |
|-----------|---------------------|
| Constructor injection | Supply composed dependencies |
| `private readonly` fields | Hold collaborators |
| Interfaces | Contracts for composed parts |
| `sealed` on leaf classes | Prevent further fragile extension |
| DelegatingHandler, middleware | Framework composition hooks |

**Refactoring steps (inheritance → composition):**

1. Identify **behavior slices** in the base class (validation, formatting, persistence).
2. Extract each slice to an interface + default implementation.
3. Replace `protected virtual` hooks with calls to injected interfaces.
4. Delete empty intermediate classes; keep inheritance only where LSP holds.

## Examples

### Example 1: Bird Hierarchy (Classic Mistake)

Inheritance-first design:

```csharp
public abstract class Bird
{
    public abstract void Fly();
}

public class Sparrow : Bird
{
    public override void Fly() => Console.WriteLine("Flying");
}

public class Penguin : Bird
{
    public override void Fly() => throw new NotSupportedException("Cannot fly");
}
```

Callers using `Bird` must handle penguins that explode. **Composition fix:**

```csharp
public interface IFlyBehavior { void Fly(); }
public sealed class Flying : IFlyBehavior { public void Fly() => Console.WriteLine("Flying"); }
public sealed class Flightless : IFlyBehavior { public void Fly() => Console.WriteLine("Walks instead"); }

public sealed class Bird
{
    private readonly IFlyBehavior _flight;
    public Bird(IFlyBehavior flight) => _flight = flight;
    public void Move() => _flight.Fly();
}
```

`Bird` is not subclassed for flight capability; capability is composed.

### Example 2: ASP.NET Core Middleware as Composition

Middleware is a **composed pipeline**, not inheritance:

```csharp
public sealed class CorrelationIdMiddleware
{
    private readonly RequestDelegate _next;

    public CorrelationIdMiddleware(RequestDelegate next) => _next = next;

    public async Task InvokeAsync(HttpContext context)
    {
        var id = context.Request.Headers["X-Correlation-ID"].FirstOrDefault()
                 ?? Guid.NewGuid().ToString("N");
        context.Response.Headers["X-Correlation-ID"] = id;
        await _next(context);
    }
}
```

Each middleware **has** the next delegate; it does not inherit from `AuthenticationMiddleware`. Order is configured in `Program.cs` — composition root.

## Common Mistakes & Anti-patterns

**Inheriting for reuse only.** Parent exposes twenty `public` methods; child needs three. Inheritance imports seventeen wrong methods (`Stack : List<T>`).

**Deep hierarchies.** More than two levels of concrete inheritance in application code is a smell. Behavior variation → interfaces + composition.

**Protected fields.** Subclasses bind to representation. Prefer `private` fields + `protected` accessors only when Template Method truly needs them — even then, consider strategy injection.

**Overriding without calling base.** Breaks when base evolves (`1.2.6` `new` vs `override` confusion worsens this).

**God abstract base controllers.** `BaseApiController` with auth, logging, mapping, and CRUD — features fight over `protected` helpers. Split cross-cutting into filters (`IActionFilter`) and compose services.

## Production & Real-World Notes

Teams enforce **composition by default** via code review checklists: "Why not an interface here?" Analyzers like Sonar flag deep inheritance trees.

**Testing:** composed classes accept mocks in constructors. Subclasses of concrete bases often need awkward partial mocks or integration tests only.

**Vertical slices** (`MediatR` handlers) favor one handler class composing `DbContext` + domain services — not `BaseCreateHandler<T>` hierarchies.

**Records and DTOs** (`1.2.13`) use inheritance sparingly; favor composition of value objects.

When migrating legacy code, **strangler approach**: extract interfaces from base class methods one by one; inject them; finally make base class `sealed` or delete.

## Comparison / Trade-offs

| Approach | Strength | Weakness | Typical use |
|----------|----------|----------|-------------|
| Inheritance | Shared algorithm, framework hooks | Fragile base, LSP risk | `Exception`, `DbContext`, Template Method |
| Composition | Flexible, testable, explicit deps | More boilerplate, wiring in DI | Application services, decorators |
| Interfaces | Loose coupling, multiple roles | No shared implementation | Contracts, strategy selection |

**Rule of thumb:** if you hesitate to mark a class `sealed` because "someone might need to extend it," you probably need an interface instead of `virtual` methods.

## Quick Reference

| Question | Prefer |
|----------|--------|
| Need substitutability in polymorphic list? | Interface + implementations |
| Share fixed algorithm with variant steps? | Abstract base (shallow) or composition + strategy |
| Reuse fields from unrelated type? | Composition, not inheritance |
| Framework requires `override`? | Inherit — minimize `protected` surface |
| Adding behavior around existing service? | Decorator / wrapper (composition) |

## Key Takeaways

- **Composition over inheritance** reduces coupling to ancestor implementation details.
- **Fragile base class:** base changes break subclasses in ways compilers cannot catch.
- **LSP** formalizes when inheritance is semantically valid.
- **Delegation + interfaces** replace most `protected virtual` extension points.
- **ASP.NET Core** extensibility (middleware, handlers, filters) is composition-first.
- Inheritance remains right for **framework types** and **true is-a** with stable contracts.
- Refactor by **extracting interfaces** and **constructor injection**, not by adding another subclass.
