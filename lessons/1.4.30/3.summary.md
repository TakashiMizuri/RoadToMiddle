# Fire-and-forget — краткая выжимка

> `1.4.30` · ~10 мин на повторение

## Главная идея

Запуск `Task` без `await` в HTTP-запросе отрезает **наблюдение исключений** и часто ломает **scoped DI**. Безопасный путь — **очередь + BackgroundService** с новым scope на job.

## Ключевые пункты

- `_ = WorkAsync()` — исключения не попадают в middleware запроса.
- Scoped `DbContext` **dispose** при конце request → race в фоне.
- **`async void`** на сервере — запрещён (кроме UI).
- **`Task.Run` + forget** — риск thread pool starvation; не замена очереди для I/O.
- **`RequestAborted`** для post-response work часто отменяет задачу рано.
- Минимум: fault logging через `ContinueWith` — **не** чинит scope.

## Самые важные примеры

```csharp
// Плохо
_ = _scopedService.DoAsync();

// Лучше
await _queue.QueueBackgroundWorkItemAsync(async sp => {
    await sp.GetRequiredService<IService>().DoAsync();
});
```

## «Под капотом» в одном абзаце

Request scope живёт только в pipeline. Discarded task продолжает на thread pool с ссылками на disposed services. Faulted task без await попадает в `UnobservedTaskException` при финализации. Hosted consumer await'ит каждый job в цикле с `IServiceScopeFactory.CreateScope()` — исключения ловятся, DI свежий.

## Частые ошибки

- Считать discard «фоновым режимом ASP.NET».
- Возвращать 200 до commit, надеясь на fire-and-forget для критичного side effect.
- Не ограничивать очередь (unbounded channel).

## Мнемоника

**«If you drop the Task, drop the request scope too — use a queue»**
