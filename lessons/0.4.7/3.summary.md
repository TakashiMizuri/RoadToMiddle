# Firewalls и порты 80/443/5432 — краткая выжимка

> `0.4.7` · ~8 мин на повторение

## Главная идея

**Firewall** решает, **какой сетевой трафик доходит до хоста** (по IP, port, protocol), но **не заменяет** auth и validation в приложении. В production наружу обычно открыты **80/443**; **5432** (PostgreSQL) — только из private network / с API-серверов.

## Ключевые пункты

- **Stateful firewall** помнит sessions и автоматически пропускает return traffic для outbound connections.
- **Port 80** — HTTP, часто redirect на HTTPS; **443** — HTTPS (TLS), главный public entry.
- **Port 5432** — PostgreSQL; **никогда** `0.0.0.0/0` в production.
- **Connection refused** — не listen'ит или explicit reject; **timeout** — silent drop (SG/firewall).
- **Least privilege:** minimum ports, minimum source IPs.
- **Defense in depth:** firewall + TLS + app security + secrets; слои дополняют друг друга.
- Docker `-p 5432:5432` на public VM = exposure независимо от app code.

## Самые важные примеры

**Typical cloud path:**
```
Internet → SG:443 → ALB → SG:app → API → SG:5432 → PostgreSQL (private)
```

**Connection string dev vs prod:**
```
Host=localhost;Port=5432          # dev (Compose)
Host=db.internal;Port=5432        # prod (private, SG-restricted)
```

## «Под капотом» в одном абзаце

Security group — stateful virtual firewall на network interface: allow rules + implicit deny. Пакет на 5432 без prior outbound session из trusted source drop'ится. Host `iptables` и Docker NAT — дополнительные слои; published port в Compose создаёт DNAT, обходящий assumption «БД только внутри контейнера».

## Частые ошибки

- Временно открыть 5432 в интернет для debug.
- Считать HTTPS на 443 автоматической защитой БД.
- Путать «порт закрыт» с «приложение безопасно».

## Мнемоника

> **443 наружу, 5432 внутри** — если сомневаешься, БД не должна видеть интернет.
