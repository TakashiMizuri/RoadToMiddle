# Covariance and Contravariance: `in`, `out`, and `IEnumerable`

> Roadmap: `1.3.2` · Node: `1.3` — C# advanced topics · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Explain **variance** in terms of **substitutability** (can I use a more specific type where a general one is expected?).
- Distinguish **covariance** (`out T`), **contravariance** (`in T`), and **invariance** (default).
- Predict assignability for **`IEnumerable<T>`**, **`IReadOnlyList<T>`**, **`Action<T>`**, **`Func<T, TResult>`**, and **`List<T>`**.
- Declare **variant generic interfaces** correctly with `in`/`out` modifiers.
- Describe **CLR enforcement** of variance and why mutable collections cannot be covariant.
- Avoid **variance pitfalls** in delegates, arrays (legacy), and generic method type inference.

---

## Why This Matters

You will write code like this and expect it to compile:

```csharp
IEnumerable<string> names = new List<string> { "Ada", "Lin" };
IEnumerable<object> objects = names; // legal — covariance
```

But this fails:

```csharp
List<string> strings = new();
List<object> objects = strings; // compile error — invariant
```

Understanding **why** prevents hours of refactoring friction and explains API design across .NET: **`IEnumerable<out T>`** is covariant; **`List<T>`** is invariant because you can **write** into it. Middle developers need variance when designing **`IReadOnlyCollection<T>`**-style APIs, consuming LINQ, passing **`Action<object>`** where **`Action<string>`** is expected (contravariance), and when **generic constraints** block an otherwise natural assignment.

Variance also appears in **type inference** edge cases and in **C# 9+ function pointers** (out of scope here) — the mental model transfers to any language with generics.

## Core Concepts

### Subtyping and Substitution

If **`Dog`** inherits **`Animal`**, a `Dog` reference can substitute for `Animal` — **subtype substitution** works for reference types in assignment:

```csharp
Animal a = new Dog(); // OK
```

Generics add a second question: if `StringList` were a `List<string>`, is it a **`List<object>`**? That would let you **`Add(new object())`** to something the compiler thinks is `List<string>` — type safety breaks. Hence **`List<T>` is invariant**: `List<string>` is **not** assignable to `List<object>` even though `string` is assignable to `object`.

Variance rules apply to **generic type parameters** in interfaces and delegates, not to arbitrary generic classes like `List<T>`.

### Covariance: `out T` — Producers

**Covariance** means: if `Derived` is substitutable for `Base`, then **`IGeneric<Derived>`** is substitutable for **`IGeneric<Base>`** — direction follows inheritance (**out** = data flows **out** of the API to the consumer).

```csharp
IEnumerable<string> strings = new[] { "a", "b" };
IEnumerable<object> objects = strings; // OK
```

You may **read** `T` from `IEnumerable<out T>` (foreach yields `T`), but the interface must not expose methods that **take `T` as input** (except constructors/destructors — special cases). **`IEnumerator<T>.Current`** returns `T` — output.

**Arrays** in C# are **covariant** for reference element types (legacy CLR rule):

```csharp
object[] arr = new string[3]; // compiles — dangerous!
arr[0] = "ok";
// arr[1] = 42; // ArrayTypeMismatchException at runtime
```

Prefer **`IReadOnlyList<T>`** or **`ReadOnlySpan<T>`** over arrays for public APIs.

### Contravariance: `in T` — Consumers

**Contravariance** reverses direction: if `Derived` : `Base`, then **`IConsumer<Base>`** can substitute for **`IConsumer<Derived>`** — you **consume** less specific types.

Classic example — **`Action<T>`** is contravariant in `T`:

```csharp
Action<object> printObject = obj => Console.WriteLine(obj);
Action<string> printString = printObject; // OK — contravariance
printString("hello"); // passes string where Action<object> expected parameter
```

The delegate accepts **`object`**; assigning to **`Action<string>`** is safe because every **`string`** is an **`object`** — the handler can process any object, so it can certainly process strings.

**`Func<in T, out TResult>`** — input parameter contravariant (`in T`), return type covariant (`out TResult`):

```csharp
Func<object, string> f = obj => obj.ToString() ?? "";
Func<string, string> g = f; // OK
```

### Invariance: Default for Classes and Unmarked Parameters

**`List<T>`**, **`Dictionary<TKey, TValue>`**, **`Span<T>`** — invariant. They have methods both **reading and writing** `T`. Marking them covariant would allow unsound writes.

Generic classes **cannot** declare variant type parameters — only **interfaces** and **delegates**.

### Declaring Variant Interfaces

```csharp
public interface IProducer<out T>
{
    T GetNext();
}

public interface IConsumer<in T>
{
    void Accept(T item);
}

// Invalid — same T cannot be both in and out unless read-only rules satisfied
// public interface IBoth<in out T> { } // not valid syntax in C#
```

**`IReadOnlyList<out T>`** — covariant because only getters expose `T`. **`IComparer<in T>`** — contravariant: `Compare(T x, T y)` takes `T` as input.

### Variance and `IEnumerable<T>`

LINQ returns **`IEnumerable<T>`** precisely because covariance lets you treat **`IEnumerable<Dog>`** as **`IEnumerable<Animal>`** when composing pipelines without copying.

```csharp
IEnumerable<Animal> all = dogs.Select(d => (Animal)d); // or covariant assign if dogs is List<Dog> via IEnumerable<Dog>
```

## Under the Hood

### CLR Type Safety Rules

The CLI specification defines **variance flags** on generic parameters (`+` covariant, `-` contravariant). The **verifier** rejects implementations that violate declared variance at JIT/load time. Interface method signatures are checked: **`out T`** methods cannot have **`T`** as parameter type (except return-only positions).

### Why Mutable Collections Stay Invariant

Imagine illegal covariant `List<out T>`:

```csharp
List<object> objects = (List<object>)(object)stringList; // hypothetical
objects.Add(42); // corrupt stringList — store int in string slot
```

Invariant **`List<T>`** prevents this at compile time. **`IReadOnlyList<T>`** removes mutators from the contract, enabling **`out T`**.

### Delegate Variance

Delegate types are generic interfaces under the hood. **`Action<in T>`**, **`Func<in T, out TResult>`** carry CLI variance metadata. Combining delegates does not change variance — assignment rules apply at conversion sites.

### Limitations

- **Value types**: variance applies to reference types only — **`IEnumerable<int>`** is not assignable to **`IEnumerable<object>`** (boxing would change semantics; `int` is not substitutable for `object` without boxing at element level).
- **Nested generics**: **`List<IEnumerable<string>>`** vs **`List<IEnumerable<object>>`** — outer `List<>` invariant; inner may covary inside interface only when assigning **`IEnumerable<string>`** to **`IEnumerable<object>`**, not the list wrapper.
- **C# 9+ `in`/`out` on interfaces** — implementing class must match; explicit implementation can be tricky in advanced scenarios.

## Syntax / Commands / API

```csharp
public interface IReadOnlyRepository<out TEntity> where TEntity : class
{
    TEntity? Find(int id);
    IEnumerable<TEntity> GetAll();
}

public interface IHandler<in TCommand> where TCommand : ICommand
{
    void Handle(TCommand command);
}

// Delegate declarations (BCL)
public delegate void Action<in T>(T obj);
public delegate TResult Func<in T, out TResult>(T arg);
```

Check variance in reflection (advanced):

```csharp
var t = typeof(IEnumerable<>).GetGenericArguments()[0];
Console.WriteLine(t.GenericParameterAttributes); // Covariant flag
```

## Examples

### Covariant Return in Pipeline

```csharp
List<Dog> dogs = GetDogs();
IEnumerable<Animal> animals = dogs; // IEnumerable<Dog> → IEnumerable<Animal>
foreach (Animal a in animals) { /* ... */ }
```

Direct `List<Dog>` → `List<Animal>` — **error**.

### Contravariant Event Handler (Pattern)

```csharp
void OnObjectChanged(object sender, EventArgs e) { }

// EventHandler<EventArgs> is not the same story, but Action pattern:
Action<object> handler = o => Log(o);
Action<string> stringHandler = handler;
```

### Custom Covariant Interface

```csharp
public interface ITree<out T>
{
    T Value { get; }
    ITree<T>? Left { get; }
    ITree<T>? Right { get; }
}
```

All members expose `T` as output only — valid `out T`.

## Common Mistakes & Anti-patterns

**Assuming all generics covary like arrays.** Arrays are covariant for references — **`object[]` = string[]`** throws at runtime on wrong store. Do not rely on this.

**Trying to covary `List<T>` via cast.** `(IEnumerable<object>)(List<string>)` works through **interface** conversion, not list covariance — you lose write API unless you cast back.

**Value type covariance confusion.** `IEnumerable<int>` → `IEnumerable<object>` — **illegal**; use **`Cast<object>()`** or **`Select(x => (object)x)`** with boxing cost.

**Declaring `out T` with input methods.** Compiler error on `void Add(T item)` in `interface IFoo<out T>`.

**Contravariance direction reversed in memory.** **Covariant**: `Derived` → `Base` implies `I<XDerived>` → `I<XBase>`. **Contravariant**: `I<XBase>` → `I<XDerived>`.

## Production & Real-World Notes

Public APIs expose **`IReadOnlyList<T>`**, **`IEnumerable<T>`**, **`IReadOnlyDictionary<,>`** instead of concrete **`List<T>`** to allow covariance at call sites and hide mutation.

LINQ methods return **`IEnumerable<T>`** — enables composable pipelines without copying collections.

Contravariance powers **`Comparison<in T>`** and **`IComparer<in T>`** — pass **`IComparer<object>`** where **`IComparer<string>`** needed.

Framework design: mutable BCL collections invariant; read-only interfaces covariant since .NET 4.

## Comparison / Trade-offs

| Type | Variance | Can assign `List<string>` to …? |
|---|---|---|
| `List<T>` | Invariant | Only `List<string>` |
| `IEnumerable<T>` | Covariant `out` | `IEnumerable<object>` ✓ |
| `IReadOnlyList<T>` | Covariant `out` | `IReadOnlyList<object>` ✓ |
| `Action<T>` | Contravariant `in` | `Action<string>` from `Action<object>` ✓ |
| `Func<T, R>` | `in T`, `out R` | See rules per position |

## Quick Reference

| Keyword | Variance | Mnemonic |
|---|---|---|
| `out T` | Covariant | T **out** to caller (producer) |
| `in T` | Contravariant | T comes **in** from caller (consumer) |
| (none) | Invariant | Read **and** write T |

**PECS** (Producer Extends, Consumer Super) — Java mnemonic; in C#: **out = producer**, **in = consumer**.

---

## Key Takeaways

- Variance governs **whether generic type arguments follow subtyping** of their type arguments.
- **`out T`** — covariant, output-only; **`in T`** — contravariant, input-only.
- **`List<T>` is invariant** because mutation would break type safety.
- **`IEnumerable<out T>`** enables LINQ composition without copying.
- **Reference types only** for variance conversions; value types need boxing strategies.
- **Arrays** are covariant for references — legacy footgun; prefer read-only abstractions.
