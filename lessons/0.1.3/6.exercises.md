# Exercises: Git Essentials — Hands-on Workflow

> Complete after reading `1.lection-eng.md`, `2.lection-ru.md` and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Initialize and first commit
**Goal:** Create a repo from scratch.

**Steps:**
1. Create folder `hello-git/` and `cd` into it.
2. Run `git init -b main`.
3. Create `greeting.txt` with content `Hello, Git!`
4. Stage and commit with message `Initial commit: add greeting`
5. Run `git log --oneline` and `git status`

**Success criteria:**
- `git log` shows exactly one commit.
- `git status` reports "nothing to commit, working tree clean".

---

### E2. Read status codes
**Goal:** Interpret short status.

After E1, append a new line to `greeting.txt` **without** staging. Run `git status -s`.

Write down:
1. The status code for `greeting.txt`
2. Which section it would appear in (long format)
3. What command shows the exact line change

**Success criteria:** Correct code (` M`), section ("Changes not staged"), command (`git diff`).

---

## Medium

### M1. Two-file commit with review
**Goal:** Practice add → diff --staged → commit.

**Setup:** In `hello-git/` (from E1):

1. Create `author.txt` with your name.
2. Change `greeting.txt` to `Hello, Git! Version 2.`

**Tasks:**
1. Stage **both** files in one logical commit.
2. Before committing, run `git diff --staged` and verify output.
3. Commit with a proper message (imperative subject + optional body).
4. Show last 2 commits: `git log --oneline -2`

**Success criteria:** One commit containing both files; message is not "update" or "fix".

---

### M2. Partial add with `-p`
**Goal:** Stage one hunk, leave another unstaged.

**Setup:** Edit `greeting.txt` to add **two separate blocks** in one save:

```
Line A: bugfix — correct spelling
Line B: DEBUG — remove before production
```

(Use two separate additions in different parts of the file.)

**Tasks:**
1. Run `git add -p greeting.txt`.
2. Stage only the bugfix hunk; skip DEBUG.
3. Commit bugfix only.
4. Run `git status` — DEBUG hunk should remain unstaged.

**Success criteria:** Two commits worth of intent split; unstaged DEBUG still visible in `git diff`.

---

### M3. Compare diff targets
**Goal:** Solidify unstaged vs staged vs HEAD.

After M2 (with unstaged DEBUG still in working tree):

Run and **document in writing** what each shows:

```bash
git diff
git diff --staged
git diff HEAD
```

**Success criteria:** You explain which tree pairs each command compares; `git diff --staged` is empty if nothing staged.

---

## Hard (stretch)

### H1. Mini archaeology
**Goal:** Use log and diff between commits.

**Tasks:**
1. Make **three** separate commits with distinct messages (e.g., add file, fix typo, add footer).
2. Run `git log --oneline --graph --decorate -5`
3. Copy the two oldest commit hashes from your log.
4. Run `git diff <older> <newer>` and describe in 2–3 sentences what changed.

**Success criteria:** Correct hash order (older → newer); diff output matches your intentional edits.

---

### H2. Clone practice (requires network)
**Goal:** Experience clone workflow.

**Steps:**
1. Clone any small public repo: `git clone https://github.com/github/gitignore.git /tmp/gitignore-demo`
2. `cd` into it; run `git log --oneline -5`
3. Create a new branch file change in README (or add a comment file), **do not push**.
4. Practice: status → diff → add → diff --staged → commit

**Success criteria:** Local commit on top of cloned history; you can explain what `origin` refers to (from clone).
