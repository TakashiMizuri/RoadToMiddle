# Списки: ul, ol, dl; вложенные списки

> Roadmap: `0.5.4` · Node: `0.5` — HTML · Depth: практика

## Learning Objectives

После этого урока ты сможешь:

- Размечать **неупорядоченные** списки через **`ul`** и **`li`**.
- Размечать **упорядоченные** списки через **`ol`** с атрибутами **`start`**, **`reversed`**, **`type`** при необходимости.
- Строить **description lists** через **`dl`**, **`dt`**, **`dd`** для пар термин–определение.
- Корректно **вкладывать** списки для outline, меню и многошаговых инструкций.
- Выбирать тип списка для navigation, procedures и glossaries.
- Не использовать списки ради layout или «одной пули».

---

## Why This Matters

Navigation из `0.5.2` часто — **список ссылок**. Инструкции требуют нумерации. Гlossary API — термины и определения. List elements кодируют **отношения**: siblings на одном уровне, важна ли последовательность, термин связан с описанием. Screen reader говорит «list, 5 items» — div с background-bullet так не умеет.

Списки часто abuse: вся страница в `<ul>` для margin, `ol` без смысла порядка, glossary на bold + br вместо dl. Middle-разработчик пишет `<nav><ul>` — правильный рефлекс, но нужно знать **когда не list** и как **nest** без поломки keyboard nav.

Списки связаны с forms `0.5.7` и CSS `0.6.x`. Сначала structure, потом presentation.

## Core Concepts

### Неупорядоченный список: ul и li

**`<ul>`** — порядок **не меняет смысл**: checklist, bullets, nav (членство важнее порядка). Каждый пункт — **`<li>`**, прямой ребёнок `ul` (или вложенного list внутри li).

Nav pattern:

```html
<nav aria-label="Primary">
  <ul>
    <li><a href="/">Home</a></li>
    <li><a href="/docs">Docs</a></li>
  </ul>
</nav>
```

Не клади `<a>` прямо в `<ul>` — только через `<li>`. Иначе invalid и неверный item count для AT.

### Упорядоченный список: ol и li

**`<ol>`** — **последовательность важна**: шаги рецепта, ranked list, numbered clauses. Пункты — **`li`**. Default: 1, 2, 3.

Атрибуты (для смысла контента, не стиля):

- **`start="5"`** — нумерация с 5
- **`reversed`** — обратный счёт
- **`type="a"`** — буквы; для стиля в новых проектах чаще CSS counters

Перестановка `li` в ol меняет meaning — в ul обычно нет. Site nav — ul, не ol, если порядок не semantic.

### Description list: dl, dt, dd

**`<dl>`** — пары **термин / описание**:

- **`<dt>`** — term
- **`<dd>`** — description

Один dt — несколько dd; несколько dt — один dd. Glossaries, metadata, FAQ «вопрос → ответ» как term/definition.

Не для fake two-column layout без term-definition semantics.

```html
<dl>
  <dt>UTF-8</dt>
  <dd>Unicode-кодировка в вебе.</dd>
</dl>
```

### Вложенные списки

Иерархия: outline, submenu, sub-steps. **Вложенный ul/ol — внутри родительского `li`**, не между siblings.

Правильно:

```html
<ul>
  <li>Fruit
    <ul>
      <li>Apple</li>
    </ul>
  </li>
</ul>
```

Неправильно: второй ul sibling сразу после li без обёртки.

Глубина >3 levels — usability страдает; flatten или headings + lists. ol с nested ul под шагом 2 — норм.

### Когда не list

Один пункт редко нужен как ul из одного li — paragraph. Не lists для columns layout. Одна ссылка — просто `<a>`.

## Under the Hood

Roles: list/listitem; dl → term/definition. Invalid nesting — browser может «починить», AT путает counts. `list-style: none` на nav OK — семантика ul/li остаётся. Flex на li — test с axe.

## Common Mistakes & Anti-patterns

a под ul без li. Nested list вне li. ol для декора. dl для layout. Пустой li для spacing.

---

## Production & Real-World Notes

Breadcrumb component — ol или nav+ul? Markdown `*` → ul. Keyboard Tab через nested nav.

---

## Comparison / Trade-offs

ul / ol / dl — по meaning. Table vs dl: table для сравнимых строк; dl для key-value prose.

---

## Quick Reference

| Элемент | Назначение |
|---------|------------|
| ul / ol / li | Списки |
| dl / dt / dd | Term + definition |
| Nesting | inner list **inside li** |

---

## Key Takeaways

- ul unordered; ol ordered; dl term/definition.
- li обязателен; nest inside li.
- Тип списка по смыслу, не по виду маркера.
