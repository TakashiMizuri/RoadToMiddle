# break, continue, return — Flow Control

> Roadmap: `1.1.11` · Node: `1.1` — C# Language Basics · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Use **`break`** to exit loops and `switch` statements at the right time.
- Use **`continue`** to skip the rest of the current loop iteration and advance to the next.
- Use **`return`** to exit a method immediately, with or without a value.
- Apply **early return (guard clauses)** to flatten nested logic in methods.
- Explain why C# has **no labeled `break`/`continue`** like Java, and what to do instead.

---

## Why This Matters

Loops from `1.1.10` repeat until a condition changes — but real programs need to **leave early** (user chose Exit, item found, fatal validation error) or **skip one item** (blank line in a file, invalid row). Three keywords — **`break`**, **`continue`**, **`return`** — control *where* execution goes next without duplicating conditions everywhere.

Junior code often uses boolean flags (`bool found = false; ... found = true`) when a simple **`break`** would tell the story. Conversely, abusing **`return`** inside deeply nested loops produces "arrow code" that reviewers reject. Middle developers balance **clarity** and **structure**: `break`/`continue` inside a single loop; **`return`** at method boundaries for guards; extract methods when nesting exceeds two levels. These tools appear in every console app, service method, and parser you will write before async and exceptions dominate error flow.

## Core Concepts

### `break`: Exit the Innermost Construct

**`break`** terminates the nearest enclosing **`switch`**, **`while`**, **`do`**, or **`for`** — not the whole method unless you are in a small `Main`.

```csharp
for (int i = 0; i < 100; i++)
{
    if (i == 5)
        break;

    Console.WriteLine(i);
}
// prints 0..4
```

In **`switch`**, `break` prevents fall-through (required in classic switch). In loops, it means **"I am done iterating"** — search found, user quit, error threshold reached.

**Nested loops:** `break` exits **only the inner loop**. To leave both, use a labeled pattern via **`goto`** (rare in app code), a **flag**, or — better — **extract a method** and `return` from it (`1.1.12`).

```csharp
bool found = false;

for (int i = 0; i < rows && !found; i++)
{
    for (int j = 0; j < cols; j++)
    {
        if (grid[i, j] == target)
        {
            found = true;
            break; // exits inner for only
        }
    }
}
```

When inner `break` is not enough, **`return`** from a helper method is often clearest.

### `continue`: Skip to Next Iteration

**`continue`** jumps to the **next iteration** of the innermost loop, skipping remaining statements in the current body:

```csharp
for (int i = 0; i < 10; i++)
{
    if (i % 2 == 0)
        continue;

    Console.WriteLine(i); // only odd: 1,3,5,7,9
}
```

Use **`continue`** to handle **edge cases at the top** of a loop body — invalid parse, empty string — without extra nesting. It pairs with **`TryParse`** patterns from `1.1.9`:

```csharp
while (true)
{
    Console.Write("Number (0 to stop): ");
    if (!int.TryParse(Console.ReadLine(), out int n))
    {
        Console.WriteLine("Try again.");
        continue;
    }

    if (n == 0)
        break;

    Process(n);
}
```

Do not sprinkle `continue` so heavily that readers lose track of the "happy path." If more than two continues appear in one loop, consider restructuring.

### `return`: Exit the Method

**`return`** ends the **current method** immediately. For **`void`** methods, bare **`return;`** is legal. For value-returning methods, **`return expression;`** supplies the result:

```csharp
static int Absolute(int x)
{
    if (x >= 0)
        return x;

    return -x;
}
```

After `return`, no further statements in that method run on that call path. The caller resumes after the call site.

In **top-level programs** and **`static void Main`**, **`return`** exits the application entry point (status code 0 by default). That is idiomatic for **guard clauses** when input is invalid:

```csharp
if (!int.TryParse(args[0], out int n))
{
    Console.Error.WriteLine("Usage: app <number>");
    return;
}
```

### Early Return and Guard Clauses

**Guard clauses** validate preconditions at the top and **`return`** (or throw) early, avoiding pyramid `if` nesting:

```csharp
static string DescribeUser(string? name, int age)
{
    if (string.IsNullOrWhiteSpace(name))
        return "Anonymous";

    if (age < 0)
        return "Invalid age";

    return $"{name} ({age})";
}
```

Compare with a single trailing return buried under nested braces — guards read like a checklist. Teams increasingly prefer this over assigning `result` in every branch. It works best when each guard is independent.

### `return` vs `break` in Loops

| Goal | Use |
|------|-----|
| Stop current loop only | `break` |
| Skip one iteration | `continue` |
| Leave entire method (and all loops) | `return` |

If a loop sits inside a method and finding one item should end processing, **`return`** from the method can be clearer than `break` plus a flag after the loop.

### No Labeled break/continue in C#

Java allows `break outerLoop;`. **C# does not** support labeled break/continue. Alternatives:

1. **`return`** from an extracted method containing the loops.
2. **Boolean flag** checked in outer loop condition.
3. **`goto`** to a label (idiomatic only in generated code or rare performance cases — avoid in application code unless team standards allow).

Knowing this prevents searching for Java syntax that does not exist in C#.

## Under the Hood

`break`, `continue`, and `return` compile to **branch instructions** that jump to IL labels — the loop's exit block, the loop's increment step, or the method epilogue. **`return`** may trigger **`finally`** blocks if the method uses `try/finally` before returning — order matters for disposal patterns you will learn with exceptions.

The compiler rejects **`break`/`continue` outside loops** — a compile-time safety net. Unreachable code after `return` produces a **warning** (CS0162) — treat warnings as quality signals.

## Syntax / Commands / API

```csharp
break;              // exit switch or innermost loop
continue;           // next loop iteration
return;             // void method exit
return value;       // value-returning method
```

## Examples

### Example 1: Search with break

```csharp
int[] ids = { 4, 7, 12, 19 };
int target = 12;
int index = -1;

for (int i = 0; i < ids.Length; i++)
{
    if (ids[i] == target)
    {
        index = i;
        break;
    }
}

Console.WriteLine(index >= 0 ? $"Found at {index}" : "Not found");
```

### Example 2: Filter with continue

```csharp
string[] lines = { "  ok ", "", "data", "   ", "end" };

foreach (string line in lines)
{
    string trimmed = line.Trim();
    if (trimmed.Length == 0)
        continue;

    Console.WriteLine(trimmed);
}
```

### Example 3: Method with early returns

```csharp
static decimal ApplyDiscount(decimal total, string? coupon)
{
    if (total <= 0)
        return 0m;

    if (string.IsNullOrEmpty(coupon))
        return total;

    if (coupon == "SAVE10")
        return total * 0.9m;

    return total;
}
```

Each rule stands alone — easy to unit test in `1.1.12` exercises.

## Common Mistakes & Anti-patterns

**Using `break` expecting to exit nested loops** — only inner loop stops; logic bug.

**`continue` skipping increment in `while`** — rare pitfall if update was at bottom and `continue` jumps before it; put update in consistent place or use `for`.

**Multiple **`return`** scattered without structure** — still better than deep nesting, but extract method if returns multiply past ~3–4.

**Confusing `return` in `Main` with `break` in a menu loop** — `return` ends app; `break` only leaves switch/loop.

## Production & Real-World Notes

Service methods use guard **`return`** for invalid input before hitting the database. **`continue`** appears in batch processors skipping bad records while logging them — combine with structured logging later.

Some style guides restrict **`continue`**; others embrace it for guard-style loop bodies. Follow team conventions; default to readability.

For **`async` methods**, **`return`** returns a completed task value — same keyword, different machinery (future lesson).

## Comparison / Trade-offs

| Technique | When |
|-----------|------|
| `break` | Found target; user exit; stop switch arm chain |
| `continue` | Skip one bad item; re-prompt |
| `return` | Method-level decision final |
| Extract method + return | Escape nested loops cleanly |

Flags vs early return: flags scale poorly across nested blocks; **`return`** documents intent at method scope.

## Quick Reference

```
break     → exit switch / innermost loop
continue  → next iteration
return    → exit method (optional value)
C#: no labeled break/continue → extract method or flag
Guards: validate early, return early
```

## Key Takeaways

- **`break`** stops the innermost loop or switch, not arbitrary nesting.
- **`continue`** skips to the next loop iteration — good for invalid rows.
- **`return`** ends the method — use guard clauses to reduce nesting.
- Choose `return` over flag + break when leaving the whole method is the real goal.
- C# has no labeled break; extract methods instead of Java-style labels.
- Combine with loops (`1.1.10`) and methods (`1.1.12`) for readable console apps.
