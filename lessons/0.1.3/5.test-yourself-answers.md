## Answers

### 1. [Recall]
**Answer:** `git init` (опционально `git init -b main` для имени default branch).

**Why:** Создаёт `.git/` и инициализирует local repository.

---

### 2. [Recall]
**Answer:** **`git init`** — новый пустой repo локально. **`git clone`** — копирует существующий remote repo с **полной history**, working tree и настраивает **`origin`**.

**Why:** Init = с нуля; clone = duplicate existing project.

---

### 3. [Understanding]
**Answer:** Файл **staged** (первая `M` в index column) **и** снова **modified** в working tree (вторая `M`) — dual state после edit post-`git add`.

**Why:** Index зафиксировал старую версию; working tree продвинулся дальше.

---

### 4. [Understanding]
**Answer:** **`git add .`** — stage изменения в **текущей директории и ниже** (subtree). **`git add -A`** — **весь repository** независимо от cwd (new, modified, deleted everywhere).

**Why:** `.` scope ограничен; `-A` repo-wide.

**Common mistake:** Думать, что `.` из subdirectory stage'ит root deletions.

---

### 5. [Understanding]
**Answer:** Auto-stage все **tracked** modified/deleted файлы и сразу commit. **Не попадут:** **untracked** (новые) файлы и изменения, которые не были tracked.

**Why:** `-a` = `--all` в контексте commit, но только для already tracked paths.

---

### 6. [Understanding]
**Answer:**
- **`git diff`** — working tree **vs** staging area (unstaged changes).
- **`git diff --staged`** — staging area **vs** last commit (HEAD) — что будет committed.

**Why:** Different comparison targets — core three-tree model from `0.1.2`.

---

### 7. [Application]
**Answer:**
```bash
git add Utils.cs Program.cs
git commit -m "Add Utils and update Program"
```
(Или два отдельных add; message лучше конкретнее.)

**Why:** Оба файла должны быть staged; `-a` не добавит новый Utils.cs без prior tracking.

---

### 8. [Application]
**Answer:** `git add -A` (или `git add --all`).

**Why:** Из subdirectory `.` не захватит changes outside subtree; `-A` — whole repo.

---

### 9. [Application]
**Answer:**

**Good:**
```
Fix 500 when order list is empty

Return empty array instead of throwing when no orders exist.
Fixes #42
```

**Bad:** `fix`, `update`, `WIP`

**Why:** Good message = imperative subject + why; bad = no context for future readers.

---

### 10. [Analysis]
**Answer:** **D** неверно.

**Why:**
- A ✅ — porcelain for scripts
- B ✅ — cached = staged (legacy alias)
- C ✅ — `-p` shows patches
- D ❌ — `-p` is **patch mode for hunks** in tracked modified files, not "only untracked"

---

### 11. [Analysis]
**Answer:** Риск **unintended files staged** — из `tests/` `git add .` stage'ит только subtree `tests/`, но если cwd был root, `.` stage'ит subtree root. Главный риск: **no review** → unrelated changes, secrets, debug code в commit.

**Why:** Blind add + no `diff --staged` = classic anti-pattern.

---

### 12. [Recall]
**Answer:** `git log --oneline -3` (или `git log -3 --oneline`).

**Why:** `--oneline` compact; `-n` limits count.

---

### 13. [Understanding]
**Answer:** Final **review checklist** — убедиться, что в commit попадёт **только** intended changes (no debug, secrets, unrelated files). `add` мог захватить лишнее.

**Why:** Last chance before immutable history entry.

---

### 14. [Application]
**Answer:**
```bash
git diff abc1234 HEAD
# or
git diff abc1234..HEAD
```

**Why:** Compares snapshot at commit to current HEAD.

---

### 15. [Application]
**Answer:** `git add -p <file>` — interactively stage selected hunks (`y`/`n`/`s`).

**Why:** Patch mode splits file into hunks for selective staging.

---

### 16. [Analysis]
**Answer:** Commit **не создастся** (или будет empty): untracked file **не staged**. Git ответит **"nothing to commit"** / no changes added to commit.

**Why:** Commit only records **staged** snapshot; untracked must be `git add` first. `-a` doesn't help new files.

**Common mistake:** Expecting commit to auto-include untracked files.
