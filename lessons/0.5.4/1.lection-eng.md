# Lists: ul, ol, dl; Nested Lists

> Roadmap: `0.5.4` · Node: `0.5` — HTML · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Mark up **unordered** lists with **`ul`** and **`li`**.
- Mark up **ordered** lists with **`ol`**, including **`start`**, **`reversed`**, and **`type`** when appropriate.
- Build **description lists** with **`dl`**, **`dt`**, **`dd`** for term-definition pairs.
- Nest lists correctly for outlines, menus, and multi-level steps.
- Choose the right list type for navigation, procedures, and glossaries.
- Avoid using lists purely for layout or single-item fake bullets.

---

## Why This Matters

Navigation from `0.5.2` often *is* a list of links. Procedural docs need numbered steps. API glossaries pair terms with definitions. HTML list elements encode **relationships**: siblings at the same level, sequence that matters (or does not), term tied to description. Screen readers announce "list, 5 items" and support jumping — semantics generic divs with bullet backgrounds do not provide.

Misused lists clutter accessibility: entire pages wrapped in `<ul>` for margin hacks, ordered lists when order is meaningless (use ul), or definition lists simulated with bold lines and br. Middle developers write `<nav><ul>` without thinking — correct instinct — but must also know **when not to list** and how **nesting** reflects hierarchy without breaking keyboard navigation.

Lists connect to forms (`0.5.7`) — grouped radio options — and to CSS (`0.6.x`) — `list-style`, `:marker`, flex on `li`. Structure first; presentation second.

## Core Concepts

### Unordered Lists: ul and li

**`<ul>`** (unordered list) groups items where **sequence does not change meaning**. Shopping checklist, feature bullets, navigation menu (order less important than set membership). Each item is **`<li>`** (list item) — direct child of `ul` (or of nested list inside `li`).

Browsers render bullets by default; CSS controls marker style. Semantics remain "unordered group" regardless of visual numbers or icons.

Typical nav pattern:

```html
<nav aria-label="Primary">
  <ul>
    <li><a href="/">Home</a></li>
    <li><a href="/docs">Docs</a></li>
  </ul>
</nav>
```

Only **`li`** should be direct child of **`ul`** (or script/template exceptions). Do not put raw `<a>` directly under `<ul>` — invalid and breaks list item counts for AT.

### Ordered Lists: ol and li

**`<ol>`** (ordered list) — sequence **matters**: recipe steps, ranked priorities, legal numbered clauses. Items still **`li`**, but order is significant. Default rendering: 1, 2, 3.

Attributes (use when content requires, not for styling alone):

- **`start="5"`** — begin numbering at 5 (continuing a prior list conceptually).
- **`reversed`** — countdown order.
- **`type="a"`** — lowercase letters (a, b, c) where allowed; prefer CSS counters in new work for styling, attributes for document meaning.

If you reorder `<ol>` items in DOM, meaning changes — unlike `ul`. Do not use `ol` for nav menus unless order is truly meaningful (step wizard breadcrumbs might be ol; main site nav is ul).

### Description Lists: dl, dt, dd

**`<dl>`** (description list) pairs **terms** with **descriptions**:

- **`<dt>`** — term (definition term)
- **`<dd>`** — description (definition description)

One `dt` may have multiple `dd` elements (synonyms, EN and RU explanation). Multiple `dt` may share one `dd` (alternate names, same definition). Good for glossaries, metadata (Author / Date), FAQ question/answer when structure is term → explanation.

Not for arbitrary two-column layout — if there is no term/definition relationship, use CSS Grid on divs.

Example:

```html
<dl>
  <dt>UTF-8</dt>
  <dd>Unicode encoding common on the web.</dd>
  <dt>DOCTYPE</dt>
  <dd>Parser mode declaration for HTML5.</dd>
</dl>
```

### Nesting Lists

**Nested lists** express hierarchy: outline with sub-points, menu with flyout sections, steps with sub-steps. Rule: **nested `ul`/`ol` goes inside the parent `li`**, not as sibling between items.

Correct:

```html
<ul>
  <li>Fruit
    <ul>
      <li>Apple</li>
      <li>Mango</li>
    </ul>
  </li>
  <li>Vegetables</li>
</ul>
```

Wrong (invalid structure):

```html
<ul>
  <li>Fruit</li>
  <ul>
    <li>Apple</li>
  </ul>
</li>
```

Deep nesting (>3 levels) challenges usability — consider flattening or splitting into headings + lists (`0.5.3`). For nav mega-menus, nested ul inside li remains standard.

Mixing types is fine: `ol` top-level steps, nested `ul` for bullet sub-details under step 2's `li`.

### When Not to Use a List

Single item rarely needs `<ul><li>only</li></ul>` — use a paragraph or heading unless it is truly one bullet in a larger template pattern. Do not use lists for page layout columns. For one link, not a list — a lone `<a>` suffices.

## Under the Hood

List elements map to **list** / **listitem** roles; description lists use **term** / **definition** in accessibility tree. Screen readers announce item counts and nesting depth. Invalid nesting (ul direct child of ul) may still render in browsers after error correction but confuses AT item numbering.

CSS **`list-style: none`** on nav removes bullets visually — keep ul/li for semantics. **`display: flex`** on `ul`/`li` is common; do not strip list semantics without `role="list"` restoration if you change display in ways that remove semantics (WebKit bug history — test with axe).

Counters for custom numbering often use CSS `@counter-style` on `ol` — HTML `type` attribute remains fallback for non-CSS contexts (print, simple readers).

## Syntax / Commands / API

Ordered procedure with nested details:

```html
<ol>
  <li>Clone the repository.</li>
  <li>Install dependencies.
    <ul>
      <li>Run <code>npm install</code>.</li>
      <li>Verify Node version ≥ 20.</li>
    </ul>
  </li>
  <li>Start the dev server.</li>
</ol>
```

FAQ as description list:

```html
<dl>
  <dt>Can I skip migrations?</dt>
  <dd>No — run <code>dotnet ef database update</code> first.</dd>
</dl>
```

## Examples

### Example 1: Nav vs steps

Site menu → **`ul`**. Deployment procedure → **`ol`**. Mixing them up hurts users who rely on "step 3 of 7" announcements for ol.

### Example 2: Metadata sidebar

Product specs as **`dl`**: dt = "Weight", dd = "1.2 kg". Cleaner than table for one column of pairs when not tabular data (tables need th scope — later HTML topics).

## Common Mistakes & Anti-patterns

**`<a>` direct child of `<ul>`** — Wrap in `<li>`.

**Nested list outside `li`** — Invalid; fix nesting under parent item.

**ol for styling numbers on unordered concepts** — Use ul + CSS if numbers are decorative.

**dl for layout only** — Need term-definition semantics.

**Empty li for spacing** — Use CSS margin.

## Production & Real-World Notes

Component libraries render `<Breadcrumb />` — know if output is ol (ordered path) or nav with ul. Markdown `*` → ul, `1.` → ol in static site generators.

Long ol in legal docs may use `start` to continue across sections. Test nested nav with keyboard Tab through links — focus order follows DOM.

## Comparison / Trade-offs

| Type | Use when |
|------|----------|
| ul | Set of items, order optional |
| ol | Sequence matters |
| dl | Term ↔ description |

Table vs dl: table for tabular data comparable across rows; dl for key-value prose pairs.

## Quick Reference

| Element | Role |
|---------|------|
| ul | Unordered list |
| ol | Ordered list |
| li | List item (child of ul/ol) |
| dl | Description list |
| dt | Term |
| dd | Description |

**Nesting:** inner list inside parent **li**.

---

## Key Takeaways

- ul = unordered; ol = ordered sequence; dl = term/definition.
- li wraps each item; links inside li, not directly under ul/ol.
- Nest lists inside li for valid hierarchy.
- Choose list type by meaning, not bullet appearance.
- Nav menus commonly ul; procedures commonly ol.
