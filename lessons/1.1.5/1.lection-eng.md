# var vs Explicit Types — Readability Rules

> Roadmap: `1.1.5` · Node: `1.1` — C# Language Basics · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Explain that **`var` is not dynamic typing** — the compiler still assigns a static type at compile time.
- Choose between **`var` and an explicit type** using readability rules used in professional C# codebases.
- Recognize when **`var` hides important information** and when it removes noise.
- Apply **`var` correctly** with required initializers, anonymous types, and LINQ projections.
- Relate type inference to the **primitive types** you learned in `1.1.3` (`int`, `long`, `decimal`, `bool`, etc.).

---

## Why This Matters

In `1.1.3` you met concrete primitive types: `int` for whole numbers, `decimal` for money, `double` for scientific approximations, `bool` for conditions. Every variable declaration looked like `int count = 0` or `decimal price = 19.99m`. That explicit style makes learning clear: you always see the type on the left side of the assignment.

Once projects grow, repeating obvious types becomes noise. C# offers **`var`** — a keyword that tells the compiler: *infer the type from the expression on the right*. Teams argue about it constantly. Some ban `var` entirely; others use it everywhere. Neither extreme is ideal. A Middle developer chooses based on **what the reader needs to know**, not personal taste.

The stakes are higher than style. Wrong assumptions about `var` lead to subtle bugs: treating inferred types as changeable at runtime, using `var` where the right-hand side is unclear, or hiding a widening conversion (`int` assigned from a `long` expression). This lesson gives you **decision rules**, not dogma.

## Core Concepts

### What `var` Actually Does

When you write:

```csharp
var count = 42;
```

the compiler sees the literal `42`, determines its type is `int`, and treats the declaration exactly as if you had written `int count = 42`. There is no `object` wrapper, no runtime type lookup, no JavaScript-style flexibility. **`var` is compile-time type inference**, full stop.

If you later write `count = "hello"`, the compiler rejects it — `count` remains `int`. You can verify this with your IDE: hover over `count` and IntelliSense shows `int`. The IL generated for `var count = 42` and `int count = 42` is identical.

This distinction matters because beginners sometimes believe `var` means "this variable can hold anything." It cannot. C# remains **statically typed**. `var` only saves you from typing the type name when the compiler can already deduce it reliably.

### The Rule You Must Never Break

**`var` requires an initializer.** You cannot write:

```csharp
var x;           // CS0818: implicitly typed variables must be initialized
var y = null;    // CS0815: cannot assign null without a cast or target type
```

The compiler learns the type from the right-hand expression. No expression, no type. For `null`, the compiler needs a concrete target type (`string? s = null`) because `null` alone carries no type information.

### When Explicit Types Improve Readability

Use an **explicit type** when the type is **not obvious from the right-hand side** or when the type itself is **the point of the declaration**.

Consider financial code:

```csharp
decimal total = CalculateOrderTotal(orderId);
```

Here `CalculateOrderTotal` might return `double`, `decimal`, or even a custom `Money` struct. The explicit `decimal` documents intent: we expect currency precision. Replacing it with `var total = CalculateOrderTotal(orderId)` forces every reader to jump to the method signature.

Another classic case is numeric literals and widening:

```csharp
long fileSize = 2_147_483_648L;
```

If you wrote `var fileSize = 2_147_483_648`, the literal fits in `int` range, so the compiler infers **`int`**, not `long`. You silently lose range. Explicit `long` (or suffix `L`) communicates and enforces the correct primitive from `1.1.3`.

Use explicit types for **public API surfaces**: method parameters, return types, public fields and properties. Callers and documentation generators depend on visible types. `var` is overwhelmingly a **local variable** convenience.

### When `var` Improves Readability

Use **`var`** when the type is **verbose and obvious** from the right-hand side — especially generic instantiations:

```csharp
var customers = new List<Customer>();
var lookup = new Dictionary<int, Order>();
```

Compare to:

```csharp
List<Customer> customers = new List<Customer>();
```

Before C# 9's target-typed `new`, the duplication (`List<Customer>` twice) added length without information. With `var`, you say the type once, on the right, where the constructor already names it.

`var` is also the **only** way to name anonymous types produced by LINQ or projection:

```csharp
var summary = orders.Select(o => new { o.Id, o.Total });
// summary has a compiler-generated anonymous type — no explicit name exists
```

For loop variables with obvious construction:

```csharp
foreach (var line in File.ReadLines(path))
{
    Console.WriteLine(line);
}
```

`line` is clearly `string` from `ReadLines`. Writing `string line` is fine too — teams differ here — but `var` avoids repeating what the API already states.

### A Practical Readability Rubric

Professional teams often adopt a rubric similar to this:

**Prefer explicit** when the right-hand side is a method call whose return type is not part of the method name, when working with numeric literals where size matters (`int` vs `long`, `float` vs `double` vs `decimal`), when declaring variables far from their initialization (split declarations are rare in modern C# anyway), and when the type communicates domain meaning (`InvoiceId` vs `int`).

**Prefer `var`** when the type appears twice in a classic `Type name = new Type(...)` pattern, when using anonymous or tuple inference where no explicit type exists, when the right-hand side is a `new`, `as`, or cast that already names the type, and when the variable name itself encodes the type (`customerCount`, `isValid`, `orderLines`).

When in doubt, ask: **"If a reviewer sees only the left side of `=`, what do they need to know?"** If the answer is obvious from context within one screen, `var` is safe. If not, spell the type.

### `var` and Primitive Types from 1.1.3

Recall that literals have default inferred types:

| Literal | Inferred type |
|---------|---------------|
| `42` | `int` |
| `42L` | `long` |
| `3.14` | `double` |
| `3.14f` | `float` |
| `19.99m` | `decimal` |
| `'A'` | `char` |
| `true` | `bool` |

So `var price = 19.99` infers **`double`**, not `decimal`. For money, either write `decimal price = 19.99m` or `var price = 19.99m` — the `m` suffix drives inference. This connects directly to your `1.1.3` knowledge: **`var` does not change numeric rules**; it follows them.

## Under the Hood

Type inference happens entirely in the **compiler front end**, before IL is emitted. Roslyn analyzes the semantic model of the initializer expression, resolves overloads, applies numeric conversions, and binds a concrete `TypeSymbol` to the variable. That symbol is frozen for the variable's lifetime in scope.

At IL level, local variables have definite types in metadata regardless of `var` in source. A decompiler shows `int32` locals, not `var`. There is zero runtime cost difference between `var` and explicit declarations.

The C# specification defines **best common type** and **target-typed inference** rules for more complex cases (conditional expressions, lambdas). You do not memorize the spec, but you should know inference can fail:

```csharp
var x = condition ? 1 : 2.0;   // infers double — int converts to double
var y = condition ? 1 : "two";   // CS0173: no common type between int and string
```

When inference fails, the compiler asks you to add an explicit type or cast — another reason explicit types remain first-class tools.

## Syntax / API

```csharp
// Legal var — initializer required
var name = "Alice";
var ids = new List<int>();

// Explicit when inference would be wrong
long big = 5_000_000_000;        // too large for int
decimal tax = 0.075m;

// Target-typed new (C# 9+) — explicit without repeating on both sides
List<string> names = new();
Dictionary<int, Order> cache = new();

// Anonymous type — var mandatory
var projection = new { Id = 1, Name = "Widget" };
```

## Examples

**Example 1 — obvious constructor, `var` reduces noise:**

```csharp
var orders = new Dictionary<Guid, Order>();
orders[Guid.NewGuid()] = new Order { Total = 100m };
```

The right-hand side names `Dictionary<Guid, Order>` explicitly. Readers who know `new` syntax know the type.

**Example 2 — hidden return type, explicit wins:**

```csharp
decimal discount = ApplyPromoCode(cart, "SAVE10");
```

`ApplyPromoCode` might live in another file. The explicit `decimal` tells maintainers we care about monetary precision.

**Example 3 — literal trap:**

```csharp
// Bug: inferred int overflows at 2_147_483_648
var size = 2_147_483_648;      // compile error — literal too big for int

// Fix:
long size = 2_147_483_648L;
```

## Common Mistakes & Anti-patterns

**Treating `var` as dynamic.** It is not. All compile-time checks apply.

**Using `var` with unclear method returns.** `var data = GetData();` tells future readers nothing. Prefer `CustomerDto data = GetData();` or rename the method.

**Forgetting suffixes with `var`.** `var rate = 0.1` is `double`; money code wants `0.1m` or explicit `decimal`.

**Banning or mandating `var` globally.** Style guides should encode the rubric above, not absolutes.

## Production & Real-World Notes

Most .NET teams use `.editorconfig` or StyleCop rules: e.g., "use `var` when type is apparent." Rider and Visual Studio analyzers can enforce team rules. Public APIs in libraries almost never use `var` in signatures — consumers need stable, visible contracts.

In code review, "Should this be `var`?" is a readability question, not a performance one. Save review energy for semantics, not religious wars.

## Comparison / Trade-offs

| Approach | Advantage | Cost |
|----------|-----------|------|
| Explicit everywhere | Maximum local clarity | Verbose; duplicated generic types |
| `var` everywhere | Less noise | Hides return types; numeric traps |
| Rubric-based (recommended) | Balanced readability | Requires team agreement |

## Quick Reference

| Situation | Recommendation |
|-----------|------------------|
| `new List<T>()` / `new Dictionary<...>()` | `var` or target-typed `new` |
| Method return not obvious | Explicit type |
| Anonymous / LINQ projection | `var` (required) |
| Numeric literals | Explicit or suffix on RHS |
| Public members | Explicit always |

---

## Key Takeaways

- `var` is **static compile-time inference**, not dynamic typing.
- **`var` requires an initializer**; type is fixed after compilation.
- Prefer **`var` when the type is obvious** from the right-hand side; prefer **explicit types when the type is the message**.
- **`var` follows literal rules from `1.1.3`** — `19.99` is `double`, `19.99m` is `decimal`.
- **No runtime difference** between `var` and explicit local declarations.
- Use a **readability rubric**, not blanket bans or mandates.
