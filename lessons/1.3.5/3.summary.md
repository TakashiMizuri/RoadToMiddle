# Events: memory leak — краткая выжимка

> `1.3.5` · ~10 мин

## Главная идея

**Event** — delegate с **add/remove** accessors; внешний код не может `=` очистить список. Delegate **сильно ссылается** на subscriber → если publisher жив дольше subscriber и нет **`-=`**, объект **не собирается GC** (memory leak).

## Ключевые пункты

- Raise: **`?.Invoke(sender, args)`** из класса publisher.
- **Static event** — держит всех subscribers до `-=`.
- **Lambda** `(s,e) => this.M()` — closure + `this` в target chain.
- Fix: **Dispose → `-=`**, named method, weak events (advanced).
- Custom **add/remove** — lock, logging, weak ref.
- Snapshot `var h = Event; h?.Invoke()` — thread race mitigation.

## Пример leak

```csharp
cache.Invalidated += listener.OnInvalidated;
// listener "out of scope" but still subscribed → not GC'd
```

Fix: `cache.Invalidated -= listener.OnInvalidated` in `Dispose`.

## «Под капотом»

Multicast delegate stores **Target** (strong). GC root path: static/singleton publisher → delegate → subscriber. `event` keyword не меняет strength references — только encapsulation.

## Частые ошибки

- Forgot `-=` on dispose.
- Lambda subscribe без сохранения reference.
- Public delegate field вместо event.

## Связи

- Предыдущая: `1.3.3`–`1.3.4` — delegates
- Следующая: `1.3.6` — lambdas (capture усиливает leak)
