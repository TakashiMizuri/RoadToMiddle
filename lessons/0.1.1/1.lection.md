# Version Control Systems: Local, Centralized, and Distributed

> Roadmap: `0.1.1` · Node: `0.1` — Git Basics · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Explain what a **Version Control System (VCS)** is and which problems it solves.
- Compare **local**, **centralized**, and **distributed** VCS architectures.
- Describe why **Git** is distributed and why that matters in real teams.
- Identify trade-offs when choosing (or migrating between) VCS models.
- Recognize historical systems (RCS, CVS, SVN) and place Git in context.

---

## Why This Matters

Imagine you maintain a file called `Report.docx`. You save copies manually:

```
Report.docx
Report_backup.docx
Report_final.docx
Report_final_v2.docx
Report_final_REAL.docx
```

This fails quickly:

| Problem | What happens |
|---------|--------------|
| **No history** | You cannot see *who* changed *what* and *when*. |
| **No safe rollback** | Restoring yesterday's version is guesswork. |
| **No collaboration** | Two people editing the same file → overwrite conflicts. |
| **No accountability** | "Who broke production?" — nobody knows. |
| **No branching** | Experimenting without breaking stable code is painful. |

A **Version Control System (VCS)** records changes to files over time so you can:

1. **Track history** — every change is stored with metadata (author, timestamp, message).
2. **Revert** — go back to any previous state.
3. **Collaborate** — multiple developers work on the same project safely.
4. **Branch & merge** — parallel lines of development (features, hotfixes).
5. **Audit** — compliance, code review, blame analysis.

Every professional software team uses a VCS. Git is the industry standard — but to understand *why* Git works the way it does, you must first understand the **three generations** of VCS architecture.

## Core Concepts

### What is Version Control?

**Version control** (also **source control** or **revision control**) is a system that:

- Stores **snapshots** or **deltas** of your project over time.
- Assigns each saved state a **revision identifier**.
- Allows **checkout** of any revision.
- Supports **merging** changes from different contributors.

Think of it as an **infinite undo stack** with a **team collaboration layer** on top.

### The Three Generations of VCS

```
Generation 1          Generation 2              Generation 3
─────────────         ─────────────             ─────────────
  LOCAL               CENTRALIZED               DISTRIBUTED

  [Dev PC]              [Dev A]                   [Dev A] ←→ [Dev B]
  history               ↓  ↓                      full repo  full repo
  on one                [Central Server]            ↓          ↓
  machine               single source               [Remote: GitHub/GitLab]
                        of truth
```

---

## Generation 1: Local VCS

### How it works

All version history lives **on one developer's machine**. No network, no server, no sharing.

**Examples (historical):**

- **RCS (Revision Control System)** — 1980s; stored deltas per file.
- Early **SCCS** — same era.

```
┌─────────────────────────┐
│   Developer's Computer  │
│                         │
│  Working files          │
│  + local history DB     │
│  (e.g. RCS ,v files)    │
└─────────────────────────┘
```

### Pros

- Simple — no infrastructure.
- Fast — everything is local.
- Works offline by definition.

### Cons

- **No collaboration** — history cannot be shared easily.
- **Single point of failure** — disk crash = history lost.
- **No backup** unless you manually copy files.
- **No code review workflow** — no shared visibility.

### When it still appears today

- macOS **Time Machine**, Windows **File History** — backup, not team VCS.
- Manual zip backups — anti-pattern in software teams.
- Some creative tools (Photoshop versions) — local only.

**Verdict:** Fine for solo experiments; **unacceptable** for team software development.

## Generation 2: Centralized VCS (CVCS)

### How it works

One **central server** holds the **official history**. Developers **check out** files, edit, and **commit** back to the server.

**Examples:**

- **CVS (Concurrent Versions System)** — 1990s.
- **SVN (Subversion)** — 2000s; still used in legacy enterprises.
- **Perforce (Helix Core)** — common in game dev / large binaries.
- **TFVC (Team Foundation Version Control)** — Microsoft stack (legacy mode).

```
        ┌──────────────┐
        │ Central      │
        │ Server       │  ← single source of truth
        │ (full history)│
        └──────┬───────┘
               │
     ┌─────────┼─────────┐
     ↓         ↓         ↓
  [Dev A]   [Dev B]   [Dev C]
  working   working   working
  copy      copy      copy
  (often    (often    (often
  partial)  partial)  partial)
```

### Typical workflow (SVN-style)

```bash
# Checkout project from central server
svn checkout https://svn.company.com/repo/trunk

# Edit files locally...

# Commit changes TO the server (requires network)
svn commit -m "Fix login bug"
```

Key idea: the **server is authoritative**. Your local machine often has **only the latest snapshot**, not full history.

### Pros

- **Team collaboration** — everyone pushes to one place.
- **Access control** — admin manages permissions on server.
- **Visibility** — managers see project state centrally.
- **Simpler mental model** than distributed (for beginners).

### Cons

| Weakness | Impact |
|----------|--------|
| **Single point of failure** | Server down → no commits, often no history access. |
| **Network dependency** | Most operations require connection to central server. |
| **No local full history** | Hard to do deep archaeology offline. |
| **Branching is heavy** | Branches often live on server; merging can be painful (SVN merge conflicts are infamous). |
| **Scaling bottleneck** | All traffic hits one server. |

### Real-world failure scenario

```
Friday 17:00 — Central SVN server disk fails.
→ No commits all weekend.
→ New developers cannot checkout.
→ Recovery depends on server's backup quality.
→ If backup is stale, days of work may be lost.
```

Centralized VCS taught the industry that **shared history matters** — but the **central server as sole custodian of history** became the bottleneck.

## Generation 3: Distributed VCS (DVCS)

### How it works

Every developer has a **full copy of the repository** — complete history, all branches, all commits — on their local machine. The "central" server (GitHub, GitLab, Bitbucket) is a **convenience for sharing**, not the sole owner of history.

**Examples:**

- **Git** — created by Linus Torvalds, 2005 (Linux kernel).
- **Mercurial (Hg)** — similar era; used by Facebook, Mozilla (historically).
- **Bazaar (Bzr)** — less common today.

```
  [Dev A — full repo]  ←—— push/pull ——→  [Remote: origin]
  [Dev B — full repo]  ←—— push/pull ——→  (GitHub/GitLab)
  [Dev C — full repo]  ←—— push/pull ——→
```

### Typical workflow (Git-style)

```bash
# Clone = get FULL history, not just latest files
git clone https://github.com/company/app.git

# Commit locally — NO network required
git add .
git commit -m "Add user profile endpoint"

# Share when ready
git push origin feature/user-profile
```

### Pros

| Advantage | Explanation |
|-----------|-------------|
| **Full local history** | `git log`, `git bisect`, archaeology — all offline. |
| **Fast operations** | Commit, branch, merge — local disk speed. |
| **Resilient** | Every clone is a backup. Server dies → push from any developer clone. |
| **Flexible workflows** | Feature branches, forks, pull requests — natural fit. |
| **Offline work** | Flights, bad Wi-Fi — keep committing locally. |
| **Cheap branches** | Branch = pointer to commit; creating one is nearly free. |

### Cons

| Disadvantage | Explanation |
|--------------|-------------|
| **Steeper learning curve** | Staging area, rebasing, detached HEAD — more concepts. |
| **Easy to misuse** | Force push, rebase on shared branches — can hurt teams. |
| **Large repos** | Full history = more disk (mitigated by shallow clone). |
| **Binary files** | Large assets bloat every clone (Git LFS helps). |

### Why Git won

1. **Performance** — designed for Linux kernel scale (millions of commits, thousands of contributors).
2. **Branching model** — branches are first-class and lightweight.
3. **Ecosystem** — GitHub (2008+) made Git + social coding mainstream.
4. **Open source** — free, cross-platform.
5. **Content-addressable storage** — integrity by design (covered in `0.1.4`).

## Under the Hood (Conceptual)

You don't need to implement Git yet — but understand the **architectural difference**:

### Centralized: client talks to server for almost everything

```
Developer → "Give me file X at revision 123" → Server
Developer → "Here is my new version of file X" → Server stores it
```

History queries, diffs, branches — often **server round-trips**.

### Distributed: client IS a full database

```
Developer → commit stored in local .git/ directory
Developer → git push sends objects server doesn't have yet
Developer → git fetch receives objects peer/server has
```

The remote is a **peer with a familiar hostname** (`origin`), not the **only place history exists**.

### Analogy: Centralized vs Distributed

| Model | Analogy |
|-------|---------|
| **Centralized** | Library with one master catalog. You borrow books; returns go to that library only. Fire destroys catalog → chaos. |
| **Distributed** | Every researcher keeps a **full copy** of the catalog + all books. They sync copies with each other. One lab burns → others still have everything. |

## Comparison Table

| Feature | Local | Centralized (SVN) | Distributed (Git) |
|---------|-------|-------------------|-------------------|
| Full history locally | ✅ (one machine) | ❌ usually partial | ✅ every clone |
| Team collaboration | ❌ | ✅ | ✅ |
| Works offline | ✅ | ❌ limited | ✅ |
| Single point of failure | Your disk | Central server | Mitigated (many full copies) |
| Branching cost | N/A | High / awkward | Very low |
| Typical remote role | None | Authority | Shared convention |
| Industry usage (2026) | Hobby | Legacy enterprise | **Standard** |

## Why Git Specifically?

Git is **distributed**, but not all DVCS are equal. Git's design choices (we'll explore in later lessons):

- **Snapshots, not just deltas** — efficient for many file types.
- **SHA-1 content addressing** — every object identified by hash of its content.
- **Three-tree model** — working directory, staging area (index), repository (`.git`).
- **Directed Acyclic Graph (DAG)** of commits — merges are first-class.

For a Middle fullstack developer, **Git is non-negotiable**. ASP.NET, React, CI/CD, code review — all assume Git.

## Common Mistakes & Anti-patterns

### 1. "We use Git but treat it like SVN"

```bash
# Anti-pattern: one long-lived branch, everyone commits to main
# No feature branches, no pull requests
git commit -m "stuff"
git push origin main   # directly to production branch
```

**Problem:** You lose Git's branching/review benefits. Might as well use SVN workflow on Git infrastructure.

### 2. "Git is just backup"

Git tracks **intent** (commits with messages), **authorship**, and **relationships** (merges, branches). Dropbox/Google Drive do **not** replace Git.

### 3. "Distributed means no central server"

Teams **still** use a central remote (`origin` on GitHub). Distributed means the server isn't the **only** copy of history — not that hierarchy disappears.

### 4. Confusing "cloud storage" with VCS

Google Docs has version history — useful, but not a replacement for code VCS (no branches, no merge model for code, no CI integration).

## Production & Real-World Notes

### What you'll see in companies

| Setup | Common in |
|-------|-----------|
| Git + GitHub/GitLab + Pull Requests | Startups, most modern teams |
| Git + Azure DevOps | .NET / Microsoft shops |
| Git + self-hosted GitLab | Regulated / on-prem industries |
| SVN → Git migration | Banks, government, legacy codebases |
| Monorepo on Git | Google-style (single huge repo) |
| Polyrepo | Many microservice repos |

### Migration from SVN to Git

Very common project. Tools: `git svn`, specialized migration scripts. Reason: developers want branches, offline work, faster operations, modern CI.

### Access control still matters

Even in DVCS, the **remote** enforces who can push to `main`. Distribution is about **data redundancy and workflow**, not "everyone can push anywhere."

---

## Trade-offs: When might centralized still appear?

- **Strict regulatory environment** with legacy SVN workflows and audit tooling.
- **Huge binary assets** (games, CAD) — Perforce still competes (partially with Git LFS).
- **Very non-technical users** — some CMS workflows mimic centralized checkout.

For **ASP.NET Core + React** fullstack work: assume **Git**.

## Quick Reference

```
LOCAL VCS        → history on one machine, no team sharing
CENTRALIZED VCS  → one server = truth (SVN, CVS)
DISTRIBUTED VCS  → every clone = full repo (Git, Mercurial)

Git workflow shorthand:
  clone  → get full repo
  commit → save snapshot locally
  push   → upload commits to remote
  pull   → download + integrate remote commits
```

## Key Takeaways

1. **VCS exists** to track change, enable collaboration, and allow safe rollback.
2. **Local VCS** cannot support real team development.
3. **Centralized VCS** (SVN) solved sharing but created a **single point of failure** and **network dependency**.
4. **Distributed VCS** (Git) gives **every developer a full repository**; the remote is for **syncing**, not sole custody.
5. **Git won** because of performance, branching, ecosystem (GitHub), and open source.
6. Using Git with an SVN mindset wastes most of Git's power.
7. As a fullstack developer, Git is the VCS you will use daily.
