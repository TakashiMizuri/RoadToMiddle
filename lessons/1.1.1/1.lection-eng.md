## Why This Matters

Phase 0 covered tooling foundations — Git, HTTP, HTML — the scaffolding around software. Phase 1 begins with the language and runtime you will use for backend work: **C# on .NET**. Before you can reason about types, memory, or async, you need a reproducible way to create projects, compile them, and run them on your machine and in CI.

The **.NET SDK** is not just a compiler. It is a toolchain: project templates, NuGet package management, build orchestration, test runners, and publish pipelines — all driven by the **`dotnet`** CLI. Teams do not hand-craft `.csproj` files from scratch every sprint; they run `dotnet new`, commit the result, and let MSBuild (invoked by `dotnet build`) produce deterministic outputs. Understanding project structure early prevents confusion when you later add ASP.NET Core packages, EF Core migrations, or Docker multi-stage builds that copy `bin/Release/publish/`.

If you treat "C# program" as a single `.cs` file you compile with `csc.exe`, you will struggle when real solutions contain dozens of projects, shared `Directory.Build.props`, and environment-specific configuration. The console template is deliberately minimal — one project, one entry point — but it already teaches the contract every .NET app shares: **source → build → assembly → runtime execution**.

## Core Concepts

### What You Are Installing

When you download **.NET SDK** from [https://dotnet.microsoft.com/download](https://dotnet.microsoft.com/download), you install several things together:

- **SDK** — compilers (`Roslyn`), MSBuild, templates, CLI tools. Needed to **build** and **publish**.
- **Runtime** — CLR + core libraries. Needed to **run** apps. End users of your published app often get runtime-only installs.
- **ASP.NET Core Runtime** — adds web server hosting stack (Kestrel). Not required for console apps but bundled in full SDK installs.

Run `dotnet --info` after install. You should see installed SDK versions, runtimes, and host information. The **host** is the `dotnet` executable that loads the correct runtime for your app's target framework.

A **target framework moniker (TFM)** like `net8.0` declares which API surface and runtime your project expects. It appears in the `.csproj` as `<TargetFramework>net8.0</TargetFramework>`. Mismatch — building for `net8.0` on a machine with only .NET 6 runtime — produces clear errors at run time, not silent corruption.

### The `dotnet` CLI as Orchestrator

Every command you run through `dotnet` delegates to MSBuild, NuGet, or the host. Common verbs:

- **`dotnet new`** — scaffold from templates.
- **`dotnet restore`** — download NuGet dependencies (often implicit on build).
- **`dotnet build`** — compile to `bin/<Configuration>/<TFM>/`.
- **`dotnet run`** — build (if needed) then execute.
- **`dotnet test`** — run test projects (later phases).

The CLI is cross-platform. The same commands work on Windows, macOS, and Linux — critical for Docker and CI pipelines.

### Creating Your First Console App

Pick a working directory and run:

```bash
dotnet new console -n HelloRoadmap -o HelloRoadmap
cd HelloRoadmap
dotnet run
```

The `-n` flag sets the project name (and default namespace). `-o` sets the output folder. Without `-o`, files are created in the current directory.

You should see `Hello, World!` printed. That output comes from the template's `Program.cs`, which uses top-level statements (C# 9+): the file is executable without an explicit `Main` method, though the compiler generates one.

Open the folder. A typical **SDK-style** console project contains:

| Path | Role |
|------|------|
| `HelloRoadmap.csproj` | Project manifest — TFM, dependencies, build settings |
| `Program.cs` | Application entry source |
| `obj/` | Intermediate build artifacts (do not commit) |
| `bin/` | Compiled output (do not commit) |

There is no `packages.config` in modern projects — dependencies are **`PackageReference`** items inside the `.csproj`, restored into a global NuGet cache (`~/.nuget/packages` or `%USERPROFILE%\.nuget\packages`).

### Anatomy of the `.csproj`

SDK-style projects are short because MSBuild imports defaults from **`Microsoft.NET.Sdk`**:

```xml
<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net8.0</TargetFramework>
    <ImplicitUsings>enable</ImplicitUsings>
    <Nullable>enable</Nullable>
  </PropertyGroup>

</Project>
```

**`OutputType`** — `Exe` for console, `Library` for class libraries, `WinExe` for Windows GUI. **`ImplicitUsings`** injects common `using` directives (`System`, `System.Linq`, etc.) so template code stays minimal. **`Nullable`** enables nullable reference type analysis — a safety feature you will use throughout the roadmap.

Adding a NuGet package modifies the same file:

```bash
dotnet add package Newtonsoft.Json
```

This inserts a `<PackageReference Include="..." Version="..." />` element. Restore happens on next build.

### Build Output and Execution Flow

When you run `dotnet build`:

1. MSBuild reads `.csproj` and imported SDK targets.
2. **Restore** ensures packages are available.
3. **Roslyn** compiles `.cs` files into IL (Intermediate Language) in an assembly — `HelloRoadmap.dll` for `OutputType=Exe`, plus a small **`HelloRoadmap.exe`** bootstrap on Windows (actually a host that loads the DLL).
4. Artifacts land in `bin/Debug/net8.0/` (configuration defaults to **Debug**).

`dotnet run` is equivalent to build + `dotnet bin/Debug/net8.0/HelloRoadmap.dll`. For Release builds: `dotnet run -c Release`.

The **entry point** is the generated or explicit `Main`. Top-level statements in `Program.cs` become a `Main` method in a compiler-generated class.

### Solution vs Project (Preview)

A **project** (`.csproj`) is one compilable unit. A **solution** (`.sln`) groups multiple projects — API, domain library, tests. You will create solutions soon with `dotnet new sln` and `dotnet sln add`. For `1.1.1`, one console project is enough; the mental model is: **solution = workspace container, project = build unit**.

## Under the Hood

### MSBuild and the SDK Import Chain

When MSBuild processes your `.csproj`, the line `<Project Sdk="Microsoft.NET.Sdk">` expands to imports from `{DOTNET_ROOT}/sdk/{version}/Sdks/Microsoft.NET.Sdk/Sdk/Sdk.props` and `Sdk.targets`. Those files define hundreds of defaults: which files to compile (`**/*.cs` except `obj/`), how to reference framework assemblies, where output goes, and how `dotnet publish` bundles dependencies.

You rarely edit those imports; you override with `<PropertyGroup>` in your project. Teams sometimes add **`Directory.Build.props`** at repo root to enforce `<Nullable>enable</Nullable>` or `<TreatWarningsAsErrors>true</TreatWarningsAsErrors>` for all projects.

### Roslyn Compilation Pipeline (Simplified)

Source text → **syntax trees** → **semantic model** (symbols, types) → **IL emit** → assembly. Errors like "type not found" fail the build at compile time. The resulting `.dll` contains IL metadata the **CLR** JIT-compiles to native code at run time (unless you use Native AOT — a later topic).

For a console app, the runtime locates the assembly containing the entry point, JITs `Main`, and executes. `Console.WriteLine` resolves to `System.Console` in `System.Console.dll` shared framework assembly — shipped with the runtime, not copied into your app folder for framework-dependent deployments.

## Syntax / Commands / API

```bash
# Verify install
dotnet --info
dotnet --version          # latest SDK version on PATH

# Create and run
dotnet new console -n MyApp -o MyApp
dotnet new list           # available templates
dotnet build
dotnet build -c Release
dotnet run
dotnet run -- arg1 arg2   # pass args to Main(string[] args)

# Clean artifacts
dotnet clean

# Pin SDK version (optional, team repos)
dotnet new globaljson --sdk-version 8.0.100
```

Default `Program.cs` (modern template):

```csharp
// See https://aka.ms/new-console-template for more information
Console.WriteLine("Hello, World!");
```

Explicit entry point equivalent (for mental model):

```csharp
namespace HelloRoadmap;

internal static class Program
{
    static void Main(string[] args)
    {
        Console.WriteLine("Hello, World!");
    }
}
```

## Examples

### Example 1: Custom Greeting with Arguments

Replace `Program.cs`:

```csharp
string name = args.Length > 0 ? args[0] : "World";
Console.WriteLine($"Hello, {name}!");
```

Run:

```bash
dotnet run -- Roadmap
```

The `--` separates `dotnet run` options from application arguments. `$"..."` is string interpolation — covered deeply in `1.1.4`.

### Example 2: Inspecting Build Output

```bash
dotnet build -c Release
dir bin\Release\net8.0\
```

You will see `MyApp.dll`, `MyApp.deps.json` (dependency manifest), `MyApp.runtimeconfig.json` (TFM and runtime options), and on Windows a host executable. Understanding these files helps when Docker COPY steps fail because publish output is incomplete.

## Common Mistakes & Anti-patterns

**Installing only the runtime** on a development machine leaves you without `dotnet new` and `dotnet build`. Developers need the **SDK**.

**Committing `bin/` and `obj/`** to Git bloats the repository and causes merge pain. Add them to `.gitignore` — `dotnet new` templates often include a suitable ignore file.

**Running `dotnet run` from the wrong directory** — MSBuild looks for a `.csproj` in the current folder. `cd` into the project directory or pass `--project path/to/App.csproj`.

**Confusing SDK version with TFM** — SDK 8.0 can target `net6.0`, `net7.0`, `net8.0`. The SDK is the toolchain; TFM is what your code compiles against.

## Production & Real-World Notes

CI pipelines run `dotnet restore`, `dotnet build -c Release`, `dotnet test`, and `dotnet publish -c Release -o ./publish`. Publish produces a self-contained or framework-dependent deployment folder — not the same as `bin/Debug`.

Teams pin SDK with **`global.json`** at repo root so local builds match CI:

```json
{
  "sdk": {
    "version": "8.0.100",
    "rollForward": "latestFeature"
  }
}
```

Docker multi-stage Dockerfiles use `mcr.microsoft.com/dotnet/sdk:8.0` for build and `mcr.microsoft.com/dotnet/runtime:8.0` (or `aspnet` for web) for the final image — smaller attack surface.

---

## Comparison / Trade-offs

**Framework-dependent deployment** — smaller output; requires matching runtime on the server. **Self-contained** — bundles runtime; larger but no server install. Console tools and microservices choose differently based on ops constraints.

**Top-level statements** vs explicit `Main` — top-level reduces ceremony for small programs; libraries and teams with strict style guides often use explicit `Main` and namespaces for clarity.

## Key Takeaways

- The **.NET SDK** includes compiler, MSBuild, templates, and CLI — required for development.
- **`dotnet new console`** scaffolds an SDK-style project with `.csproj` + `Program.cs`.
- **TFM** (`net8.0`) declares target runtime/API; must be installed on run machine for framework-dependent apps.
- **Build** produces IL assemblies in `bin/<Config>/<TFM>/`; do not commit `bin/` or `obj/`.
- **`dotnet run`** builds if needed and launches the app entry point.
- Modern templates use **top-level statements**; compiler generates `Main`.
- Real repos add **solutions**, **global.json**, and CI **`publish`** steps.
