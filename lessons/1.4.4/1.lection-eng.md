# Task.Run — Offload CPU Work, Not I/O

> Roadmap: `1.4.4` · Node: `1.4` — C# async ⚡ · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- State the **intended purpose** of **`Task.Run`**: queue CPU-bound work to the thread pool.
- Explain why **`Task.Run` + sync I/O** is an anti-pattern (fake async).
- Choose **`Task.Run`** vs **`Task.Factory.StartNew`** vs inline execution.
- Apply **`TaskCreationOptions`** (`LongRunning`, `DenyChildAttach`) when relevant.
- Bound parallel CPU offload to avoid **pool starvation**.
- Integrate with **`await`** without blocking request threads during CPU queue wait.

---

## Why This Matters

`Task.Run` is one of the most misused APIs in ASP.NET codebases. Developers wrap database calls "to make them async" or offload trivial work, adding scheduling overhead and **still blocking pool threads** during I/O waits inside the delegate.

From `1.4.2`: I/O needs true async APIs; CPU needs parallelism. **`Task.Run` is the standard way to move CPU-bound work off the ASP.NET request thread** so the request can `await` and release the thread while waiting *in queue* — but once running, the delegate still consumes a pool thread for CPU work. Used correctly, it improves **responsiveness** of the pipeline; used incorrectly, it doubles thread pool pressure.

## Core Concepts

### What Task.Run Does

**`Task.Run(Action)`** / **`Task.Run(Func<T>)`** queues a delegate to the **thread pool** (default `TaskScheduler.Default`). Returns **`Task`** / **`Task<T>`** representing that work.

It does **not** create magic parallelism beyond pool scheduling. It **does** prevent the *calling* async method from blocking the request thread *while the work item waits in queue* — but the work itself runs on a worker.

```csharp
public async Task<IActionResult> HashPassword(string pwd)
{
    var hash = await Task.Run(() => BCrypt.HashPassword(pwd));
    return Ok(hash);
}
```

Between calling `Task.Run` and delegate start, the async request method can yield — but during `BCrypt`, one pool thread is 100% busy (`1.4.2` CPU-bound).

### Why Not for I/O

```csharp
// WRONG — fake async
await Task.Run(() => File.ReadAllText(path));
```

A pool thread blocks in synchronous `ReadAllText` for the entire disk/network wait. You have **same number of blocked workers** as sync code, **plus** scheduling overhead. Use **`ReadAllTextAsync`**.

Same for **`new HttpClient().GetStringAsync().Result`** inside `Task.Run` — never do this.

### Task.Run vs Task.Factory.StartNew

**Prefer `Task.Run`** for simple CPU offload. **`StartNew`** has confusing defaults (custom scheduler, **`DenyChildAttach`**, **`LongRunning`** semantics differ). **`StartNew`** still appears in legacy code — **`Task.Run`** wraps safer options (`1.4.24`).

**`LongRunning`**: hints dedicated thread — for truly long CPU or blocking work that would starve pool; rare in web apps; prefer **`BackgroundService`** + channel.

### Bounding Concurrency

Unbounded `Task.Run` per request × 1000 requests = thousands of queued CPU jobs → **starvation** for other ASP.NET work. Use **`SemaphoreSlim`**, **`Parallel.ForEachAsync`** with `MaxDegreeOfParallelism`, or dedicated worker pool (`1.4.18`, `1.4.25`).

## Under the Hood

### Scheduling Path

`Task.Run` → internal queue to ThreadPool → worker executes delegate → task transitions to RanToCompletion/Faulted → continuations after `await Task.Run` run.

**ExecutionContext** flows to delegate (important for **`AsyncLocal`**, security context — `1.4.32`).

### ASP.NET Request Thread vs Pool Worker

Request starts on pool thread already. **`await Task.Run`** may resume continuation on *any* pool thread — often not the same as request start. **`HttpContext`** is captured in ASP.NET Core async — still available after `await` (unlike naive manual thread switch without context).

Offloading CPU from *already pool* request thread frees that worker briefly — net benefit depends on queue depth; main win is **not holding request logic synchronously** during long CPU if structured with other awaits — but CPU still costs one worker.

### When Inline Is Better

Tiny work (parse int, map DTO): **`Task.Run` overhead** (microseconds + allocation) exceeds benefit. Stay synchronous.

## Syntax / Commands / API

```csharp
await Task.Run(() => CpuHeavy());

await Task.Run(async () => { /* async lambda — runs sync portion on pool until first await */ });

Task.Run(..., cancellationToken); // token throws OCE if canceled before start

// LongRunning — dedicated thread
Task.Factory.StartNew(LongWork, TaskCreationOptions.LongRunning);
```

## Examples

### Good: CPU offload

Password hashing, image resize, large in-memory aggregation on hot API path — **`await Task.Run`** with global throttle if high QPS.

### Bad: I/O wrapper

EF sync `SaveChanges` in `Task.Run` — blocks worker; use **`SaveChangesAsync`**.

### Gray: Blocking third-party sync API

No async API available: isolate sync call in `Task.Run` **and** limit concurrency with semaphore — **tactical debt**, not architecture. Track for replacement.

## Common Mistakes & Anti-patterns

**Task.Run everywhere "because async controller".**

**Async lambda misunderstanding:** `Task.Run(async () => await Foo())` — starts on pool; first await in lambda may release worker — OK for mixed; still wrong if `Foo` is sync I/O.

**Nested Task.Run** — multiplies queue pressure.

**CPU work on UI thread** — WPF: `Task.Run` then **`Dispatcher`** back — different from ASP.NET.

## Production & Real-World Notes

Measure: if CPU offload < 1 ms median, remove `Task.Run`.

Use **`Channel<T>`** + hosted service for heavy batch CPU instead of per-request `Task.Run`.

Health: rising **threadpool-queue-length** + CPU 100% → unbounded offload.

## Comparison / Trade-offs

| Scenario | Use |
|----------|-----|
| DB/HTTP/File I/O | Native async |
| Heavy CPU on request | Bounded `Task.Run` |
| Long batch job | BackgroundService |
| Micro work | Sync inline |

## Key Takeaways

- **`Task.Run` = CPU offload to pool**, not I/O async.
- Fake async (`Task.Run` + sync I/O) **still blocks workers**.
- Prefer **`Task.Run`** over **`StartNew`**.
- **Bound** parallel CPU tasks under load.
