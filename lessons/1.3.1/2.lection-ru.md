# Generics: классы, методы и constraints

> Roadmap: `1.3.1` · Node: `1.3` — C# продвинутые темы · Depth: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Объяснить, **зачем нужны generics** и чем они отличаются от коллекций на `object`.
- Объявлять **generic-классы**, **методы** и **интерфейсы** с type parameters.
- Применять **`where`-ограничения** (`class`, `struct`, `new()`, interface, base type) и предсказывать ошибки компиляции.
- Различать **open types** и **closed constructed types**, читать generic metadata.
- Описать, как **CLR реализует generics** (reification), и почему value types не boxing'ятся.
- Находить типичные ошибки: избыточные constraints, путаница с variance (preview `1.3.2`), `new()` на abstract types.

## Why This Matters

До generics переиспользуемые контейнеры выглядели как `ArrayList` — каждый элемент хранился как `object`. Чтение `int` требовало cast и рисковало **`InvalidCastException` в runtime**. Generics переносят проверку на **compile time**: в `List<int>` нельзя случайно положить `string`. Получаешь **type safety**, **IntelliSense** и часто **нулевой boxing** для value types.

Generics повсюду в современном C#: `List<T>`, `Dictionary<TKey, TValue>`, `Task<T>`, `IEnumerable<T>`, `IRepository<TEntity>`, middleware ASP.NET Core. Middle-разработчик не должен воспринимать `<T>` как синтаксический сахар — нужно понимать **что генерирует компилятор**, **что разрешают constraints** и **как `where T : class` влияет на nullable reference types**.

Без этой базы LINQ (`1.3.11`), DI (`IOptions<T>`) и EF Core (`DbSet<T>`) остаются «магическими API», а не частью единой type system.

## Core Concepts

### Какую проблему решают generics

Стек на `object`:

```csharp
public class ObjectStack
{
    private object[] _items = new object[8];
    private int _count;

    public void Push(object item) => _items[_count++] = item;
    public object Pop() => _items[--_count];
}
```

`Push(42)` **boxing'ит** `int` в heap object. `Pop()` требует `(int)Pop()` — неверный cast падает в runtime. Компилятор не связывает тип push и pop.

**Generics** параметризуют типы. `Stack<T>` — **шаблон**; `Stack<int>` и `Stack<string>` в .NET — **разные closed types** в runtime:

```csharp
public class Stack<T>
{
    private T[] _items = new T[8];
    private int _count;

    public void Push(T item) => _items[_count++] = item;
    public T Pop() => _items[--_count];
}
```

`Stack<int>.Push(42)` кладёт значение прямо в `T[]` — без boxing. `Pop()` возвращает `int` без cast.

### Type parameters: open vs closed

`Stack<T>` — **open generic type**, `T` — placeholder. `Stack<int>` — **closed constructed type**. Open type нельзя инстанцировать:

```csharp
// Stack<T> stack = new Stack<T>(); // ошибка — T неизвестен
var stack = new Stack<int>();       // OK
```

Метод может быть generic внутри non-generic класса:

```csharp
public static class Utils
{
    public static void Swap<T>(ref T a, ref T b) => (a, b) = (b, a);
}
```

`Swap<int>` и `Swap<string>` — один исходный код, но JIT компилирует **специализированный machine code**: для reference types часто один shared path, для **каждого value type** — отдельный native code.

### Generic interfaces

```csharp
public interface IRepository<TEntity> where TEntity : class
{
    Task<TEntity?> GetByIdAsync(int id, CancellationToken ct = default);
    Task AddAsync(TEntity entity, CancellationToken ct = default);
}
```

Реализация закрывает параметр: `UserRepository : IRepository<User>`. Constraints интерфейса наследуются — struct `User` не пройдёт `where TEntity : class`.

### Constraints: что может делать T

Без constraints `T` трактуется как **`object`** для members — доступны только members `object`. Constraints открывают API:

| Constraint | Смысл |
|---|---|
| `where T : struct` | non-nullable value type |
| `where T : class` | reference type |
| `where T : notnull` | не null |
| `where T : unmanaged` | без managed полей — interop |
| `where T : new()` | parameterless constructor |
| `where T : SomeBase` | наследник или сам base |
| `where T : ISomeInterface` | implements interface |

Несколько constraints на одном параметре — **AND**. У каждого type parameter свой `where`:

```csharp
public static TOutput Convert<TInput, TOutput>(TInput input, Func<TInput, TOutput> map)
    where TInput : notnull
    where TOutput : struct
    => map(input);
```

### Type inference

Компилятор часто **выводит** type arguments:

```csharp
int max = Utils.Max(3, 7); // Max<int>

public static T Max<T>(T a, T b) where T : IComparable<T>
    => a.CompareTo(b) >= 0 ? a : b;
```

Если inference не срабатывает — явно: `Utils.Max<double>(3, 7.5)`.

### `default(T)`

Для unconstrained `T`: `default(T)` — **`null`** для reference types, **zero-bit pattern** для value types. С `where T : struct` доступен `T?`.

## Under the Hood

### Reification в CLR

.NET использует **reification**: generic info сохраняется в runtime. `typeof(List<int>)` и `typeof(List<string>)` — **разные** `Type`. В Java (erasure) info в основном compile-time; в C# можно `typeof(T)` внутри generic method.

При JIT: **reference type arguments** часто делят одну реализацию (все ссылки pointer-sized). **Каждый value type** — свой JITted code (`List<int>` vs `List<double>` — разный размер элементов в массиве). Trade-off: скорость vs размер native code.

### IL и metadata

Generic class в IL с **type parameters** в metadata. Call sites — `callvirt`/`newobj` на constructed types. Constraints в metadata — компилятор и runtime проверяют.

Disassembly `Stack<int>.Push`: нет `box int32`. `ObjectStack.Push(int)` покажет `box` перед `stelem.ref`.

### Boxing avoidance

`List<int>` → `int[]` — contiguous, cache-friendly. `ArrayList` с int — boxed object на элемент, больше GC pressure.

Generic specialization помогает **devirtualization** и inlining, когда JIT знает struct `T` с interface constraint.

### Ограничения runtime

- `class Foo : T` — illegal.
- Static fields — **per closed type**.
- `typeof(T)` OK; `new T()` только с `new()`.
- Enum как `T`: struct semantics; арифметика — с `where T : Enum` (.NET 7+).

## Common Mistakes & Anti-patterns

**Over-constraining** — лишние `where` блокируют struct projections и усложняют API.

**Escape через `object`/`dynamic`** — теряешь compile-time checks.

**Static field «один на все T»** — `GenericClass<int>.Counter` ≠ `GenericClass<string>.Counter`.

**Shadowing `T`** — method `M<T>()` в `class C<T>` — другой `T`.

**NRT + `where T : class`** — flow analysis ограничен для generic type parameters.

## Production & Real-World Notes

DI: `services.AddScoped(typeof(IRepository<>), typeof(EfRepository<>))` — container закрывает `T` при resolve `IRepository<Order>`.

`ILogger<T>`, `IOptions<T>`, `Channel<T>` — category из `typeof(T).Name`.

Source generators (System.Text.Json) — generic API, specialized code per `T`.

Code review red flag: `List<object>` в business logic.

## Comparison / Trade-offs

| Подход | Плюсы | Минусы |
|---|---|---|
| **Generics** | Type-safe, no boxing | Больше JIT code для value types |
| **`object` + cast** | Просто | Runtime failures, boxing |
| **Наследование** | Явно | Explosion of types |
| **`dynamic`** | Гибко | Медленно, нет IntelliSense |

## Key Takeaways

- Generics — **compile-time safety** и **performance** без cast/boxing.
- **Open** vs **closed** — instantiate только closed.
- **`where`** unlocks members, проверяется компилятором.
- .NET **reifies** generics; value types — specialized JIT.
- Static members — **per closed type**.
- DI использует **open generic registration**.
