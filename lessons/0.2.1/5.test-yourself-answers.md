## Answers

### 1. [Recall]
**Answer:** Одна строка — forty-character SHA-1 (или SHA-256) hash **tip commit** ветки `feature/login`.

**Why:** Branch ref — pointer file, не directory и не список файлов.

---

### 2. [Recall]
**Answer:** В detached HEAD файл `.git/HEAD` содержит **прямой commit hash**, а не `ref: refs/heads/main`. Ты не «на ветке» — новые commit'ы не привязаны к branch name.

**Why:** Symbolic ref на branch vs direct commit checkout — два режима HEAD из `0.1.4`.

**Common mistake:** Думать, что detached HEAD = повреждение репозитория.

---

### 3. [Recall]
**Answer:** **`git switch`** (для веток; restore файлов — `git restore`).

**Why:** Git 2.23 разделил checkout на switch + restore.

---

### 4. [Understanding]
**Answer:** `git branch` только **создаёт ref-файл** с hash текущего HEAD. HEAD, index и working tree **не меняются** — ты остаёшься на прежней ветке с теми же файлами на диске.

**Why:** Branch creation ≠ checkout. Нет синхронизации tree другого commit.

---

### 5. [Understanding]
**Answer:** HEAD (обычно symbolic) → branch ref → commit hash → root tree → blobs → content файлов. При **`git commit`**: создаётся **новый commit object**, **branch ref** (на которую указывает HEAD) **перезаписывается** новым hash; HEAD остаётся `ref: refs/heads/<branch>`. Objects старого commit не меняются.

**Why:** Commit append-only; двигается ref tip, не history objects.

---

### 6. [Understanding]
**Answer:** **`main` всё ещё указывает на тот же commit**, что и до commit Alice. Двигается только ref ветки, на которой был сделан commit (`feature`), потому что commit обновляет ref текущей checked-out ветки.

**Why:** Один ref файл на ветку; не checked-out refs не двигаются от local commit.

---

### 7. [Understanding]
**Answer:** Switch заменяет tracked files content из tree целевого commit. Uncommitted правки были бы **безвозвратно перезаписаны** или смешаны некontrollируемо. Git останавливается, чтобы ты явно commit/stash/discard.

**Why:** Защита данных — checkout не «мержит» silently опасные overwrites.

---

### 8. [Application]
**Answer:** `git switch -c feature/<name>` (или `git checkout -b feature/<name>`).

**Why:** `-c` создаёт ref и переключает HEAD за один шаг.

---

### 9. [Application]
**Answer:** Любое из:
- `git switch -c branch-name` — создать ветку на текущем detached commit.
- `git branch branch-name` затем позже switch (если ещё на том commit).
- Запомнить hash и восстановить через `git reflog` + `git switch -c` (менее предпочтительно, но возможно).

**Why:** Нужен named ref, иначе commit orphan при уходе с detached HEAD.

---

### 10. [Application]
**Answer:** `git restore Program.cs` (или `git checkout -- Program.cs`).

**Why:** `restore` откатывает working tree file; ветку и другие staged файлы не трогает.

---

### 11. [Analysis]
**Answer:** **B** неверно.

**Why:** `-d`/`-D` удаляет **ref file**, не commit objects. Commit'ы остаются в object store, пока достижимы или пока gc не соберёт unreachable.

---

### 12. [Analysis]
**Answer:** Git **не копирует** проект. Создаётся файл `.git/refs/heads/feature` с hash текущего commit. Все objects (blobs, trees, commits) **общие** с другими refs до divergence. Working tree один — меняется только при **switch** на другую ветку.

**Why:** Модель refs из `0.1.4` — branches as pointers, not folders.

---

### 13. [Understanding]
**Answer:** **`refs/heads/main`** — mutable branch tip, двигается при новых commit'ах на main. **`refs/tags/v1.0.0`** — обычно **фиксированная** метка release на конкретный commit; conventionally не переносится при новых commit'ах.

**Why:** Tags для immutable milestones; branches для ongoing work.

---

### 14. [Application]
**Answer:** `git switch -` (переключение на предыдущую checked-out ветку/ref).

**Why:** Git запоминает предыдущий branch context для `-`.

---

### 15. [Analysis]
**Answer:** Commit'ы могут существовать, если:
1. На них указывает **другая ветка или tag**.
2. Они ещё в **reflog** (записи о прошлых положениях HEAD/refs).
3. Другой ref достижим через **ancestry** от текущих tips.
4. **Garbage collection** ещё не запускался / objects ещё reachable.

**Why:** Delete branch = delete name, not immediate object purge.
