# LINQ Method Syntax: Where, Select, OrderBy, GroupBy

> Roadmap: `1.3.7` ¬∑ Node: `1.3` ‚Äî C#: advanced topics ¬∑ Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Explain **LINQ** as a composable query layer over sequences, built on **generics** and **delegates**.
- Use **method syntax** (`Where`, `Select`, `OrderBy`, `GroupBy`) to filter, project, sort, and aggregate in memory.
- Predict **return types** and **element types** at each stage of a chained pipeline.
- Describe how **lambda expressions** (`1.3.6`) become predicate and projection delegates at each operator.
- Articulate why operator **order matters** for both correctness and performance.
- Connect these in-memory operators to the **same vocabulary** Entity Framework Core will translate to SQL later.

## Why This Matters

In lessons `1.3.3`‚Äì`1.3.6` you learned **delegates**, **Func/Action**, and **lambdas** ‚Äî the raw machinery of "pass a function as data." **Language Integrated Query (LINQ)** is where that machinery becomes productive daily syntax. Instead of writing manual `foreach` loops with temporary lists, you declare *what* you want from a sequence and compose small operators into pipelines.

Every ASP.NET Core service you will write touches LINQ: filtering DTO lists in memory, projecting EF entities to response models, grouping analytics by date, ordering paginated results. The four operators in this lesson ‚Äî **Where**, **Select**, **OrderBy**, **GroupBy** ‚Äî are the backbone. Master them in memory first; in `1.3.11`‚Äì`1.3.12` and later EF Core topics, the *same method names* appear on `IQueryable<T>`, but the provider translates them to SQL instead of executing loops on the server process.

Middle developers do not memorize fifty LINQ methods on day one. They internalize a **pipeline mental model**: source sequence ‚Üí transform ‚Üí transform ‚Üí materialize. This lesson builds that model with the operators you will use most often, and opens the hood enough that deferred execution (`1.3.9`) and expression trees (`1.3.12`) will not feel like magic later.

## Core Concepts

### LINQ as Extension Methods on IEnumerable<T>

LINQ to Objects operates on **`IEnumerable<T>`** ‚Äî anything you can iterate with `foreach`. The operators live in static class **`System.Linq.Enumerable`** as **extension methods** (full compile-time story in `1.3.14`). When you write:

```csharp
var adults = users.Where(u => u.Age >= 18);
```

the compiler rewrites it roughly as `Enumerable.Where(users, u => u.Age >= 18)`. The lambda `u => u.Age >= 18` is a **`Func<User, bool>`** ‚Äî a predicate. **`Where`** returns another `IEnumerable<User>` that *describes* the filter; it does not necessarily loop yet (deferred execution ‚Äî `1.3.9`).

Each operator is a **pure transformation descriptor** until something forces enumeration (`ToList`, `foreach`, `Count` ‚Äî `1.3.10`).

### Where ‚Äî Filtering by Predicate

**`Where`** keeps elements for which the predicate returns `true`. Signature conceptually:

```csharp
IEnumerable<T> Where<T>(this IEnumerable<T> source, Func<T, bool> predicate)
```

An overload accepts `Func<T, int, bool>` where the `int` is the zero-based index ‚Äî useful when position matters:

```csharp
var everySecond = items.Where((item, index) => index % 2 == 0);
```

**Where** does not mutate the source collection. It returns a new sequence *view*. Multiple **Where** calls compose (each wraps the previous):

```csharp
var result = orders
    .Where(o => o.Status == OrderStatus.Paid)
    .Where(o => o.Total > 100);
```

Logically equivalent to one predicate with `&&`, but sometimes separate **Where** steps improve readability when each condition has a business name worth extracting.

**Performance note:** filter early. If you **Select** first and produce heavy objects, then **Where**, you may allocate projections you immediately discard. Prefer **Where** before **Select** when the filter applies to source fields.

### Select ‚Äî Projection (Map)

**`Select`** transforms each element to another type ‚Äî the functional **map** operation:

```csharp
IEnumerable<TResult> Select<TSource, TResult>(
    this IEnumerable<TSource> source,
    Func<TSource, TResult> selector)
```

Example ‚Äî entities to DTOs:

```csharp
var dtos = products.Select(p => new ProductDto(p.Id, p.Name, p.Price));
```

The output sequence has element type **`TResult`**, not `TSource`. Chaining changes the generic type at each step:

```csharp
var names = users.Select(u => u.Name);           // IEnumerable<string>
var lengths = users.Select(u => u.Name.Length);  // IEnumerable<int>
```

**Select** with index:

```csharp
var numbered = items.Select((item, i) => $"{i + 1}. {item}");
```

A common pattern in APIs: project early to the shape you need, so downstream code does not depend on entity types. EF Core uses **Select** heavily for **server-side projection** ‚Äî selecting only required columns instead of loading full entities (`SELECT Id, Name` not `SELECT *`).

### OrderBy, ThenBy ‚Äî Sorting

**`OrderBy`** and **`OrderByDescending`** sort by a key extracted via selector:

```csharp
IEnumerable<T> OrderBy<T, TKey>(this IEnumerable<T> source, Func<T, TKey> keySelector)
```

```csharp
var byPrice = products.OrderBy(p => p.Price);
var byPriceDesc = products.OrderByDescending(p => p.Price);
```

Return type is **`IOrderedEnumerable<T>`**, which exposes **`ThenBy`** / **`ThenByDescending`** for secondary sort keys:

```csharp
var sorted = customers
    .OrderBy(c => c.Country)
    .ThenBy(c => c.LastName)
    .ThenBy(c => c.FirstName);
```

Sorting is **stable** in LINQ to Objects: equal keys preserve relative order from the input sequence.

**Important:** **`OrderBy` after `Where`** is typical. **`Where` after `OrderBy`** also works but re-filters an already sorted sequence ‚Äî rarely what you want unless the pipeline is built dynamically.

Sorting requires knowing keys for **all** elements (or at least enough to produce correct order). That implies the sort operator must **consume** the upstream sequence through the sort algorithm ‚Äî effectively a buffering operation internally. We will revisit what that means for deferred execution in `1.3.9`.

Default comparer uses **`Comparer<TKey>.Default`** ‚Äî for `string`, ordinal by default in recent .NET; for custom types, ensure **`IComparable<T>`** or pass **`OrderBy(key, comparer)`**.

### GroupBy ‚Äî Splitting into Buckets

**`GroupBy`** partitions a sequence into **`IGrouping<TKey, TElement>`** groups sharing the same key:

```csharp
IEnumerable<IGrouping<TKey, TElement>> GroupBy<TSource, TKey>(
    this IEnumerable<TSource> source,
    Func<TSource, TKey> keySelector)
```

```csharp
var byCountry = orders.GroupBy(o => o.ShippingCountry);

foreach (var group in byCountry)
{
    Console.WriteLine($"{group.Key}: {group.Count()} orders");
    foreach (var order in group)
        Console.WriteLine($"  {order.Id}");
}
```

Each **`IGrouping<TKey, TElement>`** implements **`IEnumerable<TElement>`** and exposes **`Key`**. Default **`TElement`** equals **`TSource`** ‚Äî every order appears in its country group.

Overload with **element selector** ‚Äî group keys but project elements differently:

```csharp
var totalsByCountry = orders.GroupBy(
    o => o.ShippingCountry,
    o => o.Total);
// IGrouping<string, decimal> ‚Äî each group contains totals, not full orders
```

Overload with **result selector** ‚Äî collapse each group immediately:

```csharp
var summary = orders.GroupBy(
    o => o.ShippingCountry,
    (country, items) => new { Country = country, Count = items.Count(), Sum = items.Sum() });
```

**GroupBy** is the in-memory analogue of SQL **`GROUP BY`**. EF Core translates **`GroupBy`** only when the shape is translatable; arbitrary client logic inside grouping may force client evaluation ‚Äî a production pitfall you will meet with databases.

### Chaining: Reading Pipelines Left to Right

A typical pipeline:

```csharp
var report = sales
    .Where(s => s.Date.Year == 2026)
    .GroupBy(s => s.Region)
    .Select(g => new RegionSummary(g.Key, g.Sum(x => x.Amount), g.Count()))
    .OrderByDescending(r => r.Total)
    .Take(10);
```

Read it as: restrict to 2026 ‚Üí partition by region ‚Üí project summaries ‚Üí sort by total ‚Üí take top 10. Each step returns a new **`IEnumerable<...>`** (or **`IOrderedEnumerable<...>`**). Nothing runs until **`ToList()`**, **`foreach`**, or similar (`1.3.10`).

Operator order changes **meaning**:

```csharp
// Top 10 regions by total AFTER full aggregation
.GroupBy(...).Select(...).OrderByDescending(...).Take(10)

// WRONG if you wanted top 10 *sales* before grouping:
.Take(10).GroupBy(...)  // groups only 10 arbitrary sales
```

This is not a stylistic preference ‚Äî it is semantic. SQL queries have the same discipline.

## Under the Hood

### Iterator Blocks and Deferred Chains

Most LINQ to Objects operators are implemented with **`yield return`** iterator blocks (`1.3.20`). **`Where`**, for example, iterates the source once, testing the predicate per element and yielding matches ‚Äî no intermediate list unless the operator requires it.

**`GroupBy`** and **`OrderBy`** are heavier: they typically buffer data into internal structures (hash buckets or sort buffers) before yielding. That means the moment you **foreach** a grouped or sorted sequence, the upstream pipeline executes at least through that operator's buffering requirement.

Conceptually:

```
source ‚Üí [Where iterator] ‚Üí [Select iterator] ‚Üí [OrderBy buffer+sort] ‚Üí consumer
```

Each wrapper holds a reference to the previous **`IEnumerable`** and its delegate fields ‚Äî the lambda captures become fields in the compiler-generated iterator class.

### Closures and Captured Variables

Lambdas in LINQ often **capture** outer variables:

```csharp
int minAge = 18;
var adults = users.Where(u => u.Age >= minAge);
minAge = 21;  // changes the filter threshold before enumeration!
```

The lambda closes over **`minAge`**, not its value at pipeline construction time (unless you copy to a local inside a loop carefully). This surprises developers who build dynamic filters in loops ‚Äî a classic bug when composing **`Where`** inside **`foreach`**.

### Type Inference Through the Pipeline

The compiler infers **`T`** and **`TResult`** at each **`Select`**. A broken pipeline often fails at compile time:

```csharp
var x = users.Select(u => u.Name).Where(n => n.Age > 18); // compile error: string has no Age
```

**Where** after projecting to **`string`** cannot access **`User.Age`**. Filter on source properties **before** projecting away those properties.

### Connection to EF Core (Preview)

When the source is **`DbSet<Product>`** (`IQueryable<Product>`), identical-looking **`.Where(...).Select(...).OrderBy(...).GroupBy(...)`** calls build an **expression tree** instead of embedding delegates. The EF provider walks that tree and emits SQL. The method names match LINQ to Objects; the **execution model** differs fundamentally ‚Äî **`1.3.11`** and **`1.3.12`**. For now, master semantics in memory; the vocabulary transfers directly.

## Syntax / Commands / API

| Operator | Purpose | Typical signature |
|----------|---------|-------------------|
| `Where` | Filter | `Func<T, bool>` |
| `Select` | Project | `Func<T, TResult>` |
| `OrderBy` / `OrderByDescending` | Primary sort | `Func<T, TKey>` |
| `ThenBy` / `ThenByDescending` | Secondary sort | on `IOrderedEnumerable<T>` |
| `GroupBy` | Partition | key selector + optional element/result selectors |

All live in namespace **`System.Linq`** (included via implicit usings in modern templates).

## Examples

### Example 1 ‚Äî E-commerce order report

```csharp
public record Order(int Id, string Country, decimal Total, DateTime PlacedAt);

Order[] orders = /* ... */;

var topCountries = orders
    .Where(o => o.PlacedAt >= DateTime.UtcNow.AddMonths(-1))
    .GroupBy(o => o.Country)
    .Select(g => new { Country = g.Key, Revenue = g.Sum(o => o.Total) })
    .OrderByDescending(x => x.Revenue)
    .Take(5)
    .ToList();
```

We filter to the last month, group by country, project revenue per group, sort descending, take five, then **materialize** with **`ToList()`**. Until **`ToList()`**, the pipeline is a description.

### Example 2 ‚Äî Index in Select

```csharp
var lines = linesOfText
    .Select((line, index) => new { Number = index + 1, Text = line })
    .Where(x => !string.IsNullOrWhiteSpace(x.Text));
```

Note **`Where`** after **`Select`** ‚Äî we filter on projected shape (`Text`), not original strings alone.

## Common Mistakes & Anti-patterns

**Filtering after projecting away needed fields.** Move **`Where`** before **`Select`** when the predicate uses source properties.

**Assuming GroupBy returns a dictionary.** It returns **`IEnumerable<IGrouping<...>>`**, lazy. Multiple enumerations re-run grouping unless you **`ToList()`** the groups.

**Multiple OrderBy calls instead of ThenBy.** Second **`OrderBy`** re-sorts entirely by the new key, discarding previous ordering ‚Äî use **`ThenBy`**.

**Building pipelines with side effects inside Select.** **`Select`** should be a pure projection. Logging or saving inside **`Select`** runs on every enumeration of the sequence ‚Äî possibly zero, once, or many times (`1.3.9`).

## Production & Real-World Notes

In ASP.NET Core, in-memory LINQ on small collections (config lists, enum metadata) is fine. On large datasets, prefer **`IQueryable`** with EF so **`Where`/`Select`/`OrderBy`/`GroupBy`** run in PostgreSQL ‚Äî not in your app's RAM. Pagination belongs at the query level: **`Skip`/`Take`** (`1.3.26`) after **`OrderBy`**, not after loading all rows.

Code review often asks: "Could this **`GroupBy`** happen in SQL?" If yes, keep it on **`IQueryable`**. Profile before micro-optimizing in-memory pipelines.

## Comparison / Trade-offs

**Method syntax vs query syntax (`1.3.8`):** These operators are identical; query syntax compiles to method calls. Method syntax shines for **`Select`/`Where`** chains; query syntax for multi-table joins later.

**LINQ vs manual loops:** Loops offer slightly more control and no delegate allocation in hot paths; LINQ wins on readability and composability. For hot paths, measure ‚Äî **`1.3.29`** covers performance.

## Quick Reference

| Goal | Operator |
|------|----------|
| Filter | `Where` |
| Transform | `Select` |
| Sort | `OrderBy` / `ThenBy` |
| Partition | `GroupBy` |
| Materialize | `ToList`, `ToArray` (`1.3.10`) |

Pipeline order: **filter ‚Üí group ‚Üí project ‚Üí sort ‚Üí page**.

## Key Takeaways

- LINQ method syntax composes **`IEnumerable<T>`** transformations via extension methods and delegates.
- **`Where`** filters; **`Select`** projects; **`OrderBy`/`ThenBy`** sort; **`GroupBy`** partitions like SQL **`GROUP BY`**.
- Each step returns a new sequence type; chained generics must align.
- Operator **order changes semantics** ‚Äî especially **`Take`** vs **`GroupBy`**.
- Execution is **deferred** until enumeration (`1.3.9`); **`OrderBy`/`GroupBy`** buffer internally.
- Same names appear on **`IQueryable`** for EF Core ‚Äî execution moves to the database provider.
