# Репозиторий, working tree, staging area и commit

> Roadmap: `0.1.2` · Узел: `0.1` — Git: основы · Глубина: понимание

## Learning Objectives

После этого урока ты сможешь:

- Определить **Git repository** как **working tree** + директорию **`.git`**.
- Объяснить **три состояния** файла: modified, staged, committed.
- Различать **working directory**, **staging area (index)** и **repository (`.git`)**.
- Объяснить, **зачем нужен staging area** и когда полезны partial commits.
- Описать **commit** как snapshot с metadata (author, timestamp, message, parent).
- Проследить **lifecycle файла** от правки до commit по диаграмме.

## Why This Matters

В `0.1.1` ты узнал, что Git — **distributed** VCS: каждый clone содержит полную history проекта, а не только последние файлы. Это объясняет, почему можно commit'ить в самолёте и push'ить позже. Но это ещё не объясняет, что происходит на ноутбуке *до* push — как Git узнаёт об изменении файла, почему часть правок в одной секции `git status`, а часть в другой, и зачем профессионалы делают и `git add`, и `git commit`.

В отличие от некоторых VCS, где «сохранить» сразу пишет в permanent history, Git вставляет промежуточный слой — **staging area** (также **index**). Новички спрашивают: «Зачем два шага?» Вопрос справедливый; ответ в том, что Git оптимизирован под **осознанные snapshots**, а не случайные. Когда internalize **three-tree model** — working directory, staging area, repository — исчезают ежедневные загадки. Понятно, почему `git status` разделяет «Changes not staged for commit» и «Changes to be committed». Видно, как закоммитить **только часть** работы одним atomic snapshot. Появляется preview того, что `add` и `commit` реально записывают — разберём глубже в `0.1.4`.

Feature branches, pull requests, code review, CI — всё стоит на этом фундаменте. Remote operations позже; local state management — сначала.

## Core Concepts

### Что такое Git repository?

Фраза «clone repository» часто рисует картину папки с исходниками. Картина неполная. **Git repository** (repo) состоит из двух частей: **working tree** (**working directory**) — файлы, которые ты видишь, редактируешь и запускаешь — и **Git directory**, скрытой папки `.git/` с базой snapshots, history и config.

```
my-project/
├── .git/              ← repository (history, objects, refs)
├── src/
│   └── app.cs         ← working tree (файлы для редактирования)
├── README.md
└── Program.cs
```

При `git clone` или `git init` Git создаёт или копирует `.git`. Вся history живёт там. Видимые файлы проекта — **checkout**: content из последнего commit для работы. Working tree может расходиться с тем, что записано в `.git`; Git отслеживает различия и по твоему выбору фиксирует их в commit.

Разделение важно на практике. Можно править десять файлов, но commit'ить два. Рядом с production-ready изменениями могут лежать uncommitted эксперименты. Repository — не «всё, что сейчас на диске», а «то, что явно staged и committed в `.git`».

### Три состояния файла

Git относит каждый **tracked** файл к одному из трёх состояний. **Modified** — изменён в working tree, но не помечен для следующего commit. **Staged** — после `git add` snapshot выбранных изменений попал в staging area (index) и ждёт `git commit`. **Committed** — после commit staged snapshot записан в permanent history внутри `.git`.

Переходы образуют цикл, который ты выполнишь сотни раз:

```
                    git add                git commit
  ┌──────────┐    ──────────→    ┌──────────┐    ──────────→    ┌──────────┐
  │ Modified │                   │  Staged  │                   │Committed │
  │(working) │    ←──────────    │ (index)  │                   │  (.git)  │
  └──────────┘    git restore    └──────────┘                   └──────────┘
                  (unstage)
```

После базового пути — нюанс для новичков. Файл может быть **modified и staged одновременно**. Сделал `git add Program.cs`, снова отредактировал до commit — Git ведёт и staged portion (ранее добавленное), и новые unstaged edits. `git status` покажет одно имя в двух секциях. Это не баг — модель «что войдёт в следующий commit» vs «что ещё редактируешь».

**Untracked** файлы вне цикла трёх состояний до первого `git add`. Они в `git status`, но ещё не под version control.

### Три дерева: Working Directory, Staging Area, Repository

Внутренняя модель Git — **three-tree architecture**. Три представления проекта, которые Git постоянно сравнивает:

```
┌─────────────────┐     git add      ┌─────────────────┐    git commit    ┌─────────────────┐
│ Working         │  ──────────────→ │ Staging Area    │  ──────────────→ │ Repository      │
│ Directory       │                  │ (Index)         │                  │ (.git/)         │
│                 │  ← git checkout  │                 │                  │                 │
│ Файлы на диске  │    -- file       │ Preview         │                  │ Постоянная      │
│                 │                  │ следующего      │                  │ history         │
│                 │                  │ snapshot        │                  │                 │
└─────────────────┘                  └─────────────────┘                  └─────────────────┘
```

**Working directory** — проект как в IDE и проводнике. Печатаешь в файле — меняется working directory. Git сравнивает working files с index и последним commit (`HEAD`) для status.

**Staging area** — **index** — между правками и permanent history. `git add` записывает snapshot выбранных изменений в index. Аналогия **shipping crate**: решаешь, что войдёт в следующую «посылку», прежде чем «запечатать». Index — реальный binary file `.git/index`; связь с objects в `0.1.4`.

**Repository** — директория `.git`. **Commit** пишет staged snapshot из index в базу и связывает с предыдущей history. Committed data **content-addressed** (идентифицируется hash) — подробнее в `0.1.5`.

### Зачем нужен staging?

Почему Git не commit'ит напрямую из working directory? Реальная работа messy. За один день — typo в `README.md`, половина login API, правка local `appsettings.json` с dev-only connection string. Без staging либо commit всего вместе — docs, feature и config — либо сложный partial-commit UI.

Staging даёт **atomic commits**: один commit = одно logical change для code review. **Partial commits**: stage `AuthService.cs`, оставь `ScratchPad.cs` unstaged. **Review before commit**: `git diff --staged` — финальный checklist. **Deliberate snapshots** по hunks через `git add -p` — hands-on в `0.1.3`.

Аналогия: working directory — messy desk; staging area — inbox tray; repository — archived boxes на полке. Git выбрал explicit staging ради **control и clarity**, приняв лишний шаг ради history, которой можно доверять через полгода.

### Что такое commit?

**Commit** — не casual save. Immutable **snapshot** проекта в момент времени плюс **metadata** для social и historical context. Snapshot (tree objects) перечисляет файлы и content hashes. **Parent(s)** связывают с predecessor — граф history. **Author** / **committer** — кто написал и кто записал (могут различаться). **Timestamp**. **Commit message** — *зачем*; future teammates и future you читают в `git log`.

```
Commit abc123
├── parent: def456
├── author: Alice <alice@dev.com>  2026-07-01 10:00
├── message: "Add password validation"
└── snapshot:
    ├── src/AuthService.cs  (content hash ...)
    ├── src/Program.cs
    └── README.md
```

Git хранит **snapshots**, не только line-by-line deltas; сжимает эффективно. Неизменённые файлы между commits ссылаются на те же objects.

Commits **immutable**. History не «редактируют» — создают новые commits и двигают branch pointers. Rebase, amend — позже. Сейчас каждый commit — journal entry, которым хочешь гордиться.

## Under the Hood (Conceptual)

Читать raw files в `.git/` пока не нужно — но flow между тремя деревьями нужно понять:

```
1. Редактируешь Program.cs          → working tree меняется
2. git add Program.cs               → Git хеширует content → blob object
                                      → index entry указывает на blob
3. git commit                       → tree object из index
                                      → commit object (tree + parent + metadata)
                                      → branch ref (например main) → новый commit
```

**Index** — blueprint **следующего** commit. До commit permanent history не меняется — только index и working tree могут отличаться от `HEAD`. **`HEAD`** — указатель на текущий commit, обычно через branch `main`. Refs и objects — в `0.1.4`.

## Syntax / Commands / API

Урок концептуальный; hands-on — в `0.1.3`. Связь команд с переходами между состояниями:

- **`git status`** — состояние каждого файла.
- **`git add <path>`** — working tree → staged (index).
- **`git restore --staged <path>`** — unstage без удаления edits.
- **`git restore <path>`** — отменить working changes (опасно).
- **`git commit`** — staged snapshot → committed history в `.git`.

## Examples

### Пример 1: Простой lifecycle

`README.md` clean — committed, совпадает с `HEAD`. Редактируешь → **modified** (unstaged). `git add README.md` → **staged**. `git commit -m "Update docs"` → **committed**; при отсутствии новых правок status clean.

```
# Старт: README.md committed (clean)
# 1. Редактируешь README.md           → modified (unstaged)
# 2. git add README.md                → staged
# 3. git commit -m "Update docs"      → committed; working tree clean
```

### Пример 2: Partial commit

Правил `AuthService.cs` (feature) и `appsettings.json` (local dev URL). Stage только auth — feature без config tweak:

```
git add AuthService.cs          # только auth в staging
git commit -m "Add JWT validation"
# appsettings.json всё ещё modified, unstaged — не в commit
```

Ежедневная практика на fullstack, где config и secrets не должны попадать в shared history.

### Пример 3: Один файл, mixed state

Stage `Program.cs`, продолжаешь править до commit. Git ведёт staged content отдельно от новых unstaged строк:

```
git add Program.cs              # stage текущую версию
# снова редактируешь Program.cs → staged portion + новые unstaged edits
git status
# → "Changes to be committed" И "Changes not staged for commit" для Program.cs
```

`git diff` — unstaged; `git diff --staged` — что попадёт в commit при немедленном запуске.

## Common Mistakes & Anti-patterns

**Слепой `git add .`** перед каждым commit — feature, debug logs, config в одной куче. Сначала `git status` и `git diff --staged`.

**Commit = save в редакторе** — save обновляет working tree; commit создаёт permanent labeled history entry с message для других.

**Staging area как optional overhead** — confusion, когда один файл дважды в status. Staging — core Git design.

**Repository = GitHub** — **local `.git`** *есть* repository. GitHub — remote для sharing; commit локально без remote возможен.

**Игнор untracked** — новые файлы вне трёх состояний до первого `git add`. Замечай их до «clean» commit.

## Production & Real-World Notes

Команды ждут **atomic commits** в code review: один PR — одна coherent story, часто из commits с logical steps. Staging — production code без local `.env` experiments; большой refactor — reviewable chunks.

**CI** собирает **committed snapshots**, не unstaged WIP на desk разработчика.

Копирование source без `.git/` **уничтожает history**. Clone, pull, push — не zip без `.git`.

---

## Comparison / Trade-offs

Staging area Git — немного ceremony ради precise control. Некоторые tools commit'ят из working tree — проще сначала, сложнее partial commits и pre-commit review. Git выбрал explicit staging; принять trade-off — часть fluency. Без VCS — нет learning curve, но нет history, collaboration и rollback.

## Quick Reference

```
REPOSITORY     = working tree + .git/
WORKING TREE   = файлы, которые редактируешь
STAGING AREA   = index; preview следующего commit
.git/          = база permanent history

THREE STATES:
  modified  → edited, not staged
  staged    → git add done, ждёт commit
  committed → в .git history

FLOW:
  edit → git add → git commit

WHY STAGING:
  atomic commits | partial commits | review before seal
```

## Key Takeaways

1. Git repo = **working tree** + **`.git` directory`**.
2. Файлы проходят **modified → staged → committed**.
3. **Working directory**, **staging area (index)**, **repository** — three trees Git.
4. **Staging** нужен для atomic commits, partial commits и review перед commit.
5. **Commit** — immutable **snapshot + metadata**, не casual save.
6. **Untracked** файлы вне модели до первого `git add`.
7. Понимание states делает `git status` и `git diff` понятными в `0.1.3`.
