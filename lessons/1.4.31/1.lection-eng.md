# UnobservedTaskException — Global Handler

> Roadmap: `1.4.31` · Node: `1.4` — C# async/await · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Explain when a `Task` exception becomes **unobserved** and how .NET raises **`TaskScheduler.UnobservedTaskException`**.
- Describe **.NET 4.5+** behavior: event fires, process **usually continues**, `SetObserved()` semantics.
- Wire a **global handler** for logging and alerting without masking root causes.
- Connect unobserved faults to **fire-and-forget** (`1.4.30`) and proper **await** discipline.
- Distinguish **`OperationCanceledException`** faults from real errors in background tasks.

---

## Why This Matters

A nightly report job throws in a discarded `Task`. No log entry appears in request middleware. Days later you discover inconsistent data — not a crash, because modern .NET **does not tear down** the process on unobserved exceptions by default. The failure was **silent until something else broke**.

Understanding **`UnobservedTaskException`** is the safety net when code still fire-and-forgets — but the real fix is **observing every task** (await or hosted consumer). Middle developers register a handler for **visibility** and eliminate the class of bugs at the source.

---

## Core Concepts

### Observed vs Unobserved

A `Task` exception is **observed** when code:

- **`await`s** the task (and exception propagates), or
- Calls **`Wait()`**, **`Result`**, **`GetAwaiter().GetResult()`** (sync observation — avoid in ASP.NET), or
- Accesses **`task.Exception`** on a faulted completed task, or
- Calls **`task.GetAwaiter().GetResult()`** after completion.

If a faulted `Task` becomes eligible for **garbage collection** without any of the above, the runtime raises **`UnobservedTaskException`** on the **`TaskScheduler`** (almost always **`ThreadPoolTaskScheduler`**).

### Timeline

```
Task.Run(() => throw new InvalidOperationException());
// no await, no .Exception access
// ... eventually GC collects Task
// → UnobservedTaskException event
```

The delay makes debugging cruel — the stack trace may surface **long after** the originating request.

### .NET 4.5+ Semantics Change

Before .NET 4.5, unobserved task exceptions could **terminate the process** on finalization (similar to legacy unhandled exception policy). Since **.NET 4.5**, the default is:

1. Raise **`UnobservedTaskException`** event.
2. Mark exception **observed** if handler calls **`e.SetObserved()`** (or certain handler behaviors).
3. Process **continues** unless you escalate in the handler.

This reduces crashes but **increases silent failure risk** if nobody logs the event.

### Event Handler Registration

```csharp
TaskScheduler.UnobservedTaskException += (sender, e) =>
{
    e.SetObserved(); // prevent escalation per policy
    foreach (var ex in e.Exception.Flatten().InnerExceptions)
    {
        _logger.LogError(ex, "Unobserved task exception");
    }
};
```

Register **once** at startup (`Program.cs` / host builder), not per request. Use **`ILogger`** via factory if in generic host.

**`e.Exception`** is often **`AggregateException`** — use **`Flatten()`** for inner exceptions.

### Relationship to Fire-and-Forget

Every **`_ = WorkAsync()`** that faults is a candidate for unobserved unless something later observes that **same Task instance**. **`ContinueWith`** that checks `IsFaulted` observes the antecedent when configured correctly — but prefer **`await`** in infrastructure.

## Under the Hood

### Finalization Gate

The CLR tracks whether a faulted `Task` had its exception observed. On finalization (~GC), if still unobserved, it queues the **`UnobservedTaskException`** callback to the thread pool. This is **non-deterministic timing** — high GC pressure may surface bursts of events.

### TaskExceptionHolder

Internally, faulted tasks hold exception state in a holder object until observed or finalized. This keeps the exception alive — **memory leak** if you hoard faulted tasks in a static dictionary without observing.

### async void Exceptions

**`async void`** methods that throw propagate to **`SynchronizationContext`** or as **unhandled exception on thread pool** — a **different path** than `UnobservedTaskException`. Do not conflate; both are bad.

### Hosting and ASP.NET Core

ASP.NET Core request pipeline **does not** catch background task exceptions. **`IExceptionHandler`** / middleware only see exceptions from **awaited** request work. Global **`UnobservedTaskException`** is a **last-resort net**, not request error handling.

## Syntax / Commands / API

| API | Purpose |
|-----|---------|
| `TaskScheduler.UnobservedTaskException` | Global event |
| `UnobservedTaskExceptionEventArgs.Exception` | Usually `AggregateException` |
| `SetObserved()` | Mark handled per runtime policy |
| `task.Exception` | Observes after `IsCompleted` + `IsFaulted` |
| `await task` | Preferred observation |

## Examples

### Fault Becomes Unobserved

```csharp
public IActionResult Trigger()
{
    Task.Run(async () =>
    {
        await Task.Delay(10);
        throw new Exception("lost");
    });
    return Ok();
}
```

No log in middleware. Eventually **`UnobservedTaskException`** fires if registered.

### Proper Observation in Worker

```csharp
protected override async Task ExecuteAsync(CancellationToken stoppingToken)
{
    while (!stoppingToken.IsCancellationRequested)
    {
        var work = await _channel.Reader.ReadAsync(stoppingToken);
        try
        {
            await work(stoppingToken);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Background work failed");
        }
    }
}
```

Exception is **observed** in `catch` — never unobserved.

### Filtering Cancellation

```csharp
TaskScheduler.UnobservedTaskException += (_, e) =>
{
    e.SetObserved();
    var nonCancel = e.Exception.Flatten().InnerExceptions
        .Where(x => x is not OperationCanceledException);
    foreach (var ex in nonCancel)
        _logger.LogError(ex, "Unobserved");
};
```

Cancelled fire-and-forget still bad design, but less alarming in logs.

## Common Mistakes & Anti-patterns

Registering handler **per request** — memory leak and duplicate logs.

Calling **`SetObserved()`** without logging — swallows failures.

Assuming handler **fixes** fire-and-forget — it only reports; scope bugs remain.

Storing **`Task`** references in static collections "for later" without awaiting — delays unobserved until removal.

Using **`Task.WaitAll` without try/catch** on faulted children — observes but can deadlock in ASP.NET (`1.4.14`).

## Production & Real-World Notes

Pair handler with **metrics**: counter `unobserved_task_exceptions_total`.

**Alert** on any non-zero rate in staging — indicates discard patterns.

In **tests**, `TaskScheduler.UnobservedTaskException` can be subscribed to fail the test if unobserved faults occur.

Some teams use **`AppContext`** switches historically; modern apps rely on event + proper await discipline.

Document: **zero tolerance** for `_ =` in code review except vetted queue infrastructure.

## Comparison / Trade-offs

| Strategy | Detects fault | Fixes architecture |
|----------|---------------|-------------------|
| Global `UnobservedTaskException` | Eventually | No |
| `await` in worker | Immediately | Yes |
| `ContinueWith` log | On completion | Partial |
| Message queue + DLQ | On failure | Yes |

## Quick Reference

- Unobserved = faulted `Task` GC'd without observing exception.
- Event: **`TaskScheduler.UnobservedTaskException`** at startup.
- Use **`Flatten()`**, log, **`SetObserved()`** per policy.
- .NET 4.5+: process usually survives — **log anyway**.
- Real fix: **await every task** in hosted consumers (`1.4.30`, `1.4.34`).
- Not a substitute for request **`IExceptionHandler`**.

---

## Key Takeaways

- Discarded faulted tasks surface via **`UnobservedTaskException`**, often late.
- Register a **startup** handler for logging; call **`SetObserved()`** intentionally.
- **`await`** in background services is the correct observation point.
- **`async void`** uses a different exception path — avoid on server.
- Handler is **visibility**, not permission to fire-and-forget scoped work.
- Filter **`OperationCanceledException`** in logs when appropriate.
