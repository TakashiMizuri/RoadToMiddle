# Methods: Parameters, Return, Overloading

> Roadmap: `1.1.12` · Node: `1.1` — C# Language Basics · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Declare **methods** with access modifiers, return types, names, and **parameter lists**.
- Pass **arguments** by value for value types and by reference to object references for reference types.
- Return values with **`return`** and use **`void`** when there is no result.
- Define **overloaded** methods — same name, different signatures — and explain how the compiler resolves calls.
- Write **expression-bodied** methods for simple one-liners and organize console apps into readable static helpers.

---

## Why This Matters

Until now, most samples lived in `Main` — fine for ten lines, unmaintainable for hundreds. **Methods** (functions) bundle behavior behind a name: **`CalculateTax`**, **`ParseMenuChoice`**, **`IsValidEmail`**. They are the primary unit of reuse within a class before you reach classes and interfaces as larger units.

Every ASP.NET endpoint eventually delegates to methods on services. Every unit test calls a method and asserts its return value. Method design — clear names, minimal parameters, sensible return types — is how juniors become middles. **Overloading** lets one intuitive name (`Write`, `Add`, `CreateUser`) work with different input shapes while keeping call sites readable. Misusing overloads (same name, unrelated behaviors) confuses more than it helps. This lesson gives you the C# rules and the taste to split or overload intentionally.

## Core Concepts

### Anatomy of a Method

A method declaration combines:

```csharp
static int Add(int a, int b)
{
    return a + b;
}
```

- **`static`** — call without an instance (common in console `Program` and utility classes).
- **`int`** — **return type** (`void` if nothing returned).
- **`Add`** — **identifier** (PascalCase for methods in C# conventions).
- **`(int a, int b)`** — **parameter list** — comma-separated types and names.
- **Body** `{ }` — statements; **`return`** exits with a value matching the return type.

The **signature** (for overload resolution) includes method name and parameter types (and ref/in/out modifiers later in `1.1.13`) — **not** return type alone.

### Calling a Method

**Call site** passes **arguments** matching parameters:

```csharp
int sum = Add(3, 5);   // arguments 3 and 5
Console.WriteLine(sum);
```

Arguments are evaluated **left to right** before the call. The method receives **copies** of value-type bits; for reference types, the copy points to the same object — mutations visible to caller (more in `1.1.13` on `ref`/`out`).

### Parameters and Arguments

**Parameters** are variables in the method definition; **arguments** are expressions at the call site. Count and types must align (unless optional/default parameters — preview: C# allows **`default` values** in signature; full semantics with `ref`/`out` in `1.1.13`).

```csharp
static void Greet(string name, int times = 1)
{
    for (int i = 0; i < times; i++)
        Console.WriteLine($"Hello, {name}!");
}

Greet("Ada");       // times defaults to 1
Greet("Grace", 3);
```

Optional parameters must come **after** required ones. Callers can omit trailing defaults.

### Return Values and `void`

Value-returning methods **must** return on all paths (compiler enforces **`CS0161`** if a path lacks return):

```csharp
static string Status(bool ok) => ok ? "OK" : "Error";
```

**Expression-bodied members** (`=>`) suit single-expression methods.

**`void`** methods perform actions — logging, printing — without a result. **`return;`** alone exits early (`1.1.11`).

Use **`null`** return for reference types only when callers expect missing data; prefer **`bool TryX(out ...)`** patterns later over magic nulls.

### Method Overloading

**Overloading** — multiple methods in the same scope share a **name** but differ by **signature** (parameter count or types):

```csharp
static int Area(int side) => side * side;

static int Area(int width, int height) => width * height;

static double Area(double radius) => Math.PI * radius * radius;
```

Call **`Area(5)`** → square int. **`Area(4, 6)`** → rectangle. **`Area(2.5)`** → circle. The compiler picks the **best match** at compile time; ambiguous calls fail to compile.

Overloads should represent **the same conceptual operation** on different shapes of data — not unrelated duties sharing a name for brevity.

**Do not overload by return type only** — illegal in C#:

```csharp
// static int Parse(string s) { ... }
// static bool Parse(string s) { ... }  // CS0111 error
```

### Extracting Methods from `Main`

Console apps grow readable when `Main` orchestrates and helpers implement:

```csharp
static void Main()
{
    int[] scores = ReadScores();
    PrintReport(scores);
}

static int[] ReadScores() { /* ... */ return Array.Empty<int>(); }
static void PrintReport(int[] scores) { /* ... */ }
```

Small methods are easier to **test** and **name**. If a method needs more than ~3–4 parameters, consider a small **record** or class to group data (future topics).

### Naming and Single Responsibility

Names are verbs or verb phrases: **`GetUser`**, **`ValidateInput`**, **`ComputeTotal`**. One method — one **logical task**. If you describe a method with "and" (`ParseAndSaveAndEmail`), split it.

Keep methods **pure** when possible: same inputs → same outputs, no hidden console or global mutation — especially for business logic you will unit test.

## Under the Hood

Methods compile to **metadata** rows in the assembly and **IL instructions** with **`call`** / **`callvirt`** opcodes. Overload resolution happens at **compile time** — zero runtime cost for choosing among overloads. Virtual methods (`1.x` later) use **`callvirt`** for polymorphism.

The JIT may **inline** small methods — replace the call with body at the call site — for performance. That is invisible to you but explains why tiny getters/setters are cheap.

**Stack frames:** each call pushes a frame with locals and parameters; **`return`** pops it. Deep recursion can **`StackOverflowException`** — loops or tail recursion patterns matter for algorithm design.

## Syntax / Commands / API

```csharp
[modifiers] ReturnType MethodName(ParamType name, ...) { body }
ReturnType Name(params) => expression;
void NoResult() { return; }
// overload: same MethodName, different parameter list
```

Common modifiers: `static`, `private`, `public` (access modifiers on class members — full coverage when you study classes).

## Examples

### Example 1: Overloaded print helpers

```csharp
static void Print(int value) => Console.WriteLine($"int: {value}");
static void Print(string value) => Console.WriteLine($"string: {value}");
static void Print(decimal value) => Console.WriteLine($"decimal: {value:F2}");

static void Main()
{
    Print(42);
    Print("hello");
    Print(19.99m);
}
```

Three overloads — one name at call site, compiler picks type.

### Example 2: Validation method with early return

```csharp
static bool IsValidAge(int age)
{
    if (age < 0)
        return false;

    if (age > 120)
        return false;

    return true;
}
```

Boolean return replaces printing inside validator — caller decides UI (`1.1.9` branching).

### Example 3: Console menu using methods

```csharp
static int ReadIntPrompt(string message)
{
    while (true)
    {
        Console.Write(message);
        if (int.TryParse(Console.ReadLine(), out int n))
            return n;

        Console.WriteLine("Invalid input.");
    }
}

static void Main()
{
    int a = ReadIntPrompt("First number: ");
    int b = ReadIntPrompt("Second number: ");
    Console.WriteLine($"Sum = {a + b}");
}
```

Reusable **`ReadIntPrompt`** encapsulates loop + parse + continue pattern from prior lessons.

## Common Mistakes & Anti-patterns

**God `Main`** — hundreds of lines; fix by extraction.

**Overload abuse** — `Process(User u)` vs `Process(Order o)` unrelated; use distinct names.

**Side effects in getters** — `GetTotal()` that also saves to database surprises callers.

**Returning `-1` for error** when **`bool TryParse`** style exists — ambiguous valid data.

**Too many optional parameters** — call sites become unclear; use object parameter or builder later.

## Production & Real-World Notes

Public API surfaces (NuGet, REST clients) stabilize method signatures — breaking changes hurt consumers. Overloads extend APIs without breaking existing calls (`Write(string)` + `Write(string, object[])` pattern in `Console`).

Async methods return **`Task`/`Task<T>`** — naming convention **`Async` suffix** (`GetUserAsync`). Same method concepts, different return type story later.

XML doc comments (`/// <summary>`) on public methods generate IntelliSense — adopt early on shared libraries.

## Comparison / Trade-offs

| Choice | Prefer when |
|--------|-------------|
| Single method + params | Truly variable argument count (`params` in `1.1.13`) |
| Overloads | Fixed alternative type combinations, same concept |
| Optional parameters | Sensible defaults, avoid explosion of overloads |
| Separate methods | Different behaviors — don't overload name |

## Key Takeaways

- Methods name behavior; signatures = name + parameter types.
- **`void`** vs value return — choose what callers need.
- **Overloading** resolves at compile time by best match.
- Optional parameters must trail; defaults optional at call site.
- Extract helpers from `Main` for readable console apps.
- Overload same **concept**, not unrelated operations.
