## Answers

### 1. [Recall]
**Answer:** `var` просит компилятор **вывести статический тип** локальной переменной из типа initializer-выражения справа.
**Why:** Inference происходит на compile time; переменная имеет фиксированный тип, как при explicit declaration.

### 2. [Recall]
**Answer:** Нет. **Why:** Без initializer компилятор не знает, какой `TypeSymbol` привязать — CS0818.

### 3. [Recall]
**Answer:** **`double`**. **Why:** Литерал `19.99` без суффикса по правилам `1.1.3` имеет тип `double`; для `decimal` нужен суффикс `m`.

### 4. [Understanding]
**Answer:** Return type `CalculateOrderTotal` не виден локально; explicit `decimal` (или другой domain type) документирует контракт и ожидания (например, точность денег).
**Common mistake:** Считать, что имя метода всегда раскрывает тип.

### 5. [Understanding]
**Answer:** С суффиксом `L` литерал имеет тип `long`, и `var` выведет `long`. Без `L` значение `2_147_483_648` **не помещается в `int`**, компилятор выдаст ошибку переполнения литерала; с меньшими числами без `L` вывелся бы **`int`**, что опасно для больших размеров файлов.

### 6. [Understanding]
**Answer:** Anonymous type **генерируется компилятором**; у него нет имени, которое можно написать в source code. Единственный способ — `var` (или `dynamic`, что не рекомендуется).

### 7. [Application]
**Answer:** A: **`var customers = new List<Customer>();`** — тип obvious из constructor. B: **`double rate = GetInterestRate(accountId);`** (или фактический return type) — explicit, потому что метод может вернуть неочевидный тип.
**Why:** Rubric: RHS `new List<Customer>()` vs скрытый method return.

### 8. [Application]
**Answer:** **`string? name = null;`** — без `var`. **`var name = (string?)null;`** или target context — возможно, но менее идиоматично.
**Why:** `null` не несёт type information для inference.

### 9. [Application]
**Answer:** **`var rate = 0.075m;`**
**Why:** Суффикс `m` задаёт тип литерала `decimal`, inference следует ему.

### 10. [Analysis]
**Answer:** **B** неверно.
**Why:** C# statically typed; после inference тип нельзя «менять совместимым» значением другого типа — только assignment conversion, иначе ошибка компиляции. Runtime flexibility — не про `var`.

### 11. [Analysis]
**Answer:** **`var` не влияет на performance/runtime** — генерируется тот же IL, что и explicit. Это только readability/verbosity.
**Common mistake:** Путать `var` с `dynamic` или оптимизациями runtime.

### 12. [Understanding]
**Answer:** **Obvious:** `var items = new List<Order>();` — тип `List<Order>` уже назван constructor. **Message:** `decimal tax = 0.075m;` — важно подчеркнуть точность и domain type, не только значение.
