# Benchmark: sync vs async HTTP — использование потоков

> Roadmap: `1.4.22` · Node: `1.4` — Async + ASP.NET preview · Depth: практика

## Learning Objectives

После урока ты сможешь:

- Объяснить, **почему** синхронный HTTP под нагрузкой съедает **потоки thread pool**, а async — нет, пока идёт I/O wait.
- Настроить проект **BenchmarkDotNet** для сравнения sync и async HTTP.
- Читать результаты: **mean**, **allocated memory**, показатели **thread pool pressure**.
- Спроектировать сценарий нагрузки, похожий на **ASP.NET Core**.
- Избегать типичных ошибок бенчмаркинга: cold start, без warmup, shared mutable state, localhost vs WAN.

## Why This Matters

В ASP.NET Core каждый входящий HTTP-запрос обрабатывается на **потоке thread pool**. Если action синхронно ждёт downstream API — `GetStringAsync(url).Result` или `.GetAwaiter().GetResult()` — поток **блокируется**, пока не придут байты по сети. При сотнях одновременных запросов накапливаются заблокированные потоки. Thread pool добавляет новые медленно (примерно два в секунду после всплеска), и появляются **очереди**, **скачки latency**, **503** и таймауты.

Async HTTP (`await client.GetStringAsync(url)`) **возвращает поток в pool** на время ожидания socket I/O. Одна машина обслуживает гораздо больше concurrent requests, потому что потоки — дефицитный ресурс, а **ожидание I/O — не CPU-работа**. Middle-разработчик должен **доказать** это цифрами — BenchmarkDotNet и простой load test превращают интуицию в аргумент для code review или postmortem.

Урок связан с `1.4.20` (async all the way) и ведёт к `1.4.26`–`1.4.28` (HttpClient и resilience в production).

## Core Concepts

### Потоки не бесплатны во время I/O wait

При **синхронном** HTTP-вызове вызывающий поток блокируется внутри socket read. Для CLR этот поток не может делать другую работу. Если 500 concurrent requests и каждый блокирует поток на 200 ms — нужно **500 потоков**, которые просто ждут.

**Асинхронный** I/O использует **overlapped I/O** (Windows) и аналоги: поток запускает операцию, регистрирует callback (продолжение через `Task`) и уходит. Когда данные пришли, поток pool (тот же или другой) выполняет continuation. Во время ожидания **ноль** потоков «привязано» к idle-периоду запроса — только state machine и состояние сокета в kernel.

Суть: **async масштабирует concurrent I/O-bound work; sync — только пока хватает потоков**.

### Что мы измеряем (и что нет)

Микробенчмарк одного sync vs одного async на idle-машине почти не покажет разницы в **wall-clock** — оба ждут сеть одинаково. Нужна **concurrency**: много вызовов одновременно или симуляция saturation thread pool.

BenchmarkDotNet даёт **изолированные, повторяемые** измерения. Для HTTP + thread pool комбинируй:

1. **BenchmarkDotNet** — allocation, mean при фиксированном parallelism.
2. **Опционально:** `dotnet-counters` или сэмплы `ThreadPool.GetAvailableThreads` во время spike.

Измеряем **throughput** (ops/sec при N workers) и **доступность thread pool**, не только latency одного запроса.

### Связь с ASP.NET Core

Kestrel async-first. Action `public async Task<IActionResult> Get()` должен `await`-ить downstream. `.Result` внутри action — на каждый blocked thread меньше ресурса для других запросов Kestrel. На scale **sync-over-async в web app** — инцидент в ожидании. Бенчмарк показывает это в console app до staging.

## Under the Hood

Когда `HttpClient.SendAsync` завершается асинхронно, runtime использует **I/O completion ports** (IOCP) на Windows. `Task` переходит в `RanToCompletion` по сигналу IOCP; continuations ставятся в **ThreadPool** (если их не перехватил `SynchronizationContext` — в ASP.NET Core его нет).

Sync `GetString` блокируется в `Socket.Receive`. Stack frame активен; thread pool помечает worker как busy.

BenchmarkDotNet гоняет бенчмарки в отдельном процессе с **JIT warmup** и итерациями против tiered compilation. Для async `[Benchmark]` может возвращать `Task` — BDN await-ит корректно.

## Syntax / Commands / API

```bash
dotnet new console -n HttpSyncAsyncBench
cd HttpSyncAsyncBench
dotnet add package BenchmarkDotNet
dotnet add package Microsoft.Extensions.Http
```

```csharp
[MemoryDiagnoser]
[ThreadingDiagnoser]
public class HttpBenchmarks
{
    private HttpClient _client = null!;
    private const string Url = "https://httpbin.org/delay/1";

    [GlobalSetup]
    public void Setup() => _client = new HttpClient();

    [Benchmark(Baseline = true)]
    public async Task AsyncGet() => await _client.GetStringAsync(Url);

    [Benchmark]
    public string SyncGet() =>
        _client.GetStringAsync(Url).GetAwaiter().GetResult();
}
```

Запуск: `dotnet run -c Release`. Только **Release**.

## Common Mistakes & Anti-patterns

Измерять только один запрос. Использовать `.Result` в console и переносить привычку в ASP.NET. Новый `HttpClient` на итерацию. Debug build. Без warmup.

## Production & Real-World Notes

В production — load tests (k6, NBomber) на staging: **RPS при p95 SLO**. При инциденте — `dotnet-counters` (`threadpool-thread-count`, `threadpool-queue-length`). Fix — убрать sync-over-async, не только «добавить серверов».

## Comparison / Trade-offs

| Подход | Под нагрузкой |
|--------|---------------|
| `await` async HTTP | Низкое потребление потоков |
| `.Result` / `.Wait()` | Thread pool starvation |
| `Task.Run` + sync HTTP | Два потока на вызов — band-aid |
| BenchmarkDotNet | Micro proof | 
| k6 / NBomber | End-to-end SLO |

## Key Takeaways

- Async не ускоряет один вызов — **освобождает потоки** при wait.
- Sync под load → starvation и tail latency.
- Бенчмаркуй **concurrency**, не один call.
- ASP.NET: **await** downstream HTTP.
- Micro-benchmark + load test = production confidence.
