# Exercises: .git/ Internals

> Complete after reading `1.lection-eng.md`, `2.lection-ru.md` and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Map the `.git/` directory
**Goal:** Know where Git stores what.

**Steps:**
1. Create a new repo: `mkdir git-map && cd git-map && git init`
2. Add and commit one file.
3. List `.git/` top level and write a one-line purpose for: `HEAD`, `config`, `index`, `objects/`, `refs/`, `logs/`.

**Success criteria:** 6/6 labels correct; you can point to each without opening docs.

---

### E2. Identify object types
**Goal:** Use `git cat-file` confidently.

**Steps:**
1. In your test repo, run `git rev-parse HEAD` → save as `COMMIT`.
2. Run `git cat-file -p HEAD` and note the `tree` line → save hash as `TREE`.
3. Run `git ls-tree TREE` and pick one blob hash → save as `BLOB`.
4. For each hash, run `git cat-file -t` and record the type.

**Success criteria:** commit → `commit`, tree → `tree`, blob → `blob`.

---

### E3. One blob, two paths
**Goal:** Observe deduplication.

**Steps:**
1. Create `a.txt` and `b.txt` with identical content (`duplicate`).
2. `git add a.txt b.txt`
3. Run `git ls-files -s` — compare blob hashes for both paths.
4. Run `git count-objects` before/after (optional).

**Success criteria:** Same blob hash for both paths; you explain why in one sentence.

## Medium

### M1. Trace `git add` → `git commit`
**Goal:** See objects appear on disk.

**Steps:**
1. Fresh repo with one commit already (from E1).
2. Modify `README.md`, run `git add README.md`.
3. Before commit: `find .git/objects -type f | wc -l` (or count manually on Windows).
4. `git commit -m "Second commit"`
5. After commit: count objects again; run `git cat-file -p HEAD` and `git cat-file -p HEAD^`.

**Success criteria:** You identify new commit hash, parent line pointing to previous commit, and different tree hashes between commits.

---

### M2. Read refs and HEAD
**Goal:** Understand symbolic refs.

**Steps:**
1. `cat .git/HEAD` and `cat .git/refs/heads/main`
2. `git symbolic-ref HEAD`
3. Create branch `feature`: `git checkout -b feature`, make a commit.
4. Compare `refs/heads/main` vs `refs/heads/feature`.
5. `git checkout main` and inspect HEAD again.

**Success criteria:** You explain why HEAD content changes between branches but always starts with `ref:`.

---

### M3. Detached HEAD experiment
**Goal:** Feel the danger of detached state.

**Steps:**
1. Note current `main` tip: `git rev-parse main`
2. `git checkout HEAD~0` or checkout commit hash directly
3. Verify `cat .git/HEAD` has no `ref:` prefix
4. Make a commit with message "orphan experiment"
5. `git checkout main` — then try `git log --oneline orphan`'s hash (use reflog if needed)

**Success criteria:** You recover the "lost" commit via `git reflog` and explain why it wasn't on `main`.

---

## Hard (stretch)

### H1. Manual blob creation
**Goal:** Content-addressable storage hands-on.

**Steps:**
1. In a test repo, run `echo 'manual' | git hash-object -w --stdin`
2. Compare with `git hash-object --stdin` (without `-w`) — same hash?
3. Find the file under `.git/objects/`
4. Stage a new file whose content matches `manual\n` — does `git add` create a duplicate object?

**Success criteria:** Same hash; you locate loose object path; no duplicate on add if content exists.

---

### H2. Pack vs loose awareness
**Goal:** Read `git gc` output.

**Steps:**
1. Create repo with 20+ commits (script loop: echo >> file, git add, git commit).
2. `git count-objects -vH` — note `size`, `in-pack`, `packs`.
3. Run `git gc`
4. `git count-objects -vH` again — what changed?

**Success criteria:** You describe in 3 sentences how loose count dropped and pack size appeared.
