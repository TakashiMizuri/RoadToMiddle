# TCP: three-way handshake, reliability, flow control

> Roadmap: `0.4.3` · Node: `0.4` — Основы сетей · Depth: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Описать **TCP three-way handshake** (SYN, SYN-ACK, ACK) и states каждой стороны.
- Объяснить, как **sequence numbers**, ACK и retransmission дают reliable ordered byte delivery поверх unreliable IP.
- Сравнить гарантии TCP с IP best-effort и поведением UDP (`0.4.4`).
- Объяснить **flow control** (receive window) и back-pressure при медленном consumer.
- Outline **congestion control** (slow start, congestion avoidance) и влияние на API latency под нагрузкой.
- Связать lifecycle TCP с HTTP Keep-Alive и TLS из узла `0.3`.
- Диагностировать SYN timeout, RST, half-open, TIME_WAIT.

## Why This Matters

HTTP/1.1 и HTTP/2 из `0.3` почти всегда на **TCP**. Spinner в React часто — ожидание **TCP + TLS**, не JSON parsing. `0.3.14` (Keep-Alive, pooling) предполагает, что HTTP «connection» — **TCP byte stream**, переиспользуемый между requests; закрытие = новый handshake + возможно новый TLS.

TCP превращает IP datagrams в **reliable ordered pipe** между ports (`0.4.2`). PostgreSQL, Redis, gRPC используют его по той же причине. Middle developer настраивает Kestrel, load balancers, cloud timeouts через TCP: SYN retransmits, idle drops, `connection reset`.

Глубина TCP отличает «сеть медленная» от «SYN queue full», «window zero», «NAT убил idle flow».

## Core Concepts

### Контракт TCP

**TCP** на transport layer (`0.4.1`):

- **Reliable delivery** — потерянные segments retransmit.
- **Ordered delivery** — bytes в порядке отправки у receiver application.
- **Duplex byte stream** — обе стороны шлют после establishment.
- **Flow control** — receiver ограничивает unacknowledged data у sender.
- **Congestion control** — sender адаптирует rate к capacity сети.

TCP не гарантирует latency/bandwidth — только ordered delivery или failure. **Connection-oriented** — state на каждый socket pair.

### Three-way handshake

```
Client                              Server
   | -------- SYN, seq=x ------------> |
   | <---- SYN-ACK, seq=y, ack=x+1 --- |
   | -------- ACK, ack=y+1 -----------> |   ESTABLISHED
   | ===== HTTP / TLS data ==========> |
```

**SYN** — ISN (random для security). **SYN-ACK** — ISN server + ack client+1. **ACK** — подтверждение ISN server.

До ESTABLISHED Kestrel не принимает HTTP/TLS. Fail: **timeout** (нет SYN-ACK) или **ECONNREFUSED** (RST — port closed, `0.4.2`).

Teardown: **FIN** (graceful) или **RST** (abort).

### Sequence numbers и ACK

Каждый byte stream имеет **sequence number**. **ACK** = «получил всё до этого номера».

Sender буферизует unacked segments. **RTO** timeout → retransmit. **Fast retransmit** на duplicate ACKs.

HTTP JSON переживает Wi‑Fi loss на transport layer; HTTP видит только slower response.

### Reliability vs IP и UDP

IP и UDP — best-effort, без order guarantee. TCP — retry + order. DNS часто UDP (`0.4.4`); video/games — UDP, loss acceptable.

### Flow control — receive window

Receiver advertises **window** — сколько bytes готов принять. **Zero window** → sender pauses (**back-pressure**). Защита memory receiver; не путать с congestion control (защита network).

### Congestion control (overview)

При shared links loss и delay сигналят congestion. **Slow start** (exponential) → **congestion avoidance** (linear) → **multiplicative decrease** on loss.

Новые TCP connections под load стартуют осторожно — latency для коротких HTTP без reuse (`0.3.14`). HTTP/3/QUIC (`0.3.13`) — другой stack, та же problem domain.

### TCP и HTTP (узел 0.3)

Timeline HTTPS:

1. TCP handshake (~1 RTT)
2. TLS 1.3 (~1 RTT, `0.3.8`)
3. HTTP on encrypted stream

**Keep-Alive:** один TCP для `GET /a` и `GET /b`. **Pooling** `HttpClient` — reuse к upstream.

**HOL blocking HTTP/1.1:** большой response блокирует следующий на том же TCP — мотивация HTTP/2 (`0.3.13`).

## Under the Hood

### State machine

| State | Meaning |
|-------|---------|
| LISTEN | Server ждёт SYN |
| SYN_SENT / SYN_RECEIVED | Handshake |
| ESTABLISHED | Data |
| FIN_WAIT / CLOSE_WAIT | Close |
| TIME_WAIT | 2×MSL после close |
| CLOSED | — |

**Half-open** — crash без FIN/RST. **Keepalive probes** detect dead peer.

**TIME_WAIT** на server с many outbound closes — exhaustion ephemeral ports; tuning осторожно.

### Segments и reassembly

Sequence, ACK, window, flags, MSS (~1460 on Ethernet). Out-of-order segments в reassembly buffer. Application `read()` — contiguous bytes; HTTP parser видит stream.

**Nagle** coalesces small writes; **TCP_NODELAY** для low latency. **Delayed ACK** — trade latency vs efficiency.

### Retransmission

RTO from RTT (Jacobson/Kare). Timeout → retransmit + backoff. **Fast retransmit** after 3 duplicate ACKs. **Zero window probes** when window reopens.

Wireshark: `tcp.analysis.retransmission`.

### SYN backlog

`listen(backlog)`: **SYN queue** + **accept queue**. Burst/SYN flood → drops до Kestrel. Mitigation: backlog, SYN cookies, LB scale.

### RST

Immediate abort: closed port, firewall, crash, TLS failure after TCP up. «Connection reset by peer» — смотреть upstream, не controller.

## Examples

**SYN timeout** — no SYN-ACK: firewall, wrong IP, routing. **Refused** — RST: port closed.

**Slow API under load** — new TCP per request без Keep-Alive → handshake + slow start каждый раз. Fix: proxy + HttpClient pooling.

---

## Common Mistakes & Anti-patterns

Думать, что TCP = instant delivery. Игнорировать TIME_WAIT в tests. LB idle timeout < client Keep-Alive. Путать TCP connection и HTTP request.

---

## Production & Real-World Notes

LB TCP terminate/passthrough. NAT idle eviction — keepalive. Monitor SYN retransmit, RST, RTT. DB pools — long-lived TCP.

---

## Comparison / Trade-offs

TCP vs UDP — correctness vs timeliness. Keep-Alive vs new connection — latency vs HOL. QUIC — user-space over UDP (`0.3.13`).

## Quick Reference

| Mechanism | Purpose |
|-----------|---------|
| SYN/SYN-ACK/ACK | Establish |
| Seq + ACK | Order + reliability |
| RTO / fast retransmit | Loss recovery |
| Window | Flow control |
| Congestion control | Network load |
| FIN/RST | Close/abort |

---

## Key Takeaways

- Handshake до HTTP/TLS bytes.
- ACK + retransmit + reassembly over IP.
- Window = flow control; congestion = network pace.
- Keep-Alive/pooling amortize handshake.
- Timeout/RST/TIME_WAIT — разные fixes.
- Transport errors до HTTP status codes.
