# Remote: origin, fetch, pull, push

> Roadmap: `0.2.6` · Node: `0.2` — Git: branches and collaboration · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Define a **remote** as a named reference to another Git repository.
- Explain why **`origin`** is the conventional default remote name.
- Distinguish **`git fetch`**, **`git pull`**, and **`git push`** — what each transfers and what each changes locally.
- Describe **remote-tracking branches** (`origin/main`) and how they relate to local branches.
- Trace the flow from local commits on a feature branch to a shared remote.
- Choose the right command when you want to **see** remote changes without merging them yet.

---

## Why This Matters

In node `0.2` you learned that branches are movable pointers to commits, that merge and rebase reshape how those pointers relate, and that conflicts appear when two histories disagree. All of that work happened **inside one repository** — your laptop's `.git` directory. That is enough for solo experiments, but professional fullstack development is collaborative. Your ASP.NET Core API lives on GitHub or GitLab; your teammate's React branch lives on the same server; CI builds from what the remote contains, not from what happens to be unstaged on your machine.

A **remote** is Git's abstraction for "that other copy of the project." It is not magic cloud storage — it is a **URL plus a set of refs** that your local Git knows about. When you `git clone`, Git creates a remote called **`origin`** pointing at the server you cloned from. From that moment, most collaboration vocabulary — "push your branch," "pull latest main," "fetch before rebase" — assumes you understand three verbs: **fetch**, **pull**, and **push**.

Getting these wrong causes real incidents. A developer who always `git pull` without reading the merge output may accidentally create merge commits they did not intend. Someone who `git push` without fetching first may be rejected with "non-fast-forward" — a message that makes sense only if you know remotes advance independently. Someone who confuses fetch with pull may merge remote work they never inspected. Middle-level fluency means knowing **what moves across the network** and **what changes on disk** for each command.

## Core Concepts

### What Is a Remote?

A **remote** is a **nickname for another Git repository**, usually on a network server. Your local repo stores the remote's URL and periodically synchronizes knowledge of its branches and tags. The most common name is **`origin`**, established automatically by `git clone`. You can add more remotes — for example `upstream` when you fork an open-source project — but one remote named `origin` covers the typical team workflow.

List and inspect remotes:

```bash
git remote -v
# origin  https://github.com/team/api.git (fetch)
# origin  https://github.com/team/api.git (push)
```

The same logical remote can have different URLs for fetch and push (uncommon but supported). What matters is that **`origin` is not "GitHub itself"** — it is **your project's configured peer** on that host. Delete the remote configuration and your local history remains; you simply lose the bookmark Git used to talk to the server.

### Local Branches vs Remote-Tracking Branches

In `0.2.1` you learned that a **branch** is a pointer to a commit. Local branches like `feature/login` live under `refs/heads/`. When you **fetch**, Git also updates **remote-tracking branches** under `refs/remotes/` — for example `origin/main`. These are **read-only snapshots** of where branches were on the remote the last time you successfully contacted it. They are not your working branches; you do not commit directly onto `origin/main`.

```
Local repo after fetch:

  refs/heads/main          → your local main (you may be ahead/behind)
  refs/heads/feature/auth  → your feature work

  refs/remotes/origin/main         → where origin's main was at last fetch
  refs/remotes/origin/feature/auth → where origin's feature/auth was
```

This distinction explains `git status` messages like **"Your branch is behind 'origin/main' by 3 commits"**. Git compares your local `main` to your **cached picture** of remote `main`, not to a live network call every time. Until you fetch, that picture may be stale.

### git fetch — Download Without Integrating

**`git fetch`** contacts the remote, downloads any commits and objects you lack, and **updates remote-tracking branches**. It does **not** change your local branches, working tree, or `HEAD`. Think of fetch as **refreshing your map** of the remote world while staying parked where you are.

```bash
git fetch origin
# Updates origin/main, origin/develop, etc.
# Your local main and working files: unchanged
```

After fetch you can inspect:

```bash
git log main..origin/main --oneline   # commits on remote you lack locally
git log origin/main..main --oneline   # commits you have that remote lacks
git diff main origin/main             # file-level diff between tips
```

Fetch is the safe first step before merge, rebase, or a careful push. Teams on busy `main` branches often **`git fetch` several times a day** and integrate deliberately — a habit that pays off before `0.2.7` upstream tracking and `pull --rebase`.

### git pull — Fetch Plus Integrate

**`git pull`** is a **compound command**: **`git fetch`** followed by integration into your **current branch**. The default integration is **merge**, which may create a merge commit if your branch and the remote-tracking branch diverged. Pull therefore **can** change your working branch and working tree — unlike fetch alone.

```bash
git checkout main
git pull origin main
# Equivalent to:
# git fetch origin main
# git merge origin/main
```

Because you studied merge in `0.2.2`, you already know what can happen: fast-forward if you simply lag behind, or a merge commit if both sides added commits, or **conflicts** if the same lines changed — resolved as in `0.2.5`. Pull is convenient but **opaque** when you do not know which half of it ran; many seniors prefer **`git fetch` + explicit `merge` or `rebase`** for clarity. Rebasing on pull is the topic of `0.2.7`.

### git push — Publish Your Commits

**`git push`** uploads local commits to the remote and asks the server to **move its branch pointer** forward. Typically you push a **local branch** to a **branch on origin**:

```bash
git push origin feature/login
```

The remote accepts a push when your commits are a **fast-forward** of what it already has — your history extends theirs. If someone else pushed commits to the same branch since your last fetch, your push is **rejected** as non-fast-forward. That rejection is Git protecting shared history, not a bug. The fix is fetch, integrate (merge or rebase), then push — never blindly `push --force` on shared branches.

Push also **creates** a remote branch if it does not exist yet (when permitted by server policy). After the first push, other developers see `origin/feature/login` after they fetch.

### The origin Convention

**`origin`** is not a Git technical requirement — it is **strong convention** from `git clone`. Tools, tutorials, and CI scripts assume it. Renaming is possible (`git remote rename`), but confusing. When you fork a repo, your clone's `origin` is **your fork**; the source repo is often added as **`upstream`** — a pattern you will use when contributing to libraries your ASP.NET or React app depends on.

## Under the Hood

When you fetch or push, Git uses a **wire protocol** (HTTPS or SSH) to negotiate what the other side has. The client sends **refs** it knows; the server responds with a **packfile** of objects the client is missing. Objects are **content-addressed** (from `0.1.5`), so duplicate blobs across branches transfer once. Fetch stores downloaded objects in your local `.git/objects` and writes remote-tracking ref files such as `.git/refs/remotes/origin/main`.

A push sends **objects the remote lacks** and a request to update a ref — for example "set `refs/heads/feature/login` to SHA `abc123`." The server checks permissions and **non-fast-forward rules**. Hosting platforms may add branch protection: `main` may reject direct pushes and require pull requests (`0.2.8`).

`git pull` triggering merge invokes the same merge machinery as `git merge origin/main` after fetch — including conflict markers in your working tree. `git push` does not send your working tree or staging area — only **committed** history reachable from the pushed branch tip.

## Syntax / Commands / API

```bash
# Remotes
git remote -v
git remote show origin
git remote add upstream https://github.com/org/original.git

# Fetch (safe — no local branch movement)
git fetch                    # all remotes
git fetch origin             # one remote
git fetch origin main        # one branch

# Pull (fetch + integrate current branch)
git pull                     # default remote + upstream branch if configured
git pull origin main         # explicit
git pull --ff-only           # fail instead of merge commit if not fast-forward

# Push
git push origin feature/x    # publish branch
git push -u origin feature/x # push + set upstream (see 0.2.7)
git push origin --delete old-branch

# Inspect without integrating
git log HEAD..origin/main --oneline
git diff origin/main...HEAD   # changes on your branch since divergence
```

## Examples

### Example 1: Daily sync on main

You start the day on `main`:

```bash
git checkout main
git fetch origin
git log --oneline main..origin/main    # anything new on server?
# If commits appear:
git merge origin/main    # or: git pull origin main
```

Fetch-first lets you **read** incoming commits before merging. If CI failed on remote `main`, you see it in log messages before touching your tree.

### Example 2: Publish a feature branch

After commits on local `feature/health`:

```bash
git push -u origin feature/health
```

The remote now has your branch; teammates `git fetch` and see `origin/feature/health`. You open a PR (`0.2.8`) from that branch. Until push, the feature existed only on your machine.

### Example 3: Non-fast-forward rejection

You and a colleague both committed to `feature/shared`:

```bash
git push origin feature/shared
# ! [rejected] ... non-fast-forward
```

Your local picture was stale. Recovery:

```bash
git fetch origin
git log --oneline HEAD..origin/feature/shared   # colleague's commits
git merge origin/feature/shared                 # or rebase — 0.2.4 trade-offs
# resolve conflicts if any — 0.2.5
git push origin feature/shared
```

### Example 4: fetch without panic

Before a risky rebase onto `main`:

```bash
git fetch origin
git rebase origin/main
```

If rebase goes wrong, your old tip remains in reflog; you have not pushed rewritten history yet.

## Common Mistakes & Anti-patterns

**Treating `git pull` as harmless sync** — pull can merge or conflict; always know your current branch. Pulling on `main` while intending to update a feature branch is a classic mistake.

**Never fetching** — you operate on stale `origin/*` refs; status lies gently until push fails.

**Pushing without pulling on shared branches** — leads to rejection loops or, worse, force-push damage.

**Confusing remote-tracking branch with local branch** — checking out `origin/main` puts you in **detached HEAD** (old habit); use `git switch main` and merge/rebase from `origin/main`.

**Assuming push uploads uncommitted work** — only commits move; `git status` dirty tree stays local.

**Using `origin` when you meant `upstream`** — you may open PRs against the wrong repository after a fork.

## Production & Real-World Notes

CI/CD triggers on **remote** events — push to `main`, pull request opened. Your laptop's unpushed commits do not run the pipeline. Branch protection on `main`/`develop` blocks direct push; you push feature branches and merge via PR.

Large teams use **fork workflows** (open source) or **shared repository** workflows (most companies). SSH keys or tokens authenticate push; HTTPS with credential manager is equally common on Windows.

**Partial clone** and **sparse checkout** optimize huge monorepos — remotes still work the same conceptually. **Air-gapped** environments use remotes on internal Git servers with identical fetch/push semantics.

Monitor **how far behind** you are: `git fetch && git status` on long-lived branches prevents monster merges at release time.

## Comparison / Trade-offs

| Command | Network | Updates remote-tracking refs | Changes local branch / working tree |
|---------|---------|------------------------------|-------------------------------------|
| `fetch` | Yes | Yes | No |
| `pull` | Yes | Yes | Yes (merge/rebase) |
| `push` | Yes | No (remote server updates) | No |

**Fetch + manual merge/rebase** vs **pull**: explicit control vs one-liner convenience. **HTTPS** vs **SSH**: credential prompts vs key management — both fine when configured consistently.

## Quick Reference

```
REMOTE        = nickname (usually origin) → URL of another repo
FETCH         = download objects + update origin/* ; no local branch change
PULL          = fetch + merge into current branch (default)
PUSH          = upload commits; move remote branch pointer

origin/main   = remote-tracking branch (last known remote state)
Local main    = your branch; may diverge from origin/main

Before push on shared branch: fetch → integrate → push
Rejected push: fetch → merge/rebase → push (not --force on shared)
```

## Key Takeaways

1. A **remote** is a named peer repository; **`origin`** is the default from `clone`.
2. **Remote-tracking branches** (`origin/main`) cache remote state locally after fetch.
3. **`git fetch`** downloads history without merging into your current branch.
4. **`git pull`** = fetch + integrate; it can merge and conflict.
5. **`git push`** publishes commits; server rejects non-fast-forward when behind.
6. **Fetch before integrate** is the safe professional habit.
7. Uncommitted work never travels by push — only commits do.
