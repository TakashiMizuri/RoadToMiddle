# Func, Action, and Predicate — Built-in Delegates

> Roadmap: `1.3.4` · Node: `1.3` — C# advanced topics · Depth: практика

## Learning Objectives

After this lesson you will be able to:

- Choose **`Action`**, **`Func`**, and **`Predicate<T>`** instead of custom delegate types when appropriate.
- Read and write signatures from **`Action`** (void) through **`Func<T1,...,TResult>`** (return value).
- Use built-in delegates in **LINQ**, **List.FindAll**, **Task.Run**, and **callback parameters**.
- Apply **variance** (`in`/`out`) on `Func`/`Action` in assignments (link to `1.3.2`).
- Avoid pitfalls: **async void Action**, capturing **heavy closures**, **`Predicate<T>`** vs **`Func<T,bool>`**.

---

## Why This Matters

Custom `delegate void Notify(string msg)` made sense in C# 1.0. Today **`Action<string>`** is the lingua franca of callbacks — LINQ **`Where(x => ...)`** expects **`Func<T,bool>`**, **`Task.Run(() => ...)`** takes **`Action`**, ASP.NET **`MapGet`** handlers compile to **`RequestDelegate`** (`Func<HttpContext, Task>`). Naming fewer domain-specific delegate types reduces ceremony; understanding the matrix **`Action` / `Func` / `Predicate`** lets you read any BCL API in seconds.

**Predicate&lt;T&gt;** is legacy sugar for **`Func<T,bool>`** — still appears in **`List<T>.FindAll(Predicate<T>)`**. Middle developers wire **dependency injection** with **`Func<IServiceProvider, T>`** factory delegates and pass **lambdas** (`1.3.6`) everywhere — this lesson is the practical glue after multicast theory (`1.3.3`).

## Core Concepts

### Action — Void Callbacks

**`Action`** — no parameters, void return. **`Action<T>`**, **`Action<T1,T2>`**, … up to **16 type parameters** (BCL limit).

```csharp
Action greet = () => Console.WriteLine("Hi");
Action<string> log = msg => Console.WriteLine(msg);
Action<int, int> sumPrint = (a, b) => Console.WriteLine(a + b);

greet();
log("started");
sumPrint(2, 3);
```

Use when you **perform side effects** and return nothing. Event handlers often match **`Action`** or **`EventHandler`** (special signature with sender + args).

### Func — Returning a Value

**`Func<TResult>`** last type parameter is **return type**. **`Func<T, TResult>`** — one input.

```csharp
Func<int, int, int> add = (a, b) => a + b;
Func<string, bool> isLong = s => s.Length > 10;
int sum = add(3, 4); // 7
```

LINQ **`Select`** → **`Func<T, TResult>`**; **`Where`** → **`Func<T, bool>`** (not Predicate required).

### Predicate&lt;T&gt;

```csharp
Predicate<string> startsWithA = s => s.StartsWith('A');
List<string> filtered = names.FindAll(startsWithA);
```

**`Predicate<T>`** is defined as **`delegate bool Predicate<in T>(T obj)`** — contravariant in `T`. Interchangeable with **`Func<T, bool>`**:

```csharp
Func<string, bool> f = startsWithA; // implicit conversion both ways in many APIs
names.Where(f);
```

Prefer **`Func<T,bool>`** in new code for consistency with LINQ; **`Predicate<T>`** when calling older APIs expecting it explicitly.

### Choosing Among Them

| Need | Type |
|---|---|
| Side effect, no args | `Action` |
| Side effect, with args | `Action<T,...>` |
| Compute result | `Func<..., TResult>` |
| Boolean test (legacy API) | `Predicate<T>` or `Func<T,bool>` |

Custom named delegate when signature carries **domain meaning** repeated across assembly: **`public delegate OrderResult ProcessOrder(Order o);`** — clearer than **`Func<Order, OrderResult>`** in public API surface.

### Common BCL Patterns

```csharp
// LINQ
var adults = users.Where(u => u.Age >= 18).ToList();

// Task
await Task.Run(() => HeavyWork());

// Dictionary factory (lazy)
var cache = new Lazy<Expensive>( () => new Expensive() );

// Callback parameter
void ProcessItems(IEnumerable<Item> items, Action<Item> onEach)
{
    foreach (var item in items) onEach(item);
}
```

**`TryParse` pattern** — **`Func<string, (bool ok, int value)>`** or out params via wrapper.

### Variance Reminder

```csharp
Func<object, string> f = o => o.ToString() ?? "";
Func<string, string> g = f; // contravariant in T, covariant in return
```

## Under the Hood

Built-in **`Action`/`Func`** are **generic delegate types** in **`System` namespace** — same multicast semantics as custom delegates (`1.3.3`). Compiler desugars lambdas and method groups to delegate instances:

```csharp
Action<string> a = Console.WriteLine; // method group conversion
```

**Target**: static method → null target; instance method → `this` captured.

**Generic `Func` arity**: last type param always return; one-type-arg `Func<T>` means **`Func<T, TResult>`** with two params total — easy to miscount.

---

## Syntax / Commands / API

```csharp
Action<T1, T2>           // void(T1, T2)
Func<TResult>            // TResult()
Func<T, TResult>         // TResult(T)
Func<T1, T2, TResult>    // TResult(T1, T2)
Predicate<T>             // bool(T)

// Method group
Func<int, string> f = int.ToString;
```

## Examples

### LINQ Pipeline

```csharp
IEnumerable<Order> recent = orders
    .Where(o => o.CreatedAt > DateTime.UtcNow.AddDays(-7))
    .OrderByDescending(o => o.Total)
    .Take(10);
```

### Injecting Func Factory (DI)

```csharp
public class ReportService
{
    private readonly Func<DateTime, IReport> _reportFactory;
    public ReportService(Func<DateTime, IReport> reportFactory)
        => _reportFactory = reportFactory;

    public IReport ForMonth(DateTime month) => _reportFactory(month);
}
```

### Predicate vs Func

```csharp
bool AllPositive(List<int> nums) =>
    nums.TrueForAll(n => n > 0); // Predicate<int> inferred from lambda
```

## Common Mistakes & Anti-patterns

**`async void` with Action** — `Action` returning void cannot be async properly; use **`Func<Task>`** and await.

**Overusing `Func` in public API** — obscures contract; named delegate or interface sometimes clearer.

**Wrong `Func` arity** — `Func<string, int, bool>` is `(string, int) -> bool`, not `(string) -> int`.

**Allocating new lambda every call** in hot loop — cache delegate or use static lambda (C# 9+).

**Null delegate invoke** — always **`?.Invoke()`** or guard.

## Production & Real-World Notes

**MediatR**, **FluentValidation** use **`Func<,>`** internally sparingly; public APIs prefer interfaces.

**IEnumerable<T>.Where** accepts **`Func<T,int,bool>`** overload with index — second param easy to miss.

Logging **`Action<ILogger, Exception, string>`** patterns in Microsoft.Extensions.Logging factories.

Code review: **`Func<Dependency, Service>`** in DI module — document lifetime implications.

## Comparison / Trade-offs

| | Custom delegate | Action/Func |
|---|---|---|
| Readability in domain | High | Low |
| BCL/LINQ interop | Convert | Native |
| Multicast | Yes | Yes |
| Variance | Delegate rules | Same |

## Quick Reference

| Delegate | Signature shape |
|---|---|
| `Action` | `void()` |
| `Action<T>` | `void(T)` |
| `Func<T>` | `T()` |
| `Func<T,TResult>` | `TResult(T)` |
| `Predicate<T>` | `bool(T)` ≡ `Func<T,bool>` |

Max **16** type parameters for Action/Func (including return for Func).

---

## Key Takeaways

- **`Action`** = void side effects; **`Func`** = return value (last type param).
- **`Predicate<T>`** = **`Func<T,bool>`** for practical use.
- LINQ, Task, callbacks — built-in delegates daily.
- Method group and lambda convert to delegate instances.
- Public APIs: balance **`Func`** convenience vs named types.
