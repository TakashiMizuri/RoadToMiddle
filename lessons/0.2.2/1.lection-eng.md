# Merge: Fast-Forward vs Merge Commit vs Squash

> Roadmap: `0.2.2` · Node: `0.2` — Git: branches and collaboration · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Explain **merge** as combining divergent histories by creating a new commit or moving a ref.
- Distinguish **fast-forward merge**, **three-way merge commit**, and **squash merge** by graph shape and history preservation.
- Identify the **merge base** (common ancestor) and why Git needs three commits to merge.
- Describe what Git writes to the **index** during a merge and when **conflicts** appear.
- Choose an appropriate merge strategy for feature integration vs release hygiene.
- Read `git log --graph` output after each merge type and predict parent pointers.

---

## Why This Matters

In `0.2.1` you learned that branches are refs pointing at commit tips, and that two lines of history **diverge** when each branch receives new commits after sharing an ancestor. At some point the team wants one line again: feature work must land on `main`, hotfixes must incorporate latest `main`, release branches must absorb fixes. **Merge** is Git's primary way to **integrate histories** without throwing away either side.

Not every merge looks the same on the graph. Sometimes `main` simply **slides forward** to catch up with a feature branch — no new commit object, just a ref move. Sometimes Git must build a **merge commit** with two parents, explicitly recording that two lines of development joined. Sometimes the team wants **one clean commit** on `main` even if the feature branch had twenty messy WIP commits — squash merge. Choosing wrongly affects code review archaeology, bisect, revert, and blame. Middle developers must read a merge graph and know which operation produced it.

Merge also touches the same machinery you will reuse in conflicts (`0.2.5`) and contrasts directly with rebase (`0.2.3`–`0.2.4`). Understanding fast-forward vs true merge vs squash is the foundation for every pull request "Merge" button variant on GitHub.

## Core Concepts

### Divergence and the Merge Base

Before merging branch `feature` into `main`, Git finds the **merge base**: the nearest **common ancestor** commit reachable from both tips. With merge base `B`, `main` tip `M`, and `feature` tip `F`, a three-way merge compares **three snapshots** (trees at B, M, and F) to decide the combined result.

```
        B (merge base)
       / \
      M   ... commits ... F
     main              feature
```

If `M` is an ancestor of `F` — `main` has not moved while `feature` advanced — there is no divergence from `main`'s perspective. Git can **fast-forward**. If both moved, Git needs a **merge commit** (unless you rebase first — `0.2.3`).

Finding the merge base uses graph algorithms (Git's `merge-base` command). The base is not always visually obvious in complex graphs with criss-cross merges, but the concept — "last shared commit before forks" — anchors all merge logic.

### Fast-Forward Merge

When the target branch (`main`) has **not diverged** — its tip is an ancestor of the source tip (`feature`) — `git merge feature` while on `main` simply **moves `refs/heads/main` forward** to point at `F`. No new commit object is created. History stays **linear**; it just gets longer.

```
Before:  main → C2, feature → C4 (C2—C3—C4)
After:   main → C4, feature → C4
```

Fast-forward is the default when possible (`git merge feature`). You can forbid it with `git merge --no-ff` if you want an explicit merge commit even when FF is possible — common on `main` to preserve "this PR landed as a unit" in the graph.

Fast-forward preserves every intermediate commit from the feature branch on `main`'s line. That is good for traceability and bisect; it can clutter `main` with "fix typo" WIP commits if the branch was messy.

### Three-Way Merge Commit

When both branches advanced since the merge base, fast-forward is impossible. Git performs a **three-way merge**: it computes a new tree combining changes from both sides since `B`, stages the result, and creates a **merge commit** with **two parents** — one for the previous `main` tip, one for `feature` tip.

```
After merge commit X:
        B
       / \
      M   F
       \ /
        X   ← main (two parents: M and F)
```

The merge commit is a normal commit object (`0.1.4`) with two `parent` lines. `git log --graph` shows a diamond or fork. History is **non-linear** but **complete**: future readers see exactly when integration happened and what each parent line was.

If the same lines were edited differently, Git marks **conflicts** in the index and stops for human resolution (`0.2.5`). A successful merge completes with `git commit` (or auto-commit if no conflicts and no `--no-commit`).

### Squash Merge

**Squash merge** is a workflow variant, not a separate Git object type. Git combines all changes from `feature` since the merge base into **one staged diff** on `main`, then you (or the platform) create **one ordinary single-parent commit** on `main`. The individual commits from `feature` do **not** become part of `main`'s ancestry.

```bash
git switch main
git merge --squash feature
git commit -m "Add login feature (#42)"
```

Graph on `main` shows a single new commit whose tree reflects the net diff of the entire feature branch; `feature`'s commit chain remains on the branch ref but is not linked as parents on `main`. GitHub "Squash and merge" on a PR does this on the server side.

Squash yields a **clean, linear `main` history** at the cost of **per-commit granularity** on the integrated branch. `git bisect` on `main` jumps the whole feature at once; `git blame` on squashed lines points to one commit. Teams often squash noisy PR branches but keep detailed commits on the PR for review.

### Merge vs Rebase Preview

**Merge** preserves the true graph: both parent lines remain visible. **Rebase** (`0.2.3`) replays commits onto a new base, creating a linear history by rewriting commits. Neither is "correct" universally — `0.2.4` compares trade-offs. For now: merge **records integration**; rebase **moves commits** to pretend development happened sequentially.

## Under the Hood

### Algorithm Sketch (Three-Way)

1. Compute merge base `B` = `git merge-base main feature`.
2. Load trees at B, M (HEAD), F (MERGE_HEAD during merge).
3. For each path, combine changes:
   - Changed only on one side → take that change.
   - Changed on both sides identically → same result.
   - Changed differently → **conflict**; write conflict markers in working tree, stage number 1/2/3 in index.
4. If clean, write tree, create commit with parents M and F, move `main` ref to new commit.

During an in-progress merge, `.git/MERGE_HEAD` holds the other tip hash; `.git/MERGE_MSG` prefills the commit message. `git merge --abort` resets to pre-merge state using `ORIG_HEAD`.

### Fast-Forward Is Ref Update Only

When Git detects M is ancestor of F:

```bash
# Effectively:
echo <hash-of-F> > .git/refs/heads/main
# checkout tree of F if not already there
```

No merge commit object; `git log main` includes all feature commits as if they were always on `main`.

### Squash Staging Without Merge Commit

`git merge --squash` computes the combined tree diff vs HEAD but **does not** set MERGE_HEAD for a two-parent commit. It leaves the index and working tree staged for **you** to `git commit` once. The resulting commit has **one parent** (previous main tip) — indistinguishable from a normal commit by parent count, only by content spanning many logical changes.

### Parent Pointers and `git log`

A merge commit with two parents means `git log` default (first parent only) follows `main`'s line; `git log --merges` shows merge commits; `git log --graph --all` reveals full topology. First parent is conventionally the branch you were on when merging (`main`); second is the merged-in branch.

## Syntax / Commands / API

| Operation | Command |
|-----------|---------|
| Merge into current branch | `git merge feature` |
| Force merge commit (no FF) | `git merge --no-ff feature` |
| Squash | `git merge --squash feature` then `git commit` |
| Abort failed merge | `git merge --abort` |
| Find merge base | `git merge-base main feature` |
| Preview conflicts (diff) | `git diff --merge` (during conflict) |
| Complete after resolving | `git add ...` then `git commit` |

**GitHub PR merge buttons:**

| Button | Effect |
|--------|--------|
| Create a merge commit | `--no-ff` style two-parent merge |
| Squash and merge | squash workflow |
| Rebase and merge | rebase then FF (`0.2.3`) |

## Examples

### Example 1: Fast-forward after code review

`main` at C2. `feature` has C3, C4. No other commits on `main`.

```bash
git switch main
git merge feature
# Updating abc1234..def5678
# Fast-forward
```

`main` now points at C4. `git log --oneline` shows C3 and C4 on main's line.

### Example 2: True merge with `--no-ff`

Even when FF possible, team policy requires a merge commit for traceability:

```bash
git switch main
git merge --no-ff feature -m "Merge branch 'feature' — login API"
```

Creates merge commit X with parents (main tip, feature tip). Feature branch ref unchanged; can delete after merge.

### Example 3: Squash for clean main

Feature branch has 15 commits including "wip" and "fix fix". Team wants one commit on `main`:

```bash
git switch main
git merge --squash feature
git status   # all changes staged
git commit -m "feat(auth): add OAuth login (#128)"
```

`main` gains one commit; feature's 15 commits never appear in `main`'s first-parent log.

## Common Mistakes & Anti-patterns

**Merging without updating target branch first.** Stale `main` increases conflict size. Pull/merge `main` into feature before merging to `main`.

**Squash then continuing on same feature branch.** The branch history diverges from `main` in a confusing way; prefer new branch or rebase after squash merge.

**Assuming merge always creates a merge commit.** FF moves ref only — surprises devs expecting a diamond in the graph.

**Using `--no-ff` everywhere.** Clutters graph with empty merge bubbles; use when policy needs explicit PR boundary.

**Forgetting to delete merged branches.** Ref cleanup is hygiene, not automatic on all hosts.

## Production & Real-World Notes

**Protected `main`** requires PR + CI pass before merge. Platform enforces `--no-ff` or squash per repo settings.

**Release branches** often merge with merge commits to preserve release-line topology; hotfixes cherry-pick or merge back (`0.2.11`).

**Monorepos** may use merge queue / batch merges — same Git mechanics at scale.

Document team **merge policy** in CONTRIBUTING.md: FF allowed on dev branches? Squash on `main`? Required `--no-ff`?

## Comparison / Trade-offs

| Strategy | Graph | Pros | Cons |
|----------|-------|------|------|
| Fast-forward | Linear | Simple history, full commits | No merge boundary marker |
| Merge commit | Diamond | Preserves both lines + integration point | Noisy graph |
| Squash | Linear on main | Clean main, one revert unit | Loses commit granularity on main |

## Quick Reference

| Term | Meaning |
|------|---------|
| Merge base | Common ancestor of two tips |
| Fast-forward | Move ref; no merge commit |
| Merge commit | Two parents; three-way merge result |
| Squash merge | One staged commit; feature commits not in main ancestry |
| `--no-ff` | Always create merge commit |

## Key Takeaways

- **Merge** integrates divergent histories; mechanism depends on whether tips have diverged.
- **Fast-forward** only moves the branch ref when the target is a direct ancestor of the source.
- **Three-way merge commit** has two parents and records explicit integration.
- **Squash merge** applies net diff as one single-parent commit on the target branch.
- **Merge base** is the anchor for combining changes since divergence.
- Platform "merge buttons" map directly to these Git strategies.
