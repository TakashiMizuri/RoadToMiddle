# Answers: Custom IEnumerable with yield

## Answers

### 1. [Recall]
**Answer:** Produces next element, saves state, returns control until next `MoveNext`.
**Why:** Cooperative iterator semantics.

### 2. [Recall]
**Answer:** Cold — deferred, re-enumerate re-executes; hot — cached/broadcast (e.g. materialized list).
**Why:** LINQ chains are cold by default.

### 5. [Application]
**Answer:** `IEnumerable` with loop over pages, `yield return` each item; consumer `Take(n)` stops fetching pages early.
**Why:** Lazy pull stops producer.

### 6. [Application]
**Answer:** Да — `Dispose` on enumerator runs `finally`/`using`.
**Why:** foreach guarantees disposal.

### 7. [Analysis]
**Answer:** Небезопасно — shared `_index` mutates from both; use local state in state machine only or separate instances.

### 8. [Analysis]
**Answer:** Iterator returns via `IEnumerator`; method return type is `IEnumerable` — completion via `MoveNext` false, not `return T`.
**Why:** Language iterator model.

### 9. [Understanding]
**Answer:** State machine class with `_state` and fields for locals across yields.
**Why:** Roslyn lowering.

### 10. [Understanding]
**Answer:** `IAsyncEnumerable` pulls async stream item-by-item; `Task<IEnumerable>` waits for entire collection before consumption.
**Why:** Memory and time-to-first-item.

### 11. [Application]
**Answer:** `yield break` ends iteration early inside block; bare `return` ends iterator method without yielding more.
**Why:** Both stop further yields.

### 12. [Analysis]
**Answer:** During enumeration of pipeline — when downstream pulls, upstream runs until next matching item.
**Why:** Deferred LINQ chain.
