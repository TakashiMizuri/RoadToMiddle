# Модель OSI vs модель TCP/IP — четыре уровня

> Roadmap: `0.4.1` · Node: `0.4` — Основы сетей · Depth: понимание

- Сравнить **семиуровневую модель OSI** с **четырёхуровневой TCP/IP** и сопоставить каждый уровень с реальными протоколами.

- Использовать модели как **mental map** при чтении логов, packet capture и сообщений об ошибках в production.

## Why This Matters

В узле `0.3` ты изучал HTTP: methods, status codes, headers, TLS и REST. HTTP — **протокол прикладного уровня (application layer)**: он описывает, как выглядят *request* и *response* в виде текста (или binary в HTTP/2). Но HTTP не знает, как найти сервер в интернете, как пережить потерю пакетов в Wi‑Fi или как превратить биты в сигнал на кабеле. Эти задачи — у **нижних уровней**.

Когда ASP.NET Core API возвращает `502 Bad Gateway`, причина может быть в HTTP (upstream прислал некорректный response), transport (TCP connection reset) или network (неверный маршрут к pod). Без **слоистой mental model** отладка превращается в угадывание. С моделью ты спрашиваешь: *на каком уровне сломалась гарантия и что этот уровень вообще обещает?*

Сетевые инженеры и учебники дают две перекрывающиеся карты: **модель OSI** (7 уровней, педагогика) и **модель TCP/IP** (4 уровня, ближе к тому, как реально вырос интернет). Fullstack-разработчику не нужно зубрить каждый acronym OSI, но **нужна** надёжная картина стека — потому что каждый инструмент (браузер, Kestrel, PostgreSQL client, Docker bridge) живёт на конкретном уровне и общается с уровнем ниже через чёткий interface.

Этот урок — фундамент для `0.4.2`–`0.4.6`: IP addresses, TCP, UDP, DNS и localhost binding живут на определённых уровнях. HTTP из `0.3.14` (Keep-Alive, connection pooling) предполагает **reliable byte stream** от TCP снизу. Понимание, где HTTP заканчивается, а TCP начинается, отличает «API медленный» от «TCP handshake timeout».

## Core Concepts

### Зачем делить на уровни?

Представь web server, который одновременно должен понимать форматы Wi‑Fi frames, timing оптоволокна и JSON request bodies. Любое изменение Wi‑Fi hardware ломало бы JSON parser. **Layering** делит ответственность: каждый уровень даёт **service** уровню выше и использует **service** уровня ниже, не зная внутренних деталей.

Interface между уровнями — контракт. Transport layer обещает application layer: «Дай byte stream на host X port Y — доставлю надёжно (TCP) или быстро без гарантий (UDP).» Application layer не обязан знать, идут ли packets через Ethernet, LTE или VPN tunnel. Эта **abstraction** — почему HTTPS работает дома по Wi‑Fi и в поезде по LTE без изменения React `fetch`.

Уровни позволяют **независимую эволюцию**. IPv6 сменил IPv4 на network layer, не переписывая HTTP. TLS фактически встал между application и transport. Модели — карты, не физические коробки, но без карты легко заблудиться.

### Модель OSI — семь уровней (reference)

**Open Systems Interconnection (OSI)** — стандарт 1980‑х, семь уровней. Имена встретишь в interviews и documentation:

| Уровень | Имя | Протоколы / артефакты |
|---------|-----|------------------------|
| 7 | Application | HTTP, DNS, SMTP, FTP |
| 6 | Presentation | TLS (часто здесь), encoding, compression |
| 5 | Session | Логическое управление сессией (сегодня редко явно) |
| 4 | Transport | TCP, UDP |
| 3 | Network | IP, ICMP, routing |
| 2 | Data Link | Ethernet, Wi‑Fi (802.11), MAC addresses |
| 1 | Physical | Кабели, радио, напряжения |

На практике modern stacks **схлопывают** уровни 5–7 в «application» и часто считают TLS отдельным слоем между application и transport. OSI остаётся полезным vocabulary — «это Layer 4 problem» значит transport — даже если в daily work ты опираешься на четыре уровня TCP/IP.

### Модель TCP/IP — четыре уровня (рабочая модель)

**TCP/IP model** (Internet model) отражает, как строились ARPANET и internet:

```
┌─────────────────────────────────────────┐
│  Application  │  HTTP, DNS, TLS, SSH    │
├─────────────────────────────────────────┤
│  Transport    │  TCP, UDP               │
├─────────────────────────────────────────┤
│  Internet     │  IP (v4/v6), ICMP       │
├─────────────────────────────────────────┤
│  Link         │  Ethernet, Wi‑Fi, ARP    │
└─────────────────────────────────────────┘
```

**Link layer** — двигает frames между узлами *одного* local network segment. Wi‑Fi chip ноутбука говорит на 802.11; switch пересылает Ethernet frames по MAC address. Если destination не в local subnet, frame идёт на **default gateway** (router).

**Internet layer** — маршрутизирует **packets** между сетями по **IP addresses**. IP — *best-effort*: packets могут прийти не по порядку, продублироваться или не прийти. Routing tables и BGP выбирают путь от ISP до cloud region.

**Transport layer** — multiplexing многих приложений на одном host через **ports**. TCP даёт reliable ordered delivery и congestion control; UDP — minimal datagram service. Здесь углубляются `0.4.3` и `0.4.4`.

**Application layer** — протоколы, meaningful для программ и пользователей: HTTP, DNS queries, PostgreSQL wire protocol, gRPC. Kestrel и browser живут здесь.

### Сопоставление OSI и TCP/IP

| TCP/IP (4) | OSI (7) |
|------------|---------|
| Application | Application + Presentation + Session (7, 6, 5) |
| Transport | Transport (4) |
| Internet | Network (3) |
| Link | Data Link + Physical (2, 1) |

«Layer 3 switch» — OSI network layer (IP routing). «Layer 7 firewall» смотрит HTTP Host и path — semantics приложения.

### Encapsulation — поток данных вниз по стеку

Отправка HTTP GET иллюстрирует **encapsulation**. Каждый уровень добавляет свой header (иногда trailer) вокруг payload сверху:

```
Browser (Application):  GET /api/users HTTP/1.1\r\nHost: example.com\r\n...
        ↓ обёрнуто TLS (если HTTPS)
Transport (TCP):        [TCP header | encrypted HTTP bytes]
Internet (IP):          [IP header | TCP segment]
Link (Ethernet/Wi‑Fi):  [Frame header | IP packet | Frame trailer]
        ↓ physical bits
```

На server процесс **decapsulates** вверх: NIC принимает frame → kernel снимает link header → IP module проверяет destination address → TCP собирает segments → TLS decrypts → Kestrel парсит HTTP. Каждый уровень читает только свой header; остальное — opaque payload.

Это стыкуется с `0.3.3`: HTTP start line и headers — application payload для TCP. TCP видит byte stream и не парсит `Content-Type`.

### Где HTTP и TLS относительно узла 0.3

Из node `0.3`:

- **HTTP/1.1** — application layer text protocol; `0.3.14` Keep-Alive переиспользует один **TCP connection** для нескольких HTTP requests (transport concern, влияющий на tuning приложения).
- **HTTPS** — HTTP внутри **TLS**, шифрование до передачи bytes в TCP. TLS handshake (`0.3.8`) после TCP connect, до первого HTTP byte.
- **DNS** (`0.4.5`) — резолвит `example.com` в IP на application layer, но использует **UDP** на transport — история через уровни.

Последовательность для `https://api.example.com/users`:

1. **Application:** DNS lookup → IP address.
2. **Transport:** TCP three-way handshake на IP:443 (`0.4.3`).
3. **Application:** TLS handshake → encrypted channel.
4. **Application:** HTTP request/response (`0.3`).

Сбой на шаге 2 — `connection timed out` или `ECONNREFUSED`, не HTTP status code: HTTP не начался.

## Syntax / Commands / API

Модели концептуальны, но их видно в tools:

```bash
# Application layer — HTTP (из 0.3.12)
curl -v https://example.com/api/health

# Transport — TCP sockets (Windows)
netstat -an | findstr :443

# Internet — маршрут
tracert example.com

# Link / local — ARP cache
arp -a
```

В `curl -v` строка `* Connected to ... port 443` — transport; `> GET / ...` — application; TLS details между ними.

## Examples

### Пример 1: «Connection Refused»

React app вызывает `http://localhost:5000/api/items`. Browser: `ERR_CONNECTION_REFUSED`.

- **Application (HTTP):** не достигнут — HTTP response нет.
- **Transport (TCP):** SYN на port 5000; host отвечает RST (ничего не слушает) или SYN не завершается.
- **Internet (IP):** `127.0.0.1` маршрутизируется locally (`0.4.6`).
- **Fix:** запустить Kestrel на 5000 — binding/host issue, выглядящий как transport.

С layering не начинаешь с JSON serializers.

### Пример 2: 404 vs Unreachable Host

| Симптом | Вероятный уровень | Смысл |
|---------|-------------------|--------|
| HTTP 404 Not Found | Application | TCP+TLS OK; нет matching route |
| `Could not resolve host` | Application (DNS) | Имя не стало IP |
| `Connection timed out` | Transport / Network | Drop или silent firewall |
| `SSL certificate problem` | TLS | TCP OK; trust/identity failed |

## Common Mistakes & Anti-patterns

**Модели как literal hardware stacks.** Уровни логические. TLS может быть в kernel или user space — модель всё равно помогает.

**«Виновата сеть» при application bug.** HTTP 500 — application layer; сеть доставила bytes корректно.

**Путать IP address и port.** IP — **host** (internet layer); port — **process** (transport). `203.0.113.10:5432` — PostgreSQL на машине (`0.4.2`).

**Зубрить OSI без mapping на TCP/IP.** В работе достаточно четырёх уровней плюс «TLS wraps HTTP». OSI numbers — для interview vocabulary.

## Production & Real-World Notes

Load balancers работают на **Layer 4** (forward TCP по IP:port) или **Layer 7** (route по HTTP path, Host, cookies). L4 быстрее; L7 — canary и path routing, но нужен parse HTTP.

Kubernetes **Services** — cluster IP (network) к pod endpoints; **Ingress** — Layer 7 HTTP routing. Security groups: «allow TCP 443» (L4) vs «block `/admin`» (L7 WAF).

При correlation logs выравнивай уровни: APM — HTTP spans; `tcpdump` — packets; не смешивай tools без перевода между уровнями.

## Comparison / Trade-offs

**OSI (7)** — точное учебное vocabulary, сертификации, vendor docs («L7 DDoS»). Может казаться избыточной (session vs presentation редко разделяют).

**TCP/IP (4)** — ближе к Linux kernel (`socket` → TCP/IP → netdevice), истории internet и большинству dev docs.

Для fullstack: **TCP/IP + explicit TLS** как рабочая картина; OSI numbers — при чтении legacy material.

## Quick Reference

| TCP/IP layer | Unit | Key identifiers | Твой stack |
|--------------|------|-----------------|------------|
| Application | Message | URL, DNS name, HTTP method | React fetch, Kestrel, Npgsql |
| Transport | Segment/Datagram | IP + port, TCP/UDP | Connection pooling `0.3.14` |
| Internet | Packet | IP address | Routing, VPN, cloud VPC |
| Link | Frame | MAC address | Wi‑Fi, Ethernet, Docker bridge |

**Encapsulation:** data вниз — headers добавляются; вверх — снимаются.

## Key Takeaways

- Layered models отделяют concerns: HTTP не зависит от Wi‑Fi hardware.
- Четыре уровня TCP/IP — practical map для разработчика.
- OSI (7) — vocabulary; 5–7 часто схлопываются в «application».
- HTTP (`0.3`) — application layer; ниже нужны TCP (обычно) и IP.
- Encapsulation оборачивает upper-layer data в lower-layer headers.
- Debug: какой layer failed — потом меняй application code.
