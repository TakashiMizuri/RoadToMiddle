## Easy

### E1. Manual bisect on 7 commits
**Goal:** Feel binary search shrink range.

**Steps:**
1. Create linear history: `c1`..`c7` on `main` (empty commits OK).
2. Introduce obvious break at `c5` (e.g. break a small script).
3. `git bisect start main c3` (assume c3 good).
4. Count how many `good`/`bad` steps until first bad found.

### E2. bisect reset discipline
**Goal:** Never leave detached HEAD by accident.

**Steps:**
1. Start bisect, one step, stop mid-way.
2. `git status` â€” note detached HEAD.
3. `git bisect reset` â€” confirm back on branch.

## Medium

### M1. bisect run with shell script
**Goal:** Automate classification.

**Steps:**
1. Repo with `check.sh` returning 0 if file `ok.txt` contains `OK`, else 1.
2. History: good commits keep `OK`, bad commit changes to `FAIL`.
3. `git bisect start HEAD <good-hash>`
4. `git bisect run ./check.sh`
5. `git bisect reset`

**Success criteria:** Git prints first bad commit matching your intentional break.

---

### M2. skip untestable middle
**Goal:** Use skip sparingly.

**Steps:**
1. Insert commit that deletes `check.sh` (untestable), surrounded by good/bad known commits.
2. Run bisect; use `skip` when Git lands on broken middle.
3. Document whether Git still found correct first bad.

**Success criteria:** Paragraph on skip ambiguity risk from lecture.
