# Fire-and-Forget Task — Safe Patterns and Traps in Web Apps

> Roadmap: `1.4.30` · Node: `1.4` — C# async/await · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Define **fire-and-forget** in ASP.NET Core and why `_ = SomeAsync()` is dangerous by default.
- Explain **request lifetime**, **scoped DI**, and **exception observability** traps.
- Implement safe alternatives: **`IBackgroundTaskQueue`**, **`BackgroundService`**, **outbox**, **Hangfire**/queue workers.
- Use **`Task.Run`** only when you understand thread-pool and hosting implications.
- Attach **logging, correlation ID, and error handlers** to detached work.

---

## Why This Matters

You return `202 Accepted` and kick off email sending with `_ = SendEmailAsync()` so the user does not wait. Five minutes later, production is down because:

1. **`SendEmailAsync` threw** — nobody awaited, so the exception became **unobserved** (`1.4.31`).
2. The task captured a **scoped `DbContext`** from the HTTP request — the request ended, scope disposed, **`ObjectDisposedException`** races in background.
3. Under load, thousands of detached tasks **starve the thread pool** (`1.4.10`).

Fire-and-forget is not banned — it is **architecturally constrained**. Middle developers choose **explicit background infrastructure** instead of discarding `Task` references.

## Core Concepts

### What Fire-and-Forget Means

**Fire-and-forget** means starting asynchronous work **without awaiting** the returned `Task` in the current flow. Common spellings:

```csharp
_ = ProcessOrderAsync(orderId);           // discard
#pragma warning disable CS4014
ProcessOrderAsync(orderId);               // compiler warning suppressed
#pragma warning restore
Task.Run(() => DoWork());                 // offload + forget
```

The HTTP response may complete **before** the work finishes. That is intentional for **slow side effects** (email, webhooks, analytics). The trap is assuming the runtime will **host, scope, and observe** that work the same way as awaited code — it will not.

### The Three Traps

**1. Unobserved exceptions.** An awaited task surfaces exceptions to the caller. A discarded task surfaces them only when the `Task` is **finalized** or through **`TaskScheduler.UnobservedTaskException`** — too late for per-request handling.

**2. Captive dependencies (scoped services).** ASP.NET Core creates a **DI scope per request**. `DbContext`, `UserManager`, scoped repositories die when the request ends. A fire-and-forget task still running holds references to **disposed** instances.

**3. Wrong lifetime token.** Using **`HttpContext.RequestAborted`** for work that must complete **after** the response guarantees cancellation when the client disconnects — sometimes desired, often not for emails or billing reconciliation.

### async void — Never for Fire-and-Forget

```csharp
private async void Send() { ... }  // NEVER in app code except UI events
```

`async void` cannot be awaited; exceptions crash the **`SynchronizationContext`** or become **unhandled** on the thread pool. Use **`async Task`** and a proper host.

### Task.Run in Web Requests

`Task.Run` queues work to the **thread pool**. In ASP.NET Core it does **not** free the request thread (there is no special request thread — Kestrel uses IOCP). Using `Task.Run` to "make it background" inside a request often **adds** overhead without benefit for I/O-bound work. Reserve it for **CPU-bound** segments you intentionally offload — still await or queue properly.

## Under the Hood

### Request Scope Disposal

When the middleware pipeline completes, **`IServiceScope`** for the request is disposed. Scoped services' `Dispose`/`IAsyncDisposable` runs. Any lingering task accessing `_db.Orders` hits **undefined behavior** — usually `ObjectDisposedException`.

The fix is **`IServiceScopeFactory.CreateScope()`** inside the background worker, resolving **new** scoped services for that unit of work — not reusing the request scope.

### Exception Propagation Path

```
Fire-and-forget Task faults
  → Exception stored on Task object
  → No await → no catch in request pipeline
  → GC finalizes Task (eventually)
  → UnobservedTaskException event (.NET 4.5+ tears down less often, but still logs nowhere by default)
```

Always **`await`** in a hosted loop or attach **`.ContinueWith`** / extension that logs faults — better: use a **queue consumer** that awaits every job.

### Hosting Integration

`IHost` runs **`BackgroundService`** instances with **`CancellationToken stoppingToken`**. Work items should respect shutdown — distinct from per-request tokens. Production apps register **`IHostedService`** or **.NET 6+ `BackgroundService`** for durable background processing.

## Syntax / Commands / API

| Pattern | Safe? | Notes |
|---------|-------|-------|
| `_ = WorkAsync()` in controller | Risky | No exception path; scope issues |
| `BackgroundService` + Channel queue | Preferred | See `1.4.34` |
| `IServiceScopeFactory` in worker | Required | New scope per job |
| `async void` | No | Except UI event handlers |
| Hangfire / Azure Queue / Rabbit | Production | Durability + retry |
| `Task.Run` + forget | Risky | Pool pressure |

## Examples

### Anti-pattern — Scoped DbContext

```csharp
[HttpPost]
public async Task<IActionResult> Create(OrderDto dto, CancellationToken ct)
{
    await _db.Orders.AddAsync(Map(dto), ct);
    await _db.SaveChangesAsync(ct);

    _ = NotifyWarehouseAsync(dto); // uses injected scoped _warehouseService
    return Accepted();
}
```

When the action returns, the request scope disposes `_warehouseService` while `NotifyWarehouseAsync` still runs.

### Safer — Queue Work Item

```csharp
[HttpPost]
public async Task<IActionResult> Create(OrderDto dto, CancellationToken ct)
{
    await _db.Orders.AddAsync(Map(dto), ct);
    await _db.SaveChangesAsync(ct);

    await _queue.QueueBackgroundWorkItemAsync(async sp =>
    {
        var notifier = sp.GetRequiredService<IWarehouseNotifier>();
        await notifier.NotifyAsync(dto.Id);
    });

    return Accepted();
}
```

The queue's **`BackgroundService`** creates a **scope per item** (`1.4.34`).

### Minimal Observation Wrapper (bridge only)

If you must detach temporarily (legacy), at minimum:

```csharp
var task = ProcessAsync(orderId);
_ = task.ContinueWith(t =>
{
    if (t.Exception is not null)
        _logger.LogError(t.Exception.Flatten(), "Background work failed");
}, TaskScheduler.Default);
```

This does **not** fix scope disposal — only **visibility**. Prefer a real queue.

## Common Mistakes & Anti-patterns

Treating **`_ =`** as "run in background" without a **host** — there is no ASP.NET "background request context."

Capturing **`HttpContext`** in static or long-lived tasks — stale user, wrong culture, disposed request.

Using fire-and-forget for **transactional** work that must succeed before returning 200 — return 200 only after commit; queue **after** persistence with idempotent consumer.

Ignoring **backpressure** — unbounded fire-and-forget under spike creates **memory** (queued closures) and **thread-pool** queues.

---

## Production & Real-World Notes

**Outbox pattern**: write "send email" row in same DB transaction as order; worker polls outbox — survives process crash.

**Correlation**: copy **`Activity`** / trace ID into background work for end-to-end logs.

**Idempotency keys** on background handlers — retries are inevitable.

For **IIS in-process** legacy, app pool recycle killed fire-and-forget even harder — another reason for external queues.

## Comparison / Trade-offs

| Approach | Durability | Complexity | Exception handling |
|----------|------------|------------|-------------------|
| `_ = task` | None | Lowest | Poor |
| In-app Channel queue | Process lifetime | Medium | Good if awaited in consumer |
| Message broker | High | Higher | Dead-letter queues |
| Sync await before response | Strong consistency | Low UX latency cost | Best |

## Quick Reference

- Do not fire-and-forget **scoped services** from requests.
- Never **`async void`** for server-side background work.
- Observe tasks: **await in hosted consumer** or explicit fault logging.
- New **`IServiceScope`** per background unit of work.
- Do not use **`RequestAborted`** for post-response work unless intentional.
- Prefer **queue + BackgroundService** over discarded `Task`.

---

## Key Takeaways

- Fire-and-forget sacrifices **exception path** and **DI scope** unless engineered.
- **`_ = SomeAsync()`** in controllers is a code smell — replace with a queue.
- **`IServiceScopeFactory`** creates safe isolated scopes for background jobs.
- **`Task.Run`** is not a web-background primitive for I/O work.
- Production uses **queues, outbox, or workers** with retry and idempotency.
- Correlate and log background failures — silent loss is worse than slow response.
