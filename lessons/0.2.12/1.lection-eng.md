# git bisect — Find the Breaking Commit

> Roadmap: `0.2.12` · Node: `0.2` — Git: Branches and Collaboration · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Explain **binary search** over commit history and why Git bisect is efficient.
- Run **`git bisect start`**, **`bad`**, **`good`**, and **`reset`** workflow manually.
- Use **`git bisect run`** to automate regression search with a test script.
- Interpret **bisect log** and skip commits that cannot be tested.
- Estimate **how many steps** bisect needs for a given number of commits.
- Know when bisect beats **`git log`** guessing and when CI/git blame is enough.

---

## Why This Matters

Tests passed on Friday. On Monday `main` is red — hundreds of commits merged over the weekend. Someone asks: "Which commit broke the build?" Reading every commit message is slow. Checking out random SHAs is guesswork. **`git bisect`** turns the problem into a structured binary search: you mark a known-good commit and a known-bad commit, then Git checks out middle points and asks you (or your script) whether each state is good or bad until only one commit remains — the first bad one.

For a Middle fullstack developer, bisect is the difference between a two-hour archaeology session and a fifteen-minute root-cause story before standup. It connects directly to the commit graph from `0.1.5`, branch tips from `0.2.1`, and the discipline of **reproducible tests** — bisect is only as good as your "good/bad" signal.

## Core Concepts

### The Regression Problem

A **regression** is behavior that used to work and now fails. In Git terms, there exists some commit `G` where the test passes and a later commit `B` on the same ancestry line where it fails. Commits between `G` and `B` may mix good and bad states, but there is a **first bad commit** — the earliest commit in the `G..B` range whose introduction (alone or with its ancestors' context) makes the test fail.

If history is linear from `G` to `B` with `N` commits in between, linear search might require `N` checkouts. **Binary search** halves the remaining range each step: roughly `⌈log₂(N)⌉` tests. With 1000 commits between good and bad, about 10 steps instead of 1000.

Git bisect implements this search. It does not magically know your test — **you** (or a script) classify each checked-out commit as good or bad. Git moves the search boundaries and picks the next midpoint commit to test.

### Manual Bisect Workflow

The classic interactive session:

```bash
git bisect start
git bisect bad                    # current HEAD is bad (or: git bisect bad <commit>)
git bisect good v1.0.0            # this older commit was good
# Git checks out a commit in the middle — build/test
git bisect good                   # or git bisect bad
# repeat until Git prints the first bad commit
git bisect reset                  # return to original branch
```

While bisect runs, you are often in **detached HEAD** (`0.1.11`) — a specific commit, not a branch tip. That is normal. **`git bisect reset`** returns you to the branch you started from and ends bisect mode.

Each `good` or `bad` narrows the interval. Git maintains internally which commits are **skipped**, **untested**, **good**, or **bad**. When one commit remains as the first bad, it prints something like:

```
abc1234 is the first bad commit
```

You can then `git show abc1234` to see the diff that introduced the regression — the starting point for fix or revert (`0.2.11`).

### What "Good" and "Bad" Mean

**Bad** means "the bug / test failure is present." **Good** means "the bug is absent / test passes." Be consistent. If you accidentally mark a bad commit as good, bisect may land on wrong commit or report "no bisectable commits."

The good commit must be an **ancestor** of the bad commit on the path you search. If good and bad are on diverged branches without ancestor relationship, bisect cannot run on that pair — you need merge-base or a linear range on one branch.

**Untestable commits** (broken build, missing dependency): use `git bisect skip`. Skipping too many commits can make bisect unable to find exact first bad commit — Git warns when resolution is ambiguous.

### Automated Bisect: git bisect run

When classification is objective — unit test exit code, lint script — automate:

```bash
git bisect start HEAD v1.0.0
git bisect run npm test -- --grep "AuthService"
```

`git bisect run` executes the command at each step. Exit code **0** means good; **1–124** (except 125) means bad; **125** means skip; **126–127** abort. The script must be deterministic and fast — bisect may run it dozens of times.

This is how CI-like local scripts shine: `./run-smoke-test.sh` that returns 0/1. Keep tests isolated from local uncommitted files — stash or clean before bisect.

### How Many Steps?

If there are `N` commits strictly between good and bad, bisect needs at most about `⌈log₂(N+1)⌉` steps in the ideal case. Example: 500 commits → ~9 steps. That is why binary search scales to busy `main` branches.

Merge commits add complexity — bisect walks the graph but you must ensure good/bad span a coherent ancestry path. On heavily merged histories, prefer marking **good** at last known release tag and **bad** at current `HEAD` on `main`.

## Under the Hood

Git bisect stores state in `.git/BISECT_LOG` and related refs under `.git/refs/bisect/`. It marks commits with terms from graph theory: commits reachable from bad but not from good form the **candidate set**. Each test partitions candidates into "might still be first bad" vs "eliminated."

The algorithm prefers **commits with fewest children** among midpoints to minimize ambiguity on merged graphs — implementation detail in `git-bisect` script and C code. Conceptually it remains binary search on the ordered set of commits topologically between good and bad on the first-parent line when possible.

**First bad commit** is defined as: bad commit `C` such that `C` is bad and its parent `P` (in the bisect walk) is good. That commit's patch is the regression introducer — though sometimes the "real" cause is an interaction; bisect finds the **smallest commit unit** Git can isolate, not always the logical feature boundary.

**Bisect vs blame:** `git blame` finds who last touched a line; bisect finds which **commit** changed behavior for a test. Use blame after bisect identifies the commit.

**Performance:** each step is a checkout — working tree rebuild, maybe `npm install` if lockfile changed. Heavy projects use sparse checkout or `bisect run` with minimal test subset. Some teams bisect on CI with custom pipelines.

## Syntax / Commands / API

```bash
git bisect start [<bad> [<good>...]]
git bisect bad [<rev>]
git bisect good [<rev>]
git bisect skip [<rev>...]
git bisect run <cmd>...
git bisect reset [<commit>]
git bisect log
git bisect replay <logfile>
git bisect visualize   # needs gitk
```

## Examples

### Example 1: Interactive — API started returning 500

Last release tag `v2.4.0` good; `main` bad.

```bash
git fetch --tags
git bisect start main v2.4.0
# detached HEAD at mid commit — run dotnet test
git bisect bad    # if tests fail
# ... repeat ...
# Git: "xyz789 is the first bad commit"
git show xyz789 --stat
git bisect reset
```

### Example 2: Automated

```bash
git bisect start HEAD v2.4.0
git bisect run dotnet test --filter "FullyQualifiedName~LoginTests"
git bisect reset
```

Script returns 0 when login tests pass, non-zero when fail.

### Example 3: Skip broken build

Middle commit does not compile (missing package). `git bisect skip` — Git picks another nearby commit. Use sparingly.

## Common Mistakes & Anti-patterns

**Uncommitted changes during bisect** — checkout may fail or tests see dirty tree. Commit, stash, or clean first.

**Wrong good commit** — not ancestor of bad → bisect error or nonsense.

**Flaky tests** — random pass/fail makes bisect lie. Fix flakiness or run multiple times before marking good/bad.

**Marking "good" because build works but bug still present** — inconsistent signal.

**Forgetting `git bisect reset`** — leaves repo in detached HEAD; confuses next work.

**Slow full e2e in `bisect run`** — 20 steps × 10 min e2e = hours. Use fast unit subset.

## Production & Real-World Notes

Teams tie bisect to **last green CI tag** or nightly build numbers. **Bisect bot** issues: post comment "good=build-4421 bad=build-4499." **Monorepos** (`0.2.15`) may need path-specific test script in `bisect run`.

When first bad commit is a **merge commit**, `git show` shows combined diff — drill into parents with `-m` or bisect first-parent only (`git bisect start --first-parent`).

After identification: fix forward, cherry-pick revert (`0.2.11`), add regression test named after bisect finding.

## Comparison / Trade-offs

| Method | When | Limitation |
|--------|------|------------|
| git bisect | Known good + bad range, reproducible test | Needs clean good/bad signal |
| git log -S | Search code change | Does not prove behavior break |
| git blame | Line authorship | Not causal for regressions |
| CI bisect / history | Already have build artifacts | Infrastructure cost |

Manual bisect teaches the model; **`bisect run`** scales. For 5 commits on a feature branch, reading log may be faster than bisect setup.

## Quick Reference

| Command | Role |
|---------|------|
| `bisect start bad good` | Begin search |
| `bisect good` / `bad` | Narrow range |
| `bisect skip` | Untestable commit |
| `bisect run cmd` | Automate |
| `bisect reset` | Exit bisect, restore branch |

Steps ≈ `log₂(commits between good and bad)`.

---

## Key Takeaways

- Bisect is **binary search** on commit history between known good and bad.
- You supply the **test**; Git supplies **midpoint checkouts**.
- **`git bisect run`** automates with exit codes 0=good, 1-124=bad, 125=skip.
- Always **`bisect reset`** when done.
- First bad commit = earliest bad where parent was good — start of fix/revert.
- Flaky or dirty working tree **breaks** bisect reliability.
