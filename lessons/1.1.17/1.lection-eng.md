# Enums: Flags, Parsing, TryParse

> Roadmap: `1.1.17` · Node: `1.1` — C# Language Basics · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Define an **enum** as a named set of integral constants with a fixed underlying type.
- Apply the **`[Flags]`** attribute and combine flag values with bitwise operators.
- Parse string names into enum values using **`Enum.Parse`** and **`Enum.TryParse`** safely.
- Choose between **`HasFlag`**, **`Enum.IsDefined`**, and manual bitmask checks in production code.
- Predict default values, invalid combinations, and parsing edge cases before they reach production.

---

## Why This Matters

In earlier lessons you met enums as readable alternatives to magic numbers — `OrderStatus.Pending` instead of `0`. That covers the simplest case: one value from a small, mutually exclusive set. Real APIs rarely stop there. HTTP exposes dozens of optional headers. File permissions combine read, write, and execute. ASP.NET Core authorization policies stack requirements. Logging filters enable categories independently.

Each of these domains needs **combinable** values, not a single choice from a list. C# gives you **`[Flags]` enums** for that pattern. Separately, every HTTP query string, configuration file, and JSON payload arrives as **text**. Your application must turn `"Completed"` or `"Read, Write"` into typed enum values without crashing on typos. **`Enum.TryParse`** is the defensive gate between external input and internal logic.

Middle developers use enums daily in DTOs, options classes, and domain models. Getting flags or parsing wrong produces subtle bugs: permissions that silently grant too much access, filters that never match, or exceptions on every malformed request. This lesson connects enum syntax to the bitwise mechanics and parsing APIs you will rely on in ASP.NET Core and EF Core mapping.

## Core Concepts

### Enums as Typed Integral Constants

An **enum** declares a set of named constants backed by an integral type — by default **`int`**, but you can specify `byte`, `short`, `uint`, `long`, and others. At runtime an enum value **is** that underlying integer wearing a type-safe name. The compiler and runtime cooperate so you cannot assign arbitrary integers without an explicit cast, which catches an entire class of mistakes at compile time.

```csharp
public enum OrderStatus
{
    Pending = 0,
    Paid = 1,
    Shipped = 2,
    Cancelled = 3
}
```

When you write `OrderStatus.Paid`, the IL stores the value `1`. Serialization to JSON typically emits the name `"Paid"` (depending on serializer settings) or the number `1`. Database columns often store the integer for compactness. Understanding the underlying number matters when you debug raw SQL or inspect network payloads.

Non-flag enums usually assign **unique sequential values** starting at zero unless you have a reason to align with an external protocol. The default for any enum field you do not initialize is zero — which means the first named member should often be an explicit **`None = 0`** or **`Unknown = 0`** when zero carries semantic meaning beyond "unset."

### Flag Enums and the [Flags] Attribute

A **flags enum** represents a bitmask: each named member occupies a distinct bit (or bit pattern), and multiple members can be active simultaneously. You mark the enum with **`[Flags]`** and assign powers of two so combinations do not collide.

```csharp
[Flags]
public enum FileAccess
{
    None = 0,
    Read = 1,      // 0001
    Write = 2,     // 0010
    Execute = 4,   // 0100
    ReadWrite = Read | Write  // 0011
}
```

The **`[Flags]`** attribute does not change runtime behavior by itself. It tells **`ToString()`** to render combined values as `"Read, Write"` instead of an opaque number like `3`. Tools, debuggers, and documentation generators use the same hint. Without `[Flags]`, `ToString()` on a combined value might print `"3"`, which hides intent.

You combine flags with **bitwise OR** (`|`) to build a set and **bitwise AND** (`&`) to test membership. **XOR** (`^`) toggles a flag; **NOT** (`~`) inverts bits — useful in advanced scenarios but less common in application code.

```csharp
FileAccess access = FileAccess.Read | FileAccess.Write;

bool canRead = (access & FileAccess.Read) == FileAccess.Read;
// or, less efficient but readable:
bool canWrite = access.HasFlag(FileAccess.Write);
```

**`HasFlag`** is convenient but allocates and boxes in some paths; in hot loops prefer `(access & flag) == flag`. For `[Flags]` enums, also define **`None = 0`** explicitly so "no permissions" is clear and tests can assert empty state without magic numbers.

### Designing Flag Values Correctly

The rule of thumb for flags is **one bit per independent flag**, using `1 << n` or explicit powers of two. Composite names like **`ReadWrite`** are optional sugar; they document common combinations but are not required for the bitmask to work.

A common mistake is assigning sequential values `1, 2, 3, 4` on a flags enum without `[Flags]` discipline — then `Read | Write` becomes `3`, which equals whatever member you accidentally assigned `3`. Powers of two prevent that collision. Another mistake is reusing a bit when extending the enum later; reserved bits or versioning strategies belong in API design discussions before shipping v1.

When persisting flags to PostgreSQL, teams store **`int`** or **`bigint`** columns and map in EF Core with the enum type. JSON APIs often serialize as comma-separated strings or arrays; **`System.Text.Json`** can serialize flags as strings if configured — know your contract.

### Parsing Strings into Enums

External data arrives as strings. **`Enum.Parse<T>(string)`** and the non-generic **`Enum.Parse(typeof(T), string)`** convert a name to a value. They throw **`ArgumentException`** when the name is unknown — acceptable for internal constants, dangerous for user input.

```csharp
OrderStatus status = Enum.Parse<OrderStatus>("Paid");
// same as (OrderStatus)Enum.Parse(typeof(OrderStatus), "Paid");
```

**`ignoreCase`** overloads let `"paid"` match **`Paid`**. Default is case-sensitive; HTTP and config sources often vary in casing, so **`ignoreCase: true`** is common in parsers.

**`Enum.TryParse<T>(ReadOnlySpan<char>, out T result)`** returns **`false`** instead of throwing — the pattern you want at API boundaries.

```csharp
if (!Enum.TryParse<OrderStatus>(input, ignoreCase: true, out var status))
{
    return Results.BadRequest($"Unknown status: {input}");
}
```

For flags, **`Enum.TryParse`** accepts comma-separated names when `[Flags]` is present: `"Read, Write"` parses to **`Read | Write`**. Typos in one segment fail the whole parse unless you split and parse manually — a design choice for strict vs lenient APIs.

**`Enum.IsDefined`** checks whether a numeric or named value is a declared member. Important nuance: for flags enums, **`IsDefined`** returns **`true`** only for **exact** named values, not every valid combination. `(FileAccess)3` might be **`Read | Write`** but **`IsDefined(typeof(FileAccess), 3)`** may be **`false`** if `3` is not a named constant. Validate combinations with manual bitmask rules or allow only named composites you define.

### Enum.GetValues and Iteration

**`Enum.GetValues<T>()`** returns every named member — useful for populating dropdowns or OpenAPI enums. Remember that for flags, iterating all **combinations** is exponential; UI usually lists individual flags, not every subset.

## Under the Hood

Enums compile to integral constants. **`OrderStatus.Paid`** in IL is **`ldc.i4.1`** followed by operations that treat the stack value as **`OrderStatus`**. No heap allocation occurs for the enum value itself; it remains a value type on the stack or in a struct field.

**`[Flags]`** is metadata stored on the enum type. Reflection reads **`[FlagsAttribute]`** when formatting or validating. **`HasFlag`** implements roughly `(value & flag) == flag` but accepts **`Enum`** parameters, causing boxing when you pass a struct enum as **`Enum`**.

**`Enum.TryParse`** walks the static field table of the enum type, comparing names (and, for flags, splitting on commas). It does not consult external configuration — only declared members. Performance is fine for request parsing; avoid parsing the same string thousands of times per second without caching.

Default value **`default(OrderStatus)`** is **`0`**, which maps to the member with value zero if one exists; otherwise the numeric zero casts to the enum type without a named counterpart — a source of subtle bugs in deserialization.

## Syntax / Commands / API

```csharp
[Flags]
public enum Permissions : uint
{
    None = 0,
    View = 1 << 0,
    Edit = 1 << 1,
    Delete = 1 << 2
}

// Combine
Permissions p = Permissions.View | Permissions.Edit;

// Test
bool hasEdit = (p & Permissions.Edit) != 0;

// Parse (throws)
var x = Enum.Parse<Permissions>("View, Edit", ignoreCase: true);

// TryParse (safe)
if (Enum.TryParse<Permissions>(text, true, out var perm)) { /* use perm */ }

// Check named member only
Enum.IsDefined(typeof(Permissions), Permissions.View); // true
```

| API | Throws? | Use when |
|-----|---------|----------|
| `Enum.Parse<T>` | Yes | Trusted input, config with fail-fast |
| `Enum.TryParse<T>` | No | HTTP query, user input, JSON |
| `HasFlag` | No | Readable checks; not hot paths |
| `Enum.IsDefined` | No | Exact member validation (not all combos) |

## Examples

### Example 1: ASP.NET Core query parameter

A minimal API receives `?status=Shipped`. You must not throw on `"Shiped"`.

```csharp
app.MapGet("/orders", (string? status) =>
{
    if (status is not null &&
        !Enum.TryParse<OrderStatus>(status, ignoreCase: true, out _))
    {
        return Results.ValidationProblem(new Dictionary<string, string[]>
        {
            ["status"] = [$"Valid values: {string.Join(", ", Enum.GetNames<OrderStatus>())}"]
        });
    }
    // ...
    return Results.Ok();
});
```

The handler rejects unknown strings with **422/400** semantics instead of **500** from an unhandled **`ArgumentException`**.

### Example 2: Combining and checking flags

```csharp
[Flags]
public enum LogLevelFilter
{
    None = 0,
    Information = 1,
    Warning = 2,
    Error = 4
}

LogLevelFilter active = LogLevelFilter.Information | LogLevelFilter.Error;

bool logsErrors = (active & LogLevelFilter.Error) == LogLevelFilter.Error;
```

After combining, **`ToString()`** yields `"Information, Error"` because of **`[Flags]`**.

### Example 3: Strict vs composite parsing

```csharp
Enum.TryParse<FileAccess>("Read, Write", out FileAccess a); // OK → ReadWrite bits

Enum.IsDefined(typeof(FileAccess), a); // true only if `a` matches a single named value's integer
```

When validating API input for flags, document whether clients may send composites, comma lists, or single values — and test all three.

## Common Mistakes & Anti-patterns

**Sequential values on a flags enum** — Using `1, 2, 3` without power-of-two semantics makes **`Read | Write`** collide with another named member. Fix: assign **`1 << n`**.

**Using `Enum.Parse` on user input** — One typo crashes the request pipeline. Fix: **`TryParse`** + problem details response.

**Assuming `IsDefined` validates combinations** — `(Read | Write)` may be valid logically but undefined as a name. Fix: test bits or maintain an allow-list.

**Omitting `None = 0`** — Callers cannot distinguish "unset" from the first flag accidentally. Fix: explicit **`None`**.

**`HasFlag` in tight loops** — Minor allocation/boxing cost adds up. Fix: bitwise **`&`** comparison.

**Casting arbitrary ints from the database** — `(OrderStatus)99` compiles; no runtime check unless you validate. Fix: **`IsDefined`** or constrain DB with CHECK constraints.

## Production & Real-World Notes

Teams often expose enums in OpenAPI as string schemas; **`System.Text.Json`** serializes enum names by default in recent templates. Deserialization uses **`JsonStringEnumConverter`** — internally similar to **`TryParse`**. Keep enum names stable; renaming breaks clients.

For authorization, **`[Flags]`** on **`Claims`** or permission models is common. Unit tests should cover combination parsing and edge cases like empty string, whitespace, and duplicate names in comma lists.

When migrating enum values, never renumber — treat values as part of your persisted contract. Add new members with new bits or new integers; deprecate old names without reusing numbers.

Code analyzers (e.g. **`CA1008`**) warn when **`[Flags]`** enums lack **`None`**. Enable analyzers in CI.

## Comparison / Trade-offs

**Enum vs static readonly int constants** — Enums give type safety and tooling; constants fit when values are not a closed set. Prefer enums for domain states and flags.

**`HasFlag` vs `&`** — Readability vs micro-performance; in ASP.NET request handlers the difference rarely matters; in game loops or serializers it might.

**String enums in API vs numeric** — Strings are self-documenting; numbers are compact and immune to rename. Middle ground: string in JSON, int in database with explicit mapping.

**Manual split-parse vs built-in flags parse** — Built-in is convenient; manual allows custom separators or localized names at the cost of code.

## Quick Reference

```
FLAGS:     [Flags] + powers of two + None = 0
COMBINE:   a | b          TEST: (v & flag) == flag
PARSE:     Enum.Parse     → throws
           Enum.TryParse  → bool + out (use at boundaries)
TOSTRING:  [Flags] → "A, B" ; else often numeric
VALIDATE:  IsDefined → exact named value only
DEFAULT:   0 → first member with value 0 or unnamed zero
```

## Key Takeaways

1. Enums are **integral value types** with named constants — know the underlying type and default zero.
2. **`[Flags]`** enables bitmasks; assign **powers of two** and define **`None = 0`**.
3. Combine with **`|`**; test with **`&`**; prefer that over **`HasFlag`** in hot code.
4. **`Enum.TryParse`** is the safe choice for external strings; **`Parse`** throws.
5. **`IsDefined`** checks named members, not every valid flag combination.
6. **`ToString()`** formatting respects **`[Flags]`** metadata.
7. Persist enum **numeric values** carefully — they are part of your schema contract.
