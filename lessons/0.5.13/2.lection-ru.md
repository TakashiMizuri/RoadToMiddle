# SEO: title, meta description, canonical, Open Graph

> Roadmap: `0.5.13` · Node: `0.5` — HTML · Depth: понимание

## Learning Objectives

После этого урока ты сможешь:

- Объяснить, как поисковики и соцсети **находят и интерпретируют** metadata в HTML.
- Писать осмысленные `<title>` и `<meta name="description">` для страницы.
- Использовать `<link rel="canonical">` для решения проблем дублирующихся URL.
- Добавлять теги **Open Graph**, чтобы ссылки в мессенджерах и соцсетях показывали rich preview.
- Различать, что влияет на **ранжирование**, **snippet** в выдаче и **карточку** при шаринге — а что нет.

---

## Why This Matters

Можно собрать безупречное React-приложение с отличной accessibility — и получить почти нулевой organic traffic, если поисковые системы не понимают, что представляет каждый URL. SEO-metadata живёт в `<head>` HTML-документа — там же, где charset, viewport и ссылки на стили. Это **первый структурированный сигнал**, который crawlers читают до разбора body и выполнения JavaScript.

Для fullstack-разработчика SEO — не только зона маркетинга. Когда ты публикуешь product page, статью или документацию, ты решаешь, что увидят люди в Google и в preview при отправке ссылки в Slack или LinkedIn. Дубли URL из query string, trailing slash или `www` vs без `www` **размывают** ranking, если не указать **canonical**-источник. Open Graph определяет, покажется ли при шаринге картинка с заголовком или голый URL.

В `0.5.11` и `0.5.12` фокус был на людях с assistive technology и keyboard navigation. SEO-metadata дополняет эту работу: понятный `<title>` помогает screen reader пользователям ориентироваться во вкладках, а описательный meta content — всем, кто сканирует выдачу. Позже, в React, metadata задаётся per route — через React Helmet или metadata API фреймворка — но **HTML-контракт**, который ты изучаешь здесь, остаётся тем же.

## Core Concepts

### Как crawlers читают страницу

Когда bot поисковика запрашивает URL, он получает HTML и строит DOM (`0.5.10`). Секция `<head>` обрабатывается рано: там объявлены язык, кодировка, title, связанные ресурсы и machine-readable подсказки. Современные crawlers часто выполняют JavaScript, но **критичная metadata должна быть в начальном HTML-ответе**. Полагаться только на client-side injection — значит рисковать задержкой или пропуском индексации на медленных bots и на fetchers соцсетей, которые не запускают весь app bundle.

Думай о metadata в `<head>` как о **этикетке на коробке**: склад (search index) сортирует посылки по этикетке, прежде чем открывать. Плохая этикетка не всегда блокирует доставку, но снижает приоритет и ясность.

### Элемент `<title>`

У каждого HTML-документа должен быть ровно один `<title>` внутри `<head>`. Title виден в **вкладке браузера**, в **заголовке результата поиска** (обычно — Google может переписать) и в **закладках**. Это один из сильнейших on-page сигналов релевантности, хотя не единственный фактор ranking.

Хороший title **конкретный, читаемый и привязанный к странице**. Для карточки товара `"Wireless Headphones — Acme Shop"` лучше, чем `"Home"` или `"Acme Shop — Best Electronics Online Store Free Shipping Deals"`. Второй вариант — keyword stuffing, его обрежут в SERP. Распространённый паттерн: **Тема — Бренд** или **Страница | Сайт**, примерно до 60 символов.

```html
<head>
  <meta charset="UTF-8">
  <title>PostgreSQL Index Guide — DevDocs</title>
</head>
```

Каждый route в multi-page app или SSR-фреймворке должен отдавать **уникальный title**. Одинаковые title на десятках URL путают crawlers и пользователей.

### Meta description

`<meta name="description" content="...">` даёт **краткое описание** страницы. Google часто использует его как текст snippet под синей ссылкой — но может сгенерировать свой из body, если meta кажется нерелевантным или дублируется. Meta description — **подсказка**, не гарантия.

Пиши description **для людей**, не списком ключевых слов. Одно–два предложения (~150–160 символов): что на странице и зачем кликать. Избегай пустых, одинаковых на всех страницах и misleading clickbait — растёт bounce rate, что косвенно бьёт по сигналам качества.

```html
<meta name="description"
      content="Learn when to use B-tree, GIN, and partial indexes in PostgreSQL, with EXPLAIN examples and common mistakes.">
```

Meta description, как правило, **мало влияет на ranking напрямую**; главная ценность — **CTR** из поиска.

### Canonical URL

Duplicate content — структурная проблема веба. Одна статья доступна по:

- `https://example.com/blog/post`
- `https://example.com/blog/post/`
- `https://www.example.com/blog/post`
- `https://example.com/blog/post?utm_source=twitter`

Для crawlers это могут быть разные страницы с одинаковым текстом. **Link equity** (сигналы от внешних ссылок) распыляется вместо консолидации на одном authoritative URL.

`<link rel="canonical" href="...">` объявляет **предпочтительную версию** текущей страницы. Элемент в `<head>`, href — **абсолютный URL** с `https://`.

```html
<link rel="canonical" href="https://example.com/blog/post">
```

На URL с tracking-параметрами canonical всё равно указывает на «чистый» адрес. Поисковики трактуют canonical как **strong hint**; игнорируют при ошибках (canonical на 404, конфликтующие цепочки). У paginated lists каждая страница обычно canonicalizes to itself — если только ты сознательно не объединяешь серию (зависит от контекста).

Canonical дополняет **HTTP redirect** (`301`), но не заменяет. Redirect — при постоянном переезде URL; canonical — когда дубли остаются доступны, но не должны индексироваться отдельно.

### Open Graph и social preview

Когда ссылку вставляют в Slack, Discord, Facebook или LinkedIn, платформа делает GET и ищет **Open Graph** (`og:`) meta tags. Twitter/X исторически использовал `twitter:`, но при отсутствии fallback на OG.

Основные свойства:

- `og:title` — заголовок preview.
- `og:description` — текст preview.
- `og:image` — абсолютный URL картинки (часто 1200×630 px).
- `og:url` — canonical URL контента.
- `og:type` — например `website`, `article`.

```html
<meta property="og:title" content="PostgreSQL Index Guide">
<meta property="og:description"
      content="B-tree vs GIN, partial indexes, and EXPLAIN — practical guide for backend developers.">
<meta property="og:image" content="https://example.com/assets/og/postgres-indexes.png">
<meta property="og:url" content="https://example.com/blog/postgres-indexes">
<meta property="og:type" content="article">
```

Обрати внимание: `property="og:..."`, не `name="..."` — по конвенции RDFa. `name` на `og:` снижает совместимость с частью parsers.

Social crawlers **часто не выполняют JavaScript**. Если React inject OG только после hydration, preview покажет дефолты сайта. SSR, static generation или pre-render для публичных страниц — production necessity, не luxury.

### Robots и индексация (кратко)

`<meta name="robots" content="noindex">` — не индексировать эту страницу. Canonical — «индексируй **тот** URL вместо этого». Sitemap XML — список URL для discovery. В этом уроке фокус на title, description, canonical и OG как everyday toolkit fullstack-разработчика.

## Under the Hood

Googlebot строит DOM и извлекает metadata в внутреннее представление. Title и description участвуют в **snippet generation** — алгоритмах, которые могут переписывать title под запрос. Canonical сверяется с sitemap, internal links и redirect chains; inconsistent signals ведут к **canonical clustering**, когда Google выбирает не тот URL, который ты хотел.

OG fetchers делают HTTP GET с bot user-agent, часто парсят только `<head>`, и кешируют preview. `og:image` должен быть **публично доступен** без auth cookie и отдавать корректный `Content-Type`. Относительные пути ломают preview на многих платформах.

У SPA **two-wave indexing** Google (HTML сначала, JS потом) означает: metadata в начальном ответе индексируется раньше. Social bots вторую волну могут не пройти. Отсюда SSR metadata для marketing pages — не опциональная оптимизация.

## Syntax / Commands / API

| Элемент / атрибут | Назначение |
|---|---|
| `<title>` | Title документа; вкладка, кандидат в SERP headline |
| `<meta name="description" content="...">` | Подсказка для snippet в поиске |
| `<link rel="canonical" href="absolute-url">` | Preferred URL среди дублей |
| `<meta property="og:title" ...>` | Social preview title |
| `<meta property="og:description" ...>` | Social preview description |
| `<meta property="og:image" ...>` | Social preview image |
| `<meta property="og:url" ...>` | Canonical social URL |

Минимальный SEO-блок в `<head>`:

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>About Us — Acme Corp</title>
  <meta name="description"
        content="Acme Corp builds developer tools since 2010. Meet the team and our open-source mission.">
  <link rel="canonical" href="https://acme.example/about">
  <meta property="og:title" content="About Acme Corp">
  <meta property="og:description"
        content="Meet the team behind developer tools used by 50,000 engineers.">
  <meta property="og:url" content="https://acme.example/about">
  <meta property="og:type" content="website">
  <meta property="og:image" content="https://acme.example/og/about.png">
</head>
```

## Examples

### Статья блога с og:type article

```html
<title>Why Partial Indexes Beat Full Scans — Data Blog</title>
<meta name="description"
      content="Partial indexes reduce size and speed up queries when you filter on a predictable subset. Walkthrough with SQL examples.">
<link rel="canonical" href="https://blog.example/posts/partial-indexes">
<meta property="og:type" content="article">
<meta property="og:title" content="Why Partial Indexes Beat Full Scans">
<meta property="og:url" content="https://blog.example/posts/partial-indexes">
<meta property="og:image" content="https://blog.example/og/partial-indexes.png">
```

При шаринге с `?ref=newsletter` canonical и `og:url` остаются на чистом path.

### Product page e-commerce

```html
<title>Model X Wireless Mouse — PeripheralShop</title>
<meta name="description"
      content="Ergonomic wireless mouse, 2.4 GHz + Bluetooth, 70-day battery. Free returns.">
<link rel="canonical" href="https://shop.example/products/model-x-mouse">
```

Не копируй manufacturer boilerplate на сотни SKU — поисковики детектят **duplicate snippets**.

## Common Mistakes & Anti-patterns

**Одинаковые title и description на каждой странице** — типичный дефолт SPA без per-route настройки.

**Отсутствующий или relative canonical** — crawlers ожидают absolute URL.

**Keyword stuffing** — rewrites и низкий CTR.

**Маленькая или заблокированная og:image** — серый placeholder; проверяй через debug tools платформ.

**Конфликт canonical и redirect** — аудит после миграций.

**Client-only meta для публичных страниц** — пустые OG и слабый SEO.

## Production & Real-World Notes

Команды централизуют SEO templates в SSR: Next.js `metadata`, Remix `meta`, ASP.NET Razor `@section Head`. Marketing задаёт copy guidelines; инженеры — per-route injection и **environment-aware absolute URLs** (staging не должен попадать в production canonical).

Search Console показывает **duplicate without user-selected canonical**. После смены домена: `301`, canonical, sitemap. Для i18n рядом часто идут `hreflang` link elements. Structured data (JSON-LD) — отдельный слой поверх честного `<head>`.

## Comparison / Trade-offs

**Title vs og:title** — могут отличаться: search vs social контекст. На маленьких сайтах дублирование ок.

**Meta description vs og:description** — выравнивание снижает maintenance.

**Canonical vs 301** — canonical при analytics params; redirect когда старый URL должен исчезнуть.

**SSR vs prerender vs client meta** — SSR для динамики; prerender для docs; client-only ok за login, где SEO не нужен.

## Key Takeaways

- Metadata в `<head>` объясняет crawlers и соцсетям **смысл URL** до глубокого разбора контента.
- У каждой публичной страницы — **уникальный** `<title>`; meta description улучшает snippet и CTR.
- **Canonical** консолидирует дубли; absolute URL, согласованность с redirects.
- **Open Graph** — social preview; absolute `og:image`, server-rendered HTML.
- Metadata — **система подсказок**; качество контента важнее keyword tricks.
- В React/SPA metadata per route — осознанная задача, не побочный эффект render.
