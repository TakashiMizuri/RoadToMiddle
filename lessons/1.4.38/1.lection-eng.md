## Why This Matters

Interviewers use async-in-ctor questions because they expose whether you understand **object construction invariants**, **threading**, and **Task composition** ‚Äî not just syntax. Juniors propose `public async MyService()` or `public async string Data => await Load();`. Both are illegal or dangerous. Seniors explain **why** the CLR must return a fully constructed reference from `new` synchronously, and propose **`CreateAsync`**, **factory**, or **explicit `InitializeAsync`** called from composition root.

In production, violations appear as **random nulls** (field not loaded yet), **deadlocks** (`ctor` calls `.Wait()` on async init), and **unobserved exceptions** from `_ = LoadAsync()` in ctor. ASP.NET Core DI resolves **synchronously** ‚Äî you cannot `await` inside standard constructor injection. Middle developers design services that are **sync-constructible** and **async-initializable** at the right boundary (middleware, `IHostedService`, endpoint filter).

## Core Concepts

### Why Constructors Cannot Be Async

A constructor's job: allocate object, run base ctor chain, initialize fields, return **a valid reference** to the caller. The caller expects:

```csharp
var svc = new MyService(); // must complete before next line
```

`async` methods return **`Task`**, not `MyService`. There is no `async constructor` syntax in C#. If you write:

```csharp
public MyService() => InitializeAsync(); // fire-and-forget
```

you get a **started but unconsumed** `Task` ‚Äî ctor returns before init finishes. **Invariant violated:** object looks ready, but `_cache` is empty.

**Compiler rule:** only methods returning `void`, `Task`, `Task<T>`, `ValueTask`, etc. can be `async`. Constructors return the constructed type instance through a special CLR path ‚Äî not compatible with async state machine return.

### The `.Wait()` / `.Result` Constructor Trap

Legacy fix:

```csharp
public MyService(IApiClient api)
{
    _data = api.FetchConfigAsync().Result; // BLOCKS
}
```

Problems:

- **Deadlock** if `FetchConfigAsync` needs captured `SynchronizationContext` (less common in ASP.NET Core, still possible in libraries).
- **Thread pool starvation** under load ‚Äî many ctors block threads during resolve.
- **Hidden sync-over-async** ‚Äî violates async-all-the-way (`1.4.42`).

Interview answer: "Works sometimes, fails under load or in UI ‚Äî wrong default."

### Async Factory Pattern

Static factory returns `Task<T>` after async work **before** exposing instance to consumers who need ready state:

```csharp
public sealed class ConfigService
{
    private ConfigService(Config config) => _config = config;

    public static async Task<ConfigService> CreateAsync(IApiClient api, CancellationToken ct)
    {
        var config = await api.FetchConfigAsync(ct);
        return new ConfigService(config);
    }
}
```

Caller:

```csharp
var svc = await ConfigService.CreateAsync(api, ct);
```

Object is **fully initialized** before use. DI registration becomes async at startup:

```csharp
builder.Services.AddSingleton(await ConfigService.CreateAsync(api, ct));
```

Only valid at **composition root** during host build ‚Äî not per-request.

### IAsyncInitializer / Explicit InitializeAsync

When framework requires parameterless ctor (some serializers, ORMs):

```csharp
public interface IAsyncInitializable
{
    Task InitializeAsync(CancellationToken ct);
}
```

Host calls `InitializeAsync` once after resolve:

```csharp
foreach (var init in app.Services.GetServices<IAsyncInitializable>())
    await init.InitializeAsync(ct);
```

**Risk:** consumers may use service before init if not enforced ‚Äî document ordering or use **lazy gate**.

### Async Property Getters

C# does **not** allow `async` property accessors:

```csharp
// ILLEGAL
public async string Name => await _repo.GetNameAsync();
```

Properties are syntactic sugar for **`get`/`set` methods** that must return synchronously (except `async` is forbidden on accessors). Interview trap: "make an async property" ‚Äî correct answer: **method** `GetNameAsync()` or **cached** sync property backed by loaded state.

**Lazy async via `.Result` in getter** ‚Äî worse:

```csharp
public string Name => _name ??= _repo.GetNameAsync().Result;
```

Blocks on every first access; deadlock risk; terrible in lists (N blocking calls).

**Acceptable pattern:** sync property on **preloaded** data:

```csharp
public string Name { get; private set; } // set in InitializeAsync
```

Or expose `Task<string> GetNameAsync()` explicitly.

### Half-Initialized Objects and Races

Fire-and-forget in ctor:

```csharp
public MyService() { _ = WarmCacheAsync(); }
```

Two requests may hit before warm completes ‚Äî **race**. Tests may pass (single-threaded), production flakes.

**readonly fields** cannot be assigned in async continuation after ctor ends without unsafe patterns ‚Äî another reason init belongs **before** publish.

## Under the Hood

**Object allocation** (`newobj` IL) runs ctor synchronously on the calling thread. The GC can see the object reference only after ctor returns (unless `this` escapes early ‚Äî another anti-pattern). An async state machine would need to **return before ctor completes** to await ‚Äî impossible while preserving CLR ctor contract.

**DI containers** (Microsoft.Extensions.DependencyInjection) resolve with **synchronous reflection** `Activator.CreateInstance`. No built-in `await ResolveAsync<T>()` in default container (.NET 9 still sync resolve). Async init must happen **outside** standard ctor injection or via factory delegate registered manually.

**Property getters** compile to `get_Name()` methods ‚Äî must return `T` synchronously. `async Task<T>` would change method signature; language designers rejected async properties to avoid confusing **sync-looking** access (`obj.Name`) with **hidden suspension**.

## Syntax / Commands / API

| Pattern | Valid? |
|---------|--------|
| `async ctor` | No |
| `CreateAsync` static factory | Yes |
| `InitializeAsync()` after `new` | Yes (with ordering) |
| `async Task<T> GetXAsync()` | Yes (not property) |
| `async property get` | Compile error |
| `.Result` in ctor | Legal C#, design error |

## Examples

### DI-friendly design

```csharp
public sealed class ReportService(IConfigCache cache)
{
    // cache populated by IHostedService at startup
    public async Task<Report> BuildAsync(CancellationToken ct) =>
        await GenerateAsync(cache.Current, ct);
}
```

Startup hosted service:

```csharp
public sealed class ConfigWarmupService(IConfigCache cache, IApiClient api) : IHostedService
{
    public async Task StartAsync(CancellationToken ct) =>
        await cache.LoadAsync(api, ct);
    public Task StopAsync(CancellationToken ct) => Task.CompletedTask;
}
```

No async ctor; cache ready before Kestrel accepts traffic (if hosted service registered and awaited at startup).

### Interview one-liner

> "You cannot await in a constructor because construction must complete synchronously with a valid object reference. Use `CreateAsync`, hosted startup initialization, or lazy async methods ‚Äî never async properties or `.Wait()` in ctor."

## Common Mistakes & Anti-patterns

**`InitializeAsync().Wait()` in ctor** ‚Äî deadlock + blocked thread pool.

**Async lazy property** ‚Äî hidden I/O on every first access in hot paths.

**Assuming DI will call `InitializeAsync`** ‚Äî unless you wire it, nobody will.

**Two-phase init without enforcement** ‚Äî public methods callable before init.

**Testing ctor with `.Result`** ‚Äî tests pass, production deadlocks.

## Production & Real-World Notes

**IHostedService** and **BackgroundService** are the ASP.NET composition-root place for async warmup. **Health checks** should report "not ready" until init completes (Kubernetes readiness probe).

Some libraries expose `BuildServiceProvider()` anti-pattern with sync init ‚Äî prefer `Host.RunAsync` pipeline.

For **Blazor / MAUI**, `SynchronizationContext` makes ctor `.Wait()` especially deadly ‚Äî same guidance applies.

## Comparison / Trade-offs

| Approach | When | Risk |
|----------|------|------|
| `CreateAsync` factory | Immutable service needs remote config once | Async only at startup registration |
| `IHostedService` warmup | Shared cache, indexes | Ordering vs first request |
| Lazy `GetXAsync()` | On-demand load | Caller must await |
| `.Wait()` in ctor | Never recommended | Deadlock, starvation |
| Async property | Not allowed | ‚Äî |

## Key Takeaways

- Constructors must finish **synchronously** with valid reference ‚Äî no `async ctor`.
- **Async property getters** are illegal; use **async methods**.
- **CreateAsync** and **hosted warmup** are production patterns.
- Fire-and-forget in ctor ‚Üí **races** and **unobserved exceptions**.
- **`.Wait()` in ctor** is interview red flag and production liability.
- Default DI resolves **sync** ‚Äî design services accordingly.
