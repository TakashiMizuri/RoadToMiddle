## Why This Matters

`1.1.6` placed local variables in **stack frames** and object bodies on the heap. That diagram was correct for typical methods — but *what is* a stack frame, how deep can the stack go, and what happens when recursion never stops?

Production incidents include **`StackOverflowException`** from runaway recursion, **`Insufficient stack`** in deep pipelines, and confusing stack traces where line numbers depend on PDB metadata. GC and exception handling **scan stack frames** to find live references (`1.5.6`) — frame layout is not an abstract detail; it is correctness-critical runtime infrastructure.

Middle developers read stack traces, set thread stack limits in containers, and avoid enormous **value type** locals that bloat every frame.

## Core Concepts

### The Call Stack per Thread

Each .NET thread has a **private native stack** — contiguous virtual memory growing downward (on x64 typically). Each **method invocation** creates a **stack frame** (activation record):

```
High addresses
┌─────────────────────┐
│  Main() frame       │  locals, return to OS
├─────────────────────┤
│  Handle() frame     │  params, locals, return → Main
├─────────────────────┤
│  Process() frame    │  return → Handle
├─────────────────────┤
│  (current SP)       │ ◄── stack pointer
└─────────────────────┘
Low addresses
```

When `Process` returns, its frame is **popped** (stack pointer restored); memory is logically abandoned — no GC for stack slots; references in frames are **GC roots** while active (`1.5.6`).

### Frame Contents

A frame typically holds:

- **Return address** — where to continue after callee returns.
- **Parameters** — passed per calling convention (registers + stack spill).
- **Local variables** — `int count`, reference slots for `string s`, struct copies inline.
- **Saved callee-saved registers** — ABI requirement.
- **Frame pointer chain** (optional) — enables stack walking for debuggers and `StackTrace`.

The JIT chooses exact layout; **GC maps** describe which slots contain managed pointers at each instruction point for conservative/exact stack scanning during GC.

### Link to 1.1.6

From `1.1.6`:

- **Value type locals** (`int`, `decimal`, large `struct`) occupy bytes **inside the frame**.
- **Reference type locals** hold a **pointer-sized reference** in the frame; the **object** is on the heap.

Assignment copies bits (value) or reference (ref type) within or between frames. When a method returns, its locals die — except references to heap objects that remain reachable from caller frames or static roots.

### Stack Overflow

**StackOverflowException** occurs when the thread stack exceeds its limit — commonly:

1. **Unbounded recursion** — each call adds a frame until exhaustion.
2. **Very deep legitimate recursion** — tree depth without tail optimization.
3. **Large stack allocations** — `stackalloc byte[1_000_000]` in a loop on same thread.

```csharp
void Boom() => Boom(); // StackOverflowException
```

Unlike most exceptions, **`StackOverflowException` is often fatal** — the runtime may not recover enough stack to dispatch catch reliably. Fix the algorithm, don't try to catch routinely.

Default stack size ~1 MB on Windows threads; **Linux/containers** may differ — deep stacks in microservices need explicit configuration.

### Stack Traces and Exceptions

When you `throw`, the CLR **walks frames** using metadata and unwind info, building `StackTrace` entries (method names from metadata `1.5.3`, line numbers from **PDB**). `finally` runs during unwind; `catch` matches exception type per EH tables from IL.

Async/await complicates logical stacks — state machine types appear in traces (`1.4.x` async topics); physical frames differ from logical call graph.

## Under the Hood

### x64 Calling Convention and Prologue

JIT emits **prologue**: adjust `rsp`, save registers, allocate local space. **Epilogue** reverses on return. **Tail call** optimization (when enabled) reuses caller frame for callee — recursion can become loop at Tier 2 — not guaranteed for all C# patterns.

**Stack walking** during GC: for each frame, use **GC info** bit map to find object references → treat as **roots**. This is why **unmanaged pointers** hiding references break GC if not reported.

### Stack vs IL Evaluation Stack

IL instructions use a virtual **evaluation stack** during JIT compilation — not the same as the native call stack. One IL method compiles to native code using CPU stack/registers; confusion between the two causes wrong answers in interviews and exams.

### Stack Reserve and Commit

Windows reserves stack virtual space; pages **commit** on demand. Overflow hits guard page → `StackOverflowException`. **Edit and Continue**, **debugging**, and **instrumentation** can alter frame sizes slightly.

## Examples

### Recursion Depth

```csharp
int Depth(int n) => n == 0 ? 0 : 1 + Depth(n - 1);
// Depth(100_000) may overflow; iterative version safe
```

Fibonacci naive recursion — exponential frames **and** time — fix with iteration or memoization.

### Large Struct Local

```csharp
struct Big { public byte[256] Data; } // hypothetical large struct
void M() {
    Big b = default; // 256 bytes in THIS frame — every call
}
```

Many deep calls with huge struct locals multiply stack pressure — prefer heap (`class` or array) or ref parameters when appropriate.

## Common Mistakes & Anti-patterns

**Catching StackOverflowException to recover** — unreliable; fix recursion.

**"Stack holds all objects"** — only frames hold locals and references; object **bodies** on heap (`1.1.6`).

**Unbounded stackalloc** from user input size — stack overflow or security risk.

**Ignoring async stack traces** — physical vs logical; use `ExceptionDispatchInfo` or logging libraries that capture async context.

## Production & Real-World Notes

**Container stack limits** — Kubernetes default thread stack rarely issue unless native interop or deep XML recursion; still test edge cases.

**Library code** — XPath, JSON deserializers with depth limits prevent stack blow-up from malicious input.

**Diagnostics** — `dotnet-dump` and SOS `!clrstack` show managed frames atop native stack.

## Key Takeaways

- Each method call pushes a **stack frame** with locals and return linkage.
- **`1.1.6` locals** live in frames; **objects** on heap; frames hold references.
- **Stack overflow** = too many/large frames — often recursion.
- **Exceptions and GC** walk frames using JIT metadata.
- **IL evaluation stack ≠ native call stack**.
