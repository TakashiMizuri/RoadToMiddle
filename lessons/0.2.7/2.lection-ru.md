# Upstream tracking, push -u, pull --rebase

> Roadmap: `0.2.7` · Node: `0.2` — Git: ветки и collaboration · Depth: практика

## Learning Objectives

После этой лекции ты сможешь:

- Объяснить конфигурацию **upstream** (tracking) branch и где Git её хранит.
- Использовать **`git push -u`** для публикации ветки и настройки upstream за один шаг.
- Выполнять **`git pull --rebase`** для интеграции remote changes с linear history.
- Интерпретировать вывод **`git status`** про ahead/behind относительно upstream.
- Настроить **`branch.*.merge`** и **`branch.*.remote`** (концептуально) через `-u` и `git branch -u`.
- Решить, когда **rebase-on-pull** уместен vs merge-on-pull, опираясь на `0.2.4`.

---

## Why This Matters

В `0.2.6` ты узнал fetch, pull и push — глаголы, переносящие commits между ноутбуком и `origin`. Каждый день ты вводишь `git pull` и `git push` на feature branch, не указывая каждый раз `origin feature/login`. Это удобство зависит от **upstream tracking**: локальная ветка помнит, какой remote branch ей соответствует.

Без upstream Git не скажет **«Your branch is ahead of 'origin/feature/x' by 2 commits»** — бесценно перед открытием pull request. Без upstream голые `git pull` и `git push` падают или требуют лишних аргументов. Без team convention для **`pull --rebase`** интеграция свежего `main` в feature branch осыпает history merge commits — `git log --oneline` шумный, `git bisect` сложнее (`0.2.3`, `0.2.4`).

**`git push -u`** при первом push — привычка на секунду, экономящая недели friction. Выбор **rebase при pull** (или при обновлении ветки на `main`) держит feature history читаемой для reviewers в `0.2.8`. Middle-разработчик настраивает defaults один раз и понимает, что происходит, когда rebase переписывает commits, уже бывшие на remote.

## Core Concepts

### Что такое Upstream Tracking?

**Upstream branch** — **default remote branch**, спаренная с локальной. Git хранит это в config:

```ini
[branch "feature/login"]
    remote = origin
    merge = refs/heads/feature/login
```

При настроенном tracking **`origin/feature/login`** (remote-tracking) — upstream локальной **`feature/login`**. Команды в короткой форме:

```bash
git pull          # pull с upstream текущей ветки
git push          # push в upstream текущей ветки
git status        # ahead/behind vs upstream
```

Просмотр:

```bash
git branch -vv
# * feature/login  abc1234 [origin/feature/login: ahead 2] Add JWT middleware
```

`[origin/feature/login: ahead 2]` — связь upstream плюс сводка расхождения.

### git push -u — опубликовать и связать

**`-u`** (или **`--set-upstream`**) при первом push делает две вещи: загружает commits **и** пишет tracking config, чтобы будущие `git push` / `git pull` знали куда идти.

```bash
git push -u origin feature/login
```

Эквивалент push плюс `git branch -u origin/feature/login`.

Если remote branch не существовала, `-u` создаёт её (если policy разрешает) и связывает. Стандартный паттерн first push после `0.2.6`.

**Уже пушил без `-u`?** Исправление:

```bash
git branch -u origin/feature/login
```

### git pull --rebase — интеграция без merge commits

Обычный **`git pull`** merge'ит remote changes в текущую ветку — нормально на `main` у некоторых команд, часто шумно на **feature branches**. **`git pull --rebase`** делает fetch, затем **rebase** локальных commits поверх обновлённого upstream tip — как будто начал с latest remote state.

```bash
git checkout feature/login
git pull --rebase origin feature/login
# или с upstream:
git pull --rebase
```

Механика (`0.2.3`): Git временно снимает уникальные commits, fast-forward к upstream, накатывает commits по одному. Конфликты возможны на каждом — решаются как в `0.2.5`, затем `git rebase --continue`.

**Обновление feature свежим main** — родственный паттерн:

```bash
git fetch origin
git rebase origin/main        # на feature/login
```

Большинство команд **rebase feature на `origin/main`** перед PR, а не меняют upstream feature на `main`.

### Ahead и Behind

После `git fetch` status сравнивает tip локальной ветки с upstream:

- **Ahead** — есть commits не на remote (нужен `git push`).
- **Behind** — на remote есть commits, которых нет локально (`git pull` / `pull --rebase`).
- **Diverged** — оба (integrate, затем push).

Diverged часто после amend или rebase **после** предыдущего push — правила rewrite из `0.2.4` и `0.1.10`; может понадобиться `push --force-with-lease` только на **своей** feature branch.

### Настройка Defaults

Глобальное поведение pull (policy команды различается):

```bash
git config --global pull.rebase true
```

Многие команды используют **`pull.rebase true`** для разработчиков и защищают `main` required linear history или squash merges (`0.2.8`). Узнай правило команды до предположений.

## Under the Hood

Upstream живёт в **`.git/config`** в `[branch "name"]`. `git push -u` обновляет это после успешного push. `git branch -vv` читает `branch.*.merge` и резолвит remote-tracking ref через refspec `remote.*.fetch`.

`git pull --rebase` вызывает `git fetch`, затем `git rebase` на `@{upstream}`. Rebase создаёт новые commit objects (новые SHA) для replayed commits; старые tips в reflog.

`git status` ahead/behind считает commits, достижимые из одного ref, но не из другого — сравнение графа, без сети.

## Syntax / Commands / API

```bash
git push -u origin feature/x
git branch -u origin/feature/x
git branch --unset-upstream

git pull --rebase
git fetch origin && git rebase origin/main

git branch -vv
git rev-parse --abbrev-ref @{upstream}

git rebase --abort | --continue | --skip
git push --force-with-lease
```

## Examples

### Пример 1: Первый push новой feature

```bash
git switch -c feature/health
git push -u origin feature/health
git status
# up to date with 'origin/feature/health'
```

### Пример 2: Daily sync на tracked feature

Коллега запушил в ту же ветку:

```bash
git fetch origin
git pull --rebase
git push
```

### Пример 3: Rebase на main перед PR

```bash
git fetch origin
git rebase origin/main
git push --force-with-lease    # если feature уже была на remote
```

### Пример 4: Забыл -u

```bash
git push                     # fatal: no upstream
git branch -u origin/feature/x
git push                     # ok
```

## Common Mistakes & Anti-patterns

**Не настраивать upstream** — бесконечный copy-paste `git push origin long/branch/name`.

**`pull --rebase` на shared main** без согласования с командой.

**Rebase после того, как другие стянули твою ветку** — переписывает public commits; координация или merge (`0.2.4`).

**Слепой `push --force`** — **`--force-with-lease`** на solo feature branches.

**Думать, что `git pull` без args тянет `main`** — тянет **upstream текущей ветки**.

## Production & Real-World Notes

Hosting UI показывает ahead/behind на PR branches. CI может требовать branch **up to date with `main`** — локальный rebase или merge main в feature.

Кнопка **«Update branch»** на GitHub merge'ит или rebases через API — знай поведение платформы. Документируй в **CONTRIBUTING.md**: first push `-u`, sync `pull --rebase`, pre-PR `rebase origin/main`.

## Comparison / Trade-offs

| Подход | Форма history | Когда |
|--------|---------------|-------|
| `git pull` (merge) | Merge commits | Shared branches, merge policy |
| `git pull --rebase` | Linear | Feature branches, pre-PR |
| `fetch` + manual rebase | Полный контроль | Перед рискованной интеграцией |

## Quick Reference

```
UPSTREAM     = default remote branch (config)
git push -u  = push + set upstream
git pull --rebase = fetch + rebase на upstream

ahead  → push
behind → pull --rebase (часто на feature)

Pre-PR: git fetch && git rebase origin/main
После rebase pushed branch: push --force-with-lease
```

## Key Takeaways

1. **Upstream** связывает локальную ветку с `origin/...`.
2. **`git push -u`** при first push — стандарт.
3. **`git pull --rebase`** держит feature history linear.
4. **`git branch -vv`** показывает tracking и ahead/behind.
5. Rebase на **`origin/main`** перед PR — common pattern.
6. После rebase pushed branch — **`--force-with-lease`**.
7. Согласуй `pull.rebase` с командой (`0.2.4`).
