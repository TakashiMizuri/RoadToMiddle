# Exercises: Object.Equals, GetHashCode, IEquatable

> Complete after reading `1.lection-eng.md`, `2.lection-ru.md` and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Fix the broken key
**Goal:** Make `Dictionary<SkuKey,int>` work with value-equal keys.
**Steps:**
1. Define `class SkuKey { public string Code { get; } ... }`
2. Override `Equals` so same `Code` means equal.
3. Add matching `GetHashCode`.
4. In `Main`, `d[new SkuKey("A")] = 1;` then assert `d.ContainsKey(new SkuKey("a"))` — decide ordinal vs ignore case explicitly.
**Success criteria:** `ContainsKey` returns expected result; explain hash contract in one sentence.

### E2. Record vs class
**Goal:** Compare boilerplate for equality.
**Steps:**
1. Create `record ProductDto(Guid Id, string Name)`.
2. Create equivalent `sealed class` with manual `IEquatable<ProductDto>`.
3. List which members you had to write manually on the class.
**Success criteria:** Class compiles; written comparison of line count and maintenance cost.

## Medium

### M1. Money value object
**Goal:** Implement correct equality for `Money(decimal Amount, string Currency)`.
**Steps:**
1. `readonly record struct Money` OR sealed class — justify choice.
2. Equality: same amount **and** currency (ordinal).
3. Use as `HashSet<Money>` key; add `(10, "USD")` twice — count 1.
4. Optional: overload `==` if you used class.
**Success criteria:** Set deduplicates; `GetHashCode` uses `HashCode.Combine`.

### M2. IEquatable on struct
**Goal:** Measure awareness of boxing (conceptual).
**Steps:**
1. `struct EntityId { public Guid Value; }` — implement `IEquatable<EntityId>`.
2. Write `bool Equals(object? obj)` delegating to typed method.
3. Explain in comments when `List<EntityId>.Contains` avoids boxing.
**Success criteria:** Compiles; comment demonstrates understanding.

## Hard (stretch)

### H1. Custom comparer
**Goal:** Case-insensitive email set without wrapping `string`.
**Steps:**
1. Implement `IEqualityComparer<string>` (or use BCL) for emails: trim + ordinal ignore case.
2. `HashSet<string>` with that comparer.
3. Add `"User@Mail.com"` and test `"user@mail.com"`.
**Success criteria:** Single entry; document why normalization belongs in comparer vs in type.

## Submission checklist

- [ ] Every `Equals` override has `GetHashCode`
- [ ] No mutable dictionary keys in your designs
- [ ] Explained one bug scenario from the lecture in your own words

## Hints (read only if stuck)

<details>
<summary>Hint E1</summary>
`GetHashCode()` should derive from the same field(s) as `Equals` — typically `Code.GetHashCode()` with chosen `StringComparison` policy reflected in both methods.
</details>

<details>
<summary>Hint M1</summary>
`readonly record struct` is ideal for small immutable values; `HashCode.Combine(Amount, Currency)` pairs with structural `Equals`.
</details>

<details>
<summary>Hint H1</summary>
`StringComparer.OrdinalIgnoreCase` implements `IEqualityComparer<string>` — you can use it directly before writing your own.
</details>
