# IEnumerable vs IQueryable — The Fundamental Difference

> Roadmap: `1.3.11` · Node: `1.3` — C#: advanced topics · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Contrast **`IEnumerable<T>`** (in-memory enumeration) with **`IQueryable<T>`** (queryable expression tree + provider).
- Explain why **`AsEnumerable()`** and **`AsQueryable()`** change **where** code executes.
- Describe the role of **`IQueryProvider`** and **`Expression`** in deferred remote execution.
- Identify **client evaluation** vs **server translation** pitfalls (EF Core preview).
- Choose **`IEnumerable`** vs **`IQueryable`** for repository/API method signatures.
- Recognize when LINQ operators accept **`Func<>`** (local) vs **`Expression<Func<>>`** (translatable).
- Apply **`AsNoTracking`**-level mental model: query describes work; execution is separate (`1.3.9`–`1.3.10`).

---

## Why This Matters

You have written `.Where().Select().OrderBy()` on lists — **`IEnumerable`**, delegates, CLR loops (`1.3.7`–`1.3.10`). Entity Framework exposes **`DbSet<T>`** implementing **`IQueryable<T>`**. The **identical syntax** runs in PostgreSQL, not in your Kestrel process.

Confusing the two causes the most expensive LINQ mistakes: fetch entire tables then filter in C#; call **`ToList()`** too early; pass **`IQueryable`** outside **`DbContext`** lifetime; use non-translatable C# in **`Where`**. Middle backend developers must feel the boundary in their bones before EF chapters.

This lesson defines the **fundamental difference**. **`1.3.12`** opens expression trees and provider mechanics.

## Core Concepts

### Two Interfaces, Same Shape, Different Engine

```csharp
IEnumerable<User> e = usersList.Where(u => u.Age > 18);     // Func<User,bool> — runs in process
IQueryable<User> q = db.Users.Where(u => u.Age > 18);         // Expression<Func<User,bool>> — translated
```

Both appear **deferred** until enumeration/materialization. Difference is **what gets stored** in the pipeline:

| | `IEnumerable<T>` | `IQueryable<T>` |
|--|------------------|-----------------|
| Extends | `IEnumerable<T>` | `IEnumerable<T>` |
| Expression | No (delegates) | **`Expression`** tree per operator |
| Provider | **`Enumerable`** (LINQ to Objects) | **`IQueryProvider`** (EF, SQL, etc.) |
| Execution | CLR iterator | Provider-defined (SQL) |
| Typical source | Arrays, lists, in-memory | `DbSet`, remote stores |

**`IQueryable<T>`** extends **`IEnumerable<T>`** — every **`IQueryable`** is enumerable, but not vice versa.

### IEnumerable — LINQ to Objects

Operators take **`Func<T, bool>`**, **`Func<T, TResult>`** — compiled .NET delegates executed locally.

```csharp
List<Product> products = GetProducts();
var cheap = products.Where(p => p.Price < 10);  // Func<Product,bool>
```

When you **`foreach`**, each product is tested in your application's memory. **`ComplexLocalMethod(p)`** inside **`Where`** is fine — it's just C# code.

### IQueryable — Remote Query Description

**`IQueryable<T>`** exposes:

```csharp
Expression Expression { get; }   // tree representing the query
IQueryProvider Provider { get; }
Type ElementType { get; }
```

Each **`Where`/`Select`** on **`IQueryable`** does not filter rows in CLR. It **appends nodes** to the **`Expression`** tree. **`IQueryProvider.Execute`** (triggered by **`GetEnumerator`**, **`ToList`**, etc.) asks the provider (EF Core) to translate the tree to SQL and run it.

Lambdas in **`IQueryable`** operators are **`Expression<Func<...>>`**, not **`Func<...>`** — the compiler builds expression trees (`1.3.12`).

### AsEnumerable() — Force Client Evaluation

```csharp
var query = db.Orders
    .Where(o => o.Status == Status.Shipped)   // SQL
    .AsEnumerable()                           // switch to IEnumerable
    .Where(o => ComplexLocalLogic(o));        // CLR — all shipped rows loaded!
```

**`AsEnumerable()`** (or **`ToList()`**) is the **boundary** between server and client. Everything **after** runs in memory on whatever was yielded before the switch.

Use intentionally: rare algorithms impossible in SQL; small intermediate sets; composing with external data.

**Anti-pattern:** **`AsEnumerable()`** early to "fix" untranslatable **`Where`** — loads too much data.

### AsQueryable() — Usually Misunderstood

**`AsQueryable()`** on in-memory **`List<T>`** wraps with **`EnumerableQuery<T>`** — **`IQueryable`** interface over **local** execution. Useful for testing query providers or unified APIs; **does not** send LINQ to SQL.

```csharp
List<int> nums = new() { 1, 2, 3 };
IQueryable<int> q = nums.AsQueryable();
// Still LINQ to Objects — provider executes in memory
```

Do not confuse with EF's **`DbSet.AsQueryable()`** which participates in EF provider.

### Func vs Expression<Func> — Compiler Chooses by Target

```csharp
IEnumerable<T> e = source.Where(x => x.Id > 1);    // Func — if source is IEnumerable
IQueryable<T> q = source.Where(x => x.Id > 1);       // Expression — if source is IQueryable
```

Same syntax, different overload resolution. If you assign to **`IEnumerable`** from **`IQueryable`** without materializing, you may accidentally pull **`AsEnumerable()`** behavior depending on overload — watch assignments:

```csharp
IEnumerable<User> users = db.Users.Where(u => u.Active);
// May still be IQueryable underneath until cast/enum — prefer explicit types
```

### Repository Signatures — Middle Guidance

**Return `IQueryable<T>`** from repositories only when:

- Caller composes further filters in the same **`DbContext`** scope.
- Team accepts coupling to EF (leaky abstraction debate).

**Return `IEnumerable<T>`** or **`List<T>`** / **`Task<List<T>>`** when:

- Query is complete; materialization intentional.
- Hiding persistence technology.

**Never** return **`IQueryable`** from controllers to clients.

### EF Core: Client vs Server Evaluation (Preview)

EF Core translates most **`Where`/`Select`/`OrderBy`/`Join`** to SQL. Methods it cannot translate force **client evaluation** (older EF) or **throw** at runtime (EF Core 3+ strict mode for many patterns).

```csharp
// Often fails or splits query — local method not in SQL
db.Users.Where(u => MyHelper(u.Email));
```

Fix: translatable expression, **`EF.Functions`**, raw SQL, or fetch minimal data then **`AsEnumerable()`** consciously.

Logging **`ToQueryString()`** (EF Core 5+) shows generated SQL — essential debugging skill for later phases.

## Under the Hood

### Type Hierarchy

```
IEnumerable<T>
    ↑
IQueryable<T>  (+ Expression, Provider)
```

**`Queryable`** static class holds extension methods for **`IQueryable`**, parallel to **`Enumerable`** for **`IEnumerable`**.

### Execution Path Comparison

**IEnumerable:**

```
foreach → enumerator.MoveNext → Where delegate → Select delegate → element
```

**IQueryable (EF):**

```
ToListAsync → provider.Execute(expressionTree)
           → SQL generator → Npgsql → PostgreSQL
           → rows → materialize List<T>
```

Same deferral until terminal; different **Execute** implementation.

### EnumerableQuery — The Hybrid

When **`List.AsQueryable()`** is used, **`EnumerableQuery`** implements **`IQueryProvider`**. **`Execute`** compiles expression tree to delegates and runs LINQ to Objects. Mimics provider model for unit tests without a database.

## Examples

### Example 1 — Boundary placement

```csharp
// Good — filter in SQL, project in SQL
var dtos = await db.Products
    .Where(p => p.IsActive && p.Stock > 0)
    .Select(p => new ProductDto(p.Id, p.Name))
    .ToListAsync();

// Bad — loads all products
var dtos = (await db.Products.ToListAsync())
    .Where(p => p.IsActive)
    .Select(p => new ProductDto(p.Id, p.Name));
```

### Example 2 — Explicit AsEnumerable for hybrid

```csharp
var shipped = db.Orders.Where(o => o.Shipped).ToList(); // SQL filter + materialize moderate set
var analyzed = shipped.AsEnumerable().Where(o => Score(o) > 0.8); // local scoring OK on bounded set
```

## Common Mistakes & Anti-patterns

**`ToList()` before `Where`** — client-side filter on full table.

**Treating `IQueryable` like cheap `IEnumerable`** — each terminal may hit DB.

**Non-translatable code in `IQueryable Where`** — runtime exceptions or client eval.

**Returning `IQueryable` across layers** — disposed context, unbounded queries.

---

## Production & Real-World Notes

Always know **one SQL query** vs **N+1** — **`Include`** / projection later. **`IQueryable`** composition enables single SQL with dynamic filters; abuse enables footguns.

Use **`AsSplitQuery`**, **`AsNoTracking`**, pagination at **`IQueryable`** level — not after **`ToList()`**.

---

## Comparison / Trade-offs

| Use `IEnumerable` | Use `IQueryable` |
|-------------------|------------------|
| In-memory collections | EF `DbSet`, remote LINQ providers |
| Local arbitrary C# in operators | Translatable filters/projections |
| After intentional materialize | Before SQL generation |
| API responses (materialized) | Repository query composition (scoped) |

---

## Quick Reference

- **`IEnumerable`** + **`Func<>`** → CLR
- **`IQueryable`** + **`Expression<>`** → provider (SQL)
- **`AsEnumerable()`** → client boundary
- **`AsQueryable()`** on list → fake provider, still local
- Materialize with **`ToList`/`ToListAsync`** at scope edge

---

## Key Takeaways

- Same LINQ syntax; **execution location** differs.
- **`IQueryable`** stores **expression trees**, not compiled filters in the pipeline.
- **`AsEnumerable()`** moves execution to memory — know the cost.
- Choose signatures and materialization boundaries deliberately.
- EF Core translation is the payoff and the risk — **`1.3.12`** next.
