# Answers: CancellationToken — RequestAborted

## Scoring

- 90–100%: отлично · 80–89%: pass · <80%: повтори lection + summary

## Answers

### 1. [Recall]
**Answer:** `CancellationToken`, привязанный к HTTP-запросу; отменяется при disconnect клиента, shutdown и linked cancel.
**Why:** Kestrel/host управляют lifetime запроса через этот токен.

### 2. [Recall]
**Answer:** Нет — достаточно параметра типа `CancellationToken`.
**Why:** Built-in model binder ASP.NET Core.

### 3. [Understanding]
**Answer:** Код сам должен передавать токен в I/O и проверять его; runtime не прерывает произвольный CPU-код.
**Why:** Cooperative cancellation model в .NET.

### 4. [Understanding]
**Answer:** `RequestAborted` — внешний сигнал (клиент/хост); `CancelAfter` — ваш server-side deadline; часто link оба.
**Why:** Разные источники, один механизм через linked CTS.

### 5. [Understanding]
**Answer:** Это ожидаемое поведение, не сбой приложения; массовые Error-алерты создают шум.
**Why:** Клиентский churn нормален на публичных API.

### 6. [Application]
**Answer:** Npgsql/EF отменят команду; всплывёт `OperationCanceledException`; ответ клиенту обычно не дойдёт.
**Why:** Token propagated to database I/O.

### 7. [Application]
**Answer:** Да — иначе outbound HTTP продолжится после disconnect входящего запроса.
**Why:** End-to-end cancellation saves resources.

### 8. [Analysis]
**Answer:** `HeavyCpuWork` не получает токен и не проверяет отмену; запрос «висит» до конца CPU работы.
**Why:** `Task.Run` без token forwarding; CPU must poll `ThrowIfCancellationRequested`.
**Common mistake:** Думать, что `ct` на внешнем async автоматически отменяет всё внутри.

### 9. [Analysis]
**Answer:** После завершения response `RequestAborted` может сработать и прервать фоновую работу преждевременно.
**Why:** Request scope ends; нужна очередь/`BackgroundService` с отдельным token.

### 10. [Application]
**Answer:** `CreateLinkedTokenSource(httpContext.RequestAborted)` + `CancelAfter(30s)` на CTS.
**Why:** First-fired-wins linked token.

### 11. [Understanding]
**Answer:** Нет — cancel прерывает ожидание, не откатывает committed data.
**Why:** Cancellation ≠ transaction rollback.

### 12. [Analysis]
**Answer:** Когда `context.RequestAborted.IsCancellationRequested` — подавить 500, не логировать как fatal.
**Why:** Expected control flow for client gone.

### 13. [Recall]
**Answer:** Client disconnect; host graceful shutdown; explicit cancel via linked/custom CTS.
**Why:** Documented ASP.NET Core behavior.

### 14. [Understanding]
**Answer:** LB может оборвать раньше/позже app; несогласованность даёт «зомби»-работу или ложные cancel.
**Why:** Distributed timeout layers.

### 15. [Application]
**Answer:** Например: `app.MapGet("/", async (AppDbContext db, CancellationToken ct) => await db.Items.ToListAsync(ct));`
**Why:** Token injected and passed to EF.
