# Pull Request / Merge Request Workflow

> Roadmap: `0.2.8` · Node: `0.2` — Git: branches and collaboration · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Distinguish **Pull Request (PR)** and **Merge Request (MR)** — same concept, different platform names.
- Describe the **feature branch → push → open PR → review → merge** lifecycle end to end.
- Explain **target branch**, **source branch**, and why teams protect `main`.
- Choose merge strategies: **merge commit**, **squash merge**, **rebase merge** — and their trade-offs (`0.2.2`).
- Use PR metadata effectively: title, description, linked issues, draft PRs.
- Recognize CI gates, required reviewers, and **merge conflicts** surfaced in the UI.

---

## Why This Matters

Until `0.2.7`, collaboration was mostly Git commands on your machine and `origin`. Professional teams rarely merge feature work by pushing directly to `main`. Instead, you **publish a branch** (`0.2.6`), ensure it tracks upstream (`0.2.7`), then open a **Pull Request** on GitHub or a **Merge Request** on GitLab/Azure DevOps. The PR is a **proposal**: "here are my commits; please review before integrating."

This workflow is where **code review etiquette** (`0.2.10`), **Conventional Commits** (`0.2.9`), and your merge/rebase knowledge (`0.2.2`–`0.2.4`) meet production reality. CI runs tests against your branch. Reviewers comment on diffs. Maintainers choose how commits land on `main`. A Middle fullstack developer opens clean PRs daily — small scope, clear description, green checks, no surprise conflicts.

Understanding PR mechanics also explains why **force-push on feature branches** is normal after rebase but **never on `main`**, and why "Update branch" buttons exist when `main` moves ahead of your PR.

## Core Concepts

### PR vs MR — Same Idea, Different Labels

| Platform | Name |
|----------|------|
| GitHub | **Pull Request (PR)** |
| GitLab | **Merge Request (MR)** |
| Azure DevOps | **Pull Request** |

All mean: **request to merge source branch into target branch** with review, discussion, and automation. "Pull" emphasizes integrating changes *into* the target; "Merge" emphasizes the final action. Functionally identical.

### The Standard Lifecycle

```
1. Create feature branch from latest main
      git switch main && git pull && git switch -c feature/order-api

2. Commit locally (atomic, meaningful messages — 0.2.9)

3. Push and set upstream
      git push -u origin feature/order-api

4. Open PR: source feature/order-api → target main

5. CI runs (build, test, lint)

6. Code review — comments, approvals (0.2.10)

7. Address feedback — new commits or amend (unpushed) on same branch

8. Merge when green + approved

9. Delete feature branch (local + remote) — hygiene
```

The PR **tracks a living branch**. New pushes update the PR diff automatically. Reviewers see incremental changes.

### Source Branch, Target Branch, and Forks

- **Source (head):** your feature branch — changes you want merged.
- **Target (base):** usually `main` or `develop` — where changes should land.

In **fork workflow** (open source), source may be `yourfork:feature` and target `upstream:main`. In **shared repo** (most companies), both branches live on `origin`.

**Draft PRs** signal work-in-progress — CI may still run, but reviewers know not to expect final quality yet.

### What Reviewers See

The PR UI shows:

- **Commits list** — narrative of your branch (`0.2.9` helps readability).
- **Files changed** — unified diff vs target.
- **Checks** — CI status, required approvals.
- **Conflicts** — Git cannot auto-merge; you must resolve locally (`0.2.5`) and push.

Conflicts in UI mean target moved since you branched or rebased — fix with:

```bash
git fetch origin
git rebase origin/main    # or merge origin/main — team preference
# resolve, push
```

### Merge Strategies on the Platform

When maintainer clicks Merge, the platform applies one strategy (team-configured):

**Merge commit** — creates a merge commit on target; preserves all feature commits and branch topology (`0.2.2`). History shows the PR branch existed.

**Squash merge** — squashes all PR commits into **one** commit on target. Clean `main` log; loses per-commit detail on target (still visible in PR page). Common for small teams and ticket-based work.

**Rebase merge** (GitHub "Rebase and merge") — replays PR commits linearly onto target tip without merge commit. Linear `main`; rewrites commit SHAs on merge.

None replaces local Git knowledge — the button triggers server-side Git operations you could do manually.

### Branch Protection and Gates

**Protected branches** on `main` typically enforce:

- No direct push — PR required.
- Required status checks (CI green).
- Minimum approvals (1–2 reviewers).
- Up-to-date with target before merge.
- Signed commits (optional, advanced).

Failing checks block merge even with approvals. This is **production mindset**: broken `main` blocks everyone.

### Linking Work Items

Teams link PRs to tickets (Jira, Linear, Azure Boards):

- `Fixes #42` in PR description auto-closes GitHub issue on merge.
- Traceability from requirement → code → deploy.

## Under the Hood

A PR is **not** a Git object — it is **platform metadata** pointing at refs. GitHub stores `pull/123/head` refs that track your branch tip. When you push, the platform updates PR mergeability by attempting a **test merge** (or computing diff) against current target.

CI clones your branch (or merge result) and runs pipelines. **Merge queue** (advanced) serializes merges to keep `main` green.

Locally, nothing changes until merge completes — except your continued pushes update `origin/feature`.

## Syntax / Commands / API

Git side of PR workflow:

```bash
# Start feature
git fetch origin
git switch main && git pull --ff-only origin main
git switch -c feature/ticket-123

# Publish
git push -u origin feature/ticket-123
# Open PR in browser — platform CLI optional:
gh pr create --base main --head feature/ticket-123 --title "..." --body "..."

# Update PR after review
git add . && git commit -m "fix: address review comments"
git push

# Sync with main
git fetch origin && git rebase origin/main
git push --force-with-lease

# After merge — cleanup
git switch main && git pull origin main
git branch -d feature/ticket-123
git push origin --delete feature/ticket-123
```

Platform URLs differ; `gh` (GitHub CLI) is optional but common.

## Examples

### Example 1: Minimal ASP.NET feature PR

Branch `feature/health-endpoint` with one endpoint + test. Push with `-u`. PR title: `feat: add GET /health endpoint`. Description: what, why, how to test (`curl`, Swagger). CI runs `dotnet test`. One approval → squash merge per team policy.

### Example 2: PR blocked by failing CI

Reviewer approves, but merge button disabled — **required check** `build` failed. You fix test, push new commit; CI re-runs; merge unlocks. Approvals may reset on new pushes (team setting).

### Example 3: Conflict after main moved

PR open three days; `main` advanced. UI shows **"This branch has conflicts."**

```bash
git fetch origin
git rebase origin/main
# resolve OrderService.cs — 0.2.5
git push --force-with-lease
```

PR updates; conflicts cleared.

### Example 4: Squash vs merge commit impact

Three commits on PR: `feat: scaffold`, `fix: typo`, `test: add case`. **Squash merge** → one commit on `main`: `feat: add order validation (#88)`. **Merge commit** → three commits plus merge node. Choose per team readability vs archaeology needs.

## Common Mistakes & Anti-patterns

**Giant PRs** — hundreds of files; review paralysis. Prefer small, vertical slices.

**Empty description** — reviewers lack context; slow approvals.

**Opening PR from stale `main`** — instant conflict debt; always fetch latest main first.

**Merging with red CI** — bypass culture breaks `main`; never acceptable in production teams.

**Ignoring review comments** — resolve or reply; don't silently push unrelated fixes.

**Leaving dead branches** — clutter; delete after merge.

**Wrong target branch** — PR into `release/1.0` instead of `main` by mistake.

## Production & Real-World Notes

**Trunk-based development** — short-lived branches, PRs live hours not weeks. **GitFlow** — longer-lived `develop`, PRs into it; know which model your team uses.

**CODEOWNERS** auto-assigns reviewers by path (`/src/Api/` → backend team).

**Required reviewers** from security for auth changes.

**Deployment** often triggers on merge to `main` — PR quality equals production risk.

Monorepos may use **path filters** so only affected packages build in CI.

## Comparison / Trade-offs

| Merge style | main history | Best for |
|-------------|--------------|----------|
| Merge commit | Preserves branch graph | Audit trail, large features |
| Squash | One commit per PR | Clean log, ticket workflow |
| Rebase merge | Linear commits preserved | Strict linear main |

PR vs direct push: safety and review vs speed — protected branches remove direct push option on `main`.

## Quick Reference

```
PR/MR = proposal to merge source → target (usually feature → main)

Flow: branch → commit → push -u → open PR → CI → review → merge → delete branch

Conflicts: fetch → rebase/merge main → push

Merge options: merge commit | squash | rebase merge

Draft PR = WIP signal
Fixes #N = link issue (GitHub)
```

## Key Takeaways

1. **PR/MR** is platform review workflow around Git branches — not a Git primitive.
2. Standard path: **feature branch → push → PR → CI → review → merge**.
3. **Target** is usually protected `main`; **source** is your published branch.
4. **Merge strategy** affects `main` history — connect to `0.2.2`.
5. **Conflicts** resolve locally (`0.2.5`), push updates PR.
6. Small PRs, good descriptions, green CI — professional defaults.
7. Clean up branches after merge; sync local `main`.
