# Answers: Safe vs Idempotent

## Answers

### 1. [Recall]
**Answer:** Safe method — intended read-only request that should not alter server state relevant to the user.
**Why:** RFC 9110 safe definition.

### 2. [Recall]
**Answer:** Idempotent method — same intended server effect whether executed once or multiple times with identical request.
**Why:** Core retry/cache contract.

### 3. [Recall]
**Answer:** GET, HEAD, OPTIONS (and TRACE in spec, disabled in practice).
**Why:** Standard safe set.

### 4. [Understanding]
**Answer:** All safe methods are idempotent. Not all idempotent are safe — **PUT** and **DELETE** idempotent but not safe.
**Why:** Reading twice doesn't change more; writing twice may same state but still mutation.

### 5. [Understanding]
**Answer:** POST not safe and not idempotent; caching response would serve wrong representation to others and imply GET-like semantics.
**Why:** Shared cache rules target safe methods primarily.

### 6. [Understanding]
**Answer:** Violates **non-idempotent** POST; risk **double payment/order**.
**Why:** Server may process both if first succeeded but response lost.

### 7. [Understanding]
**Answer:** Yes idempotent — item absent after first delete; second DELETE → **404** or **204**, no additional harm.
**Why:** Absence is stable state.

### 8. [Application]
**Answer:** GET profile — safe, idempotent. PUT profile — not safe, idempotent. POST orders — neither. PATCH increment — not safe, **not idempotent** (relative increment).
**Why:** Apply matrix + PATCH nuance.

### 9. [Application]
**Answer:** Retry without dedup: GET, HEAD, PUT, DELETE, OPTIONS (caution PATCH). Cannot blindly retry POST (and risky PATCH); POST needs Idempotency-Key or dedup.
**Why:** Standard client retry guidance.

### 10. [Analysis]
**Answer:** C неверно.
**Why:** HTTPS provides transport security, not HTTP method idempotency.

### 11. [Analysis]
**Answer:** GET search **safe** (if read-only); POST search **not safe** (body processing, may log side effects still but intended read of search results via POST still classified not safe).
**Why:** POST never safe per HTTP regardless of read-like behavior.

### 12. [Understanding]
**Answer:** Prefetch may run without user intent; mutation would change state accidentally (CSRF-like), violate safe contract.
**Why:** Infrastructure trusts safe methods.

### 13. [Understanding]
**Answer:** HTTP idempotent PUT — retry converges same resource state; **exactly-once** needs dedup keys, transactions, outbox — network is at-least-once.
**Why:** Layer separation.

### 14. [Application]
**Answer:** Non-idempotent POST commands with financial/side effects: payments, order create, inventory reserve — document required Idempotency-Key and retention window.
**Why:** Enable safe client retries.

### 15. [Recall]
**Answer:** **PUT**, **DELETE** — всегда в API-наборе; третий частый пример — **PATCH** с absolute-set semantics (условно idempotent). Без оговорок: PUT и DELETE.
**Why:** Оба меняют state (not safe), но повтор не умножает effect (idempotent).
