# Exercises: Expression Trees

> Complete after `1.3.11`–`1.3.12` and passing test.

## Easy

### E1. Func vs Expression types
**Goal:** Compiler target awareness.
**Steps:** Two one-liners: `List<T>.Where` and assign lambda to `Expression<Func<T,bool>>` variable. Print `expr.ToString()` and `expr.Body.NodeType`.
**Success criteria:** Output inspected and explained.

### E2. Translation guess
**Goal:** Translatability intuition.
**Steps:** List 5 predicates — mark EF-translatable or not (e.g. `Age > 18`, `Helper(u)`, `EF.Functions.Like(...)`).
**Success criteria:** 5/5 with one-line why.

## Medium

### M1. Manual expression filter
**Goal:** Build tree by hand.
**Steps:** `Expression.Parameter` + `Expression.Property` + comparison → `Expression.Lambda<Func<User,bool>>`. Use with `List<User>.AsQueryable().Where(lambda)` or mock.
**Success criteria:** Compiles; filters correctly.

### M2. Compile trap demo
**Goal:** See EF boundary break.
**Steps:** Document (or code comment) what happens if you `Compile()` expression then pass `Func` to custom `Where` extension on `IQueryable` — why EF can't translate.
**Success criteria:** Clear prose explanation.

## Hard (stretch)

### H1. Dynamic sort (sketch)
**Goal:** Expression composition pattern.
**Steps:** Method `ApplyOrder(IQueryable<User> q, string sortBy)` — switch on property name building `Expression` for `OrderBy` (one property enough, e.g. `LastName`).
**Success criteria:** No string SQL; uses `Expression.Property`.

### H2. ToQueryString lab
**Goal:** EF integration.
**Steps:** Minimal EF Core project: two queries differing by one `Where`; log `ToQueryString()` for both; correlate SQL clause with expression branch.
**Success criteria:** Screenshot or pasted SQL + annotation.
