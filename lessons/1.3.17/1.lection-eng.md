# Nullable Reference Types — #nullable enable

> Roadmap: `1.3.17` · Node: `1.3` — C# Advanced Topics · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Enable **nullable reference types (NRT)** with `#nullable enable` and project `<Nullable>enable</Nullable>`.
- Read **`string` vs `string?`** annotations and understand they are **compile-time only**.
- Follow **null-state flow analysis**: assign, dereference, maybe-null after branches.
- Use **`!` null-forgiving operator**, **`[NotNullWhen]`**, **`[MaybeNull]`**, **`[NotNullIfNotNull]`** correctly.
- Fix **CS8600–CS8629** warnings without silencing blindly.
- Design APIs with accurate null contracts for ASP.NET DTOs and services.

---

## Why This Matters

For decades C# let **`string`** mean «maybe null» at runtime while pretending non-null at compile time — **`NullReferenceException`** was the #1 bug class. **Nullable Reference Types** (C# 8+) add a **second axis**: value types had `int?`; now **`string?`** marks reference types that may be null, and **`string`** promises «I won't assign null here» (analyzed by compiler).

This is not runtime enforcement like `Option<T>` in Rust — **no CLR change**. It's **Roslyn flow analysis** + annotations in metadata (`NullableAttribute`) for consumers. Still, teams with NRT enabled cut null bugs dramatically in code review and CI.

ASP.NET models, EF entities, JSON deserialization — all intersect NRT: `required` properties, `string? optionalField`, `[NotNullWhen(true)] TryParse` patterns. Middle developers must read warnings, not spam `!`.

## Core Concepts

### Turning NRT On

File level:

```csharp
#nullable enable
```

Project-wide (SDK-style):

```xml
<Nullable>enable</Nullable>
```

Modes: `enable`, `disable`, `warnings`, `annotations`. **`enable`** = warnings + annotations.

Legacy code migrated incrementally with `#nullable disable` in old files or `?` on public API only.

### Annotated Types

```csharp
string name;      // non-nullable intent — should not assign null
string? nickname; // may be null
```

Compiler tracks **null state** per variable at each line:

```csharp
string? s = GetMaybe();
Console.WriteLine(s.Length); // CS8602 — maybe null

if (s is not null)
    Console.WriteLine(s.Length); // OK — narrowed
```

**`string`** without `?` still can hold null at runtime if you bypass analysis (`!`, legacy interop) — NRT is **best-effort static checking**.

### Flow Analysis Rules (Simplified)

- Assignment `null` to non-nullable → warning.
- Dereference `.` on maybe-null → warning.
- After `== null` return/throw, branch knows non-null.
- **`??`**, **`?.`**, **`??=`** affect state.
- `[MaybeNullWhen(false)] bool TryGet(..., out string? value)` tells analyzer when `out` is non-null.

### Null-Forgiving Operator `!`

```csharp
string s = GetMaybe()!; // tell compiler «trust me, not null»
```

Suppresses warning **without runtime check**. Abuse hides bugs — use only with invariants compiler cannot see (tests, `Debug.Assert`, known JSON schema).

### Attributes for Libraries

| Attribute | Purpose |
|-----------|---------|
| `[NotNullWhen(true)]` on `bool` param | When method returns true, param not null |
| `[MaybeNull]` return | May return null even if `T` non-nullable |
| `[AllowNull]` input | Accept null though type lacks `?` |
| `[DisallowNull]` input | Null forbidden though `T?` |
| `[MemberNotNull(nameof(X))]` | After method, field X non-null |

BCL uses these extensively — your APIs should too for correct consumer analysis.

### Generics and NRT

```csharp
class Box<T> where T : class
{
    public T? Item { get; set; } // nullable ref type param
}
```

`T?` for unconstrained or `class` means nullable reference dimension. **`notnull` constraint** forbids null assignment to `T`.

### Interop and JSON

Deserialization may leave «non-nullable» properties null if JSON missing field — **CS8618** (non-nullable property must contain non-null value when exiting constructor). Fixes:

- `required` modifier (C# 11).
- Initialize with default `= ""`.
- Make property `string?` if truly optional.
- `[JsonRequired]` or custom validation.

EF Core: navigation properties often `Entity?` for optional relationships.

## Under the Hood

NRT annotations emit **`NullableContextAttribute`** and **`NullableAttribute`** on members — **metadata only**, no IL null checks injected for plain dereference.

Flow analysis runs in Roslyn **nullable analysis pass** — separate from definite assignment. Interprocedural analysis limited — deep calls may lose state (`MaybeNull` returns).

`!` erases warning at that point only — no IL.

Nullable reference state **ignored at runtime** by JIT — `NullReferenceException` still possible.

Cross-assembly: nullable annotations on public API affect callers when NRT enabled in consumer project.

## Syntax / Commands / API

```csharp
#nullable enable
string? maybe;
if (x is null) throw ...
var y = x!; // forgiving

[NotNullWhen(true)] bool TryGet([NotNullWhen(true)] string? key, out string? value);
```

Common warnings:

- **CS8600** — null to non-nullable.
- **CS8602** — dereference maybe null.
- **CS8618** — uninitialized non-nullable field/property.

## Examples

### Guard clause pattern

```csharp
public void Process(User user)
{
    ArgumentNullException.ThrowIfNull(user);
    // user treated as non-null below
}
```

`ThrowIfNull` annotated `[NotNull]`/`DoesNotReturnIf` — flow analysis understands.

### Try pattern

```csharp
public bool TryFind(string id, [NotNullWhen(true)] out User? user)
{
    user = _store.Find(id);
    return user is not null;
}
```

Caller:

```csharp
if (TryFind(id, out var u))
    u.Name = ...; // u non-null
```

## Common Mistakes & Anti-patterns

**`!` everywhere** to green the build — defeats NRT.

**Non-nullable string properties on DTOs** without `required`/init — CS8618 + runtime null from API.

**Ignoring warnings in only half the solution** — inconsistent contracts.

**`string?` on everything** — loses signal; use intentionally.

**Assuming NRT prevents NRE at runtime** — it doesn't without checks.

## Production & Real-World Notes

Enable `<Nullable>enable</Nullable>` on new projects; treat warnings as errors in CI (`<WarningsAsErrors>nullable</WarningsAsErrors>` optional phased rollout).

Libraries ship nullable annotations in NuGet — consumers benefit.

Code review: every `!` needs comment or invariant.

Migration: use nullable annotations package for .NET Framework consumers (limited).

## Key Takeaways

- NRT = **compile-time** null analysis for reference types; **no CLR enforcement**.
- **`string?`** may be null; **`string`** should not — warnings if violated.
- Flow analysis **narrows** after checks, patterns, attributed APIs.
- **`!`** suppresses warnings only — not a runtime guard.
- Fix **CS8618** with `required`, defaults, or honest `?`.
- Annotate library APIs with **`[NotNullWhen]`** etc. for callers.
