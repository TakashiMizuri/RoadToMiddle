# Exercises: Client-Server and Request-Response

> Complete after reading `1.lection-eng.md`, `2.lection-ru.md` and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Role identification
**Goal:** Map client/server roles in a realistic flow.

For each hop, label **client** or **server** (HTTP layer unless noted):

1. Browser → `https://cdn.example.com/app.js`
2. React app → `https://api.example.com/orders`
3. ASP.NET Core → PostgreSQL (database protocol)
4. ASP.NET Core → Stripe REST API

**Success criteria:** Four correct labels with one-sentence justification each.

---

### E2. Count the cycles
**Goal:** See one screen as many request-response pairs.

**Scenario:** User opens a product page. Sequence:
- `GET /` (HTML shell)
- `GET /assets/main.js`
- `GET /api/products/7`
- `GET /images/product-7.webp`

How many distinct HTTP request-response cycles? What role is the browser in each?

**Success criteria:** Correct count; browser always client for HTTP listed.

---

## Medium

### M1. Observe with curl
**Goal:** Read raw request-response.

Run:
```bash
curl -v https://httpbin.org/get?name=test
```

Copy (or sketch) the **request line**, one request header, **response status line**, and `Content-Type` response header.

**Success criteria:** Shows client-initiated GET; status 200; identifies where request ends and response begins in `-v` output.

---

### M2. Stateless design sketch
**Goal:** Explain session without protocol state.

Draw or write 4–6 sentences: user logs in via `POST /api/login`, then calls `GET /api/me`. How does the server know who they are on the second request without HTTP remembering the first?

**Success criteria:** Mentions token/cookie repeated on `/api/me`; no "server remembers TCP connection login."

---

## Hard (stretch)

### H1. Timeout scenario write-up
**Goal:** Production thinking on failed delivery.

**Scenario:** Mobile app calls `POST /api/orders` with 5s client timeout. Server takes 8s, creates order in DB, then tries to write response — connection already closed.

Answer:
1. What does the user likely see?
2. Is an order in the database? 
3. Name two mitigations (client and/or server).

**Success criteria:** User sees error/timeout; order may exist (duplicate risk on retry); mentions idempotency keys, async status polling, longer timeout with UX, server-side cancellation awareness.
