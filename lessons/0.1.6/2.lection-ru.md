# Паттерны .gitignore и .gitkeep

> Roadmap: `0.1.6` · Узел: `0.1` — Git: основы · Глубина: практика

## Learning Objectives

После урока ты сможешь:

- Писать корректные паттерны `.gitignore`: globs, `**`, negation (`!`), prefix `/`.
- Выбирать между **local**, **global** и **per-user exclude** источниками ignore-правил.
- Игнорировать build artifacts (`bin/`, `obj/`, `node_modules/`) и secrets (`.env`).
- Использовать `.gitkeep` для пустых директорий.
- Отлаживать правила через `git check-ignore -v`.

---

## Why This Matters

В `0.1.4` и `0.1.5` ты узнал, как Git хранит immutable objects и связывает commit'ы в граф. Всё, что попало в object database, остаётся в history — часто навсегда, даже если «удалить» в следующем commit'е. Эта необратимость делает особенно болезненным один класс ошибок: закоммитить не те файлы.

Без ignore-правил типичный full-stack repo за недели превращается в свалку. `bin/Debug/` — сотни мегабайт DLL. `node_modules/` — сотни мегабайт или гигабайты зависимостей, которые должны появляться через `npm install`, а не через Git. `.env` с паролем к БД на GitHub — инцидент, ротация ключей, post-mortem. `*.user`, `.vs/` — шум в каждом PR. Кросс-платформенные команды тащат `Thumbs.db` и `.DS_Store` в history.

У всех проблем одна причина: Git попросили отслеживать то, что не должно быть в version control. `.gitignore` — фильтр на входе: Git «не видит» certain paths при поиске untracked файлов. Это один из самых дешёвых и эффективных guardrails — и всё же команды утекают secrets и раздувают repo, потому что правила пишут один раз, копируют криво или не понимают механику.

Урок практический: что `.gitignore` реально делает и чего не исправит задним числом, откуда берутся правила и как они склеиваются, как писать паттерны для Windows и Linux, зачем `.gitkeep`, как отладить упрямый файл через `git check-ignore -v`. В конце ты сможешь собрать разумный `.gitignore` для ASP.NET Core + React и объяснить коллеге, почему после факта нужен `git rm --cached`. Эти навыки напрямую защищают object database из `0.1.4`: всё, что однажды стало blob, остаётся в истории клонов.

## Core Concepts

### Что делает `.gitignore` — и чего не делает

Первое заблуждение — про scope. `.gitignore` влияет только на **untracked** файлы. Git смотрит на него, когда сканирует working tree: «что нового?», «что возьмёт `git add .`?». Совпал с правилом — не покажет в `git status` (по умолчанию) и не добавит через `git add .` / `git add -A`.

То, чего `.gitignore` **не** делает, не менее важно. Закоммитил `secrets.env` три месяца назад — строка в `.gitignore` сегодня не вычистит history и не снимет tracking с копии в index. Файл останется в каждом clone, пока явно не сделаешь `git rm --cached` и commit — а secret останется в старых commit'ах, пока не перепишешь history. `.gitignore` — **фильтр на входе**, не ретроактивный ластик. Профилактика важнее, особенно для blob'ов, которые нельзя «раскрыть» после push.

Есть лазейка: `git add -f <path>` **force-add** игнорируемого файла. Иногда оправдано (нужен один log в repo), иногда катастрофа (credentials). Force-add sensitive paths в production — серьёзный process failure.

### Откуда берутся ignore-правила

Правила не только в `.gitignore` в корне. Git мержит паттерны из нескольких источников — понимание стека убирает «у меня работает, у тебя нет».

Committed `.gitignore` в корне — **контракт команды**. Все после clone получают те же правила: build output, dependencies, secrets, IDE noise под ваш stack. Вложенные `.gitignore` в подпапках действуют на subtree — в monorepo удобно игнорировать `docs/_build/` только под `docs/`.

Global exclude file (`core.excludesfile`) — личные паттерны для **любого** repo: OS cruft вроде `Thumbs.db`, swap-файлы редактора. Не shared, пока коллега не настроит тот же путь.

`.git/info/exclude` — per-repo, **не коммитится**: личные scratch-паттерны для одного clone без засорения общего `.gitignore`. Командная строка сильнее всего: `git add -f` обходит ignore на один вызов.

Настрой global ignore один раз:

```bash
git config --global core.excludesfile ~/.gitignore_global
```

При нескольких совпадениях Git идёт по наборам правил в заданном порядке; поздние правила могут **отменить** ранние через `!`, если path не заблокирован ignored родительской папкой. Практика: negation **после** правила, которое отменяешь; нельзя re-include файл **внутри** ignored директории без трюка с `logs/*` — ниже.

### Синтаксис паттернов: globs, anchor, double star

Паттерны `.gitignore` — **fnmatch-style globs** относительно файла `.gitignore` (если не anchor к корню repo через leading `/`). Не нужен regex — нужен набор кирпичиков.

`*.log` — любой `.log` в каталоге `.gitignore` и ниже. Обычно так и хотят для логов. Leading `/` меняет смысл: `/build` — только `build` в **корне** repo, не `src/build`. Нужно, когда в подпроектах легитимный `build`, а игноришь только корневой output.

Trailing slash (`logs/`) — **только директории**. Без slash Git может матчить и файл, и папку `logs`. Для dependencies: `node_modules/` — директория `node_modules` на любой глубине, если нет leading `/`.

`**` — ноль или больше директорий. `**/node_modules/` — каждый `node_modules` на любой глубине, критично в JS monorepo. `**/temp` — файл или папка `temp` где угодно. `*` — в пределах одного сегмента path; `?` — один символ. Классы `[abc]` — как в shell.

**Negation** — `!` в начале паттерна. Стандарт production: игнор env, разрешить template:

```gitignore
.env
.env.*
!.env.example
```

`!.env.example` **после** `.env.*`, иначе не сработает. Template в repo — новый разработчик копирует локально; secrets не коммитят.

Ловушка negation: `logs/` игнорирует всю папку — `!logs/keep.log` **не работает**, Git не заглядывает внутрь ignored directory. Workaround:

```gitignore
logs/*
!logs/.gitkeep
!logs/keep.log
```

`logs/*` игнорирует содержимое, сама `logs/` не ignored — negation может вернуть отдельные файлы.

Комментарии — `#`. Файл с `#` в имени — `\#special`.

### Порядок применения правил (практический сценарий)

Представь monorepo: корневой `.gitignore` игнорирует `dist/`, а в `frontend/.gitignore` — `coverage/`. Файл `frontend/coverage/report.html` матчится правилом из подпапки, потому что Git поднимается от файла вверх и собирает все `.gitignore` на пути. Файл `backend/dist/out.js` матчится корневым `dist/` на любой глубине. Если бы в корне стояло `/dist/` с leading slash, `backend/dist/` **не** игнорировался бы — только `dist/` в корне repo. Такие нюансы ломают CI на Linux, когда разработчик на Windows копировал `.gitignore` из другого проекта и не проверил anchor.

Ещё сценарий: в корне `*.json`, а нужно коммитить `package.json` и `tsconfig.json`. Слишком широкий паттерн скрывает критичные lock- и config-файлы; `git status` их не покажет, `git add .` их не возьмёт, и сборка у коллеги упадёт без объяснений. Правильный подход — игнорировать конкретные каталоги (`coverage/*.json`, `dist/stats.json`), а не все JSON в проекте.

### Отладка через `git check-ignore`

Когда `git status` ведёт себя не так, как ждёшь — не гадай:

```bash
git check-ignore -v node_modules/lodash/index.js
# .gitignore:3:node_modules/    node_modules/lodash/index.js
```

Verbose строка: файл, номер строки, паттерн. Быстрее всего закрыть спор «почему Git всё ещё tracking» в review.

---

## Syntax / Commands / API

Три команды в реальных workflow постоянно.

`git check-ignore -v <path>` — диагностика. `git rm --cached <file>` — снять с tracking, оставить на диске (фикс после commit до ignore). `git add -f <path>` — force; редко и не для secrets.

Global config:

```bash
git config --global core.excludesfile ~/.gitignore_global
```

## Examples

### Full-Stack .NET + React `.gitignore`

Читай как narrative намерений, не magic list для слепого paste.

```gitignore
[Bb]in/
[Oo]bj/
[Dd]ebug/
[Rr]elease/
*.dll
*.pdb
*.user
*.suo

node_modules/
dist/
build/
.npm/
*.tsbuildinfo

.env
.env.*
!.env.example

.vs/
.idea/
*.swp
.DS_Store
Thumbs.db

coverage/
TestResults/
*.coverage
```

`[Bb]in/` — case-insensitive для Windows `Bin` и Linux `bin`. `node_modules/` без `/` — на любой глубине (`client/`, `app/`). `!.env.example` — industry standard: template в repo, secrets нет. `dist/`, `build/` — compiled frontend не попадают в object database из `0.1.4`; десять мегабайт bundle в history преследуют каждый clone.

Стартуй с [github/gitignore](https://github.com/github/gitignore) — `VisualStudio.gitignore` + `Node.gitignore` — и кастомизируй под layout solution.

Пройдись по файлу перед первым commit в новом проекте так же, как по `Dockerfile` или CI config: есть ли `UserSecretsId` в `.csproj` (секреты не в repo, но `secrets.json` в UserSecrets folder и так вне repo), есть ли `appsettings.Development.json` с реальными паролями (его иногда коммитят по ошибке — лучше шаблон + локальный override в `.gitignore`), попадают ли артефакты тестов `TestResults/` и frontend `dist/` после `npm run build`. Один осознанный `.gitignore` в начале дешевле, чем `git filter-repo` через полгода.

### Пошаговый workflow: секрет случайно закоммичен

Допустим, `.env` уже в `origin/main`. Только `.gitignore` проблему не закроет.

Сначала добавь паттерн в `.gitignore`, чтобы файл не попал в следующие commit'ы. Затем `git rm --cached .env` и commit «Stop tracking .env» — в repo перестанет отслеживаться текущая версия, локально файл останется. Дальше — обязательная ротация всех ключей и паролей из утёкшего файла: history на remote всё ещё содержит старые commit'ы с secret. Если repo публичный или clone'ов много, планируй `git filter-repo` или BFG Repo-Cleaner и force-push с согласования команды — это переписывание history, затрагивает всех. Параллельно включи secret scanning на GitHub/GitLab. `.gitignore` — профилактика на будущее; прошлое лечится отдельным инцидент-процессом.

### `.gitkeep` для пустых директорий

Git отслеживает **файлы**, не папки. Пустая директория невидима для `git add` и не появится у коллег после clone. Иногда структура нужна заранее — `logs/archive/`, `uploads/` до фичи.

Конвенция — placeholder. `.gitkeep` не фича Git, а имя пустого файла, чтобы Git создал папку:

```bash
mkdir -p logs/archive
touch logs/.gitkeep
git add logs/.gitkeep
git commit -m "Add logs directory structure"
```

Альтернатива — однострочный `README.md` с назначением папки; часто яснее для maintainers. Например: «Runtime log files land here; not committed.» — и новый разработчик понимает контракт без знания конвенции `.gitkeep`. Когда появятся реальные файлы, `.gitkeep` можно убрать или оставить. Не используй `.gitkeep` для папок, которые app создаёт при runtime (`uploads/`, `temp/`), если deploy или Docker entrypoint должен делать `mkdir -p`: иначе в production папки не будет, а в dev она «есть из Git» — классический drift между окружениями. Version control для source и config, не замена deploy scripts и не substitute для healthcheck/init container.

### Снятие уже tracked файла

Кто-то закоммитил `.env` до ignore:

```bash
git rm --cached .env
echo ".env" >> .gitignore
git commit -m "Stop tracking .env"
```

`--cached` — только index; локальный файл остаётся. Ротируй всё, что когда-либо ушло в remote. Публичный или широко клонированный repo — `git filter-repo`, secret scanning на платформе; один `.gitignore` history не вычистит.

## Under the Hood

Связь с `0.1.4`: ignore не создаёт objects. Пока путь ignored и untracked, байты файла не попадают в `.git/objects/` — ни blob, ни tree, ни commit. Именно поэтому правильный `.gitignore` защищает не только «чистый» `git status`, но и размер repo и скорость clone: двести мегабайт `node_modules` в history — это двести мегабайт в каждом clone навсегда, даже если позже удалить из последнего commit'а.

Ignore engine Git работает при **обнаружении untracked** — `git status`, `git add`. Для каждого path собираются правила: global excludes, `.git/info/exclude`, цепочка `.gitignore` от path вверх к родителям. Matching — fnmatch относительно расположения rule file.

Первое совпадение побеждает, если позже negation не вернёт path. Ignored directory обрезает обход внутренностей — отсюда `logs/` vs `logs/*`. Весь pipeline **только для файлов не в index**. Tracked остаются tracked, пока не `git rm --cached`.

Отсюда половина internal Git tickets и почему CI гоняет `git status --porcelain` после build: untracked artifacts — нет ignore; modified tracked — кто-то закоммитил, ignore без untrack не поможет.

## Common Mistakes & Anti-patterns

Закоммитить `.env`, потом `.gitignore` — secret в history; untrack, rotation, возможно rewrite. Думать, что «удалил файл в последнем commit — secret исчез из GitHub»: remote и старые clone всё ещё имеют objects в history до filter-repo. `git add -f` на credentials — снесён guardrail. `node_modules/` ok, но `*.json` зацепил `package-lock.json` — сломан reproducible install. `.gitkeep` для runtime dirs — путают VCS и deploy.

Копировать `.gitignore` со старого проекта без проверки: `/build` в корне не игнорирует `client/dist/` без `dist/` или `**/dist/`.

## Production & Real-World Notes

В ASP.NET Core типичный набор дополнений к шаблону: папки `bin/`, `obj/`, артефакты publish `publish/`, пользовательские `*.user`, `.vs/`, иногда `appsettings.*.local.json` для локальных override без секретов в shared config. В React/TypeScript — `node_modules/`, `dist/`, `.vite/`, `*.tsbuildinfo`, кэш ESLint/Prettier если пишется в проект. В Docker-проектах часто игнорируют временные volume-mount артефакты, но **не** игнорируют `Dockerfile` и `docker-compose.yml`.

Корневой `.gitignore` — как code: review в PR при смене stack. `!.env.example` + документация `cp .env.example .env`. Global ignore для OS/IDE — меньше споров про `Thumbs.db`. gitleaks, trufflehog в CI — detection; `.gitignore` — prevention. CI с fail на неожиданный `git status` после compile ловит дыры в ignore рано.

## Comparison / Trade-offs

Широкие `*.log` — просто, но может скрыть нужный log; узкие `/logs/*.log` — точнее, больше поддержки. Global ignore чистит все repo на машине, но невидим команде — в onboarding стоит упомянуть `core.excludesfile`, иначе новичок удивится, почему у него в `git status` появляется `Thumbs.db`, а у остальных нет. `.git/info/exclude` — локальные эксперименты без commit. `.gitkeep` сохраняет форму папок, но чуть hacky; README в папке часто понятнее следующему разработчику.

## Quick Reference

| Задача | Команда / Паттерн |
|--------|-------------------|
| Ignore `node_modules` | `node_modules/` |
| Только root `build` | `/build/` |
| Un-ignore файл | `!path/to/file` |
| Debug | `git check-ignore -v <path>` |
| Global ignore | `git config --global core.excludesfile ~/.gitignore_global` |
| Untrack, оставить файл | `git rm --cached <file>` |
| Пустая папка | `touch dir/.gitkeep` |

## Key Takeaways

- `.gitignore` — только **untracked**; committed → `git rm --cached`.
- Globs: `*`, `?`, `**`, `/` для root, `!` для negation.
- Три личных слоя + shared `.gitignore`.
- Игнорь `bin/`, `obj/`, `node_modules/`, `.env`; коммить `.env.example`.
- `.gitkeep` — конвенция для пустых папок.
- `git check-ignore -v` — какое правило сработало.
