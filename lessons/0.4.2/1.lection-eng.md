# IP Addresses, IPv4, Ports, and Sockets

> Roadmap: `0.4.2` · Node: `0.4` — Networking basics · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Explain the role of IP addresses at the internet layer and how they differ from ports at the transport layer.
- Read and write IPv4 addresses in dotted-decimal form and understand network vs host portions with CIDR basics.
- Describe why ports exist and how `(IP, port, protocol)` uniquely identifies a communication endpoint on a host.
- Define a **socket** as the programming API endpoint and relate it to `socket()` / `bind()` / `connect()` / `listen()`.
- Connect URL components from `0.3.2` (host, port) to underlying identifiers used before HTTP starts.
- Predict common errors (`address already in use`, `connection refused`) from address/port configuration.

## Why This Matters

In `0.3.2` you decomposed a URL: `https://api.example.com:443/v1/users?page=1`. The **host** and **port** are not decorative — they become concrete numbers that TCP and IP use after DNS resolves the name. Without understanding IP and ports, Kestrel binding errors, Docker port mapping, PostgreSQL connection strings, and firewall rules feel arbitrary.

Every concurrent connection on a server is distinguished by a **5-tuple** at transport layer: `(source IP, source port, dest IP, dest port, protocol)`. Your browser might open twenty tabs to different sites; each HTTPS connection uses ephemeral source ports while destination port stays 443. Middle developers configure apps, containers, and cloud security groups daily using these identifiers.

This lesson bridges `0.4.1` (layers) to `0.4.3` (TCP behavior): TCP segments carry IP addresses in the network header and port numbers in the transport header. HTTP Keep-Alive from `0.3.14` reuses one TCP connection — one socket pair — for multiple requests. When that connection breaks, the browser opens a new socket to the same `(IP, 443)`.

## Core Concepts

### IP Addresses — Logical Host Identity

An **IP address** identifies a host's attachment to an IP network at the **internet layer**. It is logical, not physical: one laptop can have Wi‑Fi and VPN addresses simultaneously; addresses change when you move networks.

**IPv4** uses 32 bits, written as four octets in **dotted decimal**: `203.0.113.45`. Roughly 4.3 billion addresses — insufficient for today's internet, mitigated by **NAT**, **private ranges**, and migration to **IPv6** (`2001:db8::1`, 128 bits).

**Public vs private (RFC 1918):**

| Range | Typical use |
|-------|-------------|
| `10.0.0.0/8` | Large corporate / cloud VPC |
| `172.16.0.0/12` | Docker default bridges, medium networks |
| `192.168.0.0/16` | Home routers, small LAN |

Your home laptop might be `192.168.1.105` privately while the router NATs outbound traffic to one public IP. Servers in AWS have private VPC addresses plus optional public Elastic IPs.

**CIDR notation** (`192.168.1.0/24`) means the first 24 bits are network prefix; remaining bits identify hosts. The `/24` subnet has 256 addresses (254 usable hosts classically). Routers use prefix length to decide local vs remote delivery — if destination is same subnet, link-layer delivery; else forward to gateway.

### Ports — Process Multiplexing on One Host

One public IP might host nginx, Kestrel, PostgreSQL, and Redis. **Ports** (16-bit, 0–65535) let transport layer demultiplex incoming segments to the correct process.

**Well-known ports (0–1023):** HTTP 80, HTTPS 443, SSH 22, DNS 53, PostgreSQL 5432. Require elevated privileges on many OSes to bind.

**Registered (1024–49151):** conventions, not enforced.

**Ephemeral / dynamic (49152–65535):** client-side source ports chosen by OS for outbound connections.

The combination **`203.0.113.10:5432`** means PostgreSQL on that host. **`0.3.2` URL default ports:** HTTP omits `:80`, HTTPS omits `:443` — browsers insert them implicitly.

### Sockets — The Developer Interface

A **socket** is an endpoint for sending/receiving data, exposed by the OS as a file descriptor (Unix) or `SOCKET` handle (Windows). In the **Berkeley sockets API** (used by .NET, Node, browsers internally):

```csharp
// Conceptual — actual .NET uses TcpClient / Socket class
// socket(domain, type, protocol) → fd
// bind(fd, local IP:port)      → server listens
// connect(fd, remote IP:port)  → client initiates TCP
// listen / accept               → server accepts clients
```

**Socket address** = IP + port (+ address family AF_INET / AF_INET6). Application code rarely manipulates raw IPs for HTTP — you use hostnames — but Kestrel **binds** a socket to `IPAddress.Any` or `127.0.0.1` and a port (`0.4.6`).

Two types matter early:

- **Stream socket (SOCK_STREAM)** — TCP, reliable byte stream; used by HTTP, HTTPS, PostgreSQL.
- **Datagram socket (SOCK_DGRAM)** — UDP; used by DNS queries, some games (`0.4.4`).

### From URL to Connection (Linking 0.3)

```
https://api.shop.com:443/orders
         │            │
         │            └── transport port (default 443 if omitted)
         └── host → DNS → 198.51.100.22 (internet layer)
```

Steps before HTTP (`0.3.3`):

1. Resolve `api.shop.com` → IPv4/AAAA (`0.4.5`).
2. Create TCP socket; `connect(198.51.100.22, 443)`.
3. TLS handshake on that socket (`0.3.8`).
4. Write HTTP request bytes on encrypted stream.

Connection pooling (`0.3.14`) keeps step 2–3 alive for subsequent HTTP requests on the same socket.

### IPv4 Address Special Cases (Preview 0.4.6)

- **`127.0.0.1`** — loopback; packets never leave host (`0.4.6`).
- **`0.0.0.0`** — not a valid destination; used in **bind** to mean "all interfaces."
- **`255.255.255.255`** — limited broadcast (local segment).

## Syntax / Commands / API

```bash
# Show local IP addresses (Windows)
ipconfig

# Test reachability (internet layer ICMP)
ping 8.8.8.8

# List listening TCP ports and owning PIDs
netstat -ano | findstr LISTENING

# curl with explicit port (0.3.12)
curl http://127.0.0.1:5000/health
```

**ASP.NET Core / Kestrel** (`launchSettings.json`, `appsettings.json`):

```json
"applicationUrl": "https://localhost:7150;http://localhost:5150"
```

**Npgsql connection string:**

```
Host=db.internal;Port=5432;Database=app;Username=app
```

**Docker port publish:** `-p 8080:80` maps host port 8080 → container port 80 (NAT at host).

## Examples

### Example 1: Two Services, One IP

A VM at `10.0.1.50` runs Kestrel on `:5000` and PostgreSQL on `:5432`. Firewall allows `5000` from load balancer subnet only; `5432` from app subnet only. Same IP — different ports route traffic to different processes via kernel demultiplexing.

### Example 2: Ephemeral Client Port

Browser connects to `93.184.216.34:443`. OS assigns source `192.168.1.105:52441`. Server sees connection from that 5-tuple. Opening a second tab creates `192.168.1.105:52442` → same destination — two independent TCP connections for HTTP/1.1 or parallel HTTP/2 streams on one connection.

## Common Mistakes & Anti-patterns

**Confusing IP with MAC address.** MAC is link layer (local); IP is end-to-end routable.

**Assuming `localhost` is always `127.0.0.1`.** Usually true; IPv6 `::1` also loopback (`0.4.6`).

**Binding to wrong interface.** `127.0.0.1` is unreachable from other machines; `0.0.0.0` listens on all (`0.4.6`).

**Port collision.** Two processes cannot bind same `(IP, port, TCP)` — `EADDRINUSE`.

**Forgetting default ports in URLs.** `http://host` uses 80; `https://host` uses 443 — security groups must allow them.

## Production & Real-World Notes

Cloud **security groups** / **NSGs** filter on IP + port + protocol (Layer 4). **Kubernetes Services** allocate cluster IPs and target ports; **NodePort** maps high host ports. **PgBouncer** listens on 6432 while Postgres stays 5432 — port indirection for pooling.

Document which ports each environment exposes; scan with tools in CI. Never expose PostgreSQL `:5432` publicly without strong auth and network isolation.

---

## Comparison / Trade-offs

**IPv4 vs IPv6:** IPv6 removes NAT complexity long-term and restores end-to-end addressing; dual-stack is common. Developers still see IPv4 in logs and private VPCs daily.

**Hostname vs IP in config:** Hostnames survive IP changes (DNS update); hard-coded IPs brittle except for static infrastructure endpoints.

## Key Takeaways

- IP identifies the host on the network; port identifies the service/process on that host.
- IPv4 is 32-bit dotted decimal; private ranges and NAT extend usability.
- Sockets are the OS programming interface combining address family, IP, port, and protocol.
- URLs from `0.3.2` map to `(resolved IP, port)` before HTTP/TLS bytes flow.
- Connection pooling reuses one TCP socket (one port pair) for multiple HTTP requests.
- Binding and firewall rules are configured in terms of IP + port + protocol.
