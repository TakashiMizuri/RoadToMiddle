# Answers: Code review etiquette

## Scoring

- 90–100%: отлично
- 80–89%: pass (13/16+)
- <80%: повтори лекции

---

## Answers

### 1. [Recall]
**Answer:** Например: **knowledge sharing** (распространение контекста), **enforcement standards** (стиль, patterns), **design improvement** / **shared ownership**.

**Why:** Review — социальный и архитектурный механизм, не только QA.

---

### 2. [Recall]
**Answer:** **Non-blocking** замечание по мелочи (style, naming); не должно блокировать merge.

**Why:** Разделение приоритетов сохраняет velocity.

---

### 3. [Understanding]
**Answer:** В тексте нет tone of voice и body language; персональные атаки читаются жёстче и демотивируют; фокус на code снижает конфликт и держит discussion technical.

**Why:** Async review — письменный контракт между профессионалами.

---

### 4. [Understanding]
**Answer:** **Blocking** — must fix before merge (или explicit team waiver). **Suggestion** — улучшение на усмотрение author; merge возможен без выполнения.

**Why:** Trust зависит от честной маркировки severity.

---

### 5. [Understanding]
**Answer:** **Rubber-stamping** — approve без реального чтения. Опасно: баги и security попадают в `main`, review gate бесполезен.

**Why:** Ложное ощущение безопасности.

---

### 6. [Application]
**Answer:** Например: **[blocking]** Submit остаётся enabled при пустом email — reproduce: clear email, click Submit, request уходит. Expected: disabled until valid per form spec.

**Why:** Конкретные steps + expected behavior actionable.

---

### 7. [Application]
**Answer:** Вежливо объяснить с evidence: «Tracking needed because global soft-delete filter in DbContext requires tracked entity — see `Docs/ef-soft-delete.md`. Happy to add comment in code.»

**Why:** Fix or explain с документацией, не спор о статусе.

---

### 8. [Analysis]
**Answer:** **B** — rubber-stamping на большом auth PR.

**Why:** A, C, D — здоровые practices.

---

### 9. [Analysis]
**Answer:** Прочитать **все** comments; сначала **blocking** + security; batch fix nits где дёшево; отдельно ответить на suggestions с fix или rationale; не спорить по каждому nit в 40 replies.

**Why:** Prioritization и batching экономят время обеих сторон.

---

### 10. [Understanding]
**Answer:** Показывает reviewer'у завершённость; снижает повторные вопросы; очищает mental model PR state.

**Why:** UI state = communication protocol.

---

### 11. [Application]
**Answer:** Сначала **[blocking]** SQL injection с конкретным fix (parameterized query). CSS пробелы — **[nit]** optional, отдельно и ниже по приоритету.

**Why:** Risk-first ordering; не bury critical в style noise.

---

### 12. [Understanding]
**Answer:** **Нет** при branch protection с required checks (`0.2.8`). Approval human gate дополняет CI, не заменяет.

**Why:** Люди не запускают тесты за author'а.

---

### 13. [Understanding]
**Answer:** Долгие споры о **тривиальном** (style, naming), игнорируя серьёзные issues (security, logic).

**Why:** Parkinson's law applied to review — лёгкие темы dominate.

---

### 14. [Application]
**Answer:** Не эскалировать публично; перевести в **PR thread** или DM: «Давай обсудим конкретные строки в PR — готов поправить blocking points.» Involve lead если harassment.

**Why:** Public shaming ломает culture; конкретика в PR — норма.

---

### 15. [Understanding]
**Answer:** Закрепляет **хорошие patterns** для всей команды; балансирует criticism; повышает morale и повторение best practices.

**Why:** Review shapes learning, не только фильтрацию.

---

### 16. [Application]
**Answer:** (примеры) **Security:** webhook signature verification, idempotency, no secret log. **Correctness:** handling duplicate events, error paths. **Maintainability:** tests, clear naming, Conventional follow-ups, docs for ops.

**Why:** Middle reviewer сканирует risk layers, не только style.
