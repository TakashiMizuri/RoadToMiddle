# SemaphoreSlim for Throttling

> Roadmap: `1.4.18` · Node: `1.4` — C# async · Depth: практика

## Learning Objectives

After this lesson you will be able to:

- Use **`SemaphoreSlim`** to limit **concurrent** async operations (not just mutual exclusion).
- Contrast **`SemaphoreSlim`** with **`lock`** / **`Monitor`** for async code.
- Implement **`WaitAsync`** / **`Release`** with **`try/finally`** to avoid permit leaks.
- Build a **throttled `HttpClient`** or DB call pattern with configurable **`MaxConcurrency`**.
- Combine semaphores with **`CancellationToken`** and **`Task.WhenAll`** safely.
- Diagnose **permit leaks** and starvation in production.

---

## Why This Matters

Calling 10,000 external APIs or DB queries **at once** after **`Task.WhenAll`** feels fast locally — until production hits **connection limits**, **429 Too Many Requests**, **thread pool pressure**, and **database max connections**. Unlimited parallelism is a common outage cause.

**`SemaphoreSlim`** is the standard .NET primitive for **"at most N at a time"** in **async** code. Unlike **`lock`**, **`WaitAsync`** doesn't block a thread pool thread while waiting — it registers a continuation when a permit frees.

You'll use this for **bulk imports**, **parallel HTTP fan-out**, **EF concurrent saves limits**, and guarding shared **non-thread-safe** resources. Related **`1.4.33`** covers **`lock` vs SemaphoreSlim`** in depth; this lesson focuses on **hands-on throttling**.

## Core Concepts

### Counting Semaphore Mental Model

**`SemaphoreSlim(initialCount, maxCount)`** maintains a pool of **permits**:

```csharp
var gate = new SemaphoreSlim(maxConcurrency: 10); // 10 permits available
```

- **`WaitAsync()`** — take one permit; if none left, await until **`Release()`**.
- **`Release()`** — return one permit (must not exceed **`maxCount`** — throws **`SemaphoreFullException`**).

Pattern for each operation:

```csharp
await gate.WaitAsync(ct);
try
{
    await DoWorkAsync(ct);
}
finally
{
    gate.Release();
}
```

**`finally`** is non-negotiable — skipped **`Release`** permanently reduces concurrency (leak).

### Throttling Many Tasks

```csharp
var semaphore = new SemaphoreSlim(5);
var tasks = urls.Select(async url =>
{
    await semaphore.WaitAsync(ct);
    try
    {
        return await _http.GetStringAsync(url, ct);
    }
    finally
    {
        semaphore.Release();
    }
});
string[] results = await Task.WhenAll(tasks);
```

All tasks **start**, but only **5** execute HTTP calls concurrently; others await permits.

Alternative: **`Parallel.ForEachAsync`** (.NET 6+) with **`MaxDegreeOfParallelism`** — higher-level; semaphore gives explicit control in custom pipelines.

### SemaphoreSlim vs lock

| | `lock` | `SemaphoreSlim` |
|---|--------|-----------------|
| Async wait | ❌ Blocks thread | ✅ `WaitAsync` |
| Typical use | Short sync critical section | Limit concurrency / async gate |
| Re-entrant | Yes (same thread) | No |
| Cross-process | No | No |

**Never `lock` around `await`** — thread holds lock while suspended → deadlocks and pool starvation.

### InitialCount and maxCount

**`new SemaphoreSlim(0, 1)`** — binary semaphore (mutex-like), starting with **no** permits until **`Release`** — rare pattern.

For throttling: **`new SemaphoreSlim(N, N)`** — **N** concurrent slots from the start.

### Cancellation

**`WaitAsync(CancellationToken)`** — if canceled while waiting, **no permit taken**; don't **`Release`** in that path.

```csharp
if (!await gate.WaitAsync(TimeSpan.FromSeconds(30), ct))
    throw new TimeoutException("Throttled wait timeout");
```

## Syntax / API

| Member | Purpose |
|--------|---------|
| `WaitAsync()` / `WaitAsync(ct)` | Acquire permit async |
| `WaitAsync(timeout)` | Timed wait |
| `Release()` | Return permit |
| `CurrentCount` | Available permits (diagnostics) |
| `Dispose()` | Cleanup (waiting tasks may throw) |

## Examples

### ThrottledDbCaller service

```csharp
public sealed class ThrottledDbCaller
{
    private readonly SemaphoreSlim _dbGate = new(8, 8); // max 8 concurrent queries

    public async Task<T> QueryAsync<T>(Func<Task<T>> query, CancellationToken ct)
    {
        await _dbGate.WaitAsync(ct);
        try
        {
            return await query();
        }
        finally
        {
            _dbGate.Release();
        }
    }
}
```

Inject as singleton; **`SemaphoreSlim`** is **not** thread-safe for **`Dispose`** while waits pending — dispose only on shutdown.

### Rate limit companion

Semaphore limits **concurrency** (parallel in-flight), not **requests per second**. For RPS limits add **`TokenBucket`** or **`RateLimiter`** (.NET 7+ **`System.Threading.RateLimiting`**) — complementary tools.

## Common Mistakes & Anti-patterns

**Missing `finally Release`** — slow leak of permits → eventually all callers block forever.

**Double `Release`** — **`SemaphoreFullException`** or inflated count.

**Using `Wait()` instead of `WaitAsync` in async method** — blocks thread during wait.

**Semaphore per request (new in action)** — defeats global limit; use **singleton** gate.

**Max concurrency >> DB pool size** — still exhausts connections; align **`N`** with **`Max Pool Size`**.

---

## Production & Real-World Notes

Log **`CurrentCount`** periodically or when waits exceed threshold — detect leaks.

Tune **`N`** from **downstream limits** (API docs, DB **`max_connections`**, **`HttpClient`** **`MaxConnectionsPerServer`**).

Combine with **Polly** retry — retries multiply concurrency; account in **`N`** or separate retry semaphore.

On shutdown, cancel token → waiting **`WaitAsync`** throw **`OperationCanceledException`** — don't dispose mid-flight without cancel.

## Key Takeaways

- **`SemaphoreSlim`** = async-friendly concurrency limiter.
- Always **`try/finally`** with **`Release`**.
- Prefer **`WaitAsync`** over **`Wait`** in async code.
- Throttling ≠ rate limiting — know which you need.
- Singleton semaphore for global caps; tune from downstream constraints.
- **`lock` + await`** is forbidden — use semaphore or redesign.
