## Why This Matters

**`1.3.11`**: **`IEnumerable`** = delegates local; **`IQueryable`** = provider remote. Bridge — **expression trees**: EF queries — не functions in pipeline, а **AST** для inspect, rewrite, SQL.

Без модели EF errors кажутся random. С моделью: **nodes без SQL mapping → fail or split**. Debug через **`ToQueryString()`**, refactor predicates, compose **`Expression<Func<>>`** — не string SQL.

## Core Concepts

### Code as data vs instructions

```csharp
Func<User, bool> pred = u => u.Age > 18;
pred(user);  // JIT runs IL

Expression<Func<User, bool>> expr = u => u.Age > 18;
// object graph: parameter u, member Age, constant 18, GreaterThan
```

**`expr`** **describes** predicate; EF walk **`expr.Body`** → **`WHERE "Age" > 18`**.

### Dual nature of lambdas

**`db.Users.Where(u => ...)`** → **`Expression<Func<>>`** (**`Queryable.Where`**).  
**`usersList.Where(u => ...)`** → **`Func<>`** (**`Enumerable.Where`**).

Static type receiver определяет artifact.

### Anatomy of tree

**`u => u.Age > 18`:**

```
LambdaExpression
├── Parameters: [ u ]
└── Body: GreaterThan(MemberAccess u.Age, Constant 18)
```

Node types: **`ParameterExpression`**, **`MemberExpression`**, **`BinaryExpression`**, **`MethodCallExpression`**, …

### Building tree per operator

Each **`Queryable.Where`** → **`MethodCallExpression`** wrapping inner expression. Still no SQL until **`Execute`**.

### Execution — provider as interpreter

**`ToListAsync()`** → EF **`Execute(expression)`** → query compiler → SQL → Npgsql → materialize.

**`CreateQuery`** — extend deferral; **`Execute`** — run now.

### Translatability

| Translatable | Often not |
|--------------|-----------|
| Property compare, `&&` | Local method calls |
| Mapped `string.Contains` | Custom static helpers |

EF Core 3+ throws on bad client eval (default strict). Fix: **`EF.Functions`**, split query, **`AsEnumerable()`** after SQL narrow.

### Manual composition (intro)

```csharp
var param = Expression.Parameter(typeof(User), "u");
var body = Expression.Equal(Expression.Property(param, "IsActive"), Expression.Constant(true));
Expression<Func<User, bool>> lambda = Expression.Lambda<Func<User, bool>>(body, param);
db.Users.Where(lambda);
```

### Compile() — trap

```csharp
expr.Compile();  // → Func — local only, EF translation lost
```

## Under the Hood

**`IQueryProvider`**: **`CreateQuery`**, **`Execute`**.  
EF pipeline: Expression → translation → SQL → command → materialization.  
**`EnumerableQuery`**: compile tree to delegates — tests without DB.

---

## Common Mistakes

- **`Compile()`** on EF-bound expressions
- Local functions in **`IQueryable`** lambdas
- SQL string concat instead of composable **`Expression`**

---

## Key Takeaways

- Trees = **code structure as data** for SQL translation.
- **`IQueryable`** accumulates **`MethodCallExpression`** until terminal.
- Compiler emits **Expression vs Func** by target type.
- EF errors name **untranslatable nodes** — refactor that branch.
- EF chapters assume this model.
