# Keep-Alive и connection pooling

> Roadmap: `0.3.14` · Node: `0.3` — HTTP и Web · Depth: понимание

## Learning Objectives

После этого урока ты сможешь:

- Объяснить **стоимость setup TCP connection** (DNS, handshake, TLS) и зачем reuse.
- Описать **`Connection: keep-alive`** в HTTP/1.1 и отличие от implicit reuse в HTTP/2.
- Определить **connection pooling** на client (`HttpClient`, browser) и server.
- Распознать **pool exhaustion**, **stale connections**, **socket leaks**.
- Настроить **`IHttpClientFactory`** и `SocketsHttpHandler` conceptually в ASP.NET Core.
- Связать pooling с HTTP/2 single-connection multiplexing (`0.3.13`).

## Why This Matters

Каждый HTTP call начинается ниже JSON parsing: DNS, TCP three-way handshake, часто **full TLS**, потом HTTP. На high QPS или chatty API setup доминирует в latency и съедает file descriptors.

**Keep-alive** — несколько HTTP exchanges на одном TCP. **Connection pooling** — warm connections ready вместо close после каждого use. Misconfigured pools → **`SocketException`**, hung requests, **503** при исчерпании connections. Middle debug'ит через metrics и понимает **`HttpClient`** defaults — почему **`new HttpClient()` per request** был anti-pattern до `IHttpClientFactory`.

## Core Concepts

### Cost нового connection

Cold HTTPS: DNS → TCP → TLS → HTTP. **TLS session resumption** короче, но хуже reuse established connection. Localhost скрывает проблему; 200 ms RTT — больно.

### HTTP/1.1 Keep-Alive

HTTP/1.1 **default persistent** unless `Connection: close`. Sequential requests на **same socket** без repeat handshake. Limit: **one in-flight request per connection** — browsers **~6 persistent connections** per host. Server: **`Keep-Alive: timeout, max`**.

Idle connections — memory и FDs; server enforce max idle timeout.

### HTTP/2 и reuse

Many streams на **one TCP** — aggressive reuse. Extra TCP per host discouraged. Single connection stall → need replacement; pool health checks.

### Client-side pooling

Pool holds open sockets по **destination** (scheme+host+port). Borrow → request → return или discard.

**.NET `SocketsHttpHandler`:** `PooledConnectionLifetime`, `PooledConnectionIdleTimeout`, `MaxConnectionsPerServer`. **`IHttpClientFactory`** — handler lifetime, **DNS refresh** при rotate (~2 min default).

Anti-pattern: **`new HttpClient()` в loop** — socket exhaustion, **`TIME_WAIT`**.

Browser pools per origin — DevTools "Queueing" при saturation.

### Server-side

**Kestrel:** `MaxConcurrentConnections`, `KeepAliveTimeout`. **nginx** `keepalive` к upstream — отдельные pools client↔nginx и nginx↔Kestrel.

**PgBouncer/Npgsql** — тот же pattern на DB layer.

### Stale connections

NAT timeout, server restart, LB idle cutoff → **connection reset**. Pools retry on fresh conn (GET safe). **`PooledConnectionLifetime`** rotates proactively.

## Under the Hood

**`TIME_WAIT`** при close — ephemeral port exhaustion. Keep-alive avoids per-request close.

HTTP/1.1 parser читает full response before next on same socket.

Separate **`HttpClient` instances** — separate pools. HTTP/2 **GOAWAY** — drain и new conn.

**`ulimit -n`**, container limits — production incidents.

## Syntax / Commands / API

```bash
curl -v https://httpbin.org/get 2>&1 | findstr /i "Connection keep-alive"
```

```csharp
builder.Services.AddHttpClient("payments", c =>
    c.BaseAddress = new Uri("https://payments.internal/"))
.ConfigurePrimaryHttpMessageHandler(() => new SocketsHttpHandler
{
    PooledConnectionLifetime = TimeSpan.FromMinutes(5),
    MaxConnectionsPerServer = 20
});
```

## Examples

Microservice fan-out: 3 calls без pool = 3× TLS; shared handler reuse. Load test: `new HttpClient()` per iter → socket errors; fix factory. Rolling deploy: `PooledConnectionLifetime` → fresh DNS.

---

## Common Mistakes & Anti-patterns

`new HttpClient()` per request. Infinite pool. No timeouts. nginx без upstream keepalive. HTTP/2 не снимает все pool concerns. Лишний `Connection: close`.

---

## Production & Real-World Notes

Metrics: active connections, queue time, OpenTelemetry. Tune HTTP + DB pools together. Lambda — короткий lifetime. Corporate proxies.

---

## Comparison / Trade-offs

| Approach | Pros | Cons |
|----------|------|------|
| New conn each time | Simple | Latency, ports |
| HTTP/1.1 keep-alive | Reuse TCP | Limited parallel/conn |
| HTTP/2 one conn | Many streams | Single failure impact |
| Pool + lifetime | Reuse + DNS refresh | Tuning |

## Quick Reference

```
Cost = DNS + TCP + TLS
Keep-alive: reuse TCP (HTTP/1.1 default)
Pool: IHttpClientFactory, SooledConnectionLifetime
Symptoms: SocketException, TIME_WAIT, slow under load
```

---

## Key Takeaways

1. TCP+TLS expensive — reuse.
2. HTTP/1.1 keep-alive default; ~6 conns/host.
3. HTTP/2 — few long-lived conns, many streams.
4. Shared `HttpClient`, not per request.
5. Factory → DNS refresh.
6. Server/proxy limits and timeouts.
7. Misconfig → latency spikes, socket errors.
