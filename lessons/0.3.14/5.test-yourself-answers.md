## Answers

### 1. [Recall]
**Answer:** **Persistent TCP connection** для нескольких HTTP request/response без close после каждого exchange.
**Why:** HTTP/1.1 default unless `Connection: close`.

### 2. [Recall]
**Answer:** Каждый instance свой handler/pool; sockets не reuse эффективно → **socket exhaustion**, **`TIME_WAIT`**, DNS не refresh.
**Why:** `HttpClient` disposable but underlying sockets linger; factory shares handler.

### 3. [Understanding]
**Answer:** Например: **DNS resolve**, **TCP handshake**, **TLS handshake** (then HTTP request).
**Why:** Cold path cost before application data.

### 4. [Understanding]
**Answer:** **Pool** управляет **набором** connections per destination (borrow/return, limits, lifetime); один keep-alive socket — single reuse path without queue/limit abstraction.
**Why:** Pool scales concurrent requests to same host.

### 5. [Understanding]
**Answer:** **Proactive rotation** connections/handlers — refresh **DNS**, drop connections before NAT/middlebox kill, pick up new server IPs after deploy.
**Why:** Infinite lifetime caches stale topology.

### 6. [Application]
**Answer:** **`MaxConnectionsPerServer`** на `SocketsHttpHandler`.
**Why:** Caps parallel connections per host:port.

### 7. [Application]
**Answer:** Handler recycle after **`PooledConnectionLifetime`** → new connections resolve DNS → new pod IPs.
**Why:** Old TCP tied to old IP until rotated.

### 8. [Analysis]
**Answer:** **Closing connection after each request** (no keep-alive) или слишком aggressive close → many sockets in **TIME_WAIT**.
**Why:** Churn vs reuse trade-off.

### 9. [Analysis]
**Answer:** **Нет полностью** — still need managed long-lived conn(s), handle GOAWAY, limits, stale detection; pool abstraction remains.
**Why:** Multiplexing ≠ eliminate connection management.

### 10. [Recall]
**Answer:** Registers typed/named **`HttpClient`**, manages **`HttpMessageHandler`** lifetime, avoids socket leaks, centralizes config.
**Why:** DI-friendly pattern for outgoing HTTP.

### 11. [Understanding]
**Answer:** Server/LB/NAT closed idle socket; pool не знает до следующего write/read → **reset**.
**Why:** Idle pooled conn looks valid locally.

### 12. [Application]
**Answer:** **Частично теряется** — nginx opens new TCP to Kestrel per request without upstream keepalive; client pool only helps to nginx.
**Why:** Weakest link in chain.

### 13. [Analysis]
**Answer:** **POST payment** retry risky — может **duplicate charge** if first request succeeded but client saw reset; need idempotency keys, not blind retry.
**Why:** Non-idempotent ops vs safe GET.

### 14. [Understanding]
**Answer:** Мало parallel connections → **bundle** many assets into fewer files to reduce requests.
**Why:** HTTP/1.1 parallelism limit drove bundling (`0.3.13` link).
