## Answers

### 1. [Recall]
**Answer:** Switch **expression** — `value switch { pat => result, ... }` — это **expression**, возвращает значение; arms через `=>`. Switch **statement** — `switch (value) { case pat: ... break; }` — выполняет statements, нужен `break`/`goto`.
**Why:** Expressions для pure mapping; statements для imperatives и side effects.

### 2. [Recall]
**Answer:** `_` — discard, matches anything not caught earlier; делает switch exhaustive.
**Why:** Без `_` компилятор требует покрыть все случаи (для bool, enum с настройками и т.д.).

### 3. [Recall]
**Answer:** `{ Total: > 1000 }` или внутри larger property pattern `{ Total: > 1000, ... }`.
**Why:** Property name + nested pattern (relational `> 1000`).

### 4. [Understanding]
**Answer:** Сработает **первая** matching arm. `>= 0` первым поглотит все int ≥ 0, включая 90+ — более специфичные arms никогда не выполнятся.
**Common mistake:** Думать, что компилятор «выберет лучший» match — это first-match semantics.

### 5. [Understanding]
**Answer:** Type pattern делает **одну** runtime check + bind; переменная `s` уже typed и в scope только где match успешен. Старый стиль — двойная работа (as + null + возможно повторный cast).
**Why:** Меньше boilerplate, меньше риска забыть null.

### 6. [Understanding]
**Answer:** Guard уместен для **внешних** условий (Parse, permissions), которые не выражаются pattern syntax. Если условие — часть **структуры** (`> 1000`), лучше relational/property pattern без `when`.
**Why:** Дублирование логики в pattern и guard усложняет review.

### 7. [Application]
**Answer:** `big-int`
**Why:** `x` — `int` 42; arm `int n and > 40` совпадает раньше, чем общий `int n`.

### 8. [Application]
**Answer:** `on Y axis`
**Why:** Positional pattern `(0, _)` — X=0, Y любой (включая 5); `(0,0)` не подходит.

### 9. [Application]
**Answer:** `"delete"`
**Why:** List pattern `["delete", var id]` + guard `int.TryParse` для `"42"` успешен.

### 10. [Analysis]
**Answer:** Switch **не exhaustive** для невалидного enum value `(Color)99` — **runtime exception** `SwitchExpressionException` (или fall-through отсутствует). Нужен `_ => ...` arm.
**Common mistake:** Думать, что enum switch всегда безопасен при cast из int/API.

### 11. [Analysis]
**Answer:** Зависит от контекста; **B** centralizes classification но теряет per-branch logic. Если дальше разная обработка для string vs int — **A** или switch **statement** с разными телами лучше. **B** хорош, если нужен только bool flag.
**Why:** Switch expression не заменяет разную imperative logic без последующего второго switch.

### 12. [Application]
**Answer:**
```csharp
nums switch
{
    [] => "empty",
    [var _] => "single",
    [var _, var __] => "pair",
    [..] => "many"
};
```
**Why:** List patterns по длине; `[..]` — любой непустой с 3+ (после более специфичных arms) или явно `[_, _, ..]` для exactly 3+.

### 13. [Understanding]
**Answer:** **Count/Length ≥ 1** (минимум один элемент для head), indexer `[0]` для head, остаток через slice/rest — без LINQ, прямые проверки длины и индексов.
**Why:** Compiler lowers to length + indexer IL.

**Answer:** Любые три: property, relational, list, constant, discard, positional, logical (and/or/not).

### 15. [Application]
**Answer:**
```csharp
ex switch
{
    ValidationException => "validation",
    KeyNotFoundException => "not-found",
    _ => "error"
};
```
**Why:** Type patterns; порядок специфичных до общего `_`.
