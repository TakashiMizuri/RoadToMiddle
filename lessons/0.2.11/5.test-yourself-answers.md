## Answers

### 1. [Recall]
**Answer:** Создаёт **новый** commit на текущей ветке с тем же patch; оригинальный commit не перемещается.
**Why:** Cherry-pick replay через merge machinery; parent нового commit — текущий HEAD, hash другой.

### 2. [Recall]
**Answer:** Добавляет в message строку `(cherry picked from commit <hash>)` для traceability backport'ов.
**Why:** Audit и release management — grep по источнику fix.

### 3. [Understanding]
**Answer:** Merge втянет **все** commits feature, включая незавершённую работу; cherry-pick переносит **только** fix.
**Why:** Hotfix требует минимального blast radius и быстрого ship.
**Common mistake:** Merge «потому что так проще» — риск регрессий.

### 4. [Understanding]
**Answer:** Revert добавляет **новый** commit-отмену; history остаётся линейной для pull без force. Reset + force переписывает remote history.
**Why:** Коллеги с bad commit в локальном clone не ломаются при revert.

### 5. [Understanding]
**Answer:** Возникнет **merge conflict**; нужно resolve как в `0.2.5`, затем `git cherry-pick --continue`.
**Why:** Patch применяется на другое tree, чем original parent.

### 6. [Application]
**Answer:** `git cherry-pick -x deadbeef` (на hotfix-ветке от актуального `main`).
**Why:** `-x` для audit; ветку hotfix создают отдельно до pick.

### 7. [Application]
**Answer:** `git revert bad99 --no-edit` затем `git push origin main`.
**Why:** Publish revert commit, не rewrite.

### 8. [Application]
**Answer:** `git cherry-pick --abort`.
**Why:** Сбрасывает CHERRY_PICK_HEAD и состояние до начала pick.

### 9. [Analysis]
**Answer:** Примеры: duplicate commits с разными hash; сложнее merge/rebase позже; шумный log; риск пропустить/дублировать commits.
**Why:** Cherry-pick для **изолированных** commits, не замена merge всей ветки.

### 10. [Analysis]
**Answer:** **Неверно** — bad99 остаётся в log; добавляется revert commit **после** него.
**Why:** Revert не удаляет objects из graph.

### 11. [Application]
**Answer:** `-n` / `--no-commit`.
**Why:** Применяет изменения без commit — можно править перед `git commit`.

### 12. [Understanding]
**Answer:** У merge commit **два parents**; Git не знает, какую линию «отменять» без `-m <parent-number>`.
**Why:** Revert строит inverse относительно выбранного mainline parent.

### 13. [Application]
**Answer:** `git cherry-pick -x <fix-commit-hash>` (или hash merge commit с `-x`, если так принято в команде).
**Why:** Тот же patch на LTS без merge feature.

### 14. [Analysis]
**Answer:** **Revert** — bad deploy уже на shared `main`, нужен быстрый rollback. **Cherry-pick** — good fix уже есть на другой ветке, нужно **добавить** на `main`/LTS.
**Why:** Revert убирает эффект; cherry-pick добавляет эффект.

### 15. [Recall]
**Answer:** **Author** и author date сохраняются; **committer** — тот, кто выполнил pick (ты), commit date — сейчас.
**Why:** Разделение «кто написал» vs «кто backport'нул».

**Answer:** c → d → a → f → b → e  
**Why:** Актуальный main → hotfix branch → pick → PR → merge+tag → backport LTS.

### P2. Choose the tool
1. **reset --hard** (локально, unpushed — из `0.1.8`; не revert если не shared)  
2. **cherry-pick**  
3. **revert**  
4. **merge**
