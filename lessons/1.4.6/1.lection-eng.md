## Why This Matters

You write readable linear code:

```csharp
var json = await httpClient.GetStringAsync(url);
var doc = JsonDocument.Parse(json);
```

The compiler rewrites this into a **state machine class** with fields for locals, **`awaiter`**, and **state integer**. Understanding the rewrite explains: why you cannot **`await`** in certain contexts (lock, ctor), why **`async void`** is special, why exceptions propagate on **`await`**, and why **no extra thread** exists during I/O wait (`1.4.2`).

Without this mental model, async remains syntax sugar mystery — Middle developers inspect **IL / generated code** in SharpLab or dotnet disasm.

## Core Concepts

### What the Compiler Generates

An **`async Task<int> FooAsync()`** method becomes:

1. A **struct or class** implementing **`IAsyncStateMachine`** (often nested private).
2. **`MoveNext()`** containing switch on **state** — body split at each **`await`**.
3. **`AsyncTaskMethodBuilder<int>`** to create/set **`Task`** returned to caller.
4. Original method becomes **stub** that initializes state machine and calls **`Start(ref stateMachine)`**.

Caller receives **`Task`** immediately; **`MoveNext`** runs until first **incomplete** await, then **returns**; later **continuation** calls **`MoveNext`** again at next state.

### await Decomposition

```csharp
await expr;
// roughly:
var awaiter = expr.GetAwaiter();
if (!awaiter.IsCompleted)
{
    state = 1; // next state
    builder.AwaitUnsafeOnCompleted(ref awaiter, ref stateMachine);
    return; // async method returns; thread free
}
// continue synchronously if already completed
result = awaiter.GetResult();
```

**`GetAwaiter()`** / **`IsCompleted`** / **`GetResult()`** — **awaitable pattern** (`1.4.23`).

### async Task vs async void

**`async Task`**: builder exposes **`.Task`** to caller — observable, awaitable.

**`async void`**: **`AsyncVoidMethodBuilder`** — no Task; exceptions crash process/event handler (`1.4.13`). Only event handlers.

### Exception Path

Throw before await → builder **`SetException`**. Throw after await → captured in **`GetResult()`**, builder notified. **`await`** in caller unwraps.

## Under the Hood

### State Machine Fields

Compiler hoists **locals** used across awaits into fields. **`__state`**, **`__builder`**, **`__awaiter1`**, **`__u1`** (user variables). Large state machines → **heap allocation** (class) vs **struct** optimization when escape analysis allows.

### MoveNext Switch

```csharp
switch (__state)
{
    case -1: // start
        // code before first await
        goto case 0;
    case 0:
        // resume after first await
        break;
}
```

Each **`await`** boundary = new case label.

### Continuation Registration

**`AwaitUnsafeOnCompleted`** registers **`MoveNext`** as continuation on **`Task`**. When I/O completes, thread pool runs continuation → **`MoveNext`** resumes — **may be different thread** (`1.4.1`).

**No thread blocked** during wait — thread runs other work.

### AsyncMethodBuilder

**`AsyncTaskMethodBuilder`** creates **`Task`**, **`SetResult`/`SetException`**, **`AwaitOnCompleted`**. Custom builders enable **`ValueTask`** pooling (`1.4.15`).

Paste async method at [sharplab.io](https://sharplab.io) — view generated **`<>c__DisplayClass`** / **`d__`** state machine type.

## Syntax / Commands / API

```csharp
public async Task<string> FetchAsync(HttpClient http)
{
    var response = await http.GetAsync("https://example.com");
    response.EnsureSuccessStatusCode();
    return await response.Content.ReadAsStringAsync();
}

// Compiler produces roughly:
// struct FetchAsyncd__X : IAsyncStateMachine { void MoveNext() { ... } }
```

**Disable async** — none; always compiler transform in C# 5+.

## Examples

### Example 1: Two Awaits — Three States

Start → await HTTP → await ReadContent → return. Two suspension points; **`MoveNext`** invoked up to three times.

### Example 2: Synchronous Completion

**`await Task.FromResult(1)`** — **`IsCompleted` true** — no suspension; entire method runs synchronously on caller thread until first incomplete await (or end).

### Example 3: try/catch Around await

Compiler emits try/finally blocks inside **`MoveNext`** preserving **`try/catch`** semantics across states.

## Common Mistakes & Anti-patterns

**Thinking async creates a thread** — it registers continuations; I/O wait has no dedicated thread.

**Huge async methods** — giant state machine, harder debugging, allocation.

**Capturing large state** in many awaits — all hoisted to fields — GC pressure.

**await inside lock** — **`lock` not async-compatible** — deadlock risk (`1.4.33`).

## Production & Real-World Notes

Stack traces show **`MoveNext`** — normal; use **Portable PDB** / **async stack traces** improvements in .NET for readability (`1.4.37`).

Performance: **`ValueTask`** when often synchronous complete (`1.4.15`).

Analyzer CS1998 — async without await — useless state machine.

## Comparison / Trade-offs

| You write | Runtime sees |
|-----------|--------------|
| `await` | Continuation + state transition |
| `async Task` | Task returned immediately |
| Sequential awaits | Sequential states, not parallel |
| `Task.WhenAll` | Single await on combined task |

Parallel I/O: **`await Task.WhenAll(a,b)`** — one suspension for both (`1.4.12`).

- **`async`/`await`** = compiler state machine + Task builder.
- **`MoveNext`** runs until next incomplete **`await`**, then returns.
- I/O wait ** frees thread** via continuation model.
