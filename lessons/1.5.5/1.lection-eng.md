# Heap Allocation, Object Headers, and Alignment

> Roadmap: `1.5.5` · Node: `1.5` — .NET Runtime and Memory · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Describe **object header** and **MethodTable pointer** layout on 64-bit CLR.
- Explain **alignment** and padding for fields in objects and structs.

## Why This Matters

In `1.1.6`, reference types live on the **managed heap**. Every `new Customer()`, every `string` concatenation that allocates, every LINQ chain creating iterators adds pressure. The heap is not a simple malloc pool — objects carry **headers**, **type pointers**, and **alignment padding** the GC uses to track type, sync blocks, and compaction (where applicable).

ASP.NET Core under load allocates constantly — request buffers, JSON nodes, EF entities. Middle developers reduce allocations (`ArrayPool`, `Span`, object pooling) because **allocation rate drives Gen0 frequency** (`1.5.7`), not because a single `new` is "slow" in isolation.

Understanding headers explains **memory overhead** beyond field sizes and why **false sharing** and **cache lines** matter at scale.

## Core Concepts

### When Does the Heap Grow?

The CLR allocates reference type objects and **boxed value types** on the **managed heap** (small object heap by default; large objects on LOH — `1.5.8`). Triggers include:

- `new` on **class**, **array**, **string**, **delegate**.
- **Boxing** — storing value type as `object` or interface.
- Runtime internals — exceptions, async state machines, reflection caches.

**Value types** embedded in objects occupy space **inside** the object's payload; standalone locals stay on stack (`1.5.4`).

Allocation path (simplified): **allocate pointer bump** in SOH arena → if full, trigger **Gen0 collection** → if still failing, escalate or OOM.

### Object Layout on 64-bit

Every heap object starts with:

1. **Object header** (8 bytes on 64-bit) — sync block index, hash code cache flags (implementation details vary by CLR version).
2. **MethodTable pointer** — points to type's **MethodTable** (vtable, GC desc, type info).

Then **instance fields** follow with alignment padding.

```
┌──────────────────┬──────────────────┬─────────┬─────────┐
│ Object header    │ MethodTable ptr  │ Field1  │ Field2  │ ...
│ (8 bytes)        │ (8 bytes)        │         │ + pad   │
└──────────────────┴──────────────────┴─────────┴─────────┘
```

Minimum object size is **24 bytes** on 64-bit CLR for empty `class` instance (header + MT + alignment) — empty `new object()` is not free in memory footprint.

**Arrays** add **length** (+ optional multi-dim bounds) after header before elements.

### MethodTable and Type Identity

The **MethodTable pointer** identifies runtime type for casts, virtual dispatch, and GC **exact scanning** of reference fields. Cast checks compare MethodTable hierarchy.

Static fields live on **type-associated memory**, not per instance — one copy per AppDomain/load context.

### Alignment and Padding

CLR lays out fields to satisfy **alignment rules** — typically reference and `long` on 8-byte boundaries, `int` on 4-byte. Struct layout can reorder with `[StructLayout(LayoutKind.Auto)]` for classes; **`StructLayout.Explicit`** manual for interop.

```csharp
class PaddingExample {
    byte a;      // 1 byte
    // 7 bytes padding
    long b;      // 8 at offset 8
    byte c;      // 1
    // 7 padding → instance may be 24 bytes fields + header
}
```

**Pack** attribute reduces padding at cost of unaligned access — interop scenarios.

### Allocation vs Collection

**Allocation** is often a **pointer bump** — very fast if Gen0 has space. **GC** cost comes when generations fill — mark, sweep, compact (`1.5.6`–`1.5.8`). High **allocation rate** → frequent Gen0 → CPU and pause (even if short in workstation GC; server GC behaves differently).

Micro-optimization without measurement misses the point; **allocation profiling** (`dotnet-gcdump`, ETW) finds hot types.

## Under the Hood

### SOH Arenas and Allocation Pointer

Each heap generation uses **allocation contexts** (per thread caches in modern GC) — thread-local bump pointer reduces contention. On fill, **GC trigger** or steal from global heap.

**Pinned objects** (GCHandle) block compaction in affected generations — fragment heap over time.

### Sync Block and Thin Lock

Object header ties to **sync block table** for `lock (obj)` — fat lock promotion on contention. Hash codes for `GetHashCode` without override may use header bits — rare edge cases in identity.

### GC Descriptors in MethodTable

MethodTable references **GC info** describing which instance offsets are pointers — GC marks traverse object graphs without scanning arbitrary bytes as references (unlike conservative C heaps).

## Examples

### Hidden Allocations

```csharp
string s = "a" + "b" + "c"; // multiple string allocations unless optimized
foreach (var x in Enumerable.Range(0, 1000).Where(n => n % 2 == 0))
    ; // delegate + enumerator allocations
```

### Array vs List

`new int[1000]` — one array object + header + length + 1000×4 bytes inline.  
`List<int>` — array + list object overhead — extra indirection, useful growth semantics.

## Common Mistakes & Anti-patterns

**Assuming only `new` matters** — boxing, closures capturing locals, string ops allocate silently.

**Huge object graphs per request without disposal** — not leak if unreachable, but GC churn hurts latency.

**Ignoring struct padding in hot structs** — reorder fields largest-first to shrink size (`1.1.6` value types on stack copy size matters).

**Measuring empty object as 0 bytes** — header overhead dominates small DTOs.

## Production & Real-World Notes

**Object pooling** (`Microsoft.Extensions.ObjectPool`) for ASP.NET Core — reuse `StringBuilder`, byte buffers.

**System.Text.Json** source generators reduce reflection allocations.

**EF Core** tracking allocates identity maps — `AsNoTracking` for read-only queries reduces heap growth per request.

**Server GC** tolerates larger heaps; still minimize short-lived garbage for tail latency.

## Comparison / Trade-offs

| Choice | Heap impact | Notes |
|--------|-------------|-------|
| class instance | Header + fields | Reference type semantics |
| struct | Embedded or stack | Copy cost vs heap object overhead |
| ArrayPool rent | Reuse arrays | Manual return; no zero-init guarantee on rent |
| stackalloc | Stack not heap | Bounded size (`1.5.4`) |

## Key Takeaways

- Reference objects = **header + MethodTable + fields (+ padding)**.
- **Minimum size** makes tiny objects expensive in memory.
- **Allocation is cheap** until GC triggers from volume.
- **MethodTable** enables dispatch and precise GC scanning.
- Links **`1.1.6` heap**, **`1.5.6` marking**, **`1.5.7` Gen0**.
