# Remote: origin, fetch, pull, push — краткая выжимка

> `0.2.6` · ~8 мин на повторение

## Главная идея

**Remote** — именованная ссылка (обычно **`origin`**) на другой Git-репозиторий. **`fetch`** скачивает history и обновляет **`origin/*`** без изменения локальных веток; **`pull`** = fetch + merge в текущую ветку; **`push`** публикует локальные commits на сервер.

## Ключевые пункты

- **`origin`** — конвенция от `git clone`, не обязательное имя, но стандарт в командах и CI.
- **Remote-tracking branch** (`origin/main`) — снимок remote при last fetch; не коммитят напрямую.
- **`git fetch`** — безопасное обновление карты remote; working tree не трогает.
- **`git pull`** — может создать merge commit или конфликт (как в `0.2.2`, `0.2.5`).
- **`git push`** — только **committed** history; отклоняется при non-fast-forward.
- Перед push на shared branch: **fetch → merge/rebase → push**.

## Самые важные примеры

**Fetch-first sync:**
```bash
git fetch origin
git log main..origin/main --oneline   # что пришло с сервера?
git merge origin/main                 # или pull
```

**Публикация feature:**
```bash
git push -u origin feature/health     # -u → upstream (0.2.7)
```

**Отклонённый push:**
```bash
git fetch origin
git merge origin/feature/shared       # или rebase (0.2.4)
git push origin feature/shared
```

## «Под капотом» в одном абзаце

Fetch/push обмениваются **packfile** content-addressed objects по HTTPS/SSH. Fetch пишет `.git/refs/remotes/origin/*`; push просит сервер сдвинуть `refs/heads/...` при fast-forward. Pull после fetch вызывает ту же merge machinery, что `git merge origin/main`.

## Частые ошибки

- Путать **fetch** (только скачать) и **pull** (скачать + влить).
- Push без fetch на ветку, куда коммитили другие.
- Считать `origin/main` локальной рабочей веткой (detached HEAD при checkout).
- Ожидать, что push отправит unstaged изменения.

## Мнемоника

> **Fetch = карта. Pull = карта + слияние. Push = отправить свои коммиты.**
