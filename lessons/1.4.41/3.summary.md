# Sync-over-async fixes — краткая выжимка

> `1.4.41` · ~7 мин на повторение

## Главная идея

**Sync-over-async** (`.Result`/`.Wait()`) и **async-over-sync** (`Task.Run`+sync I/O) вредят scalability. **Лучший fix** — refactor **async all-the-way**. **Task.Run** — временный мост при низком QPS.

## Ключевые пункты

- Sync-over-async: sync метод блокирует thread на async I/O.
- Async-over-sync: `async` wrapper вокруг sync — fake async, лишний pool thread.
- ASP.NET Core: deadlock реже (нет SyncContext), **starvation** от `.Result` реальна.
- Dual API: `*Async` + `[Obsolete]` sync shim для migration.
- Bounded `SemaphoreSlim` + `.Result` — last resort с cap.
- Не вызывать blocking legacy с hot request path без изоляции.

## Comparison (коротко)

| Подход | Когда |
|--------|-------|
| Async refactor | Контролируешь stack |
| Task.Run bridge | Short-term, low traffic |
| .Result в request | Избегать |
| Dual API | Постепенная миграция |

## Частые ошибки

- «.Result безопасен в Core»
- Task.Run на high-QPS API
- Удалить sync shim до миграции consumers

## Мнемоника

> **Await the stack, don't block the pool** — await по стеку, не блокируй pool.
