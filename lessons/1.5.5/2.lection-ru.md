# Heap allocation, object headers, alignment

> Roadmap: `1.5.5` · Depth: **глубоко**

## Learning Objectives

- Объяснить heap allocation: `new`, boxing, runtime internals.
- Описать **object header** + **MethodTable pointer** на 64-bit.
- Объяснить **alignment** и padding.
- Связать с Gen0 (`1.5.7`) и LOH (`1.5.8`).
- Различить cost **allocation** vs **GC collection**.

---

## Why This Matters

`1.1.6` — reference types на **managed heap**. Каждый `new`, boxing, LINQ iterators — pressure. Objects несут **headers** и **padding** для GC и sync.

High **allocation rate** → частый Gen0 — не потому что один `new` «медленный», а из-за объёма garbage.

---

## Core Concepts

### Когда растёт Heap?

`new` class/array/string/delegate, **boxing**, runtime objects (exceptions, async machines).

Value types embedded in objects — inside payload; standalone locals — stack (`1.5.4`).

### Layout 64-bit

1. **Object header** (~8 bytes) — sync/hash flags.
2. **MethodTable pointer** — type identity, vtable, GC desc.
3. **Instance fields** + alignment padding.

Minimum empty `class` ~**24 bytes** on 64-bit.

Arrays: length + elements. Static fields — per-type storage, not per instance.

### Allocation vs GC

Allocation often **pointer bump** in Gen0 — fast until collection. GC cost when generations fill (`1.5.6`–`1.5.8`).

---

## Under the Hood

**Allocation contexts** per thread — bump pointer. **Pinned** objects block compaction.

MethodTable **GC descriptors** — exact pointer scanning.

---

## Key Takeaways

- Header + MethodTable + fields (+ pad).
- Tiny objects — overhead significant.
- MethodTable — dispatch + GC.
- Volume drives Gen0, not single alloc.
