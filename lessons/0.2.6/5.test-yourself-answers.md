## Answers

### 1. [Recall]
**Answer:** **Remote** — именованная ссылка на другой Git-репозиторий (обычно URL на сервере). При `git clone` по умолчанию создаётся remote **`origin`**.

**Why:** Remote — это bookmark для синхронизации; локальный `.git` остаётся полноценным repo, а `origin` указывает на peer для fetch/push.

---

### 2. [Recall]
**Answer:** `origin/main` — **remote-tracking branch**: read-only снимок того, где `main` был на remote при last fetch. Локальный `main` — твоя рабочая ветка; на неё коммитят и merge/rebase.

**Why:** Разделение позволяет сравнивать «у меня» vs «на сервере (по last fetch)» без автоматического слияния.

---

### 3. [Understanding]
**Answer:** `fetch` только скачивает objects и обновляет `origin/*`, **не** меняя текущую ветку и working tree. `pull` сразу интегрирует (merge/rebase) — возможны merge commits и конфликты.

**Why:** Fetch даёт время изучить `git log`/`git diff` до изменения кода.

**Common mistake:** Думать, что после fetch локальный `main` уже обновлён.

---

### 4. [Understanding]
**Answer:** Локальный `main` отстаёт на 3 коммита от **кэшированного** `origin/main` (после последнего fetch). Git сравнивает refs локальной ветки и remote-tracking branch — без live network call.

**Why:** Если давно не fetch'ил, цифра может быть неактуальной относительно реального сервера.

---

### 5. [Understanding]
**Answer:** `git fetch` (скачать) + **`git merge`** tracked remote branch (по умолчанию) в текущую локальную ветку.

**Why:** `pull --rebase` меняет второй шаг на rebase (`0.2.7`).

---

### 6. [Application]
**Answer:**
```bash
git fetch origin
git log HEAD..origin/feature/api --oneline
# опционально: git diff HEAD..origin/feature/api
```

**Why:** Только fetch + inspect; не делать merge/rebase/pull.

---

### 7. [Application]
**Answer:**
1. `git fetch origin`
2. Посмотреть чужие коммиты: `git log HEAD..origin/feature/login`
3. Интегрировать: `git merge origin/feature/login` или `git rebase origin/feature/login`
4. Разрешить конфликты если есть (`0.2.5`)
5. `git push origin feature/login`

**Why:** Non-fast-forward значит remote tip впереди или истории разошлись; нужно объединить перед push.

---

### 8. [Analysis]
**Answer:** **C** — неверно.

**Why:** При расхождении веток `git pull` (merge mode) **создаёт** merge commit. A, B, D — верны.

---

### 9. [Analysis]
**Answer:** (1) Неожиданные **merge commits** и сложный history без осознанного выбора merge vs rebase. (2) **Конфликты** в working tree без подготовки; возможна потеря контекста, что именно влилось.

**Why:** Pull скрывает два шага; без review легко пропустить проблемные коммиты на `main`.

---

### 10. [Understanding]
**Answer:** В конфигурации локального repo: `.git/config` (секция `[remote "origin"]`) и после fetch — refs в `.git/refs/remotes/origin/`.

**Why:** Remote — локальная конфигурация + обновляемые remote-tracking refs.

---

### 11. [Application]
**Answer:** `git push -u origin fix/typo` (или `git push origin fix/typo` — создаст ветку на сервере; `-u` настраивает upstream для удобства).

**Why:** Перый push публикует ветку; `-u` связывает локальную с remote (`0.2.7`).

---

### 12. [Understanding]
**Answer:** Когда локальный `main` — **прямой предок** `origin/main` (ты только отстаёшь, своих уникальных коммитов на `main` нет). Тогда merge = fast-forward.

**Why:** Fast-forward возможен только без diverging commits на обеих сторонах.
