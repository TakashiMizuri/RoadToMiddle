# HTTP Headers: Content-Type, Accept, Authorization, Cookie, Cache-Control, ETag

> Roadmap: `0.3.7` · Node: `0.3` — HTTP и Web · Depth: **глубоко**

## Learning Objectives

После этой лекции ты сможешь:

- Объяснить headers как **metadata** к start line и body из `0.3.3`.
- Согласовывать format representation через **Content-Type** и **Accept** (content negotiation).
- Передавать credentials через **Authorization** (Bearer, Basic), не путая с **Cookie**.
- Описать директивы **Cache-Control** и связь с **304** из `0.3.6`.
- Использовать **ETag** и conditional headers (`If-None-Match`) для revalidation.
- Предсказывать поведение browser vs API client в ASP.NET Core и fetch.

---

## Why This Matters

Status codes (`0.3.6`) отвечают, *удался ли* request. Headers отвечают, *как интерпретировать* сообщение, *кто вызывает*, *как долго кэшировать*, *в каком format* body. JSON API без `Content-Type: application/json` работает, пока strict client или proxy не отклонит. Browser с `Accept: text/html` к API может получить HTML error page вместо JSON. Cache misconfiguration отдаёт stale prices или **leaks private data** в shared CDN.

Fullstack developer настраивает headers ежедневно — controllers, middleware, nginx, CloudFront, frontend fetch. Middle-level work требует знать не только *имена*, но **directive semantics**, **security**, **interaction** с status codes и methods (`0.3.4`–`0.3.5`).

## Core Concepts

### Headers: Case-Insensitive Key–Value Metadata

HTTP headers — строки `Name: value` после start line. Имена case-insensitive. Значения с parameters: `application/json; charset=utf-8`. Request headers, response headers, entity headers.

Browser JavaScript **forbidden/restricted** headers (Host и др.). Server — `[FromHeader]`, middleware.

### Content-Type: What the Body Is

**Content-Type** — media type **body**. Server на response; client на request с body (POST/PUT/PATCH).

- `application/json` — JSON APIs.
- `application/x-www-form-urlencoded` — HTML forms.
- `multipart/form-data` — uploads (`0.3.11`).

Wrong Content-Type → **415** или **400**. ASP.NET `[Consumes("application/json")]`.

Charset для text: `text/html; charset=utf-8`. JSON — UTF-8.

### Accept: What the Client Wants Back

**Accept** — **content negotiation**:

```
Accept: application/json, text/plain;q=0.9, */*;q=0.8
```

Server выбирает best match. APIs часто always JSON; MVC — HTML vs JSON по Accept.

Mismatch → **406 Not Acceptable** (редко).

**Accept** = desired **response** format. **Content-Type** = actual **body** format. Не путать direction.

### Authorization: Proving Identity

```
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
Authorization: Basic dXNlcjpwYXNz
```

**Bearer** — JWT/OAuth tokens. **Basic** — base64 credentials, **только HTTPS**.

Explicit per request — SPAs/mobile prefer. ASP.NET JWT Bearer middleware.

Не логировать Authorization — compliance incident.

**401** + `WWW-Authenticate` — пара с `0.3.6`.

### Cookie: Session State on the Wire

**Cookie** **request** header — name=value pairs от **Set-Cookie** response (`0.3.9`):

```
Cookie: session_id=abc; theme=dark
```

Browser auto-sends по domain/path rules. **Cookie** (request) vs **Set-Cookie** (response).

ASP.NET cookie authentication validates encrypted session cookie.

### Cache-Control: Directing Caches

Primary modern caching directive.

Response:
- `no-store` — не хранить (sensitive).
- `private, max-age=3600` — browser cache, не shared CDN.
- `public, max-age=86400, immutable` — CDN static assets.

Request:
- `no-cache` — store ok, **must revalidate** before use.

`public` на `/api/me` → **leak через CDN**.

Связь с **304** (`0.3.6`).

### ETag: Entity Tag for Conditional Requests

Opaque validator версии representation:

```
ETag: "33a64df55142582555b22e4d9586"
```

Strong vs weak (`W/"..."`). Client:

```
If-None-Match: "33a64df55142582555b22e4d9586"
```

Unchanged → **304**. Changed → **200** + new ETag.

**If-Match** + PUT — optimistic concurrency → **412**/**409**.

## Under the Hood

### Header Processing in ASP.NET Core

```
Kestrel parse → Middleware (CORS) → Auth (Authorization/Cookie) →
Content negotiation → Action → Response headers (Cache-Control, ETag)
```

**X-Forwarded-*** за reverse proxy — HTTPS и client IP.

### Browser Request Headers

Navigation: Accept, Cookie, User-Agent automatic. fetch adds Authorization; forbidden headers blocked. CORS preflight для custom headers.

### Cache Layers

```
Browser → CDN → Reverse proxy → Origin
```

**Vary: Accept-Encoding**, **Vary: Authorization** — critical для authenticated caching (usually avoid).

## Syntax / Commands / API

```bash
curl -H "Accept: application/json" https://api.example.com/items/1
curl -X POST ... -H "Content-Type: application/json" -H "Authorization: Bearer TOKEN" -d '{}'
curl -H 'If-None-Match: "abc"' -i https://api.example.com/items/1
```

```javascript
fetch('/api/items', {
  headers: {
    'Content-Type': 'application/json',
    Accept: 'application/json',
    Authorization: `Bearer ${token}`,
  },
});
```

## Common Mistakes & Anti-patterns

Swap Accept/Content-Type. Missing charset. Log Authorization. public cache на auth JSON. Ignore Vary. JSON без Content-Type.

---

## Production & Real-World Notes

Gateways inject correlation IDs. CDN + ETag. OpenAPI documents required headers. Response caching middleware — осторожно с auth.

---

## Comparison / Trade-offs

| Mechanism | Strength | Weakness |
|-----------|----------|----------|
| Bearer | Stateless, mobile | Token theft |
| Cookie session | HttpOnly vs XSS | CSRF → SameSite |
| ETag | Bandwidth save | Compute hash |
| max-age | Speed | Stale risk |
| no-store | Privacy | No cache benefit |

---

## Quick Reference

| Header | Direction | Role |
|--------|-----------|------|
| Content-Type | Both | Body media type |
| Accept | Request | Preferred response |
| Authorization | Request | Credentials |
| Cookie | Request | Stored cookies |
| Cache-Control | Both | Cache directives |
| ETag | Response | Version tag |
| If-None-Match | Request | Conditional GET |

---

## Key Takeaways

- Headers = format, auth, caching — orthogonal to status codes.
- Content-Type vs Accept — body is vs want back.
- Authorization explicit; Cookie browser-managed.
- Cache-Control + ETag → 304.
- Misconfigured cache on private data = security issue.
