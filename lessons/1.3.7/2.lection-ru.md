# LINQ: method syntax — Where, Select, OrderBy, GroupBy

> Roadmap: `1.3.7` · Node: `1.3` — C#: продвинутые темы · Depth: **глубоко**

## Learning Objectives

После этой лекции ты сможешь:

- Объяснить **LINQ** как composable query-слой над sequences, построенный на **generics** и **delegates**.
- Использовать **method syntax** (`Where`, `Select`, `OrderBy`, `GroupBy`) для filter, project, sort и aggregate в памяти.
- Предсказывать **return types** и **element types** на каждом этапе pipeline.
- Описать, как **lambda expressions** (`1.3.6`) становятся predicate и projection delegates у каждого operator.
- Обосновать, почему **порядок operators** важен для correctness и performance.
- Связать эти in-memory operators с тем же vocabulary, который Entity Framework Core позже переведёт в SQL.

## Why This Matters

В уроках `1.3.3`–`1.3.6` ты изучил **delegates**, **Func/Action** и **lambdas** — механизм «передать функцию как данные». **Language Integrated Query (LINQ)** — место, где этот механизм превращается в ежедневный productive syntax. Вместо ручных `foreach` с временными списками ты декларируешь, *что* нужно из sequence, и composable маленькие operators в pipelines.

Каждый ASP.NET Core service использует LINQ: фильтрация DTO в памяти, projection EF entities в response models, grouping analytics по дате, ordering paginated results. Четыре operator из этого урока — **Where**, **Select**, **OrderBy**, **GroupBy** — backbone. Освой их in-memory; в `1.3.11`–`1.3.12` и далее EF Core те же **method names** появятся на `IQueryable<T>`, но provider переведёт их в SQL вместо loops в process.

Middle-разработчик не заучивает пятьдесят LINQ methods в первый день. Он internalize **pipeline mental model**: source sequence → transform → transform → materialize. Этот урок строит модель на самых частых operators и открывает hood достаточно, чтобы deferred execution (`1.3.9`) и expression trees (`1.3.12`) не казались magic.

## Core Concepts

### LINQ как extension methods на IEnumerable<T>

LINQ to Objects работает на **`IEnumerable<T>`** — всё, что можно iterate через `foreach`. Operators живут в static class **`System.Linq.Enumerable`** как **extension methods** (compile-time story в `1.3.14`). Когда пишешь:

```csharp
var adults = users.Where(u => u.Age >= 18);
```

compiler переписывает примерно как `Enumerable.Where(users, u => u.Age >= 18)`. Lambda `u => u.Age >= 18` — **`Func<User, bool>`**, predicate. **`Where`** возвращает другой `IEnumerable<User>`, который *описывает* filter; loop не обязательно запускается сразу (deferred execution — `1.3.9`).

Каждый operator — **pure transformation descriptor**, пока что-то не force enumeration (`ToList`, `foreach`, `Count` — `1.3.10`).

### Where — фильтрация по predicate

**`Where`** оставляет elements, для которых predicate возвращает `true`. Концептуальная signature:

```csharp
IEnumerable<T> Where<T>(this IEnumerable<T> source, Func<T, bool> predicate)
```

Overload принимает `Func<T, int, bool>`, где `int` — zero-based index:

```csharp
var everySecond = items.Where((item, index) => index % 2 == 0);
```

**Where** не мутирует source collection. Возвращает новый sequence *view*. Несколько **Where** composable (кадый оборачивает предыдущий):

```csharp
var result = orders
    .Where(o => o.Status == OrderStatus.Paid)
    .Where(o => o.Total > 100);
```

Логически эквивалентно одному predicate с `&&`, но отдельные **Where** иногда улучшают readability, когда каждое условие — отдельное business rule.

**Performance note:** filter early. Если **Select** первым создаёт heavy objects, потом **Where** — можешь allocate projections, которые сразу discard. Prefer **Where** before **Select**, когда filter на source fields.

### Select — projection (map)

**`Select`** transform каждый element в другой type — functional **map**:

```csharp
IEnumerable<TResult> Select<TSource, TResult>(
    this IEnumerable<TSource> source,
    Func<TSource, TResult> selector)
```

Пример — entities в DTOs:

```csharp
var dtos = products.Select(p => new ProductDto(p.Id, p.Name, p.Price));
```

Output sequence имеет element type **`TResult`**, не `TSource`. Chaining меняет generic type на каждом шаге:

```csharp
var names = users.Select(u => u.Name);           // IEnumerable<string>
var lengths = users.Select(u => u.Name.Length);  // IEnumerable<int>
```

**Select** с index:

```csharp
var numbered = items.Select((item, i) => $"{i + 1}. {item}");
```

Typical API pattern: project early к нужной shape, чтобы downstream code не зависел от entity types. EF Core heavily использует **Select** для **server-side projection** — только нужные columns вместо full entities.

### OrderBy, ThenBy — sorting

**`OrderBy`** и **`OrderByDescending`** sort по key из selector:

```csharp
var byPrice = products.OrderBy(p => p.Price);
var byPriceDesc = products.OrderByDescending(p => p.Price);
```

Return type — **`IOrderedEnumerable<T>`** с **`ThenBy`** / **`ThenByDescending`** для secondary keys:

```csharp
var sorted = customers
    .OrderBy(c => c.Country)
    .ThenBy(c => c.LastName)
    .ThenBy(c => c.FirstName);
```

Sorting в LINQ to Objects **stable**: equal keys сохраняют relative order из input sequence.

**`OrderBy` after `Where`** — typical. **`Where` after `OrderBy`** тоже работает, но re-filters уже sorted sequence — редко нужно.

Sorting требует знать keys для **всех** elements — operator internally **consume** upstream через sort algorithm, effectively buffering. Связь с deferred execution — в `1.3.9`.

Default comparer — **`Comparer<TKey>.Default`**. Для custom types нужен **`IComparable<T>`** или explicit **`OrderBy(key, comparer)`**.

### GroupBy — разбиение на buckets

**`GroupBy`** partition sequence на **`IGrouping<TKey, TElement>`** с одинаковым key:

```csharp
var byCountry = orders.GroupBy(o => o.ShippingCountry);

foreach (var group in byCountry)
{
    Console.WriteLine($"{group.Key}: {group.Count()} orders");
    foreach (var order in group)
        Console.WriteLine($"  {order.Id}");
}
```

Каждый **`IGrouping<TKey, TElement>`** implements **`IEnumerable<TElement>`** и expose **`Key`**. Default **`TElement`** = **`TSource`**.

Overload с **element selector** — group by key, но project elements иначе:

```csharp
var totalsByCountry = orders.GroupBy(
    o => o.ShippingCountry,
    o => o.Total);
```

Overload с **result selector** — collapse group сразу:

```csharp
var summary = orders.GroupBy(
    o => o.ShippingCountry,
    (country, items) => new { Country = country, Count = items.Count(), Sum = items.Sum() });
```

**GroupBy** — in-memory analogue SQL **`GROUP BY`**. EF Core translate **GroupBy** только когда shape translatable; arbitrary client logic в grouping может force client evaluation — production pitfall с databases.

### Chaining: читать pipeline слева направо

Typical pipeline:

```csharp
var report = sales
    .Where(s => s.Date.Year == 2026)
    .GroupBy(s => s.Region)
    .Select(g => new RegionSummary(g.Key, g.Sum(x => x.Amount), g.Count()))
    .OrderByDescending(r => r.Total)
    .Take(10);
```

Читай: restrict 2026 → partition by region → project summaries → sort by total → top 10. Каждый step возвращает новый **`IEnumerable<...>`**. Ничего не run, пока **`ToList()`**, **`foreach`** или similar (`1.3.10`).

Operator order меняет **meaning**:

```csharp
// Top 10 regions ПОСЛЕ full aggregation — правильно
.GroupBy(...).Select(...).OrderByDescending(...).Take(10)

// НЕПРАВИЛЬНО, если нужны top 10 sales ДО grouping:
.Take(10).GroupBy(...)
```

Это не stylistic preference — semantics. SQL queries требуют той же discipline.

## Under the Hood

### Iterator blocks и deferred chains

Большинство LINQ to Objects operators реализованы через **`yield return`** iterator blocks (`1.3.20`). **`Where`** iterate source once, test predicate per element, yield matches — без intermediate list, если operator не требует иного.

**`GroupBy`** и **`OrderBy`** heavier: typically buffer data во internal structures (hash buckets, sort buffers) before yielding. Когда **foreach** grouped/sorted sequence — upstream pipeline execute минимум до buffering requirement этого operator.

Conceptually:

```
source → [Where iterator] → [Select iterator] → [OrderBy buffer+sort] → consumer
```

Каждый wrapper держит reference на previous **`IEnumerable`** и delegate fields — lambda captures становятся fields в compiler-generated iterator class.

### Closures и captured variables

Lambdas в LINQ часто **capture** outer variables:

```csharp
int minAge = 18;
var adults = users.Where(u => u.Age >= minAge);
minAge = 21;  // меняет threshold ДО enumeration!
```

Lambda closes over **`minAge`**, не over value на момент construction pipeline. Surprises при dynamic filters в loops — classic bug при composing **Where** inside **foreach**.

### Type inference через pipeline

Compiler infer **`T`** и **`TResult`** на каждом **Select**. Broken pipeline часто fail at compile time:

```csharp
var x = users.Select(u => u.Name).Where(n => n.Age > 18); // error: string has no Age
```

**Where** после projection на **`string`** не видит **`User.Age`**. Filter на source properties **до** project away.

### Связь с EF Core (preview)

Когда source — **`DbSet<Product>`** (`IQueryable<Product>`), те же **`.Where(...).Select(...).OrderBy(...).GroupBy(...)`** build **expression tree** вместо delegates. EF provider walk tree и emit SQL. Method names совпадают с LINQ to Objects; **execution model** fundamentally другой — **`1.3.11`**, **`1.3.12`**. Пока master semantics in-memory; vocabulary transfer directly.

## Syntax / Commands / API

| Operator | Purpose | Typical signature |
|----------|---------|-------------------|
| `Where` | Filter | `Func<T, bool>` |
| `Select` | Project | `Func<T, TResult>` |
| `OrderBy` / `OrderByDescending` | Primary sort | `Func<T, TKey>` |
| `ThenBy` / `ThenByDescending` | Secondary sort | on `IOrderedEnumerable<T>` |
| `GroupBy` | Partition | key + optional element/result selectors |

Namespace **`System.Linq`**.

## Examples

### Example 1 — E-commerce order report

```csharp
var topCountries = orders
    .Where(o => o.PlacedAt >= DateTime.UtcNow.AddMonths(-1))
    .GroupBy(o => o.Country)
    .Select(g => new { Country = g.Key, Revenue = g.Sum(o => o.Total) })
    .OrderByDescending(x => x.Revenue)
    .Take(5)
    .ToList();
```

Filter month → group → project revenue → sort → take five → **materialize** с **`ToList()`**.

### Example 2 — Index in Select

```csharp
var lines = linesOfText
    .Select((line, index) => new { Number = index + 1, Text = line })
    .Where(x => !string.IsNullOrWhiteSpace(x.Text));
```

**Where** after **Select** — filter на projected shape.

## Common Mistakes & Anti-patterns

**Filter после project away fields.** **Where** before **Select**, когда predicate на source.

**GroupBy как dictionary.** Это lazy **`IEnumerable<IGrouping<...>>`**. Multiple enumerations re-run grouping без **`ToList()`**.

**Второй OrderBy вместо ThenBy.** Re-sorts entirely, теряя previous order.

**Side effects в Select.** Может run zero, once или many times (`1.3.9`).

## Production & Real-World Notes

In-memory LINQ на малых collections — OK. На large datasets — **`IQueryable`** + EF, чтобы operators run в PostgreSQL. Pagination: **`Skip`/`Take`** после **`OrderBy`**, не после load all rows.

Code review: «Could this **GroupBy** happen in SQL?»

## Comparison / Trade-offs

**Method vs query syntax (`1.3.8`):** identical после desugaring. **LINQ vs loops:** loops — control в hot paths; LINQ — readability. Measure в hot paths.

## Quick Reference

| Goal | Operator |
|------|----------|
| Filter | `Where` |
| Transform | `Select` |
| Sort | `OrderBy` / `ThenBy` |
| Partition | `GroupBy` |

Pipeline order: **filter → group → project → sort → page**.

## Key Takeaways

- LINQ method syntax composes **`IEnumerable<T>`** через extension methods и delegates.
- **`Where`** filter; **`Select`** project; **`OrderBy`/`ThenBy`** sort; **`GroupBy`** partition как SQL **`GROUP BY`**.
- Operator **order меняет semantics**.
- Execution **deferred** до enumeration (`1.3.9`).
- Те же names на **`IQueryable`** для EF Core.
