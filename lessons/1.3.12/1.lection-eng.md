## Why This Matters

Lesson **`1.3.11`** separated **`IEnumerable`** (delegates, local) from **`IQueryable`** (provider, remote). The bridge is **expression trees**: LINQ queries EF can translate are not "functions stored in a pipeline" — they are **abstract syntax trees** the provider inspects, rewrites, and converts to SQL.

Without this model, EF errors look random: "could not be translated," "client evaluation," sudden memory spikes. With it, you predict: **if the tree contains nodes SQL has no mapping for, translation fails or splits**. Middle developers debug with **`ToQueryString()`**, refactor predicates into translatable shapes, and build dynamic queries by composing **`Expression<Func<>>`** — not concatenating SQL strings.

This is **`глубоко`** depth: you will not implement a LINQ provider, but you must understand the provider's input format and the compiler rules that produce it.

## Core Concepts

### Code as Data vs Code as Instructions

Normal lambda assigned to **`Func<T, bool>`**:

```csharp
Func<User, bool> pred = u => u.Age > 18;
bool ok = pred(user);  // JIT runs IL directly
```

Same syntax assigned to **`Expression<Func<User, bool>>`**:

```csharp
Expression<Func<User, bool>> expr = u => u.Age > 18;
// expr is an object graph describing "parameter u, member Age, constant 18, GreaterThan"
```

Nothing is "called" on **`user`** when building **`expr`**. The tree **describes** the predicate. EF walks **`expr.Body`**, finds **`u.Age > 18`**, emits **`WHERE "Age" > 18`** (simplified).

### Dual Nature of Lambdas — Compiler Magic

When you write:

```csharp
db.Users.Where(u => u.IsActive);
```

the compiler sees **`IQueryable<User>`** extension from **`Queryable.Where`**, which requires **`Expression<Func<User, bool>>`**. It **builds an expression tree** automatically — you write familiar syntax.

When you write:

```csharp
usersList.Where(u => u.IsActive);
```

**`Enumerable.Where`** requires **`Func<User, bool>`** — compiler emits a **delegate**.

One syntax, two targets — resolved by **static type** of the receiver. This is why changing a variable's declared type changes semantics silently.

### Anatomy of a Simple Tree

For **`u => u.Age > 18`**:

```
LambdaExpression
├── Parameters: [ u (User) ]
└── Body: GreaterThan
         ├── MemberAccess: u.Age
         └── Constant: 18
```

Node types inherit from **`Expression`**: **`ParameterExpression`**, **`MemberExpression`**, **`ConstantExpression`**, **`BinaryExpression`**, **`MethodCallExpression`**, etc.

**`Select`** adds **`MemberInit`** or **`New`** nodes for projections. **`OrderBy`** wraps in **`MethodCallExpression`** nodes chaining **`Queryable.OrderBy`**.

### How IQueryable Builds the Tree

Each operator call:

```csharp
IQueryable<User> q = db.Users
    .Where(u => u.IsActive)
    .Where(u => u.Age >= 18);
```

Conceptually:

```
q.Expression = MethodCallExpression(
    Queryable.Where,
    inner: MethodCallExpression(
        Queryable.Where,
        inner: ConstantExpression(db.Users),
        predicate1),
    predicate2)
```

**`IQueryProvider.CreateQuery`** returns new **`IQueryable`** wrapper pointing to extended tree — still no SQL, no row touches.

### Execution — Provider as Interpreter

On **`ToListAsync()`** / **`GetEnumerator()`**:

1. EF's **`IQueryProvider`** receives final **`Expression`**.
2. **Query compiler** visits nodes, maps to SQL AST.
3. **SQL generator** (Npgsql for PostgreSQL) produces parameterized command.
4. Driver executes; materializer builds **`User`** entities or projections.

**`Execute<T>`** returns scalar; **`CreateQuery`** returns deferred **`IQueryable`**.

Different providers (EF Core, NHibernate, **`EnumerableQuery`**) share **`IQueryable`** surface, different **`Execute`** implementations.

### Translatability — Why Your C# Fails in SQL

EF maps subset of expression nodes to SQL:

| Often translatable | Often not |
|--------------------|-----------|
| Property access, comparisons | Arbitrary local method calls |
| `&&`, `\|\|`, `!` | Complex loops in predicate |
| `string.StartsWith` (mapped) | `MyRegex.IsMatch` |
| `EF.Functions.Like` | Most static custom helpers |

```csharp
// May translate
.Where(u => u.Email.Contains("@") && u.IsActive)

// Often fails
.Where(u => ValidateEmail(u.Email))
```

EF Core 3+ **throws** when non-translatable nodes would silently client-evaluate (configurable in older patterns). Fix: **`EF.Functions`**, SQL **`FUNCTION`**, split query, or **`AsEnumerable()`** after narrowing in SQL.

### Building Expressions Manually (Intro)

Dynamic sorting/filter APIs sometimes compose trees by hand:

```csharp
ParameterExpression param = Expression.Parameter(typeof(User), "u");
MemberExpression prop = Expression.Property(param, nameof(User.IsActive));
ConstantExpression trueConst = Expression.Constant(true);
BinaryExpression eq = Expression.Equal(prop, trueConst);
Expression<Func<User, bool>> lambda = Expression.Lambda<Func<User, bool>>(eq, param);

IQueryable<User> q = db.Users.Where(lambda);
```

Libraries (**LinqKit**, specification pattern) automate this. Manual building teaches that **query syntax is just tree construction**.

### Expression Trees vs Delegates — Performance Note

Building trees has compile-time cost; translation cached in EF for repeated query shapes. At runtime, **one SQL round-trip** beats shipping rows for local **`Func`** filter. Do not "optimize" to **`IEnumerable`** without measuring.

## Under the Hood

### Compiler: Lambda to Expression

Roslyn emits **`Expression.Lambda`** calls when target type is **`Expression<TDelegate>`**. Reflection or **`Compile()`** can turn expression back to **`Func<>`**:

```csharp
Func<User, bool> pred = expr.Compile();  // now local delegate — loses translatability for EF
```

**`Compile()`** on EF-bound expressions defeats provider — use only for in-memory after **`AsEnumerable()`**.

### IQueryProvider Interface

```csharp
public interface IQueryProvider
{
    IQueryable CreateQuery(Expression expression);
    object? Execute(Expression expression);
    TResult Execute<TResult>(Expression expression);
}
```

**`CreateQuery`** — extend deferral. **`Execute`** — run now. EF implements both with SQL translation pipeline.

### EF Core Pipeline (Conceptual)

```
Expression Tree
    → Query Translation (method call mapping)
    → SQL Expression
    → SQL Generation (provider-specific)
    → DbCommand (parameters)
    → Result materialization
```

Logging: **`ToQueryString()`**, **`EnableSensitiveDataLogging`** in dev only.

### IEnumerable Provider — EnumerableQuery

In-memory **`AsQueryable()`** uses **`EnumerableQuery`**: **`Execute`** compiles tree to delegates — same **`Expression`** API, local semantics. Useful for tests asserting expression composition without PostgreSQL.

## Examples

### Example 1 — Inspect tree (debug learning)

```csharp
Expression<Func<User, bool>> expr = u => u.Age > 18;
Console.WriteLine(expr);
// prints something like: u => (u.Age > 18)
Console.WriteLine(expr.Body.NodeType);  // GreaterThan
```

### Example 2 — Dynamic filter

```csharp
public static IQueryable<User> WhereActive(IQueryable<User> q, bool? active)
{
    if (!active.HasValue) return q;
    return q.Where(u => u.IsActive == active.Value);
}
```

Each **`Where`** adds translatable **`MethodCallExpression`** if property maps to column.

## Common Mistakes & Anti-patterns

**Calling `Compile()` on EF predicates** before provider — SQL lost.

**Assuming identical syntax implies identical execution** — check receiver type.

**Local functions in `IQueryable` lambdas** — translation failure.

**String concatenation for SQL** instead of composable **`Expression`** — injection risk and unmaintainable.

---

## Production & Real-World Notes

Use **specification pattern** or predicate builders for complex dynamic filters — still expression-based, testable with **`ToQueryString()`**.

Integration tests with **Testcontainers** validate translation against real PostgreSQL, not just in-memory **`List`**.

When translation fails, read exception **which node** failed — refactor that expression branch.

## Comparison / Trade-offs

| Expression trees | Delegates (`Func`) |
|------------------|-------------------|
| Inspectable, translatable | Opaque, fast local call |
| Provider executes remotely | CLR executes locally |
| Dynamic query composition | Dynamic via chained `Func` (client only) |

## Quick Reference

- **`Expression<Func<T,bool>>`** — tree, not delegate
- Compiler picks Expression vs Func by **`IQueryable`** vs **`IEnumerable`**
- Each **`Queryable.*`** operator → **`MethodCallExpression`**
- **`IQueryProvider.Execute`** → SQL (EF)
- **`Compile()`** → local delegate (post-EF boundary only)
- **`ToQueryString()`** — debug translation

## Key Takeaways

- Expression trees represent **code structure as data** for providers to translate.
- **`IQueryable`** pipelines accumulate **`MethodCallExpression`** nodes until execution.
- EF Core maps translatable nodes to SQL; non-translatable nodes cause failures or client eval (avoid).
- Lambdas compile to **different artifacts** depending on extension method target type.
- Understanding trees explains EF errors and enables safe dynamic LINQ.
- Next EF chapters assume this model — **`DbSet`**, migrations, **`Include`**, projections all live here.
