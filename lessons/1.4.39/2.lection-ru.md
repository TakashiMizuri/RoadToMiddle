# Task vs ValueTask — дерево решений

> Roadmap: `1.4.39` · Node: `1.4` — C# async · Depth: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Объяснить **профиль аллокаций** `Task` vs **pooled** `ValueTask` / `IValueTaskSource`.
- Применять **decision tree** для return type в библиотеках и application code.
- Сформулировать правило: **`ValueTask` нельзя await дважды** без materialize.
- Различать **`async ValueTask`** и sync result в `ValueTask`.
- Сравнить **`ValueTask<T>`** с **`Task<T>`** на hot paths (cache hit, buffered I/O).
- Избегать breaking changes при смене return type в public API.

---

## Why This Matters

`Task` — стандарт async в .NET: composable, безопасно await повторно (для completed). **`ValueTask`** (C# 7+) — **производительность**: при **частом синхронном завершении** cached `Task` всё равно аллоцирует. `ValueTask` может обернуть **`T` напрямую** или pooled `IValueTaskSource` — zero alloc на sync path.

Misuse → **subtle bugs**: double await, хранение в field. EF Core, `Stream.ReadAsync` используют `ValueTask` на hot paths; в application code чаще **`Task`** без профилирования.

На interview: «Когда ValueTask?» — **decision tree + double-await rule**, не «всегда быстрее».

## Core Concepts

### Task — reference type, default

`Task`/`Task<T>` — **classes**, heap. `async Task` с suspend аллоцирует state machine + Task. **Повторный await** того же completed `Task` — OK.

**Default** для public API, controllers, services.

### ValueTask — struct wrapper

Оборачивает:
1. **`T`** — sync complete
2. **`Task`** — async path
3. **`IValueTaskSource`** — pooled (advanced)

```csharp
public ValueTask<User?> GetCachedUserAsync(int id)
{
    if (_cache.TryGetValue(id, out var user))
        return new ValueTask<User?>(user);
    return new ValueTask<User?>(FetchFromDbAsync(id));
}
```

### Double-await rule

**Не await `ValueTask` дважды** (кроме редких гарантий Task inside). После await struct **consumed**.

```csharp
// WRONG — два await
// RIGHT
var task = GetAsync().AsTask();
```

**`.AsTask()`** для WhenAll, storage, multiple awaits.

### async ValueTask

Генерирует state machine. Выгода когда **часто return без await**. Если всегда suspend — мало пользы → `Task`.

### Decision tree

```
Public API / app code? → Task (default)
Hot path + profiling показал alloc pressure?
  Often sync-complete + single immediate await? → ValueTask
  Need WhenAll / store / multiple await? → Task или AsTask()
  Library tight loop + pooling? → IValueTaskSource
```

### Примеры framework

`SaveChangesAsync` → `Task` — typical app не micro-hot.

`Stream.ReadAsync` → `ValueTask<int>` — tight loop, часто sync из buffer.

## Under the Hood

`ValueTask` struct — union-like. Awaiter: если `T` — sync complete; если `IValueTaskSource` — GetResult + **return to pool**.

Pooling требует корректный `IValueTaskSource` — double release, use-after-free. Поэтому pooling в library code.

`async ValueTask` → `AsyncValueTaskMethodBuilder` — может избежать Task alloc при sync complete.

---

## Common Mistakes

- Все методы на ValueTask «для perf» без measurement
- ValueTask в `Task.WhenAll` без `AsTask()`
- Public API breaking change Task→ValueTask
- Double await pooled ValueTask

---

## Quick Reference

```
Default → Task<T>
Hot + sync-complete + single await → ValueTask<T>
WhenAll/store → AsTask() или Task
Never double-await ValueTask
```

---

## Key Takeaways

- **Task** — default.
- **ValueTask** — perf при частом sync-complete и single await.
- **Double-await** — bug; **AsTask()** materialize.
- Profile before change.
- Public API — conservative.
