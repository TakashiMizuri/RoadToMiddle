# Semantic layout: header, nav, main, article, section, aside, footer

> Roadmap: `0.5.2` · Node: `0.5` — HTML · Depth: практика

## Learning Objectives

После этого урока ты сможешь:

- Выбирать **semantic landmarks** вместо безликих `<div>` для регионов страницы.
- Строить типичный сайт с **`header`**, **`nav`**, **`main`**, **`footer`** и опциональным **`aside`**.
- Различать **`article`** и **`section`** и вкладывать их без поломки логики outline.
- Объяснить, как landmarks помогают **screen reader** и keyboard-пользователям пропускать повторяющийся chrome.
- Избегать дублирования **`main`** и «спрятанной» навигации.
- Набросать layout блога, документации и dashboard одним и тем же словарём тегов.

---

## Why This Matters

В `0.5.1` ты собрал конверт документа — DOCTYPE, charset, lang. Следующий вопрос: *что внутри `<body>`?* Долгое время страницы резали на `<div id="header">` и `<div class="sidebar">`. Браузер рисовал, Grid позиционировал — но **смысл** регионов жил только в class names.

**Секционные элементы HTML5** открывают структуру платформе: assistive technology видит «navigation», «main», «complementary»; SEO оценивает фокус страницы; коллеги читают разметку без CSS. Middle-разработчик на React всё равно отдаёт HTML в DOM — `<header>` в JSX тот же контракт, что в статике.

Ошибки в landmarks — частый a11y-провал: два `<main>`, `<section>` без заголовков, меню в div-soup. Правильная семантика дешёва при написании и дорога при рефакторинге SPA на пятьдесят роутов.

## Core Concepts

### Landmarks — именованные области страницы

**Landmarks** — крупные зоны: шапка с логотипом, глобальное меню, основной контент, сайдбар, подвал. HTML5 даёт теги вместо анонимных div:

| Элемент | Типичная роль |
|---------|----------------|
| `header` | Вводная часть секции или страницы — лого, заголовок, иногда nav |
| `nav` | Блок **навигационных ссылок** |
| `main` | **Уникальный** основной контент документа — один на страницу |
| `article` | Самодостаточный блок — пост, карточка, комментарий |
| `section` | Тематическая группа **с заголовком** |
| `aside` | Косвенно связанный контент — сайдбар, реклама |
| `footer` | Подвал ближайшего sectioning-предка |

Теги не заменяют CSS — они **называют** регионы. Flexbox/Grid (`0.6.x`) по-прежнему выстраивает колонки.

### header и footer: контекст важен

`<header>` и `<footer>` не глобальны как `<main>`. У `<article>` может быть свой `<header>` с заголовком поста и `<footer>` с тегами. Шапка **страницы** — лого и top nav; **страницы** footer — copyright.

Ошибка: весь контент внутри одного огромного `<header>`. Основная статья в `<header>` — AT воспринимает её как «вводную», не main. Правило: page `<header>` = верхняя полоса; **`main`** = то, ради чего пришли.

### nav: ссылки для перемещения

`<nav>` — когда блок **в первую очередь навигация**: меню, breadcrumbs, оглавление. Не каждая группа ссылок требует nav: три ссылки «Privacy» в footer могут жить в `<footer>` без отдельного nav.

Несколько `<nav>` допустимы — глобальное меню и TOC статьи. Дубли подписывай `aria-label` (`0.5.11`):

```html
<nav aria-label="Primary">...</nav>
```

### main: одна главная история

**Ровно один** видимый `<main>` на документ. Уникальное для URL: тело статьи, карточка товара, виджеты dashboard. Повторяющийся chrome — **вне** main.

В SPA один main на route; не вкладывай main в main. Layout с `{children}` в `<main>` — хороший паттерн.

### article vs section: автономность vs группа

**`article`** — **самодостаточен**: понятен отдельно в RSS или по прямой ссылке. Пост, новость, карточка товара в ленте.

**`section`** — **тематическая часть** целого, почти всегда с **heading**. «Глава 3: Deploy» — section. Section без заголовка — smell; для стилей — `<div>`.

Блог-индекс: `<main>` с несколькими `<article>` (превью). Страница поста: `<main>` → один `<article>` → несколько `<section>`.

Вопрос: «Имеет смысл отдельно в RSS?» Да → article. Нет, глава одного doc → section.

### aside: рядом, но не главное

`<aside>` — связано с контентом, но не центральный поток: bio автора, «читайте также», glossary. Комментарии — часто `<section>`; «Trending» — `<aside>`.

Не клади основную статью в aside из-за CSS-колонки — семантика по **назначению**, не по пикселям.

## Under the Hood

Браузеры мапят элементы на **implicit ARIA landmarks**: banner (header на уровне body), navigation, main, complementary, contentinfo. Screen readers дают shortcuts между landmarks.

Document outline в HTML5 менялся — не полагайся только на авто-outline для SEO. Заголовки внутри section всё равно нужны (`0.5.3`). Validator ругается на два main.

Div-soup загрузится, но потеряешь affordances платформы и провалишь axe/Lighthouse.

## Examples

**Product page:** один main, один article (товар), specifications — section с h2, «Also bought» — aside, nav в header вне main.

**Dashboard:** main = charts/tables; sidebar links — nav вне main; help — aside. Не оборачивай каждый widget в article без причины syndication.

---

## Common Mistakes & Anti-patterns

Два `<main>`. `<section>` без heading. Всё в `<article>`. Nav только через div. Основной контент в `<header>`.

---

## Production & Real-World Notes

Design systems с классами — семантику выбираешь ты. Layout-компонент React → literal `<main>`. Micro-frontends: не по main на фрагмент. Lighthouse «no main landmark» — быстрый fix.

---

## Comparison / Trade-offs

`<div role="navigation">` работает, но native first (first rule of ARIA). Сомнение article/section → div + хорошие headings лучше, чем неверный section.

## Quick Reference

| Элемент | Один на страницу? | Нужен heading? |
|---------|-------------------|----------------|
| main | Да | Нет |
| nav | Несколько OK | Нет |
| article | Несколько OK | Обычно да |
| section | Несколько OK | Должен быть |
| aside | Несколько OK | Часто h2 |

---

## Key Takeaways

- Landmarks = **назначение**, не layout.
- Один `<main>`; chrome снаружи.
- article = standalone; section = глава с заголовком.
- nav для навигации; aside для tangential.
- header/footer могут повторяться внутри article.
