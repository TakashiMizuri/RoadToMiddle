# CLR, managed code и IL — краткая выжимка

> `1.5.1` · ~8 мин на повторение

## Главная идея

**CLR** — execution engine .NET: загружает **assembly** (IL + metadata), **JIT**-компилирует в native code, управляет **heap**, **stack frames**, **GC** и **exceptions**. **Managed code** выполняется под контролем CLR; это реализует модель памяти из **`1.1.6`**.

## Ключевые пункты

- **C# → IL** (Roslyn), не напрямую machine code (кроме AOT/R2R).
- **IL** — stack-based bytecode; **metadata** — type system для JIT и GC.
- **Managed heap** — reference objects; **stack frames** — locals (`1.1.6`).
- **Unmanaged** — native вне GC (P/Invoke); leaks/crashes — отдельный класс проблем.
- **Verification** + GC — type-safe references; **unsafe** ломает assumptions.
- **Assembly** — deployment unit с manifest и references.

## Самые важные примеры

Pipeline: `C# → dotnet build → .dll (IL+metadata) → CLR load → JIT → native`.

```csharp
public int Add(int a, int b) => a + b;
// IL: ldarg.1, ldarg.2, add, ret
```

## «Под капотом» в одном абзаце

CoreCLR (EE + JIT + GC + type loader) парсит ECMA-335 metadata в **MethodTable**, JIT эмитит native code с **GC maps** для stack roots. IL evaluation stack — абстракция компилятора; native **call stack** — реальные frames (`1.5.4`).

## Частые ошибки

- «C# = native на CPU» — нужен JIT/AOT.
- «IL интерпретируется» — steady-state = native.
- Путать IL stack и thread stack.

## Мнемоника

**CLR = Load · JIT · Run · GC** — четыре глагола runtime.
