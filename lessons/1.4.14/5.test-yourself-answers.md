# Answers: .Result, .Wait(), Deadlocks

## Answers

### 1. [Recall]
**Answer:** Blocking call thread on **`Task`** from **`async`** API instead of **`await`**.
**Why:** Classic anti-pattern in mixed sync/async code.

### 2. [Recall]
**Answer:** **`.Wait()`** → **`AggregateException`**; **`GetResult()`** → **direct** inner exception.
**Why:** Different unwrap behavior only.

### 3. [Understanding]
**Answer:** UI blocks in `.Result`; continuation **Post** to UI SyncContext; UI can't process queue → **MoveNext** never runs → circular wait.
**Why:** `1.4.7` + `1.4.9`.

### 4. [Understanding]
**Answer:** **Thread pool starvation** — each request blocks pool thread (`1.4.10`).
**Why:** No SyncContext but finite pool.

### 5. [Application]
**Answer:** **Deadlock** (classic) if library uses default capture.
**Why:** UI SyncContext scenario.

### 6. [Application]
**Answer:** **Blocks pool thread** per request waiting async completion.
**Why:** Starvation under concurrency.

### 7. [Analysis]
**Answer:** **Неверно** — все block одинаково.
**Why:** Deadlock mechanism unchanged.

### 8. [Analysis]
**Answer:** **Often gone** for SyncContext deadlock, but **still wrong** — use async; pool issues remain on Core.
**Why:** Mitigation not architecture.

### 9. [Understanding]
**Answer:** Continuations on **pool**, not UI — UI block doesn't prevent **MoveNext**.
**Why:** `1.4.8` breaks Post-to-UI requirement.

### 10. [Application]
**Answer:** `public async Task<IActionResult> Get() => Ok(await _svc.GetAsync(id));`
**Why:** async all the way.

### 11. [Analysis]
**Answer:** **Нет** — outer **still blocks**; can worsen pool usage.
**Why:** Nested blocking.

### 12. [Application]
**Answer:** **Pool starvation**, rising latency, hangs (`1.4.10`).
**Why:** 100 blocked workers.
