# Answers: Thread Pool Starvation

## Answers

### 1. [Recall]
**Answer:** Исчерпание свободных pool workers из-за blocking; очередь work/continuations растёт, work не выполняется вовремя.
**Why:** Finite pool + sync block = throughput collapse.

### 2. [Understanding]
**Answer:** Async I/O releases thread during kernel wait; `.Wait()` **blocks** thread until complete — worker unavailable.
**Why:** State machine continuation тоже needs pool thread.

### 3. [Understanding]
**Answer:** Starvation — **нет свободных pool threads**; Framework deadlock — **blocked thread ждёт Post на тот же SyncContext**.
**Why:** Core has no request SyncContext (`1.4.9`).

### 4. [Understanding]
**Answer:** After await, **`MoveNext` queued** to pool; if all threads blocked, continuations delay.
**Why:** `1.4.7` resume path uses pool without SyncContext.

### 5. [Application]
**Answer:** **Да, высокий риск** — каждый call blocks worker inside Task.Run under concurrency.
**Why:** Pool flood + blocking.

### 6. [Application]
**Answer:** **Найти и убрать blocking**; MinThreads — временная маска.
**Why:** Root cause fix required.

### 7. [Analysis]
**Answer:** **Нет** — не добавляет threads.
**Why:** Only changes SyncContext capture.

### 8. [Analysis]
**Answer:** Занимает pool threads unnecessarily; native async I/O не блокирует.
**Why:** Misuse of Task.Run for I/O.

### 9. [Recall]
**Answer:** `threadpool-thread-count`, `threadpool-queue-length`, lock contention counters.
**Why:** Standard Runtime counters.

### 10. [Application]
**Answer:** **`.Wait()` блокирует** pool thread на каждый request до Init complete.
**Why:** Classic starvation contributor.

### 11. [Understanding]
**Answer:** **Blocks pool thread** 1 sec per call under load.
**Why:** Use `await Task.Delay` with token instead.

### 12. [Analysis]
**Answer:** **Starvation** (threads blocked waiting, not computing).
**Why:** Low CPU + hangs suggests blocking not CPU saturation.
