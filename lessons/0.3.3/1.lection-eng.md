# HTTP Message Structure: Start Line, Headers, Body

> Roadmap: `0.3.3` · Node: `0.3` — HTTP and the Web · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Decompose an HTTP **request** and **response** into start line, headers, and optional body.
- Read and write raw HTTP/1.1 messages using correct **CRLF** line endings and header grammar.
- Explain how **Content-Length** and **chunked transfer encoding** delimit message bodies.
- Distinguish **header names** (case-insensitive) from **header values** and common standard headers.
- Describe how Kestrel and browsers parse messages into `HttpRequest` / `HttpResponse` abstractions.
- Identify **hop-by-hop** vs **end-to-end** headers and why proxies care.
- Debug malformed messages: missing blank line, wrong length, chunked parsing errors.

---

## Why This Matters

In `0.3.1` you saw the request-response cycle; in `0.3.2` you targeted a URL. Now you open the envelope: **what bytes actually cross the wire** after TCP and TLS deliver a stream. Frameworks hide this — `HttpClient`, `fetch`, ASP.NET Core controllers — but production failures often require reading raw traffic: Postman raw view, `curl -v`, Fiddler, Wireshark, or Kestrel connection logs when a client sends a bad `Content-Length` and the server hangs waiting for body bytes.

HTTP messages are deliberately simple text (HTTP/1.1) or framed binary (HTTP/2+). Understanding HTTP/1.1 message grammar remains essential because debugging tools decode to it, many concepts transfer to HTTP/2 **pseudo-headers** (`:method`, `:path`), and misconfigured proxies still speak HTTP/1.1 to backends. Middle developers configure request size limits, streaming uploads, gzip compression, and CORS headers — all of which are header and body mechanics.

A message without a correct **blank line** after headers is unparseable. A body without matching **Content-Length** or **chunked** encoding causes truncated reads or security issues. Methods and status codes (`0.3.4`, `0.3.6`) attach to the start line; this lesson is the chassis they ride in.

## Core Concepts

### Three Parts of Every Message

Every HTTP/1.1 message — request or response — has:

1. **Start line** (one line) — different for request vs response.
2. **Header section** — zero or more header lines, terminated by empty line.
3. **Body** (optional) — raw bytes whose length is defined by headers or encoding.

Structure:

```
start-line CRLF
header-name: header-value CRLF
...
CRLF
[message body]
```

**CRLF** is carriage return + line feed (`\r\n`). Unix tools may show `\n` only; on the wire HTTP/1.1 expects `\r\n`. A lone `\n` sometimes works but violates spec and breaks strict parsers.

### Request Start Line

Format:

```
METHOD request-target HTTP-version
```

Example:

```
GET /api/products?page=2 HTTP/1.1
```

**Method** (`GET`, `POST`, …) — `0.3.4`. **Request-target** — usually origin-form path + query from `0.3.2`. **Version** — almost always `HTTP/1.1` today for HTTP/1.1 connections.

After the start line come headers. Minimum for HTTP/1.1: **Host** header (mandatory except some ancient cases). Then blank line. **GET** typically has no body; **POST**/`PUT`/`PATCH` often do.

Full minimal request:

```
GET /health HTTP/1.1
Host: localhost:5000
Accept: application/json
Connection: close

```

Note the **empty line** after headers — parser signal that body follows (or message ends if no body).

### Response Start Line

Format:

```
HTTP-version status-code reason-phrase
```

Example:

```
HTTP/1.1 200 OK
```

**Status code** (`200`, `404`, …) — `0.3.6`. Reason phrase is informational; clients rely on numeric code. Headers follow, then optional body (HTML, JSON, file bytes).

```
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
Content-Length: 18

{"status":"ok"}
```

### Headers: Metadata and Control

Headers are `name: value` lines. Names are **case-insensitive** (`Content-Type` = `content-type`). Values may fold in obsolete HTTP/1.0 syntax; HTTP/1.1 uses one line per header.

Headers carry:

- **Representation metadata** — `Content-Type`, `Content-Encoding` (gzip)
- **Caching** — `Cache-Control`, `ETag` (`0.3.7`)
- **Auth** — `Authorization`, `WWW-Authenticate`
- **Connection control** — `Connection: close`, `Upgrade`
- **Custom** — `X-Correlation-ID` (prefer registered names without `X-` when possible)

ASP.NET Core exposes request headers as `IHeaderDictionary`; you add response headers in middleware or `Results.WithHeaders`.

Duplicate header names: some headers allow multiple values (Set-Cookie); others should be combined per spec (e.g. multiple `Accept` — unusual). Frameworks often expose **comma-separated** combination for certain headers.

### Message Body: When and How Much

Not every message has a body. **HEAD** responses must not include a body (but headers as if body existed). **204 No Content**, **304 Not Modified** — no body.

When a body exists, the receiver must know **where it ends**:

**Content-Length: N** — body is exactly N bytes after the blank line. Simple and common for JSON APIs.

**Transfer-Encoding: chunked** — body split into chunks for streaming; each chunk prefixed with size in hex; terminated by zero-size chunk and trailing headers. Used when length unknown upfront or for compression pipelines.

You must not send **both** Content-Length and chunked on HTTP/1.1 (conflict). HTTP/2 handles framing differently — length in DATA frames — but semantically still headers + body.

**Content-Type** tells how to interpret body bytes: `application/json`, `text/html; charset=utf-8`, `multipart/form-data` (`0.3.11`).

### Request vs Response Symmetry

Both share header + body grammar. Asymmetric details:

| | Request | Response |
|---|---------|----------|
| Start line | METHOD target HTTP/1.1 | HTTP/1.1 code phrase |
| Required Host | Yes (HTTP/1.1) | N/A |
| Body on errors | Client may send on 4xx | Server may send Problem Details JSON |

## Under the Hood

### Parsing State Machine (HTTP/1.1)

A typical server parser (Kestrel `HttpParser`):

1. Read bytes until `\r\n` → parse start line.
2. Read header lines until empty `\r\n` → build header collection.
3. If `Transfer-Encoding: chunked` → enter chunked decoder loop.
4. Else if `Content-Length` → read exactly that many bytes.
5. Else if method allows body and HTTP/1.1 rules say no body → body length 0.
6. Dispatch to application; repeat for next message on **keep-alive** connection (`0.3.14`).

If client declares `Content-Length: 100` but sends 50 bytes and closes, server waits or times out. If client sends 1000 bytes, overflow buffers may reject with **413 Payload Too Large** when limits configured (`MaxRequestBodySize` in Kestrel).

### Chunked Encoding Detail

```
HTTP/1.1 200 OK
Transfer-Encoding: chunked

1a
{"id":1,"name":"Alice"
0

```

Each chunk: **hex size** + CRLF + data + CRLF. Final `0\r\n\r\n` ends body. Trailer headers after final chunk are rare in APIs.

Chunked allows **streaming response** — ASP.NET Core can flush JSON array elements or SSE without knowing total length first.

### Hop-by-Hop vs End-to-End

**End-to-end** headers (`Authorization`, `Content-Type`) must propagate to final server. **Hop-by-hop** (`Connection`, `Keep-Alive`, `Transfer-Encoding`, proxy auth) meaningful only for one connection leg; proxies may strip or consume them. Mis-set `Connection: close` forces new TCP per request — latency cost.

### HTTP/2 Mapping (Preview)

HTTP/2 compresses headers (HPACK) and splits messages into **frames**. Pseudo-headers replace start line: `:method GET`, `:path /api`, `:status 200`. Body in DATA frames with length prefix. Conceptual model — headers + body — unchanged; wire format differs (`0.3.13`).

### Security and Limits

- **Request smuggling** — conflicting Content-Length and Transfer-Encoding on proxies; modern servers reject ambiguous messages.
- **Header injection** — CRLF in header values if not validated breaks message boundaries.
- **Decompression bombs** — huge body after gzip; limit sizes.

## Syntax / Commands / API

**Raw HTTP with netcat (lab only, HTTPS needs openssl s_client):**

```bash
printf 'GET / HTTP/1.1\r\nHost: example.com\r\nConnection: close\r\n\r\n' | nc example.com 80
```

**curl verbose** shows outgoing request headers and incoming response headers separately from bodies.

**ASP.NET Core read body:**

```csharp
app.MapPost("/echo", async (HttpRequest req) =>
{
    using var reader = new StreamReader(req.Body);
    var text = await reader.ReadToEndAsync();
    return Results.Text(text, "text/plain");
});
```

**Set response headers:**

```csharp
return Results.Json(dto, statusCode: 200)
    .WithHeader("Cache-Control", "no-store");
```

## Examples

**Example 1 — JSON POST.** Client sends:

```
POST /api/orders HTTP/1.1
Host: api.shop.com
Content-Type: application/json
Content-Length: 47

{"productId":7,"qty":2,"coupon":"SAVE10"}
```

Server responds with `201 Created`, `Location` header, JSON body — start line + headers + body again.

**Example 2 — No body GET.** Response:

```
HTTP/1.1 204 No Content
Cache-Control: no-store

```

Blank line, zero body bytes — still valid message.

**Example 3 — gzip.** Response headers include `Content-Encoding: gzip`; body bytes are compressed binary; client decompresses using Content-Length or chunked on compressed stream per spec rules.

## Common Mistakes & Anti-patterns

**Forgetting Host header** in hand-crafted HTTP/1.1 requests → 400 from many servers.

**Assuming `\n` alone is enough** — strict parsers and some load balancers fail.

**Reading body without checking Content-Type** — JSON parsed as text fails silently or throws.

**Ignoring charset** — `application/json` defaults UTF-8 in modern APIs; HTML needs `charset=utf-8` explicitly often.

**Manual Content-Length miscount** — off-by-one on byte length breaks next keep-alive message on same connection.

**Streaming entire upload into memory** — use `Request.Body` as stream with size limits for large files.

## Production & Real-World Notes

**Kestrel limits:** `Limits.MaxRequestBodySize`, `MaxRequestHeaderCount`, `MaxRequestLineSize` — tune for DoS protection.

**Middleware order** affects headers: exception handler may replace body; compression middleware wraps body stream.

**Problem Details** (`application/problem+json`) — standardized error body shape (RFC 7807).

**Reverse proxies** may buffer full body before forwarding; streaming WebSockets need special handling.

Logging: log **header names** carefully; never log full `Authorization` or cookies in production info level.

## Comparison / Trade-offs

**Content-Length vs chunked** — Fixed length simple for JSON APIs; chunked for streaming unknown size. HTTP/2+ framing reduces visible chunked on wire to client.

**Text HTTP/1.1 vs binary HTTP/2** — Human-readable vs efficient multiplexing; same semantic headers + body.

**Single large header vs many cookies** — Header size limits (~8–32 KB total varies) affect JWT in cookie vs split sessions.

## Quick Reference

```
REQUEST:  METHOD /path HTTP/1.1 \r\n + headers \r\n\r\n + [body]
RESPONSE: HTTP/1.1 CODE Reason \r\n + headers \r\n\r\n + [body]
```

| Delimit body | Header |
|--------------|--------|
| Fixed size | Content-Length |
| Streaming | Transfer-Encoding: chunked |
| No body | 204, HEAD rules, GET typically |

Mandatory request header HTTP/1.1: **Host**.

## Key Takeaways

- HTTP/1.1 message = **start line + headers + blank line + optional body**.
- Request start line: **method, target, version**; response: **version, status, phrase**.
- **CRLF** terminates lines; empty line ends headers.
- Body length via **Content-Length** or **chunked** — never guess from TCP stream alone.
- Header names **case-insensitive**; Content-Type defines body interpretation.
- Parsers are **state machines**; wrong length causes hang or truncation.
- HTTP/2 reframes same concepts into pseudo-headers and DATA frames.
