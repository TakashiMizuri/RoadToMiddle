# LINQ First vs Single vs FirstOrDefault — Semantics

> Roadmap: `1.3.27` · Node: `1.3` — C#: Advanced Topics · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- State **cardinality contracts** of **`First`**, **`Single`**, **`Last`**, and their **`*OrDefault`** variants.
- Predict which methods **throw**, which return **`default`**, and when **`InvalidOperationException`** vs **`ArgumentNullException`** applies.
- Explain **short-circuit enumeration** for **`First`** vs full scan for **`Single`** on sequences.
- Choose the correct operator for **business invariants** (unique email, optional header row, latest item).
- Map operators to **SQL** (`TOP 1`, unique constraint violations) on **`IQueryable`**.
- Avoid **`First()`** when **`Single()`** documents uniqueness, and **`Single()`** on unbounded duplicates.

---

## Why This Matters

These four methods look interchangeable to juniors — "give me one element". They encode **different promises** to readers and the runtime:

- **`First`** — "I need **some** match; order defines which."
- **`Single`** — "There must be **exactly one** match; otherwise it's a bug or data corruption."
- **`*OrDefault`** — "Zero matches is **OK**; return **`default`** instead of throwing."

Using **`First`** on a query that must be unique hides duplicate data until it causes financial or security bugs. Using **`Single`** on a filtered list that may legitimately be empty forces exceptions where **`SingleOrDefault`** fits. Middle developers read **`Single`** as an **assertion** about data integrity — similar to a unique index in the database.

## Core Concepts

### The Cardinality Matrix

| Method | 0 elements | 1 element | 2+ elements |
|--------|------------|-----------|-------------|
| `First` | throws | returns | returns first |
| `FirstOrDefault` | default(T) | returns | returns first |
| `Single` | throws | returns | throws |
| `SingleOrDefault` | default(T) | returns | throws |

**`Last`/`LastOrDefault`** mirror **`First`** but require the sequence to be fully enumerable (or buffered) unless the source is a list with O(1) tail access — **`Last`** on infinite sequence hangs.

All throwing cases use **`InvalidOperationException`** with messages like "Sequence contains no elements" or "Sequence contains more than one element" — not **`IndexOutOfRangeException`**.

### First — At Least One, Take the First

```csharp
var admin = users.First(u => u.Role == "Admin");
```

Stops at the **first** match — **does not** verify uniqueness. If zero matches → throw. If three admins → silently returns one — **may be wrong** if business rule says exactly one admin.

**`FirstOrDefault`** returns **`null`** for reference types (or **`0`**, **`default struct`**) when empty — **does not** distinguish "missing" from "legit default value" for value types without nullable wrapper.

### Single — Exactly One

```csharp
var settings = db.AppSettings.Single(s => s.TenantId == tenantId);
```

**`Single`** continues enumeration after the first match to detect a **second** — O(n) worst case even if duplicate is second element. This is **intentional**: it validates uniqueness at read time.

Use when a **unique constraint** should exist (primary key lookup by non-key field is suspicious — prefer **`First`** on key or **`Find`**).

**`SingleOrDefault`**: zero → default; one → element; many → throw. Ideal for "optional singleton" configs.

### Predicate Overloads

```csharp
sequence.First(predicate);
sequence.Single(predicate);
```

Equivalent to **`Where(predicate).First()`** but **stops early** for **`First`** when match found. For **`Single`**, must prove no second match — may scan entire sequence.

### Default Value and Nullability

```csharp
User? u = users.FirstOrDefault(x => x.Id == id);
if (u is null) { /* not found */ }
```

With **nullable reference types**, annotate as **`T?`**. For **`int`**, consider **`int?`** or **`Try`-pattern** because **`0`** might be valid.

**.NET 6+** **`FirstOrDefault(defaultValue)`** overload supplies explicit default when empty.

## Under the Hood

### Iterator Implementation Sketch

**`First`** (simplified):

```csharp
foreach (var item in source)
    return item; // no predicate
throw new InvalidOperationException("Sequence contains no elements.");
```

With predicate: return on first match; if loop completes → throw.

**`Single`**:

```csharp
T? found = default;
bool seen = false;
foreach (var item in source) {
    if (!predicate(item)) continue;
    if (seen) throw new InvalidOperationException("Sequence contains more than one matching element");
    found = item;
    seen = true;
}
if (!seen) throw ...;
return found;
```

Second matching element triggers throw — even if identical values (reference types: two distinct objects both match).

### IQueryable Translation

| LINQ | Typical SQL |
|------|-------------|
| `First()` / `First(predicate)` | `TOP 1` / `LIMIT 1` — **does not** check uniqueness server-side |
| `Single()` with key equality | often `TOP 2` or fetch one + client check; provider-dependent |
| `FirstOrDefault` | `TOP 1` nullable |

EF Core **`Single`** may translate to SQL that fetches limited rows and validates count in memory for some providers, or generates subqueries. **Never assume** SQL enforces **`Single`** semantics unless the query uses a **unique key**.

**`FirstAsync`/`SingleAsync`** — same semantics with async enumeration against DB.

### ValueTask and IAsyncEnumerable

**`AsyncEnumerable.FirstAsync`** follows same rules. For **`IAsyncEnumerable`**, cancellation propagates through **`WithCancellation`**.

## Syntax / Commands / API

```csharp
source.First()
source.First(predicate)
source.FirstOrDefault()
source.FirstOrDefault(predicate)
source.FirstOrDefault(defaultValue)          // .NET 6+
source.Single()
source.Single(predicate)
source.SingleOrDefault()
source.SingleOrDefault(predicate)

// Last / LastOrDefault — same cardinality, different position
```

Immediate execution — not deferred. Returns **`T`**, not **`IEnumerable<T>`**.

## Examples

### Lookup by Unique Key — Single

```csharp
var user = await db.Users.SingleAsync(u => u.Email == normalizedEmail, ct);
```

Documents expectation: email is unique (DB unique index should back this). Duplicate emails → exception surfaces data bug.

### Optional Config Row — SingleOrDefault

```csharp
var banner = await db.FeatureFlags
    .SingleOrDefaultAsync(f => f.Key == "HomeBanner", ct);
if (banner is null) return DefaultBanner;
```

Zero rows OK; two rows → throw (misconfiguration).

### Wrong: First When Uniqueness Required

```csharp
// BAD if Email must be unique:
var user = users.First(u => u.Email == email);
// Returns arbitrary duplicate; Single would throw
```

### Ordered First — Latest Order

```csharp
var latest = orders.OrderByDescending(o => o.CreatedAt).FirstOrDefault();
```

**`First`** after **`OrderBy`** = "latest" by sort key — not **`Last`** unless order is explicit ascending and you want tail.

## Common Mistakes & Anti-patterns

**`First` for unique lookup.** Use **`Single`** or query by primary key.

**`Single` on large table without index** — full table scan to detect duplicates.

**`FirstOrDefault` + null-forgiving `!`** without check — **`NullReferenceException`**.

**`Single()` on `GroupBy` result** when group may be empty — use **`SingleOrDefault`**.

**Assuming `Single` compares values** — two different objects matching predicate still throw.

## Production & Real-World Notes

Align LINQ choice with **DB constraints**: **`Single`** pairs with **`UNIQUE`** indexes. Integration tests should seed duplicate rows and assert **`Single`** fails loudly.

Logging: catch **`InvalidOperationException`** at API boundary only if converting to **404** vs **409 Conflict** — "no rows" vs "duplicate rows" have different HTTP semantics.

Performance: **`Any()` + `First()`** double enumeration — use one call or materialize.

## Comparison / Trade-offs

| Intent | Operator |
|--------|----------|
| Any match, order picks | `First` / `FirstOrDefault` |
| Exactly one must exist | `Single` |
| Zero or one OK | `SingleOrDefault` |
| Max/min by sort | `OrderBy` + `First` |
| PK lookup | `Find`, `First` on key, or `Single` if non-PK unique |

## Key Takeaways

- **`First`** = existence + first match; **does not** enforce uniqueness.
- **`Single`** = **exactly one**; scans for second match — data integrity signal.
- **`*OrDefault`** converts empty to **`default`** — handle null/zero explicitly.
- Immediate execution; **`First`** short-circuits, **`Single`** may not.
- Match operator to DB constraints and API error semantics.
