# Exercises: Tags and Semver Releases

> Complete after lectures and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Create both tag types
**Goal:** See object type difference.

**Steps:**
1. On `main`, create lightweight `demo-l` and annotated `demo-a` with message.
2. `git cat-file -t demo-l` and `demo-a`.
3. `git show demo-a` vs `git show demo-l`.

**Success criteria:** Table: tag name | cat-file -t | has tagger line in show?

---

### E2. Semver bump decision
**Goal:** Map commits to version.

**Given** current tag `v1.2.3`, write next tag for each commit message:

1. `fix: handle null userId`
2. `feat: export CSV dashboard`
3. `feat!: remove legacy /api/v1`

**Success criteria:** Three version strings with reasoning.

---

## Medium

### M1. Release tag workflow
**Goal:** End-to-end release ref.

**Steps:**
1. Branch `release-prep`, merge a small change to `main`.
2. After merge commit, annotated tag `v0.1.0` with release message.
3. Push branch and tag separately; verify `git ls-remote --tags origin`.

**Success criteria:** Remote lists tag; `git rev-parse v0.1.0^{commit}` matches merge commit.

---

### M2. Changelog between tags
**Goal:** Release notes source.

**Steps:**
1. Create tags `v0.1.0` and `v0.2.0` on two commits.
2. `git log --oneline v0.1.0..v0.2.0`
3. Draft release notes from Conventional Commits in range.

**Success criteria:** Markdown bullet list grouped by feat/fix.

---

## Hard (stretch)

### H1. Hotfix double tag
**Goal:** Connect `0.2.11` + semver.

Simulate: `main` at v3.0.0, `release/2.9` maintained. Document tag commands after cherry-pick hotfix to both lines (v3.0.1 and v2.9.8).

**Success criteria:** Four commands minimum; no retag of existing versions.

---

### H2. Tag policy document
Write team policy (1 page): annotated only, who pushes, protected tags, no `--tags` bulk push, signing optional requirement.

**Success criteria:** Addresses retag prohibition and CI gate before tag.
