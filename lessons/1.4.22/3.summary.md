# Benchmark sync vs async HTTP — краткая выжимка

> `1.4.22` · ~8 мин на повторение

## Главная идея

Async HTTP не делает **один** вызов быстрее — он **не держит поток thread pool** во время сетевого ожидания. Sync blocking под concurrency съедает workers и вызывает starvation, очереди и рост p95 latency.

## Ключевые пункты

- Синхронный HTTP: поток blocked в socket read → не доступен для других запросов.
- Async: IOCP + continuation; во время wait поток свободен.
- Один вызов sync ≈ async по времени; разница видна при **многих concurrent** вызовах.
- BenchmarkDotNet: `[MemoryDiagnoser]`, `[ThreadingDiagnoser]`, только **Release**.
- ASP.NET action должен `await`, не `.Result` / `.Wait()`.
- Локальный delay-server стабильнее публичного HTTP для lab.

## Самые важные примеры

```csharp
// Масштабируется
await client.GetStringAsync(url);

// Под load — thread pool starvation
client.GetStringAsync(url).GetAwaiter().GetResult();
```

```csharp
var tasks = Enumerable.Range(0, 200)
    .Select(_ => client.GetStringAsync(url));
await Task.WhenAll(tasks);
```

## Частые ошибки

- Бенчмарк одного запроса без concurrency.
- Debug configuration.
- Новый `HttpClient` на каждую итерацию.
- Вывод «async быстрее» из single-call benchmark.

## Мнемоника

**Async масштабирует ожидание; sync масштабирует потоки — а потоков мало.**
