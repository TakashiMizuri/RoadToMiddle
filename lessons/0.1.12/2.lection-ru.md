# Git clean: удаление untracked files

> Roadmap: `0.1.12` · Узел: `0.1` — Git: основы · Глубина: практика

## Цели урока

После этой лекции ты сможешь:

- Объяснить, что **`git clean`** удаляет (и что **никогда** не трогает).
- Использовать **dry-run** (`-n`) перед каждым destructive clean.
- Применять флаги **`-f`**, **`-d`**, **`-x`** по назначению.
- Отличать **`git clean`** от **`git restore`**, **`git rm`**, **`git stash -u`**.
- Понимать взаимодействие с **`.gitignore`** (default vs `-x`).
- Безопасно чистить build artifacts в реальном workflow.

---

## Зачем это нужно

Ты собрал .NET solution, запустил frontend dev server, может быть выполнил разовый SQL-скрипт. `git status` завален paths, которые Git не track'ит: `bin/`, `obj/`, `scratch.sql`, временный log, возможно `node_modules/` при неполном `.gitignore`. Ни один из этих файлов не попадёт в commit, пока ты явно не сделаешь `git add`, но они засоряют output, мешают заметить реальные изменения перед commit и иногда попадают в `git add .` по ошибке.

Нужен **clean working tree** — но знакомые инструменты решают другие задачи. **`git restore`** отбрасывает изменения **tracked** files; файл, о котором Git не знает, не удалит. **`git rm`** убирает **tracked** files из index и working tree, staging deletion для следующего commit. **`git stash -u`** сохраняет untracked в stash stack, а не удаляет. **`git clean`** — команда, чья работа **удалить untracked files и directories с диска**.

Deletion **permanent** с точки зрения Git. Нет recycle bin, нет undo stack, нет commit с записью удалённого. Удалил afternoon uncommitted prototype через `git clean` — recovery зависит от IDE local history, backup'ов или удачи, не от Git. С **`git clean -n`** (dry-run) сначала — essential daily hygiene. Без preview — один из fastest способов потерять never-committed work.

В fullstack-проекте ASP.NET Core + React картина типична: после `dotnet build` и `npm run dev` в status или в dry-run появляются `bin/`, `obj/`, `dist/`, `.vite/`, локальные `.env` (если в `.gitignore`), временные SQL-дампы. Без clean status output превращается в шум; с неосторожным clean можно снести `.env` с connection string, который никогда не был в repo, но нужен для локального запуска. Отсюда правило: **`-x` только после `-ndx` и осознанного чтения каждой строки**.

## Основные концепции

### Три категории файлов в working tree

Git классифицирует каждый path в project directory; инструмент removal зависит от категории.

**Tracked files** — paths в index или в tree commit'а `HEAD`; Git actively manages. Изменил `Program.cs` после commit — Git покажет modified. Removal или revert tracked content — **`git rm`** (перестать track'ить и удалить) или **`git restore`** (discard working tree changes к `HEAD`).

**Untracked, non-ignored** — paths на диске, но не в index и не в last commit — и **не** matched `.gitignore`. Scratch `temp.log`, папка `experiments/` — territory **`git clean`**.

**Ignored files** — patterns из `.gitignore`. Build output `bin/`, `obj/` в типичном .NET — здесь. По умолчанию **`git clean` skip'ает ignored** — invisible для команды, пока не передашь **`-x`**. Default защищает от случайного удаления `.env` и secrets, которые never commit, но must stay on disk.

Invariant: **`git clean` never removes tracked files**, даже heavily modified. Только paths, которые Git считает untracked. `modified: Program.cs` в status — clean не тронет.

### Dry-run — non-negotiable habit

Deletion irreversible — Git даёт preview. **`-n`** (`--dry-run`) list'ит, что **would** remove, без touch filesystem:

```bash
git clean -n
git clean -nd
```

Output с prefix `Would remove`. Прочитай каждую строку. Safe to lose? Только потом те же flags с **`-f`** вместо **`-n`**. Professionals treat `-n` как part of command — как preview перед `DROP TABLE`.

Представь dry-run как **контракт с самим собой**: ты явно подтверждаешь список жертв. Если в preview есть сюрприз — `.env`, `appsettings.Local.json`, папка с несохранённым прототипом — остановись. Добавь нужное в `.gitignore`, сделай `git stash -u`, или commit, **до** `-f`. Dry-run не замедляет работу; он предотвращает единственный необратимый сценарий в этой команде.

### Force flag и почему Git требует его

Actual deletion — **`-f`** (force). Без `-f` clean refuse (кроме dry-run). Git заставляет typed flag «понимаю, destructive». Комбинации: **`-fd`** — untracked files + directories; **`-fdx`** — + ignored («nuclear» для build artifacts, скрытых `.gitignore` от normal status).

## Под капотом

При `git clean` Git обходит working directory от root repository (или указанного path), evaluate'ит каждый untracked path против index, tree `HEAD` и ignore rules, затем filesystem delete (`unlink` для files, recursive removal для directories при `-d`).

Decision logic для каждого path: tracked в index — skip, даже если working copy differs. Match ignore rule без `-x` — skip. Untracked directory без `-d` — skip directory. Прошёл filters и есть `-f` — delete с диска.

**`.gitignore`** surprises в обе стороны. Допустим:

```
bin/
obj/
.env
```

**`git clean -fd`** удаляет untracked **не** ignored. **`bin/`, `obj/`, `.env` не тронет** — ignore exclude из default scope. **`git clean -fdx`** с `-x` — «also remove ignored». Теперь `bin/`, `obj/`, `.env` в dry-run preview и будут deleted. Именно это нужно для «fresh clone» после corrupted build — и именно **не** нужно, если `.env` с database credentials.

Редкий **`-X`** (capital X): remove **only** ignored, non-ignored untracked оставить. Targeted cleanup build output без scratch scripts в repo root.

Internally clean не создаёт commits, не update index, не touch refs. Purely working directory operation. `git log` и `git reflog` не вернут cleaned files — Git never recorded их как objects (если не add+commit — тогда tracked, clean не удалил бы).

Важный edge case: **untracked file inside tracked directory**. Если `src/` tracked, а `src/scratch.txt` untracked, clean удалит `scratch.txt` при `-f`. Если `experiments/` целиком untracked directory, без `-d` Git может не зайти внутрь — отсюда habit `-nd` вместо `-n`. Ещё edge case: **nested git repos** (submodule или случайный `.git` в подпапке) — clean ведёт себя иначе; в monorepo всегда смотри preview.

Interactive mode (`git clean -i`) полезен, когда untracked много, но удалить нужно выборочно: Git группирует paths и спрашивает confirmation по группам. Медленнее, но безопаснее blind `-fd` на незнакомом repo после clone legacy-проекта.

## Синтаксис

```bash
# Dry-run — always start
git clean -n
git clean -nd
git clean -nx

# Execute
git clean -f
git clean -fd
git clean -fdx             # + ignored (bin, obj, node_modules, .env if ignored)
git clean -fdX             # only ignored

# Interactive
git clean -i
git clean -id
```

**`-n`** dry-run; **`-f`** force; **`-d`** directories; **`-x`** ignored; **`-X`** only ignored; **`-i`** interactive. **`git clean -fdx`** и **`git clean -xdf`** equivalent.

Advanced: **`-e`** exclude specific paths в monorepo.

## Примеры

### Пример 1: Безопасный daily workflow

Эксперимент закончен — убрать scratch перед новой задачей:

```bash
git status
git clean -nd              # "Would remove scratch.txt", "Would remove tmp/"
# read carefully
git clean -fd              # execute if preview OK
```

Preview → execute — пять секунд, prevents wrong folder stories. Muscle memory.

Разберём output dry-run построчно. Строка `Would remove scratch.sql` — untracked file в корне, не ignored: default `-fd` удалит. `Would remove tmp/` — directory: без `-d` в execute не удалилась бы вся папка; `-nd` показывает, что `-d` нужен. Если `-ndx` показывает `Would remove bin/` — это ignored artifact; execute потребует `-x`. Если в preview внезапно `.env` — stop: либо exclude, либо backup, либо не используй `-x`.

### Пример 2: Reset build artifacts после corrupted .NET build

`bin/`, `obj/` в `.gitignore` — не в plain `git status`, но disk space и stale build errors. Pristine tree:

```bash
git clean -ndx
git clean -fdx
dotnet restore
dotnet build
```

После `-x` — reinstall npm/NuGet. Rebuild cost intentional — certainty, что stale artifact не остался.

Почему `bin/` не виден в обычном `git status`? Потому что `.gitignore` скрывает ignored paths от status и от default clean. Это feature, не bug: build output не должен засорять status. Но когда tree «грязный» ignored файлами, status выглядит clean при том, что disk забит. Тогда `-ndx` — единственный способ увидеть полную картину перед nuclear `-fdx`. После clean с `-x` закладывай время на restore: `dotnet restore`, `npm ci`, иногда `docker compose build` — это плата за confidence, что локальная среда совпадает с fresh clone.

### Пример 3: Правильный инструмент

`Program.cs` tracked, edits плохие; `temp.sql` untracked scratch; `OldService.cs` tracked, obsolete, must leave repo:

```bash
git restore Program.cs          # tracked — NOT clean

git clean -fd                   # untracked scratch — clean
# или git clean -f -- temp.sql

git rm OldService.cs              # tracked removal — NOT clean
git commit -m "Remove obsolete OldService"
```

Confusion этих трёх — most common clean mistake. «Git track'ит path?» Tracked modification → **restore**. Tracked removal → **rm**. Untracked on disk → **clean**.

Проведём contrast явно. **`git restore Program.cs`** говорит: «верни tracked file к версии из HEAD/index» — файл остаётся tracked, содержимое меняется. **`git rm OldService.cs`** говорит: «убери file из repo и staging» — deletion станет частью следующего commit. **`git clean -fd`** говорит: «удали с диска всё, о чём Git не ведёт учёт (с учётом ignore rules)» — repo history не меняется, потому что эти paths никогда не были в commits. Три разные оси: tracked/untracked, keep/delete, history/no history.

### Пример 4: Когда stash beats clean

Untracked files might need tomorrow — WIP notes, generated config не ready to commit. **`git stash push -u`** saves в stash, clean tree без deletion. **`git clean`** destroy. Uncertain — stash first; clean when sure deletion OK.

### Пример 5: Полный сценарий «как на CI»

Коллега говорит: «у меня build падает, у тебя проходит». Ты подозреваешь stale artifacts:

```bash
git status
git clean -ndx                    # смотрим всё, включая ignored
# убеждаемся, что .env не в списке — или сознательно исключаем
git clean -fdx
dotnet restore
dotnet build --no-incremental
cd app && npm ci && npm run build
```

Если после этого build проходит — проблема была в local debris, не в коде. Если нет — дело не в clean. Такой сценарий повторяют десятки раз в карьере; dry-run каждый раз занимает секунды, одна ошибка без preview — часы восстановления `.env` и dependencies.

## Частые ошибки

Skip dry-run hurts most. **`git clean -fdx`** в repo root без `-ndx` — `.env.local`, hours of `node_modules`, uncommitted prototypes одной командой. Clean вместо revert modified tracked — `Program.cs` stays modified. Clean вместо save — **stash -u** preserves. Clean вместо commit wanted work — irreversible без backup.

Subtle trap: `.gitignore` не защищает files you care about, если не ignored. `important-draft.md` never committed, not in `.gitignore` — untracked non-ignored — **`git clean -fd`** delete. Important scratch — commit, stash, или conscious ignore rule.

Ещё ошибка — запуск clean в **неправильной directory**. `git clean` работает относительно root repository (или указанного path). Если ты в subdirectory и думаешь, что clean'ишь только её — проверь preview: по умолчанию команда может затронуть весь repo. В monorepo используй path argument или `-e` exclude patterns после dry-run.

Команда **`git clean -fdX`** (capital X) удаляет **только** ignored files — полезно, когда нужно снести `bin/` и `node_modules/`, но оставить untracked `notes.txt` в корне. Реже `-fdx`, но worth knowing при selective cleanup.

## Production notes

Teams script **`git clean -fdx && dotnet restore`** (or `npm ci`) для CI-like pristine tree при «works on my machine». CI — fresh clone each run; clean simulates locally.

Never **`-x`** на monorepo root без preview — gitignored secrets и local overrides — targets `-x`. Solid **`.gitignore`** в repo; clean не substitute proactive ignore.

IDE **Local History** (Rider, VS Code) sometimes recovers deleted untracked — last resort, not strategy. Onboarding mantra: **`status` → `clean -nd` → read → `clean -fd`**.

«Nuclear reset» alias в wikis:

```bash
git reset --hard HEAD && git clean -fdx
```

Wipes tracked modifications **and** all untracked/ignored debris. Only when fully accept losing all uncommitted work.

Перед code review полезно `git clean -fd` после `git status`: в diff остаются только намеренные tracked changes, scratch не попадёт в PR случайно.

### Практический checklist перед `-fd` / `-fdx`

Перед любым destructive clean пройди мысленно пять пунктов. Первый: есть ли untracked work, который не в `.gitignore` и не в commit — notes, drafts, generated configs? Если да — stash или commit. Второй: нужен ли `-d` для directories или достаточно `-f`? Третий: нужен ли `-x` — и если да, проверил ли `.env`, `.env.local`, secrets в preview? Четвёртый: запускал ли `-n` / `-nd` / `-ndx` с **теми же flags**, что будут в execute? Пятый: есть ли backup или IDE Local History для worst case?

Этот checklist занимает полминуты. Он скучный — и именно поэтому спасает от единственной необратимой ошибки в workflow с clean.

Если работаешь в Windows, помни: `git clean` удаляет файлы мимо Recycle Bin — восстановление только через backup или IDE Local History. На WSL paths в preview должны совпадать с тем, где реально лежат файлы; сюрпризы редки, но dry-run всё равно обязателен каждый раз.

## Сравнение подходов

**`git clean -fd`** — untracked non-ignored. Reversible только backup/IDE — не Git. **`git clean -fdx`** — + ignored; full rebuild deps. **`git restore .`** — tracked к `HEAD`; untracked не трогает. **`git stash -u`** — untracked в stash; `stash pop` reverses. **`git rm`** — tracked removal + stage; до commit reversible через restore.

Clean — files **gone**. Stash — **preserved but out of way**. Restore — tracked changed, untracked не problem.

Итог для daily practice: **`git status`** показывает проблему, **`git clean -nd`** показывает лекарство, **`git clean -fd`** применяет его — только после прочтения preview. Для build artifacts добавь `-x` только когда готов к `dotnet restore` / `npm ci`. Эта тройка команд закрывает 90% локальных «грязных tree» ситуаций без риска для tracked work и без потери secrets при дисциплине dry-run.

## Шпаргалка

Preview: `git clean -nd`. Untracked respect ignore: `git clean -fd`. + build output: `git clean -fdx` after `git clean -ndx`. Interactive: `git clean -i`. Golden: **`git status` → `git clean -nd` → read → `git clean -fd`**.

---

## Главное

- **`git clean`** — **untracked** с диска; never tracked modifications.
- **`-n` перед `-f`** always.
- **`-d`** — directories; **`-x`** — ignored (осторожно).
- Default **respects `.gitignore`**.
- Не путать с **restore / rm / stash -u**.
- Deletion **permanent** для never-committed files.
