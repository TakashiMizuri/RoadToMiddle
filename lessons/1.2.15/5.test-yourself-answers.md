# Answers: readonly struct, ref struct

## Scoring

- 90–100%: отлично
- 80–89%: pass
- <80%: Core + Under the Hood

---

## Answers

### 1. [Recall]
**Answer:** Instance state **не изменяется** после construction; все instance fields readonly; instance methods не мутируют fields.
**Why:** Compiler-enforced immutability.

### 2. [Recall]
**Answer:** Примеры: **нельзя box**; **нельзя** (в общем случае) поле обычного class; **нельзя** безопасно capture в async/iterator; ограничения generics.
**Why:** Stack-only lifetime safety.

### 3. [Recall]
**Answer:** **`ref struct`** (special category of value type, stack-only).
**Why:** Not ordinary struct for heap/box rules.

### 4. [Understanding]
**Answer:** Boxing положил бы view в heap с lifetime дольше underlying buffer → **dangling reference**.
**Why:** CLR prevents use-after-free.

### 5. [Understanding]
**Answer:** Class может пережить stack buffer; `Span` как field продлит view → **memory safety violation** — CS8345.
**Why:** ref struct must not outlive stack allocation incorrectly.

### 6. [Understanding]
**Answer:** **`in`** передаёт readonly **reference** — избегает копирования  большого struct.
**Why:** Performance при immutability.

### 7. [Application]
**Answer:** **`Memory<byte>`** или **`byte[]`** — heap lifetime, safe в instance field; `Span` только как local derived from them.
**Why:** Long-lived storage needs heap.

### 8. [Application]
**Answer:** `public readonly record struct Point(int X, int Y);`
**Why:** Positional readonly record struct syntax.

### 9. [Application]
**Answer:** **`ReadOnlySpan<char>`** overload — меньше allocation при slice parsing.
**Why:** Zero-allocation API pattern.

### 10. [Analysis]
**Answer:** **B** неверно — `ref struct` **не может** быть полем обычного class (без специальных ref field rules).
**Common mistake:** Treat Span like normal field.

### 11. [Analysis]
**Answer:** `await` разрешает continuation после stack frame; local `Span` указывал бы на **invalid stack memory**.
**Why:** Lifetime mismatch (rules relaxing in newer C# with constraints).

### 12. [Understanding]
**Answer:** **`ref`** — read/write alias; **`in`** — readonly alias, callee cannot mutate (except via `ref` escape tricks blocked).
**Why:** `in` = cheap read-only pass.

### 13. [Application]
**Answer:** **Да** — `Add` returns `new Money(...)`; fields readonly; fits value object pattern.
**Why:** Immutability enforced.

### 14. [Analysis]
**Answer:** **Нет полностью** — readonly убирает mutation/defensive copy issues, но **200-byte copy** при by-value pass остаётся — нужен `in`/`ref` или smaller type/class.
**Why:** readonly ≠ eliminates copy cost (`1.2.14`).

### 15. [Application]
**Answer:** **`stackalloc` + Span** — нет GC pressure для small bounded buffer; `new char[128]` — heap allocation. Trade-off: stack overflow risk if huge/recursive.
**Why:** Typical local optimization.
