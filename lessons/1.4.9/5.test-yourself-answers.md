# Answers: SynchronizationContext

## Answers

### 1. [Recall]
**Answer:** `Post` — async queue callback; `Send` — sync dispatch (blocks until run).
**Why:** UI frameworks override для message pump.

### 2. [Recall]
**Answer:** **`null`**.
**Why:** Core deliberately не устанавливает request context.

### 3. [Understanding]
**Answer:** Захватывает `Current` при await; на completion `Post` вызывает continuation → **`MoveNext` state machine**.
**Why:** `1.4.7` — тот же callback, другой scheduler.

### 4. [Understanding]
**Answer:** Thread agility для I/O scalability; нет ложной «request thread affinity».
**Why:** Kestrel model — pool threads обслуживают many requests.

### 5. [Understanding]
**Answer:** **`AsyncLocal<HttpContext>`** через `HttpContextAccessor` / middleware setup.
**Why:** Logical call context, не SyncContext.

### 6. [Application]
**Answer:** **Cross-thread UI access** — race или exception.
**Why:** Нет capture → не UI thread.

### 7. [Application]
**Answer:** **SyncContext deadlock** (Framework): block request thread, continuation Post на тот же context.
**Why:** Classic AspNetSynchronizationContext scenario.

### 8. [Analysis]
**Answer:** **Неверно** — HttpContext не через SyncContext.
**Common mistake:** Путать capture policies.

### 9. [Analysis]
**Answer:** **Thread pool** (или inline если allowed).
**Why:** `Current == null` → no Post.

### 10. [Understanding]
**Answer:** SyncContext — **scheduling**; ExecutionContext — **ambient data** (`AsyncLocal`, culture, security).
**Why:** `ConfigureAwait` affects only SyncContext.

### 11. [Application]
**Answer:** **UI-like** — component dispatch single-threaded per circuit.
**Why:** Blazor Server имеет dispatcher semantics.

### 12. [Recall]
**Answer:** **`WindowsFormsSynchronizationContext`**.
**Why:** Installed on UI thread at app start.
