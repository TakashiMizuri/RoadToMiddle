# IDisposable vs Finalizer — Dispose Pattern

> Roadmap: `1.5.10` · Node: `1.5` — C#: .NET Runtime и память · Depth: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Различать **deterministic disposal** (`IDisposable`) и **non-deterministic finalization**; объяснить, почему GC не заменяет `Dispose`.
- Реализовать **стандартный dispose pattern** для sealed/unsealed типов: **`Dispose(bool disposing)`**, **`GC.SuppressFinalize(this)`**.
- Описать **finalization queue**, **f-reachable queue** и почему finalizer **удлиняет** lifetime минимум на один GC-цикл.
- Решать, когда finalizer **оправдан** (`SafeHandle`), а когда — anti-pattern.
- Связать с **`1.3.16`** (`using`, `await using`) как consumer-side контракта.
- Диагностировать **double dispose**, use-after-dispose, **resurrection**.

---

## Why This Matters

Managed memory освобождается при unreachable. **Unmanaged resources** — descriptors, sockets, DB handles — **вне** основной модели GC. Ten thousand `FileStream` без `Dispose` → OS отказывает в handles, **connection pool exhaustion**, locked files — **до** OOM.

`IDisposable` — **prompt cleanup под контролем разработчика**. Finalizers — **страховка**, когда caller забыл: **позже**, на **отдельном потоке**, **непредсказуемый порядок**, **дополнительные GC-фазы**.

Урок глубже **`1.3.16`**: **почему** pattern такой, **что делает runtime**, **base/derived** иерархии.

## Core Concepts

### Два канала cleanup

1. **`Dispose()`** — вызов или `using`. **Deterministic**, на calling thread, сразу. Release handle, **`SuppressFinalize`**.

2. **Finalizer `~Type()`** — регистрация при allocation. При unreachable → **finalization queue**, **re-root**. **Finalizer thread** позже — секунды под load.

**GC не вызывает `Dispose`**. Finalizer и `Dispose` связаны только через **`SuppressFinalize`**.

### Standard Dispose Pattern

```csharp
public class ResourceHolder : IDisposable
{
    private IntPtr _handle;
    private bool _disposed;

    public void Dispose()
    {
        Dispose(disposing: true);
        GC.SuppressFinalize(this);
    }

    protected virtual void Dispose(bool disposing)
    {
        if (_disposed) return;
        _disposed = true;

        if (disposing)
        {
            // managed IDisposable children
        }

        if (_handle != IntPtr.Zero)
        {
            NativeMethods.CloseHandle(_handle);
            _handle = IntPtr.Zero;
        }
    }

    ~ResourceHolder()
    {
        Dispose(disposing: false);
    }
}
```

**`disposing == true`** — из public `Dispose`; можно трогать **managed** disposables (с guard).

**`disposing == false`** — из finalizer; **только прямой unmanaged** — другие managed могут быть уже finalized.

**`SuppressFinalize`** — не тратить finalizer queue, если cleanup уже сделан.

**Sealed managed-only** — упрощённый pattern из **`1.3.16`**, без finalizer.

### Когда finalizer

**Да:** прямой unmanaged, callers могут забыть `Dispose`. **Предпочтительно `SafeHandle`** — finalizer на handle, не на outer class.

**Нет:** только managed; "на всякий случай"; logging в finalizer.

### Derived classes

```csharp
protected override void Dispose(bool disposing)
{
    if (disposing)
        _stream?.Dispose();
    base.Dispose(disposing);
}
```

Public `Dispose()` **не virtual**; **`Dispose(bool)` protected virtual**. Derived finalizer нужен редко — если **свой** unmanaged не через SafeHandle.

### IAsyncDisposable

См. **`1.3.16`**, **`1.4.35`**. `DisposeAsync` → `DisposeAsyncCore` → при необходимости sync `Dispose` + `SuppressFinalize`. Без `.Wait()` на I/O.

## Under the Hood

Unreachable + finalizer → **не collect сразу** → **finalization queue** → re-root → finalizer thread → снова unreachable (если нет **resurrection**).

**f-reachable** backlog под load → память и handles дольше живут.

Finalizer registration — cost **на каждую** allocation до `SuppressFinalize`.

**Resurrection** — `this` в static из finalizer — почти всегда bug.

**Thread safety:** `_disposed`; concurrent dispose — lock/`Interlocked` по необходимости.

---

## Common Mistakes & Anti-patterns

Finalizer на managed-only. Managed dispose в `Dispose(false)`. Public virtual `Dispose()`. Throw из finalizer. Забытый `SuppressFinalize`. Use-after-dispose без `ObjectDisposedException`.

## Production & Real-World Notes

ASP.NET **scoped** dispose в конце request. Singleton без per-request disposable без factory. **HttpClient** через `IHttpClientFactory`. Code review: ownership каждого handle.

## Comparison / Trade-offs

| Подход | Плюсы | Минусы |
|--------|-------|--------|
| `using` на BCL | Просто | Нет custom types |
| Sealed IDisposable | Ясно | Нет inheritance |
| Full pattern | Extensible | Boilerplate |
| Finalizer | Safety net | Медленно |
| SafeHandle | Централизованный native | Learning curve |

## Quick Reference

```
Dispose() → Dispose(true) → SuppressFinalize
~Type()   → Dispose(false) → только unmanaged
Derived   → override Dispose(bool), base.Dispose
Consumer  → using / await using (1.3.16)
```

---

## Key Takeaways

- **Dispose now; finalize later.**
- Full pattern: public Dispose, protected virtual Dispose(bool), optional finalizer, SuppressFinalize.
- Finalizers **дороги** и **non-deterministic**.
- **`SafeHandle`** для native.
- **`disposing`** разделяет managed vs unmanaged paths.
