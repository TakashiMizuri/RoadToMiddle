# CancellationToken in ASP.NET — RequestAborted

> Roadmap: `1.4.29` · Node: `1.4` — C# async/await · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Explain what **`HttpContext.RequestAborted`** represents and when it fires.
- Thread **`CancellationToken`** from controller/minimal API through services to EF Core and `HttpClient`.
- Distinguish **client disconnect** from **server shutdown** and **explicit timeout** cancellation.
- Avoid wasted CPU, DB connections, and downstream calls after the client is gone.
- Recognize when **not** to cancel (side effects, fire-and-forget) and how to document that choice.

---

## Why This Matters

A user clicks "Search" and immediately navigates away. Without cancellation, your API keeps querying PostgreSQL, calling a third-party API, and holding a thread-pool continuation — work nobody will ever see. Under load, this becomes **connection pool pressure**, **higher latency** for active users, and **money burned** on external APIs.

ASP.NET Core wires **`RequestAborted`** into every HTTP request. Middle developers must treat it as a first-class parameter — not an optional nicety — and propagate it **async all the way** to I/O boundaries. This is the production complement to `CancellationToken` basics from `1.4.11`: same type, different **lifecycle owner** (Kestrel + client TCP state, not your `CancelAfter` timer).

## Core Concepts

### What RequestAborted Is

Every HTTP request in ASP.NET Core exposes **`HttpContext.RequestAborted`**, a **`CancellationToken`** that transitions to the **cancelled** state when:

- The **client closes the connection** (browser tab closed, mobile app backgrounded, load balancer idle timeout).
- The **server is stopping** (graceful shutdown via `IHostApplicationLifetime.ApplicationStopping` links into request tokens).
- Middleware or your code **links** another token that fires (e.g. Polly timeout, custom deadline).

It does **not** mean "the user pressed Cancel in the UI" unless the client actually **aborts the HTTP request** (e.g. `AbortController` in `fetch`, Axios cancel token). Many SPAs simply ignore in-flight responses — then cancellation only happens on navigation/unmount if you wired it.

The token is **cooperative**: nothing stops your `for` loop unless **you** check `token.IsCancellationRequested`, pass the token to `Task.Delay`, `HttpClient.SendAsync`, `DbContext` queries, etc.

### Propagation Pattern

The canonical pattern is **parameter threading**:

```csharp
app.MapGet("/orders/{id}", async (int id, IOrderService orders, CancellationToken ct) =>
{
    return await orders.GetOrderAsync(id, ct);
});
```

ASP.NET Core model binding supplies **`CancellationToken`** automatically when the parameter is named `cancellationToken` or typed as `CancellationToken` — you do not need `[FromServices]` or attributes. The same token is **`HttpContext.RequestAborted`** (or a linked composite during shutdown).

Your service layer should accept **`CancellationToken cancellationToken = default`** on every I/O method and pass it downstream. **`default`** means "no cancellation" for unit tests and console callers; in a web request, the framework injects the real token.

### What Gets Cancelled — and What Does Not

When the token fires, **in-flight I/O** that respects cancellation throws **`OperationCanceledException`** (often wrapping **`TaskCanceledException`** from HTTP). EF Core translates cancelled commands; Npgsql closes the command; `HttpClient` aborts the socket read.

**CPU-bound work** on the thread pool does not magically stop. A tight loop must poll the token:

```csharp
for (var i = 0; i < huge; i++)
{
    cancellationToken.ThrowIfCancellationRequested();
    // work
}
```

**Side effects already committed** are not rolled back by cancellation. If you charged a card and then the client disconnected, cancellation does not undo the payment — you need **idempotency** and **compensation**, not a token.

### Linked Tokens and Timeouts

Production APIs often combine **client abort** with a **server-side deadline**:

```csharp
using var cts = CancellationTokenSource.CreateLinkedTokenSource(
    httpContext.RequestAborted);
cts.CancelAfter(TimeSpan.FromSeconds(30));
var token = cts.Token;
await _db.Orders.ToListAsync(token);
```

Whichever fires first wins. **Link disposal** releases registrations; `using` on the `CancellationTokenSource` is important when you create linked sources per request.

## Under the Hood

### Kestrel and Connection Lifetime

Kestrel tracks each connection's **`ConnectionAborted`** token. When the client resets TCP (RST) or completes a half-close that implies abandonment, Kestrel cancels the request's token. This is **not instantaneous** on all platforms — there can be a short delay before the stack detects a dead peer.

For **HTTP/2** and **HTTP/3**, stream reset maps to request cancellation. Multiplexing means one connection carries many requests; only the affected stream's token fires.

### Middleware and Hosting

During **graceful shutdown**, the host signals cancellation so in-flight requests can finish within **`ShutdownTimeout`** or abort. `BackgroundService` operations are separate — they use **`stoppingToken`** from `ExecuteAsync`, not `RequestAborted`.

### EF Core and Database Providers

When cancellation reaches EF Core, the provider sends cancellation to the underlying ADO.NET command. PostgreSQL supports query cancel via a separate cancel message on the connection. The **`OperationCanceledException`** you catch in the app may surface **after** the DB already did partial work — another reason cancellation is not a transaction rollback.

### Exception Semantics

**`OperationCanceledException`** on cancelled requests is often **expected**, not a bug. Many teams filter it in exception middleware:

```csharp
catch (OperationCanceledException) when (context.RequestAborted.IsCancellationRequested)
{
    // client gone — don't log as 500
}
```

Logging every client disconnect as **Error** creates alert noise. At scale, treat it as **Information** or do not log at all.

## Syntax / Commands / API

| API | Role |
|-----|------|
| `HttpContext.RequestAborted` | Per-request token from Kestrel/host |
| `CancellationToken` action parameter | Automatic injection in controllers/minimal APIs |
| `ThrowIfCancellationRequested()` | Fast fail in CPU loops |
| `CreateLinkedTokenSource` | Combine abort + timeout + custom |
| `IHttpClientFactory` + `SendAsync(..., ct)` | Propagate to outbound HTTP |
| `ToListAsync(ct)`, `SaveChangesAsync(ct)` | EF Core cancellation |

## Examples

### Minimal API with EF Core

```csharp
app.MapGet("/products", async (AppDbContext db, CancellationToken ct) =>
{
    return await db.Products.AsNoTracking().ToListAsync(ct);
});
```

If the client disconnects mid-query, EF/Npgsql aborts the read; the delegate throws `OperationCanceledException`. ASP.NET Core typically does not write a response body — the connection is already gone.

### Service Layer

```csharp
public async Task<OrderDto?> GetOrderAsync(int id, CancellationToken ct)
{
    var order = await _db.Orders
        .AsNoTracking()
        .FirstOrDefaultAsync(o => o.Id == id, ct);

    if (order is null) return null;

    // Outbound call — same token
    var shipping = await _http.GetFromJsonAsync<ShippingStatus>(
        $"/status/{order.TrackingId}", ct);

    return Map(order, shipping);
}
```

Each `await` passes `ct` so cancellation propagates through the **async chain** without manual plumbing at every layer if you are disciplined about signatures.

### Controller (same mechanism)

```csharp
[HttpGet("{id}")]
public async Task<ActionResult<OrderDto>> Get(int id, CancellationToken cancellationToken)
{
    var dto = await _orders.GetOrderAsync(id, cancellationToken);
    return dto is null ? NotFound() : Ok(dto);
}
```

## Common Mistakes & Anti-patterns

**Ignoring the token entirely** is the most common mistake. Methods accept `CancellationToken` but never pass it to I/O — the parameter becomes documentation theater.

**Catching `OperationCanceledException` and rethrowing as 500** trains operators to ignore real errors and hides client-driven aborts from metrics.

**Using `CancellationToken.None` in library code** when the caller passed a token — always forward the caller's token unless you have a documented reason to detach (rare).

**Assuming cancellation rolls back transactions.** Use explicit `IDbContextTransaction` and business rules; cancel only stops **waiting** for more work.

**Starting long work after the response is sent** without a separate token — `RequestAborted` may fire as soon as the response completes; background work needs **`IHostApplicationLifetime`** or a queue (`1.4.34`), not the request token alone.

## Production & Real-World Notes

Teams often add a **global request timeout** middleware (e.g. 30s) linked to `RequestAborted` so slow dependencies cannot hold resources forever even if the client stays connected.

**Load balancers** (ALB, nginx) have idle timeouts shorter than your DB query — the client may still appear connected through the LB while the app has already been cancelled from the LB's perspective. Align timeouts across layers.

**Observability**: tag spans with `cancellation_requested=true` when handling `OperationCanceledException` so you can separate **SLA violations** from **benign disconnects**.

For **minimal APIs**, consider **`[AsParameters]`** DTOs that include `CancellationToken` as a property — binding still works.

## Comparison / Trade-offs

| Approach | Pros | Cons |
|----------|------|------|
| Thread `RequestAborted` everywhere | Saves resources, standard | Verbose signatures; must discipline team |
| Server timeout only | Predictable max latency | Client may still wait until timeout if connected |
| No cancellation | Simpler code | Wastes resources under churn |
| Fire-and-forget after response | Fast UX | Must **not** use `RequestAborted` alone (`1.4.30`) |

## Quick Reference

- Inject: `CancellationToken ct` in endpoint → pass to all I/O.
- Source: `HttpContext.RequestAborted` ≡ endpoint `CancellationToken` in ASP.NET Core.
- Fires on: client disconnect, host shutdown, linked token cancel.
- Catch: `OperationCanceledException` when `RequestAborted.IsCancellationRequested` — usually not 500.
- Link: `CreateLinkedTokenSource(RequestAborted)` + `CancelAfter` for deadlines.
- EF/HttpClient: always pass token into `*Async(..., cancellationToken)`.

---

## Key Takeaways

- **`RequestAborted`** is the request-scoped cancellation signal from Kestrel/hosting.
- Cancellation is **cooperative** — pass the token to every I/O call.
- Client disconnect ≠ exception bug; filter **`OperationCanceledException`** appropriately.
- Combine with **timeouts** via linked `CancellationTokenSource`.
- Cancellation does **not** undo committed side effects or replace transactions.
- Propagate through services, EF Core, and `HttpClient` for real resource savings.
