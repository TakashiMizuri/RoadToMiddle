# Constructors, `: this()`, Primary Constructors

> Roadmap: `1.2.3` · Node: `1.2` — C#: OOP · Depth: **practice**

## Learning Objectives

After this lesson you will be able to:

- Explain the role of **constructors** in initializing new instances.
- Write **instance constructors** and distinguish them from ordinary methods.
- Chain constructors with **`: this(...)`** and **`: base(...)`** (base preview for `1.2.6`).
- Use **constructor initializers** and **object initializers** together without redundancy.
- Apply **primary constructors** (C# 12+) and understand what the compiler generates.
- Choose between constructors, static factories, and object initializers for clarity.

---

## Why This Matters

`new Customer()` allocates memory (`1.2.1`), but until fields and invariants are set, the object is not trustworthy. **Constructors** run immediately after allocation to put the instance into a **valid state**. Domain rules like "every order must have a customer id" belong in construction paths — alongside **`required`** properties from `1.2.2`.

Teams accumulate **multiple construction shapes**: empty + setters (bad), many overloads (noisy), or chained constructors that funnel into one **canonical** constructor. Primary constructors reduce boilerplate for simple types but are not free magic — captured parameters have visibility rules that surprise people.

ASP.NET Core does not call your constructors for JSON bodies directly the way `new` does — but your domain entities and immutable commands still rely on constructor design for correctness.

## Core Concepts

### What a Constructor Is

A **constructor** is a special method that runs at **object creation**. It has the **same name as the class**, no return type (not even `void`), and is invoked by **`new`**:

```csharp
public class Customer
{
    public int Id { get; }
    public string Name { get; }

    public Customer(int id, string name)
    {
        Id = id;
        Name = name ?? throw new ArgumentNullException(nameof(name));
    }
}

var c = new Customer(1, "Ada");
```

If you define **no constructors**, the compiler emits an implicit **parameterless constructor** that calls base and leaves fields at defaults. Once you define **any** constructor, the parameterless one is **not** generated unless you declare it explicitly.

Constructors are not inherited (`1.2.6` covers base calls). Each class must ensure its own instances become usable.

### Object Initializers vs Constructor

**Object initializer** syntax runs **after** the constructor body:

```csharp
var c = new Customer(1, "Ada") { /* only settable members here */ };
```

With `{ get; init; }` properties, initializer assigns in **init context**. Constructor should establish **invariants**; initializer fills optional or supplemental values.

Avoid duplicating the same validation in both constructor and setter unless intentional.

### Constructor Chaining: `: this()`

Multiple **`this(...)`** overloads funnel common work into one place:

```csharp
public class Rectangle
{
    public double Width { get; }
    public double Height { get; }

    public Rectangle(double side) : this(side, side)
    {
    }

    public Rectangle(double width, double height)
    {
        if (width <= 0 || height <= 0)
            throw new ArgumentOutOfRangeException();
        Width = width;
        Height = height;
    }
}
```

**`: this(...)`** must be the **first statement** — it delegates to another constructor on the **same class** before your body runs (the body of the target ctor runs, then returns). This avoids copy-pasting validation.

You cannot chain both `: this()` and `: base()` on one constructor — pick one delegation target.

### `: base()` — Brief Preview

When a class **inherits** another (`1.2.6`), a constructor can call a base class constructor:

```csharp
public class PremiumCustomer : Customer
{
    public PremiumCustomer(int id, string name, int tier) : base(id, name)
    {
        Tier = tier;
    }
    public int Tier { get; }
}
```

**`: base(...)`** runs base initialization before the derived constructor body. Every derived constructor must ultimately call a base constructor (explicitly or via default chain rules).

### Static Constructors and `required`

**Static constructor** (`static ClassName()`) initializes type-level state once — covered in `1.2.12`.

**`required`** members (`1.2.2`) complement constructors: compiler verifies object creation assigns them. You can still use a constructor to assign required properties:

```csharp
public class Invoice
{
    public required Guid Id { get; init; }
    public required decimal Total { get; init; }

    public Invoice(Guid id, decimal total)
    {
        Id = id;
        Total = total >= 0 ? total : throw new ArgumentOutOfRangeException();
    }
}
```

### Primary Constructors (C# 12+)

**Primary constructor** puts parameters on the **class declaration**:

```csharp
public class Point(int x, int y)
{
    public int X { get; } = x;
    public int Y { get; } = y;
}
```

The compiler synthesizes a constructor from the parameter list. Parameters are in scope in the class body for initializing members. You can add **additional constructors** that chain `: this(...)`:

```csharp
public class Point(int x, int y)
{
    public int X { get; } = x;
    public int Y { get; } = y;

    public Point() : this(0, 0) { }
}
```

**Caution:** primary constructor parameters may be **captured** into synthetic fields when used in instance methods — understand visibility and mutability if parameters are reference types. For data-holding types and services with injected dependencies, primary constructors reduce noise:

```csharp
public class OrderService(IOrderRepository repo)
{
    public void Ship(Guid id) => repo.MarkShipped(id);
}
```

This resembles constructor injection without explicit field assignment boilerplate — common in modern ASP.NET Core.

### Private Constructors and Factories

**Private constructor** blocks external `new` — used with **static factory methods**:

```csharp
public class Money
{
    public decimal Amount { get; }
    private Money(decimal amount) => Amount = amount;

    public static Money FromMajorUnits(decimal amount) =>
        amount >= 0 ? new Money(amount) : throw new ArgumentOutOfRangeException();
}
```

Factories name intent (`FromMajorUnits`) and can cache or reuse instances (see `string.Intern` patterns conceptually).

## Syntax / API

```csharp
// Instance constructor
public MyClass(int id) { ... }

// Chaining
public MyClass() : this(0) { }
public MyClass(int id) : base() { ... }   // inheritance

// Primary constructor (C# 12)
public class Service(ILogger logger) { }

// Static factory + private ctor
public static MyClass Create(...) { ... }
```

## Examples

**Example 1 — single canonical constructor:**

```csharp
public class EmailAddress
{
    public string Value { get; }

    public EmailAddress(string value)
    {
        if (!value.Contains('@'))
            throw new ArgumentException("Invalid email");
        Value = value.Trim().ToLowerInvariant();
    }

    public EmailAddress(string value, bool normalize) : this(normalize ? value.Trim().ToLowerInvariant() : value)
    {
    }
}
```

Second ctor delegates validation path — adjust to taste; point is **one validation core**.

**Example 2 — primary constructor service:**

```csharp
public class Greeter(string prefix)
{
    public string Greet(string name) => $"{prefix}, {name}!";
}
```

**Example 3 — explicit parameterless when needed:**

```csharp
public class Config
{
    public string Environment { get; set; } = "Development";

    public Config() { }  // required for some serializers if no other ctor
}
```

Some serializers need parameterless ctor; **`[JsonConstructor]`** and records alter the story — know your serializer.

## Common Mistakes & Anti-patterns

**God constructor** with 20 parameters — use object builder, record, or grouped value objects.

**Chaining loop** (`A(): this()` calling `B(): this()` circular) — compile error; design carefully.

**Throwing in constructor** leaves partially constructed object if base ctor succeeded — prefer validating before `base` when possible; document failure modes.

**Primary ctor reference capture** — mutating injected mutable singleton affects all uses.

**Public parameterless + mutable setters** bypassing invariants — prefer `required` + ctor/factory.

## Production & Real-World Notes

DI containers call the **public constructor** with most parameters resolvable. Primary constructors work well with ASP.NET Core minimal hosting.

EF Core often needs **parameterless constructor** (can be private with backing field mapping) for materialization.

Unit tests use constructors or object initializers to build valid fixtures quickly.

## Comparison / Trade-offs

| Approach | Pros | Cons |
|----------|------|------|
| Single ctor + validation | One invariant gate | Verbose for many optional params |
| `: this()` chaining | DRY validation | Indirection |
| Object initializer | Readable optional fields | Can skip required if not enforced |
| Primary constructor | Less boilerplate | Capture semantics learning curve |
| Static factory | Named intents, caching | Extra type discovery step |

## Quick Reference

```
Constructor     → same name as class, no return type
new T(args)     → alloc + ctor chain
: this(args)    → delegate to sibling ctor on same class
: base(args)    → delegate to base class ctor
Primary ctor    → params on class line (C# 12)
required        → must set on creation (with ctor/init)
```

---

## Key Takeaways

- Constructors **initialize** new instances to valid state after allocation.
- **`: this()`** centralizes validation across overloads.
- **`: base()`** initializes inherited portion (`1.2.6`).
- **Object initializers** complement, not replace, core invariants.
- **Primary constructors** reduce boilerplate; know capture and DI patterns.
- **Private ctor + factory** expresses intent and controls creation.
