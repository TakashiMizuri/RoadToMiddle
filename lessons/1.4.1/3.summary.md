# Threads 101 — краткая выжимка

> `1.4.1` · ~10 мин на повторение

## Главная идея

**Thread** — единица выполнения ОС со своим stack; создание дорого. **Thread pool** переиспользует worker-потоки для `Task.Run`, ASP.NET и т.д. **`Task` ≠ thread**. Блокировка pool thread под I/O — путь к starvation.

## Ключевые пункты

- **Kernel thread** планирует ОС; **managed thread** — обёртка CLR.
- **Context switch** — микросекунды × тысячи потоков = потери CPU.
- Pool **инжектит** threads медленно (throttling) при росте очереди.
- **Parallelism** — одновременно на cores; **concurrency** — прогресс многих задач.
- Связь с `1.3.30`: LINQ — CPU в одном потоке; `1.4` — потоки и ожидание.

## Самые важные примеры

```csharp
Task.Run(() => Work()); // очередь в pool, не new Thread()
Thread.CurrentThread.IsThreadPoolThread // true на worker
```

## «Под капотом» в одном абзаце

Scheduler ОС выделяет quantum; blocking syscall снимает thread с CPU. CLR pool следит за queue depth и throughput, добавляет workers порциями. I/O completion ports связывают async I/O с pool continuations. Каждый thread ~1 MB stack — thread-per-request не масштабируется.

## Частые ошибки

- Thread на каждый HTTP request.
- `.Result` / sync I/O на pool thread в ASP.NET.
- Думать, что 1000 `Task` = 1000 threads.

## Мнемоника

**Pool — общий автобус; не паркуй его на светофоре I/O.**
