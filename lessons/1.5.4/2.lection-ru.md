# Stack frames, call stack и stack overflow

> Roadmap: `1.5.4` · Node: `1.5` — .NET Runtime и память · Depth: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Определить **stack frame** и механизм **call stack** на каждом thread.
- Объяснить содержимое frame: return address, parameters, locals, saved registers.
- Связать frames с **`1.1.6`**: value types inline, references как pointer-sized slots.
- Описать **StackOverflowException** и типичные причины.
- Объяснить stack walk для exceptions и **GC roots** (`1.5.6`).
- Отличить **native call stack** от **IL evaluation stack** (`1.5.1`).

## Why This Matters

В `1.1.6` locals value types и reference slots были нарисованы **внутри stack frame**, а тела объектов — на heap. Это верная картина для типичного метода, но не ответ на вопросы: что такое frame физически, почему бесконечная recursion падает, и откуда берутся строки в stack trace.

Production регулярно показывает **`StackOverflowException`** (XML/JSON recursion без depth limit), «Insufficient stack» в глубоких pipeline, и stack traces, где async скрывает логический call graph. GC во время mark phase сканирует **managed references в активных frames** как roots — layout frame и **GC maps**, которые эмитит JIT (`1.5.2`), напрямую влияют на корректность collection.

Middle-разработчик читает stack trace осмысленно, понимает размер struct locals при deep calls и не путает thread stack с IL stack из `1.5.1`.

## Core Concepts

### Call stack на каждом thread

У каждого .NET thread есть **собственный native stack** — непрерывная область virtual memory (на x64 обычно растёт вниз). Каждый **вызов метода** создаёт **stack frame** (activation record). Цепочка frames от текущего метода до `Main` (или thread start) — **call stack**.

```
High addresses
┌─────────────────────┐
│  Main()             │
├─────────────────────┤
│  HandleRequest()    │
├─────────────────────┤
│  ParseJson()        │  ◄── текущий SP
└─────────────────────┘
Low addresses
```

При **return** frame **pop**'ается: stack pointer восстанавливается, память frame логически мёртва. GC не «собирает stack», но **ссылки в живых frames** — strong roots, пока метод не вернулся.

### Что внутри frame

Типичное содержимое (точный layout выбирает JIT):

- **Return address** — куда продолжить после callee.
- **Parameters** — по calling convention (registers + stack).
- **Local variables** — inline value types; для reference types — **slot с reference** (адрес), не сам object.
- **Saved registers** — ABI callee-saved.
- **Frame pointer chain** (опционально) — для debuggers и stack walk.

JIT генерирует **GC info**: bitmap, какие slots в каких instruction points содержат managed pointers — GC и exception unwinding используют это для exact scanning.

### Связь с 1.1.6

Правило из `1.1.6` уточняем runtime-терминами:

- `int count = 42` — значение **внутри frame**.
- `Customer? c = new Customer()` — reference slot в frame указывает на **object body на managed heap**.
- Большой **struct local** — все байты struct **копируются в frame** на каждый вызов; deep recursion с большим struct ускоряет stack overflow.

Value type как **field класса** живёт **внутри heap object**, не в caller frame — это уже не stack frame local, а часть heap layout (`1.5.5`).

### StackOverflowException

Исключение при превышении лимита stack thread. Частые причины:

1. **Unbounded recursion** — `void F() => F();`
2. **Очень глубокая легитимная recursion** без tail optimization.
3. **Огромный stackalloc** — память на stack, не heap.

```csharp
void Boom() => Boom(); // StackOverflowException
```

**Важно:** в отличие от большинства exceptions, **StackOverflowException часто fatal** — runtime может не иметь достаточно stack, чтобы надёжно выполнить `catch`. Лечится алгоритмом (iteration, explicit stack data structure на heap), а не try/catch.

Default stack ~1 MB на Windows thread; Linux/containers — проверяй лимиты.

### Stack traces и exceptions

При `throw` CLR **unwind**'ит stack, ищет `catch` по EH metadata из IL (`1.5.3`), выполняет `finally`. `StackTrace` строится обходом frames: имена методов из metadata, номера строк из **PDB**.

**async/await** — compiler генерирует state machine; physical frames (`MoveNext`) не совпадают с логическим async call graph — traces выглядят «шумно» без async-aware logging.

## Under the Hood

### Prologue/epilogue и tail calls

JIT prologue: adjust `rsp`, save registers, выделить space для locals. Epilogue — обратно перед `ret`. **Tail call optimization** (Tier 2, `1.5.2`) может переиспользовать frame callee как caller — некоторые recursive patterns превращаются в loop; для произвольного C# **не гарантировано**.

### Stack walk для GC

During GC mark phase (`1.5.6`) runtime обходит **roots на stacks** всех managed threads. Для каждого slot, помеченного в GC map как pointer, если значение указывает на heap object — object **marked** и трассируется дальше. **Unmanaged pointer**, притворяющийся ссылкой без reporting, ломает модель — отсюда ограничения unsafe.

### IL evaluation stack ≠ call stack

IL метод использует виртуальный **evaluation stack** при verification и JIT import — это не физический call stack thread. Один IL method компилируется в native code, использующий CPU stack/registers. Путаница двух «стеков» — частая ошибка на собеседованиях.

### Guard page и commit

Stack растёт commit'ом страниц; guard page at limit → StackOverflowException.

## Examples

### Recursion vs iteration

```csharp
int DepthRecursive(int n) => n == 0 ? 0 : 1 + DepthRecursive(n - 1);
// DepthRecursive(200_000) — риск overflow

int DepthIterative(int n) {
    int d = 0;
    while (n > 0) { d++; n--; }
    return d;
}
```

### stackalloc на stack

```csharp
Span<char> buf = stackalloc char[256]; // OK для bounded size
// stackalloc char[10_000_000] — StackOverflowException
```

## Common Mistakes & Anti-patterns

**Ловить StackOverflowException для recovery** — ненадёжно.

**«Stack хранит все C# objects»** — только locals/refs; **payload** reference types на heap.

**stackalloc с размером от user input** — overflow + security.

**Игнорировать struct size в deep recursion** — каждый frame += sizeof struct.

## Production & Real-World Notes

Deserializers (XML/JSON) — **MaxDepth** limits против malicious/recursive input. **dotnet-dump** + SOS `!clrstack` — managed frames. Container **stack size** редко проблема для CRUD, но native interop stacks matter.

---

## Key Takeaways

- Frame = один activation record на thread stack.
- **`1.1.6`**: locals/refs in frame; object bodies on heap.
- Active frame refs = **GC roots**.
- Stack overflow ≠ OOM heap.
- IL stack ≠ native call stack.
