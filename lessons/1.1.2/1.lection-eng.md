# Namespaces, `using`, and File-Scoped Namespaces

> Roadmap: `1.1.2` · Node: `1.1` — C#: language basics · Depth: **практика**

## Learning Objectives

After this lesson you will be able to:

- Explain **namespaces** as logical containers that prevent name collisions across types.
- Resolve type names via **fully qualified names** vs **`using` directives**.
- Distinguish **`using` import**, **`using static`**, and **`using alias`**.
- Apply **file-scoped namespaces** (C# 10+) and compare them to block-scoped `namespace { }`.
- Predict how **default namespace** from the project relates to folder structure and `RootNamespace`.
- Organize types in multi-file console or library projects without ambiguity.

---

## Why This Matters

In lesson `1.1.1` you created a single-file console app. Real solutions contain hundreds of types across dozens of assemblies — `OrderService`, `UserController`, `StringExtensions` may all coexist. Without a naming system, two teams could both define `class Customer` and the compiler would fail or worse — silently bind the wrong type.

**Namespaces** are C#'s primary mechanism for grouping types and controlling visibility of names at compile time. They mirror (loosely) folder hierarchy and company/product boundaries: `Contoso.Billing.Invoices`. Combined with **`using`**, they keep call sites readable: `List<Order>` instead of `System.Collections.Generic.List<Contoso.Billing.Order>`.

Middle developers read and refactor namespace layouts daily — moving a DTO to a shared library, fixing ambiguous references after a NuGet upgrade, or aligning folder names with namespace for discoverability. File-scoped namespaces reduce indentation noise in every file; understanding both styles lets you navigate legacy and modern codebases.

## Core Concepts

### Namespaces as Qualified Name Prefixes

Every type has a **fully qualified name (FQN)**: namespace + nested namespaces + type name. Example:

```csharp
namespace Contoso.App.Services;

public class EmailSender { }
```

FQN: `Contoso.App.Services.EmailSender`.

The compiler uses namespaces to **disambiguate** homonyms. `Contoso.App.Models.Customer` and `Contoso.Crm.Models.Customer` are different types. At the call site you either use the FQN or import a namespace so short names resolve.

Namespaces do **not** dictate physical folders — convention links them (`Services/EmailSender.cs` → `Contoso.App.Services`), but C# only requires logical consistency for human maintainability, not for compilation.

### Block-Scoped vs File-Scoped Namespace

**Classic (block-scoped)** — everything nested inside braces:

```csharp
namespace Contoso.App
{
    public class Program
    {
        public static void Main() { }
    }
}
```

**File-scoped (C# 10+)** — one namespace for the entire file, no extra indent:

```csharp
namespace Contoso.App;

public class Program
{
    public static void Main() { }
}
```

Only **one** file-scoped namespace declaration is allowed per file, and it must appear before other members. Mixed files (half inside block namespace, half outside) are invalid. Templates in modern SDK default to file-scoped for readability.

### The `using` Directive

Place `using` directives at the top of the file (after file-scoped namespace if present). They tell the compiler: when you see an unqualified type name, also search these namespaces.

```csharp
using System;
using System.Collections.Generic;

namespace MyApp;

class Demo
{
    static void Run()
    {
        var list = new List<int>(); // resolves to System.Collections.Generic.List<int>
        Console.WriteLine(list.Count);
    }
}
```

**Global usings** — in `GlobalUsings.cs` or injected by `<ImplicitUsings>enable</ImplicitUsings>` in `.csproj` — apply to every file in the project without repeating imports.

### Fully Qualified Names When Needed

If two imported namespaces expose the same short name, the compiler reports **ambiguity**. Fix options:

1. Use FQN at call site: `System.Text.Json.JsonSerializer` vs `Newtonsoft.Json.JsonSerializer`.
2. **Using alias**:

```csharp
using SystemJson = System.Text.Json.JsonSerializer;
using NewtonsoftJson = Newtonsoft.Json.JsonSerializer;
```

3. Remove one `using` and qualify only that side.

Explicit qualification is preferred in libraries when clarity beats brevity.

### `using static`

Imports **static members** of a type into scope — useful for `Math`, or carefully for extension-method hosting types:

```csharp
using static System.Console;

WriteLine("No Console. prefix");
```

Overuse harms readability (`WriteLine` from where?). Production code often limits `using static` to test helpers or domain-specific DSLs.

### Default Namespace and Project Settings

`dotnet new console -n MyApp` sets root namespace from project name. In `.csproj`:

```xml
<PropertyGroup>
  <RootNamespace>MyApp</RootNamespace>
</PropertyGroup>
```

New types in templates align with this. Renaming the project does not automatically rename namespaces in existing files — a common refactor task.

### Nested Namespaces

Equivalent forms:

```csharp
namespace A.B.C { class T { } }

// same as:
namespace A
{
    namespace B
    {
        namespace C
        {
            class T { }
        }
    }
}
```

Prefer dotted single declaration (`namespace A.B.C;`) for flat readability.

## Under the Hood

### Name Binding at Compile Time

Roslyn's **semantic model** resolves each identifier to a symbol. For `List<int>`, the compiler searches:

1. Local scope, enclosing scopes
2. Imported namespaces (order matters for extensions, not for type lookup among usings — ambiguity if multiple types match)
3. Global namespace

Namespaces affect **compile-time binding only**. They are erased in IL metadata as organizing names but **namespace-qualified type names** remain in assembly metadata for reflection and tooling.

### Assemblies vs Namespaces

An **assembly** (`.dll`) is a deployment/compilation unit; a **namespace** is logical. One assembly can contain many namespaces; one namespace can span assemblies (e.g. `System.*` split across many framework DLLs). `using` imports namespaces; **project references** and **PackageReference** bring assemblies into compilation.

## Syntax / Commands / API

```csharp
// File-scoped
namespace MyCompany.MyApp.Features.Auth;

// Block-scoped
namespace MyCompany.MyApp.Features.Auth
{
    // types
}

// Usings (file top, after namespace declaration in file-scoped style)
using System;
using System.Linq;
using Alias = Some.Long.Namespace.Type;
using static System.Math;

// Global usings file (SDK-style project)
// GlobalUsings.cs
global using System;
global using System.Collections.Generic;
```

Disable implicit usings to see explicit imports:

```xml
<ImplicitUsings>disable</ImplicitUsings>
```

## Examples

### Example 1: Splitting Types Across Files

**Models/User.cs**

```csharp
namespace RoadmapApp.Models;

public record User(int Id, string Name);
```

**Services/Greeter.cs**

```csharp
using RoadmapApp.Models;

namespace RoadmapApp.Services;

public static class Greeter
{
    public static string Greet(User user) => $"Hello, {user.Name}!";
}
```

**Program.cs**

```csharp
using RoadmapApp.Models;
using RoadmapApp.Services;

var user = new User(1, "Ada");
Console.WriteLine(Greeter.Greet(user));
```

Each file declares its namespace; `using` connects them within the same assembly.

### Example 2: Resolving Ambiguity

```csharp
using TextJson = System.Text.Json;
using NewtonsoftJson = Newtonsoft.Json;

namespace SerializationDemo;

class Compare
{
    static string ToSystemJson(object o) =>
        TextJson.JsonSerializer.Serialize(o);

    static string ToNewtonsoftJson(object o) =>
        NewtonsoftJson.JsonConvert.SerializeObject(o);
}
```

## Common Mistakes & Anti-patterns

**Namespace does not match folder structure** — not a compiler error, but hurts navigation; align in greenfield code.

**Wildcard usings** — C# has no `using System.*`; must import each namespace explicitly (unlike Java).

**Duplicate file-scoped namespace lines** — compile error CS8954.

**Relying on implicit usings in learning exercises** — temporarily disable to understand what `System` imports provide.

**God namespace** — dumping all types in `MyApp` without feature folders/namespaces scales poorly.

## Production & Real-World Notes

Teams adopt conventions: `Company.Product.Feature`, test projects mirror `*.Tests`, internal types in `*.Internal`. Analyzers (StyleCop, ReSharper) enforce namespace-folder alignment.

Libraries expose public API in stable namespaces; breaking changes sometimes require **namespace versioning** (`V2` subnamespace) alongside API revision.

IDE **Rename refactoring** updates namespace across files when moving types — prefer tooling over manual edits.

## Comparison / Trade-offs

| Style | Pros | Cons |
|-------|------|------|
| File-scoped | Less indent, modern default | One namespace per file only |
| Block-scoped | Multiple namespaces per file (rare) | Extra indentation |
| FQN at call site | Zero ambiguity | Verbose |
| `using` + short names | Readable | Ambiguity if careless |

## Quick Reference

```csharp
namespace A.B;           // file-scoped
using X = Long.Type;     // alias
using static System.Math; // static import
global using System;      // project-wide
```

FQN: `Namespace.Subnamespace.TypeName`

---

## Key Takeaways

- **Namespaces** group types and prevent name collisions; FQN = full path to type.
- **`using`** imports namespaces for short name resolution; ambiguity → qualify or alias.
- **File-scoped namespace** (`namespace X.Y;`) applies to entire file — C# 10+.
- **Assemblies** deploy code; **namespaces** organize names — related but independent.
- **RootNamespace** in `.csproj` influences default namespace for new files.
- **`using static`** imports static members — use sparingly in production code.
