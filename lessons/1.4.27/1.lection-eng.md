# IHttpClientFactory — Named Clients, Typed Clients, Handlers

> Roadmap: `1.4.27` · Node: `1.4` — Async + ASP.NET preview · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Register and consume **named HttpClients** with **`AddHttpClient`** and **`CreateClient(name)`**.
- Build **typed clients** (`AddHttpClient<IPaymentsApi, PaymentsApi>`) encapsulating routes and serialization.
- Chain **DelegatingHandler** middleware for logging, auth headers, correlation IDs, and resilience.
- Explain **handler lifetime** (default ~2 minutes) and its effect on **DNS** and **connection pools**.
- Configure **`ConfigurePrimaryHttpMessageHandler`** for **`SocketsHttpHandler`** tuning.
- Integrate factory clients in **Minimal APIs**, **controllers**, and **`Parallel.ForEachAsync`** safely.

---

## Why This Matters

**`IHttpClientFactory`** solves the **`HttpClient` lifetime problem** (`1.4.26`) in DI-friendly way. ASP.NET Core registers it by default when you call **`AddHttpClient`**. Each **logical downstream** (payments, catalog, identity) gets **named configuration**: base address, timeout, default headers, retry policies (`1.4.28`), and **handler chain** — without static singletons scattered across the codebase.

Middle developers structure microservice clients as **typed wrappers** testable with **`WebApplicationFactory`** and **`Mock<HttpMessageHandler>`**. **DelegatingHandler** is the extension point for **OAuth token refresh**, **Polly**, **OpenTelemetry** propagation — same pattern as ASP.NET middleware but for outbound HTTP.

Misconfiguration — wrong handler order, disposing typed client like transient resource, or **capturing scoped services in singleton handler** — causes subtle production bugs. This lesson is the bridge from raw **`HttpClient`** to production **outbound HTTP architecture**.

## Core Concepts

### Registration and Named Clients

```csharp
builder.Services.AddHttpClient("catalog", client =>
{
    client.BaseAddress = new Uri(builder.Configuration["Catalog:BaseUrl"]!);
    client.DefaultRequestHeaders.Add("Accept", "application/json");
    client.Timeout = TimeSpan.FromSeconds(30);
});
```

Consumption:

```csharp
public class CatalogController(IHttpClientFactory factory) : ControllerBase
{
    public async Task<IActionResult> GetItem(int id, CancellationToken ct)
    {
        var client = factory.CreateClient("catalog");
        var item = await client.GetFromJsonAsync<ItemDto>($"/items/{id}", ct);
        return Ok(item);
    }
}
```

**`CreateClient(name)`** returns **`HttpClient`** instance **lightweight** — underlying **`HttpMessageHandler`** rotated by factory. Do not register **`HttpClient`** as singleton yourself when using factory.

### Typed Clients

```csharp
public interface ICatalogApi
{
    Task<ItemDto?> GetItemAsync(int id, CancellationToken ct);
}

public class CatalogApi(HttpClient http) : ICatalogApi
{
    public Task<ItemDto?> GetItemAsync(int id, CancellationToken ct) =>
        http.GetFromJsonAsync<ItemDto>($"/items/{id}", ct);
}

// Registration
builder.Services.AddHttpClient<ICatalogApi, CatalogApi>(client =>
{
    client.BaseAddress = new Uri(config["Catalog:BaseUrl"]!);
});
```

Inject **`ICatalogApi`** — cleaner than string names, enables unit tests with **`HttpMessageHandler`** stub. Typed client class **must not store mutable request state** across calls unless thread-safe — one instance per typed client registration lifetime (transient handler chain, but typed client often **transient** or **scoped** depending on registration — default typed client is **transient**).

### Handler Lifetime and Rotation

Factory maintains **`DefaultHttpClientFactory`** with **`HandlerLifetime`** default **2 minutes**. After expiry, new **`SocketsHttpHandler`** — **DNS refresh**, new connection pool generation. Prevents stale DNS from immortal singleton.

Override:

```csharp
builder.Services.AddHttpClient("catalog")
    .SetHandlerLifetime(TimeSpan.FromMinutes(5));
```

Trade-off: longer lifetime = fewer handler allocations, slower DNS refresh.

### DelegatingHandler Pipeline

Handlers wrap inner handler — last registered runs **outermost** (first to see outgoing request):

```csharp
builder.Services.AddTransient<CorrelationIdHandler>();
builder.Services.AddHttpClient("catalog")
    .AddHttpMessageHandler<CorrelationIdHandler>()
    .AddHttpMessageHandler<AuthTokenHandler>();
```

```csharp
public class CorrelationIdHandler : DelegatingHandler
{
    private readonly IHttpContextAccessor _ctx;
    public CorrelationIdHandler(IHttpContextAccessor ctx) => _ctx = ctx;

    protected override Task<HttpResponseMessage> SendAsync(
        HttpRequestMessage request, CancellationToken cancellationToken)
    {
        var id = _ctx.HttpContext?.TraceIdentifier ?? Guid.NewGuid().ToString();
        request.Headers.TryAddWithoutValidation("X-Correlation-ID", id);
        return base.SendAsync(request, cancellationToken);
    }
}
```

**Order matters:** auth handler before logging vs after changes what gets logged. **Polly** handlers (`1.4.28`) insert here.

### Primary Handler Configuration

```csharp
builder.Services.AddHttpClient("catalog")
    .ConfigurePrimaryHttpMessageHandler(() => new SocketsHttpHandler
    {
        PooledConnectionLifetime = TimeSpan.FromMinutes(5),
        MaxConnectionsPerServer = 50,
        EnableMultipleHttp2Connections = true
    });
```

Use for TLS, HTTP/2, proxy, certificate pinning — not for business logic (that belongs in **DelegatingHandler**).

## Under the Hood

**`IHttpClientFactory.CreateClient`** looks up **`ActiveHandlerEntry`** in **`HttpClientFactoryOptions`** registry. Expired handlers disposed asynchronously. **`HttpClient`** instances are **not** pooled — **handlers** are managed; creating **`HttpClient`** per operation from factory is **cheap** and **correct**.

**Captive dependency:** registering **scoped** `DbContext` inside **singleton** `DelegatingHandler` fails DI validation — handlers often **transient**, resolve scoped via **`IServiceScopeFactory`** inside **`SendAsync`** when needed.

## Syntax / Commands / API

| API | Purpose |
|-----|---------|
| `AddHttpClient(name, configure)` | Named client |
| `AddHttpClient<TClient,TImplementation>()` | Typed client |
| `AddHttpMessageHandler<T>()` | Pipeline handler |
| `ConfigurePrimaryHttpMessageHandler` | SocketsHttpHandler |
| `SetHandlerLifetime` | DNS/pool rotation interval |
| `AddStandardResilienceHandler()` | Polly (.NET 8+) |

Testing: **`Mock<HttpMessageHandler>`** or **`WebApplicationFactory`** with **`ConfigureTestServices`**.

## Examples

### Example 1: Options pattern binding

```csharp
builder.Services.Configure<CatalogOptions>(config.GetSection("Catalog"));
builder.Services.AddHttpClient<ICatalogApi, CatalogApi>((sp, client) =>
{
    var opts = sp.GetRequiredService<IOptions<CatalogOptions>>().Value;
    client.BaseAddress = new Uri(opts.BaseUrl);
});
```

### Example 2: Minimal API injection

```csharp
app.MapGet("/proxy/{id}", async (int id, ICatalogApi api, CancellationToken ct) =>
    await api.GetItemAsync(id, ct));
```

### Example 3: Wrong — IHttpClientFactory in singleton without scope

Background **`HostedService`** singleton injecting **scoped typed client** directly — use **`IServiceScopeFactory`** per operation.

## Common Mistakes & Anti-patterns

**Static HttpClient** when factory available.

**Wrong handler order** — retry before auth adds token.

**Scoped service in singleton handler** — captive dependency.

**Disposing `CreateClient()` result** unnecessarily each time thinking it closes pool wrongly — understand factory ownership.

**One giant "default" client** for all downstreams — different timeouts/policies need **named** clients.

## Production & Real-World Notes

Teams map **one named client per downstream bounded context**. Configuration via **Options + IOptionsMonitor** for reload. **Health checks** call **`/health`** via same client config.

OpenTelemetry: **`AddHttpClientInstrumentation`** traces outbound calls. **YARP** reverse proxy is alternative when many routes — factory still relevant for service-to-service.

**.NET 8+** **`AddStandardResilienceHandler`** replaces manual Polly wiring for common cases (`1.4.28`).

## Comparison / Trade-offs

| Pattern | Pros | Cons |
|---------|------|------|
| Named client | Simple, flexible | String name typos |
| Typed client | Type-safe DI | Extra class per service |
| Raw HttpClient DI | None | Lifetime bugs |
| Refit / NSwag clients | Codegen DTOs | Build step |

## Quick Reference

```csharp
services.AddHttpClient("name", c => { c.BaseAddress = ...; })
    .AddHttpMessageHandler<MyHandler>()
    .SetHandlerLifetime(TimeSpan.FromMinutes(2));

// Inject IHttpClientFactory or ICatalogApi
var client = factory.CreateClient("name");
```

Handler chain = outbound middleware. Factory fixes DNS + sockets. Typed client = preferred for microservices.

## Key Takeaways

- **`IHttpClientFactory`** manages **handler lifetime**, not short **`HttpClient`** disposal per call.
- **Named clients** per downstream; **typed clients** for clean DI and tests.
- **DelegatingHandler** for cross-cutting outbound concerns.
- **Handler rotation** (~2 min default) refreshes **DNS** and pools.
- Configure **`SocketsHttpHandler`** via **ConfigurePrimaryHttpMessageHandler**.
- Avoid **captive dependencies** in handlers; use scopes in **`HostedService`**.
- Foundation for **Polly** resilience in `1.4.28`.
