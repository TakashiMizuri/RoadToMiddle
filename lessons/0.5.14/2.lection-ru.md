# iframe, embed — Security Sandbox

> Roadmap: `0.5.14` · Node: `0.5` — HTML · Depth: понимание

## Learning Objectives

После этого урока ты сможешь:

- Объяснить, когда использовать `<iframe>` vs `<embed>` и что каждый элемент делает в браузере.
- Описать **same-origin policy** и почему embedded content по умолчанию изолирован.
- Настраивать атрибут **`sandbox`** для минимальных прав untrusted frames.
- Распознавать риски **clickjacking**, **XSS через embeds** и **mixed content**.
- Применять **`allow`**, **`referrerpolicy`** и **`loading`** осознанно в production embeds.

---

## Why This Matters

Современные приложения постоянно встраивают сторонний контент: YouTube, Stripe payment forms, Google Maps, чат-виджеты, Figma previews, PDF viewers. Каждый embed — **отдельный browsing context**, вложенная «вкладка» внутри страницы. С этой силой приходят security obligations. Неверно настроенный iframe может дать атакующему read cookies, обмануть пользователя скрытыми кликами или выполнить scripts с избыточными правами.

Fullstack-разработчик не только потребляет embeds, но и **хостит** embeddable widgets (charts, calendars). Понимание sandbox boundaries помогает проектировать API с least privilege. CSP `frame-ancestors` и iframe `sandbox` дополняют друг друга против **clickjacking** — когда ваш сайт frame'ят maliciously для перехвата кликов.

Урок связан с DOM tree (`0.5.10`): у каждого iframe свой nested `Document` и window. Позже React portals рендерят вне subtree компонента, но в том же document — unlike iframe, это настоящая origin boundary. Security thinking переносится на OAuth popups, PCI scope платежей и micro-frontends.

## Core Concepts

### iframe: документ внутри документа

`<iframe>` загружает другую HTML-страницу в прямоугольную область. Embedded page имеет свой **browsing context**: отдельные `window`, `document`, history и origin (scheme + host + port). Parent JavaScript не может access iframe DOM при разных origins — **same-origin policy** бросает ошибку на cross-origin `contentDocument`.

```html
<iframe
  src="https://www.youtube.com/embed/dQw4w9WgXcQ"
  title="Tutorial video"
  width="560"
  height="315"
  loading="lazy"
  allow="accelerometer; autoplay; clipboard-write; encrypted-media; gyroscope; picture-in-picture"
  allowfullscreen>
</iframe>
```

Всегда задавай **`title`** для accessibility (`0.5.11`). Без title screen reader объявляет unlabeled embedded content.

**`src`** определяет origin embed. **`srcdoc`** — inline HTML без network request; опасен при untrusted strings (XSS vector).

### embed vs iframe

`<embed>` — **replaced element** для external resources (исторически Flash, иногда PDF/SVG). Void element, менее гибок для полных documents. Для HTML pages и большинства widgets **предпочитай iframe** — полный navigable document со standard security model.

```html
<embed src="/manual.pdf" type="application/pdf" width="600" height="800">
```

Когда говорят «embed video/map», обычно имеют iframe с vendor URL. Знай оба тега для legacy markup.

### Same-Origin Policy

Два URL share **origin**, если совпадают scheme, host, port. `https://app.example.com` и `https://api.example.com` — **разные origins**. Parent и child frames разных origins визуально соседствуют, но не читают DOM/localStorage друг друга. Обмен — через **`postMessage`** с explicit origin checks.

Изоляция защищает пользователя: compromised ad iframe не крадёт session cookies banking tab. Ты не можешь стилизовать internals vendor iframe — только outer frame element.

### Атрибут sandbox

Cross-origin iframe и так не access parent DOM. **`sandbox`** ограничивает **capabilities embedded document** даже при same-origin или untrusted HTML. Пустой `sandbox` — maximum restrictions:

- **Opaque origin** для контента
- Блок script (без `allow-scripts`)
- Блок form submission (без `allow-forms`)
- Блок top-level navigation (без allow variants)
- Блок plugins, popups, auto downloads и т.д.

```html
<iframe sandbox srcdoc="<p>Preview</p>" title="Email preview"></iframe>
<iframe sandbox="allow-scripts" src="/user-widget.html" title="Widget"></iframe>
```

Каждый **`allow-*`** снимает одно ограничение. **`allow-same-origin` + `allow-scripts`** на untrusted same-site HTML часто **слишком permissive** — sandbox escape классика.

| Token | Эффект |
|---|---|
| `allow-scripts` | JavaScript |
| `allow-forms` | Submit forms |
| `allow-popups` | `window.open` |
| `allow-same-origin` | Сохраняет origin |
| `allow-top-navigation-by-user-activation` | Top nav по user gesture |

**Least privilege**: полный sandbox, добавляй tokens только по необходимости.

### Clickjacking и frame-ancestors

**Clickjacking** — ваш сайт в прозрачном iframe на attacker page, клики попадают на «Transfer money». Защита:

1. **`X-Frame-Options: DENY`** / `SAMEORIGIN`
2. **CSP `frame-ancestors 'self'`** (modern)
3. JS frame-busting (unreliable)

Как consumer embed — trust reputable `src`. Как provider — headers против unauthorized framing.

### Permissions Policy через allow

**`allow`** на iframe делегирует features embedded document — camera, microphone, geolocation, fullscreen, payment. Cross-origin sensitive features по умолчанию deny.

```html
<iframe src="https://meet.vendor.com/room/abc"
        allow="camera; microphone; fullscreen"
        title="Video meeting">
</iframe>
```

### referrerpolicy и loading

**`referrerpolicy`** — сколько referrer URL утекает при load subresources.

**`loading="lazy"`** — отложенный fetch offscreen iframe; security не меняет.

### Mixed Content

HTTPS page + **`http://`** iframe = **mixed content** — block/weaken browsers. Embed URLs только HTTPS на secure sites.

## Under the Hood

Браузер при parse `<iframe src>` создаёт **nested browsing context** в дереве frames. Spec определяет sandbox flags на navigation и при run scripts. Пустой sandbox → **unique origin** (`null` в checks) — scripts не access parent cookies.

Composite model: URL origin + sandbox + CSP parent/child + Permissions Policy. **`allow-scripts allow-same-origin`** на user HTML same site — script может read parent cookies.

`postMessage` — queue между windows; handlers **must validate `event.origin`**.

## Syntax / Commands / API

```html
<iframe
  src="https://player.example.com/v/123"
  title="Product demo video"
  width="640"
  height="360"
  loading="lazy"
  referrerpolicy="strict-origin-when-cross-origin"
  allow="fullscreen"
  allowfullscreen>
</iframe>

<iframe sandbox="allow-scripts" src="/render/user-preview/abc"
        title="User HTML preview" width="100%" height="400">
</iframe>
```

Hosting headers:

```
Content-Security-Policy: frame-ancestors 'self' https://partner.example.com;
X-Frame-Options: SAMEORIGIN
```

## Examples

### Payment Element (conceptual)

```html
<iframe src="https://js.stripe.com/v3/..." title="Secure payment input" allow="payment"></iframe>
```

Card data в PCI scope провайдера, не в твоём DOM.

### Sandboxed user preview

```html
<iframe sandbox="" srcdoc="&lt;p&gt;Author preview&lt;/p&gt;" title="Content preview"></iframe>
```

Live JS для untrusted — отдельный subdomain + strict CSP, не loosen sandbox на main origin.

### Block framing admin

```
Content-Security-Policy: frame-ancestors 'none';
```

## Common Mistakes & Anti-patterns

**Нет iframe `title`** — WCAG violation.

**`allow-scripts allow-same-origin` на untrusted HTML** — escape sandbox.

**Embed arbitrary user URLs** — phishing в chrome вашего domain.

**Нет frame-ancestors на admin** — clickjacking.

**Over-broad `allow`** — лишние camera/mic.

**postMessage без origin check** — exfiltration.

## Production & Real-World Notes

Vendor embeds — official snippets. First-party widgets — document allowed parent origins, version `postMessage`. Security review: «что если src подменят client-side?» — validate server-side.

Micro-frontends через iframe — JS isolation vs UX cost. Module federation — same origin, другой security profile.

CSP **`frame-ancestors`** сначала report-only.

---

## Comparison / Trade-offs

**iframe vs third-party script** — script в твоём origin; iframe isolates. Untrusted/PCI → iframe.

**Strict sandbox vs widget** — больше tokens = больше surface.

**XFO vs CSP frame-ancestors** — CSP preferred.

**embed vs iframe PDF** — iframe + titling/sandbox чаще.

## Quick Reference

| Mechanism | Защита от |
|---|---|
| Same-origin policy | Cross-frame DOM/cookie access |
| `sandbox` | Script/form/navigation abuse |
| CSP `frame-ancestors` | Clickjacking вашего UI |
| `allow` | Over-delegation features |
| HTTPS embeds | Mixed content |
| `postMessage` + origin | Safe cross-frame comms |

## Key Takeaways

- `<iframe>` — nested browsing context со своим origin.
- Cross-origin parent не access iframe DOM — **`postMessage`** + origin validation.
- **`sandbox`** — least privilege; избегай scripts+same-origin на untrusted HTML.
- **Clickjacking** — CSP **`frame-ancestors`** на sensitive pages.
- **`title`** обязателен; **`allow`** — narrowly.
- iframe > embed для documents; vendor embed ≠ roll your own без модели угроз.
