# Exercises: Conventional Commits

> Complete after lectures and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Classify the change
**Goal:** Pick correct type.

For each scenario, write the **type** only (`feat`, `fix`, `chore`, `docs`, `test`, `ci`):

1. Add `GET /api/version` endpoint.
2. Correct null reference in `OrderService`.
3. Update GitHub Actions workflow to run `dotnet format`.
4. Add unit tests for `TaxCalculator`.
5. Fix typo in `CONTRIBUTING.md`.

**Success criteria:** 5/5 types correct with one-line justification each.

---

### E2. Rewrite vague messages
**Goal:** Practice imperative conventional subjects.

Rewrite:

| Bad | Your conventional subject |
|-----|---------------------------|
| `wip` | |
| `fix` | |
| `updated packages` | |
| `changes after review` | |

**Success criteria:** Each has type, optional scope, imperative description.

---

## Medium

### M1. Commit a small change properly
**Goal:** Real commit with convention.

**Tasks:**
1. On a practice branch, make a tiny doc or comment change.
2. Commit: `docs(<scope>): <imperative description>`.
3. `git log -1 --format=full`

**Success criteria:** Header passes conventional format; subject ≤ 72 chars.

---

### M2. Breaking change narrative
**Goal:** Document API break.

**Scenario:** You rename JSON field `userName` → `username` in API response.

**Tasks:**
Write full commit message (header + `BREAKING CHANGE:` footer) explaining consumer impact.

**Success criteria:** Contains `!` or footer; migration hint for React client.

---

### M3. Split a messy commit plan
**Goal:** Atomic commits.

**Scenario:** You changed `AuthController.cs` (feature), `package-lock.json` (chore), and `AuthControllerTests.cs` (test) in one sitting.

**Tasks:**
Plan **three** separate commit messages (do not need to execute if no repo).

**Success criteria:** Three distinct types/scopes; no combined "and" in subjects.

---

## Hard (stretch)

### H1. PR squash title
**Goal:** Connect to `0.2.8`.

**Tasks:**
Given PR with commits:
- `feat(cart): add line item quantity validation`
- `test(cart): cover edge case quantity zero`
- `fix(cart): address review comment on error message`

Write one **squash merge title** that best represents the PR for `main` history.

**Success criteria:** Single conventional line; explain why you kept `feat` vs `fix` as primary type.

---
