# ValueTask — Pooling and the "Don't Await Twice" Rule

> Roadmap: `1.4.15` · Node: `1.4` — C# async · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Explain why `ValueTask` and `ValueTask<T>` exist and when they outperform `Task`.
- Describe **pooling** of `IValueTaskSource` implementations and what gets reset between uses.
- State the **"don't await twice"** rule and predict undefined behavior when violated.
- Choose between `Task`, `ValueTask`, and synchronous return in library APIs.
- Recognize when `AsTask()` is the safe escape hatch for consumers.
- Debug production issues caused by cached `ValueTask` misuse.

---

## Why This Matters

Every `await` in a hot path that returns `Task<T>` allocates a **Task object** on the heap — even when the operation completes synchronously (cache hit, in-memory read, fast validation). In high-throughput ASP.NET Core endpoints and database drivers, those allocations add GC pressure and latency tails.

`ValueTask` was introduced in .NET Core 2.1 as a **struct-based awaitable** that can represent either a **synchronous result** (no allocation) or a **wrapped `Task`** (when the operation is truly asynchronous). Libraries like **System.Text.Json**, **Entity Framework Core**, and **Stream.ReadAsync** overloads use `ValueTask`/`ValueTask<T>` for exactly this reason.

The trade-off is strict: a pooled `ValueTask` may be **invalidated after the first await**. Awaiting the same instance twice, or storing it and awaiting later after the pool reclaimed it, produces **undefined behavior** — silent corruption, wrong results, or rare crashes. Middle developers must internalize this contract when consuming and especially when **authoring** async APIs.

## Core Concepts

### Task Recap: Reference Type Overhead

`Task` and `Task<T>` are **reference types**. When an async method suspends, the compiler-generated state machine often completes on a **cached completed task** (`Task.CompletedTask`, `Task.FromResult`) for sync paths — but many code paths still allocate. Each `new TaskCompletionSource<T>()` or async state machine that transitions to async mode creates heap objects tracked by the GC.

For operations that are **async only sometimes** — e.g., read from buffer (sync) vs read from network (async) — returning `Task<T>` forces either always-async (waste) or manual `TaskCompletionSource` juggling.

### ValueTask as a Discriminated Struct

`ValueTask` and `ValueTask<T>` are **structs** implementing the awaitable pattern (`GetAwaiter()` → `ValueTaskAwaiter`). Internally they hold one of:

1. **Synchronous result** — `T` value (for `ValueTask<T>`) or void completion (`ValueTask`).
2. **Wrapped `Task` / `Task<T>`** — delegates to an ordinary task when the operation went async.
3. **`IValueTaskSource` / `IValueTaskSource<T>`** — custom completion source, often **pooled**.

From the caller's perspective, `await valueTask` looks identical to `await task`. Under the hood, the awaiter checks whether the operation completed synchronously and either continues inline or registers a continuation on the underlying task/source.

### The Pooling Model

When a library uses **manual `IValueTaskSource`** with an object pool (common in BCL and high-performance IO), the flow looks like this:

```
Caller invokes ReadAsync() → pool rents ManualResetValueTaskSourceCore
  → if sync complete: return ValueTask with result, maybe no async path
  → if async: register continuation, on complete: signal awaiter, RETURN TO POOL
```

After the **first await completes**, the `ValueTask` instance is a **one-time ticket**. The pooled `IValueTaskSource` may be **reset and handed to another caller**. A second await on the same `ValueTask` might observe another operation's completion — classic data race from the application perspective.

**Rule:** Treat every `ValueTask` as **single-consumer, single-use**. Do not cache in a field, do not pass to multiple concurrent awaiters, do not `await` twice.

### When ValueTask Helps

`ValueTask` shines when:

- The method completes synchronously **most of the time** (cached config, buffered stream read, fast path validation).
- You control the implementation and can use **`ManualResetValueTaskSourceCore<T>`** with pooling.
- Callers are unlikely to need multiple awaits, `.Result`, or composition requiring a stable `Task`.

It does **not** help when the operation is **always async** — you still pay for the underlying `Task` and add struct copying overhead without benefit.

### Safe Consumption Patterns

**Correct — await once:**

```csharp
ValueTask<int> vt = stream.ReadAsync(buffer);
int bytes = await vt;
```

**Correct — convert if you need multiple consumers or storage:**

```csharp
ValueTask<int> vt = GetValue();
Task<int> task = vt.AsTask(); // allocates Task if needed; safe to await multiple times
int a = await task;
int b = await task; // OK on Task
```

**Incorrect — double await:**

```csharp
ValueTask<int> vt = GetValue();
int a = await vt;
int b = await vt; // UNDEFINED — may throw, return garbage, or appear to work
```

**Incorrect — fire two parallel awaits:**

```csharp
ValueTask work = DoWorkAsync();
await Task.WhenAll(work.AsTask(), work.AsTask()); // still wrong if AsTask() not called first once
// Correct: Task t = work.AsTask(); await Task.WhenAll(t, SomethingElse());
```

### ValueTask vs Task — API Author Guidance

Microsoft guidance for **public library APIs**:

- Default to **`Task` / `Task<T>`** — safest for consumers, composable, multiple awaits OK.
- Use **`ValueTask`** when profiling proves allocation cost matters **and** you document single-await semantics.
- Never return **`ValueTask` from async method** that always suspends — the compiler wraps in `Task` anyway; use `async Task` instead.

In **application code** (controllers, handlers), prefer `Task` unless you wrap a library returning `ValueTask` — just await once and move on.

## Under the Hood

### ManualResetValueTaskSourceCore

.NET provides `ManualResetValueTaskSourceCore<TResult>` — a struct that implements `IValueTaskSource<TResult>` without allocating per operation when pooled:

```csharp
// Simplified mental model — real BCL code is more complex
sealed class PooledSource : IValueTaskSource<int>
{
    ManualResetValueTaskSourceCore<int> _core;
    public ValueTask<int> Rent() => new(_core, _core.Version);
    public void ReturnToPool() { _core.Reset(); /* return to pool */ }
}
```

`ValueTask` stores a **version token** paired with the source. After reset, version increments — stale awaiters detect invalidation (in debug/correct usage you never hit this).

### Awaiter Path

When you `await valueTask`:

1. `GetAwaiter()` returns `ValueTaskAwaiter` or `ValueTaskAwaiter<T>`.
2. `IsCompleted` — if true, `GetResult()` runs synchronously (no continuation).
3. If false, `OnCompleted` registers on underlying `Task` or `IValueTaskSource`.
4. After completion, **consumer must not touch the `ValueTask` again** if backed by pooled source.

Disassembling a method that awaits `ValueTask` shows similar state machine to `Task`, but `GetResult` may call `IValueTaskSource.GetResult(short token)`.

### Boxing and Conversions

Implicit conversion `ValueTask` → `ValueTask<T>` doesn't exist carelessly. Calling `.AsTask()` **may allocate** a `Task` wrapper — that's the price of safety. For hot paths, avoid `.AsTask()` unless required.

## Syntax / API

| API | Purpose |
|-----|---------|
| `new ValueTask()` / `new ValueTask<T>(T result)` | Sync-completed awaitable |
| `new ValueTask(Task)` / `new ValueTask<T>(Task<T>)` | Wrap existing task |
| `ValueTask.FromCanceled()` / `FromException()` | Canceled/faulted sync paths |
| `valueTask.AsTask()` | Convert to `Task` for safe reuse |
| `ManualResetValueTaskSourceCore<T>` | Build pooled custom sources |
| `IValueTaskSource.GetResult(short token)` | Completion callback surface |

## Examples

### Example 1: Fast Path in a Cache Service

```csharp
public ValueTask<User?> GetUserAsync(int id, CancellationToken ct)
{
    if (_cache.TryGetValue(id, out User? user))
        return new ValueTask<User?>(user); // zero allocation async path

    return new ValueTask<User?>(FetchFromDbAsync(id, ct)); // wraps Task when slow
}
```

The caller **awaits once**. Cache hits avoid `Task` allocation entirely.

### Example 2: Consumer Safety with AsTask()

```csharp
public async Task<int> ProcessAllAsync(IEnumerable<ValueTask<int>> items)
{
    // Must materialize to Task — each ValueTask consumed once
    Task<int>[] tasks = items.Select(vt => vt.AsTask()).ToArray();
    int[] results = await Task.WhenAll(tasks);
    return results.Sum();
}
```

Without `.AsTask()`, storing `ValueTask` in a collection and awaiting later violates single-use semantics.

### Example 3: Stream.ReadAsync — BCL Pattern

```csharp
byte[] buffer = new byte[4096];
ValueTask<int> read = stream.ReadAsync(buffer.AsMemory());
int count = await read;
// Do NOT: await stream.ReadAsync(...) stored in variable twice
```

## Common Mistakes & Anti-patterns

**Awaiting twice "because it worked in tests"** — undefined behavior is timing-dependent; tests may pass until production load reuses pooled sources.

**Returning `async ValueTask`** — an `async ValueTask` method that always hits `await` allocates a state machine **and** often a `Task` backing store, defeating the purpose. Use `async Task` or manual `IValueTaskSource`.

**Storing `ValueTask` in instance fields** for "later" — use `Task` if you need deferred completion.

**Using `.Result` on `ValueTask`** — same sync-over-async problems as `Task`, plus single-use violation if combined with other awaits.

**Assuming `ValueTask` is always allocation-free** — wrapping `Task` still holds a reference; only sync-complete paths avoid heap allocation.

## Production & Real-World Notes

Profile before switching public APIs to `ValueTask`. Teams often add **`AsTask()` at boundaries** — library returns `ValueTask`, application layer converts once at the edge of composition (e.g., before `Task.WhenAll`).

In **EF Core 5+**, many APIs return `ValueTask`; await directly in handlers — don't store query results' tasks.

Code analyzers (custom Roslyn) sometimes flag double-await patterns; no built-in analyzer ships for all cases — code review matters.

When upgrading packages, read release notes: pooling implementations may change internal behavior; single-await contract remains.

## Comparison / Trade-offs

| | `Task` | `ValueTask` |
|---|--------|-------------|
| Multiple awaits | ✅ Safe | ❌ Undefined (pooled) |
| Sync fast path allocations | Often allocates | Often zero alloc |
| Composability (`WhenAll`, etc.) | Native | Needs `.AsTask()` |
| Public API default | ✅ Recommended | Only when measured |
| Debugging | Easier | Harder (pooled sources) |

## Key Takeaways

- `ValueTask` reduces allocations when operations often complete synchronously.
- Pooled `IValueTaskSource` makes **second await undefined** — not just "bad style."
- Use `.AsTask()` when you need multiple awaits, storage, or `Task.WhenAll`.
- Library authors: default to `Task`; adopt `ValueTask` with profiling and clear docs.
- `async ValueTask` rarely helps; prefer manual completion or `async Task`.
- Application code: await library `ValueTask` once inline; don't cache.
- Same discipline applies to `ValueTask<T>` as to non-generic `ValueTask`.
