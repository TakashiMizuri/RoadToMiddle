# UnobservedTaskException — глобальный обработчик

> Roadmap: `1.4.31` · Node: `1.4` — C# async/await · Depth: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Объяснить, когда исключение `Task` становится **unobserved** и как срабатывает **`TaskScheduler.UnobservedTaskException`**.
- Описать поведение **.NET 4.5+**: событие, процесс **обычно живёт**, `SetObserved()`.
- Подключить **глобальный handler** для логов и алертов.
- Связать unobserved faults с fire-and-forget (`1.4.30`).
- Отличать **`OperationCanceledException`** от реальных ошибок в фоне.

## Why This Matters

Ночной отчёт падает в discarded `Task`. В middleware тишина. Данные расходятся, процесс не упал — .NET **по умолчанию не роняет** app на unobserved. Сбой был **невидим**, пока не проявился elsewhere.

**`UnobservedTaskException`** — страховочная сетка, когда fire-and-forget ещё есть. Настоящий fix — **observe каждый Task** (await в consumer). Middle регистрирует handler для **visibility** и устраняет класс багов в источнике.

## Core Concepts

### Observed vs Unobserved

Исключение **observed**, если:

- **`await`** task (exception propagates),
- **`Wait()`/`Result`** (избегай в ASP.NET),
- чтение **`task.Exception`** на faulted completed task,
- **`GetAwaiter().GetResult()`** после completion.

Иначе при GC faulted `Task` → **`UnobservedTaskException`**.

### Timeline

Discarded faulted task → **позже GC** → event. Отладка сложна: stack trace **далеко** от причины.

### .NET 4.5+

1. Event **`UnobservedTaskException`**.
2. **`SetObserved()`** в handler.
3. Процесс **продолжается** (если handler не эскалирует).

Меньше crash — больше риск **тихих** сбоев без логов.

### Регистрация handler

```csharp
TaskScheduler.UnobservedTaskException += (sender, e) =>
{
    e.SetObserved();
    foreach (var ex in e.Exception.Flatten().InnerExceptions)
        _logger.LogError(ex, "Unobserved task exception");
};
```

Один раз при **startup**. **`Flatten()`** для `AggregateException`.

### Связь с fire-and-forget

**`_ = WorkAsync()`** при fault → unobserved, если тот же `Task` никто не observe. **`ContinueWith`** с проверкой fault — костыль; лучше **await** в инфраструктуре.

## Under the Hood

### Finalization

CLR при финализации неobserved faulted task ставит callback в thread pool — **недетерминированное** время.

### TaskExceptionHolder

Держит exception до observe/finalize. Static hoard faulted tasks → **утечка** памяти.

### async void

Другой путь — **unhandled** на SyncContext/pool, не `UnobservedTaskException`.

### ASP.NET Core

Pipeline ловит только **awaited** work. Handler — **last resort**, не замена middleware.

## Examples

### Unobserved fault

```csharp
Task.Run(async () => { await Task.Delay(10); throw new Exception("lost"); });
return Ok();
```

### Правильно в worker

```csharp
try { await work(stoppingToken); }
catch (Exception ex) { _logger.LogError(ex, "Background failed"); }
```

## Common Mistakes & Anti-patterns

Handler **per request** — утечки и дубли логов.

**`SetObserved()`** без лога — проглатывание.

Думать, что handler **чинит** scoped fire-and-forget.

Static коллекция `Task` без await.

---

## Quick Reference

- Unobserved = faulted Task без observe до GC.
- Event при **startup**.
- `Flatten()`, log, `SetObserved()`.
- .NET 4.5+: процесс жив — **логируй**.
- Fix: **await** в consumers.
- Не замена `IExceptionHandler`.

---

## Key Takeaways

- Discarded faults → event, часто **с задержкой**.
- Startup handler + осознанный **`SetObserved()`**.
- **Await** в `BackgroundService` — правильная точка.
- **`async void`** — отдельный ад на сервере.
- Handler = visibility, не разрешение на `_ =` scoped work.
