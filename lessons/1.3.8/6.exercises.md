# Exercises: LINQ Query Syntax

> Complete after lectures and passing `4.test-yourself.md`. Hints only — not full solutions.

## Easy

### E1. Translate both ways
**Goal:** Fluency converting syntaxes.
**Steps:** Write filter+sort+project active users to emails in query syntax; rewrite method syntax; verify same output.
**Success criteria:** Side-by-side equivalence.

### E2. `let` readability
**Goal:** Use **`let`** clause.
**Steps:** Users with First/Last name — query with **`let fullName`**, filter where fullName length > 10, select fullName.
**Success criteria:** Single query expression; no duplicated concatenation.

## Medium

### M1. Inner join
**Goal:** Query join.
**Steps:** `Order { CustomerId }`, `Customer { Id, Name }` — join, select customer name and order total for totals > 1000.
**Success criteria:** **`equals`** in join; **`where`** after join.

### M2. Group with `into`
**Goal:** **`group by ... into`**.
**Steps:** Group orders by month (use `OrderDate.Month`), select month + count + sum total.
**Success criteria:** Result selector via **`select`** after **`into`**.

## Hard (stretch)

### H1. Left join pattern
**Goal:** **`join ... into` + `DefaultIfEmpty`**.
**Steps:** All customers with optional latest order id (0 if none). Sketch method-syntax equivalent.
**Success criteria:** Query runs; explanation of **`GroupJoin`**.

### H2. Style guide decision
**Goal:** Team trade-off.
**Steps:** Pick one pipeline from `1.3.7` M1 — write both syntaxes; document which you'd merge in PR and why (3–5 sentences).
**Success criteria:** Justified choice, not "always query".

## Submission checklist

- [ ] At least one join exercise uses **`equals`**
- [ ] One exercise mixes query + method with parentheses
- [ ] Compiles with `dotnet run`
