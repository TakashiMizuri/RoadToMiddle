# HTTP Status Codes — краткая выжимка

> `0.3.6` · ~10 мин на повторение

## Главная идея

**Status code** в response start line (`0.3.3`) — главный machine-readable сигнал об исходе запроса. Класс по первой цифре (2xx success, 3xx redirect/cache, 4xx client fix, 5xx server/retry) определяет поведение client, cache и monitoring.

## Ключевые пункты

- **200** — success с body; **201** — создан resource (+ **Location**); **204** — success без body.
- **301** — permanent URL move; **304** — conditional GET, cache still valid (no body).
- **401** — не authenticated; **403** — authenticated, но не allowed (не путать).
- **404** — resource not found; **409** — conflict со state; **422** — validation/semantics.
- **429** — rate limit + **Retry-After**; client backoff.
- **500** — app bug; **502** — gateway/upstream bad; **503** — temporary unavailable.
- Anti-pattern: **200** с `{ error: true }` — ломает HTTP semantics и SLO.

## Самые важные примеры

**Created order:**
```
HTTP/1.1 201 Created
Location: /orders/991
```

**Conditional GET unchanged:**
```
HTTP/1.1 304 Not Modified
(no body)
```

**Auth:**
```
No token     → 401
Wrong role   → 403 (retry с тем же token бесполезен)
```

## «Под капотом» в одном абзаце

Status code commit'ится в response start line до body stream. ASP.NET Core: middleware (exceptions → 500), authorization filters (401/403), action results (404, 201, 204). Proxies translate upstream failures → 502. Caches и conditional requests используют **304** вместе с **ETag**/`If-None-Match` (`0.3.7`). Monitoring строит error budget по **5xx**, не по fake 200 errors.

## Частые ошибки

- 401 вместо 403 и наоборот.
- 201 без Location.
- 500 для validation (нужен 422/400).
- 404 для всех ошибок вместо 409/422.
- Ignore Retry-After на 429.

## Мнемоника

> **4xx** — client исправь request; **5xx** — server/gateway, часто retry; **401** = «кто ты?», **403** = «знаю, но нельзя».
