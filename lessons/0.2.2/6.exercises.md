# Exercises: Merge

> Complete after lectures and passing `4.test-yourself.md`.

## Easy

### E1. Name the merge type
**Goal:** Recognize graph outcomes.

For each scenario, label **FF**, **merge commit**, or **squash**:

1. `main` at C10, `feature` at C12, path C10→C11→C12 only on feature.
2. Both branches committed since C5; `git merge feature` on main creates C13 with two parents.
3. `git merge --squash feature` then one commit on main.

**Success criteria:** 3/3 correct.

---

## Medium

### M1. Draw before and after
**Goal:** Visualize three-way merge.

Draw ASCII graph: merge base B, main tip M, feature tip F, merge commit X after `git merge feature` on main.

**Success criteria:** X has two parents M and F; main points to X.

---

### M2. Policy pick
**Goal:** Apply trade-offs.

Your team values (a) clean linear `main` for releases, (b) full commit history for debugging. Pick **FF**, **--no-ff**, or **squash** for integrating feature PRs. Write 5 sentences defending choice and one downside.

**Success criteria:** Names trade-off explicitly.

---

## Hard

### H1. Simulate FF vs no-ff
**Goal:** Hands-on in throwaway repo.

Create repo with `main` and `feature` (2 commits on feature, main idle). Record `git log --oneline --graph` after:
1. FF merge on fresh clone branch setup.
2. Reset, recreate, merge with `--no-ff`.

Compare commit count and graph shape.

**Success criteria:** Can explain parent count difference.
