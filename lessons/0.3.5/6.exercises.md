# Exercises: Safe vs Idempotent Matrix

> Complete after lectures and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Fill the matrix
**Goal:** Memorize classification.

Fill Safe (Y/N) and Idempotent (Y/N) for: GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS.

**Success criteria:** Matches lecture matrix; PATCH marked idempotent "depends" with note.

---

### E2. Safe or idempotent?
**Goal:** Distinguish properties.

For each scenario, which property is violated or at risk?

1. Retry POST create user on timeout
2. CDN caches GET that increments view counter in DB
3. DELETE resource twice

**Success criteria:** (1) idempotent/retry (2) safe (3) idempotent OK.

---

## Medium

### M1. Idempotency-Key spec sketch
**Goal:** Application-layer design.

Write 6–8 bullet API spec for `POST /api/transfers` requiring `Idempotency-Key`: header name, TTL, duplicate behavior, status codes, response body on replay.

**Success criteria:** Same key → same response without second transfer; 409 or 200 replay documented.

---

### M2. PATCH idempotency review
**Goal:** Analyze patch semantics.

Two PATCH bodies for `/settings/1`:

- A: `{ "theme": "dark" }`
- B: `{ "loginCount": { "op": "increment" } }`

Which is idempotent if applied twice? Explain in 3–5 sentences.

**Success criteria:** A yes if absolute; B no (counter increments twice).

---

## Hard (stretch)

### H1. Retry policy table
**Goal:** Production HttpClient policy.

Design retry rules for outbound calls: GET catalog, PUT update profile, POST checkout. Table columns: Method | Auto-retry? | Max retries | Required extra mechanism.

**Success criteria:** GET/PUT retry with backoff; POST no retry without idempotency key; mention 503 Retry-After preview.
