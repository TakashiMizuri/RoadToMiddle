## Answers

### 1. [Recall]
**Answer:** **`Func<T, bool>`** (или **`Func<T, int, bool>`** с index overload).
**Why:** Predicate возвращает `true` для элементов, которые остаются в sequence.

### 2. [Recall]
**Answer:** **`ThenBy`** добавляет secondary sort key, сохраняя ordering по первому ключу. Второй **`OrderBy`** **пересортировывает** всю sequence только по новому ключу, игнорируя предыдущий порядок.
**Common mistake:** два **`OrderBy`** подряд вместо **`OrderBy` + `ThenBy`**.

### 3. [Recall]
**Answer:** **`IEnumerable<IGrouping<string, Order>>`** (если `Country` — `string`).
**Why:** Каждая группа — **`IGrouping<TKey, TElement>`** с **`Key`** и элементами группы.

### 4. [Understanding]
**Answer:** **`Where`** на source не создаёт лишних projected objects. **`Select`** first может allocate DTO/ anonymous objects для элементов, которые потом отфильтруешь.
**Why:** Меньше allocations и work; для EF — filter в SQL до projection.

### 5. [Understanding]
**Answer:** Первый вариант группирует **только 10** произвольных элементов (после take). Второй — группирует **все** элементы, затем берёт 10 **групп** (или элементов результата — зависит от типа после GroupBy; здесь речь о semantic «10 sales vs full aggregation then take»).
**Why:** **`Take`** ограничивает upstream sequence **до** того, как downstream operator видит полный набор.

### 6. [Understanding]
**Answer:** Pipeline только **описывает** transformations; реальный проход по data происходит при enumeration (**`foreach`**, **`ToList()`**, **`Count()`** и т.д.).
**Why:** Deferred execution (`1.3.9`) — operators return iterators, не results immediately.

### 7. [Application]
**Answer:**
```csharp
var emails = users
    .Where(u => u.IsActive)
    .OrderBy(u => u.Email)
    .Select(u => u.Email);
```
**Why:** **`OrderBy`** до **`Select`** допустим на `User`; можно и **`Select` then `OrderBy`** на string — оба OK здесь.

### 8. [Application]
**Answer:**
```csharp
var result = orders
    .GroupBy(o => o.Country, (country, items) => new { Country = country, Sum = items.Sum(o => o.Total) })
    .OrderByDescending(x => x.Sum);
```
**Why:** Result selector collapses groups; **`Sum`** inside group.

### 9. [Analysis]
**Answer:** Выведет count пользователей с **`Age >= 21`**, не 18.
**Why:** Lambda **captures** variable **`threshold`**, читает текущее значение при **execution**, не при построении **`q`**.

### 10. [Analysis]
**Answer:** После **`Select`** элементы — **`string`** (`Name`); у **`string`** нет **`Price`**.
**Why:** Filter по **`Price`** нужен **до** projection на name.

### 11. [Understanding]
**Answer:** **`IGrouping<TKey, TElement>`** extends **`IEnumerable<TElement>`** — группу можно iterate; плюс property **`Key`**.
**Why:** Модель «bucket of elements under one key».

### 12. [Application]
**Answer:** `.OrderBy(c => c.Country).ThenBy(c => c.LastName)`
**Why:** **`ThenBy`** только на **`IOrderedEnumerable<T>`** после первого **`OrderBy`**.

### 13. [Analysis]
**Answer:** **Не совсем.** **`GroupBy`** возвращает lazy **`IEnumerable<IGrouping<...>>`**. При enumeration строит группы (often с internal buffering), но это не **`Dictionary`** API и не eager materialize по умолчанию.
**Common mistake:** думать, что **`GroupBy`** = **`ToDictionary`**.

### 14. [Understanding]
**Answer:** Те же extension method names на **`IQueryable<T>`**; provider (EF Core) переводит expression tree в SQL. In-memory — delegates + **`IEnumerable`** iterators.
**Why:** Vocabulary общий; execution model разный (`1.3.11`–`1.3.12`).

### 15. [Application]
**Answer:** (один из вариантов)
```csharp
var list = new List<int>();
foreach (var o in orders)
{
    if (o.Total <= 100) continue;
    list.Add(o.Id);
}
list.Sort();
```
**Why:** Эквивалент **`Where` → Select → OrderBy` → materialize.

## Model answers for practical sections

Pipeline: **`Where` (year) → `GroupBy` → sum → `OrderByDescending` → `Take(3)` → `ToList()`**. Проверь, что **`Date.Year`** или диапазон дат корректен для «текущий год».
