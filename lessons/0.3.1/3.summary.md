# Client-Server и request-response — краткая выжимка

> `0.3.1` · ~8 мин на повторение

## Главная идея

Web и API построены на **client-server**: client **инициирует** HTTP **request**, server **отвечает** **response**. Один пользовательский экран часто = много таких циклов. HTTP **stateless** на уровне протокола; login и session — поверх (cookies, JWT).

## Ключевые пункты

- **Client/server** — логические роли; одна машина может быть и client, и server (browser + Kestrel локально; API + Stripe).
- **Request-response**: method + URL + headers + body → status + headers + body.
- HTTP — **layer 7**; ниже TCP (или QUIC), часто TLS (HTTPS).
- **Stateless**: server не помнит прошлые requests; state через token/cookie в каждом request.
- **TTFB** — время до первого byte response; latency складывается из DNS, TLS, server, БД.
- Backend — **server** для frontend и **client** к PostgreSQL, Redis, внешним API.

## Самые важные примеры

**Цикл:**
```
Client -- HTTP Request --> Server
Client <-- HTTP Response -- Server
```

**Много циклов на одну страницу:** HTML → JS bundle → `GET /api/products` → images.

**curl с verbose:**
```bash
curl -v https://httpbin.org/get
```

## «Под капотом» в одном абзаце

Kestrel парсит bytes в `HttpContext`, pipeline выполняет handler, сериализует response в TCP. Client (browser/`HttpClient`) делает обратное. Timeout'ы на TCP, TLS и application layer; abort client'а не отменяет автоматически work на server.

## Частые ошибки

- Думать, что server = одна физическая машина (есть LB, pool).
- Ожидать server-initiated push без WebSockets/SSE/polling.
- Путать stateless HTTP с «нет сессий» — сессии на application layer.

## Мнемоника

> **Client спрашивает, server отвечает** — один request, один response; state живёт в headers/cookies, не в «памяти TCP».
