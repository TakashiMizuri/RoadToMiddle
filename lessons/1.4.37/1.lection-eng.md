# Debugging Async: Tasks Window and Async Stack Traces

> Roadmap: `1.4.37` · Node: `1.4` — C# async · Depth: **практика**

## Learning Objectives

After this lesson you will be able to:

- Use the **Tasks window** in Visual Studio to inspect **active, scheduled, and completed** `Task` instances during a debug session.
- Interpret **async stack traces** that cross `await` boundaries (with and without **Enhanced Stack Trace**).
- Set breakpoints on **continuations** and diagnose **deadlocks** and **never-completing** tasks.
- Enable **.NET Async** debugger features and understand **"async void"** debugging pitfalls.
- Use **parallel stacks** and **Tasks** together to map thread pool work to logical async operations.
- Apply practical workflows: "hangs on await", "wrong exception stack", "fire-and-forget lost error".

---

## Why This Matters

Synchronous debugging is intuitive: call stack shows who called whom. **Async breaks the illusion.** When you pause inside `await db.QueryAsync()`, the call stack often shows only the **current continuation** — not the controller that started the request. A **deadlock** from `.Result` may show threads blocked with **no obvious owner**. A **fire-and-forget** `Task` fails on a thread pool thread with no link back to the HTTP request.

Middle developers spend real time on production-like bugs: API timeout, stuck spinner, `AggregateException` with nested inner exceptions. Without async-aware debugging, you guess. With the **Tasks window**, **Parallel Stacks**, and modern **async stack trace** support (.NET 6+ / VS 2022+), you see **which tasks are waiting**, **what they wait on**, and **the logical async causality chain**.

This lesson is hands-on tooling — the complement to theory in `1.4.1`–`1.4.35`. You will use it weekly in ASP.NET + EF Core projects.

## Core Concepts

### Why Call Stacks Lie in Async

When the compiler processes `await`, it splits the method into a **state machine**. After an await, execution resumes on a **possibly different thread pool thread**. The physical call stack at the resume point does not include the original synchronous caller frame in a simple linear way.

Before enhanced diagnostics, stack traces at an exception often looked like:

```
at System.Runtime.CompilerServices.TaskAwaiter.ThrowForNonSuccess(...)
at MyApp.Data.UserRepository.GetByIdAsync(...) line 42
```

The **controller action** frame was missing — you knew the repository failed, not which endpoint triggered it.

**.NET 6+** improved **async stack trace** reconstruction for thrown exceptions. **Visual Studio 2022 17.4+** adds **Async Stack Trace** in the debugger UI when **Enable .NET Framework debugging helpers** / managed async debugging is on. You still need the **Tasks** view for **in-flight** waits without an exception.

### The Tasks Window (Visual Studio)

Open while debugging: **Debug → Windows → Tasks** (or **Parallel Windows → Tasks** in some layouts).

The window lists **Task** objects the debugger knows about:

| Column / info | Meaning |
|---------------|---------|
| Id | Task identifier |
| Status | Created, WaitingForActivation, WaitingToRun, Running, WaitingForChildrenToComplete, RanToCompletion, Canceled, Faulted |
| Method | Originating async method when available |
| Location | Where task logic is or was |

**Use cases:**

1. **Hang** — all thread pool threads busy? Tasks window shows dozens of `Waiting` on the same sync primitive.
2. **Never completes** — find `WaitingForActivation` tasks that never transition (missing `SetResult`, deadlock).
3. **Fire-and-forget** — faulted tasks still appear if GC has not collected them and debugger tracks them.

Right-click a task → **Flag** to watch it across steps. Combine with **break on thrown** `TaskScheduler.UnobservedTaskException` for background failures.

### Parallel Stacks and Async

**Debug → Windows → Parallel Stacks** shows threads and, in async-aware mode, **task-based views**: which tasks run on which threads, **blocked** relationships. When a request "hangs", you may see the ASP.NET thread returned to Kestrel but **logical work** continues on thread pool — Parallel Stacks + Tasks reconnect the story.

### Diagnosing Common Scenarios

**Stuck on `await`:** Pause debugger. Check Tasks for your `Task`. If `Waiting` — inspect **sync block** (often `.Result` on another thread). Use **Threads** window: two threads, one waiting on `Monitor`, one holding it inside legacy sync code.

**Missing exception context:** Enable **Tools → Options → Debugging → General → Enable .NET Async** (wording varies by VS version). Re-run; exception dialog shows **async call chain**.

**async void event handler:** Fault appears as **unhandled on thread pool** with shallow stack. Tasks window may not list it as a normal `Task` — prefer `async Task` event handlers in app code.

### VS Code / `dotnet` ecosystem

VS Code C# Dev Kit debugger has more limited **Tasks** UI than full Visual Studio. Practical approach:

- **`dotnet-dump analyze`** on hung process — `clrthreads`, `syncblk`
- **Logging** with correlation IDs at each `await` boundary
- **DiagnosticSource** / OpenTelemetry spans (see observability phase)

For deep async hangs in CI, Visual Studio or WinDbg still win; know the limitation.

## Under the Hood

The debugger hooks into the **CLR thread pool** and **Task** infrastructure via **ICorDebug** and **IAsyncDebug** interfaces. When a `Task` transitions state, debug events can surface it in the Tasks window. **Stack trace enhancement** uses **execution context** and **captured `Activity`**-like trace metadata stored in `Task` objects (.NET 5+ improvements to `Exception.StackTrace` for async).

`ConfigureAwait(false)` does **not** remove tasks from the Tasks window — it only affects **SynchronizationContext** resume target. You still see the `Task`; restoring context is a separate column in advanced views.

**IAsyncStateMachine** types are compiler-generated — stack frames show **MoveNext** and state numbers (`state: 2`). Learn to read: `state: -1` often means completed; positive states map to await points in source (Roslyn maps via `IAsyncMethodBuilder`).

## Syntax / Commands / API

| Action | Visual Studio |
|--------|---------------|
| Open Tasks window | Debug → Windows → Tasks |
| Parallel Stacks | Debug → Windows → Parallel Stacks |
| Break on task cancel | Breakpoint on `Task.ThrowIfCancellationRequested` |
| Unobserved exception | Break on `TaskScheduler.UnobservedTaskException` |
| Freeze threads | Threads window → Freeze inactive threads while stepping |

**LaunchSettings / env for development:**

```json
"COMPlus_ThreadPool_UsePortableThreadPool": "1"
```

(Usually default on .NET 6+; affects thread pool debugging consistency.)

## Examples

### Breakpoint workflow for "API never returns"

1. Reproduce under F5 with breakpoint in controller `await _service.GetAsync()`.
2. When hit, open **Tasks** — note `GetAsync` task Id.
3. Step over await — if never returns, pause all (Break All).
4. Tasks: same Id still `Running` or `Waiting`?
5. Parallel Stacks: thread blocked on SQL/network?
6. If `Waiting` on `ManualResetEventSlim` — search codebase for `.Result` / `.Wait()` in that call chain.

### Reading a state machine frame

```
MyApp.OrdersService.<ProcessOrderAsync>d__3.MoveNext() line 28
```

Line 28 is an `await` boundary. Open `ProcessOrderAsync` — count `await` from start; third await is likely the hang point.

## Common Mistakes & Anti-patterns

**Only watching the main thread** — ASP.NET request thread often idles after first `await`; real work is on thread pool.

**Ignoring faulted tasks in Tasks window** — scroll to **Faulted**; double-click inner exception.

**Assuming empty Tasks means no async work** — very short tasks complete between breaks; use **tracepoints** or logging.

**Debugging Release without PDBs** — async stacks degrade; keep **Debug** symbols for investigations.

**Stepping into `TaskAwaiter` internals** — waste of time; step **out** to user code.

## Production & Real-World Notes

Production hangs use **dotnet-trace**, **dump collection**, **Application Insights** dependency duration — not interactive Tasks window. The **mental model** from Tasks debugging transfers: identify **waiting task**, **what it waits on**, **who started it**.

Enable **OpenTelemetry** `Activity` around each service call — you get distributed traces that resemble an always-on Tasks window across machines.

In code reviews, ask: "If this `await` hangs, what shows in Tasks?" — drives fixes for sync-over-async.

| Tool | Best for | Weakness |

| Logging + correlation ID | All environments | Must instrument ahead |

## Key Takeaways

- Async splits call stacks; **Tasks window** shows logical task state.
- **Parallel Stacks** links thread pool work to blocked/waiting tasks.
- **.NET 6+** improves exception async stacks; still instrument for in-flight waits.
- Hangs often involve **sync-over-async** — visible as cross-thread waits in Tasks/Threads.
- **async void** is hard to debug — avoid in app logic.
- Production: dumps and tracing; local: Tasks + breakpoints on await boundaries.
