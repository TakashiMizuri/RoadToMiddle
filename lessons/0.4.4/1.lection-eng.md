# UDP — When Datagrams Beat Byte Streams

> Roadmap: `0.4.4` · Node: `0.4` — Networking basics · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Explain UDP semantics: connectionless, unordered, unreliable datagram delivery over IP.
- Contrast UDP with TCP reliability and state from `0.4.3` and choose appropriately for a scenario.
- Describe why DNS, live video, VoIP, and online games often prefer UDP.
- Understand message boundaries vs byte streams and the impact on application protocol design.
- Recognize UDP's role beneath QUIC/HTTP/3 (`0.3.13`) as a modern exception to "UDP is only for lossy media."
- Identify when to stay on TCP (HTTP APIs, databases) despite UDP's lower overhead.

---

## Why This Matters

Lesson `0.4.3` established TCP as the workhorse for HTTP, TLS, and PostgreSQL — reliable bytes in order. Yet the same laptop uses **UDP** constantly: every browser navigation starts with a **DNS query** over UDP (`0.4.5`); Zoom and Discord ship audio/video in UDP; multiplayer games send position updates without waiting for retransmissions.

Choosing UDP is not laziness — it is a **trade-off**. Retransmitting a 200 ms old video frame is useless; delivering it late is worse than dropping it. DNS queries are tiny; TCP's three-way handshake would add latency to every name lookup. Middle developers still deploy mostly TCP services, but misconfiguring firewalls (blocking UDP 53), misunderstanding DNS fallback to TCP, or assuming "UDP is insecure" blocks legitimate traffic.

HTTP/3 (`0.3.13`) runs over **QUIC**, which implements reliability and encryption in user space **on top of UDP** — UDP as a substrate, not as "unreliable only." Knowing classic UDP use cases clarifies why QUIC made that design choice.

## Core Concepts

### UDP in One Sentence

**User Datagram Protocol (UDP)** wraps application data in a small header (source port, dest port, length, checksum) and hands it to IP. No connection state, no ACKs, no retransmission, no ordering — **fire and forget** (best effort, like IP, with port multiplexing from `0.4.2`).

Each `send()` produces one **datagram** bounded by size (often practical limit ~1472 bytes payload on Ethernet without fragmentation anxiety; larger may IP-fragment — fragile).

Receiver gets discrete datagrams or nothing — not a continuous stream.

### UDP vs TCP — Decision Matrix

| Need | Prefer |
|------|--------|
| Correct every byte (HTTP, JSON API, SQL) | **TCP** |
| Low latency, tolerate loss (live video, game positions) | **UDP** (often + app logic) |
| Small query/response, speed (DNS) | **UDP** first |
| Multicast/broadcast to many hosts | **UDP** |
| Built-in congestion + flow control without custom code | **TCP** or QUIC |

TCP pays **3-way handshake** (`0.4.3`) and header overhead per segment. UDP sends immediately — one RTT saved for short exchanges.

### DNS — The Canonical UDP Example (`0.4.5` preview)

Resolvers send DNS questions to port **53** as UDP datagrams. Response fits one datagram in common cases → fast. If response exceeds 512 bytes (traditionally) or TC (truncated) flag set, client **retries over TCP** — hybrid design.

Blocking UDP 53 breaks name resolution even when TCP 443 works — classic ops mistake.

### Live Video and VoIP

Video frames are **time-sensitive**. TCP retransmission causes **jitter buffer** growth and delay — viewer sees stall. UDP allows drop frame and move on; codecs interpolate. **RTP** often runs over UDP with sequence numbers at application layer for partial ordering without TCP's head-of-line blocking.

VoIP similarly: late packet discarded; consecutive loss degrades quality briefly rather than freezing conversation seconds.

### Online Games

Player position updates sent 20–60 times per second. Missing one update is superseded by the next. TCP retransmitting old state causes **lag** and weird rubber-banding. Games use UDP with custom prediction, interpolation, and sometimes reliable UDP channels for critical events (inventory) only.

Anti-cheat and encryption add application-layer protocols (not "raw UDP to internet").

### Message Boundaries Matter

TCP: two `write()` calls may merge in one `read()` — HTTP parsers handle framing via Content-Length. UDP: one datagram = one unit; if you send two small datagrams, receiver gets two reads (unless OS coalescing quirks). Protocol designers on UDP must define framing explicitly.

### QUIC and HTTP/3 — UDP as Transport Shell

QUIC (`0.3.13`) uses UDP ports (443 often) so middleboxes that only allow UDP for "known" patterns can pass it, while implementing streams, reliability, and TLS 1.3 inside QUIC — not in kernel TCP. Firewall teams now treat "UDP 443" as first-class for HTTP/3.

This does not replace classic UDP use cases; it **reuses the datagram channel** for a new stack.

## Syntax / Commands / API

```bash
# DNS query over UDP (see 0.4.5)
nslookup example.com

# Test UDP reachability — no standard "connect"; use app-specific tools
# PowerShell: Test-NetConnection -ComputerName 8.8.8.8 -Port 53 -Udp
Test-NetConnection 8.8.8.8 -Port 53
```

**.NET UDP send (conceptual):**

```csharp
using var udp = new UdpClient();
var data = Encoding.UTF8.GetBytes("hello");
await udp.SendAsync(data, data.Length, "host", 9999);
```

No `Connect` required (though connected UDP mode exists for ICMP errors).

## Examples

### Example 1: API vs Game Server

REST API on Kestrel: **TCP 443** — JSON must arrive complete. FPS game server: **UDP** for movement; maybe TCP for login/auth once — hybrid architecture.

### Example 2: Firewall Rule Gap

Security group allows inbound TCP 443 to web tier. Developers enable HTTP/3. Clients fall back to TCP HTTP/2 — works. If you want HTTP/3 performance, add **UDP 443**.

## Common Mistakes & Anti-patterns

**"UDP is always faster than TCP."** For large transfers TCP's congestion control often wins; UDP without control can collapse Wi‑Fi.

**Blocking all UDP in corporate firewall** — breaks DNS, QUIC, NTP, DHCP.

**Using UDP for file upload** without custom reliability — silent data loss.

**Assuming datagram order** — application must sequence if order matters.

## Production & Real-World Notes

Monitor DNS over UDP vs TCP ratio. VoIP/WebRTC uses UDP with STUN/TURN; TURN may relay over TCP/TLS if UDP blocked — degraded experience.

DDoS amplification attacks abuse UDP services (DNS, NTP) — rate limit and BCP38 at providers.

Game backends and realtime use **custom protocols** on UDP; still need auth and encryption (DTLS).

---

## Comparison / Trade-offs

**TCP:** simplicity for apps, kernel maturity, HTTP ecosystem. **UDP:** control, latency, custom semantics; you own reliability where needed.

**DNS UDP+TCP fallback:** best of both for variable response sizes.

## Key Takeaways

- UDP delivers independent datagrams without connection setup or retransmission.
- Choose UDP when timeliness beats completeness or messages are small/request-like (DNS).
- Video, VoIP, and games accept loss to avoid TCP delay and head-of-line blocking.
- DNS primarily UDP port 53 with TCP fallback for large responses (`0.4.5`).
- HTTP/3/QUIC uses UDP as a carrier while providing reliability in user space (`0.3.13`).
- Business APIs and databases remain TCP-first unless you have explicit realtime requirements.
