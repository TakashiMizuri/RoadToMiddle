# Covariance/contravariance — краткая выжимка

> `1.3.2` · ~10 мин на повторение

## Главная идея

**Variance** определяет, можно ли подставить `IGeneric<Derived>` там, где ожидается `IGeneric<Base>`. **`out T`** (covariant) — `T` только **выходит** из API; **`in T`** (contravariant) — `T` только **входит**. **`List<T>` invariant** — есть запись.

## Ключевые пункты

- `IEnumerable<string>` → `IEnumerable<object>` ✓ (covariance).
- `List<string>` → `List<object>` ✗ (invariance).
- `Action<object>` → `Action<string>` ✓ (contravariance).
- `Func<in T, out R>` — input contravariant, return covariant.
- Variance — только **interfaces** и **delegates**, не generic classes.
- **Value types** — `IEnumerable<int>` → `IEnumerable<object>` ✗.
- **Arrays** reference types — covariant (legacy, runtime exception при неверной записи).

## Самые важные примеры

```csharp
IEnumerable<object> o = new List<string>(); // OK
Action<string> a = (Action<object>)(obj => { }); // OK
```

## «Под капотом» в одном абзаце

CLR хранит variance flags на generic parameters. Verifier блокирует interface methods, нарушающие `in`/`out`. Covariant `List` был бы unsound: запись через `List<object>` испортила бы `List<string>`. `IReadOnlyList<out T>` без mutators — безопасен. Delegates — generic interfaces с теми же правилами.

## Частые ошибки

- Ожидать covariance у `List<T>`.
- Путать `(IEnumerable<object>)list` (OK) с covariance самого `List`.
- Variance для `int` → `object` без boxing.

## Мнемоника

**PECS / out=producer, in=consumer**
