# Repository, Working Tree, Staging Area, and Commit

> Roadmap: `0.1.2` · Node: `0.1` — Git Basics · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Define a **Git repository** as the combination of a **working tree** and the **`.git` directory**.
- Explain the **three states** of a file: modified, staged, and committed.
- Distinguish **working directory**, **staging area (index)**, and **repository (`.git`)**.
- Articulate **why the staging area exists** and when partial commits are useful.
- Describe a **commit** as a snapshot with metadata (author, timestamp, message, parent).
- Trace a file's **lifecycle** from edit to commit using a diagram.

---

## Why This Matters

In `0.1.1` you learned that Git is **distributed**: every clone holds the full history of a project, not just the latest files. That architectural choice explains why you can commit on a plane and push later. It does not yet explain what happens on your laptop *before* you push — how Git knows you changed a file, why some changes appear in one section of `git status` and others in another, or why professionals insist on both `git add` and `git commit`.

Unlike some version control tools where saving work immediately writes to permanent history, Git inserts an intermediate layer called the **staging area** (also the **index**). Beginners often ask, "Why do I need two steps?" The question is fair; the answer is that Git optimizes for **deliberate snapshots** rather than accidental ones. Once you internalize the **three-tree model** — working directory, staging area, repository — several daily mysteries disappear. You understand why `git status` distinguishes "Changes not staged for commit" from "Changes to be committed." You see how to commit **only part** of your work in one atomic snapshot. You gain a preview of what `add` and `commit` actually write, which we will open further in `0.1.4`.

Every workflow you will use as a fullstack developer — feature branches, pull requests, code review, CI builds — rests on this foundation. Remote operations come later; local state management comes first.

## Core Concepts

### What Is a Git Repository?

When someone says "clone the repository," it is easy to imagine a folder of source files. That picture is incomplete. A **Git repository** (repo) consists of two cooperating parts: the **working tree** (also **working directory**) — the project files you see, edit, and run — and the **Git directory**, the hidden `.git/` folder that stores Git's database of snapshots, history, and configuration.

```
my-project/
├── .git/              ← repository (history, objects, refs)
├── src/
│   └── app.cs         ← working tree (files you edit)
├── README.md
└── Program.cs
```

When you run `git clone` or `git init`, Git creates or copies the `.git` directory. Everything Git "knows" about history lives there. The visible project files are your **checkout**: content extracted from the latest commit so you can work. The working tree is allowed to diverge from what is recorded in `.git`; Git's job is to track and eventually record those differences when you choose to commit.

This separation matters in practice. You can edit ten files but commit only two. You can have uncommitted experiments sitting beside production-ready changes. The repository is not "whatever is on disk right now" — it is "what has been explicitly staged and committed into `.git`."

### The Three States of a File

Git classifies each **tracked** file into one of three states at any moment. A file is **modified** when you have changed it in the working tree but have not yet marked those changes for the next commit. It becomes **staged** after `git add` places a snapshot of selected changes into the staging area (index), ready for `git commit`. It is **committed** once the commit command writes that staged snapshot into permanent history inside `.git`.

The transitions form a loop you will execute hundreds of times:

```
                    git add                git commit
  ┌──────────┐    ──────────→    ┌──────────┐    ──────────→    ┌──────────┐
  │ Modified │                   │  Staged  │                   │Committed │
  │(working) │    ←──────────    │ (index)  │                   │  (.git)  │
  └──────────┘    git restore    └──────────┘                   └──────────┘
                  (unstage)
```

Having seen the basic path, we must add a nuance that surprises many newcomers. A file can be **modified and staged at the same time**. If you `git add Program.cs` and then edit `Program.cs` again before committing, Git tracks both the staged portion (what you added earlier) and new unstaged edits. `git status` will list the same filename under two headings. That is not a bug — it reflects Git's precise model of "what the next commit will contain" versus "what you are still editing."

**Untracked** files sit outside this three-state cycle until their first `git add`. They appear in `git status` but are not yet part of version control.

### Three Trees: Working Directory, Staging Area, Repository

Git's internal model is often called the **three-tree architecture**. The names refer to three views of your project that Git compares constantly:

```
┌─────────────────┐     git add      ┌─────────────────┐    git commit    ┌─────────────────┐
│ Working         │  ──────────────→ │ Staging Area    │  ──────────────→ │ Repository      │
│ Directory       │                  │ (Index)         │                  │ (.git/)         │
│                 │  ← git checkout  │                 │                  │                 │
│ Files you edit  │    -- file       │ Next snapshot   │                  │ Permanent       │
│ on disk         │                  │ preview         │                  │ history         │
└─────────────────┘                  └─────────────────┘                  └─────────────────┘
```

The **working directory** is the project as your IDE and file explorer show it. Typing in a file changes the working directory. Git compares working files against the index and against the last commit (`HEAD`) to produce status output.

The **staging area** — the **index** — sits between casual edits and permanent history. `git add` records a snapshot of chosen changes into the index. Think of it as a **shipping crate**: you decide exactly which changes belong in the next package before sealing it. The index is not a metaphor only; it is a real binary file at `.git/index`, which we will relate to objects in `0.1.4`.

The **repository** is the `.git` directory itself. A **commit** writes the staged snapshot from the index into this database and links it to previous history. Committed data is **content-addressed** (identified by hash), making accidental silent corruption unlikely — details in `0.1.5`.

### Why Does Staging Exist?

The natural follow-up question is why Git does not commit directly from the working directory. The staging area exists because real work is messy. In one afternoon you might fix a typo in `README.md`, implement half of a login API, and tweak a local `appsettings.json` with a dev-only connection string. Without staging, you would either commit everything together — mixing documentation, feature code, and local config — or rely on a complex partial-commit interface.

Staging enables **atomic commits**: one commit should represent one logical change understandable in code review. It enables **partial commits**: stage `AuthService.cs` while leaving experimental `ScratchPad.cs` unstaged. It enables **review before commit**: `git diff --staged` shows precisely what will enter history — a final checklist. It enables **deliberate snapshots** at hunk granularity via `git add -p` (patch mode), covered hands-on in `0.1.3`.

An analogy many find helpful: the working directory is a messy desk; the staging area is the inbox tray for the next shipment; the repository is archived, labeled boxes on the shelf. Git chose explicit staging for **control and clarity**, accepting a small extra step in exchange for history you can trust six months later.

### What Is a Commit?

A **commit** is not a casual save. It is an immutable **snapshot** of your project at a point in time, plus **metadata** that makes the snapshot meaningful in a social and historical context. The snapshot itself (via tree objects) lists which files existed and what content hashes they pointed to. **Parent** pointer(s) link the commit to its predecessor(s), forming the history graph. **Author** and **committer** record who wrote and who recorded the change (they can differ when someone applies another person's patch). **Timestamp** anchors the event in time. **Commit message** explains *why* — the human-readable story future teammates and future you will read in `git log`.

```
Commit abc123
├── parent: def456
├── author: Alice <alice@dev.com>  2026-07-01 10:00
├── message: "Add password validation"
└── snapshot:
    ├── src/AuthService.cs  (content hash ...)
    ├── src/Program.cs
    └── README.md
```

Git stores **snapshots**, not merely line-by-line deltas between commits, though it compresses efficiently. Unchanged files across commits can reference the same underlying objects — no wasteful duplication of identical content.

Commits are **immutable**. You do not edit history in place; you create new commits and move branch pointers. "Changing history" later (rebase, amend) means creating new objects — topics for advanced lessons. For now, treat each commit as a permanent journal entry you want to be proud of.

## Under the Hood (Conceptual)

You do not need to read raw files inside `.git/` yet, but you should understand the flow that connects the three trees:

```
1. You edit Program.cs          → working tree changes
2. git add Program.cs           → Git hashes content → blob object
                                 → index entry updated to point at blob
3. git commit                   → tree object built from index
                                 → commit object created (tree + parent + metadata)
                                 → branch ref (e.g. main) moves to new commit
```

The **index** is Git's blueprint for the **next** commit. Until you commit, permanent history is unchanged — only the index and working tree may differ from `HEAD`. **`HEAD`** is a pointer to your current commit, usually through a branch name like `main`. We unpack refs and objects in `0.1.4`.

## Syntax / Commands / API

This lesson is conceptual; hands-on commands arrive in `0.1.3`. For now, associate each command with movement between states:

- **`git status`** — shows which state each file occupies.
- **`git add <path>`** — moves changes from working tree to staged (index).
- **`git restore --staged <path>`** — unstages: staged back toward working tree without deleting edits.
- **`git restore <path>`** — discards working tree changes (dangerous: can lose work).
- **`git commit`** — moves staged snapshot into committed history in `.git`.

## Examples

### Example 1: Simple file lifecycle

Imagine `README.md` is clean — committed and matching `HEAD`. You edit the file: it becomes **modified** (unstaged). Running `git add README.md` promotes those edits to **staged**. Running `git commit -m "Update docs"` writes them into history; the file is **committed**, and if you made no further edits, `git status` reports a clean working tree.

```
# Start: README.md is committed (clean)
# 1. Edit README.md           → modified (unstaged)
# 2. git add README.md        → staged
# 3. git commit -m "Update docs" → committed; working tree clean again
```

### Example 2: Partial commit

You edited both `AuthService.cs` (a real feature) and `appsettings.json` (a local dev URL you must not share). Staging only the auth file lets you commit the feature without the config tweak:

```
git add AuthService.cs          # only auth goes to staging
git commit -m "Add JWT validation"
# appsettings.json still modified, unstaged — not in the commit
```

This pattern is daily practice on fullstack projects where configuration and secrets must stay out of shared history.

### Example 3: Same file, mixed state

You stage `Program.cs`, then continue editing it before committing. Git now tracks staged content separately from newer unstaged lines:

```
git add Program.cs              # stage current version
# edit Program.cs again       # now: staged portion + new unstaged edits
git status
# → "Changes to be committed" AND "Changes not staged for commit" for Program.cs
```

When this happens, `git diff` shows unstaged edits; `git diff --staged` shows what the commit would include if you ran it immediately.

## Common Mistakes & Anti-patterns

**Blind `git add .` before every commit** trains you to commit whatever happens to be changed — feature code, debug logging, accidental config. Review with `git status` and `git diff --staged` first.

**Treating commit as "save" in your editor** misleads teammates. Saving updates the working tree; committing creates a permanent, labeled history entry with a message others will read.

**Dismissing the staging area as optional overhead** leads to confusion when one file appears twice in status output. Staging is core Git design, not a beginner hurdle to skip mentally.

**Equating repository with GitHub** forgets that your **local `.git`** *is* the repository. GitHub is a remote copy for sharing; you can commit locally with no remote at all.

**Ignoring untracked files** leaves new files invisible to the three-state model until the first `git add`. They still show in status — learn to notice them before assuming a "clean" commit.

## Production & Real-World Notes

Teams expect **atomic commits** in code review: one pull request should tell one coherent story, often built from commits that each represent a logical step. Staging lets you commit production code without local `.env` experiments and split a large refactor into reviewable chunks.

**CI pipelines build committed snapshots**, not whatever is unstaged on a developer's machine. Uncommitted work-in-progress never reaches the build server — another reason to understand states precisely.

Copying source files without the `.git` directory **destroys history**. Share work through clone, pull, or push — not email attachments of folders minus `.git`.

## Comparison / Trade-offs

Git's staging area trades a little ceremony for precise control. Some tools commit directly from the working tree, which feels simpler initially but makes partial commits and pre-commit review harder. Git chose explicit staging; accepting that trade-off is part of becoming fluent. The alternative — no VCS — avoids learning curve entirely at the cost of history, collaboration, and rollback.

## Quick Reference

```
REPOSITORY     = working tree + .git/
WORKING TREE   = files you edit
STAGING AREA   = index; next commit preview
.git/          = permanent history database

THREE STATES:
  modified  → edited, not staged
  staged    → git add done, waiting for commit
  committed → in .git history

FLOW:
  edit → git add → git commit

WHY STAGING:
  atomic commits | partial commits | review before seal
```

## Key Takeaways

1. A Git repo = **working tree** + **`.git` directory`**.
2. Files move through **modified → staged → committed**.
3. **Working directory**, **staging area (index)**, and **repository** are Git's three trees.
4. **Staging exists** for atomic commits, partial commits, and pre-commit review.
5. A **commit** is an immutable **snapshot + metadata**, not a casual save.
6. **Untracked** files are outside this model until first `git add`.
7. Understanding states makes `git status` and `git diff` readable in `0.1.3`.
