# CancellationToken в ASP.NET — RequestAborted

> Roadmap: `1.4.29` · Node: `1.4` — C# async/await · Depth: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Объяснить, что представляет **`HttpContext.RequestAborted`** и когда он срабатывает.
- Пробрасывать **`CancellationToken`** из endpoint через сервисы в EF Core и `HttpClient`.
- Различать **отключение клиента**, **остановку сервера** и **явный timeout**.
- Не тратить CPU, соединения с БД и внешние вызовы после ухода клиента.
- Понимать, когда отмену применять **нельзя** (side effects, fire-and-forget).

---

## Why This Matters

Пользователь нажал «Поиск» и сразу ушёл со страницы. Без cancellation API продолжает ходить в PostgreSQL, звонить во внешний сервис и держать continuation в thread pool — работу, которую никто не увидит. Под нагрузкой это **давление на connection pool**, **рост latency** для активных пользователей и **деньги на внешних API**.

ASP.NET Core подключает **`RequestAborted`** к каждому HTTP-запросу. Middle-разработчик обязан относиться к нему как к полноценному параметру и пробрасывать **async all the way** до границ I/O. Это production-дополнение к основам `CancellationToken` из `1.4.11`: тот же тип, другой **владелец жизненного цикла** (Kestrel и состояние TCP-соединения клиента, а не ваш `CancelAfter`).

## Core Concepts

### Что такое RequestAborted

Каждый HTTP-запрос в ASP.NET Core даёт **`HttpContext.RequestAborted`** — **`CancellationToken`**, который переходит в состояние **cancelled**, когда:

- **Клиент закрыл соединение** (вкладка, фоновый режим приложения, idle timeout балансировщика).
- **Сервер останавливается** (graceful shutdown; `IHostApplicationLifetime.ApplicationStopping` связывается с токенами запросов).
- Middleware или ваш код **связывает** другой токен (Polly timeout, собственный deadline).

Это **не** означает «пользователь нажал Cancel в UI», если клиент реально **не прервал HTTP-запрос** (`AbortController` в `fetch`, cancel token в Axios). Многие SPA просто игнорируют in-flight ответы — тогда отмена произойдёт только при unmount, если вы это настроили.

Токен **кооперативный**: цикл `for` не остановится, пока вы не проверите `IsCancellationRequested` или не передадите токен в `Task.Delay`, `SendAsync`, запрос EF и т.д.

### Паттерн проброса

Канонический подход — **параметр в сигнатуре**:

```csharp
app.MapGet("/orders/{id}", async (int id, IOrderService orders, CancellationToken ct) =>
{
    return await orders.GetOrderAsync(id, ct);
});
```

ASP.NET Core **автоматически** подставляет `CancellationToken` при биндинге параметра — атрибуты не нужны. Это тот же токен, что и **`HttpContext.RequestAborted`** (или связанный композит при shutdown).

Сервисный слой принимает **`CancellationToken cancellationToken = default`** на каждом I/O-методе и передаёт дальше. **`default`** — «без отмены» для тестов и консоли; в web-запросе framework подставляет реальный токен.

### Что отменяется — и что нет

При срабатывании токена **I/O, уважающий cancellation**, бросает **`OperationCanceledException`** (часто с **`TaskCanceledException`** от HTTP). EF Core отменяет команду; Npgsql закрывает чтение; `HttpClient` прерывает сокет.

**CPU-bound** работа сама не останавливается — в цикле нужен опрос токена:

```csharp
for (var i = 0; i < huge; i++)
{
    cancellationToken.ThrowIfCancellationRequested();
}
```

**Уже зафиксированные side effects** отменой не откатываются. Списали деньги — disconnect клиента платёж не отменит; нужны **idempotency** и **compensation**, не токен.

### Связанные токены и таймауты

В production часто комбинируют **abort клиента** и **серверный deadline**:

```csharp
using var cts = CancellationTokenSource.CreateLinkedTokenSource(
    httpContext.RequestAborted);
cts.CancelAfter(TimeSpan.FromSeconds(30));
await _db.Orders.ToListAsync(cts.Token);
```

Срабатывает тот, кто первый. **`using`** на `CancellationTokenSource` освобождает регистрации linked source.

## Under the Hood

### Kestrel и lifetime соединения

Kestrel отслеживает **`ConnectionAborted`** на соединении. RST от клиента или half-close, означающий abandonment, ведёт к отмене токена запроса. На части платформ есть **задержка** до обнаружения «мёртвого» peer.

В **HTTP/2** и **HTTP/3** reset stream отменяет конкретный запрос на мультиплексированном соединении.

### Middleware и hosting

При **graceful shutdown** хост сигнализирует отмену: запросы должны завершиться в **`ShutdownTimeout`** или прерваться. `BackgroundService` использует **`stoppingToken`**, не `RequestAborted`.

### EF Core и провайдеры

Отмена доходит до ADO.NET; PostgreSQL шлёт cancel message. **`OperationCanceledException`** может прийти **после** частичной работы БД — отмена ≠ rollback транзакции.

### Семантика исключений

**`OperationCanceledException`** при отмене клиентом часто **ожидаема**, не баг. Фильтруют в exception middleware:

```csharp
catch (OperationCanceledException) when (context.RequestAborted.IsCancellationRequested)
{
    // клиент ушёл — не 500
}
```

Логировать каждый disconnect как **Error** — шум для алертов. На scale — **Information** или без лога.

## Syntax / Commands / API

| API | Роль |
|-----|------|
| `HttpContext.RequestAborted` | Токен запроса от Kestrel/host |
| `CancellationToken` в action | Автоматический injection |
| `ThrowIfCancellationRequested()` | Быстрый выход из CPU-циклов |
| `CreateLinkedTokenSource` | Abort + timeout + custom |
| `SendAsync(..., ct)` | Проброс во внешний HTTP |
| `ToListAsync(ct)` | Отмена запросов EF |

## Examples

### Minimal API + EF Core

```csharp
app.MapGet("/products", async (AppDbContext db, CancellationToken ct) =>
{
    return await db.Products.AsNoTracking().ToListAsync(ct);
});
```

Клиент отключился mid-query — EF/Npgsql прерывает чтение; `OperationCanceledException`. Тело ответа часто не пишется — соединение уже мертво.

### Сервисный слой

```csharp
public async Task<OrderDto?> GetOrderAsync(int id, CancellationToken ct)
{
    var order = await _db.Orders.AsNoTracking()
        .FirstOrDefaultAsync(o => o.Id == id, ct);
    if (order is null) return null;

    var shipping = await _http.GetFromJsonAsync<ShippingStatus>(
        $"/status/{order.TrackingId}", ct);
    return Map(order, shipping);
}
```

Каждый `await` с `ct` — отмена идёт по **async chain** при дисциплине в сигнатурах.

## Common Mistakes & Anti-patterns

**Игнорировать токен** — метод принимает `CancellationToken`, но не передаёт в I/O.

**Ловить `OperationCanceledException` и отдавать 500** — шум в мониторинге.

**`CancellationToken.None` в библиотеке**, когда caller передал токен — почти всегда нужно форвардить.

**Думать, что cancel откатывает транзакцию** — только явный rollback.

**Долгая работа после response** только с `RequestAborted` — после ответа токен может сработать; нужна очередь (`1.4.34`) или `stoppingToken`.

## Production & Real-World Notes

Часто добавляют **глобальный request timeout** middleware, связанный с `RequestAborted`.

**Load balancer** idle timeout может быть короче DB query — выровняйте таймауты по слоям.

В **tracing** помечайте `cancellation_requested` для отделения SLA от benign disconnects.

## Comparison / Trade-offs

| Подход | Плюсы | Минусы |
|--------|-------|--------|
| `RequestAborted` везде | Экономия ресурсов | Многословные сигнатуры |
| Только server timeout | Предсказуемый max | Клиент ждёт до timeout |
| Без cancellation | Проще код | Потери при churn |
| Fire-and-forget | Быстрый UX | Не только `RequestAborted` (`1.4.30`) |

## Quick Reference

- Параметр `CancellationToken ct` в endpoint → во всё I/O.
- `RequestAborted` ≡ `CancellationToken` в ASP.NET Core endpoint.
- Срабатывает: disconnect, shutdown, linked cancel.
- `OperationCanceledException` + `RequestAborted` — обычно не 500.
- Link + `CancelAfter` для deadlines.
- EF/HttpClient: `*Async(..., cancellationToken)`.

---

## Key Takeaways

- **`RequestAborted`** — сигнал отмены на время запроса.
- Отмена **кооперативная** — передай токен в каждый I/O.
- Disconnect клиента ≠ баг сервера.
- Комбинируй с **timeouts** через linked CTS.
- Отмена **не откатывает** committed side effects.
- Пробрасывай через сервисы, EF, `HttpClient`.
