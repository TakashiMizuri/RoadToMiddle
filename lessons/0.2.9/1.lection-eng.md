# Conventional Commits: feat, fix, chore, BREAKING

> Roadmap: `0.2.9` · Node: `0.2` — Git: branches and collaboration · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- State the **Conventional Commits** format: `type(scope)!: description`.
- Use common types: **`feat`**, **`fix`**, **`chore`**, **`docs`**, **`refactor`**, **`test`**, **`ci`**.
- Mark **breaking changes** with `!` and/or `BREAKING CHANGE:` footer.
- Explain how conventional messages power **changelogs**, **semantic versioning**, and **release automation**.
- Write **imperative**, concise subject lines suitable for PR squash titles.
- Avoid vague messages (`update`, `fix stuff`) that break team automation.

---

## Why This Matters

You can merge perfect code through a flawless PR (`0.2.8`) and still hurt your team with commit messages like `wip`, `fix`, or `changes`. Six months later, `git log` should answer: *What changed? Was it user-visible? Did it break the API?* **Conventional Commits** — a lightweight convention on top of ordinary Git messages — makes history **machine-readable** and **human-scannable**.

Tools parse `feat:` vs `fix:` to build **changelogs** without manual copy-paste. **Semantic Release** and similar pipelines bump **MAJOR.MINOR.PATCH** from commit types: `feat` → minor, `fix` → patch, breaking → major. Even without full automation, reviewers scanning a PR commit list instantly see intent. Squash-merge titles often copy the convention: `feat(auth): add refresh token rotation (#102)`.

For a fullstack Middle developer shipping ASP.NET APIs and React apps, consistent messages connect Git history to **release notes**, **client SDK versioning**, and **incident postmortems** ("which commit introduced the regression?").

## Core Concepts

### The Format

```
<type>[optional scope][optional !]: <description>

[optional body]

[optional footer(s)]
```

**Type** — what kind of change (required). **Scope** — optional subsystem hint (`api`, `auth`, `ui`). **!** — breaking change indicator. **Description** — short imperative summary, lowercase, no period at end. **Body** — context, motivation, migration notes. **Footer** — metadata like `BREAKING CHANGE:`, `Fixes #42`.

Examples:

```
feat(orders): add PATCH endpoint for partial updates
fix(auth): reject expired refresh tokens
chore(deps): bump EF Core to 8.0.4
docs(readme): document local Docker setup
```

### Core Types

**`feat`** — new feature; **user-visible capability** or new API surface. Maps to **MINOR** in SemVer when releasing.

**`fix`** — bug fix for users or production behavior. Maps to **PATCH**.

**`chore`** — maintenance without production code behavior change: deps, tooling, `.gitignore`. Not user-facing.

**`docs`** — documentation only.

**`refactor`** — code structure change without fixing a bug or adding feature behavior.

**`test`** — adding or correcting tests.

**`ci`** — CI/CD configuration changes.

**`perf`** — performance improvement.

**`style`** — formatting, whitespace (not CSS); rare in backend repos.

Teams may add custom types (`build`, `revert`) — document in CONTRIBUTING.md.

### Breaking Changes

A **breaking change** alters API or behavior in a way consumers must adapt — removed endpoint, renamed JSON field, changed auth flow.

Signal two ways:

1. **`!` after type/scope:** `feat(api)!: remove v1 orders endpoint`
2. **Footer:** 
```
BREAKING CHANGE: Orders v1 endpoints removed; use /api/v2/orders.
```

Breaking commits trigger **MAJOR** version bumps in automated release. Even manual teams flag them loudly for coordination between backend and frontend.

### Subject Line Rules

- **Imperative mood:** "add endpoint" not "added endpoint" (matches Git's own generated messages).
- **~50 chars** for subject when possible — readable in `git log --oneline`.
- **No vague words:** `update`, `misc`, `fix issue` — useless for bisect and changelog.
- **One logical change per commit** — connects to atomic commits from `0.1.2`.

Body optional but valuable for **why**, migration steps, and security implications.

### Conventional Commits and PR Workflow

In PR with multiple commits, each message should follow convention. With **squash merge**, edit squash title to best represent whole PR — often one `feat(scope): summary`. Reviewers may request message fixes before merge (`git commit --amend` locally if unpushed, or new commit).

Link issues in footer:

```
fix(payments): handle null currency code

Stripe webhook crashed on missing currency.

Fixes #218
```

## Under the Hood

Conventional Commits is **not** a Git feature — it is a **parsing convention**. Tools like **commitlint**, **semantic-release**, **standard-version** regex-match headers. Angular convention historically influenced this spec.

```
^(\w+)(\(.+\))?(!)?: (.+)$
```

Breaking footer detected separately. CI may **reject** non-conforming commits on PR branches.

Squash merge composes one message — teams sometimes enforce PR title format via templates.

## Syntax / Commands / API

Writing commits:

```bash
git commit -m "feat(health): expose liveness probe endpoint"
git commit -m "fix(auth): validate audience claim" -m "Fixes #42"

# Breaking
git commit -m "feat(api)!: rename UserDto to UserResponse" \
  -m "BREAKING CHANGE: clients must update JSON mapping for user endpoints."
```

Amend before push (`0.1.10`):

```bash
git commit --amend -m "fix(orders): correct tax calculation for EU"
```

Interactive rebase to reword history before PR (`0.2.3`):

```bash
git rebase -i origin/main   # reword commits with bad messages
```

## Examples

### Example 1: ASP.NET API feature

```
feat(notifications): add POST /api/notifications/preferences

Persists per-user channel settings (email, push).
Requires migration 20260703_AddNotificationPrefs.

Closes LINEAR-441
```

### Example 2: React fix

```
fix(checkout): disable submit while payment intent loads

Prevents double charge when user double-clicks Pay.
```

### Example 3: Chore vs fix

```
chore(ci): cache NuGet packages in GitHub Actions
```

Not `fix(ci)` unless pipeline was **broken** and this restores correctness for developers.

### Example 4: Breaking API change

```
feat(users)!: return ISO-8601 dates only in UserResponse

BREAKING CHANGE: CreatedAt and UpdatedAt no longer use Unix milliseconds.
Mobile clients must update parsing logic before deploy.
```

## Common Mistakes & Anti-patterns

**Everything is `fix` or `update`** — changelog useless; automation wrong version bumps.

**Multi-paragraph subject** — belongs in body; breaks `oneline` tools.

**Past tense** — `added feature` vs imperative `add feature`.

**Scope spam** — `feat(app): ...` every time adds noise; scope when it helps filters.

**Breaking change hidden in body** without `!` or footer — consumers miss migration notes.

**Mixing unrelated changes** — one commit `feat: login and fix footer typo` — split commits.

## Production & Real-World Notes

**semantic-release** on `main` publishes NuGet/npm versions from commits. **Conventional Changelog** generates RELEASE notes.

Monorepos use **scopes** per package: `feat(api-client): ...`.

Some teams enforce via **commitlint** hook — commit rejected before PR.

**Revert** commits often: `revert: feat(auth): experimental SSO` — documents rollback.

Align with **API versioning** — breaking `!` should correlate with `/v2` routes.

## Comparison / Trade-offs

| Approach | Pros | Cons |
|----------|------|------|
| Conventional Commits | Automation, clarity, SemVer | Learning curve, rigid CI |
| Free-form messages | Flexible | Poor search, manual changelogs |
| Single squash message only | Simple main log | Loses step-by-step PR narrative |

Many teams: conventional on PR commits, squash to one conventional title on merge.

## Quick Reference

```
Format: type(scope)!: imperative description

Common types:
  feat     → new capability (MINOR)
  fix      → bug fix (PATCH)
  chore    → tooling/deps
  docs     → documentation
  test     → tests
  refactor → restructure, same behavior
  ci       → pipeline

Breaking: ! in header and/or BREAKING CHANGE: footer (MAJOR)

Bad:  "update", "wip", "fix stuff"
Good: "fix(auth): expire session on password change"
```

## Key Takeaways

1. **Conventional Commits** = `type(scope)!: description` + optional body/footer.
2. **`feat`** and **`fix`** are the primary release-related types.
3. **`chore`** for non-user-facing maintenance.
4. **Breaking changes** need `!` or `BREAKING CHANGE:` — MAJOR SemVer.
5. **Imperative**, specific subjects help log, bisect, and PR review.
6. Tools can **automate** changelogs and versions from messages.
7. Pair with atomic commits and PR hygiene from `0.2.8`.
