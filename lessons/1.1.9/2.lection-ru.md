# Управление потоком: if/else и switch (классический + expression)

> Roadmap: `1.1.9` · Node: `1.1` — C#: основы языка · Depth: practice

## Learning Objectives

После этого урока ты сможешь:

- Писать **ветвления** через `if`, `else if` и `else`, направляя выполнение программы в зависимости от данных.
- Корректно вычислять **boolean-выражения**, понимая **short-circuit** evaluation и влияние на side effects.
- Выбирать между **классическим `switch`** и **`switch` expression** для многовариантных решений.
- Использовать **pattern matching** в современном `switch` для типов, констант и relational patterns.
- Избегать типичных ошибок: **dangling else**, **присваивание вместо сравнения**, пропуск **`break`** в classic switch.

## Why This Matters

В `1.1.8` ты изучил операторы — «словарь» для вопросов к данным. Они дают **bool**-ответ: проходит ли балл? admin ли пользователь? null ли input? **Control flow** — это то, что ты **делаешь** с этими ответами. Без ветвлений каждая программа шла бы сверху вниз как магнитофонная дорожка. Реальные приложения постоянно решают: валидировать input, выбрать HTTP status, tier цены, пропустить пустую строку в отчёте.

Fullstack-разработчик пишет conditionals в controllers (`if (ModelState.IsValid)`), domain logic (`if (order.Total > threshold)`) и на frontend — но в C# механика начинается здесь. Middle-разработчик выбирает структуру, которая **передаёт intent**. Пятиступенчатый `if/else`, который должен быть `switch` по enum — smell читаемости. `Switch` expression, возвращающий value, часто яснее, чем мутация переменной в каждой ветке. Этот урок даёт и классические инструменты, и modern C# idioms production-кода.

## Core Concepts

### Пути выполнения и оператор `if`

Программа на C# — последовательность statements, но **control flow statements** меняют, какой statement выполнится следующим. Простейшее ветвление — **`if`**: если boolean condition `true`, выполни block; иначе пропусти.

```csharp
int score = 72;

if (score >= 60)
{
    Console.WriteLine("Pass");
}
```

Condition в скобках должен быть **`bool`**. C# не трактует `0`/`1` как false/true, в отличие от C — это strictness предотвращает целый класс багов. Когда condition true, runtime входит в **block** `{ }`. Для одного statement скобки опциональны, но style guides почти всегда требуют их: второй добавленный line может случайно оказаться вне ветки.

**`else`** выполняется, когда `if` false:

```csharp
if (score >= 60)
{
    Console.WriteLine("Pass");
}
else
{
    Console.WriteLine("Fail");
}
```

**`else if`** — цепочка проверок сверху вниз; выполняется **только первая** подходящая ветка:

```csharp
if (score >= 90)
    Console.WriteLine("A");
else if (score >= 80)
    Console.WriteLine("B");
else if (score >= 70)
    Console.WriteLine("C");
else
    Console.WriteLine("Below C");
```

Порядок важен. Если поменять первые две проверки местами, score 95 ошибочно станет `"B"`, потому что сначала сработает `>= 80`. Это не ошибка компилятора — **logic bug**, и careful ordering — часть craft.

### Boolean conditions и short-circuit evaluation

Conditions часто комбинируют comparisons с **logical operators** из `1.1.8`. **`&&` и `||` short-circuit**: правая часть может не вычисляться.

```csharp
string? name = null;

if (name != null && name.Length > 0)
{
    Console.WriteLine(name);
}
```

Так как `name != null` false, C# **не** вычисляет `name.Length` — нет `NullReferenceException`. Этот pattern повсюду в production. Если нужны **обе** части (редко), используй `&` и `|` для bool — но по умолчанию `&&` и `||`.

Классическая ловушка — **`if (x = 5)`** (assignment вместо comparison). C# часто отвергает это в bool context, но паттерны вроде **`if (flag = GetFlag())`** всё ещё кусают juniors.

### Тернарный conditional operator

Когда нужно выбрать **два value** (не два произвольных block statements), **ternary operator** `? :` компактен:

```csharp
string label = score >= 60 ? "Pass" : "Fail";
```

Читай: *если condition — это value, иначе то*. Ternary — expression, даёт value для assign, argument или return. Не вкладывай пять уровней — тогда `if/else` или `switch` яснее. В API часто: `var status = isActive ? UserStatus.Active : UserStatus.Inactive;`.

### Классический `switch`: многовариантное ветвление

Когда одну переменную (или expression) сравнивают с **множеством constant cases**, **`switch`** часто яснее длинной цепочки `if/else if`:

```csharp
DayOfWeek day = DateTime.Today.DayOfWeek;

switch (day)
{
    case DayOfWeek.Saturday:
    case DayOfWeek.Sunday:
        Console.WriteLine("Weekend");
        break;
    case DayOfWeek.Monday:
        Console.WriteLine("Start of work week");
        break;
    default:
        Console.WriteLine("Midweek");
        break;
}
```

Каждый **`case`** — метка ветки. **`break`** выходит из switch — в отличие от C/C++, C# **не** проваливается в следующий case без явного **`goto case`** (почти никогда в application code). **`default`** — когда ничего не совпало; опционален, но полезен для defensive handling.

**Switch expression** должен быть совместим с case labels: числа, `char`, `string`, enums и (в modern C#) patterns. Не switch по float/double для exact equality — IEEE арифметика; используй ranges или integer cents для денег.

### `switch` expressions (C# 8+)

C# 8 ввёл **`switch` expressions** — компактная expression-oriented форма, **возвращающая value**:

```csharp
string mood = day switch
{
    DayOfWeek.Saturday or DayOfWeek.Sunday => "Relax",
    DayOfWeek.Monday => "Coffee",
    _ => "Work"
};
```

Синтаксис: **`value switch { patterns => results }`**. **`_`** — discard pattern («всё остальное», как `default`). **`or`** объединяет patterns. Каждая arm даёт compatible type; compiler предупреждает о пропущенных case для **enum** без `_`.

Сравни imperative vs expression style:

```csharp
string tier;
switch (score)
{
    case >= 90: tier = "Gold"; break;
    case >= 70: tier = "Silver"; break;
    default:    tier = "Bronze"; break;
}

string tier = score switch
{
    >= 90 => "Gold",
    >= 70 => "Silver",
    _     => "Bronze"
};
```

Expression form снижает баги с uninitialized variables и упрощает reasoning про **exhaustiveness**.

### Pattern matching в `switch`

Modern C# — **pattern matching**, не только constant equality:

```csharp
object input = "hello";

string description = input switch
{
    null       => "Missing input",
    string s   => $"Text with length {s.Length}",
    int n      => $"Number {n}",
    bool b     => b ? "True" : "False",
    _          => "Unknown type"
};
```

**Relational patterns** (`>= 90`) для numeric switched value. **Type patterns** (`string s`) проверяют runtime type и bind variable. Заменяет verbose `if (input is string s)` chains. В ASP.NET позже увидишь pattern matching на records и HTTP results — mental model начинается здесь.

## Under the Hood

JIT компилирует `if` и `switch` в **branch instructions** на уровне IL — conditional jumps по flags после comparison. **Branch predictor** CPU делает предсказуемые ветки дёшевыми; непредсказуемые в hot loops могут иметь значение, но в business logic побеждает readability.

Для **`switch` на dense integer/enum** compiler может emit **jump table** — O(1) dispatch. Для **`string switch`** — hash-based strategy (по духу `Dictionary`). Micro-optimize редко нужен; это объясняет, почему switch на enum с пятьюдесятью values нормален.

**Short-circuit** — **defined behavior**, не hint оптимизатору. Reviewers полагаются на него для null checks. Не меняй порядок `&&` operands, если left guards right.

## Syntax / Commands / API

| Construct | Purpose |
|-----------|---------|
| `if (condition) { }` | Block при true |
| `else if (condition) { }` | Альтернативная ветка |
| `else { }` | Default |
| `condition ? a : b` | Ternary value choice |
| `switch (x) { case ...: break; }` | Classic multi-way |
| `x switch { pattern => value }` | Switch expression |
| `_` | Discard / default pattern |
| `case A or B:` | Combined patterns (C# 9+) |

## Examples

### Пример 1: валидация input в console app

```csharp
Console.Write("Enter age: ");
string? text = Console.ReadLine();

if (!int.TryParse(text, out int age))
{
    Console.WriteLine("Invalid number.");
    return;
}

if (age < 0 || age > 120)
{
    Console.WriteLine("Age out of realistic range.");
}
else if (age < 18)
{
    Console.WriteLine("Minor");
}
else
{
    Console.WriteLine("Adult");
}
```

`TryParse` избегает exceptions для ожидаемого bad input — production-friendly pattern. **`return`** выходит из `Main` при ошибке parse; подробно в `1.1.11`.

### Пример 2: mapping status codes через switch expression

```csharp
int statusCode = 404;

string message = statusCode switch
{
    200 => "OK",
    201 => "Created",
    400 => "Bad Request",
    401 => "Unauthorized",
    404 => "Not Found",
    >= 500 and < 600 => "Server Error",
    _ => "Unknown"
};

Console.WriteLine($"{statusCode}: {message}");
```

Похоже на mapping API results в ASP.NET — decision table видна сразу.

## Common Mistakes & Anti-patterns

**Dangling else** редок при braces, но nested `if` без скобок путает ассоциацию. В team code всегда brace nested branches.

**Switch на doubles/floats** для equality хрупок. Для денег — integer cents или thresholds.

**Забытый `break`** в classic switch — compile error в C# (в отличие от C). Намеренная safety.

**Каждый `if` в ternary** бьёт readability. Ternary — для simple value selection.

**Duplicate case labels** или unreachable arms после `_` — warnings/errors компилятора; слушай их.

## Production & Real-World Notes

В code review предпочитай **`switch` на enums** magic numbers. Enum self-documents cases; compiler предупредит о новом member без update switch (exhaustive analysis или намеренный omit `_`).

Guard clauses — early `if` returns — flatten nested logic в service methods. Свяжешь с `return` в `1.1.11` и methods в `1.1.12`.

Configuration-driven behavior иногда заменяют giant switches на dictionaries или strategy — но понимание `switch` обязательно: те patterns компилируются в ту же decision logic.

## Comparison / Trade-offs

| Situation | Prefer |
|-----------|--------|
| Два исхода, simple value | Ternary |
| Boolean rules с ranges | `if / else if` |
| Одна variable vs many constants | Classic или expression `switch` |
| Type dispatch, small scale | Pattern switch expression |
| Сложные rules, много variables | Extract method; polymorphism позже |

Classic switch поддерживает **statements** в arm; switch expressions требуют **expression-oriented** arms. Несколько statements в arm — classic switch или `if/else`.

## Key Takeaways

- Control flow превращает bool **answers** в **actions** или **values**.
- C# требует explicit `bool` conditions — нет implicit truthiness.
- `&&` и `||` short-circuit; guard checks в правильном порядке.
- Classic `switch` использует `break`; C# блокирует accidental fall-through.
- Switch expressions — чистый assign и modern pattern matching.
- Выбирай структуру, которая лучше показывает decision table следующему читателю.
