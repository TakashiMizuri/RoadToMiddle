# Lock в async — SemaphoreSlim vs lock

> Roadmap: `1.4.33` · Node: `1.4` — C# async/await · Depth: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Объяснить, почему **`lock`** нельзя держать через **`await`**.
- Использовать **`SemaphoreSlim.WaitAsync`** для async mutual exclusion.
- Выбирать между mutex **`(1,1)`** и **throttling `(n,n)`**.
- Избегать **deadlock** и **thread pool starvation** от sync-over-async в lock.
- Понимать, когда **`lock`** всё ещё уместен (короткий sync).

---

## Why This Matters

`lock (_gate) { await RefreshAsync(); }` — **CS1996** или unsafe обходы. `lock` + **`.Result`** — deadlock под нагрузкой (`1.4.14`). Async-серверу нужны замки, **отпускающие поток** на время ожидания. **`WaitAsync`** — стандарт. Middle знает, когда **короткий sync lock** OK, а когда **semaphore обязателен**.

---

## Core Concepts

### Почему lock и await несовместимы

**`lock`** = **`Monitor.Enter/Exit`** на **текущем потоке**. **`await`** отдаёт поток в pool — критическая секция логически не завершена, другой код может войти в lock. C# **запрещает** `await` в `lock`.

### SemaphoreSlim как async mutex

```csharp
await _mutex.WaitAsync(ct);
try { await RefreshFromDbAsync(ct); }
finally { _mutex.Release(); }
```

**`WaitAsync`** — continuation вместо blocked thread. **`Release`** всегда в **`finally`**.

**`(1,1)`** — mutex **без re-entrancy** (в отличие от `lock` на том же потоке). Повторный `WaitAsync` без Release → self-deadlock.

### Когда lock OK

Нет **await** — чисто in-memory:

```csharp
lock (_cacheLock) { _cache[key] = value; }
```

Коротко. Без I/O внутри.

### Throttling

**`SemaphoreSlim(10, 10)`** — до 10 параллельных операций (`1.4.18`), не то же самое, что exclusive lock.

---

## Under the Hood

**`lock`** re-entrant per thread; async continuation может быть на **другом** потоке — семантика ломается.

**`.Result` в lock** — pool starvation (`1.4.10`).

**`ReaderWriterLockSlim`** — sync; для async read-heavy — immutable snapshots или спец. библиотеки.

## Common Mistakes & Anti-patterns

`WaitAsync` без **`finally Release`**.

Вложенный Wait на том же `(1,1)`.

**Global semaphore** на всё — лишняя сериализация; per-key locks с cleanup.

---

## Quick Reference

- Не **`await` в lock**.
- **`WaitAsync` + `finally Release`**.
- **`lock`** — короткий sync.
- Не **`.Result`** в lock.
- Multi-instance — **distributed** lock.

---

## Key Takeaways

- **`lock`** thread-bound — не через suspension.
- **`SemaphoreSlim.WaitAsync`** — async mutex.
- Cancellation в WaitAsync.
- Sync-over-async в lock — deadlock/starvation.
- Immutable snapshots уменьшают need в locks.
