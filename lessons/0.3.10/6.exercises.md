# Exercises: REST Principles

> Complete after lectures and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Resource or RPC?
**Goal:** URI design sense.

Label each as **resource-oriented** or **RPC-style**:

1. `GET /customers/10`
2. `POST /api/runReport`
3. `DELETE /sessions/abc`
4. `POST /payments/processPayment`
5. `PUT /settings/theme`

**Success criteria:** 1,3,5 resource-oriented; 2,4 RPC — with brief why.

---

### E2. Stateless check
**Goal:** Identify violations.

Which designs violate REST **statelessness**?

- A) JWT in Authorization on every API call
- B) Server remembers filter preferences in RAM keyed only by TCP connection
- C) Session cookie + Redis session store looked up each request
- D) Shopping cart only in one server's memory, no cart id in client

**Success criteria:** B and D violate; A and C acceptable in practical REST (auth context in each request).

## Medium

### M1. Design mini API
**Goal:** Level 2 resource map.

For domain **library books**, write:

- 5 endpoints (method + path)
- Expected success status for each (from `0.3.6`)
- One error case with status (404, 409, or 422)

No implementation — design only.

**Success criteria:** Nouns in paths; methods match semantics; statuses not all 200.

---

### M2. Richardson level
**Goal:** Classify APIs.

Describe three fictional APIs (2–3 sentences each) at levels **0**, **2**, and **3**. Include one feature proving the level.

**Success criteria:** Level 3 mentions hypermedia links driving client; level 0 single POST tunnel.

## Hard (stretch)

### H1. Conscious compromise
**Goal:** Evaluation not dogma.

Pick one scenario where strict REST resource model is awkward:

- bulk import 10k rows
- `cancel order` action
- full-text search

For each, propose **REST-leaning** and **pragmatic** endpoint design. Write 2–3 sentences trade-offs each.

**Success criteria:** Acknowledges level 2 ideals vs shipping velocity.

---

### H2. Connect the node
**Goal:** Synthesize `0.3.x`.

Write one paragraph linking: methods (`0.3.4`), idempotency (`0.3.5`), status codes (`0.3.6`), caching headers (`0.3.7`), HTTPS (`0.3.8`), cookies (`0.3.9`) into coherent RESTful GET on public resource.

**Success criteria:** Mentions safe GET, 200/304, ETag, TLS for integrity/confidentiality.
