# Exercises: HTTP Message Structure

> Complete after lectures and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Label message parts
**Goal:** Recognize structure in raw traffic.

Given this response, label start line, headers, body:

```
HTTP/1.1 404 Not Found
Content-Type: application/json
Content-Length: 25

{"error":"not found"}
```

**Success criteria:** Three regions identified; body exactly 25 bytes if counted.

---

### E2. Missing piece
**Goal:** Spot invalid raw request.

What's wrong with:

```
GET /test HTTP/1.1
Content-Type: application/json
{"a":1}
```

**Success criteria:** Mentions missing Host (HTTP/1.1), missing blank line, missing Content-Length for body.

---

## Medium

### M1. Craft raw GET
**Goal:** Write valid HTTP/1.1 by hand.

Write full raw HTTP/1.1 request (with `\r\n` noted) for `GET /get` to `httpbin.org` with header `Accept: application/json` and `Connection: close`.

Test with curl compare or nc if you have lab access.

**Success criteria:** Valid start line, Host header, blank line, no body.

---

### M2. Content-Length calculation
**Goal:** Understand byte counting.

Body string (UTF-8): `{"ok":true}` — what Content-Length? What if emoji `{"ok":"✓"}` — does length change?

**Success criteria:** Correct byte counts, not character counts; UTF-8 multi-byte awareness.

---

## Hard (stretch)

### H1. Chunked vs length design
**Goal:** Choose encoding in API design.

Scenario: ASP.NET Core streams 1 GB export CSV to client without buffering full file in RAM.

Explain in 6–10 sentences: chunked vs Content-Length, impact on client, Kestrel streaming, proxy buffering caveat.

**Success criteria:** Mentions unknown length → chunked or HTTP/2 DATA frames; memory benefit; possible proxy buffer issue.
