## Answers

### 1. [Recall]
**Answer:** Предпочитай собирать функциональность через владение и делегирование другим объектам/интерфейсам, а не через наследование реализации базового класса.
**Why:** Наследование связывает подкласс с деталями и эволюцией предка; композиция делает зависимости явными и заменяемыми.

### 2. [Recall]
**Answer:** Изменения в базовом классе, которые кажутся безопасными авторам base, ломают подклассы, полагавшиеся на старое поведение `virtual`/`protected` hooks.
**Why:** Компилятор не проверяет семантические контракты overrides в consumer-сборках.

### 3. [Recall]
**Answer:** **Is-a** — подтип логически *является* родителем и заменяем везде; **has-a** — объект *содержит* или *использует* другую capability.
**Common mistake:** Путать code reuse с истинным is-a (`Stack : List`).

### 4. [Understanding]
**Answer:** Подклассы **override** `protected virtual` и встраиваются в алгоритм base; изменение base меняет контекст вызова override. В composed-классе `private` helper — деталь реализации; снаружи виден только публичный контракт, подмена идёт через **injected interface**, не через hidden override.
**Why:** Virtual hooks — неявный контракт между base и всеми descendants.

### 5. [Understanding]
**Answer:** `Resize(Rectangle)` ожидает независимые `Width` и `Height`. `Square` синхронизирует стороны — нарушает ожидаемый invariant area = width × height после независимой установки.
**Why:** LSP требует, чтобы подтип не удивлял код, написанный для базового типа.

### 6. [Understanding]
**Answer:** Каждый middleware — отдельный, опциональный слой с единственной зависимостью `RequestDelegate _next`; порядок настраивается в composition root без subclassing; нет fragile base с `protected virtual Invoke`.
**Why:** Pipeline — классический has-a chain, не is-a tree.

### 7. [Application]
**Answer:** Извлечь `Validate` и/или `Write` в интерфейсы (`IMessageValidator`, `IMessageSink`) и внедрить в concrete `Notifier` без наследования.
**Why:** Первый шаг — отделить вариативные срезы от фиксированного алгоритма `Send`.

### 8. [Application]
**Answer:** `Bird` с полем `IFlyBehavior` (`Flying`, `Flightless`); метод `Move()` делегирует `_flight.Fly()`; без `Penguin : Bird` с исключением.
**Why:** Flight — capability, не типовая идентичность птицы.

### 9. [Application]
**Answer:** **Decorator:** `FraudCheckingValidator : IOrderValidator` с `private readonly IOrderValidator _inner`; сначала fraud-check, затем `_inner.Validate(order)`.
**Why:** Open/closed без изменения исходного validator class.

### 10. [Analysis]
**Answer:** Неверно категорично: наследование нужно для framework extension (`ControllerBase`, `Exception`), Template Method с shallow hierarchy, истинного is-a с LSP.
**Why:** Dogma «никогда» игнорирует legitimate substitutability.

### 11. [Analysis]
**Answer:** **Deep inheritance for cross-cutting concerns** — audit и cache накладываются слоями subclassing; хрупко, сложно тестировать, нельзя включить cache без audit.
**Why:** Cross-cutting лучше composition/decorators/DI.

### 12. [Understanding]
**Answer:** **Inheritance** сливает implementation и vtable; **delegation** держит отдельный объект и пересылает вызовы — поведение меняется заменой collaborator.
**Why:** Delegation не обещает substitutability всего родительского API.

### 13. [Application]
**Answer:** Когда алгоритм **фиксирован** (последовательность шагов), варианты — только отдельные шаги, команда владеет всей иерархией, глубина мала (1–2 уровня).
**Why:** Template Method контролирует skeleton; риск LSP ниже при узком `protected` surface.

### 14. [Analysis]
**Answer:** (1) Email не **is-a** SMS — неверная доменная модель. (2) Наследует лишний API `SmsSender` (номера, провайдеры SMS).
**Why:** Reuse через inheritance импортирует нерелевантный контракт.

### 15. [Application]
**Answer:** Примеры: подкласс использует <30% API родителя; override бросает `NotSupportedException`; иерархия >2 уровней для features; нужны optional layers (cache, audit) combinable по-разному.
**Why:** Каждый признак указывает на has-a, не is-a.
