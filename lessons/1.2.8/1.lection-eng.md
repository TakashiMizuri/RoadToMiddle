# Abstract Classes vs Interfaces — Decision Guide

> Roadmap: `1.2.8` · Node: `1.2` — C# OOP · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- State the structural differences between **`abstract class`** and **`interface`** in modern C#.
- Choose **abstract base class** when shared state, constructors, or template methods matter.
- Choose **interface** when defining cross-cutting contracts and supporting multiple inheritance of type.
- Combine **class inheritance + interface implementation** in layered designs (BCL and ASP.NET patterns).
- Explain why C# allows **multiple interfaces** but **single base class**.
- Apply a practical **decision checklist** for new domain abstractions in production code.

---

## Why This Matters

Lessons `1.2.6`–`1.2.7` built inheritance and polymorphism on **classes**. Real architectures split two questions that beginners merge: **"What behavior must every implementation expose?"** (contract) and **"What shared implementation or state can we centralize?"** (reuse). **`interface`** answers the first; **`abstract class`** often answers the second.

Getting the choice wrong produces **`IBlob`** interfaces with forty methods copied into every implementer, or deep **`MegaBaseController`** hierarchies that fight composition. ASP.NET Core itself **`Controller : ControllerBase`** (shared implementation) plus **`IActionFilter`**, **`IDisposable`**, **`IAsyncActionFilter`** (cross-cutting contracts). EF entities inherit **`DbContext`** patterns through composition, not one entity base for everything — but **`IdentityUser`** is a concrete base with optional interfaces.

Middle developers author abstractions consumed by teams for years. **`IRepository<T>`**, **`IEmailSender`**, **`IPipelineBehavior`** — interfaces. **`Stream`**, **`DbContext`**, **`BackgroundService`** — abstract bases with shared machinery. The decision guide in this lesson is what senior reviewers apply in design meetings.

## Core Concepts

### What an Abstract Class Is

An **`abstract class`** is a **class** that cannot be instantiated directly. It may contain:

- **Abstract members** — no implementation; concrete derived classes must override.
- **Virtual members** — default implementation overridable like any class.
- **Concrete members** — full methods with bodies.
- **Fields, constructors, access modifiers** — everything a normal class has (within rules).

```csharp
public abstract class RepositoryBase<T> where T : class
{
    protected readonly DbContext Context;

    protected RepositoryBase(DbContext context)
    {
        Context = context ?? throw new ArgumentNullException(nameof(context));
    }

    public abstract Task<T?> FindByIdAsync(Guid id, CancellationToken ct);

    public virtual IQueryable<T> Query() => Context.Set<T>();
}
```

**`RepositoryBase`** centralizes **`DbContext`** storage and a default **`Query()`**. Derived **`CustomerRepository`** must implement **`FindByIdAsync`** but inherits constructor chain and optional **`Query()`** override. **Single inheritance**: **`CustomerRepository : RepositoryBase<Customer>`** — cannot also extend another class.

Abstract classes express **"is-a"** with **partially defined** identity. They are ideal when every subtype shares **initialization**, **protected helpers**, or ** invariant enforcement** in one place.

### What an Interface Is

An **`interface`** defines a **contract** — member signatures that implementing types must provide. Before C# 8, interfaces were purely abstract (except static rules in later versions). Modern interfaces may include **default interface methods** (`1.2.10`), but the primary role remains **capability declaration** without forcing a common base class.

```csharp
public interface IEmailSender
{
    Task SendAsync(string to, string subject, string body, CancellationToken ct = default);
}

public interface ITrackable
{
    DateTime CreatedAt { get; }
    DateTime? UpdatedAt { get; }
}
```

A **`OrderService`** can implement **`IEmailSender`** only in test doubles; production uses **`SmtpEmailSender : IEmailSender`**. **`Order`** entity might implement **`ITrackable`** while **`Product`** implements **`ITrackable`** too — **no shared class required**.

C# allows **multiple interface implementation**:

```csharp
public sealed class Invoice : Entity, IPrintable, IAuditable
{
    // ...
}
```

**`Invoice`** has one base **`Entity`** (if chosen) plus two interface contracts. Interfaces do not declare instance fields (until static/abstract static members in C# 11 — advanced); state lives in the implementing class.

### Side-by-Side Comparison

Both **`abstract class`** and **`interface`** support polymorphic references and **`abstract`/virtual-style** extension. The differences that drive decisions:

| Aspect | Abstract class | Interface |
|--------|----------------|-----------|
| Instantiation | No | No (for interface itself) |
| Inheritance limit | One base class | Many interfaces |
| Fields | Yes (instance) | No instance fields in classic model |
| Constructors | Yes | No instance constructors |
| Access modifiers on members | full range | public by default; private/protected members C# 8+ |
| Shared implementation | Yes, extensive | Default methods (C# 8+, limited) |
| Versioning | Breaking changes hurt derived | Default methods help additive change (`1.2.10`) |
| Semantic | is-a with shared DNA | can-do / has-capability |

Neither replaces the other. **`Stream`** is abstract class with shared **`ReadByte`** loops; **`IAsyncDisposable`** is interface for async cleanup capability across unrelated types.

### When to Prefer Abstract Class

Prefer **`abstract class`** when:

1. **Shared state** — all derivatives need the same fields (**`_context`**, **`_logger`**, **`Id`**).
2. **Constructor enforcement** — base ctor establishes invariants all children must pass through.
3. **Template Method** — non-virtual orchestration calling virtual hooks (`1.2.7`).
4. **Protected helpers** — **`protected`** methods reused by subclasses without public API noise.
5. **Single lineage** — domain truly is one hierarchy (**`Exception`**, **`Attribute`**, **`DbConnection`**).

Example: **`BackgroundService`** in **`Microsoft.Extensions.Hosting`** — abstract base with **`ExecuteAsync`** abstract, host integration concrete in base.

### When to Prefer Interface

Prefer **`interface`** when:

1. **Unrelated types** share capability — **`IComparable`**, **`IFormattable`**, **`ILogger<T>`**.
2. **Multiple contracts** on one class — repository + disposable + cacheable.
3. **Testing and DI** — mock/stub **`IOrderRepository`** without inheriting production base.
4. **Avoid fragile base class** — no forced constructor chain from library base.
5. **Consumer owns hierarchy** — library defines **`IHandler`**, app defines base **`HandlerBase`** if needed.

Example: ASP.NET **`IMiddleware`**, MediatR **`IRequestHandler<TRequest,TResponse>`** — apps implement without extending framework class.

### Combining Both — Layered Pattern

Production pattern:

```csharp
public abstract class HandlerBase<TRequest> : IRequestHandler<TRequest, Unit>
    where TRequest : IRequest<Unit>
{
    protected ILogger Logger { get; }

    protected HandlerBase(ILogger logger) => Logger = logger;

    public async Task<Unit> Handle(TRequest request, CancellationToken ct)
    {
        Logger.LogInformation("Handling {Name}", typeof(TRequest).Name);
        await HandleCore(request, ct);
        return Unit.Value;
    }

    protected abstract Task HandleCore(TRequest request, CancellationToken ct);
}
```

App handlers **`CreateOrderHandler : HandlerBase<CreateOrderCommand>`** — team-owned abstract base. Framework defines **`IRequestHandler<,>`** interface; team adds **optional** internal abstract base for logging/error wrapping. **Interface for DI registration; abstract base for DRY** — not either/or.

### Abstract Class vs Concrete Base

Sometimes a **non-abstract** base with **`virtual`** defaults suffices — **`ControllerBase`**. Use **`abstract class`** when **instantiating the base is always wrong** — **`Shape`**, **`RepositoryBase`**. **`abstract`** documents intent and blocks **`new Shape()`** at compile time.

## Under the Hood

### CLR Type Layout

An **`abstract class`** instance is a normal object with method table and fields. **`abstract`** methods have **no method body RVA** until a concrete override supplies IL.

An **interface** is a **type** with metadata table **`InterfaceImpl`** on implementing class listing which class methods satisfy which interface slots. Calling **`((IEnumerable<int>)list).GetEnumerator()`** may use **interface map** entry distinct from class vtable path — same implementation object, different dispatch stub if explicit implementation (`1.2.9`).

Multiple interfaces → multiple entries in **`InterfaceImpl`**; no diamond problem because **classes implement interfaces separately**, not multiply inherit implementation from two class bases.

### Default Interface Methods (Preview)

C# 8+ allows bodies on interfaces (`1.2.10`). That blurs the "interfaces have no implementation" line but **does not add instance fields** to implementers automatically. CLR reuses most specific override: class override beats default interface method when conflict rules apply.

### Abstract vs Interface in API Evolution

Adding **`abstract`** method to released abstract class **breaks all derived** until they implement. Adding member to **interface** without default **breaks all implementers**. Default interface methods mitigate interface breakage; abstract class still prefers **`virtual`** default over new **`abstract`** when extending shipped bases.

## Examples

### Example 1: Interface for DI

```csharp
public interface IClock
{
    DateTime UtcNow { get; }
}

public sealed class SystemClock : IClock
{
    public DateTime UtcNow => DateTime.UtcNow;
}

// Test double
public sealed class FixedClock : IClock
{
    public DateTime UtcNow { get; init; }
}
```

No shared base needed; **`FixedClock`** and **`SystemClock`** unrelated except contract.

### Example 2: Abstract for Shared Wiring

```csharp
public abstract class ApiControllerBase : ControllerBase
{
    protected IMediator Mediator { get; }

    protected ApiControllerBase(IMediator mediator) =>
        Mediator = mediator;
}
```

Controllers inherit **mediator field + ctor** once; **`ControllerBase`** already provides ASP.NET surface.

### Example 3: Decision Walkthrough

**Requirement:** Plugin **`IExportFormat`** (**Pdf**, **Csv**, **Xlsx**) — no shared state, host loads by DI.

→ **`interface IExportFormat`** with **`Export(DataSet data, Stream output)`**.

**Requirement:** **`FileProcessor`** pipeline with shared retry/logging in base, variants only parse bytes.

→ **`abstract class FileProcessor`** with **`Process(Stream)`** template and **`abstract Parse`** hook.

## Common Mistakes & Anti-patterns

**God interface** — **`IDataAccess`** with CRUD for every entity. Split by capability or use generics.

**Abstract class for only one method** — likely **`interface`** unless template/state follows.

**Inheritance for code reuse only** — **`1.2.11`**: **`LoggedList : List<T>`** wrong motivation; compose **`ILogger`**.

**Implementing interface explicitly without reason** — **`1.2.9`**; default implicit unless name collision.

**Choosing abstract base to "prevent instantiation" of utility class** — use **`static class`** or private ctor instead.

## Production & Real-World Notes

Clean Architecture inner rings define **interfaces**; infrastructure implements them. Domain entities rarely inherit heavy bases except **`Entity<TId>`** patterns in some templates — teams debate **`record`** vs base vs pure interface **`IEntity`**.

Microsoft guidance evolves: favor **small interfaces** (Interface Segregation Principle). **`IReadOnlyList<T>`** vs fat **`IList<T>`** when consumer only reads.

## Comparison / Trade-offs

**Abstract class vs interface vs composition.** Abstract class shares **implementation lineage**; interface shares **type contract**; composition shares **behavior via delegation** without is-a. **`1.2.11`** completes the triangle.

**Multiple interfaces vs single abstract base.** Interfaces compose capabilities; abstract base unifies one family. **`class Bird : Animal, IFlyable, IWalkable`** — common pattern.

## Quick Reference

| Need | Choose |
|------|--------|
| Shared fields/ctors | Abstract class |
| Template Method base | Abstract class |
| Multiple unrelated implementers | Interface |
| DI/mock boundary | Interface |
| Framework extension point (single tree) | Abstract class |
| Add capability to existing types | Interface |

**Decision question:** *"Is there shared state or constructor wiring every subtype must use?"* → Yes: consider abstract class. *"Is it only a role/capability?"* → Interface.

---

## Key Takeaways

- **Abstract class** = single-inheritance base with partial implementation and optional abstract members.
- **Interface** = contract; multiple per class; ideal for capabilities and DI.
- Both enable polymorphism; CLR representation differs (vtable vs interface map).
- Production uses **both together** — interface for registration, team abstract base for DRY.
- Prefer **small interfaces**; avoid deep abstract hierarchies.
- **`abstract`** keyword on class blocks meaningless instantiation.
