# Accessibility: ARIA Roles, Keyboard Navigation, Focus

> Roadmap: `0.5.11` · Node: `0.5` — HTML · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- State the **first rule of ARIA**: use native HTML elements before adding roles and attributes.
- Distinguish **implicit roles** from **explicit** `role` attributes and know when explicit roles help or harm.
- Implement **keyboard-operable** custom widgets using tabindex, roving tabindex, and key handlers aligned with WAI-ARIA Authoring Practices.
- Manage **focus order**, **focus traps** in modals, and **focus restoration** on close.
- Describe how **focus** moves between DOM elements and how `:focus-visible` differs from `:focus`.
- Connect ARIA attributes (`aria-label`, `aria-labelledby`, `aria-expanded`, `aria-live`) to the accessibility tree built from DOM nodes (`0.5.10`).

---

## Why This Matters

Accessibility is not a polish layer applied after the UI "works" for mouse users. It is the same UI experienced through **keyboard**, **screen reader**, **voice control**, and **switch devices**. In many markets, inaccessible software excludes users and creates legal exposure; in all markets, it signals engineering maturity. You already structure tables with `scope` (`0.5.9`) and understand that the browser exposes an accessibility tree derived from DOM — this lesson covers what happens when native elements are insufficient and how **focus** — the cursor of keyboard and assistive tech — must be deliberate.

React component libraries often abstract away `<button>` into `<div role="button">` without keyboard support. ASP.NET Core Razor pages may emit custom dropdowns missing `aria-expanded`. Middle developers are expected to catch these in review, fix focus traps in modals, and choose **semantic HTML first**, ARIA second. Keyboard navigation is also power-user UX: engineers live in keyboards; broken Tab order frustrates everyone.

## Core Concepts

### Semantic HTML First, ARIA Second

**WAI-ARIA** (Accessible Rich Internet Applications) supplements HTML when native elements cannot express role, state, or properties. The **first rule of ARIA**: if you can use a native HTML element with built-in semantics and behavior, **do not** change it with ARIA. A `<button>` gets `role="button"`, `tabindex` focusability, Enter/Space activation, and disabled state for free. A `<div role="button" tabindex="0">` requires you to implement keyboard activation, disabled semantics, and focus styling — easy to get wrong.

Use ARIA when building **composite widgets** not mapped 1:1 to a single tag — tabs, combobox, tree view — or when **augmenting** native controls (`aria-describedby` linking error text to an input). **Do not** use ARIA to fix bad markup when a native element exists.

### Roles: Implicit and Explicit

Every element node in the accessibility tree carries a **role** — `button`, `link`, `heading`, `table`, `navigation`, etc. Native HTML tags assign **implicit roles** via the HTML Accessibility API Mapping. Explicit **`role`** attribute overrides default in limited cases — misuse creates **role conflicts** (e.g. `role="heading"` on `<div>` without `aria-level`).

Roles divide into:

- **Landmark roles** — `banner`, `navigation`, `main`, `complementary`, `contentinfo`; prefer `<header>`, `<nav>`, `<main>`, `<aside>`, `<footer>` when structure matches.
- **Widget roles** — `button`, `checkbox`, `tab`, `tabpanel`, `dialog`.
- **Document structure** — `heading`, `list`, `listitem`, `article`.

Screen readers use roles to announce element type: "Edit, Email" for `<input type="email">`. Wrong role — "Clickable" on a div — misleads.

### Keyboard Navigation Fundamentals

**Tab** moves focus among **tabbable** elements: links, buttons, form fields, `tabindex="0"`. **Shift+Tab** reverses. **`tabindex="-1"`** removes from tab order but allows **programmatic** `.focus()` — useful for moving focus into a opened dialog heading.

Default tab order follows **DOM order** — visual CSS reorder (`order`, `grid`) does not change tab order unless you also reorder DOM or misuse positive `tabindex`. **`tabindex` > 0** creates a custom tab order that fights maintainers — avoid.

Interactive widgets need **keyboard patterns** from [ARIA APG](https://www.w3.org/WAI/ARIA/apg/):

- **Button** — Enter and Space activate.
- **Tabs** — Arrow keys move between tabs; Tab moves into panel.
- **Menu** — Arrow keys, Escape closes, typeahead optional.
- **Dialog** — Focus trapped inside; Escape closes; restore focus to trigger.

Without key handlers, `role="button"` is a accessibility failure.

### Focus Management

**Focus** is the active element receiving keyboard events — one per document (conceptually). **`element.focus()`** moves it programmatically. Opening a modal: move focus to first focusable or title inside; **trap** Tab within modal (`aria-modal="true"` helps AT); on close, **restore** focus to the button that opened it — users lose context otherwise.

**Roving tabindex** — in toolbar or tablist, only one item has `tabindex="0"`, siblings `-1`; arrow keys move focus and swap tabindex values. Keeps Tab count sane in complex widgets.

Skip links (`0.5.12`) jump focus to `#main` — same focus API, different UX purpose.

### Essential ARIA Attributes

| Attribute | Purpose |
|-----------|---------|
| `aria-label` | Accessible name when visible text absent |
| `aria-labelledby` | Points to element id(s) providing name |
| `aria-describedby` | Extra description (hint, error) |
| `aria-expanded` | Collapsible/disclosure state |
| `aria-selected` | Selected item in set |
| `aria-hidden="true"` | Excludes subtree from AT (not focusable children!) |
| `aria-live` | Announces dynamic updates (`polite`, `assertive`) |
| `aria-disabled` | Disabled state when native disabled unavailable |

**Accessible name computation** merges visible text, `aria-label`, `aria-labelledby`, `alt` on images, `<label for>`. Duplicating or contradicting sources confuses users.

**`aria-live` regions** — toast notifications, form errors after submit — screen readers poll or get events when content changes.

## Under the Hood

### From DOM to Accessibility Tree

After layout, the browser builds a parallel **accessibility tree** (AX tree): nodes with role, name, state, value, relations (` labelledby`, `controls`, `describedby`). Not every DOM node appears — decorative icons with `alt=""` may be ignored; text nodes merge into parent names.

Platform bridges expose AX tree to **NVDA**, **JAWS**, **VoiceOver**, **TalkBack** via OS APIs (UI Automation on Windows, NSAccessibility on macOS). Changes batch on **mutation**; heavy DOM churn causes announcement lag. **`aria-hidden="true"`** on an ancestor hides descendants from tree — but **never** hide focusable content; focused element must remain accessible.

### Focus Ring and :focus-visible

Browser default **focus outline** shows which element is focused. Designers often `outline: none` without replacement — WCAG **2.4.7 Focus Visible** failure. **`:focus-visible`** applies outline when keyboard focus expected, not always on mouse click (browser heuristics). Pair `:focus-visible` with visible custom ring — `0.5.12` expands contrast requirements.

### tabindex and Sequential Focus Navigation

HTML defines **focusable** and **tabindex** sorting: positive tabindex elements come first (in numeric order), then tabindex 0/undefined in DOM order, tabindex -1 skipped. This algorithm explains why positive tabindex harms predictability.

In shadow DOM, focus can move across shadow boundaries; **delegatesFocus** on shadow host affects behavior. Web Components authors must forward focus intentionally.

### ARIA in React and SSR

Client-only `aria-expanded={open}` must match server render for hydration. Conditional roles must reflect real keyboard behavior on first paint. **ESLint jsx-a11y** catches some missing `alt`, `role`, click-without-keyboard — not exhaustive.

## Syntax / Commands / API

```html
<button type="button" aria-expanded="false" aria-controls="menu" id="menu-btn">
  Menu
</button>
<ul id="menu" role="menu" hidden>...</ul>

<div role="dialog" aria-modal="true" aria-labelledby="dlg-title">
  <h2 id="dlg-title">Confirm</h2>
  ...
</div>

<div role="status" aria-live="polite" id="toast"></div>
```

```javascript
menuBtn.addEventListener('click', () => {
  const open = menuBtn.getAttribute('aria-expanded') === 'true';
  menuBtn.setAttribute('aria-expanded', String(!open));
  if (!open) firstMenuItem.focus();
});

dialog.querySelector('.close').addEventListener('click', () => {
  dialog.close?.(); // or hide
  menuBtn.focus(); // restore
});
```

Keyboard on custom button:

```javascript
el.addEventListener('keydown', (e) => {
  if (e.key === 'Enter' || e.key === ' ') {
    e.preventDefault();
    el.click();
  }
});
```

## Examples

**Example 1 — Native wins.** Dropdown built with `<select>` and `<option>` — full keyboard and AT support. Custom div dropdown needs `role="listbox"`, option roles, arrow keys, `aria-activedescendant` or roving tabindex — ten times the code.

**Example 2 — Disclosure.** `<button aria-expanded="false" aria-controls="panel">` toggles panel visibility; update `aria-expanded` on click; panel `id` matches `aria-controls`.

**Example 3 — Live region.** After AJAX form error, set `errorRegion.textContent = 'Invalid email'`; `aria-live="polite"` on region announces without moving focus — do not steal focus for non-critical errors.

## Common Mistakes & Anti-patterns

**Click-only `<div onClick>`** — no keyboard, no role. Fix: `<button>` or full keyboard + role pattern.

**Positive tabindex** — jumps tab order. Fix: DOM order + CSS layout.

**aria-label overriding visible text** incorrectly — screen reader hears wrong label. Prefer visible text + `aria-labelledby` if needed.

**Focus trap without Escape** — users stuck in modal. Provide close + restore focus.

**aria-hidden on focused element** — violates accessibility; focused nodes must not be hidden from AT.

**Redundant ARIA** — `role="button"` on `<button>`. Noise, occasionally harmful.

## Production & Real-World Notes

Run **axe**, **Lighthouse accessibility**, and manual **keyboard-only** pass before release. Test **NVDA** or VoiceOver on one critical flow. Design system documents keyboard specs per component.

Legal: WCAG 2.2 Level AA common contract — **2.1.1 Keyboard**, **4.1.2 Name, Role, Value**.

SPAs route changes should **move focus to `<h1>`** or use `aria-live` to announce new page title — otherwise screen reader users miss navigation.

ASP.NET tag helpers: ensure `asp-for` generates `<label>` association; validation summaries need `role="alert"` or live region thoughtfully.

## Comparison / Trade-offs

**Native vs custom widget** — Native cheaper, consistent; custom only when design/function needs exceed native.

**Moving focus vs aria-live** — Critical errors may need focus to error summary; inline field errors often `aria-describedby` without focus steal.

**Fully focusable modal vs non-modal dialog** — Modal traps focus; non-modal (`role="dialog"` without modal) allows interaction outside — rare in web apps.

**ARIA vs automated tests** — axe catches missing names; cannot prove good UX — manual AT still required.

## Quick Reference

1. Native HTML first.
2. Tab order = DOM order (avoid tabindex > 0).
3. Every interactive control — keyboard operable.
4. Modal: trap focus, Escape, restore on close.
5. Name via visible text, label, or aria-label.
6. Dynamic updates — aria-live or focus management.

| Key | Typical use |
|-----|-------------|
| Tab / Shift+Tab | Move between tabbables |
| Enter / Space | Activate button |
| Arrow keys | Tabs, menus, listbox |
| Escape | Close overlay |

## Key Takeaways

- **First rule of ARIA**: prefer native HTML semantics and behavior.
- **Roles** tell AT what an element is; wrong role worse than none on native tags.
- **Keyboard** support is mandatory — match ARIA APG patterns for custom widgets.
- **Focus** is shared UX for keyboard and AT — manage on open/close/route change.
- **tabindex="-1"** for programmatic focus; avoid positive tabindex.
- **aria-* attributes** express name, state, relationships, live updates.
- Accessibility tree is derived from DOM — align both (`0.5.10`).
