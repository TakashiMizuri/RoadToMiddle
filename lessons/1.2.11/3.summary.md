# Composition over Inheritance — краткая выжимка

> `1.2.11` · ~10 мин на повторение

## Главная идея

**Composition over inheritance** — предпочитай собирать поведение из отдельных частей (has-a + delegation), а не наследовать реализацию (is-a), потому что наследование жёстко связывает подкласс с каждым изменением базового класса (**fragile base class**).

## Ключевые пункты

- **Наследование** покупает substitutability и shared implementation, но создаёт **tight coupling** к `protected`/`virtual` API предка.
- **Композиция** — объект **имеет** collaborators (`private readonly`) и **делегирует** им; wiring через **constructor injection** (DI).
- **Is-a** — для стабильной доменной таксономии; **has-a** — для capabilities и reuse без LSP.
- **Fragile base class:** «безопасное» изменение в base ломает overrides в consumer-сборках.
- **LSP (Liskov):** подтип должен быть заменяем без сюрпризов; `Square : Rectangle` — классическое нарушение.
- **Decorator / strategy** — композиция вместо `protected virtual` hooks.
- **Наследование уместно:** framework hooks (`ControllerBase`, `Exception`), shallow Template Method, истинный is-a.
- **ASP.NET Core:** middleware, filters, `DelegatingHandler` — composition-first.

## Самые важные примеры

```csharp
// Composition: OrderService HAS repository, payments, email
public sealed class OrderService
{
    private readonly IOrderRepository _repository;
    private readonly IPaymentGateway _payments;
    public OrderService(IOrderRepository r, IPaymentGateway p) { ... }
}

// Decorator: stack validators without inheritance
public sealed class FraudCheckingValidator : IOrderValidator
{
    private readonly IOrderValidator _inner;
    public ValidationResult Validate(Order o) => /* check */ _inner.Validate(o);
}
```

## «Под капотом» в одном абзаце

Virtual dispatch (`callvirt`) встраивает overrides в единую vtable-цепочку — любое изменение virtual-метода затрагивает все подклассы во всех сборках. Композиция выносит вызовы на **interface-поля** в конструкторе: зависимости видны статическому анализу и тестам; JIT может devirtualize вызовы на sealed-типах. Подкласс **is** базовый объект в памяти (общий header + layout полей); композиция позволяет менять части независимо и избегать god base classes с накопленными зависимостями.

## Частые ошибки

- Наследовать ради трёх методов из двадцати (`Stack : List<T>`).
- Глубокие деревья (>2 уровня) в application code.
- `protected` fields и overrides без `base` при эволюции base.
- `BaseApiController` с auth + CRUD + mapping вместо filters + services.

## Мнемоника / правило большого пальца

> **Если сомневаешься в `sealed` — нужен interface, не subclass. Делегируй, не наследуй implementation.**
