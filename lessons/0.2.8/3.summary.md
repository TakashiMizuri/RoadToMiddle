# Pull Request / Merge Request workflow — краткая выжимка

> `0.2.8` · ~7 мин на повторение

## Главная идея

**PR/MR** — proposal влить **source branch** (feature) в **target** (обычно `main`) через review и CI. Это platform workflow поверх Git branches из `0.2.6`–`0.2.7`.

## Ключевые пункты

- GitHub **PR**, GitLab **MR** — одна механика.
- Flow: **branch → commits → push -u → open PR → CI → review → merge → delete branch**.
- **Draft PR** — сигнал WIP.
- Конфликты в UI → `fetch`, `rebase origin/main`, resolve (`0.2.5`), `push --force-with-lease`.
- Merge styles: **merge commit** | **squash** | **rebase merge** (`0.2.2`).
- **Branch protection**: no direct push на `main`, required checks, approvals.

## Самые важные примеры

```bash
git push -u origin feature/ticket-123
# open PR in UI
git fetch origin && git rebase origin/main
git push --force-with-lease
```

PR description: what / why / how to test; `Fixes #42` для issues.

## Частые ошибки

- Giant PR, пустое описание, merge при red CI.
- PR без sync с latest `main`.
- Забыть удалить branch после merge.

## Мнемоника

> **PR = предложение, не факт.** Green CI + approval = можно вливать.
