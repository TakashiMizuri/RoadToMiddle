# REST principles — краткая выжимка

> `0.3.10` · ~8 мин на повторение

## Главная идея

**REST** — architectural style: model **resources** (nouns) с URIs, manipulate через **representations** и HTTP **methods** (`0.3.4`), каждый request **self-contained** (stateless interaction), единые правила сообщений (uniform interface). Не синоним «любой JSON API».

## Ключевые пункты

- **Resource** + URI: `GET /users/42`, `POST /orders` — verbs в HTTP method, не в path (`/createOrder` — RPC smell).
- **Stateless**: server не держит «шаг мастера» между requests без id в request; DB resource state — OK.
- **Uniform interface**: standard methods, status codes (`0.3.6`), headers (`0.3.7`), documented representations.
- **Richardson levels**: большинство API — **level 2** (resources + verbs + codes); **HATEOAS** (level 3) optional.
- REST leverages **caching** safe GET + ETag; RPC POST tunnels теряют cache benefits.
- Cookies/JWT (`0.3.9`) не ломают stateless, если credentials в **каждом** request.

## Самые важные примеры

**Resource-oriented:**
```
GET    /products/7
PUT    /products/7
DELETE /products/7
→ 200/204/404/409 по смыслу
```

**Stateless auth:**
```
Every request: Authorization: Bearer …
Any app server can handle — no in-memory user wizard
```

## Частые ошибки

- Verbs в URL + только POST.
- 200 для всех outcomes.
- Путать «stateless» с «server без database».
- Dogma HATEOAS vs practical OpenAPI.

## Мнемоника

> **REST** на HTTP: **R**esources (nouns), **E**xchange representations, **S**elf-contained requests, **T**rust methods/status (uniform interface).
