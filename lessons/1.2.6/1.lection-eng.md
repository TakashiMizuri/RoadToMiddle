# Inheritance: base, override, sealed, hidden (new)

> Roadmap: `1.2.6` · Node: `1.2` — C# OOP · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Explain **single inheritance** in C# and how a derived class extends a base class.
- Use **`base`** to call parent constructors and inherited members from a derived class.
- Distinguish **`override`** (polymorphic replacement) from **`new`** (member hiding).
- Apply **`sealed`** on methods and classes to freeze an inheritance chain.
- Predict which method runs when base and derived types expose the same signature through different mechanisms.
- Recognize when hiding with **`new`** creates maintenance traps versus intentional API surface separation.

---

## Why This Matters

In lessons `1.2.1`–`1.2.5` you built classes as self-contained blueprints: fields behind properties, constructors that establish invariants, access modifiers that guard boundaries. Real domain models rarely stop at one class. An **`Invoice`** is a **`Document`**; a **`CreditCardPayment`** is a **`Payment`**. **Inheritance** models an **is-a** relationship: the derived type **is a specialized kind of** the base type and should be usable anywhere the base type is expected — with caveats you will master here and in `1.2.7`.

C# allows **single inheritance** for classes: one direct base class only. That constraint avoids the diamond ambiguity of C++ multiple inheritance while still giving you a powerful extension mechanism. The language gives you four related tools — **`base`**, **`override`**, **`sealed`**, and **`new`** — that look similar in tutorials but behave very differently at compile time and runtime. Confusing **`override`** with **`new`** is one of the most common sources of "I called the wrong method" bugs in code review. Confusing **`sealed`** with **`private`** leads to types that look extensible but are not.

Middle developers inherit from framework base classes daily — **`ControllerBase`**, **`DbContext`**, **`Exception`**. They override **`ToString`**, seal performance-critical paths, and occasionally hide an obsolete base member. Understanding **what the compiler records in metadata** and **what the runtime dispatches** separates intentional design from accidental fragility.

## Core Concepts

### Single Inheritance and the Is-A Relationship

Every class in C# implicitly derives from **`object`** unless you specify otherwise. When you write **`class Dog : Animal`**, **`Dog`** inherits instance fields (respecting accessibility), instance methods, properties, and events that the base exposes to derived code. The derived class **adds** members and **may replace** behavior for virtual members (covered in depth in `1.2.7`).

Inheritance is not free composition. A **`Dog`** object **is** an **`Animal`** for assignment:

```csharp
Animal pet = new Dog(); // legal — Dog is-a Animal
```

That assignment is the foundation of polymorphism. It also means **breaking base invariants** in a derived constructor or overridden method can violate **Liskov Substitution Principle**: code that worked with **`Animal`** must not break when given a **`Dog`**. Inheritance couples types across releases; favor it when the specialization is stable and shared behavior is substantial — a theme **`1.2.11`** develops further with composition.

### The base Keyword

**`base`** appears in two primary places: **constructor chaining** and **member access**.

When a derived class constructor runs, the runtime must initialize the base part of the object first. If you do not call **`: base(...)`** explicitly, the compiler inserts a call to the parameterless base constructor — which fails to compile if the base has no accessible parameterless constructor.

```csharp
public class Animal
{
    protected string Name { get; }

    protected Animal(string name) => Name = name;
}

public class Dog : Animal
{
    public string Breed { get; }

    public Dog(string name, string breed) : base(name)
    {
        Breed = breed;
    }
}
```

Here **`base(name)`** runs **`Animal`'s** constructor before **`Dog`** assigns **`Breed`**. The object is not fully constructed until the entire chain completes; do not pass **`this`** to external code from a constructor — the derived half may still be uninitialized.

Inside instance methods, **`base.Method()`** calls the **base class implementation** even if the derived class **overrides** that method. That is useful when the override wraps or extends behavior rather than replacing it entirely:

```csharp
public class LoggingDog : Dog
{
    public LoggingDog(string name, string breed) : base(name, breed) { }

    public override string ToString()
    {
        return $"[LOG] {base.ToString()}";
    }
}
```

**`base`** resolves statically to the **immediate base type** in the inheritance chain. It never skips levels or dynamically dispatch — the compiler knows exactly which method to call.

### override: Replacing Virtual Behavior

**`override`** replaces a **`virtual`**, **`abstract`**, or **`override`** member in the base class. The replacement participates in **runtime polymorphism**: when code holds a reference typed as the base class but pointing at a derived instance, the **derived override** runs. That dispatch model is the subject of `1.2.7`; for now, internalize that **`override`** is a **contract** with the base: same signature, compatible behavior expectations, and metadata that links override to **`slot`** in the virtual method table.

```csharp
public class Animal
{
    public virtual void Speak() => Console.WriteLine("...");
}

public class Dog : Animal
{
    public override void Speak() => Console.WriteLine("Woof");
}
```

Without **`virtual`** on the base member, **`override`** is a compile error. You cannot override non-virtual methods — only hide them (next section). Compiler error **CS0506** ("cannot override inherited member because it is not marked virtual, abstract, or override") protects you from assuming polymorphism where none exists.

### new: Hiding (Non-Polymorphic Replacement)

When a base member is **not virtual**, a derived class can still declare a member with the **same name and signature** using **`new`**:

```csharp
public class Report
{
    public void Print() => Console.WriteLine("Base report");
}

public class DetailedReport : Report
{
    public new void Print() => Console.WriteLine("Detailed report");
}
```

**`new`** tells the compiler: "I am **hiding** the inherited member, not overriding it." Without **`new`**, C# still compiles but emits warning **CS0108** — the hiding happened implicitly. Explicit **`new`** documents intent.

The critical behavioral difference:

```csharp
Report r = new DetailedReport();
r.Print(); // "Base report" — compile-time type Report

DetailedReport d = new DetailedReport();
d.Print(); // "Detailed report"
```

Hiding is **static dispatch** based on the **compile-time type** of the reference. **`override`** would use **runtime type**. That is why **`new`** is appropriate only when you **deliberately** do not want polymorphic behavior — often a sign the design should use a different method name or composition instead.

Properties and fields can be hidden too; the same static resolution applies. Hiding **does not remove** the base member from the object — both methods exist; the accessible one depends on the reference type.

### sealed: Stopping Further Overrides or Inheritance

**`sealed`** has two uses.

**Sealed method** — applied to an **`override`**, prevents further derived classes from overriding again:

```csharp
public class Animal
{
    public virtual void Speak() => Console.WriteLine("...");
}

public class Dog : Animal
{
    public sealed override void Speak() => Console.WriteLine("Woof");
}

public class RobotDog : Dog
{
    // CS0239: cannot override sealed Speak()
}
```

Framework authors seal methods when further customization would break security or invariants — **`string.GetHashCode`** is sealed in the BCL.

**Sealed class** — no class may inherit from it:

```csharp
public sealed class PaymentResult { /* ... */ }
```

**`string`**, many **`struct`-like** service types, and performance-sensitive sealed hierarchies use this. You trade extensibility for clarity and JIT devirtualization opportunities in hot paths.

**`sealed`** on a class is stronger than sealing every virtual member: inheritance of the type itself is forbidden. A **non-sealed** class with all virtual methods sealed still allows **subclassing** for new members, just not further overrides of those methods.

## Under the Hood

### Object Layout and Inheritance

A derived class instance is **one object** on the heap containing **all instance fields** from base and derived, laid out according to CLR rules (base fields typically precede derived fields; the exact layout is an implementation detail). There is not a separate **`Animal`** sub-object and **`Dog`** sub-object — there is a **`Dog`** whose memory includes **`Animal`'s** fields.

Method tables (**vtable** for reference types with virtual methods) live per **actual runtime type**. When **`Dog`** overrides **`Speak`**, the **`Dog`** method table slot for **`Speak`** points to **`Dog.Speak`**. **`new`-hidden** methods do not replace that slot; they add a separate method entry associated with the derived type's metadata, while the base slot still points to the base implementation.

### IL and Dispatch Hints

**`callvirt`** is emitted for virtual instance calls and resolves using the runtime type of **`this`**. Non-virtual calls use **`call`**, bound at compile time to a specific method definition.

When you **`override`**, the derived method's metadata includes **`.override`** directive linking to the base virtual slot. When you **`new`**, metadata records **`hidebysig`** semantics — the derived method is unrelated for virtual dispatch purposes.

Inspecting IL (ILDasm, ILSpy) on a **`new`-hidden** vs **`override`** pair makes the polymorphism difference concrete — a exercise worth doing once in your career.

### Constructor Chain and Field Initialization

Field initializers run **before** constructor bodies, **base to derived**:

```
1. Animal field initializers
2. Animal constructor body (via base chain)
3. Dog field initializers
4. Dog constructor body
```

If a base constructor calls a **virtual** method overridden in the derived class, the **derived override may run on a partially constructed object** — a classic fragile base class pattern. Production code often avoids virtual calls from constructors for that reason.

## Syntax / API

| Mechanism | Applies to | Effect |
|-----------|------------|--------|
| `: BaseType` | class | Inherit from single base |
| `: base(args)` | constructor | Run base constructor |
| `base.Member()` | method/property | Call base implementation |
| `virtual` | base member | Allows override |
| `override` | derived member | Polymorphic replacement |
| `new` | derived member | Hide non-virtual base member |
| `sealed override` | method | Block further overrides |
| `sealed class` | class | Block inheritance |

## Examples

### Example 1: Correct base Constructor Chain

```csharp
public abstract class Entity
{
    public Guid Id { get; }

    protected Entity(Guid id) => Id = id;
}

public class Customer : Entity
{
    public string Email { get; }

    public Customer(Guid id, string email) : base(id)
    {
        Email = email ?? throw new ArgumentNullException(nameof(email));
    }
}
```

**`Customer`** cannot be constructed without supplying **`Entity`'s** required **`id`**. The **`protected`** base constructor prevents external code from creating a bare **`Entity`** while allowing derived types to initialize **`Id`**.

### Example 2: override vs new Side by Side

```csharp
public class Notifier
{
    public virtual void Send(string message) =>
        Console.WriteLine($"Default: {message}");

    public void Log(string message) =>
        Console.WriteLine($"Log: {message}");
}

public class SmsNotifier : Notifier
{
    public override void Send(string message) =>
        Console.WriteLine($"SMS: {message}");

    public new void Log(string message) =>
        Console.WriteLine($"SMS log: {message}");
}

// Usage:
Notifier n = new SmsNotifier();
n.Send("Hi");  // SMS: Hi — override, runtime dispatch
n.Log("Hi");   // Log: Hi — hidden, static dispatch on Notifier
```

The asymmetry surprises juniors once; after this lesson it should not.

### Example 3: sealed Class for Value-Like Results

```csharp
public sealed class Money
{
    public decimal Amount { get; }
    public string Currency { get; }

    public Money(decimal amount, string currency)
    {
        Amount = amount;
        Currency = currency;
    }
}
```

Sealing prevents **`class DiscountedMoney : Money`** from breaking equality or currency invariants you intended to keep centralized.

## Common Mistakes & Anti-patterns

**Using `new` when you meant `override`.** Symptom: polymorphic code path calls base behavior. Fix: mark base **`virtual`**, use **`override`**, or rename the derived method.

**Calling virtual methods from base constructors.** Derived overrides may observe default field values. Defer to **`Init`** methods or factory patterns when necessary.

**Deep inheritance towers.** More than two or three levels often signals missing composition. **`1.2.11`** reframes this as a design smell.

**Public `new`-hidden members on framework base types.** Callers holding base references never see your API. Prefer distinct names or true overrides.

**Forgetting `base` in override that should extend behavior.** Empty override that duplicates base logic via copy-paste drifts when base changes — call **`base`** instead.

## Production & Real-World Notes

ASP.NET **`ControllerBase`**, EF **`DbContext`**, and **`Exception`** hierarchies are intentional inheritance. Teams document which methods are **`virtual`** for extension vs **`sealed`** for stability. When overriding **`Dispose`**, the pattern **`protected virtual void Dispose(bool disposing)`** with **`sealed public void Dispose()`** combines **`sealed`** public surface with controlled extensibility — you will implement this pattern in resource-owning types.

Code analyzers flag **`new`** on inherited members (FxCop/CA1060 family) because hiding often confuses consumers of libraries you do not control.

## Comparison / Trade-offs

**Inheritance vs composition (`1.2.11`).** Inheritance shares interface and implementation along an is-a axis; composition delegates behavior without coupling type identity. Prefer inheritance when polymorphic substitution is required and base behavior is substantial.

**override vs new.** **`override`** = participate in virtual dispatch, LSP obligations. **`new`** = separate method per compile-time type, no polymorphism — use sparingly with clear naming.

**sealed class vs unsealed with sealed methods.** Sealed class blocks subclassing entirely; sealed methods block overriding while still allowing new members in subclasses.

## Quick Reference

| Goal | Mechanism |
|------|-----------|
| Call parent ctor | `: base(args)` |
| Extend parent method | `override` + `base.Method()` |
| Replace non-virtual member without polymorphism | `new` |
| Freeze override chain | `sealed override` |
| Prevent subclassing | `sealed class` |
| Polymorphic call | `virtual` + `override` (see `1.2.7`) |

## Key Takeaways

- C# classes inherit from **one** base class; **`object`** is the root.
- **`base`** invokes parent constructors and members; resolution is static.
- **`override`** replaces **virtual/abstract/override** members for **runtime dispatch**.
- **`new` hides** inherited members; dispatch uses **compile-time** reference type.
- **`sealed`** on methods stops further overrides; on classes stops inheritance.
- Constructor order runs **base first**; avoid virtual calls from base constructors.
- Choose **`override`** for intentional polymorphism; **`new`** is a last resort with eyes open.
