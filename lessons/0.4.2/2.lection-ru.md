# IP addresses, IPv4, ports и sockets

> Roadmap: `0.4.2` · Node: `0.4` — Основы сетей · Depth: понимание

## Learning Objectives

После урока ты сможешь:

- Объяснить роль **IP addresses** на internet layer и чем они отличаются от **ports** на transport layer.
- Читать и записывать **IPv4** в dotted-decimal и понимать network vs host части (basics CIDR).
- Объяснить, зачем нужны ports и как `(IP, port, protocol)` однозначно идентифицирует endpoint на host.
- Определить **socket** как programming API endpoint и связать с `bind()` / `connect()` / `listen()`.
- Связать компоненты URL из `0.3.2` (host, port) с идентификаторами до старта HTTP.
- Предсказывать ошибки (`address already in use`, `connection refused`) из конфигурации address/port.

## Why This Matters

В `0.3.2` ты разбирал URL: `https://api.example.com:443/v1/users?page=1`. **Host** и **port** не декорация — после DNS они становятся числами для TCP и IP. Без понимания IP и ports ошибки Kestrel binding, Docker port mapping, PostgreSQL connection strings и firewall rules кажутся магией.

Каждое concurrent connection на server различается **5-tuple** на transport layer: `(source IP, source port, dest IP, dest port, protocol)`. Браузер открывает много вкладок; у каждого HTTPS connection свой ephemeral source port, destination часто `:443`. Middle developer ежедневно настраивает apps, containers и cloud security groups через эти идентификаторы.

Урок связывает `0.4.1` (layers) с `0.4.3` (TCP): TCP segments несут IP в network header и ports в transport header. HTTP Keep-Alive из `0.3.14` переиспользует один TCP connection — одну пару sockets — для нескольких requests.

## Core Concepts

### IP addresses — logical host identity

**IP address** идентифицирует host в IP network на **internet layer**. Это logical identity: у laptop может быть Wi‑Fi и VPN address; адреса меняются при смене сети.

**IPv4** — 32 bit, **dotted decimal**: `203.0.113.45`. ~4.3 млрд адресов — мало для сегодняшнего internet; mitigation: **NAT**, **private ranges**, **IPv6** (`2001:db8::1`, 128 bit).

**Public vs private (RFC 1918):**

| Range | Typical use |
|-------|-------------|
| `10.0.0.0/8` | Corporate / cloud VPC |
| `172.16.0.0/12` | Docker bridges |
| `192.168.0.0/16` | Home LAN |

Домашний laptop может быть `192.168.1.105` privately; router NAT'ит outbound на один public IP. AWS servers имеют private VPC + optional Elastic IP.

**CIDR** (`192.168.1.0/24`): первые 24 bit — network prefix; остальное — hosts. Router сравнивает destination с prefix: same subnet → local link delivery; иначе → gateway.

### Ports — multiplexing процессов на одном host

Один public IP может обслуживать nginx, Kestrel, PostgreSQL, Redis. **Ports** (16 bit, 0–65535) позволяют transport demultiplex incoming segments к нужному process.

**Well-known (0–1023):** HTTP 80, HTTPS 443, SSH 22, DNS 53, PostgreSQL 5432.

**Ephemeral (49152–65535):** client source ports, выбирает OS.

**`203.0.113.10:5432`** — PostgreSQL на этом host. **URL defaults из `0.3.2`:** HTTP без `:80`, HTTPS без `:443`.

### Sockets — interface для разработчика

**Socket** — endpoint send/receive, OS даёт file descriptor / `SOCKET` handle. **Berkeley sockets API** ( .NET `TcpClient`, Node, browsers):

- `socket()` — создать endpoint
- `bind(local IP:port)` — server слушает
- `connect(remote IP:port)` — client инициирует TCP
- `listen` / `accept` — server принимает clients

**Socket address** = IP + port + family (AF_INET / AF_INET6). Kestrel **binds** socket к `IPAddress.Any` или `127.0.0.1` (`0.4.6`).

- **SOCK_STREAM** — TCP (HTTP, HTTPS, PostgreSQL)
- **SOCK_DGRAM** — UDP (DNS, games — `0.4.4`)

### От URL к connection (связь с 0.3)

```
https://api.shop.com:443/orders
         │            └── port (default 443)
         └── host → DNS → 198.51.100.22
```

До HTTP (`0.3.3`): DNS → TCP `connect(IP, 443)` → TLS (`0.3.8`) → HTTP bytes. Pooling (`0.3.14`) держит шаги 2–3 alive.

### Special cases (preview 0.4.6)

- **`127.0.0.1`** — loopback
- **`0.0.0.0`** — bind «all interfaces», не destination
- **`255.255.255.255`** — limited broadcast

## Examples

### Пример 1: два сервиса, один IP

VM `10.0.1.50`: Kestrel `:5000`, PostgreSQL `:5432`. Firewall: `5000` только от LB; `5432` только от app subnet. Один IP — разные ports → разные processes через kernel demultiplexing.

### Пример 2: ephemeral client port

Browser → `93.184.216.34:443`, source `192.168.1.105:52441`. Вторая вкладка — `:52442`, тот же destination — два TCP connection (или HTTP/2 streams на одном — `0.3.13`).

## Common Mistakes & Anti-patterns

Путать IP и MAC. Считать `localhost` всегда IPv4. Bind на `127.0.0.1` — недоступен с других машин. **EADDRINUSE** — два process на одном `(IP, port, TCP)`. Забыть default ports в security groups.

---

## Production & Real-World Notes

Security groups фильтруют IP + port + protocol (L4). Kubernetes Services, NodePort, PgBouncer на `:6432`. Не expose PostgreSQL `:5432` в public без isolation.

---

## Comparison / Trade-offs

IPv4 vs IPv6: dual-stack common. Hostname vs hard-coded IP в config — DNS гибче при смене IP.

## Key Takeaways

- IP — host; port — service/process на host.
- IPv4 dotted decimal; private ranges + NAT.
- Socket — OS API для TCP/UDP endpoints.
- URL → `(resolved IP, port)` до HTTP.
- Pooling переиспользует TCP socket.
- Firewall и bind — IP + port + protocol.
