# Loops: for, while, do-while, foreach

> Roadmap: `1.1.10` · Node: `1.1` — C# Language Basics · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Repeat work with **`for`**, **`while`**, **`do-while`**, and **`foreach`** loops and explain when each fits best.
- Control loop counters and **termination conditions** to avoid infinite loops and off-by-one errors.
- Iterate **arrays**, **`List<T>`**, and other **`IEnumerable<T>`** sequences with `foreach`.
- Recognize why you must not **modify a collection** while iterating it with `foreach`.
- Combine loops with **`break` and `continue`** (detailed in `1.1.11`) for practical console programs.

---

## Why This Matters

Branching from `1.1.9` decides *which path* to take once. Loops decide *how many times* to take a path. Process every line in a CSV, retry a flaky HTTP call up to three times, render each row in a table, sum numbers until the user types `quit` — all loops. Junior developers often memorize syntax; middle developers pick the loop that makes **invariants** obvious to the reader and the compiler.

In backend C# you will loop over query results, batch database updates, and stream `IAsyncEnumerable<T>` (later). On day one, loops appear in console tools, unit test data setup, and algorithm warm-ups. Choosing `for` when you need an index versus `foreach` when you only need values prevents subtle bugs. Misusing `while (true)` without a clear exit breeds unmaintainable code. This lesson builds the loop toolkit you will use until LINQ replaces many explicit loops — but LINQ itself is implemented with iterators that **`foreach`** consumes, so the model matters even when you write `.Where(...).Select(...)`.

## Core Concepts

### The Idea of Iteration

An **iteration** executes a block repeatedly until a **termination condition** becomes true (or forever if you forget the condition — the **`infinite loop`** nightmare). Every loop has:

1. **Initialization** — starting state (counter, enumerator, flag).
2. **Condition** — whether to run another iteration.
3. **Update** — how state changes between iterations (sometimes implicit).

Different loop keywords emphasize different parts of that triple. Picking the keyword that puts the important part **most visible** reduces mistakes.

### The `for` Loop

**`for`** is ideal when you know — or can compute — **how many iterations** you need, especially with an **index**:

```csharp
for (int i = 0; i < 5; i++)
{
    Console.WriteLine($"Iteration {i}");
}
```

The header has three clauses separated by semicolons: **init** (`int i = 0`), **condition** (`i < 5`), **iterator** (`i++`). All are optional; `for (;;)` is an infinite loop (rare; prefer `while` with a named condition for clarity).

**Off-by-one errors** are the classic loop bug: using `i <= array.Length` instead of `i < array.Length` reads past the end. When indexing arrays, the safe pattern is **`0` to `Length - 1`**, or:

```csharp
int[] ids = { 10, 20, 30 };

for (int i = 0; i < ids.Length; i++)
{
    Console.WriteLine(ids[i]);
}
```

Use `for` when you need the **index** — for example, pairing with another array at the same position — or when traversing a string by character index.

### The `while` Loop

**`while`** checks the condition **before** each iteration. If the condition is false initially, the body may **never run**:

```csharp
int remaining = 3;

while (remaining > 0)
{
    Console.WriteLine($"Retries left: {remaining}");
    remaining--;
}
```

Choose `while` when the number of iterations is **unknown upfront** — read until EOF, poll until a flag flips, process a queue until empty. The condition should become false eventually; if it cannot, you have an infinite loop and a frozen console app.

A common pattern is **`while (condition)`** with complex state updated inside the body. Keep the body focused; if it grows beyond ~15 lines, extract a method (`1.1.12`).

### The `do-while` Loop

**`do-while`** checks the condition **after** the body, so the body runs **at least once**:

```csharp
string? input;

do
{
    Console.Write("Enter yes or no: ");
    input = Console.ReadLine();
}
while (input != "yes" && input != "no");
```

Use this for **menu prompts** and **validation loops** where showing the prompt at least once is required. If zero iterations could be valid, prefer `while`. Note the trailing semicolon after the `while (...)` line — required syntax.

### The `foreach` Loop

**`foreach`** iterates any **`IEnumerable<T>`** (arrays, lists, dictionaries' keys/values with appropriate enumeration, LINQ results) without manual index management:

```csharp
List<string> names = new() { "Ada", "Grace", "Linus" };

foreach (string name in names)
{
    Console.WriteLine(name);
}
```

The compiler transforms `foreach` into a **`try/finally`** block that disposes an **enumerator** when the sequence implements `IDisposable` — important for file streams and database readers later. You do not see that machinery, but you benefit from it.

**Read-only iteration:** `foreach` gives you each **element value** (or `ref readonly` in advanced scenarios). For **`struct`** types, the iteration variable is a **copy** unless you use `ref readonly foreach` (C# 12+) — mutating the loop variable does not change the collection.

**Do not modify the collection** during `foreach` (add/remove elements). That throws **`InvalidOperationException: Collection was modified`**. If you must remove items, use a **`for` loop backwards**, **`RemoveAll`**, or build a new list.

### Choosing the Right Loop

| Need | Loop |
|------|------|
| Index 0..n-1, fixed count | `for` |
| Unknown count, maybe zero runs | `while` |
| At least one run (prompt) | `do-while` |
| Each element, no index | `foreach` |

When you only need transformation/filtering, LINQ (`Select`, `Where`) expresses intent declaratively — but it still iterates under the hood.

### Nested Loops

Loops inside loops model **2D structures** — tables, grids, matrices:

```csharp
for (int row = 1; row <= 3; row++)
{
    for (int col = 1; col <= 3; col++)
    {
        Console.Write($"({row},{col}) ");
    }
    Console.WriteLine();
}
```

Inner loop runs completely for each outer step. Total iterations = product of counts. Nested loops can be O(n²) — fine for small console demos, suspect in hot paths on large data without measurement.

## Under the Hood

`foreach` desugars roughly to:

```csharp
var enumerator = collection.GetEnumerator();
try
{
    while (enumerator.MoveNext())
    {
        var item = enumerator.Current;
        // body
    }
}
finally
{
    enumerator.Dispose(); // if IDisposable
}
```

That is why modifying the collection invalidates the enumerator state. **`for` on arrays** is often optimized by the JIT similarly to indexed access with bounds checks; in modern .NET some bounds checks can be elided when proven safe.

**Iterator variables** in `for` are scoped to the loop header in C# — redeclaring `int i` in an nested `for` is legal and shadows the outer `i`. This prevents accidental carry-over between loops.

## Syntax / Commands / API

```csharp
for (init; condition; iterator) { }
while (condition) { }
do { } while (condition);
foreach (Type item in enumerable) { }
```

Common helpers with loops: `.Length`, `.Count`, `Enumerable.Range(0, n)` for index sequences.

## Examples

### Example 1: Sum numbers until sentinel

```csharp
Console.WriteLine("Enter numbers (0 to finish):");

int sum = 0;

while (true)
{
    Console.Write("> ");
    if (!int.TryParse(Console.ReadLine(), out int n))
        continue; // skip bad input — see 1.1.11

    if (n == 0)
        break;

    sum += n;
}

Console.WriteLine($"Sum = {sum}");
```

Here `while (true)` works because **`break`** provides a visible exit when the user enters zero. Prefer a named condition when it stays simple: `while (!done)`.

### Example 2: Multiplication table with nested `for`

```csharp
int size = 5;

for (int i = 1; i <= size; i++)
{
    for (int j = 1; j <= size; j++)
    {
        Console.Write($"{i * j,4}");
    }
    Console.WriteLine();
}
```

Formatting `{value,4}` right-aligns columns — small console UX polish.

### Example 3: foreach over char array

```csharp
string word = "C#";

foreach (char c in word)
{
    Console.WriteLine(c);
}
```

`string` implements `IEnumerable<char>` — you iterate Unicode UTF-16 **char** units, not full grapheme clusters; that distinction matters for internationalized UI later.

## Common Mistakes & Anti-patterns

**Off-by-one** — `<= Length` when accessing `array[i]`. Use `< Length`.

**Infinite `while`** — forgetting to decrement a counter or advance an enumerator.

**Modifying list in `foreach`** — add/remove throws; refactor to `for` backwards or filter to new list.

**Using `for` everywhere** — when index is unused, `foreach` is clearer and harder to mis-index.

**Heavy work in loop condition** — `while (ComputeExpensive())` runs compute every check; cache in a variable if needed.

## Production & Real-World Notes

In ASP.NET, avoid long-running **blocking loops** on request threads — async iteration comes later. For batch jobs and console tools, loops remain standard.

Prefer **`foreach` + LINQ** for business collections when index is irrelevant — reviewers recognize idiomatic C#. Keep **`for`** when indexing parallel arrays or building fixed-size buffers.

Logging inside tight loops can flood observability backends — log aggregates, not every iteration, in production.

## Comparison / Trade-offs

| Loop | Pros | Cons |
|------|------|------|
| `for` | Index explicit; control iteration | Verbose for simple enumeration |
| `while` | Flexible precondition | Easy to forget update → infinite |
| `do-while` | Guaranteed first run | Always at least one run even if undesired |
| `foreach` | Clean, no off-by-one on index | No index; no collection modify |

```
for (int i = 0; i < n; i++) { }

Never modify collection during foreach
Off-by-one: use i < Length
```

## Key Takeaways

- Loops repeat until a termination condition is met — design that condition first.
- `for` = counted/indexed; `while` = precondition; `do-while` = at least once; `foreach` = sequence.
- Off-by-one bugs dominate loop errors — `< Length` is the default safe pattern.
- `foreach` uses an enumerator; do not structurally modify the collection mid-flight.
- Nested loops multiply iterations — know your complexity.
- Combine with `break`/`continue` (`1.1.11`) for real control flow inside loops.
