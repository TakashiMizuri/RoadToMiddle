# Cookies: Set-Cookie Attributes и SameSite

> Roadmap: `0.3.9` · Node: `0.3` — HTTP и Web · Depth: **глубоко**

## Learning Objectives

После этой лекции ты сможешь:

- Различать **Set-Cookie** (response instruction) и **Cookie** (request echo) из `0.3.7`.
- Объяснить attributes: **Expires/Max-Age**, **Domain**, **Path**, **Secure**, **HttpOnly**, **SameSite**.
- Предсказывать, когда browser **отправляет** или **не отправляет** cookies на cross-site requests.
- Описать trade-offs **SameSite=Lax/Strict/None** для CSRF vs OAuth/deep links.
- Настроить cookie authentication в ASP.NET Core с HTTPS (`0.3.8`).
- Сравнить cookie sessions vs **Authorization: Bearer** для SPA.

---

## Why This Matters

HTTP **stateless** by design (`0.3.10`). Servers идентифицируют users через **session id** в **cookie**. Browser автоматически шлёт `Cookie` header — login, carts, analytics без tokens в URL.

Cookies — **XSS и CSRF battlefield**. XSS крадёт `session_id` → impersonation. CSRF trick browser отправить cookie на malicious form. Browsers ужесточили defaults: **SameSite=Lax**, **Secure для None**, **HttpOnly** против script theft.

Middle developer configures `Set-Cookie` в ASP.NET Core, debugs «works in Postman, not browser» (SameSite + CORS + credentials), выбирает BFF vs Bearer. Misconfiguration → silent login failures после Chrome updates.

## Core Concepts

### Set-Cookie → Storage → Cookie

1. Server: **Set-Cookie**.
2. Browser stores name=value + scope.
3. Matching requests: **Cookie** header.

```
Set-Cookie: session=abc123; Path=/; HttpOnly; Secure; SameSite=Lax
Cookie: session=abc123
```

Multiple cookies — multiple Set-Cookie lines. Session id — **opaque**; не passwords в cookie.

### Expires и Max-Age

**Session cookie** — без Expires/Max-Age, до закрытия browser session.

**Persistent:** `Max-Age=2592000` (preferred) или `Expires=...`.

Короткий TTL на auth — меньше theft window.

### Domain и Path

**Path=/admin** — только под `/admin`.  
**Domain=.example.com** — subdomains (осторожно с scope).

Default **host-only** — safer. Не ставь Domain на public suffix hosts.

### Secure: HTTPS Only

Cookie только на **secure requests**. Production auth **обязательно** с `0.3.8`.

**SameSite=None** **requires Secure**.

### HttpOnly

**HttpOnly** — недоступен `document.cookie`. Mitigates **XSS theft**.

Не stop CSRF. Не stop XSS actions in page — но blocks simple exfiltration.

ASP.NET auth cookies → **HttpOnly** always (unless deliberate anti-pattern).

### SameSite

**Same-site** — same registrable domain (`app.example.com` + `api.example.com` = same-site; `evil.com` = cross-site).

**Strict** — never cross-site send. Strong CSRF; breaks some email deep links.

**Lax** (default) — block cross-site **subresources** (XHR from evil); allow **top-level GET navigation**.

**None** — all cross-site if **Secure**. OAuth/embed/cross-site `credentials: 'include'`. Needs **CSRF defenses**.

Third-party cookie restrictions — ads/trackers; first-party Lax auth standard.

### Limits

~4KB per cookie, ~50 per domain. Large JWT in cookie — breaks.

## Under the Hood

### Browser Algorithm (Simplified)

Parse Set-Cookie → reject invalid (None without Secure) → store → on request match URL + **SameSite context** → Cookie header.

Chrome 2020+ defaults broke cross-subdomain until explicit None or unified domain.

### ASP.NET Core

```csharp
options.Cookie.HttpOnly = true;
options.Cookie.SecurePolicy = CookieSecurePolicy.Always;
options.Cookie.SameSite = SameSiteMode.Lax;
```

Encrypted/signed cookie. Antiforgery for CSRF on forms.

SPA `app.` + API `api.` — cross-site; cookie auth needs None+Secure+CORS+CSRF or Bearer/BFF.

### CORS + credentials

`credentials: 'include'` + `Allow-Credentials` + non-wildcard Origin. SameSite still applies.

## Examples

Login HTTPS → HttpOnly Secure Lax. Evil.com XHR → cookie not sent.

OAuth cross-site → temporary None;Secure.

XSS without HttpOnly → document.cookie theft.

---

## Common Mistakes

No HttpOnly/Secure. Lax on cross-origin SPA API. JWT in non-HttpOnly for JS. Huge cookies. Broad Domain.

---

## Production & Real-World Notes

Rotate session on privilege change. Server-side logout invalidation. GDPR cookie inventory. BFF pattern for SPAs.

---

## Comparison / Trade-offs

| Approach | Pros | Cons |
|----------|------|------|
| HttpOnly session | Auto send, XSS harder | CSRF, cross-site |
| Bearer memory | No auto cross-site | XSS steals memory |
| Strict SameSite | Strong isolation | Breaks flows |
| Lax | Balance | Rare edge CSRF |
| None | Cross-site flows | Secure + CSRF required |

---

## Key Takeaways

- Set-Cookie instructs; Cookie echoes.
- HttpOnly + Secure + SameSite=Lax baseline on HTTPS.
- None requires Secure + CSRF.
- Cross-origin cookie auth hard — Bearer/BFF alternative.
- TLS + cookies + auth headers work together.
