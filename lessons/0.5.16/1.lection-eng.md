# HTML Validation — W3C Validator

> Roadmap: `0.5.16` · Node: `0.5` — HTML · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Explain **why HTML validation** matters for accessibility, SEO, and cross-browser consistency.
- Validate documents using the **W3C Nu Html Checker** (validator.w3.org) by URL, file upload, and direct input.
- Interpret **error vs warning** messages and locate the offending line in source.
- Fix common validation failures: unclosed tags, invalid nesting, missing `<title>`, duplicate IDs, obsolete attributes.
- Integrate validation into a **local workflow** before shipping markup (hand-written or generated).

---

## Why This Matters

Browsers are famously forgiving — they apply the **HTML parsing algorithm** and construct a DOM even from seriously broken markup. That forgiveness hides problems. A missing `</div>` might "work" visually because the parser implicitly closes elements, but the resulting DOM tree may differ from what you assumed in CSS selectors or JavaScript queries. Assistive technologies, search crawlers, and email HTML renderers vary in error recovery; valid markup reduces **surprise**.

Validation is a fast feedback loop when learning HTML (`0.5.x`) and when reviewing **template output** from Razor, React SSR, or CMS exports. It catches mistakes early: a `<ul>` directly inside another `<ul>` without `<li>`, a `<button>` nested inside `<button>`, `id` duplicated across components, or `data-productId` camelCase violating the data attribute rules you learned in `0.5.15`. Invalid arbitrary attributes fail validation — confirming why `data-*` exists.

This is the capstone practice lesson for node **0.5 — HTML**. Next you move to **CSS cascade** (`0.6.1`). Valid semantic HTML gives CSS and React a predictable structure to style and hydrate. Treat the validator as a **lint pass for markup**, complementary to ESLint for JavaScript and axe for accessibility audits.

## Core Concepts

### What HTML Validation Checks

The **W3C Nu Html Checker** (successor to the classic SGML validator) parses your document against **HTML living standard** rules (WHATWG-compatible checking). It reports:

- **Errors** — conformance failures (invalid element in context, forbidden attribute, bad character reference).
- **Warnings** — questionable but not always fatal (obsolete features, redundant roles).
- **Info** — stylistic or minor notes.

Validation checks **syntax and content models** — which elements may contain which children, required attributes, uniqueness of `id`, valid attribute names and values. It does **not** fully verify accessibility (use axe/WAVE), performance, or semantic quality of your copy. A page can validate and still have poor UX.

### The Validator Interface

Official tool: **[https://validator.w3.org/](https://validator.w3.org/)** (Nu Html Checker).

Three input modes:

1. **Validate by URI** — enter a public URL; validator fetches over HTTP(S). Requires deployed or tunnel-accessible page.
2. **Validate by File Upload** — upload `.html` file from disk; best for local work-in-progress.
3. **Validate by Direct Input** — paste markup into textarea; fastest for snippets during learning.

Recommended settings (show under "More options" or checkboxes):

- **Document type:** HTML5 (default for modern work)
- **Encoding:** UTF-8 (match your `<meta charset="UTF-8">`)
- **Show source** — displays numbered lines for error correlation
- **Show outline** — heading structure preview (helps spot skipped levels from `0.5` semantics lessons)

After submission, results list messages with **line:column**, message text, and excerpt of source.

### Reading Results

Example error pattern:

```
Error: Element div not allowed as child of element ul in this context.
From line 24, column 5; to line 24, column 30
```

Workflow:

1. Open your source at **line 24**.
2. Identify structural mistake — likely `<ul><div>` instead of `<ul><li>`.
3. Fix and **re-validate** until zero errors (warnings optional triage).

Some messages reference **content model** vocabulary — "phrasing content", "flow content". MDN element reference lists permitted contents; validator teaches you those rules experientially.

**False positives** are rare but possible with cutting-edge features not yet in checker rules, or with experimental attributes. Document and suppress only when you understand the trade-off; do not ignore real errors as "probably fine."

### Common Error Categories

**Unclosed or mis-nested tags** — classic beginner bug; parser recovery creates unexpected DOM (`0.5.10`).

**Missing required document elements** — `<!DOCTYPE html>`, `<html lang="...">`, `<title>`, charset meta.

**Duplicate `id`** — invalid; `getElementById` and labels break.

**Invalid attribute names** — `onclick` is valid; `productId="1"` without `data-` prefix is not (`0.5.15`).

**Interactive content nesting** — `<a>` inside `<a>`, `<button>` inside `<button>`.

**Obsolete attributes** — `align` on `<div>`, `border` on `<table>` (use CSS in `0.6`).

**Bad character references** — unescaped `&` in text (`AT&T` → `AT&amp;T`).

### Valid Document Skeleton

Start from a known-good baseline when debugging large files — binary search by commenting sections, or validate skeleton then add chunks.

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Validation Practice Page</title>
</head>
<body>
  <main>
    <h1>Hello</h1>
    <p>Valid minimal body.</p>
  </main>
</body>
</html>
```

Upload this — expect **no errors**. Then introduce one intentional bug, validate, observe message, fix — muscle memory for the practice exercises.

### Local and CI Validation

Beyond the website:

- **Nu Html Checker CLI** (`vnu.jar` / `html5validator` Python wrapper) for scripts and CI.
- Editor extensions (VS Code "Validate HTML") give inline hints.

For ASP.NET Razor, validate **rendered output** in browser (View Source or Save As), not raw `.cshtml` — server syntax is not HTML yet. For React, validate SSR HTML or static export; client-only SPA root `<div id="root"></div>` validates trivially while app markup does not — SSR snapshot is the right artifact.

## Under the Hood

The checker uses the **same parsing rules** as browsers (WHATWG tokenizer + tree construction), then applies **constraints** from the specification: each element has a **content model** defining valid children. When you omit optional tags (`<tbody>` auto-inserted), validator may accept what parser inserts — but explicit markup is clearer.

`DOCTYPE html` triggers **no-quirks mode** (standards mode) in browsers — important for consistent CSS layout later. Legacy DOCTYPE strings trigger quirks; validator flags obsolete DOCTYPEs.

The validator fetches **external entities** only for URI mode; file and direct input stay offline aside from posting to W3C service. For sensitive unreleased HTML, prefer file upload on local CLI if policy forbids sending markup externally.

## Syntax / Commands / API

### W3C Web UI Steps

1. Go to `https://validator.w3.org/`
2. Choose tab: **Validate by File Upload** (or Direct Input)
3. Select `.html` file or paste source
4. Click **Check**
5. Review errors list; click line numbers when "Show source" enabled
6. Fix source; repeat until clean

### Direct Input Quick Test

Paste broken snippet:

```html
<!DOCTYPE html>
<html lang="en">
<head><meta charset="UTF-8"><title>Test</title></head>
<body>
  <ul>
    <div>List item?</div>
  </ul>
</body>
</html>
```

Expect error about `div` not allowed as direct child of `ul`. Fix to `<ul><li>List item</li></ul>`.

### Optional CLI (concept)

```bash
# Python html5validator (wraps vnu) — if installed locally
html5validator --format text practice.html
```

Exit code non-zero when errors exist — suitable for CI gate on static site output.

## Examples

### Example 1: Missing Title

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
</head>
<body><p>No title</p></body>
</html>
```

Validator: **Error** — missing `title` element. Fix: add `<title>Descriptive page title</title>` (`0.5.13` SEO tie-in).

### Example 2: Duplicate ID

```html
<section id="main"></section>
<div id="main"></div>
```

Validator: **Error** — duplicate ID. Fix: unique ids or remove redundant wrappers.

### Example 3: Invalid Custom Attribute

```html
<p sku="WDG-42">Widget</p>
```

Validator: **Error** — attribute `sku` not allowed on `p`. Fix: `data-sku="WDG-42"` per `0.5.15`.

### Example 4: Button Nesting

```html
<button><button>Nested</button></button>
```

Validator: **Error** — interactive content nesting. Fix: single button or restructure with `<div>` wrapper outside.

## Common Mistakes & Anti-patterns

**"Browser shows it fine, skip validation"** — DOM recovery != intended structure; bugs surface on email clients, older AT, or SSR/hydration mismatch.

**Validating `.cshtml` or `.tsx` raw** — validate **rendered HTML** only.

**Ignoring warnings forever** — obsolete attributes hint at maintenance debt; plan CSS migration.

**Chasing zero warnings on third-party embed snippets** — sometimes vendor iframe markup is imperfect; isolate and document exceptions.

**Using presentational tags to silence errors** without semantic refactor — `<b>` vs `<strong>` debate is semantic; validator may accept both but accessibility may not.

## Production & Real-World Notes

Teams gate **static marketing pages** and **email templates** with validator in CI. Dynamic app shells often validate storybook HTML snapshots or component SSR fixtures. Pair validation with **accessibility** scanners — orthogonal checks.

Document template **components must output valid HTML** as code review criterion. When migrating legacy Bootstrap 3 sites, validation surfaces `type="text/css"` on `<style>` in body and other legacy patterns.

Keep a **known-good boilerplate** partial in Razor/React layout shared across pages — reduces repeated head/body mistakes.

---

## Comparison / Trade-offs

**W3C validator vs browser DevTools** — DevTools shows parsed DOM after fixes; validator shows **source-level** conformance before browser repair.

**Errors vs warnings** — fix all errors before merge; triage warnings by risk (obsolete vs informational).

**URI vs file validation** — URI tests production CDN output; file tests local branch without deploy.

**HTML validation vs ESLint (html plugin)** — overlap; validator is authoritative on content models; ESLint faster in editor loop. Use both if team agrees.

## Quick Reference

| Step | Action |
|---|---|
| 1 | Complete `<!DOCTYPE html>`, `lang`, charset, `<title>` |
| 2 | Upload or paste at validator.w3.org |
| 3 | Fix errors starting at lowest line numbers |
| 4 | Re-run until zero errors |
| 5 | Optionally fix high-value warnings |

| Frequent error | Typical fix |
|---|---|
| div in ul | Wrap in `<li>` |
| duplicate id | Unique `id` or class |
| unknown attribute | Use `data-*` or valid standard attr |
| missing title | Add `<title>` |
| unescaped & | Use `&amp;` |

## Key Takeaways

- **Valid HTML** aligns source with spec content models — predictable DOM for CSS, JS, SEO, and a11y.
- **validator.w3.org** (Nu Html Checker) supports URI, file upload, and direct input.
- Fix **errors** systematically using line numbers; re-validate after each fix batch.
- Common failures: nesting, duplicate IDs, missing `<title>`, non-`data-*` custom attrs.
- Validate **rendered output** from templates/SSR, not server template syntax files.
- Validation closes HTML node `0.5`; next is **CSS cascade** at `0.6.1`.
