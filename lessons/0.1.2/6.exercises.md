# Exercises: Repository, Working Tree, Staging Area

> Complete after reading `1.lection-eng.md`, `2.lection-ru.md` and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Label the states
**Goal:** Map Git terminology to everyday language.

For each situation, write **modified**, **staged**, **committed**, or **untracked**:

1. You created `notes.txt` but never ran `git add`.
2. You edited `Program.cs` after your last commit; no `git add` yet.
3. You ran `git add README.md`; no `git commit` yet.
4. `AuthService.cs` matches exactly what's in the latest commit on `main`.

**Success criteria:** 4/4 correct with one-sentence justification each.

---

### E2. Draw the three trees
**Goal:** Visual memory of Git's model.

On paper or Excalidraw, draw three boxes:

- Working Directory
- Staging Area (Index)
- Repository (`.git/`)

Add arrows labeled: `git add`, `git commit`, `git restore --staged`.

**Success criteria:** Arrows show correct direction; you can explain each arrow verbally in 30 seconds.

---

## Medium

### M1. Partial commit scenario
**Goal:** Apply staging for atomic commits.

**Setup:** Imagine you changed three files in one session:

| File | Change |
|------|--------|
| `OrderService.cs` | Bug fix (production-ready) |
| `OrderServiceTests.cs` | Tests for the fix |
| `appsettings.Development.json` | Local DB connection string |

Write the exact command sequence to create **one commit** containing only the bug fix and tests — **not** the config file. Include a good commit message.

**Success criteria:** Only two files staged; config remains unstaged; message describes the fix, not "update files".

---

### M2. Explain to a junior
**Goal:** Feynman technique — teach the three-tree model.

Write 5–8 sentences explaining to a junior developer:

1. Why Git has both `git add` and `git commit`.
2. What would go wrong if teams always used `git add .` without looking.

**Success criteria:** Mentions atomic commits, review, and at least one concrete bad outcome (mixed changes, secrets in history, hard to revert).

---

## Hard (stretch)

### H1. Mixed state puzzle
**Goal:** Deep understanding of index vs working tree.

**Scenario:**

```bash
# Starting point: clean working tree, tracked file hello.txt contains "Hello"
git add hello.txt          # you had changed it to "Hello, World"
# Before committing, you edit hello.txt again to "Hello, World!"
git status
```

Without running commands, predict what `git status` shows for `hello.txt`. Then explain what `git diff` vs `git diff --staged` would each show.

**Success criteria:** Recognizes dual state (staged + unstaged); diff shows different hunks for working vs index.
