# Partial Classes — Generated Code Pattern

> Roadmap: `1.2.18` · Node: `1.2` — C# OOP · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Explain what the **`partial`** keyword does at compile time.
- Split a **class**, **struct**, **record**, or **interface** across multiple source files safely.
- Distinguish **hand-written** partial parts from **tool-generated** partial parts.
- Use **`partial` methods** with optional implementations (call site in generated code, logic in your file).
- Navigate real-world patterns: **WinForms/WPF designers**, **EF Core**, **source generators**, **ASP.NET scaffolding**.
- Follow team rules: which file to edit, how to survive regenerating code.

---

## Why This Matters

You open a solution and see `MainForm.cs` beside `MainForm.Designer.cs`, or `AppDbContext.cs` beside `AppDbContext.g.cs`. The class name is the same; the files differ wildly — one holds your event handlers, the other auto-generated `InitializeComponent` and control field declarations. If you merged them manually, every time the designer saved, it would overwrite your edits. If you did not understand **partial classes**, you would either fear touching generated files or accidentally edit code that the next build replaces.

Modern .NET development is **half hand-written, half generated**. Source generators (compile-time), EF migrations and compiled models, OpenAPI clients, gRPC stubs, and UI designers all emit C# that must **compile as one type** with your code. The `partial` keyword is the seam: tools own one file; you own another; the compiler merges them into a single type definition before semantic analysis.

This is not academic. A Middle developer merges a PR that edits `*.Designer.cs` by hand, the next designer save deletes the fix. Another adds business logic to a generated gRPC partial incorrectly and wonders why the build overwrote it. Knowing the pattern prevents wasted hours and teaches a broader lesson: **separate stable intent from repeatable boilerplate**.

## Core Concepts

### What Partial Means

The **`partial`** modifier on a type declaration tells the compiler: "This is only **one fragment** of the type; other fragments may exist in other compilation units (typically other files)." All fragments must use the **`partial`** keyword with the **same type name** and compatible modifiers in the same namespace and assembly.

At **compile time**, the C# compiler **merges** all partial declarations into a **single** type. Members from all parts are combined. There is still **one** class in the final assembly — not nesting, not inheritance from yourself.

```csharp
// File: Customer.cs
public partial class Customer
{
    public void SendInvoice() { /* your logic */ }
}

// File: Customer.generated.cs
public partial class Customer
{
    public int Id { get; set; }
    public string Name { get; set; }
}
```

After merge, `Customer` has `SendInvoice`, `Id`, and `Name`. Cross-file visibility is total — private members in one part are visible to the other part of the same type (unique among C# visibility rules).

### Which Types Can Be Partial

**Classes**, **structs**, **records**, and **interfaces** can be partial. Enums cannot. Nested types can be partial if each part declares the same nesting.

**Methods**, **properties**, **indexers**, and **events** can also be **`partial`** — the method signature is split: one part declares signature (often generated), another part provides **implementation** (often your code). This is the hook for source generators that call into your extension points without you editing generated bodies.

### Why Split Files

**Human organization:** large types split by concern — validation vs mapping vs API surface — though over-splitting hurts discoverability; use when files exceed comfortable review size.

**Tool ownership:** designers and generators **rewrite** their file on each run. Your file stays untouched. The contract is: **never edit `*.Designer.cs` / `*.g.cs` / files marked auto-generated** unless the team explicitly says otherwise.

**Source generators:** run during compilation, emit `partial` types; you implement the other half in normal `.cs` files. Example pattern from community generators: `partial class MyController` with `[GenerateController]` — generator adds actions skeleton; you add custom actions in another partial.

### Partial Methods

A **`partial method`** has its declaration in one part and optionally an **implementing** part:

```csharp
// Generated part
public partial class Logger
{
    partial void OnLogWritten(string message);

    public void Write(string message)
    {
        // ...
        OnLogWritten(message);
    }
}

// Your part — optional; if omitted, calls are erased at compile time
public partial class Logger
{
    partial void OnLogWritten(string message)
    {
        Console.WriteLine($"Logged: {message}");
    }
}
```

Rules (simplified): partial methods return **`void`**, are **`private`** (C# 9+ allows more visibility in some cases for partial events), cannot be **`virtual`**. If **no implementing part** exists, the compiler **removes** the call sites — no runtime cost, no need for empty stubs. That is why generators use them as **optional hooks**.

If an implementing part exists, it behaves like a normal private method called from generated code.

### The Generated Code Pattern in Practice

A recurring layout:

| File | Owner | Contents |
|------|-------|----------|
| `Entity.cs` | You | Domain behavior, computed properties |
| `Entity.Designer.cs` or `Entity.g.cs` | Tool | Serialized fields, wiring, metadata |

**Header comments** often say `// <auto-generated>` or `<auto-generated />` — treat as "do not edit."

**Regeneration triggers:** save in UI designer, `dotnet ef dbcontext scaffold`, re-run source generator on build, update connected service reference.

**Customization strategies:**
1. **`partial` class** — add methods/properties in your file (safest).
2. **Partial method hooks** — implement optional callbacks.
3. **Inheritance / wrapping** — when generation is not partial-friendly (older tools).
4. **Configuration outside generated file** — EF `OnModelCreating` in your `DbContext` partial.

### Interfaces as Partial

**Partial interfaces** let generators add members to an interface you implement:

```csharp
// Your file
public partial interface ICustomerService
{
    Task<Customer> GetByIdAsync(Guid id);
}

// Generated adds more members in another partial
```

Implementing class must satisfy **all** parts after merge.

### Records and Partial

**`partial record`** works like partial class — useful when a generator adds positional syntax pieces or serialization helpers while you add behavior in another file. Same merge semantics.

## Examples

### EF Core DbContext Split

```csharp
// AppDbContext.cs — you maintain
public partial class AppDbContext : DbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options) : base(options) { }

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.ApplyConfigurationsFromAssembly(typeof(AppDbContext).Assembly);
    }
}

// AppDbContext.g.cs — compiled model or tooling (illustrative)
public partial class AppDbContext
{
    public DbSet<Order> Orders => Set<Order>();
}
```

You configure relationships in **your** partial; generated part exposes `DbSet` properties.

### WinForms-Style (Conceptual)

Designer file declares `partial class MainForm` with `InitializeComponent()` and `private Button btnOk;`. Your `MainForm.cs` handles `btnOk_Click`. Saving the form updates designer file only.

### Source Generator Hook

```csharp
[GenerateBuilder]
public partial class EmailMessage
{
    partial void OnBuilding(); // you implement to validate defaults
}
```

Generator emits builder calls `OnBuilding()` if you provided implementation.

## Common Mistakes & Anti-patterns

**Editing auto-generated files** — fix lost on regen. Move change to your partial or configuration.

**Forgetting `partial` on one file** — compile error: duplicate type definition or missing modifier.

**Mismatching namespace/class name** across parts — fragments do not merge; duplicate type errors.

**Putting business logic in Designer.cs** because "it is the same class" — works until someone saves the designer.

**Assuming partial means thread-safe or lazy merge** — merge is compile-time only; runtime is ordinary single type.

**Public partial methods** without implementation when required — know optional vs required partial method rules for your language version.

## Production & Real-World Notes

**.editorconfig** and analyzers sometimes flag edits to generated files — follow CI rules.

**Git:** designer noise in diffs is normal; some teams collapse designer commits.

**Source generators** (C# 9+): inspect `obj/generated` in build output when debugging missing partial pieces.

**Blazor, MAUI, ASP.NET** use similar splits — search `partial class` in template projects.

When scaffolding gRPC/OpenAPI clients, extend via **`partial` class** in separate file rather than modifying `*.cs` from tool.

Code reviews: flag manual edits inside `#region auto generated`.

## Comparison / Trade-offs

| Approach | Pros | Cons |
|----------|------|------|
| Partial split | Clean tool/human boundary | Harder to see full type in one view |
| Single file | Full visibility | Tools overwrite human edits |
| Inheritance from generated base | Works with non-partial output | Fragile hierarchy, less idiomatic in modern .NET |
| Composition wrapper | No codegen coupling | More boilerplate |

**Partial** is the idiomatic .NET answer when both tool and developer must extend the **same** type name.

## Quick Reference

| Element | Rule |
|---------|------|
| Keyword | `partial` on **every** fragment |
| Merge time | Compile time → one type |
| Generated files | `*.Designer.cs`, `*.g.cs` — do not edit |
| partial method | Optional impl; no impl → calls stripped |
| Types | class, struct, record, interface |
| Visibility | private in one part visible in other |

## Key Takeaways

- **`partial`** merges multiple declarations into **one** type at compile time.
- Tools own generated fragments; you own the other partial file(s).
- **Partial methods** are optional extension hooks with zero cost if unimplemented.
- Do **not** hand-edit auto-generated regions — customize via your partial.
- EF, UI designers, and source generators rely on this pattern daily.
- Same type name, namespace, and `partial` on all parts — or merge fails.
