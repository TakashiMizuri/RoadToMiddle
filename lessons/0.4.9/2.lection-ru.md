## Why This Matters

`npm run build` на React даёт static files: `index.html`, JS bundles, CSS, fonts, images. В production они должны быстро доходить до users worldwide. Один server во Frankfurt — European users OK; Singapore платит full RTT на **каждый** asset, latency из `0.4.11` умножается на десятки файлов.

**CDN** кладёт копии cacheable content в **edge locations** близко к users. Первый request в регионе — fetch с **origin** (S3, Blob, nginx); следующие — с edge до expiry. ASP.NET API обычно на origin/app servers — dynamic JSON не кешируют на edge без explicit design.

Middle dev настраивает CDN URLs в pipeline, debug'ит «users видят старый UI после deploy», выбирает cache headers. CDN ≠ load balancer (`0.4.8`): CDN = **delivery cacheable content**, не **compute scaling** API.

## Core Concepts

### Origin и Edge

**Origin** — authoritative source: S3 с React build, storage account, web server. **Edge servers (PoPs)** — globally distributed. PoP cache'ирует responses по URL (и иногда headers).

User → `https://cdn.example.com/assets/logo.png` → nearest PoP. Cached + fresh = **hit**. Иначе **miss** → fetch origin → store → respond. User может never touch origin для static.

### Что на CDN

**Good:** hashed JS/CSS (`app.[hash].js`), images, fonts, video. URL меняется каждый deploy.

**Usually not:** `index.html` (low TTL / invalidation — ссылается на bundles), user-specific API JSON, authenticated endpoints, WebSockets к API.

```
https://app.example.com/       → CDN → S3 (React)
https://api.example.com/orders → ALB → Kestrel
```

React с CDN вызывает API — **cross-origin** → CORS (`0.4.10`).

### TTL и Headers

CDN respect'ит **Cache-Control** от origin (unless CDN policy override). **`max-age=31536000, immutable`** на hashed assets — год cache. **`no-cache`** на `index.html` — revalidation для новых bundle refs.

**TTL** — сколько edge отдаёт copy без check origin. Long TTL = less load + latency; stale = old UI без **invalidate**/purge или new URLs.

**Cache busting:** content hashes в Vite/Webpack — deploy = new URL, лучше manual purge.

### Invalidation workflow

1. Upload build to origin.
2. Invalidate `/index.html` или paths (cost on some providers).
3. Hash filenames — purge только HTML часто enough.

Invalidation minutes; edges могут temporarily serve stale HTML.

### CDN vs другие слои

**Reverse proxy/LB** (`0.4.8`) — перед **app servers** для dynamic. **CDN** — перед **cacheable content**, often object storage. **Both:** CDN static, ALB API. DNS (`0.4.5`): `app` → CDN, `api` → ALB.

## Under the Hood

```
User (Tokyo) → Edge (Tokyo) → hit? yes → 200 (ms)
                    miss → Origin (S3 EU) → store → user
```

Hit = RTT до nearby city. Miss = edge RTT + edge-to-origin once per TTL/PoP.

DNS CNAME на CloudFront; anycast/geo routing к nearest PoP.

## Syntax / Commands / API

```nginx
location /assets/ {
    add_header Cache-Control "public, max-age=31536000, immutable";
}
location = /index.html {
    add_header Cache-Control "no-cache";
}
```

```bash
curl -I https://cdn.example.com/assets/app.js
# Cache-Control, Age, X-Cache, CF-Cache-Status
```

## Common Mistakes & Anti-patterns

- Cache `index.html` на год.
- API на CDN без rules → cached personalized JSON.
- Purge-only deploy без hashes.

---

## Production & Real-World Notes

Brotli/gzip на edge. DDoS absorption. Origin IP hide за CDN. Cost: origin egress vs edge egress.

---

## Comparison / Trade-offs

CDN + S3 — global latency, complexity. nginx same VM — simple, single region. SSR Kestrel ≠ CDN для bundles.

---

## Quick Reference

Hit/miss, hash filenames, short TTL для index.html, API не на CDN by default.

---

## Key Takeaways

- CDN — latency + origin load для static React assets.
- Hash filenames + long max-age; index.html — short TTL/purge.
- app (CDN) vs api (dynamic) split.
- Small internal apps — CDN optional.
