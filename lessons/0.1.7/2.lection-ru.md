# git show и git blame — Forensics

> Roadmap: `0.1.7` · Узел: `0.1` — Git: основы · Глубина: практика

## Learning Objectives

После урока ты сможешь:

- Инспектировать commits, tags и blobs через **`git show`**.
- Использовать **`git blame`** для поиска автора последнего изменения каждой строки.
- Применять флаги **`-L`**, **`-C`**, **`-M`** для точного forensic-анализа.
- Пройти практический **workflow** от симптома бага до ответственного commit.
- Использовать **`git log -S`** (pickaxe) для поиска появления или удаления строки.

---

## Why This Matters

Представь production-инцидент в два часа ночи. Login перестал работать после вчерашнего deploy, stack trace указывает на строку 847 в `AuthService.cs`. Кто-то задаёт вопрос, которого боится каждый on-call инженер: какой commit добавил плохой null check? Без Git forensics остаётся гадать по памяти, листать merged pull requests или спрашивать в Slack в надежде, что кто-то вспомнит. С forensics ответ можно получить за минуты — часто, не выходя из терминала.

Инструменты этого урока — `git show`, `git blame` и pickaxe search через `git log -S` — это read-only запросы к базе истории Git. Они не переписывают history, не меняют working tree и не требуют сети, если репозиторий уже есть локально. Это делает их безопасными во время инцидента, code review или отладки регрессии на ветке коллеги.

Важно явно сказать, чем forensics *не* является. Blame — не оружие культуры наказания. Название неудачное: в профессиональных командах `git blame` означает «кто последним трогал эту строку, чтобы спросить о намерении», а не «кого уволить». Строка может быть attributed Alice, хотя она написала её правильно; баг мог прийти из позднего merge, изменившегося API или данных, которые больше не соответствуют предположениям. Forensics даёт точку входа в расследование, а не приговор.

## Core Concepts

Каждый commit в Git — snapshot проекта в момент времени, связанный назад с parent или parents. Под капотом snapshot хранится не как полная копия каждого файла. Git хранит **tree** — структуру директорий, **blob** — содержимое файлов с адресацией по hash, и metadata: author, committer, timestamps, message. Parent pointers образуют directed acyclic graph, который ты знаешь как history.

Когда ты запускаешь forensic-команды, ты делаешь запросы к этой object database. `git show` отвечает: «что это за object и чем он отличается от parent?» `git blame` отвечает: «для каждой строки этого файла на этой revision — какой commit последним ввёл именно такое содержимое строки?» Обе команды вычисляют ответ в момент запроса из stored diffs и objects — Git не ведёт отдельный per-line audit log внутри каждого commit.

Думай о Git как о forensic database, где каждая запись неизменяема. Commit, once created, не меняется. Новые commits добавляют новые факты. Именно эта immutability делает историческое расследование надёжным: ты не читаешь log file, который кто-то мог отредактировать; ты обходишь content-addressed graph, где подделка сломала бы hashes и цепочку.

Различие annotated и raw views важно на практике. `git show` на commit даёт metadata плюс **patch** — diff относительно первого parent. Это человекочитаемая история изменений. `git blame` даёт per-line map от содержимого обратно к commits. `git log -S` ищет по history commits, где *количество* вхождений строки в tree изменилось — совсем другой вопрос, но он естественно дополняет blame, когда нужно найти, когда identifier впервые появился.

## Under the Hood

Понимание работы `git blame` помогает правильно интерпретировать output и знать, когда нужны extra flags. Git не хранит authorship per line внутри commit objects. При blame Git начинает с содержимого файла на запрошенной revision (обычно `HEAD`) и идёт по history назад. Для каждой строки находит commit, который последним ввёл её content, сравнивая blob diffs между revisions. Алгоритм связан с `git annotate` и использует caching для performance на больших histories, но концептуально это reverse engineering из snapshots.

Когда передаёшь `-C` или `-M`, Git включает copy и move detection heuristics. Plain blame смотрит только history текущего file path. Если функцию перенесли из `OldService.cs` в `NewService.cs` при refactor, plain blame на новом файле может attribute строки commit'у refactor — человеку, который cut-paste, — а не original author. Flag `-C` ищет строки, скопированные из других файлов в том же commit; `-C -C` ищет шире по history. Flag `-M` обнаруживает строки, moved или re-indented внутри файла или между файлами. Heuristics не идеальны, но сильно улучшают attribution после крупных refactors.

Для `git show` Git сначала разрешает аргумент через `rev-parse`. Commit hash или branch name — load commit, find parent, generate diff. Tag — dereference annotated tag к underlying commit. Syntax `rev:path`, например `git show v2.0.0:appsettings.json`, трактует `rev` как tree snapshot и идёт по path к blob, печатая raw file content без patch. Raw blob hash — decompress object и print. Путь разрешения:

```
git show <argument>
        │
        ▼
   rev-parse
        │
        ├── commit? → load parent → generate patch + metadata
        ├── tag?    → dereference → commit
        ├── rev:path? → tree lookup → blob content
        └── blob hash? → decompress → print
```

Merge commits заслуживают отдельной заметки. По умолчанию `git show` на merge commit даёт **combined diff**, который может сбивать с толку, потому что сливает changes от нескольких parents. Для forensic work на merges флаг `-m` показывает отдельные diffs против каждого parent. Реже, чем single-parent commits, но спасает от путаницы.

Pickaxe search (`git log -S`) работает иначе, чем blame. Он обходит commits и сравнивает количество вхождений строки до и после каждого commit. Если count изменился — commit попадает в список. Идеально для «когда впервые добавили `ValidateToken`?», но ужасно для «когда добавили `return`?» — common strings match почти каждый commit. Сужай path через `-- src/` или конкретный файл.

## Syntax / Commands / API

Базовая форма `git show` проста: без аргuments — последний commit на текущей ветке; с любой revision — commit hash, branch, tag или `rev~n` — inspect эту точку history. Output включает commit header (hash, author, date, message) и patch. `--no-patch` — только metadata без diff. `--stat` — компактная сводка: какие файлы и сколько строк.

Syntax `rev:path` — один из самых недооценённых инструментов forensic work. Команда `git show a1b2c3d:src/Program.cs` печатает `Program.cs` как он существовал в commit `a1b2c3d`, без checkout и без изменения working tree. Можно сравнить две historical versions двумя вызовами show с разными revisions. Бесценно, когда баг есть в production на tag `v2.0.0`, а на `main` уже fix — читаешь старый файл прямо из tag.

Для `git blame` default invocation аннотирует каждую строку файла. На файле в тысячу строк — шум. Flag `-L` ограничивает range: `git blame -L 840,860 Program.cs` blame только строки 840–860. В incident response почти всегда известен line number из stack trace — blame сначала эту строку, не весь файл.

Default blame output: abbreviated commit hash, author name, timestamp, line number, line content. Hash — ссылка на следующий шаг: `git show <that-hash>` для полного change в context.

Flags `-C`, `-C -C` и `-M` управляют copy и move detection. Flag `-w` говорит blame игнорировать revisions, где изменился только whitespace — essential после formatting pull request, иначе каждая строка выглядит newly touched formatter'ом.

Pickaxe search использует log: `git log -S "NullReferenceException" --oneline -- src/` — commits, где count этой строки изменился под `src/`. `-p` добавляет patch в output. Related flag `-G` — regex против diff hunks, может быть noisier; `-S` count-based и часто лучше для identifiers и API strings.

## Examples

Пройдём полный forensic workflow. Сервис бросает `NullReferenceException` на строке 847 `src/Services/AuthService.cs` после release. Шаг один: что на этой строке и кто последним модифицировал.

```bash
git blame -L 847,847 -- src/Services/AuthService.cs
```

Допустим, output показывает commit `f4e8b21`, Jane Doe, 15 июня. Шаг два: inspect этот commit полностью.

```bash
git show f4e8b21
```

Читай commit message, full diff, surrounding changes. Null check убрали intentionally при refactor? Commit трогал related tests? Часто ответ очевиден из patch. Если message говорит «Remove redundant null guard — UserService now guarantees non-null», дальше копай `UserService`, не intent Jane.

Если blame указывает на недавний refactor commit, а logic выглядит ancient — строку могли move. Re-run с copy и move detection:

```bash
git blame -C -M -L 847,847 -- src/Services/AuthService.cs
```

Теперь blame может attribute строку older commit до file rename. Follow hash через `git show`. Опционально, если подозреваешь, что expression ввели давно — pickaxe:

```bash
git log -S "user.Profile.Settings" --oneline -- src/
```

Flow от symptom к root cause повторяется: stack trace → line number, blame → commit, show → context, pickaxe или `-C -M` — когда refactors скрыли trail.

```
Bug report (line 847 NRE)
        │
        ▼
  git blame -L 847,847
        │
        ▼
  commit f4e8b21 ──► git show f4e8b21
        │
        ├── makes sense? → fix forward / revert
        │
        └── moved code? → git blame -C -M
                    │
                    └── older commit? → git log -S
```

Для release forensics tags сочетаются с show. `git show v2.0.0 --stat` — сводка всего, что ушло в release. `git show v2.0.0:appsettings.json` — configuration на момент release — полезно сравнивать environment-specific settings между versions без deploy старых builds.

Перед code review legacy code blame fragile block показывает, у кого спросить context:

```bash
git blame -L 200,250 src/Legacy/PaymentGateway.cs
git blame -w -L 200,250 src/Legacy/PaymentGateway.cs
```

Вторая команда игнорирует whitespace-only commits — видишь, кто последним менял meaningful logic, а не кто запускал formatter.

## Common Mistakes & Anti-patterns

Самая частая cultural mistake — treat blame output как proof of fault. Blame показывает последний commit, изменивший content строки, а не commit, введший bug. Строка, написанная правильно два года назад, будет «blamed» на тот old commit, хотя bug от new data или changed caller. Всегда interpret blame как pointer, затем `git show` и domain knowledge для полной картины.

После mass reformatting plain blame вводит всех в заблуждение. Каждая строка выглядит touched formatting commit. Используй `git blame -w` или blame файл на commit до merge format PR. Команды, которым важна accurate attribution, иногда делают blame до merge крупных format changes.

Refactors без `-C` и `-M` указывают на wrong author и wrong commit. Если investigation говорит «строка добавлена вчера», а code читается как six-month-old logic — rerun с copy detection до выводов.

На merge commits default `git show` может выглядеть empty или confusing — combined diffs скрывают per-parent changes. `git show -m`, когда нужно inspect вклад каждого parent.

Pickaxe на common strings заливает useless results. Поиск `return`, `null`, `{` match почти каждый commit. Всегда scope pickaxe на directory или file; prefer distinctive identifiers — method names, error message strings.

## Production & Real-World Notes

Современные IDE — Visual Studio, Rider, Cursor, VS Code с GitLens — показывают inline blame annotations на тех же данных, что CLI. Workflow тот же: suspicious line → identify commit → read full change. CLI skills нужны при SSH на build agent, minimal container или без IDE.

Во время incidents сочетай blame с deployment timestamps и `git log --since`, чтобы сузить commit range до line-by-line blame. Deploy в 18:00 UTC — `git log --since="2026-06-14 18:00" --until="2026-06-15 18:00" --oneline` даёт candidates; blame подтверждает, какой commit трогал failing line.

Large generated files — migrations, lock files, protobuf output — дают slow, meaningless blame. Не blame десять тысяч строк generated JSON; blame generator script или commit, который regenerated файл. Некоторые teams помечают generated paths в `.gitattributes`.

Author dates в blame expose work patterns. Используй для technical investigation, не surveillance. Healthy teams обсуждают code и intent открыто; blame — shortcut к правильному разговору, не scoreboard.

На code review forensics помогает быстро понять контекст спорной строки: blame → show → обсуждение в PR, а не догадки в комментариях.

## Comparison / Trade-offs

Иногда после blame и show картина всё ещё неполная: ты знаешь, кто изменил строку, но не знаешь, в каком именно commit впервые появился символ или паттерн, который сейчас ломает систему. Тогда pickaxe дополняет blame, а не заменяет его. Если же у тебя есть только «вчера работало, сегодня нет» и нет stack trace с номером строки, `git bisect` (тема другого урока) делает бинарный поиск по history быстрее, чем ручной перебор blame на каждом файле. Опытный разработчик держит в голове набор инструментов и выбирает по формулировке вопроса, а не по привычке всегда запускать одну и ту же команду.

`git blame` отлично отвечает «кто последним менял эту строку и когда», но ничего не говорит о *why* — commit message и review thread дают that. `git show` — полная история одной revision, но по одной за раз; chain следуешь вручную. Pickaxe (`git log -S`) находит introduction или removal строки по history, но только count changes, не moves где count тот же. Regex pickaxe (`git log -G`) match patterns в diff hunks — noisier. Для binary search по history при good/bad commit без line number — `git bisect`; blame не единственный forensic instrument.

## Quick Reference

Show commit: `git show <hash>`. Файл на revision без checkout: `git show <rev>:path/to/file`. Blame line range: `git blame -L 10,20 file.cs`. Follow moved/copied code: `git blame -C -M file.cs`. Ignore whitespace blame: `git blame -w file.cs`. Когда строка добавлена/удалена: `git log -S "text" -p -- path/`.

## Key Takeaways

- `git show` показывает commits, tags и file snapshots через syntax `rev:path`.
- `git blame` map'ит каждую строку на commit, последним изменивший её content — вычисляется из history, не stored per line.
- `-L` ограничивает blame range; `-C` и `-M` track copied/moved code через refactors.
- Стандартный forensic workflow: blame → hash commit → show для полного change.
- `git log -S` находит commits, где count строки изменился — сужай paths.
- Blame — investigation tool; interpret с context, не как automatic fault assignment.
