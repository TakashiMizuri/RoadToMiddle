# Exercises: First vs Single vs FirstOrDefault

> Complete after lectures and test.

## Easy

### E1. Cardinality table
**Goal:** For a given sequence and predicate, predict throw/return for First, Single, FirstOrDefault, SingleOrDefault.
**Success criteria:** Fill 4×3 matrix for `[1,2,3]` and `x => x > 1`.

### E2. Optional header
**Goal:** Read first line if file non-empty; null if empty — choose operator.
**Success criteria:** No exception on empty; code idiomatic.

## Medium

### M1. Repository method
**Goal:** Implement `GetByEmailAsync` (unique) and `GetBannerFlagAsync` (0–1 row).
**Success criteria:** Correct Single vs SingleOrDefault; XML doc states invariant.

### M2. Refactor review
**Goal:** Given `users.First(u => u.Id == id)` where Id is PK — is Single needed? Write 3-sentence justification.
**Success criteria:** PK guarantees ≤1 row; First vs Single trade-off articulated.

## Hard (stretch)

### H1. Map to HTTP
**Goal:** Design error mapping: SingleOrDefault empty → 404; Single with duplicates → 409.
**Success criteria:** Pseudocode catch/filter by exception message or custom helper.
