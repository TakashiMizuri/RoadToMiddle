# IAsyncEnumerable — Async Streams

> Roadmap: `1.4.16` · Node: `1.4` — C# async · Depth: понимание

## Learning Objectives

After this lesson you will be able to:

- Contrast **`IEnumerable<T>`** (sync pull) with **`IAsyncEnumerable<T>`** (async pull).
- Write **`async IAsyncEnumerable<T>`** methods with **`yield return`** and **`await`**.
- Consume streams with **`await foreach`** and **`ConfigureAwait`**.
- Apply **`CancellationToken`** via **`[EnumeratorCancellation]`** and `[AsyncEnumeratorCancellation]`.
- Explain memory and backpressure differences vs buffering entire collections.
- Recognize when async streams fit ASP.NET Core, EF Core, and HTTP streaming.

---

## Why This Matters

Classic **`IEnumerable<T>`** with **`yield return`** lets you **pull** items lazily — but each **`MoveNext()`** is synchronous. If the next element requires a **database page**, **HTTP chunk**, or **file read**, you block a thread or fake asynchrony with awkward patterns.

**C# 8** introduced **`IAsyncEnumerable<T>`** and **`await foreach`**: the consumer asks for the next item **asynchronously**. The producer can **`await`** I/O between yields. This models **streaming** naturally — export million rows without loading them into a `List<T>`, stream JSON lines from an API, or paginate EF queries with **`AsAsyncEnumerable()`**.

Without this model, teams buffer entire result sets (memory spikes) or spawn **`Task.Run`** per page (thread waste). Middle fullstack developers use async streams at the **service → controller** boundary and when integrating **System.Text.Json** async serialization.

## Core Concepts

### Sync vs Async Iteration

**`IEnumerable<T>`** implements the iterator pattern: **`GetEnumerator()`** returns **`MoveNext()`** + **`Current`**. The caller drives pace; work between items runs **synchronously** on the caller's thread unless the iterator itself wraps async incorrectly (anti-pattern).

**`IAsyncEnumerable<T>`** adds **`GetAsyncEnumerator()`** returning **`IAsyncEnumerator<T>`** with **`MoveNextAsync()`** returning **`ValueTask<bool>`**. The consumer uses **`await foreach`**:

```csharp
await foreach (var item in GetItemsAsync())
{
    Process(item);
}
```

Each iteration may **suspend** without blocking a thread pool thread during I/O.

### Authoring with async iterators

An **`async IAsyncEnumerable<T>`** method can **`yield return`** values and **`await`** between them:

```csharp
async IAsyncEnumerable<OrderDto> StreamOrdersAsync(
    [EnumeratorCancellation] CancellationToken ct = default)
{
    int page = 0;
    while (true)
    {
        var batch = await _repo.GetPageAsync(page++, ct);
        if (batch.Count == 0) yield break;
        foreach (var o in batch)
            yield return Map(o);
    }
}
```

The compiler generates a state machine similar to **`async Task`** but for iteration. **`yield break`** ends the stream.

**`[EnumeratorCancellation]`** injects the token from **`await foreach`'s** cancellation into the async iterator — so when the consumer stops or passes **`CancellationToken`**, the producer's pending **`await`** gets canceled.

### Consumption patterns

```csharp
// Basic
await foreach (var line in ReadLinesAsync(path))
    Console.WriteLine(line);

// With cancellation
await foreach (var item in StreamAsync(ct).WithCancellation(ct))
{ }

// ConfigureAwait in library code
await foreach (var x in GetAsync().ConfigureAwait(false))
{ }
```

**`WithCancellation`** extension attaches token to the enumerator.

### Relationship to IAsyncEnumerable and LINQ

**System.Linq.Async** (NuGet) adds **`Where`**, **`Select`**, etc. for async streams — deferred like LINQ to Objects. Without it, compose manually in the async iterator or materialize sparingly.

EF Core: **`context.Orders.AsAsyncEnumerable()`** streams query results; still mind **`Include`** cartesian cost — streaming doesn't fix bad SQL.

### Memory and backpressure

Async streams **pull** one item (or batch) at a time — consumer speed limits producer work **if** the iterator awaits before producing the next item. This is **implicit backpressure**: a slow consumer delays the next **`MoveNextAsync()`**.

This is **not** a full reactive push pipeline — for producer-driven push with buffering, see **`System.Threading.Channels`** (**`1.4.17`**).

## Under the Hood

The async iterator state machine implements **`IAsyncEnumerator<T>`**. Each **`await`** in the iterator method becomes a state transition; **`MoveNextAsync()`** returns a **`ValueTask<bool>`** — often completing synchronously for in-memory yields.

Disposal: **`await foreach`** generates **`try/finally`** calling **`DisposeAsync()`** on the enumerator — release DB connections, HTTP streams. Use **`await using`** when implementing custom enumerators wrapping resources.

## Examples

### Paginated API client

```csharp
async IAsyncEnumerable<GitHubIssue> GetIssuesAsync(
    string repo,
    [EnumeratorCancellation] CancellationToken ct = default)
{
    int page = 1;
    while (true)
    {
        using var response = await _http.GetAsync(
            $"repos/{repo}/issues?page={page}&per_page=100", ct);
        response.EnsureSuccessStatusCode();
        var issues = await response.Content.ReadFromJsonAsync<List<GitHubIssue>>(ct);
        if (issues is null || issues.Count == 0) yield break;
        foreach (var issue in issues) yield return issue;
        page++;
    }
}
```

Consumer processes issues as they arrive; peak memory ≈ one page, not full history.

### ASP.NET Core minimal API ( .NET 6+ )

```csharp
app.MapGet("/orders/stream", async (OrderService svc, CancellationToken ct) =>
{
    return Results.Stream(async stream =>
    {
        await foreach (var order in svc.StreamOrdersAsync(ct).WithCancellation(ct))
        {
            await JsonSerializer.SerializeAsync(stream, order, cancellationToken: ct);
            await stream.WriteAsync("\n"u8.ToArray(), ct);
        }
    }, "application/x-ndjson");
});
```

## Common Mistakes & Anti-patterns

**Using `IEnumerable` + `.Result` inside iterator** to "make it async" — blocks threads; use **`IAsyncEnumerable`**.

**Forgetting cancellation** — long streams without **`[EnumeratorCancellation]`** keep running after client disconnect.

**Multiple enumeration** — async iterators may re-query DB each time; cache or document single-pass.

**`ToList()` too early** — defeats streaming; only materialize when you need random access or count.

**Mixing `foreach` with `IAsyncEnumerable`** — won't compile; must **`await foreach`**.

## Production & Real-World Notes

ASP.NET Core **`IAsyncEnumerable<T>`** as action return type can serialize as JSON array **while streaming** in some configurations — understand serializer behavior.

gRPC server streaming and SignalR use different abstractions but same **pull/push** mental model.

For **HTTP response streaming**, set appropriate headers and avoid middleware that buffers entire body.

Test with **slow consumers** to verify connection and token cleanup.

## Comparison / Trade-offs

| | `IEnumerable` + yield | `IAsyncEnumerable` |
|---|----------------------|----------------------|
| I/O between items | Blocks or hacks | Native `await` |
| Consumer syntax | `foreach` | `await foreach` |
| LINQ | Built-in | System.Linq.Async |
| Backpressure | Pull (sync) | Pull (async) |
| vs Channels | Single producer iterator | Multi-producer queues |

## Key Takeaways

- **`IAsyncEnumerable<T>`** = lazy async pull sequence.
- **`await foreach`** drives **`MoveNextAsync()`** without blocking during I/O.
- **`[EnumeratorCancellation]`** wires consumer cancellation to producer.
- Prefer async streams over loading huge lists into memory.
- Implicit backpressure via pull — slow consumer slows production.
- For push/multi-producer scenarios, consider **Channels** next.
- Always dispose enumerators holding connections or streams.
