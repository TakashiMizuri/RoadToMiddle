# Exercises: HTTP Methods

> Complete after lectures and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Method picker
**Goal:** Match intent to method.

| Intent | Method |
|--------|--------|
| Download JSON profile | ? |
| Check ETag without body | ? |
| Create invoice (server assigns id) | ? |
| Remove session cookie resource | ? |

**Success criteria:** GET, HEAD, POST, DELETE (or POST logout if modeled as action — justify if chosen).

---

### E2. Safe or not?
**Goal:** Apply safe definition.

Classify **safe** or **not safe** (intended semantics): GET `/stats`, POST `/login`, DELETE `/files/1`, OPTIONS `/api`.

**Success criteria:** Only GET and OPTIONS safe; POST and DELETE not safe.

---

## Medium

### M1. Design mini API
**Goal:** REST-style method map.

Resources: `articles` and `articles/{id}/comments`. Define methods + paths for: list articles, create article, update title only, delete comment.

Write table: Method | Path | Success code.

**Success criteria:** GET collection, POST create, PATCH partial, DELETE comment; no verbs in path except optional nested resource names.

---

### M2. Fix the PR
**Goal:** Code review method misuse.

Review snippet:

```csharp
[HttpGet("orders/{id}/cancel")]
public IActionResult Cancel(int id) { _orders.Cancel(id); return Ok(); }
```

Explain two problems and propose fix (method + route + CSRF note).

**Success criteria:** GET mutation unsafe; suggest POST/DELETE appropriate + anti-forgery for cookie auth.

---

## Hard (stretch)

### H1. Idempotency-Key design
**Goal:** Bridge POST to safe retries.

Describe how you would document and implement `Idempotency-Key` header for `POST /api/payments` so network retries do not double-charge. 8–12 sentences; HTTP vs application layer.

**Success criteria:** Server stores key→result; duplicate key returns same response; POST remains non-idempotent at HTTP level but retry-safe at app level.
