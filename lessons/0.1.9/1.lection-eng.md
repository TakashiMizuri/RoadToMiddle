# Git Stash: Saving Work-in-Progress Without Committing

> Roadmap: `0.1.9` · Node: `0.1` — Git Basics · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Explain **why** `git stash` exists and when to use it instead of a commit.
- Save and restore **work-in-progress (WIP)** with `git stash push`, `pop`, and `apply`.
- List, inspect, and delete stashes with `stash list`, `show`, `drop`, and `clear`.
- Include **untracked files** in a stash using `-u` / `--include-untracked`.
- Describe how stashes are stored internally as **commits** in `refs/stash`.
- Switch branches safely when you have uncommitted changes.

---

## Why This Matters

You are halfway through implementing login validation on `feature/login`. Modified files litter your working tree — half-written tests, a refactored handler, config tweaks you have not verified yet. Then production pages you: a critical bug on `main` needs a hotfix now. Your manager expects you to switch branches, pull latest, fix, and push within the hour.

Git refuses a clean branch switch when uncommitted changes would be overwritten by checkout. Even when Git allows the switch, carrying dirty changes into `main` risks committing them to the wrong branch or creating merge confusion. You could commit everything with message `WIP`, but that pollutes history with meaningless commits you will squash or revert later. You could copy files to a backup folder outside Git — error-prone and untracked. You could discard the work — unacceptable when the feature is half done.

`git stash` exists for exactly this interruption. It temporarily saves your uncommitted work — both staged and unstaged tracked changes — and restores your working tree to a clean state matching `HEAD`. You switch branches, do the hotfix, return to your feature, and restore the saved work. The operation takes seconds and leaves no permanent scar on branch history unless you choose to commit afterward.

Stash is one of the most-used commands in professional daily workflows. Teams juggling code review feedback, urgent fixes, and parallel tasks reach for it constantly. Understanding stash internals also reinforces how Git stores snapshots — stashes are not a separate subsystem but clever reuse of the commit object model.

---

## Core Concepts

A stash is a saved snapshot of your working directory and staging area at a moment in time, stored outside your current branch history until you apply it. When you run `git stash push`, Git records what was modified in tracked files — both changes you staged with `git add` and changes that existed only in the working tree. Optionally, with flags discussed later, it can also capture untracked and even ignored files.

After a successful stash, your working tree matches `HEAD` cleanly. `git status` shows nothing to commit. You can switch branches, pull remote updates, or run commands that require a clean tree. The saved work lives in a stack structure referenced by `refs/stash`, waiting for you to reapply.

Stashes organize as a **stack** with last-in-first-out semantics. Git names them `stash@{0}` for the newest entry, `stash@{1}` for the next older, and so on. Pushing adds to the top. `git stash pop` applies the top entry and removes it from the stack. `git stash apply` applies the top entry but leaves the stash in place — useful when you want the same saved changes on multiple branches or want a backup until you confirm the apply succeeded.

Think of the stash stack as a pile of plates in a cafeteria. You put a plate on top when saving work. Pop takes the top plate and uses it. Apply photocopies the top plate onto your table but leaves the original in the stack.

The mental model matters: stash is **temporary** storage for **uncommitted** work. It is not a substitute for commits on a feature branch. Commits belong on branches, appear in history, can be pushed to remotes, and survive team collaboration. Stashes are local by default, invisible to teammates, and easy to forget. Use stash for context switches measured in hours or days at most — not as archival storage for finished work.

## Under the Hood

Stashes are not stored in a magic folder beside your repository. Each stash entry is a real Git commit object — typically a **merge commit** with up to three parents — linked through the special reference `refs/stash`.

When you push a stash, Git creates a merge commit whose parents represent three different snapshots of your project state at stash time:

```
                    stash@{0} (merge commit)
                   /      |      \
                  /       |       \
         base commit   index tree  working tree tree
         (HEAD)        (staged)    (unstaged changes)
```

The first parent is `HEAD` at the moment you stashed — the baseline commit you were on. The second parent captures the index state — what was staged. The third parent captures the working tree state including unstaged modifications to tracked files. This three-parent structure lets Git reconstruct both staged and unstaged portions separately when you apply.

The `refs/stash` reference points to the newest stash commit. Each stash commit's first parent links to the previous stash, forming a chain Git walks when you run `git stash list`. When you pop, Git applies the diff between the working-tree parent and the base, restores the index from the index parent, and moves `refs/stash` to the previous stash in the chain.

You can inspect this directly:

```bash
cat .git/refs/stash
git log --oneline refs/stash
git stash show -p stash@{0}
```

The last command shows the patch of unstaged changes relative to the base. Add `--include-untracked` context when you used `-u` during push.

Why implement stash as commits? Git already has efficient snapshot storage, diff generation, and metadata. Reusing the object model avoids a parallel format and lets stash participate in the same integrity checks — content addressing, zlib compression, reflog on the stash ref itself. When apply conflicts occur, Git uses the same conflict markers and resolution workflow as merges.

Stash entries also have reflog entries under `refs/stash`, providing limited recovery if you drop the wrong stash — though dropped stashes become unreachable and eventually garbage-collected, so treat `git stash drop` and especially `git stash clear` as irreversible without immediate reflog intervention.

## Syntax / Commands / API

Saving work starts with `git stash push`, abbreviated simply as `git stash` in common usage. The basic form stashes all tracked modifications — staged and unstaged — and cleans the working tree.

```bash
git stash push
git stash push -m "login validation WIP"
```

Always prefer `-m` with a descriptive message in real projects. Default messages like `WIP on feature/login: abc1234 Some commit message` tell you the branch and base commit but not what you were actually doing. Ticket IDs and brief descriptions — `stash push -m "ticket-1234 login validation WIP"` — save future you from guessing which stash entry to pop.

To include **untracked files** — files Git knows about on disk but never committed — use `-u` or `--include-untracked`:

```bash
git stash push -u -m "include scaffold files"
```

Without `-u`, untracked files remain in the working tree after stash and can still block branch switches if they would collide with files on the target branch. The rare `-a` or `--all` flag additionally stashes ignored files matching `.gitignore`. That is seldom needed and can pull in build artifacts if you are not careful.

You can stash only specific paths instead of everything:

```bash
git stash push -- src/auth/login.ts appsettings.Development.json
```

Restoring work uses `pop` or `apply`. Pop applies and removes from the stack:

```bash
git stash pop
git stash pop stash@{2}
```

Apply keeps the stash entry:

```bash
git stash apply
git stash apply stash@{1}
```

Management commands inspect and delete entries:

```bash
git stash list
git stash show
git stash show -p
git stash show -p stash@{1}
git stash drop stash@{0}
git stash clear
```

`git stash list` shows each entry with index, branch context, base commit, and message. `git stash show` without `-p` gives a stat summary; with `-p` gives the full patch. `git stash clear` removes every stash — destructive and without confirmation prompt — so always run `list` first.

## Examples

Walk through the emergency branch switch that motivates most stash usage. You are on `feature/login` with modified `src/auth/login.ts`. Production needs a hotfix on `main`.

```bash
git status
# modified: src/auth/login.ts

git stash push -m "login validation WIP"
git switch main
git pull origin main
# fix bug, commit, push hotfix

git switch feature/login
git stash pop
```

Line by line: the push saves your changes and leaves a clean tree. Switch to `main` succeeds because nothing is dirty. After the hotfix, you return to the feature branch and pop reapplies the stash. If `main` changed the same file in conflicting ways, pop may produce merge conflicts — resolve them like any merge, then `git stash drop` if pop failed partway and left the entry in the stack.

Untracked files illustrate a common gotcha. You create a scratch file for experiments:

```bash
echo "temp scaffold" > scratch.txt
git stash push
# scratch.txt still present — untracked files not included by default

git stash push -u -m "include scaffold"
# scratch.txt stashed; working tree fully clean
```

For reuse across branches, apply instead of pop:

```bash
git stash apply stash@{0}
# work on branch A

git switch branch-B
git stash apply stash@{0}
# same saved changes on branch B; stash still in stack

git stash drop stash@{0}
# remove when confirmed both applies succeeded
```

Inspect before applying when the stack has grown:

```bash
git stash list
git stash show -p stash@{2}
# confirm this is the login WIP, not the experiment from Tuesday

git stash pop stash@{2}
```

## Common Mistakes & Anti-patterns

Using stash as long-term storage is the most frequent abuse. Stashes have no branch, no remote backup, and no code review trail. Developers accumulate dozens of entries, forget contents, and lose work after `git stash clear` or disk failure. If work represents a meaningful milestone, commit it on a feature branch — even a `WIP` commit you squash later is safer than a six-month-old stash.

Never naming stashes makes `git stash list` a wall of identical `WIP on main` lines. The `-m` flag costs five seconds and pays off the first time you have more than two stashes.

Running `git stash pop` after the codebase changed significantly — you switched branches, pulled a hundred commits, rebased — invites conflicts. Treat pop like a merge: expect conflict markers, resolve carefully, and verify tests before continuing.

Forgetting `-u` when untracked files exist leaves those files behind. They block `git switch` when the target branch has a file at the same path, or they accidentally get committed later because they were never stashed and never ignored.

`git stash clear` without reviewing `list` permanently deletes all WIP snapshots. There is no recycle bin. Some teams alias clear to a confirmation prompt; default Git does not.

Stashing instead of committing finished work delays integration and hides progress from teammates. Stash is for interruption, not for "I will commit tomorrow" procrastination on completed tasks.

After a failed pop due to conflicts, Git may leave the stash entry in the stack even though partial application occurred. Read the output carefully, finish resolution, run tests, then manually `git stash drop` if pop did not auto-remove.

## Production & Real-World Notes

Teams often standardize stash messages with ticket IDs for traceability across interruptions. Some developers prefer WIP commits on feature branches and squash before merge — stash versus WIP commit is a team preference. Stash avoids history pollution; WIP commits survive machine restarts and appear on remotes if pushed.

CI/CD never sees stashes. Pipelines run on commits pushed to remotes. Stashed work is invisible until applied and committed. That is another reason not to treat stash as backup — your build system cannot compile what only exists in `refs/stash`.

IDE integrations in VS Code, Rider, and JetBrains Git UI expose stash as graphical lists with apply and pop buttons. The underlying commands are identical; understanding CLI behavior helps when the GUI shows a conflict you need to diagnose in the terminal.

When multiple developers share a machine or CI agent — uncommon but happens in training environments — stashes are per-repository and local to that Git config. They do not travel with the remote.

For simultaneous work on two branches without stashing back and forth, `git worktree` checks out multiple branches in separate directories — advanced topic beyond this lesson, but worth knowing stash is not the only answer to context switching.

## Comparison / Trade-offs

Stash fits temporary WIP during short interruptions: hotfixes, urgent reviews, pulling updates that require a clean tree. WIP commits on a feature branch fit work that might sit for days or needs to survive laptop sleep, travel, or sharing with a pair partner via push to a remote feature branch. Regular commits fit completed milestones worth preserving in history with messages and review.

Stash is faster and cleaner for history than a commit you plan to squash immediately. Commit is safer for anything you cannot afford to lose. Apply the rule: if losing the stash would ruin your day, it should have been a commit ten minutes ago.

## Quick Reference

Save WIP: `git stash push -m "msg"`. Include untracked: `git stash push -u`. List stack: `git stash list`. View diff: `git stash show -p`. Apply and remove: `git stash pop`. Apply and keep: `git stash apply`. Delete one: `git stash drop stash@{n}`. Delete all: `git stash clear`.

---

## Key Takeaways

- Stash saves uncommitted work and returns a clean working tree matching `HEAD`.
- Stashes form a stack — `stash@{0}` is newest; pop is LIFO.
- `pop` applies and removes; `apply` keeps the stash entry for reuse.
- Use `-u` to include untracked files that would otherwise block branch switches.
- Internally, stashes are merge commits referenced by `refs/stash`.
- Stash is local and temporary — not a substitute for commits on branches.
- Always use descriptive messages with `-m` in real projects.
