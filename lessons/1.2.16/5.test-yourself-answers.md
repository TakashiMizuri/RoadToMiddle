## Answers

### 1. [Recall]
**Answer:** Reference equality сравнивает **ссылки** (один объект в памяти или нет). Value equality сравнивает **значение/содержимое** по правилам типа.
**Why:** Default `Object.Equals` для class — reference; для многих BCL-типов (`string`) и кастомных override — value semantics.

### 2. [Recall]
**Answer:** Если `a.Equals(b)` == true, то `a.GetHashCode() == b.GetHashCode()` обязательно.
**Why:** Иначе hash table положит равные ключи в разные бакеты и lookup failed.
**Common mistake:** Переопределить только `Equals`.

### 3. [Recall]
**Answer:** Чтобы generic equality вызывал `Equals(T)` **без boxing** struct в `object`.
**Why:** `Equals(object)` на struct аллоцирует box; в циклах это дорого.

### 4. [Understanding]
**Answer:** `GetHashCode` остаётся identity-based (default), а `Equals` — по `Id`. Равные по `Equals` объекты имеют разные hash → разные бакеты.
**Why:** Контракт нарушен; `ContainsKey` ищет в wrong bucket.

### 5. [Understanding]
**Answer:** Сначала **`GetHashCode`** (через comparer) для выбора бакета; затем **`Equals`** по кандидатам в цепочке.
**Why:** Hash — быстрый фильтр; Equals — точное сравнение.

### 6. [Understanding]
**Answer:** Изменение поля меняет `GetHashCode` после insert; запись «теряется» в таблице.
**Why:** Bucket вычислен по старому hash; цепочка не совпадает.

### 7. [Application]
**Answer:** `override Equals(object?)`, `override GetHashCode()`, желательно `IEquatable<Order>` с `Equals(Order?)`.
**Why:** Минимальная корректная пара для class + performance path.

### 8. [Application]
**Answer:** `new HashSet<string>(StringComparer.OrdinalIgnoreCase)` или передать `StringComparer` в конструктор `Dictionary`.
**Why:** `IEqualityComparer<T>` задаёт правила без правки `string`.

### 9. [Analysis]
**Answer:** Нет `GetHashCode` — нарушен контракт с `Equals`; `Dictionary`/`HashSet` сломаются.
**Why:** Наследуется identity hash при value-based `Equals`.

### 10. [Analysis]
**Answer:** `==` не перегружен — default reference equality на class.
**Fix:** Перегрузить `==` и `!=`, делегируя в `Equals` (с null-handling).
**Common mistake:** Думать, что override `Equals` меняет `==`.

### 11. [Understanding]
**Answer:** Structural `Equals`, `GetHashCode`, `==`, `!=` по свойствам `Name` и `Age`.
**Why:** Это стандартное поведение positional record.

### 12. [Application]
**Answer:** `IEquatable<Tag>` с `bool Equals(Tag other)`.
**Why:** `List<Tag>.Contains` через comparer избежит boxing при сравнении.

### 13. [Analysis]
**Answer:** `Equals(NaN, NaN)` → true (BCL override); `==` → false (IEEE). Рефлексивность для `==` нарушена у float.
**Why:** При кастомном equality для float нужна явная политика (bitwise / tolerance).

### 14. [Understanding]
**Answer:** Нет. Коллизии допустимы; требование уникальных hash для неравных объектов невыполнимо (бесконечно много объектов, 2³² hash codes).
**Why:** Hash — сжатие в 32 bit; correctness через `Equals`.

### 15. [Application]
**Answer:** `return HashCode.Combine(LastName, FirstName, BirthDate);`
**Why:** Порядок фиксирован и согласован с полями в `Equals`.
