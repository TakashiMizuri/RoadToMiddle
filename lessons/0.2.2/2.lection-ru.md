# Merge: fast-forward vs merge commit vs squash

> Roadmap: `0.2.2` · Node: `0.2` — Git: branches and collaboration · Depth: **глубоко**

## Learning Objectives

После этого урока ты сможешь:

- Объяснить **merge** как объединение divergent histories через новый commit или движение ref.
- Различить **fast-forward merge**, **three-way merge commit** и **squash merge** по форме graph и сохранению history.
- Найти **merge base** (common ancestor) и понять, зачем Git нужны три commit'а для merge.
- Описать, что Git пишет в **index** при merge и когда появляются **conflicts**.
- Выбрать подходящую merge strategy для feature integration vs release hygiene.
- Читать `git log --graph` после каждого типа merge и предсказывать parent pointers.

## Why This Matters

В `0.2.1` ты узнал, что branches — refs на tip commit'ы, и две линии **diverge**, когда каждая ветка получает новые commit'ы после общего ancestor. Команда хочет снова одну линию: feature на `main`, hotfix с актуальным `main`, release с fixes. **Merge** — основной способ Git **интегрировать histories** без отбрасывания сторон.

Не каждый merge одинаков на graph. Иногда `main` просто **скользит вперёд** к feature — без нового commit object, только move ref. Иногда Git строит **merge commit** с двумя parents, явно фиксируя слияние двух линий. Иногда нужен **один чистый commit** на `main`, хотя на feature было двадцать WIP — squash merge. Неверный выбор бьёт по archaeology review, bisect, revert и blame. Middle должен читать merge graph и знать, какая операция его создала.

Merge использует ту же machinery, что и conflicts (`0.2.5`), и контрастирует с rebase (`0.2.3`–`0.2.4`). FF vs true merge vs squash — основа каждой кнопки «Merge» на GitHub.

## Core Concepts

### Divergence и merge base

Перед merge `feature` в `main` Git находит **merge base** — ближайший **common ancestor**, достижимый с обоих tips. С base `B`, tip `M` на `main` и `F` на `feature` **three-way merge** сравнивает **три snapshot** (trees в B, M, F).

```
        B (merge base)
       / \
      M   ... F
     main    feature
```

Если `M` — ancestor `F` (main не двигался, feature ушла вперёд) — возможен **fast-forward**. Если обе двигались — нужен **merge commit** (если не rebase сначала — `0.2.3`).

### Fast-forward merge

Когда target (`main`) **не diverged** — его tip ancestor tip source (`feature`) — `git merge feature` на `main` **двигает `refs/heads/main`** на `F`. Новый commit object не создаётся. History **linear**, просто длиннее.

```
До:  main → C2, feature → C4 (C2—C3—C4)
После: main → C4, feature → C4
```

FF — default, если возможен. **`git merge --no-ff`** запрещает FF и создаёт merge commit даже когда можно проскочить — часто на `main`, чтобы сохранить «этот PR landed как unit».

FF сохраняет каждый промежуточный commit feature на линии `main` — хорошо для traceability; может засорить `main` «fix typo».

### Three-way merge commit

Обе ветки ушли от merge base — FF невозможен. Git делает **three-way merge**: новый tree, combining changes с обеих сторон с B, и **merge commit** с **двумя parents** — M и F.

```
После merge commit X:
        B
       / \
      M   F
       \ /
        X  ← main (parents: M, F)
```

Merge commit — обычный commit object (`0.1.4`) с двумя строками `parent`. `git log --graph` показывает diamond. History **non-linear**, но **полная**.

При конфликтующих правках одних строк Git ставит **conflicts** (`0.2.5`). Успешный merge — `git commit` (или auto, если clean).

### Squash merge

**Squash** — workflow variant, не отдельный тип object. Git собирает все изменения `feature` с merge base в **один staged diff** на `main`, затем один **ordinary single-parent commit**. Commit'ы feature **не входят** в ancestry `main`.

```bash
git switch main
git merge --squash feature
git commit -m "Add login feature (#42)"
```

На `main` — один commit с net diff всей feature; chain feature остаётся на ref ветки, но не linked как parents на `main`. GitHub «Squash and merge» делает это на сервере.

Squash даёт **чистую linear main** ценой **granularity** commit'ов. Bisect на `main` прыгает feature целиком; blame на squashed lines — один commit.

### Merge vs rebase (preview)

**Merge** сохраняет true graph. **Rebase** (`0.2.3`) replay commit'ы на новый base, linear history через rewrite. Сравнение — `0.2.4`. Merge **records integration**; rebase **moves commits**, будто development был sequential.

## Under the Hood

### Алгоритм three-way (sketch)

1. Merge base B = `git merge-base main feature`.
2. Trees в B, M (HEAD), F (MERGE_HEAD во время merge).
3. По каждому path: changed one side → взять; both same → OK; both different → **conflict**, markers + stages 1/2/3 в index.
4. Если clean — tree, commit с parents M и F, ref `main` → новый commit.

В процессе merge: `.git/MERGE_HEAD` — hash другого tip; `.git/MERGE_MSG` — message. `git merge --abort` откатывает через `ORIG_HEAD`.

### Fast-forward — только ref update

Когда M ancestor of F:

```bash
echo <hash-F> > .git/refs/heads/main
```

Нет merge commit; `git log main` включает все feature commit'ы.

### Squash без merge commit

`git merge --squash` считает combined tree vs HEAD, **не** ставит MERGE_HEAD для two-parent. Index staged для одного **`git commit`**. Результат — **one parent** — как normal commit по parent count.

### Parents и `git log`

Two parents: default `git log` (first parent) идёт по линии `main`; `--graph --all` — полная topology. First parent — ветка, на которой был merge (`main`).

## Syntax / Commands / API

| Операция | Команда |
|----------|---------|
| Merge в current branch | `git merge feature` |
| Merge commit (no FF) | `git merge --no-ff feature` |
| Squash | `git merge --squash feature` + `git commit` |
| Abort | `git merge --abort` |
| Merge base | `git merge-base main feature` |

**GitHub PR:** Create merge commit | Squash and merge | Rebase and merge (`0.2.3`)

## Examples

### FF после review

```bash
git switch main
git merge feature
# Fast-forward
```

### `--no-ff` по policy

```bash
git merge --no-ff feature -m "Merge branch 'feature'"
```

Diamond с merge commit X.

### Squash для чистого main

```bash
git merge --squash feature
git commit -m "feat(auth): OAuth login (#128)"
```

Один commit на `main`; 15 commit'ов feature не в first-parent log.

## Common Mistakes & Anti-patterns

**Merge без обновления target** — больше conflicts. Сначала sync `main` в feature.

**Squash и продолжение той же feature branch** — путаница; новая ветка или rebase.

**Ожидание merge commit всегда** — FF только двигает ref.

**`--no-ff` везде** — пустые merge bubbles.

**Не удалять merged branches** — hygiene.

## Production & Real-World Notes

**Protected main** — PR + CI. Platform задаёт squash/no-ff.

**Release branches** — merge commits для topology; hotfixes cherry-pick (`0.2.11`).

Документируй **merge policy** в CONTRIBUTING.md.

## Comparison / Trade-offs

| Strategy | Graph | Плюсы | Минусы |
|----------|-------|-------|--------|
| FF | Linear | Просто, все commit'ы | Нет boundary marker |
| Merge commit | Diamond | Обе линии + integration point | Шумный graph |
| Squash | Linear на main | Чистый main | Нет granularity на main |

## Key Takeaways

- **Merge** интегрирует divergent histories; механика зависит от divergence tips.
- **FF** — только move ref, когда target — ancestor source.
- **Merge commit** — два parents, explicit integration.
- **Squash** — net diff одним single-parent commit на target.
- **Merge base** — anchor для combining changes.
- Кнопки platform = эти Git strategies.
