# Deferred Execution — Lazy IEnumerable Evaluation

> Roadmap: `1.3.9` · Node: `1.3` — C#: advanced topics · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Define **deferred execution** (lazy evaluation) for **`IEnumerable<T>`** LINQ queries.
- Distinguish **query construction** from **query execution** (enumeration).
- Explain what triggers execution: **`foreach`**, **`ToList`**, **`Count`**, terminal operators (`1.3.10`).
- Predict behavior when a LINQ query is **enumerated multiple times**.
- Identify **captured mutable state** and **changing data sources** as sources of subtle bugs.
- Describe which operators **stream** vs **buffer** the entire sequence internally.
- Relate deferred in-memory LINQ to **IQueryable** deferral until the provider runs SQL (`1.3.11`).

---

## Why This Matters

In `1.3.7`–`1.3.8` you wrote pipelines like `.Where(...).Select(...)`. A common beginner assumption: "I called **Where**, so it filtered the list." It did not — not yet. LINQ to Objects builds a **recipe** (a chain of iterators) and waits until something **pulls** elements through that chain.

Deferred execution is powerful: compose filters without copying data; build queries conditionally before running once. It is also dangerous: enumerate twice → run twice; hold a query while the database changes → inconsistent results; side effects in lambdas → unpredictable call counts. Production incidents from "I materialized too early" or "I never materialized and enumerated thrice" are routine.

Entity Framework Core relies on the same idea at a higher level: **`IQueryable`** stays deferred until **`ToListAsync`**, **`CountAsync`**, etc. — then one SQL round-trip. Understanding **`IEnumerable`** deferral first makes EF behavior predictable instead of mystical.

## Core Concepts

### Query Construction vs Execution

```csharp
var query = numbers.Where(n => n > 0).Select(n => n * 2);  // construction
// no elements processed yet

foreach (var x in query)   // execution begins
    Console.WriteLine(x);
```

**Construction** allocates iterator objects and stores delegates. **Execution** happens when the consumer calls **`MoveNext()`** on the enumerator — via **`foreach`**, LINQ terminal methods, or manual enumeration.

Assigning **`var q = ...`** never implies work on elements. This differs from eager APIs that compute immediately ( **`List.FindAll`**, **`Array.Sort`**, **`ToList`** on an already-built list).

### What Implements Deferred Execution?

**LINQ to Objects** standard query operators on **`IEnumerable<T>`** — **`Where`**, **`Select`**, **`Take`**, **`Skip`**, **`Concat`**, etc. — return deferred sequences.

**Exceptions within LINQ (operators that execute immediately on call):**

- **`Count`**, **`LongCount`** — must scan (or use fast path if source is **`ICollection<T>`** with **`Count`**).
- **`ToList`**, **`ToArray`**, **`ToDictionary`**, **`ToHashSet`** — materialize.
- **`First`**, **`Single`**, **`Max`**, **`Min`**, **`Sum`**, **`Average`** — consume until satisfied or end.
- **`Aggregate`** — fold, always runs.

These are **terminal** or **eager** operators (`1.3.10`). Everything before the first terminal in a chain remains deferred until that terminal pulls.

### Multiple Enumeration — The Query Runs Again

```csharp
var evens = numbers.Where(n => n % 2 == 0);

var count = evens.Count();      // full pass 1
var first = evens.First();      // full pass 2 (or partial for First)
```

Each enumeration **re-executes** the pipeline from the source **`numbers`**. If **`numbers`** is an **`List<T>`**, the source is stable. If **`numbers`** is **`File.ReadLines(...)`** or an EF **`IQueryable`** converted poorly, costs explode.

**Fix:** materialize once when you need reuse:

```csharp
var evens = numbers.Where(n => n % 2 == 0).ToList();
```

Trade-off: memory for stability and **`O(1)`** re-enumeration.

### Changing the Underlying Data

```csharp
var list = new List<int> { 1, 2, 3 };
var q = list.Where(x => x > 1);

list.Add(4);

foreach (var x in q)  // may include 4 — source mutated before enum
    Console.WriteLine(x);
```

Deferred query **reads current source state at execution time**, not at construction time (unless the operator snapshot — rare). Concurrent modification during enumeration can throw **`InvalidOperationException`** ("Collection was modified").

### Changing Captured Variables

```csharp
int min = 10;
var q = values.Where(v => v > min);

min = 20;
Console.WriteLine(q.Count());  // uses 20, not 10
```

Lambdas close over **variables**, not snapshots. Dynamic query building in loops must copy locals intentionally:

```csharp
foreach (var threshold in thresholds)
{
    var local = threshold;  // copy for closure
    queries.Add(data.Where(x => x.Score > local));
}
```

### Streaming vs Buffering Operators

**Streaming (typical single-pass):** **`Where`**, **`Select`**, **`Take`**, **`Skip`**, **`SelectMany`** (with caveats) — yield elements as upstream produces them.

**Buffering (must see more or all data):** **`OrderBy`**, **`Reverse`**, **`GroupBy`**, **`Join`**, **`GroupJoin`**, **`Distinct`**, **`Except`**, **`Intersect`** — internal collections before yielding.

When you **`foreach`** an **`OrderBy`**, the sort runs when enumeration starts — the entire upstream sequence up to that operator is consumed into a sort buffer. Memory and latency spike compared to a pure **`Where`→`Select`** chain.

Diagram:

```
Source ──► Where ──► Select ──►  foreach  (one pass, O(1) extra per element typical)

Source ──► Where ──► OrderBy ──► foreach  (buffer all, sort, then yield)
```

### Side Effects in Lambdas

```csharp
int calls = 0;
var q = items.Where(x => { calls++; return x.Valid; });

// calls still 0

q.Count();   // calls = items.Count
q.Count();   // calls += items.Count again
```

Never put logging, mutation, or I/O in **`Where`/`Select`** unless you explicitly want execution-frequency coupling. Use explicit loops or materialize first for deterministic side effects.

### Conditional Query Building — Why Deferral Helps

```csharp
IEnumerable<Order> query = dbOrders; // in-memory or IQueryable

if (filterByDate)
    query = query.Where(o => o.Date >= start);
if (minTotal.HasValue)
    query = query.Where(o => o.Total >= minTotal.Value);

var page = query.OrderBy(o => o.Id).Skip(pageIndex * size).Take(size).ToList();
```

Each **`Where`** wraps the previous **`IEnumerable`**. **One enumeration** at **`ToList()`** applies all filters — no intermediate lists per **`if`**. EF translates the final composed **`IQueryable`** to one SQL **`WHERE`** clause with **`AND`** — the payoff of deferral in APIs.

## Under the Hood

### Iterator Pattern and `yield return`

Deferred LINQ operators compile to **iterator blocks** (`1.3.20`). The compiler generates a state machine class implementing **`IEnumerator<T>`** with **`MoveNext`**, **`Current`**, **`Dispose`**.

**`Where`** pseudocode concept:

```csharp
foreach (var item in source)
    if (predicate(item))
        yield return item;
```

The **`foreach`** over **`query`** drives nested **`MoveNext`** calls — Russian doll enumerators, one per LINQ operator in the chain.

### No Magic Caching (by Default)

LINQ to Objects does **not** cache results of **`Where`/`Select`**. **`ICollection<T>`** or **`IList<T>`** sources may allow **`Count()`** without full scan, but **`Where`** wrapper still re-filters on each enum.

**`IQueryable`** caching is a separate concern (EF identity map, second-level cache) — not **`IEnumerable`** deferral.

### Relationship to `IQueryable`

Both **`IEnumerable`** and **`IQueryable`** LINQ appear deferred at construction. Difference is **what runs at execution**:

| | IEnumerable deferral | IQueryable deferral |
|--|---------------------|---------------------|
| Stores | Delegates (`Func<>`) | Expression trees |
| Executes | CLR loops in process | Provider (SQL) |
| Re-enumeration | Re-runs loops | Re-sends query (unless materialized) |

Lesson **`1.3.11`**–**`1.3.12`** deepen **`IQueryable`**. The **mental habit** is identical: build first, execute once intentionally.

## Examples

### Example 1 — Double enumeration cost

```csharp
IEnumerable<int> expensive = ReadNumbersFromNetwork()
    .Where(n => SlowIsPrime(n));

bool any = expensive.Any();  // primes computed
bool again = expensive.Any();  // computed again from network
```

Materialize **`ToList()`** after **`Where`** if **`SlowIsPrime`** is costly and reuse is needed.

### Example 2 — Empty query until terminal

```csharp
var q = logs.Where(l => l.Level == LogLevel.Error);
// logging pipeline not scanned

if (!q.Any())
    return;

foreach (var log in q)  // second scan if Any() already ran full sequence
    Process(log);
```

Better: **`var list = q.ToList()`** once, or use **`FirstOrDefault`** pattern consciously.

## Common Mistakes & Anti-patterns

**Assuming `var q = ...` already filtered** — only a plan until enumeration.

**Calling `Count()` then iterating** — two passes unless source is cheap.

**Side effects in `Select`** — may run zero or N times.

**Modifying collection during `foreach` on deferred query** — undefined behavior / exceptions.

**Using deferred query after `DbContext` disposed** — EF-specific; materialize inside scope.

## Production & Real-World Notes

Profile with intention: **`ToList()`** too early pulls entire tables into RAM; too late causes duplicate SQL. Middle rule: **materialize at boundary** — API response, view model, end of request — once semantics are fixed.

For infinite or live streams (**`IAsyncEnumerable`**, channels), deferral never ends until cancellation — different but analogous pull model.

## Comparison / Trade-offs

| Deferred | Eager (materialize) |
|----------|---------------------|
| Compose filters cheaply | Stable snapshot |
| Single pass if enum once | Safe multiple iteration |
| Reads live source | Predictable with mutations |

## Quick Reference

- **Deferred:** most **`IEnumerable`** LINQ operators
- **Execute on:** **`foreach`**, **`ToList`**, **`Count`**, **`First`**, etc.
- **Re-enumerate** = re-run pipeline
- **Buffering:** **`OrderBy`**, **`GroupBy`**, **`Distinct`**, ...
- **Fix reuse:** **`.ToList()`** once

---

## Key Takeaways

- LINQ builds a **description**; enumeration **executes** it.
- Multiple enumerations → multiple executions from source.
- Captured variables and mutable sources affect results at **run time**.
- Some operators **buffer** entire sequences before yielding.
- Same deferral discipline applies to **`IQueryable`** / EF — execute once at the right boundary.
