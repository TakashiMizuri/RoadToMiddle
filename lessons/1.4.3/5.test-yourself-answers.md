# Answers: Task Class

## Answers

### 1. [Recall]
**Answer:** `IsCompleted` — любое финальное состояние (success/fault/cancel). `IsCompletedSuccessfully` — только RanToCompletion без cancel.
**Why:** Faulted task IsCompleted true but not Successfully.

### 2. [Recall]
**Answer:** `AggregateException?` (non-null when Faulted).
**Why:** TPL aggregates; await unwraps.

### 3. [Understanding]
**Answer:** await не блокирует pool thread; throws inner exception без Aggregate wrapper; preserves async flow.
**Common mistake:** Wait in ASP.NET.

### 4. [Understanding]
**Answer:** Task faulted, но никто не await/check — runtime может поднять UnobservedTaskException при GC/finalization.
**Why:** Observation requirement.

### 5. [Application]
**Answer:** Exception после response; unobserved/lost in logs; пользователь видит 200 при фоновой ошибке.
**Why:** No observation in request scope.

### 6. [Application]
**Answer:** RanToCompletion (after await completes).
**Why:** Terminal success state.

### 7. [Analysis]
**Answer:** await WhenAll throws first inner; полный список — в Exception property агрегированных tasks или inspect each task.
**Why:** WhenAll AggregateException design.

### 8. [Recall]
**Answer:** false; IsCanceled true (unless mishandled).
**Why:** Separate terminal states.

### 9. [Analysis]
**Answer:** GetResult throws inner directly; Result throws AggregateException wrapper on faulted completed task.
**Why:** Historical TPL API difference.

### 10. [Application]
**Answer:** Sync known result — zero thread pool schedule, cached task, less allocation.
**Why:** CompletedTask optimization.
