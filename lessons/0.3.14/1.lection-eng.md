## Why This Matters

Every HTTP call your React app or microservice makes starts lower on the stack than JSON parsing. Opening a **new TCP connection** per request means DNS lookup, three-way handshake, often **TLS 1.3 full handshake** (1-RTT or more), then HTTP. On high QPS or chatty APIs (GraphQL micro-queries, health-check storms), connection setup can dominate latency and exhaust file descriptors.

**Keep-alive** lets multiple HTTP exchanges share one TCP connection. **Connection pooling** keeps warm connections ready instead of closing after each use. Misconfigured pools cause **`SocketException`**, hung requests, or **`503`** under load when the server stops accepting new connections. Middle developers debug these with metrics (active connections, pool wait time) and understand defaults in **`HttpClient`** ‚Äî including why **`new HttpClient()` per request** was an anti-pattern before `IHttpClientFactory`.

## Core Concepts

### The Cost of a New Connection

A cold HTTPS request path:

```
Client                    Server
  |---- DNS resolve --------> (cache miss: extra RTT)
  |---- TCP SYN ------------>
  |<--- SYN-ACK ------------|
  |---- ACK + TLS ClientHello ->
  |<--- ServerHello, cert ---|
  |---- Finished ----------->|
  |<--- Finished ------------|
  |---- HTTP request -------->|
  |<--- HTTP response --------|
```

Each new connection pays this tax. **TLS session resumption** (session tickets, PSK) shortens repeat handshakes but still worse than **reusing an established connection**. On localhost the cost looks negligible; at 200 ms RTT to a remote API it is painful.

### HTTP/1.1 Keep-Alive

Historically HTTP/1.0 closed connections after each response unless `Connection: keep-alive`. **HTTP/1.1 defaults to persistent connections** unless `Connection: close` is sent.

With keep-alive, sequential requests on the **same TCP socket** avoid repeated handshakes. Limits remain: **one in-flight request per connection** in typical HTTP/1.1 usage (pipelining unused), so browsers open **multiple persistent connections** per host (~6). Each stays open for **`Keep-Alive: timeout=N, max=M`** (server advertises idle timeout and max requests).

Idle connections consume memory and file descriptors on both ends. Servers (Kestrel, nginx) enforce **max keep-alive timeout** to reclaim resources.

### HTTP/2 and Connection Reuse

HTTP/2 multiplexes many streams on **one TCP connection** ‚Äî natural aggressive reuse. Opening extra TCP connections per host is discouraged (**connection coalescing** rules apply for different hostnames with same cert). Pooling at client level often means **one or few HTTP/2 connections** carrying hundreds of streams.

When that single connection stalls (TCP loss, server goaway), client must open another ‚Äî pool health checks matter.

### Connection Pooling on the Client

A **pool** holds open sockets keyed by **destination** (scheme + host + port). When code `GET`s a URL:

1. Borrow idle connection from pool (or create if under limit).
2. Send request, read response.
3. Return connection to pool if reusable; else discard.

**.NET `SocketsHttpHandler`** (default for `HttpClient`) maintains **`PooledConnectionLifetime`**, **`PooledConnectionIdleTimeout`**, **`MaxConnectionsPerServer`**. **`IHttpClientFactory`** in ASP.NET Core registers typed/named clients with sane handler lifetime ‚Äî **DNS refresh** when handler rotates (default two minutes) fixes stale IP after deploy.

Anti-pattern: **`using var client = new HttpClient()`** inside a loop ‚Äî socket exhaustion under load because old handler disposed slowly while sockets linger in **`TIME_WAIT`**.

**Browser** pools aggressively per origin ‚Äî DevTools shows **connection ID** or timing "Queueing" when pool saturated.

### Connection Handling on the Server

**Kestrel** accepts connections, parses HTTP, dispatches to middleware. Limits: **`MaxConcurrentConnections`**, **`KeepAliveTimeout`**, **`MaxRequestBodySize`**. Reverse proxies (nginx) buffer and may use **`keepalive` to upstream** ‚Äî separate **client‚Üînginx** and **nginx‚ÜîKestrel** pools.

**PostgreSQL connection pooling** (Npgsql, PgBouncer) is a parallel idea at the database layer ‚Äî not HTTP, but same pattern: expensive connect, cheap reuse, pool size tuning. Fullstack Middle connects both when API latency spikes from DB pool wait vs HTTP pool wait.

### Stale Connections and Health

A pooled connection may die silently (NAT timeout, server restart, load balancer idle cutoff). Next request fails with **connection reset**. Robust pools detect failed requests and **retry on fresh connection** (idempotent GET safe; POST needs care). **`PooledConnectionLifetime`** proactively rotates connections before middleboxes drop them.

## Under the Hood

TCP **`TIME_WAIT`** after close ties up quadruples on client ‚Äî why churning short connections exhausts ephemeral ports. Keep-alive avoids close per request.

HTTP/1.1 **`Content-Length`** or chunked encoding defines message boundaries on persistent socket ‚Äî parser must read full response before next request on same connection.

`HttpClient` **does not pool across unrelated instances** ‚Äî each handler has its own pool. Factory pattern shares handler lifetime across injections in DI scope.

HTTP/2 **GOAWAY** frame signals connection shutdown; client should drain and open replacement. Pool must honor this.

Linux **`ulimit -n`**, Windows dynamic ports, container cgroup limits ‚Äî production incidents when defaults too low for peak concurrent connections.

## Syntax / Commands / API

**Observe keep-alive (curl):**

```bash
curl -v https://httpbin.org/get 2>&1 | findstr /i "Connection keep-alive"
curl -v --http1.1 https://example.com/get  # force HTTP/1.1 behavior
```

**ASP.NET Core ‚Äî HttpClient factory (concept):**

```csharp
builder.Services.AddHttpClient("payments", client =>
{
    client.BaseAddress = new Uri("https://payments.internal/");
})
.ConfigurePrimaryHttpMessageHandler(() => new SocketsHttpHandler
{
    PooledConnectionLifetime = TimeSpan.FromMinutes(5),
    PooledConnectionIdleTimeout = TimeSpan.FromMinutes(2),
    MaxConnectionsPerServer = 20
});
```

**Kestrel limits (appsettings):**

```json
"Kestrel": {
  "Limits": {
    "MaxConcurrentConnections": 1000,
    "KeepAliveTimeout": "00:02:00"
  }
}
```

**nginx upstream keepalive (concept):**

```nginx
upstream api {
    server 127.0.0.1:5000;
    keepalive 32;
}
```

## Examples

### Example 1: Microservice fan-out

Order service calls inventory, payments, and shipping APIs ‚Äî 3 HTTP calls per checkout. Without pooling: 3√ó TLS handshakes. With shared `HttpClient` handler: reuse to each host ‚Äî latency drops especially on warm traffic.

### Example 2: Load test socket exhaustion

Integration test creates **`new HttpClient()` per iteration** ‚Äî passes locally (50 iterations), fails in CI at 5000 with **`Only one usage of each socket address`**. Fix: single factory-managed client; tune **`MaxConnectionsPerServer`**.

### Example 3: Deploy behind new IP

Long-lived **`SocketsHttpHandler`** caches DNS forever in older patterns. **`PooledConnectionLifetime`** forces handler recycle ‚Äî factory creates new handler, fresh DNS ‚Äî traffic shifts to new pods after Kubernetes rolling update.

## Common Mistakes & Anti-patterns

**New `HttpClient()` per request in ASP.NET** ‚Äî socket leak classic; use **`IHttpClientFactory`**.

**Infinite pool size** ‚Äî starves server or hits OS limits; tune **`MaxConnectionsPerServer`**.

**No timeout on pooled connections** ‚Äî hung upstream blocks pool slot until timeout.

**Ignoring proxy keepalive** ‚Äî nginx without `keepalive` to upstream opens TCP per request negating client pooling.

**Assuming HTTP/2 removes all pooling concerns** ‚Äî one bad stream or connection still affects many in-flight calls until detected.

**Closing connections with `Connection: close` unnecessarily** ‚Äî forces reconnect tax on every request.

## Production & Real-World Notes

Monitor: **active connections**, **request queue time**, **`http_client_open_connections`** (OpenTelemetry .NET). Alert on **`TIME_WAIT`** counts on load balancers.

**PgBouncer** + HTTP pooling: tune both; bottleneck moves to whichever pool is smaller.

Serverless (AWS Lambda) may not benefit from long keep-alive ‚Äî cold starts and short lifetimes favor **HTTP/2** to same API gateway with reuse within invocation container.

Corporate **HTTP proxies** break keep-alive or require **`HTTP_PROXY`** env ‚Äî `HttpClient` supports default proxy credentials.

## Comparison / Trade-offs

| Approach | Pros | Cons |
|----------|------|------|
| New connection per request | Simple | High latency, port exhaustion |
| HTTP/1.1 keep-alive | Reuse TCP, widely supported | Limited parallel requests per conn |
| HTTP/2 single conn | Many parallel streams | Single conn failure impact |
| Client pool + lifetime limits | Balance reuse and DNS refresh | Needs tuning per workload |

**Keep-alive timeout shorter** ‚Äî less server memory, more handshake overhead. **Longer** ‚Äî risk stale NAT, more FDs.

## Quick Reference

```
Connection cost = DNS + TCP + TLS (+ HTTP)

HTTP/1.1 keep-alive: reuse TCP for sequential requests (default on)
HTTP/2: multiplex many requests on one TCP

Client pool: borrow ‚Üí request ‚Üí return (per host)
.NET: IHttpClientFactory + SocketsHttpHandler
  PooledConnectionLifetime, MaxConnectionsPerServer

Symptoms: SocketException, slow under load, TIME_WAIT spike
Fix: shared HttpClient, tune pool/timeouts, proxy keepalive upstream
```

## Key Takeaways

1. **TCP + TLS setup** is expensive ‚Äî reuse beats reconnect per request.
2. **HTTP/1.1 keep-alive** persists connections; browsers use several per host.
3. **HTTP/2** multiplexes on one connection ‚Äî pooling looks like few long-lived conns.
4. **Client pools** (`HttpClient`) must be shared ‚Äî not new instance per call.
5. **`IHttpClientFactory`** manages handler lifetime and DNS refresh in ASP.NET Core.
6. **Servers and proxies** have keep-alive timeouts and connection limits.
7. Pool misconfiguration shows as **latency spikes and socket errors under load**.
