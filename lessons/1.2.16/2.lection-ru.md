# Object.Equals, GetHashCode и IEquatable

> Roadmap: `1.2.16` · Node: `1.2` — C# OOP · Depth: **глубоко**

## Learning Objectives

После этой лекции ты сможешь:

- Различать **reference equality** и **value equality** и объяснять, когда применяется каждый вид.
- Описать **контракт** между `Equals`, `GetHashCode` и операторами `==` / `!=`.
- Корректно переопределять `Equals` и `GetHashCode` для своих типов, включая **структурное** сравнение.
- Реализовывать **`IEquatable<T>`** для избежания boxing и ускорения hot paths.
- Объяснить, как **`Dictionary<TKey,TValue>`** и **`HashSet<T>`** используют hash code и equality.
- Понимать, когда **records** (`1.2.13`) уже дают equality, а когда нужна ручная реализация.
- Диагностировать типичные баги: нарушенный hash-контракт, асимметричное сравнение, сюрпризы с float.

## Why This Matters

Две переменные могут указывать на один и тот же объект в памяти, а могут указывать на разные объекты, которые представляют одно и то же логическое значение. Путаница между этими смыслами — один из самых частых источников тонких багов в C#. Ты добавляешь `Customer` в `HashSet<Customer>`, позже создаёшь другой экземпляр с тем же ID, вызываешь `Contains` — и получаешь `false`. Unit-тест сравнивает два `Money` через `==` и падает, хотя оба означают `10.00 USD`. API дедуплицирует DTO в словаре и молча оставляет дубликаты, потому что equality не переопределён.

Каждая коллекция, которая ключирует по объекту — `Dictionary`, `HashSet`, `ConcurrentDictionary`, LINQ `Distinct`, поиск в change tracker EF Core — зависит от согласованной модели равенства. BCL даёт defaults, но для **reference types** default означает «тот же экземпляр на heap», а не «тот же смысл в бизнес-логике». Value types (`struct`) по умолчанию сравниваются **по полям**, что часто неверно для составных ключей или доменных value objects.

Middle-разработчик не заучивает рецепт — он понимает **контракт** и реализует его один раз правильно. Контракт связывает `Object.Equals`, `GetHashCode`, опциональный `IEquatable<T>` и при необходимости перегруженные `==`. Сломаешь одну ножку — стол шатается, обычно в production, под нагрузкой, внутри hash table, о которой ты забыл.

## Core Concepts

### Reference equality vs value equality

При сравнении двух переменных **reference type** (`class`) через `ReferenceEquals(a, b)` или default `Object.Equals` вопрос звучит так: «Указывают ли эти ссылки на **один и тот же объект** в памяти?» Это **reference equality**. Значения полей не важны. Два `new Customer()` никогда не равны по умолчанию, даже если все свойства совпадают.

**Value equality** спрашивает: «Представляют ли объекты **одно логическое значение**?» Две разные `string` с `"hello"` value-equal, потому что `string` переопределяет `Equals` и сравнивает содержимое, а не ссылки. Для своих типов value equality — **opt-in**.

Связь с `1.2.1`: reference types — handles на heap; default equality следует за handle, а не за данными. Records (`1.2.13`) меняют картину для data-centric типов, генерируя value-based equality автоматически.

### Контракт Object.Equals

`Object.Equals(object? obj)` — virtual-метод в корне иерархии. Default в `System.Object` для reference types — **reference equality**. Для **value types** runtime синтезирует override с посравнением полей (с оговорками для полей-ссылок).

При override Microsoft формулирует контракт:

1. **Рефлексивность:** `x.Equals(x)` → `true` (кроме `NaN` у float — см. pitfalls).
2. **Симметрия:** `x.Equals(y)` ↔ `y.Equals(x)`.
3. **Транзитивность:** если `x.Equals(y)` и `y.Equals(z)`, то `x.Equals(z)`.
4. **Стабильность:** повторные вызовы дают тот же результат, пока объекты не менялись.
5. **Non-null:** `x.Equals(null)` → `false` для ненулевого `x`.

Нарушение симметрии или транзитивности ломает коллекции и сортировку. Классическая ошибка — асимметричное сравнение `Person` и `Employee`.

**Null:** всегда `if (obj is null) return false;` или pattern matching.

### GetHashCode и связь с Equals

`GetHashCode()` возвращает 32-bit `int` для **бакетирования** в hash-based коллекциях. Главное правило:

> Если `a.Equals(b)` == `true`, то `a.GetHashCode() == b.GetHashCode()` **обязательно**.

Обратное **не требуется**: неравные объекты могут коллидировать. Коллизии замедляют lookup (больше вызовов `Equals` в бакете), но не ломают корректность при правильном `Equals`.

Если переопределить только `Equals`, унаследованный `GetHashCode()` обычно основан на **identity** — value-equal экземпляры попадают в разные бакеты, и `Dictionary` их «не видит».

Для **immutable** value objects hash можно кэшировать. **Mutable** ключи в словаре — антипаттерн: изменение поля после insert ломает цепочку бакета.

### Реализация Equals для class

Надёжный шаблон для reference type с бизнес-ключом `Id`:

```csharp
public sealed class Customer : IEquatable<Customer>
{
    public Guid Id { get; }
    public string Name { get; }

    public Customer(Guid id, string name)
    {
        Id = id;
        Name = name;
    }

    public bool Equals(Customer? other)
    {
        if (other is null) return false;
        if (ReferenceEquals(this, other)) return true;
        return Id == other.Id;
    }

    public override bool Equals(object? obj) =>
        Equals(obj as Customer);

    public override int GetHashCode() => Id.GetHashCode();
}
```

**Sealed** снижает сюрпризы наследования. **`IEquatable<Customer>`** — typed path без boxing. **`ReferenceEquals`** — дешёвый short-circuit. **Бизнес-ключ** определяет равенство.

Для **структурного** равенства по всем полям:

```csharp
public override int GetHashCode() =>
    HashCode.Combine(Id, Name);
```

`HashCode.Combine` (.NET Core 2.1+) заменяет хрупкие формулы `hash * 31 + field.GetHashCode()`.

### IEquatable<T> — производительность и boxing

`IEquatable<T>` объявляет `bool Equals(T other)`. Generic-коллекции и `EqualityComparer<T>.Default` предпочитают этот метод.

Вызов `Equals(object? obj)` на **struct** без `IEquatable<T>` **боксирует** struct на heap. В tight loops (`List.Contains`, dictionary probe) это заметно. `IEquatable<MyStruct>` позволяет generic path вызывать typed overload напрямую.

В application code ты редко вызываешь `IEquatable<T>` явно — ты реализуешь его для инфраструктуры.

### Операторы == и !=

Для **reference types** `==` / `!=` — **reference equality**, пока ты не **перегрузишь** операторы (`1.2.17`). `Equals` и `==` могут расходиться, если override есть только у `Equals`.

Для **value types** `==` сравнивает значения.

**Strings** перегружают `==` для сравнения содержимого.

**Правило:** если переопределяешь `Equals` на class и хочешь интуитивного `==`, перегрузи `==` и `!=`, делегируя в `Equals`:

```csharp
public static bool operator ==(Customer? left, Customer? right) =>
    left is null ? right is null : left.Equals(right);

public static bool operator !=(Customer? left, Customer? right) => !(left == right);
```

### Как Dictionary и HashSet используют equality

На `Add` / lookup:

1. `hashCode = comparer.GetHashCode(key)`.
2. Индекс бакета из hash code.
3. Обход цепочки: `comparer.Equals(storedKey, searchKey)`.

Несогласованный `GetHashCode` → lookup не находит entry. Мутация ключа после insert → «осиротевшая» запись.

Кастомный **`IEqualityComparer<T>`** (например `StringComparer.OrdinalIgnoreCase`) даёт другие правила без изменения типа ключа.

### Records и сгенерированное equality

Records (`1.2.13`) генерируют `Equals`, `GetHashCode`, `==`, `!=` по позиционным и object properties. Для DTO часто достаточно `record` вместо ручной реализации. Сложное наследование или инкапсуляция — повод для manual equality.

## Under the Hood

### Virtual dispatch для Equals

`Object.Equals` — **virtual**. Вызов на переменной типа `Object` или базового class идёт в most derived override — так работает `string.Equals` через `object`.

`EqualityComparer<T>.Default` — struct-обёртка, создаваемая per `T`; JIT специализирует вызовы где возможно.

### Default equality для struct

Для struct без override runtime сравнивает каждое поле. Поле типа reference type использует equality этого типа (`string` — по содержимому, `byte[]` — по **ссылке на массив**, не по байтам — частая ловушка).

### Стратегии GetHashCode

`Object.GetHashCode()` для class без override исторически связан с identity — **не используй** как стабильный ID между процессами.

`HashCode.Combine` перемешивает hash полей; порядок аргументов важен для семантики.

### Float и nullable

`double.NaN`: `NaN == NaN` → false, но `double.Equals(NaN, NaN)` → true. Будь последователен в доменной модели.

`Dictionary` не принимает `null` ключ для non-nullable `TKey`; nullable reference types помогают поймать это на compile time.

### JIT и inlining

Маленькие `Equals` / `GetHashCode` на **sealed** типах часто **инлайнятся**. `IEquatable<T>` помогает devirtualization в generic code для struct.

| Member | Назначение |
|--------|------------|

## Examples

### Сломано: Equals без GetHashCode

```csharp
var d = new Dictionary<BadKey, int>();
var k1 = new BadKey(1);
var k2 = new BadKey(1);
d[k1] = 42;
Console.WriteLine(d.ContainsKey(k2)); // False
```

Fix: `GetHashCode() => Id.GetHashCode()`.

### Record struct

```csharp
readonly record struct Point(int X, int Y);
// Compiler-generated Equals/GetHashCode
```

### Comparer без изменения типа

```csharp
var set = new HashSet<string>(StringComparer.OrdinalIgnoreCase);
set.Add("Hello");
set.Contains("hello"); // True
```

## Common Mistakes & Anti-patterns

**Equals без GetHashCode** — silent bugs в Dictionary/HashSet. Всегда пара.

**Mutable ключи** — изменение после insert. Ключи immutable.

**Асимметрия в иерархии** — seal или явное сравнение типов.

**Float через `==` в Equals** — tolerance или bit patterns.

**Deep equality с циклами** — stack overflow; сравнивай по ID или visited set.

**Стабильность GetHashCode между запусками** — не для persistence.

## Production & Real-World Notes

EF: tracked entities и identity по ключу — отдельная история от DTO equality в API.

Distributed: in-memory equality ≠ equality между сервисами; для dedup сообщений — явные versioned ID.

Тесты: `Assert.Equal` использует `EqualityComparer<T>.Default` — тип должен соблюдать контракт.

Performance: `IEquatable<T>`, sealed types, дешёвый hash mix; профилируй перед микро-оптимизациями.

## Comparison / Trade-offs

| Подход | Когда |
|--------|-------|
| Default reference equality | Singleton, identity objects |
| Manual Equals + GetHashCode | Кастомные правила |
| `record` | DTO, structural equality |
| `IEqualityComparer<T>` | Альтернативное сравнение |
| `IEquatable<T>` | Structs, hot paths |

Выровняй `==` с `Equals` на class, если ожидается value semantics.

## Key Takeaways

- Reference types по умолчанию — **identity**; value equality — осознанный дизайн.
- `GetHashCode` должен согласовываться с `Equals`, иначе hash collections ломаются молча.
- `IEquatable<T>` — performance path, особенно для struct.
- `Dictionary` / `HashSet`: hash, затем `Equals`.
- Records для data models; manual — для сложных доменов.
- Выровняй `==` с `Equals` на class.
- Ключи словаря — **immutable** value objects.
