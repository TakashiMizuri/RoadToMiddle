# Abstract classes vs Interfaces — краткая выжимка

> `1.2.8` · ~8 мин на повторение

## Главная идея

**Abstract class** — один base с shared state, ctors и partial implementation. **Interface** — contract и **multiple capabilities** без общего class base. Оба поддерживают polymorphism; выбор — по **shared wiring** vs **role/capability**.

## Ключевые пункты

- **Single class inheritance** + **multiple interfaces**.
- Abstract: fields, protected helpers, **Template Method**, **abstract** + **virtual** members.
- Interface: DI/mock, unrelated types, ISP (мелкие контракты).
- **`abstract class`** нельзя `new`; concrete base с **virtual** — когда instantiation base допустим (`ControllerBase`).
- Pattern: **`IRequestHandler`** (framework) + **`HandlerBase`** (команда/логирование в команде).
- Versioning: новый **abstract** member ломает derived; interface — default methods (`1.2.10`).

## Самые важные примеры

```csharp
public abstract class RepositoryBase<T> {
    protected readonly DbContext Context;
    public abstract Task<T?> FindByIdAsync(...);
}

public interface IEmailSender { Task SendAsync(...); }
public sealed class OrderService : IEmailSender { ... } // + другие interfaces
```

## «Под капотом» в одном абзаце

Abstract class — обычный object + vtable. Interface — **`InterfaceImpl`** mapping на class methods; dispatch через interface reference может отличаться от class path (**explicit** — `1.2.9`). Multiple interfaces без diamond class problem.

## Частые ошибки

- Fat interface вместо нескольких small.
- Abstract hierarchy только ради copy-paste (composition — `1.2.11`).
- Interface там, где всем subtypes нужен один ctor/state.

## Мнемоника

> **Shared DNA + ctor → abstract. Can-do across types → interface. Often both.**
