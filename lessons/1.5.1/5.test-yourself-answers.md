## Answers

### 1. [Recall]
**Answer:** CLR (Common Language Runtime) — execution engine .NET: загружает assemblies, верифицирует IL, JIT-компилирует в native code, управляет памятью (GC), потоками и исключениями.
**Why:** Без CLR IL не исполняется; это центральный runtime process.

### 2. [Recall]
**Answer:** Managed code исполняется под контролем CLR (IL, GC, structured exceptions). Unmanaged — native machine code вне GC/verifier (C/C++, часть P/Invoke).
**Why:** Граница определяет memory safety и diagnostics.

### 3. [Recall]
**Answer:** Metadata (types, methods, fields, attributes), manifest (identity, references), optional embedded resources.
**Why:** Deployment unit = IL + type database + identity.

### 4. [Understanding]
**Answer:** C# компилируется Roslyn в **CIL/IL** bytecode в assembly; CPU исполняет native code после **JIT** (или AOT at build).
**Why:** Portability и unified type system across languages.

### 5. [Understanding]
**Answer:** Тела reference types — на **managed heap** (GC-tracked). Locals (value types inline, references as slots) — в **stack frames** per thread (`1.1.6`, `1.5.4`).
**Why:** CLR реализует модель копирования и reachability из 1.1.6.

### 6. [Understanding]
**Answer:** Metadata описывает type layout, methods, attributes — нужна JIT, GC scanning, reflection, exception handling, virtual dispatch.
**Why:** IL alone не несёт полную type system information in human form.

### 7. [Understanding]
**Answer:** IL stack — виртуальная модель для verification/JIT compilation одного метода. Native call stack — реальные activation records вызовов на thread (`1.5.4`).
**Common mistake:** Считать их одним стеком.

### 8. [Application]
**Answer:** Нет — память native allocator вне GC. Нужен explicit free или обёртка (`SafeHandle`, `Dispose`).
**Why:** GC roots только managed references.

### 9. [Application]
**Answer:** `ildasm`, ILSpy, dotPeek, `ilspycmd`.
**Why:** Disassembly/decompile показывает IL или C# equivalent.

### 10. [Analysis]
**Answer:** После first call JIT компилирует IL в **native code**; steady-state выполняется native, не interpreter loop. Tiered JIT (`1.5.2`) может recompile, но не interpret IL each call.
**Why:** Performance model .NET CoreCLR.

### 11. [Analysis]
**Answer:** **C** — `List<int>` — reference type; object на **heap**, `int` elements inline в array object на heap.
**Why:** Generic class is reference type regardless of type parameter.

### 12. [Understanding]
**Answer:** Стандарт CLI — формат IL, metadata, verification rules — основа cross-language .NET.
**Why:** Explains portable bytecode.

### 13. [Application]
**Answer:** Примеры: wrong bitness (x86 vs x64), corrupted/invalid PE, runtime version mismatch, wrong RID-specific build, mixing incompatible frameworks.
**Why:** Loader validates PE/CLI header before execution.

### 14. [Analysis]
**Answer:** CLR **exactly** knows reference fields via metadata/GC maps; can update all refs when compacting. C++ raw pointers без tracing GC — moving objects breaks pointers unless pinned/handles.
**Why:** Tracing GC requires controlled reference model.

### 15. [Understanding]
**Answer:** Native AOT precompiles IL→native at **build**; clrjit at runtime отсутствует или minimal; faster startup, limited dynamic codegen/reflection.
**Why:** Alternative pipeline to first-call JIT.
