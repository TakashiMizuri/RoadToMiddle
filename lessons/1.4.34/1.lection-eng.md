# BackgroundTaskQueue in ASP.NET — Channel Pattern

> Roadmap: `1.4.34` · Node: `1.4` — C# async/await · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Implement **`IBackgroundTaskQueue`** using **`System.Threading.Channels`**.
- Host a **`BackgroundService`** consumer that **awaits** each work item with **scoped DI**.
- Replace unsafe **fire-and-forget** (`1.4.30`) with **bounded backpressure**.
- Pass **correlation** and **idempotent** identifiers into queued delegates.
- Shut down gracefully with **`CancellationToken`** and channel completion.

---

## Why This Matters

Controllers should return fast; emails, webhooks, and file virus scans should run **outside** the request thread but **inside** the process — with **observed exceptions**, **fresh scopes**, and **bounded memory**. Microsoft's documented pattern — **queue + `BackgroundService`** — is the standard answer to fire-and-forget traps.

**`Channel<T>`** (`1.4.17`) gives you **producer/consumer** with optional **bounded capacity** so spikes do not enqueue unlimited closures. This lesson wires the pattern end-to-end for ASP.NET Core DI.

## Core Concepts

### Roles

| Component | Responsibility |
|-----------|----------------|
| **`IBackgroundTaskQueue`** | Enqueue `Func<CancellationToken, ValueTask>` or `Func<IServiceProvider, CancellationToken, ValueTask>` |
| **`Channel<T>`** | Thread-safe buffer between HTTP and worker |
| **`QueuedHostedService`** | `BackgroundService` loop: read → scope → await work |
| **`IServiceScopeFactory`** | New scope per item — no disposed `DbContext` |

### Queue Interface

```csharp
public interface IBackgroundTaskQueue
{
    ValueTask QueueBackgroundWorkItemAsync(
        Func<IServiceProvider, CancellationToken, ValueTask> workItem);

    ValueTask<Func<IServiceProvider, CancellationToken, ValueTask>> DequeueAsync(
        CancellationToken cancellationToken);
}
```

Using **`ValueTask`** for enqueue when often synchronous (unbounded channel write completes inline) reduces allocations.

### Channel Implementation

```csharp
public sealed class BackgroundTaskQueue : IBackgroundTaskQueue
{
    private readonly Channel<Func<IServiceProvider, CancellationToken, ValueTask>> _queue;

    public BackgroundTaskQueue(int capacity = 100)
    {
        var options = new BoundedChannelOptions(capacity)
        {
            FullMode = BoundedChannelFullMode.Wait
        };
        _queue = Channel.CreateBounded(options);
    }

    public async ValueTask QueueBackgroundWorkItemAsync(
        Func<IServiceProvider, CancellationToken, ValueTask> workItem)
    {
        ArgumentNullException.ThrowIfNull(workItem);
        await _queue.Writer.WriteAsync(workItem);
    }

    public async ValueTask<Func<IServiceProvider, CancellationToken, ValueTask>> DequeueAsync(
        CancellationToken cancellationToken)
    {
        var workItem = await _queue.Reader.ReadAsync(cancellationToken);
        return workItem;
    }
}
```

**`BoundedChannelFullMode.Wait`** applies **backpressure**: producers `await` when full — better than dropping or throwing for HTTP handlers (they slow down instead of exhausting memory).

### BackgroundService Consumer

```csharp
public sealed class QueuedHostedService : BackgroundService
{
    private readonly IBackgroundTaskQueue _queue;
    private readonly IServiceScopeFactory _scopeFactory;
    private readonly ILogger<QueuedHostedService> _logger;

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            var work = await _queue.DequeueAsync(stoppingToken);
            try
            {
                await using var scope = _scopeFactory.CreateAsyncScope();
                await work(scope.ServiceProvider, stoppingToken);
            }
            catch (Exception ex)
            {
                _logger.LogError(ex, "Queued background work failed");
            }
        }
    }
}
```

**`CreateAsyncScope()`** supports **`IAsyncDisposable`** services (`1.4.35`). Each work item is **fully awaited** — exceptions **observed**, not unobserved (`1.4.31`).

### Enqueue from Controller

```csharp
await _queue.QueueBackgroundWorkItemAsync(async (sp, ct) =>
{
    var mailer = sp.GetRequiredService<IEmailSender>();
    await mailer.SendOrderConfirmationAsync(orderId, ct);
});
return Accepted();
```

Capture **`orderId`** (value), not **`DbContext`** or request-scoped service references.

## Under the Hood

### Single Consumer vs Pool

The sample uses **one** consumer loop — serial execution. For parallelism, run **N** consumers reading the same channel (competing readers) or use **`Channel` + `Parallel.ForEachAsync`** with care and **idempotency**. More consumers increase throughput but complicate ordering guarantees.

### Registration

```csharp
services.AddSingleton<IBackgroundTaskQueue, BackgroundTaskQueue>();
services.AddHostedService<QueuedHostedService>();
```

Queue is **singleton**; work resolves **scoped** services inside per-item scope.

### Shutdown

On host stop, **`stoppingToken`** cancels **`ReadAsync`**. Decide policy for **in-flight** items — drain vs abandon. Optionally call **`_queue.Writer.Complete()`** on shutdown and finish remaining items.

### vs External Broker

In-process channel dies with the process — acceptable for **best-effort** notifications. **Outbox + Rabbit/Azure Service Bus** for **must-deliver** (`1.4.30` production notes).

## Examples

### Correlation in Work Item

```csharp
var correlation = Activity.Current?.Id ?? httpContext.TraceIdentifier;
await _queue.QueueBackgroundWorkItemAsync(async (sp, ct) =>
{
    using (LogContext.PushProperty("CorrelationId", correlation))
    {
        var svc = sp.GetRequiredService<IProcessor>();
        await svc.RunAsync(id, ct);
    }
});
```

### Multiple Work Items

Each enqueue is independent; order preserved with **single reader**. If order matters per aggregate, use **partitioned channels** or **sequential keys**.

## Common Mistakes & Anti-patterns

**Singleton service capturing scoped `DbContext`** in closure — still wrong; only resolve inside `work` delegate.

**Unbounded channel** under attack — memory grows; always set **capacity** in production unless proven safe.

**Throwing from controller if queue full** without policy — document whether to return **503** or await (backpressure).

**One long work item blocks** entire queue with single consumer — split stages or add workers.

Forgetting **`stoppingToken`** in work delegate — prevents clean shutdown.

---

## Production & Real-World Notes

Expose **metrics**: queue depth, processing duration, failure rate.

**Retry** with Polly inside work item or move to broker with DLQ.

**Idempotency key** per message — `BackgroundService` may run **at-least-once** if crash after work but before ack (design accordingly).

Health check: flag if queue depth > threshold sustained.

## Comparison / Trade-offs

| Pattern | Durability | Complexity | Backpressure |
|---------|------------|------------|--------------|
| `_ = task` | None | Low | None |
| Channel + hosted | Process | Medium | Bounded channel |
| Outbox + broker | High | High | Broker-defined |

## Quick Reference

- **Singleton queue** + **`BackgroundService`** consumer.
- **`IServiceScopeFactory.CreateAsyncScope()`** per item.
- **`Channel.CreateBounded`** + **`FullMode.Wait`**.
- Capture **IDs**, not scoped instances.
- **Await** work in `try/catch` — observe exceptions.
- For must-deliver → **outbox/broker**.

---

## Key Takeaways

- Channel queue is the **structured** replacement for fire-and-forget in ASP.NET Core.
- **Scoped DI** per work item via **`IServiceScopeFactory`**.
- **Bounded** channels provide **backpressure** under load.
- Consumer **must await** — links to **`UnobservedTaskException`** prevention.
- Single vs multiple consumers trades **simplicity** for **throughput**.
- Process-local — not a substitute for **durable messaging** when required.
