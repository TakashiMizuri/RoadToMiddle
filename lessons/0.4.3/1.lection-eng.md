## Why This Matters

HTTP/1.1 and HTTP/2 from node `0.3` run on **TCP** almost exclusively. When React shows a spinner, the delay is often not JSON parsing — it is **waiting for TCP + TLS** before the first byte of HTTP response. Lesson `0.3.14` (Keep-Alive, connection pooling) assumes you understand that an HTTP "connection" is a **TCP byte stream** reused across requests; closing it forces another three-way handshake and possibly another TLS session.

TCP is the transport protocol that turns IP's datagrams into a **reliable, ordered pipe** between two ports (`0.4.2`). PostgreSQL, Redis, gRPC-over-h2, and SMTP use it for the same reason: applications want to `write()` bytes and `read()` them in order without reinventing loss recovery. Middle developers tune Kestrel, load balancers, and cloud timeouts using TCP concepts — `SYN` retransmits, idle connection drops, `SO_KEEPALIVE`, upstream `connection reset`.

Understanding TCP internals separates superficial "network is slow" from actionable "SYN queue full," "client window zero," or "NAT killing idle flows." That depth is what this lesson builds.

## Core Concepts

### TCP's Contract

**Transmission Control Protocol (TCP)** sits at transport layer (`0.4.1`). It offers:

- **Reliable delivery** — lost segments are retransmitted.
- **Ordered delivery** — bytes arrive in send order at the receiver's application.
- **Duplex byte stream** — bidirectional; either side can send anytime after establishment.
- **Flow control** — receiver can limit how much unacknowledged data the sender buffers.
- **Congestion control** — sender adapts rate to network capacity (shared global behavior).

TCP does **not** guarantee latency or bandwidth — only eventual ordered delivery or connection failure. It is **connection-oriented**: state is maintained per flow (socket pair).

IP below delivers packets independently; TCP reassembles and fills gaps.

### The Three-Way Handshake

Before application data (HTTP, TLS ClientHello), client and server **establish** a TCP connection:

```
Client                              Server
   |                                   |
   | -------- SYN, seq=x ------------> |   (Client → SYN_SENT)
   |                                   |
   | <---- SYN-ACK, seq=y, ack=x+1 --- |   (Server → SYN_RECEIVED)
   |                                   |
   | -------- ACK, ack=y+1 -----------> |   (Both → ESTABLISHED)
   |                                   |
   | ===== application data =======> |
```

**SYN (Synchronize)** — initiates connection; carries initial **sequence number** (ISN), randomly chosen for security (avoid predictable seq attacks).

**SYN-ACK** — server agrees; sends its ISN and acknowledges client's ISN+1.

**ACK** — client confirms server's ISN+1. Both sides now have synchronized sequence space.

Only after ESTABLISHED does Kestrel accept HTTP or TLS bytes. A failed handshake surfaces as browser `ERR_CONNECTION_TIMED_OUT` (no SYN-ACK) or `ECONNREFUSED` (RST in response to SYN — nothing listening on port `0.4.2`).

**Simultaneous open** (both send SYN) is rare in client-server HTTP but supported by TCP state machine.

Connection teardown uses **FIN** four-way close (or RST for abrupt abort). Half-closed states allow one side to finish sending while the other still transmits — relevant for graceful shutdown of long responses.

### Sequence Numbers and Acknowledgments

TCP assigns each byte in the stream a **sequence number**. The **ACK number** means "I have received all bytes up to this number; send from here next."

Sender buffers unacknowledged segments. If ACK doesn't arrive within **RTO (retransmission timeout)**, segment resends. Duplicate ACKs trigger **fast retransmit** in modern TCP when gaps are detected — recovering without waiting full timeout.

This is how HTTP response bodies survive Wi‑Fi packet loss without corrupting JSON — TCP retries at transport layer; HTTP remains unaware except slower response.

**Header fields (conceptual):** Sequence, Acknowledgment, Window (flow control), flags (SYN/FIN/RST/PSH/ACK), checksum.

### Reliability vs IP and UDP

| Aspect | IP | UDP | TCP |
|--------|----|-----|-----|
| Delivery | Best-effort | Best-effort | Reliable (retry) |
| Ordering | Not guaranteed | Not guaranteed | Ordered byte stream |
| Connection | Connectionless | Connectionless | Connection-oriented |
| Overhead | Low header | Low header | Higher (state, ACKs) |

DNS often uses UDP for small queries (`0.4.4`, `0.4.5`); large responses may fall back to TCP. Video/gaming may prefer UDP and accept loss.

### Flow Control — Receive Window

The receiver advertises a **window** size: how many bytes it can accept given its buffer and application read rate. If your ASP.NET app reads slowly from a socket, the window shrinks to zero (**zero window**); sender pauses — **back-pressure** without dropping the connection.

This protects memory on the receiver. It differs from **congestion control**, which protects the network from overload.

Application developers see this indirectly: slow consumers cause upload stalls; blocking reads on server can stall clients.

### Congestion Control (Overview)

When many flows share links, TCP senders infer congestion from **packet loss** and **delay**. Classic algorithms (Reno, CUBIC on Linux) use:

- **Slow start** — exponential increase until threshold.
- **Congestion avoidance** — linear increase.
- **Multiplicative decrease** on loss — reduce sending rate.

Under load, new connections start cautiously — **slow start** adds latency to short HTTP requests unless connection reuse (`0.3.14`) avoids repeated ramp-up. HTTP/3 over QUIC (`0.3.13`) uses different congestion machinery but same problem domain.

### TCP and HTTP from Node 0.3

Typical HTTPS request timeline:

1. **TCP handshake** (~1 RTT if no packet loss).
2. **TLS 1.3 handshake** (~1 RTT, sometimes 0-RTT with session resumption — `0.3.8`).
3. **HTTP request/response** on encrypted TCP stream.

**Keep-Alive:** same TCP connection serves `GET /a` then `GET /b` — one handshake amortized. **Connection pooling** in `HttpClient` reuses sockets to upstream APIs.

**Head-of-line blocking (HTTP/1.1):** one large response blocks subsequent responses on same TCP connection — motivation for HTTP/2 multiplexing on one TCP (`0.3.13`).

Timeouts you configure (Kestrel, reverse proxy, load balancer) often align with TCP idle timers and total handshake + TLS budget.

## Under the Hood

### TCP State Machine (Client Server View)

Key states on Linux/Windows stacks:

| State | Meaning |
|-------|---------|
| LISTEN | Server socket waiting for SYN |
| SYN_SENT | Client sent SYN, awaiting SYN-ACK |
| SYN_RECEIVED | Server sent SYN-ACK, awaiting ACK |
| ESTABLISHED | Data transfer |
| FIN_WAIT_1 / FIN_WAIT_2 | Active close in progress |
| CLOSE_WAIT | Peer sent FIN; app must close |
| TIME_WAIT | Wait 2×MSL after close (avoid stale segments) |
| CLOSED | No connection |

A **half-open** connection occurs when one side crashes without FIN/RST — peer may send data until timeout. **Keepalive probes** detect dead peers.

**TIME_WAIT** on busy servers closing many outbound connections can exhaust ephemeral ports — production tuning uses `SO_REUSEADDR`, connection pooling, or kernel tweaks (careful).

### Segment Structure and Reassembly

TCP segments carry a **sequence number** (first byte in segment), **ACK**, **window**, options (MSS, window scaling, SACK). Maximum Segment Size (**MSS**) negotiated in SYN — typically ~1460 bytes payload on Ethernet (1500 MTU minus headers).

Receiver **reassembly buffer** holds out-of-order segments until gaps fill. Application `read()` consumes contiguous bytes from the front of the stream — HTTP parsers see a continuous byte sequence.

**Nagle's algorithm** coalesces small writes (disabled for low-latency with `TCP_NODELAY` — common in games and some RPC). **Delayed ACK** batches ACKs — trade latency for efficiency.

### Retransmission Logic

RTO computed from smoothed RTT measurements (Jacobson/Kare algorithm). On timeout, segment retransmitted; **exponential backoff** may apply. **Fast retransmit** after three duplicate ACKs suggests single segment loss without waiting RTO.

Persistent timers handle **zero window** — probe with small segments when receiver opens window again.

Wireshark filter `tcp.analysis.retransmission` visualizes loss — invaluable when "API flaky on Wi‑Fi only."

### SYN Backlog and Accept Queue

Server `listen(backlog)` creates **SYN queue** (incomplete handshakes) and **accept queue** (ESTABLISHED waiting for app `accept()`). Under SYN flood or burst traffic, drops happen before your Kestrel code runs — symptoms: clients timeout despite CPU idle.

Mitigations: increase backlog, SYN cookies, rate limiting at load balancer, horizontal scale.

### RST — Reset

**RST** aborts connection immediately — no graceful FIN. Causes: connect to closed port, firewall reject, app crash, TLS mismatch after TCP established. HTTP client sees "connection reset by peer" — check upstream logs, not just controllers.

## Examples

### Example 1: SYN Timeout vs Refused

- **Timeout:** SYN sent repeatedly, no SYN-ACK — wrong IP, firewall drop, server down, routing black hole.
- **Refused:** RST to SYN — port closed on reachable host (`0.4.2`).

Different fixes: firewall rule vs start Kestrel.

### Example 2: Slow API Under Load

Many new TCP connections per request (no Keep-Alive) → each pays handshake + slow start → higher P99 latency. Fix: enable Keep-Alive on reverse proxy and `HttpClient` pooling — transport tuning, not faster JSON serializer.

## Common Mistakes & Anti-patterns

**Assuming TCP guarantees instant delivery.** It guarantees eventual order or failure — latency varies with RTT, loss, congestion.

**Ignoring TIME_WAIT on integration tests** spinning up thousands of short connections to localhost.

**Sync-over-async blocking reads** starving thread pool while TCP window backs up — looks like "network hang."

**Closing load balancer idle timeout below client Keep-Alive** — silent half-open; next request fails mysteriously.

**Confusing TCP connection with HTTP request.** One TCP connection carries many HTTP requests when Keep-Alive enabled.

## Production & Real-World Notes

Load balancers terminate TCP (L4) or pass through. **TLS passthrough** still needs TCP established to backend. **Health checks** often pure TCP connect to port.

Cloud **NAT gateways** track connection tables; idle TCP flows evicted — use keepalive or shorter client refresh for long-lived connections.

Monitor: SYN retransmit rate, active connections, RST rate, RTT percentiles. APM "external call" duration includes TCP+TLS unless broken out.

PostgreSQL and Redis expect long-lived TCP — pool connections at application layer.

## Comparison / Trade-offs

**TCP vs UDP (`0.4.4`):** TCP for correctness (HTTP, DB); UDP when timeliness beats completeness (live video, DNS query speed).

**HTTP/1.1 Keep-Alive vs new connection:** Reuse saves handshake; risk HOL blocking without HTTP/2.

**Kernel TCP vs user-space stacks (QUIC):** HTTP/3 moves reliability to user space over UDP — different deployment and firewall implications (`0.3.13`).

## Quick Reference

| Mechanism | Purpose |
|-----------|---------|
| SYN / SYN-ACK / ACK | Establish connection |
| Sequence + ACK | Ordered reliable delivery |
| Retransmission / RTO | Recover loss |
| Receive window | Flow control (receiver pace) |
| Congestion control | Network protection |
| FIN / RST | Close or abort |
| TIME_WAIT | Safe close, wait 2×MSL |

**HTTPS cold start ≈ 1 RTT TCP + 1 RTT TLS (1.3) before HTTP.**

## Key Takeaways

- TCP three-way handshake synchronizes sequence numbers before any HTTP/TLS bytes.
- Reliability comes from ACKs, retransmission, and ordered reassembly over best-effort IP.
- Flow control uses receive window; congestion control adapts sending rate to network conditions.
- HTTP Keep-Alive and HttpClient pooling reuse TCP to avoid repeated handshake and slow start.
- SYN timeout, RST, and TIME_WAIT have distinct causes and fixes in production.
- Diagnose at transport layer when errors occur before HTTP status codes exist.
