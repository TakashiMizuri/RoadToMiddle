# Polymorphism: virtual/override Runtime Dispatch

> Roadmap: `1.2.7` · Node: `1.2` — C# OOP · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Define **polymorphism** as one interface (base reference), many runtime behaviors (derived implementations).
- Explain how **`virtual`**, **`override`**, and **`abstract`** cooperate in an inheritance chain.
- Trace **virtual method dispatch** from source code through IL **`callvirt`** to vtable slot selection.
- Distinguish **compile-time type** from **runtime type** and predict which method executes.
- Contrast polymorphic **instance methods** with **static**, **private**, and **`new`-hidden** methods.
- Apply polymorphism in realistic abstractions without violating **Liskov Substitution Principle**.

---

## Why This Matters

Lesson `1.2.6` introduced **`override`** versus **`new`**: one participates in polymorphism, the other hides. This lesson answers the deeper question — **how** does the runtime know which implementation to run when you write **`animal.Speak()`** and **`animal`** might be a **`Dog`**, **`Cat`**, or **`Robot`**?

Polymorphism is the payoff of object-oriented design in C#. Payment processors, notification channels, pricing strategies, and ASP.NET middleware pipelines all rely on **substitutable** implementations behind a shared abstraction. Without understanding **runtime dispatch**, you misread code that "looks like" it calls a base method, write **`new`** when you need **`override`**, or break consumers by tightening base behavior in a derived class.

Middle developers debug production issues where the wrong **`ToString`**, **`Equals`**, or **`Handle`** runs because someone held an interface reference but registered the wrong concrete type — or because a method was never virtual. They also benefit from knowing when the JIT **devirtualizes** hot calls. Polymorphism is not magic; it is a **vtable indirection** with clear rules.

## Core Concepts

### Polymorphism in One Sentence

**Polymorphism** means the same operation — a method call on a shared abstraction — executes **different code** depending on the **actual type of the object at runtime**, while the caller uses a **base class or interface reference**.

```csharp
IEnumerable<Animal> zoo = new[] { new Dog(), new Cat() };
foreach (Animal a in zoo)
    a.Speak(); // each iteration: different override
```

The variable **`a`** has compile-time type **`Animal`** (or **`IEnumerable`** element type). Each object has runtime type **`Dog`** or **`Cat`**. The **`Speak`** call resolves to the correct override without **`if (a is Dog)`** branching — that branching moves into the virtual mechanism.

This is **subtype polymorphism**. C# also has **ad hoc polymorphism** (method overloads resolved at compile time) and **parametric polymorphism** (generics — node `1.3`). Do not conflate them.

### virtual, override, and abstract

A method becomes polymorphic when the base declares it **`virtual`** or **`abstract`**:

- **`virtual`** — base provides a default implementation; derived may **`override`**.
- **`abstract`** — base provides no implementation; derived **must override** (unless derived is also abstract).
- **`override`** in derived — replaces the slot in the virtual chain; may call **`base.Method()`**.

```csharp
public abstract class Shape
{
    public abstract double Area();

    public virtual string Describe() => "A shape";
}

public class Circle : Shape
{
    private readonly double _radius;
    public Circle(double radius) => _radius = radius;

    public override double Area() => Math.PI * _radius * _radius;

    public override string Describe() => $"Circle r={_radius}";
}
```

**`Area`** is pure polymorphic contract. **`Describe`** is virtual with a default that **`Circle`** specializes. Code holding **`Shape s = new Circle(5)`** calls **`s.Area()`** and gets circle math without casting.

**`virtual`** methods can be **`sealed override`** in intermediate classes (`1.2.6`) to stop further customization.

### Compile-Time Type vs Runtime Type

Every reference carries two type notions:

| Concept | Meaning | Example |
|---------|---------|---------|
| Compile-time (static) type | Type the compiler uses for overload resolution, member visibility | **`Shape s = ...`** → **`Shape`** |
| Runtime type | Actual object on the heap | **`new Circle(5)`** → **`Circle`** |

Polymorphic **virtual** calls use **runtime type** to pick the implementation. **Non-virtual** calls, **`new`-hidden** members, **`static`** methods, and **`private`** methods use **compile-time type** (or exact declaring type).

```csharp
Shape s = new Circle(3);
s.Area();       // Circle.Area — runtime type
s.ToString();   // Circle.ToString unless not overridden — still virtual on object
```

Casting **`((Circle)s).Area()`** does not change dispatch for virtual methods — it already used runtime type. Casting is for accessing members not on the static type.

### What Is Not Polymorphic

Understanding boundaries prevents false expectations:

**Static methods** — no **`virtual`/`override`**; **`call`** to declaring type. Hiding with **`new`** on static is rare and confusing — avoid.

**Private methods** — not part of inheritance surface; no override.

**Struct instance methods** — default **non-virtual**; **`virtual`** on structs forbidden except explicit interface implementation patterns. Struct "polymorphism" goes through **boxing** to interface (`1.2.14`).

**`new`-hidden instance methods** — static dispatch on reference type (`1.2.6`).

**Default interface methods** — special rules (`1.2.10`); dispatch through interface reference differs slightly from class hierarchy.

### Liskov Substitution Principle (LSP)

Polymorphism implies a **contract**: derived types must be **substitutable** for their base without breaking callers. If **`Bird.Fly()`** throws for **`Penguin`**, code iterating **`Bird`** fails unexpectedly. LSP is not a syntax rule — it is a design discipline. Valid **`override`** signatures can still violate LSP through stricter preconditions, weakened postconditions, or thrown **`NotSupportedException`** in "impossible" overrides.

Production teams document which virtual hooks are safe to override and which base assumptions must hold. Framework **`OnModelCreating`**, **`OnActionExecuting`**, and **`Dispose(bool)`** are extension points with documented contracts.

## Under the Hood

### Vtables and callvirt

Reference types with **any virtual methods** (including inherited from **`object`**: **`ToString`**, **`Equals`**, **`GetHashCode`**) carry a **method table pointer**. Each virtual method occupies a **slot** indexed consistently along the inheritance chain. **`Dog`** overriding **`Speak`** updates **`Dog`'s** table slot to point to **`Dog.Speak`**.

A call **`animal.Speak()`** compiles to **`callvirt`** (for instance virtual methods):

```
1. Load reference (animal)
2. callvirt instance void Animal::Speak()
   → JIT loads method table from object header
   → reads slot for Speak
   → jumps to actual code (Dog.Speak, Cat.Speak, ...)
```

**`callvirt`** also performs a **null check** — calling virtual on null throws **`NullReferenceException`** before dispatch. Non-virtual **`call`** skips that pattern for the dispatch path (still may null-check separately).

**`object.GetType()`** returns **`RuntimeType`** — the runtime type — useful for logging, not for replacing polymorphism with type switches in business logic.

### Override Metadata Chain

The compiler builds an **override chain** in metadata: **`Dog.Speak`** overrides **`Animal.Speak`**. Reflection and debuggers walk this chain. **`abstract`** roots have **`RVA`** absent until concrete override supplies IL body.

Interface methods on classes use **implicit or explicit** implementation (`1.2.9`); their slots map through **interface maps** on the object — separate from class vtable but same dispatch idea when calling through **`IEnumerable`**.

### Devirtualization and sealed

The JIT may **devirtualize** calls when it proves runtime type — e.g. **`sealed`** class, **` sealed override`**, or local analysis. **`sealed`** on **`string`** helps hot **`ToString`** paths. Do not seal everything prematurely; measure when needed.

### Covariance of Return Types (C# 9+)

Derived **`override`** may return a **more derived type** (**covariant return**):

```csharp
public class Factory
{
    public virtual Animal Create() => new Animal();
}

public class DogFactory : Factory
{
    public override Dog Create() => new Dog(); // OK — Dog is Animal
}
```

Dispatch rules unchanged; callers see **`Animal`** from base signature but may receive **`Dog`** instance.

## Syntax / API

```csharp
public virtual void M() { }      // allow override
public abstract void M();        // must override in concrete derived
public override void M() { }     // replace virtual/abstract
public sealed override void M() { } // final override
```

Interfaces (pre–default methods): all instance members implicitly abstract until implemented.

## Examples

### Example 1: Strategy Without switch

```csharp
public interface IPricingStrategy
{
    decimal GetPrice(decimal basePrice);
}

public class StandardPricing : IPricingStrategy
{
    public decimal GetPrice(decimal basePrice) => basePrice;
}

public class VipPricing : IPricingStrategy
{
    public decimal GetPrice(decimal basePrice) => basePrice * 0.85m;
}

public class CheckoutService(IPricingStrategy pricing)
{
    public decimal Total(decimal amount) => pricing.GetPrice(amount);
}
```

**`CheckoutService`** is polymorphic over **`IPricingStrategy`** — interface dispatch, same mental model as virtual classes.

### Example 2: Template Method Pattern

```csharp
public abstract class DataExporter
{
    public void Export()
    {
        var data = LoadData();   // abstract — polymorphic
        WriteHeader();
        WriteRows(data);
        WriteFooter();
    }

    protected abstract IEnumerable<string> LoadData();
    protected virtual void WriteHeader() { }
    protected virtual void WriteRows(IEnumerable<string> rows) { /* default */ }
    protected virtual void WriteFooter() { }
}
```

**`Export`** is non-virtual **orchestration** in base; hooks are virtual/abstract for derived specialization — classic **Template Method**, relies entirely on runtime dispatch in hooks.

### Example 3: Debugging Dispatch

```csharp
Animal a = GetFromFactory(); // could be Dog or Cat
Console.WriteLine(a.GetType().Name);
a.Speak();
```

Log **runtime type** when behavior surprises; compare to **compile-time type** of **`a`**. Mismatch often explains "wrong" method if hiding or non-virtual involved.

## Common Mistakes & Anti-patterns

**Replacing polymorphism with long if/else on type.** **`is`/`switch` on types** for behavior belongs in factory; repeated in business logic violates Open/Closed Principle.

**Forgetting to mark base `virtual`.** Derived **`override`** fails; accidental **`new`** hides — silent bug through base references.

**Overriding and breaking LSP.** **`Square : Rectangle`** with settable width/height is a famous modeling failure — override **`SetWidth`** breaks rectangle invariant for square substituted as rectangle.

**Calling virtual methods from constructors.** Derived override runs before derived fields initialized (`1.2.6`).

**Assuming structs polymorph like classes without interface/boxing.**

## Production & Real-World Notes

ASP.NET Core **`Controller`** actions are not classic virtual polymorphism on each request, but **middleware** and **`AuthorizationHandler`** hierarchies are. EF Core **`ValueConverter`**, logging **`ILogger`**, and **`HttpMessageHandler`** chains use interface/class polymorphism extensively.

Unit tests mock interfaces with Moq — polymorphism enables substituting fakes without changing system under test. Understanding dispatch clarifies why **`virtual`** members need **`Mock<>.Setup`** on the right declared member.

## Comparison / Trade-offs

| Mechanism | Dispatch | When |
|-----------|----------|------|
| `virtual`/`override` | Runtime (callvirt) | Extensible class hierarchies |
| `new` hide | Compile-time | Rare; versioning edge cases |
| Interface call | Runtime (interface map) | Multiple contracts, DI |
| overload | Compile-time | Same name, different signatures |
| `switch` on type | Manual | Factory, serialization — not repeated domain logic |

**Polymorphism vs generics.** Generics choose code at compile time for type parameters (with JIT specialization); virtual chooses at runtime per object. Combine: **`Repository<T>`** with **`IEntity`** polymorphic instances.

## Quick Reference

- **Polymorphism** = base reference, derived behavior at runtime.
- **`virtual`/`abstract` + `override`** = polymorphic instance methods.
- **`callvirt`** + vtable slot = dispatch mechanism.
- **Compile-time type** → visibility, overloads; **runtime type** → virtual target.
- **LSP** = substitutability obligation, not compiler-enforced.
- **Not polymorphic**: static, private, struct default, `new` hide.

---

## Key Takeaways

- Polymorphism eliminates type-switch behavior for extensible designs.
- Runtime dispatch follows **actual object type** for virtual/abstract overrides.
- IL **`callvirt`** implements virtual calls with vtable lookup and null check.
- **`abstract`** forces derived implementations; **`virtual`** provides overridable default.
- Interface references dispatch similarly through interface method mapping.
- Good polymorphism respects **LSP**; bad overrides compile yet break callers.
