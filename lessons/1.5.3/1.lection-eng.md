After this lesson you will be able to:

- Explain the **manifest** — assembly name, version, culture, and references.
- Describe **metadata** as the CLR's embedded type system (types, methods, fields, attributes).
- Distinguish **assembly** from **namespace** and from **NuGet package**.

## Why This Matters

In `1.5.1` you learned that the CLR loads **assemblies** containing IL and metadata. In `1.5.2` the JIT reads method bodies and metadata tokens to generate native code. Without understanding what sits inside that `.dll` file, versioning conflicts (`FileLoadException`), missing dependencies, and reflection failures look arbitrary.

Every `dotnet publish` output, every NuGet dependency graph, and every `[Authorize]` attribute resolved at runtime flows through **metadata**. Middle developers read assembly names in fusion logs, fix binding issues, and know why **internalsVisibleTo** is an assembly-level attribute — not a namespace trick.

## Core Concepts

### What Is an Assembly?

An **assembly** is the smallest deployable unit that the CLR version-resolves and loads. Typically one `.dll` or `.exe` per project output (multi-target builds emit per-RID binaries). It contains:

1. **PE file** wrapper (Windows portable executable format; same container on Linux/macOS with appropriate loader).
2. **CLR header** pointing to metadata and IL.
3. **IL streams** — method bodies (`1.5.1`).
4. **Metadata tables** — type system database.
5. **Manifest** — identity and assembly references.
6. **Optional resources** — embedded `.resx` strings, icons, WPF assets.

Multiple **namespaces** can live in one assembly; one namespace can span assemblies. **Namespace** organizes source code; **assembly** organizes deployment and loading.

### The Manifest

The **assembly manifest** is metadata table `#~` (Assembly row) plus **AssemblyRef** rows. It answers:

- **Who am I?** Name (`MyApp.Api`), version (`1.0.0.0`), culture (`neutral`), public key token (if strong-named).
- **What do I depend on?** References to `System.Runtime`, `Microsoft.AspNetCore.Mvc`, etc.
- **Where are types?** Entry point (for executables), module list.

At load time, the loader walks **AssemblyRef** and loads dependencies recursively (respecting `AssemblyLoadContext` policy in modern .NET).

Version mismatch example: app compiled against `PackageA 2.0` but `1.0` on disk → `FileLoadException` or `MissingMethodException` if API changed.

### Metadata Tables

Metadata is stored in normalized **tables** linked by indexes (ECMA-335):

| Table (examples) | Contents |
|------------------|----------|
| TypeDef | Types defined in this module |
| MethodDef | Methods, flags, IL RVA |
| FieldDef | Fields, offsets (for layout) |
| MemberRef | References to external methods/fields |
| TypeRef | External types |
| CustomAttribute | `[Serializable]`, `[HttpGet]`, etc. |
| Property, Event | Properties and events |

The JIT uses **MethodDef** → IL RVA; the GC uses **layout information** from metadata and compiler-emitted **GC maps**; serializers use **property names** from metadata unless overridden.

**Tokens** in IL (`0x06000001`) index into these tables — `call` instructions reference MethodDef/MemberRef.

### Resources

**Embedded resources** (.resources binary format) store localized strings, images, default configs compiled from `.resx`. **Satellite assemblies** (`fr/MyApp.resources.dll`) hold culture-specific resources; manifest culture `neutral` on main assembly.

ASP.NET Core often uses JSON/appsettings instead of `.resx`, but libraries still embed resources; tools like `dotnet resource` inspect them.

### Reflection — Reading Metadata at Runtime

```csharp
var asm = typeof(Program).Assembly;
Console.WriteLine(asm.FullName);

foreach (var type in asm.GetExportedTypes())
    Console.WriteLine(type.FullName);

var method = typeof(string).GetMethod("Concat", new[] { typeof(string), typeof(string) });
Console.WriteLine(method?.MetadataToken.ToString("X8"));
```

Reflection walks the same tables the JIT uses — slower, for discovery (DI registration, OpenAPI, mapping) not hot paths.

## Under the Hood

### PE + CLI Header Structure

A managed `.dll` is a **PE file** with an extra **CLI metadata root**. Native tools (`dumpbin /headers`, `dotnet file`) show **COR20 header** (legacy name) pointing to `#~` heap. The loader validates PE, maps sections, then initializes the CLR which parses metadata into runtime structures (`MethodTable`, `EEClass` — internal names from `1.5.1`).

**Multi-module assemblies** (rare) had multiple files one manifest — essentially obsolete; single-file publish bundles differently.

### Strong Names and Identity

**Strong naming** signs an assembly with a key pair — unique identity for GAC-era binding. Modern apps rely on **NuGet version** + `AssemblyLoadContext`. **PublicKeyToken** still appears in fusion logs and `InternalsVisibleTo` friend declarations.

### Single-File and Bundling

**.NET single-file publish** extracts assemblies to temp or loads from memory — logical assembly model unchanged; physical file layout differs. Metadata still drives loading.

## Syntax / Commands / API

```bash
# Assembly identity
dotnet list package
ildasm /text MyApp.dll | findstr ".assembly"

# Reflection in code
Assembly.Load("MyLib, Version=1.0.0.0, Culture=neutral, PublicKeyToken=...");
typeof(Foo).Assembly.GetName().Version
```

Common attributes on assembly (in `AssemblyInfo` or `Properties/AssemblyInfo.cs` or `[assembly: ...]` in any file):

```csharp
[assembly: AssemblyVersion("1.0.0.0")]
[assembly: InternalsVisibleTo("MyTests")]
```

## Examples

### Dependency Chain

```
MyWebApp.dll
  ├── references Microsoft.AspNetCore.Mvc.dll
  │     └── references System.Runtime.dll
  └── references MyDomain.dll
```

Loader starts at `MyWebApp`, reads manifest AssemblyRefs, loads tree. Missing intermediate package → load failure at first use of type from that assembly.

### Metadata Drives JSON Property Names

System.Text.Json can use `[JsonPropertyName("id")]` — stored as **CustomAttribute** on property metadata. Serializer reads attributes via reflection or source generator from compile-time metadata.

## Common Mistakes & Anti-patterns

**Confusing namespace with assembly** — `using System.Collections.Generic` does not tell you which assembly provides a type; `List<T>` is in `System.Collections`.

**Ignoring version in binding** — "same DLL name" ≠ same assembly if version/public key differs.

**Reflection in hot loops** — metadata access is cached after first `GetMethod`, but unoptimized patterns hurt; prefer delegates or source generators.

---

## Production & Real-World Notes

**Central Package Management** (`Directory.Packages.props`) aligns versions across projects — reduces manifest mismatch.

**Trimming / Native AOT** — metadata pruned when unused; reflection-only code paths break unless `[DynamicallyAccessedMembers]` annotated — critical for library authors.

**AssemblyLoadContext** isolation in plugin hosts — unload collectible contexts when metadata/IL no longer referenced.

## Comparison / Trade-offs

| Concept | Scope | Purpose |
|---------|-------|---------|
| Namespace | Source organization | Avoid name collisions |
| Assembly | Deploy/load unit | CLR resolution, versioning |
| NuGet package | Distribution | May contain multiple assemblies |
| Module | Single `.dll` file | One manifest per module typically |

## Key Takeaways

- **Assembly** = IL + metadata + manifest (+ resources).
- **Manifest** names the assembly and lists dependencies.
- **Metadata** is the CLR type database for JIT, GC, attributes, reflection.
- **Namespace ≠ assembly**; know both when debugging load errors.
- **`1.5.1`/`1.5.2`** depend on metadata tokens and method defs.
