# Pattern matching: switch, property, list, relational

> Roadmap: `1.3.13` · Узел: `1.3` — C#: продвинутые темы · Глубина: практика

## Learning Objectives

После этого урока ты сможешь:

- Заменять длинные цепочки `if/else` на **switch expressions** и **switch statements** с pattern matching.
- Сопоставлять **типы**, **константы**, **свойства** и **relational conditions** в одном `case`.
- Использовать **list patterns** (`[a, b, ..]`) для декомпозиции последовательностей.
- Применять **`when` guards** и комбинаторы **`or`/`and`/`not`** без потери читаемости.
- Осознанно выбирать между **exhaustiveness**, **discard `_`** и default-ветками.
- Рефакторить legacy control flow в поддерживаемый pattern-based код.

## Why This Matters

До C# 7 `switch` работал только с дискретными константами — в основном `int` и `enum`. Бизнес-логика превращалась в вложенные `if (x is Foo f && f.Status == Active)` или длинные `else if`, где дублировались проверки типа и чтение свойств. Код был корректным, но хрупким: добавил новый derived type — забыл ветку; переименовал property — сломались три блока.

Современный **pattern matching** превращает `switch` в **структурный классификатор**: ты описываешь *форму* данных (тип, свойства, длина коллекции, числовой диапазон) и одновременно **bind** переменные. Компилятор может предупредить о неполном `switch` (exhaustiveness для enum с analyzers). В LINQ и ASP.NET это ежедневная практика — маппинг HTTP results, routing domain events, разбор API payloads.

Pattern matching — не синтаксический сахар ради красоты. Он кодирует **таблицы решений** читаемо и держит **binding** рядом с **условием**, что снижает количество багов.

## Core Concepts

### От классического switch к patterns

Классический switch сравнивает значение с константами:

```csharp
switch (status)
{
    case OrderStatus.Pending: ...
    case OrderStatus.Shipped: ...
    default: ...
}
```

Pattern matching обобщает метку `case` до **pattern** — шаблона, который может совпасть и **ввести переменные**:

```csharp
var message = status switch
{
    OrderStatus.Pending => "Waiting",
    OrderStatus.Shipped => "On the way",
    _ => "Unknown"
};
```

**Switch expression** (`value switch { ... }`) возвращает значение; **switch statement** выполняет statements. Синтаксис patterns общий.

### Type patterns и declaration patterns

**Type pattern** проверяет runtime type и делает bind:

```csharp
object input = GetPayload();
if (input is string s)
    Console.WriteLine(s.Length);
```

В switch:

```csharp
var result = input switch
{
    string s => $"text:{s}",
    int n => $"number:{n}",
    _ => "other"
};
```

**Declaration pattern** объединяет тип и имя: `CaseType variable`. Переменная в scope только в этой ветке. Это замена `(input as string) != null`.

### Property patterns

Можно сопоставлять **вложенную структуру** без ручных null checks:

```csharp
var discount = order switch
{
    { Total: > 1000, Customer.IsVip: true } => 0.15m,
    { Total: > 1000 } => 0.10m,
    { Customer.IsVip: true } => 0.05m,
    _ => 0m
};
```

Каждый `{ Property: pattern }` — **property pattern**. `Total: > 1000` — **relational pattern** на значение свойства. Компилятор генерирует null-safe доступ (short-circuit при null посередине цепочки).

Property patterns удобны для **DTO**, **records** и **anonymous types** в тестах — проверяешь структуру, а не последовательность getters.

### Relational patterns

Relational patterns сравнивают с константами через `<`, `>`, `<=`, `>=`:

```csharp
var band = score switch
{
    >= 90 => "A",
    >= 80 => "B",
    >= 70 => "C",
    _ => "F"
};
```

Порядок важен: побеждает **первая** подошедшая ветка. Это замена `if (score >= 90) ... else if ...`.

Комбинация с type patterns:

```csharp
var label = value switch
{
    int n and >= 0 and < 100 => "valid percent",
    int n and < 0 => "negative",
    _ => "not a percent"
};
```

### Logical patterns: and, or, not

Комбинаторы C# 9+:

- `p1 and p2` — оба должны совпасть (на одном input).
- `p1 or p2` — любой из них.
- `not p` — отрицание.

Пример:

```csharp
var kind = obj switch
{
    string s and ("" or " ") => "blank",
    string s => "text",
    int i and > 0 => "positive",
    not null => "something",
    _ => "null"
};
```

Комбинаторы — умеренно. Глубокая вложенность `and`/`or` ухудшает читаемость; лучше property patterns или local function.

### List patterns (C# 11)

**List patterns** работают с массивами, `List<T>`, любыми типами с `Count`/`Length` и indexer:

```csharp
var summary = numbers switch
{
    [] => "empty",
    [var only] => $"single:{only}",
    [var first, var second] => $"pair:{first},{second}",
    [var head, .. var tail] => $"head:{head}, rest:{tail.Count}",
};
```

- `[]` — пустая коллекция.
- `[a, b, ..]` — минимум два элемента; `..` — **rest pattern**.
- `[.., var last]` — захват последнего.

Подходят для **routing по arity аргументов**, **результатов parse**, **коротких CLI-команд** без ручного `args[0]`, `args.Length`.

### when Guards

Дополнительное условие через **`when`**:

```csharp
switch (order)
{
    case { Total: var t } when t > 10_000:
        RequireApproval();
        break;
    case { Total: var t }:
        AutoApprove();
        break;
}
```

Guards не вводят bindings — они фильтруют уже совпавший pattern. Частые `when` часто означают, что pattern стоит уточнить.

### Discard и exhaustiveness

`_` — **discard pattern**, ловит всё не пойманное раньше. Для enum включай analyzer **CS8509** (non-exhaustive switch).

У records с positional syntax работают **positional patterns**:

```csharp
public record Point(int X, int Y);

var quadrant = point switch
{
    ( > 0, > 0) => "Q1",
    ( < 0, > 0) => "Q2",
    ( < 0, < 0) => "Q3",
    ( > 0, < 0) => "Q4",
    (0, 0) => "origin",
    _ => "on axis"
};
```

Positional patterns — по сути property patterns для records.

## Under the Hood

Pattern matching компилируется в **decision tree**, а не в наивную цепочку `is`. Roslyn строит **pattern graph**, упорядочивает проверки (часто сначала дешёвые type checks, потом property fetches) и может переиспользовать **temps** для повторных обращений к свойствам.

На уровне IL switch expression часто становится **`switch` opcode** (для плотных int/enum) или последовательностью **conditional branches** с **`isinst`/`castclass`** для type patterns. Property patterns вызывают getters; компилятор вставляет **null checks**.

List patterns превращаются в **проверку длины** + **indexer** + рекурсивный match хвоста через slice. LINQ на runtime не участвует — для типичного array/list без лишних аллокаций.

Switch expressions должны быть **exhaustive** или иметь discard; компилятор это контролирует для `bool` и некоторых enum в зависимости от настроек.

## Syntax / Commands / API

| Construct | Example |
|-----------|---------|
| Switch expression | `x switch { 1 => "a", _ => "b" }` |
| Type pattern | `case string s:` |
| Property pattern | `case { Id: > 0, Name: not null }:` |
| Relational | `case >= 18 and <= 65:` |
| List pattern | `case [var h, ..]:` |
| Combinator | `string s and not ""` |
| Guard | `case int n when n % 2 == 0:` |

## Examples

### Маппинг HTTP results (ASP.NET)

```csharp
IResult MapError(Exception ex) => ex switch
{
    ValidationException ve => Results.ValidationProblem(ve.Errors),
    NotFoundException nf => Results.NotFound(nf.Message),
    UnauthorizedAccessException => Results.Unauthorized(),
    _ => Results.Problem(ex.Message)
};
```

Новый тип exception — одна строка в switch.

### Разбор heterogeneous JSON nodes

```csharp
string Describe(object? node) => node switch
{
    null => "null",
    bool b => b ? "true" : "false",
    string s => $"\"{s}\"",
    int or long or double => "number",
    IList list => $"array[{list.Count}]",
    _ => "object"
};
```

### Command arity router

```csharp
bool TryParseCommand(string[] args, out Command cmd) =>
    args switch
    {
        ["create", var name] => Success(new CreateCommand(name), out cmd),
        ["delete", var id] when int.TryParse(id, out _) => Success(new DeleteCommand(id), out cmd),
        ["list"] => Success(ListCommand.Instance, out cmd),
        _ => Fail(out cmd)
    };
```

List patterns заменяют ручной `args.Length` switch.

## Common Mistakes & Anti-patterns

**Неверный порядок relational arms.** `>= 0` перед `>= 90` поглощает все scores. Ставь **самые специфичные** (высокий порог) первыми.

**Switch вместо каждого if.** Две ветки — оставь `if`. Patterns выигрывают при **3+ ветках** или **структурных** условиях.

**Side effects в switch expression.** Expressions должны быть **pure**; для императивной логики — switch **statement**.

**Non-exhaustive enum switch** без `_` — сюрприз в runtime при новом значении enum из API.

**List pattern без `..`** — `[a,b,c]` только ровно три элемента; для «минимум два» используй `[a,b,..]`.

## Production & Real-World Notes

Команды активно используют switch expressions в mapping layers (domain → DTO, exception → ProblemDetails). Analyzers + nullable context делают property patterns безопасными на partial API models.

На **high-frequency** путях (миллионы сообщений/сек) иногда выгоднее span-based parser или state machine — clarity patterns важнее micro-optimization, пока profiling не сказал иное.

Code review: порядок arms задокументирован? Новые enum values обработаны? Guards не дублируют pattern?

## Comparison / Trade-offs

| Подход | Плюсы | Минусы |
|--------|-------|--------|
| if/else | Привычно, произвольная логика | Verbose, дублирование casts |
| Classic switch | Быстро для int/enum | Без структуры |
| Pattern switch | Читаемая структура, binding | Кривая обучения, порядок arms |
| Polymorphism | Open extension (OOP) | Нужна иерархия типов |

**Virtual dispatch** — когда поведение принадлежит типу; **pattern matching** — когда поведение внешнее (functional style, cross-cutting mappers).

## Quick Reference

```
value switch { pattern => result, _ => default }

Patterns: constant | type var | property { A: pat } | relational >= n
          list [a, b, ..] | and | or | not | discard _

Statement: switch (e) { case pat when cond: ... break; }
```

## Key Takeaways

- Pattern matching классифицирует **данные по форме**, не только по скалярному значению.
- **Switch expressions** возвращают значение; держи их без side effects.
- **Property** и **relational** patterns убирают boilerplate null-check + compare.
- **List patterns** выражают arity и head/tail декомпозицию.
- **Порядок arms** критичен для relational и first-match semantics.
- Компилятор строит decision trees; list patterns без LINQ allocations.
- Patterns для mappers; polymorphism для intrinsic type behavior.
