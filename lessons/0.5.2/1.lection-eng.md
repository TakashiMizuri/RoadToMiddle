# Semantic Layout: header, nav, main, article, section, aside, footer

> Roadmap: `0.5.2` · Node: `0.5` — HTML · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Choose **semantic landmarks** instead of generic `<div>` wrappers for page regions.
- Structure a typical site with **`header`**, **`nav`**, **`main`**, **`footer`**, and optional **`aside`**.
- Distinguish **`article`** vs **`section`** and nest them without breaking outline logic.
- Explain how landmarks help **screen readers** and keyboard users skip repetitive chrome.
- Avoid duplicate **`main`** regions and orphaned navigation blocks.
- Sketch a blog, docs, and dashboard layout using the same vocabulary.

---

## Why This Matters

In lesson `0.5.1` you built the document envelope — DOCTYPE, charset, lang. The next question is: *what lives inside `<body>`?* For years developers sliced pages with `<div id="header">` and `<div class="sidebar">`. Browsers rendered them fine. CSS Grid could position them. But the **meaning** of each region was locked in class names only humans (sometimes) read.

**Semantic HTML5 sectioning elements** expose structure to the platform: assistive technology can list "navigation", "main content", "complementary" regions; SEO crawlers infer page focus; your teammates read markup without opening CSS. A Middle developer shipping React components still outputs HTML in the DOM — `<header>` in JSX is the same contract as in a static page.

Getting landmarks wrong is a common accessibility failure: two `<main>` elements, every `<section>` without a heading, navigation buried in `<div>` soup so screen reader users cannot jump to links. Correct semantics are cheap to add at authoring time and expensive to retrofit after a SPA grows fifty routes.

## Core Concepts

### Landmarks: Named Regions of the Page

**Landmarks** are major areas users recognize: branding header, global nav, primary content, related sidebar, footer with legal links. HTML5 gives dedicated elements instead of anonymous divs:

| Element | Typical role |
|---------|----------------|
| `header` | Introductory content for its section or page — logo, title, sometimes nav |
| `nav` | Block of **navigation links** — primary menu, breadcrumbs, table of contents |
| `main` | **Unique** primary content of the document — one per page |
| `article` | Self-contained composition — blog post, card, comment thread |
| `section` | Thematic grouping **with a heading** inside a document or article |
| `aside` | Tangentially related content — sidebar, pull quote, ads |
| `footer` | Footer for its nearest sectioning ancestor — copyright, meta links |

These elements do not replace CSS layout. They **label** regions. You still use Flexbox or Grid (`0.6.x`) for visual placement.

### header and footer: Scoped to Context

`<header>` and `<footer>` are not global singletons like `<main>`. An `<article>` can have its own `<header>` with the post title and author, and its own `<footer>` with tags and share links. The page-level `<header>` often wraps site logo and top navigation; the page-level `<footer>` holds site-wide copyright.

Confusion arises when juniors put the entire page chrome inside one giant `<header>`. If primary content sits inside `<header>`, assistive tech may treat your article as "introductory" rather than main content. Rule of thumb: page `<header>` = top bar; **`main`** holds what the user came for.

### nav: Links That Move You Around

Use `<nav>` when the primary purpose of the block is **navigation** — a list of links to other pages or in-page anchors. Not every group of links needs `<nav>`: a footer with three legal links might stay in `<footer>` without a separate `<nav>`; a primary site menu belongs in `<nav>`.

Multiple `<nav>` elements are valid: one for global menu, one for in-article table of contents. Label secondary nav with `aria-label` when needed (`0.5.11`):

```html
<nav aria-label="Primary">
  <ul>...</ul>
</nav>
```

### main: One Primary Story Per Page

There must be **exactly one** visible `<main>` per document (excluding hidden templates). Everything unique to this URL's purpose — article body, product details, dashboard widgets — belongs here. Repeated site chrome (logo bar, global nav) stays **outside** `<main>`.

In SPAs, ensure each route renders one logical main region; do not nest `<main>` inside `<main>`. React Router layouts often wrap `{children}` in `<main>` once at the layout level — a solid pattern.

### article vs section: Independence vs Grouping

An **`article`** is **self-contained**: understandable on its own if syndicated in an RSS reader or shared as a link. Blog post, news item, forum post, product card in a list — each can be `<article>`.

A **`section`** is a **thematic group** within a larger whole, almost always with a **heading** (`h2`–`h6`). "Chapter 3: Deployment" on a docs page is a `<section>`. A `<section>` without a heading is usually a smell — use `<div>` if you only need a styling hook.

Relationship: a blog index page might have `<main>` containing multiple `<article>` elements (previews). Each full post page has `<main>` with one `<article>` containing several `<section>` elements (Introduction, Steps, Conclusion).

When unsure: ask "Would this make sense alone in an RSS feed?" Yes → `article`. No, it's a chapter inside one doc → `section`.

### aside: Related but Not Central

`<aside>` holds content **related** to nearby content but not part of the primary flow: author bio, related links, glossary, ad slot. On a news article, comments might be `<section>`; a "Trending topics" box is `<aside>`.

Do not put the entire primary article in `<aside>` because it looks like a sidebar in CSS — semantics follow **purpose**, not visual column.

## Under the Hood

Browsers map these elements to **implicit ARIA landmark roles** (with nuances): `banner` for header at body level, `navigation` for nav, `main`, `complementary` for aside, `contentinfo` for footer at body level. Screen readers expose landmark shortcuts (NVDA/JAWS/VoiceOver) to jump between them.

The **document outline** algorithm changed over HTML5 history; do not rely on automatic heading outlines alone for SEO. Still use headings inside sections for human and AT structure (`0.5.3`). Validators warn on multiple `<main>` elements or missing `<main>` on long pages.

Parsing never requires these tags — a div-soup page loads — but you lose platform affordances and fail audits (axe, Lighthouse).

## Syntax / Commands / API

Typical blog post page:

```html
<body>
  <header>
    <p><a href="/">Site Name</a></p>
    <nav aria-label="Primary">
      <ul>
        <li><a href="/blog">Blog</a></li>
        <li><a href="/about">About</a></li>
      </ul>
    </nav>
  </header>

  <main>
    <article>
      <header>
        <h1>Post Title</h1>
        <p>Published <time datetime="2026-07-03">July 3, 2026</time></p>
      </header>
      <section>
        <h2>Introduction</h2>
        <p>...</p>
      </section>
      <footer>
        <p>Tags: html, semantics</p>
      </footer>
    </article>
  </main>

  <aside>
    <h2>Related posts</h2>
    <ul>...</ul>
  </aside>

  <footer>
    <p>&copy; 2026 Company</p>
  </footer>
</body>
```

Docs landing with multiple articles in main:

```html
<main>
  <h1>Documentation</h1>
  <article>
    <h2><a href="/docs/install">Installation</a></h2>
    <p>Short teaser...</p>
  </article>
  <article>
    <h2><a href="/docs/config">Configuration</a></h2>
    <p>Short teaser...</p>
  </article>
</main>
```

## Examples

### Example 1: Product page

Primary content is the product — one `<main>`, one `<article>` (self-contained product listing). Specifications split into `<section>` with headings. "Customers also bought" is `<aside>`. Site nav stays in top `<header>` outside main.

### Example 2: Dashboard (conceptual)

`<main>` wraps charts and tables unique to this view. App shell sidebar links might live in `<nav>` outside main; contextual help panel in `<aside>`. Do not wrap every widget in `<article>` unless each widget is independently syndicable — usually `<section>` with headings is enough.

---

## Common Mistakes & Anti-patterns

**Two `<main>` elements** — Fails accessibility audits; confuses landmark navigation. Keep one per page view.

**`<section>` without heading** — Use `<div>` for styling-only wrappers; sections need titles for outline clarity.

**Everything is `<article>`** — Cards in a feed yes; arbitrary layout columns no.

**Navigation only as styled `<div>`** — Loses navigation landmark; hurts keyboard/AT users.

**Primary content inside `<header>`** — Main story should be in `<main>`, not introductory landmark.

---

## Production & Real-World Notes

Design systems (Material, Bootstrap) use class names; you still choose semantic tags in components. `<MainLayout>` in React should render literal `<main>`. Storybook stories can axe-check landmarks.

Micro-frontends embedding fragments must not each ship a full `<main>` — coordinate with shell team. Email HTML often ignores these tags; web apps should not.

Lighthouse "Document does not have a main landmark" is a quick win for a11y score.

---

## Comparison / Trade-offs

`<div>` + ARIA `role="navigation"` works but prefer native elements first (**first rule of ARIA**). `<section>` vs `<article>` vs `<div>`: when semantics unclear, `<div>` with good headings beats wrong sectioning. React `<Fragment>` does not add nodes — landmarks must be real elements in DOM output.

## Quick Reference

| Element | One per page? | Needs heading? |
|---------|---------------|----------------|
| `main` | Yes (one) | No (but often contains h1) |
| `nav` | Multiple OK | No |
| `article` | Multiple OK | Usually has title/h1 |
| `section` | Multiple OK | Should have heading |
| `aside` | Multiple OK | Often has h2 |
| `header`/`footer` | Multiple OK | No |

---

## Key Takeaways

- Semantic landmarks describe **purpose**, not visual layout.
- Exactly one `<main>` per page view; site chrome outside it.
- `<article>` = standalone; `<section>` = themed chapter with heading.
- `<nav>` for navigation blocks; label duplicates with `aria-label`.
- `<aside>` for tangential content; header/footer can repeat per article.
