# Merge vs rebase — краткая выжимка

> `0.2.4` · ~10 мин на повторение

## Главная идея

**Merge** сохраняет реальный graph; **rebase** переписывает feature для linear story. Выбор — trade-off, не dogma. **Golden rules:** не rebase shared published tips; **force-with-lease**; короткоживущие ветки; policy в CONTRIBUTING.md.

## Ключевые пункты

- **Golden #1:** не rewrite remote history, от которой зависят другие (`main`, shared `develop`).
- **Golden #2:** `--force-with-lease`, не `--force`.
- **Golden #3:** integrate often, small PRs.
- Типичный SaaS: rebase private feature → squash merge PR → linear `main`.
- Типичный GitFlow: `--no-ff` в `develop`; rebase только до merge в shared branch.
- Final **tree** может совпадать; **hashes/blame/bisect/revert** — нет.
- После rebase pushed branch: announce + teammates `reset --hard origin/branch`.

## Частые ошибки

- Rebase shared `develop` без координации.
- Спор «только merge» vs «только rebase» без policy.
- Force-push `main`.

## Мнемоника

> **Shared = append-only.** **Private feature = editable draft.** **Main = published book** — не перепечатываем, только новые издания (commits).
