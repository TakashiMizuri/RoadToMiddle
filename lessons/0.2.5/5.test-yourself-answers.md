# Answers: Conflict resolution

## Scoring: 80%+ → `0.2.6`

### 1. [Recall]
**Answer:** `<<<<<<<` , `=======` , `>>>>>>>` (plus optional labels like HEAD / branch name).

### 2. [Recall]
**Answer:** **Stage 2 = ours (HEAD/current branch side)**; **Stage 3 = theirs (incoming side)**.

### 3. [Understanding]
**Answer:** Cannot know **product intent**; auto-pick risks silent loss of logic/reviewed code. Human decision required.

### 4. [Understanding]
**Answer:** **`HEAD` = main** (branch rebased onto) — «ours» in rebase; incoming replayed commit under `>>>>>>>` is feature side («theirs»).

### 5. [Application]
**Answer:** Open `App.tsx` → edit remove markers → `git add App.tsx` → (all unmerged resolved) → **`git commit`**.

### 6. [Application]
**Answer:** **`git rebase --continue`**

### 7. [Application]
**Answer:** **`git merge --abort`**

### 8. [Analysis]
**Answer:** **Syntax/runtime errors**, wrong behavior, markers in production if review misses.

### 9. [Analysis]
**Answer:** **`--ours` = main** (current HEAD during rebase onto main), **not** feature commit being replayed.

### 10. [Application]
**Answer:** Regenerate: e.g. checkout one side or delete lock → **`npm install`** → `git add package-lock.json` → continue merge/rebase.

### 11. [Recall]
**Answer:** **`git ls-files -u`**

### 12. [Understanding]
**Answer:** **`|||||||` section** showing **common ancestor** (merge base) hunk for 3-way context.

### 13. [Application]
**Answer:** **`git merge --abort`** (or `git rebase --abort` if rebase).

### 14. [Analysis]
**Answer:** **HEAD/main section** typically "B"; **incoming/feature** "C"; diff3 **ancestor** "A".

### 15. [Understanding]
**Answer:** Merge can **compile** but combine logic incorrectly (both changes partially kept wrong). Tests catch semantic errors.
