# Answers: First vs Single vs FirstOrDefault

## Scoring

- 90–100%: отлично · 80–89%: pass · <80%: повтори лекции

## Answers

### 1. [Recall]
**Answer:** 0 → throw; 2+ → throw «more than one element».
**Why:** Exactly-one contract.

### 2. [Recall]
**Answer:** `First` throws; `FirstOrDefault` returns `default(T)`.
**Why:** OrDefault absorbs empty case.

### 3. [Understanding]
**Answer:** Must detect second match to enforce uniqueness.
**Why:** Stopping at first would miss duplicate later in sequence.

### 4. [Understanding]
**Answer:** When email must be unique — `First` returns arbitrary duplicate silently.
**Why:** No validation of cardinality.

### 5. [Application]
**Answer:** `SingleAsync(u => u.Email == normalizedEmail)`.
**Why:** Documents + enforces unique email invariant.

### 6. [Application]
**Answer:** `SingleOrDefault` / `SingleOrDefaultAsync`.
**Why:** Zero OK, two misconfig → throw.

### 7. [Analysis]
**Answer:** `InvalidOperationException` (more than one).
**Why:** Three elements, not exactly one.

### 8. [Analysis]
**Answer:** `InvalidOperationException` — two elements match `== 2`.
**Why:** Single on predicate counts matching elements.

### 9. [Understanding]
**Answer:** Use `int?`, `TryGet` pattern, or `Any` + cast; or `FirstOrDefault` with sentinel not in domain.
**Why:** `default(int)` is 0 — ambiguous.

### 10. [Application]
**Answer:** Prefer `OrderByDescending(o => o.CreatedAt).FirstOrDefault()` (explicit sort key).
**Why:** `Last()` on unordered/queryable depends on provider order; explicit sort is clear and SQL-translatable.

### 11. [Analysis]
**Answer:** Throw on second object — both match predicate.
**Why:** Single checks count of matches, not reference identity alone.

### 12. [Understanding]
**Answer:** `InvalidOperationException`.
**Why:** Standard LINQ element operator behavior.
