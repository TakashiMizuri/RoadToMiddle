# Exception Hierarchy — throw vs throw ex

> Roadmap: `1.5.15` · Node: `1.5` — C#: .NET Runtime and Memory · Depth: понимание

## Learning Objectives

After this lesson you will be able to:

- Navigate the **System.Exception** hierarchy: **`SystemException`**, application exceptions, **`ArgumentException`**, **`InvalidOperationException`**, etc.
- Explain when to **catch**, **rethrow**, and **wrap** exceptions in API layers.
- Distinguish **`throw;`** from **`throw ex;`** and preserve **stack trace** with **`ExceptionDispatchInfo`**.
- Use **exception filters** (`when`) and avoid catching **`Exception`** blindly in libraries.
- Map exceptions to **ProblemDetails** / HTTP status in ASP.NET Core (conceptual link).

---

## Why This Matters

Exceptions are control flow for ** exceptional** conditions — not validation shortcuts. Wrong rethrow destroys **stack traces** and turns a one-hour debug into a three-day mystery: logs show error at **logging line**, not where **`NullReferenceException`** originated.

Middle developers read production logs, write middleware, and decide **`ArgumentNullException`** vs **`ValidationException`** vs custom **`OrderNotFoundException`**. **`throw ex;`** resets stack — **`throw;`** preserves it. **`ExceptionDispatchInfo.Capture(ex).Throw()`** preserves across **`await`** boundaries.

## Core Concepts

### Hierarchy Basics

All exceptions derive from **`Exception`** (except **`ObjectDisposedException`** etc. still under it). Common branches:

- **`SystemException`** — BCL runtime-ish (`NullReferenceException`, `IndexOutOfRangeException`) — less common to catch specifically today.
- **`ArgumentException`** family — **`ArgumentNullException`**, **`ArgumentOutOfRangeException`** — bad inputs.
- **`InvalidOperationException`** — object state wrong for operation.
- **`NotSupportedException`**, **`NotImplementedException`** — capability gaps.
- **`IOException`** / **`SqlException`** — I/O and data access (provider-specific).

**`ApplicationException`** — legacy marker; **avoid** for new code — use concrete types.

**Custom exceptions** when callers need **typed catch** or domain semantics — not for every 404.

### throw vs throw ex

```csharp
catch (Exception ex)
{
    _logger.LogError(ex, "Failed");
    throw;           // ✅ preserves stack trace
}

catch (Exception ex)
{
    throw ex;        // ❌ resets stack to this line
}
```

**`throw;`** rethrows **current exception** with original stack (and **`ExceptionDispatchInfo`** state).

**`throw ex;`** throws **`ex`** as **new throw site** — stack trace **starts at catch block**.

**Wrap with inner:**

```csharp
throw new OrderProcessingException("Payment failed", ex);
```

Stack shows outer throw; **`InnerException`** chain preserves cause.

### ExceptionDispatchInfo

When you must store and rethrow later (async, background queue):

```csharp
var edi = ExceptionDispatchInfo.Capture(ex);
// later on another thread:
edi.Throw();
```

Restores **original stack trace** — unlike **`throw ex`**.

### catch When Filters

```csharp
catch (SqlException ex) when (ex.Number == 1205) // deadlock
{
    // retry logic
}
```

Filter runs **before** binding catch; **does not** catch if `when` false.

### What to Catch

**Libraries:** catch **specific** exceptions you can handle; let others propagate.

**App boundary** (middleware): catch **`Exception`**, log, return **500 ProblemDetails** — **once**, not in every repository method.

**Never** catch **`Exception`** and swallow without log.

**OperationCanceledException** — often **rethrow** or treat as normal cancel, not error.

## Under the Hood

Stack trace stored in exception **`StackTrace`** string at throw time (unless **`StackTrace`** hidden in some rethrow paths). **`Environment.StackTrace`** separate.

**Rethrow** opcode vs **throw** new — CLR distinguishes **`throw;`** preservation.

---

## Common Mistakes & Anti-patterns

**`throw ex;`** in catch — lost context.

**Empty catch** — silent failure.

**Exceptions for flow control** on hot paths (use **`TryParse`**, **`Result`** where appropriate).

**Catching all in data layer** and returning null — hides bugs.

---

## Production & Real-World Notes

ASP.NET Core **DeveloperExceptionPage** vs **ExceptionHandler** middleware. Log **full exception** + **correlation ID**. Don't expose stack to clients in Production.

Map domain errors: **`NotFoundException` → 404**, validation → **422 ProblemDetails**.

## Quick Reference

| Pattern | Stack preserved | Use |
|---------|-----------------|-----|
| `throw;` | Yes | Same exception after log |
| `throw ex;` | No | Avoid |
| `throw new Wrap(..., ex)` | New outer | API boundary |
| `EDI.Capture(ex).Throw()` | Yes | Async/deferred |

---

## Key Takeaways

- Prefer **specific** exception types and messages.
- **`throw;`** not **`throw ex;`** when rethrowing same exception.
- **Wrap** with **InnerException** at boundaries.
- **ExceptionDispatchInfo** for deferred rethrow.
- Catch at **appropriate layer** — not everywhere.
