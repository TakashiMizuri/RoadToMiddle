# SynchronizationContext — ASP.NET Core Has None

> Roadmap: `1.4.9` · Node: `1.4` — Async/await · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Define **SynchronizationContext** and its role in posting work back to a **single-threaded scheduler**.
- Explain why **ASP.NET Core** deliberately does not set a request `SynchronizationContext`, unlike **ASP.NET Framework**.
- Connect SyncContext to **`ConfigureAwait`** (`1.4.8`) and **state machine continuations** (`1.4.7`).
- Compare **UI SyncContext** (WPF, WinForms) with **null context** behavior in server apps.
- Identify when **`TaskScheduler.FromCurrentSynchronizationContext`** still matters (`1.4.21` preview).

## Why This Matters

Before async/await, developers manually marshaled callbacks to the UI thread with `Control.Invoke` or `Dispatcher.BeginInvoke`. `SynchronizationContext` abstracts that pattern: a hook the runtime uses to say “run this delegate on the right thread.” When `await` captures context, it uses this hook — which is essential in UI apps and was a footgun in **ASP.NET Framework** where a request context existed.

ASP.NET Core removed that request context by design. That single architectural choice explains why **`ConfigureAwait(false)` is optional** in Core controllers and why many Framework-era deadlock recipes do not apply verbatim. Middle developers still encounter SyncContext in desktop tools, Blazor Server, Unity game loops, and legacy Framework maintenance — and must not confuse “web server” with “no threading concerns.”

## Core Concepts

### What SynchronizationContext Is

`SynchronizationContext` is an abstract class with **`Post(SendOrPostCallback, state)`** and **`Send(...)`** (synchronous wait for execution). The **current** context is stored in thread-local/static slot accessible as `SynchronizationContext.Current`.

When a thread has no special scheduler, **`Current` is `null`**. Async continuations with default `ConfigureAwait` check: if `Current != null`, queue via `Post`; else run on thread pool.

UI frameworks install a custom context on the UI thread at startup. **WPF** uses `DispatcherSynchronizationContext`; **WinForms** uses `WindowsFormsSynchronizationContext`. Only one thread “owns” the UI; all control updates must happen there — SyncContext is the front door.

### How await Uses It

From `1.4.7`, the state machine registers a continuation when an await is not completed synchronously. `TaskAwaiter.OnCompleted` captures `SynchronizationContext.Current` at the **await site** (when `continueOnCapturedContext` is true). Completion calls `Post` to run `MoveNext` on the captured context.

This is **orthogonal** to **ExecutionContext**, which flows `AsyncLocal`, security, and culture. Both can be captured; `ConfigureAwait` only toggles SyncContext capture.

```mermaid
flowchart LR
    A[await on UI thread] --> B{SyncContext null?}
    B -->|No| C[Capture context]
    B -->|Yes| D[No capture]
    C --> E[I/O completes on pool thread]
    E --> F[Post MoveNext to UI]
    D --> G[Queue MoveNext on pool]
```

### ASP.NET Framework vs ASP.NET Core

**ASP.NET Framework** (System.Web) installed **`AspNetSynchronizationContext`** per request in many pipeline configurations, especially after `HttpContext.Current` async changes (.NET 4.5+). Awaiting with capture could resume on the **request context**, giving illusions of “thread affinity” for a request. Combined with **sync-over-async** (`.Result`, `.Wait()`), this produced classic deadlocks: block the request thread waiting for async work that tries to post back to the same context.

**ASP.NET Core** (Kestrel pipeline) **does not set** a `SynchronizationContext` on request threads. `SynchronizationContext.Current` is **`null`** during normal controller and middleware execution. Continuations after `await` run on thread pool threads — which is appropriate for scalable I/O-bound web apps. **`HttpContext` is not tied to a single thread**; it flows via **`AsyncLocal`** and logical request scope in the DI scope, not via SyncContext.

This is why Stephen Toub and ASP.NET team guidance says: **in ASP.NET Core application code, you do not need `ConfigureAwait(false)`** — there is nothing to capture. Thread agility is a feature, not a bug.

### Console and Generic Host

Console apps without a custom context also have `Current == null`. **BackgroundService** workers same. **Blazor Server** is different: circuit dispatch may use a sync context analogous to UI — treat as UI-like for component state updates.

## Under the Hood

`SynchronizationContext` default implementation **`Post`** delegates to **`ThreadPool.QueueUserWorkItem`**. Derived types override to target UI message pumps.

On **ASP.NET Framework**, the request context serialized posts to avoid concurrent re-entrancy on `HttpContext` in some eras; behavior evolved across patches. Migration guides emphasize: **remove assumptions that continuations return to “the request thread.”**

**Kestrel** uses IOCP and thread pool for connection handling; there is no “request thread” identity to preserve. **`HttpContextAccessor`** uses `AsyncLocal<HttpContext>` so `HttpContext` remains available after awaits on any pool thread within the same logical request — as long as you stay in the async flow and scoped services are resolved correctly.

State machine link: without SyncContext, **`MoveNext`** after await typically schedules via **`TaskScheduler.Default`** (thread pool). With SyncContext, **`Post`** wraps the same `MoveNext` delegate — the state machine code is identical; only the queue target differs.

## Syntax / API

```csharp
// Inspect at runtime (diagnostics)
var ctx = SynchronizationContext.Current; // null in ASP.NET Core controller

// Manual post (rare in app code; used by frameworks)
SynchronizationContext.Current?.Post(_ => UpdateUi(), null);

// Opt out of capture — see 1.4.8
await work.ConfigureAwait(false);

// Schedule to current SyncContext explicitly
await Task.Factory.StartNew(
    () => DoUiWork(),
    CancellationToken.None,
    TaskCreationOptions.None,
    TaskScheduler.FromCurrentSynchronizationContext());
```

## Examples

### WPF: Context Required

```csharp
private async void LoadButton_Click(object sender, RoutedEventArgs e)
{
    var data = await _api.GetDataAsync(); // captures Dispatcher SyncContext
    StatusLabel.Content = data;           // safe — resumed on UI thread
}
```

If you `ConfigureAwait(false)` before updating `StatusLabel`, you must dispatch manually.

### ASP.NET Core: Null Context

```csharp
public async Task<IActionResult> Get()
{
    Debug.Assert(SynchronizationContext.Current == null);
    var x = await _http.GetStringAsync("https://example.com");
    // Continuation may run on different pool thread — HttpContext still valid via AsyncLocal
    return Ok(x);
}
```

## Common Mistakes & Anti-patterns

**Assuming Framework deadlock fixes apply to Core without analysis.** Core deadlocks more often from **thread pool starvation** (`1.4.10`) or **sync-over-async blocking pool threads**, not SyncContext.

**Using `ConfigureAwait(true)` “to keep HttpContext.”** Wrong mechanism — use async all the way; context flows via `IHttpContextAccessor` / scope.

**Calling `SynchronizationContext.SetSynchronizationContext` in middleware** without deep reason — fights the framework model.

## Production & Real-World Notes

Legacy **ASP.NET MVC on .NET Framework** maintenance still hits SyncContext deadlocks. **Core** teams focus on **not blocking** (`Task.Wait`, `Thread.Sleep` in request path) and **DbContext** scoping.

Desktop developers embedding **WebView2** or **CEF** in LOB apps cross UI SyncContext and async HTTP daily — know both worlds.

## Comparison / Trade-offs

| Host | SyncContext on request/UI | ConfigureAwait in app code |
|------|---------------------------|----------------------------|
| WPF / WinForms | Yes (UI thread) | Usually omit `false` for UI |
| ASP.NET Framework | Often yes | Sometimes critical |
| ASP.NET Core | No | Optional |
| Console | No | Optional |

## Quick Reference

- **`SynchronizationContext.Current`** — scheduler for `Post`/`Send`; `null` = thread pool continuations.
- **await default** — captures SyncContext if non-null.
- **ASP.NET Core** — no request SyncContext; thread-agile requests.
- **UI** — custom SyncContext; single-threaded updates.

---

## Key Takeaways

- SyncContext is the **await continuation marshaling** abstraction.
- UI has it; **ASP.NET Core does not** — major difference from Framework.
- **`ConfigureAwait(false)`** skips SyncContext capture (`1.4.8`).
- State machine **`MoveNext`** target depends on capture policy (`1.4.7`).
- **`HttpContext`** in Core uses **AsyncLocal**, not SyncContext.
- Deadlock class shifts from context re-entrancy to **pool starvation** on Core.
