# Amending Commits: When It's Safe and When It's Dangerous

> Roadmap: `0.1.10` · Node: `0.1` — Git Basics · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Explain what **`git commit --amend`** does to the most recent commit.
- Identify when amending is **safe** (local, unpushed work).
- Identify when amending is **dangerous** (pushed or shared history).
- Compare **`--amend`** vs creating a **new commit** for fixing mistakes.
- State the **golden rule of public history**: do not rewrite commits others may have pulled.
- Know when **interactive rebase** is the broader tool (without deep-diving into rebase mechanics).

---

## Why This Matters

Picture a familiar moment. You finish a feature, run `git commit -m "Add user login endpoint"`, and push nothing yet — you are still on a local branch, polishing before opening a pull request. Then you notice a typo in the commit message, or you realize that `appsettings.Development.json` never made it into the staging area. Without amend, the natural instinct is to fix things with follow-up commits: one for the typo, one for the forgotten file. Before long your history looks like three separate events when there was really one logical change.

That noise matters more than it first appears. Code review becomes harder because reviewers must mentally squash three commits into one intent. `git bisect` — the tool that finds which commit introduced a bug — treats each commit as a potential culprit; three commits for one feature triples the search space for no good reason. And when someone reads `git log` six months later, they see a trail of "fix typo" and "add forgotten file" instead of a clean record of what actually landed.

**`git commit --amend`** exists to solve exactly this problem on the **most recent commit**. It lets you fold corrections into the commit you just made, keeping history linear and readable. But amend is not a free pass to rewrite history whenever you feel like it. The same mechanism that cleans up your local work can break a teammate's clone, invalidate open pull requests, and force dangerous force-pushes if you use it on commits that already left your machine. Understanding when amend is a convenience and when it is a hazard is one of the clearest markers of moving from junior habits to middle-level Git fluency.

## Core Concepts

### What Does `--amend` Actually Do?

The name "amend" suggests editing something in place, as if Git opened your last commit and changed a few bytes. That mental model is wrong, and holding onto it causes confusion later. Git commits are **immutable objects**. Once created, a commit's SHA-1 hash is a fingerprint of its entire contents: the tree of files, the parent commit, the author, the committer, and the message. Change anything — even a single character in the message — and you get a completely different hash.

When you run `git commit --amend`, Git does not modify the existing commit. Instead, it builds a **brand-new commit object** that replaces the old one as far as your current branch is concerned. The new commit can include freshly staged changes (if you ran `git add` before amending), a revised message (if you passed `-m` or edited in your editor), or both. Your branch pointer — say `refs/heads/main` or `refs/heads/feature/login` — moves forward to this new SHA. The previous commit, the one you thought you were "editing," loses its branch reference. It still exists on disk for a while, but nothing points to it except the reflog.

Visually, the transition looks like this:

```
Before amend:
  main → A → B → C (HEAD)
                  ↑ forgot a file

After git add forgotten.txt && git commit --amend:
  main → A → B → C' (HEAD)   ← new SHA, includes forgotten.txt
                  C is orphaned (still in reflog briefly)
```

The orphaned commit `C` is not immediately destroyed. Git's garbage collector will eventually reclaim it, but locally you can still find it through `git reflog` for roughly ninety days. That safety net matters when you amend by mistake — but it does **not** help teammates who never had `C` in their reflog because they pulled a different history from the remote.

Three scenarios cover most day-to-day amend usage. When you only need to fix the commit message, `git commit --amend -m "Correct message"` opens no editor and sets the new text directly. When you forgot to stage files, you stage them first with `git add`, then run `git commit --amend --no-edit` — the `--no-edit` flag tells Git to keep the existing message while incorporating the newly staged content into the replacement commit. When you want to rewrite the message thoughtfully, plain `git commit --amend` opens your configured editor with the current message ready for editing.

### The Golden Rule of Public History

Every team eventually codifies some version of this rule: **never rewrite commits that others have already based work on.** "Public" history means any commit that has been pushed to a shared remote branch — `main`, `develop`, a release branch — where teammates, CI pipelines, or deployment systems treat those SHAs as ground truth.

The reason is mechanical, not social. When you amend a pushed commit, your local branch now points to `C'` while the remote still points to `C`. These histories have **diverged**. A normal `git push` is rejected because Git refuses non-fast-forward updates that would discard remote commits. The "fix" people reach for is `git push --force`, which overwrites the remote branch with your rewritten history. Anyone who already pulled `C` now has a clone whose history does not match the server. Their next push may fail; they may need to rebase or reset; open pull requests may show confusing diffs; CI may re-run against SHAs that no longer exist on the remote.

On a local branch that has never been pushed, amend is almost always safe because you are the only consumer of that history. On a feature branch that only you use, amend is still safe until someone else pulls it — at which point the same divergence problem appears. On `main` after a merge, the answer is no: add a new commit or use `git revert`. On a release tag that has already been deployed to production, rewriting is never acceptable; the deployed artifact is tied to that immutable SHA.

## Under the Hood

When you execute `git commit --amend`, Git performs a sequence of steps that mirrors an ordinary commit, but with a different parent selection. Instead of using the current `HEAD` commit as the parent, amend uses **the parent of the current `HEAD`**. In effect, Git says: "Build a new commit on top of what came before my last commit, incorporating whatever is staged now, and make that the new tip."

If you changed files, Git constructs a new **tree object** reflecting the updated snapshot. The **author date** may be preserved from the original commit (depending on Git version and options), while the **committer date** updates to now — a subtle detail that shows up in `git log --format=fuller`. The branch ref file — for example `.git/refs/heads/main` — is atomically updated to the new SHA. The old tip commit becomes unreachable from any branch, though it remains in `refs/logs/HEAD` and `refs/logs/refs/heads/main`.

You can see this history in the reflog:

```bash
git reflog
# abc1234 HEAD@{0}: commit (amend): Add user login
# def5678 HEAD@{1}: commit: Add user login   ← old commit, recoverable locally
```

If you already pushed `def5678` to origin before amending, the remote still references that SHA. Your local `main` now references `abc1234`. Until you force-push, two truths coexist: your machine says the latest commit is `abc1234`, the server says `def5678`. That is the divergence problem in concrete terms.

Amend is strictly limited to **`HEAD`**, the commit at the tip of your current branch. If you need to change a commit three steps back, amend cannot help. That is the domain of **interactive rebase** (`git rebase -i`), which we will cover in a later lesson. The same golden rule applies there: rewriting commits that others have pulled causes the same collateral damage, just across multiple SHAs instead of one.

## Syntax / Commands / API

The amend command family is small, but each flag has a precise meaning:

```bash
# Amend message only
git commit --amend -m "New message"

# Add staged files to last commit, keep message
git add forgotten.cs
git commit --amend --no-edit

# Amend author/date (rare — usually for correcting metadata mistakes)
git commit --amend --author="Name <email>" --no-edit
git commit --amend --date="2026-07-01T12:00:00"

# Open editor for message
git commit --amend
```

Remember the prerequisite for adding forgotten files: **`git add` must happen before `--amend`**. Staging is what tells Git which content belongs in the replacement commit. Without staged changes, amend only affects the message.

## Examples

### Example 1: Safe amend on unpushed work

Consider a feature branch `feature/health` that exists only on your laptop. You commit the controller but forget a helper class:

```bash
git commit -m "Add health check endpoint"
git add HealthCheckExtensions.cs    # forgot this file
git commit --amend --no-edit
git push origin feature/health       # first push — remote never saw old commit
```

This workflow is the ideal amend case. The remote never received the incomplete commit. When you push for the first time, the server sees a single, correct commit. No force push, no divergence, no teammate impact. Many teams encourage exactly this pattern on feature branches before opening a pull request.

### Example 2: Dangerous amend after push to shared branch

Now imagine the same mistake, but you already pushed to `main`:

```bash
git commit -m "Add health check"
git push origin main                 # shared branch — others may have pulled
git add forgotten.cs
git commit --amend --no-edit
git push origin main                 # REJECTED: non-fast-forward
git push --force origin main         # ⚠️ DANGEROUS — rewrites shared history
```

After the amend, colleagues who pulled the original commit have `def5678` in their history. Your force-push replaces remote history with `abc1234`. Their local `main` still points to `def5678`, which is no longer on the server. They must run something like `git fetch origin && git reset --hard origin/main` — losing any local commits based on the old tip — or rebase their work onto the new history. That is a high price for forgetting one file.

The safer alternative after push is a follow-up commit:

```bash
git add forgotten.cs
git commit -m "Add missing HealthCheckExtensions to health check commit"
```

The log is slightly noisier, but history remains consistent for everyone. No force push required.

### Example 3: Choosing between amend and a new commit

The decision tree is simpler than it feels. Ask one question: **has anyone else possibly built on this commit?** If no — amend freely. If yes — add a new commit or revert. On open pull requests where only you work on the branch, amending and force-pushing the PR branch (`git push --force-with-lease origin feature/health`) is common and accepted on many teams. The `--force-with-lease` variant refuses to push if someone else updated the remote branch since your last fetch, adding a small safety check over bare `--force`.

## Common Mistakes & Anti-patterns

The most damaging mistake is amending a commit on `main` or `develop` and force-pushing because "it's just a small fix." Small fixes still produce new SHAs, and protected branches with CI, deployment hooks, and colleague clones amplify the blast radius. Another frequent error is running `git commit --amend` without staging first, expecting it to pick up unstaged working tree changes. Amend only includes what is in the index; unstaged modifications stay in your working directory, unchanged and uncommitted.

Amending merge commits without understanding the result can land you in a confusing conflict state — merge commits have two parents, and replacing them is rarely what you want. Using amend to hide mistakes from code review erodes trust; reviewers approved a specific diff at a specific SHA, and amend silently changes that diff. Finally, reaching for amend when you need to squash three local commits before a PR is the wrong tool; that is interactive rebase territory, where the same public-history rules apply at scale.

## Production & Real-World Notes

Professional teams often combine amend with **squash merge** workflows. Developers amend freely on feature branches, keeping the branch tidy, and the platform merges the PR as a single commit onto `main`. The rewrite happens only on the private branch; `main` never sees force-pushed amendments.

Branch protection on `main` — blocking force pushes, requiring pull requests, running CI — exists partly to prevent history rewrites on shared lines of development. Signed commits and GPG verification add another wrinkle: amend creates a new commit object, so the signature must be regenerated. In regulated environments, audit trails may require that production-deployed SHAs remain immutable; amending a tagged release is a compliance violation, not merely a social faux pas.

When a mistake is already published and amend is off the table, **`git revert`** creates a new commit that undoes a previous one without erasing it from history. That is the production-safe path for "we shipped something broken on `main`."

## Comparison / Trade-offs

**`git commit --amend`** shines when you want a clean, linear history and the commit is still private. The cost is a new SHA and the risk of divergence if you amend too late. A **new fixup commit** is noisier but safe on any shared branch — the honest record of "we forgot something and added it later." **`git revert`** is the choice when the bad commit is public and you need to undo its effect without pretending it never happened. **Interactive rebase** is the power tool for reshaping multiple local commits before a PR; it shares amend's rewrite mechanics but operates on more than just `HEAD`.

None of these approaches is universally "best." Middle-level judgment means matching the tool to who else depends on the commits you are changing.

---

## Quick Reference

Typo in message, not pushed: `git commit --amend -m "..."`. Forgot a file, not pushed: `git add ... && git commit --amend --no-edit`. Mistake already on shared `main`: new commit or `git revert`. Cleanup of several local commits before a PR: `git rebase -i` (covered later). Already pushed to a branch others use: do not amend — add a follow-up commit.

---

## Key Takeaways

- **`--amend` replaces the `HEAD` commit** with a new object bearing a new SHA — it does not edit in place.
- **Safe when local and unshared** — the remote never saw the old commit.
- **Dangerous after push to shared branches** — requires force push and breaks others' clones.
- **Golden rule:** do not rewrite public history that teammates or CI depend on.
- Prefer **new commit** or **revert** on shared branches; reserve amend for private work.
- **`--no-edit`** adds staged files without changing the commit message.
- For commits older than the last one, use **interactive rebase** — with the same rewrite cautions.
