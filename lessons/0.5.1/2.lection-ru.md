## Why This Matters

Прежде чем браузер применит Flexbox, загрузит React или отобразит JSON от ASP.NET Core API, он получает **байты** и должен понять, как их интерпретировать. Первые сигналы в HTML-файле не визуальны — они договорённость между автором и парсером. **DOCTYPE** сообщает, по каким правилам разбирать разметку. **`charset`** — как декодировать байты в символы. **`lang`** — на каком человеческом языке написана страница для скринридеров и поисковиков.

Эти три строки часто выглядят как шаблонная «обёртка», которую генерирует фреймворк. Но это фундамент каждой страницы. Неверная или отсутствующая кодировка даёт **mojibake** — вместо «Привет» на экране `ÐŸÑ€Ð¸Ð²ÐµÑ‚`. Без `lang` screen reader угадывает произношение. Без корректного DOCTYPE браузер может войти в **quirks mode**, где CSS ведёт себя как в старых версиях IE. Middle-разработчик не игнорирует каркас: он знает, *зачем* каждая часть нужна, чтобы чинить кодировку в production, закрывать предупреждения Lighthouse и ревьюить PR, ломающие структуру документа.

В узле `0.5` ты строишь язык веба с нуля. Ты уже знаешь HTTP request-response (`0.3.1`): сервер шлёт байты, браузер их парсит. HTML — частый формат этих байтов. Этот урок — первый лист этого формата: конверт до содержания письма.

## Core Concepts

### HTML-документ — дерево элементов

HTML-документ — иерархическое дерево. Корень — элемент `<html>`. У него два практически обязательных ребёнка: `<head>` для метаданных и `<body>` для видимого контента. Всё остальное вкладывается в одну из этих веток. Браузер не исполняет твои намерения — он **парсит** текст в DOM по спецификации HTML. Если разметка кривая, срабатывают **правила восстановления после ошибок**: страница часто всё равно что-то покажет, но не обязательно то, что ты задумал.

Прощающий парсер — палка о двух концах. В Chrome «всё выглядит нормально», а структура неправильная — страдают accessibility, SEO и селекторы CSS. Корректный каркас уменьшает тихие поломки.

### DOCTYPE: включение standards mode

**DOCTYPE** — не HTML-элемент. Это **объявление** в самой первой строке файла, до `<html>`. В HTML5 оно короткое:

```html
<!DOCTYPE html>
```

Раньше строки DOCTYPE ссылались на DTD и версии HTML. HTML5 упростил до одного токена: «разбирай документ по современному living standard».

Браузеры держат два режима: **no-quirks mode** (standards) и **quirks mode** (совместимость со старыми багами). Quirks mode эмулирует, например, иной расчёт ширины с padding. Корректный HTML5 DOCTYPE включает standards mode во всех актуальных браузерах. Без DOCTYPE или с древним объявлением layout может вести себя непредсказуемо — и твой CSS из `0.6.x` даст сюрпризы.

DOCTYPE не попадает в DOM для JavaScript. DevTools не покажет его как элемент. Он нужен только на этапе парсинга — но эффект на вёрстку сохраняется на всю жизнь страницы.

### Корневой элемент и атрибут lang

`<html>` оборачивает весь документ. На нём указывают **`lang`** с тегом языка **BCP 47**:

```html
<html lang="ru">
```

Screen reader выбирает правила произношения по `lang`. Если страница в основном на русском, а цитата на английски — у цитаты свой `lang="en"` на `<span>` или `<blockquote>`, а не смена корня без смены языка всей страницы.

Поисковики используют `lang` для региональной выдачи. Браузеры предлагают перевод, когда `lang` не совпадает с локалью пользователя. Без атрибута всё это работает хуже.

Обычно один URL — один основной язык. Альтернативы между URL — через hreflang (`0.5.13`); `lang` описывает текущий документ.

### head: метаданные, которые пользователь не видит

`<head>` собирает **метаданные** — информацию *о* документе. Типичные дети:

- `<meta charset="UTF-8">` — кодировка
- `<title>` — заголовок вкладки и сниппет в поиске
- `<meta name="viewport" ...>` — мобильная вёрстка
- `<link rel="stylesheet">` — CSS
- `<script>` — JavaScript

`head` парсится до отрисовки `body`. Критичные вещи вроде charset ставят **рано**, чтобы браузер не декодировал первые килобайты неверно, пока ждёт meta внизу.

### UTF-8 и meta charset

**UTF-8** кодирует латиницу, кириллицу, CJK, emoji — практически всё нужное в вебе. Это **дефолтная кодировка веба**. Редактор сохраняет UTF-8; сервер шлёт `Content-Type: text/html; charset=utf-8` в HTTP (`0.3.3`).

В HTML это дублируют так:

```html
<meta charset="UTF-8">
```

В HTML5 эта форма предпочтительнее старого `http-equiv`. Meta должна оказаться в первых **1024 байтах**, лучше первым ребёнком `<head>`.

Без charset браузер **угадывает** — часто ошибается на кириллице. Видишь `` или кракозябры. Лечат выравнивание **кодировки файла**, **meta** и **HTTP-заголовка**, а не CSS.

### body: видимое содержимое

В `<body>` — всё, что видит пользователь: заголовки, абзацы, картинки, формы. `head` настраивает, `body` показывает. Видимые заголовки в `head` не кладут.

## Under the Hood

Kestrel или nginx отдаёт `index.html` как поток байтов. Парсер HTML читает байты, определяет кодировку (HTTP-заголовок может перебить meta в пограничных случаях), строит DOM, потом CSS и скрипты.

DOCTYPE обрабатывается **до** построения дерева. Совпадение с HTML5-триггером → no-quirks. Meta charset при позднем обнаружении может **перезапустить** парсинг с начала — ещё один аргумент ставить charset первым.

View Source — твой исходник; панель Elements — **распарсенный** DOM с автоисправлениями. Сравнение учит видеть, что браузер «додумал» сам.

## Syntax / Commands / API

Минимальный шаблон:

```html
<!DOCTYPE html>
<html lang="ru">
<head>
  <meta charset="UTF-8">
  <title>Моя первая страница</title>
</head>
<body>
  <p>Привет, мир.</p>
</body>
</html>
```

Проверка: [W3C Validator](https://validator.w3.org/) — подробнее в `0.5.16`.

## Examples

### Пример 1: русская страница

Файл сохранён UTF-8; сервер отдаёт UTF-8:

```html
<!DOCTYPE html>
<html lang="ru">
<head>
  <meta charset="UTF-8">
  <title>Команда разработки</title>
</head>
<body>
  <p>Добро пожаловать.</p>
</body>
</html>
```

Сохранишь файл в Windows-1251 или сервер объявит другую кодировку — кириллица сломается при «правильной» разметке.

### Пример 2: английская страница с русским именем

```html
<html lang="en">
...
<p>Lead: <span lang="ru">Иван Петров</span>.</p>
```

Произношение переключается только внутри span.

## Common Mistakes & Anti-patterns

**Нет DOCTYPE** — quirks mode и странный CSS. **Charset в конце head** — риск неверного первого декодирования. **Meta charset в body** — невалидно. **lang не совпадает с языком контента** — вред a11y и SEO. **Уроки без DOCTYPE** — не тащи в production.

---

## Production & Real-World Notes

Шаблоны Vite, CRA, Razor `_Layout.cshtml` уже включают каркас. При кастомизации layout сохраняй DOCTYPE, charset, lang. В ASP.NET Core можно выставлять `lang` из culture запроса. Middleware должен отдавать charset для HTML.

Lighthouse ругается на отсутствие `lang` и `title`. Баги кодировки часто всплывают на staging с реальными ФИО на кириллице — тестируй рано.

## Comparison / Trade-offs

Для новых проектов только `<!DOCTYPE html>`. UTF-8 vs UTF-16: на вебе доминирует UTF-8. Charset только в HTTP достаточен, если контролируешь сервер полностью; meta помогает при `file://` и хостинге без заголовка.

## Key Takeaways

- DOCTYPE — переключатель режима парсера, не элемент DOM.
- `lang` на `<html>` — a11y, SEO, перевод.
- UTF-8 + ранний `<meta charset="UTF-8">`.
- Согласуй кодировку файла, meta и HTTP.
- Сначала каркас, потом стили и скрипты.
