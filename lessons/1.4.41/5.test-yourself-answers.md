# Answers: Sync-over-async fixes

## Scoring

- 90–100%: отлично · 80–89%: pass · <80%: повтори lection

---

## Answers

### 1. [Recall]
**Answer:** Синхронный код блокирует поток, ожидая async операцию через `.Result`, `.Wait()`, `.GetAwaiter().GetResult()`.
**Why:** Blocking wait on Task.

### 2. [Recall]
**Answer:** `async` метод оборачивает синхронный blocking вызов (часто `Task.Run`), не используя async I/O end-to-end.
**Why:** Fake async facade.

### 3. [Understanding]
**Answer:** Thread pool **starvation** — каждый request блокирует worker на I/O; under load queue растёт, latency ↑, timeouts.
**Why:** No SyncContext ≠ free threads.

### 4. [Understanding]
**Answer:** Sync HTTP/socket внутри `Task.Run` всё ещё блокирует **worker thread**; только перенос с request async path на другой pool thread.
**Why:** No OS async I/O benefit.

### 5. [Understanding]
**Answer:** Low QPS internal tool, short-term bridge, нельзя изменить legacy сразу, CPU-bound sync work (не I/O).
**Why:** Tactical trade-off.

### 6. [Application]
**Answer:** Добавить `GetUserAsync`; пометить sync `[Obsolete]`; новый controller `await GetUserAsync`; удалить sync после consumers migrated.
**Why:** Incremental safe path.

### 7. [Application]
**Answer:** Dedicated microservice с async outward API; queue-based worker с limited parallelism; не вызывать sync SDK на Kestrel request thread.
**Why:** Isolate blocking boundary.

### 8. [Analysis]
**Answer:** Sync-over-async / blocking sync EF на thread pool thread в async action — fix: `await ToListAsync()` или projection async.
**Why:** Async action should not block on sync EF.

### 9. [Analysis]
**Answer:** Нет — уменьшает UI deadlock risk; thread всё ещё blocked на `.GetResult()`.
**Why:** ConfigureAwait не создаёт async I/O.

### 10. [Understanding]
**Answer:** `GetResult()` бросает original exception без обёртки в `AggregateException` (в отличие от `.Result`).
**Why:** Slight ergonomics difference; still blocks.

### 11. [Application]
**Answer:** 1) Async refactor 2) Task.Run bridge 3) .Result in request (worst).
**Why:** Scalability ranking from lection.

### 12. [Analysis]
**Answer:** Максимум **10 concurrent** blocking calls — caps blast radius; threads всё ещё blocked, но не unbounded.
**Why:** Damage control, not fix.
