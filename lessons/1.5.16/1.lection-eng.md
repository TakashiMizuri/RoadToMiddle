# NuGet Packages, Transitive Dependencies, and Central Package Management

> Roadmap: `1.5.16` · Node: `1.5` — C#: .NET Runtime and Memory · Depth: практика

## Learning Objectives

After this lesson you will be able to:

- Add and update **NuGet packages** in SDK-style projects with **`PackageReference`**.
- Explain **direct vs transitive** dependencies and how they appear in **`obj/project.assets.json`**.
- Audit transitive versions with **`dotnet list package --include-transitive`** and **`dotnet nuget why`**.
- Enable **Central Package Management (CPM)** with **`Directory.Packages.props`** for monorepo version control.
- Apply **version pinning**, **floating versions**, and **security audit** (`dotnet list package --vulnerable`).

## Why This Matters

Modern .NET apps are assembled from **NuGet packages** — EF Core, ASP.NET middleware, logging, test frameworks. You declare **`PackageReference`** in `.csproj`, but **`dotnet restore`** pulls a **graph**: your package depends on others **transitively**. A vulnerability or breaking change in a **transitive** library can affect production even if you never heard of the package name.

Teams with five microservices and a shared library duplicate **version drift** — Service A uses `Serilog 3.1.0`, Service B uses `3.0.0`. **Central Package Management** (.NET 6+) centralizes versions in **`Directory.Packages.props`**: projects reference **`PackageReference Include="Serilog" Version=""`** or omit version entirely with **`ManagePackageVersionsCentrally`**.

Middle developers run **`dotnet restore`**, read **dependency graphs**, fix **NU1605** downgrade errors, and coordinate upgrades in **solution** or **repo** scope — daily workflow before SQL in **`1.6.1`**.

## Core Concepts

### PackageReference and Restore

SDK-style project:

```xml
<ItemGroup>
  <PackageReference Include="Npgsql.EntityFrameworkCore.PostgreSQL" Version="9.0.2" />
</ItemGroup>
```

**`dotnet restore`** (implicit on build) resolves versions from **nuget.org** (or private feeds), writes **`project.assets.json`** — full closure of packages and DLL paths.

**`dotnet add package SomeLib`** adds reference and runs restore.

Packages land in **global cache** (`~/.nuget/packages` or `%USERPROFILE%\.nuget\packages`) — not copied into repo unless **PackageReference** with **`PrivateAssets`** nuances for publish.

### Direct vs Transitive

**Direct** — you listed in csproj.

**Transitive** — required by your direct packages.

Example: you reference **`Microsoft.AspNetCore.OpenApi`** → pulls **`Microsoft.OpenApi`**, **`Swashbuckle`** pieces, etc.

Inspect:

```bash
dotnet list package --include-transitive
dotnet list package --outdated
dotnet list package --vulnerable
```

**.NET 8+:**

```bash
dotnet nuget why MyApp.csproj Newtonsoft.Json
```

Shows **why** a package entered the graph — invaluable for conflicts.

### Version Ranges and Floating

```xml
<PackageReference Include="Serilog" Version="3.*" />
```

**Floating** — gets latest matching on restore — convenient in dev, **risky** in CI reproducibility. Teams often **pin exact** versions in CPM.

**NU1605** — detected **package downgrade** — two dependencies want incompatible versions of same package; fix by **aligning versions** or **explicit PackageReference** to force higher version.

### Central Package Management (CPM)

**`Directory.Packages.props`** at repo root:

```xml
<Project>
  <PropertyGroup>
    <ManagePackageVersionsCentrally>true</ManagePackageVersionsCentrally>
  </PropertyGroup>
  <ItemGroup>
    <PackageVersion Include="Npgsql.EntityFrameworkCore.PostgreSQL" Version="9.0.2" />
    <PackageVersion Include="Serilog.AspNetCore" Version="8.0.3" />
  </ItemGroup>
</Project>
```

Project `.csproj`:

```xml
<ItemGroup>
  <PackageReference Include="Serilog.AspNetCore" />
  <!-- Version from central props -->
</ItemGroup>
```

**Benefits:** one place to bump versions; PR reviews see dependency changes clearly; less drift across projects.

**`GlobalPackageReference`** — packages applied to all projects (analyzers, SourceLink).

**`CentralPackageTransitivePinningEnabled`** — pin transitive versions centrally (advanced).

### Private Feeds and nuget.config

**`nuget.config`** at solution root:

```xml
<configuration>
  <packageSources>
    <add key="nuget.org" value="https://api.nuget.org/v3/index.json" />
    <add key="company" value="https://pkgs.dev.azure.com/.../index.json" />
  </packageSources>
</configuration>
```

CI uses **`dotnet restore --configfile nuget.config`** with credentials via **`NUGET_AUTH_TOKEN`**.

### Security and Supply Chain

- **`dotnet list package --vulnerable`**
- Dependabot / Renovate on **`Directory.Packages.props`**
- Lock files: **`RestorePackagesWithLockFile`** + **`packages.lock.json`** for reproducible CI

## Examples

### Add package CLI

```bash
dotnet add src/Api/Api.csproj package Microsoft.EntityFrameworkCore.Design
dotnet restore
dotnet build
```

### Enable CPM in existing repo

1. Create **`Directory.Packages.props`** with **`ManagePackageVersionsCentrally`**
2. Move all **`Version="..."`** from csproj files to **`PackageVersion`**
3. Remove Version from individual **`PackageReference`**
4. **`dotnet restore`** verify

### Force transitive version

If transitive pulls old **`System.Text.Json`**, add direct reference to safe version or update parent package.

## Common Mistakes & Anti-patterns

**Duplicate versions** across microservices without CPM.

**Floating `*` in production** — non-reproducible builds.

**Ignoring NU1605** — runtime **`FileNotFoundException`** or subtle bugs.

**Committing `packages/` folder** — legacy; modern uses global cache.

**Binding redirects** — largely **.NET Framework** concern; .NET Core uses unified deps.

## Production & Real-World Notes

Upgrade workflow: branch → bump **`Directory.Packages.props`** → **`dotnet test`** all affected projects → security scan.

Document **breaking** major upgrades (EF Core major, ASP.NET major) in team changelog.
