# Exercises: HTTP Status Codes

> Complete after reading `1.lection-eng.md`, `2.lection-ru.md` and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Code picker
**Goal:** Map scenarios to the correct status code.

For each scenario, write one status code from this lesson's list:

1. Successful GET returning JSON entity.
2. POST created a new user; URI is `/users/42`.
3. DELETE succeeded; no response body needed.
4. JWT expired or missing on protected endpoint.
5. Valid JWT but user lacks `Admin` role.
6. Rate limit exceeded; client should wait 30 seconds.

**Success criteria:** Six correct codes with one-line justification each.

---

### E2. True or false
**Goal:** Fix misconceptions.

1. Reason phrase `Not Found` must match exactly across all servers.
2. 304 responses include a full JSON body with unchanged data.
3. 502 typically indicates a problem at a gateway/proxy layer.
4. Returning 200 for all outcomes simplifies frontend error handling long-term.

**Success criteria:** 4/4 with RFC-aligned reasoning.

## Medium

### M1. Design an error matrix
**Goal:** API consistency.

Sketch a table for a `Products` API with columns: **Scenario | Status | Body shape (brief)**. Include at least:

- Product not found by id
- Invalid JSON in POST body
- Validation: negative price
- Duplicate SKU on create
- Successful create
- Successful update with no body returned

**Success criteria:** No scenario uses 200 for errors; 401/403/404/409/422 used appropriately; 201 has Location note.

---

### M2. curl status inspection
**Goal:** Read status line from real responses.

Using a public API (e.g. `https://httpbin.org/status/404`) or local ASP.NET minimal API:

1. Request a known **404** endpoint; paste status line.
2. Request **204** if available; confirm empty body.
3. Use `curl -w "%{http_code}"` and compare to status line.

Write what `-i` shows vs `-w`.

**Success criteria:** Correct identification of numeric code independent of reason phrase.

## Hard (stretch)

### H1. Incident triage script
**Goal:** 500 vs 502 vs 503 in layered architecture.

**Scenario:** Users report errors. Metrics show spike in **502** at nginx, **500** flat at app, **503** on `/health`.

Write a 6–10 sentence postmortem hypothesis: where failed, what to check first, what status clients should retry.

**Success criteria:** Distinguishes gateway upstream failure from app exceptions and readiness drain.

---

### H2. ASP.NET Core mapping
**Goal:** Connect codes to `Results` / filters.

List which ASP.NET Core construct likely produces each code (without full code):

- `[Authorize]` without cookie/JWT
- `return Results.NotFound()`
- `return Results.Created(...)`
- Unhandled `NullReferenceException` in middleware pipeline
- Model validation filter (your team's 400 vs 422 policy)

**Success criteria:** Maps to 401/403/404/201/500/422(or 400) with layer note (middleware vs action).
