# Git Stash: сохранение незавершённой работы без commit

> Roadmap: `0.1.9` · Узел: `0.1` — Git: основы · Глубина: практика

## Learning Objectives

После урока ты сможешь:

- Объяснить, **зачем** нужен `git stash` и когда использовать его вместо commit.
- Сохранять и восстанавливать **work-in-progress (WIP)** через `git stash push`, `pop` и `apply`.
- Просматривать и удалять stashes через `stash list`, `show`, `drop` и `clear`.
- Включать **untracked files** в stash через `-u` / `--include-untracked`.
- Описать, как stashes хранятся internally как **commits** в `refs/stash`.
- Безопасно переключать branches при uncommitted changes.

---

## Why This Matters

Ты на полпути реализации валидации login на ветке `feature/login`. В working tree лежат изменённые файлы — недописанные тесты, переработанный handler, правки конфига, которые ты ещё не проверил. В этот момент production зовёт: на `main` критический баг, нужен hotfix немедленно. Менеджер ждёт, что ты переключишь ветку, подтянешь последние изменения, исправишь проблему и запушишь fix в течение часа.

Git не даёт чисто переключить ветку, если uncommitted changes будут перезаписаны checkout'ом. Даже когда switch формально разрешён, «грязное» дерево на `main` рискует случайным commit не туда или путаницей при merge. Можно закоммитить всё с message `WIP`, но это засоряет history бессмысленными commits, которые потом придётся squash'ить или revert'ить. Можно скопировать файлы в backup вне Git — это error-prone и без version control. Можно выбросить работу — но это unacceptable, когда feature наполовину готов.

`git stash` существует именно для такого прерывания. Он временно сохраняет незавершённую работу — и staged, и unstaged изменения tracked files — и возвращает working tree в clean state, совпадающий с `HEAD`. Ты переключаешь ветку, делаешь hotfix, возвращаешься на feature и восстанавливаешь сохранённое. Операция занимает секунды и не оставляет постоянного следа на branch history, пока ты сам не решишь сделать commit после apply.

Stash — одна из самых частых команд в ежедневной работе профессиональных разработчиков. Команды, которые совмещают code review, срочные fix'ы и параллельные задачи, обращаются к ней постоянно. Понимание того, как stash устроен под капотом, также укрепляет общую картину хранения snapshots в Git: stash — не отдельная подсистема, а умное переиспользование commit object model.

## Core Concepts

Stash — это сохранённый snapshot working directory и staging area в конкретный момент времени, который хранится вне history текущей ветки до тех пор, пока ты его не apply. Когда запускаешь `git stash push`, Git записывает все изменения в tracked files — и то, что ты staged через `git add`, и то, что существовало только в working tree. С дополнительными flags, о которых речь ниже, можно захватить также untracked и даже ignored files.

После успешного stash working tree совпадает с `HEAD`. `git status` показывает, что commit'ить нечего. Можно переключать branches, pull'ить remote updates или запускать команды, требующие clean tree — например, cherry-pick или rebase, которые иначе отказали бы из-за локальных правок. Сохранённая работа живёт в stack structure под reference `refs/stash`, ожидая, когда ты её вернёшь.

Stashes организованы как **stack** с семантикой last-in-first-out. Git именует их `stash@{0}` для самой новой записи, `stash@{1}` для следующей более старой и так далее. Push добавляет запись на вершину. `git stash pop` применяет верхнюю запись и удаляет её из stack. `git stash apply` применяет верхнюю запись, но оставляет stash на месте — полезно, когда нужны те же saved changes на нескольких branches или когда хочешь backup, пока не убедишься, что apply прошёл успешно.

Удобная аналогия — стопка тарелок в столовой. Кладёшь тарелку сверху при save, pop снимает верхнюю и используешь её содержимое, apply — как будто копируешь верхнюю тарелку на стол, но оригинал остаётся в стопке.

Mental model важен: stash — **временное** хранилище для **uncommitted** work. Это не substitute для commits на feature branch. Commits живут на branches, видны в history, push'атся на remote и переживают team collaboration. Stashes по умолчанию local, невидимы для teammates и легко забываются. Используй stash для context switches на часы или максимум дни — не как archival storage для finished work.

## Under the Hood

Stashes не хранятся в «магической» папке рядом с репозиторием. Каждая stash entry — настоящий Git commit object, обычно **merge commit** с до трёх parents, связанный через special reference `refs/stash`.

Когда делаешь push stash, Git создаёт merge commit, parents которого представляют три разных snapshot состояния проекта в момент stash:

```
                    stash@{0} (merge commit)
                   /      |      \
                  /       |       \
         base commit   index tree  working tree tree
         (HEAD)        (staged)    (unstaged changes)
```

First parent — `HEAD` в момент stash, baseline commit, на котором ты стоял. Second parent захватывает состояние index — что было staged. Third parent захватывает working tree, включая unstaged изменения tracked files. Эта трёхродительская структура позволяет Git отдельно восстановить staged и unstaged части при apply.

Reference `refs/stash` указывает на самый новый stash commit. First parent каждого stash commit связывает его с предыдущим stash, образуя цепочку для `git stash list`. При pop Git накладывает diff между working-tree parent и base, восстанавливает index из index parent и сдвигает `refs/stash` к предыдущей записи.

Можно посмотреть это напрямую:

```bash
cat .git/refs/stash
git log --oneline refs/stash
git stash show -p stash@{0}
```

Последняя команда показывает patch unstaged changes относительно base. Учитывай `--include-untracked`, если использовал `-u` при push.

Почему stash реализован как commits? Git уже умеет эффективно хранить snapshots, генерировать diffs и metadata. Переиспользование object model избегает параллельного формата и даёт stash те же проверки целостности — content addressing, zlib compression, reflog на stash ref. При конфликтах apply Git использует те же conflict markers и workflow resolution, что при merge.

Записи stash также имеют reflog под `refs/stash` — ограниченное восстановление, если drop'нул не тот stash. Dropped stashes становятся unreachable и eventually garbage-collected, поэтому `git stash drop` и особенно `git stash clear` считай необратимыми без немедленного reflog.

## Syntax / Commands / API

Сохранение начинается с `git stash push`, в разговорной речи часто просто `git stash`. Базовая форма сохраняет все tracked modifications — staged и unstaged — и очищает working tree.

```bash
git stash push
git stash push -m "login validation WIP"
```

В реальных проектах всегда добавляй `-m` с осмысленным message. Сообщения по умолчанию вроде `WIP on feature/login: abc1234 Some commit message` говорят ветку и base commit, но не то, над чем ты реально работал. Ticket ID и краткое описание — `stash push -m "ticket-1234 login validation WIP"` — избавят тебя от угадывания, какой stash pop'ать через неделю.

Чтобы включить **untracked files** — файлы, которые Git видит на диске, но которые never committed — используй `-u` или `--include-untracked`:

```bash
git stash push -u -m "include scaffold files"
```

Без `-u` untracked files остаются в working tree после stash и могут по-прежнему block branch switch, если пути collide с files на target branch. Редкий flag `-a` или `--all` additionally stashes ignored files по `.gitignore`. Это нужно seldom и может затянуть build artifacts, если не careful.

Можно stash только specific paths вместо всего репозитория:

```bash
git stash push -- src/auth/login.ts appsettings.Development.json
```

Восстановление — через `pop` или `apply`. Pop apply и remove from stack:

```bash
git stash pop
git stash pop stash@{2}
```

Apply оставляет запись в stack:

```bash
git stash apply
git stash apply stash@{1}
```

Команды управления inspect и delete entries:

```bash
git stash list
git stash show
git stash show -p
git stash show -p stash@{1}
git stash drop stash@{0}
git stash clear
```

`git stash list` показывает каждую entry с index, branch context, base commit и message. `git stash show` без `-p` даёт stat summary; с `-p` — full patch. `git stash clear` удаляет все stashes — destructive и без confirmation prompt — поэтому always сначала `list`.

## Examples

Пройдём сценарий срочного переключения ветки — главную причину, по которой stash используют каждый день. Ты на `feature/login`, изменён `src/auth/login.ts`. На `main` нужен production hotfix.

```bash
git status
# modified: src/auth/login.ts

git stash push -m "login validation WIP"
git switch main
git pull origin main
# fix bug, commit, push hotfix

git switch feature/login
git stash pop
```

По шагам: `push` сохраняет правки и оставляет чистое дерево. Переключение на `main` проходит без проблем, потому что working tree clean. После hotfix возвращаешься на feature branch, `pop` накладывает сохранённый snapshot обратно. Если за время hotfix на `main` тот же файл изменился конфликтно, `pop` может выдать merge conflicts — разреши их как при обычном merge, затем при необходимости `git stash drop`, если `pop` не удалил запись из stack.

Untracked files — частая ловушка. Создаёшь временный файл для эксперимента:

```bash
echo "temp scaffold" > scratch.txt
git stash push
# scratch.txt остаётся — untracked по умолчанию не включаются

git stash push -u -m "include scaffold"
# scratch.txt в stash; working tree полностью чистое
```

Чтобы применить одни и те же сохранённые правки на нескольких ветках, используй `apply` вместо `pop`:

```bash
git stash apply stash@{0}
# работа на branch A

git switch branch-B
git stash apply stash@{0}
# те же changes на branch B; запись в stack остаётся

git stash drop stash@{0}
# удалить, когда оба apply прошли успешно
```

Если stack разросся, перед apply посмотри содержимое:

```bash
git stash list
git stash show -p stash@{2}
# убедись: это login WIP, а не эксперимент со вторника

git stash pop stash@{2}
```

Ещё типичный случай: на feature branch нужен `pull` с `main`, но локальные правки мешают. Цепочка stash → pull → pop часто быстрее, чем commit WIP только ради pull. Конфликты после pop нормальны — Git накладывает старый snapshot на новый tip; разреши как merge и прогоняй tests.

## Common Mistakes & Anti-patterns

Использовать stash как долгосрочное хранилище — самая частая ошибка. У stash нет branch, remote backup и code review trail. Накапливаются десятки записей, содержимое забывается, работа теряется после `git stash clear` или сбоя диска. Если работа — осмысленный milestone, commit на feature branch; даже WIP commit для последующего squash безопаснее полугодовалого stash.

Stash без имён превращает `git stash list` в стену одинаковых `WIP on main`. Флаг `-m` стоит пяти секунд и окупается, когда записей больше двух.

`git stash pop` после сильного изменения codebase — switch ветки, pull сотни commits, rebase — почти гарантирует conflicts. Относись к pop как к merge: ожидай conflict markers, аккуратно resolve, проверяй tests.

Забытый `-u` при untracked files оставляет их на месте. Они block `git switch` при collision path или случайно попадают в commit, потому что не были stashed и не в `.gitignore`.

`git stash clear` без просмотра `list` безвозвратно удаляет все WIP snapshots. Корзины нет.

Stash вместо commit готовой работы откладывает integration и скрывает progress от команды. Stash — для прерывания, не для откладывания «commit завтра».

После failed pop из-за conflicts Git может оставить stash entry в stack при partial apply. Читай output, заверши resolution, прогоняй tests, вручную `git stash drop`, если pop не удалил запись автоматически.

## Production & Real-World Notes

Команды часто стандартизируют stash messages с ticket IDs для traceability. Некоторые предпочитают WIP commits на feature и squash before merge — stash vs WIP commit вопрос team preference. Stash не засоряет history; WIP commits переживают restart и remote, если pushed.

CI/CD не видит stashes. Pipelines работают только с pushed commits. Stashed work невидим до apply и commit. Не используй stash как backup — build system не соберёт то, что живёт только в `refs/stash`.

IDE VS Code, Rider, JetBrains показывают stash в graphical UI; underlying commands те же. CLI помогает diagnose conflicts, когда GUI не даёт полной картины.

На pair programming иногда удобнее WIP commit и push на feature branch, чтобы напарник видел progress. Stash остаётся личным буфером одного clone — коллега не увидит его на remote, пока ты не apply и не commit.

Stashes local и per-repo; не синхронизируются с remote. `git worktree` — альтернатива для одновременной работы на двух branches без ping-pong stash; advanced topic, но полезно знать, что stash не единственный ответ на context switching.

## Comparison / Trade-offs

Stash подходит для temporary WIP при коротких прерываниях: hotfixes, срочные reviews, pull с требованием clean tree. WIP commits на feature branch — когда работа может лежать days или нужна на remote через push. Regular commits — для completed milestones в history с review.

Stash быстрее и чище для history, чем commit «на один squash». Commit безопаснее для всего, что нельзя afford to lose. Правило: если потеря stash испортит день — ten minutes ago это должен был быть commit.

Если сомневаешься между stash и WIP commit, спроси себя: «Смогу ли я объяснить коллеге через неделю, что лежит в этой записи?» Нет — добавь `-m` или commit. «Нужно ли это на remote или только пережить обед?» Только обед — stash. «Может ли apply конфликтовать с pull?» Да — после pull inspect stack через `git stash show -p` перед pop.

## Quick Reference

Save WIP: `git stash push -m "msg"`. Untracked: `git stash push -u`. List: `git stash list`. Diff: `git stash show -p`. Apply и remove: `git stash pop`. Apply и keep: `git stash apply`. Delete one: `git stash drop stash@{n}`. Delete all: `git stash clear`.

## Key Takeaways

- Stash сохраняет uncommitted work и возвращает clean working tree, совпадающий с `HEAD`.
- Stashes образуют stack — `stash@{0}` newest; pop работает по LIFO.
- `pop` apply и remove; `apply` оставляет запись для reuse.
- `-u` включает untracked files, которые иначе block branch switch.
- Internally stashes — merge commits под `refs/stash`.
- Stash local и temporary — не substitute для commits на branches.
- В реальных проектах всегда используй descriptive messages с `-m`.
