# StringBuilder vs String Concatenation

> Roadmap: `1.1.16` · Node: `1.1` — C# Language Basics · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Use **`StringBuilder`** for efficient multi-part string construction.

## Why This Matters

In lesson **`1.1.4`** you learned that **`string` is immutable**: after creation, content cannot change. Any operation that "modifies" a string creates a **new string object** on the heap. One concatenation `a + b` is cheap. Ten thousand in a loop is not — you allocate thousands of intermediate strings, increase GC pressure, and hurt latency in ASP.NET request paths building HTML, SQL, logs, or CSV exports.

**`StringBuilder`** (mutable character buffer) exists to amortize allocation: append many times, grow internal storage, produce **one final `string`** (or expose buffer via `ToString()`). Middle developers must not cargo-cult `StringBuilder` everywhere — **`string.Join`**, **`$"..."` interpolation**, and modern **`String.Create`** often beat naive `StringBuilder` for small cases. The skill is **matching tool to pattern** and understanding **what the runtime allocates**.

Every log formatter, template engine, and serializer touches this trade-off. Misjudging it shows up in profiles as Gen0 collections and LOH churn for large texts.

## Core Concepts

### Immutability Recap (Link to 1.1.4)

A **`string`** is a sequence of UTF-16 code units (`char`) with immutable length and content. The CLR may **intern** literal strings (intern pool from `1.1.4`), but concatenation produces **new instances**:

```csharp
string s = "Hello";
s += " World"; // new string allocated; old "Hello" eligible for GC
```

**`+` operator** for strings compiles to **`string.Concat`** (or similar), which allocates a result buffer sized for combined length and copies both parts. Chaining **`a + b + c`** may still create **intermediate strings** depending on compiler optimizations — do not assume single allocation for long chains in loops.

### Concatenation in Loops — The Classic Problem

```csharp
string csv = "";
foreach (var row in rows)
    csv += row.ToCsvLine() + "\n"; // BAD at scale
```

Each iteration: new string for inner concat, another for assignment — **O(n²)** total character copying over the loop lifetime. For large `rows`, this dominates.

**Fix patterns:**
1. **`StringBuilder`** with **`Append`** / **`AppendLine`**
2. **`string.Join`** when you already have **`IEnumerable<string>`**
3. **Write to `StreamWriter` / `TextWriter`** without materializing one giant string

### StringBuilder Basics

```csharp
var sb = new StringBuilder();
sb.Append("Hello");
sb.Append(' ');
sb.Append("World");
sb.AppendLine();
string result = sb.ToString(); // one final string allocation
```

**Constructors:** default, **`StringBuilder(int capacity)`**, **`StringBuilder(string initial)`**. Setting **initial capacity** near final size avoids repeated internal reallocations (typically double when full, similar to `List<T>`).

**Not thread-safe** — one builder per thread or external synchronization.

**Methods:** `Append`, `AppendLine`, `AppendFormat`, `Insert`, `Remove`, `Replace`, `Clear`, `EnsureCapacity`, `Length`, **`Capacity`**.

### string.Join and Concat

```csharp
string joined = string.Join(", ", names); // efficient for known parts
string all = string.Concat(part1, part2, part3); // params overload
```

**`Join`** computes total length (for ICollection), allocates **one** result string, copies once — often better than manual `StringBuilder` for joining collection elements.

### Interpolation

```csharp
string msg = $"User {userId} logged in at {DateTime.UtcNow:O}";
```

Compiler lowers interpolation to **`string.Format`**-like logic or **`DefaultInterpolatedStringHandler`** (.NET 6+) with stack buffer for small strings — **efficient for few parts**, not a loop replacement.

For loop building, do not use **`$"{sb}..."`** each iteration — that still allocates new strings per iteration unless using **`StringBuilder.AppendInterpolatedStringHandler`** patterns (advanced).

## Under the Hood

### String Concatenation IL and Runtime

**`string.Concat`** receives two or more strings, computes total length, allocates **`string`** of that length once, **`Span<char>`** or internal copy routines fill buffer. Two-string concat is highly optimized in modern .NET.

Long chains in a **loop** defeat optimization because each step produces new immutable intermediate.

### StringBuilder Internal Buffer

**`StringBuilder`** holds:
- **`char[] m_ChunkChars`** (older) or **linked chunks** in some implementations — modern .NET uses **chunked linked list of char buffers** for very large builders to avoid single huge array LOH issues.
- **`Length`** — logical character count; **`Capacity`** — current storage.

On **`Append`**, if room exists, write in place; else grow (new chunk or resize). **`ToString()`** allocates **one immutable string** and copies all chunks — you pay copy cost once at the end.

**`StringBuilderCache`** (internal) and **`ValueStringBuilder`** (stack-friendly struct in BCL internals) — patterns for short-lived builds without heap `StringBuilder` — awareness for reading framework source.

### GC and LOH

Very large final strings (> ~85 KB on 64-bit) may land on **Large Object Heap** — Gen2 collections costlier. Streaming output may beat one **`StringBuilder`** megastring.

**Intern pool** (`1.1.4`) does not help concatenation results unless you explicitly **`string.Intern`** — rarely done for dynamic text.

### Modern Alternatives (.NET 6+)

- **`String.Create`** — supply length and callback writing into **`Span<char>`** — zero intermediate strings when used well.
- **`CompositeFormat`** / **`Utf8JsonWriter`** for structured output without UTF-16 intermediate.
- **`ArrayPool<char>`** rented buffers in high-performance code.

Middle tier: know **`StringBuilder`** deeply; recognize when BCL APIs already optimized your scenario.

## Syntax / Commands / API

```csharp
var sb = new StringBuilder(capacity: 256);
sb.AppendFormat("{0}: {1}", name, value);
sb.Clear();

string j = string.Join('\n', lines);
string c = string.Concat(a, b, c);
```

**Thread safety:** use separate instances or lock; no concurrent **`Append`** on one builder.

## Examples

### Example 1: CSV with StringBuilder

```csharp
static string BuildCsv(IEnumerable<Row> rows)
{
    var sb = new StringBuilder(capacity: 4096);
    foreach (var row in rows)
    {
        sb.Append(row.Id).Append(',')
          .Append(row.Name).AppendLine();
    }
    return sb.ToString();
}
```

Single **`ToString()`** allocation for result (plus builder internal churn — pre-set capacity helps).

### Example 2: Join beats loop concat

```csharp
string header = string.Join(" | ", columnNames);
```

If you already have **`string[]`**, **`Join`** is clear and efficient.

### Example 3: When + is fine

```csharp
string path = basePath + "/" + fileName; // two or three parts, once
```

Readable, one concat — **no StringBuilder needed**.

## Common Mistakes & Anti-patterns

**`StringBuilder` for 2–3 parts** — overhead of builder object vs one **`Concat`**.

**Loop with `+=`** on large data — quadratic copying.

**Calling `ToString()` inside loop** — defeats purpose; build once at end.

**Shared static `StringBuilder`** without lock — corruption.

**Ignoring `capacity`** on known large output — extra reallocations.

**Building giant string** when streaming to HTTP response — memory spike; use **`WriteAsync`**.

## Production & Real-World Notes

Logging frameworks buffer with **`StringBuilder`** or write structured properties without string concat at all (Serilog message templates).

SQL dynamic filters: parameterize — do not **`StringBuilder`**-concat user input (injection). **`StringBuilder`** for SQL structure with **`SqlParameter`** is separate concern.

Roslyn and ASP.NET use **`ValueStringBuilder`**-style patterns internally — profile before micro-optimizing app code.

Benchmark with **`BenchmarkDotNet`** — assumptions about **`StringBuilder`** vs **`Join`** vary by size and runtime version.

## Comparison / Trade-offs

| Scenario | Prefer |
|----------|--------|
| 2–4 fixed parts | `+`, `Concat`, `$"..."` |
| Many parts in loop | `StringBuilder` or stream |
| Join collection | `string.Join` |
| Known exact format, perf critical | `String.Create`, UTF-8 writers |
| Huge text | Stream / chunked, not one string |

**Readability vs perf:** small hot path — measure; typical business code — prefer clarity unless profile says otherwise.

## Key Takeaways

- **`string` immutability** (`1.1.4`) makes repeated **`+=`** expensive in loops.
- **`StringBuilder`** amortizes growth; set **initial capacity** when size known.
- **`string.Join`** / **`Concat`** optimized for many scenarios — not obsolete.
- **`ToString()`** on builder copies to final immutable string once.
- Match tool to pattern; profile large text and hot paths.
- Consider streaming instead of one giant in-memory string.
