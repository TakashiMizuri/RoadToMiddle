# TimeProvider (.NET 8+): тестируемое время и задержки

> Roadmap: `1.4.36` · Node: `1.4` — C# async · Depth: **практика**

## Learning Objectives

После урока ты сможешь:

- Объяснить, почему **`DateTime.UtcNow`** и **`Task.Delay`** делают unit-тесты **медленными и нестабильными**.
- Использовать **`TimeProvider`** и внедрять его через DI для **виртуального времени** в сервисах.
- Регистрировать **`TimeProvider.System`** в production и подменять **`FakeTimeProvider`** в тестах.
- Продвигать виртуальное время через **`Advance`** / **`SetUtcNow`** и наблюдать завершение **`Task.Delay`** без реального ожидания.
- Применять **`TimeProvider`** к **таймаутам**, **retry backoff** и проверке **истечения токенов**.
- Избегать типичных ошибок: статические вызовы времени, смешение реальных и fake-часов.

## Why This Matters

Production-код постоянно зависит от **времени**: истечение сессии, окна rate limiting, задержки retry, TTL кэша, фоновые задачи по расписанию. Естественная реализация — `DateTime.UtcNow` и `Task.Delay(500)`. В разработке это работает. В тестах становится проблемой.

Тест с `await Task.Delay(TimeSpan.FromMinutes(5))` буквально ждёт пять минут — неприемлемо в CI. Разработчики сокращают задержки через `#if DEBUG`, и тогда **тесты не проверяют production-логику таймингов**. Другие мокают `Task.Delay` через reflection — хрупко и борется с фреймворком.

В **.NET 8** появился **`TimeProvider`** — абстрактные часы и фабрика таймеров. **`TimeProvider.System`** делегирует реальному OS-времени. **`FakeTimeProvider`** (пакет `Microsoft.Extensions.TimeProvider.Testing`) позволяет **мгновенно продвигать время** в тестах, пока async-код с `Delay` ведёт себя так, будто прошло реальное время. ASP.NET Core регистрирует `TimeProvider.System` в host по умолчанию.

Это не академия. Middle-разработчики пишут retry-политики, Polly timeouts, фоновые sweepers. На собеседованиях спрашивают: «Как протестировать "подожди 30 секунд и упади"?» **`TimeProvider`** — современный ответ, выровненный с фреймворком.

## Core Concepts

### Проблема прямых time API

`DateTime.UtcNow`, `DateTimeOffset.UtcNow`, `Stopwatch` и `Task.Delay` — **статические** или глобальные для runtime. Они привязывают логику к системным часам. Последствия:

- **Flaky tests** — падения около полуночи, DST, медленного CI.
- **Медленные тесты** — реальные задержки накапливаются.
- **Нетестируемые ветки** — «токен истекает через 15 минут» требует time travel.

Решение — **dependency inversion**: код, которому нужно время, получает абстракцию. До .NET 8 изобретали `IClock`, `ISystemClock`. .NET 8 стандартизировал **`TimeProvider`**.

### Что даёт TimeProvider

`TimeProvider` — abstract class в корневом namespace `System`. Ключевые члены:

- **`GetUtcNow()`** — текущий UTC-момент как `DateTimeOffset`.
- **`GetLocalNow()`** — локальное время.
- **`CreateTimer(callback, state, dueTime, period)`** — таймеры, привязанные к timeline провайдера.
- **`Delay(delay, cancellationToken)`** — async-задержка на timeline этого провайдера.
- **`TimeProvider.System`** — singleton с реальным временем.

Когда вызываешь `await timeProvider.Delay(TimeSpan.FromSeconds(30), ct)`, **`FakeTimeProvider`** не блокируется на 30 секунд. Задержка регистрирует continuation; когда тест вызывает **`fake.Advance(TimeSpan.FromSeconds(30))`**, pending delays завершаются — **синхронно для теста**, но через реальные async state machines.

### FakeTimeProvider в тестах

Пакет: **`Microsoft.Extensions.TimeProvider.Testing`**. Типичный workflow:

```csharp
var fake = new FakeTimeProvider(startDateTime: new DateTimeOffset(2026, 1, 1, 12, 0, 0, TimeSpan.Zero));
var service = new TokenService(fake);

var refreshTask = service.RefreshAfterDelayAsync(TimeSpan.FromMinutes(5));
fake.Advance(TimeSpan.FromMinutes(5));
await refreshTask; // без wall-clock ожидания
```

`SetUtcNow` прыгает на абсолютный момент. `Advance` двигает относительно. Оба запускают callback таймеров и delays, которые due.

### DI в ASP.NET Core

С .NET 8 `WebApplication.CreateBuilder` регистрирует:

```csharp
builder.Services.AddSingleton(TimeProvider.System);
```

Внедрение в сервисы:

```csharp
public sealed class RateLimiterService(TimeProvider timeProvider)
{
    public async Task<bool> TryAcquireAsync(CancellationToken ct)
    {
        await timeProvider.Delay(TimeSpan.FromMilliseconds(100), ct);
        // ...
    }
}
```

В тестах — замена:

```csharp
services.AddSingleton<TimeProvider>(new FakeTimeProvider());
```

Production — реальное время; тесты — подмена провайдера.

## Under the Hood

`FakeTimeProvider` держит внутреннюю **очередь с приоритетом** запланированной работы: delays, timers, аналоги `PeriodicTimer` с due time на fake timeline. `Advance` сливает все записи с `due <= newNow`, вызывает callbacks и завершает `Task` от `Delay`. Это имитирует thread pool timer callbacks — но под контролем теста.

`TimeProvider` **не** подменяет `CancellationTokenSource.CancelAfter` автоматически — тот всё ещё использует реальные таймеры. Для полной детерминизма предпочитай `await timeProvider.Delay(timeout, ct)` в своих timeout-обёртках.

`PeriodicTimer` в .NET 8 имеет overload с `TimeProvider` — периодические polling-циклы тестируются так же.

## Syntax / Commands / API

| API | Назначение |
|-----|------------|
| `TimeProvider.System` | Production, реальное время |
| `timeProvider.GetUtcNow()` | Injectable «сейчас» |
| `await timeProvider.Delay(ts, ct)` | Injectable async delay |
| `new FakeTimeProvider(start)` | Test double |
| `fake.Advance(delta)` | Сдвинуть fake clock вперёд |
| `fake.SetUtcNow(instant)` | Прыжок на абсолютное время |

## Examples

### Retry с backoff

```csharp
public sealed class ApiClient(TimeProvider time)
{
    public async Task<string> GetWithRetryAsync(Func<Task<string>> call, CancellationToken ct)
    {
        var delay = TimeSpan.FromSeconds(1);
        for (var attempt = 0; attempt < 5; attempt++)
        {
            try { return await call(); }
            catch (HttpRequestException) when (attempt < 4)
            {
                await time.Delay(delay, ct);
                delay += delay;
            }
        }
        throw new InvalidOperationException("Unreachable");
    }
}
```

Тест: `FakeTimeProvider`, `Advance` на 1s, 2s, 4s — проверка числа попыток без sleep.

### Проверка истечения токена

```csharp
public sealed class SessionValidator(TimeProvider time)
{
    public bool IsExpired(DateTimeOffset expiresAt) =>
        time.GetUtcNow() >= expiresAt;
}
```

Тест: `fake.SetUtcNow(expiresAt.AddSeconds(1))` → `true`.

## Common Mistakes & Anti-patterns

**Вызов `DateTime.UtcNow` внутри класса с injected `TimeProvider`** — тестируется только injected path; смешанные часы дают confusing failures.

**Advance до старта async-операции** — delays, запланированные после advance, требуют следующего advance; порядок важен.

**Один `Task.Delay` «просто так»** — ломает детерминизм.

**Ожидание, что `CancelAfter` уважает `FakeTimeProvider`** — не уважает.

**Один `FakeTimeProvider` на параллельные тесты** — изоляция per-test.

## Production & Real-World Notes

Команды иногда оборачивают `TimeProvider` в domain `IClock` только с `GetUtcNow()`, если таймеры редки — но для нового .NET 8+ кода предпочитай raw `TimeProvider` для согласованности с `PeriodicTimer`, Polly v8.

EF Core не использует `TimeProvider` для DB timeouts — они остаются на уровне драйвера.

Логи в production должны использовать реальное время в инфраструктуре логирования; `TimeProvider` — для **application logic**.

## Comparison / Trade-offs

| Подход | Плюсы | Минусы |
|--------|-------|--------|
| `TimeProvider` + `FakeTimeProvider` | Официальный, Delay/timers | .NET 8+, test package |
| Custom `IClock` | Старые TFM | Дублирует framework |
| `Task.Delay` только в prod | Просто | Нетестируемо |
| Короткие delays в `#if DEBUG` | Быстрый hack | Tests ≠ prod |

## Key Takeaways

- Прямой `DateTime` и `Task.Delay` привязывают к wall clock — плохо для тестов.
- **`TimeProvider`** — абстракция .NET 8+ для now, delays, timers.
- **`FakeTimeProvider.Advance`** завершает pending delays без реального ожидания.
- Регистрируй в DI; подменяй в `WebApplicationFactory`.
- Используй последовательно — один `Task.Delay` ломает паттерн.
- `CancelAfter` всё ещё на реальном времени.
