# Answers: OrderBy, ThenBy — Stable Sort

## Answers

### 1. [Recall]
**Answer:** `IOrderedEnumerable<T>` (enables `ThenBy`).
**Why:** Type encodes "already ordered" state.

### 2. [Recall]
**Answer:** `ThenBy` adds secondary key; second `OrderBy` replaces entire sort with new primary key only.
**Why:** Common API mistake.

### 3. [Understanding]
**Answer:** Preserved from input (stable sort in LINQ to Objects).
**Why:** Merge sort stability property.

### 4. [Understanding]
**Answer:** Unique key guarantees deterministic total order across queries/pages.
**Why:** SQL without unique ORDER BY can shuffle ties between requests.

### 5. [Application]
**Answer:** `.OrderBy(e => e.Department).ThenByDescending(e => e.Salary).ThenBy(e => e.Name)`
**Why:** ThenBy chain for multi-level.

### 6. [Application]
**Answer:** Before `Skip/Take` on `IQueryable` — same query pipeline.
**Why:** SQL ORDER BY ... OFFSET ... FETCH.

### 7. [Analysis]
**Answer:** Only by `Name` (Department sort lost).
**Why:** Second OrderBy overwrites.

### 8. [Analysis]
**Answer:** Forces client evaluation — may load entire table into memory.
**Why:** AsEnumerable breaks IQueryable provider.

### 9. [Understanding]
**Answer:** `.Select((item,i) => new {item,i}).OrderBy(x=>x.item.P).ThenBy(x=>x.i).Select(x=>x.item)` or stable sort relies on input order + ThenBy unique field.
**Why:** Index as explicit tie-break.

### 10. [Application]
**Answer:** `.OrderBy(s => s, StringComparer.OrdinalIgnoreCase)`
**Why:** Comparer overload on key or element.
