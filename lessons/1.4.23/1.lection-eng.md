# The Awaitable Pattern: GetAwaiter and INotifyCompletion

> Roadmap: `1.4.23` · Node: `1.4` — Async + ASP.NET preview · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Describe the **awaitable pattern** contract: `GetAwaiter()`, `IsCompleted`, `GetResult()`, `OnCompleted`.
- Explain how the C# compiler **lowers** `await` into state machine calls on an **awaiter**.
- Distinguish **INotifyCompletion** vs **ICriticalNotifyCompletion** and when `UnsafeOnCompleted` is used.
- Implement a **custom awaitable** for yielding, throttling, or bridging legacy APIs.
- Recognize **TaskAwaiter**, **ValueTaskAwaiter**, and **ConfiguredTaskAwaitable** in disassembled async code.
- Avoid pitfalls: double `GetResult()`, awaiting non-task types incorrectly, capturing execution context in custom awaiters.

## Why This Matters

Every `await` in C# looks like syntactic sugar over a small, fixed protocol — not magic tied exclusively to `Task`. When you `await task`, the compiler calls `task.GetAwaiter()`, checks `IsCompleted`, either calls `GetResult()` immediately or registers `OnCompleted` with a continuation delegate. Libraries like **Dapper**, **Polly**, and **Entity Framework** expose `Task`-based APIs, but framework authors sometimes expose **custom awaitables** (`YieldAwaitable`, `ConfiguredTaskAwaitable`, `ValueTask`) to control scheduling and allocations.

Middle developers who understand the pattern can **debug** weird continuation ordering, **wrap** legacy callback APIs (`BeginInvoke`/`EndInvoke`, socket callbacks) into `await`-friendly surfaces, and **read** Roslyn-generated state machines without fear. In ASP.NET Core, most awaits use `Task` with no `SynchronizationContext`, but **custom awaiters** still appear in throttling (`SemaphoreSlim.WaitAsync`), **channels**, and **IAsyncEnumerable**. Knowing `GetAwaiter()` is the universal key.

## Core Concepts

### The Pattern, Step by Step

For expression `await expr`, the compiler roughly generates:

```csharp
var awaiter = expr.GetAwaiter();
if (!awaiter.IsCompleted)
{
    // state machine: register continuation, return from MoveNext
    awaiter.OnCompleted(stateMachine.MoveNext);
    return;
}
var result = awaiter.GetResult();
// continue with result
```

**Requirements on `expr`:** must expose instance or extension method `GetAwaiter()` returning a type that implements the **awaiter pattern** (C# spec: struct or class with `IsCompleted`, `GetResult()`, and optionally `OnCompleted` / `UnsafeOnCompleted`).

**GetAwaiter()** can be an extension method — `Task` uses `TaskAwaiter GetAwaiter(this Task task)` in `System.Runtime.CompilerServices`.

**IsCompleted** — synchronous fast path. Many operations complete immediately (cached result, completed task, lock already acquired).

**GetResult()** — produces the outcome or throws stored exception. For `Task`, exceptions are **marshaled** here, not on the task object directly when observed via await.

**OnCompleted(Action continuation)** — registers callback when operation pending. Must not run continuation synchronously if still incomplete (unless completing between check and register — race handled by awaiter).

### INotifyCompletion and ICriticalNotifyCompletion

```csharp
public interface INotifyCompletion
{
    void OnCompleted(Action continuation);
}

public interface ICriticalNotifyCompletion : INotifyCompletion
{
    void UnsafeOnCompleted(Action continuation);
}
```

If the awaiter implements **ICriticalNotifyCompletion**, the compiler prefers **UnsafeOnCompleted**, which **does not flow ExecutionContext** (logical call context: `AsyncLocal`, security context in older models). That reduces capture cost for high-performance awaitables (`Task`, `ValueTask`) at the expense of not restoring context in the continuation unless the awaiter chooses to.

**ExecutionContext** capture matters for `AsyncLocal<T>` and ambient logging scopes. Most ASP.NET Core middleware sets context explicitly; still, library awaiters must document whether they capture context.

### Custom Awaitable Example: Yield to Thread Pool

```csharp
public readonly struct YieldAwaitable
{
    public YieldAwaiter GetAwaiter() => new YieldAwaiter();
    public struct YieldAwaiter : ICriticalNotifyCompletion
    {
        public bool IsCompleted => false;
        public void GetResult() { }
        public void OnCompleted(Action continuation) =>
            ThreadPool.QueueUserWorkItem(_ => continuation());
        public void UnsafeOnCompleted(Action continuation) =>
            ThreadPool.UnsafeQueueUserWorkItem(_ => continuation(), preferLocal: false);
    }
}

// Usage: await Task.Yield(); // built-in equivalent idea
```

This forces async method to **resume on thread pool**, splitting long synchronous sections — useful to avoid blocking ASP.NET request thread in edge cases (prefer true async I/O instead when possible).

### TaskAwaiter and ConfiguredTaskAwaitable

`task.ConfigureAwait(false)` returns **ConfiguredTaskAwaitable** whose awaiter **does not** marshal back to captured `SynchronizationContext`. In ASP.NET Core there is no request `SyncContext`, so `ConfigureAwait(false)` is often a no-op for correctness but still documents intent in libraries.

Disassemble async method with [SharpLab](https://sharplab.io/) — you will see `AwaitUnsafeOnCompleted` vs `AwaitOnCompleted` based on awaiter type.

## Under the Hood

Roslyn async state machine implements **IAsyncStateMachine**. Fields hold awaiter instances, `__state` integer, and hoisted locals. Each `await` splits `MoveNext` into blocks; `BuildAsync` wraps it in a `Task` or `AsyncTaskMethodBuilder`.

**ValueTask** can wrap **IValueTaskSource** — pooling completion sources to avoid `Task` allocation for hot paths (`HttpClient` internal paths, `MemoryStream.ReadAsync` fast paths). The awaiter still follows the same pattern but may complete synchronously without allocating a `Task`.

**GetResult()** on faulted awaiter **throws** the stored exception with preserved stack when using `ExceptionDispatchInfo.Capture` inside the task machinery — why `await` gives clean stacks vs `.Result`.

Race between `IsCompleted == false` and completion before `OnCompleted` register: correct awaiters use **CompareExchange** or gate so continuation runs **exactly once**.

## Syntax / Commands / API

| Member | Role |
|--------|------|
| `GetAwaiter()` | Entry point from awaited expression |
| `IsCompleted` | Sync completion check |
| `GetResult()` | Result or throw |
| `OnCompleted(Action)` | Schedule continuation with context |
| `UnsafeOnCompleted(Action)` | Schedule without context capture |

Extension pattern:

```csharp
public static TaskAwaiter GetAwaiter(this Task task) => ...;
```

Custom wrapper for legacy callback API:

```csharp
public Task<T> ToTask<T>(...) // TaskCompletionSource + callback
// consumer: await ToTask(...)
```

## Examples

### Example 1: Manual await expansion (mental model)

```csharp
Task<string> t = http.GetStringAsync(url);
TaskAwaiter<string> awaiter = t.GetAwaiter();
if (!awaiter.IsCompleted) { /* register MoveNext */ }
string body = awaiter.GetResult();
```

Same semantics as `var body = await http.GetStringAsync(url);`.

### Example 2: TaskCompletionSource bridge

```csharp
public Task<int> DelayResult(int ms, int value)
{
    var tcs = new TaskCompletionSource<int>();
    _ = Task.Run(async () =>
    {
        await Task.Delay(ms);
        tcs.SetResult(value);
    });
    return tcs.Task; // awaitable via standard Task awaiter
}
```

### Example 3: Invalid double-await on one-shot awaiter

Some custom awaiters are **single-use**. Awaiting twice or calling `GetResult()` twice may throw — document lifetime.

## Common Mistakes & Anti-patterns

**Assuming only `Task` is awaitable.** `ValueTask`, `Task<T>`, `ConfiguredTaskAwaitable`, `IAsyncEnumerable`, custom types — all via pattern.

**Calling `GetResult()` on incomplete awaiter** — undefined behavior; always check `IsCompleted` or use compiler `await`.

**Custom awaiter that invokes continuation synchronously on wrong thread** — can cause deep recursion or UI thread violations.

**Ignoring `UnsafeOnCompleted` contract** — must not throw after taking responsibility for continuation.

**async void** event handlers — different exception path; not awaitable pattern issue but often confused.

## Production & Real-World Notes

Library code targeting ASP.NET and UI should expose **`Task`/`ValueTask`** return types, not custom awaitables, unless optimization demands it. Custom awaitables appear inside frameworks (`Polly` resilience contexts, `ChannelReader` `WaitToReadAsync`).

When wrapping **fire-and-forget** legacy APIs, **`TaskCompletionSource`** with proper exception mapping is the production-standard bridge — implements awaiter via `Task`.

Performance-sensitive code uses **`ValueTask`** when operation often completes synchronously (cache hit) — but **do not await same ValueTask twice** unless documented reusable.

## Comparison / Trade-offs

| Type | Allocation | Await twice? | Typical use |
|------|------------|--------------|-------------|
| `Task` | Often allocates | Yes | Public APIs, composition |
| `ValueTask` | Pooling possible | Careful | Hot path, sync-complete common |
| Custom awaitable | Varies | Read docs | Scheduling, interop |
| `ConfigureAwait(false)` | None | N/A | Library code, UI optional |

## Quick Reference

```
await expr;
  → expr.GetAwaiter()
  → if !IsCompleted → OnCompleted/UnsafeOnCompleted(continuation)
  → GetResult()
```

Built-ins: **TaskAwaiter**, **ValueTaskAwaiter**, **YieldAwaitable**.

## Key Takeaways

- `await` is a fixed protocol: **GetAwaiter**, **IsCompleted**, **GetResult**, **OnCompleted**.
- Compiler lowers `await` to state machine blocks — not runtime reflection.
- **ICriticalNotifyCompletion** avoids ExecutionContext capture via **UnsafeOnCompleted**.
- **GetResult()** throws faulted exceptions with proper dispatch.
- Custom awaitables enable interop and scheduling; most app code uses **Task**.
- **ValueTask** follows same pattern with pooling and stricter reuse rules.
- Understanding the pattern explains ConfigureAwait, continuations, and debugging async.
