## Why This Matters

Узел `1.4` дал mechanics. Реальный ASP.NET solution — **сотни методов**. Один `.Result` в singleton или `.ToList()` в hot endpoint ломает архитектуру. Нужен **checklist** — как security review — на каждый PR с I/O.

Урок синтезирует `1.4.1`–`1.4.41` в **actionable gates**. После `1.4.42` — **CLR** (`1.5.1`) с operationalized async в web stack.

## Core Concepts

### Принцип

**Async all-the-way** ≠ «каждый метод async». Это: от **Kestrel** до **DB/network** нет sync blocking на I/O. CPU-bound — отдельное решение.

Правило: **если есть `*Async` — await; никогда `.Result`.**

### Category 1: Endpoints

- `Task`/`Task<T>` return
- `CancellationToken`
- Нет `.Result`/`.Wait()`
- Нет sync EF
- Нет `Task.Run` для DB/network

### Category 2: Services

- I/O → `Task` + `*Async` naming
- Token propagated
- Нет `.Result` в properties
- Один scoped `DbContext` per flow

### Category 3: EF Core

- `*Async` queries и `SaveChangesAsync`
- `await using` для manual connections
- Projection в query

### Category 4: HttpClient

- `IHttpClientFactory`
- Await `SendAsync`/`GetAsync`
- Linked cancellation tokens

### Category 5: Middleware/Filters

- `await next(context)` / `await next()`

### Category 6: Background

- Нет blind `_ = DoAsync()`
- `IServiceScopeFactory` для background scope
- `IHostedService` для warmup

### Category 7: DI

- No async ctor
- No singleton scoped `DbContext`

### Category 8: CI

- Grep `.Result`, `.Wait(`
- Analyzers AsyncFixer/VSTHRD
- Load test thread pool

## Audit Workflow

1. **Grep** automated
2. **Trace** 3 critical endpoints (`1.4.40`)
3. **Load test** 100 concurrent
4. **PR checkbox**
5. **Allowlist** legacy shims

```bash
rg "\.(Result|Wait\(|GetAwaiter\(\)\.GetResult\()" src/ --glob "*.cs"
```

## Common Mistakes

- Audit без grep
- Исключения «маленьким» sync calls
- `async void` в app code
- Singleton `DbContext`
