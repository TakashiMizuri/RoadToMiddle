# async void — exception handling nightmare

> Roadmap: `1.4.13` · Node: `1.4` — Async/await · Depth: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Объяснить, зачем **`async void`** и почему только **event handlers**.
- Описать propagation exceptions из **`async void`** vs **`async Task`**.
- Связать **`async void`** с **state machine** (`1.4.6`–`1.4.7`), но другим completion channel.
- Избегать **`async void`** в business logic, tests, ASP.NET.
- Оборачивать event handlers в **try/catch** — caller не может **await**.

---

## Why This Matters

Event handlers требуют **`void`** — **`async void`**. Exceptions после первого `await` **не** на faulted **`Task`**; они идут в **unhandled** / **`SynchronizationContext`**. В ASP.NET Core **`async void`** action — process crash или lost errors.

Middle refactor → **`async Task`** + safe fire-and-forget (`1.4.30`).

---

## Core Concepts

### async Task vs async void

**`async Task`**: exceptions на **`Task`**; caller **`await`** + catch.

**`async void`**: та же state machine **`MoveNext`**, но **нет Task**. Exception → **`AsyncVoidMethodBuilder`** → unhandled / UI pipeline.

```csharp
private async void Button_Click(object sender, EventArgs e)
{
    try { await LoadDataAsync(); }
    catch (Exception ex) { _logger.LogError(ex); }
}

public async void ProcessOrder(int id) { } // NEVER
```

### ASP.NET Core

Actions — **`Task<IActionResult>`**, не **`async void`**.

### Why void for events?

Signature **`void`**. Extract to **`async Task`** + void wrapper с try/catch.

---

## Under the Hood

**`AsyncVoidMethodBuilder`** vs **`AsyncTaskMethodBuilder`**. Оба **`MoveNext`**.

Нет Task → нельзя **`await voidMethod()`**.

xUnit: **`async void`** tests broken.

---

## Key Takeaways

- **`async void`** — UI events + try/catch only.
- Exceptions не к caller.
- **`async Task`** everywhere else.
- ASP.NET: never **`async void`** actions.
