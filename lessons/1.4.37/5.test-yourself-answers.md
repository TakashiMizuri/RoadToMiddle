## Answers

### 1. [Recall]
**Answer:** Debug → Windows → Tasks (или Parallel Windows → Tasks).
**Why:** Стандартное окно списка Task при отладке managed кода.

### 2. [Recall]
**Answer:** Task создан, но ещё не запланирован/не начал выполнение — ждёт активации (часто незавершённая цепочка или deadlock до scheduler).
**Why:** Застрявшие в этом статусе tasks — красный флаг при hang.

### 3. [Recall]
**Answer:** `Namespace.Type.<MethodName>d__N.MoveNext()` — IAsyncStateMachine.
**Why:** `d__N` — generated struct; MoveNext — шаг state machine.

### 4. [Understanding]
**Answer:** `await` разрывает синхронный stack; continuation на другом thread; без enhanced async trace физические frames обрываются на TaskAwaiter.
**Why:** Нужны Tasks window и .NET 6+ async stack reconstruction.

### 5. [Understanding]
**Answer:** Request thread мог вернуться в Kestrel; реальная работа на thread pool — Parallel Stacks показывает blocked threads, Tasks — logical operations.
**Why:** ASP.NET async освобождает request thread после await.

### 6. [Understanding]
**Answer:** `async void` не возвращает Task — нельзя await/track; исключения идут напрямую в SynchronizationContext как unhandled.
**Why:** Нет объекта Task в Tasks window для полноценного tracking.

### 7. [Application]
**Answer:** (1) Breakpoint/step to await (2) Open Tasks, note task Id/status (3) Break All if stuck (4) Parallel Stacks + Threads for block on I/O or sync primitive; search `.Result`/`.Wait()`.
**Why:** Стандартный Middle workflow для hang.

### 8. [Application]
**Answer:** Debugger break on `TaskScheduler.UnobservedTaskException`; или смотреть Faulted tasks в Tasks window до GC.
**Why:** Fire-and-forget без await теряет exception path.

### 9. [Analysis]
**Answer:** Sync-over-async deadlock — один thread `.Result` на async method, другой ждёт context/lock.
**Common mistake:** Legacy library `.Wait()` в filter/middleware.

### 10. [Analysis]
**Answer:** UI/state bug отдельно от backend task — проверь client await, SignalR callback, другой Task, cancellation, wrong task awaited.
**Why:** Completed server task ≠ UI updated.

### 11. [Understanding]
**Answer:** Нет — Task всё ещё виден; меняется только контекст resume, не регистрация в debugger.
**Why:** ConfigureAwait — про SyncContext, не про Task lifetime.

### 12. [Application]
**Answer:** `dotnet-dump` / WinDbg, `dotnet-trace`, structured logs + correlation ID, APM (Application Insights, OpenTelemetry).
**Why:** Production lacks interactive Tasks UI.
