# var и явные типы — правила читаемости

> Roadmap: `1.1.5` · Node: `1.1` — C#: основы языка · Depth: понимание

## Learning Objectives

После этого урока ты сможешь:

- Объяснить, что **`var` — не dynamic typing**: компилятор по-прежнему назначает **статический тип** на этапе компиляции.
- Выбирать между **`var` и явным типом** по правилам читаемости, принятым в профессиональных C#-командах.
- Распознавать, когда **`var` скрывает важную информацию**, а когда убирает шум.
- Корректно применять **`var`** с обязательным initializer, anonymous types и LINQ-проекциями.
- Связать type inference с **примитивными типами** из `1.1.3` (`int`, `long`, `decimal`, `bool` и др.).

---

## Why This Matters

В `1.1.3` ты познакомился с конкретными примитивами: `int` для целых, `decimal` для денег, `double` для приближённых вычислений, `bool` для условий. Объявления выглядели прозрачно: `int count = 0`, `decimal price = 19.99m`. Явный тип слева помогает учиться — всегда видно, с чем работаешь.

Когда проект растёт, повторять очевидные типы становится шумом. C# предлагает **`var`** — ключевое слово, которое говорит компилятору: *выведи тип из выражения справа*. Команды спорят об этом постоянно: одни запрещают `var`, другие пишут его везде. Ни один крайний вариант не идеален. Middle-разработчик выбирает исходя из **того, что нужно читателю кода**, а не из личных предпочтений.

Ставки выше, чем «стиль». Ошибочные представления о `var` ведут к багам: ожидание смены типа в runtime, `var` там, где правая часть неочевидна, или скрытое сужение/расширение (`int` вместо `long`). Этот урок даёт **правила решения**, а не догму.

## Core Concepts

### Что на самом деле делает `var`

Когда ты пишешь:

```csharp
var count = 42;
```

компилятор видит литерал `42`, определяет тип `int` и обрабатывает объявление **точно так же**, как `int count = 42`. Нет обёртки `object`, нет lookup типа в runtime, нет гибкости как в JavaScript. **`var` — compile-time type inference**, и точка.

Если потом написать `count = "hello"`, компилятор отклонит код — `count` остаётся `int`. IDE при наведении покажет `int`. IL для `var count = 42` и `int count = 42` **идентичен**.

Новички иногда думают, что `var` означает «переменная может хранить что угодно». Не может. C# остаётся **statically typed**. `var` лишь избавляет от повторения имени типа, когда компилятор уже надёжно его выводит.

### Правило, которое нельзя нарушать

**`var` требует initializer.** Нельзя написать:

```csharp
var x;           // CS0818
var y = null;    // CS0815 — null без целевого типа
```

Компилятор узнаёт тип из правого выражения. Нет выражения — нет типа. Для `null` нужен конкретный target type: `string? s = null`, потому что один `null` не несёт информации о типе.

### Когда явный тип улучшает читаемость

Используй **явный тип**, когда тип **неочевиден из правой части** или когда **сам тип — суть объявления**.

Финансовый пример:

```csharp
decimal total = CalculateOrderTotal(orderId);
```

`CalculateOrderTotal` может вернуть `double`, `decimal` или `Money`. Явный `decimal` документирует намерение: нужна точность денег. С `var total = CalculateOrderTotal(orderId)` каждый читатель должен открыть сигнатуру метода.

Ещё случай — числовые литералы из `1.1.3`:

```csharp
long fileSize = 2_147_483_648L;
```

При `var fileSize = 2_147_483_648` литерал попадает в диапазон `int`, компилятор выведет **`int`**, не `long`. Диапазон теряется незаметно. Явный `long` или суффикс `L` фиксируют правильный примитив.

Явные типы обязательны на **публичной поверхности API**: параметры, return types, public fields/properties. Потребители и документация опираются на видимые типы. **`var` — почти всегда для локальных переменных**.

### Когда `var` улучшает читаемость

Используй **`var`**, когда тип **многословен и очевиден** справа — особенно для generic-инстанцирований:

```csharp
var customers = new List<Customer>();
var lookup = new Dictionary<int, Order>();
```

До target-typed `new` (C# 9) приходилось дублировать `List<Customer>` с обеих сторон без добавления информации. `var` оставляет имя типа там, где его уже задаёт constructor.

`var` **единственный** способ назвать anonymous type из LINQ:

```csharp
var summary = orders.Select(o => new { o.Id, o.Total });
```

Для `foreach` с очевидным API:

```csharp
foreach (var line in File.ReadLines(path))
{
    Console.WriteLine(line);
}
```

`line` явно `string`. Писать `string line` тоже нормально — команды расходятся — но `var` не дублирует то, что API уже объявил.

### Практическая rubric для читаемости

**Явный тип**, когда правая часть — вызов метода с неочевидным return type, когда важен размер числа (`int` vs `long`, `float` vs `double` vs `decimal`), когда тип несёт domain-смысл (`InvoiceId` vs `int`).

**`var`**, когда классический паттерн `Type name = new Type(...)`, anonymous/tuple без имени типа, когда справа уже есть `new`, `as` или cast с именем типа, когда имя переменной кодирует тип (`customerCount`, `isValid`).

Если сомневаешься: **«Что reviewer должен понять, глядя только на левую часть `=`?»** Если очевидно в пределах экрана — `var` безопасен. Если нет — пиши тип явно.

### `var` и примитивы из 1.1.3

Литералы имеют типы по умолчанию:

| Literal | Inferred type |
|---------|---------------|
| `42` | `int` |
| `42L` | `long` |
| `3.14` | `double` |
| `3.14f` | `float` |
| `19.99m` | `decimal` |
| `'A'` | `char` |
| `true` | `bool` |

`var price = 19.99` → **`double`**, не `decimal`. Для денег: `decimal price = 19.99m` или `var price = 19.99m`. **`var` не меняет правила чисел** — следует им.

## Under the Hood

Type inference выполняется в **compiler front end** до генерации IL. Roslyn анализирует semantic model initializer, разрешает overloads, применяет numeric conversions и привязывает `TypeSymbol` к переменной. Символ зафиксирован на lifetime scope.

В IL локальные переменные имеют конкретные типы в metadata — decompiler покажет `int32`, не `var`. **Нулевая разница в runtime** между `var` и явным объявлением.

Спецификация C# задаёт **best common type** и **target-typed inference**. Inference может провалиться:

```csharp
var x = condition ? 1 : 2.0;   // double
var y = condition ? 1 : "two";   // CS0173
```

Тогда компилятор просит явный тип или cast.

## Common Mistakes & Anti-patterns

**`var` как dynamic** — неверно; все compile-time проверки на месте.

**`var data = GetData();`** — тип скрыт; лучше явный тип или говорящее имя метода.

**Забытые суффиксы:** `var rate = 0.1` → `double`; для денег нужен `m`.

**Глобальный запрет или mandate `var`** — используй rubric, не крайности.

## Production & Real-World Notes

Команды настраивают `.editorconfig` / StyleCop: «`var`, когда тип apparent». Analyzers в Rider/VS могут enforce правила. Public API библиотек почти никогда не использует `var` в сигнатурах.

Code review: «`var` или explicit?» — вопрос читаемости, не performance.

---

## Comparison / Trade-offs

Явный тип везде даёт максимальную локальную ясность, но verbose и дублирует generics. `var` везде сокращает шум, но скрывает return types и создаёт numeric traps. Rubric-based подход — баланс, но требует договорённости в команде.

## Quick Reference

| Situation | Recommendation |
|-----------|------------------|
| `new List<T>()` | `var` или target-typed `new` |
| Неочевидный return method | Explicit |
| Anonymous / LINQ | `var` (required) |
| Numeric literals | Explicit или suffix |
| Public members | Explicit always |

---

## Key Takeaways

- `var` = **static compile-time inference**, не dynamic.
- **Initializer обязателен**; тип фиксирован после компиляции.
- **`var`, когда тип obvious справа**; **explicit, когда тип — сообщение**.
- **`var` следует правилам литералов из `1.1.3`**.
- **Нет runtime-разницы** с explicit locals.
- **Rubric, не dogma.**
