## Answers

### 1. [Recall]
**Answer:** `ValueTask` / `ValueTask<T>` — **value types (struct)**, `Task` — **reference type (class)**.

**Why:** Struct может не аллоцировать heap-объект на sync path; `Task` почти всегда heap-backed.

---

### 2. [Recall]
**Answer:** После **первого** завершённого await повторный await **не определён** — может вернуть мусор, чужой результат, throw или «случайно» работать.

**Why:** Pooled `IValueTaskSource` reset и reuse для другого caller'а.

---

### 3. [Understanding]
**Answer:** Sync path может вернуть `new ValueTask<T>(result)` **без создания `Task`**. `Task<T>` как return type почти всегда требует heap `Task` или обёртку state machine.

**Why:** Discriminated struct хранит результат inline.

---

### 4. [Understanding]
**Answer:** `async ValueTask` компилятор генерирует state machine; при suspend создаётся backing **`Task`** — теряется смысл struct + добавляется overhead. Лучше `async Task`.

**Why:** ValueTask выигрывает на **sync-complete** и manual `IValueTaskSource`, не на always-async.

---

### 5. [Understanding]
**Answer:** `WhenAll` нужны **stable `Task`**, которые можно await'ить без нарушения single-use. `.AsTask()` материализует безопасный `Task` (возможна аллокация).

**Why:** Нельзя передать один `ValueTask` в несколько awaiter'ов.

---

### 6. [Application]
**Answer:**
```csharp
Task<int> t1 = ReadChunkAsync().AsTask();
Task<int> t2 = ReadChunkAsync().AsTask();
await Task.WhenAll(t1, t2);
```
Два **отдельных** вызова `ReadChunkAsync()`, каждый `ValueTask` → один `.AsTask()` → один await path.

**Common mistake:** Один `ValueTask`, два await или два `AsTask()` на **одной** instance без понимания single-use.

---

### 7. [Application]
**Answer:** **`Task` / `Task<T>`** — безопасен для multiple awaits, composition, понятен потребителям.

**Why:** Public API не должен навязывать single-await contract без крайней необходимости.

---

### 8. [Analysis]
**Answer:** **`ValueTask` нельзя хранить** для отложенного await — к моменту второго использования pooled source может быть invalid / reused.

**Fix:** Хранить `Task` (`vt.AsTask()` сразу) или await inline при получении.

---

### 9. [Analysis]
**Answer:** Undefined behavior **не обязан** проявляться в single-threaded тестах; под concurrency и pooling reuse баг **проявится недетерминированно**.

**Why:** Тест ≠ production load; контракт BCL явный.

---

### 10. [Understanding]
**Answer:** (1) синхронный результат `T`, (2) wrapped `Task`/`Task<T>`, (3) `IValueTaskSource`/`IValueTaskSource<T>` (часто pooled).

---

### 11. [Recall]
**Answer:** Реализация **pooled async completion** без аллокации `Task` на каждый вызов; `Reset()` готовит source к reuse.

---

### 12. [Application]
**Answer:** **Нет** — даже в том же методе **два await на одной instance** нарушают контракт для pooled BCL implementation.

**Why:** `ReadAsync` может использовать pool; второй await undefined.

---

### 13. [Analysis]
**Answer:** Когда значение **уже есть** synchronously (cache, validation fail fast, in-memory) и async path обёрнут отдельно в `new ValueTask<T>(FetchAsync(...))`.

---

### 14. [Understanding]
**Answer:** Плюс к deadlock/starvation — **single-use violation**: `.Result` блокирует и «потребляет» completion; повторное использование `ValueTask` после этого ещё опаснее.

---

### 15. [Application]
**Answer:** **Не нужен** — один вызов, один немедленный `await` inline; single-use соблюдён.

**Why:** `.AsTask()` нужен при storage, `WhenAll`, или multiple awaits.

---

## Model answers for practical sections

Pooled flow: `Rent source` → `return new ValueTask(_source, version)` → caller `await` → `GetResult` → `source.Reset(); pool.Return(source)`. **Право на второй await теряется сразу после первого `GetResult`/completion** — source может обслужить другой `ReadAsync`.
