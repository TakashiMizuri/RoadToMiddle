# async void — Exception Handling Nightmare

> Roadmap: `1.4.13` · Node: `1.4` — Async/await · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Explain why **`async void`** exists and why it is **restricted to event handlers** (UI and similar).
- Describe how exceptions propagate from **`async void`** vs **`async Task`** — **`SynchronizationContext`** and **`AppDomain.UnhandledException`** / crash.
- Connect **`async void`** to the same **state machine** codegen as `async Task` (`1.4.6`–`1.4.7`) but different completion channel.
- Avoid **`async void`** in business logic, unit tests, and ASP.NET (except rare patterns).
- Wrap event handlers with **try/catch** and logging because callers cannot **`await`** or observe faults.

---

## Why This Matters

Event handlers in WinForms/WPF must match **`void Click(...)`** signature — hence **`async void`**. Exceptions thrown after the first `await` in `async void` **do not** return to the caller as a faulted **`Task`**; they rise through the **`SynchronizationContext`** as **unhandled** on UI apps, crashing or hitting **`Application.ThreadException`**. In ASP.NET Core, **`async void`** action methods are especially dangerous — request pipeline cannot observe completion or exceptions reliably.

Middle developers refactor **`async void`** helpers to **`async Task`** with safe fire-and-forget only via explicit patterns (`1.4.30`).

## Core Concepts

### async Task vs async void

**`async Task`** (or **`Task<T>`**): compiler generates state machine; exceptions after await are stored on the **`Task`**; awaiter observes them. Caller can **`try/catch`** around **`await`**.

**`async void`**: same state machine machinery (`IAsyncStateMachine`, `MoveNext` — `1.4.7`), but **no Task object** to hold completion or exception. When an exception escapes the method, runtime posts it to **`SynchronizationContext`** (if captured) or raises on thread pool as **unhandled** — behavior depends on host.

```csharp
// OK pattern for events ONLY
private async void Button_Click(object sender, EventArgs e)
{
    try
    {
        await LoadDataAsync();
    }
    catch (Exception ex)
    {
        _logger.LogError(ex, "Click failed");
        ShowError(ex.Message);
    }
}

// NEVER in services
public async void ProcessOrder(int id) { ... } // untestable, unobservable
```

### Exception Propagation Path

In **`async Task`**, faulted state machine calls **`SetException`** on **`TaskCompletionSource`** backing the task.

In **`async void`**, there is no task — exception propagates to **`AsyncVoidMethodBuilder`**, which typically notifies **`SynchronizationContext`** via **`OperationStarted`/`OperationCompleted`** tracking and throws to **`Application`** unhandled exception pipeline on UI.

**Unit tests** cannot `await async void` — test runners may not fail on exceptions from void async methods.

### Why Not async Task for Events?

C# event signatures require **`void`**. **`async Task`** handler would return before work finishes — compiler allows it but callers ignore returned Task — **fire-and-forget** with better exception surface if you assign and observe (rare in generated event wire-up).

Preferred: keep **`async void`** for UI events with **top-level try/catch**, or extract logic to **`async Task`** called from void wrapper.

### ASP.NET Core

Controller actions should return **`Task<IActionResult>`**, not **`async void`**. **`async void`** action — exception may **terminate process** or become unhandled; middleware **`try/catch`** may not run as expected for faults after await.

## Under the Hood

Roslyn uses **`AsyncVoidMethodBuilder`** instead of **`AsyncTaskMethodBuilder`**. Both drive **`MoveNext`** on the state machine struct (`1.4.7`).

**`AsyncVoidMethodBuilder`** maintains **`SynchronizationContext`** and **`ExecutionContext`** for **`OperationCompleted`** semantics so UI knows when async operation started from event finishes.

No **`Task`** means no **`await voidMethod()`** — compiler forbids awaiting void.

Testing frameworks (xUnit) warn: **`async void`** tests are broken by design.

---

## Common Mistakes & Anti-patterns

**`async void`** service methods “because fire-and-forget.” Use **`_ = ProcessAsync()`** with logged **`Task`** or **`BackgroundService`**.

**Empty catch in event handler** — swallows production failures silently.

**Assuming `async void` exceptions reach caller** — they do not.

**Using `async void` in libraries** — consumers cannot compose.

---

## Production & Real-World Notes

Global handlers: **`AppDomain.UnhandledException`**, **`TaskScheduler.UnobservedTaskException`** (for Task, not void). UI: **`DispatcherUnhandledException`**.

Refactor pattern:

```csharp
private async void OnSave(object? s, EventArgs e) => await RunSafe(SaveAsync);

private async Task RunSafe(Func<Task> action)
{
    try { await action(); }
    catch (Exception ex) { ... }
}
```

## Key Takeaways

- **`async void`** — only **UI/event handlers** with try/catch.
- Same **state machine**, no **Task** to observe faults.
- Exceptions → **unhandled** / SyncContext path, not caller.
- **`async Task`** everywhere else — testable, awaitable.
- ASP.NET: **never `async void`** actions.
- Fire-and-forget: explicit **`Task`** + logging (`1.4.30`).
