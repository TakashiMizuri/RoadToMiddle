# Records: Positional Syntax, with-Expressions, and Equality

> Roadmap: `1.2.13` · Node: `1.2` — C# OOP · Depth: практика

## Learning Objectives

After this lesson you will be able to:

- Declare **positional records** and explain what the compiler synthesizes.
- Use **`with`-expressions** for non-destructive mutation (immutable updates).
- Predict **value equality** behavior for records vs classes.
- Choose **record class** vs **record struct** vs plain **class** for DTOs and domain values.
- Apply **init-only** properties and immutability patterns in API models.
- Avoid pitfalls: mutable reference types inside records, inheritance with records, and large struct copies.

---

## Why This Matters

ASP.NET Core APIs move data constantly: request DTOs, response models, MediatR commands, EF projections. Classes with boilerplate constructors, `Equals`, `GetHashCode`, and `==` overloads waste time and invite bugs — two instances with same values compare false because reference identity differs.

**Records** (C# 9+) are **reference types** (by default) optimized for **immutable data with value semantics**. One line:

```csharp
public record CreateProductCommand(string Name, decimal Price, int CategoryId);
```

gives you constructor, read-only properties, value equality, `ToString`, and `with` copies. You will see records in minimal APIs, CQRS handlers, and `System.Text.Json` deserialization. Middle developers must know when a record is perfect, when a mutable class is still right, and when `record struct` from C# 10 fits.

## Core Concepts

### What Is a Record?

A **record** is a reference type (unless `record struct`) whose primary purpose is **carrying data** with **structural equality**: two records are equal if their **data members** have equal values, not if they are the same object reference.

```csharp
public record Point(int X, int Y);

var a = new Point(1, 2);
var b = new Point(1, 2);
Console.WriteLine(a == b);           // True — value equality
Console.WriteLine(ReferenceEquals(a, b)); // False — different objects
```

For a normal **class** without overrides, `a == b` would be **false** (reference equality). Records synthesize `Equals`, `GetHashCode`, and `==`/`!=` to compare each property in order.

### Positional Records

**Positional syntax** declares primary constructor parameters that become **init-only public properties** (same name, PascalCase):

```csharp
public record CustomerDto(int Id, string Name, string Email);
```

Compiler generates roughly:

- A constructor `(int Id, string Name, string Email)`
- Properties `public int Id { get; init; }` etc.
- `Equals`/`GetHashCode` using all positional members
- `PrintMembers` for `ToString()` → `CustomerDto { Id = 1, Name = Ann, Email = ... }`
- `Deconstruct` for `(int id, string name, string email) = customer;`

You can add more members in the body:

```csharp
public record OrderLine(int ProductId, int Quantity)
{
    public decimal UnitPrice { get; init; }
    public decimal LineTotal => UnitPrice * Quantity;
}
```

`UnitPrice` participates in equality if it is a property included in equality contract — compiler includes **all instance properties** unless you customize.

### with-Expressions: Non-Destructive Copy

Records favor **immutability**: instead of `dto.Name = "x"`, create a **copy with changes**:

```csharp
var original = new CustomerDto(1, "Ann", "ann@x.com");
var updated = original with { Email = "ann@new.com" };
```

`with` clones the record (shallow copy) and applies property overrides. `original` unchanged. Under the hood the compiler uses `<Clone>$()` and object initializer syntax.

**Shallow copy caveat:** if a property is a `List<T>` or array, `with` copies the **reference** — both records share the same list unless you replace the whole property:

```csharp
var r1 = new Container(new List<int> { 1 });
var r2 = r1 with { }; // same list reference
r2.Items.Add(2);       // mutates shared list — r1 sees change too
```

For immutable records, prefer `IReadOnlyList<T>` or immutable collections.

### record class vs class with manual equality

| Feature | `record` | Regular `class` |
|---------|----------|-----------------|
| Value equality | Synthesized | Manual override |
| `with` | Yes | No (unless manual) |
| Primary constructor | Positional syntax | C# 12+ primary constructors differ semantically |
| Inheritance | `record Derived : Base` with rules | Full OOP |
| Mutability | Typically init-only | Any |

Use **records for DTOs, commands, events, value-like aggregates**. Use **classes** for entities with identity and behavior that mutate state (`DbContext` tracked entities often stay classes).

### record struct (C# 10+)

```csharp
public readonly record struct Pixel(byte R, byte G, byte B);
```

**Value type** with record equality and `with`. Lives on stack when not boxed; no heap allocation for local use. Good for small immutable bundles. **`readonly record struct`** prevents mutating struct state after construction.

Trade-off from `1.2.14`: copying large structs is expensive — keep record structs small.

### Inheritance with Records

Records can inherit from records; **sealed** is common for leaf DTOs.

```csharp
public abstract record Shape(string Name);
public sealed record Circle(string Name, double Radius) : Shape(Name);
```

Equality includes base properties. **Do not** mix arbitrary class hierarchies with records lightly — `record class Child : SomeRegularClass` is allowed but equality semantics get subtle.

### Init and Required Members

Records pair naturally with **`init`** accessors (`1.2.2`):

```csharp
public record Product(string Name)
{
    public DateTime CreatedAt { get; init; } = DateTime.UtcNow;
}
```

**`required`** (C# 11) on properties ensures deserialization/object creation sets them:

```csharp
public record ApiRequest
{
    public required string CorrelationId { get; init; }
}
```

## Under the Hood

### Compiler Synthesis

Roslyn emits a **synthesized private `Clone` method** and overrides `Equals(object?)`, `bool Equals(T?)`, `GetHashCode`, and operators. Equality compares each included field using `EqualityComparer<T>.Default`.

For `record struct`, `Equals` avoids boxing when typed as generic `T`.

### Equality and Collections

Records work well as **`Dictionary` keys** and **`HashSet`** elements because stable value `GetHashCode` — unlike mutable classes where hash changes after mutation break collections.

### JSON Deserialization

`System.Text.Json` deserializes positional records via constructor parameters matching JSON property names (case-insensitive by default). Missing required members fail validation.

```json
{ "name": "Widget", "price": 9.99, "categoryId": 3 }
```

maps to `CreateProductCommand("Widget", 9.99m, 3)` with property name / parameter matching rules.

## Syntax / API

```csharp
// Positional record class (reference type)
public record Person(string First, string Last);

// Mutable-style init properties in body
public record Address(string City) { public string? Zip { get; init; } }

// Non-destructive update
var p2 = p1 with { Last = "Smith" };

// Deconstruction
var (first, last) = p2;

// record struct
public readonly record struct Vector2(double X, double Y);

// Equality
bool same = p1 == p2;
```

**Not synthesized for you:** deep clone of nested mutable graphs — application responsibility.

## Examples

### Example 1: API Command

```csharp
public record UpdateStockCommand(int ProductId, int Delta, string Reason);

// Handler
public class UpdateStockHandler
{
    public Task Handle(UpdateStockCommand cmd, CancellationToken ct)
    {
        // cmd is immutable — safe to pass without defensive copy
        return Task.CompletedTask;
    }
}
```

### Example 2: Immutable domain update

```csharp
public record Cart(IReadOnlyList<CartLine> Lines)
{
    public Cart AddLine(CartLine line) =>
        this with { Lines = Lines.Append(line).ToList() };
}
```

Functional style: each change returns new `Cart` instead of mutating list in place.

### Example 3: with and nested mutation trap

```csharp
public record Settings(Dictionary<string, string> Values);

var s1 = new Settings(new Dictionary<string, string> { ["k"] = "v" });
var s2 = s1 with { };
s2.Values["k"] = "changed";
Console.WriteLine(s1.Values["k"]); // "changed" — shared reference
```

**Fix:** `ImmutableDictionary<string, string>` or copy dictionary in constructor.

## Common Mistakes & Anti-patterns

**Mutable collections inside immutable record** — breaks immutability illusion.

**Using records for EF entities** with navigation properties and change tracking — EF expects mutable classes; records confuse update semantics.

**Huge positional records** (20+ parameters) — readability suffers; nest value objects or use builder.

**Assuming deep equality** — records compare references for reference-type properties shallowly.

**record vs class for behavior-heavy types** — records excel at data; put logic in services, not 500-line record methods.

## Production & Real-World Notes

Teams standardize on **`public record`** for commands/queries in vertical slices. API versioning: adding optional init property is backward compatible; adding positional parameter to primary constructor is **breaking**.

**OpenAPI** generators map records to schemas like classes. **Validation:** use `FluentValidation` or data annotations on record properties.

**Performance:** records are reference types — heap allocated like classes; benefit is correctness and less boilerplate, not automatic stack allocation (use `record struct` for that).

## Comparison / Trade-offs

| Type | Use when |
|------|----------|
| `record class` | DTOs, commands, events, API models |
| `record struct` | Small immutable value bundles (coordinates, pixels) |
| `class` | Mutable entities, rich behavior, ORM models |
| `struct` | Performance-critical value types without synthesized equality |

## Quick Reference

| Syntax | Effect |
|--------|--------|
| `record R(int A, int B)` | Positional props, value equality, `with` |
| `x with { A = 1 }` | Copy with changed members |
| `readonly record struct` | Immutable value type record |
| `sealed record` | No further inheritance |

## Key Takeaways

- **Records** = data carriers with **synthesized value equality** and **`with` copies**.
- **Positional syntax** → constructor + init properties + `Deconstruct`.
- **`with`** is **shallow** — watch mutable reference properties.
- Prefer records for **commands/DTOs**; not ideal for **mutable EF entities**.
- **`record struct`** for small stack-friendly immutable values.
- Equality includes **all instance properties** unless customized.
