# Pull Request / Merge Request workflow

> Roadmap: `0.2.8` · Node: `0.2` — Git: ветки и collaboration · Depth: практика

## Learning Objectives

После этой лекции ты сможешь:

- Различить **Pull Request (PR)** и **Merge Request (MR)** — одна концепция, разные названия платформ.
- Описать lifecycle **feature branch → push → open PR → review → merge** end to end.
- Объяснить **target branch**, **source branch** и зачем защищают `main`.
- Выбрать стратегии merge: **merge commit**, **squash merge**, **rebase merge** — и trade-offs (`0.2.2`).
- Использовать metadata PR: title, description, linked issues, draft PRs.
- Распознать CI gates, required reviewers и **merge conflicts** в UI.

---

## Why This Matters

До `0.2.7` collaboration — в основном Git-команды на машине и `origin`. Профессиональные команды редко мержат feature work direct push в `main`. Вместо этого **публикуешь branch** (`0.2.6`), настраиваешь upstream (`0.2.7`), открываешь **Pull Request** на GitHub или **Merge Request** на GitLab/Azure DevOps. PR — **proposal**: «вот мои commits; review перед интеграцией».

Здесь сходятся **code review etiquette** (`0.2.10`), **Conventional Commits** (`0.2.9`) и знания merge/rebase (`0.2.2`–`0.2.4`). CI гоняет тесты на branch. Reviewers комментируют diffs. Maintainers выбирают, как commits попадут на `main`. Middle fullstack открывает чистые PR ежедневно — маленький scope, ясное описание, green checks, без сюрпризных conflicts.

Понимание PR объясняет, почему **force-push на feature** нормален после rebase, а на **`main`** — никогда, и зачем существуют кнопки «Update branch», когда `main` ушёл вперёд PR.

## Core Concepts

### PR vs MR — одна идея, разные ярлыки

| Платформа | Название |
|-----------|----------|
| GitHub | **Pull Request (PR)** |
| GitLab | **Merge Request (MR)** |
| Azure DevOps | **Pull Request** |

Смысл один: **запрос слить source branch в target** с review, обсуждением и automation. «Pull» — втянуть изменения *в* target; «Merge» — финальное действие.

### Стандартный lifecycle

```
1. Feature branch от свежего main
2. Локальные commits (atomic, осмысленные — 0.2.9)
3. git push -u origin feature/...
4. Открыть PR: source feature → target main
5. CI (build, test, lint)
6. Code review (0.2.10)
7. Feedback → новые commits на той же ветке
8. Merge при green + approvals
9. Удалить feature branch (local + remote)
```

PR **отслеживает живую ветку**. Новые push обновляют diff автоматически.

### Source, Target и Forks

- **Source (head):** feature branch — что хотят влить.
- **Target (base):** обычно `main` или `develop` — куда вливают.

В **fork workflow** source может быть `yourfork:feature`, target `upstream:main`. В **shared repo** обе ветки на `origin`.

**Draft PR** — work-in-progress; reviewers знают, что quality ещё не финальная.

### Что видят reviewers

- **Commits list** — narrative ветки (`0.2.9`).
- **Files changed** — diff относительно target.
- **Checks** — CI, required approvals.
- **Conflicts** — auto-merge невозможен; чинить локально (`0.2.5`) и push.

Конфликты в UI — target сдвинулся с момента branch/rebase:

```bash
git fetch origin
git rebase origin/main
git push --force-with-lease
```

### Стратегии merge на платформе

**Merge commit** — merge commit на target; сохраняет все feature commits (`0.2.2`).

**Squash merge** — все commits PR в **один** на target. Чистый log `main`; деталь per-commit на target теряется (остаётся на странице PR).

**Rebase merge** — replay commits PR на tip target без merge commit. Linear `main`; новые SHA при merge.

Кнопка триггерит server-side Git — то, что можно сделать вручную локально.

### Branch protection

На `main` часто:

- PR обязателен, direct push запрещён.
- Required CI checks.
- Минимум approvals.
- Up-to-date с target перед merge.

Failing checks блокируют merge. Broken `main` блокирует всех.

### Связь с тикетами

`Fixes #42` в description закрывает GitHub issue при merge. Traceability: requirement → code → deploy.

## Under the Hood

PR — **не** Git object, а **metadata платформы** на refs. GitHub хранит `pull/123/head`, отслеживающие tip branch. При push платформа пересчитывает mergeability test merge против current target.

CI клонирует branch (или merge result) и гоняет pipelines. Локально ничего не меняется до merge — кроме твоих push на `origin/feature`.

## Syntax / Commands / API

```bash
git fetch origin
git switch main && git pull --ff-only origin main
git switch -c feature/ticket-123
git push -u origin feature/ticket-123

gh pr create --base main --head feature/ticket-123 --title "..." --body "..."

git commit -m "fix: address review"
git push

git fetch origin && git rebase origin/main
git push --force-with-lease

git switch main && git pull origin main
git branch -d feature/ticket-123
git push origin --delete feature/ticket-123
```

## Examples

### Пример 1: ASP.NET feature PR

`feature/health-endpoint`, push `-u`, title `feat: add GET /health`, description с how to test. CI `dotnet test`. Squash merge по policy.

### Пример 2: CI блокирует merge

Approval есть, но check `build` red. Фикс → push → CI green → merge разблокирован.

### Пример 3: Конфликт после движения main

Rebase `origin/main`, resolve `0.2.5`, `push --force-with-lease`.

### Пример 4: Squash vs merge commit

Три commits на PR → squash даёт один на `main`; merge commit сохраняет тройку + merge node.

## Common Mistakes & Anti-patterns

**Огромные PR** — review paralysis; маленькие vertical slices.

**Пустое description** — медленные approvals.

**PR со stale main** — conflict debt; fetch latest main first.

**Merge при red CI** — ломает культуру качества.

**Игнор review comments** — отвечай или исправляй.

**Мёртвые ветки** после merge — clutter.

**Неверный target** — PR в `release/1.0` вместо `main`.

## Production & Real-World Notes

**Trunk-based** — короткие ветки, PR часами. **GitFlow** — PR в `develop`. **CODEOWNERS**, security reviewers на auth. Deploy часто на merge в `main`.

## Comparison / Trade-offs

| Стиль | History main | Когда |
|-------|--------------|-------|
| Merge commit | Граф веток | Audit, крупные features |
| Squash | Один commit/PR | Чистый log, tickets |
| Rebase merge | Linear | Strict linear main |

## Quick Reference

```
PR/MR = source → target (feature → main)
Flow: branch → push -u → PR → CI → review → merge → cleanup
Conflicts: fetch → rebase main → push
Draft PR = WIP
```

---

## Key Takeaways

1. PR/MR — platform workflow, не Git primitive.
2. Path: **branch → push → PR → CI → review → merge**.
3. **Target** защищённый `main`; **source** — published branch.
4. **Merge strategy** влияет на history (`0.2.2`).
5. Conflicts — локально (`0.2.5`), push обновляет PR.
6. Маленькие PR, описание, green CI.
7. Удаляй ветки после merge.
