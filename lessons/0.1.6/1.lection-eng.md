# .gitignore Patterns and .gitkeep

> Roadmap: `0.1.6` · Node: `0.1` — Git Basics · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Write correct `.gitignore` patterns using globs, `**`, negation (`!`), and directory prefixes (`/`).
- Choose between **local**, **global**, and **per-user exclude** ignore sources.
- Ignore common build artifacts (`bin/`, `obj/`, `node_modules/`) and secrets (`.env`) safely.
- Use `.gitkeep` to track empty directories without tracking their contents.
- Debug ignore rules with `git check-ignore -v`.

---

## Why This Matters

In lessons `0.1.4` and `0.1.5` you learned how Git stores immutable objects and links commits into a graph. Everything that enters that object database stays in history — often forever, even if you "delete" it in a later commit. That permanence makes one class of mistakes especially painful: committing the wrong files in the first place.

Without ignore rules, a typical full-stack repository becomes a junk drawer within weeks. `bin/Debug/` fills with hundreds of megabytes of compiled assemblies. `node_modules/` swells to hundreds of megabytes or gigabytes of dependencies that belong in `npm install`, not in Git. A `.env` file with database credentials lands on GitHub and triggers an incident, key rotation, and post-mortem. IDE-specific files like `*.user` or `.vs/` create meaningless noise in every pull request. Cross-platform teams fight `Thumbs.db` on Windows and `.DS_Store` on macOS polluting history.

Each problem has the same root cause: Git was asked to track files that should never have been versioned. `.gitignore` is the front-door filter that tells Git to treat certain paths as if they do not exist when discovering untracked files. It is one of the cheapest, highest-leverage guardrails in professional development — yet teams still leak secrets and bloat repositories because ignore rules are written once, copied incorrectly, or misunderstood.

This lesson is practical. You will learn what `.gitignore` actually affects (and what it cannot fix retroactively), where rules come from and in what order they combine, how to write patterns that work on Windows and Linux, how to use `.gitkeep` for empty directories, and how to debug a stubborn ignored file with `git check-ignore -v`. By the end you should be able to stand up a sensible `.gitignore` for an ASP.NET Core plus React project and explain to a teammate why `git rm --cached` is required after the fact.

## Core Concepts

### What `.gitignore` Does — and What It Cannot Do

The first misconception to clear is scope. `.gitignore` affects **untracked** files only. It is consulted when Git scans your working tree to answer questions like "what is new?" and "what should `git add .` pick up?" If a path matches an ignore rule, Git will not offer it as untracked in `git status` (by default) and will not add it when you run `git add .` or `git add -A`.

What `.gitignore` does **not** do is equally important. If you already committed `secrets.env` three months ago, adding `secrets.env` to `.gitignore` today does not remove it from history and does not stop Git from tracking the copy already in the index. The file remains in every clone until you explicitly untrack it with `git rm --cached` and commit that change — and the secret remains in old commits unless you rewrite history. Think of `.gitignore` as a **filter on the front door**, not a retroactive eraser. Prevention beats cure, especially for blobs that cannot be truly "un-published" once pushed.

There is one escape hatch: `git add -f <path>` **force-adds** an ignored file. That is occasionally legitimate (you really need one log file in the repo) and occasionally catastrophic (forcing a credential file). Production teams treat force-add of sensitive paths as a serious process failure.

### Where Ignore Rules Come From

Ignore rules are not only in the `.gitignore` file at your repository root. Git merges patterns from several sources, and understanding the stack prevents "it works on my machine" ignore behavior.

The committed `.gitignore` at the repository root is the **team contract**. Everyone who clones gets the same rules. It should list build outputs, dependencies, secrets, and common IDE noise for your stack. Nested `.gitignore` files in subdirectories apply to their subtree — useful in monorepos where `docs/_build/` should be ignored only under `docs/`, not project-wide.

Your personal global exclude file (configured via `core.excludesfile`) holds patterns you never want to see in any repository — often OS cruft like `Thumbs.db` or editor swap files. It is not shared with teammates unless they configure the same file path themselves.

The file `.git/info/exclude` is per-repository but **not committed** — personal scratch patterns for one clone without polluting the shared `.gitignore`. Finally, the command line can override everything: `git add -f` bypasses ignore for that invocation.

Configure a global ignore once and maintain it for years:

```bash
git config --global core.excludesfile ~/.gitignore_global
```

When multiple rules could match a path, Git walks the rule sets in a defined order; later rules can **negate** earlier ones with `!` if the path is not blocked by an ignored parent directory. The exact precedence is documented in `gitignore(5)`; the practical lesson is that negation must come **after** the rule it overrides, and you cannot un-ignore a file inside an ignored directory without ignoring the directory's contents differently — a trap we address below.

### Pattern Syntax: Globs, Anchors, and Double Stars

`.gitignore` patterns are **fnmatch-style globs** interpreted relative to the location of the `.gitignore` file (unless anchored to the repository root with a leading slash). You do not need regex mastery; you need a handful of reliable building blocks.

A pattern like `*.log` matches any file ending in `.log` in the directory of the `.gitignore` and in subdirectories. That is usually what you want for log files scattered through the project. A leading slash changes the meaning: `/build` matches only a `build` directory or file at the **repository root**, not `src/build`. This anchor is essential when you have legitimate `build` folders inside subprojects but want to ignore only the root output folder.

A trailing slash on a pattern such as `logs/` means "directories only." Without the slash, Git may match both files and directories named `logs`. For dependency folders, `node_modules/` is the idiomatic form — directory named `node_modules` at any depth when the pattern has no leading slash.

The double-star pattern `**` matches zero or more directories. `**/node_modules/` ignores every `node_modules` folder no matter how deep — critical in JavaScript monorepos with nested packages. `**/temp` matches any file or directory named `temp` anywhere. Single `*` matches within one path segment; `?` matches one character. Character classes like `[abc]` work as in shell globs.

**Negation** re-includes a path with `!` at the start of the pattern. A common production pattern ignores all env files then allows the safe template:

```gitignore
.env
.env.*
!.env.example
```

The `!.env.example` line must appear **after** `.env.*` or it will not win. The template is committed so new developers copy it locally; real secrets stay untracked.

The negation trap catches everyone eventually. If you ignore an entire directory with `logs/`, you **cannot** un-ignore `logs/keep.log` with `!logs/keep.log` — Git never looks inside an ignored directory. The workaround is to ignore contents, not the directory itself:

```gitignore
logs/*
!logs/.gitkeep
!logs/keep.log
```

Here `logs/*` ignores everything inside `logs/`, but the directory `logs/` itself is not ignored, so negation can re-include specific files.

Comments start with `#`. To ignore a file whose name starts with `#`, escape it: `\#special`.

### Debugging with `git check-ignore`

When `git status` does not show a file you expect, or shows one you thought was ignored, stop guessing. `git check-ignore -v <path>` prints which pattern matched and from which file:

```bash
git check-ignore -v node_modules/lodash/index.js
# .gitignore:3:node_modules/    node_modules/lodash/index.js
```

The verbose line tells you the file, the line number, and the pattern. That is the fastest way to resolve "why is Git still tracking this?" disputes in code review.

## Syntax / Commands / API

Beyond patterns, three commands appear constantly in real workflows.

`git check-ignore -v <path>` diagnoses matches. `git rm --cached <file>` removes a tracked file from the index while keeping it on disk — the fix when something was committed before ignore rules existed. `git add -f <path>` force-adds despite ignore; use rarely and never for secrets.

Global configuration:

```bash
git config --global core.excludesfile ~/.gitignore_global
```

## Examples

### A Full-Stack .NET + React `.gitignore`

The following file is representative of what a professional full-stack repository needs. Read it as a narrative of intent, not a magic list to paste blindly.

```gitignore
# ── Build output (.NET) ──
[Bb]in/
[Oo]bj/
[Dd]ebug/
[Rr]elease/
*.dll
*.pdb
*.user
*.suo

# ── Node / frontend ──
node_modules/
dist/
build/
.npm/
*.tsbuildinfo

# ── Secrets & local config ──
.env
.env.*
!.env.example

# ── IDE / OS ──
.vs/
.idea/
*.swp
.DS_Store
Thumbs.db

# ── Test / coverage ──
coverage/
TestResults/
*.coverage
```

The `[Bb]in/` pattern illustrates case-insensitivity for Windows `Bin` vs Linux `bin` without two separate lines. `node_modules/` without a leading slash matches at any depth — necessary when frontend lives in `client/` or `app/`. The `!.env.example` negation is the industry-standard split between committable template and non-committable secrets. `dist/` and `build/` keep compiled frontend assets out of the object database you studied in `0.1.4` — once a ten-megabyte bundle enters history, it haunts every clone forever.

Start from community templates ([github/gitignore](https://github.com/github/gitignore)) — `VisualStudio.gitignore` plus `Node.gitignore` — then customize for your solution layout.

### `.gitkeep` for Empty Directories

Git tracks **files**, not directories. An empty folder is invisible to `git add` and will not exist for teammates after clone. Teams sometimes need the folder structure anyway — for example `logs/archive/` where runtime will write files, or `uploads/` created before the upload feature ships.

The convention is to place a placeholder file. `.gitkeep` is not a Git feature; it is a community name for an empty or nearly empty file whose sole job is to force Git to create the directory:

```bash
mkdir -p logs/archive
touch logs/.gitkeep
git add logs/.gitkeep
git commit -m "Add logs directory structure"
```

Alternatives include a one-line `README.md` explaining the directory's purpose — often clearer for future maintainers. When real files arrive, you may delete `.gitkeep` or leave it; either is fine. Do not use `.gitkeep` for directories your application creates at runtime if deploy scripts should create them instead — version control is for source and configuration, not for substituting proper deployment setup.

### Removing Already-Tracked Files

The correction path when someone committed `.env` before ignore rules existed:

```bash
git rm --cached .env
echo ".env" >> .gitignore
git commit -m "Stop tracking .env"
```

`--cached` removes only from the index; the developer keeps the local file. Rotate every secret that was ever pushed. If the repository is public or widely cloned, consider `git filter-repo` or platform secret scanning — `.gitignore` alone does not scrub history.

## Under the Hood

Git's ignore engine runs during **untracked file discovery** — when `git status` walks the working tree and when `git add` considers what to stage. For each candidate path, Git collects applicable rules from global excludes, `.git/info/exclude`, and `.gitignore` files from the path upward through parent directories. Patterns are matched with fnmatch semantics against paths relative to the rule file's location.

The first matching rule wins unless a later negation rule re-includes the path. Ignored directories short-circuit examination of their contents, which is why the `logs/` vs `logs/*` distinction matters. Critically, this entire pipeline applies only to files **not already in the index**. Tracked files are tracked regardless of `.gitignore` until you `git rm --cached` them.

That implementation detail explains half the support tickets in internal Git channels. It also explains why CI pipelines sometimes run `git status --porcelain` after a build: if artifacts appear as untracked, ignore rules are missing; if they appear as modified tracked files, someone committed them and ignore cannot help without untracking.

## Common Mistakes & Anti-patterns

Committing `.env` and then adding it to `.gitignore` leaves secrets in history forever — fix with untrack, rotation, and possibly history rewriting. Using `git add -f` on credential files bypasses your only guardrail. Ignoring `node_modules/` but accidentally ignoring `package-lock.json` with an overly broad `*.json` pattern breaks reproducible installs — lockfiles must be committed. Relying on `.gitkeep` for directories the app creates at runtime confuses version control with deployment configuration.

Another subtle mistake is copying a `.gitignore` from an old project without checking paths: a root-only `/build` pattern will not ignore `client/dist/` unless you add `dist/` or `**/dist/` appropriately.

## Production & Real-World Notes

Treat the root `.gitignore` as code: review it in pull requests when stack or tooling changes. Pair `!.env.example` with documentation telling developers to `cp .env.example .env` locally. Maintain a personal global ignore for OS and editor noise so you never debate `Thumbs.db` in team chat. Use secret scanning tools (gitleaks, trufflehog) in CI — `.gitignore` is prevention; scanners are detection. In CI, fail builds that produce unexpected `git status` output after compile steps to catch missing ignore rules early.

---

## Comparison / Trade-offs

Broad patterns like `*.log` are easy to maintain but may hide a log file you someday need in the repo — narrow paths like `/logs/*.log` are more precise but require more thought. Global ignores clean all repositories on your machine but are invisible to teammates. `.git/info/exclude` is useful for local experiments without committing personal paths. `.gitkeep` preserves directory shape in Git but is slightly hacky; a short README in the folder often communicates intent better to the next developer.

## Key Takeaways

- `.gitignore` filters **untracked** files only — already-committed files need `git rm --cached`.
- Patterns use globs: `*`, `?`, `**`, `/` prefix for root, `!` for negation.
- Three personal layers: global excludesfile, `.git/info/exclude`, plus shared `.gitignore`.
- Always ignore `bin/`, `obj/`, `node_modules/`, and `.env`; commit `.env.example` instead.
- `.gitkeep` is a convention to track empty directories.
- `git check-ignore -v` tells you exactly which rule matched.
