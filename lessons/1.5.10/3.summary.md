# IDisposable vs Finalizer — Dispose Pattern — краткая выжимка

> `1.5.10` · ~10 мин · Depth: **глубоко**

## Главная идея

**GC** освобождает managed memory; **OS handles** — через **`IDisposable.Dispose`** (deterministic). **Finalizer** — last resort: позже, на finalizer thread, удлиняет lifetime. Полный **dispose pattern** связывает оба пути через **`Dispose(bool)`** и **`GC.SuppressFinalize(this)`**.

## Ключевые пункты

- **`disposing == true`**: managed + unmanaged; **`false`**: только unmanaged (из `~Type()`).
- Public **`Dispose()`** не virtual; **`protected virtual Dispose(bool)`** для inheritance.
- Finalizer только при **прямом unmanaged**; лучше **`SafeHandle`**.
- **`SuppressFinalize`** после успешного `Dispose` — must.
- Consumer: **`using` / `await using`** — см. **`1.3.16`**.
- Finalization queue → object **re-rooted** → extra GC cycles.

## Самые важные примеры

```csharp
public void Dispose()
{
    Dispose(disposing: true);
    GC.SuppressFinalize(this);
}
```

```csharp
~ResourceHolder() => Dispose(disposing: false);
```

## «Под капотом» в одном абзаце

Object с finalizer при unreachable попадает в **finalization queue**, снова reachable для runtime, собирается только после `~Type()` (если нет resurrection). `SuppressFinalize` убирает необходимость этого пути.

## Частые ошибки

- Finalizer на managed-only class.
- Managed cleanup в `Dispose(false)`.
- Забытый `SuppressFinalize`.
- Полагаться на finalizer under load.

## Мнемоника

> **Dispose is a promise; finalize is a backup you should never need.**
