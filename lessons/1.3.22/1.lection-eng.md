# LINQ SelectMany — Flatten Nested Collections

> Roadmap: `1.3.22` · Node: `1.3` — C#: Advanced Topics · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Explain **`SelectMany`** as **map + flatten** (monadic bind over sequences).
- Use one-sequence and **two-sequence overloads** of **`SelectMany`**.
- Complete **left outer join** patterns with **`GroupJoin` + SelectMany + DefaultIfEmpty`**.
- Contrast **`Select` + nested loop** vs **`SelectMany`** for readability and deferred execution.
- Recognize **`SelectMany`** in EF Core SQL (**`CROSS APPLY`**, subqueries) and cartesian risks.
- Build **pipelines** over **`IEnumerable<IEnumerable<T>>`** and optional collections.

---

## Why This Matters

Real models are nested: **`Order`** has **`Lines`**, **`BlogPost`** has **`Tags`**, API returns **`Page<Batch<Item>>`**. You need a flat stream of line items or tags for filtering, summing, or export. **`SelectMany`** projects each outer element to an **inner sequence** and concatenates — one operator replacing nested **`foreach`** and **`AddRange`**.

It is also the second half of the **LINQ left join** idiom from **`GroupJoin`**. Without **`SelectMany`**, you hold grouped hierarchies; with it, you flatten while preserving outer context in the **`resultSelector`**.

Confusing **`Select`** (returns nested **`IEnumerable`**) with **`SelectMany`** (returns flat stream) produces **`IEnumerable<IEnumerable<T>>`** bugs and double **`foreach`** in views. Middle developers see **`SelectMany`** in Rx, functional **`Bind`**, and SQL **`UNNEST`**-like patterns — same idea, different surface.

## Core Concepts

### Map vs Flatten

**`Select`**: one outer → one inner **value** (which may itself be a sequence).

```csharp
var nested = orders.Select(o => o.Lines); // IEnumerable<IEnumerable<Line>>
```

**`SelectMany`**: one outer → **many** output elements, concatenated.

```csharp
var flat = orders.SelectMany(o => o.Lines); // IEnumerable<Line>
```

Equivalent mental model:

```csharp
foreach (var o in orders)
    foreach (var line in o.Lines)
        yield return line;
```

**Deferred** until enumeration — inner sequences not walked until **`MoveNext`** reaches that outer element.

### One-Sequence Overload

```csharp
source.SelectMany(x => x.Children)
```

For each **`x`**, **`collectionSelector(x)`** returns **`IEnumerable<TChild>`**; all child sequences concatenate in order — outer order preserved, inner order per collection preserved.

### Two-Sequence Overload (with resultSelector)

```csharp
source.SelectMany(
    collectionSelector,
    (outer, inner) => result);
```

Flattens like one-sequence overload but **`resultSelector`** combines outer and each inner element — avoids anonymous **`Select`** step after flatten.

```csharp
orders.SelectMany(
    o => o.Lines,
    (o, line) => new { o.Id, line.ProductId, line.Qty });
```

This is the flatten step in **left join**:

```csharp
customers
    .GroupJoin(orders, c => c.Id, o => o.CustomerId, (c, g) => new { c, g })
    .SelectMany(x => x.g.DefaultIfEmpty(), (x, o) => new { x.c, o });
```

**`DefaultIfEmpty()`** yields one **`null`** when group empty — **`SelectMany`** still emits one row per outer customer.

### SelectMany vs Select + SelectMany

```csharp
orders.Select(o => o.Lines).SelectMany(x => x);
```

Works but noisy. Prefer single **`SelectMany(o => o.Lines)`**.

### Empty and Null Inner Sequences

If **`collectionSelector`** returns **empty** sequence, outer contributes **nothing** — not an error. If it returns **`null`**, **`SelectMany`** throws **`ArgumentNullException`** — guard with **`o.Lines ?? Enumerable.Empty<Line>()`** or **`?? []`**.

## Under the Hood

**`Enumerable.SelectMany`** iterates outer; for each outer, iterates inner enumerator, yields each element — **O(total output count)** time, **O(1)** extra space aside from enumerators (not counting output storage).

If inner collections are **`ICollection`** with known count, implementation may optimize pre-sizing when materialized — still lazy for deferred chain.

**Cartesian feel:** **`SelectMany`** is **not** cartesian product of two independent lists unless you write:

```csharp
as.SelectMany(a => bs, (a, b) => ...); // pairs each a with EVERY b — O(n×m)
```

That overload pairs **each outer with all inners from selector returning same sequence** — often bug when developer meant **`Join`**.

**EF Core:** **`SelectMany`** over navigation collections translates to **`CROSS APPLY`** or join expansion. Multiple **`SelectMany`** on unrelated collections can explode row counts — same as SQL cross join semantics when mistyped.

**`IQueryable.SelectMany`** builds expression tree; provider must translate **`collectionSelector`** — method calls may force client evaluation.

## Syntax / API

```csharp
// Flatten
source.SelectMany(x => x.Items);

// Flatten + project
source.SelectMany(x => x.Items, (x, item) => new { x.Id, item });

// Query syntax — multiple from
from o in orders
from line in o.Lines
select new { o.Id, line.Qty };
```

Multiple **`from`** without **`join`** desugar to **`SelectMany`**.

## Examples

### All line items across orders

```csharp
var allLines = orders.SelectMany(o => o.Lines);
var totalQty = allLines.Sum(l => l.Qty);
```

### Tags on posts with post id

```csharp
var tagRows = posts.SelectMany(
    p => p.Tags,
    (p, tag) => new TagRow(p.Id, tag));
```

### Paginated API flatten

```csharp
IEnumerable<Item> AllItems(IEnumerable<Page<Item>> pages) =>
    pages.SelectMany(p => p.Items);
```

Fetch pages lazily if **`pages`** itself is **`yield`**-based — items stream without loading all pages into one list first.

## Common Mistakes & Anti-patterns

**Using `Select` when flattening needed** — **`IEnumerable<IEnumerable<T>>`** in Razor/API responses.

**Null inner sequence** — NRE inside **`SelectMany`**.

**Accidental cross product** — **`SelectMany(a => bs, ...)`** when **`Join`** on key was intended.

**Multiple SelectMany on large EF graphs** — SQL row explosion; use **`Include`**, split query, or projection.

**Eager `.ToList()` inside collectionSelector** per outer — allocates full inner lists repeatedly; often necessary for EF client eval but watch performance.

## Production & Real-World Notes

Report queries: **`SelectMany`** to lines then **`Where`/`Sum`** — keep **`IQueryable`** through flatten when navigation is translatable.

For optional 1:0..1 navigation, **`SelectMany`** with empty default:

```csharp
orders.SelectMany(o => o.Coupon != null ? new[] { o.Coupon } : Enumerable.Empty<Coupon>());
```

Sometimes **`Left Join`** clearer than **`SelectMany`** hack.

Testing: assert flat count equals sum of inner counts; test empty outer, empty inner, null guard.

## Comparison / Trade-offs

| Task | Operator |
|---|---|
| 1:1 projection | **`Select`** |
| 1:many flatten | **`SelectMany`** |
| Key-based pairing | **`Join`** |
| Left join flat | **`GroupJoin` + SelectMany + DefaultIfEmpty** |
| Cartesian all pairs | **`SelectMany`** with constant inner (rare, explicit) |

**`SelectMany`** = **`Bind`** in functional languages — composes monadic sequences.

```csharp
SelectMany(collectionSelector)
SelectMany(collectionSelector, resultSelector)

```

---

## Key Takeaways

- **`SelectMany`** = project to sequence + concatenate.
- Two overloads: flatten only vs flatten + combine with outer.
- Query **`from ... from`** → **`SelectMany`**.
- Guard **null** inner sequences.
- Wrong overload → accidental O(n×m) cross product.
- Completes **left join** pattern after **`GroupJoin`**.
