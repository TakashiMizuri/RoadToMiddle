# IDisposable, using, await using, finalizers

> Roadmap: `1.3.16` · Узел: `1.3` — C#: продвинутые темы · Глубина: **глубоко**

## Learning Objectives

После этого урока ты сможешь:

- Объяснить **unmanaged vs managed** resources и почему одного GC недостаточно.
- Реализовать **`IDisposable.Dispose`** корректно, включая **`GC.SuppressFinalize`**.
- Использовать **`using`** и **`await using`** для deterministic cleanup.
- Реализовать **`IAsyncDisposable`** для async release (streams, DB).
- Описать **finalizers**, **finalization queue** и почему их избегают.
- Применять **dispose pattern** (base + derived) для классов с handles.

---

## Why This Matters

File handles, sockets, DB connections, `MemoryPool`, HTTP streams держат ресурсы **вне** простой модели GC «освободи когда unreachable». Ждать GC — **handle leaks**, **connection pool exhaustion**, **locked files**.

`IDisposable` — контракт **deterministic cleanup**: освободи сейчас. `using` гарантирует `Dispose` при success **и** exception. В ASP.NET Core `DbContext`, streams, locks опираются на это.

Ошибки disposal — production incidents: `SqlConnection` leaks, temp files не удаляются, **double dispose**, **never dispose** → native OOM. Middle dev знает sync vs async disposal и когда finalizers ещё уместны (редко).

## Core Concepts

### Managed vs Unmanaged

**Managed** — память под GC (`new List<int>()`). **Unmanaged** — OS handles: descriptors, sockets, GDI, raw `IntPtr` из P/Invoke.

Managed wrappers (`FileStream`, `SqlConnection`) держат unmanaged handles — реализуют `IDisposable` для **prompt** release.

GC освобождает **managed object memory**; **`Dispose` автоматически не вызывает** (кроме finalizer path).

### IDisposable

```csharp
public interface IDisposable
{
    void Dispose();
}
```

`Dispose` должен:

- Освободить unmanaged resources.
- Опционально — expensive managed (`HttpClient` по policy).
- Быть **idempotent** — guard `_disposed`.

```csharp
public sealed class LogWriter : IDisposable
{
    private readonly Stream _stream;
    private bool _disposed;

    public void Dispose()
    {
        if (_disposed) return;
        _stream.Dispose();
        _disposed = true;
        GC.SuppressFinalize(this);
    }
}
```

### using Statement

```csharp
using (var writer = new LogWriter())
{
    writer.Write("...");
} // Dispose даже при exception
```

Lowering — try/finally. C# 8 **using declaration**:

```csharp
using var writer = new LogWriter();
// Dispose в конце block (reverse order для нескольких)
```

### IAsyncDisposable и await using

```csharp
public interface IAsyncDisposable
{
    ValueTask DisposeAsync();
}

await using var conn = new AsyncDbConnection();
await conn.ExecuteAsync(...);
```

`await using` — finally с `DisposeAsync`. Тип может implement **оба** интерфейса. Prefer **`await using`** когда cleanup делает I/O.

### Finalizers

```csharp
~LogWriter() { /* last resort */ }
```

Object с finalizer: unreachable → **finalization queue** → `Finalize` на background thread → потом collect.

Проблемы: **non-deterministic**, **дорого**, **нет порядка** между finalizers.

**Правило:** избегать, кроме raw unmanaged pointer без `Dispose`. С finalizer — full dispose pattern + `SuppressFinalize` в `Dispose`.

### Dispose Pattern

```csharp
public void Dispose()
{
    Dispose(disposing: true);
    GC.SuppressFinalize(this);
}

protected virtual void Dispose(bool disposing)
{
    if (_disposed) return;
    if (disposing) { /* managed IDisposable fields */ }
    /* unmanaged always */
    _disposed = true;
}
```

Modern: **`SafeHandle`** для unmanaged — меньше custom finalizers.

### Что не dispose

- Shared **`HttpClient`** singleton — не dispose per request pattern.
- **`Task`**, **`string`**, **`int`** — не disposable.
- **`DbContext`** — always dispose (scoped в ASP.NET).

## Under the Hood

`using` → try/finally + `Dispose`. Finalizer регистрирует object для **extra GC cycle** — **extends lifetime**. `SuppressFinalize` — skip finalizer при correct dispose.

`Dispose` idempotent; thread-safety — по документации типа. Analyzers: CA1816 и др.

## Examples

```csharp
using var stream = File.OpenRead(path);
using var reader = new StreamReader(stream);
return await reader.ReadToEndAsync();
```

```csharp
public sealed class TempFile : IDisposable
{
    public string Path { get; } = System.IO.Path.GetTempFileName();
    public void Dispose() => File.Delete(Path);
}
```

## Common Mistakes & Anti-patterns

- No `using` on DbContext/Stream.
- Dispose shared singleton.
- Throw from `Dispose` — masks original exception.
- Block in `Dispose()` вместо `DisposeAsync`.
- Empty `Dispose` «for convention» на pure managed class.

---

## Production & Real-World Notes

ASP.NET DI **scoped** disposes end of request. Singleton ≠ scoped disposable без factory.

Metrics: connection pool spike без dispose — `pg_stat_activity`, SQL DMVs.

## Key Takeaways

- GC ≠ release OS handles — **`Dispose`** promptly.
- **`using`** / **`await using`** = try/finally.
- **Finalizers** — slow, last resort.
- **DbContext**, streams — always dispose in scope.
