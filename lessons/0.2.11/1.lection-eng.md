# git cherry-pick and git revert for Hotfixes

> Roadmap: `0.2.11` · Node: `0.2` — Git: Branches and Collaboration · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Explain **cherry-pick** as replaying a single commit onto another branch.
- Describe a **hotfix workflow** using branches, cherry-pick, and revert.
- Use **`git cherry-pick`** with conflict resolution and `-x` / `--no-commit` options.
- Use **`git revert`** to undo a bad release commit on shared branches.
- Choose **cherry-pick vs merge vs revert** for production incidents.
- Recover from a failed cherry-pick with **`git cherry-pick --abort`**.

---

## Why This Matters

Production is on fire. A critical bug exists on `main`, but the fix was already written three days ago on a long-running feature branch — or on last week's release tag. You cannot wait for the entire feature branch to merge. You also cannot force-push `main` to rewrite history while customers depend on the current deployment. Teams need surgical tools: copy exactly the fix commit elsewhere, or undo a bad deploy without erasing audit history.

**Cherry-pick** and **revert** are those surgical tools. Cherry-pick says: "Take this one commit's changes and apply them here." Revert says: "The change in this commit was wrong — add a new commit that undoes it." Both preserve the social contract of shared Git history that you learned in `0.1.8`: on branches others have pulled, you add truth forward rather than rewriting the past.

Hotfix workflows combine branching discipline from `0.2.1`, merge/rebase trade-offs from `0.2.4`, and safe undo from `0.1.8`. Middle developers are expected to execute a hotfix branch, cherry-pick a fix onto `main` and a long-term support line, revert a broken release if rollback is faster than forward-fix, and explain to QA exactly which commits shipped.

## Core Concepts

### What Cherry-pick Does

When you run `git cherry-pick <commit>`, Git creates a **new commit** on your current branch whose **parent** is the current branch tip, but whose **tree diff** matches what the picked commit introduced. Git does not move the original commit. It **replays** the patch.

Imagine history like this:

```
feature/login:  ... ─ A ─ B ─ FIX
main:           ... ─ A ─ B
```

If `FIX` contains only the security patch you need, checking out `main` and running `git cherry-pick FIX` produces:

```
main:  ... ─ A ─ B ─ FIX'   (FIX' has new hash, same code change as FIX)
```

The original `FIX` still exists on `feature/login`. Two commits now contain the same logical change under different hashes. That duplication is intentional in hotfix scenarios — you ship the fix without merging unfinished feature work.

Cherry-pick uses the same **three-way merge** machinery as `git merge` and `git revert`. Git finds the picked commit's parent as the "base," compares base → picked commit as the patch, and applies that patch onto your current HEAD. If lines changed on your branch since the base diverged, you get conflicts — resolve them like any merge conflict from `0.2.5`.

### Cherry-pick vs Merge vs Revert

These three commands solve different problems and are often confused because all three touch history without a simple "undo last action" mental model.

**Merge** integrates an entire branch's history — all commits reachable from the branch tip. Use merge when you want the full feature branch on `main`, not one commit in isolation.

**Cherry-pick** copies **one or more selected commits** by hash. Use it when only specific commits are production-ready — classic hotfix from a feature branch, backporting a fix to `release/1.4` while `main` is already on `2.0`, or pulling a commit from a fork without merging unrelated work.

**Revert** creates a commit that **undoes** a prior commit's effect. Use it when a bad commit already reached a shared branch and you must roll back without force push — failed deployment, broken migration, accidental secret in a tagged release.

A useful decision tree: need **more** code from a branch → merge (or rebase then merge). Need **one** commit elsewhere → cherry-pick. Need to **remove** effect of a shipped commit → revert.

### Hotfix Branch Workflow

A disciplined hotfix pattern keeps `main` stable and traceable:

```
1. git checkout main && git pull
2. git checkout -b hotfix/SEC-401-null-check
3. Fix bug, commit (or cherry-pick existing fix commit)
4. Test, open PR, fast review
5. Merge to main, tag release (see 0.2.13)
6. Cherry-pick same commit to release/1.x if still supported
7. git push origin main release/1.x --tags
```

If the fix already exists as commit `abc1234` on a feature branch, step 3 becomes `git cherry-pick abc1234` instead of rewriting the fix. That reduces regression risk — QA validates the same patch that was already code-reviewed.

For **multiple commits** that form one logical fix, cherry-pick a range:

```bash
git cherry-pick A^..C    # commits after A through C (A not included)
# or
git cherry-pick A..C     # commits after A through C
```

Range syntax follows the same `A..B` "reachable from B excluding A" rules as `git log`. When in doubt, list commits with `git log --oneline A..C` before picking.

### Revert in Hotfix and Rollback Scenarios

Revert complements cherry-pick in incident response. Cherry-pick **forward-applies** good work; revert **forward-removes** bad work.

Scenario: commit `bad99` on `main` broke login. Forward fix will take two hours; rollback restores service in minutes:

```bash
git checkout main
git pull
git revert bad99 --no-edit
git push origin main
```

The bug commit remains in history — compliance and postmortems can reference it. Production runs code equivalent to the parent of `bad99` (assuming no later commits depend on the reverted lines).

If **later commits** modified the same lines, revert may conflict. Resolve like a merge, then `git revert --continue`. For **merge commits**, use `git revert -m 1 <merge-hash>` to specify which parent line to keep — the same rule from `0.1.8`.

**Revert then cherry-pick:** sometimes you revert a bad release on `main`, fix properly on a branch, then cherry-pick the good fix commit. History shows revert + fix — verbose but honest.

### Options That Matter in Production

**`-x`** appends `(cherry picked from commit <hash>)` to the commit message. Many teams require this for audit trails linking backports to source commits.

**`--no-commit`** applies changes to the working tree and index without creating a commit — useful when you need to adjust paths or combine with local edits before committing (e.g., different config file in LTS branch).

**`-n` / `--no-commit`** pairs with running tests before `git commit`.

**`--abort`** stops an in-progress cherry-pick and returns to pre-pick state — your escape hatch when conflicts are too messy.

**`--continue`** completes after resolving conflicts (like `git merge --continue`).

**`-e`** opens editor for commit message — default unless `--no-edit`.

## Under the Hood

Cherry-pick is implemented as a **sequential merge** of patches. For each commit to pick, Git:

1. Loads the picked commit's **parent** as merge base.
2. Computes diff from parent tree → picked commit tree.
3. Applies that diff onto current HEAD's tree via merge/recursive strategy.
4. Creates a new commit object with current HEAD as parent, new tree, and metadata (author/committer may differ — see below).

The new commit gets a **fresh SHA-1 hash** even if the tree ends up identical to the original picked commit's tree on another branch. Identity is hash-based; content equality does not imply same commit object.

**Author vs committer:** cherry-pick preserves **author** and **author date** from the original commit by default but sets **committer** to you and **commit date** to now. `git log` shows both — important when tracing who wrote code vs who backported it.

**Duplicate patches:** cherry-picking the same change twice often produces an **empty commit** or Git may skip with "previous cherry-pick is now empty" depending on version and options. Teams track cherry-picked commits in release notes to avoid double-applying.

**Reflog:** failed cherry-picks leave entries; `--abort` resets `CHERRY_PICK_HEAD` and index state. Like merge, Git stores `.git/CHERRY_PICK_HEAD` during operation.

Revert at object level **does not delete** the reverted commit. It runs `git merge-tree` / patch inversion logic to build a tree where the picked commit's changes are negated. The reverted commit stays in the graph as an ancestor of the revert commit — unlike `reset`, which moves branch pointers backward.

## Syntax / Commands / API

```bash
# Pick single commit onto current branch
git cherry-pick <commit-hash>

# Pick range (exclusive..inclusive semantics — verify with log)
git cherry-pick <older>^..<newer>

# Audit trail in message
git cherry-pick -x <commit-hash>

# Apply without committing
git cherry-pick -n <commit-hash>

# After conflicts: edit files, git add, then
git cherry-pick --continue

# Give up
git cherry-pick --abort

# Revert bad commit on shared main
git revert <bad-commit> --no-edit
git push origin main

# Revert merge (keep first parent = usually mainline)
git revert -m 1 <merge-commit-hash>
```

## Examples

### Example 1: Hotfix from feature branch

The security fix is commit `f7a2b1c` on `feature/payments`. Production tracks `main`.

```bash
git fetch origin
git checkout main
git pull origin main
git checkout -b hotfix/payment-null-ref
git cherry-pick -x f7a2b1c
# run tests
git push -u origin hotfix/payment-null-ref
# open PR → merge → tag v2.3.1
```

After merge, backport to supported LTS:

```bash
git checkout release/2.2
git pull
git cherry-pick -x f7a2b1c   # or cherry-pick merge commit from hotfix PR
git push origin release/2.2
```

The `-x` line in the message lets release managers grep for `(cherry picked from commit f7a2b1c)` across branches.

### Example 2: Rollback bad deploy with revert

Commit `d4e5f6a` on `main` enabled a feature flag that crashes startup. Immediate rollback:

```bash
git checkout main
git pull
git revert d4e5f6a -m 1 --no-edit   # -m 1 only if d4e5f6a is a merge commit
git push origin main
# redeploy from main
```

Post-incident: fix forward on `fix/startup-crash`, merge via PR, deploy again. History shows bad commit, revert, and fix — ideal for blameless postmortems.

### Example 3: Cherry-pick conflict

```bash
git cherry-pick abc123
# CONFLICT in src/AuthService.cs
# fix markers, git add src/AuthService.cs
git cherry-pick --continue
```

If you realize the pick was wrong branch entirely: `git cherry-pick --abort` restores pre-pick HEAD and working tree (uncommitted local edits outside conflict may need stash first).

## Common Mistakes & Anti-patterns

**Cherry-picking entire feature branches commit-by-commit** instead of merging creates duplicate commits with different hashes — painful merges later and confusing `git log --cherry-mark`. Prefer merge or rebase for whole features; cherry-pick for isolated commits.

**Force-pushing `main` to remove a bad release** breaks teammates and CI. Revert is slower to read in log but safe for shared branches.

**Forgetting LTS backport** leaves older supported versions vulnerable. Hotfix checklist should list all active release branches.

**Cherry-pick without `-x` in regulated environments** loses traceability between original and backported fix.

**Assuming cherry-pick copies the commit object** — it creates a new one. Links in issue trackers to commit URLs on one branch may not match the hash on another.

**Revert without testing** — revert can fail or partially undo if later commits touched same code. Always run CI on revert PRs.

## Production & Real-World Notes

Many teams use **GitFlow** or **trunk-based** variants: hotfix branches from `main` or from latest release tag, cherry-pick to maintenance branches, semver tag on merge (`0.2.13`). GitHub/GitLab **cherry-pick buttons** on merged PRs automate backport PRs to release branches.

**Release trains** may forbid direct cherry-pick to `main` without PR — same review rules as any merge. **Automated cherry-pick bots** open backport PRs when label `backport/release-1.4` is applied.

For **database migrations**, revert of application code without reverting migration may leave schema inconsistent — coordinate revert commits across repos or use forward-fix migrations only. Monorepo vs polyrepo (`0.2.15`) affects how many cherry-picks you run during an incident.

**Empty cherry-picks** after a fix was already merged manually — use `git cherry -v` to see commits in one branch not in another (patch-id comparison), not just hash presence.

## Comparison / Trade-offs

| Approach | Best for | Risk |
|----------|----------|------|
| Cherry-pick | Single commit backport, hotfix from feature | Duplicate history, merge conflicts per pick |
| Merge branch | Entire feature ready | Ships unwanted commits if branch dirty |
| Revert | Rollback on shared `main` | Messy log; conflicts if history moved on |
| Reset + force push | Solo unpushed mistake | Forbidden on shared production branches |

Cherry-pick vs **rebase then merge**: rebase replays commits on a new base linearly; cherry-pick is explicit one-off replay without rebasing the whole branch. For one commit, cherry-pick is simpler.

## Quick Reference

| Command | Effect |
|---------|--------|
| `git cherry-pick <hash>` | New commit with same patch |
| `git cherry-pick -x <hash>` | Same + source hash in message |
| `git cherry-pick --abort` | Cancel in-progress pick |
| `git revert <hash>` | New commit undoing `<hash>` |
| `git revert -m 1 <merge>` | Revert merge, keep mainline parent |
| `git cherry -v main feature` | Commits in feature not in main (patch level) |

## Key Takeaways

- Cherry-pick **replays** a commit's patch as a **new** commit on the current branch.
- Hotfixes often combine **branch from main → pick or fix → merge → backport** to LTS lines.
- Revert **safely rolls back** shared history without force push; bad commit stays visible.
- Both commands use **merge machinery** — expect conflicts and resolve like `0.2.5`.
- Use **`-x`** for audit trail; **`--abort`** when a pick goes wrong.
- On shared branches: **revert to remove**, **cherry-pick to add** — not reset.
