# TaskCompletionSource — Manual Task Control

> Roadmap: `1.4.5` · Node: `1.4` — C# async ⚡ · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Explain **`TaskCompletionSource<T>`** (TCS) as a manual **`Task`** factory.
- Use **`SetResult`**, **`SetException`**, **`SetCanceled`** exactly once semantics.
- Bridge **event-based** or **callback-based** APIs to **`await`**.
- Avoid **`TrySet*`** races and double-completion bugs.
- Recognize TCS in **`HttpClient`**, timers, **`Channel`**, custom async primitives.
- Apply **`TaskCreationOptions.RunContinuationsAsynchronously`**.

---

## Why This Matters

Not every asynchronous operation starts as `async Task`. Legacy APIs fire events: `DownloadCompleted`, `socket.BeginReceive`. **`TaskCompletionSource`** lets you **produce** a `Task` that completes when *you* decide — connecting foreign callbacks to modern `await` code.

Without TCS, developers block with **`ManualResetEvent`** or spin — pool starvation. With wrong TCS usage, **`SetResult` twice** throws **`InvalidOperationException`** or loses race. Middle developers read library source (`Task.Delay` implementation pattern) and build reliable wrappers.

## Core Concepts

### The Pattern

```csharp
var tcs = new TaskCompletionSource<string>();

SomeEventBasedApi.OnDone += (_, result) => tcs.TrySetResult(result);
SomeEventBasedApi.OnError += (_, ex) => tcs.TrySetException(ex);

string value = await tcs.Task;
```

**`tcs.Task`** is the **`Task<string>`** consumers await. Completing the TCS transitions that task to terminal state and runs continuations.

### SetResult / SetException / SetCanceled

Each TCS completes **at most once**. **`SetResult`** on already completed task throws. **`TrySetResult`** returns `false` if already completed — use in callbacks to handle races.

**`SetCanceled`** / **`TrySetCanceled`** — optional token linking for canceled state.

### Task vs TaskCompletionSource

| | Task (from async/Run) | TCS |
|--|----------------------|-----|
| Who completes? | Runtime/delegate | Your code |
| Start | Automatic | WaitingForActivation until Set* |
| Use | Normal async | Callback/event bridge |

### RunContinuationsAsynchronously

Default (modern): continuations after Set* **may** run synchronously on completing thread — risk stack overflow or deadlock if completing thread holds lock caller awaits on.

**`new TaskCompletionSource<T>(TaskCreationOptions.RunContinuationsAsynchronously)`** — continuations always queued — safer default in libraries.

## Under the Hood

### Internal Link

TCS wraps a **`Task`** in **`WaitingForActivation`** state. **`TrySetResult`** atomically transitions task and queues continuations registered by `await tcs.Task`.

### HttpClient / Sockets (Concept)

Many BCL types use TCS internally per operation: register IOCP callback → **`TrySetResult(bytesRead)`**.

### Timeouts with TCS

```csharp
var tcs = new TaskCompletionSource<bool>();
using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(5));
await using var reg = cts.Token.Register(() => tcs.TrySetCanceled());
DoWorkWithCallback(success => tcs.TrySetResult(success));
await tcs.Task;
```

Combine with **`CancellationToken`** for cancel propagation.

### Multiple Waiters

One TCS, one **`Task`** — **many** can `await tcs.Task`; all resume on single completion (broadcast).

## Syntax / Commands / API

```csharp
var tcs = new TaskCompletionSource<int>(
    TaskCreationOptions.RunContinuationsAsynchronously);

tcs.TrySetResult(42);
tcs.TrySetException(new InvalidOperationException());
tcs.TrySetCanceled();

// Property
Task<int> task = tcs.Task;
```

## Examples

### Example 1: Timer to Task

```csharp
Task DelayFromTimer(int ms)
{
    var tcs = new TaskCompletionSource<object?>(
        TaskCreationOptions.RunContinuationsAsynchronously);
    var timer = new Timer(_ => tcs.TrySetResult(null));
    timer.Change(ms, Timeout.Infinite);
    return tcs.Task;
}
```

(BCL uses **`Task.Delay`** — study reference impl concept.)

### Example 2: Legacy Event

Wrap `WebClient`/`BackgroundWorker`-style event API: subscribe once, **`TrySet*`** in handler, unsubscribe in continuation.

### Example 3: Race — First Wins

```csharp
var tcs = new TaskCompletionSource<string>();
StartFastPath(s => tcs.TrySetResult(s));
StartSlowPath(s => tcs.TrySetResult(s)); // second TrySetResult returns false — OK
return await tcs.Task;
```

## Common Mistakes & Anti-patterns

**Forgetting TrySet** — exception on double callback.

**Never completing TCS** — await hangs forever (memory leak of continuations).

**SetResult on wrong thread with sync continuations** — deadlock with UI/locks.

**Creating new TCS per operation inside shared object without cleanup** — leak handlers.

---

## Production & Real-World Notes

Library code: always **`RunContinuationsAsynchronously`**.

Always link **`CancellationToken`** for client cancel.

In ASP.NET, don't hold request scope in TCS callback after request ended — capture needed data early.

**Channels** (`1.4.17`) often replace hand-rolled TCS queues.

## Comparison / Trade-offs

| Approach | When |
|----------|------|
| `async Task` | You control full flow |
| TCS | External callback completes |
| `Task.FromResult` | Already have result |
| `IValueTaskSource` | High-perf pooling (`ValueTask`) |

## Key Takeaways

- TCS **manufactures** Tasks completed manually.
- **TrySet*** for callback safety; complete exactly once.
- **RunContinuationsAsynchronously** in library wrappers.
- Foundation for bridging legacy async models.
