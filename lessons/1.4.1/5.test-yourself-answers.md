## Answers

### 1. [Recall]
**Answer:** Kernel thread — поток, создаваемый и планируемый ОС. Managed thread — объект `System.Threading.Thread` в CLR, обычно 1:1 с kernel thread.
**Why:** CLR — прослойка над ОС; планирование и context switch делает kernel.

### 2. [Recall]
**Answer:** Создание OS thread дорого (stack, handles, scheduler). Pool переиспользует workers и ограниченно расширяется под нагрузку.
**Why:** Amortized cost + контроль oversubscription.

### 3. [Recall]
**Answer:** Переключение CPU с одного thread на другой: сохранение/восстановление регистров, stack pointer, scheduler state.
**Why:** Позволяет multiprogramming; имеет measurable cost.

### 4. [Understanding]
**Answer:** 10k blocked threads ≈ 10k stacks + scheduler overhead; cores простаивают в переключениях. Async I/O освобождает workers на время wait — меньше blocked threads, выше throughput.
**Common mistake:** Думать, что «больше threads = больше concurrent users» для I/O.

### 5. [Understanding]
**Answer:** Concurrency — много запросов «в полёте»; parallelism — одновременное выполнение на нескольких cores. API может быть concurrent на малых threads (async) без parallelism на каждый request.
**Why:** I/O wait не требует dedicated running thread.

### 6. [Understanding]
**Answer:** `Task.Run` ставит work item в очередь pool; выполняет существующий worker или pool инжектит новый с throttling — не `new Thread()` каждый раз.
**Why:** Queue + reuse — модель pool.

### 7. [Application]
**Answer:** Каждый request блокирует pool worker на 5 сек. При ~min threads (например 16–32) очередь растёт; новые requests ждут worker — latency взлетает, возможен deadlock-подобный hang (starvation).
**Why:** Shared pool + blocking.

### 8. [Application]
**Answer:** Плохая идея без evidence. 200 idle threads — memory, лишние context switches; может маскировать sync-over-async вместо fix root cause.
**Why:** Tuning knob после metrics, не default.

### 9. [Analysis]
**Answer:** Неверно для I/O-bound (threads блокируются) и для CPU-bound сверх core count (oversubscription, context switches). Нужен model work: async I/O vs `Parallel`/pool до ProcessorCount.
**Why:** Threads помогают CPU parallelism до числа cores; I/O — другая модель.

### 10. [Analysis]
**Answer:** 1000 CPU jobs в очереди; pool injects workers медленно; много runnable threads на few cores → context switching; CPU 100% на overhead + work, queue не успевает drain.
**Why:** Throttled injection + oversubscription.

### 11. [Recall]
**Answer:** Текущий код выполняется на worker thread из .NET thread pool.
**Why:** ASP.NET, `Task.Run`, timers — pool threads.

### 12. [Understanding]
**Answer:** При burst queue заполняется быстро; injection ~2 threads/500ms — latency spike до появления достаточных workers.
**Why:** Защита от runaway thread creation.
