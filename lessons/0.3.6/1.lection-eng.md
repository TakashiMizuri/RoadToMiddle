# HTTP Status Codes: Success, Redirection, Client Error, Server Error

> Roadmap: `0.3.6` · Node: `0.3` — HTTP and the Web · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Explain how HTTP status codes fit into the **response start line** you studied in `0.3.3`.
- Classify codes by **first digit** (1xx–5xx) and predict client/server behavior from the class.
- Choose the correct code for common API scenarios: **200 vs 201 vs 204**, **401 vs 403**, **404 vs 409 vs 422**.
- Describe **301 vs 304** and when conditional requests avoid redundant body transfer.
- Interpret **429** and **503** in production: rate limits, retries, and `Retry-After`.
- Distinguish **500 vs 502 vs 503** when debugging layered architectures.
- Map status codes to ASP.NET Core patterns (`Results`, `ProblemDetails`, middleware).

---

## Why This Matters

In lessons `0.3.1` through `0.3.5` you built the request side of HTTP: the client-server model, URL anatomy, message structure (start line, headers, body), methods (GET, POST, PUT, …), and the safe/idempotent matrix. Every response your server sends back includes a **three-digit status code** on the first line — for example `HTTP/1.1 404 Not Found`. That number is not decorative metadata. It is the **primary signal** telling the client what happened and what to do next.

Browsers, mobile SDKs, API gateways, caches, and retry libraries all branch on status codes before they ever parse JSON. A SPA might redirect on **301**, skip re-downloading a bundle on **304**, show a login form on **401**, display "access denied" on **403**, surface validation errors on **422**, back off on **429**, and fail over on **503**. If you return **200 OK** with an error payload, you break this ecosystem — clients cannot use standard HTTP semantics, caches may store failures, and monitors miss real outages.

Middle developers must pick codes **deliberately**, not accidentally. Returning **500** for a validation mistake hides bugs in dashboards. Returning **404** for "you are not allowed to see this resource" leaks information (often **403** is correct). Returning **201** without a **Location** header misses a REST convention clients expect. This lesson turns status codes from a memorized list into a **decision framework** tied to HTTP semantics and real production behavior.

## Core Concepts

### The Status Line: Code + Reason Phrase

An HTTP response begins with a **status line**:

```
HTTP/1.1 404 Not Found
```

The **numeric code** is authoritative. The **reason phrase** (`Not Found`) is human-readable and may vary by server; clients should not depend on exact wording. In HTTP/2 and HTTP/3 the reason phrase is largely omitted on the wire, but the numeric code remains the contract.

Status codes are grouped by **hundreds digit**:

| Class | Meaning | Who "failed"? | Typical client action |
|-------|---------|---------------|----------------------|
| **1xx** | Informational | — | Wait for final response |
| **2xx** | Success | Nobody | Use response body/headers |
| **3xx** | Redirection | — | Follow `Location` or use cache |
| **4xx** | Client error | Client request | Fix request or auth |
| **5xx** | Server error | Server/gateway | Retry, failover, alert |

The distinction between **4xx** and **5xx** is crucial for **observability** and **retry policy**. Clients should generally **not** retry the same 4xx request unchanged (except **429** with backoff). Many **5xx** responses are safe to retry idempotently (especially GET) — a topic that connects directly to idempotency from `0.3.5`.

### 2xx Success: 200, 201, 204

**200 OK** is the default success for requests that return a representation. GET returns the resource; PUT/PATCH often return the updated resource or a summary. It means "the request succeeded and here is content (or headers) answering it."

**201 Created** signals that a **new resource** was created — typically in response to POST (or sometimes PUT when the server assigns a new URI). Best practice includes a **Location** header pointing to the new resource URI and often a body with the created entity. Using 201 for "we inserted a row" helps clients distinguish creation from mere updates.

**204 No Content** means success **without a response body**. Common for DELETE, PUT/PATCH when the client does not need the updated representation, or POST actions that only trigger side effects. Do not send a JSON body with 204 — some clients ignore it; it violates the semantic contract.

Choosing among 200/201/204 is an API design decision: does the client need a body? was a new URI born? ASP.NET Core: `Results.Ok()`, `Results.Created(uri, value)`, `Results.NoContent()`.

### 3xx Redirection: 301 and 304

**301 Moved Permanently** tells clients and caches the resource has a **new canonical URL**. Search engines and browsers update bookmarks; caches may store the mapping. Use when a path permanently changes (`/api/v1/users` → `/api/v2/users`). Include **Location** with the new URI.

**304 Not Modified** is special: it is **not** a redirect to a new URL. It answers conditional GET: "your cached copy is still valid." The client sent validators (`If-None-Match` with ETag or `If-Modified-Since`); the server compares and, if nothing changed, returns **304 with no body** — saving bandwidth. This connects to caching headers in `0.3.7`.

Do not confuse 301 (URL moved) with 304 (representation unchanged at same URL).

### 4xx Client Errors: 400, 401, 403, 404, 409, 422, 429

**400 Bad Request** — the server cannot process the request due to **malformed syntax**: invalid JSON, missing required header, wrong Content-Type. It is a generic "your message is broken" signal. Prefer more specific codes when applicable.

**401 Unauthorized** — authentication **failed or is missing**. The client should obtain credentials (login, refresh token) and retry. Often paired with `WWW-Authenticate`. Despite the name "Unauthorized," RFC usage means **unauthenticated**. In ASP.NET Core, `[Authorize]` without valid credentials yields 401.

**403 Forbidden** — the client **is authenticated** but **not permitted** to perform this action on this resource. Retrying with the same credentials will not help. Example: logged-in user trying to delete another user's order.

**404 Not Found** — no resource matches the URI (or the server chooses not to reveal existence — see security note below). Wrong id, typo in path, deleted entity.

**409 Conflict** — request conflicts with **current server state**: duplicate unique key, optimistic concurrency failure (ETag mismatch), illegal state transition (ship an already-cancelled order). The client may retry after resolving the conflict.

**422 Unprocessable Content** (formerly 422 Unprocessable Entity) — syntax is valid but **semantics fail**: validation errors, business rules. Popular in APIs with structured error bodies (Problem Details, field-level errors). ASP.NET Core model validation often maps to 400 or 422 depending on team convention — **pick one and stay consistent**.

**429 Too Many Requests** — **rate limiting**. Include **Retry-After** (seconds or HTTP-date). Clients must backoff; APIs protect shared capacity.

### 5xx Server Errors: 500, 502, 502, 503

**500 Internal Server Error** — unexpected failure inside the **origin application**: unhandled exception, database down from the app's perspective, bug. Log with correlation id; alert on rate.

**502 Bad Gateway** — a **proxy or gateway** received an **invalid response** from an upstream server (or none at all). Typical in nginx → Kestrel misconfiguration, crashed upstream, TLS errors between layers. The app might be fine; the edge is reporting upstream failure.

**503 Service Unavailable** — server **temporarily** cannot handle traffic: maintenance, overload, dependency circuit-open. Often includes **Retry-After**. Load balancers drain nodes during deploy; health checks return 503.

In incident triage: **502** → look at gateway/upstream chain; **503** → capacity/maintenance/degraded mode; **500** → application logs and exceptions.

### Security and Semantics: 404 vs 403

Returning **404** for resources the caller is not allowed to see is a **deliberate obfuscation** pattern (avoid confirming resource existence). Returning **403** confirms the resource exists but access is denied. Teams choose based on threat model. Neither is universally "wrong" — but mixing them inconsistently confuses API consumers.

## Under the Hood

### Where the Code Is Set in the Pipeline

In ASP.NET Core, status codes emerge at different layers:

```
Request → Middleware → Routing → Filters → Action → Result execution → Response start line
```

An unhandled exception may become **500** via exception middleware. `[Authorize]` failures produce **401/403** before your action runs. `return NotFound()` sets **404** in the result executor. A reverse proxy (nginx, Azure Front Door) may translate upstream failures to **502/504**.

The status code is written when the server commits the **response start line** — before the body streams. Middleware can still change headers early; changing status after headers flush causes errors.

### Caching and 3xx/304 Interaction

When a cache receives **301**, it may store the redirect mapping long-term. **304** requires the client to have sent conditional headers; the cache revalidates with the origin. Wrong use of **200** with long `Cache-Control` on dynamic API data causes stale reads; wrong use of **404** on cacheable assets prevents CDN efficiency — status codes and headers work together (`0.3.7`).

### Monitoring and SLOs

Production systems aggregate by status class:

- **2xx rate** — success ratio (careful: 200 with `{ "error": true }` poisons this).
- **4xx rate** — often split: 401/403 auth noise vs 422 validation vs 404 client mistakes.
- **5xx rate** — primary **error budget** burn; page on-call.

Alerting on **500** spike without ignoring **502/503** at the edge gives full picture in microservice + gateway setups.

## Syntax / Commands / API

### curl: observe status line

```bash
curl -i https://api.example.com/orders/42
# HTTP/1.1 200 OK
# Content-Type: application/json
```

```bash
curl -i -X POST https://api.example.com/orders -d '{}' -H "Content-Type: application/json"
# 400 or 422 depending on API
```

```bash
curl -i -o /dev/null -w "%{http_code}\n" https://api.example.com/health
# 503 during maintenance
```

### ASP.NET Core (minimal examples)

```csharp
app.MapGet("/items/{id}", (int id) =>
    id > 0 ? Results.Ok(new { id }) : Results.NotFound());

app.MapPost("/items", (Item dto) =>
    Results.Created($"/items/{dto.Id}", dto));

app.MapDelete("/items/{id}", () => Results.NoContent());
```

Problem Details (RFC 7807) — structured errors often with 4xx/5xx:

```csharp
return Results.Problem(
    statusCode: StatusCodes.Status422UnprocessableEntity,
    title: "Validation failed",
    extensions: new Dictionary<string, object?> { ["errors"] = fieldErrors });
```

## Examples

### Example 1: Order API lifecycle

POST `/orders` with valid cart → **201 Created**, `Location: /orders/991`, body with order id.  
GET `/orders/991` → **200 OK** with order JSON.  
PATCH status with invalid transition → **409 Conflict**.  
PATCH with invalid email format → **422** (or **400** if your API standard says so).  
DELETE `/orders/991` → **204 No Content**.  
GET `/orders/999999` → **404 Not Found**.

### Example 2: Auth flow

GET `/admin/reports` without token → **401** + `WWW-Authenticate`.  
GET with valid token but role `User` → **403 Forbidden**.  
Same credentials, retry → still **403** (not 401).

### Example 3: Rate limit

API gateway counts requests per API key. Burst exceeded → **429 Too Many Requests**, `Retry-After: 60`. Client waits 60 seconds before retry — not immediate tight loop.

## Common Mistakes & Anti-patterns

**Everything is 200 with `{ success: false }`.** Breaks HTTP semantics, caching, monitoring, and generic HTTP clients. Use appropriate 4xx/5xx.

**401 vs 403 swapped.** 401 = who are you? 403 = I know you, but no.

**201 without Location.** Clients cannot discover the new resource URI from headers alone.

**500 for validation errors.** Pollutes error budgets; client retries uselessly.

**404 for every failure.** Hides whether auth, validation, or missing id caused the problem.

**Ignoring 429 Retry-After.** Causes ban loops and angry platform teams.

## Production & Real-World Notes

Teams publish **API error catalogs** mapping codes to client actions. Gateways normalize upstream errors (hide internal 500 details, return correlation id). **Idempotent GET retries** on 503 are common; **POST retries** need idempotency keys (later phases). Health endpoints: **200** for live, **503** when dependencies fail readiness checks — Kubernetes uses this for traffic routing.

ASP.NET Core `UseStatusCodePages` can rewrite empty 404/403 bodies for browser users while APIs return JSON Problem Details.

## Comparison / Trade-offs

| Situation | Weak choice | Better choice |
|-----------|-------------|---------------|
| Created resource | 200 + body | 201 + Location |
| Success, no body | 200 + empty | 204 |
| Invalid JSON | 422 | 400 |
| Field validation | 400 generic | 422 + field errors |
| Not logged in | 403 | 401 |
| Wrong role | 401 | 403 |
| Duplicate email | 400 | 409 |
| Permanent URL change | 302 temporary | 301 permanent |
| Unchanged cache | 200 full body | 304 |

**400 vs 422:** both valid; 422 emphasizes "understood but semantically invalid." Consistency beats dogma.

## Key Takeaways

- Status codes are the **first-class contract** for machine behavior, not optional decoration.
- **2xx** success, **3xx** redirection/revalidation, **4xx** client fix, **5xx** server/retry (often).
- **201 + Location**, **204** without body, **304** without body — respect semantics.
- **401** authentication, **403** authorization — do not swap them.
- **409** for state conflicts, **422** for validation (or consistent **400**).
- **502** at gateway vs **500** at app vs **503** temporary — different debug paths.
- Align with monitoring, caching, and client retry policies.
