## Answers

### 1. [Recall]
**Answer:** Translates method IL to **native machine code** at runtime, patches entry points, emits GC/unwind metadata.
**Why:** Bridge between IL (`1.5.1`) and CPU.

### 2. [Recall]
**Answer:** Tier 0 — fast compile, lower opt; Tier 1 — transition; Tier 2 — slow compile, full optimization (inlining, loops, devirtualization).
**Why:** Startup vs throughput balance.

### 3. [Recall]
**Answer:** First call pays **JIT compile** (+ possibly class static ctor); no native body yet.
**Why:** Lazy compilation model.

### 4. [Understanding]
**Answer:** **Call count** crosses threshold (~30 default, configurable); runtime queues **background reJIT** to optimizing tier.
**Why:** Profile hot paths without slow compile for cold code.

### 5. [Understanding]
**Answer:** Replace running loop's code with optimized tier **without waiting for method return** — important for long server loops.
**Why:** Promotion mid-execution.

### 6. [Understanding]
**Answer:** JIT chooses frame layout; emits **GC info** marking which stack slots hold managed refs at each instruction — used in stack walks (`1.5.4`, `1.5.6`).
**Why:** Correct roots during GC/exceptions.

### 7. [Application]
**Answer:** First iterations run **Tier 0** (+ JIT cost); after warmup **Tier 2** native code — orders of magnitude faster.
**Why:** Always warmup benchmarks.

### 8. [Application]
**Answer:** Many short-lived or frequently scaled containers where **cold start SLA** matters; accept larger publish size and possibly less peak opt than Tier 2.
**Why:** R2R reduces JIT work at startup.

### 9. [Application]
**Answer:** Potentially **two** distinct native bodies — generic value/reference type instantiations compile separately.
**Why:** Generic sharing limits but many closed types = more JIT memory/time.

### 10. [Analysis]
**Answer:** Disabling tiering removes fast Tier 0 startup path and/or tier promotion story; may **increase** compile latency on all methods or lose Tier 2 peak — hurts steady-state throughput.
**Why:** Tiering optimizes both phases.

### 11. [Analysis]
**Answer:** **C** — CoreCLR JIT-compiles to native; no perpetual IL interpreter for hot server paths.
**Why:** Modern .NET execution model.

### 12. [Understanding]
**Answer:** AOT: best startup, no runtime JIT, smaller attack surface; less dynamic reflection/codegen, compile-time constraints.
**Why:** Different deployment axis.

### 13. [Application]
**Answer:** (1) JIT of host, Kestrel pipeline, middleware on first requests; (2) Tier 0 code before promotion — CPU spent in clrjit + slower unoptimized code.
**Why:** Cold start + warm-up.

### 14. [Analysis]
**Answer:** Ensure measurements hit **Tier 2** (or steady tier), exclude one-time JIT compile from stats.
**Why:** Otherwise misleading perf numbers.

### 15. [Understanding]
**Answer:** **Code heap** holds **native method bodies** (unmanaged executable pages). **Managed heap** holds **objects** (`1.5.5`) tracked by GC.
**Why:** Separate allocators and lifetimes.
