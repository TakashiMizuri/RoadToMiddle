## Answers

### 1. [Recall]
**Answer:** VCS = **Version Control System**. Primary problem: tracking changes over time so teams can **collaborate safely**, **revert** mistakes, and **audit** who changed what.

**Why:** Without VCS, manual copies and overwrites make teamwork and rollback unreliable.

**Common mistake:** Saying "backup only" — backup is one benefit, not the core (collaboration + branching matter more for code).

---

### 2. [Recall]
**Answer:**
- Centralized: **SVN**, CVS, Perforce (TFVC in centralized mode)
- Distributed: **Git**, Mercurial

**Why:** The classification depends on **where full history lives**, not brand popularity.

---

### 3. [Understanding]
**Answer:** Local VCS keeps history on **one machine**. Other developers cannot access shared history, cannot merge work reliably, and there is no team-wide source of truth.

**Why:** Collaboration requires **shared, synchronized history** — local-only cannot provide that.

---

### 4. [Understanding]
**Answer:** Developers typically **cannot commit** (and often cannot checkout/update reliably). Operations depend on the server as the authority.

**Why:** Centralized model routes most write operations through the central server — network/server availability is critical.

**Common mistake:** Saying "nothing works at all" — local edits may still exist, but **version control operations** are blocked.

---

### 5. [Understanding]
**Answer:** **Yes, valid.** `git commit` records a snapshot in the **local** `.git` repository. Network is only needed for `push`/`pull`/`fetch`.

**Why:** Distributed VCS decouples **recording history** from **sharing history**.

---

### 6. [Understanding]
**Answer:** Every clone already has **complete history**. GitHub/GitLab is a **convention point** to exchange commits — if it disappears temporarily, history survives in every developer's clone.

**Why:** Custody is **replicated** across clones; remote is not the sole database.

---

### 7. [Application]
**Answer:** **Centralized** generation (SVN). Git's **local commits** and **full local repository** allow work to continue offline; sync happens later via push.

**Why:** The pain point (commit blocked without server) is the classic centralized weakness.

---

### 8. [Application]
**Answer:** GitHub is a **hosting service** for Git remotes. Git remains **distributed** because each developer holds a **full repo** locally. GitHub being "central" for the team doesn't make Git centralized — it makes the **remote** central for sync.

**Common mistake:** Confusing **hosting UI** with **VCS architecture**.

---

### 9. [Analysis]
**Answer:** **C** is wrong.

**Why:**
- A ✅ — default clone includes full history
- B ✅ — SVN model: server holds authoritative full history
- C ❌ — teams still use central remotes; "no central server" is false
- D ✅ — Git branches are lightweight pointers

---

### 10. [Analysis]
**Answer:** Anti-pattern: **"Git as SVN"** — using Git infrastructure but a **linear, server-centric workflow**. Resembles **centralized** habits (everyone pushes to one shared line without isolation/review).

**Why:** Git's strength is cheap branches + distributed workflows (PRs, feature branches).

---

### 11. [Recall]
**Answer:** Any three of:
- Full history on every clone
- Commits work offline
- No single point of failure for history (clones are backups)
- Cheap/fast branching
- Faster local operations

**Why:** These follow directly from **replicated full repositories**.

---

### 12. [Understanding]
**Answer:** Examples:
- **GitHub ecosystem** — social coding, PRs, massive adoption
- **Performance** on large repos (Linux kernel)
- **Open source** and community momentum
- **Branching model** fit modern CI/CD

**Why:** Mercurial is also DVCS; Git's dominance is ecosystem + design fit, not "distributed vs not."

---

### 13. [Application]
**Answer:**

| Step | Local | Centralized | Distributed |
|------|-------|-------------|-------------|
| `commit` | ✅ primary on machine | ❌ usually requires server | ✅ primary locally |
| `view full project history` | ✅ if local tool keeps it | ❌ often limited locally | ✅ full history in clone |

**Why:** Centralized clients often have **partial** local data; distributed clones are **complete**.

---

### 14. [Analysis]
**Answer:** **No.**

**Why:** Cloud storage lacks **commit graph**, **meaningful merge model for code**, **branching workflow**, and **CI/code-review integration**. Version history in Drive is not equivalent to structured VCS semantics.

---

### 15. [Understanding]
**Answer:**
- Technical: feature branches, offline commits, faster local ops, better merge tools, CI integration
- Organizational: Pull Request code review culture, parallel feature development, clearer audit trail

**Why:** Migration benefits are both **tooling** and **process**.
