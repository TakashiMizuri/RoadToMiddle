# struct vs class: Performance, Mutability, and Semantics

> Roadmap: `1.2.14` · Node: `1.2` — C# OOP · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Explain **value semantics** (`struct`) vs **reference semantics** (`class`) at the CLR level.
- Describe **stack vs heap** allocation patterns and when the distinction blurs (boxing, closures, async).
- Predict **copy behavior** when passing structs to methods or assigning variables.
- Choose **struct vs class** using size, mutability, identity, and lifetime criteria.
- Understand **mutating struct** pitfalls through hidden copies (`foreach`, properties).
- Relate structs to **`1.2.13` records** and **`1.2.15` readonly/ref struct** restrictions.

## Why This Matters

Every C# type is either a **value type** or **reference type**. This single choice drives allocation, GC pressure, equality, threading, and API design. `int`, `DateTime`, `Guid` are structs; `string`, `List<T>`, your domain services are classes. Wrong choice costs money at scale: copying a 256-byte struct on every loop iteration; boxing value types into `object` for collections before generics; mutable struct surprises causing "my changes disappeared."

From `1.2.1` you learned objects are references on the heap. This lesson completes the picture for **stack-allocated value types** and when the runtime still puts data on the heap. Middle developers justify `readonly struct Vector3` in hot paths and explain why EF entities must stay **classes**.

## Core Concepts

### Two Families of Types

**Value types** (`struct`, primitives, enums) default to **copy-by-value**: assignment duplicates bits (field-wise copy). **Reference types** (`class`, `interface`, `delegate`, `record class`) copy a **pointer** (4 or 8 bytes); two variables can alias the same object.

```csharp
var a = new Point(1, 2);   // struct
var b = a;                 // full copy — b is independent
b.X = 99;
Console.WriteLine(a.X);    // 1

var c = new Customer();    // class
var d = c;                 // copy reference
d.Name = "X";
Console.WriteLine(c.Name);   // "X" — same object
```

**Identity:** reference types have stable **object identity** (`ReferenceEquals`). Structs have no separate identity — equality is value-based (unless boxed).

### struct Declaration and Defaults

```csharp
public struct Point
{
    public int X;
    public int Y;
}
```

`new Point()` zero-initializes: `X = 0`, `Y = 0`. Cannot inherit from another struct or class (except `System.ValueType` implicitly). Can implement interfaces — but interface calls may **box** the struct.

Structs cannot have parameterless instance constructor before C# 10 rules evolved; modern C# allows explicit parameterless ctor but **default** still zeroes unless you define otherwise carefully.

**Size guideline:** Microsoft historically suggested **< 16 bytes** as rough struct size budget for performance; larger structs cost copy overhead. Not a hard law — profile hot paths.

### Copy Semantics in Methods

Structs passed **by value** to methods copy the entire struct unless you use **`ref`**, **`in`**, or **`out`** (`1.2.15` preview):

```csharp
void Move(Point p) { p.X++; }           // copy — caller unchanged
void MoveRef(ref Point p) { p.X++; }     // alias — caller sees change
```

**Mutating struct** methods should often be `ref readonly` returns or return new struct (immutable pattern) to avoid confusion.

### class: Reference on the Heap

```csharp
public class Customer
{
    public int Id { get; set; }
    public string Name { get; set; } = "";
}
```

`new Customer()` allocates on the **managed heap**; GC reclaims when unreachable. Reference type fields inside a struct still point to heap objects — the struct copy duplicates pointers, not deep graph.

```csharp
public struct Wrapper
{
    public string Label;  // reference field — struct copy shares string instance
}
```

### Mutability Traps with Structs

**foreach** copies:

```csharp
foreach (var p in points)
    p.X++;   // modifies copy — does NOT update list
```

**Auto-properties** on structs in readonly contexts, **nullable structs**, and **defensive copies** in properties — C# compiler inserts copies to preserve struct semantics when `this` could mutate readonly struct.

Prefer **immutable structs** (`readonly struct`, init-only) from `1.2.15` when possible.

### When to Prefer struct

- Small, **logical value** with no identity (coordinates, money amount pair, date range).
- **High churn** local variables where avoiding heap allocation helps (profile first).
- **Immutable** data passed frequently by value.
- **`record struct`** (`1.2.13`) when you want synthesized equality on value types.

### When to Prefer class

- **Identity matters** (customer id 42 is one entity, not a value tuple).
- **Large** or **growing** state (collections, many fields).
- **Polymorphism** via inheritance (structs cannot derive).
- **Null** needed as sentinel — reference types nullable by default; structs need `Nullable<T>` or `T?`.
- **EF Core entities**, long-lived graphs, shared mutable state.

### Boxing and Unboxing

Assigning struct to `object` or non-generic collection **boxes**: allocates heap object wrapping struct copy. **Unboxing** copies out.

```csharp
object o = 42;        // boxes int
int x = (int)o;       // unbox copy
```

Generics (`List<int>`, `Dictionary<Guid, T>`) avoid boxing for type parameters constrained to `struct` or unconstrained with known value types. **`1.1.7` boxing** — GC pressure and cache misses.

### string Is a Special Reference Type

`string` behaves as immutable reference type with value-like equality semantics — exception to "reference = identity for ==".

## Under the Hood

### Memory Layout

**Stack (conceptual):** each thread has stack frames for locals and parameters. Small **value types** often live in stack slots — fast allocate/deallocate with frame pop. **Not guaranteed:** JIT may enregister locals or place them on heap if escaped (closures, `async` state machines, `ref` to heap).

**Heap:** reference type objects include **object header** (sync block + method table pointer) + fields. Struct inside class object lives **inline** in object's payload — no separate struct allocation.

```
class Container { public Point P; }  →  [header][Point.X][Point.Y] on heap (one object)

struct Point on stack frame:  [X][Y] in stack slot
```

### Copy Cost

Assignment `b = a` for struct of size N copies N bytes (field-wise, sometimes with padding for alignment). For large N, **`in` parameter** passes readonly reference to avoid copy:

```csharp
void Print(in LargeStruct s) { ... }
```

### GC Interaction

Structs themselves are not GC-tracked unless **boxed** or contain reference fields. Struct with `string` field: copying struct copies string **reference**; GC tracks the string object separately.

**Array of structs** (`Point[]`) — contiguous value storage, cache-friendly vs array of class references (pointer chasing).

### Interface Method on Struct

Calling interface method on struct through interface type **boxes** (unless generic constrained call with struct type parameter and JIT devirtualizes):

```csharp
IComparable c = myStruct;  // box
```

Design APIs with **generics `where T : struct`** or concrete types in hot paths to avoid boxing.

### ref struct and stack-only Types

Preview `1.2.15`: `Span<T>`, `ReadOnlySpan<T>` are **ref struct** — must stay on stack, cannot box, cannot be field of regular class. Bridges performance and safety for buffers.

## Syntax / API

```csharp
public struct Point { public int X, Y; }
public class Entity { public int Id { get; set; } }

void ByValue(Point p);
void ByRef(ref Point p);
void ByReadonly(in Point p);

public readonly struct ImmutablePoint { public int X { get; } public int Y { get; } }
```

| Modifier | Effect |
|----------|--------|
| `struct` | Value type |
| `readonly struct` | All instance members readonly |
| `ref struct` | Stack-only, no boxing (`1.2.15`) |
| `record struct` | Value type + record features |

## Examples

### Example 1: Hidden foreach Copy

```csharp
var list = new List<Point> { new(1, 1) };
foreach (var p in list)
    p.X = 10;
Console.WriteLine(list[0].X);  // 1 — not 10
```

**Fix:** `for` loop with index, or immutable update pattern.

### Example 2: struct with Reference Field

```csharp
public struct LabelBox
{
    public string Text;
}

var a = new LabelBox { Text = "hi" };
var b = a;
b.Text = "bye";
Console.WriteLine(a.Text);  // "bye" — shared string on heap
```

Value copy duplicated pointer, not string content (`string` immutable anyway — reassignment affects shared reference field).

### Example 3: Array of Struct vs Class

```csharp
Point[] points = new Point[1_000_000];      // contiguous structs
Customer[] customers = new Customer[1_000_000];  // array of references — 1M heap objects if all new'd
```

Memory and cache behavior differ dramatically.

## Common Mistakes & Anti-patterns

**Large mutable struct** — expensive silent copies.

**Mutable struct in collections + foreach mutation** — changes lost.

**struct for OOP hierarchy** — impossible; forces switches instead of polymorphism.

**Boxing in hot loops** — `ArrayList` era pattern; use generics.

**Every DTO as struct** — large copies on every ASP.NET binding path; classes often fine.

**Assuming struct always on stack** — escaping to heap is legal and happens.

## Production & Real-World Notes

**Performance:** profile with `dotnet-counters`, Allocation Tracker. `struct` wins in tight numeric loops; loses when copied often due to size.

**API design:** public APIs rarely expose mutable structs — confusing. BCL uses `DateTime`, `Guid`, `TimeSpan` as well-behaved structs (mostly immutable).

**EF Core:** entities are classes; owned types map value object columns but are complex.

**ASP.NET:** request models are classes/records; small internal value types OK in domain core.

## Comparison / Trade-offs

| Criterion | struct | class |
|-----------|--------|-------|
| Assignment | Copy all fields | Copy reference |
| Null | Needs `T?` | Nullable reference |
| Inheritance | No | Yes |
| Identity | No object identity | `ReferenceEquals` |
| Typical size | Small | Any |
| GC pressure | Lower if no boxing | Heap object per instance |
| Polymorphism | Interfaces (may box) | Virtual methods |

---

## Quick Reference

| Choose **struct** | Choose **class** |
|-------------------|------------------|
| < ~16–32 bytes, immutable | Identity, ORM entity |
| Value object, no id | Inheritance needed |
| High-frequency local math | Large/mutable graph |
| `record struct` equality | `null` sentinel |

## Key Takeaways

- **struct** = copy-by-value; **class** = copy reference, shared object.
- **Stack vs heap** is JIT/runtime decision; structs can box or escape.
- **Mutable struct** + foreach/copy = subtle bugs — prefer immutable.
- **Boxing** hurts GC — generics avoid it.
- **Large structs** copy often — expensive; use `in`/`ref` or class.
- **EF / identity / polymorphism** → class; **small values** → struct.
