## Easy

### E1. Classify the scenario
**Goal:** Recognize VCS generation from a description.

For each scenario, write **Local**, **Centralized**, or **Distributed** and one sentence why:

1. A developer saves `project_v3_final.zip` on a USB drive.
2. A team uses SVN; when the server is offline, nobody can commit.
3. A developer on a flight runs `git commit` without Wi-Fi, then `git push` at the hotel.
4. One person uses RCS on a Linux workstation; no network involved.

**Success criteria:** 4/4 correct classifications with valid reasoning.

## Medium

### M1. Failure analysis
**Goal:** Understand single point of failure.

**Scenario:** A company uses SVN. On Friday at 5 PM the central server disk dies. Backups are 3 days old.

Answer in writing (5–8 sentences):

1. What immediate problems do developers face?
2. What work might be lost?
3. How would the same situation differ with Git + GitHub if GitHub went down for 4 hours?

**Success criteria:** You mention network dependency, checkout/commit blockage, and that Git clones retain full local history.

---

### M2. Draw the architecture
**Goal:** Visual memory.

Draw (on paper or Excalidraw) two diagrams:

1. **Centralized VCS** — 3 developers + 1 central server. Show direction of checkout and commit.
2. **Distributed VCS** — 3 developers + 1 remote (origin). Show that each dev has full repo; push/pull arrows.

Label arrows: `checkout`, `commit`, `push`, `fetch`.

**Success criteria:** Diagrams show that centralized commits go *to server only*; distributed commits happen *locally first*.

## Hard (stretch)

### H1. Migration argument
**Goal:** Production thinking.

Your team maintains a 15-year-old ASP.NET Framework monolith in **SVN**. Management asks: "Why spend 3 months migrating to Git?"

Write a short proposal (200–300 words) covering:

- Risks of staying on SVN
- Benefits of Git for CI/CD and Pull Requests
- One risk of migration itself

**Success criteria:** Balanced argument; not "Git is better because it's new."
