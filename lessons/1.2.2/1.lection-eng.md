# Fields and Properties — Auto, Full, Init-Only, Required

> Roadmap: `1.2.2` · Node: `1.2` — C#: OOP · Depth: **practice**

## Learning Objectives

After this lesson you will be able to:

- Distinguish **fields** (storage) from **properties** (controlled accessors).
- Write **auto-properties**, **full properties** with backing fields, and choose when each fits.
- Use **`init`** accessors for immutable-after-construction state.
- Apply **`required`** members (C# 11+) so callers must set critical values at initialization.
- Understand **default values**, **property initializers**, and compiler-generated backing fields.
- Avoid common leaks of invariants through poorly designed public setters.

---

## Why This Matters

In `1.2.1` you saw classes as blueprints with state and behavior. The next question is **how that state is stored and exposed**. Novices often use public fields because they compile quickly. Professional C# overwhelmingly uses **properties** — a thin API surface that can evolve (validation, logging, change notification) without breaking callers.

ASP.NET Core model binding, JSON serializers, EF Core mappings, and React form DTOs all interact with **property names**, not raw fields. `init` and `required` support **immutable configuration objects** and safer construction — patterns you will see in options classes, record-like DTOs, and domain entities where identity fields must be set exactly once.

Understanding the compiler's translation from `{ get; set; }` to a hidden backing field helps you read decompiled code, debug serialization issues, and know when a **full property** is worth the verbosity.

## Core Concepts

### Fields: Raw Storage

A **field** is a variable declared inside a type. Instance fields exist **per object**; static fields belong to the **type** (`1.2.12`).

```csharp
public class TemperatureReading
{
    private decimal _celsius;   // instance field — storage
}
```

Fields can be **`private`** (default choice for storage), **`public`** (discouraged for most cases — see `1.2.5`), **`readonly`**, or **`const`/`static readonly`** for constants.

Fields hold the **actual bits** (or references) inside the object layout on the heap. Properties usually **front** fields, not replace the need for storage.

### Properties: Controlled Access

A **property** looks like a field to callers but is implemented with **`get`** and **`set`** accessors (or only one of them):

```csharp
public decimal Celsius
{
    get => _celsius;
    set => _celsius = value;
}
```

Call site syntax is uniform: `reading.Celsius = 20m;` whether auto-generated or hand-written. That uniformity is why libraries bind to properties.

**Expression-bodied accessors** (`=>`) keep simple pass-through properties readable. **Full properties** add logic:

```csharp
public decimal Fahrenheit
{
    get => _celsius * 9m / 5m + 32m;
    set => _celsius = (value - 32m) * 5m / 9m;
}
```

Validation, rounding, and side effects belong here — not scattered at every assignment site.

### Auto-Properties

When you need **no extra logic**, the compiler generates a hidden **backing field**:

```csharp
public string Sku { get; set; }
public int Quantity { get; init; }
```

This is an **auto-property**. IL contains a compiler-generated field like `<Sku>k__BackingField`. You cannot reference that field name in source unless you declare your own backing field for a full property.

**Property initializers** run for new instances:

```csharp
public bool IsActive { get; set; } = true;
public List<string> Tags { get; set; } = new();
```

Be careful with **mutable reference defaults** (`new List<>()`): each instance gets its own list, but exposing `public set` still allows replacing the whole list from outside.

### Read-Only Surface: get-only and init

**Get-only** (set only in constructor or initializer):

```csharp
public class OrderLine
{
    public Guid LineId { get; } = Guid.NewGuid();
}
```

**`init` accessor** (C# 9+): settable only during **object initialization** — object initializer or constructor body before construction completes:

```csharp
public class Product
{
    public required string Sku { get; init; }
    public required string Name { get; init; }
    public decimal UnitPrice { get; init; }
}

// OK:
var p = new Product { Sku = "A1", Name = "Widget", UnitPrice = 9.99m };

// Error after construction:
// p.Sku = "B2";
```

`init` enables **immutable-after-build** DTOs without verbose constructor-only APIs. Combined with **`required`**, the compiler enforces that callers supply `Sku` and `Name` at creation.

### Required Members (C# 11+)

**`required`** on a property or field means object creation **must** assign it:

```csharp
public required string Email { get; set; }
```

Valid assignments: object initializer, constructor assigning the member, `with` expression on records. Missing `Email` at creation is a **compile error**:

```csharp
// var user = new UserAccount();  // CS9035 — required member not set
var user = new UserAccount { Email = "a@example.com" };
```

`required` improves **construction completeness** without giant constructor parameter lists. It works with **`init`** or **`set`**.

### Full Property with Explicit Backing Field

Use a **full property** when auto-property is insufficient — validation, lazy initialization, computed cache:

```csharp
private string _email = string.Empty;

public string Email
{
    get => _email;
    set
    {
        if (string.IsNullOrWhiteSpace(value))
            throw new ArgumentException("Email required.");
        _email = value.Trim().ToLowerInvariant();
    }
}
```

Naming convention: **`_camelCase`** private field, **`PascalCase`** public property. Some teams use `email` field without underscore for full properties — consistency matters more than the exact style.

### Fields vs Properties — Quick Decision

Use **private fields** for storage. Expose **properties** publicly. Use **full properties** when invariants or computed values matter. Use **`init` + required** for create-once configuration objects.

## Syntax / API

```csharp
// Auto-property
public int Count { get; set; }

// Init-only
public string Code { get; init; }

// Required
public required string Title { get; init; }

// Get-only with initializer
public DateTime CreatedUtc { get; } = DateTime.UtcNow;

// Full property
private int _age;
public int Age
{
    get => _age;
    set => _age = value >= 0 ? value : throw new ArgumentOutOfRangeException();
}

// Read-only auto (private set)
public string Status { get; private set; } = "Draft";
```

## Examples

**Example 1 — validation in setter:**

```csharp
public class BankAccount
{
    private decimal _balance;

    public decimal Balance
    {
        get => _balance;
        private set
        {
            if (value < 0) throw new InvalidOperationException("Negative balance.");
            _balance = value;
        }
    }

    public void Deposit(decimal amount) => Balance = _balance + amount;
}
```

External code reads `Balance` but cannot assign directly; mutations go through methods that respect rules.

**Example 2 — required init DTO:**

```csharp
public class CreateCustomerCommand
{
    public required string Name { get; init; }
    public required string Email { get; init; }
    public string? Phone { get; init; }
}
```

ASP.NET Core minimal APIs and `[FromBody]` deserialization can populate `init` properties; missing required members fail at compile time in your code and at validation time for incoming JSON if configured.

**Example 3 — computed property (no backing field):**

```csharp
public class Rectangle
{
    public double Width { get; set; }
    public double Height { get; set; }
    public double Area => Width * Height;   // get-only computed
}
```

`Area` has no storage; it derives from other state. Marking it `{ get; }` without `set` documents read-only derived data.

## Common Mistakes & Anti-patterns

**Public fields for DTOs "because it's faster."** Breaks encapsulation and serializer conventions; use properties (`1.2.5`).

**Exposing mutable collections with public `set`.** Prefer `IReadOnlyList<T>` surface or init-only collection properties.

**Validation only in UI**, not in property/domain layer — API clients bypass UI.

**`required` without `init`/constructor path** — understand all legal assignment contexts.

**Auto-property with logic needed later** — refactor to full property; callers unchanged (good design win).

## Production & Real-World Notes

System.Text.Json deserializes into **public properties** with setters or `init` by default. `[JsonInclude]` can include fields — rare in app code.

EF Core maps **properties** (and sometimes fields) to columns. Backing field access is configured for encapsulation (`UsePropertyAccessMode`).

`IOptions<T>` options classes often use `{ get; init; }` bound from configuration — immutable after binding.

## Comparison / Trade-offs

| Mechanism | Mutability after ctor | Compiler enforcement | Typical use |
|-----------|----------------------|----------------------|-------------|
| `{ get; set; }` | Fully mutable | None | Entities, view models |
| `{ get; init; }` | Frozen after init | Init context only | Commands, DTOs |
| `{ get; private set; }` | Mutated inside type | External read-only | Domain entities |
| `required` | N/A | Must assign at creation | Critical identifiers |
| Full property | Controlled | Custom validation | Invariants |

---

## Quick Reference

```
field     → storage (prefer private)
property  → get/set API
{ get; set; }           → auto-property + backing field
{ get; init; }          → set only during initialization
required                → must assign when creating object
private set / init      → limit who can mutate
```

---

## Key Takeaways

- **Fields store**; **properties expose** with a stable API.
- **Auto-properties** compile to hidden backing fields — fine when no logic needed.
- **Full properties** centralize validation and computed values.
- **`init`** freezes properties after object construction.
- **`required`** forces callers to supply critical values at creation time.
- Libraries bind to **properties** — design public surface accordingly.
