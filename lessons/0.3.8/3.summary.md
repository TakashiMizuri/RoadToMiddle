# HTTPS и TLS — краткая выжимка

> `0.3.8` · ~10 мин на повторение

## Главная идея

**HTTPS** = HTTP (`0.3.3`) поверх **TLS** на TCP:443. Шифрует и проверяет целостность channel; semantics methods/headers/status codes не меняются. Без HTTPS credentials в headers (`0.3.7`) readable on the wire.

## Ключевые пункты

- TLS handshake: negotiate ciphers, **verify server certificate chain**, derive **symmetric session keys** для bulk HTTP.
- **X.509 cert**: public key + identity (SAN hostnames) + CA signature; failures — expired, wrong name, untrusted/missing chain.
- **Asymmetric** (cert) для auth/key agreement; **symmetric** (AES-GCM) для data — hybrid model.
- Production: TLS at **load balancer** или **Kestrel**; `X-Forwarded-Proto` за proxy; **HSTS**.
- **Secure** cookie (`0.3.9`) отправляется только по HTTPS.
- Dev: `dotnet dev-certs https --trust`; prod: Let's Encrypt / managed certs, monitor expiry.

## Самые важные примеры

**Stack:**
```
HTTP messages → TLS encrypt → TCP:443
```

**Inspect:**
```bash
openssl s_client -connect host:443 -servername host
```

**Proxy pattern:** Client—HTTPS—nginx—HTTP—Kestrel; app needs forwarded headers for redirects.

## «Под капотом» в одном абзаце

Client TCP connect → ClientHello (SNI hostname, ciphers) → ServerHello + certificate chain → key exchange (ECDHE, forward secrecy) → Finished → encrypted Application Data records carry HTTP bytes. Browser validates chain to trust store + SAN match. TLS 1.3 cuts RTTs vs 1.2. Resumption speeds repeat visits. ASP.NET `UseHttpsRedirection`/`UseHsts` enforce HTTPS at app edge when configured correctly behind proxy.

## Частые ошибки

- HTTP login/API in production.
- Incomplete cert chain (works desktop, fails mobile).
- TLS terminated but app generates `http://` URLs.
- Ignore certificate expiry.

## Мнемоника

> **HTTPS** защищает **дорогу** (transport), не **содержимое логики** (auth still required). Cert = «паспорт server identity»; session key = « одноразовый сейф для этого разговора».
