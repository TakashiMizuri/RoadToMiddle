## Answers

### 1. [Recall]
**Answer:** **Working tree** (рабочие файлы проекта) и **`.git` directory** (Git directory — база history, objects, refs, config).

**Why:** Repository — не только исходники; history живёт в `.git/`.

**Common mistake:** Называть repository только папку с `.cs` / `.tsx` файлами.

---

### 2. [Recall]
**Answer:** **Modified**, **staged**, **committed**.

**Why:** Это три состояния tracked файла в Git lifecycle.

---

### 3. [Understanding]
**Answer:** **Working directory** — файлы на диске, которые ты редактируешь. **Staging area (index)** — промежуточная зона: preview того, что войдёт в **следующий** commit. Changes попадают в index через `git add`.

**Why:** Working = «что на столе»; staging = «что упаковано для отправки в history».

---

### 4. [Understanding]
**Answer:** Staging позволяет **atomic commits** (одно логическое изменение), **partial commits** (stage только нужные файлы/hunks), **review before commit** (`git diff --staged`).

**Why:** Без staging commit'ил бы всё подряд или нужен был бы другой механизм split changes.

**Common mistake:** «Staging — лишний шаг для новичков» — это core design, не optional hack.

---

### 5. [Understanding]
**Answer:** Git может показать **два набора** изменений для одного файла: **staged** (версия на момент `git add`) и **unstaged/modified** (новые правки после add).

**Why:** Index зафиксировал snapshot при add; последующие edits не попадают в index автоматически.

---

### 6. [Understanding]
**Answer:** Metadata: **parent commit(s)**, **author**, **committer**, **timestamp**, **commit message**; structurally — **tree** (snapshot файлов).

**Why:** Commit — не только файлы, но и связь с history и контекст «кто/когда/зачем».

---

### 7. [Application]
**Answer:**
```bash
git add OrderService.cs
git commit -m "Fix order total calculation bug"
# appsettings.Development.json остаётся unstaged
```
(Опционально: `git diff --staged` перед commit для review.)

**Why:** Stage только нужный файл — partial commit через staging area.

**Common mistake:** `git add .` — закоммитит и appsettings.

---

### 8. [Application]
**Answer:** **Repository** — локальная сущность: working tree + `.git/`. GitHub — **remote** (hosted copy для sync и collaboration). У каждого разработчика свой local repository; GitHub не единственное место history.

**Common mistake:** Путать hosting UI с определением repository.

---

### 9. [Analysis]
**Answer:** **C** неверно (не всегда).

**Why:**
- A ✅ — `.git/` хранит Git database
- B ✅ — index = `.git/index`
- C ❌ — после commit **закоммиченные** изменения совпадают с HEAD, но могут остаться **другие** unstaged/untracked изменения
- D ✅ — untracked вне трёх состояний до `git add`

---

### 10. [Analysis]
**Answer:** Anti-pattern: **blind `git add .`** + **meaningless commit messages**. Проблемы: unrelated changes в одном commit, сложный review/blame, debug/secrets в history.

**Why:** Staging и messages существуют для intentional, atomic history.

---

### 11. [Recall]
**Answer:** `git restore --staged <file>` (или legacy `git reset HEAD <file>`).

**Why:** Unstage убирает из index, не трогая working directory edits.

---

### 12. [Understanding]
**Answer:**
- **Messy desk** = working directory — все текущие правки
- **Inbox tray** = staging area — что выбрано для следующей «отправки»
- **Archived boxes** = repository (`.git`) — permanent snapshots с labels (commits)

**Why:** Аналогия отражает deliberate selection перед записью в history.

---

### 13. [Application]
**Answer:** Пример flow:
```
new file (untracked) → git add → staged → git commit → committed (tracked)
edit file → modified → git add → staged → git commit → committed
```

**Why:** Untracked → first add enters tracking; далее цикл modified → staged → committed.

---

### 14. [Understanding]
**Answer:** Git хранит **полный snapshot** дерева файлов на момент commit (через tree objects), а не только diff. Это эффективно при повторяющемся content и упрощает checkout любой revision.

**Why:** Snapshot model — fundamental Git design (Pro Git: "Snapshots, Not Differences").

**Common mistake:** Думать, что commit = только diff как в RCS.

---

### 15. [Analysis]
**Answer:** Ошибка **до commit** — debug-код попал в **staged snapshot** (скорее всего через `git add .` без review). Помогло бы: `git diff --staged`, selective `git add`, atomic commits, code review до push.

**Why:** CI строит committed state; unstaged WIP не попадает в commit, если не был staged.
