# Generations: Gen0, Gen1, Gen2, and Promotion

> Roadmap: `1.5.7` · Node: `1.5` — .NET Runtime and Memory · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Explain **generational hypothesis** — most objects die young.
- Describe **Gen0**, **Gen1**, **Gen2** roles and **promotion** between them.
- Predict when **Gen0 vs full GC** occurs and typical pause characteristics.
- Distinguish **Workstation GC** vs **Server GC** for ASP.NET Core.
- Connect high allocation rate to ** frequent Gen0 collections**.
- Explain **ephemeral segment** and why full GC is expensive.

---

## Why This Matters

Every allocation in `1.5.5` lands in **Gen0** (unless LOH — `1.5.8`). Most ASP.NET objects — request DTOs, JSON buffers, temporary lists — die after one request. Generational GC collects **young garbage often and cheaply**, touching Gen2 rarely.

When Gen2 fills or memory pressure hits, **full collections** pause longer and compact less frequently for LOH. Middle developers tune **Server GC** for web hosts, interpret **% Time in GC**, and avoid promoting long-lived garbage into Gen2 via caches and statics (`1.5.6` roots).

Misattributing latency to "database slow" when **Gen2 GC** spikes after cache warm-up is a common production mistake.

## Core Concepts

### The Generational Hypothesis

Empirical observation: **most objects become unreachable quickly**; older objects tend to stay live. Scanning the entire heap every allocation would waste CPU. .NET splits the **small object heap (SOH)** into **three generations**:

| Generation | Typical contents | Collection frequency |
|------------|------------------|----------------------|
| **Gen0** | New allocations | Very often |
| **Gen1** | Survivors of Gen0 | Less often |
| **Gen2** | Long-lived objects | Rarely; most expensive |

Objects **promote** upward when they survive a collection of their generation.

### Allocation and Survivorship

```
new obj → Gen0
Gen0 GC:
  unreachable → reclaimed
  reachable → promoted to Gen1
Gen1 GC:
  survivors → Gen2
Gen2 GC:
  full collection of SOH (+ often Gen1/0); compact SOH
```

**Promotion** copies or compacts survivors to older generation segments — costs CPU but reduces repeated scanning of long-lived graphs.

### When Each Generation Collects

- **Gen0** — when Gen0 budget exhausted (allocation threshold).
- **Gen1** — after Gen0 collections, Gen1 considered periodically.
- **Gen2** — explicit `GC.Collect(2)`, low memory, or Gen1 pressure.

You cannot rely on exact thresholds — implementation-dependent; measure don't guess.

### Workstation vs Server GC

**Workstation GC** (default for client apps): smaller heaps, may favor lower latency on single-core interactive apps.

**Server GC** (default for ASP.NET Core): **GC heap per logical CPU**, background/concurrent collections optimized for throughput on multi-core servers. Multiple heaps reduce lock contention; total process memory often higher.

```xml
<PropertyGroup>
  <ServerGarbageCollection>true</ServerGarbageCollection>
</PropertyGroup>
```

**Concurrent GC** / **Background GC** — Gen2 parts run concurrently with app (server mode), reducing STW pauses (`1.5.6` mark phases).

### GC Latency Modes

`GCLatencyMode` — `Interactive`, `Batch`, `LowLatency`, `SustainedLowLatency` for short critical sections (games, trading) — trade throughput for pause bounds temporarily.

## Under the Hood

### Ephemeral Segments

Gen0 and Gen1 live in **ephemeral segments** — fast bump allocation. When Gen0 fills, **ephemeral GC** runs; survivors promoted to Gen1/Gen2 segments. Gen2 uses separate segments; **compaction** moves survivors and fixes references (`1.5.6` mark).

### Card Table and Write Barriers

When Gen2 object gets field pointing to Gen0 object, **write barrier** marks **card** in card table. Gen0 GC scans cards in older gens for extra roots — without scanning entire Gen2.

### Finalizers and Generations

First GC puts finalizable objects on queue; may **promote** to higher gen prematurely — another reason to use `Dispose` (`1.5.10`).

### POH (Preview)

**.NET 5+ Pinned Object Heap** — pinned allocations separated to reduce SOH fragmentation — related to `1.5.8` fragmentation story.

## Syntax / Commands / API

```csharp
GC.GetGeneration(obj);      // 0, 1, or 2
GC.CollectionCount(0);        // Gen0 collections so far
GC.GetTotalMemory(forceFullCollection: false);

// Avoid in libraries — forces full GC
GC.Collect(2, GCCollectionMode.Forced, blocking: true);
```

Diagnostics:

```bash
dotnet-counters monitor --counters System.Runtime[gc-heap-size,gen-0-gc-count,time-in-gc]
```

## Examples

### Request-Scoped Objects

ASP.NET Core creates controller, view models per request — Gen0. After response, unreachable → next Gen0 sweep reclaims — ideal pattern.

### Static Cache Promotion

```csharp
static ConcurrentDictionary<int, byte[]> Cache = new();
// Items live for process lifetime → Gen2
// High Gen2 size; Gen2 GC pauses matter
```

Long-lived roots skip generational benefit for those objects — they become **Gen2 residents**.

### Premature Promotion

Large survive rate in Gen0 (e.g., store every request object in static list) fills Gen1/Gen2 with short-lived data that should have died — **promotion undoing generational optimization**.

---

## Common Mistakes & Anti-patterns

**Calling `GC.Collect()` routinely** — disrupts generational tuning; hurts perf.

**Huge Gen2 on "memory leak" without gcroot** — may be legitimate cache; may be leak — diagnose.

**Client GC on server** — throughput loss under parallel load.

**Ignoring allocation rate** — focus only on Gen2; Gen0 churn still costs CPU.

## Production & Real-World Notes

**Kestrel + Server GC** — monitor `gen-0-gc-count` rate vs RPS; spikes correlate with JSON/EF allocations.

**Container memory limits** — OOMKill when heap + native exceeds limit; GC tries compact; may fail → `OutOfMemoryException`.

**.NET 8+ DATAS** — Dynamic Adaptation To Application Sizes — adjusts heap hard limits — know your runtime version behavior.

**Load tests** — warm Gen2 with realistic caches before measuring steady-state GC.

## Comparison / Trade-offs

| Pattern | Gen impact | Notes |
|---------|------------|-------|
| Short-lived locals | Gen0 collect | Ideal |
| Request cache | Gen0–Gen1 | OK if cleared |
| Process-wide static | Gen2 | Survives all GCs |
| LOH objects | Gen2 + LOH rules | `1.5.8` |

## Quick Reference

| Gen | Role | Collection cost |
|-----|------|-----------------|
| 0 | New objects | Lowest |
| 1 | Buffer between 0 and 2 | Medium |
| 2 | Long-lived SOH | Highest (full) |
| LOH | Large objects | Collected with Gen2; rarely compact |

---

## Key Takeaways

- **Generations** exploit short object lifetimes — collect young often.
- **Promotion** moves survivors Gen0 → Gen1 → Gen2.
- **Server GC** suits ASP.NET Core multi-core throughput.
- **Allocation rate** drives Gen0 frequency (`1.5.5`).
- **Long-lived roots** defeat generational benefits — Gen2 pressure.
- **Mark phase** (`1.5.6`) runs per collected generation scope.
