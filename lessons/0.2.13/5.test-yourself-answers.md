# Answers: Tags и semver

## Scoring

## Answers

### 1. [Recall]
**Answer:** Lightweight — ref напрямую на commit. Annotated — ref на **tag object**, который указывает на commit + metadata.
**Why:** `cat-file -t`: lightweight → `commit`, annotated → `tag`.

### 2. [Recall]
**Answer:** `git tag -a <name> -m "<message>"`.
**Why:** `-a` создаёт annotated object.

### 3. [Understanding]
**Answer:** Message, tagger, date, optional GPG — audit trail и release notes; `git show` показывает полный контекст.
**Why:** Lightweight не хранит metadata object.

### 4. [Understanding]
**Answer:** **PATCH** — backward-compatible bug fix. **MINOR** — backward-compatible new functionality.
**Why:** Semver 2.0 rules.

### 5. [Understanding]
**Answer:** Consumers (CI artifacts, npm, Docker) доверяют `v2.0.0` = конкретный hash; смена hash без смены версии ломает traceability и кэши.
**Why:** Tags должны быть immutable на remote.

### 6. [Application]
**Answer:** `git tag -a v2.3.2 -m "v2.3.2: ..."` затем `git push origin v2.3.2`.
**Why:** Annotated + explicit push.

### 7. [Application]
**Answer:** `git push origin v2.3.2` (или имя tag).
**Why:** Tags не всегда push'ятся автоматически с branch push.

### 8. [Application]
**Answer:** **PATCH** (например v2.3.1 → v2.3.2).
**Why:** `fix:` → patch per Conventional Commits mapping.

### 9. [Application]
**Answer:** **MAJOR** (например v2.x → v3.0.0).
**Why:** Breaking change semver rule.

### 10. [Analysis]
**Answer:** **Lightweight** (ref указывает сразу на commit).
**Why:** Annotated top-level type — `tag`.

### 11. [Analysis]
**Answer:** Branch **двигается** с новыми commits; tag **фиксирует** один commit как label (conventionally immutable). Разные цели: dev vs release.
**Why:** refs/heads vs refs/tags semantics.

### 12. [Application]
**Answer:** `git push origin --delete v1.0.0-test` (или `:refs/tags/...`).
**Why:** Удаление ref на remote.

### 13. [Understanding]
**Answer:** **Detached HEAD** at tagged commit.
**Why:** Tag не branch — HEAD указывает на commit напрямую.

### 14. [Application]
**Answer:** `git tag -l "v2.*"`.
**Why:** Pattern listing.

### 15. [Analysis]
**Answer:** Примеры: tag указывает на broken commit; prod/artifacts deploy'ят сломанную версию под «официальным» semver.
**Why:** Tag должен ставиться после verified CI.
