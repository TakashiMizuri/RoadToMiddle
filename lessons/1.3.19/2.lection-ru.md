# Локальные функции vs private methods

> Roadmap: `1.3.19` · Node: `1.3` — C#: продвинутые темы · Depth: понимание

## Learning Objectives

После урока ты сможешь:

- Объяснить, что такое **local function** и где она «живёт» в модели типов.
- Выбирать между **local function** и **private method** по контексту, а не по привычке.
- Использовать **`static` local functions**, чтобы не захватывать внешнее состояние случайно.
- Понимать связь local functions с **generics**, **async/await** и **iterator blocks**.
- Рефакторить длинные методы без раздувания private API класса.

---

## Why This Matters

Классы обрастают helper-методами: `ValidateHeader`, `ParseSegment`, `NormalizeEmail` — и каждый вызывается ровно из одного public-метода. Читатель прыгает по файлу на сотни строк, теряя нить одного workflow. **Local functions** (C# 7.0+) объявляются **внутри** другого member — helper оказывается рядом с кодом, который его использует.

Но local function — не «private method со скидкой». Она может **захватывать** переменные outer scope, быть **`static`**, жить внутри **`async`** и **`iterator`** методов, вызываться **до** текстового объявления в том же outer method. Middle-разработчик видит это в ASP.NET handlers, EF-пайплайнах, validation — и ошибается, когда helper нужен в трёх endpoint'ах, а его спрятали как local function в одном.

Ключ — trade-off: **локальность и capture vs переиспользование и тестируемость**.

## Core Concepts

### Что такое local function?

**Local function** — метод, объявленный **внутри** тела другого метода, конструктора, accessor'а property или другой local function. Это **не** member класса; видимость ограничена enclosing member.

```csharp
public decimal CalculateShipping(Order order)
{
    decimal BaseRate() => order.TotalWeight * 0.5m;

    decimal ApplyDiscount(decimal rate)
    {
        if (order.IsPremiumCustomer)
            return rate * 0.9m;
        return rate;
    }

    var rate = BaseRate();
    return ApplyDiscount(rate);
}
```

`BaseRate` и `ApplyDiscount` недоступны снаружи `CalculateShipping`. Это детали одного сценария, а не API типа — даже private.

По сравнению с **private instance method**: local function не «видит» `this` явно, но non-static local в instance member может использовать поля через capture или outer scope. По сравнению с **lambda** в `Func<>`: local function имеет имя, поддерживает **`ref`/`out`**, при прямом вызове часто **не требует** delegate allocation.

### Local functions vs private methods

**Private methods** — часть типа. Вызываются из любого member класса, попадают в IntelliSense всего класса, их проще выносить в тесты (через public API, `InternalsVisibleTo` или отдельный helper class). Выбор private method, когда логика нужна в **нескольких** местах, **сложная и требует отдельных тестов**, или когда важно **стабильное имя** в stack trace.

**Local functions** — часть одного enclosing member. Когда логика **одноразовая**, **тесно связана с локальными переменными**, или длинный метод нужно разбить без роста списка private methods. Типичный case: action method на 40 строк с тремя шагами validation.

Решение — про **структуру и intent**, не про микроперформанс: JIT часто одинаково inline'ит маленькие private и local functions.

### Static local functions (C# 8+)

Если helper не читает и не пишет captured variables outer scope — пометь **`static`**:

```csharp
public IReadOnlyList<string> SplitTags(string input)
{
    static IEnumerable<string> SplitCore(string text, char separator)
    {
        foreach (var part in text.Split(separator, StringSplitOptions.RemoveEmptyEntries))
            yield return part.Trim();
    }

    return SplitCore(input, ',').Distinct(StringComparer.OrdinalIgnoreCase).ToList();
}
```

**`static`** запрещает случайный capture `input` или `this` — данные идут через parameters явно. Analyzers часто предлагают **`static`**, когда capture не нужен. Если outer variables нужны — либо убери **`static`**, либо передавай значения параметрами (предпочтительнее для нетривиальных helper'ов).

### Capture semantics

Non-static local function использует variables из enclosing scope — parameters, locals, **`ref` locals**. Удобно для incremental parsing:

```csharp
public bool TryParseRange(string text, out int min, out int max)
{
    min = max = 0;
    var span = text.AsSpan();

    bool TryReadInt(ref ReadOnlySpan<char> s, out int value)
    {
        value = 0;
        var i = 0;
        while (i < s.Length && char.IsDigit(s[i]))
        {
            value = value * 10 + (s[i] - '0');
            i++;
        }
        if (i == 0) return false;
        s = s[i..];
        return true;
    }

    if (!TryReadInt(ref span, out min)) return false;
    if (span.Length == 0 || span[0] != '-') return false;
    span = span[1..];
    return TryReadInt(ref span, out max) && span.IsEmpty;
}
```

Private method потребовал бы struct state или много parameters. Capture держит код линейным. Цена: **`TryReadInt` не тестируется изолированно**; over-capture усложняет reasoning.

### Local functions vs lambdas

Оба могут capture. **Local function** — когда нужны **`ref`/`out`**, **`yield return`**, именованная **рекурсия**, **`async`** local. **Lambda** — когда behavior передаётся как **delegate** в LINQ или API с `Func<>`/`Action<>`.

```csharp
int Fibonacci(int n)
{
    int Fib(int k) => k < 2 ? k : Fib(k - 1) + Fib(k - 2);
    return Fib(n);
}

var doubled = numbers.Select(x => x * 2);
```

**`async` local function** внутри **`async` outer** получает свой state machine — «local» не значит «легче lambda».

## Under the Hood

Roslyn lowering превращает local functions в **compiler-generated nested methods** (или generic nested для generic outer) с mangled именами вроде `<CalculateShipping>g__BaseRate|0_0`. Это не IL-local variables, а настоящие methods.

При **capture** компилятор создаёт **closure class** (как у lambda) с полями captured state; local function становится instance method на closure. **`static` local** — без closure, только arguments.

В **stack traces** и APM видны generated names — иногда private method с осмысленным именем читабельнее в production logs.

Local functions можно **вызывать до объявления** в том же outer method — у private methods достаточно порядка в файле класса, forward declaration в C# не нужен.

## Syntax / Commands / API

```csharp
void Outer()
{
    int Add(int a, int b) => a + b;
}

static IEnumerable<int> RangeLocal(int n)
{
    static IEnumerable<int> Range(int count)
    {
        for (int i = 0; i < count; i++)
            yield return i;
    }
    return Range(n);
}
```

Local functions **не** имеют access modifiers и **не** могут быть `virtual`.

## Examples

### Validation в minimal API

```csharp
app.MapPost("/orders", (CreateOrderDto dto, IOrderService svc) =>
{
    static bool IsValidSku(string sku) =>
        sku.Length is >= 3 and <= 20 && sku.All(char.IsLetterOrDigit);

    void Reject(string message) =>
        throw new BadHttpRequestException(message);

    if (string.IsNullOrWhiteSpace(dto.CustomerEmail))
        Reject("Email required.");
    if (!IsValidSku(dto.Sku))
        Reject("Invalid SKU.");

    return svc.CreateAsync(dto);
});
```

`IsValidSku` — **`static`**, pure. Если правило нужно в `MapPut` и import job — вынеси в **`private static`** или validator class.

## Common Mistakes & Anti-patterns

**Все private → local.** Два public method с одной логикой → duplication или private method, не local в одном месте.

**Огромные local functions (50+ строк).** Nesting не уменьшает complexity — extract private method или type.

**Over-capture.** Д десять outer locals → явные parameters.

**`static` «ради скорости» без профиля.** Сначала clarity data flow.

**Рекурсия с лишним capture.** Fibonacci-style → **`static`** local + parameters.

## Production & Real-World Notes

Local functions часты в **`Program.cs`**, filters, сложном **`MapGroup`**. Многие команды лимитируют ~15 строк на local; дальше — private static или `Validator` class.

Business rules (деньги, security) — в injectable services или testable static helpers, не спрятаны в 200-line method.

## Comparison / Trade-offs

| Критерий | Local function | Private method |
|---|---|---|
| Видимость | Один member | Весь класс |
| Reuse | Нет | Да |
| Изолированный unit test | Практически нет | Да |
| Capture | Да (non-static) | Только через params |
| Surface area класса | Не растёт | Растёт |

Для **inheritance** и **override** — только private/protected instance methods.

## Quick Reference

| Задача | Выбор |
|---|---|
| Helper один раз в одном method | Local function |
| 2+ call sites | Private method |
| Pure, без capture | `static` local |
| Delegate в LINQ | Lambda |
| Именованная рекурсия | Local function |
| Отдельный unit test rule | Private static / service |

---

## Key Takeaways

- Local function живёт **внутри member**, не в API класса.
- Local — для **одноразового** workflow; private — для **reuse и тестов**.
- **`static` local** — без closure, явный data flow.
- Capture = та же machinery, что у lambda.
- Local function не заменяет **архитектуру** при росте логики.
