## Answers

### 1. [Recall]
**Answer:** **`101 Switching Protocols`**
**Why:** Handshake completes upgrade; then non-HTTP WebSocket frames.

### 2. [Recall]
**Answer:** **SSE — только server → client**; client commands через отдельные HTTP requests. **WebSocket — bidirectional** на одном channel.
**Why:** SSE is one-way event stream over HTTP.

### 3. [Understanding]
**Answer:** Load растёт **даже без новых данных** — N clients × frequency requests; wasted bandwidth и server CPU.
**Why:** Pull model не event-driven.

### 4. [Understanding]
**Answer:** **WebSocket frames** (text/binary) **full-duplex** до close — не новые HTTP request/response pairs.
**Why:** Same TCP, different protocol after 101.

### 5. [Understanding]
**Answer:** **Keep-alive** through NAT/firewall/proxy idle timeouts; detect dead peers.
**Why:** Silent drop без traffic (`0.3.14` idle).

### 6. [Application]
**Answer:** Scores **one-way** frequent → **SSE** enough. Chat **both ways** low latency → **WebSocket**. Разделение снижает complexity scores channel.
**Why:** Match transport to direction and frequency.

### 7. [Application]
**Answer:** **HTTP + TanStack Query** (`refetchInterval` 30s or manual) — WebSocket overkill.
**Why:** Low frequency, no bidirectional need.

### 8. [Analysis]
**Answer:** REST **cacheable** GET responses at CDN edge; WebSocket **long-lived custom protocol** — CDN не cache WS как static HTTP page; needs dedicated WS routing/sticky.
**Why:** Request-response cache model ≠ duplex stream.

### 9. [Analysis]
**Answer:** **Broadcast между users на разных nodes** — message to user on other instance не дойдёт без **shared backplane**.
**Why:** WS state local to server process.

### 10. [Recall]
**Answer:** **`Upgrade: websocket`**, **`Connection: Upgrade`**, **`Sec-WebSocket-Key`**, **`Sec-WebSocket-Version: 13`**
**Why:** RFC 6455 handshake requirements (any two of first group OK for recall).

### 11. [Understanding]
**Answer:** **Нет** — HTTP/2 still **request-response** per stream; не arbitrary duplex app messaging без upgrade.
**Why:** Different semantics (`0.3.13`).

### 12. [Application]
**Answer:** **Long polling** (or SSE if server→client enough).
**Why:** SignalR fallback transports when WS blocked.

### 13. [Analysis]
**Answer:** **Credentials leak** on wire without TLS; MITM reads first message.
**Why:** Always **`wss://`**; auth at secure handshake.

### 14. [Understanding]
**Answer:** Long polling **держит many HTTP connections open** waiting — similar connection pressure; WS **one conn** but long-lived — оба consume server resources differently.
**Why:** Both tie up connections vs short REST.

### 15. [Recall]
**Answer:** **Pub/sub layer** (Redis) синхронизирует messages между SignalR server instances.
**Why:** Cross-node hub broadcast.
