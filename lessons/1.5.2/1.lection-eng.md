# JIT Compilation: Tier 0, Tier 1, and Tier 2

> Roadmap: `1.5.2` · Node: `1.5` — .NET Runtime and Memory · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Explain **JIT compilation** as the step that translates IL to **native machine code** at runtime.
- Describe **tiered compilation** in modern .NET: **Tier 0**, **Tier 1**, and **Tier 2** (formerly Quick JIT vs optimizing JIT).
- Predict **startup vs steady-state** performance trade-offs from tiered behavior.
- Identify when **ReadyToRun (R2R)** and **Native AOT** bypass or replace tiered JIT.
- Connect JIT activity to **CPU spikes**, **code memory**, and **recompilation** events in production.
- Explain why **first call** to a method can be slower than subsequent calls.

## Why This Matters

In `1.5.1` you traced C# → IL → CLR. IL is not what your CPU runs. The **Just-In-Time (JIT) compiler** inside CoreCLR turns each method's IL into native instructions **when needed** — typically on first invocation. That design enables cross-platform deployment (same IL, different JIT backends) but introduces **latency on cold paths** and **CPU work during compilation**.

ASP.NET Core apps pay a **warm-up tax**: first requests compile controllers, EF queries, JSON serializers. Microservices that scale horizontally see this on every new pod. Middle developers interpret **P99 spikes after deploy**, tune **ReadyToRun**, use **TieredCompilation** settings, and avoid mistaking JIT time for database slowness.

Understanding tiers explains why benchmarks must **warm up** before measuring — Tier 0 code can be dramatically slower than Tier 2.

## Core Concepts

### What JIT Does

When the CLR resolves a method call target for the first time, the **JIT compiler** (`clrjit.dll`):

1. Reads IL and metadata for that method.
2. Applies IL verification rules (unless skipped).
3. Builds an internal representation (trees, SSA in optimizing tiers).
4. Emits **native code** into **executable memory** (RX pages).
5. Patches call sites to jump to native entry point.

The emitted code knows **GC pointers** locations (GC info), **exception handlers**, and **stack frame layout** — linking directly to stack frames in `1.5.4` and GC roots in `1.5.6`.

Subsequent calls jump directly to native code — no re-interpretation of IL in steady state.

### Pre-Tiered vs Tiered Compilation

Historically, .NET had a **single optimizing JIT** — good peak performance, **slow first compile**. **Quick JIT** (Tier 0) added fast, minimal optimization compile for faster startup; later **tier recompilation** promoted hot methods to fully optimized code.

**.NET Core 3+ / .NET 5+** use **tiered compilation** by default:

| Tier | Name (conceptual) | Speed to compile | Quality of code | Typical trigger |
|------|-------------------|------------------|-----------------|-----------------|
| **Tier 0** | Quick / min-opt | Very fast | Lower throughput | First execution |
| **Tier 1** | Default opt (transition) | Medium | Good | Call count threshold |
| **Tier 2** | Full optimization | Slower | Best throughput | Hot methods (OSR / reJIT) |

Exact naming in docs varies; think **fast bootstrap → better → best** as call counts and profiling counters increase.

### The Promotion Story

```
First call ──► Tier 0 native code (fast JIT, few opts)
       │
       │  method called N times (threshold ~30 in many builds; configurable)
       ▼
Background reJIT ──► Tier 1/2 native code (inlining, loop opts, SIMD)
       │
       └── Call sites updated to new entry (transparent to managed code)
```

**On-Stack Replacement (OSR)** allows replacing a long-running loop's code mid-execution when promotion happens — important for hot loops in server jobs without waiting for method exit.

Cold methods stay Tier 0 forever — fine for error paths and one-time startup.

### What the Optimizing JIT Changes

Higher tiers apply:

- **Inlining** — copy callee into caller, removing call overhead.
- **Loop optimizations** — unrolling, strength reduction.
- **Devirtualization** — turn virtual calls into direct calls when type known.
- **Register allocation** — keep locals in CPU registers vs stack slots.
- **Intrinsics** — map `System.Math`, `Vector<T>`, `BitOperations` to CPU instructions.

Tier 0 may leave calls intact, spill more to stack, skip expensive analyses — correct but slower.

### JIT and Memory

JIT consumes:

- **Code heap** — native method bodies (not GC heap, but process memory).
- **Temporary JIT heap** during compilation.

Under load, **reJIT** can cause brief CPU bumps. Tools like **EventPipe** (`Jit/Start`, `Jit/Stop`, `TieredCompilation/BackgroundJIT`) expose this.

Object allocations in **your** code still go to managed heap (`1.5.5`); JIT code itself is unmanaged executable memory until collectible in some dynamic scenarios.

## Under the Hood

### clrjit and the Compiler Pipeline

CoreCLR's JIT (`ClrJit`) implements multiple **compilation tiers** via `CorJitFlag` and tiering API. Rough pipeline for optimizing tier:

```
IL bytes → importer (build IR) → morph → flow graph → optimizations → emit (x64/ARM64)
```

**GC info** records where managed references live at each instruction point — GC can find roots during stack walks. **EH clauses** map IL exception regions to OS unwind data.

**PGO (Profile-Guided Optimization)** in .NET 6+ can use runtime instrumentation to guide inlining and layout — tiering + PGO interact in modern releases.

### Call Counting and Background JIT Thread

The runtime increments **call counters** on method entry (with overhead small enough for production). When threshold crossed, work queues to **background JIT thread** — avoids blocking app threads on long compiles.

`DOTNET_TieredCompilation` (default on), `DOTNET_TC_QuickJitForLoops`, `DOTNET_ReadyToRun` environment variables tune behavior — use deliberately in benchmarks, not random production toggles.

### ReadyToRun and Crossgen2

**ReadyToRun (R2R)** embeds **precompiled native code** in the assembly alongside IL. At runtime, R2R code may run immediately; fallback to JIT if runtime mismatch. **Crossgen2** (NativeAOT tool chain related) precompiles for faster startup in containers.

Trade-off: larger assemblies, less flexibility for tier promotion on R2R bodies (policy evolves per .NET version).

### Native AOT — No JIT at Runtime

**Native AOT** (`PublishAot`) compiles at build time to a single native binary — no clrjit at runtime. Startup and size win; reflection/dynamic limited. Different axis from tiered JIT but answers the same "IL to native" question at build time instead of first call.

## Syntax / Commands / API

| Setting | Effect |
|---------|--------|
| `DOTNET_TieredCompilation=0` | Disable tiering (legacy single-tier behavior approx.) |
| `DOTNET_ReadyToRun=0` | Disable R2R use at runtime |
| `TieredCompilationQuickJit` in `.csproj` | Configure quick JIT |
| `PublishReadyToRun` | Embed R2R in publish output |

```bash
dotnet-trace collect --providers Microsoft-Windows-DotNETRuntime:0x1:4
# JitKeyword for JIT events
```

BenchmarkDotNet **WarmupCount** — essential so measurements hit Tier 2.

## Examples

### First-Call vs Warmed-Call Latency

```csharp
// Cold: first invocation pays JIT + any static ctor
for (int i = 0; i < 1; i++) HotPath();

// Warm: Tier 2 likely after many iterations in real traffic
for (int i = 0; i < 100_000; i++) HotPath();

void HotPath() {
    // enough work to matter in benchmark
    double x = 0;
    for (int j = 0; j < 100; j++) x += Math.Sqrt(j);
}
```

Microbenchmark without warmup measures Tier 0 — misleading for API SLA planning.

### ASP.NET Core Cold Start

New pod after deploy:

1. CLR loads, JITs Main, host builder, Kestrel pipeline.
2. First HTTP request JITs controller, model binders, System.Text.Json helpers.
3. EF first query JITs query compiler paths.

Health checks should **warm critical paths** before marking pod ready — readiness probe design (`11.5.x` later).

## Common Mistakes & Anti-patterns

**Benchmarking without warmup** — reports Tier 0 performance as "production speed."

**Disabling tiered compilation globally** to "fix" startup — hurts steady-state throughput; prefer R2R or AOT for startup-sensitive apps.

**Assuming JIT runs once per method forever** — tier promotion replaces code; rare deoptimization on invalid assumptions (guard failure) in advanced scenarios.

**Huge generic explosion** — each value-type generic instantiation can JIT separate native bodies (`List<int>` vs `List<string>`) — increases memory and compile time.

---

## Production & Real-World Notes

**Horizontal scaling** multiplies JIT cost — each instance warms independently. **Warmup endpoints** or synthetic traffic after deploy reduce user-facing cold latency.

**Container CPU limits** — background JIT competes with request handling; throttled CPU stretches warm-up.

**.NET 8+** continues refining PGO and tiering — stay on LTS, measure after upgrades.

**Server GC** (`1.5.7`) runs concurrently but JIT spikes still show as CPU — use APM CPU flame graphs separating `clrjit!` from app code.

## Comparison / Trade-offs

| Approach | Startup | Peak perf | Flexibility |
|----------|---------|-----------|-------------|
| Tiered JIT (default) | Good | Excellent after warm | Full dynamic features |
| R2R | Better | Good (may not reach Tier 2 opts) | Larger deploy |
| Native AOT | Best | Good (compile-time opts) | Limited reflection/dynamic |
| Interpreter (legacy/debug) | N/A | Poor | Rare in prod CoreCLR |

Default tiered JIT fits most ASP.NET APIs; choose R2R/AOT for edge, functions cold start, or security-hardened single-binary deploys.

## Key Takeaways

- **JIT** bridges IL (`1.5.1`) and CPU execution on first need.
- **Tiered compilation** balances **startup** (Tier 0) and **throughput** (Tier 2).
- **Hot methods** promote via call counting and background reJIT.
- **JIT consumes CPU and code memory** — visible on deploy and scale-out.
- **Benchmarks and probes** must account for warm-up.
- **R2R / Native AOT** are alternatives when JIT at runtime is too costly.
