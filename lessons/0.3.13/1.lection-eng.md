# HTTP/1.1 vs HTTP/2 vs HTTP/3 (QUIC) — Overview

> Roadmap: `0.3.13` · Node: `0.3` — HTTP and the Web · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Summarize how **HTTP/1.1** models connections and why **head-of-line blocking** at the connection level mattered historically.
- Explain **HTTP/2** improvements: binary framing, multiplexing, header compression (HPACK), and server push (legacy concept).
- Describe **HTTP/3** over **QUIC** (UDP) and why it moves multiplexing below the transport layer.
- Recognize which version a client negotiated (browser DevTools, `curl -v`).
- Discuss **trade-offs** when tuning Kestrel, reverse proxies, and CDNs — without implementing protocol stacks.
- Relate version choice to **latency**, **TLS**, and **lossy networks** at a Middle-level mental model.

---

## Why This Matters

You already send requests with curl and see status codes; what you may not see is **which HTTP version** carried them. That version shapes performance under load: a React SPA opening twenty asset requests, an API gateway fan-out, mobile clients on flaky Wi‑Fi. HTTP/1.1 served the web for decades but encourages workarounds (domain sharding, bundling) to avoid connection limits. HTTP/2 fixed many wire inefficiencies but kept TCP's **head-of-line blocking** when packets drop. HTTP/3 reimplements reliability on **QUIC** so one lost packet does not stall all streams.

You rarely choose HTTP/3 in application code — Kestrel, nginx, Cloudflare, and browsers negotiate it. Middle developers still need the overview: why HTTP/2 is default on HTTPS, why debugging looks different, when fallback to HTTP/1.1 happens, and why "just enable HTTP/2" does not replace good API design or caching (`0.3.14` connects here).

## Core Concepts

### HTTP/1.1 — Text, Sequential, Keep-Alive

**HTTP/1.1** (1997, refined in RFC 9112) speaks **plain text** on the wire: request line `GET /path HTTP/1.1`, headers `Key: value`, blank line, optional body. Responses mirror that shape. **`Connection: keep-alive`** (default in modern stacks) reuses one **TCP connection** for multiple requests sequentially — one in flight at a time per connection unless **pipelining** (rare, broken in practice).

Browsers open **several parallel TCP connections** per host (historically ~6) to fetch CSS, JS, and images concurrently. That causes **connection churn** and duplicate TLS handshakes before widespread keep-alive and TLS session resumption.

**Head-of-line (HoL) blocking at HTTP/1.1 level:** if request A is slow, request B on the **same connection** waits behind A even if B would be cheap. Parallel connections mitigate but do not eliminate overhead.

For APIs returning JSON, HTTP/1.1 remains perfectly adequate: few requests, small bodies. Problems show with **many small assets** on one origin — hence bundling in frontend toolchains.

### HTTP/2 — One Connection, Many Streams

**HTTP/2** (RFC 9113) keeps **semantic HTTP** (methods, headers, status codes) but changes **framing**: messages split into binary **frames** (HEADERS, DATA, SETTINGS, etc.) Multiplexed **streams** share one TCP connection; stream IDs interleave so response B can arrive while A is still streaming.

**HPACK** compresses headers with static and dynamic tables — important when cookies and JWT-heavy headers repeat every request. **Server push** (prefetch assets before client asks) saw limited adoption; many CDNs deprecate it in favor of preload hints in HTML.

Negotiation: almost always **TLS ALPN** — client offers `h2`, server agrees during handshake. Cleartext HTTP/2 (`h2c`) exists but is rare outside internal meshes.

HTTP/2 does **not** change REST semantics. Your ASP.NET Core controller is unaware; Kestrel terminates HTTP/2 and presents the same `HttpContext`.

**TCP-level HoL remains:** if a TCP packet is lost, **all streams** on that connection pause until retransmit — HTTP/2 multiplexing does not escape TCP.

```
HTTP/1.1 (one connection, sequential):
  Req A ----------> Resp A ----------> Req B ----------> Resp B

HTTP/2 (one connection, interleaved streams):
  Stream 1: Req A -----> Resp A fragments...
  Stream 3:     Req B -> Resp B fragments...  (parallel on wire)
```

### HTTP/3 and QUIC — UDP-Based Transport

**HTTP/3** (RFC 9114) maps HTTP semantics onto **QUIC** — a transport protocol over **UDP** implemented in user space (typical libraries: quiche, msquic, nginx QUIC module). QUIC provides:

- **Encrypted by default** (TLS 1.3 integrated; no cleartext QUIC in practice).
- **Stream-multiplexing independent of packet loss** — loss on stream A does not block stream B's delivery the same way TCP blocks all byte stream order.
- **Connection migration** — client IP change (Wi‑Fi to LTE) can preserve connection ID without full reconnect (useful mobile; less visible in datacenter API calls).
- **0-RTT** resumption (with replay caveats for non-idempotent requests).

Browsers and CDNs negotiate HTTP/3 via **Alt-Svc** header or HTTPS DNS records; fallback chain often **HTTP/3 → HTTP/2 → HTTP/1.1**.

For typical CRUD API behind nginx to Kestrel, HTTP/3 reduces pain mainly at **edge-to-client** last mile, not inside your VPC where HTTP/2 over stable TCP suffices.

### What Stays the Same Across Versions

Methods, status codes, header names (mostly), URI scheme `https://`, cookies, CORS, and JSON bodies are unchanged. **Version is transport optimization**, not a new API style. Security rules (HTTPS, auth headers) apply identically.

## Under the Hood

**ALPN** during TLS handshake lists protocols client supports (`http/1.1`, `h2`). Server picks highest mutually supported. `curl -v` prints `ALPN, server accepted to use h2`.

HTTP/2 frames carry **priority** and **flow control** windows — misconfigured proxies can buffer entire responses and defeat streaming. **gRPC** requires HTTP/2; REST on Kestrel benefits opportunistically.

QUIC runs over **UDP port 443** (often alongside TCP 443). Firewalls that block UDP break HTTP/3 fallback to TCP-based HTTP/2. **Load balancers** must speak QUIC or pass through to backends on HTTP/1.1 or HTTP/2.

Kestrel enables HTTP/1.1, HTTP/2, and HTTP/3 (where supported) simultaneously on HTTPS endpoints. Internal service-to-service calls may stay HTTP/2 over TCP in mesh sidecars.

## Syntax / Commands / API

**Observe version:**

```bash
curl -sI -o /dev/null -w "%{http_version}\n" https://example.com
curl -v https://cloudflare.com 2>&1 | findstr /i "HTTP/2 ALPN"
```

**Browser:** DevTools → Network → Protocol column (`h2`, `h3`, `http/1.1`).

**Response header (HTTP/3 advertisement):**

```
Alt-Svc: h3=":443"; ma=86400
```

No application C# attribute switches version — hosting and reverse proxy configuration.

## Examples

### Example 1: SPA asset loading

HTTP/1.1: six connections, each JS chunk queued. HTTP/2: one connection, dozens of parallel streams — faster first paint on high-latency links. API calls to `/api/me` still one stream; difference is smaller unless many parallel XHR/fetch.

### Example 2: Mobile lossy network

Packet loss on TCP stalls entire HTTP/2 page load. HTTP/3 QUIC isolates stream loss — video chunk and API JSON may progress independently. Visible in consumer apps; less in office API testing.

### Example 3: Internal microservice

Service mesh sidecar speaks HTTP/2 between pods. External client hits CDN on HTTP/3, CDN to origin HTTP/2 or HTTP/1.1 — **version may change per hop**.

## Common Mistakes & Anti-patterns

**Assuming HTTP/2 fixes slow APIs** — N+1 queries and missing indexes dominate; protocol version does not.

**Disabling HTTP/1.1 without testing legacy clients** — old embedded devices may only speak HTTP/1.1.

**Expecting server push to replace cache headers** — prefer `Cache-Control`, ETag, CDN cache rules.

**Ignoring UDP firewall rules** when enabling HTTP/3 — mysterious fallback only on corporate networks.

**Confusing HTTP/2 with WebSockets** — different upgrade path and use case (`0.3.15`).

## Production & Real-World Notes

Cloudflare, Fastly, AWS CloudFront terminate HTTP/3 at edge. Origin often sees HTTP/1.1 or HTTP/2. **TLS everywhere** is prerequisite for HTTP/2 in browsers.

Monitoring: access logs may log `%HTTP2` or protocol version. Latency SLIs split by version to compare rollout. **Certificate rotation** and ALPN misconfiguration cause sudden HTTP/1.1-only fallback.

HTTP/2 **single connection** increases importance of **`0.3.14` connection pooling** — one bad hung stream ties up connection unless timeouts configured.

## Comparison / Trade-offs

| | HTTP/1.1 | HTTP/2 | HTTP/3 (QUIC) |
|---|----------|--------|---------------|
| Transport | TCP | TCP | UDP (QUIC) |
| Multiplexing | Parallel TCP conns | Many streams / one TCP | Many streams / QUIC |
| HoL on packet loss | Per connection | Whole TCP connection | Per-stream isolation |
| Header compression | None (gzip body only) | HPACK | QPACK |
| Typical deployment | Legacy, simple | Default HTTPS | Edge + modern browsers |
| API JSON workload | Fine | Fine | Marginal gain internally |

**When to care deeply:** high-latency clients, mobile, asset-heavy sites, gRPC. **When overview suffices:** typical BFF + REST behind CDN with sane caching.

## Quick Reference

```
HTTP/1.1  text, keep-alive, ~6 conns/host, sequential per conn
HTTP/2    binary frames, multiplex on ONE TCP, HPACK, ALPN h2
HTTP/3    HTTP over QUIC (UDP), better loss behavior, Alt-Svc

Same: methods, status codes, headers semantics, JSON
Negotiated: TLS ALPN + CDN/browser support
Debug: curl -v, DevTools Protocol column
```

## Key Takeaways

1. **HTTP/1.1** uses text and parallel TCP connections; HoL per connection.
2. **HTTP/2** multiplexes streams on one TCP connection with binary framing and HPACK.
3. **TCP packet loss** still blocks all HTTP/2 streams — motivation for HTTP/3.
4. **HTTP/3** runs over **QUIC/UDP** with per-stream loss isolation and built-in TLS.
5. Application code rarely selects version — **Kestrel, proxy, CDN, browser** negotiate.
6. Faster protocol **does not replace** good caching, pooling, or query design.
7. Observe version with **curl -v** and DevTools when debugging latency.
