## Answers

### 1. [Recall]
**Answer:** SYN → SYN-ACK → ACK.
**Why:** Synchronizes ISNs; only then ESTABLISHED.

### 2. [Recall]
**Answer:** «Получены все bytes до этого номера; следующий ожидаемый byte для send side peer.»
**Why:** Cumulative ACK model.

### 3. [Understanding]
**Answer:** **TCP** reorders segments, буферизует out-of-order, retransmits loss; application read() получает ordered byte stream. IP только доставляет datagrams.
**Why:** Reliability layer above IP.

### 4. [Understanding]
**Answer:** **Flow control** — receiver buffer/app read rate (window to sender). **Congestion control** — sender rate vs network capacity (shared links, loss).
**Why:** Different feedback signals and goals.

### 5. [Understanding]
**Answer:** Avoid repeated **three-way handshake** (+ often TLS) on каждый HTTP request; reuse **same socket/5-tuple** for multiple HTTP messages on one TCP pipe.
**Why:** `0.3.14` performance — transport amortization.

### 6. [Application]
**Answer:** **TLS handshake** (and possibly slow first HTTP byte); connect = TCP ESTABLISHED only.
**Why:** `time_starttransfer` ≈ first response byte.

### 7. [Application]
**Answer:** После TCP, during **TLS** or immediate **RST** from app/firewall; HTTP not started — no status line.
**Why:** Reset during TLS alert or proxy rejection common.

### 8. [Analysis]
**Answer:** **B** неверно — TCP не гарантирует bounded latency.
**Why:** A, C, D — correct TCP behavior.

### 9. [Analysis]
**Answer:** (1) Repeated **three-way handshake** RTT. (2) **Slow start / congestion control** on each new flow.
**Why:** No reuse = pay setup + cautious ramp every request.

### 10. [Understanding]
**Answer:** One side thinks connection live, other crashed/ gone — no FIN/RST. **TCP keepalive** sends probes; no response → close.
**Why:** Detect dead peer without app traffic.

### 11. [Recall]
**Answer:** ~**1 RTT** for TLS 1.3 full handshake (order of magnitude).
**Why:** After TCP, crypto negotiation — `0.3.8`.

### 12. [Application]
**Answer:** **Stops sending** (waits); may send **zero-window probes** when window reopens.
**Why:** Flow control — cannot exceed advertised window.

### 13. [Understanding]
**Answer:** On one TCP connection HTTP/1.1 responses sequential; large/slow first response delays start of second response even if ready at server.
**Why:** Single byte stream ordering — motivation HTTP/2 multiplexing.

### 14. [Application]
**Answer:** **Transport TCP OK** (port open, SYN-ACK). Problem **TLS / application** — certificate validation (`0.3.8`), not TCP reachability.
**Why:** Test-NetConnection is L4 connect only.
