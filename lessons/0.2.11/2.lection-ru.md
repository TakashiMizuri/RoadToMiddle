# git cherry-pick и git revert для hotfix

> Roadmap: `0.2.11` · Узел: `0.2` — Git: ветки и collaboration · Глубина: практика

## Learning Objectives

После этой лекции ты сможешь:

- Объяснить **cherry-pick** как воспроизведение одного commit на другой ветке.
- Описать **hotfix workflow** с ветками, cherry-pick и revert.
- Использовать **`git cherry-pick`** с разрешением конфликтов и опциями `-x` / `--no-commit`.
- Использовать **`git revert`** для отмены плохого release commit на shared ветках.
- Выбирать **cherry-pick vs merge vs revert** при production-инцидентах.
- Восстанавливаться после неудачного cherry-pick через **`git cherry-pick --abort`**.

---

## Why This Matters

Production горит. Критический баг есть на `main`, но fix уже написан три дня назад на длинной feature-ветке — или в прошлом release tag. Ждать merge всей feature нельзя. Переписывать `main` через force push тоже нельзя: на текущем deploy завязаны клиенты и CI.

**Cherry-pick** и **revert** — хирургические инструменты. Cherry-pick: «возьми изменения **этого** commit и примени **здесь**». Revert: «этот commit был ошибкой — добавь **новый** commit, который его отменяет». Оба сохраняют договорённость shared history из `0.1.8`: на ветках, которые уже pull'нули коллеги, правда добавляется **вперёд**, а не стирается прошлое.

Hotfix workflow объединяет дисциплину веток из `0.2.1`, trade-offs merge/rebase из `0.2.4` и безопасный undo из `0.1.8`. Middle-разработчик должен уметь: завести hotfix-ветку, cherry-pick'нуть fix на `main` и LTS, revert'нуть сломанный release, если rollback быстрее forward-fix, и объяснить QA, какие commits реально уехали в prod.

## Core Concepts

### Что делает cherry-pick

Команда `git cherry-pick <commit>` создаёт **новый commit** на текущей ветке. Его **parent** — tip текущей ветки, а **diff дерева** совпадает с тем, что внёс выбранный commit. Оригинальный commit **не перемещается** — Git **переигрывает** patch.

```
feature/login:  ... ─ A ─ B ─ FIX
main:           ... ─ A ─ B
```

После `git checkout main` и `git cherry-pick FIX`:

```
main:  ... ─ A ─ B ─ FIX'   (новый hash, та же логическая правка)
```

`FIX` остаётся на `feature/login`. Дублирование patch'а намеренное: ship fix без merge незавершённой feature.

Cherry-pick использует **three-way merge**, как `git merge` и `git revert`. Base — parent выбранного commit; patch — diff base → picked commit; применение — на текущий HEAD. Конфликты — как в `0.2.5`.

### Cherry-pick vs merge vs revert

**Merge** втягивает **всю** историю ветки. Нужен, когда feature целиком готова на `main`.

**Cherry-pick** копирует **один или несколько commits по hash**. Hotfix с feature-ветки, backport на `release/1.4`, пока `main` уже на `2.0`.

**Revert** создаёт commit, **отменяющий** эффект прошлого commit. Bad deploy на shared `main` без force push.

Правило: нужен **весь** branch → merge. **Один** commit elsewhere → cherry-pick. **Убрать** эффект shipped commit → revert.

### Hotfix branch workflow

```
1. git checkout main && git pull
2. git checkout -b hotfix/SEC-401-null-check
3. Fix + commit (или cherry-pick существующего fix commit)
4. Test, PR, быстрый review
5. Merge в main, tag release (0.2.13)
6. Cherry-pick тот же commit на release/1.x если поддерживается
7. git push origin main release/1.x --tags
```

Если fix уже есть как `abc1234` на feature, шаг 3 — `git cherry-pick abc1234`. QA проверяет тот же patch, что уже прошёл review.

Для **нескольких commits** одного fix:

```bash
git cherry-pick A^..C
```

Перед pick проверь диапазон: `git log --oneline A..C`.

### Revert в hotfix и rollback

Revert дополняет cherry-pick. Cherry-pick **добавляет** хорошее; revert **убирает** плохое.

Commit `bad99` сломал login. Rollback за минуты:

```bash
git checkout main
git pull
git revert bad99 --no-edit
git push origin main
```

Bad commit остаётся в history — postmortem и compliance. Если **позже** меняли те же строки — конфликт revert, resolve + `git revert --continue`. Для **merge commit** — `git revert -m 1` (как в `0.1.8`).

## Under the Hood

Cherry-pick — **последовательный merge** patch'ей. Для каждого commit:

1. Parent picked commit = merge base.
2. Diff parent tree → picked tree.
3. Применение на HEAD через merge strategy.
4. Новый commit: parent = текущий HEAD, новое tree, metadata.

Новый **SHA-1** даже при идентичном tree на другой ветке — identity по hash, не по содержимому.

**Author vs committer:** author и author date сохраняются; committer — ты, commit date — сейчас.

**Duplicate patches:** повторный pick часто даёт empty commit или skip — веди учёт backport'ов.

Во время операции Git хранит `.git/CHERRY_PICK_HEAD`. **`--abort`** сбрасывает состояние.

Revert **не удаляет** reverted commit — строит tree с инверсией patch'а. Commit остаётся предком revert commit'а. `reset` двигает указатель назад; revert — нет.

## Syntax / Commands / API

```bash
git cherry-pick <commit-hash>
git cherry-pick -x <commit-hash>
git cherry-pick -n <commit-hash>
git cherry-pick --continue
git cherry-pick --abort
git revert <bad-commit> --no-edit
git revert -m 1 <merge-commit-hash>
```

## Examples

### Hotfix с feature-ветки

Fix — `f7a2b1c` на `feature/payments`:

```bash
git fetch origin
git checkout main && git pull
git checkout -b hotfix/payment-null-ref
git cherry-pick -x f7a2b1c
git push -u origin hotfix/payment-null-ref
```

Backport на LTS:

```bash
git checkout release/2.2 && git pull
git cherry-pick -x f7a2b1c
git push origin release/2.2
```

### Rollback через revert

```bash
git revert d4e5f6a --no-edit
git push origin main
```

## Common Mistakes & Anti-patterns

Cherry-pick **всей** feature по одному commit — duplicate history и боль при merge. Merge/rebase для целой feature; cherry-pick для изолированных commits.

**Force push на main** вместо revert — ломает команду.

**Забытый LTS backport** — уязвимость на старых версиях.

**Cherry-pick без `-x`** в regulated средах — теряется traceability.

**Revert без тестов** — partial undo при пересекающихся правках.

## Production & Real-World Notes

GitFlow/trunk-based: hotfix от `main` или tag, cherry-pick на maintenance, semver tag. GitHub/GitLab **cherry-pick в UI** для backport PR. **DB migrations:** revert кода без revert миграции — schema drift; согласовывай с `0.2.15` monorepo/polyrepo.

`git cherry -v` сравнивает patch-id, не только hash — видно, применён ли fix уже другим путём.

## Comparison / Trade-offs

| Подход | Когда | Риск |
|--------|-------|------|
| Cherry-pick | Один commit, backport | Дубликаты, конфликты |
| Merge branch | Вся feature | Лишние commits |
| Revert | Shared rollback | Шумный log |
| Reset + force | Solo, unpushed | Запрет на prod |

## Quick Reference

| Команда | Эффект |
|---------|--------|
| `git cherry-pick <hash>` | Новый commit с тем же patch |
| `git cherry-pick -x` | + ссылка на источник в message |
| `git cherry-pick --abort` | Отмена pick |
| `git revert <hash>` | Commit-отмена |
| `git revert -m 1` | Revert merge |

## Key Takeaways

- Cherry-pick **переигрывает** patch как **новый** commit.
- Hotfix: **ветка → pick/fix → merge → backport** на LTS.
- Revert **безопасно откатывает** shared history.
- Оба используют **merge machinery** — жди конфликты.
- **`-x`** для audit; **`--abort`** при ошибке.
- Shared: **revert убрать**, **cherry-pick добавить**.
