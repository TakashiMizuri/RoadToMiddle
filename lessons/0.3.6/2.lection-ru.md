# HTTP Status Codes: Success, Redirection, Client Error, Server Error

> Roadmap: `0.3.6` · Node: `0.3` — HTTP и Web · Depth: **глубоко**

## Learning Objectives

После этой лекции ты сможешь:

- Объяснить, как status codes встраиваются в **response start line** из `0.3.3`.
- Классифицировать коды по **первой цифре** (1xx–5xx) и предсказывать поведение client/server.
- Выбирать правильный код для типичных API-сценариев: **200 vs 201 vs 204**, **401 vs 403**, **404 vs 409 vs 422**.
- Описать **301 vs 304** и когда conditional requests экономят bandwidth.
- Интерпретировать **429** и **503** в production: rate limits, retries, `Retry-After`.
- Различать **500 vs 502 vs 503** при отладке многослойной архитектуры.
- Связать status codes с паттернами ASP.NET Core (`Results`, `ProblemDetails`, middleware).

## Why This Matters

В уроках `0.3.1`–`0.3.5` ты разобрал request-сторону HTTP: client-server модель, anatomy URL, структуру сообщения (start line, headers, body), methods (GET, POST, PUT, …) и матрицу safe/idempotent. Каждый response сервера содержит **трёхзначный status code** в первой строке — например `HTTP/1.1 404 Not Found`. Это число не декоративная метаданная. Это **главный сигнал** клиенту: что произошло и что делать дальше.

Браузеры, mobile SDK, API gateways, caches и retry-библиотеки ветвятся по status codes **до** парсинга JSON. SPA может редиректить на **301**, не перекачивать bundle на **304**, показывать login на **401**, «access denied» на **403**, validation errors на **422**, backoff на **429**, failover на **503**. Если вернуть **200 OK** с `{ "error": true }`, ломается вся экосистема — клиенты не могут использовать стандартную HTTP-семантику, caches сохраняют failures, мониторинг не видит outages.

Middle-разработчик выбирает коды **осознанно**, не случайно. **500** за validation mistake засоряет dashboards. **404** вместо «нет прав видеть ресурс» может leak information (часто нужен **403**). **201** без **Location** — упущенная REST-конвенция. Эта лекция превращает status codes из списка для заучивания в **decision framework**, связанный с HTTP semantics и production behavior.

## Core Concepts

### Status Line: Code + Reason Phrase

HTTP response начинается со **status line**:

```
HTTP/1.1 404 Not Found
```

**Числовой code** — authoritative. **Reason phrase** (`Not Found`) — для людей; формулировка может отличаться. В HTTP/2 и HTTP/3 reason phrase на wire почти не передаётся, но numeric code остаётся контрактом.

Status codes группируются по **сотням**:

| Class | Meaning | Кто «виноват»? | Типичное действие client |
|-------|---------|----------------|--------------------------|
| **1xx** | Informational | — | Ждать финальный response |
| **2xx** | Success | Никто | Использовать body/headers |
| **3xx** | Redirection | — | Следовать `Location` или cache |
| **4xx** | Client error | Client request | Исправить request или auth |
| **5xx** | Server error | Server/gateway | Retry, failover, alert |

Различие **4xx** и **5xx** критично для **observability** и **retry policy**. Client обычно **не** retry тот же 4xx без изменений (кроме **429** с backoff). Многие **5xx** безопасно retry для idempotent GET — связь с `0.3.5`.

### 2xx Success: 200, 201, 204

**200 OK** — default success с representation. GET возвращает resource; PUT/PATCH часто — updated entity. «Request succeeded, вот content.»

**201 Created** — создан **новый resource**, обычно после POST. Best practice: **Location** header с URI нового resource и часто body с entity. Помогает client отличить creation от update.

**204 No Content** — success **без response body**. Типично для DELETE, PUT/PATCH когда representation не нужна, или POST side effects. Не отправляй JSON body с 204 — clients могут игнорировать; нарушается semantic contract.

ASP.NET Core: `Results.Ok()`, `Results.Created(uri, value)`, `Results.NoContent()`.

### 3xx Redirection: 301 и 304

**301 Moved Permanently** — resource получил **новый canonical URL**. Search engines и browsers обновляют bookmarks; caches хранят mapping. Используй при permanent path change. Include **Location**.

**304 Not Modified** — **не** redirect на новый URL. Ответ на conditional GET: «твой cached copy ещё valid». Client прислал validators (`If-None-Match`, `If-Modified-Since`); server сравнивает и при отсутствии изменений возвращает **304 без body**. Связь с caching headers в `0.3.7`.

Не путай 301 (URL moved) с 304 (representation unchanged at same URL).

### 4xx Client Errors: 400, 401, 403, 404, 409, 422, 429

**400 Bad Request** — server не может обработать request из‑за **malformed syntax**: invalid JSON, missing header, wrong Content-Type. Generic «сообщение сломано».

**401 Unauthorized** — authentication **failed или отсутствует**. Client должен получить credentials и retry. Часто `WWW-Authenticate`. Несмотря на имя, RFC означает **unauthenticated**. ASP.NET Core `[Authorize]` без valid credentials → 401.

**403 Forbidden** — client **authenticated**, но **не permitted** для этого action на этом resource. Retry с теми же credentials бесполезен. Пример: User пытается удалить чужой order.

**404 Not Found** — нет resource по URI (или server скрывает existence — см. security note).

**409 Conflict** — request конфликтует с **current server state**: duplicate unique key, optimistic concurrency (ETag mismatch), illegal state transition.

**422 Unprocessable Content** — syntax valid, **semantics fail**: validation, business rules. Популярен с Problem Details и field-level errors. ASP.NET Core часто 400 или 422 — **выбери одно и будь consistent**.

**429 Too Many Requests** — **rate limiting**. Include **Retry-After**. Client должен backoff.

### 5xx Server Errors: 500, 502, 503

**500 Internal Server Error** — unexpected failure **origin application**: unhandled exception, bug. Log с correlation id.

**502 Bad Gateway** — **proxy/gateway** получил **invalid response** от upstream (или none). nginx → crashed Kestrel, TLS между слоями. App может быть fine; edge сообщает upstream failure.

**503 Service Unavailable** — server **временно** не может обработать traffic: maintenance, overload. Часто **Retry-After**. Load balancers drain nodes при deploy.

При incident triage: **502** → gateway/upstream chain; **503** → capacity/maintenance; **500** → application logs.

### Security: 404 vs 403

**404** для resources без доступа — **obfuscation** (не подтверждать existence). **403** — resource exists, access denied. Выбор по threat model. Inconsistent mix путает API consumers.

## Under the Hood

### Где code устанавливается в pipeline

В ASP.NET Core status codes появляются на разных слоях:

```
Request → Middleware → Routing → Filters → Action → Result execution → Response start line
```

Unhandled exception → **500** через exception middleware. `[Authorize]` → **401/403** до action. `return NotFound()` → **404** в result executor. Reverse proxy может переводить upstream failures в **502/504**.

Status code пишется при commit **response start line** — до streaming body. Middleware может менять headers рано; смена status после flush headers — error.

### Caching и 3xx/304

Cache на **301** может хранить redirect mapping long-term. **304** требует conditional headers от client. Wrong **200** с long `Cache-Control` на dynamic API → stale reads. Status codes и headers работают вместе (`0.3.7`).

### Monitoring и SLOs

Production агрегирует по class:

- **2xx rate** — success ratio (осторожно: 200 с `{ "error": true }` отравляет метрику).
- **4xx** — split: auth vs validation vs client mistakes.
- **5xx** — **error budget**; page on-call.

Alert на **500** без учёта **502/503** на edge даёт неполную картину.

## Syntax / Commands / API

### curl

```bash
curl -i https://api.example.com/orders/42
curl -i -X POST https://api.example.com/orders -d '{}' -H "Content-Type: application/json"
curl -i -o NUL -w "%{http_code}\n" https://api.example.com/health
```

### ASP.NET Core

```csharp
app.MapGet("/items/{id}", (int id) =>
    id > 0 ? Results.Ok(new { id }) : Results.NotFound());

app.MapPost("/items", (Item dto) =>
    Results.Created($"/items/{dto.Id}", dto));

return Results.Problem(
    statusCode: StatusCodes.Status422UnprocessableEntity,
    title: "Validation failed");
```

## Examples

### Order API

POST valid cart → **201** + `Location`. GET → **200**. Invalid transition → **409**. Invalid email → **422**. DELETE → **204**. Unknown id → **404**.

### Auth

No token → **401**. Valid token, wrong role → **403** (retry не поможет).

### Rate limit

Burst exceeded → **429**, `Retry-After: 60`.

## Common Mistakes & Anti-patterns

**Всё 200 с `{ success: false }`.** Ломает semantics, caching, monitoring.

**401/403 swapped.** 401 = who are you? 403 = I know you, but no.

**201 без Location.** Client не discovers new URI from headers.

**500 для validation.** Pollutes error budgets.

**404 для every failure.** Скрывает причину.

**Ignore 429 Retry-After.** Ban loops.

## Production & Real-World Notes

Teams публикуют **API error catalogs**. Gateways normalize upstream errors. **GET retry** на 503 common; **POST retry** needs idempotency keys. Health: **200** live, **503** failed readiness — Kubernetes routing.

## Comparison / Trade-offs

| Situation | Weak | Better |
|-----------|------|--------|
| Created resource | 200 | 201 + Location |
| Success, no body | 200 empty | 204 |
| Invalid JSON | 422 | 400 |
| Field validation | 400 generic | 422 + errors |
| Not logged in | 403 | 401 |
| Wrong role | 401 | 403 |
| Duplicate email | 400 | 409 |
| Permanent URL | 302 | 301 |
| Unchanged cache | 200 full | 304 |

## Key Takeaways

- Status codes — **first-class contract** для machine behavior.
- **2xx/3xx/4xx/5xx** — разные client actions и retry policies.
- **201 + Location**, **204** без body, **304** без body — respect semantics.
- **401** authentication, **403** authorization.
- **409** state conflicts, **422** validation.
- **502/503/500** — разные debug paths.
- Align с monitoring, caching, retries.
