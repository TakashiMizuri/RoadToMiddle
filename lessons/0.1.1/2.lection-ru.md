## Why This Matters

Прежде чем говорить о командах Git или папке `.git`, нужно понять задачу, ради которой вообще придумали version control. Представь разработчика с файлом `Report.docx`. Без инструментов единственный способ «подстраховаться» — сохранять копии вручную:

```
Report.docx
Report_backup.docx
Report_final.docx
Report_final_v2.docx
Report_final_REAL.docx
```

Этот паттерн знаком почти каждому — и почти каждый хотя бы раз терял работу из‑за него. Подход ломается не потому, что люди неаккуратны. Нет настоящей **history**: нельзя надёжно ответить, кто, что и когда изменил. **Rollback** к вчерашней версии превращается в угадывание среди похожих имён файлов. Два человека, правящие один документ одновременно, получают тихую перезапись. Когда что‑то падает в production, **accountability** исчезает. А параллельный эксперимент «не ломая stable» — это боль с копиями папок.

**Version Control System (VCS)** — также **source control** или **revision control** — заменяет этот хаос осознанной записью изменений. Система хранит **snapshots** или **deltas** проекта во времени, присваивает каждому состоянию **revision identifier**, позволяет сделать **checkout** любой прошлой revision и поддерживает **merge** вкладов разных людей. На практике это значит: **track history** с metadata (author, timestamp, message), **revert** к любому состоянию, **collaborate** безопасно, **branch & merge** параллельные линии работы и **audit** для compliance и code review.

Каждая профессиональная команда использует VCS. В 2026 году Git — industry standard. Но Git не появился из ниоткуда. Чтобы понять, *почему* Git устроен так — локальные commits до сети, дешёвые branches, remote как удобство, а не единственный владелец history — нужно пройти **три поколения** архитектур VCS. Этот исторический путь — тема этой лекции.

## Core Concepts

### Что такое version control?

В основе version control — система, которая помнит, как эволюционировали файлы. Вместо того чтобы считать «текущую папку на диске» единственной правдой, VCS ведёт структурированную временную шкалу состояний проекта. Каждое сохранённое состояние получает идентификатор; любое из них можно извлечь; когда двое меняют одну кодовую базу, система даёт механизмы объединения, а не правило «последний save победил».

Полезная аналогия: **бесконечный undo stack** с **слойом командной работы** сверху. Undo stack даёт безопасность — ошибки обратимы. Collaboration layer даёт масштаб — десятки или тысячи contributors работают над одним продуктом без рассылки zip-архивов. *Где* хранится этот stack — на одной машине, на central server или на ноутбуке каждого разработчика — и определяет поколение VCS.

Разобравшись с тем, *что* делает version control, можно разложить три поколения на временной шкале. Каждое следующее решало проблемы предыдущего и приносило новые trade-offs.

### Три поколения VCS

Поле эволюционировало тремя волнами; инструменты ранних волн до сих пор живут в legacy-средах.

```
Поколение 1         Поколение 2              Поколение 3
─────────────         ─────────────             ─────────────
  LOCAL               CENTRALIZED               DISTRIBUTED

  [Dev PC]              [Dev A]                   [Dev A] ←→ [Dev B]
  history               ↓  ↓                      full repo  full repo
  на одном              [Central Server]            ↓          ↓
  компьютере            single source               [Remote: GitHub/GitLab]
                        of truth
```

**Поколение 1 (local)** хранит всю history на машине одного разработчика. **Поколение 2 (centralized)** вводит shared server как официального хранителя history; разработчики **check out** работу, правят локально и **commit** обратно на server. **Поколение 3 (distributed)** даёт каждому участнику **полную копию repository** — всю history, все branches, все commits — а «central» server рассматривает как одного из peers, нужного в основном для sync и social convention.

Отсюда важный вопрос: если centralized системы уже дали teamwork, зачем industry двинулся дальше? Ответ — в том, *где* живёт history, кто может работать при падении сети и насколько дорого branch'ить. Разберём каждое поколение по порядку.

## Поколение 1: Local VCS

Ранние инструменты были полностью локальными. Вся version history — на компьютере одного разработчика. Без сети, без shared server, без встроенного способа передать history коллеге. Исторические примеры: **RCS (Revision Control System)** 1980‑х с deltas в companion-файлах `,v` и ранний **SCCS** той же эры.

```
┌─────────────────────────┐
│   Компьютер разработчика│
│  Working files          │
│  + local history DB     │
└─────────────────────────┘
```

Local VCS — реальный шаг вперёд от неструктурированного копирования файлов. Один разработчик получал structured history, быстрые операции и offline по определению. Для solo-экспериментов этого часто достаточно.

Для командной разработки local-only history **неприемлема**. Collaboration по сути отсутствует: передать history — значит вручную копировать папки или ленты backup. Диск машины — **single point of failure**: диск умер — history умерла, если не было отдельных backup. Нет shared visibility для code review, нет central place для лида, нет стандартного workflow параллельной работы.

Local version control встречается и сегодня, но редко как *командный* source control. macOS **Time Machine** и Windows **File History** — backup, не collaboration platform. Zip перед «большим refactor» на профессиональных командах — anti-pattern. Творческие приложения с локальными версиями внутри одной программы полезны одному человеку, но не заменяют multi-developer API и frontend.

Вердикт: поколение 1 показало ценность записанной history, но не поддерживает то, как строят реальные продукты.

## Поколение 2: Centralized VCS (CVCS)

Centralized version control решил проблему sharing через **central server** с **официальной history**. Разработчики **check out** файлы (или часть проекта), правят локально и **commit** изменения обратно. Server **authoritative**: что на server — то команда считает правдой.

Известные centralized системы: **CVS** 1990‑х, **SVN (Subversion)** 2000‑х (ещё в legacy enterprise), **Perforce (Helix Core)** в game dev и workflows с большими binary, **TFVC** в старых Microsoft-стеках.

```
        ┌──────────────┐
        │ Central      │
        │ Server       │  ← единственный source of truth
        └──────┬───────┘
               │
     ┌─────────┼─────────┐
     ↓         ↓         ↓
  [Dev A]   [Dev B]   [Dev C]
  working   working   working
  copy      copy      copy
```

Типичный SVN-style workflow делает зависимость от сети явной:

```bash
svn checkout https://svn.company.com/repo/trunk
# правки локально...
svn commit -m "Fix login bug"   # нужна сеть, server — authority
```

Mental model: локально часто **только последний snapshot**, не полная archaeological record. Глубокие history queries, многие diffs и часть branch-операций — **server round-trips**. Каноническая timeline — база на server.

Centralized VCS принёс реальные плюсы: collaboration через одно место, **access control** на server, centralized visibility для менеджмента. Новичкам «один server, одна правда» иногда кажется проще distributed concepts.

Слабости стали болезненными на масштабе. Central server — **single point of failure**: упал — commits останавливаются, history может стать недоступной. Большинство операций требуют сети — offline ограничен. Без полной local history сложно **bisect** старые баги в самолёте. **Branching** на многих centralized системах тяжёлый; merge conflicts в SVN — часть migration-историй. Весь трафик сходится на одну машину — **scaling bottleneck**.

Конкретный сценарий сбоя:

```
Пятница 17:00 — диск central SVN server умер.
→ Весь weekend без commit.
→ Новые dev не могут checkout.
→ Recovery = качество backup server.
```

Centralized VCS научил industry, что **shared history важна**, но server как **единственный хранитель** history создал хрупкий hub. Поколение 3 переосмыслило, где history должна жить.

## Поколение 3: Distributed VCS (DVCS)

В distributed модели у каждого разработчика **полная копия repository** — вся history, все branches, все commits — на локальной машине. GitHub, GitLab, Bitbucket дают привычный **remote**, но remote — **удобство для sharing**, не единственное место history.

Крупные distributed системы: **Git** (2005, Linus Torvalds), **Mercurial (Hg)**, **Bazaar (Bzr)**.

```
  [Dev A — full repo]  ←—— push/pull ——→  [Remote: origin]
  [Dev B — full repo]  ←—— push/pull ——→  (GitHub/GitLab)
  [Dev C — full repo]  ←—— push/pull ——→
```

Git-style workflow показывает инверсию authority. **Clone** скачивает **полную history**, не только последние файлы. **Commit** локально без сети. **Push** и **pull** — когда готов:

```bash
git clone https://github.com/company/app.git   # полная history локально
git commit -m "Add user profile endpoint"        # без сети
git push origin feature/user-profile            # share когда готов
```

Distributed VCS меняет крутизну learning curve на resilience и flexibility. Каждый clone — backup: server сгорел — ноутбук разработчика может содержать объекты для восстановления. Commit, branch, merge — скорость локального диска. Feature branches, forks, pull requests естественно ложатся на full local repos. Самолёт и плохой Wi‑Fi не блокируют progress — commit локально, push позже.

Цена тоже реальна: staging area, rebase, detached HEAD требуют времени. Misuse — force push на shared branches — больно для команды. Full history занимает диск (помогает shallow clone). Большие binary раздувают каждый clone без Git LFS.

### Почему победил Git среди DVCS

Не каждая distributed система взаимозаменяема. Git выиграл mainstream по нескольким причинам: design под scale Linux kernel, **branches** как лёгкие pointers, ecosystem GitHub (2008+), open source, cross-platform, **content-addressable storage** с integrity by design (см. `0.1.4`).

Для Middle fullstack с ASP.NET Core, React, CI/CD и code review **Git non-negotiable**. Инструменты предполагают Git semantics, даже за GUI.

## Under the Hood (Conceptual)

Implement Git не обязательно — но architectural shift нужно *чувствовать*.

В **centralized** client просит server почти обо всём meaningful. «Дай file X revision 123» — ответ server. Новая версия — server сохраняет. History, diffs, branches — часто **server round-trips**. Каноническая timeline — база server.

В **distributed** client **сам** — полноценная база. Commit в local `.git/`. `git push` шлёт объекты, которых нет на server. `git fetch` получает объекты от server или peer. Remote `origin` — **peer с привычным hostname**, не **единственное место history**.

Аналогия: **Centralized** — библиотека с одним master-каталогом; пожар — хаос до backup. **Distributed** — у каждого полная копия каталога и книг; sync между копиями; одна лаборатория сгорела — остальные сохранили всё.

## Comparison / Trade-offs

Таблица сжимает пройденное в prose. Это карта, не замена narrative — «why» живёт в поколениях выше.

| Feature | Local | Centralized (SVN) | Distributed (Git) |
|---------|-------|-------------------|-------------------|
| Full history locally | ✅ (one machine) | ❌ usually partial | ✅ every clone |
| Team collaboration | ❌ | ✅ | ✅ |
| Works offline | ✅ | ❌ limited | ✅ |
| Single point of failure | Your disk | Central server | Mitigated |
| Branching cost | N/A | High | Very low |
| Industry (2026) | Hobby | Legacy enterprise | **Standard** |

Centralized tooling ещё встречается в regulated средах со SVN audit tooling, в game/CAD с Perforce, в CMS-подобных checkout workflows. Для **ASP.NET Core + React** assume **Git**, если employer не сказал иное.

## Why Git Specifically?

Git distributed, но конкретные design choices важны для daily work — разберём в следующих уроках. **Snapshots**, не только deltas. **SHA-1 content addressing**. **Three-tree model** — working directory, staging area (index), `.git`. Commits — **DAG**, merges first-class.

Понимание поколений VCS объясняет, почему outage remote не блокирует commits. Понимание модели Git — *как* пользоваться этой свободой, не делая history нечитаемой для команды.

## Common Mistakes & Anti-patterns

Частая ошибка — Generation 3 tooling с Generation 2 mindset.

**«Git, но workflow как SVN»** — commits напрямую в `main` без feature branches и pull requests:

```bash
git commit -m "stuff"
git push origin main   # напрямую в production branch
```

**«Git = backup»** — cloud sync не записывает intent, authorship, merge relationships так, как ждут CI и review.

**«Distributed = нет central server»** — remote `origin` **есть**. Distributed: server **не единственная** копия history.

**Document history ≠ code VCS** — Google Docs полезен для prose, но не заменяет branching, merge model и CI для кода.

## Production & Real-World Notes

В production: **Git + GitHub/GitLab + PR** у большинства modern teams; **Git + Azure DevOps** в .NET/Microsoft; self-hosted GitLab on-prem; **SVN → Git migration** в banks, government, legacy. **Monorepo** vs **polyrepo** — оба на Git; trade-off coordination vs isolation.

Даже в DVCS **access control** важен: remote решает, кто push'ит в `main`. Distribution — про **redundancy и workflow**, не про anarchy.

## Quick Reference

```
LOCAL VCS        → history на одной машине
CENTRALIZED VCS  → один server = truth (SVN)
DISTRIBUTED VCS  → каждый clone = full repo (Git)

Git shorthand:
  clone  → полный repo
  commit → snapshot локально
  push   → upload commits
  pull   → download + integrate
```

## Key Takeaways

1. **VCS** — track change, collaboration, rollback.
2. **Local VCS** не для команд.
3. **Centralized** (SVN) — sharing, но **single point of failure**.
4. **Distributed** (Git) — **full repo у каждого**; remote для **sync**.
5. **Git won** — performance, branching, GitHub, open source.
6. Git с SVN mindset — теряешь большую часть пользы.
7. Как fullstack developer, Git — ежедневный инструмент.
