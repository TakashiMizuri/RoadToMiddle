# Nullable Reference Types — #nullable enable

> Roadmap: `1.3.17` · Узел: `1.3` — C#: продвинутые темы · Глубина: **глубоко**

## Learning Objectives

После этого урока ты сможешь:

- Включать **NRT** через `#nullable enable` и `<Nullable>enable</Nullable>`.
- Читать **`string` vs `string?`** — annotations **только compile-time**.
- Использовать **null-state flow analysis**: assign, dereference, narrowing после branches.
- Применять **`!`**, **`[NotNullWhen]`**, **`[MaybeNull]`** осознанно.
- Исправлять **CS8600–CS8629** без слепого подавления.
- Проектировать API с точными null contracts для DTO и services.

---

## Why This Matters

Долго **`string`** означал «maybe null» at runtime, но compile-time притворялся non-null — **`NullReferenceException`** был классикой багов. **Nullable Reference Types** (C# 8+) добавили вторую ось: **`string?`** — may be null, **`string`** — intent «не присвою null» (analyzer проверяет).

Это **не runtime enforcement** как Rust `Option` — **нет изменений CLR**. **Roslyn flow analysis** + metadata (`NullableAttribute`). Команды с NRT режут null bugs в CI и review.

ASP.NET models, EF, JSON deserialization — `required`, `string?`, `[NotNullWhen] TryParse`. Middle dev читает warnings, не спамит `!`.

## Core Concepts

### Включение NRT

```csharp
#nullable enable
```

Project:

```xml
<Nullable>enable</Nullable>
```

Modes: `enable`, `disable`, `warnings`, `annotations`.

### Анnotations

```csharp
string name;      // non-nullable intent
string? nickname; // may be null
```

```csharp
string? s = GetMaybe();
Console.WriteLine(s.Length); // CS8602

if (s is not null)
    Console.WriteLine(s.Length); // OK — narrowed
```

**`string`** без `?` всё ещё может быть null at runtime при обходе analysis — **best-effort static checking**.

### Flow analysis (упрощённо)

- `null` → non-nullable assignment → warning.
- Dereference maybe-null → warning.
- После `== null` throw/return — branch knows non-null.
- `??`, `?.`, `??=` меняют state.
- Attributes на BCL methods guide analysis.

### Null-forgiving `!`

```csharp
string s = GetMaybe()!; // «trust me»
```

Suppress warning **без runtime check**. Abuse скрывает баги.

### Attributes для libraries

| Attribute | Purpose |
|-----------|---------|
| `[NotNullWhen(true)]` | Param non-null when true returned |
| `[MaybeNull]` | Return may be null |
| `[MemberNotNull]` | Field non-null after method |

### Generics

```csharp
class Box<T> where T : class
{
    public T? Item { get; set; }
}
```

`notnull` constraint запрещает null для `T`.

### JSON / EF

CS8618 — non-nullable property без init. Fixes: `required`, `= ""`, honest `string?`, validation.

## Under the Hood

Annotations → **`NullableContextAttribute`**, **`NullableAttribute`** в metadata — **без IL null checks** на обычный dereference.

Roslyn **nullable analysis pass** — interprocedural analysis ограничен.

`!` только suppress — runtime NRE возможен.

Cross-assembly annotations влияют на callers с NRT enabled.

## Examples

```csharp
public void Process(User user)
{
    ArgumentNullException.ThrowIfNull(user);
    // user non-null ниже
}
```

```csharp
public bool TryFind(string id, [NotNullWhen(true)] out User? user)
{
    user = _store.Find(id);
    return user is not null;
}
```

## Common Mistakes

- **`!` everywhere**
- Non-nullable DTO props без `required`
- Думать, что NRT = runtime safety

---

## Key Takeaways

- NRT = **compile-time**; **no CLR enforcement**.
- **`string?`** vs **`string`** — flow warnings.
- **`!`** — suppress, not guard.
- **CS8618** — `required`, defaults, `?`.
- Library attributes для callers.
