# Safe vs Idempotent — краткая выжимка

> `0.3.5` · ~10 мин на повторение

## Главная идея

**Safe** = read-only **intent** (GET, HEAD, OPTIONS) — replay не должен менять business state; важно для cache/prefetch/CSRF. **Idempotent** = один или N одинаковых requests — same **intended effect** (GET, PUT, DELETE, HEAD, OPTIONS) — важно для **retry** после network failure. **POST** не idempotent; retry — через **Idempotency-Key** на application layer.

## Ключевые пункты

- **Safe ⇒ idempotent**, но idempotent **не** implies safe (PUT, DELETE меняют state).
- **GET mutation** — CSRF, crawler, CDN cache bugs.
- **Blind POST retry** — duplicate orders/payments.
- **PATCH** idempotency зависит от semantics (`status=active` vs `count+1`).
- CDN/proxy/LB опираются на properties — broken server semantics = production incidents.
- DELETE twice → 404/204, не «double delete harm».

## Матрица (кратко)

| | Idempotent Yes | Idempotent No |
|--|----------------|---------------|
| **Safe Yes** | GET, HEAD, OPTIONS | — |
| **Safe No** | PUT, DELETE | POST, PATCH* |

## Самые важные примеры

**Bad:** `GET /api/cart/add?sku=1`  
**Good:** `POST /api/cart/items` + body

**Retry safe:** same PUT after timeout  
**Retry unsafe:** same POST without `Idempotency-Key`

## «Под капотом» в одном абзаце

HttpClient не retry POST by default. Polly policies часто retry только idempotent methods. Server хранит `(Idempotency-Key → response)` для dedup. Shared cache по RFC кэширует GET/HEAD, не POST. LB failover retry idempotent PUT converges; POST без key — duplicate rows в БД.

## Частые ошибки

- Путать safe и authenticated/public.
- Retry POST в generic HTTP wrapper.
- Считать PATCH всегда idempotent.
- Side effects на GET «потому что только admin URL».

## Мнемоника

> **Safe = read, Idempotent = repeat** — safe answers «can I prefetch?»; idempotent answers «can I retry?»
