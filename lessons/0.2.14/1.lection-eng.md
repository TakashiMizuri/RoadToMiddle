## Learning Objectives

After this lesson you will be able to:

- Configure **`.gitattributes`** with `text`, `eol=lf`, and `* text=auto`.

- Set sensible defaults for **cross-platform** .NET + React teams.

---

## Why This Matters

Your team ships ASP.NET Core on Linux containers and React from developers on Windows. Without a line-ending policy, `git diff` shows every line changed when only line endings differ ‚Äî code review becomes impossible, merge conflicts multiply, and CI may treat files as modified when content is identical. Some tools break on mixed endings in shell scripts or Docker entrypoints.

**`.gitattributes`** is the repository-level contract: "store text files as LF in Git; checkout appropriately on each OS." It travels with the repo (unlike each developer's local `core.autocrlf` alone), so clones behave consistently. Middle developers diagnose "phantom diffs," fix policy once, renormalize, and prevent recurrence ‚Äî essential for polyglot monorepos (`0.2.15`) and open-source collaboration.

## Core Concepts

### LF, CRLF, and the Problem

**Line ending** characters terminate lines in text files:

- **LF** (`\n`, ASCII 10) ‚Äî Unix, Linux, macOS default, most server tooling.
- **CRLF** (`\r\n`, 13+10) ‚Äî Windows default in many editors.

Git compares **bytes**. Same logical text with different endings is **different content** ‚Äî full-file diff, blame noise, unnecessary conflicts.

Historically Windows checked out CRLF and converted on commit; Linux stayed LF. Mixed team without rules ‚Üí chaos in `Program.cs`, `package.json`, shell scripts.

### .gitattributes Basics

**`.gitattributes`** lives in the repo root (and can exist in subdirs). Git applies rules on **checkout** and **commit** to normalize line endings for matched paths.

Common starter for cross-platform teams:

```gitattributes
# Default: normalize text files, LF in repo
* text=auto eol=lf

# Explicit binary ‚Äî no conversion
*.png binary
*.dll binary
*.woff2 binary

# Shell scripts must stay LF even on Windows checkout
*.sh text eol=lf
```

**`text=auto`** ‚Äî Git detects text vs binary; if text, normalize line endings.

**`eol=lf`** ‚Äî force LF in working tree on checkout (and normalized LF in index/repo for text).

**`binary`** ‚Äî no EOL conversion; byte-for-byte.

Rules are **path patterns** like `.gitignore`; later rules can override earlier.

### How Normalization Works on Commit and Checkout

On **commit** (add to index): Git converts text files to **LF in the object database** (when normalization enabled) ‚Äî canonical repo storage.

On **checkout**: Git converts from stored LF to **`eol=` setting** for working tree (often LF everywhere for server-aligned teams even on Windows).

```
Developer on Windows edits file (CRLF locally possible)
        ‚Üì git add
Index / Git object ‚Üí LF (normalized)
        ‚Üì git checkout (eol=lf)
Working tree ‚Üí LF
```

Without `.gitattributes`, behavior depends heavily on **`core.autocrlf`**:

| Setting | Windows typical effect | Linux typical effect |
|---------|------------------------|----------------------|
| `true` | checkout CRLF, commit LF | no conversion |
| `input` | no checkout conv | commit LF only |
| `false` | no conversion | no conversion |

**Best practice today:** `.gitattributes` with `* text=auto eol=lf` and **`core.autocrlf=false`** (or `input` on Windows if needed) ‚Äî **repo file is source of truth**, not each laptop's global config.

Document in README for Windows devs: EditorConfig `end_of_line = lf` aligns editors.

### Fixing an Existing Mess

After adding or changing `.gitattributes`, already-committed files may still carry old endings until renormalized:

```bash
git add --renormalize .
git status   # may show many files ‚Äî line ending only changes
git commit -m "chore: normalize line endings per .gitattributes"
```

One painful commit cleans history **forward**; avoids perpetual phantom diffs. Coordinate with team ‚Äî merge window, no mixed in-flight branches if possible.

**`.gitattributes` itself** should use LF.

### EditorConfig Synergy

Many repos add `.editorconfig`:

```ini
root = true

[*]
end_of_line = lf
insert_final_newline = true
charset = utf-8
```

EditorConfig affects **editor save**; `.gitattributes` affects **Git**. Both together reduce surprises. `.editorconfig` is not enforced by Git alone ‚Äî `.gitattributes` is.

## Under the Hood

Git's **clean/smudge filters** (internal) implement EOL conversion when `text` is set. On `git add`, the **clean** filter converts working tree text to LF for the blob written to the object database. On checkout, **smudge** may convert LF to configured `eol` for working tree.

The index stores blob hashes **after** clean filter ‚Äî what gets committed. `git diff` compares index to working tree (unstaged) and index to HEAD (staged). Wrong autocrlf ‚Üí "modified" status with empty-looking diff or `^M` in diffs.

**`text=auto`** uses content inspection (heuristic) to guess binary ‚Äî not perfect; mark known binaries explicitly.

**Renormalize** (`--renormalize`): re-run clean filter on all tracked files per current `.gitattributes` without changing logical content ‚Äî updates index blobs to LF-normalized form.

**Merge conflicts** on line endings alone should be rare after normalization; if one branch pre-dates policy, merge may touch whole files ‚Äî fix with renormalize commit on main first.

Windows **warning:** `core.safecrlf` (legacy) warns on irreversible conversions ‚Äî mostly replaced by explicit `.gitattributes` policy.

## Syntax / Commands / API

**.gitattributes patterns:**

```gitattributes
* text=auto eol=lf
*.cs text diff=csharp
*.sln text eol=crlf          # rare: VS solution sometimes CRLF ‚Äî team choice
*.bat text eol=crlf          # Windows batch often CRLF
```

**Commands:**

```bash
git add --renormalize .
git config core.autocrlf false   # local, document for team
git diff --check                 # whitespace errors including CRLF
git ls-files --eol               # inspect index/working tree EOL info
```

## Common Mistakes & Anti-patterns

**Relying only on global `core.autocrlf`** ‚Äî new clone on Linux ignores Windows laptop setting; inconsistent.

**No `binary` for images** ‚Äî Git corrupts or slow-diff binary as text.

**Renormalize without team notice** ‚Äî giant PR, merge hell.

**Mixing `eol=crlf` globally and LF CI** ‚Äî runtime surprises in Docker.

**Forgetting `.gitattributes` in root** ‚Äî subfolder-only rules miss most files.

## Production & Real-World Notes

**CI on Linux** assumes LF; Windows devs without `.gitattributes` commit CRLF ‚Üí pipeline linters fail. **Docker COPY** scripts with CRLF break shebang lines.

Microsoft/dotnet templates increasingly ship `.gitattributes`. **GitHub** displays diffs normalized; still store LF in repo for consistency.

Large renormalize commit: announce freeze, merge quickly. Some teams use **`.git-blame-ignore-revs`** for formatting-only commit (`0.1.7` blame context).

## Comparison / Trade-offs

| Approach | Pros | Cons |
|----------|------|------|
| .gitattributes in repo | Same rules for all clones | One-time renormalize pain |
| core.autocrlf per user | No repo change | Inconsistent across OS |
| Ignore problem | None | Permanent noisy diffs |

**LF everywhere** (including Windows checkout) vs **CRLF on Windows checkout**: industry trend LF everywhere for web/backend; exceptions for `.bat`, legacy VS files.

## Key Takeaways

- LF vs CRLF byte differences cause **full-file false diffs**.
- **`.gitattributes`** is the shared, versioned line-ending policy.
- **`text=auto eol=lf`** is the common default for cross-platform repos.
- Mark **binaries** explicitly; renormalize after policy changes.
- **`core.autocrlf`** supplements but does not replace `.gitattributes`.
- Coordinate one **renormalize commit** when fixing legacy repos.
