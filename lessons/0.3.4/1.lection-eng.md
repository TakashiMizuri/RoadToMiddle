# HTTP Methods: GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS

> Roadmap: `0.3.4` · Node: `0.3` — HTTP and the Web · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- State the purpose of **GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS** in HTTP semantics.
- Map methods to typical **CRUD** and REST operations without treating method as the only design tool.
- Explain why **POST** is the “catch-all” for non-idempotent actions and **GET** must not mutate server state.
- Choose **PUT vs PATCH** for full vs partial resource updates with correct idempotency expectations.
- Use **HEAD** for metadata-only checks and **OPTIONS** for capability discovery (including CORS preflight).
- Configure ASP.NET Core routing and Minimal APIs to accept or restrict methods per endpoint.
- Avoid anti-patterns: GET for delete, POST for everything, ignoring **405 Method Not Allowed**.

---

## Why This Matters

The **method** is the verb in the first line of every HTTP request (`0.3.3`). It tells the server *what kind of operation* the client intends on the **request-target** (URL from `0.3.2`). Two requests to the same URL with different methods can mean entirely different things: `GET /api/orders/42` reads an order; `DELETE /api/orders/42` removes it. Frameworks route on method plus path — `[HttpGet]`, `[HttpPost]`, `MapGet`, `MapDelete`.

Teams that treat every endpoint as POST with `{ "action": "delete" }` in JSON lose HTTP’s built-in semantics, cache behavior, intermediary safety, and clarity for API consumers. Teams that put destructive actions in GET links create **CSRF** vulnerabilities and cache poisoning. Middle developers must choose methods deliberately and understand that **safe** and **idempotent** properties (`0.3.5`) are not academic — they drive retry logic, CDN caching, and browser prefetch behavior.

This lesson covers the core method set you will use daily in ASP.NET Core APIs and see in browser traffic. Method semantics interact with status codes (`0.3.6`), headers like `Allow` and `Location`, and REST design (`0.3.10`).

## Core Concepts

### Method as Intended Semantics, Not Enforcement

HTTP methods express **intent**. The server **may** reject a method with **405 Method Not Allowed** and an `Allow` header listing supported methods. Nothing in HTTP stops a server from deleting data on GET except standards, security, and your code review — but violating semantics breaks caches, proxies, and client assumptions.

Methods are registered with IANA. Custom methods exist (`PURGE` on CDNs) but stick to standard set for public APIs unless you have strong reasons.

### GET — Retrieve a Representation

**Purpose:** Fetch a representation of a resource. **Safe** (no server state change intended) and **idempotent** (`0.3.5`).

Typical use: load product list, download file, search with query parameters. Body in GET requests is **undefined** by spec — avoid sending one; many servers ignore or reject it.

Caching: GET responses cacheable when headers allow (`Cache-Control`, `ETag` in `0.3.7`). Browsers may prefetch GET links.

ASP.NET Core:

```csharp
app.MapGet("/api/products/{id}", (int id) => GetProduct(id));
```

**Never** use GET for operations that change data (logout via GET link, delete user via `<a href>`).

### POST — Process, Create, Non-Idempotent Actions

**Purpose:** Submit data to be **processed**; often **create** a subordinate resource when the server assigns URI. **Not safe**, generally **not idempotent** — repeating POST may create duplicate resources.

Typical use: `POST /api/orders` creates order; `POST /api/search` complex query body; login forms; file upload start. Success often **201 Created** with `Location` header pointing to new resource.

```csharp
app.MapPost("/api/orders", (CreateOrderDto dto) => CreateOrder(dto));
```

POST is the right choice when:

- Server assigns new resource ID/URL.
- Action is a **command** (place order, send email) where duplicates are harmful without idempotency keys.
- Semantics do not fit PUT/PATCH cleanly.

**Idempotency-Key** header (Stripe-style) can make POST safe to retry in production — application layer, not HTTP property.

### PUT — Replace Resource at URI

**Purpose:** Replace the resource at the target URI with the request representation. **Idempotent** — sending the same PUT twice leaves server in same state (same resource content). **Not safe**.

`PUT /api/users/42` with full user JSON replaces user 42. If resource exists, update; if not, some APIs create at that URI (**PUT create** pattern).

Contrast with POST create: `POST /api/users` — server picks id and returns `Location: /api/users/42`.

PUT requires client to send **full representation** (or consistent server-defined full replace). Large partial edits waste bandwidth — use PATCH.

### PATCH — Partial Update

**Purpose:** Apply partial modification to a resource. **Not necessarily idempotent** in all cases (RFC allows either; patch formats like JSON Patch can be idempotent if same patch applied twice). **Not safe**.

`PATCH /api/users/42` with `{ "email": "new@example.com" }` or JSON Patch `[{ "op": "replace", "path": "/email", ... }]`.

ASP.NET Core supports PATCH with `[HttpPatch]`; validate partial DTOs carefully — null vs missing field semantics matter.

Teams debate PUT vs PATCH consistency; document whether PATCH merges or requires patch document format.

### DELETE — Remove Resource

**Purpose:** Delete the resource at URI. **Idempotent** — deleting already deleted resource often returns **204** or **404**; repeated DELETE should not multiply effect. **Not safe**.

`DELETE /api/products/7` removes product 7. Soft-delete in DB still counts as DELETE at HTTP layer if API contract says resource gone (404) or returns deleted representation.

Do not implement DELETE via GET query param `?delete=true`.

### HEAD — GET Without Body

**Purpose:** Identical to GET but response **must not** include a message body. Headers (Content-Length, ETag, Last-Modified) tell metadata. **Safe** and **idempotent**.

Use: check if resource exists, validate cache before download, health of large object. `HEAD /api/reports/export` before `GET` same URL.

Kestrel and controllers support HEAD when GET exists — often automatic mapping.

### OPTIONS — Discover Supported Methods

**Purpose:** Describe communication options for target resource. **Safe**; idempotent in practice.

Returns **204** or **200** with **`Allow`** header: `Allow: GET, POST, OPTIONS`. Used for **CORS preflight** — browser sends `OPTIONS` with `Access-Control-Request-Method` before cross-origin POST with custom headers (`0.3.7`).

ASP.NET Core CORS middleware handles preflight; you may expose OPTIONS explicitly for API discovery.

### Method Summary Table

| Method | Safe | Typical idempotent | Body in request | Common success codes |
|--------|------|-------------------|-----------------|----------------------|
| GET | Yes | Yes | No | 200 |
| HEAD | Yes | Yes | No | 200 |
| POST | No | No* | Often | 201, 200 |
| PUT | No | Yes | Often | 200, 204 |
| PATCH | No | Sometimes | Often | 200 |
| DELETE | No | Yes | Rare | 204, 200 |
| OPTIONS | Yes | Yes | No | 204, 200 |

*POST with Idempotency-Key — application-level retry safety.

## Under the Hood

### Routing and Method Matching

ASP.NET Core endpoint routing matches **HTTP method + route template**. Order of middleware matters: `UseRouting` → `UseEndpoints` / `MapControllers`. Unmatched method on matched path → **405** if configured; otherwise **404**.

`HttpMethodMetadata` on minimal routes restricts verbs. `[AcceptVerbs("GET", "HEAD")]` on controllers.

### GET Caching Stack

Browser cache, CDN, reverse proxy use **method + URL + vary headers**. GET response with `Cache-Control: public` may be served without hitting origin. That is why GET must not change server state — stale cache would serve wrong assumptions.

POST responses rarely cached by default (`Cache-Control: no-store` common).

### CORS Preflight as OPTIONS

Cross-origin request with `Content-Type: application/json` and custom headers triggers browser **OPTIONS** preflight. Server must respond with `Access-Control-Allow-Methods` including actual method. Failure → browser blocks JavaScript from reading response — not the same as network failure.

### Method Override Workarounds (Legacy)

Some clients only support GET/POST. `_method=PUT` in form or `X-HTTP-Method-Override` header — discouraged for new APIs; use correct method over HTTPS JSON APIs.

### Linking Method to Resource State Machine

REST thinking: resources have URIs; methods map to transitions — GET read state, PUT replace state, PATCH merge, DELETE remove, POST create collection member or action. **RPC-style POST** `/api/orders/42/cancel` is valid when modeled as sub-resource or action URI — still POST for non-idempotent cancel side effects.

## Syntax / Commands / API

**curl methods:**

```bash
curl -X GET    https://api.example.com/users/1
curl -X POST   https://api.example.com/users -H "Content-Type: application/json" -d '{"name":"Ann"}'
curl -X PUT    https://api.example.com/users/1 -d '{...full user...}'
curl -X PATCH  https://api.example.com/users/1 -d '{"name":"Bob"}'
curl -X DELETE https://api.example.com/users/1
curl -X HEAD   -I https://api.example.com/users/1
curl -X OPTIONS -i https://api.example.com/users
```

**Controller attributes:**

```csharp
[HttpGet("{id}")]
public IActionResult Get(int id) => Ok(...);

[HttpPost]
public IActionResult Create([FromBody] CreateDto dto) => Created(...);

[HttpDelete("{id}")]
public IActionResult Delete(int id) => NoContent();
```

## Examples

**Example 1 — CRUD users.** `GET /users` list; `GET /users/5` one; `POST /users` create; `PUT /users/5` replace; `PATCH /users/5` partial; `DELETE /users/5` remove. Clear method-per-operation API.

**Example 2 — Wrong GET delete.** `<a href="/admin/deleteUser?id=5">Delete</a>` — crawler or prefetch may trigger; user opens cached page — accidental delete. Fix: POST/DELETE with anti-forgery token.

**Example 3 — HEAD before download.** Client HEAD checks `Content-Length` and `ETag`; if local copy current, skip GET — saves bandwidth.

## Common Mistakes & Anti-patterns

**POST for all mutations** — loses idempotent PUT/DELETE retry and clear Allow lists.

**GET with side effects** — breaks safe assumption; CSRF and cache bugs.

**PUT with partial JSON** without documenting merge rules — clients wipe unspecified fields with null defaults.

**Ignoring 405** — client sends PATCH, server only has PUT; return 405 + Allow, not 404.

**DELETE returning 200 with huge body** — prefer **204 No Content** for simple deletes.

## Production & Real-World Notes

Document method behavior in OpenAPI/Swagger — consumers rely on it. ASP.NET Core generates OpenAPI from attributes.

**Rate limiting** may differ by method — stricter on POST login.

**Audit logs** often record method + path + user — DELETE and PATCH need strong authZ.

GraphQL uses POST for all queries — different paradigm; single HTTP method by design (`0.3.10` contrast).

---

## Comparison / Trade-offs

**PUT vs POST create** — PUT when client controls URI (`/users/42`); POST when server assigns id.

**PATCH vs PUT** — PATCH bandwidth and partial edits; PUT simpler mental model full replace.

**DELETE vs soft-delete** — HTTP DELETE still valid; body optional with deleted entity JSON for audit APIs.

**RPC POST vs REST verbs** — `POST /orders/42/cancel` acceptable for actions; do not overload GET.

## Key Takeaways

- Method is first component of request start line — expresses **intent** on URL.
- **GET/HEAD** safe; **GET/PUT/DELETE/HEAD/OPTIONS** idempotent by standard semantics.
- **POST** creates and commands; duplicates possible without idempotency keys.
- **PUT** full replace idempotent; **PATCH** partial update.
- **OPTIONS** + **Allow** for discovery and CORS preflight.
- Violating semantics breaks caches, security, and retry logic — choose methods deliberately.
- ASP.NET Core routes by method + path; return **405** when method unsupported.
