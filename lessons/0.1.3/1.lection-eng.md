## Core Concepts

### Two Ways to Get a Repository

Every Git story begins with a repository on disk â€” a working tree plus a `.git` directory. You arrive there in one of two ways.

**`git init`** starts a **new** project locally. Git creates an empty repository skeleton inside `.git/` in the current folder (or a path you specify). No remote exists yet; no history exists yet. All project files begin life as **untracked** until you deliberately `git add` them. This path suits greenfield work: a new ASP.NET minimal API, a React app scaffold, a learning exercise.

**`git clone`** copies an **existing** repository from a URL â€” typically GitHub, GitLab, or Azure DevOps. Clone downloads **full history**, checks out the default branch into your working tree, and configures a remote named `origin` pointing back at the source. You use clone when joining a team, contributing to open source, or continuing work on another machine.

Both commands give you the same structural result: editable files plus `.git`. Clone additionally wires you to upstream from day one. Having chosen your entry point, the daily rhythm is identical.

### The Daily Workflow

Professional Git use is not a bag of isolated tricks; it is a repeating cycle that mirrors the three-tree model from `0.1.2`. You edit files in the working directory. You inspect state with `status` and compare content with `diff`. You stage intentionally with `add`. You review staged content with `diff --staged`. You record history with `commit`. You orient yourself in time with `log`. Remote sync comes later; this loop works fully offline.

```mermaid
flowchart LR
    A[Edit files] --> B[git status]
    B --> C[git diff]
    C --> D[git add]
    D --> E[git diff --staged]
    E --> F[git commit]
    F --> G[git log]
```

The diagram is deliberately linear. Real sessions loop back â€” after `log` you edit again â€” but the order inside a single commit matters: never commit blindly before looking. The sections below walk through each command with enough detail to use them confidently on a real repository.

## Under the Hood (Brief)

When you run these commands, Git is not magic â€” it is maintaining the three trees you already know.

**`git init`** creates the `.git/` skeleton: directories for objects and refs, a `HEAD` pointer, and configuration. **`git clone`** fetches all objects from the remote, writes them into a new `.git`, checks out the default branch into the working tree, and records `origin`. **`git add`** reads file content from the working tree, hashes it into blob objects as needed, and updates `.git/index` so the staging area reflects your choices. **`git commit`** builds a tree object from the index, wraps it in a commit object with metadata and parent pointer, and moves the current branch ref forward. **`git log`** walks the commit graph backward from `HEAD`. **`git diff`** compares blob content between whichever two trees you specify â€” working vs index, index vs `HEAD`, or commit vs commit.

The on-disk layout of objects and refs is the subject of **`0.1.4`**. For now, let the commands be levers on a model you already understand conceptually.

## Syntax / Commands / API

### `git init`

`git init` creates a new repository. By default it initializes the current directory; you can pass a path to create a subdirectory instead. Modern Git lets you name the initial branch explicitly â€” `main` is conventional today:

```bash
mkdir my-app && cd my-app
git init
# Creates .git/ â€” repository is born

git init --initial-branch=main   # default branch name (modern Git)
git init -b main                 # short form
```

After initialization, every file in the folder is **untracked**. Git sees them in `git status` but does not version them until `git add`. That distinction trips people who run `git commit` immediately and see "nothing to commit" while new files still show as untracked.

### `git clone`

`git clone` downloads a repository and its full history, then checks out files for you to work on:

```bash
git clone https://github.com/user/repo.git
# Creates ./repo/ with .git and checked-out files

git clone https://github.com/user/repo.git my-folder
# Clone into custom directory name

git clone --depth 1 https://github.com/user/huge-repo.git
# Shallow clone â€” only latest commit (advanced; not default)
```

Clone implicitly runs the equivalent of `git remote add origin <url>`. You can commit locally immediately; pushing requires credentials and permission on the remote â€” topics for later lessons. Shallow clone (`--depth 1`) trades history depth for speed and disk; use it when you understand what you are giving up.

### `git status`

`git status` is your dashboard. It answers: which branch am I on, what is staged, what is modified but unstaged, what is untracked? Learn to read it before every commit.

```bash
git status              # long format â€” verbose, beginner-friendly
git status -s           # short format
git status --short      # same as -s
git status --porcelain  # machine-readable; used by scripts and tools
```

The **long format** groups files into sections whose names map directly to the three-tree model. "On branch `main`" tells you where `HEAD` points. "Changes to be committed" lists **staged** files â€” what `git commit` would record now. "Changes not staged for commit" lists **modified** tracked files whose working tree differs from the index. "Untracked files" lists paths Git is not yet tracking.

The **short format** compresses each path to one line with a two-column code. A space in the first column and `M` in the second (` M`) means modified in the working tree but not staged. `M` in the first column (`M `) means staged. `MM` means both â€” staged content plus newer unstaged edits on the same file. `??` marks untracked paths. Reading short status quickly becomes second nature in busy repos.

Example:

```
$ git status -s
 M Program.cs
M  README.md
?? notes.txt
```

Here `Program.cs` has unstaged edits, `README.md` is staged, and `notes.txt` is new and untracked.

### `git add`

`git add` moves changes from the working directory into the staging area. Scope matters: adding a single file, a directory, or "everything" are different decisions with different consequences.

```bash
git add Program.cs           # single file
git add src/                 # entire directory (recursive)
git add .                    # current directory and below (not parent)
git add -A                   # all changes in entire repo (from any cwd)
git add --all                # same as -A
git add -u                   # only tracked files (updates/deletes, no new untracked)
git add -p                   # patch mode â€” stage hunks interactively
git add -i                   # interactive mode (menu)
```

The difference between `.` and `-A` bites people who run `git add .` from a subdirectory and wonder why sibling folders were not staged. Dot adds from the current directory downward; `-A` stages new, modified, and deleted paths across the **whole repository** regardless of where you stand. Flag `-u` updates already tracked files without picking up new untracked ones â€” useful when you want to commit deletions and modifications but not accidental new files.

**Patch mode (`-p`)** splits a file's changes into **hunks** and asks what to stage. You might accept a bugfix hunk with `y`, skip a debug print with `n`, split a large hunk with `s`, or edit a hunk with `e`. Patch mode is how you keep unrelated edits in the same file out of the same commit â€” essential for atomic history.

### `git commit`

`git commit` records the staged snapshot into repository history. Unstaged changes stay out; untracked files stay out unless you added them first.

```bash
git commit -m "Add user registration endpoint"
# -m = message on command line (required for non-interactive)

git commit -a -m "Fix validation typo"
# -a = auto-stage all TRACKED modified/deleted files (NOT new untracked files)

git commit                       # opens editor for message (vim/nano/etc.)
git commit --amend               # replace last commit (advanced â€” later topic)
```

The `-a` flag is convenient for small fixes on files Git already tracks, but it **does not** add new files. That limitation causes recurring confusion: you create `NewService.cs`, run `commit -a`, and the new file remains untracked.

Commit messages are communication, not bureaucracy. The subject line should summarize in imperative mood â€” "Add feature," not "Added feature" â€” because Git itself generates messages that way ("Merge branch..."). Keep the subject roughly within fifty characters when possible. Use the body to explain **why**, not to list filenames. Reference issues when your team expects it.

Good:

```
Fix null reference when order has no items

Orders without line items returned 500. Return empty array instead.
Fixes #128
```

Bad messages like `fix`, `updated stuff`, `WIP`, or `asdfasdf` destroy the value of `git log` when you bisect a regression six months later.

### `git log`

`git log` walks history backward from `HEAD`, showing commits newest first by default. It is how you answer "what changed recently" and "when did this line appear."

```bash
git log                        # full log with hash, author, date, message
git log -n 5                     # last 5 commits (--max-count)
git log --oneline                # one line per commit
git log --graph                  # ASCII graph of branches/merges
git log --decorate               # show branch/tag names
git log --oneline --graph --decorate --all   # popular combo

git log -p                       # show patch (diff) for each commit
git log -p -2                    # patch for last 2 commits
git log --stat                   # file change stats per commit

git log -- README.md             # history of one file
git log -S "functionName"        # pickaxe â€” commits that changed string count
```

For daily orientation, `--oneline --graph --decorate` on recent history is enough. For archaeology, `-p` shows the actual diff each commit introduced. Filtering by path (`-- README.md`) or pickaxe search (`-S`) narrows noise in large repos.

Example output:

```
* a1b2c3d (HEAD -> main) Add JWT validation
* d4e5f6g Initial commit
```

### `git diff`

`git diff` compares content between Git's trees. Choosing the wrong variant is one of the most common beginner mistakes, so learn the mapping by heart.

```bash
git diff                         # working tree vs staging area (unstaged changes)
git diff --staged                # staging area vs last commit (staged changes)
git diff --cached                # same as --staged (legacy name)

git diff HEAD                    # working tree vs last commit (staged + unstaged)
git diff HEAD -- Program.cs      # same, one file

git diff abc123 def456           # between two commits
git diff abc123..def456          # equivalent range syntax
git diff abc123 HEAD             # commit to current
```

When you edit but have not staged, `git diff` shows what would change if you staged now. After `git add`, `git diff --staged` shows what the next commit will contain â€” your pre-flight check. `git diff HEAD` collapses both against the last commit when you want the full picture.

Reading unified diff output:

```diff
--- a/Program.cs
+++ b/Program.cs
@@ -1,3 +1,4 @@
 unchanged line
-removed line
+added line
 another unchanged line
```

Lines prefixed with `-` were removed; lines with `+` were added. The `@@` hunk header gives line context. Color terminals highlight red and green; the meaning is the same in plain text.

## Examples

### Example 1: Brand-new project

Starting from zero exercises the full path from untracked to committed:

```bash
mkdir todo-api && cd todo-api
git init -b main

echo "# Todo API" > README.md
git status                    # README.md untracked

git add README.md
git commit -m "Initial commit: add README"
git log --oneline
# abc1234 Initial commit: add README
```

Notice the pause at `status`: untracked is not staged. Notice `add` before `commit`. Notice `log` confirming what history now contains.

### Example 2: Clone and first change

Joining an existing project reverses the origin story but the local loop is identical:

```bash
git clone https://github.com/example/shop-api.git
cd shop-api
git status                    # clean

# edit src/Orders/OrderController.cs
git status                    # modified, unstaged
git diff                      # see unstaged changes

git add src/Orders/OrderController.cs
git diff --staged             # review before commit

git commit -m "Return 404 when order not found"
git log --oneline -3
```

The discipline of `diff` before add and `diff --staged` before commit is what keeps debug prints and secrets out of shared history.

### Example 3: Partial staging with `-p`

When one file contains two logical changes, patch mode preserves atomic commits:

```bash
# Program.cs has two unrelated edits in one file
git add -p Program.cs
# y for bugfix hunk, n for debug print hunk

git diff --staged             # only bugfix hunk
git commit -m "Fix divide-by-zero in calculator"
```

The debug print remains unstaged for a separate commit or deletion.

### Example 4: Comparing commits

History is not only forward motion; you compare past states to understand regressions:

```bash
git log --oneline
# f1a2b3c Add feature
# d4e5f6a Initial commit

git diff d4e5f6a f1a2b3c       # everything that changed between commits
git diff d4e5f6a..HEAD         # from initial to now
```

Use commit hashes from `log --oneline`; range syntax `A..B` means "changes reachable from B since A."

## Common Mistakes & Anti-patterns

Running **`git commit` without `git add` on new files** leaves them untracked. If nothing is staged, Git reports nothing to commit while new files still lurk in status. Always add new paths explicitly.

Assuming **`git commit -a` picks up new files** repeats the same failure. `-a` auto-stages modifications and deletions on **tracked** files only.

Running **`git add .` from the wrong directory** stages only the subtree below your current path. When in doubt, run `status` from the repository root or use `git add -A`.

Writing **meaningless commit messages** makes `git log`, `git bisect`, and code review painful. The message is part of the artifact; treat it that way.

Skipping **`git diff --staged`** before commit is how debug code and credentials enter history. The staged diff is your last line of defense before `.gitignore` lessons in `0.1.6`.

Confusing **`git diff` with `git diff --staged`**: unstaged edits vs what will be committed are different questions. Ask which you need before typing.

## Production & Real-World Notes

IDEs and CI scripts often parse **`git status --porcelain`** and **`git diff --name-only`** instead of human-oriented output because porcelain formats stay stable for machines. You should still read long status fluently; tools hide nothing you cannot see in the terminal.

Many teams adopt **Conventional Commits** prefixes â€” `feat:`, `fix:`, `docs:`, `chore:` â€” for example `fix(orders): return 404 when not found`. That convention powers changelogs and semantic release tooling. It is optional but widespread.

Companies may require **GPG-signed commits**, **commit-msg hooks** that lint message format, or templates in `.gitmessage`. Learn the basics here; polish follows team policy.

Never commit **secrets** â€” `.env`, API keys, connection strings. `.gitignore` helps; **`git diff --staged`** confirms what you are about to immortalize.

## Comparison / Trade-offs

There is no single "best" invocation for every situation â€” only habits matched to intent. **`git add` with specific paths** is the default for careful work. **`git add .`** is fine when you have reviewed everything in scope. **`git add -A`** stages repo-wide from any subdirectory. **`git add -p`** shines when one file mixes concerns. **`git commit -a -m`** suits tiny tracked-file fixes where staging separately adds no value. **`git log --oneline -10`** serves daily recall; full `log -p` serves deep investigation. Choosing consciously beats muscle memory that always runs `add .`.
