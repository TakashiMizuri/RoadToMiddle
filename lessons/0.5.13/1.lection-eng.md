# SEO: title, meta description, canonical, Open Graph

> Roadmap: `0.5.13` · Node: `0.5` — HTML · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Explain how search engines and social platforms **discover and interpret** HTML metadata.
- Write an effective `<title>` and `<meta name="description">` for a page.
- Use `<link rel="canonical">` to resolve duplicate URL problems.
- Add **Open Graph** tags so shared links render rich previews.
- Distinguish what metadata affects **ranking**, **snippets**, and **social cards** — and what does not.

---

## Why This Matters

You can build a flawless React SPA with perfect accessibility and still receive almost no organic traffic if search engines cannot understand what each URL represents. SEO metadata lives in the `<head>` of your HTML document — the same region where charset, viewport, and stylesheet links sit. It is the **first structured signal** crawlers read before they parse body content or execute JavaScript.

For a fullstack developer, SEO is not a marketing-only concern. When you ship a product page, a blog post, or a documentation article, you decide what appears in Google results and in Slack/Twitter/LinkedIn link previews. Duplicate URLs from query strings, trailing slashes, or `www` vs non-`www` variants can dilute ranking unless you declare a **canonical** source. Open Graph tags control whether a shared link shows a compelling image and title or a bare URL string.

In `0.5.11` and `0.5.12` you focused on humans using assistive technology and keyboard navigation. SEO metadata complements that work: a clear `<title>` helps screen reader users identify tabs, and descriptive meta content helps everyone scanning search results. Later, when you build React apps, you will set these tags per route — often via libraries like React Helmet or framework metadata APIs — but the **HTML contract** you learn here remains the same.

## Core Concepts

### How Crawlers Read Your Page

When a search engine bot requests your URL, it receives HTML and parses it into a DOM tree (`0.5.10`). The `<head>` section is processed early because it declares document-wide facts: language, encoding, title, linked resources, and machine-readable hints. Modern crawlers also execute JavaScript on many sites, but **critical metadata should be present in the initial HTML response** whenever possible. Relying solely on client-side injection delays or prevents indexing on slower crawlers and on social preview fetchers that do not run your full app bundle.

Think of `<head>` metadata as the **label on a shipping box**: the warehouse (search index) sorts packages by label before opening them. A missing or misleading label does not always stop delivery, but it reduces priority and clarity.

### The `<title>` Element

Every HTML document should have exactly one `<title>` inside `<head>`. The title appears in three prominent places: the **browser tab**, the **search result headline** (usually — Google may rewrite it), and **bookmarks**. It is one of the strongest on-page signals for relevance, though not the only ranking factor.

A good title is **specific, readable, and scoped to the page**. For a product detail page, `"Wireless Headphones — Acme Shop"` beats `"Home"` or `"Acme Shop — Best Electronics Online Store Free Shipping Deals"`. The latter is keyword-stuffed and truncated awkwardly in SERPs (Search Engine Result Pages). A common pattern is **Primary topic — Brand** or **Page name | Site name**, kept roughly under 60 characters so truncation is rare on desktop (mobile SERPs are narrower).

```html
<head>
  <meta charset="UTF-8">
  <title>PostgreSQL Index Guide — DevDocs</title>
</head>
```

Each route in a multi-page app or SSR framework should emit a **unique title**. Duplicate titles across dozens of URLs confuse crawlers and users alike.

### Meta Description

The `<meta name="description" content="...">` element provides a **summary** of the page. Google often uses it as the snippet text below the blue link in results — but Google may also generate its own snippet from body content if it judges the meta description irrelevant or duplicate. Other engines behave similarly: the meta description is a **hint**, not a guarantee.

Write descriptions for **humans**, not keyword lists. One or two sentences (~150–160 characters is a practical target) that state what the page offers and why someone should click. Avoid empty descriptions, identical descriptions on every page, and misleading clickbait that does not match content — that erodes trust and increases bounce rate, which indirectly hurts performance signals.

```html
<meta name="description"
      content="Learn when to use B-tree, GIN, and partial indexes in PostgreSQL, with EXPLAIN examples and common mistakes.">
```

Meta description is widely believed to have **little or no direct ranking weight** today; its primary value is **click-through rate** from search results. Still, CTR influences how search systems evaluate satisfaction over time.

### Canonical URLs

Duplicate content is a structural problem on the web. The same article might be reachable at:

- `https://example.com/blog/post`
- `https://example.com/blog/post/`
- `https://www.example.com/blog/post`
- `https://example.com/blog/post?utm_source=twitter`

To crawlers, these can look like separate pages with identical body text. **Link equity** (ranking signals from inbound links) splits across copies instead of consolidating on one authoritative URL.

The `<link rel="canonical" href="...">` element declares **which URL is the preferred version** of the current page. It goes in `<head>` and should be an **absolute URL** (including `https://`) pointing to the definitive location.

```html
<link rel="canonical" href="https://example.com/blog/post">
```

If you are on a tracking URL variant, the canonical still points to the clean URL. Search engines treat canonical as a **strong hint**; they may ignore it if implemented incorrectly (canonical pointing to a 404, or chains of conflicting canonicals). For paginated lists, each page typically canonicalizes to itself, not to page 1 — unless you intentionally merge series (context-dependent).

Canonical complements **HTTP redirects** (`301`) but does not replace them. Redirect when you permanently move a URL; use canonical when duplicates remain accessible but should not be indexed separately.

### Open Graph and Social Previews

When someone pastes your link into Slack, Discord, Facebook, or LinkedIn, the platform fetches your HTML and looks for **Open Graph** (`og:`) meta tags — a convention originally from Facebook, now widely supported. Twitter/X historically used its own `twitter:` tags but falls back to Open Graph when Twitter Card tags are absent.

Core Open Graph properties:

- `og:title` — preview title (can match `<title>` or be shorter/punchier for social).
- `og:description` — preview text.
- `og:image` — absolute URL to an image (often 1200×630 px for safe cropping).
- `og:url` — canonical URL of the content.
- `og:type` — e.g. `website`, `article`.

```html
<meta property="og:title" content="PostgreSQL Index Guide">
<meta property="og:description"
      content="B-tree vs GIN, partial indexes, and EXPLAIN — practical guide for backend developers.">
<meta property="og:image" content="https://example.com/assets/og/postgres-indexes.png">
<meta property="og:url" content="https://example.com/blog/postgres-indexes">
<meta property="og:type" content="article">
```

Note `property="og:..."` rather than `name="..."` — Open Graph uses the `property` attribute per the RDFa convention. Mixing `name` on `og:` tags reduces compatibility with some parsers.

Social crawlers often **do not execute JavaScript**. If your React app injects OG tags only after hydration, previews may show generic site defaults. SSR, static generation, or pre-rendering for public pages solves this — a production pattern you will revisit in the React phase.

### Robots and Indexing (Brief Context)

Related but distinct: `<meta name="robots" content="noindex">` tells crawlers **not to index** this page. Canonical says "index **that** URL instead of this one." Sitemap XML lists URLs you **want discovered**. These tools work together in site architecture; this lesson focuses on title, description, canonical, and OG as the everyday fullstack toolkit.

## Under the Hood

When Googlebot fetches a document, it builds a DOM and extracts candidate metadata into an internal representation. Title and meta description feed **snippet generation** heuristics — algorithms that may rewrite titles for query relevance. Canonical tags are cross-checked against sitemap entries, internal links, and redirect chains; inconsistent signals trigger **canonical clustering** where Google picks a URL you did not intend.

Open Graph fetchers issue a normal HTTP GET with a bot user-agent, parse `<head>` only in many cases, and cache preview metadata. Image URLs must be **publicly reachable** (no auth cookie required) and return correct `Content-Type`. Relative `og:image` paths break on many platforms; always use absolute URLs.

For SPAs, **Google's two-wave indexing** (HTML first, JavaScript later) means metadata in the initial response is indexed sooner. Social bots may never run the second wave. That asymmetry is why SSR metadata is a production requirement for marketing pages, not an optimization luxury.

## Syntax / Commands / API

| Element / attribute | Purpose |
|---|---|
| `<title>` | Document title; tab label, primary SERP headline candidate |
| `<meta name="description" content="...">` | Snippet hint for search results |
| `<link rel="canonical" href="absolute-url">` | Preferred URL among duplicates |
| `<meta property="og:title" content="...">` | Social preview title |
| `<meta property="og:description" content="...">` | Social preview description |
| `<meta property="og:image" content="absolute-url">` | Social preview image |
| `<meta property="og:url" content="...">` | Canonical social URL |

Minimal `<head>` SEO block:

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>About Us — Acme Corp</title>
  <meta name="description"
        content="Acme Corp builds developer tools since 2010. Meet the team and our open-source mission.">
  <link rel="canonical" href="https://acme.example/about">
  <meta property="og:title" content="About Acme Corp">
  <meta property="og:description"
        content="Meet the team behind developer tools used by 50,000 engineers.">
  <meta property="og:url" content="https://acme.example/about">
  <meta property="og:type" content="website">
  <meta property="og:image" content="https://acme.example/og/about.png">
</head>
```

## Examples

### Blog Article with Article Type

An article benefits from `og:type="article"` and optional article-specific tags (publish time, author — omitted here for brevity). The canonical URL excludes tracking parameters someone might append when sharing.

```html
<title>Why Partial Indexes Beat Full Scans — Data Blog</title>
<meta name="description"
      content="Partial indexes reduce size and speed up queries when you filter on a predictable subset. Walkthrough with SQL examples.">
<link rel="canonical" href="https://blog.example/posts/partial-indexes">
<meta property="og:type" content="article">
<meta property="og:title" content="Why Partial Indexes Beat Full Scans">
<meta property="og:url" content="https://blog.example/posts/partial-indexes">
<meta property="og:image" content="https://blog.example/og/partial-indexes.png">
```

When this URL is shared with `?ref=newsletter`, the canonical and `og:url` still point to the clean path, consolidating signals.

### E-commerce Product Page

Product pages need unique titles including product name and brand. Descriptions should mention key attributes buyers search for, written naturally.

```html
<title>Model X Wireless Mouse — PeripheralShop</title>
<meta name="description"
      content="Ergonomic wireless mouse, 2.4 GHz + Bluetooth, 70-day battery. Free returns.">
<link rel="canonical" href="https://shop.example/products/model-x-mouse">
```

Avoid copying manufacturer boilerplate verbatim across hundreds of SKUs — search engines detect **duplicate snippets** and may substitute body text.

## Common Mistakes & Anti-patterns

**Identical title and description on every page** — typical of SPA defaults left unchanged per route. Fix by templating metadata per view.

**Missing or relative canonical URLs** — relative hrefs are invalid for canonical in practice; crawlers expect absolute URLs.

**Keyword stuffing in title and description** — triggers rewrites and looks spammy; hurts CTR.

**og:image too small or blocked** — previews show a grey placeholder; verify with platform debug tools (Facebook Sharing Debugger, etc.).

**Conflicting canonical and redirect** — page A canonicalizes to B but A returns 200 with duplicate content while B redirects elsewhere. Audit redirect chains after migrations.

**Relying on client-only meta for public pages** — social previews and some crawlers see empty OG tags. Serve metadata in server HTML.

## Production & Real-World Notes

Teams usually centralize SEO templates in SSR frameworks: Next.js `metadata` export, Remix `meta` function, ASP.NET Core Razor layouts with `@section Head`. Product and marketing often own **copy guidelines**; engineers implement **per-route injection** and **environment-aware absolute URLs** (staging must not leak into production canonicals).

Monitor Search Console for **duplicate without user-selected canonical** warnings. After domain migrations, combine `301` redirects, canonical updates, and sitemap resubmission. For international sites, `hreflang` link elements complement canonical — a separate topic, but often adjacent in the same `<head>` review checklist.

Structured data (JSON-LD) enhances rich results — recipes, products, FAQs — but builds on the same principle: **machine-readable head metadata** plus honest content alignment.

## Comparison / Trade-offs

**Title vs og:title** — they can differ: `<title>` optimized for search queries; `og:title` optimized for social feed context (shorter, more conversational). Duplicating exactly is fine for small sites.

**Meta description vs og:description** — same trade-off; keeping them aligned reduces maintenance. Large sites sometimes diverge intentionally.

**Canonical vs 301 redirect** — canonical when duplicates must remain reachable (analytics params); redirect when old URL should die. Redirect is stronger for users and bots.

**SSR vs prerender vs client meta** — SSR best for dynamic per-user-less pages; static prerender for docs/marketing; client-only acceptable for authenticated app shells behind login where SEO is irrelevant.

## Key Takeaways

- SEO metadata in `<head>` tells crawlers and social platforms what each URL **means** before they deep-parse content.
- Every public page needs a **unique**, human-readable `<title>`; meta description improves snippets and CTR.
- **Canonical** consolidates duplicate URLs; use absolute URLs and align with redirects.
- **Open Graph** tags power social previews; use absolute `og:image` and server-rendered HTML for reliability.
- Metadata is a **hint system** — correct implementation plus quality content beats keyword tricks.
- React/SPA apps must deliberately per-route metadata — raw client render is insufficient for OG and often for SEO.
