## Answers

### 1. [Recall]
**Answer:** Skips **finalization** because unmanaged cleanup already done in `Dispose(true)`.
**Why:** Avoids extra GC cycle and finalizer thread work.

### 2. [Recall]
**Answer:** **`true`**: managed disposables + unmanaged. **`false`**: **unmanaged only** — no other managed objects.
**Why:** Other managed may already be finalized during finalizer pass.

### 3. [Recall]
**Answer:** **`1.3.16`** — `using`, `await using`, `IAsyncDisposable`.
**Why:** Pattern is implemented by library; consumers use language constructs.

### 4. [Understanding]
**Answer:** Object is **re-rooted** to **finalization queue** instead of collected when first found unreachable.
**Why:** Runtime must run finalizer before reclaiming memory.

### 5. [Understanding]
**Answer:** **`protected virtual Dispose(bool)`** is the extensibility point; public non-virtual ensures **`SuppressFinalize`** always runs once from public API.
**Why:** Prevents override skipping suppress or double paths.

### 6. [Understanding]
**Answer:** **`SafeHandle`** centralizes **correct** handle lifetime, **critical finalizer**, **`ReleaseHandle`** contract — less error-prone than raw `IntPtr`.
**Why:** Microsoft-recommended interop pattern.

### 7. [Application]
**Answer:** Typically **neither** unless implementing aggregate disposable of owned disposables.
**Why:** No unmanaged; finalizer adds cost only.

### 8. [Application]
**Answer:** After cleaning **derived** resources (when `disposing`), call **`base.Dispose(disposing)`** — often last in derived override.
**Why:** Base releases shared handle.

### 9. [Analysis]
**Answer:** **Bad** — non-deterministic, may throw, slows finalizer thread, unreliable under shutdown.
**Why:** Finalizers are not logging infrastructure.

### 10. [Analysis]
**Answer:** **Bug** — must guard `_disposed`, only dispose `_stream` when **`disposing == true`**, handle null.
**Why:** `_stream` may be invalid in finalizer path.

### 11. [Application]
**Answer:** **Idempotency** — second call no-op, no throw.
**Why:** `using` + manual dispose + fault paths.

### 12. [Understanding]
**Answer:** Reassigning **`this`** to static/global inside finalizer makes object **reachable again**.
**Why:** GC treats it as alive until next collection.

### 13. [Recall]
**Answer:** **Finalizer thread** (not arbitrary app threads).
**Why:** CLR dedicated finalizer worker drains queue.

### 14. [Application]
**Answer:** **Deterministic** — DI scope / request pipeline disposes at end of request.
**Why:** Not relying on GC finalizers for connections.

### 15. [Analysis]
**Answer:** **Registration cost** on every instance, **extra GC pressure**, **no benefit** if no unmanaged.
**Why:** Finalizers are expensive safety nets, not default.
