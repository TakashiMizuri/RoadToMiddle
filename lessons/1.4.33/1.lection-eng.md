# Lock in Async Code — SemaphoreSlim vs lock

> Roadmap: `1.4.33` · Node: `1.4` — C# async/await · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Explain why **`lock`** cannot be held across **`await`** safely.
- Use **`SemaphoreSlim`** with **`WaitAsync`** for async-compatible mutual exclusion.
- Choose between **semaphore (1,1)**, **`SemaphoreSlim(1,1)` as mutex**, and **fine-grained locking** strategies.
- Avoid **deadlocks** and **thread-pool starvation** from sync-over-async inside locks.
- Apply **throttling** with `SemaphoreSlim(n,n)` beyond exclusive locks (`1.4.18`).

---

## Why This Matters

You protect a shared cache with `lock (_gate) { await RefreshAsync(); }` — the compiler rejects it (`CS1996`), or worse, a teammate uses **`Monitor.Enter`** + await in unsafe patterns. Another path wraps **`lock`** around **`.Result`** — deadlock under load (`1.4.14`).

Async server code needs **locks that release the thread** while waiting. **`SemaphoreSlim.WaitAsync`** is the standard primitive. Middle developers must know **when a simple lock still fits** (pure sync critical sections) and when **semaphore + async** is mandatory.

---

## Core Concepts

### Why lock and await Do Not Mix

**`lock`** compiles to **`Monitor.Enter`** / **`Monitor.Exit`** on the **current thread**. While held, the thread must not leave the critical section. **`await`** may **return** the thread to the pool until continuation — another caller could enter the same `lock` on the **same async flow's continuation thread** while logically still "inside" the first operation — or you block the thread synchronously waiting — defeating async.

C# **disallows** `await` inside `lock` statement. The language enforces: **do not hold thread-bound locks across suspension points**.

### SemaphoreSlim as Async Mutex

```csharp
private readonly SemaphoreSlim _mutex = new(1, 1);

public async Task UpdateCacheAsync(CancellationToken ct)
{
    await _mutex.WaitAsync(ct);
    try
    {
        await RefreshFromDbAsync(ct);
    }
    finally
    {
        _mutex.Release();
    }
}
```

**`WaitAsync`** registers a continuation when the semaphore is busy — **no blocked thread** (unless sync `Wait()`). Always **`Release()`** in `finally` — even on exceptions.

**`SemaphoreSlim(1, 1)`** behaves like a mutex with optional async wait. It is **not** re-entrant — same async flow calling twice without release **deadlocks** (unlike `lock` which is re-entrant for same thread — and that re-entrancy does not map cleanly to async).

### Sync Critical Section — lock Still OK

If the protected region has **no await** — pure in-memory update:

```csharp
lock (_cacheLock)
{
    _cache[key] = value;
}
```

Keep it **short**. Never call external I/O inside `lock`.

### Throttling vs Exclusion

**`SemaphoreSlim(10, 10)`** allows ten concurrent operations — useful for limiting parallel DB or HTTP calls (`1.4.18`). Different problem than exclusive cache update — choose count deliberately.

## Under the Hood

### Monitor Reentrancy vs Async

**`lock`** is **re-entrant** per thread: same thread can enter twice. Async continuations may run on **different threads** — re-entrancy semantics break. Hence semaphore without re-entrancy is actually **clearer** for async — document "do not nested wait same instance."

### Thread Pool Starvation

```csharp
lock (_gate) { return _http.GetStringAsync(url).Result; }
```

Blocks a pool thread waiting for I/O; under load → **starvation** (`1.4.10`). Fix: **`await`** outside lock or use **`WaitAsync`** throughout.

### ReaderWriterLockSlim

**`ReaderWriterLockSlim`** has **`EnterReadLock`** sync only — for read-heavy sync caches. For async read paths, patterns include **immutable snapshots**, **`AsyncReaderWriterLock`** from libraries (e.g. **Nito.AsyncEx**), or **Channels** for single-writer — evaluate complexity.

## Syntax / Commands / API

| Primitive | Async-safe | Re-entrant | Use case |
|-----------|------------|------------|----------|
| `lock` | No await inside | Yes (thread) | Short sync |
| `SemaphoreSlim.WaitAsync` | Yes | No | Async mutex |
| `SemaphoreSlim(n,n)` | Yes | No | Throttle |
| `Monitor.TryEnter` | No await | Yes | Legacy sync |

## Examples

### Cache Stampede Prevention

```csharp
public async Task<Data> GetAsync(string key, CancellationToken ct)
{
    if (_cache.TryGetValue(key, out var hit)) return hit;

    await _loadLock.WaitAsync(ct);
    try
    {
        if (_cache.TryGetValue(key, out hit)) return hit; // double-check
        var data = await _db.LoadAsync(key, ct);
        _cache[key] = data;
        return data;
    }
    finally
    {
        _loadLock.Release();
    }
}
```

Double-checked locking **after** acquiring semaphore prevents thundering herd on cold key.

### Anti-pattern

```csharp
lock (_gate)
{
    var data = _client.GetAsync(url).GetAwaiter().GetResult();
}
```

Sync-over-async inside lock — deadlock + pool starvation risk.

## Common Mistakes & Anti-patterns

**`WaitAsync` without `finally Release`** — semaphore leak, permanent stall.

**Nested `WaitAsync` on same `SemaphoreSlim(1,1)`** from same flow — self-deadlock.

Using **`lock`** + **`Task.Run`** to "fix" await — races unless entire critical path moves inside Run with clear ordering — usually wrong.

**Global singleton semaphore** for unrelated resources — unnecessary serialization; partition locks per key (`ConcurrentDictionary<string, SemaphoreSlim>`) with disposal strategy.

---

## Production & Real-World Notes

Prefer **immutable cache swap** (`Interlocked.Exchange` reference) for read-heavy, write-rare — fewer locks.

**Distributed locks** (Redis Redlock) when multiple **instances** share resource — `SemaphoreSlim` is **in-process only**.

Always pass **`CancellationToken`** to `WaitAsync` so shutdown does not hang on lock.

Dispose **`SemaphoreSlim`** when per-key semaphores removed from dictionary — avoid handle leaks.

## Comparison / Trade-offs

| Approach | Pros | Cons |
|----------|------|------|
| `lock` sync only | Fast, simple | No await |
| `SemaphoreSlim(1,1)` | Async mutex | Not re-entrant |
| Per-key locks | Parallelism | Cleanup complexity |
| Immutable snapshots | Lock-free reads | Memory churn |

## Quick Reference

- **Never `await` in `lock`.**
- Async mutex: **`await sem.WaitAsync(ct)`** + **`finally Release()`**.
- **`lock`** only for **short sync** regions.
- No **`.Result`** inside locks.
- Throttle: **`SemaphoreSlim(n,n)`**, not always 1.
- Multi-instance → **distributed** lock, not semaphore alone.

---

## Key Takeaways

- Thread-bound **`lock`** incompatible with **`await`** suspension.
- **`SemaphoreSlim.WaitAsync`** is the async mutual exclusion primitive.
- Always **`Release`** in `finally`; pass **cancellation** to wait.
- Sync-over-async in locks causes **deadlock** and **starvation**.
- Separate **throttling** (n>1) from **exclusive** (1,1) needs.
- Consider **immutable** structures to reduce locking.
