# Exercises: Undo — restore, revert, reset

> Complete after reading `1.lection-eng.md`, `2.lection-ru.md` and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Three trees on paper
**Goal:** Internalize HEAD, index, working tree.

**Steps:**
1. Draw three boxes: HEAD, Index, Working tree.
2. Annotate what happens after: (a) edit file, (b) `git add`, (c) `git commit`.
3. Add a fourth row for `git restore --staged` and `git reset --mixed HEAD~1`.

**Success criteria:** Diagram matches lecture; you can explain it aloud in 60 seconds.

---

### E2. Restore practice
**Goal:** Use `git restore` without touching HEAD.

**Steps:**
1. In a test branch, edit `practice.txt`: add line A, `git add`, add line B (do not stage B).
2. `git status` — confirm staged vs unstaged.
3. `git restore practice.txt` — what happened to line B? line A?
4. Re-create line B; run `git restore --staged practice.txt` — staged state?
5. Document final content vs HEAD.

**Success criteria:** Written before/after for each command.

## Medium

### M1. Reset modes lab
**Goal:** Feel soft vs mixed vs hard.

**Setup:** Branch `undo-lab`, three empty commits on top of base.

**Steps:**
1. On commit #3, change `lab.txt` (+ "step3"), commit.
2. **Experiment A:** `git reset --soft HEAD~1` — status, recommit or note staged state.
3. Reset branch to commit #3 again (use `git reset --hard` to tip or re-do commits).
4. **Experiment B:** `git reset HEAD~1` — status; is "step3" staged?
5. Reset to tip again; add uncommitted line "dirty"; **Experiment C:** `git reset --hard HEAD~1` — is "dirty" recoverable?

**Success criteria:** Table: mode | HEAD position | staged? | uncommitted survivability?

---

### M2. Revert on shared-style history
**Goal:** Safe undo without rewrite.

**Steps:**
1. Create branch `revert-demo` with commits: `Good` → `Bad` (introduce obvious typo) → `Also good`.
2. `git revert <Bad-hash> --no-edit`
3. `git log --oneline` — how many commits? Is `Bad` still visible?
4. `git show HEAD` — patch should undo typo.

**Success criteria:** Explain why this is safe if branch were pushed; contrast with `git reset --hard` + force push.

---

### M3. Reflog rescue
**Goal:** Recover from hard reset panic.

**Steps:**
1. Create commit with unique message `RESCUE-ME-123`.
2. `git reset --hard HEAD~1`
3. `git log --oneline` — is `RESCUE-ME-123` gone from branch?
4. `git reflog` — find hash.
5. `git reset --hard <hash>` — restore.

**Success criteria:** Screenshot or paste of reflog line + restored log entry.

## Hard (stretch)

### H1. Decision matrix (production)
**Goal:** Choose undo strategy like a tech lead.

Write recommendations (command + 1 sentence why) for:

1. Junior staged `appsettings.Development.json` with real connection string — not committed yet.
2. Bad commit on `feature/x`, never pushed.
3. Bad commit on `main`, 10 teammates may have pulled.
4. Need to remove file from last commit only (unpushed) and fix `.gitignore`.
5. Accidentally ran `git reset --hard` losing 2 local commits never pushed.

**Success criteria:** No use of `reset --hard` on scenario 3; reflog mentioned in 5.

---

### H2. Staged + unstaged same file
**Goal:** Master P1-style reasoning.

**Steps:**
1. In `mixed.txt`: add staged line `STAGED`, save unstaged `UNSTAGED` in same file.
2. Predict outcome, then run: `restore`, `restore --staged`, `restore --staged --worktree`.
3. After each run, reset file state and repeat next command from clean C3 state.

**Success criteria:** Prediction table matches actual `git diff` and `git diff --staged` output.

---

### H3. Revert merge (optional)
**Goal:** Understand `-m 1`.

**Steps:**
1. Create `main-line` and `side` branches; commit on each; `git checkout main-line; git merge side`.
2. Attempt `git revert -m 1 HEAD --no-edit` (if conflicts, document — do not require full resolve).
3. Explain in writing what `-m 1` selected.

**Success criteria:** Two sentences on why merge revert differs from normal revert.

## Hints (read only if stuck)

<details>
<summary>Hint E2</summary>

`git restore file` resets working tree to **index** — unstaged line B disappears; staged line A (in index) remains, so file may still differ from HEAD until you unstage or commit.
</details>

<details>
<summary>Hint M1</summary>

After `--soft`, `git status` shows all changes from undone commit as **Changes to be committed**. After `--mixed`, as **Changes not staged**.
</details>

<details>
<summary>Hint H1 scenario 4</summary>

`git reset --soft HEAD~1` → update `.gitignore` → `git rm --cached` if needed → recommit. Do not use `--hard` if you want to keep other changes from that commit.
</details>
