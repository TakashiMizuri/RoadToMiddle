# GC Modes: Workstation vs Server

> Roadmap: `1.5.9` · Node: `1.5` — C#: .NET Runtime and Memory · Depth: понимание

## Learning Objectives

After this lesson you will be able to:

- Explain what **workstation GC** and **server GC** optimize for and how they differ in thread model.
- Describe **background GC**, **concurrent GC**, and how they affect pause times on desktop vs server workloads.
- Choose or configure GC mode for ASP.NET Core, console tools, and Windows Services.
- Read **`DOTNET_gcServer`**, **`runtimeconfig.json`**, and **`Environment.IsServerGC`** to verify runtime behavior.
- Relate GC mode choices to **throughput**, **latency**, and **memory footprint** trade-offs.

---

## Why This Matters

The garbage collector is not a single algorithm frozen in time — it is a **policy engine** tuned for different machine roles. A laptop running a WPF editor cares about **UI responsiveness**: long stop-the-world pauses feel like freezes. A 16-core Linux VM running ASP.NET Core behind a load balancer cares about **aggregate throughput**: pausing one thread briefly while others keep serving may be acceptable if total allocations per second stay high.

Choosing the wrong mode does not break correctness, but it can silently degrade performance. Teams deploy an API to production with **workstation GC** (the SDK default for some project types) and wonder why p99 latency spikes under load, or they enable **server GC** on a single-core container and pay for extra GC threads that contend for one CPU. Middle developers must know which mode is active, how to change it, and what symptoms point to a mismatch — not memorize registry keys from a blog post.

This lesson builds on generations, LOH, and mark-sweep from earlier `1.5.x` topics. GC **mode** decides *how many collectors run* and *how aggressively* they interleave with application threads; generations decide *what* gets collected and *how often*.

## Core Concepts

### Two GC Flavors at a Glance

**.NET exposes two primary GC flavors** (plus variations like **DATAS GC** in .NET 8+ for very high-throughput server scenarios):

**Workstation GC** assumes a **single interactive application** on a machine where the user notices pause times. Historically it favored **shorter pauses** even if that meant less aggressive parallel collection. On modern .NET (Core / 5+), workstation GC still typically uses **one logical GC heap per process** (unless **GCHeapCount** or container limits apply) and may run **background/concurrent** collections for Gen2 to reduce blocking.

**Server GC** assumes a **dedicated server process** on a multi-processor machine. It creates **one managed heap per logical processor** (by default), each with its own allocation thread affinity patterns, and uses **multiple GC threads** to collect in parallel. The goal is **higher throughput** on parallel hardware: while one thread collects heap 3, heaps 0–2 may still accept allocations (subject to phase and version details). Pause times can be **longer in wall-clock terms** on a given thread but **aggregate application throughput** often rises for allocation-heavy server apps.

Neither mode eliminates pauses entirely — both are generational, tracing collectors. The difference is **parallelism**, **heap layout**, and **default tuning** for latency vs throughput.

### How You Know Which Mode Is Active

At runtime:

```csharp
Console.WriteLine($"Server GC: {GCSettings.IsServerGC}");
Console.WriteLine($"Latency mode: {GCSettings.LatencyMode}");
```

For ASP.NET Core published apps, check **`runtimeconfig.json`**:

```json
{
  "runtimeOptions": {
    "configProperties": {
      "System.GC.Server": true
    }
  }
}
```

Environment variables (examples):

- `DOTNET_gcServer=1` — enable server GC
- `DOTNET_gcServer=0` — workstation GC

**ASP.NET Core** projects typically enable **server GC** by default in the `.csproj`:

```xml
<PropertyGroup>
  <ServerGarbageCollection>true</ServerGarbageCollection>
</PropertyGroup>
```

Console apps often default to **workstation** unless you opt in. Always **measure** — defaults change across templates and SDK versions.

### Pause Time vs Throughput

When a **full blocking collection** runs, application threads that allocate or need safe object graphs may be **suspended** (details vary by generation and background GC phase). Workstation GC on an interactive app tries to keep those pauses short enough that UI threads recover quickly. Server GC may accept a longer coordinated pause if **all cores** finish collecting faster in parallel and the process returns to **saturated throughput** sooner.

For **ASP.NET Core**, request threads hit the thread pool; a Gen2 pause can delay many concurrent requests if they trigger allocation or need GC synchronization. That is why server GC is the common default: web APIs are usually **throughput-bound** on multi-core servers, not single-thread UI-bound.

For **desktop**, **game loops**, or **real-time audio**, developers sometimes force **workstation GC** or set **`GCLatencyMode.LowLatency`** / **`SustainedLowLatency`** during critical sections — trading higher memory use or deferred collection for fewer disruptive pauses.

### Background and Concurrent Collection

**Concurrent / background GC** (terminology evolved across .NET Framework → Core) allows **Gen2** collection work to proceed **alongside** application threads for portions of the mark phase, reducing stop-the-world time. Workstation GC has long used concurrent Gen2 on desktop; server GC gained background collection improvements over releases.

You do not usually toggle "background GC" separately in app code — it is part of the runtime flavor and version. What matters operationally: **server GC + many cores** → parallel heaps; **profiler traces** show concurrent mark threads. When diagnosing latency, distinguish **Gen0/Gen1 ephemeral pauses** (usually sub-millisecond to low ms) from **Gen2/LOH** events (can be tens of ms or more depending on heap size and mode).

### Container and Cloud Considerations

In **Docker/Kubernetes**, CPU limits affect GC thread usefulness. A pod limited to **250m CPU** with server GC may spin GC threads that **fight** with app threads for fractional cores — sometimes **workstation GC** or reduced **`GCHeapCount`** performs better. .NET reads ** cgroup memory limits** and adjusts heap hard limits (`DOTNET_GCHeapHardLimit` etc.).

Rule of thumb: **match GC to actual vCPU and workload**. Multi-core API server with dedicated CPUs → server GC. Single-replica sidecar, CLI tool, or UI → workstation GC. **Load test** both if unsure.

## Under the Hood

Each **server GC heap** is associated with a **heap index**; objects stay on the heap where they were allocated (no cross-heap movement for SOH in normal operation), which improves locality but means **imbalanced heaps** if one thread allocates disproportionately. The runtime may **balance** over time depending on version and settings.

**GC threads** are runtime threads (not thread-pool workers). In server mode you may see multiple **`/.NET GC`** or **`GC Server #`** threads in a debugger. **Ephemeral segments** (Gen0/Gen1) and **LOH** exist per heap in server mode.

**`GCSettings.LatencyMode`** interacts with mode:

| Mode | Effect (simplified) |
|------|---------------------|
| `Batch` | Full blocking collections; predictable, higher pauses |
| `Interactive` | Default; balances pauses |
| `LowLatency` | Defers Gen2; use briefly |
| `SustainedLowLatency` | Longer deferral window; monitor memory |
| `NoGCRegion` | Try to avoid GC in a scope (`GC.TryStartNoGCRegion`) |

Workstation vs server is orthogonal to latency mode but **combining** server GC with sustained low latency for hours can cause **memory pressure** — understand the trade-off.

## Examples

### Verify Mode in a Console App

```csharp
// Program.cs — run after publish, not just F5 defaults
Console.WriteLine($"Server GC: {GCSettings.IsServerGC}");
Console.WriteLine($"Heap count (approx): {GC.GetGCMemoryInfo().HeapCount}");
```

If you toggle `<ServerGarbageCollection>` and republish, `IsServerGC` should flip. Development inner-loop and published output can differ — test the **deployed artifact**.

### ASP.NET Core Default

A typical web `.csproj` enables server GC. Under load (`wrk`, `k6`), Gen2 pauses may be less frequent per request than workstation on 8 cores, but individual pause duration in traces may look larger — **percentile latency** improves when throughput is the goal.

## Common Mistakes & Anti-patterns

**Assuming F5 behavior equals production.** Debug builds, different `runtimeconfig`, and environment overrides change GC. Always verify on the deployed host.

**Server GC on single-core containers without testing.** Extra GC threads add overhead when no parallelism exists.

**Chasing zero GC pauses on a server API.** Some allocation is inevitable; fix ** unnecessary allocations** (LINQ, strings) before disabling server GC.

**Ignoring `GCLatencyMode` leaks.** `LowLatency` left enabled globally can inflate memory until an forced full GC.

## Production & Real-World Notes

Teams document GC settings in **runbooks** alongside CPU/memory limits. **Azure App Service** and **IIS** host ASP.NET with server GC by convention. **Lambda / short-lived functions** may prefer workstation or aggressive trimming — lifetime is seconds.

When migrating **.NET Framework → .NET Core**, GC behavior changed (better background GC, LOH compaction options). Re-benchmark; do not copy Framework registry hacks blindly.

Pair GC mode tuning with **`GC.GetTotalMemory`**, **Application Insights**, or **OpenTelemetry** .NET runtime metrics. Compare **GC pause time %** in dotnet-counters before and after config changes.

## Comparison / Trade-offs

| Aspect | Workstation GC | Server GC |
|--------|----------------|-----------|
| Target | Interactive, client | Server, multi-core |
| Heaps | Typically one | One per logical CPU (default) |
| Pause goal | Shorter per thread | Higher parallel throughput |
| Default | Many console/UI apps | ASP.NET Core |
| Risk | Lower throughput on many-core | Longer pauses possible per event |

When **latency SLO** is strict (p99 < 50ms) on small instances, profile Gen2 frequency; consider **object pooling**, **ArrayPool**, fewer LOH allocations — not only GC mode flips.

## Key Takeaways

- Workstation GC optimizes for **interactive latency**; server GC for **multi-core throughput**.
- ASP.NET Core usually runs **server GC** — verify with `GCSettings.IsServerGC`.
- GC mode does not replace **allocation discipline**; it shapes pause and parallelism characteristics.
- Containers with tiny CPU quotas may need **non-default** GC settings — benchmark.
- Use **dotnet-counters** and load tests to validate config, not folklore.
