# Object.Equals, GetHashCode, and IEquatable

> Roadmap: `1.2.16` · Node: `1.2` — C# OOP · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Distinguish **reference equality** from **value equality** and explain when each applies.
- Describe the **contract** between `Equals`, `GetHashCode`, and `==` / `!=`.
- Override `Equals` and `GetHashCode` correctly for custom types, including **structural** comparison.
- Implement **`IEquatable<T>`** to avoid boxing and improve performance in hot paths.
- Explain how **`Dictionary<TKey,TValue>`** and **`HashSet<T>`** use hash codes and equality.
- Recognize when **records** (from `1.2.13`) already satisfy equality and when you must implement it manually.
- Diagnose common bugs: broken hash contract, asymmetric equality, floating-point surprises.

## Why This Matters

Two variables can point at the same object, or they can point at different objects that still represent the same logical value. Confusing those two ideas is one of the most common sources of subtle bugs in C#. You add a `Customer` to a `HashSet<Customer>`, later create another `Customer` instance with the same ID, call `Contains` — and get `false`. Your unit test compares two `Money` objects with `==` and fails even though both represent `10.00 USD`. Your API deduplicates DTOs in a dictionary and silently keeps duplicates because you never overrode equality.

Every collection that keys by object identity — `Dictionary`, `HashSet`, `ConcurrentDictionary`, LINQ's `Distinct`, EF Core's change tracker lookups — depends on a consistent equality story. The BCL gives you defaults, but defaults for **reference types** mean "same instance on the heap," not "same business meaning." Value types (`struct`) get **value equality** by default (field-by-field), which is often wrong for types like `DateTime` ranges or composite keys where you want custom rules.

Middle developers do not memorize recipes; they understand the **contract** and implement it once, correctly. That contract ties together `Object.Equals`, `GetHashCode`, optional `IEquatable<T>`, and optionally overloaded `==`. Break one leg of the table and the whole structure wobbles — usually in production, under load, inside a hash table you forgot was there.

## Core Concepts

### Reference Equality vs Value Equality

When you compare two variables of a **reference type** (`class`) with `ReferenceEquals(a, b)` or the default `Object.Equals` implementation, you ask: "Do these two references point to the **same object** in memory?" That is **reference equality**. It ignores field values. Two distinct `new Customer()` instances are never equal by default, even if every property matches.

**Value equality** asks: "Do these two objects represent the **same logical value**?" Two different `string` instances containing `"hello"` are value-equal because `string` overrides `Equals` to compare character content, not references. `int`, `decimal`, and other primitives compare by value. For your own types, value equality is **opt-in** — you implement it.

This distinction connects directly to `1.2.1` (reference semantics): reference types are handles to heap objects; equality default follows the handle, not the payload. Records (`1.2.13`) flip the story for data-centric types by generating value-based equality automatically.

### The Object.Equals Contract

`Object.Equals(object? obj)` is the virtual method at the root of the type hierarchy. The default implementation in `System.Object` uses **reference equality** for reference types. For **value types** (`struct`), the runtime provides a synthesized override that compares fields bitwise / structurally (with caveats for fields containing reference types).

When you override `Equals`, Microsoft documents a contract (paraphrased for clarity):

1. **Reflexive:** `x.Equals(x)` is `true` (except `NaN` cases for floats — see pitfalls).
2. **Symmetric:** `x.Equals(y)` implies `y.Equals(x)`.
3. **Transitive:** if `x.Equals(y)` and `y.Equals(z)`, then `x.Equals(z)`.
4. **Consistent:** repeated calls return the same result if neither object changed.
5. **Non-null:** `x.Equals(null)` is `false` for non-null `x`.

Violating symmetry or transitivity breaks collections and sorting. A classic mistake: comparing a `Person` to a `Employee` subclass with asymmetric rules.

**Null handling:** always guard `if (obj is null) return false;` (or use pattern matching). For nullable value types, consider `Nullable<T>` semantics separately.

### GetHashCode and Why It Must Agree with Equals

`GetHashCode()` returns a 32-bit `int` used by hash-based collections to **bucket** objects. The algorithm is not cryptographic; it is a **hash function** for speed. The essential rule:

> If `a.Equals(b)` is `true`, then `a.GetHashCode() == b.GetHashCode()` **must** be `true`.

The converse is **not** required: unequal objects may collide (same hash code). Collisions degrade performance (more equality checks in a bucket) but do not break correctness if `Equals` is correct.

If you override `Equals` without overriding `GetHashCode`, you inherit `Object.GetHashCode()` which is typically based on **object identity** (reference) for classes. Two value-equal instances then land in different buckets — `Dictionary` lookups fail. This is the bug from the introduction.

For **immutable** value objects, the hash code can be cached. For **mutable** types used as dictionary keys — avoid that pattern entirely; mutating a key after insertion corrupts the bucket chain.

### Implementing Equals for a Class

A robust pattern for a reference type with an `Id` field:

```csharp
public sealed class Customer : IEquatable<Customer>
{
    public Guid Id { get; }
    public string Name { get; }

    public Customer(Guid id, string name)
    {
        Id = id;
        Name = name;
    }

    public bool Equals(Customer? other)
    {
        if (other is null) return false;
        if (ReferenceEquals(this, other)) return true;
        return Id == other.Id; // business key
    }

    public override bool Equals(object? obj) =>
        Equals(obj as Customer);

    public override int GetHashCode() => Id.GetHashCode();
}
```

Several deliberate choices appear here. **Sealed** class avoids subclass equality surprises. **`IEquatable<Customer>`** provides a typed `Equals` without boxing when the caller knows the type. **`ReferenceEquals` short-circuit** is cheap. **Business key** (`Id`) defines equality — two customers with the same ID are the same customer, regardless of name changes.

`override Equals(object? obj)` delegates to the typed method via `as` or pattern `obj is Customer c`. Always override **both** `Equals` and `GetHashCode` together.

For **structural** equality (all fields), combine contributing fields:

```csharp
public override int GetHashCode() =>
    HashCode.Combine(Id, Name);
```

`HashCode.Combine` ( .NET Core 2.1+) replaces error-prone manual `hash * 31 + field.GetHashCode()` recipes.

### IEquatable<T> — Performance and Boxing

`IEquatable<T>` declares `bool Equals(T other)`. Generic collections and `EqualityComparer<T>.Default` prefer this method when available.

When you call `Equals(object? obj)` on a **struct** that does not implement `IEquatable<T>`, the struct is **boxed** to `object` on the heap — allocation + indirection. In tight loops (`List.Contains`, dictionary probes), that cost matters. Implementing `IEquatable<MyStruct>` lets the generic path call the typed overload directly.

`EqualityComparer<T>.Default` resolution order (simplified): if `T` implements `IEquatable<T>`, use it; else if `T` is a nullable value type, special rules; else fall back to `Object.Equals`.

You rarely call `IEquatable<T>` explicitly in application code — you implement it so infrastructure does.

### The == and != Operators

For **reference types**, `==` and `!=` are **reference equality** unless you **overload** the operators (covered in `1.2.17`). `Equals` and `==` can disagree if you override `Equals` but forget operators — a frequent source of confusion.

For **value types**, `==` compares values using built-in or overloaded operators.

**Strings** overload `==` to compare content (ordinal by default in modern guidance; culture-aware comparisons use `Equals` overloads with `StringComparison`).

**Rule of thumb:** if you override `Equals` on a reference type and want `==` to match, overload `==` and `!=` too, and delegate to `Equals`.

```csharp
public static bool operator ==(Customer? left, Customer? right) =>
    left is null ? right is null : left.Equals(right);

public static bool operator !=(Customer? left, Customer? right) => !(left == right);
```

Handle **null** on both sides symmetrically.

### How Dictionary and HashSet Use Equality

Internally, `Dictionary<TKey,TValue>` stores entries in an array of buckets. On `Add` or lookup:

1. Compute `hashCode = comparer.GetHashCode(key)` (via `IEqualityComparer<T>` or default).
2. Map hash code to bucket index (mask / modulo depending on implementation).
3. Walk the chain in that bucket; for each candidate, call `comparer.Equals(storedKey, searchKey)`.

If `GetHashCode` is inconsistent with `Equals`, the search never finds the entry even though it exists. If keys mutate after insertion, hash code changes and the entry becomes unreachable — **orphaned** in the table.

Pass a custom **`IEqualityComparer<T>`** (e.g. `StringComparer.OrdinalIgnoreCase`) when you need different rules without changing the key type itself.

### Records and Generated Equality

Records (`1.2.13`) generate `Equals`, `GetHashCode`, `==`, and `!=` from **positional and object properties** (excluding settable-only nuances). Value semantics are the default for `record` and `record class`.

`record struct` also gets synthesized equality; `readonly record struct` is common for small immutable values (`1.2.14`, `1.2.15`).

When a hand-written `class` models a DTO, consider `record` instead of manual equality. When inheritance or encapsulated behavior complicates the shape, manual implementation remains appropriate.

## Under the Hood

### Runtime Dispatch for Equals

`Object.Equals` is **virtual**. Calling `a.Equals(b)` on a variable typed as `Object` or a base class uses **virtual dispatch** — the runtime selects the most derived override. That is how `string.Equals` runs even when referenced as `object`.

`EqualityComparer<T>.Default` is a struct wrapper created per `T` at JIT time. For many `T`, the JIT specializes calls to avoid virtual dispatch where possible.

### Default Struct Equality

For unmodified `struct` types, the runtime emits **EqualityComparer** logic that compares each field using its own equality. If a field is a reference type, that field uses **reference equality** unless the field type overrides equality (`string` does). A struct containing two `string` fields compares strings by value; a struct containing two `byte[]` arrays compares **array references**, not contents — a common trap.

### GetHashCode Implementation Strategies

`Object.GetHashCode()` for reference types without override historically tied to sync block or identity — **not stable across runs** in some CLR versions for certain objects. Never persist `GetHashCode()` to disk as a long-lived identifier.

`HashCode.Combine` mixes field hash codes with a fixed algorithm (similar in spirit to FNV or xxHash-style mixing). Order matters: `Combine(a, b)` ≠ `Combine(b, a)` in general — consistent with asymmetric field importance only if you are deliberate.

For **large** composite keys, consider hashing a canonical serialized form — trade CPU vs collision rate.

### Floating-Point and Nullable Edge Cases

`double.NaN` breaks reflexivity in IEEE 754: `NaN == NaN` is false. `double.Equals(NaN, NaN)` is **true** per `Object.Equals` override for floating types. Be consistent within your domain model.

`null` reference keys are rejected by `Dictionary<TKey,TValue>` when `TKey` is non-nullable reference type; nullable reference type annotations (`#nullable enable`) help catch this at compile time.

### JIT and Inlining

Small `Equals` and `GetHashCode` overrides on **sealed** types are often **inlined** in optimized builds. `IEquatable<T>` enables devirtualization in generic code when `T` is a known struct.

## Syntax / API

| Member | Purpose |
|--------|---------|
| `virtual bool Equals(object? obj)` | Polymorphic equality entry point |
| `virtual int GetHashCode()` | Hash for collections |
| `bool IEquatable<T>.Equals(T other)` | Typed, often boxing-free equality |
| `static bool ReferenceEquals(object? a, object? b)` | Identity check |
| `HashCode.Combine(...)` | Mix multiple fields into one hash |
| `EqualityComparer<T>.Default` | BCL default strategy for `T` |
| `IEqualityComparer<T>` | Pluggable equality for collections |

**Analyzer:** override `Equals` → Roslyn CA rules and IDE suggest overriding `GetHashCode`.

## Examples

### Broken: Equals without GetHashCode

```csharp
var d = new Dictionary<BadKey, int>();
var k1 = new BadKey(1);
var k2 = new BadKey(1);
d[k1] = 42;
Console.WriteLine(d.ContainsKey(k2)); // False — hash mismatch
```

`BadKey` overrides `Equals` to compare `Id` but inherits identity `GetHashCode`. Fix: `GetHashCode() => Id.GetHashCode()`.

### Structural Equality for a Small Struct

```csharp
readonly record struct Point(int X, int Y);
// Compiler-generated Equals/GetHashCode — preferred over manual struct
```

### Custom Comparer Without Touching the Type

```csharp
var set = new HashSet<string>(StringComparer.OrdinalIgnoreCase);
set.Add("Hello");
set.Contains("hello"); // True
```

## Common Mistakes & Anti-patterns

**Overriding Equals but not GetHashCode** — dictionary and set bugs as above. Always pair them.

**Using mutable objects as dictionary keys** — changing a property changes `GetHashCode` after insert. Keys should be immutable value objects.

**Asymmetric equality across inheritance** — `if (obj is not Employee e) return false` in a `Person.Equals` that allows comparing to subclasses inconsistently. Often seal the type or use explicit type match.

**Comparing floating point with `==` in Equals** — use tolerance or `BitConverter` bit patterns for strict IEEE semantics.

**Deep equality on graphs** — reference cycles cause infinite recursion. Use visited sets or compare by ID.

**Relying on `GetHashCode()` stability** — not guaranteed for all types across processes; use explicit IDs for persistence.

## Production & Real-World Notes

Entity Framework keys: primary key equality often **is** identity in the database — two different tracked instances with same ID may need reference equality in the change tracker context; DTOs use value equality for API comparison.

Distributed systems: equality in memory does not imply equality across services — use explicit versioned IDs, not `GetHashCode`, in message deduplication.

Unit testing: `Assert.Equal` uses `EqualityComparer<T>.Default` — your type must satisfy the contract or tests lie about production behavior.

Performance: in hot `HashSet` lookups, prefer `IEquatable<T>`, sealed types, and cheap hash mixing; profile before micro-optimizing.

Logging and serialization: never log raw hash codes as PII substitutes.

## Comparison / Trade-offs

| Approach | When to use |
|----------|-------------|
| Default reference equality | Singleton services, explicit identity objects |
| Manual `Equals` + `GetHashCode` | Custom rules, non-record classes |
| `record` | DTOs, value objects with structural equality |
| `IEqualityComparer<T>` | Alternate comparison without modifying type |
| `IEquatable<T>` | Structs and performance-sensitive classes |

**Equals vs `==`:** align them on reference types when value semantics matter. **GetHashCode quality:** better distribution → fewer collisions → faster dictionaries; perfection is unnecessary.

## Key Takeaways

- Reference types default to **identity** equality; value equality is intentional design.
- `GetHashCode` must be **consistent** with `Equals` or hash collections break silently.
- `IEquatable<T>` is the performance path for generic equality, especially for structs.
- `Dictionary` / `HashSet` probe with hash first, then `Equals` — both must be correct.
- Records solve equality for data models; manual implementation remains for complex domains.
- Align `==` with `Equals` when developers expect intuitive comparison.
- Treat dictionary keys as **immutable** value objects.
