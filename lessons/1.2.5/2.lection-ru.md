# Encapsulation — why public fields are bad

> Roadmap: `1.2.5` · Node: `1.2` — C#: ООП · Depth: **глубоко**

## Learning Objectives

После этого урока ты сможешь:

- Определить **encapsulation** как объединение data с behavior и hiding implementation.
- Объяснить, почему **`public` fields** ломают invariants и coupling callers к layout.
- С contrast properties, methods, factories как controlled access.
- Находить **leaky encapsulation** через mutable collections и `protected` fields.
- Связать encapsulation с maintainability, testing, API versioning.
- Применять **`private set`**, **`init`**, guard clauses, **Tell, Don't Ask**.

## Why This Matters

Ты знаешь classes (`1.2.1`), properties (`1.2.2`), constructors (`1.2.3`), visibility (`1.2.4`). **Encapsulation** — design principle, связывающий их: object enforce rules; callers используют **stable surface**.

Public fields удобны в tutorials — меньше строк, JSON «просто работает». В production — **silent corruption**: negative balance, пустой email, ship order без lines. Public field = **каждый external assignment** в semver contract. Rename field — breaking change по всей codebase.

Middle защищает invariants на **object boundaries**, не в каждом controller. Domain model переживает REST, jobs, consumers без duplicated validation.

## Core Concepts

### Что такое encapsulation

**Encapsulation** скрывает **how** state хранится и обновляется, expose **what** object может. External code зависит от **operations**, не memory layout.

Encapsulated `BankAccount` не даёт писать в `_balance` снаружи. **`Deposit`**, **`Withdraw`**, read-only **`Balance`**. Object **owns** rules: no overdraft, audit, rounding — centralized.

### Public fields

```csharp
public class Account { public decimal Balance; }
account.Balance = -1_000_000m;  // no hook
```

Refactor на computed balance из transactions → менять **every assignment site**.

Properties/methods — **seam** для evolution без перекомпиляции callers (часто).

### Properties — seam, не magic

`{ get; set; }` public — **unrestricted mutation** без validation. Нужно решить **who/when**:

- `{ get; private set; }`
- `{ get; init; }`
- full property с validation
- mutation только через domain verbs

### Tell, Don't Ask

```csharp
// Ask-heavy
if (order.Total > 1000 && order.Customer.IsVip) order.ApplyDiscount(0.1m);
// Tell
order.ApplyBestDiscount();
```

Invariants не должны зависеть от правильной последовательности reads/writes public fields снаружи.

### Leaky encapsulation

**`public List<OrderLine> Lines { get; set; }`** — Clear/Replace мимо rules.  
**`protected` fields** — fragile base class (`1.2.11`).  
**Anemic domain** — data bags, logic только в services.

### DTO vs domain

На **boundaries** (HTTP, DB) anemic DTO OK. Ошибка — **весь** domain как public DTO, rules в controllers. Map DTO → **`Order`** через factory/ctor с invariants.

## Under the Hood

### Compiler view

**Field** — IL `ldfld`/`stfld`. **Property** — accessor methods `get_X`/`set_X`. Indirection enables breakpoints, logging, INotifyPropertyChanged, EF change tracking. Public fields **skip seam** — frameworks не intercept uniformly.

### Serialization

System.Text.Json — public properties by default; fields need `[JsonInclude]`. Reflection читает private fields — encapsulation **не security** против malicious same-process code; это **discipline** против accidental coupling.

### Versioning

Public field rename/type change breaks compiled callers. Private field + stable property — **evolvable** NuGet API.

### EF Core

Property changes tracked; public field bypass может ломать expectations change tracker.

## Examples

**Email** — ctor-only validation, no setter.  
**ShoppingCart** — private list, `IReadOnlyList`, `Add`.  
**Job** — `Status { get; private set; }`, `Start()` state machine.

---

## Common Mistakes

Public fields for JSON. Public `{ get; set; }` без rules. Getter returns mutable internal collection. God service mutating DTO graphs.

---

## Production

Analyzers **CA1051**. P/Invoke structs — public fields at boundary only. Immutability (`init`, records) в CQRS commands.

---

## Key Takeaways

- Encapsulation — controlled mutation paths.
- Public fields — no validation hook, high coupling.
- Property/method seam — evolution.
- Guard collections and domain vs DTO separation.
