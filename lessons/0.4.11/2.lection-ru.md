# Latency vs Bandwidth vs Throughput

> Roadmap: `0.4.11` · Node: `0.4` — Сети · Depth: понимание

## Learning Objectives

После урока ты сможешь:

- Определить **latency**, **bandwidth**, **throughput** и корректно использовать термины.
- Объяснить, почему **1 Gbps** link feels slow для маленького API request.
- Связать **RTT** с latency и TCP handshake (`0.4.3`).
- Описать, как **CDN** (`0.4.9`) снижает latency для static.
- Определить, problem latency-bound или bandwidth-bound.
- Использовать ms, Mbps, MB/s, RPS.

---

## Why This Matters

«Сеть медленная» без precision → wrong fixes. Upgrade 100→1000 Mbps не поможет API с p95 400 ms из-за cross-region DB. CDN helps static, не serial chain из ten API calls по RTT каждый.

Metrics в DevTools, AWS network caps, load tests (RPS). Путать bandwidth и throughput → surprise на «gigabit» pipe с 80 Mbps single flow.

---

## Core Concepts

### Latency — время ожидания

**Latency** — delay, обычно **ms**. **RTT** — request до server и ack/response bit обратно. Same city ~20 ms; Sydney–London ~250 ms **до** server code.

Latency dominates **interactive** UX: click, TTFB, TCP handshake (+1 RTT, `0.4.3`). Ten serial 50 ms calls = 500 ms minimum при infinite bandwidth.

Components: propagation (physics), processing (TLS, Kestrel, EF), queuing (load).

### Bandwidth — ёмкость pipe

**Bandwidth** — **max data rate** link, **Mbps/Gbps** (bits; 100 Mbps ≈ 12.5 MB/s). Office «500 Mbps» — capacity; не говорит, сколько small packet ждёт в queue.

Helps **large payloads**: video, backups, big bundles. Tiny JSON почти не uses bandwidth; wider pipe не shrinks RTT.

Cloud instances имеют network caps — sizing по bandwidth для heavy transfer.

### Throughput — фактическая скорость

**Throughput** — **what you achieve**: Mbps, MB/s, **RPS**. Throughput ≤ bandwidth always. Limits: latency, loss, CPU, DB locks, connections.

Load test «2000 RPS, 50 ms avg» — throughput при тех conditions. DB contention doubles latency → throughput may halve.

### Interaction

1 KB JSON, 1 Gbps, 100 ms RTT — dominated by **latency** (TCP+TLS+processing). Bandwidth upgrade barely helps.

50 MB file, 10 Mbps, 20 ms RTT — **bandwidth** caps ~1.25 MB/s → ~40 s download.

10k clients × 50 MB — aggregate **throughput** saturates **bandwidth**; latency rises (queuing).

### Связи

TCP handshake, CDN edge, DNS lookup, CORS preflight — всё добавляет RTT/latency.

---

## Under the Hood

**BDP** = bandwidth × RTT — data in flight; high BDP needs large TCP windows для full **throughput**. HTTP/2 multiplexing — one connection, many streams; RTT still matters per response start.

## Examples

2 KB API, p95 300 ms — latency/DB/region, не office bandwidth. 4K video buffering — throughput < encode rate. CDN 200→20 ms RTT per static file.

---

## Common Mistakes

Gigabit internet как proof API fast. Optimize bundle size, ignore serial API waterfall. Bandwidth vs throughput confusion.

---

## Production & Real-World Notes

APM p50/p95/p99. API+DB same region. SLAs на latency; capacity: peak RPS × response size.

---

## Quick Reference

| | Unit | Question |
|---|------|----------|
| Latency | ms | How long wait? |
| Bandwidth | Mbps | Pipe size? |
| Throughput | RPS, MB/s | Actual delivery? |

Small payload → latency. Large file → bandwidth. Many users → throughput capacity.

---

## Key Takeaways

Precise terms → right fixes. Gbps ≠ low RTT. Throughput ≤ bandwidth. Measure, don't assume.
