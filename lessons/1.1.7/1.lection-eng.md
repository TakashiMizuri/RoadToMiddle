# Boxing and Unboxing — When It Happens

> Roadmap: `1.1.7` · Node: `1.1` — C# Language Basics · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Define **boxing** as wrapping a value type in a heap-allocated **`object`** (or compatible interface).
- Define **unboxing** as extracting the value type with a **type-exact cast**.
- Identify **common boxing sites** in everyday C# before generics.
- Read **`box` / `unbox` / `unbox.any`** patterns in IL conceptually.
- Explain **performance and GC impact** of unnecessary boxing.
- Avoid boxing using **generics**, **`Span<T>`**, and API choices — connecting to `1.1.6` value semantics and `1.1.3` primitives.

---

## Why This Matters

In `1.1.6` you learned that an `int` local holds its value inline, while a `string` variable points to a heap object. That model feels clean until you write:

```csharp
object o = 42;
```

Suddenly a primitive from `1.1.3` — a pure value type — becomes a **heap object**. The CLR must allocate, copy the `int` into the object payload, and track it for GC. This is **boxing**, and it is one of the most common hidden allocation sources in .NET code.

Boxing bridges the gap between **value types** and **reference-based APIs** (`object`, non-generic `IList`, `string.Format(object arg0, ...)`). Before generics (`List<int>`), almost every collection boxed your integers. Middle developers still hit boxing through **`object` parameters**, **`interface` dispatch on structs**, **`enum` to `object`**, and some **`string` APIs**.

Understanding boxing turns mysterious GC pressure into predictable engineering: profile, find `box` in IL or allocation traces, replace with generic or specialized APIs.

## Core Concepts

### What Boxing Is

**Boxing** converts a **value type** instance into a **reference type** by creating a new object on the **managed heap** that **contains a copy** of the value.

```csharp
int n = 42;
object boxed = n;   // boxing occurs here
```

After boxing:

1. CLR allocates an object large enough to hold the value plus object header / method table.
2. The **`int` value is copied** into that object (the box).
3. The `object` variable holds a **reference** to the box.

The original `int` local is unchanged; the box is a **separate copy** on the heap.

**Boxing happens implicitly** when a value type is converted to `object` or to an **interface type** implemented by the value type (for example `IComparable`, `IFormattable` on many primitives).

### What Unboxing Is

**Unboxing** extracts the value type from a boxed object back into a **value type variable**:

```csharp
object boxed = 42;
int m = (int)boxed;   // unboxing
```

Rules:

- The boxed object must **actually contain** that exact value type (or compatible nullable unwrap rules).
- You must cast to the **exact underlying type** — `(int)boxed`, not `(long)boxed` on an `int` box.
- Invalid cast throws **`InvalidCastException`**.

Unboxing copies bits from the box payload into the target local — it does **not** remove the box object; GC reclaims it when unreachable.

### The Bridge Problem Boxing Solves

Reference-type APIs pre-generics needed a **common supertype**. **`System.Object`** is that supertype. Value types do not inherit from arbitrary classes, but they **can be boxed** to `object` so legacy APIs (`ArrayList`, `Hashtable`, logging `object value`) could store `int`, `DateTime`, and custom structs uniformly.

Generics (`List<T>`, `Dictionary<TKey,TValue>`) store **`T` directly** for value type type arguments — **no boxing per element** in the collection's storage (with JIT specialization).

### When Boxing Happens — Practical Catalog

**1. Assignment or conversion to `object`:**

```csharp
object o = 1;              // boxes int
object[] arr = { 1, 2 };   // each int element boxes
```

**2. Non-generic collections:**

```csharp
var list = new ArrayList();
list.Add(42);              // boxes int
```

**3. Interface dispatch on struct/primitive without generics:**

```csharp
IComparable c = 10;        // boxes int
void Log(IFormattable f) { }
Log(3.14m);                // boxes decimal
```

**4. `enum` to `object` or non-generic interface:**

```csharp
Day day = Day.Monday;
object o = day;            // boxes enum underlying int
```

**5. `string.Format` / composite formatting with value types:**

```csharp
string s = string.Format("{0}", 42);   // may box 42 as object arg
// Prefer: 42.ToString() or $"{42}" (optimized paths often avoid boxing in modern runtime)
```

**6. Virtual calls on value types through `object` or base `ValueType`/`Enum` APIs** — patterns like `value.GetType()` on unboxed struct vs boxed paths differ; `GetType()` on struct boxes.

**7. Nullable value types:** boxing `int?` with value boxes underlying `int`; **null** `int?` assigns **`null` reference** without boxing a value.

```csharp
int? a = 5;
object o = a;      // boxes 5

int? b = null;
object p = b;      // p is null, no box
```

### When Boxing Does *Not* Happen

- **`List<int>.Add(42)`** — stores `int` directly in array (no box per element).
- **Assignment between same value types** (`int a = b`).
- **Generics with struct type parameter** — JIT generates specialized code operating on `T` inline.
- **`ref struct` types** — cannot be boxed at all (C# 7.2+).
- Most **`$"{intValue}"`** interpolation paths in modern .NET use **`Span`-based formatting** without boxing (still profile critical paths).

Always verify hot paths with **dotnet-trace / allocation profiler** — optimizations evolve per release.

### Unboxing Pitfalls

**Wrong type cast:**

```csharp
object o = 42;
long x = (long)o;   // InvalidCastException — must unbox as (int) first
```

**Boxing null nullable:**

```csharp
int? n = null;
object o = n;       // o is null — not a box of int
int? m = (int?)o;   // OK
```

**Repeated box/unbox in tight loops** — allocation + cache pressure.

## Under the Hood

### IL Instructions

Roslyn emits explicit IL for boxing conversions.

**Boxing** — instruction **`box`** followed by type token:

```csharp
int n = 42;
object o = n;
```

Conceptual IL:

```il
ldc.i4.s 42
box [System.Runtime]System.Int32
stloc.1
```

**`box`** allocates object, copies value from evaluation stack into object payload, pushes **reference** to stack.

**Unboxing** — **`unbox`** or **`unbox.any`** + load:

```csharp
int m = (int)o;
```

Conceptual IL:

```il
ldloc.1
unbox.any [System.Runtime]System.Int32
ldind.i4
stloc.2
```

**`unbox`** yields **managed pointer** to payload (`TypedReference` scenarios); **`unbox.any`** + **`ldind`** pattern loads value. Wrong token → **`InvalidCastException`** at runtime.

You can inspect your own samples with **`dotnet build`** and `ildasm` / ILSpy / dotPeek.

### Object Layout of a Box

A boxed `int` object contains:

- **Method table pointer** (like any reference type)
- **Sync block index** (optional header)
- **Payload**: the **`int` bits** (4 bytes), aligned per CLI rules

Boxing **always copies** — mutating unboxed copy does not change box content unless you use rare `ref` patterns with `TypedReference`.

### Performance and GC

Each **`box`** is typically a **small object allocation** (Gen0). High-frequency boxing (logging every event as `object`, `ArrayList` of numbers) drives **allocation rate** and **GC pauses**.

**Unboxing** is cheaper (no allocation) but still type check + copy.

**Generics** eliminate boxing for `T` storage. **`where T : struct`** constraints still avoid converting to `object` unless you explicitly cast `T` to `object`.

### Connection to `1.1.6`

Boxing is the clearest example of **value types on the heap**: the value leaves its inline slot and becomes **embedded payload** inside a reference type object. After boxing, you have **reference semantics** — aliasing, `null` for nullable references to boxes, GC tracking.

## Syntax / API

```csharp
// Boxing
object o1 = 42;
IComparable c = 42;

// Unboxing
int n = (int)o1;

// Avoid
var list = new List<int>();   // not ArrayList

// enum
enum Status { Active, Inactive }
object o2 = Status.Active;    // boxes

// Nullable
int? maybe = 7;
object o3 = maybe;            // boxes 7
object o4 = (int?)null;       // null reference
```

## Examples

**Example 1 — hidden boxing in API:**

```csharp
void Print(object value) => Console.WriteLine(value);

Print(100);   // boxes int at call site
```

Fix: overload `Print(int value)` or generic `Print<T>(T value)`.

**Example 2 — interface on struct:**

```csharp
public struct Temperature : IFormattable
{
    public double Celsius;
    public string ToString(string? format, IFormatProvider? provider) =>
        Celsius.ToString(format, provider);
}

IFormattable f = new Temperature { Celsius = 36.6 };  // boxes struct
```

Fix: call `ToString` on struct directly or use generic constraint without converting to interface type variable.

**Example 3 — safe unbox:**

```csharp
object unknown = GetObject();
if (unknown is int i)
    Use(i);   // pattern avoids InvalidCastException
```

## Common Mistakes & Anti-patterns

**Using `ArrayList` / non-generic collections** for value types in hot code.

**Storing everything as `object`** in DTOs or event args.

**Casting unbox to wrong numeric type** — `(long)(object)42` fails.

**Assuming `$"{x}"` never boxes** — usually true for primitives in modern .NET, but don't treat as law without profiling exotic types.

**Boxing large structs** — copies entire struct into heap object (expensive).

## Production & Real-World Notes

Legacy BCL APIs (`Console.WriteLine(object)`) accept `object` — passing `int` boxes. For tight loops, prefer **`WriteLine(int)`** overload or **`Utf8Formatter`**.

Logging frameworks (Serilog, etc.) use **`destructuring`** and templates — value types may box if passed as `object`; structured logging with typed templates reduces this.

Analyzers: some Roslyn analyzers flag **`object`** overload usage where typed overload exists.

ASP.NET Core model binding and JSON serialization paths are highly optimized — different from `ArrayList` era; still watch custom code using **`object`**.

## Comparison / Trade-offs

| Approach | Boxing | Flexibility |
|----------|--------|-------------|
| `ArrayList` | Yes for value types | Heterogeneous |
| `List<int>` | No for storage | Homogeneous typed |
| `object` parameter | Often boxes | Single API surface |
| Generic `T` | Avoids for `T` struct | Compile-time type safety |

## Quick Reference

```
Value → object/interface     = box (heap alloc + copy)
(object) → exact value type  = unbox (check + copy)
int? null → object           = null ref, no box
Generics List<T>             = no per-element box for struct T
Wrong unbox cast             = InvalidCastException
IL: box / unbox / unbox.any
```

## Key Takeaways

- **Boxing** wraps value type in heap **object** with **copy** of value.
- **Unboxing** requires **exact type** cast; copies value out.
- Triggered by **`object`**, **non-generic interfaces**, **non-generic collections**, many **`enum`/`object` paths**.
- **`List<T>` / generics** avoid storage boxing for value type `T`.
- IL uses **`box`** and **`unbox`/`unbox.any`** — visible in decompilers.
- Unnecessary boxing increases **allocations and GC** — fix API choice and typing.
