# Exercises: Monorepo vs Polyrepo

> Complete after lectures and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Map your current or sample project
**Goal:** Classify layout.

**Steps:**
1. List components: API, web, mobile, infra, shared lib.
2. Sketch current or hypothetical repo boundaries (boxes + repo names).
3. Label monorepo or polyrepo.

**Success criteria:** One paragraph justification.

---

### E2. Trade-off table from scratch
**Goal:** Internalize comparison.

Fill 5 rows (clone size, atomic PR, CI, permissions, hotfix) with monorepo vs polyrepo one-liners each.

**Success criteria:** No empty cells; at least one row shows disadvantage for each model.

---

## Medium

### M1. Hotfix scenario
**Goal:** Connect `0.2.11` + repo model.

**Scenario:** XSS fix in shared validation library used by API and web.

Write step lists for:
1. Polyrepo (3 repos)
2. Monorepo (single repo, packages/contracts)

**Success criteria:** Polyrepo has ≥3 release steps; monorepo ≤2 tags if convention allows.

---

### M2. CI design sketch
**Goal:** Path filters.

For monorepo `apps/api/**`, `apps/web/**`, draw GitHub Actions pseudo-yaml: when to run `dotnet test` vs `npm test`.

**Success criteria:** Both path filters and a shared `contracts/` change triggering both.

---

## Hard (stretch)

### H1. Architecture decision record (ADR)
Write ADR: "Monorepo for Acme CRM (5 devs, API+React)". Include context, decision, consequences (+/-), rejected polyrepo reason.

**Success criteria:** Mentions atomic change and CI affected builds; honest minus (clone size).

---

### H2. Migration thought experiment
Team has polyrepo API+web with synchronized releases weekly. Argue **for** or **against** merging to monorepo in 800 words max.

**Success criteria:** Addresses anti-pattern "polyrepo with no independence"; no dogma.
