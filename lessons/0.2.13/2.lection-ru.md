# Tags: lightweight vs annotated, semver releases

> Roadmap: `0.2.13` · Узел: `0.2` — Git: ветки и collaboration · Глубина: практика

## Learning Objectives

После лекции ты сможешь:

- Различать **lightweight** и **annotated** tags и выбирать тип.
- Создавать tags: **`git tag`**, **`git tag -a`**, push на remote.
- Объяснять **semver** (MAJOR.MINOR.PATCH) для релизов.
- Использовать tags в deploy, bisect, hotfix.
- Просматривать и **безопасно удалять** tags на remote.
- Связать tags с Conventional Commits (`0.2.9`).

---

## Why This Matters

Ветки **двигаются**; tags помечают **конкретный commit** как release. «Что в prod v2.3.1?» — hash commit'а, лучше через **annotated tag** с message и датой. CI/CD, Docker labels, `git bisect good v2.3.0` опираются на стабильные refs.

Lightweight — закладка. Annotated — release artifact. Semver синхронизирует коммуникацию с `fix:`/`feat:`/`BREAKING` из `0.2.9`. Middle dev создаёт tag после hotfix merge (`0.2.11`), не перетирает опубликованный semver.

## Core Concepts

### Что такое tag

**Tag** — именованный указатель на commit (обычно), **не** двигается с новой работой: `v1.0.0`, `release-2026-07-01`.

```
refs/tags/v2.3.1  →  commit abc1234
refs/heads/main   →  commit def5678  (двигается)
```

Deploy «v2.3.1» = build commit'а tag'а — reproducible rollback target.

### Lightweight tags

```bash
git tag v2.3.1
```

Только ref в `refs/tags/` → hash. Без tagger/message/GPG. OK для local bookmark; **слабо для production release**.

### Annotated tags

```bash
git tag -a v2.3.1 -m "Release 2.3.1: hotfix SEC-401"
```

Полноценный Git object type `tag` с message, tagger, date, optional **GPG** (`-s`, verify `-v`). **`git show v2.3.1`** — tag object + commit. **Production → annotated.**

### Listing и push

```bash
git tag -l "v2.*"
git push origin v2.3.1
git push origin --tags   # осторожно — все local tags
```

Удаление remote (редко, согласованно): `git push origin --delete v2.3.1`.

**Не retag'ай** опубликованный `v2.3.1` на другой commit — ship `v2.3.2`.

### Semver

**MAJOR.MINOR.PATCH:**

- **MAJOR** — breaking changes.
- **MINOR** — backward-compatible features.
- **PATCH** — backward-compatible fixes.

| Commit | Bump |
|--------|------|
| fix: | PATCH |
| feat: | MINOR |
| BREAKING / feat! | MAJOR |

Tag **после** green CI на merge commit в `main`/release branch:

```bash
git tag -a v2.3.2 -m "v2.3.2: login hotfix"
git push origin v2.3.2
```

## Under the Hood

Lightweight: файл ref → commit hash.

Annotated: ref → **tag object** → commit. `git cat-file -t` показывает `tag` vs `commit`.

**Peeling:** `git rev-parse v2.3.1^{commit}` — commit за annotated tag.

Tags в namespace `refs/tags/`; checkout tag → detached HEAD (`0.1.11`).

## Syntax / Commands / API

```bash
git tag -a v1.0.0 -m "Release"
git tag -a v1.0.0 <commit-hash>
git push origin v1.0.0
git tag -d v1.0.0
git log v1.0.0..v1.1.0
```

## Common Mistakes & Anti-patterns

Lightweight для semver prod; tag до CI; retag published version; забытый push tag; tag не на том commit (feature tip вместо merge).

---

## Production & Real-World Notes

GitHub/GitLab Releases; protected tags; LTS + cherry-pick + patch tags; monorepo может tag'ать `api/v1.2.0` (`0.2.15`).

## Comparison / Trade-offs

| Тип | Metadata | Когда |
|-----|----------|-------|
| Lightweight | Нет | Local |
| Annotated | Message, GPG | Release |
| Branch | Двигается | Dev |

Semver vs calver — team convention.

---

## Quick Reference

`git tag -a vX.Y.Z -m "..."` → `git push origin vX.Y.Z`. Не retag published semver.

---

## Key Takeaways

- Tags = **immutable** release points.
- **Annotated** для prod; lightweight для local.
- **Semver** + Conventional Commits.
- Push tag явно; hotfix заканчивается tag + push.
