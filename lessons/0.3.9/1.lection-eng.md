# Cookies: Set-Cookie Attributes and SameSite

> Roadmap: `0.3.9` · Node: `0.3` — HTTP and the Web · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Distinguish **Set-Cookie** (response instruction) from **Cookie** (request echo) introduced in `0.3.7`.
- Explain each major attribute: **Expires/Max-Age**, **Domain**, **Path**, **Secure**, **HttpOnly**, **SameSite**.
- Predict when browsers **send** or **withhold** cookies on cross-site requests.
- Describe **SameSite=Lax/Strict/None** trade-offs for CSRF vs OAuth/deep links.
- Configure cookie authentication safely in ASP.NET Core with HTTPS from `0.3.8`.
- Compare cookie sessions vs **Authorization: Bearer** for SPA architecture.

## Why This Matters

HTTP is **stateless** by design (`0.3.10` expands this). Servers identify returning users by issuing a **session identifier** — often stored in a **cookie**. The browser automatically attaches matching cookies on subsequent requests (`Cookie` header), enabling login sessions, shopping carts, and analytics without putting tokens in every URL.

Cookies are also the **#1 XSS and CSRF battlefield**. Steal `session_id` via injected JavaScript and you impersonate the user. Trick a browser into sending cookies on a malicious cross-site form and you perform actions as the victim. Modern browsers tightened defaults: **`SameSite=Lax`** by default, **`Secure` required for `SameSite=None`**, **`HttpOnly`** to block script access.

Middle developers configure `Set-Cookie` in ASP.NET Core cookie authentication, debug "works in Postman, not in browser" (SameSite + CORS + credentials), and choose between cookie-based BFF patterns vs Bearer tokens. Misconfigured cookies cause silent login failures after Chrome updates — not mysterious "browser bugs."

## Core Concepts

### How Cookies Work: Set-Cookie → Storage → Cookie

1. Server responds with **Set-Cookie** header(s).
2. Browser **stores** name=value plus scope rules (domain, path, expiry).
3. On matching future requests, browser sends **Cookie** header automatically.

```
HTTP/1.1 200 OK
Set-Cookie: session=abc123; Path=/; HttpOnly; Secure; SameSite=Lax

--- next request ---

GET /dashboard HTTP/1.1
Cookie: session=abc123
```

Multiple cookies → multiple Set-Cookie lines (not comma-separated in modern HTTP). Server reads `Cookie` in middleware; ASP.NET Core decrypts/auth cookie payload.

Cookies are **not** secret channels — users can inspect them in DevTools. Treat session id as **opaque pointer** to server-side session or signed/encrypted token — never store raw passwords in cookies.

### Expires and Max-Age: Session vs Persistent

**Session cookie** (no Expires/Max-Age): deleted when browser **session** ends (often when last window closes — behavior varies).

**Persistent cookie:**

```
Set-Cookie: theme=dark; Max-Age=2592000
Set-Cookie: remember=1; Expires=Wed, 01 Jul 2026 12:00:00 GMT
```

**Max-Age** (seconds from now) preferred over **Expires** (absolute HTTP-date) — avoids clock skew confusion.

Shorter TTL on auth cookies limits theft window. **Refresh tokens** in HttpOnly cookies with rotation — advanced auth patterns.

### Domain and Path Scope

**Path=/admin** — cookie sent only to URLs under `/admin`.  
**Domain=.example.com** — cookie sent to `example.com` and subdomains (leading dot legacy semantics; modern browsers treat subdomain scope carefully).

Broader Domain increases exposure — set **minimum scope** needed. Avoid `Domain` on public suffix domains (hosting platforms) — browser blocks.

Default: **host-only** cookie for exact host that set it (no subdomain sharing) — safer default.

### Secure: HTTPS Only

**Secure** attribute: browser sends cookie **only on secure requests** (HTTPS). Mandatory companion for production auth cookies after `0.3.8`.

On HTTP development localhost, browsers often treat `http://localhost` as **trustworthy** exception — know environment differences.

**SameSite=None** **requires Secure** — browsers reject None without Secure.

### HttpOnly: Keep JavaScript Out

**HttpOnly** — cookie **inaccessible** to `document.cookie` and most JS APIs. Mitigates **XSS token theft** — attacker script cannot exfiltrate session cookie easily.

Does **not** stop CSRF — browser still sends cookie on qualifying requests. Does not stop XSS from **performing actions as user** within the page — but blocks simple cookie string theft.

ASP.NET Core auth cookies should always be **HttpOnly** unless you deliberately expose API tokens to JS (generally avoid).

### SameSite: Cross-Site Sending Policy

**Same-site** roughly means same **registrable domain** (eTLD+1) — `app.example.com` and `api.example.com` are same-site; `evil.com` is cross-site.

**SameSite=Strict** — cookie **never** sent on cross-site requests (even top-level navigation from external link in some cases). Strong CSRF protection; breaks flows like "email link opens logged-in dashboard" until second navigation.

**SameSite=Lax** (modern default) — cookie withheld on **cross-site subresource** requests (img, iframe, XHR from evil.com) but sent on **top-level GET navigation** (click link from email). Balanced default for session cookies.

**SameSite=None** — cookie sent on **all** cross-site requests **if Secure**. Needed for embedded widgets, some OAuth/SAML flows, cross-site `fetch` with `credentials: 'include'`. Requires explicit CSRF defenses (anti-forgery tokens, double-submit, custom headers).

```
Evil.com form POST → bank.com/transfer
SameSite=Lax/Strict: session cookie often NOT sent → CSRF blocked
SameSite=None + Secure: cookie sent → need CSRF token
```

Third-party cookie phase-out in browsers affects ads/trackers; **first-party** auth cookies with Lax remain standard for same-site SPAs.

### Cookie Size and Count Limits

Browsers limit ~**4KB per cookie**, ~**50 cookies per domain** (approximate). Large JWTs in cookies break — another reason for session id + server store or chunked patterns.

## Under the Hood

### Browser Cookie Jar Algorithm (Simplified)

On response:

1. Parse Set-Cookie name=value and attributes.
2. Reject if **Secure** required but response was HTTP (except localhost quirks).
3. Reject **SameSite=None** without Secure.
4. Store in cookie jar keyed by (name, domain, path).
5. Evict expired cookies.

On request:

1. Collect cookies matching URL (scheme secure check, domain path match).
2. Apply **SameSite** context: cross-site vs same-site, request type (navigate vs XHR).
3. Serialize **Cookie** header.

Changes in Chrome/Firefox SameSite defaults (2020+) broke cross-subdomain logins until teams set explicit `SameSite=None; Secure` or unified domains.

### ASP.NET Core Cookie Authentication

```csharp
builder.Services.AddAuthentication(CookieAuthenticationDefaults.AuthenticationScheme)
    .AddCookie(options =>
    {
        options.Cookie.HttpOnly = true;
        options.Cookie.SecurePolicy = CookieSecurePolicy.Always;
        options.Cookie.SameSite = SameSiteMode.Lax;
    });
```

Cookie is **encrypted/signed** by Data Protection API — wire value not plain user id. **`Antiforgery`** uses separate cookie + token for CSRF on cookie-auth form posts.

SPA on `app.example.com` calling API on `api.example.com`: cross-site — cookie auth needs `SameSite=None; Secure` + CORS `AllowCredentials` + explicit CSRF strategy, or switch to Bearer tokens.

### Interaction with CORS and credentials

```javascript
fetch('https://api.example.com/me', { credentials: 'include' });
```

Requires server `Access-Control-Allow-Credentials: true` and **non-wildcard** `Allow-Origin`. SameSite rules still apply — credentialed cross-site XHR blocked unless None.

## Syntax / Commands / API

### Set-Cookie examples

```
Set-Cookie: SESSION=opaque; Path=/; HttpOnly; Secure; SameSite=Lax

Set-Cookie: prefs=dark; Max-Age=31536000; Path=/; SameSite=Lax

Set-Cookie: embed=token; Path=/; Secure; SameSite=None
```

### Inspect

Browser DevTools → Application → Cookies.  
curl does not persist cookies unless `-c`/`-b` cookie jar.

## Examples

### Example 1: Login session

POST `/login` over HTTPS → `Set-Cookie: .AspNetCore.Cookies=...; HttpOnly; Secure; SameSite=Lax`. Subsequent same-site GET `/dashboard` includes cookie → 200. Evil.com XHR POST to your API → cookie **not** sent (Lax) → CSRF mitigated for API JSON (if CORS also blocks).

### Example 2: OAuth cross-site

Auth callback on `auth.example.com`, app on `app.example.com` — may need `SameSite=None; Secure` on correlation cookie during flow; tighten after login.

### Example 3: XSS without HttpOnly

Attacker injects script; reads `document.cookie` for non-HttpOnly session → account takeover. **HttpOnly** blocks read — still patch XSS.

## Common Mistakes & Anti-patterns

**Session cookie without HttpOnly/Secure.** XSS and network theft.

**SameSite=Lax API on cross-origin SPA** — silent auth failure; fix architecture or None+CSRF.

**Storing JWT with sensitive claims in non-HttpOnly cookie for JS.** XSS wins everything.

**Huge cookies** — 4xx errors, perf hit every request.

**Domain=.com** or overly broad scope — excessive exposure.

---

## Production & Real-World Notes

Rotate session on privilege change. Invalidate server-side session on logout (not just client delete). Monitor cookie size. Document cookie inventory for GDPR consent where required (analytics vs essential).

**BFF pattern:** SPA talks same-origin to backend; backend holds HttpOnly session — avoids browser third-party cookie issues.

## Comparison / Trade-offs

| Approach | Pros | Cons |
|----------|------|------|
| HttpOnly session cookie | Automatic send, XSS-theft harder | CSRF care, cross-site complexity |
| Bearer in memory | No automatic cross-site send | XSS steals from memory; refresh patterns |
| SameSite=Strict | Strong isolation | Breaks some legit cross-site flows |
| SameSite=Lax | Default balance | Some CSRF via top-level GET rare edge cases |
| SameSite=None | Cross-site embed/OAuth | Requires Secure + CSRF defenses |

## Quick Reference

| Attribute | Effect |
|-----------|--------|
| Expires / Max-Age | Lifetime (persistent vs session) |
| Domain | Subdomain scope (careful) |
| Path | URL path scope |
| Secure | HTTPS only |
| HttpOnly | No JS access |
| SameSite=Strict | No cross-site send |
| SameSite=Lax | Block cross-site subresources; allow top GET nav |
| SameSite=None | Cross-site if Secure |

## Key Takeaways

- **Set-Cookie** instructs storage; **Cookie** echoes on matching requests.
- **HttpOnly + Secure + SameSite=Lax** — baseline for auth cookies on HTTPS.
- **SameSite=None** needs **Secure** and **CSRF** strategy.
- Cross-origin SPA + cookie auth is hard — consider Bearer or BFF.
- Cookies complement but do not replace TLS (`0.3.8`) and app authorization (`0.3.7`).
