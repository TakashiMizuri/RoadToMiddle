## Learning Objectives

After this lesson you will be able to:

- Distinguish a **class** (blueprint / type) from an **object instance** (runtime entity on the heap).
- Explain **reference semantics**: assignment copies a reference, not the object body.
- Predict **aliasing behavior** when two variables refer to the same instance.
- Describe what **`new`** does at a high level and what **`null`** means for a reference variable.
- Connect **`1.1.6`** reference-type memory model to everyday OOP code.
- Read a simple class diagram and map it to C# fields, methods, and instances.

## Why This Matters

In `1.1.6` you learned that reference types store a pointer-like **reference** to heap data, and assignment **aliases** objects instead of cloning them. Until now, most examples used built-in types like `List<int>` or `string`. Real applications model **domain concepts** — `Customer`, `Order`, `InvoiceLine` — with **`class`**.

A class is not a thing you "run." It is a **blueprint**: a named contract that describes **state** (fields) and **behavior** (methods). An **instance** is the live object created from that blueprint at runtime. Confusing the two leads to classic junior mistakes: putting business logic in static helpers while forgetting instance state, comparing two references with `==` when you meant value equality, or assuming `customerCopy = customer` duplicated a customer when you only duplicated the address.

Middle developers think in three layers at once: **source code** (class definition), **runtime object** (heap instance with identity), and **variables** (reference slots that may or may not point to an instance). ASP.NET Core controllers, EF Core entities, React DTOs deserialized to C# classes — all assume you understand this triangle.

## Core Concepts

### Class as Blueprint

A **class** declares a new reference type. It groups data and operations under one name:

```csharp
public class BankAccount
{
    public string Owner { get; set; }
    public decimal Balance { get; set; }

    public void Deposit(decimal amount)
    {
        Balance += amount;
    }
}
```

This code does **not** create an account in memory yet. It tells the compiler and CLR: "Objects of type `BankAccount` have at least these members." The class lives in **metadata** (assembly). Instances appear only when you **construct** them.

Think of the class as an architectural floor plan. The plan is not a house; you can build many houses from one plan, each with its own furniture inside.

### Object Instance and Identity

An **object instance** (or simply **object**) is a concrete chunk of memory on the **managed heap** created for a reference type. You typically create it with **`new`**:

```csharp
var account = new BankAccount { Owner = "Ada", Balance = 100m };
```

After this line, three ideas exist:

1. **Type** — `BankAccount` (shared by all accounts).
2. **Object** — one heap instance with its own `Owner` and `Balance` values.
3. **Variable** — `account`, a local reference slot pointing at that object.

Each `new BankAccount()` produces a **new object** with a **new identity**, even if field values match another account. Two accounts both named `"Ada"` with balance `100` are still **two objects** unless your code intentionally shares one reference.

### Reference Semantics in Action

Variables of a class type hold **references**, not embedded object bodies. Assignment copies the reference:

```csharp
var a = new BankAccount { Owner = "Ada", Balance = 50m };
var b = a;                    // b points to the same object as a
b.Deposit(25m);
Console.WriteLine(a.Balance); // 75 — shared mutation
```

This is exactly the `List<int>` story from `1.1.6`, now applied to your own types. **`a` and `b` are aliases** for one object. There is still only **one** `BankAccount` instance on the heap; you merely have two names for it.

If you need an independent copy, you must implement copying explicitly (copy constructor, factory, record `with`, manual clone). The language does not deep-clone classes by default.

### `null`: Absence of an Instance

A reference variable can hold **`null`**, meaning it does not currently refer to any object:

```csharp
BankAccount? savings = null;
// savings.Owner;  // NullReferenceException if dereferenced
```

`null` is not an empty object; it is **no target**. This differs from value types where a variable always holds a value bit pattern (default `0` for `int`). Nullable reference types (`BankAccount?`) and the `#nullable` context help the compiler warn you before dereferencing.

Production code checks `null` at boundaries: API inputs, database lookups, deserialization. **`?.` and `??`** operators express safe navigation, but the underlying model is unchanged — references may aim at nothing.

### Methods Operate on Instances

Instance methods receive an implicit **`this`** reference — the object the call runs on:

```csharp
account.Deposit(10m);   // "this" inside Deposit is that account instance
```

Different instances keep separate state:

```csharp
var x = new BankAccount { Balance = 10m };
var y = new BankAccount { Balance = 20m };
x.Deposit(5m);   // x.Balance == 15, y.Balance still 20
```

The **method table** is shared (defined by the class); **field values** live per instance. That separation is the heart of OOP: one definition, many independent states.

### Diagram: Blueprint vs Instances

```
        CLASS (metadata)                 HEAP (runtime)
   ┌─────────────────────┐         ┌──────────────────┐
   │ BankAccount         │         │ Instance #1      │
   │  Owner : string     │  new    │  Owner → "Ada"   │◄── ref account
   │  Balance : decimal  │ ──────► │  Balance = 100   │
   │  Deposit(...)       │         └──────────────────┘
   └─────────────────────┘         ┌──────────────────┐
                                   │ Instance #2      │
                                   │  Owner → "Bob"   │◄── ref backup
                                   │  Balance = 50    │
                                   └──────────────────┘

   var backup = account;  // two refs → ONE instance (alias)
```

## Under the Hood

### From Source to Heap Object

When JIT compiles `new BankAccount()`, the CLR:

1. Resolves the **type handle** and **method table** for `BankAccount`.
2. Allocates memory on the **managed heap** sized for the object header plus instance fields (`Owner` reference slot, `decimal` value embedded inline).
3. Initializes the object header (sync block index, method table pointer).
4. Runs the **constructor** chain to set initial field values.
5. Returns a **managed reference** stored in the local variable slot.

The **class definition** itself is not duplicated per instance. Metadata (field offsets, method signatures, attributes) lives in the **assembly** and is shared. Each instance stores **its own field values** according to that layout.

### Object Header and Method Table

Every reference-type object on the heap begins with a small **header** pointing to a **method table** (type information used for virtual dispatch and casting). Instance methods use `this` as a pointer to that object; virtual calls consult the method table at runtime (`1.2.7` will go deeper).

Field layout is deterministic: reference fields are pointer-sized slots; value-type fields like `decimal` are embedded **inside** the object body (still value semantics for the decimal bits, but stored in the heap object as part of its payload).

### GC Roots and Aliasing

The garbage collector reclaims heap objects unreachable from **GC roots** (stack references, static fields, handles). When `a` and `b` alias the same account, both references keep **one** object alive. When **all** references disappear, the object becomes eligible for collection.

Copying references does **not** allocate a new object. **`var b = a`** is cheap — pointer copy. **`new BankAccount()`** allocates. Confusing those costs matters in hot paths.

### Reference Equality vs Value Equality

By default, **`==` for classes** (unless overridden) compares **references** — same object or not. Two distinct instances with identical field values are **not** equal under reference equality:

```csharp
var c1 = new BankAccount { Owner = "Ada", Balance = 0m };
var c2 = new BankAccount { Owner = "Ada", Balance = 0m };
Console.WriteLine(c1 == c2);   // false — different objects
Console.WriteLine(c1 == c1);   // true
```

Value-based equality requires explicit design (`1.2.16`, records). Reference semantics explain why.

## Syntax / Commands / API

```csharp
// Class declaration
public class Customer
{
    public int Id { get; set; }
    public string Name { get; set; } = string.Empty;
}

// Instance creation
Customer c = new Customer();
Customer c2 = new() { Id = 1, Name = "Lin" };   // target-typed new

// Reference assignment (alias)
Customer alias = c2;

// Null
Customer? maybe = null;

// Identity check (reference)
bool same = ReferenceEquals(c2, alias);   // true
```

## Examples

**Example 1 — two instances, independent state:**

```csharp
var first = new BankAccount { Owner = "A", Balance = 10m };
var second = new BankAccount { Owner = "B", Balance = 10m };
second.Deposit(5m);
// first.Balance == 10, second.Balance == 15
```

Two `new` calls ⇒ two heap objects. Same initial numbers do not imply sharing.

**Example 2 — alias through parameter:**

```csharp
void AddBonus(BankAccount acc)
{
    acc.Deposit(100m);
}

var mine = new BankAccount { Balance = 0m };
AddBonus(mine);
// mine.Balance == 100 — parameter copied reference, not object
```

Default pass-by-value for references copies the **reference** into the parameter slot (`1.1.6`). Callee mutates the shared object.

**Example 3 — reassignment does not affect caller's other reference:**

```csharp
void Replace(BankAccount acc)
{
    acc = new BankAccount { Balance = 999m };
}

var original = new BankAccount { Balance = 1m };
Replace(original);
// original.Balance still 1 — local parameter reassigned, caller unchanged
```

## Common Mistakes & Anti-patterns

**Treating class variables like value copies.** `var copy = customer` does not clone; it aliases.

**Assuming equal field values mean equal objects.** Reference equality and value equality diverge unless you override `Equals`.

**Forgetting `null` at API edges.** Dereference without check ⇒ `NullReferenceException` — still one of the most common production exceptions.

**Creating unnecessary instances in loops.** Each `new` hits the allocator; sometimes you reuse a builder or clear state instead (context-dependent).

## Production & Real-World Notes

EF Core **tracks entity instances by reference** in the DbContext identity map. Two variables pointing to the same tracked `Order` see one persisted row; attaching duplicates requires understanding instance identity.

Dependency injection registers **services as types** but resolves **instances** per lifetime (Singleton one instance, Scoped per request). Confusing type registration with object identity causes subtle bugs in custom caches.

JSON deserialization creates **new objects**; two deserialized graphs are unrelated references even if JSON text matched.

## Comparison / Trade-offs

| Concept | Class (blueprint) | Instance (object) | Reference variable |
|---------|-------------------|-------------------|---------------------|
| Exists when | Compile time / load time | After `new` (or other creation) | Always in scope |
| Count per app | One definition per type | Many | Many slots |
| Assignment | N/A (types are not assigned) | N/A | Copies reference |
| Mutations | N/A | Affects shared aliases | Reassigning variable points elsewhere |

Classes vs **structs** (`1.2.14`): classes are reference types — identity, heap allocation, inheritance. Structs are value types — copying semantics differ.

## Quick Reference

```
class          → type blueprint (metadata)
new T()        → heap instance + reference returned
var a = b      → alias same object (reference copy)
null           → no instance targeted
== (default)   → reference equality for classes
this           → current instance in instance method
```

## Key Takeaways

- A **class** defines structure and behavior; an **instance** is a live heap object with its own state.
- Class-type variables store **references**; assignment **aliases** objects (`1.1.6`).
- **`new`** allocates a new instance; **`null`** means no instance.
- Instance methods run with a **`this`** reference; state is per object, methods shared via type metadata.
- **Reference equality** is default; value equality must be designed explicitly.
- GC tracks **objects**, not class definitions; aliases affect reachability.
