# Task.Factory.StartNew — почему избегать, использовать Task.Run

> Roadmap: `1.4.24` · Node: `1.4` — Async + ASP.NET preview · Depth: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Объяснить overload resolution **Task.Factory.StartNew** и defaults **TaskScheduler** / **TaskCreationOptions**.
- Найти баг **StartNew + async lambda** → **`Task<Task>`** без **Unwrap**.
- Выбирать **`Task.Run`** для CPU-bound offload с предсказуемыми defaults.
- Описать, когда **LongRunning** и custom schedulers — legitimate use **StartNew**.
- Связать misuse **StartNew** с **thread pool starvation** и **SyncContext**.
- Рефакторить legacy **StartNew** в **Task.Run** или true async I/O.

---

## Why This Matters

`Task.Factory.StartNew` (.NET 4.0) — гибкое scheduling API. Правило после years of incidents: **`Task.Run` для thread pool**, **StartNew только** при явной нужде в non-default scheduler/options. Defaults StartNew удивляют: scheduler может быть **не thread pool**. Хуже: **`StartNew(async () => await X())`** возвращает **`Task<Task>`** — outer complete до завершения inner async work.

В ASP.NET Core **Task.Run** для CPU иногда OK; **StartNew с wrong options** → race, premature response. Middle dev распознаёт StartNew в legacy и мигрирует safely.

## Core Concepts

### StartNew и ловушка async lambda

```csharp
Task<Task> outer = Task.Factory.StartNew(async () =>
{
    await client.GetStringAsync(url);
});
// outer.IsCompleted true до конца HTTP!
```

Fix:

```csharp
await Task.Factory.StartNew(
    () => client.GetStringAsync(url),
    CancellationToken.None,
    TaskCreationOptions.DenyChildAttach,
    TaskScheduler.Default).Unwrap();

// Preferred:
await Task.Run(async () => await client.GetStringAsync(url));
```

**Task.Run** ≈ StartNew на **TaskScheduler.Default** с safer defaults для common cases.

### Task.Run — когда

CPU-bound блокирует request thread → `Task.Run(() => HeavyCpu(), token)`.

**Не** для I/O — `await` truly async API.

**LongRunning** — dedicated thread для blocking legacy loop — редко.

### Legitimate StartNew

UI marshal: **FromCurrentSynchronizationContext**. **LongRunning** + blocking listener. Framework internals — не typical controllers.

---

## Under the Hood

**Task.Run** queue через thread pool. **StartNew** создаёт Task с scheduler reference; **`ScheduleAndStart`**.

**Unwrap()`** flatten **`Task<Task>`** → **`Task`**. **Task.Run** с async delegate делает это корректно.

---

## Syntax / Commands / API

Migration:

```csharp
Task.Factory.StartNew(() => DoCpu(), token);
// →
Task.Run(() => DoCpu(), token);
```

Broken async:

```csharp
Task.Factory.StartNew(async () => await DoIo());
// →
await DoIo();
```

## Examples

**Task<Task> surprise** — await outer завершается сразу.

**ASP.NET CSV parse** — `CopyToAsync` без Run; `ParseCsv` в `Task.Run` с **RequestAborted**.

**LongRunning listener** — StartNew + **LongRunning**, не для HTTP.

## Common Mistakes & Anti-patterns

StartNew + async без Unwrap. Думать, что всегда thread pool. Task.Run на каждый метод. StartNew вместо fix sync-over-async.

---

## Production & Real-World Notes

Code review flag **StartNew** → **Task.Run** или justify. Long work — **BackgroundService** + **Channel**. Port legacy **Parallel + StartNew** carefully.

---

## Comparison / Trade-offs

| | Task.Run | StartNew |
|---|----------|----------|
| Intent | Pool queue | Full control |
| Footguns | Меньше | Task<Task>, scheduler |
| App default | **Да** | **Legacy** |

---

## Quick Reference

- Rule: **Task.Run** для pool.
- **Never** StartNew(async ...) без **Unwrap**.
- I/O: **await**, не Run wrapper на sync HTTP.
- ASP.NET CPU: Task.Run + **RequestAborted**.

---

## Key Takeaways

- **StartNew** confusing — **Task.Run** safe default.
- **async lambda** → **Task<Task>** trap.
- **Task.Run** не делает I/O async.
- **LongRunning** — rare, blocking legacy.
- ASP.NET: CPU **Run**, I/O **await**, background **BackgroundService**.
