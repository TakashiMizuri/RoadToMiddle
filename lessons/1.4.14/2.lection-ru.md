# .Result, .Wait(), GetAwaiter().GetResult() — deadlocks

> Roadmap: `1.4.14` · Node: `1.4` — Async/await · Depth: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Объяснить **sync-over-async**: block на **`Task`**, пока async work нужен **blocked thread** для **`MoveNext`** (`1.4.7`).
- Сравнить **`.Wait()`**, **`.Result`**, **`GetAwaiter().GetResult()`** — exceptions.
- Диагностировать **SyncContext deadlock** (`1.4.9`) vs **pool starvation** (`1.4.10`).
- Fix: **async all the way**, library **`ConfigureAwait(false)`** (`1.4.8`).
- Refactor legacy без sync-over-async.

---

## Why This Matters

Legacy и SDK толкают к **`.Result`** в controllers. Under load — **hangs**. Blocker ждёт continuation, который не может run — SyncContext или pool.

---

## Core Concepts

### Sync-over-async

```csharp
GetDataAsync().Result;
GetDataAsync().Wait();
GetDataAsync().GetAwaiter().GetResult();
```

Async method на первом **`await`** schedule'ит **`MoveNext`**. Blocker **держит thread**. Если тот же thread нужен для continuation — **deadlock** / stall.

### SyncContext deadlock (UI / Framework)

1. UI thread **`.Result`**
2. Await с capture SyncContext
3. Continuation **Post** на UI
4. UI **blocked** — deadlock

**ConfigureAwait(false)** в library ломает цикл, но fix — **async handler** (`1.4.8`).

### Pool starvation (Core)

Нет SyncContext, но **`.Wait()`** block pool thread → starvation (`1.4.10`).

### API comparison

| API | Exception |
|-----|-----------|
| `.Wait()` | AggregateException |
| `.Result` | unwrap inner |
| `GetResult()` | direct throw |

Все **block** одинаково для deadlock analysis.

---

## Examples

```csharp
// BAD
label.Text = _api.GetLabelAsync().Result;

// GOOD
label.Text = await _api.GetLabelAsync();
```

---

## Key Takeaways

- Block = sync-over-async.
- SyncContext: blocked thread = Post target (`1.4.7`, `1.4.9`).
- Core: blocked pool (`1.4.10`).
- Fix **`await`** (`1.4.20`).
- GetResult ≠ safe from deadlock.
