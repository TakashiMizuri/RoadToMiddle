# Control Flow: if/else and switch (Classic + Expression)

> Roadmap: `1.1.9` · Node: `1.1` — C# Language Basics · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Write **conditional branches** with `if`, `else if`, and `else` to route program execution based on runtime data.
- Evaluate **boolean expressions** correctly, including how **short-circuit** evaluation affects side effects.
- Choose between **classic `switch`** and **`switch` expressions** for multi-way decisions.
- Use **pattern matching** in modern `switch` to match types, constants, and relational conditions.
- Avoid common pitfalls such as **dangling else**, **assignment instead of comparison**, and **missing `break`** in classic switch.

---

## Why This Matters

In `1.1.8` you learned operators — the vocabulary for asking questions about data. Operators produce **boolean** answers: is this score passing? is this user an admin? is the input null? Control flow is what you do **with** those answers. Without branching, every program would run the same path from top to bottom like a tape recorder. Real applications constantly decide: validate input, choose an HTTP status code, pick a pricing tier, skip empty rows in a report.

As a fullstack developer you will write conditionals in controllers (`if (ModelState.IsValid)`), in domain logic (`if (order.Total > freeShippingThreshold)`), and in React-like UI code on the frontend — but in C# the mechanics start here. Middle developers do not merely make code compile; they choose structures that **communicate intent**. A five-level `if/else` chain that should be a `switch` on an enum is a readability smell. A `switch` expression that returns a value is often clearer than mutating a variable in each branch. This lesson gives you both the classic tools and the modern C# idioms teams expect in production codebases.

## Core Concepts

### Execution Paths and the `if` Statement

A C# program is a sequence of statements, but **control flow statements** change which statement runs next. The simplest branch is **`if`**: when a boolean condition is `true`, execute a block; otherwise skip it.

```csharp
int score = 72;

if (score >= 60)
{
    Console.WriteLine("Pass");
}
```

The condition inside parentheses must be **`bool`**. C# does not treat `0` or `1` as false/true the way C does — that strictness prevents an entire class of bugs. When the condition is true, the runtime enters the **block** delimited by `{ }`. If you have exactly one statement, braces are optional in C#, but teams almost always require them in style guides because a second added line might accidentally stay outside the branch.

**`else`** runs when the `if` condition is false:

```csharp
if (score >= 60)
{
    Console.WriteLine("Pass");
}
else
{
    Console.WriteLine("Fail");
}
```

**`else if`** chains multiple tests in order. The runtime evaluates from top to bottom and executes **only the first** matching branch:

```csharp
if (score >= 90)
    Console.WriteLine("A");
else if (score >= 80)
    Console.WriteLine("B");
else if (score >= 70)
    Console.WriteLine("C");
else
    Console.WriteLine("Below C");
```

Order matters. If you reversed the first two checks, a score of 95 would incorrectly become `"B"` because `>= 80` would match first. This is not a compiler error — it is a **logic bug**, which makes careful ordering part of the craft.

### Boolean Conditions and Short-Circuit Evaluation

Conditions often combine comparisons with **logical operators** from `1.1.8`. Remember that **`&&` and `||` short-circuit**: the right-hand side may never run.

```csharp
string? name = null;

if (name != null && name.Length > 0)
{
    Console.WriteLine(name);
}
```

Because `name != null` is false, C# does **not** evaluate `name.Length`, avoiding a `NullReferenceException`. This pattern appears constantly in production code. Conversely, if you need **both** sides evaluated (rare), use `&` and `|` on booleans — but default to `&&` and `||`.

Watch the classic mistake **`if (x = 5)`** — assignment instead of comparison. C# rejects it for `bool` contexts in most cases, but **`if (flag = GetFlag())`**-style patterns still bite juniors who confuse `=` with `==`.

### The Ternary Conditional Operator

When you need to choose between **two values** (not two arbitrary statement blocks), the **ternary operator** `? :` keeps code compact:

```csharp
string label = score >= 60 ? "Pass" : "Fail";
```

Read it as: *if condition, then this value, else that value*. Ternary expressions are expressions — they produce a value you can assign, pass as an argument, or return. Do not nest five levels deep; at that point an `if/else` or `switch` is clearer. In APIs you will see ternaries for simple mappings: `var status = isActive ? UserStatus.Active : UserStatus.Inactive;`.

### Classic `switch`: Multi-Way Branching

When a single variable (or expression) must be compared against **many constant cases**, **`switch`** is often clearer than a long `if/else if` chain:

```csharp
DayOfWeek day = DateTime.Today.DayOfWeek;

switch (day)
{
    case DayOfWeek.Saturday:
    case DayOfWeek.Sunday:
        Console.WriteLine("Weekend");
        break;
    case DayOfWeek.Monday:
        Console.WriteLine("Start of work week");
        break;
    default:
        Console.WriteLine("Midweek");
        break;
}
```

Each **`case`** labels a branch. **`break`** exits the switch — unlike C/C++, C# does **not** fall through to the next case unless you explicitly use **`goto case`** (almost never in application code). **`default`** catches no match; it is optional but recommended when you want defensive handling.

The **`switch` expression** must be compatible with case labels: numeric types, `char`, `string`, enums, and (in modern C#) patterns. You cannot switch on floating-point types with exact equality safely — `0.1 + 0.2 == 0.3` is false in IEEE floating point — so use ranges or integers instead.

### `switch` Expressions (C# 8+)

C# 8 introduced **`switch` expressions** — a concise, expression-oriented form that **returns a value**:

```csharp
string mood = day switch
{
    DayOfWeek.Saturday or DayOfWeek.Sunday => "Relax",
    DayOfWeek.Monday => "Coffee",
    _ => "Work"
};
```

Note the syntax: **`value switch { patterns => results }`**. The underscore **`_`** is a **discard pattern** meaning "anything else" (like `default`). **`or`** combines patterns. Because this is an expression, every arm must produce a compatible type, and the compiler can warn if you miss a case when switching on an **enum** without `_`.

Compare imperative vs expression style:

```csharp
// Classic: mutate a variable
string tier;
switch (score)
{
    case >= 90: tier = "Gold"; break;
    case >= 70: tier = "Silver"; break;
    default:    tier = "Bronze"; break;
}

// Expression: assign once
string tier = score switch
{
    >= 90 => "Gold",
    >= 70 => "Silver",
    _     => "Bronze"
};
```

The expression form often reduces bugs from uninitialized variables and makes **exhaustiveness** easier to reason about.

### Pattern Matching in `switch`

Modern C# treats `switch` as **pattern matching**, not only constant equality:

```csharp
object input = "hello";

string description = input switch
{
    null       => "Missing input",
    string s   => $"Text with length {s.Length}",
    int n      => $"Number {n}",
    bool b     => b ? "True" : "False",
    _          => "Unknown type"
};
```

**Relational patterns** (`>= 90`) work when the switched value is numeric. **Type patterns** (`string s`) test runtime type and bind a variable. This replaces verbose `if (input is string s)` chains for polymorphic dispatch at small scale. In ASP.NET you will later see similar ideas in **pattern matching** on records and HTTP results — the mental model starts here.

## Under the Hood

When the JIT compiles your method, `if` and `switch` become **branch instructions** at the IL level — conditional jumps based on flags set by comparison instructions. The CPU's **branch predictor** makes heavily predictable branches cheap; unpredictable branches in hot loops can matter for performance, but for typical business logic readability wins.

For **`switch` on dense integer or enum values**, the compiler may emit a **jump table** — O(1) dispatch — rather than a chain of comparisons. For **`string switch`**, the compiler uses a hash-based strategy (similar in spirit to `Dictionary`). You rarely need to micro-optimize this; knowing it exists explains why switching on an enum with fifty values is still fine.

**Short-circuit evaluation** is not an optimization hint — it is **defined behavior**. Code reviewers depend on it for null checks. Never reorder `&&` operands if the left side guards the right.

## Syntax / Commands / API

| Construct | Purpose |
|-----------|---------|
| `if (condition) { }` | Execute block when true |
| `else if (condition) { }` | Alternative branch |
| `else { }` | Default when all prior conditions false |
| `condition ? a : b` | Ternary value choice |
| `switch (x) { case ...: break; }` | Classic multi-way branch |
| `x switch { pattern => value }` | Switch expression |
| `_` | Discard / default pattern |
| `case A or B:` | Combined patterns (C# 9+) |

## Examples

### Example 1: Input validation in a console app

```csharp
Console.Write("Enter age: ");
string? text = Console.ReadLine();

if (!int.TryParse(text, out int age))
{
    Console.WriteLine("Invalid number.");
    return;
}

if (age < 0 || age > 120)
{
    Console.WriteLine("Age out of realistic range.");
}
else if (age < 18)
{
    Console.WriteLine("Minor");
}
else
{
    Console.WriteLine("Adult");
}
```

`TryParse` avoids exceptions for expected bad input — a production-friendly pattern. The method uses **`return`** early to exit `Main` when parsing fails; we explore `return` deeply in `1.1.11`.

### Example 2: HTTP-style status mapping with switch expression

```csharp
int statusCode = 404;

string message = statusCode switch
{
    200 => "OK",
    201 => "Created",
    400 => "Bad Request",
    401 => "Unauthorized",
    404 => "Not Found",
    >= 500 and < 600 => "Server Error",
    _ => "Unknown"
};

Console.WriteLine($"{statusCode}: {message}");
```

This mirrors how you will map API results in later ASP.NET lessons — the decision table is visible at a glance.

## Common Mistakes & Anti-patterns

**Dangling else confusion** is rare in well-braced code, but nested `if` without braces causes mis-association. Always brace nested branches in team code.

**Using `switch` on doubles or floats** for equality is fragile. Prefer thresholds or integer cents for money.

**Forgetting `break`** in classic switch causes compile errors in C# — unlike C. That is intentional safety.

**Replacing every `if` with ternary** hurts readability. Reserve ternary for simple value selection.

**Duplicate case labels** or unreachable cases after `_` in expressions trigger compiler warnings or errors — treat them as signals, not noise.

## Production & Real-World Notes

In code review, prefer **`switch` on enums** over magic numbers. Enums make cases self-documenting and let the compiler warn when you add a new enum member without updating the switch (if you enable exhaustive analysis or omit `_` deliberately during development).

Guard clauses — early `if` returns — flatten nested logic in service methods. You will combine that style with `return` in `1.1.11` and methods in `1.1.12`.

For configuration-driven behavior, teams sometimes replace giant switches with dictionaries or strategy patterns — but understanding `switch` remains essential because those patterns compile down to the same decision logic.

## Comparison / Trade-offs

| Situation | Prefer |
|-----------|--------|
| Two outcomes, simple value | Ternary |
| Boolean business rules with ranges | `if / else if` |
| Same variable vs many constants | Classic or expression `switch` |
| Type-based dispatch, small scale | Pattern `switch` expression |
| Complex rules spanning many variables | Extract method; consider polymorphism later |

Classic switch supports **statements** in each arm; switch expressions require **expression-oriented** arms (use blocks with `=>` carefully). When arms need multiple statements, classic switch or `if/else` may win.

## Key Takeaways

- Control flow turns boolean **answers** into **actions** or **values**.
- C# requires explicit `bool` conditions — no implicit truthiness.
- `&&` and `||` short-circuit; order guard checks accordingly.
- Classic `switch` uses `break`; C# prevents accidental fall-through.
- Switch expressions assign cleanly and support modern pattern matching.
- Choose the structure that best shows your decision table to the next reader.
