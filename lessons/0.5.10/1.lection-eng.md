# DOM Tree: Document, Element, Text Nodes

> Roadmap: `0.5.10` · Node: `0.5` — HTML · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Describe the browser **Document Object Model** as a tree of nodes rooted at `document`.
- Distinguish **node types**: `Document`, `Element`, `Text`, `Comment`, `DocumentFragment`, and others.
- Explain how HTML parsing builds the DOM and when **script** and **stylesheet** loading blocks parsing.
- Navigate and mutate the tree with `parentNode`, `childNodes`, `firstChild`, `nextSibling`, and modern `children` / `firstElementChild`.
- Relate DOM structure to the **accessibility tree** and to React's virtual DOM reconciliation.
- Predict the effect of `textContent` vs `innerHTML` on node creation and security.

---

## Why This Matters

Every HTML lesson so far — semantic elements in `0.5.x`, forms in `0.5.8`, tables in `0.5.9` — is a **serialization** of a tree the browser will build in memory. When you write `<p>Hello</p>`, the parser does not store a string; it creates nodes with types, parent/child links, and live references. JavaScript in the browser — and React, Vue, Svelte on top — reads and writes that tree. ASP.NET Core returns HTML bytes; the client's parser turns them into the same model. Middle developers debug "my click handler doesn't fire", "hydration mismatch", "duplicate IDs in the a11y tree", and "XSS via innerHTML" by understanding nodes, not tags as text.

The DOM is also the boundary between ** markup** and **runtime**. CSS selectors match element nodes; `getComputedStyle` reads cascaded values on elements. Screen readers consume a filtered **accessibility tree** derived largely from element nodes and their attributes. Confusing `innerText` with `textContent` or assuming `div.textContent = userInput` is safe while `innerHTML` is not — these are node-level decisions. Depth here pays off across the entire frontend stack and when reviewing SSR output from Razor or Blazor.

## Core Concepts

### The Document as Root

When a browser loads a page, the HTML parser constructs a **`Document`** node — the root of the tree. In JavaScript you access it via `document` (or `window.document`). The document owns **doctype**, the single **`documentElement`** (the `<html>` element), and metadata such as `title`, `URL`, and **ready state** (`loading`, `interactive`, `complete`).

Everything visible and invisible in the page markup hangs under `document.documentElement`. The DOM is **live**: if you remove a node in script, layout and event listeners update; if React replaces a subtree, the browser disconnects old nodes (modulo pooling in frameworks). This mutability distinguishes DOM from one-shot parsed JSON.

### Node Interface and nodeType

All nodes implement the base **`Node`** interface. Each node has:

- **`nodeType`** — numeric constant identifying kind
- **`nodeName`** — tag name for elements, `#text` for text nodes, etc.
- **`parentNode`**, **`childNodes`** — tree links
- **`textContent`** — concatenated text of node and descendants (for elements)

Common `nodeType` values:

| Constant | Value | Example |
|----------|-------|---------|
| `DOCUMENT_NODE` | 9 | `document` |
| `ELEMENT_NODE` | 1 | `<div>`, `<p>`, `<html>` |
| `TEXT_NODE` | 3 | `"Hello"` between tags |
| `COMMENT_NODE` | 8 | `<!-- note -->` |
| `DOCUMENT_FRAGMENT_NODE` | 11 | `document.createDocumentFragment()` |

**Element nodes** (`nodeType === 1`) represent HTML/SVG elements. They have **`tagName`**, **attributes** (`id`, `class`, `href`), **`children`** (element-only collection), and methods like `getAttribute`, `querySelector`, `append`, `remove`. **Text nodes** (`nodeType === 3`) hold raw character data — often whitespace between tags. They have no attributes; **`nodeValue`** / **`textContent`** on the text node itself is the string.

Why whitespace text nodes matter: `<div>\n  <span>x</span>\n</div>` often yields three children of `div`: text, element, text. Code using `div.firstChild` may hit a newline text node instead of `span`. Hence **`firstElementChild`** and **`children`** — element-only traversals — are preferred in modern scripts.

### Building the Tree from HTML

Parsing proceeds roughly: **tokenize** HTML → **construct tree** following insertion rules in the HTML spec → **attach** to document. Misnested tags like `<p><div></div></p>` get **auto-corrected** — the `<div>` closes the `<p>`. This is why DevTools DOM may differ slightly from your source file; the **canonical DOM** is what scripts and CSS see.

**Void elements** (`<img>`, `<br>`, `<input>`) have no children. **Raw text** elements (`script`, `style`) treat content specially. **Template** contents live in a **`DocumentFragment`** until activated.

External resources interact with parsing:

- **Classic scripts** without `async`/`defer` **block** parsing while fetched and executed.
- **`defer`** scripts run after document parsed, before `DOMContentLoaded`.
- **`async`** scripts run when ready, may interrupt parsing.
- **CSS** is render-blocking by default; DOM construction can continue but rendering waits.

For Middle+ SSR: the server sends HTML; the browser builds DOM once; hydration scripts **attach** to existing element nodes — mismatches occur when server HTML produces a different node tree than client render expects.

### Traversing and Mutating

Legacy traversal uses **`childNodes`** (NodeList including text/comments) and **`firstChild` / `nextSibling`**. Element-focused APIs:

```javascript
const list = document.querySelector('ul');
const firstItem = list.firstElementChild;
const items = list.children; // HTMLCollection, live
```

**Mutation** methods: `appendChild`, `insertBefore`, `removeChild`, modern **`append`**, **`remove`**, **`replaceWith`**. **`document.createElement('button')`** creates detached element nodes; **`document.createTextNode('Hi')`** creates text nodes. **`element.textContent = 'Hi'`** replaces all children with one text node — safe from HTML injection. **`element.innerHTML = userInput`** parses HTML and creates new element/text nodes — **XSS risk**.

**`DocumentFragment`** is an off-DOM container: batch inserts minimize reflows when appending many nodes.

### Document vs Window

**`window`** is the JavaScript global object in browsers; **`document`** is the active document (one per tab in classic model). **`document.getElementById`** searches the tree; **`document.querySelector`** uses Selectors API. Events **bubble** through element nodes from target up through ancestors to `document` and `window` — the path is node-linked.

## Under the Hood

### Parser and DOM Implementation

Browser engines (Blink, Gecko, WebKit) implement HTML parsing as a **state machine** aligned with the WHATWG HTML spec. Tokens become nodes; the **open elements stack** decides where to insert the next node. **Incremental parsing** allows DOM exposure before full download — scripts can run mid-parse if not deferred.

Internally, nodes are C++ objects (Chromium) or similar with cached layout information on elements after style calculation. Text nodes store UTF-16 code units; line breaks from HTML source become text node data. **Shadow DOM** attaches separate subtrees to **shadow roots** — still nodes, but scoped for encapsulation; `element.shadowRoot` exposes host-specific tree.

### Live Collections and Performance

**`HTMLCollection`** and **`NodeList`** from `getElementsByTagName` can be **live** — mutating DOM updates the collection. Caching `length` in a loop matters. **`querySelectorAll`** returns static NodeList. Heavy read/write interleaving causes **layout thrashing** — read geometric property, write style, read again forces reflow each time.

### Accessibility Tree Mapping

Not every DOM node appears in the **accessibility tree**. Plain text nodes inside buttons merge into the button's **accessible name**. Decorative elements with `aria-hidden="true"` are pruned. **`display:none`** and **`visibility:hidden`** remove from a11y tree (with nuances for visibility:hidden on focusable descendants). The browser's **Accessibility API** maps element roles from implicit HTML semantics (`<button>` → button) plus ARIA. Understanding DOM helps explain why a `<span onclick>` lacks button role until you add `role="button"` and keyboard handlers — the node type is still `HTMLSpanElement`.

### React and the Virtual DOM

React keeps a **JavaScript tree** mirroring UI; on update it **diffs** and applies minimal DOM operations — `createElement`, `insertBefore`, `removeChild` under the hood. **Fiber** schedules work per unit; concurrent mode can pause. Hydration matches **existing DOM nodes** from SSR to client vdom — text node mismatches (extra whitespace) cause warnings. Keys on lists preserve element identity across reorders — without keys, React may reuse wrong DOM nodes.

## Syntax / Commands / API

```javascript
// Node type check
node.nodeType === Node.ELEMENT_NODE
node.nodeType === Node.TEXT_NODE

// Create and attach
const el = document.createElement('p');
const text = document.createTextNode('Hello');
el.appendChild(text);
document.body.appendChild(el);

// Safe vs unsafe
el.textContent = userInput;   // no HTML parse
el.innerHTML = userInput;     // parses markup — XSS if untrusted

// Traversal
parent.children
parent.firstElementChild
node.parentElement
```

| Property / method | Returns |
|-------------------|---------|
| `childNodes` | All child nodes (incl. text) |
| `children` | Element children only |
| `textContent` | Text descendants as string |
| `innerHTML` | HTML serialization of children |
| `cloneNode(deep)` | Copy node subtree |

## Examples

**Example 1 — Whitespace trap.**

```html
<ul id="list">
  <li>One</li>
</ul>
<script>
  const list = document.getElementById('list');
  console.log(list.childNodes.length); // often 3: text, li, text
  console.log(list.children.length);   // 1
</script>
```

**Example 2 — textContent wipe.**

```javascript
const box = document.querySelector('#box');
box.innerHTML = '<span old></span>';
box.textContent = 'Plain';
// box has one child: TEXT_NODE "Plain"
```

**Example 3 — Fragment batch.**

```javascript
const frag = document.createDocumentFragment();
for (const name of ['A', 'B', 'C']) {
  const li = document.createElement('li');
  li.textContent = name;
  frag.appendChild(li);
}
document.querySelector('ul').append(frag);
// single reflow for append
```

## Common Mistakes & Anti-patterns

**Using `firstChild` expecting element** — hits whitespace text node. Prefer **`firstElementChild`**.

**innerHTML for user data** — creates element nodes from attacker markup. Use **`textContent`** or sanitize.

**Assuming DOM equals source HTML** — parser fixes nesting; always inspect DevTools Elements panel.

**Ignoring live collections** — caching `HTMLCollection` then mutating DOM in loop → infinite loop or skipped nodes.

**Memory leaks via detached DOM** — nodes removed from tree but referenced in JS closures retain listeners and memory.

**Confusing `innerText` vs `textContent`** — `innerText` triggers layout, respects CSS visibility; `textContent` is raw tree text.

## Production & Real-World Notes

**SSR + hydration:** Ensure server and client produce identical element/text structure; avoid `Date.now()` or random IDs in first client render.

**MutationObserver** watches node insert/remove/attribute changes — used by frameworks and extensions; expensive on hot paths.

**CSP** restricts inline script execution — affects how scripts mutate DOM at parse time.

**Web Components** combine shadow DOM nodes with custom element lifecycle callbacks.

**Testing:** `@testing-library/react` queries the DOM as users see it — roles from element nodes, not implementation details.

---

## Comparison / Trade-offs

**DOM APIs vs HTML strings** — Building via `createElement` is verbose but safe; templating + `innerHTML` fast but needs trust boundary.

**Synchronous DOM mutation vs DocumentFragment** — Fragment batches inserts; single `innerHTML` parse is one operation but replaces all children.

**childNodes vs children** — Complete vs element-only; choose based on whether text nodes matter.

**Imperative DOM vs declarative React** — React abstracts node ops; debugging still requires DOM mental model.

## Key Takeaways

- DOM is a **live tree of nodes**, not a static HTML string.
- **Document** is root; **elements** have tags and attributes; **text nodes** hold character data including whitespace.
- Parser **repairs** invalid HTML; DevTools shows canonical tree.
- Use **element-only** traversal APIs to avoid whitespace text node surprises.
- **`textContent`** sets text safely; **`innerHTML`** parses HTML and enables XSS.
- **Accessibility tree** is derived from DOM elements, not a 1:1 copy.
- React **reconciles** to DOM node operations — hydration requires matching server tree.
