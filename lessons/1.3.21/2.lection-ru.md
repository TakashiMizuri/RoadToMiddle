# LINQ Join и GroupJoin — SQL-like joins в памяти

> Roadmap: `1.3.21` · Node: `1.3` — C#: продвинутые темы · Depth: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Различать **`Join`** (inner) и **`GroupJoin`** (группировка правой стороны на ключ).
- Переводить **`INNER JOIN`** / **`LEFT JOIN`** в LINQ method и query syntax.
- Строить left outer join через **`DefaultIfEmpty`** после **`GroupJoin`**.
- Оценивать cost join на **`IEnumerable`** vs **`IQueryable`**.
- Избегать cartesian explosion и ошибок composite keys.

---

## Why This Matters

Данные relational — orders + customers, posts + tags. SQL выражает связи **`JOIN`**; LINQ — то же для objects и (через EF) для SQL. **`Join`** отбрасывает non-matching rows как **`INNER JOIN`**. **`GroupJoin`** даёт на каждый outer element **`IEnumerable`** inner matches — основа **`LEFT JOIN`** и иерархий «customer + orders».

Nested **`foreach`** + **`Dictionary`** вручную — медленнее писать и легче ошибиться с null. Middle dev читает join pipelines в repositories и debug'ит EF SQL, где **`GroupJoin`** стал subquery или cartesian product.

## Core Concepts

### Join — inner join

```csharp
customers.Join(
    orders,
    c => c.Id,
    o => o.CustomerId,
    (c, o) => new { c.Name, o.Total });
```

**`FROM customers INNER JOIN orders ON ...`**. Customer без orders исчезает. Order с несуществующим **`CustomerId`** — тоже.

Keys — **`Equals`** (default comparer или overload). Composite keys — anonymous type / tuple с согласованными именами и типами в query syntax.

### GroupJoin — группа справа на ключ

```csharp
customers.GroupJoin(
    orders,
    c => c.Id,
    o => o.CustomerId,
    (c, orderGroup) => new { c.Name, Orders = orderGroup });
```

Каждый customer **один раз**, даже без orders — **`orderGroup`** пустой. Left join + nested collection pattern.

Flat left outer:

```csharp
customers.GroupJoin(orders, c => c.Id, o => o.CustomerId, (c, g) => new { c, g })
    .SelectMany(x => x.g.DefaultIfEmpty(), (x, o) => new { x.c.Name, Order = o });
```

**`o == null`** — customer без order.

### Query syntax

```csharp
from c in customers
join o in orders on c.Id equals o.CustomerId
select new { c.Name, o.Total };

from c in customers
join o in orders on c.Id equals o.CustomerId into custOrders
from o in custOrders.DefaultIfEmpty()
select new { c.Name, Total = o != null ? o.Total : 0 };
```

**`join ... into`** = **`GroupJoin`**. **`equals`**: outer key слева, inner справа.

## Under the Hood

**`Enumerable.Join`**: lookup (hash) по inner — O(m), scan outer — O(n), average **O(n+m)**. Duplicate inner keys → несколько output rows на outer match (как SQL).

**`GroupJoin`**: тот же lookup, но **`resultSelector`** получает **`IEnumerable`** без flatten.

**`IQueryable`**: provider translate в SQL **`JOIN`**. Client eval **`GroupJoin`** в EF 3+ — exception если не translate.

Deferred до enumeration / **`ToList()`**.

## Examples

### Inner join: orders с именем customer

Когда нужен flat report только по orders с валидным customer, **`Join`** выражает intent напрямую:

```csharp
var report = orders.Join(
    customers,
    o => o.CustomerId,
    c => c.Id,
    (o, c) => new OrderReportRow(c.Name, o.Id, o.Total));
```

Orphan orders ( **`CustomerId`** без matching customer) исчезают — как при **`INNER JOIN`**. Если orphan'ы важны для audit, outer side должна быть **`orders`**, а pattern — **`GroupJoin`** + **`DefaultIfEmpty`**.

### Иерархия: customer + список orders

**`GroupJoin`** сохраняет customers без orders:

```csharp
var dto = customers.GroupJoin(
    orders,
    c => c.Id,
    o => o.CustomerId,
    (c, ords) => new CustomerDto(
        c.Id,
        c.Name,
        ords.Select(o => o.Total).ToList()));
```

**`.ToList()`** внутри selector на **`IQueryable`** может translate в SQL subquery или уехать на client — всегда смотри **`ToQueryString()`** и план.

### EF-style left join

```csharp
from c in db.Customers
join o in db.Orders on c.Id equals o.CustomerId into og
from o in og.DefaultIfEmpty()
select new { c.Name, Total = o == null ? 0 : o.Total };
```

Provider emit'ит **`LEFT JOIN`**, когда keys translatable.

## Common Mistakes & Anti-patterns

**`Join` вместо left join** — customers без orders silently пропадают; в отчётах «все клиенты» это частая production ошибка.

**Забытый `DefaultIfEmpty`** после **`join into`** — empty group не даёт flat row с null inner.

**Composite keys** — `new { Id, R }` и `new { Id, Region }` — разные anonymous types, join never matches.

**Join на float/double** — unstable equality; join по business keys (int, Guid, string codes).

**Double enumeration `IQueryable`** — два **`ToList()`** → два SQL round-trips; materialize once.

---

## Production & Real-World Notes

На больших datasets держи join в **`IQueryable`** — пусть SQL join'ит indexed FK (**`CustomerId`**). **`Include`** тянет entities целиком; explicit **`Join`** + projection экономит columns и memory.

При cartesian symptoms проверь duplicate keys на inner side — **`Join`** умножает rows как SQL. **`EXPLAIN ANALYZE`** на emitted SQL обязателен при review тяжёлых report queries.

## Comparison / Trade-offs

| SQL | LINQ |
|---|---|
| INNER | **`Join`** |
| LEFT flat | **`GroupJoin` + DefaultIfEmpty** |
| LEFT nested | **`GroupJoin`** |

---

## Key Takeaways

- **`Join`** — inner, unmatched gone.
- **`GroupJoin`** — outer + group, empty group OK.
- Left flat — **`SelectMany` + DefaultIfEmpty`** (1.3.22).
- Stable keys; duplicate inner keys multiply rows.
