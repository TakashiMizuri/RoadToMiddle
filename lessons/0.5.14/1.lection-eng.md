# iframe, embed — Security Sandbox

> Roadmap: `0.5.14` · Node: `0.5` — HTML · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Explain when to use `<iframe>` vs `<embed>` and what each element does in the browser.
- Describe the **same-origin policy** and why embedded content is isolated by default.
- Configure the **`sandbox`** attribute to grant minimal permissions to untrusted frames.
- Identify **clickjacking**, **XSS via embeds**, and **mixed content** risks.
- Apply **`allow`**, **`referrerpolicy`**, and **`loading`** attributes responsibly in production embeds.

---

## Why This Matters

Modern web applications constantly embed third-party content: YouTube tutorials, Stripe payment forms, Google Maps, Intercom chat widgets, Figma previews, and PDF viewers. Each embed is a **separate browsing context** — essentially a nested browser tab inside your page. That power comes with security obligations. A misconfigured iframe can let an attacker read your cookies, trick users into clicking hidden buttons, or execute scripts with more privilege than you intended.

As a fullstack developer you will not only consume embeds but also **host** embeddable widgets (dashboard charts, booking calendars). Understanding sandbox boundaries helps you design APIs that assume least privilege. Content Security Policy (CSP) `frame-ancestors` and iframe `sandbox` are complementary defenses against **clickjacking** — when your site is framed maliciously to capture clicks.

This lesson connects to the DOM tree (`0.5.10`): each iframe owns a nested `Document` with its own window object. Later, React portals render outside the component subtree but still within the same document — unlike iframes, which are true origin boundaries. Security thinking here transfers directly to OAuth popups, payment PCI scope, and micro-frontend architectures.

## Core Concepts

### iframe: A Document Inside a Document

The `<iframe>` element loads another HTML page (or navigable content) into a rectangular region on your page. The embedded page has its own **browsing context**: separate `window`, `document`, history stack, and origin (scheme + host + port). Parent JavaScript cannot access iframe DOM if origins differ — the **same-origin policy** throws on cross-origin `contentDocument` access.

```html
<iframe
  src="https://www.youtube.com/embed/dQw4w9WgXcQ"
  title="Tutorial video"
  width="560"
  height="315"
  loading="lazy"
  allow="accelerometer; autoplay; clipboard-write; encrypted-media; gyroscope; picture-in-picture"
  allowfullscreen>
</iframe>
```

Always set a descriptive **`title`** for accessibility — screen reader users need context for embedded content (`0.5.11`). Without `title`, the iframe is announced as unlabeled embedded content.

The **`src`** URL determines the embedded origin. **`srcdoc`** allows inline HTML without a network request — useful for sandboxed previews but dangerous if fed untrusted strings (XSS vector).

### embed vs iframe

`<embed>` is a **replaced element** for external resources — historically plugins (Flash), now occasionally PDFs or SVG. It is void (no closing tag), less flexible than iframe for full documents. For HTML pages and most third-party widgets, **prefer iframe** because it provides a complete navigable document with standard security model and better accessibility patterns.

```html
<!-- Legacy-style PDF embed — iframe often preferred today -->
<embed src="/manual.pdf" type="application/pdf" width="600" height="800">
```

When documentation says "embed a map/video," they usually mean iframe with vendor URL. Know both tags for reading legacy markup and MDN accurately.

### Same-Origin Policy Recap

Two URLs share an **origin** if scheme, host, and port match. `https://app.example.com` and `https://api.example.com` are **different origins** (subdomain differs). Parent and child frames from different origins can coexist visually but cannot read each other's DOM or localStorage. They **can** communicate via **`postMessage`** with explicit origin checks — the safe bridge pattern for widget integration.

This isolation protects users: a compromised ad iframe cannot steal session cookies from your banking tab's origin. Conversely, you must not assume you can style or script vendor iframe internals — only the outer frame element.

### The sandbox Attribute

By default, a cross-origin iframe already cannot access parent DOM. **`sandbox`** adds restrictions **on the embedded document's capabilities** even when same-origin or when you host untrusted HTML. An iframe with `sandbox` (empty value) applies maximum restrictions:

- Treats content as from a unique **opaque origin**
- Blocks script execution (unless `allow-scripts`)
- Blocks form submission (unless `allow-forms`)
- Blocks top-level navigation (unless `allow-top-navigation` variants)
- Blocks plugins, popups (unless allowed), automatic downloads, etc.

```html
<!-- Highly restricted: static HTML preview only -->
<iframe sandbox srcdoc="<p>Preview</p>" title="Email preview"></iframe>

<!-- Allow scripts but not same-origin escalation -->
<iframe sandbox="allow-scripts" src="/user-generated-widget.html" title="Widget"></iframe>
```

Each **`allow-*`** token removes one restriction. **`allow-same-origin`** is subtle: combined with `allow-scripts`, embedded same-origin content regains normal origin privileges — often **too permissive** for untrusted user HTML. Security guides recommend avoiding `allow-same-origin` + `allow-scripts` together for untrusted content.

Common tokens:

| Token | Effect |
|---|---|
| `allow-scripts` | JavaScript runs |
| `allow-forms` | Form submission |
| `allow-popups` | `window.open` |
| `allow-same-origin` | Keeps origin (not opaque) |
| `allow-top-navigation-by-user-activation` | User-initiated top navigation |

Apply **least privilege**: start with full sandbox, add tokens only when required.

### Clickjacking and frame-ancestors

**Clickjacking** overlays your site in a transparent iframe on an attacker page, tricking clicks onto "Transfer money" buttons. Defenses:

1. **`X-Frame-Options: DENY`** or `SAMEORIGIN` (legacy header)
2. **CSP `frame-ancestors 'self'`** (modern, preferred)
3. JavaScript frame-busting (unreliable; do not rely alone)

As embed **consumer**, trust only reputable `src`. As **provider**, send headers preventing unauthorized framing.

### Permissions Policy via allow

The **`allow`** attribute on iframe lists **Permissions Policy** features delegated to the embedded document — camera, microphone, geolocation, fullscreen, payment, etc. Default is deny for sensitive features cross-origin.

```html
<iframe src="https://meet.vendor.com/room/abc"
        allow="camera; microphone; fullscreen"
        title="Video meeting">
</iframe>
```

Narrow delegation reduces abuse if the embedded origin is compromised.

### referrerpolicy and loading

**`referrerpolicy`** controls how much referrer URL is sent when the iframe loads subresources — privacy and leakage concern when internal URLs appear in query strings.

**`loading="lazy"`** defers offscreen iframe fetch until near viewport — performance win for pages with many embeds; does not change security model.

### Mixed Content

An HTTPS page embedding **`http://`** iframe is **mixed content** — blocked or weakened by browsers. Always use HTTPS embed URLs on secure sites.

## Under the Hood

When the browser parses `<iframe src="...">`, it creates a **nested browsing context** linked in a tree of frames. The **HTML specification** defines sandbox flags as a set of restrictions enforced at navigation time and when scripts run in the nested context. With empty sandbox, the embedded document gets a **unique origin** serialized as `null` in security checks — scripts cannot access cookies/storage of parent origin.

The **Composite Security Model** combines: URL origin, sandbox flags, CSP of both parent and child, and Permissions Policy. A vulnerability often requires chaining misconfigurations — e.g. `sandbox="allow-scripts allow-same-origin"` on user HTML allowing script to read parent cookies when hosted on same site subdomain.

`postMessage` events queue between windows; handlers must validate **`event.origin`** against an allowlist — never trust message content alone.

## Syntax / Commands / API

```html
<!-- Third-party video (vendor-controlled security) -->
<iframe
  src="https://player.example.com/v/123"
  title="Product demo video"
  width="640"
  height="360"
  loading="lazy"
  referrerpolicy="strict-origin-when-cross-origin"
  allow="fullscreen"
  allowfullscreen>
</iframe>

<!-- Untrusted HTML sandbox -->
<iframe
  sandbox="allow-scripts"
  src="/render/user-preview/abc"
  title="User HTML preview"
  width="100%"
  height="400">
</iframe>
```

Hosting side HTTP headers (ASP.NET Core example concept):

```
Content-Security-Policy: frame-ancestors 'self' https://partner.example.com;
X-Frame-Options: SAMEORIGIN
```

## Examples

### Stripe-like Payment Element (Conceptual)

Payment providers supply iframe URLs so card data touches **their PCI scope**, not yours. You frame their origin; sensitive fields never enter your DOM.

```html
<iframe src="https://js.stripe.com/v3/..." title="Secure payment input" allow="payment"></iframe>
```

Your JavaScript communicates via provider SDK using `postMessage` internally — you still must load official scripts only.

### Sandboxed User Content Preview

A CMS lets authors paste HTML snippets. Rendering raw HTML in parent page = XSS. Sandboxed iframe without `allow-same-origin`:

```html
<iframe
  sandbox=""
  srcdoc="&lt;p&gt;Author preview&lt;/p&gt;"
  title="Content preview">
</iframe>
```

Scripts in srcdoc do not run with empty sandbox. If previews need live JS, isolate on **separate subdomain** with strict CSP instead of loosening sandbox on main origin.

### Blocking Your App from Being Framed

In middleware or reverse proxy for admin panel:

```
Content-Security-Policy: frame-ancestors 'none';
```

Prevents any ancestor frame — clickjacking defense for privileged UI.

## Common Mistakes & Anti-patterns

**Omitting iframe `title`** — accessibility failure; WCAG violation for embedded content.

**`sandbox="allow-scripts allow-same-origin"` on untrusted same-site HTML** — restores full script power on your origin; classic sandbox escape mistake.

**Embedding arbitrary user URLs** — attacker embeds phishing page in your domain's chrome; users trust your shell.

**Ignoring CSP frame-ancestors on sensitive routes** — admin panels frameable by malicious sites.

**Over-broad `allow` features** — camera/microphone on embeds that do not need them.

**Trusting postMessage without origin check** — cross-site data exfiltration.

## Production & Real-World Notes

Vendor embeds (Maps, YouTube) document required iframe attributes — copy official snippets. For **first-party widgets**, document allowed parent origins and use `postMessage` protocol versioning. Security reviews often ask: "What happens if iframe src is swapped client-side?" — validate URLs server-side when generating embed codes.

Micro-frontends sometimes use iframes for **JS isolation** between teams; trade-off is UX integration cost (shared routing, styling). Module federation is an alternative with different security profile (same origin, shared runtime).

Log and monitor CSP **`frame-ancestors` violations** in report-only mode before enforcing.

## Comparison / Trade-offs

**iframe vs inline third-party script** — script runs in your origin (full DOM access if compromised); iframe isolates by origin. Prefer iframe for untrusted or PCI-sensitive UI.

**Strict sandbox vs functional widget** — more tokens = more attack surface; product must negotiate minimum set.

**X-Frame-Options vs CSP frame-ancestors** — implement CSP; keep XFO for legacy browsers if needed.

**embed vs iframe for PDF** — both work; iframe gives consistent sandbox and titling; PDF.js in iframe common pattern.

## Quick Reference

| Mechanism | Protects against |
|---|---|
| Same-origin policy | Cross-frame DOM/cookie access |
| `sandbox` | Script, form, navigation abuse in frame |
| CSP `frame-ancestors` | Your site framed maliciously (clickjacking) |
| `allow` attribute | Over-delegation of camera/mic/payment |
| HTTPS embeds | Mixed content downgrade/block |
| `postMessage` + origin check | Safe cross-frame communication |

## Key Takeaways

- `<iframe>` creates a **nested browsing context** with its own origin and DOM.
- Cross-origin parent cannot access iframe DOM — use **`postMessage`** with origin validation.
- **`sandbox`** applies least-privilege flags to embedded content; avoid `allow-scripts` + `allow-same-origin` for untrusted HTML.
- Prevent **clickjacking** with CSP **`frame-ancestors`** on sensitive pages.
- Always set iframe **`title`**; use **`allow`** narrowly for Permissions Policy features.
- Prefer iframe over embed for documents; understand vendor embed security vs rolling your own.
