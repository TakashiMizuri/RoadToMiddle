# Answers: ref, out, in, params

## Scoring

## Answers

### 1. [Recall]
**Answer:** При pass by value копируется сам reference (указатель на объект); callee не может переприсвоить переменную caller'а. При `ref` передаётся alias на **слот переменной** caller'а — переприсваивание параметра меняет переменную caller'а.
**Why:** Мутация объекта через общий reference возможна и без `ref`; `ref` нужен для замены **переменной** или in-place update value type.

### 2. [Recall]
**Answer:** Callee **обязан присвоить** `out`-параметр на каждом control-flow пути, ведущем к `return`, **до** выхода из метода.
**Why:** Caller не инициализирует аргумент; чтение до assign — undefined behavior на уровне логики.

### 3. [Recall]
**Answer:** Ровно **один** `params`; он должен быть **последним** параметром в списке.
**Why:** Компилятор должен однозначно знать, какие аргументы идут в variadic-массив.

### 4. [Understanding]
**Answer:** Caller часто передаёт **неинициализированную** переменную; `out` это разрешает. `ref` требовал бы инициализации и семантически обещал бы «входное значение имеет смысл», чего нет при parse.
**Why:** `Try*` = «запишу результат только при успехе»; `out` + `bool` — установленный контракт BCL.
**Common mistake:** Думать, что `out` и `ref` взаимозаменяемы на call site.

### 5. [Understanding]
**Answer:** `in` даёт byref **без права записи** — защита от случайной мутации и defensive copy semantics для readonly struct. `ref` разрешает read+write.
**Why:** Performance + immutability contract в одном модификаторе.

### 6. [Understanding]
**Answer:** Компилятор генерирует передачу **нового массива** `new int[] { 1, 2, 3 }` (heap allocation), если не использован overload с готовым массивом.
**Why:** `params` — syntactic sugar над single-dimensional array parameter.

### 7. [Understanding]
**Answer:** **Нет.** `n` definitely assigned только в ветке, где условие `true`; в `else` компилятор считает `n` possibly unassigned.
**Why:** Flow analysis привязана к результату `TryParse`.

### 8. [Application]
**Answer:** `static bool TryMinMax(int a, int b, out int min, out int max)` — при `true` записывает меньшее в `min`, большее в `max`; caller не обязан их инициализировать до вызова.
**Why:** Два независимых output через `out` — типичный pattern.

### 9. [Application]
**Answer:** Когда нужна **in-place мутация** большого struct или обновление конкретного slot (например элемент массива через `ref`), и return copy дороже или менее ясен по intent.
**Why:** `ref` документирует aliasing; return — functional style для маленьких immutable-friendly типов.

### 10. [Analysis]
**Answer:** **Читает `value` до присвоения** — нарушение контракта `out`.
**Why:** Компилятор должен выдать error CS0170 (use of unassigned out parameter).
**Common mistake:** Забыть assign на всех путях return.

### 11. [Analysis]
**Answer:** Каждый variadic вызов может **аллоцировать массив**; в tight loop это давление на GC.
**Why:** Prefer `ReadOnlySpan<T>`, overloads, или переиспользуемый buffer.

### 12. [Application]
**Answer:** `Process(in x)` для readonly field struct — **да** (read-only alias). `Process(ref x)` для readonly field — **нет** (compile error CS1654).
**Why:** `ref` позволил бы нарушить readonly guarantee поля.

### 13. [Understanding]
**Answer:** На IL **практически не отличаются** — оба `byref`; различие только в **compile-time definite assignment rules**.
**Why:** Runtime calling convention одна; контракты — job компилятора C#.

### 14. [Analysis]
**Answer:** `out User`: привычен BCL-стилю, хорош с existing Try patterns; tuple: удобнее deconstruct, functional composition, но для legacy API менее привычен.
**Why:** Trade-off readability vs idiomatic .NET.

### 15. [Application]
**Answer:** `Log("INFO", "count", 42);` или `Log("INFO", new object[] { "count", 42 });`
**Why:** Первый аргумент после `level` попадает в `params` array.

## Model answers for practical sections

**Swap generic:** стандартный swap через temp; для struct `T` generic constraint не требует class — **no boxing**, потому что `T` остаётся unconstrained value или ref type без conversion to object.
