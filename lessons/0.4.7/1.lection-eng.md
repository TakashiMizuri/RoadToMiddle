# Firewalls, Ports 80/443/5432 — Security Basics

## Learning Objectives

After this lesson you will be able to:

- Explain what a **firewall** does at the network boundary and why it is not a substitute for application security.
- Describe how **ports** relate to services on a host and why well-known ports matter for operations.
- State the typical roles of ports **80**, **443**, and **5432** in a fullstack stack.
- Distinguish **host firewall** rules from **cloud security groups** and **network ACLs** conceptually.
- Apply the **principle of least privilege** when deciding which ports to expose publicly.
- Diagnose a common "connection refused" vs "timeout" scenario using port and firewall knowledge from `0.4.2`.

## Why This Matters

In lesson `0.4.2` you learned that a TCP connection is identified by an IP address and a port number, and that a **socket** binds a process to a port on a host. In `0.4.6` you saw how binding to `127.0.0.1` vs `0.0.0.0` controls whether a service accepts connections only from localhost or from the network. Those lessons describe what happens *inside* a machine once traffic arrives. This lesson asks the question that sits one layer outside: **which traffic is allowed to arrive at all?**

A **firewall** is the gatekeeper. It inspects packets — typically by source IP, destination IP, destination port, and protocol — and applies a policy: allow, deny, or drop. In development you often run everything on one laptop with no firewall friction. In production, a PostgreSQL database listening on port 5432 must almost never be reachable from the public internet. Your ASP.NET Core API might listen on 5000 internally but be exposed only through port 443 on a reverse proxy (covered in `0.4.8`). Misconfigured port exposure is one of the fastest paths to a breach: scanners constantly probe the internet for open databases, Redis instances, and admin panels.

Middle developers do not need to become network security engineers, but they must speak the language of ports and firewall rules when debugging deploys, writing Docker Compose files, configuring cloud VMs, and reviewing infrastructure pull requests. When someone says "security group allows 443 inbound," you should know exactly what that means for your React app and your API.

## Core Concepts

### What a Firewall Actually Does

A firewall enforces a **policy on network traffic**. At its simplest, the policy is a ordered list of rules: "allow TCP 443 from anywhere," "allow TCP 22 from office IP range," "deny everything else." When a packet reaches the firewall — whether implemented as Linux `iptables`/`nftables`, Windows Defender Firewall, an AWS Security Group, or a hardware appliance — it is matched against these rules. The first matching rule wins (in most implementations), and the packet is forwarded, rejected, or silently dropped.

Firewalls operate at different layers depending on type. A **stateless** rule might say "allow inbound TCP 443" without tracking whether a connection was initiated from inside. A **stateful** firewall tracks connection state: return traffic for an outbound request is automatically allowed because it belongs to an established session. Stateful inspection is what makes "allow outbound, block unsolicited inbound" the default posture on most servers.

Critically, a firewall controls **network reachability**, not **authorization**. Opening port 443 lets HTTPS traffic reach your web server; it does not prove the caller is allowed to delete a user record. Authentication, authorization, input validation, and encryption at the application layer remain your responsibility. Treating "port closed" as "secure" is a dangerous shortcut.

### Ports as Service Addresses

A port is a 16-bit number (0–65535) that multiplexes many services on one IP address. Ports 0–1023 are **well-known** and conventionally require elevated privileges on Unix systems. Ports 1024–49151 are **registered**, and 49152–65535 are **dynamic/ephemeral** — often used as the source port on the client side of a connection.

When you run Kestrel on `http://localhost:5000`, clients connect to port 5000. When Nginx terminates TLS in front of your app, browsers connect to port 443 on the public IP; Nginx forwards internally to port 5000 or 8080. The **same application** can therefore appear on different ports depending on where you stand in the topology.

**Connection refused** usually means nothing is listening on that port, or a host firewall explicitly rejected the connection. **Timeout** often means a packet was dropped silently — typical of a network firewall or security group with no matching allow rule — so the client retries until it gives up.

### Port 80 — HTTP

Port **80** is the default for unencrypted **HTTP**. Browsers assume port 80 when you type `http://example.com` without a port. In modern production, port 80 still serves an important role even when all real traffic uses HTTPS: many setups run a lightweight HTTP listener that responds with **301/302 redirects** to the HTTPS URL on port 443. Let's Encrypt certificate issuance (HTTP-01 challenge) also requires temporary reachability on port 80.

From a security standpoint, serving sensitive data over plain HTTP on port 80 is unacceptable on the public internet because the traffic is readable and modifiable in transit. Closing port 80 entirely is possible but uncommon; redirect-and-upgrade is the usual compromise.

### Port 443 — HTTPS

Port **443** is the default for **HTTPS** — HTTP over **TLS**. This is where your users' browsers connect when they load your React SPA or call your REST API through a public URL. TLS provides confidentiality and integrity on the wire; the firewall's job is simply to permit inbound TCP 443 to the load balancer or web server.

In a typical fullstack deployment, only 443 (and sometimes 80 for redirect) is open to `0.0.0.0/0` — the entire internet. Everything else — Kestrel's internal port, PostgreSQL, Redis, message queues — stays on a private network or localhost. When you configure CORS later (`0.4.10`), remember that the browser's same-origin policy cares about scheme, host, and port; `https://api.example.com` on 443 is a different origin from `http://api.example.com:5000`.

### Port 5432 — PostgreSQL

Port **5432** is the default for **PostgreSQL**. Your EF Core application connects to PostgreSQL using a connection string that specifies host and port: `Host=db.internal;Port=5432;Database=app;...`. In Docker Compose, the service name `postgres` resolves to a container IP, and 5432 is the container port.

**Production rule of thumb:** port 5432 should be reachable only from application servers (or a bastion/VPN for admin access), never from the public internet. Automated scans for open 5432 are constant; weak or default passwords lead to ransomware and data exfiltration within hours. Cloud providers place PostgreSQL in a private subnet; the security group allows 5432 **only** from the API tier's security group, not from `0.0.0.0/0`.

Developers often expose 5432 to localhost during local dev (`ports: "5432:5432"` in Compose) — that maps the container port to your laptop's loopback, which is acceptable for development if your host firewall blocks external access. Mapping 5432 to `0.0.0.0` on a cloud VM is a recurring incident pattern.

### Defense in Depth

Firewall rules are one layer in **defense in depth**. Others include: TLS everywhere externally, strong authentication, parameterized SQL (no injection), secrets not in git, regular patches, and database roles with minimal permissions. Closing port 5432 does not fix SQL injection; fixing SQL injection does not replace closing 5432. Both matter.

The **principle of least privilege** applied to ports means: expose the minimum set of services to the minimum set of source IPs required for the system to function. Your React build artifacts are static files served over 443; they never need direct access to 5432. Only the backend does.

## Under the Hood

### Where Rules Live in a Typical Cloud Stack

Consider a common AWS-style layout: a React app on CloudFront or S3 behind HTTPS, an Application Load Balancer on public subnets accepting 443, ASP.NET Core on private subnets, RDS PostgreSQL on database subnets.

```
Internet → [SG: allow 443 in] → ALB (public)
                ↓
         [SG: allow app port from ALB only] → API servers (private)
                ↓
         [SG: allow 5432 from API SG only] → RDS (database subnet)
```

Each **security group** is a stateful virtual firewall attached to an ENI (network interface). Rules are allow-only; implicit deny applies to everything else. **Network ACLs** operate at the subnet boundary and can deny explicitly — rarely the first tool you reach for, but useful for coarse blocks.

On the Linux VM itself, `iptables` or `nftables` may further restrict traffic even after cloud SG allows it — "belt and suspenders." Docker adds its own `DOCKER-USER` chain; publishing a port with `-p 5432:5432` creates a NAT rule that bypasses some host assumptions — another reason to understand the full path.

### Stateful vs Stateless Matching

When your API server initiates an outbound connection to fetch a NuGet package or call a payment API, the response packets are **related** to that session. A stateful firewall remembers `(src, dst, src_port, dst_port, protocol)` and permits the return path. Without statefulness, you would need separate inbound allow rules for every possible external response — impractical.

Inbound connection attempts from random internet hosts to port 5432 have no matching outbound session. The packet hits the default deny and is dropped → client sees **timeout** if nothing responds, or **ICMP unreachable** / TCP RST depending on configuration.

## Syntax / Commands / API

### Checking What Is Listening (Linux/macOS)

```bash
# Linux — show listening TCP ports and processes
ss -tlnp

# macOS / some Linux
sudo lsof -iTCP -sTCP:LISTEN -P -n
```

### Windows

```powershell
netstat -ano | findstr LISTENING
Get-NetTCPConnection -State Listen | Select LocalAddress, LocalPort, OwningProcess
```

### Testing Reachability

```bash
# Is anything accepting TCP on port 443 at host?
nc -zv example.com 443

# From outside — online port checker or curl
curl -v --connect-timeout 5 https://example.com
```

### UFW Example (Ubuntu host firewall)

```bash
sudo ufw default deny incoming
sudo ufw allow 22/tcp    # SSH — restrict source IP in production
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
```

PostgreSQL on the same host would **not** get a public allow rule; it listens only on the Docker bridge or private interface.

## Examples

### Local Development vs Production

**Development:** Docker Compose runs `postgres:16` with `ports: ["5432:5432"]`. Your ASP.NET app uses `Host=localhost;Port=5432`. Your laptop firewall usually allows loopback; the database is reachable only from your machine unless you bind `0.0.0.0` and open the host firewall — avoid that.

**Production:** RDS endpoint `mydb.xxxx.region.rds.amazonaws.com:5432` resolves to a private IP. Security group on RDS allows inbound 5432 from the API server's security group ID only. No public IP on the database. Connection string lives in secrets manager, not in source code.

### Reading a Connection Failure

A junior deploys the API but the health check fails connecting to PostgreSQL. Error: `Npgsql.NpgsqlException: Connection timed out`. That pattern suggests packets reach a firewall or SG that drops them — not "wrong password" (that would authenticate and fail differently). The fix is network policy: add SG rule from API tier to database tier on 5432, verify subnet routing and NACLs.

If the error were `Connection refused`, the next check is whether PostgreSQL is running and listening on the expected interface — tying back to `0.4.6` binding semantics.

## Common Mistakes & Anti-patterns

**Exposing PostgreSQL to the internet "temporarily for debugging."** Temporary rules become permanent; bots find open ports within minutes. Use SSH tunnel, VPN, or a managed console instead.

**Assuming HTTPS on 443 means the database is encrypted too.** TLS to the load balancer protects browser-to-ALB traffic. The ALB-to-API hop and API-to-PostgreSQL hop may still be plain TCP on a private network — acceptable inside a VPC if trust boundaries are clear, but not the same as end-to-end encryption.

**Opening wide port ranges "just in case."** `allow 4000-9000` creates unnecessary attack surface. Specify exact ports and sources.

**Confusing Docker port publish with cloud firewall.** `-p 5432:5432` on a public VM exposes the database even if your application code is perfect. Check both container publish bindings and security groups.

## Production & Real-World Notes

Teams document **port matrices**: service, port, protocol, source, destination, justification. This becomes essential during audits and onboarding.

**Infrastructure as Code** (Terraform, CloudFormation) defines security groups in version control — reviewable like application code. A PR that adds `0.0.0.0/0` on 5432 should trigger the same scrutiny as a PR removing authorization checks.

**Zero trust** trends push beyond IP-based rules toward mTLS between services (service mesh), but port restriction remains the baseline. You will still encounter classic SG + private subnet designs for years.

Penetration tests routinely verify that only 80/443 (and perhaps 22 from bastion) are exposed. Failing that test blocks release in regulated environments.

## Comparison / Trade-offs

**Host firewall vs cloud security group:** SG protects at the hypervisor/network edge before traffic hits the VM; host firewall protects if another instance in the same VPC is compromised. Using both adds operational complexity but improves depth.

**Deny vs drop:** Explicit **reject** sends ICMP or TCP RST — faster failure for legitimate misconfigurations. **Drop** silently discards — slightly better against reconnaissance but slower debugging (timeouts).

**Single vs multi-tier subnets:** Putting the database in a subnet with no route to an internet gateway makes accidental public exposure harder than relying on SG rules alone. Routing + SG + host firewall stack for high-value data stores.

## Quick Reference

| Port | Service (default) | Typical public exposure |
|------|-------------------|-------------------------|
| 80 | HTTP | Yes — redirect to HTTPS |
| 443 | HTTPS (TLS) | Yes — primary user/API entry |
| 5432 | PostgreSQL | **No** — private network only |
| 22 | SSH | Restricted source IPs / bastion |
| 5000/8080 | Dev Kestrel / app | Internal or behind proxy only |

| Symptom | Likely cause |
|---------|----------------|
| Connection refused | Nothing listening; or explicit reject |
| Connection timed out | Firewall/SG drop; wrong route; wrong IP |
| Auth failed | Network OK; credentials/SSL issue |

## Key Takeaways

- Firewalls control **which packets reach a host**, not who is authorized to perform actions in your app.
- Port **443** is the public entry for HTTPS; **80** often exists for redirect and certificate challenges.
- Port **5432** (PostgreSQL) must stay off the public internet in production.
- **Least privilege** on ports: open only what is needed, from only the sources that need it.
- **Refused** vs **timeout** narrows whether the problem is listening vs network policy.
- Defense in depth combines firewall rules with app security, TLS, and secrets hygiene.
