# SemaphoreSlim vs lock — краткая выжимка

> `1.4.33` · ~10 мин на повторение

## Главная идея

**`lock`** нельзя сочетать с **`await`** — поток отдаётся в pool, Monitor thread-bound. Для async mutual exclusion: **`SemaphoreSlim.WaitAsync`** + **`Release`** в `finally`.

## Ключевые пункты

- `await` внутри `lock` — ошибка компиляции CS1996.
- **`SemaphoreSlim(1,1)`** — async mutex, **не re-entrant**.
- **`lock`** — только короткие sync-секции без I/O.
- **`.Result`/`.Wait()` в lock** — deadlock и pool starvation.
- **`(n,n)`** — throttling, не только exclusion.
- In-process only; multi-node — distributed lock.

## Самые важные примеры

```csharp
await _sem.WaitAsync(ct);
try { await WorkAsync(ct); }
finally { _sem.Release(); }
```

## «Под капотом» в одном абзаце

Monitor привязан к потоку; await разрывает непрерывное владение потоком. WaitAsync регистрирует continuation и освобождает поток до Release. Re-entrancy lock на том же потоке не переносится на async модель — повторный Wait на (1,1) зависнет.

## Частые ошибки

- Забыть Release при exception.
- Один глобальный sem на несвязанные ресурсы.
- Task.Run + lock как «fix» для await.

## Мнемоника

**«await needs WaitAsync, not lock»**
