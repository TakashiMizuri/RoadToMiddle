# LINQ Aggregate и Scan — fold operations

> Roadmap: `1.3.23` · Node: `1.3` · Depth: **глубоко**

## Learning Objectives

- **`Aggregate`** как left fold: no-seed, seed, resultSelector.
- Когда **`Sum`/`Max`** лучше custom **`Aggregate`**.
- **`Scan`** (running fold) — нет в **`System.Linq`**, custom/MoreLINQ.
- Empty sequence: no-seed throws, seed returns seed.
- **`Aggregate`** — immediate terminal; **`Scan`** — deferred iterator.

---

## Why This Matters

Не каждая reduction имеет named operator: product, merge patches, longest string, custom rollup — **`Aggregate`**. **`Scan`** даёт промежуточные accumulators (running totals) — prefix sum, charts, reducers. Ошибки: no-seed на empty, string **`+`** in fold, ожидание EF translate.

---

## Core Concepts

### No-seed

```csharp
names.Aggregate((acc, next) => next.Length > acc.Length ? next : acc);
```

First element = initial acc; **empty → exception**; single element → без вызова func.

### Seed

```csharp
numbers.Aggregate(0, (acc, n) => acc + n);
```

**Empty → seed** (0). **`TAccumulate`** может отличаться от **`TSource`**.

### Result selector

```csharp
orders.Aggregate(new StringBuilder(), (sb,o) => sb.AppendLine(...), sb => sb.ToString());
```

### Scan

```csharp
public static IEnumerable<TAcc> Scan<TSource, TAcc>(
    this IEnumerable<TSource> source, TAcc seed, Func<TAcc, TSource, TAcc> func)
{
    var acc = seed;
    foreach (var item in source)
    {
        acc = func(acc, item);
        yield return acc;
    }
}
```

Convention: emit seed first or not — document. MoreLINQ has standard overloads.

| | Aggregate | Scan |
|---|---|---|
| Output | One value | Sequence of states |
| Empty+seed | seed | seed only or empty per design |

---

## Under the Hood

Foreach loop; no short-circuit. PLINQ **`Aggregate`** needs **associative** op. EF: generic **`Aggregate`** usually client-side. String concat **`Aggregate`** — O(n²); **`StringBuilder`**.

---

## Examples

### Product с multiplicative identity

```csharp
var product = numbers.Aggregate(1, (acc, n) => acc * n);
// Empty → 1
```

Seed задаёт identity для empty — осознанный выбор.

### Running totals через Scan

```csharp
var running = balances.Scan(0m, (acc, tx) => acc + tx.Amount);
foreach (var rt in running)
    Console.WriteLine(rt);
```

**`Scan`** deferred — partial sums по мере enumeration.

### Reducer / event sourcing shape

```csharp
var finalState = events.Aggregate(InitialState, (state, e) => state.Apply(e));
```

Тот же mental model, что Redux **`reduce`** — fold как design pattern.

---

## Common Mistakes & Anti-patterns

**No-seed на possibly empty** — exception; для aggregates с identity используй seed overload.

**`Aggregate` вместо `Sum`** — шум и хуже optimization/translations.

**String `acc + item` in fold** — O(n²) allocations; **`StringBuilder`** seed или **`string.Join`**.

**PLINQ `Aggregate` с non-associative op** — wrong results under partition.

**`Scan` на `IQueryable` ожидая SQL window** — push running totals в SQL **`SUM() OVER`**.

---

## Production & Real-World Notes

Dashboard running metrics на большой history — SQL window functions, не in-memory **`Scan`**. Configuration merge через ordered **`Aggregate`** — порядок patches matters; fold не commutative unless designed.

Reviewers flag «clever **`Aggregate`**» когда есть **`Sum`/`ToDictionary`/`MaxBy`**.

---

## Key Takeaways

- Terminal fold vs running **`Scan`**.
- Seed overload для empty safety.
- Dedicated operators when possible.
- SQL running totals → window functions.
