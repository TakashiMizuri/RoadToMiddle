# Naming Conventions and XML Documentation Comments

> Roadmap: `1.1.20` · Node: `1.1` — C# Language Basics · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Apply **Microsoft C# naming conventions** consistently for types, members, parameters, and private fields.
- Write **XML documentation comments** (`///`) that populate IntelliSense and generated API docs.
- Use **`param`**, **`returns`**, **`exception`**, **`see`**, **`seealso`**, and **`cref`** correctly.
- Enable **`GenerateDocumentationFile`** and resolve **CS1591** warnings pragmatically in team projects.
- Review names and docs in PRs for clarity, not ceremony — production mindset.

## Why This Matters

You are closing the **C# language basics** node (`1.1`). You know types, control flow, methods, enums, nullability, and immutability keywords. Code that compiles is not yet code a team can maintain. **Naming** is the first documentation layer: **`CalculateTotal`** vs **`Calc`**, **`cancellationToken`** vs **`ct`**, **`_repository`** vs **`repo1`**. Inconsistent naming forces readers to decode every file anew; consistent conventions let them focus on business logic.

**XML documentation comments** are the second layer — machine-readable summaries attached to public API surfaces. Visual Studio, Rider, and VS Code show them in IntelliSense; **DocFX**, **Sandcastle**, and NuGet.org render them for package consumers. ASP.NET Core OpenAPI tools can surface descriptions from XML in some setups. Middle developers write libraries consumed by other teams, expose Minimal API endpoints documented for frontend developers, and review juniors whose **`///`** blocks copy the method name without adding meaning.

This lesson is practical: conventions used in real .NET repositories (including ASP.NET Core itself), plus how to write docs that help under pressure — not boilerplate that rots.

## Core Concepts

### Framework Design Guidelines: Naming Overview

Microsoft's **Framework Design Guidelines** (FDG), adapted by **.NET** style analyzers, establish defaults teams follow unless documented otherwise.

**Types** (classes, structs, enums, records, delegates, interfaces) use **PascalCase**: **`OrderService`**, **`PaymentStatus`**, **`IEmailSender`**. **Interfaces** prefix **`I`**: **`IRepository`**, not **`RepositoryInterface`**. Generic type parameters use **`T`** or descriptive **`TKey`**, **`TValue`**, **`TRequest`**.

**Methods**, **properties**, **events**, and **public fields** (rare) use **PascalCase**: **`GetByIdAsync`**, **`CreatedAt`**, **`OrderPlaced`**.

**Parameters** and **local variables** use **camelCase**: **`orderId`**, **`cancellationToken`**. Avoid Hungarian notation (`strName`, `iCount`) — type is visible from IDE and compiler.

**Private instance fields** in modern .NET code commonly use **`_camelCase`** with leading underscore: **`_logger`**, **`_dbContext`**. Some codebases use **`m_`** legacy prefix — match project style. **Static fields** sometimes **`s_`** or PascalCase for **`const`**/**`static readonly`** public constants.

**Async methods** suffix **`Async`** by convention: **`SendEmailAsync`**. Returns **`Task`** or **`Task<T>`** — the suffix signals awaitable intent to callers (FDG; analyzers may warn if missing).

**Namespaces** use **PascalCase** dotted hierarchy matching folder structure: **`Company.Product.Feature`**.

Names should be **nouns** for types, **verbs or verb phrases** for methods (**`Calculate`**, **`Validate`**, **`ToString`** exception). Boolean properties read as questions: **`IsActive`**, **`HasPermission`**, **`CanExecute`**.

Avoid abbreviations unless universal (**`Id`**, **`Url`**, **`Async`**). **`IO`** is fine; **`OrdProc`** is not.

### Consistency Beats Personal Taste

Teams document exceptions in **`CONTRIBUTING.md`** or **`.editorconfig`** + **Roslyn analyzers** (**`IDE1006`**, **`CA1707`** identifiers should not contain underscores — except **`_field`** pattern often suppressed in style rules). When joining a project, **match existing style** over FDG pedantry — consistency reduces cognitive load.

Public API surface demands stricter naming — breaking renames cost consumers. Internal private names can be refactored freely.

### XML Documentation Comments Basics

Place **`///`** immediately above types and members. The compiler extracts comments into an **`.xml`** file when **`GenerateDocumentationFile`** is enabled.

```csharp
/// <summary>
/// Calculates the total amount for an order including tax and shipping.
/// </summary>
/// <param name="orderId">The unique identifier of the order.</param>
/// <param name="cancellationToken">Token to cancel the database lookup.</param>
/// <returns>The total in the order's currency.</returns>
/// <exception cref="OrderNotFoundException">Thrown when no order exists for <paramref name="orderId"/>.</exception>
public Task<decimal> CalculateTotalAsync(Guid orderId, CancellationToken cancellationToken = default)
```

**`<summary>`** — what it does, in complete sentences. First sentence often appears alone in IntelliSense tooltips — put the gist there.

**`<param name="...">`** — must match parameter name exactly; documents each input.

**`<returns>`** — meaning of return value; omit for **`void`** / **`Task`** without meaningful result.

**`<exception cref="...">`** — documents thrown exception types; **`cref`** must resolve to a type (use full name if ambiguous).

**`<paramref name="..."/>`** — inline reference to parameter inside other tags.

**`<see cref="..."/>`** — inline link to another member. **`<seealso href="..."/>`** — external links.

**`<remarks>`** — extended notes, usage caveats, thread safety.

**`<example>`** with **`<code>`** — usage sample; valuable for libraries.

### cref and Cross-References

**`cref`** attributes encode references the doc compiler resolves:

```csharp
/// <see cref="CalculateTotalAsync(Guid, CancellationToken)"/>
/// <see cref="OrderNotFoundException"/>
```

Angle brackets in generics are encoded: **`List{T}`** in cref strings. Wrong **`cref`** fails doc build or produces broken links.

### What to Document

Document **public** and **protected** API. **`internal`** if exposed via **`InternalsVisibleTo`** for test or friend assemblies. Private members usually skip XML unless complex algorithm needs maintainer note (regular **`//`** comment suffices).

Do not restate obvious signatures:

```csharp
/// <summary>Gets or sets the name.</summary>  // weak
public string Name { get; set; }
```

Better for non-obvious property:

```csharp
/// <summary>
/// Display name shown in the storefront; may differ from the internal SKU label.
/// </summary>
public string DisplayName { get; set; }
```

Document **null behavior** when nullable reference types enabled:

```csharp
/// <param name="email">Must not be null or empty.</param>
public void Send(string email)
```

Document **units** (**milliseconds**, **UTC** vs local), **valid ranges**, and **side effects** (sends email, writes file).

### Enabling Documentation Output

In **`.csproj`**:

```xml
<PropertyGroup>
  <GenerateDocumentationFile>true</GenerateDocumentationFile>
  <!-- Optional: treat missing docs as warnings -->
  <NoWarn>$(NoWarn);1591</NoWarn>
</PropertyGroup>
```

**`CS1591`** warns on undocumented public members. Teams either document all public API, suppress selectively, or disable warning in early prototypes — production libraries should not silence permanently without policy.

NuGet packages include **`.xml`** alongside **`.dll`** for consumer IntelliSense.

## Under the Hood

The C# compiler parses **`///`** into **`DocumentationComment`** nodes in the syntax tree. Roslyn emits **`AssemblyMetadataAttribute`**-linked **`*.xml`** in the [Sandcastle/doc comment XML format](https://learn.microsoft.com/en-us/dotnet/csharp/language-reference/xmldoc/).

Structure:

```xml
<member name="M:Namespace.OrderService.CalculateTotalAsync(System.Guid,System.Threading.CancellationToken)">
  <summary>...</summary>
  <param name="orderId">...</param>
</member>
```

IDE loads XML from referenced assemblies from **`lib/net8.0/YourLib.xml`**. No runtime cost — comments stripped from IL; metadata lives in separate file.

Analyzers (**`SA1600`** StyleCop, **`CS1591`**) enforce team doc coverage in CI.

## Syntax / Commands / API

Common XML tags:

| Tag | Purpose |
|-----|---------|
| `<summary>` | Short description |
| `<remarks>` | Extended discussion |
| `<param>` | Parameter docs |
| `<returns>` | Return value |
| `<exception cref="">` | Thrown exceptions |
| `<see cref="">` | Inline cross-ref |
| `<seealso>` | See also section |
| `<example>` | Usage example |
| `<typeparam>` | Generic parameter |

Naming quick table:

| Element | Convention | Example |
|---------|------------|---------|
| Class | PascalCase noun | `OrderRepository` |
| Interface | I + PascalCase | `IOrderRepository` |
| Method | PascalCase verb + Async | `GetByIdAsync` |
| Property | PascalCase | `IsPublished` |
| Parameter | camelCase | `orderId` |
| Private field | _camelCase | `_logger` |
| Const | PascalCase | `DefaultPageSize` |

## Examples

### Example 1: Well-named Minimal API handler

```csharp
public static class OrderEndpoints
{
    public static RouteGroupBuilder MapOrderEndpoints(this RouteGroupBuilder group)
    {
        group.MapGet("/{orderId:guid}", GetOrderByIdAsync);
        return group;
    }

    private static async Task<IResult> GetOrderByIdAsync(
        Guid orderId,
        IOrderQueries queries,
        CancellationToken cancellationToken)
    {
        var order = await queries.FindByIdAsync(orderId, cancellationToken);
        return order is null ? Results.NotFound() : Results.Ok(order);
    }
}
```

**`MapOrderEndpoints`** — extension naming pattern ASP.NET uses. Parameters descriptive; **`cancellationToken`** full word.

### Example 2: Library-quality XML on exception type

```csharp
/// <summary>
/// Thrown when an order identifier does not match any persisted order.
/// </summary>
public sealed class OrderNotFoundException : Exception
{
    /// <summary>
    /// Initializes a new instance with the missing order id.
    /// </summary>
    /// <param name="orderId">The identifier that was not found.</param>
    public OrderNotFoundException(Guid orderId)
        : base($"Order '{orderId}' was not found.")
    {
        OrderId = orderId;
    }

    /// <summary>Gets the order identifier that was not found.</summary>
    public Guid OrderId { get; }
}
```

### Example 3: Generic interface documentation

```csharp
/// <summary>
/// Reads entities by primary key.
/// </summary>
/// <typeparam name="TEntity">The entity type.</typeparam>
/// <typeparam name="TKey">The key type.</typeparam>
public interface IReadRepository<TEntity, TKey>
    where TEntity : class
{
    /// <summary>
    /// Finds an entity by key, or returns null if none exists.
    /// </summary>
    /// <param name="id">The primary key value.</param>
    /// <param name="cancellationToken">Cancellation token.</param>
    /// <returns>The entity, or null when not found.</returns>
    Task<TEntity?> FindAsync(TKey id, CancellationToken cancellationToken = default);
}
```

## Common Mistakes & Anti-patterns

**Cryptic abbreviations** — **`ProcOrd`** saves keystrokes, wastes review time. Fix: full words.

**Missing `Async` suffix** on async public methods — callers miss await contract. Fix: rename + analyzer.

**XML that duplicates signature** — adds noise, drifts when signature changes. Fix: document behavior, edge cases, units.

**Wrong `param name`** — doc build warning or misleading IntelliSense. Fix: match identifier exactly.

**Public API rename without `[Obsolete]` path** — breaking change. Fix: semver + migration notes in **`remarks`**.

**Inconsistent field naming** — mix **`_field`**, **`m_field`**, **`field`** in one project. Fix: team rule in editorconfig.

## Production & Real-World Notes

Open-source .NET packages treat missing public docs as CI failure. Internal apps may document only shared **Application** / **Domain** layers.

Swagger/OpenAPI descriptions often duplicated from XML via Swashbuckle **`IncludeXmlComments`** — single source of truth in **`///`**.

Code review: reject PRs that add public members without summary when project requires docs.

Localization: XML docs usually **English** in industry codebases regardless of team spoken language — matches BCL and NuGet ecosystem.

---

## Comparison / Trade-offs

**XML `///` vs regular `//`** — XML for public API tooling; **`//`** for implementation notes.

**Verbose names vs brevity** — clarity wins in shared code; tight scope locals can be short (**`i`** loop index acceptable).

**Document everything vs document behavior** — latter scales; former creates stale noise.

**StyleCop strictness vs velocity** — startups relax **`SA1600`**; libraries tighten before 1.0.

## Key Takeaways

1. **PascalCase** for types and members; **camelCase** for parameters; **`_camelCase`** private fields is mainstream .NET.
2. Interfaces **`I` prefix**; async methods **`Async` suffix**; generics **`T`** or descriptive **`TKey`**.
3. **`///` XML docs** feed IntelliSense and package documentation — not runtime behavior.
4. **`<summary>` first sentence** is the high-visibility line; document behavior, units, null rules.
5. **`cref`** must resolve; match **`param name`** to identifiers.
6. Enable **`GenerateDocumentationFile`** for libraries; manage **CS1591** with team policy.
7. Consistency within the repo matters more than debating underscore fields in isolation.
