# Exercises: Naming Conventions and XML Documentation

> Complete after lectures and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Rename refactor
**Goal:** Apply FDG naming.

**Tasks:**
Refactor this class names only (keep logic):

```csharp
public class ord_proc
{
    private ILogger log;
    public async Task<dto> get(int ID, CancellationToken ct) { /* ... */ }
}
```

**Success criteria:** PascalCase type, `_logger` or team convention, `GetAsync` suffix, camelCase params `id`, `cancellationToken`.

---

### E2. First XML docs
**Goal:** Write meaningful `///` blocks.

**Tasks:**
Add XML to:

```csharp
public sealed class EmailValidator
{
    public bool IsValid(string email) { /* ... */ }
}
```

Document: empty/null invalid; does not throw.

**Success criteria:** `<summary>` explains behavior, not «Checks validity» only; `<param>` for email.

---

## Medium

### M1. Document a small service
**Goal:** Library-style public API.

**Tasks:**
Create `IProductCatalog` with `Task<Product?> GetBySkuAsync(string sku, CancellationToken cancellationToken)`. Full XML on interface + implementation class summary.

Enable `<GenerateDocumentationFile>true</GenerateDocumentationFile>`; build with zero CS1591 on public members you added.

**Success criteria:** Build succeeds; XML file contains your `<member>` entries.

---

### M2. Fix broken documentation
**Goal:** Code review skill.

**Tasks:**
Given intentionally broken XML (wrong param names, empty summary, bad cref), fix all doc compiler warnings.

**Success criteria:** List each fix and why.

---

### M3. OpenAPI-ready endpoint
**Goal:** Connect XML to Swagger (if Swashbuckle present) or describe steps.

**Tasks:**
1. Minimal API with one documented extension method in static class.
2. XML on method describing 200/404 responses in `<remarks>`.
3. Note how `IncludeXmlComments` would wire XML into Swagger.

**Success criteria:** Names follow `MapXEndpoints` pattern; parameter `cancellationToken` spelled out.

---

## Hard (stretch)

### H1. Style guide excerpt
**Goal:** Team policy document.

**Tasks:**
Write 10-line `CONTRIBUTING` naming section for your repo: fields, async, interfaces, abbreviations allowed (`Id`, `Url`).

**Success criteria:** Resolves underscore vs `m_` explicitly; English XML docs policy.
