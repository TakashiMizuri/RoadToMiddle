# Fire-and-forget Task — безопасные паттерны и ловушки

> Roadmap: `1.4.30` · Node: `1.4` — C# async/await · Depth: **глубоко**

## Why This Matters

Вы отдаёте `202 Accepted` и запускаете отправку письма через `_ = SendEmailAsync()`. Через время production падает или теряет события, потому что:

1. **`SendEmailAsync` бросил исключение** — никто не await'ил → **unobserved** (`1.4.31`).
2. Task удерживает **scoped `DbContext`** — запрос завершился, scope disposed → **`ObjectDisposedException`**.
3. Под нагрузкой тысячи detached tasks **голодают thread pool** (`1.4.10`).

Fire-and-forget не запрещён — он **архитектурно ограничен**. Middle выбирает **явную фоновую инфраструктуру** вместо отбрасывания `Task`.

## Core Concepts

### Что значит fire-and-forget

**Fire-and-forget** — запуск async-работы **без await** текущего `Task`:

```csharp
_ = ProcessOrderAsync(orderId);
ProcessOrderAsync(orderId); // CS4014 без await
Task.Run(() => DoWork());
```

HTTP-ответ может уйти **раньше** завершения работы — так задумано для email, webhooks, аналитики. Ошибка — думать, что runtime **хостит, скоупит и наблюдает** как при await.

### Три ловушки

**1. Ненаблюдаемые исключения.** Discarded task не пробрасывает fault в pipeline запроса — только `UnobservedTaskException` или финализация.

**2. Captive scoped dependencies.** DI scope на запрос; `DbContext` и scoped-сервисы **dispose** при завершении request. Фоновый task держит **мертвые** ссылки.

**3. Неверный токен.** `RequestAborted` для работы **после** response отменит задачу при disconnect клиента — часто нежелательно для email/биллинга.

### async void — не для сервера

```csharp
private async void Send() { ... }  // NEVER (кроме UI events)
```

Исключения из `async void` — unhandled; нельзя await. Используй **`async Task`** и хост.

### Task.Run в web-запросе

`Task.Run` гоняет на **thread pool**. В ASP.NET Core это **не** «освобождает request thread» (его нет в классическом смысле). Для I/O-bound — лишний overhead. Для CPU-bound offload — всё равно **await** или очередь.

## Under the Hood

### Disposal request scope

По завершении pipeline **`IServiceScope`** запроса dispose'ится. Фоновый task с `_db` → **`ObjectDisposedException`**.

Fix: **`IServiceScopeFactory.CreateScope()`** в worker'е — **новые** scoped-сервисы на job.

### Путь исключения

```
Task fault → нет await → нет catch в request
→ (позже) UnobservedTaskException
```

**Await** в consumer очереди или явный fault handler.

### Hosting

`BackgroundService` + **`stoppingToken`** — отдельный lifecycle от HTTP. Регистрируй **`IHostedService`** для фоновой обработки.

## Examples

### Антипаттерн

```csharp
[HttpPost]
public async Task<IActionResult> Create(OrderDto dto, CancellationToken ct)
{
    await _db.SaveChangesAsync(ct);
    _ = NotifyWarehouseAsync(dto); // scoped service
    return Accepted();
}
```

### Безопаснее — очередь

```csharp
await _queue.QueueBackgroundWorkItemAsync(async sp =>
{
    var notifier = sp.GetRequiredService<IWarehouseNotifier>();
    await notifier.NotifyAsync(dto.Id);
});
return Accepted();
```

Consumer создаёт **scope на item** (`1.4.34`).

## Common Mistakes & Anti-patterns

**`_ =`** без хоста — нет «фонового контекста запроса».

**Захват `HttpContext`** в долгоживущих задачах.

Fire-and-forget для **транзакционной** работы до 200 OK.

**Backpressure** — unbounded detached tasks → память и pool starvation.

---

## Production & Real-World Notes

**Outbox** — событие в той же транзакции, что заказ; worker читает outbox.

**Correlation** — проброс trace ID в фон.

**Idempotency** на consumer — retries неизбежны.

---

## Quick Reference

- Не fire-and-forget **scoped** из request.
- Не **`async void`** на сервере.
- **Await** в hosted consumer.
- **`IServiceScopeFactory`** на job.
- Не **`RequestAborted`** для post-response без намерения.
- **Очередь + BackgroundService** вместо `_ =`.

---

## Key Takeaways

- Fire-and-forget ломает **exception path** и **DI scope**.
- **`_ = SomeAsync()`** в controller — smell → очередь.
- Новый scope на фоновую единицу работы.
- **`Task.Run`** ≠ web background для I/O.
- Production: **очереди, outbox, workers**.
- Логируй фоновые сбои — тихая потеря хуже медленного ответа.
