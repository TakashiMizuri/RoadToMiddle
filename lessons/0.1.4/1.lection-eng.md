# .git/ Structure: Objects, Refs, and HEAD

After this lesson you will be able to:

- Navigate the `.git/` directory and explain the purpose of each major component.
- Describe the four Git object types: **blob**, **tree**, **commit**, and **tag**.
- Explain **content-addressable storage** — why the hash *is* the identity of an object.

## Why This Matters

In lesson `0.1.3` you learned the everyday commands: `git init`, `git add`, `git commit`, `git log`. Those commands are the **interface** — the steering wheel and pedals. The `.git/` directory is the **engine room** — the machinery that actually stores history, resolves branches, and makes every command possible.

Most developers can drive Git for months without ever opening `.git/`. That works until something breaks. A colleague reports "detached HEAD" and you do not know whether their work is safe. A CI job claims a commit is missing. Someone accidentally runs a destructive command and asks whether history can be recovered. A repository feels corrupted after a bad merge or a partial clone. In each of these situations, the fix begins with one question: *what does Git actually store, and where?*

Middle-level developers do not memorize every file inside `.git/`. They carry a mental model instead. History is not a list of diffs stored in a database table. It is a **graph of immutable objects** linked by cryptographic hashes. Branch names are not folders full of files; they are **lightweight pointers** — tiny text files that say "the tip of `main` is commit `abc123`." `HEAD` is usually just another small file that answers "where am I right now?" Once you internalize that model, commands like `git reflog`, `git fsck`, merge conflicts, rebases, and detached HEAD stop feeling like magic and start feeling like predictable mechanics.

This lesson opens the hood. We will walk through the layout of `.git/`, understand the four object types, see how refs and HEAD sit above the object store, and trace exactly what happens on disk when you stage and commit a file. By the end, you should be able to explain Git's storage model to a teammate on a whiteboard — and know which lever to pull when something goes wrong.

## Core Concepts

### The `.git/` Directory: The Repository Itself

When you run `git init` in a project folder, Git creates a hidden directory named `.git/`. That directory **is** the repository. Everything else — your source files, README, configuration you edit in the IDE — lives in the **working tree**, which is simply a checkout of one particular snapshot. The working tree is mutable and disposable; you can delete every tracked file outside `.git/` and still recover the entire project from the object database, as long as `.git/` remains intact.

Think of the relationship this way. The working tree is your draft on the desk. `.git/` is the archive in the basement — every version you ever saved, indexed and cross-referenced. Commands like `git checkout` copy a snapshot from the archive to the desk. Commands like `git add` and `git commit` decide what from the desk gets sealed into the archive.

A freshly initialized repository contains a skeleton. At minimum you will find `HEAD` (a pointer to your current position), `config` (repository settings including remotes once you add them), `description` (used mainly in bare repositories), `hooks/` (scripts that can run on events like pre-commit), `info/` (including global exclude patterns), `objects/` (initially almost empty, but this is where all content will live), and `refs/` (where branch and tag pointers will appear). After your first `git add`, Git also creates `index` — the staging area as a binary file on disk.

As the repository matures, more files and folders appear. `refs/heads/main` (or `master`) will hold the forty-character hash of your latest commit on `main`. `refs/remotes/origin/main` will appear after you fetch from a remote, recording where the remote branch was last seen. The `logs/` directory stores **reflog** data — a safety net that remembers where branch tips used to point. Files like `COMMIT_EDITMSG`, `FETCH_HEAD`, and `ORIG_HEAD` are ephemeral bookkeeping; they help Git remember context across operations.

The following diagram is not something you must memorize file-by-file, but it anchors the mental map. Notice the separation: `objects/` holds **content**, `refs/` holds **names for commit tips**, and `HEAD` sits above refs telling Git which name (or raw commit) is currently checked out.

```
my-project/
├── .git/                 ← THE repository
│   ├── HEAD              ← "where am I?"
│   ├── config            ← repo settings, remotes
│   ├── hooks/            ← scripts on events (pre-commit, etc.)
│   ├── info/             ← global exclude patterns
│   ├── objects/          ← ALL content (blobs, trees, commits, tags)
│   ├── refs/             ← branch/tag pointers
│   └── index             ← staging area (binary)
├── README.md             ← working tree
└── src/
```

### Content-Addressable Storage: Identity Equals Content

Before we name the four object types, we need the principle that unifies them. Git does not assign random serial numbers to stored data. It uses **content-addressable storage**: the identifier of an object **is** a cryptographic hash of its contents.

For most repositories today, that hash is **SHA-1** — forty hexadecimal characters. (Git 2.29+ optionally supports SHA-256 for new repositories; the storage layout and object model are the same, only the hash length changes.) When Git stores an object, it computes `hash = SHA-1(type + " " + size + "\0" + content)`, then writes the compressed object to a path derived from that hash. The first two hex digits become a subdirectory name; the remaining thirty-eight become the filename. A blob storing the bytes `hello\n` might live at `.git/objects/e6/9de29bb2d1d6434b8b29ae775ad8c2e48c5391`.

This design has profound consequences. **Determinism:** the same content always produces the same hash, on every machine, forever. **Integrity:** if a single bit on disk flips, the stored hash will no longer match the recomputed hash, and Git can detect corruption. **Deduplication:** if two files — or two versions of history — contain identical bytes, they share one blob object. Storage grows with *changes*, not with redundant copies.

The hash is not metadata attached to content. It **is** the object's identity. When one commit points to a parent hash, it is pointing to an exact, verified bundle of bytes. That chain is what makes Git history tamper-evident — a topic we develop further in lesson `0.1.5`.

### The Four Object Types

Git's object database is deliberately minimal. Every piece of stored history is one of exactly four types: **blob**, **tree**, **commit**, or **tag**. Each type is a particular structured use of the same content-addressing mechanism.

A **blob** (binary large object) is the simplest: raw file content and nothing else. No filename, no path, no permission bits — just bytes. When you `git add README.md`, Git reads the file from the working tree, wraps the bytes in a blob header, hashes the result, and stores it. If you later add another file with identical content under a different name, the index will reference the same blob hash. Two paths, one object. This is deduplication in action.

A **tree** is Git's way of representing a directory. It is a flat list of entries. Each entry says: this filename (or subdirectory name), this mode (regular file, executable, subdirectory), this object hash. A tree might map `README.md` to a blob hash and `src/` to another tree hash. Trees are built bottom-up during commit: the tree for `src/` is constructed first, then the root tree references it. A commit's snapshot is not a tarball of files; it is a pointer to a root tree, which fans out to more trees and blobs.

A **commit** ties a snapshot to history and metadata. Its body contains a `tree` line (the root tree hash), zero or more `parent` lines (previous commits — omitted on the very first commit), `author` and `committer` lines with timestamps, and a free-text message after a blank line. Critically, a commit does **not** store diffs. It stores a full snapshot reference via the tree. When you run `git diff` between two commits, Git compares their trees and derives the diff on demand. That choice — store snapshots, compute diffs — makes branching cheap and random access fast.

A **tag** marks a named point in history. A **lightweight tag** is simply a ref file under `refs/tags/` containing a commit hash — no tag object involved. An **annotated tag** (created with `git tag -a`) is a full **tag object** with its own message, tagger identity, and pointer to a commit (or other object). Release engineering often prefers annotated tags because they carry release notes and can be signed.

The conceptual layout of a blob header illustrates how type and size are part of what gets hashed — not just the raw user bytes:

```
blob object on disk (conceptual):

  blob 6\0hello\n
        ↑   ↑
     header  content
```

A tree object, by contrast, reads like a directory listing tied together by hashes:

```
tree object (conceptual):

  tree 72\0
  100644 README.md\0<blob-hash>
  100644 app.cs\0<blob-hash>
  040000 src\0<tree-hash-for-src/>
```

Mode `100644` means an ordinary file; `040000` means a subdirectory whose contents live in a nested tree object. The commit sitting above this tree adds the time dimension — who, when, and which parent commits led here.

### Refs: Human-Readable Names for Commits

If objects are identified only by forty-character hashes, navigation would be painful. **Refs** (references) are Git's solution: small, mutable files that map friendly names to commit hashes.

Local branches live under `.git/refs/heads/`. The file `.git/refs/heads/main` might contain a single line: `a1b2c3d4e5f6789...`. That is the **tip** of `main` — the newest commit on that branch. When you create a branch with `git branch feature/login`, Git creates `refs/heads/feature/login` pointing at the current commit. No files are copied; only a new pointer is added.

Tags live under `.git/refs/tags/`. Remote-tracking branches — Git's memory of what the remote looked like after the last fetch — live under `.git/refs/remotes/<remote>/`, for example `refs/remotes/origin/main`.

The crucial distinction is mutability. **Objects are immutable.** Once written, a blob, tree, or commit never changes. **Refs are mutable.** Every new commit on `main` updates `refs/heads/main` to the new hash. Old commits remain in the object database; they simply are no longer the tip unless some ref or tag still points to them or they are reachable through ancestry.

This is why deleting a branch is cheap: you delete a forty-one-byte file, not gigabytes of history. The commits may still exist until garbage collection removes unreachable objects — but the branch name is gone.

### HEAD: Where You Are in the Graph

**HEAD** answers the question "what commit is my working tree based on right now, and usually which branch am I on?" It is stored in `.git/HEAD`. In the common case, HEAD is a **symbolic ref** — it does not contain a commit hash directly. It contains a line like `ref: refs/heads/main`, meaning "resolve my position by following the `main` branch ref."

The chain looks like this:

```
HEAD ──→ refs/heads/main ──→ commit abc1234
         (symbolic)           (40-char hash)
```

When you `git checkout feature`, Git updates HEAD to `ref: refs/heads/feature`. Your working tree files are replaced with the snapshot from that branch's tip commit. When you `git commit`, Git creates a new commit whose parent is the current HEAD commit, then moves the branch ref forward. HEAD still says `ref: refs/heads/feature`, but that ref now points to the new hash.

**Detached HEAD** is the exceptional state. If HEAD contains a raw commit hash instead of `ref: refs/heads/...`, you are checked out directly on a commit, not on a branch. You can look around and experiment, but new commits you make are not attached to any branch name. Switch back to a branch and those commits can become unreachable — saved temporarily by reflog, but easy to lose if you do not create a branch or tag. Understanding HEAD as a file — sometimes symbolic, sometimes direct — makes detached HEAD a diagnostic state, not a mystery error.

## Under the Hood

### How `git add` Creates Objects

Let us trace `git add README.md` at the byte level, because this is where the working tree first meets the object database.

Git reads the file bytes from your working tree. It constructs a header consisting of the word `blob`, a space, the size of the content, a null byte, then the content itself. It feeds that entire byte sequence into SHA-1. The resulting hash is the object's identity. Git zlib-compresses the header-plus-content bundle and writes it to `.git/objects/ab/cdef...` where `ab` is the first two hex digits of the hash. This on-disk format is called a **loose object** — one file per object, written immediately.

The add operation also updates the **index** — the file `.git/index`. The index is not a Git object; it is mutable staging state. For each path you have staged, the index records the path, the blob hash, file metadata (stat information), and a **stage number** used during merge conflict resolution. Think of the index as the blueprint for the next commit: a list of paths mapped to blob hashes that will be assembled into trees when you commit.

The flow from working tree to objects can be sketched as follows. Notice that unchanged files may skip re-hashing if Git determines — via metadata comparison — that the working tree file matches what is already staged. That optimization matters on large repositories.

```
Working tree          Index (.git/index)         Objects
─────────────         ──────────────────         ───────
README.md  ──add──→   README.md → blob:abc       abc = blob "..."
src/app.cs ──add──→   src/app.cs → blob:def      def = blob "..."
```

You can even create a blob manually without the index, which is instructive when learning:

```bash
git hash-object -w README.md   # write blob, print hash
git ls-files -s                  # staged: mode hash stage path
```

The first command proves that blobs exist independently of commits. The second shows how the index references those blobs by path.

### How `git commit` Creates Objects

`git commit` is where snapshots become history. Git reads every entry in the index — each path and its blob hash — and constructs **tree objects** from the bottom up. For a path like `src/app.cs`, the blob hash is known. Git builds a tree for `src/` listing `app.cs`. At the repository root, it builds a tree listing `README.md` and the `src/` subtree. Each tree is hashed and stored as its own object, just like blobs.

Next Git assembles the **commit object**. It writes the root tree hash, the parent commit hash (unless this is the first commit), author and committer lines, the message, then hashes and stores the commit. Finally it updates the current branch ref — for example `refs/heads/main` — to the new commit hash. If HEAD is symbolic and points at that branch, no HEAD update is needed; the branch moved, and HEAD still means "follow that branch."

Consider a second commit that changes only `README.md`. Before the commit, `HEAD` points through `refs/heads/main` to commit A. The index holds a new blob for `README.md` but the same blob for `src/app.cs`. After the commit, `refs/heads/main` points to commit B. Commit B's parent is A. Its tree references the new README blob and reuses the existing `src/` tree structure where possible. Commit A and all its objects still exist unchanged. History is append-only.

```
Before commit:
  HEAD → refs/heads/main → commit A
  Index: README.md (blob v2), src/app.cs (blob v1)

After commit:
  HEAD → refs/heads/main → commit B
                              ├── parent: A
                              └── tree: T2
                                    ├── README.md → blob v2
                                    └── src/ → tree
                                          └── app.cs → blob v1

  commit A and its tree still exist — immutable history
```

This diagram is worth staring at. It explains why Git is fast at branching (new ref, shared objects), why rebasing creates new commits (new parents mean new hashes), and why `git reset --soft` can move a branch pointer without touching objects at all.

### Loose Objects vs Packfiles

Early in a repository's life, every `git add` and `git commit` creates loose objects under `.git/objects/`. Each object is one zlib-compressed file in a two-level directory. Writes are simple and fast.

Over months and thousands of commits, loose objects accumulate. Similar blobs — successive versions of the same source file — differ slightly. Storing each version as a full blob works but wastes space. Git's maintenance command `git gc` (garbage collection) packs reachable objects into **packfiles**: `.git/objects/pack/pack-*.pack` with a companion `.idx` index file. Inside a pack, Git can store objects **delta-compressed** against similar objects. A recent version of `Program.cs` might be encoded as a thin delta on top of an older revision.

You rarely manage packs manually. They matter when you read `git count-objects -vH` and wonder why `.git` is smaller after gc, or when a fetch delivers a pack from a remote. Conceptually: **loose objects are the write path; packs are the optimized read/storage path after housekeeping.**

```
Loose:     .git/objects/e6/9de29bb2d1d6434b8b29ae775ad8c2e48c5391
Packed:    .git/objects/pack/pack-a1b2c3d4.pack
           .git/objects/pack/pack-a1b2c3d4.idx
```

### Inspecting Objects Yourself

Middle developers should know the inspection commands. They turn abstract theory into concrete evidence.

```bash
# Find current branch tip
cat .git/refs/heads/main

# See what HEAD points to
cat .git/HEAD

# Object type and pretty-print
git cat-file -t abc1234      # commit | tree | blob | tag
git cat-file -p abc1234      # pretty-print content

# List all objects (can be huge)
find .git/objects -type f
```

Run these on a small practice repository after a couple of commits. `git cat-file -p HEAD` shows the commit; copy the tree hash from that output and run `git ls-tree <tree-hash>` to see directory entries. Within minutes you have walked from branch name to commit to tree to blob — the full chain Git uses on every operation.

## Syntax / Commands / API

The plumbing commands below are the ones you will reach for when debugging or learning internals. You do not use them daily in normal feature work, but they map directly to the storage model described above.

| Command | Purpose |
|---------|---------|
| `git init` | Create `.git/` skeleton |
| `git hash-object -w <file>` | Create blob object manually |
| `git cat-file -t <hash>` | Object type |
| `git cat-file -p <hash>` | Object contents |
| `git ls-tree <tree-hash>` | List directory tree |
| `git rev-parse HEAD` | Resolve HEAD to commit hash |
| `git symbolic-ref HEAD` | Show branch HEAD points to |
| `git update-ref refs/heads/test <hash>` | Move branch pointer (advanced) |
| `git count-objects -vH` | Loose vs packed statistics |
| `git gc` | Pack objects, optimize repo |

`git rev-parse HEAD` is especially useful: it follows symbolic refs for you and prints the raw commit hash that HEAD ultimately references.

## Examples

### Example 1: Fresh Repository Anatomy

The best way to cement this lesson is a five-minute experiment. Create a tiny repository, make one commit, then read `.git/` directly.

```bash
mkdir demo && cd demo
git init
echo "# Demo" > README.md
git add README.md
git commit -m "Initial commit"
```

After the commit, list `.git/`. You should see `HEAD`, `config`, `objects`, `refs`, `index`, `logs`, and `COMMIT_EDITMSG` among others. Reading `HEAD` shows `ref: refs/heads/main` — symbolic. Reading `.git/refs/heads/main` shows the full commit hash. Running `git cat-file -p HEAD` pretty-prints the commit object with its tree, author, and message. You have just traversed refs → commit → (next) tree → blob without any GUI.

### Example 2: Same Content, One Blob

Deduplication is not theoretical. Create two files with identical bytes, stage both, and inspect the index.

```bash
echo "shared" > file-a.txt
echo "shared" > file-b.txt
git add file-a.txt file-b.txt
```

`git ls-files -s` will show the same blob hash in the second column for both paths. Only one loose object was written. In a large monorepo with shared assets or generated code, this property keeps storage manageable.

### Example 3: Symbolic vs Detached HEAD

Checking out a branch keeps HEAD symbolic. Checking out a raw hash does not.

```bash
git checkout main
cat .git/HEAD
# ref: refs/heads/main

git checkout abc1234   # raw commit hash
cat .git/HEAD
# abc1234...   (detached — no "ref:" prefix)
```

The detached case is your signal to create a branch if you plan to commit, or to treat the session as read-only exploration.

## Common Mistakes & Anti-patterns

The most dangerous mistake is editing or deleting files inside `.git/objects/` by hand. Hashes will no longer match content; `git fsck` will report corruption; recovery may require restoring from a remote clone. Another catastrophic shortcut is deleting the entire `.git/` folder to "fix" merge conflicts — that destroys every object and ref; only the working tree files remain, often without complete history.

Many juniors imagine that a branch "contains" copies of project files. In reality a branch is a tiny ref file pointing to one commit, which points to a tree, which points to blobs. Confusing the **index** with the object store causes staging bugs: the index is mutable staging state; objects, once written, are permanent.

Assuming `git commit` stores diffs leads to confusion about why amending or rebasing rewrites history. Commits store **trees**; diffs are derived. Finally, committing in detached HEAD without creating a branch produces commits that become unreachable when you checkout elsewhere — reflog may rescue you for a while, but do not rely on luck.

## Production & Real-World Notes

On servers you will encounter **bare repositories** (`core.bare=true`): no working tree, only what would normally live in `.git/` at the repository root. This is standard for Git hosting. **Shallow clones** (`git clone --depth N`) omit most historical objects; some operations require full history. **Partial clones** and **sparse checkouts** fetch only needed objects or paths — essential in large monorepos.

When integrity is in question, `git fsck` walks objects and verifies linkage and hashes. The **reflog** under `logs/refs/heads/` records where branch tips used to point — a lifeline after accidental resets. CI pipelines often cache `.git/objects` or use shallow clones to save time; that is a deliberate trade-off between speed and full history availability.

## Comparison / Trade-offs

Git maintains several layers of state, and they differ in mutability and location. Your working tree files are fully mutable — ordinary source code. The index is mutable staging ground. Objects in `.git/objects/` are immutable once written. Branch refs and HEAD are mutable pointers that move forward as you commit or checkout.

| Concept | Mutable? | Stored where |
|---------|----------|--------------|
| Working tree files | Yes | Project directory |
| Index (staging) | Yes | `.git/index` |
| Objects (blob/tree/commit) | No | `.git/objects/` |
| Branch refs | Yes (tip moves) | `.git/refs/heads/` |
| HEAD | Yes | `.git/HEAD` |

Loose objects favor write speed during active development; packfiles favor disk efficiency and transfer size after `git gc`. Git uses both over a repository's lifetime.

## Quick Reference

```
.git/
  HEAD              → symbolic ref → branch → commit
  index             → staging: paths → blob hashes
  objects/          → blob | tree | commit | tag (by hash)
  refs/heads/       → local branches
  refs/tags/        → tags
  refs/remotes/     → remote-tracking branches
  config            → remotes, user, core settings
```

Object chain for one commit:

```
commit → tree → (blobs + subtrees) → file contents
       → parent commit(s)
```

## Key Takeaways

1. `.git/` **is** the repository; the working tree is a checkout of one commit.
2. Git stores **four object types**: blob, tree, commit, tag — all content-addressed by hash.
3. **Refs** are named pointers to commits; **HEAD** is usually a symbolic ref to a branch.
4. `git add` writes **blobs** and updates the **index**; `git commit` builds **trees** and a **commit**, then moves the branch ref.
5. Objects are **immutable**; branches and HEAD are **mutable**.
6. **Packfiles** compress and delta-store objects after `git gc` — loose objects are the write path.
7. Understanding this model explains branching, merging, detached HEAD, and recovery tools.
