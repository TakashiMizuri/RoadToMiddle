# ARIA, keyboard, focus — краткая выжимка

> `0.5.11` · ~10 мин на повторение

## Главная идея

**Сначала native HTML** (button, nav, input), **потом ARIA**. Keyboard и **focus** — обязательная часть UI, не опция. Custom widgets должны следовать **ARIA APG** patterns.

## Ключевые пункты

- **First rule of ARIA** — не дублируй/не ломай native semantics.
- **Implicit role** от тега; explicit `role` — когда native не хватает.
- **Tab order** ≈ DOM order; **`tabindex > 0`** — избегать; **`-1`** — programmatic `.focus()`.
- Modal: **focus trap**, Escape, **restore focus** к trigger.
- **`aria-expanded`**, **`aria-controls`** — disclosure/tabs/menus.
- **`aria-live="polite"`** — dynamic updates без focus steal.
- **`aria-hidden="true"`** — не на focusable/focused content.

## Самые важные примеры

```html
<button aria-expanded="false" aria-controls="panel-id">More</button>
<div role="dialog" aria-modal="true" aria-labelledby="title-id">...</div>
```

Enter/Space на custom `role="button"` — обязательны.

## «Под капотом» в одном абзаце

Browser строит accessibility tree parallel DOM: role, name, state. Platform APIs → screen readers. Focus navigation algorithm сортирует tabindex. `:focus-visible` отделяет keyboard focus от mouse.

## Частые ошибки

- `<div onclick>` без keyboard и role pattern.
- `outline: none` без visible focus style.
- Positive tabindex.
- Modal без restore focus.

## Мнемоника

> **Native, Tab, Trap, Talk** — native control, Tab order, trap in modal, name/state via ARIA/live.
