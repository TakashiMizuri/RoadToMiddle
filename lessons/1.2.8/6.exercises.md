# Exercises: Abstract classes vs Interfaces

> Complete after lectures and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Classify the abstraction
**Goal:** Apply decision checklist.

**Tasks:**
For each, choose **interface**, **abstract class**, or **concrete base** + one line why:
1. **`SendEmail`** for SMTP and fake test sender
2. **`Entity`** with **`Guid Id`** ctor for all domain entities
3. ASP.NET **`ControllerBase`**-style HTTP helpers (already exists ‚Äî what category?)
4. **`CompareTo`**-like ordering on unrelated types

**Success criteria:** Four choices with shared-state vs capability reasoning.

---

### E2. Minimal interface
**Goal:** ISP practice.

**Tasks:**
Split fat **`IUserService`** (Get, Create, Delete, SendNewsletter, ExportCsv) into **2‚Äì3** focused interfaces. Sketch class **`UserAdminService`** implementing subset.

**Success criteria:** No interface forces implementer to stub unrelated methods.

## Medium

### M1. Repository base
**Goal:** Abstract class with shared ctor.

**Tasks:**
**`abstract class EfRepository<T> : IReadRepository<T>, IWriteRepository<T>`** with **`DbContext`**, implement **`FindById`** concrete, leave **`Add`** abstract (or inverse).

**Success criteria:** Compiles; one **`DbContext`** field in base only.

---

### M2. Handler layering
**Goal:** Interface + team base.

**Tasks:**
Define **`INotificationHandler`**. Add **`NotificationHandlerBase`** with **`Handle`** calling **`protected abstract HandleCore`**. **`EmailHandler : NotificationHandlerBase`**.

**Success criteria:** Call via **`INotificationHandler`** reference works; logging in base optional.

---

### M3. Wrong tool refactor
**Goal:** Spot misuse.

**Tasks:**
Given **`abstract class StringUtils`** with only static **`Join`** ‚Äî propose fix. Given **`interface IAnimal`** with 20 methods ‚Äî propose split.

**Success criteria:** Correct type kind after refactor named.

## Hard (stretch)

### H1. Breaking change thought experiment
**Goal:** API evolution awareness.

**Success criteria:** Table: who must change (derived, implementers, callers).
