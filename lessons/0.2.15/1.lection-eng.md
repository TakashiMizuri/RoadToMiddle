# Monorepo vs Polyrepo (Concept)

> Roadmap: `0.2.15` · Node: `0.2` — Git: Branches and Collaboration · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Define **monorepo** and **polyrepo** (multi-repo) organization models.
- List **trade-offs** for versioning, CI, ownership, and developer experience.
- Explain how **Git workflows** differ (tags, cherry-pick, bisect, line endings) in each model.
- Recognize when startups vs enterprises lean toward each approach.
- Describe **tooling** concepts: workspace packages, shared CI, submodule/subtree (awareness).
- Choose a model for a typical **ASP.NET Core + React** product with informed trade-offs — not dogma.

---

## Why This Matters

Your fullstack product has a .NET API, React SPA, shared DTO contracts, and infrastructure scripts. Should everything live in one Git repository, or should `api`, `web`, and `infra` be separate repos with separate pipelines? The answer shapes daily Git commands, release tags (`0.2.13`), hotfix cherry-picks across services (`0.2.11`), and how painful cross-cutting changes become.

There is no universal winner. Google and Meta famously use monorepos at scale; many product companies use polyrepo with clear service boundaries. A Middle developer participates in architecture discussions, estimates cost of "change API + frontend + mobile together," and understands why the team's repo layout exists. This lesson closes Git node `0.2` before HTTP (`0.3.1`) — repository structure sits beside collaboration patterns as organizational context.

## Core Concepts

### Polyrepo (Multi-Repo)

**Polyrepo** means **multiple Git repositories**, typically one per deployable unit, library, or team boundary:

```
github.com/acme/api-backend
github.com/acme/web-frontend
github.com/acme/shared-contracts
github.com/acme/terraform-infra
```

Each repo has its own **history**, **issues**, **CI pipeline**, **access control**, and **release tags**. A feature touching API and UI requires coordinated PRs across repos, version bumps in package feeds (NuGet/npm), and often a **compatibility matrix** ("frontend 2.4 requires API ≥ 2.3").

**Advantages:** clear ownership; independent release cadence; smaller clone size; simpler permissions (contractors see frontend only); failure isolation in CI (API build red does not block docs repo).

**Disadvantages:** cross-repo atomic changes are hard; dependency drift; duplicated tooling (.editorconfig, `.gitattributes`, GitHub templates); more overhead for wide refactors; integration testing across repos needs orchestration.

Git operations you learned map **1:1 per repo**: cherry-pick hotfix inside `api-backend`; tag `v2.1.0` on that repo only; bisect API regression without cloning frontend.

### Monorepo

**Monorepo** means **one Git repository** containing many projects/packages:

```
acme-platform/
├── apps/
│   ├── api/          # ASP.NET Core
│   └── web/          # React
├── packages/
│   └── contracts/    # shared types
├── infra/
└── .gitattributes
```

One clone; one (or coordinated) CI pipeline; **atomic commits** can change API contract and React consumer in single PR — compiler/tests fail together if mismatched.

**Advantages:** single source of truth; atomic cross-cutting changes; unified lint/line-ending policy (`.gitattributes` once); shared CI cache; easier large-scale refactors; one bisect can span breaking commit across layers.

**Disadvantages:** clone size and history grow; CI complexity (path filters, "only build what changed"); permission granularity coarser unless path-based; tooling investment (Nx, Turborepo, Bazel, MSBuild solutions); release tagging must define **what** version means (whole repo vs per-package tags like `api/v1.2.0`).

### Versioning and Releases

**Polyrepo:** semver **per repository** is natural — `api v3.2.1`, `web v1.8.0`. Compatibility documented externally.

**Monorepo:** choices:

- **Single version** for entire repo (mobile app style) — one tag `v2026.07.0`.
- **Per-package semver** inside monorepo — tags `contracts/v2.0.0`, `api/v2.0.0`; tools like Changesets or Rush.

Hotfix (`0.2.11`) in polyrepo: cherry-pick in two repos, two tags, deploy two artifacts. Monorepo: one cherry-pick commit may fix both if change shared, one coordinated tag strategy.

### CI/CD and Developer Experience

**Polyrepo CI** — simple per repo; cross-repo integration tests in a third "e2e" repo or nightly pipeline triggering multiple checkouts.

**Monorepo CI** — **affected detection** (`paths:` filters in GitHub Actions, `nx affected`, `dotnet build --filter`): run API tests only when `apps/api/**` changes. Without this, every commit runs everything — slow at scale.

Developer laptop: monorepo one `git clone`, one IDE workspace; polyrepo multiple clones or meta-tooling (see below).

### Hybrid Patterns (Awareness)

Teams mix models:

- **Git submodules / subtrees** — polyrepo consumption with monorepo-like checkout (complexity cost).
- **Meta-repo** — empty repo with scripts cloning all services (orchestration only).
- **Monorepo + published packages** — code in one repo, NuGet/npm packages released from subsets.

Middle level: know names and pain points; deep submodule mastery optional unless your team uses them.

## Under the Hood

Git itself does not have a "monorepo" feature — it is **organizational**. One `.git` directory still holds one object graph. Monorepo scale challenges:

- **History depth** — `git clone --depth` partial clone, sparse checkout for huge trees.
- **Bisect** (`0.2.12`) spans all paths — first bad commit might be in unrelated package; path-specific test scripts in `bisect run` help.
- **Blame** across boundaries — same repo, same blame; polyrepo needs cross-repo archaeology.

**Access control:** Git hosting (GitHub/GitLab) applies permissions per repo (polyrepo natural) or per path rules (monorepo enterprise features).

**Line endings** ( `0.2.14`) — monorepo: one `.gitattributes` at root rules all packages; polyrepo: drift if `api` has attributes and `web` does not.

Object storage is shared in monorepo — identical files dedupe by hash across packages (`0.1.4`); polyrepo duplicates blob storage across clones but smaller individual repos.

## Syntax / Commands / API

No special Git command for monorepo — operational patterns:

```bash
# Monorepo: path-scoped log
git log --oneline -- apps/api/

# Polyrepo: clone multiple
git clone git@github.com:acme/api.git
git clone git@github.com:acme/web.git

# Monorepo per-package tag (convention)
git tag -a api/v2.1.0 -m "API release 2.1.0"
```

Tooling examples (conceptual): **npm workspaces**, **pnpm**, **Nx**, **Turborepo**, **.NET solution** with multiple projects — not Git features but monorepo enablers.

## Examples

### Example 1: Startup fullstack product

Small team, single product — **monorepo** with `backend/`, `frontend/`, shared `contracts/`. One PR updates endpoint + React hook; CI runs both. Simple mental model.

### Example 2: Enterprise platform

Fifty teams, independent SLAs — **polyrepo** per service, API gateway composition. Shared library published to private NuGet. Cross-service change needs RFC + multiple release trains.

### Example 3: Hotfix incident

**Polyrepo:** security patch in auth library repo `v1.0.1`; bump dependency in API and web repos separately.

**Monorepo:** one commit updates library + callers; tag strategy defines whether one or multiple version numbers ship.

## Common Mistakes & Anti-patterns

**Monorepo without CI path filters** — every push runs 2-hour pipeline.

**Polyrepo with tight coupling** — constant synchronized releases anyway; worst of both worlds.

**Dogma** — "monorepos always better" or "microservices mean polyrepo always."

**Duplicated `.gitattributes`/CI** across polyrepos without template — drift (`0.2.14`).

**Wrong tag scope** — tagging monorepo root `v1.0.0` when only API changed but web unrelated — document meaning.

## Production & Real-World Notes

Google-scale monorepos use custom infrastructure (Bazel, Piper). Most teams use **pragmatic monorepo** with Nx/Turborepo or .NET solution. **Open source** often monorepo for docs + code together.

**Contract testing** (Pact) popular in polyrepo. **Feature flags** decouple deploy from release in both models.

Migrating polyrepo → monorepo: history preservation hard (sometimes merge unrelated histories once); **strangler** migration common.

## Comparison / Trade-offs

| Dimension | Monorepo | Polyrepo |
|-----------|----------|----------|
| Atomic cross-project change | Easy (one PR) | Hard (multiple PRs) |
| Clone size | Larger | Smaller per repo |
| CI complexity | Path/affected logic | Per-repo simpler |
| Permissions | Coarser | Finer per repo |
| Release versioning | Needs convention | Natural per repo |
| Refactors spanning codebases | Easier | Harder |
| Team autonomy | Can conflict | Strong boundaries |

**When monorepo fits:** one product, shared release, small/medium team, heavy fullstack coupling.

**When polyrepo fits:** independent services, different teams/SLAs, legal separation, open vs closed components.

| Term | Meaning |
|------|---------|
| Monorepo | Many projects, one Git repo |

| Affected CI | Build only changed paths |

No single correct answer — align with team boundaries and release model.

## Key Takeaways

- Monorepo vs polyrepo is **organization**, not a Git feature.
- Monorepo enables **atomic** API+frontend changes; polyrepo enables **independent** releases.
- CI, tags, hotfixes, and `.gitattributes` **multiply or simplify** depending on choice.
- Monorepos need **path-based CI** at scale; polyrepos need **dependency/version discipline**.
- Choose based on **coupling and team structure**, not hype.
- Hybrid patterns exist — submodules/meta-repo as compromise with cost.
