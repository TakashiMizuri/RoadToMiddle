# git bisect — поиск breaking commit

> Roadmap: `0.2.12` · Узел: `0.2` — Git: ветки и collaboration · Глубина: понимание

## Learning Objectives

После этой лекции ты сможешь:

- Объяснить **binary search** по commit history и почему bisect эффективен.
- Провести **`git bisect start`**, **`bad`**, **`good`**, **`reset`** вручную.
- Использовать **`git bisect run`** с test script.
- Применять **`skip`** для untestable commits.
- Оценить **число шагов** bisect для N commits.
- Понимать, когда bisect лучше `git log`, а когда достаточно blame.

---

## Why This Matters

В пятницу тесты зелёные. В понедельник `main` красный — за выходные сотни merges. «Какой commit сломал build?» Читать все messages долго; случайные checkout'ы — угадайка. **`git bisect`** превращает задачу в **двоичный поиск**: отмечаешь known-good и known-bad, Git checkout'ит середины диапазона, ты (или script) говоришь good/bad, пока не останется **first bad commit**.

Для Middle это разница между двухчасовой археологией и fifteen-minute root cause перед standup. Bisect опирается на commit graph (`0.1.5`), ветки (`0.2.1`) и **воспроизводимый** test signal.

## Core Concepts

### Задача регрессии

**Regression** — работало, перестало. Есть commit `G` (good) и более новый `B` (bad) на одной ancestry-линии. Между ними — **first bad commit**: earliest, где test падает.

Linear search — до N checkout'ов. **Binary search** — ~`⌈log₂(N)⌉` шагов. 1000 commits → ~10 шагов.

Git bisect реализует поиск. **Ты** классифицируешь good/bad; Git двигает границы и выбирает midpoint.

### Ручной workflow

```bash
git bisect start
git bisect bad
git bisect good v1.0.0
# detached HEAD — build/test
git bisect good   # или bad
# ...
git bisect reset
```

**Detached HEAD** (`0.1.11`) — норма. **`bisect reset`** возвращает на исходную ветку.

Результат: `abc1234 is the first bad commit` → `git show` → fix или revert (`0.2.11`).

### Good и Bad

**Bad** — баг/тест fail **есть**. **Good** — бага **нет**. Good должен быть **ancestor** bad на искомой линии.

**Untestable** (не собирается): `git bisect skip`. Много skip → ambiguous result.

### git bisect run

```bash
git bisect start HEAD v1.0.0
git bisect run npm test -- --grep "AuthService"
```

Exit **0** = good; **1–124** = bad; **125** = skip. Script должен быть **deterministic** и **быстрым**.

### Сколько шагов?

~`⌈log₂(N+1)⌉` для N commits между good и bad. 500 commits → ~9 шагов.

## Under the Hood

Состояние в `.git/BISECT_LOG`, refs в `.git/refs/bisect/`. Candidate set — commits между good и bad. Каждый шаг делит множество.

**First bad** — bad commit `C`, parent `P` good в walk. Patch `C` — unit регрессии; иногда «логическая» причина шире одного commit.

**Bisect vs blame:** blame — кто тронул строку; bisect — какой **commit** сломал **поведение** test'а.

Каждый шаг = checkout — в тяжёлых repo ускоряют subset test'ов или `--first-parent`.

## Syntax / Commands / API

```bash
git bisect start [<bad> [<good>...]]
git bisect bad | good [<rev>]
git bisect skip [<rev>...]
git bisect run <cmd>...
git bisect reset
git bisect log
git bisect start --first-parent
```

---

## Examples

**Interactive:** `git bisect start main v2.4.0` → test → good/bad → reset.

**Automated:** `git bisect run dotnet test --filter "LoginTests"`.

**Skip:** middle commit не компилируется → `git bisect skip`.

---

## Common Mistakes & Anti-patterns

Dirty working tree; wrong good (не ancestor); flaky tests; забытый `bisect reset`; полный e2e в run — часы работы.

---

## Production & Real-World Notes

Good = last green CI tag; bad = current `main`. Merge commit — смотри combined diff или `--first-parent`. После find — regression test + fix/cherry-pick/revert.

---

## Comparison / Trade-offs

| Метод | Когда |
|--------|-------|
| bisect | Good+bad range, стабильный test |
| log -S | Поиск изменения текста, не поведения |
| blame | Автор строки, не causal regression |

---

## Quick Reference

`start` → `bad`/`good` (repeat) → `reset`. Run: 0=good, 125=skip. Steps ≈ log₂(N).

---

## Key Takeaways

- Bisect = binary search по history.
- Signal good/bad — **твоя** ответственность.
- `bisect run` для automation.
- Всегда `reset` в конце.
- Flaky test = ложный first bad.
