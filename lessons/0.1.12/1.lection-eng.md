# Git Clean: Removing Untracked Files Safely

> Roadmap: `0.1.12` · Node: `0.1` — Git Basics · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Explain what **`git clean`** removes (and what it **never** touches).
- Use **dry-run** (`-n` / `--dry-run`) before every destructive clean.
- Apply **`-f`**, **`-d`**, and **`-x`** flags appropriately.
- Distinguish **`git clean`** from **`git restore`**, **`git rm`**, and **`git stash -u`**.
- Understand interaction with **`.gitignore`** (default vs `-x`).
- Follow a safe production workflow for cleaning build artifacts and scratch files.

---

## Why This Matters

You build a .NET solution, run the frontend dev server, maybe execute a one-off SQL script. When you run `git status`, the output is cluttered with paths Git does not track: `bin/`, `obj/`, `scratch.sql`, a temporary log, perhaps `node_modules/` if your `.gitignore` is incomplete. None of these files appear in commits unless you explicitly add them, but they pollute status output, make it harder to spot real changes before a commit, and occasionally get swept into `git add .` by mistake.

You want a **clean working tree** — but the tools you already know target different problems. **`git restore`** discards modifications to **tracked** files; it will not delete a file Git never knew about. **`git rm`** removes **tracked** files from the index and working tree, staging a deletion for the next commit. **`git stash -u`** saves untracked files into the stash stack rather than deleting them. **`git clean`** is the command whose job is to **delete untracked files and directories from disk**.

That deletion is **permanent** from Git's perspective. There is no recycle bin, no undo stack, no commit that captures what was removed. If you delete an afternoon of uncommitted prototype code with `git clean`, recovery depends on IDE local history, filesystem backups, or luck — not on Git itself. Used with **`git clean -n`** (dry-run) first, clean is essential daily hygiene. Used without reading the preview, it is one of the fastest ways to lose work that was never committed.

## Core Concepts

### Three Categories of Files in Your Working Tree

Git classifies every path in your project directory into overlapping categories, and the removal tool depends entirely on which category applies.

**Tracked files** are paths that appear in the index or in the `HEAD` commit tree — files Git is actively managing. If you modify `Program.cs` after committing it, Git reports it as modified. Removing or reverting tracked content uses **`git rm`** (to stop tracking and delete) or **`git restore`** (to discard working tree changes back toward `HEAD`).

**Untracked, non-ignored files** are paths that exist on disk but are not in the index and not in the last commit — and they are **not** matched by any `.gitignore` rule. A scratch file `temp.log` or a folder `experiments/` you created yesterday fall here. These are **`git clean`** territory.

**Ignored files** match patterns in `.gitignore` (or other ignore sources). Build output like `bin/` and `obj/` in a typical .NET project lives here. By default, **`git clean` skips ignored files** — they are invisible to the command unless you pass **`-x`**. That default protects you from accidentally deleting `.env` or other secrets that should never be committed but must stay on disk during development.

The invariant worth memorizing: **`git clean` never removes tracked files**, not even if they are heavily modified. It only deletes paths Git considers untracked. If `git status` shows `modified: Program.cs`, clean will leave it alone.

### Dry-Run Is Non-Negotiable Habit

Because deletion is irreversible, Git provides a preview mode. The **`-n`** flag (short for **`--dry-run`**) lists what **would** be removed without touching the filesystem:

```bash
git clean -n        # preview untracked files at top level (respects .gitignore)
git clean -nd       # preview including untracked directories
```

The output uses the prefix `Would remove` for each path. Read every line. Ask whether each path is safe to lose. Only then rerun the same flags with **`-f`** instead of **`-n`** to execute. Professional developers treat `-n` as part of the command, not an optional extra — the same way they read a `DROP TABLE` preview before confirming.

### The Force Flag and Why Git Requires It

Actual deletion requires **`-f`** (force). Without `-f`, `git clean` refuses to run (except in dry-run mode). Git intentionally forces you to type the flag that means "I understand this is destructive." Combining flags is common: **`-fd`** removes untracked files and directories; **`-fdx`** additionally removes ignored files — the "nuclear" option for wiping build artifacts that `.gitignore` hides from normal status output.

## Under the Hood

When you invoke `git clean`, Git walks the working directory starting from the repository root (or a specified path), evaluates each untracked path against the index, the `HEAD` tree, and ignore rules, then calls filesystem delete operations (`unlink` for files, recursive removal for directories when `-d` is present).

The decision logic for each path follows roughly this sequence. If the path is tracked in the index, Git skips it — even if the working tree copy differs from the index entry. If the path matches an ignore rule and you did not pass `-x`, Git skips it. If the path is an untracked directory and you did not pass `-d`, Git skips the directory (but might still remove untracked files inside tracked directories in some edge cases — another reason to dry-run). If the path passes all filters and `-f` is present, Git deletes it from disk.

The interaction with **`.gitignore`** deserves explicit attention because it surprises people in both directions. Suppose `.gitignore` contains:

```
bin/
obj/
.env
```

Running **`git clean -fd`** removes untracked files and directories that are **not** ignored. It will **not** delete `bin/`, `obj/`, or `.env` because ignore rules exclude them from clean's default scope. Running **`git clean -fdx`** adds `-x`, which means "also remove ignored files." Now `bin/`, `obj/`, and `.env` appear in the dry-run preview and will be deleted on execution. That is exactly what you want when simulating a fresh clone after a corrupted build — and exactly what you do **not** want if `.env` holds database credentials you never committed.

The less common **`-X`** flag (capital X) inverts the filter: remove **only** ignored files, leaving non-ignored untracked files alone. Teams use it less often, but it exists for targeted cleanup of build output without touching scratch scripts sitting untracked in the repo root.

Internally, clean does not create commits, update the index, or touch refs. It is purely a working directory operation. That is why `git log` and `git reflog` cannot bring back cleaned files — Git never recorded them as objects in the first place (unless you had previously added and committed them, in which case they were tracked and clean would not have removed them).

## Syntax / Commands / API

The command surface is small but composable:

```bash
# Dry-run (always start here)
git clean -n
git clean -nd
git clean -nx              # preview including ignored files

# Execute deletion
git clean -f               # untracked files (not ignored, not dirs without -d)
git clean -fd              # + untracked directories
git clean -fdx             # + ignored files (bin, obj, node_modules, .env if ignored)
git clean -fdX             # only ignored files

# Interactive mode — choose per group
git clean -i
git clean -id              # interactive + directories
```

Flag meanings in prose: **`-n`** dry-run; **`-f`** force actual deletion; **`-d`** include untracked directories; **`-x`** include ignored paths; **`-X`** only ignored paths; **`-i`** interactive prompts. You can combine them in any sensible order (`git clean -fdx` and `git clean -xdf` are equivalent).

For advanced use, Git supports exclude patterns via **`-e`** to skip specific paths even when they would otherwise match — useful in large monorepos when you want to clean most artifacts but preserve one local folder.

## Examples

### Example 1: The safe daily workflow

You finished experimenting and want to remove scratch files before starting a new task:

```bash
git status                 # see what's untracked
git clean -nd              # preview: "Would remove scratch.txt", "Would remove tmp/"
# read output carefully
git clean -fd              # execute only if preview looks correct
```

The two-step rhythm — preview, then execute — takes five seconds and prevents stories about deleting the wrong folder. Make it muscle memory.

### Example 2: Resetting build artifacts after a corrupted .NET build

Your `bin/` and `obj/` directories are listed in `.gitignore`, so they do not show in plain `git status`, but they are consuming disk space and possibly causing stale build errors. You want a pristine tree:

```bash
git clean -ndx             # preview ignored removal — expect bin/, obj/ in list
git clean -fdx             # delete ignored build output
dotnet restore             # rebuild dependency graph
dotnet build               # fresh compile
```

After `-x`, expect to reinstall npm packages or restore NuGet packages — that rebuild cost is intentional. You traded time for certainty that no stale artifact remains.

### Example 3: Choosing the right tool

Suppose `Program.cs` is tracked and you edited it badly, `temp.sql` is untracked scratch, and `OldService.cs` is tracked but obsolete and should leave the repository entirely:

```bash
# Discard edits to tracked file — NOT clean
git restore Program.cs

# Delete untracked scratch — clean
git clean -fd    # or git clean -f -- temp.sql for single file

# Remove tracked file from repo — NOT clean
git rm OldService.cs
git commit -m "Remove obsolete OldService"
```

Confusing these three commands is the most common clean-related mistake. Ask: "Does Git track this path?" Tracked modification → **`restore`**. Tracked removal from repo → **`rm`**. Untracked existence on disk → **`clean`**.

### Example 4: When stash beats clean

You have untracked files you might need tomorrow — work-in-progress notes, a generated config you are not ready to commit. **`git stash push -u`** saves untracked content into the stash stack and restores a clean tree without deleting anything. **`git clean`** would destroy those files. When uncertain, stash first; clean when you are sure deletion is acceptable.

## Common Mistakes & Anti-patterns

Skipping dry-run is the mistake that hurts most. Running **`git clean -fdx`** at the repository root without reading `-ndx` output has deleted `.env.local`, hours of `node_modules` download time, and uncommitted prototypes in one command. Expecting clean to revert modified tracked files leads to frustration — `Program.cs` stays modified after clean because clean never touches it. Using clean when you meant to save work — **`git stash -u`** preserves; clean destroys. Running clean instead of committing work you actually want to keep is irreversible if no backup exists.

Another subtle trap: assuming `.gitignore` protects files you care about. If you never added `important-draft.md` to `.gitignore` and never committed it, it is untracked non-ignored — **`git clean -fd`** will delete it. Only ignored untracked files are safe from default clean. Important scratch work either belongs in a commit, a stash, or an ignore rule you consciously add.

## Production & Real-World Notes

Many teams script **`git clean -fdx && dotnet restore`** (or `npm ci`) to reproduce CI-like pristine trees when debugging "works on my machine" issues. CI itself usually checks out a fresh clone each run, so clean is a local simulation of that cleanliness.

Never run **`-x`** on a monorepo root without preview — gitignored secrets and local overrides are often exactly the files `-x` targets. Pair clean habits with a solid **`.gitignore`** maintained in the repo; clean is not a substitute for ignoring build output proactively.

IDE **Local History** (JetBrains Rider, VS Code with extensions) sometimes recovers deleted untracked files — treat that as a last resort, not a strategy. When onboarding juniors, teach the mantra: **`status` → `clean -nd` → read → `clean -fd`**.

Some organizations document a "nuclear reset" alias:

```bash
git reset --hard HEAD && git clean -fdx
```

That wipes tracked modifications **and** all untracked/ignored debris. It is appropriate only when you fully accept losing all local uncommitted work — document it loudly in team wikis.

In pull request preparation, running `git clean -fd` after reviewing `git status` ensures you do not accidentally commit local scratch files that slipped past review attention. The clean tree matches what reviewers expect from the diff — only intentional, tracked changes remain visible.

## Comparison / Trade-offs

**`git clean -fd`** targets untracked non-ignored files and directories. Reversible only through backups or IDE history — not through Git. **`git clean -fdx`** extends to ignored artifacts; expect full rebuild of dependencies. **`git restore .`** resets tracked working tree files to match `HEAD`; it does not delete untracked paths. **`git stash -u`** copies untracked files into the stash stack; reversible with `git stash pop`. **`git rm`** removes tracked files and stages the deletion; reversible with `git restore --staged` and `git restore` before commit.

Choose clean when you want files **gone**. Choose stash when you want files **preserved but out of the way**. Choose restore when tracked content changed and untracked content is not the problem.

---

## Quick Reference

Always preview: `git clean -nd`. Delete untracked respecting ignore: `git clean -fd`. Delete including build output: `git clean -fdx` after `git clean -ndx`. Interactive: `git clean -i`. Golden workflow: **`git status` → `git clean -nd` → read every line → `git clean -fd`**.

---

## Key Takeaways

- **`git clean`** removes **untracked** paths from disk — never tracked modifications.
- **Always dry-run with `-n`** before adding **`-f`**.
- **`-d`** includes untracked **directories**; **`-x`** includes **ignored** files — use `-x` with extreme care.
- Default clean **respects `.gitignore`** — ignored files survive unless `-x`.
- Different from **`restore`** (tracked changes), **`rm`** (stop tracking), **`stash -u`** (save untracked).
- Deletion is **permanent** — no Git undo for files that were never committed.
