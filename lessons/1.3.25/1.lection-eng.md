# LINQ Distinct, Union, Intersect, Except

> Roadmap: `1.3.25` · Node: `1.3` — C#: Advanced Topics · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Explain **set operations** on sequences and how they differ from mathematical sets (multiset vs set, order preservation).
- Use **`Distinct`**, **`Union`**, **`Intersect`**, and **`Except`** with default and custom **`IEqualityComparer<T>`**.
- Predict **deferred vs immediate** behavior and when operators **buffer** the entire source.
- Translate set operations to **SQL** when working with **`IQueryable`** (EF Core).
- Choose between **LINQ set operators** and **HashSet<T>** for performance and clarity.
- Avoid **reference equality** bugs on custom types and **multiple enumeration** pitfalls.

---

## Why This Matters

Real applications constantly merge, deduplicate, and compare collections: merge tag lists from two products, find users in group A but not group B, list permissions common to two roles, remove duplicates from an import file. SQL has **`DISTINCT`**, **`UNION`**, **`INTERSECT`**, **`EXCEPT`**; LINQ mirrors them on **`IEnumerable<T>`** and **`IQueryable<T>`**.

Choosing the wrong operator or equality semantics causes subtle bugs: two **`Person`** objects with the same email treated as different because **`Distinct()`** uses reference equality; **`Union`** silently deduplicates when you wanted concatenation; **`Except`** on two **`IQueryable`** sources generates a subquery that scans both tables. Middle developers need the **semantics** (what counts as "equal", what happens to order, whether results are lazy) and the **implementation cost** (HashSet allocations, full materialization).

## Core Concepts

### Sets vs Sequences

A **mathematical set** has no duplicates and no order. A **sequence** (`IEnumerable<T>`) has order and may contain duplicates. LINQ set operators sit between these worlds: they **consume ordered sequences** and produce **sequences without duplicates** (except where noted), generally **preserving first-seen order** from the left operand.

This matters when you write `first.Union(second)`: the result contains each distinct element once, ordered by first appearance in `first`, then first appearance in `second` for elements not already seen. It is **not** "concatenate then sort".

### Distinct — Remove Duplicates

```csharp
var ids = new[] { 1, 2, 2, 3, 1, 4 };
var unique = ids.Distinct(); // 1, 2, 3, 4
```

**`Distinct()`** without arguments uses **`EqualityComparer<T>.Default`**. For **`string`**, that is ordinal equality. For **reference types** without overridden **`Equals`**, each object is unique even if fields match — a classic junior bug.

**`DistinctBy(keySelector)`** (.NET 6+) deduplicates by a projected key while returning the **original element** (first occurrence per key):

```csharp
var people = new[] {
    new { Name = "Ada", Dept = "Eng" },
    new { Name = "Lin", Dept = "Eng" },
    new { Name = "Ada", Dept = "Ops" }
};
var byName = people.DistinctBy(p => p.Name); // Ada(Eng), Lin
```

**Deferred execution:** `Distinct` wraps the source; work happens on enumeration. Internally it maintains a **set of seen keys** — O(n) time, O(n) extra space for the hash set.

### Union — A ∪ B (Distinct Merge)

```csharp
var a = new[] { 1, 2, 3 };
var b = new[] { 3, 4, 5 };
var u = a.Union(b); // 1, 2, 3, 4, 5
```

**`Union`** is equivalent to **`Concat` followed by `Distinct`** with the same comparer, but implemented as a single pass over both sequences using a hash set. **`UnionBy`** (.NET 6+) projects keys before deduplication.

There is no **`Union`** overload that keeps duplicates — use **`Concat`** when you need a multiset merge.

### Intersect — A ∩ B

```csharp
var a = new[] { 1, 2, 3, 3 };
var b = new[] { 3, 4, 5 };
var i = a.Intersect(b); // 3 — once, from order of first sequence
```

**`Intersect`** returns elements that appear in **both** sequences, each **once** in the order of the **first** sequence's first occurrence. Both sequences are fully enumerated (unless short-circuited by implementation details — do not rely on side effects).

### Except — A \ B

```csharp
var a = new[] { 1, 2, 3, 4 };
var b = new[] { 3, 5 };
var diff = a.Except(b); // 1, 2, 4
```

**`Except`** returns elements in the **first** sequence that are **not** in the second, deduplicated, preserving order of first sequence. It does **not** return elements in B that are missing from A — set difference is **not symmetric**.

For "symmetric difference" (elements in either but not both), compose: `a.Except(b).Union(b.Except(a))`.

### Custom Equality — IEqualityComparer<T>

When identity is not default equality:

```csharp
var emails = users.Select(u => u.Email).Distinct(StringComparer.OrdinalIgnoreCase);

var setA = rolesA.Union(rolesB, StringComparer.OrdinalIgnoreCase);
```

For custom types, implement **`IEqualityComparer<Person>`** or use **`DistinctBy(p => p.Email)`** when the key is simple.

Passing the **same comparer** to both operands in **`Union`/`Intersect`/`Except`** is required for consistent behavior; the second sequence's membership test uses that comparer.

## Under the Hood

### Iterator Chains and HashSet Backing

Set operators in **`System.Linq.Enumerable`** are implemented as **iterator blocks** (`yield return`) that lazily produce output, but they **must remember what they have seen**. For **`Distinct`**, the iterator creates a **`Set<T>`** (internal hash set wrapper) on first **`MoveNext`** and adds each element (or key) before yielding.

Rough mental model for **`Union(first, second)`**:

```
seen = empty HashSet
foreach x in first:
    if seen.Add(x): yield x
foreach x in second:
    if seen.Add(x): yield x
```

**`Intersect`** typically builds a set from the **second** sequence (or iterates second into set), then walks **first** and yields elements found in the set that haven't been yielded yet (tracking another set for deduplication on the output side).

Because of hash sets, **memory is O(unique elements)** and **hash quality** matters: poor **`GetHashCode`** distributions degrade performance toward O(n²).

### IQueryable Translation (EF Core)

On **`IQueryable`**, set operators translate to SQL set operations where the provider supports them:

| LINQ | Typical SQL |
|------|-------------|
| `Distinct()` | `SELECT DISTINCT ...` |
| `Union(other)` | `UNION` (deduplicating) |
| `Concat(other).Distinct()` | similar to UNION |
| `Intersect(other)` | `INTERSECT` (PostgreSQL, SQL Server) |
| `Except(other)` | `EXCEPT` |

**`Union`** in LINQ maps to SQL **`UNION`**, not **`UNION ALL`**. If you need all rows including duplicates from SQL, use **`Concat`** at the query level or raw SQL.

Provider differences exist: some older providers expand **`Except`** as **`WHERE NOT IN`** or **`NOT EXISTS`**. Always inspect generated SQL for hot paths.

Client evaluation: if one side is in-memory and the other is **`IQueryable`**, EF may **client-evaluate** part of the expression — often a performance and correctness trap.

### DistinctBy Implementation Sketch

**`DistinctBy`** uses **`HashSet<TKey>`** to track keys; on first unseen key it yields the **source element**. The key selector runs once per element; if the selector is expensive, profile it.

## Syntax / Commands / API

```csharp
// Distinct
source.Distinct()
source.Distinct(comparer)
source.DistinctBy(keySelector)           // .NET 6+
source.DistinctBy(keySelector, comparer)

// Union / Intersect / Except — same pattern for *By variants
first.Union(second)
first.Union(second, comparer)
first.Intersect(second)
first.Except(second)
```

All return **`IEnumerable<T>`** (deferred) except when followed by immediate operators like **`ToList()`**.

## Examples

### Deduplicating Import Rows

```csharp
var imported = await ReadCsvRowsAsync();
var uniqueBySku = imported.DistinctBy(r => r.Sku, StringComparer.OrdinalIgnoreCase);
```

Each SKU appears once — the **first** row wins. If you need "last wins", reverse before **`DistinctBy`** or use a **`GroupBy`**.

### Permission Diff Between Roles

```csharp
var editorPerms = GetPermissions("Editor");
var adminPerms = GetPermissions("Admin");

var onlyEditor = editorPerms.Except(adminPerms);
var onlyAdmin = adminPerms.Except(editorPerms);
var shared = editorPerms.Intersect(adminPerms);
```

This mirrors how authorization audits are described to security reviewers.

### EF Core — Active Users Not in Blocklist

```csharp
var blockedIds = db.Blocklist.Select(b => b.UserId);
var activeNotBlocked = db.Users
    .Where(u => u.IsActive)
    .Select(u => u.Id)
    .Except(blockedIds);
```

Translated to **`EXCEPT`** or anti-join depending on provider — inspect **`ToQueryString()`** in development.

## Common Mistakes & Anti-patterns

**Using `Distinct()` on objects without equality.** Fix with **`DistinctBy`** on a stable key or implement **`Equals`/`GetHashCode`** (or **`IEquatable<T>`**).

**Expecting `Union` to preserve duplicates.** Use **`Concat`**.

**Wrong operand order in `Except`.** `a.Except(b)` ≠ `b.Except(a)`.

**Assuming `Intersect` counts multiplicity.** `[1,1,1].Intersect([1])` yields **one** `1`, not three.

**Side effects in source sequences.** Set operators may enumerate sources multiple times in composed pipelines — prefer pure sequences or materialize once.

**Mixing comparers.** `a.Union(b, StringComparer.OrdinalIgnoreCase)` but `b` built with different casing rules — inconsistent.

## Production & Real-World Notes

For **in-memory** large datasets (millions of rows), **`HashSet<T>`** with explicit loops may beat chained LINQ due to fewer iterator allocations — measure. **`DistinctBy`** on a hot path after **`Select`** may allocate less than full **`Distinct`** on heavy objects if the key is a **`int`** or **`string`**.

In **API pagination**, never **`Distinct`** after **`Skip/Take`** on SQL without understanding that dedup may require a subquery — duplicates might reappear across pages if dedup is only per page.

Logging: when debugging EF, capture SQL for **`Intersect`/`Except`** — ORMs sometimes generate plans that miss indexes on the inner set.

## Comparison / Trade-offs

| Approach | When |
|----------|------|
| `Distinct` / `Union` / … | Readable, composable LINQ pipelines |
| `HashSet<T>` manual | Hot path, single operation, mutable accumulation OK |
| SQL set ops via EF | Large data, push work to server |
| `GroupBy` + `First` | Need to pick *which* duplicate wins with rules |

**`Concat` + `Distinct`** vs **`Union`**: same result for many cases; **`Union`** expresses intent and may share implementation.

## Quick Reference

| Operator | Meaning | Duplicates in result | Order |
|----------|---------|----------------------|-------|
| `Distinct` | unique elements | no | first-seen |
| `Union` | in A or B | no | A then B |
| `Intersect` | in A and B | no | order of A |
| `Except` | in A not B | no | order of A |

Default equality: **`EqualityComparer<T>.Default`**. Override with comparer or **`DistinctBy`/`UnionBy`**.

---

## Key Takeaways

- Set operators remove duplicates and preserve **first-seen order** from the left/first sequence.
- **`Distinct()`** on reference types without **`Equals`** does not deduplicate by value.
- **`Union`** ≠ **`Concat`**; **`Except`** is asymmetric.
- Internally, hash sets drive O(n) behavior and O(unique) memory.
- On **`IQueryable`**, operators map to SQL set operations — verify generated SQL.
- Use **`DistinctBy`/`UnionBy`** (.NET 6+) for key-based identity without full object equality.
