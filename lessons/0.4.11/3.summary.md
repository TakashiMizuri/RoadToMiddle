# Latency vs Bandwidth vs Throughput — краткая выжимка

> `0.4.11` · ~8 мин на повторение

## Главная идея

**Latency** — задержка (ms, RTT). **Bandwidth** — ёмкость канала (Mbps). **Throughput** — реально достигнутая скорость (RPS, MB/s). Маленький API JSON упирается в **latency**; большой файл — в **bandwidth**; много клиентов — в **throughput** и capacity.

## Ключевые пункты

- **RTT** — round-trip; TCP handshake + TLS + CORS preflight добавляют RTT.
- **1 Gbps** не убирает 200 ms RTT между континентами.
- **Throughput ≤ bandwidth**; CPU/DB/loss тоже limit.
- Serial API calls: latencies **суммируются**.
- CDN снижает **latency** до static (`0.4.9`).
- **BDP** = bandwidth × RTT — in-flight data для заполнения pipe.

## Самые важные примеры

```bash
ping -c 4 api.example.com          # latency / RTT
hey -n 1000 -c 50 URL              # throughput RPS + latency
```

## Частые ошибки

- «Gigabit internet» как аргумент быстрого API.
- Путать Mbps (ISP) с measured throughput.

## Мнемоника

> **Latency** = сколько ждёшь стакан воды; **Bandwidth** = ширина трубы; **Throughput** = сколько воды реально налили за минуту.
