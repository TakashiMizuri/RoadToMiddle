# Detached HEAD — краткая выжимка

> `0.1.11` · ~10 мин на повторение

## Главная идея

**Detached HEAD** — состояние, когда **HEAD** указывает **напрямую на commit SHA**, а не на **branch ref** (`refs/heads/...`). Это не ошибка, но **новые commits без branch** становятся **unreachable**, если уйти на другую ветку без сохранения.

## Ключевые пункты

- **Normal:** `HEAD → refs/heads/main → commit`
- **Detached:** `HEAD → abc1234` (commit напрямую)
- Попадаешь через: **`git checkout SHA`**, **tag**, **`git switch --detach`**, **`git bisect`**
- Файл **`.git/HEAD`**: attached = `ref: refs/heads/main`; detached = raw SHA
- **Сохранить работу:** `git switch -c new-branch-name` (или `checkout -b`)
- **Вернуться:** `git switch main` (если не было новых commits — ничего не теряется)
- **Recovery:** `git reflog` → найти SHA → `git switch -c rescue SHA`

## Самые важные примеры

**Безопасный просмотр:**
```bash
git switch --detach v1.0.0
# смотришь код...
git switch main
```

**Опасно:**
```bash
git checkout v2.0.0
git commit -am "fix"      # commit без branch
git switch main           # commit "потерян" (но в reflog!)
```

**Правильно от tag:**
```bash
git checkout v2.0.0
git switch -c hotfix/2.0.1
git commit -am "fix"
```

## «Под капотом» в одном абзаце

Git хранит commits, пока на них есть **ref** (branch, tag, remote). В detached HEAD новые commits reachable только через **HEAD**. Когда HEAD переезжает на branch, orphaned commits остаются в **reflog** ~90 дней, затем **GC** может удалить. **Branch ref** (`git switch -c`) делает линию commits permanently reachable.

## Частые ошибки

- Игнорировать warning Git про detached HEAD.
- Думать, что **tag** — это ветка (tag не двигается вперёд).
- `git checkout origin/feature` вместо `git switch -c feature origin/feature`.

## Мнемоника

> **Detached = HEAD без «фамилии» ветки.** Commit'ишь — **создай branch (`-c`)**, иначе reflog — единственный друг.
