## Why This Matters

You have written `a + b` for integers since your first program. The `+` symbol is not magic built into the language for all types â€” for `int`, the compiler maps it to IL `add`. For your own `Money` or `Vector2` types, nothing happens unless **you** define what `+` means. Without that definition, `left + right` is a compile error. With a thoughtful definition, domain code reads like the problem statement: `total = price + tax` instead of `total = price.Add(tax)`.

Operator overloading is often misunderstood as "clever syntax." In professional C#, it is a **narrow tool**: make value objects feel natural in expressions while keeping **named methods** for anything ambiguous. Frameworks you use every day rely on it â€” `string` has `+`, `TimeSpan` has `+` and `-`, `decimal` has full arithmetic. Misuse â€” overloading `+` to mean "merge databases" â€” confuses readers and breaks the principle of least surprise.

You just studied equality in `1.2.16`. If you overload `==`, you enter the same contract world as `Equals`. Operators are not isolated syntax; they participate in the type's public API and must stay **consistent** with the rest of the type.

## Core Concepts

### What Operator Overloading Is

An **operator overload** is a **static method** with a special name (`operator +`, `operator ==`, etc.) that lets the compiler rewrite an expression into a method call. When you write `c = a + b`, the compiler looks for `operator +(A a, B b)` (with compatible types) and emits a call to that static method.

Operators are **syntactic sugar**. They do not grant new runtime powers. Everything you can express with an operator you can express with `static` methods like `Add`, `Equals`, or `CompareTo` â€” operators exist for **readability** when the meaning is universally clear in the domain.

C# deliberately restricts which operators you may overload. You cannot invent `operator**` or overload the assignment `=`, the short-circuit `&&` / `||` (they require `operator true`/`false` patterns on custom types â€” rare), the ternary `?:`, member access `.`, or casting `as` / `is` (though **conversion operators** exist separately).

### Overloadable Operators (Basics)

**Unary:** `+`, `-`, `!`, `~`, `++`, `--`, `true`, `false` (for custom `&&`/`||` in some scenarios).

**Binary:** `+`, `-`, `*`, `/`, `%`, `&`, `|`, `^`, `<<`, `>>`, `==`, `!=`, `<`, `>`, `<=`, `>=`.

**Cannot overload:** `=`, `.`, `?:`, `->`, `=>`, `f(x)` invocation (except implicit/explicit **conversion** operators), `checked` variants as separate keywords (operands use your overload).

For this lesson, focus on **arithmetic** (`+`, `-`) and **equality** (`==`, `!=`) â€” the ones you will see and occasionally implement.

### Syntax Rules

Operator overloads must be **`public static`**. At least one parameter must be the declaring type (or a type you are converting from/to for conversion operators).

Binary example:

```csharp
public readonly struct Vector2
{
    public double X { get; }
    public double Y { get; }

    public Vector2(double x, double y) => (X, Y) = (x, y);

    public static Vector2 operator +(Vector2 a, Vector2 b) =>
        new(a.X + b.X, a.Y + b.Y);

    public static Vector2 operator -(Vector2 a, Vector2 b) =>
        new(a.X - b.X, a.Y - b.Y);
}
```

Usage `var v = a + b` compiles because the compiler finds `operator +`. The struct remains immutable â€” each operation returns a **new** instance, which aligns with value object practice from `1.2.14`.

Unary negation:

```csharp
public static Vector2 operator -(Vector2 v) => new(-v.X, -v.Y);
```

### Equality Operators and the 1.2.16 Contract

If you overload `==` and `!=` for a **reference type** where you care about value semantics, implement them **together** with `Equals` and `GetHashCode`:

```csharp
public static bool operator ==(Money? left, Money? right) =>
    left is null ? right is null : left.Equals(right);

public static bool operator !=(Money? left, Money? right) => !(left == right);
```

Microsoft guidance: if you override `Equals`, override `GetHashCode`; if you provide value `Equals` on a class, provide `==`/`!=` that delegate to it. Users expect `a == b` to agree with `a.Equals(b)`.

For **structs**, `==` is not defined by default unless you define it â€” default struct comparison uses reflection-based equality in some contexts, but `==` itself requires overload or you use `Equals`.

**Records** (`1.2.13`) generate `==`/`!=` automatically from property values.

### Conversion Operators (Awareness)

**Implicit** and **explicit** conversion operators let you write `(TargetType)source` or silent promotion:

```csharp
public static implicit operator decimal(Money m) => m.Amount; // use sparingly
```

Implicit conversions that lose information (money without currency) are dangerous. Prefer **explicit** casts when precision or context matters. Deep treatment is optional; know they exist when reading BCL types like `System.Xml.Linq.XName`.

### When Overloading Helps

Good fits share traits: **small immutable value types**, operation meaning **fixed by convention**, used in **expressions** repeatedly.

- **Money / Currency:** `+`, `-`, unary `-` for refunds; sometimes `*` with `decimal` scalar.
- **Vector / Matrix math:** `+`, `-`, `*` (static method may be clearer for matrix multiply).
- **Date ranges:** intersection might be `&` â€” but named `Intersect` is often clearer than obscure overloads.
- **Strongly typed IDs:** usually **no** arithmetic; maybe explicit `==` only.

### When to Avoid Overloading

If the operation is **ambiguous**, **side-effect-heavy**, or **not symmetric**, use a named method.

**Bad:** `database + migration` to mean "apply migration." **Good:** `database.Apply(migration)`.

**Bad:** overloading `<` when the sort order is configurable per query. **Good:** `CompareTo` or `IComparer<T>`.

Team style guides (including many .NET shops) say: **when in doubt, use `Add`, `Subtract`, `Combine`.** Operators are for types that behave like numbers or vectors, not like services.

### Operators vs Static Methods â€” No Runtime Difference

```csharp
var c = a + b;
var c2 = Vector2.operator +(a, b); // legal but never written by humans
```

The JIT inlines small operator bodies same as static methods. Choose based on **API clarity**, not performance.

## Examples

### Money â€” Add Same Currency Only

```csharp
public sealed class Money : IEquatable<Money>
{
    public decimal Amount { get; }
    public string Currency { get; }

    public Money(decimal amount, string currency) =>
        (Amount, Currency) = (amount, currency);

    public static Money operator +(Money left, Money right)
    {
        if (left.Currency != right.Currency)
            throw new InvalidOperationException("Currency mismatch");
        return new Money(left.Amount + right.Amount, left.Currency);
    }

    // Equals, GetHashCode, ==, != as in 1.2.16 ...
}
```

The `+` overload **encodes an invariant** (same currency) at the expression site. A static `Add` would be equally valid; `+` is idiomatic for numeric-like aggregates.

### Do Not Overload Everything

```csharp
// Prefer:
var merged = invoice.Merge(adjustment);
// Over invoice + adjustment only if your entire team agrees it reads clearly
```

## Common Mistakes & Anti-patterns

**Overloading `==` without `Equals`/`GetHashCode`** â€” three-way inconsistency; dictionaries and `==` disagree.

**Mutating in operator** â€” `operator +` should return new instances for value objects, not modify operands (unless documented mutable type â€” rare).

**Throwing surprising exceptions** â€” currency mismatch on `+` is fine; null reference because you forgot nullable annotations is not.

**Overloading for DSL cleverness** â€” cute today, unmaintainable after turnover.

**Symmetric types** â€” if `operator +(A,B)` exists, consider whether `B + A` needs support or document order dependence.

## Production & Real-World Notes

Many enterprise codebases **ban** custom operator overloads except in approved math/money libraries â€” check team conventions before adding them.

Unit tests should cover operators the same as public methods â€” they are public API.

Serialization and logging do not use operators; do not rely on `+` for business workflows that must be auditable â€” named methods grep better.

Source generators and records may emit operators; read generated code in IDE without editing (`1.2.18`).

## Comparison / Trade-offs

| Approach | Pros | Cons |
|----------|------|------|
| Operator `+` | Concise math-like expressions | Surprise if meaning non-obvious |
| `static Add(a,b)` | Explicit, grep-friendly | Verbose in formulas |
| Record `==` | Free structural equality | Limited to generated shape |

**Rule of thumb:** arithmetic on numeric-like value types â†’ operators OK. Business workflows â†’ methods.

## Key Takeaways

- Operators are **static methods** with compiler sugar â€” limited set in C#.
- Use for **clear, conventional** operations on value types (money, vectors).
- `==`/`!=` must align with **`Equals`/`GetHashCode`** (`1.2.16`).
- Prefer **named methods** when meaning is ambiguous or domain-heavy.
- Immutability: operators return **new** values for value objects.
- Records generate equality operators; custom classes must coordinate manually.
- Team conventions may restrict operators â€” follow project style.
