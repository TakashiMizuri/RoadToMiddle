## Answers

### 1. [Recall]
**Answer:** **tree**, **parent** (может быть несколько строк), **author**, **committer**, пустая строка перед message (message — тело, не header-поле в том же смысле).

**Why:** Эти поля сериализуются в commit object и участвуют в вычислении hash.

---

### 2. [Recall]
**Answer:** **Root commit** — первый commit в репозитории. **Ноль** parent'ов (строка `parent` отсутствует).

**Why:** Нечему указывать назад — начало history.

---

### 3. [Recall]
**Answer:** **Directed Acyclic Graph** — направленный ациклический граф. Directed: связи parent ← child; Acyclic: нет циклов.

**Why:** Git history идёт только назад во времени, never loops.

---

### 4. [Understanding]
**Answer:** Hash commit'а вычисляется от **всего** serialized content, включая message. Новый message → другой input → другой SHA-1.

**Why:** Commit object immutable; любое изменение metadata = новый object.

---

### 5. [Understanding]
**Answer:** **Author** — кто создал изменение (original writer). **Committer** — кто записал commit в repo. Различаются после **rebase**, **cherry-pick**, когда другой человек/процесс применяет patch и создаёт новый commit object.

**Why:** Committer timestamp/name обновляются при перезаписи history.

**Common mistake:** Считать их синонимами.

---

### 6. [Understanding]
**Answer:** **First parent** — tip ветки, на которой ты стоял при `git merge` (обычно main). Second — tip влитой ветки. `git log --first-parent` идёт только по first parent на merge'ах, показывая **mainline** без деталей side-branch.

**Why:** Convention для linearized view CI/history.

---

### 7. [Understanding]
**Answer:** Каждый commit указывает на **более старые** parent'ы. Цикл означал бы commit, который ancestor сам себе — невозможно при нормальной работе Git. Parent hash включён в content child — цикл сломал бы модель времени.

**Why:** DAG модель time arrow backward.

---

### 8. [Understanding]
**Answer:** **Reachable** — commit'ы, до которых можно дойти от HEAD (или другого ref), следуя parent pointers. `git log` по умолчанию показывает reachable commit'ы от текущего HEAD, в порядке от новых к старым.

**Why:** Git показывает «вашу» history от текущей позиции.

---

### 9. [Understanding]
**Answer:** Commit hash зависит от tree hash и parent hash(es). Tree зависит от blob/tree детей. Изменение любого object меняет его hash → parent commit указывает на старый hash → несоответствие обнаруживается при чтении/fsck.

**Why:** Tamper-evident chain без central authority.

---

### 10. [Application]
**Answer:** `git rev-parse HEAD^2` или `git cat-file -p HEAD | grep parent` (вторая строка parent).

**Why:** `^` = first parent, `^2` = second parent; на non-merge `^2` ошибка.

**Common mistake:** Использовать `HEAD~2` — это «дедушка» по first-parent chain, не second parent merge.

---

### 11. [Application]
**Answer:** Commit hash включает **parent hash(es)**, **tree hash**, **author/committer timestamps**. Rebase создаёт **новые** commit objects на новом base — parent'ы другие, timestamps другие → hash'и другие, даже если patch к файлам идентичен.

**Why:** Identity commit = весь object, не только diff.

---

### 12. [Application]
**Answer:** `git rev-list --max-parents=0 HEAD` (или `--all` для всех root'ов при нескольких orphan histories).

**Why:** Фильтр commits с zero parents.

---

### 13. [Analysis]
**Answer:** **B** неверно.

**Why:** Одинаковый tree + **разные** parent/metadata/message → **разные** commit hash'и.

---

### 14. [Analysis]
**Answer:** **Да**, objects могут оставаться на диске, пока commit **reachable** из reflog (`HEAD`, `feature@{n}`) или другого ref, или пока не выполнен `git gc --prune`. После истечения reflog и prune — удалятся.

**Why:** Refs и reflog удерживают reachable objects.

---

### 15. [Recall]
**Answer:** (любые два)
1. Одинаковый file content → один blob hash → хранится один раз.
2. Unchanged files между commit'ами → тот же blob/tree reused.
3. Global identity по hash — два clone deduplicate при fetch одинаковых objects.

**Why:** Content-addressed storage экономит место и упрощает sync.

## Model answers for practical sections

После сценария merge:

```
*   merge commit (HEAD, main)  ← 2 parents
|\
| * on feature (feature tip)
* | on main
|/
* root
```

- First parent merge: commit «on main» (куда влили).
- Second parent: «on feature».
- `git cat-file -p HEAD` — две строки `parent`.
