# Exercises: Git Stash

> Complete after reading `1.lection-eng.md`, `2.lection-ru.md` and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Basic stash cycle
**Goal:** Experience push → list → pop.

**Steps:**
1. Create a test repo: `mkdir stash-lab && cd stash-lab && git init`
2. Create `app.txt`, commit it.
3. Modify `app.txt` (do not commit).
4. Run `git stash push -m "my first stash"`.
5. Verify `git status` is clean.
6. Run `git stash list` and `git stash show -p`.
7. Run `git stash pop` and verify content restored.

**Success criteria:** Clean status after push; file content restored after pop; stash list empty after successful pop.

---

### E2. Untracked with `-u`
**Goal:** See difference between stash with and without `-u`.

**Steps:**
1. In the same repo, create untracked `scratch.md` and modify `app.txt`.
2. Run `git stash push` (no flags). Check: is `scratch.md` still present?
3. Run `git stash pop` to restore.
4. Run `git stash push -u -m "with untracked"`. Check: is working tree fully clean?
5. `git stash pop` to restore.

**Success criteria:** You can explain why `-u` was needed for step 4.

## Medium

### M1. Branch switch workflow
**Goal:** Simulate hotfix interrupt.

**Steps:**
1. Create branch `feature/demo`, make uncommitted changes.
2. Stash with message including ticket id: `-m "DEMO-42 WIP"`.
3. Switch to `main`, make a commit (simulate hotfix).
4. Switch back to `feature/demo`.
5. `git stash pop`. Resolve any conflicts if they appear.

**Success criteria:** Both hotfix commit and feature WIP exist; you documented commands used.

---

### M2. Apply vs pop on two branches
**Goal:** Reuse one stash entry.

**Steps:**
1. Stash WIP on `main`.
2. Create branch `experiment/a`, `git stash apply`.
3. Switch to `experiment/b`, `git stash apply` again.
4. Drop stash manually with `git stash drop`.

**Success criteria:** Same changes applied twice; stash removed only after `drop`.

## Hard (stretch)

### H1. Inspect internal stash structure
**Goal:** Connect commands to `refs/stash`.

**Steps:**
1. Create a stash with both staged and unstaged changes (stage one file, modify another, then stash).
2. Run: `git log --oneline --graph refs/stash`
3. Run: `git show --format=fuller stash@{0}` and count parents.
4. Write 3–5 sentences explaining what the three parents represent.

**Success criteria:** You identify merge commit with 3 parents and relate them to HEAD/index/worktree.

## Hints (read only if stuck)

<details>
<summary>Hint E2</summary>
Default stash only tracks files Git already knows. Untracked files have no blob in the index until `-u` tells Git to include them in the stash commit's working tree snapshot.
</details>

<details>
<summary>Hint M1 conflicts</summary>
If pop conflicts, fix files, `git add`, then `git stash drop` if the stash entry remains. The stash is like applying a patch — conflicts mean branches diverged since stash was created.
</details>

<details>
<summary>Hint H1</summary>
Use `git cat-file -p stash@{0}` or `git rev-parse stash@{0}^1 stash@{0}^2 stash@{0}^3` to see the three parent SHAs.
</details>
