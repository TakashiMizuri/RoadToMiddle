# Benchmark: Sync vs Async HTTP Call — Thread Usage

> Roadmap: `1.4.22` · Node: `1.4` — Async + ASP.NET preview · Depth: практика

## Learning Objectives

After this lesson you will be able to:

- Explain **why** blocking sync HTTP under load consumes **thread pool threads** while async HTTP does not hold them during I/O wait.
- Set up a **BenchmarkDotNet** project to compare sync and async HTTP call patterns.
- Interpret benchmark results: **mean**, **allocated memory**, and **thread pool pressure** indicators.
- Design a realistic load scenario that mirrors **ASP.NET Core** request handling.
- Avoid common benchmarking mistakes: cold start, no warmup, shared mutable state, measuring localhost incorrectly.

---

## Why This Matters

In ASP.NET Core, every incoming HTTP request is handled on a **thread pool thread**. When your action calls a downstream API synchronously — `client.GetStringAsync(url).Result` or `GetString(url).GetAwaiter().GetResult()` — the thread **blocks** until bytes arrive over the network. Under moderate concurrency (hundreds of simultaneous requests), blocked threads accumulate. The thread pool injects new threads slowly (roughly two per second after a burst), and you see **queueing**, **latency spikes**, and eventually **503** responses or timeouts.

Async HTTP (`await client.GetStringAsync(url)`) releases the thread back to the pool while the OS completes the socket I/O. The same machine can serve far more concurrent requests because threads are a scarce resource and **I/O wait is not CPU work**. Middle developers must be able to **prove** this difference with numbers, not slogans — BenchmarkDotNet and a simple load test turn intuition into evidence you can show in a code review or incident postmortem.

This lesson connects directly to `1.4.20` (async all the way) and foreshadows `1.4.26`–`1.4.28` (HttpClient lifetime and resilience in production).

## Core Concepts

### Threads Are Not Free During I/O Wait

When code performs a **synchronous** HTTP call, the calling thread enters a blocked state inside the socket read. From the CLR's perspective, that thread cannot process other work. If you have 500 concurrent requests and each blocks a thread for 200 ms waiting on a remote API, you need **500 threads** just to sit idle. The default thread pool might start with ~CPU-count threads and grow under pressure, but growth is throttled to avoid melting the OS scheduler.

**Asynchronous** I/O uses **overlapped I/O** (Windows) or analogous mechanisms: the thread initiates the operation, registers a callback (continued via `Task`), and returns. When data arrives, a thread pool thread (often the same one, often a different one) runs the continuation. During the wait, **zero** threads are dedicated to that request's idle period — only a small state machine object and kernel socket state.

This is the core economic argument: **async scales concurrent I/O-bound work; sync scales only until threads run out**.

### What We Are (and Are Not) Measuring

A micro-benchmark comparing one sync vs one async call on an idle machine shows almost no difference in **wall-clock time** for a single call — both wait on the network equally. The lesson's benchmark must therefore include **concurrency**: many calls in flight at once, or a simulation of thread pool saturation.

BenchmarkDotNet excels at **isolated, repeatable** measurements. For thread pool behavior under HTTP, combine:

1. **BenchmarkDotNet** — allocation, mean time per operation at fixed parallelism.
2. **Optional: `dotnet-counters`** or **ThreadPool.GetAvailableThreads** sampling during a load spike — qualitative confirmation.

We measure **throughput** (operations per second at N concurrent workers) and **thread pool availability**, not just single-request latency.

### ASP.NET Core Preview Connection

In Kestrel, request processing is async-first. A controller action like `public async Task<IActionResult> Get()` should `await` downstream calls. If the team uses `.Result` inside that action, each blocked thread is one fewer thread available for other Kestrel requests and for the thread pool's global queue. At scale, **sync-over-async in a web app** is a production incident waiting to happen — the benchmark makes that visible in a controlled console app before it happens in staging.

## Under the Hood

When `HttpClient.SendAsync` completes asynchronously, the runtime uses **I/O completion ports** (IOCP) on Windows. The `Task` transitions from `WaitingForActivation` to `RanToCompletion` when the IOCP signals completion; continuations are queued to the **ThreadPool** (unless a `SynchronizationContext` captures them — ASP.NET Core has none on modern versions).

A sync `HttpClient.GetString` call blocks in `Socket.Receive` (simplified path). The thread's stack frame remains active; GC cannot reclaim that stack; the thread pool marks the worker as busy.

BenchmarkDotNet runs benchmarks in a separate process with **JIT warmup** and multiple iterations to defeat tiered compilation noise. It also isolates GC via logical groups. For async benchmarks, use `[Benchmark]` methods that return `Task` — BenchmarkDotNet awaits them correctly in modern versions.

## Syntax / Commands / API

### BenchmarkDotNet setup

```bash
dotnet new console -n HttpSyncAsyncBench
cd HttpSyncAsyncBench
dotnet add package BenchmarkDotNet
dotnet add package Microsoft.Extensions.Http
```

### Minimal benchmark skeleton

```csharp
using BenchmarkDotNet.Attributes;
using BenchmarkDotNet.Running;

[MemoryDiagnoser]
[ThreadingDiagnoser]
public class HttpBenchmarks
{
    private HttpClient _client = null!;
    private const string Url = "https://httpbin.org/delay/1"; // or local test server

    [GlobalSetup]
    public void Setup() => _client = new HttpClient();

    [Benchmark(Baseline = true)]
    public async Task AsyncGet()
    {
        await _client.GetStringAsync(Url);
    }

    [Benchmark]
    public string SyncGet()
    {
        return _client.GetStringAsync(Url).GetAwaiter().GetResult();
    }
}

// Program.cs
BenchmarkRunner.Run<HttpBenchmarks>();
```

Run with `dotnet run -c Release`. Always **Release** — Debug distorts timings.

### Concurrency benchmark (conceptual extension)

To stress threads, use `[Params(1, 50, 200)] public int Concurrency` and launch `Task.WhenAll(Enumerable.Range(0, Concurrency).Select(_ => AsyncGet()))` vs blocking parallel sync — or use BenchmarkDotNet's `Parallelism` job attribute where appropriate.

## Examples

### Example 1: Single-call benchmark (baseline)

Run async vs sync-over-async for one URL. You will likely see similar **mean** times (~RTT + server delay). **Allocated** bytes may differ slightly due to async state machine vs blocking path. This establishes that async is not "faster" for one call — it is **more scalable**.

### Example 2: Concurrent workers console

```csharp
var client = new HttpClient();
var url = "https://httpbin.org/delay/0.5";
var sw = Stopwatch.StartNew();
var tasks = Enumerable.Range(0, 200)
    .Select(_ => client.GetStringAsync(url));
await Task.WhenAll(tasks);
Console.WriteLine($"Async 200: {sw.ElapsedMilliseconds} ms");
```

Compare to 200 tasks each calling `.GetAwaiter().GetResult()` — watch **ThreadPool.GetAvailableWorkerThreads** before and after (available drops sharply under sync blocking).

### Example 3: Local controlled delay server

Relying on public HTTP adds network jitter. For repeatable labs, spin a minimal ASP.NET Core app:

```csharp
app.MapGet("/delay/{seconds:int}", async (int seconds) =>
{
    await Task.Delay(seconds * 1000);
    return "ok";
});
```

Point the benchmark at `http://localhost:5000/delay/1` to isolate thread behavior from WAN variance.

## Common Mistakes & Anti-patterns

**Measuring only one request.** Sync and async look equivalent; concurrency exposes the thread pool cliff.

**Using `.Result` in benchmarks without understanding deadlock risk.** In console apps with no `SynchronizationContext`, `.Result` often "works" — that misleads developers into using it in ASP.NET where context or dependencies differ.

**Creating new `HttpClient` per iteration.** Distorts results with socket setup cost; use one client for the benchmark (production lifetime is covered in `1.4.26`).

**Running in Debug configuration.** JIT optimizations disabled; times meaningless.

**Ignoring warmup.** First calls include DNS, TLS handshake, connection establishment — use `[GlobalSetup]` or warmup loops.

## Production & Real-World Notes

Production teams rarely keep BenchmarkDotNet in the main app; they run it in a **benchmark project** or use **load tests** (k6, NBomber, Azure Load Testing) against staging. The same principle applies: compare **RPS at p95 latency SLO** with sync-blocked vs async downstream calls.

During incidents, **`dotnet-counters`** (`System.Runtime` → `threadpool-thread-count`, `threadpool-queue-length`) on the ASP.NET process confirms thread pool starvation. The fix is almost always **remove sync-over-async** and ensure HttpClient is managed correctly — not "add more servers" alone.

## Comparison / Trade-offs

| Approach | Best for | Risk under load |
|----------|----------|-----------------|
| `await` async HTTP | ASP.NET, I/O-bound | Low thread usage |
| Sync `.Result` / `.Wait()` | Legacy interop (avoid) | Thread pool starvation |
| `Task.Run` + sync HTTP | Offloading (band-aid) | Wastes two threads per call |
| BenchmarkDotNet | Micro proof, CI regression | Not a substitute for load test |
| k6 / NBomber | End-to-end SLO validation | Heavier setup |

Async adds **state machine allocation** and cognitive overhead; sync is simpler to read. The trade-off for server-side I/O is universally in favor of async when concurrency exceeds a handful of requests.

## Key Takeaways

- Async HTTP does not make one call faster; it **frees threads during wait**, enabling higher concurrency.
- Sync blocking under load consumes thread pool workers and causes queueing and tail latency.
- Benchmark **concurrent** scenarios, not single calls, to see the difference.
- Use BenchmarkDotNet in **Release** with warmup and stable target (local delay server preferred).
- ASP.NET Core actions should **await** downstream HTTP — benchmark evidence supports code review policy.
- Thread pool growth is slow; starvation is a **cliff**, not a linear degradation.
- Combine micro-benchmarks with load tests for production confidence.
