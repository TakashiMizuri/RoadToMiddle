# Cookies и SameSite — краткая выжимка

> `0.3.9` · ~10 мин на повторение

## Главная идея

**Set-Cookie** (response) instructs browser хранить name=value; **Cookie** (request, `0.3.7`) auto-sent на matching URLs. Attributes определяют lifetime, scope и **cross-site policy** — критично для auth security.

## Ключевые пункты

- **HttpOnly** — JS не читает cookie (XSS theft harder). **Secure** — только HTTPS (`0.3.8`).
- **SameSite=Lax** (default) — block cross-site subresources; top-level GET nav ok.
- **SameSite=Strict** — strongest isolation; ломает некоторые cross-site flows.
- **SameSite=None** — cross-site send только с **Secure**; нужен CSRF defense.
- **Max-Age** preferred over **Expires**; session cookie без обоих.
- **Domain/Path** — minimum scope; host-only default safer.
- Cross-origin SPA (`app.` vs `api.`) + cookies → сложно; часто Bearer или BFF.

## Самые важные примеры

**Auth baseline:**
```
Set-Cookie: session=...; Path=/; HttpOnly; Secure; SameSite=Lax
```

**Cross-site embed/OAuth:**
```
Set-Cookie: ...; Secure; SameSite=None
(+ antiforgery / CSRF tokens)
```

## «Под капотом» в одном абзаце

Browser cookie jar matches URL (scheme, domain, path), then applies SameSite context (same-site vs cross-site, navigate vs XHR). ASP.NET Core signs/encrypts auth cookie via Data Protection. CORS `credentials: 'include'` не обходит SameSite — нужен None для cross-site credentialed requests. Third-party cookie phase-out влияет на trackers, не на типичный first-party Lax session.

## Частые ошибки

- Auth cookie без HttpOnly/Secure.
- SameSite=Lax при API на другом subdomain без плана.
- Sensitive JWT в readable JS cookie.
- Думать HttpOnly = полная защита от CSRF.

## Мнемоника

> **HttpOnly** — спрятать от JS; **Secure** — только по HTTPS; **SameSite** — когда browser **несёт** cookie на чужой site.
