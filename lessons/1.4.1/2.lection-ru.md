# Threads 101: kernel thread, thread pool и стоимость context switch

## Learning Objectives

После этой лекции ты сможешь:

- Различать **kernel thread** (поток ОС), **managed thread** и **`Task`**.
- Объяснить, как **.NET thread pool** планирует работу и зачем он нужен.
- Описать **стоимость context switch** и почему создание потоков дорого.

- Связать эту базу с тем, зачем существует **`async/await`** (preview `1.4.2`–`1.4.7`).

## Why This Matters

В `1.3.30` ты разобрал **multiple enumeration** в LINQ — лишние проходы по коллекции в одном потоке. Это проблема **CPU-эффективности**. Узел `1.4` открывает другой слой — **concurrency**: как программа использует **threads** при ожидании I/O или параллельной CPU-работе.

Каждый запрос ASP.NET Core, каждый `HttpClient`, каждый запрос через `DbContext` в итоге выполняется на **потоках**. Если считать потоки «бесплатными», API будет работать на ноутбуке с десятью пользователями и падать в production. Middle-разработчик должен понимать, что такое thread, что делает **thread pool** и почему **блокировка** worker-потока опасна — до изучения `async/await`.

Эта лекция — фундамент. Без него `Task.Run`, `async` и «thread pool starvation» остаются магическими словами, а не инженерными trade-offs.

## Core Concepts

### Что такое thread?

**Thread** — единица выполнения, которую планировщик ОС назначает на ядро CPU. Процесс содержит один или много потоков; у каждого свой **stack** (часто ~1 MB на Windows по умолчанию), **instruction pointer** и регистры. **Kernel** создаёт и планирует **kernel threads** (OS threads). CLR отображает **managed threads** (`System.Threading.Thread`) на kernel threads — обычно один к одному в современном .NET.

`new Thread(() => { ... }).Start()` просит CLR создать managed thread, а тот — kernel thread у ОС. Это **дорого**: структуры ядра, stack, регистрация в scheduler. Тысячи потоков в секунду для web-сервера — нереалистично.

**Parallelism** — несколько потоков одновременно на нескольких ядрах. **Concurrency** — несколько задач «движутся вперёд», не обязательно в один момент. Один поток может обслуживать много concurrent I/O, если не блокируется в ожидании (async-модель в следующих уроках). Путать parallelism и concurrency — путь к «thread на каждый request».

### .NET Thread Pool

Вместо бесконечного `new Thread()` .NET даёт **thread pool**: набор worker-потоков, которые runtime создаёт, переиспользует и расширяет под нагрузку. `Task.Run`, `ThreadPool.QueueUserWorkItem`, приём HTTP в ASP.NET Core — типично очередь в **ThreadPool**.

Pool держит **global queue** и (с .NET 4) **local queues** на worker с work-stealing. Небольшое число потоков при старте процесса; pool **инжектит** новые, когда есть очередь и workers заняты — но **throttled** (классически ~2 потока за 500 ms).

Worker-потоки **общие**. ASP.NET Core, `Timer`, `Task.Run`, библиотеки — все конкурируют за один pool. Блокировка одного потока синхронным I/O или `.Result` уменьшает число idle workers для всех.

### Стоимость context switch

Когда ОС **вытесняет** thread A ради thread B, происходит **context switch**: сохранить регистры и stack pointer A, восстановить состояние B, обновить метаданные scheduler, возможно сбросить CPU cache. Стоимость — **микросекунды–десятки микросекунд** на switch; при масштабе — катастрофа.

10 000 runnable threads на 8 ядрах — ОС тратит время на переключения, а не на полезную работу. Высокий **ready-thread count** увеличивает tail latency. Цель production — достаточно потоков, чтобы загрузить CPU, без массового oversubscription. Для I/O-heavy приложений часто нужно **меньше** blocked threads, не больше.

### Threads vs Tasks (preview)

**`Task`** — абстракция асинхронной работы; это **не** thread. Task может выполниться на pool thread, может завершиться без user code на потоке (cached result), может продолжиться после I/O. **Task — абстракция; thread — ресурс выполнения.** Разбор `Task` — в `1.4.3`; пока запомни: schedule task ≠ create thread.

## Under the Hood

### Kernel scheduling и quantum

В Windows **scheduler** выдаёт runnable thread **quantum** CPU на ядре. Quantum истёк, появился более приоритетный thread, или thread **заблокировался** (syscall I/O, lock) — kernel делает context switch. **Blocking syscall** убирает thread из runnable set — ядро свободно для других. **Spin-wait** держит thread runnable и жжёт CPU.

Linux (CFS) — другая механика, тот же вывод: runnable threads конкурируют за cores; blocking освобождает core.

### Алгоритм injection pool (концептуально)

CLR thread pool смотрит **throughput**: завершённые work items vs глубина очереди. Очередь растёт, completion rate падает — добавляются threads, медленно. Burst из 10 000 `Task.Run` сначала только очередь; injection догоняет за сотни ms. **`ThreadPool.SetMinThreads`** поднимает floor — tuning, не default fix.

**I/O completion ports** (Windows) связывают async I/O completions с pool: когда async read завершён, pool thread (или completing thread — зависит от API/версии) запускает continuation. Мост ОС → managed callbacks — база для `1.4.6`.

### Stack memory и лимиты процесса

Stack каждого thread — virtual address space. 64-bit process имеет огромное virtual space, но **commit charge** и handles важны. Thread-per-request с 10k blocked connections — memory pressure до насыщения CPU. Pool переиспользует ограниченный набор stacks.

### Thread vs ThreadPool vs Task

| Механизм | Новый OS thread? | Типичное use |
|----------|------------------|--------------|
| `new Thread()` | Да | Долгий background, STA |
| `ThreadPool.QueueUserWorkItem` | Reuse pool | Низкоуровневая очередь |
| `Task.Run` | Очередь в pool | CPU offload (`1.4.4`) |
| `async/await` | Нет thread при await I/O | I/O scalability |

## Syntax / Commands / API

```csharp
var t = new Thread(() => Console.WriteLine(Thread.CurrentThread.ManagedThreadId));
t.IsBackground = true;
t.Start();

ThreadPool.QueueUserWorkItem(_ => DoWork());

Console.WriteLine(Thread.CurrentThread.IsThreadPoolThread);

ThreadPool.SetMinThreads(minWorker: 100, minIo: 100); // с measurement!
```

Диагностика: `dotnet-counters` (`threadpool-thread-count`, `threadpool-queue-length`), APM metrics.

## Examples

### Пример 1: один поток, последовательная работа

```csharp
for (int i = 0; i < 5; i++)
    Console.WriteLine($"Work {i} on thread {Thread.CurrentThread.ManagedThreadId}");
```

Все итерации на одном потоке (часто main). Нет parallelism.

### Пример 2: pool workers

```csharp
var tasks = Enumerable.Range(0, 8).Select(i => Task.Run(() =>
{
    Thread.Sleep(100);
    return i;
})).ToArray();
Task.WaitAll(tasks);
```

`Task.Run` ставит 8 work items в pool. Может быть меньше 8 distinct thread IDs — reuse.

### Пример 3: oversubscription

50 000 длинных `Task.Run` заполняют очередь. Injection throttling + context switches растягивают completion — «больше threads» не всегда «быстрее».

## Common Mistakes & Anti-patterns

**Thread per request** — legacy ASP.NET, naive sockets. Тысячи blocked threads → memory + scheduler collapse. ASP.NET Core — pool + async I/O.

**Blocking pool threads** — sync `.Result`, `Thread.Sleep` в request path. Symptom: hang при moderate concurrency, low CPU — **starvation** (`1.4.10`).

**Task = thread** — 100 tasks ≠ 100 threads.

**`new Thread()` для короткой работы** — выше overhead, хуже в shared ASP.NET environment.

## Production & Real-World Notes

Kestrel dispatch в thread pool. `SetMinThreads` иногда после burst load tests — не по умолчанию.

Kubernetes CPU limits влияют на полезный parallelism.

`IHostedService` для long-running — не unbounded `Task.Run` без `LongRunning`.

Мониторь queue length и active pool threads в APM.

## Comparison / Trade-offs

**Dedicated vs pool** — isolation vs shared scarce resource.

**Больше threads vs async I/O** — I/O: async освобождает threads (`1.4.2`); CPU: parallelism до числа cores, дальше diminishing returns.

**Horizontal scale** — больше pods; внутри процесса pool + async максимизирует throughput.

## Key Takeaways

- Threads — **OS resources** со stack и scheduling overhead.
- **Thread pool** — общий ресурс ASP.NET и `Task.Run`.
- **Context switches** и oversubscription бьют по scalability.
- **`Task` ≠ thread**.
- Blocking pool threads — частая причина production hangs.
- Далее: **I/O vs CPU** (`1.4.2`).
