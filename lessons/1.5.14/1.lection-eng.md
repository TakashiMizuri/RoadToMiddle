# ArrayPool — Rent and Return Pattern

> Roadmap: `1.5.14` · Node: `1.5` — C#: .NET Runtime and Memory · Depth: понимание

## Learning Objectives

After this lesson you will be able to:

- Explain **`ArrayPool<T>.Shared`** as a **rent/return** pool reducing allocations for temporary **`T[]`** buffers.
- Implement **`Rent` → use → `Return`** correctly, including **`clearArray`** semantics.
- Avoid **use-after-return**, **double return**, and **retaining rented arrays** in fields.
- Choose between **`ArrayPool`**, **`stackalloc`**, and **`new`** for buffer sizing.
- Use **`ArrayPool<byte>`** in serialization, I/O, and logging hot paths responsibly.

---

## Why This Matters

High-throughput services allocate **`byte[]`** and **`char[]`** constantly — JSON serialization scratch buffers, copying streams to memory, building log messages. Each **`new byte[4096]`** hits Gen0; under load, allocation rate drives **GC pauses**. **`System.Buffers.ArrayPool<T>`** (.NET Core 2.1+) rents arrays from **shared pools**; when you **`Return`**, they reuse — ** amortized zero allocation** if pool warm.

Getting rent/return wrong causes **data leaks between requests** (sensitive bytes in pooled buffer), **corruption** (use after return), or **pool pollution** (never returned arrays → pool grows then falls back to GC anyway). Middle developers use **`ArrayPool`** where BCL already does (`PipeReader`, Kestrel) and apply the same discipline in app code.

## Core Concepts

### Rent and Return

```csharp
byte[] buffer = ArrayPool<byte>.Shared.Rent(minimumLength: 512);
try
{
    int written = FillBuffer(buffer.AsSpan());
    Process(buffer.AsSpan(0, written));
}
finally
{
    ArrayPool<byte>.Shared.Return(buffer, clearArray: true);
}
```

**`Rent(minimumLength)`** returns array with **`.Length >= minimumLength`** — often **larger** than requested (bucket sizes). Use **logical length** separately; do not assume exact size.

**`Return(array)`** — must be **same array reference** from Rent (same pool instance). **`clearArray: true`** zeroes sensitive data; **`false`** faster but next renter may see old bytes.

### Shared vs Custom Pools

**`ArrayPool<T>.Shared`** — global buckets per power-of-two-ish sizes; thread-safe.

**`ArrayPool<T>.Create(maxArrayLength, maxArraysPerBucket)`** — isolated pool for **predictable** retention (tests, subsystems that shouldn't starve shared pool).

Most app code: **Shared** only.

### Lifetime Rules

1. **Do not** store rented array in **instance field** beyond method unless you transfer ownership explicitly (rare anti-pattern).
2. **Do not** **`Return`** then use array.
3. **Do not** **`Return`** twice.
4. **Do not** **`Return`** array you **`new`**'d — only rented arrays.
5. Prefer **`try/finally`** or **`using` wrapper** (`ArrayPoolBufferWriter`, custom disposable).

### clearArray Trade-off

| clearArray | Pros | Cons |
|------------|------|------|
| `true` | No data leak between tenants | CPU zero-fill |
| `false` | Faster | Next renter sees stale data |

For crypto/password buffers — **always clear**. For idempotent parse scratch — often **`false`** acceptable in single-tenant batch jobs; **public multi-tenant API** — **`true`** or use Span only portion you wrote.

### Relationship to Span and stackalloc

```csharp
byte[] rented = ArrayPool<byte>.Shared.Rent(1024);
Span<byte> span = rented.AsSpan(0, 1024);
```

Large or unknown size → **ArrayPool**. Tiny fixed → **stackalloc** (`1.5.13`). Long-lived → **`new`** or **`MemoryPool`** / **`IMemoryOwner`**.

## Under the Hood

Pools maintain **buckets** of arrays by size. **`Rent`** pops from bucket or allocates new array if empty. **`Return`** pushes back until **maxArraysPerBucket**; excess arrays become eligible for GC.

Arrays are **never shrunk** — returned buffer may be **8192** when you asked **512** — always track **used length**.

---

## Common Mistakes & Anti-patterns

**Forgot Return** — pool depletion + LOH pressure from orphaned large arrays.

**Return in wrong finally after exception that still references buffer elsewhere** — rare but document ownership.

**Assuming Rent length == minimumLength** — buffer overruns if you use full `.Length` without intent.

**Pooling tiny arrays** — overhead may exceed benefit; profile.

## Production & Real-World Notes

**`Microsoft.IO.RecyclableMemoryStream`** builds on similar ideas for streams.

ASP.NET **`ArrayPoolWriter`** patterns in custom middleware — rent once per request.

Code review: every **`Rent`** has matching **`Return`** in **`finally`**.

## Quick Reference

```csharp
var pool = ArrayPool<byte>.Shared;
var buf = pool.Rent(size);
try { /* use buf.AsSpan(0, size) */ }
finally { pool.Return(buf, clearArray: sensitive); }
```

---

## Key Takeaways

- **Rent ≥ size**; track logical length.
- **Always Return** same reference in **finally**.
- **clearArray** when security matters.
- Don't pool **long-lived** or **field** storage without design.
- Complements **Span** (`1.5.12`) and **stackalloc** (`1.5.13`).
