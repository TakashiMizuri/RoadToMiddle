# Postman, Bruno, and curl — Manual API Testing

> Roadmap: `0.3.12` · Node: `0.3` — HTTP and the Web · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Explain **why manual HTTP clients** remain essential alongside automated tests and Swagger UI.
- Send requests with **curl** — methods, headers, body types, and reading status plus response headers.
- Use **Postman** (or equivalent GUI) to organize collections, environments, and reusable auth.
- Use **Bruno** as a **Git-friendly**, file-based alternative to cloud-synced collections.
- Reproduce bugs by crafting exact requests (content types from `0.3.11`, auth headers, query strings).
- Export or share requests in formats teammates and CI can reuse.

---

## Why This Matters

Swagger UI and integration tests cover happy paths, but production debugging starts with a precise question: **what exactly did we send, and what came back?** A frontend bug might be a missing header; a 403 might be a stale token; a 415 might be wrong `Content-Type`. Manual API tools let you **isolate the HTTP layer** — strip away React state, EF Core, and browser cache — and iterate in seconds.

**curl** lives on every server and in every CI image; knowing it is non-negotiable for Middle developers. **Postman** dominates teams that want GUI, documentation, and mock servers. **Bruno** gained traction where **collections belong in the repository** next to code, not in a vendor cloud. All three implement the same mental model: build a request, observe the raw response, adjust one variable at a time. Fluency here accelerates everything from OAuth debugging to webhook replay.

## Core Concepts

### Manual Testing vs Other Layers

**Unit tests** mock dependencies; they may never hit real HTTP serialization. **E2E tests** run the full stack but hide request details in trace logs. **OpenAPI/Swagger** documents contracts but often auto-generates naive requests. Manual clients fill the gap: you control **every byte and header**, save reproducible examples, and share them in Slack or tickets.

The workflow is scientific: **hypothesis** (server rejects JSON without charset) → **single change** (add header) → **observe** (200 vs 415) → **document** (save request in collection). This is how seniors debug before touching application code.

### curl — The Universal CLI Client

**curl** (and Windows **curl.exe**, not legacy PowerShell alias confusion) speaks HTTP directly. It is ideal for scripts, README examples, and SSH sessions on production jump boxes where GUI tools do not run.

Core flags:

- **`-X METHOD`** — HTTP verb (curl defaults to GET; POST often inferred with `-d`).
- **`-H "Name: value"`** — repeat for multiple headers.
- **`-d 'body'`** — sends body; implies POST unless `-X` overrides.
- **`-G`** — append `-d` data as query string on GET.
- **`-i`** — include response headers in output.
- **`-v`** — verbose (TLS handshake, request/response headers) — essential for debugging.
- **`-F "key=value"`** — multipart form (sets boundary automatically).
- **`--data-urlencode`** — safe urlencoded fields.

```bash
curl -sS -w "\nHTTP %{http_code}\n" \
  -X POST "https://httpbin.org/post" \
  -H "Content-Type: application/json" \
  -H "X-Request-Id: debug-001" \
  -d '{"probe":true}' \
  -i
```

Reading output: status line first (`HTTP/2 200`), then headers, blank line, body. Piping to `jq` parses JSON in the shell.

### Postman — GUI Collections and Environments

**Postman** organizes work into **collections** (grouped requests), **environments** (variable sets: `baseUrl`, `accessToken`), and **pre-request scripts** / **tests** (JavaScript hooks). You pick method, URL, Params tab for query string, Headers, Body (raw JSON, form-data, x-www-form-urlencoded), then **Send**.

**Environments** solve "works on my machine": `{{baseUrl}}/api/users` resolves to localhost in Dev and staging URL in QA. After login request, a test script can `pm.environment.set("accessToken", json.access_token)` for Bearer auth on subsequent calls.

Teams share collections via workspace sync or exported JSON. Downsides: large collections drift from code; secrets in shared exports; cloud sync may conflict with compliance policies.

### Bruno — Collections as Files in Git

**Bruno** stores each request as text files in a folder (`.bru` format) inside your repo — e.g. `bruno/api/login.bru`. Variables live in `environments/local.bru`. **Git diff** shows when someone changed the health-check URL; PR review includes API changes.

Bruno supports the same HTTP features: methods, headers, body types, auth helpers. Trade-off: smaller ecosystem than Postman; no heavyweight cloud — often a feature for regulated or offline teams.

Conceptually: **Postman** optimizes individual productivity and sharing via platform; **Bruno** optimizes **version control** and team review alongside source.

### Building a Reproducible Request

Regardless of tool, a complete manual test captures:

1. **Method and full URL** (including query).
2. **Headers** — especially `Authorization`, `Content-Type`, `Accept`, correlation IDs.
3. **Body** — exact encoding (JSON, form, multipart).
4. **Expected status** and key response headers (`Location`, `Set-Cookie`).
5. **Environment** — which server, which user role.

From `0.3.11`: wrong content type is the fastest way to waste an hour; set it explicitly in every tool's Body tab or curl `-H`.

### Auth Patterns in Manual Tools

- **Bearer JWT:** `Authorization: Bearer eyJ...` — paste token from login response or script refresh.
- **Basic:** `-u user:pass` in curl; Postman Authorization tab.
- **API key:** custom header `X-Api-Key` or query param (prefer header).
- **Cookie:** `-b "session=..."` or jar file `-c cookies.txt` after login.

Always use **environment variables** for secrets — never commit tokens; `.env` or Bruno secrets stay local.

## Under the Hood

curl uses **libcurl** — same library many languages bind. It handles DNS, TCP, TLS (certificate verification with `-k` only for local dev), HTTP/1.1 or HTTP/2 as negotiated. **Verbose mode** shows whether TLS failed, redirect followed (`-L`), or HTTP/2 push occurred.

Postman/Bruno embed HTTP clients (Node or similar). They serialize multipart boundaries like browsers. **Code generation** (Postman → curl, C#, JavaScript fetch) helps paste into apps once manual request works.

**localhost vs Docker:** `localhost:5000` from host vs `http://host.docker.internal` from container — manual tools expose wrong-host mistakes immediately.

## Syntax / Commands / API

**curl cheat sheet:**

```bash
# GET with query
curl "https://api.example.com/items?status=open&limit=10"

# PUT JSON
curl -X PUT "$BASE/items/1" -H "Content-Type: application/json" -d @body.json

# DELETE with auth
curl -X DELETE "$BASE/items/1" -H "Authorization: Bearer $TOKEN"

# Save cookies / use cookies
curl -c cookies.jar -X POST .../login -d ...
curl -b cookies.jar .../me

# Follow redirects
curl -L -o /dev/null -w "%{http_code}" https://example.com/redirect

# Timing
curl -w "@curl-format.txt" -o /dev/null -s https://api.example.com/health
```

**Postman/Bruno parity:**

| Task | Postman | Bruno |
|------|---------|-------|
| Variables | Environment | Environment folder |
| Share | Export / workspace | Git commit |
| Tests | Tests tab (JS) | Assertions in `.bru` |
| Multipart | Body → form-data | Body → multipart |

## Examples

### Example 1: Debug 401 on API

Frontend reports unauthorized. Copy failing URL and headers from browser DevTools Network tab. Paste into Bruno/Postman. Remove frontend-only headers (`Sec-Fetch-*`). Add valid Bearer from separate login request. If manual call succeeds, bug is client token storage; if it fails, bug is server auth or wrong audience claim.

### Example 2: curl in ticket comment

```bash
curl -X POST "$STAGING/api/orders" \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"sku":"WIDGET-1","qty":2}'
# Expected: 201 + Location header
```

QA reruns exact steps without Postman account — reproducibility.

### Example 3: Collection in repo (Bruno)

```
bruno/
  bruno.json
  environments/local.bru
  orders/create-order.bru
  auth/login.bru
```

PR changes `create-order.bru` URL path when API version bumps — visible in code review.

## Common Mistakes & Anti-patterns

**Trusting Swagger "Try it" without checking Content-Type** — UI may omit charset or auth your real client sends.

**Copy-paste secrets into shared collections** — rotate leaked keys; use environment placeholders.

**PowerShell `curl` alias** — on Windows, `curl` may invoke `Invoke-WebRequest`; use **`curl.exe`** for real curl.

**Not reading response headers** — `429` with `Retry-After`, `401` with `WWW-Authenticate` explain failures body JSON hides.

**Testing only through GUI** — production SSH has curl, not Postman.

**Ignoring `-v` on TLS errors** — certificate mismatch shows in verbose output first.

## Production & Real-World Notes

Store **golden requests** in repo (Bruno) or runbook (curl blocks) for incident response: health check, admin read-only query, cache bust. CI can run ** Newman** (Postman CLI) or **bru CLI** for smoke tests against staging.

Rate limits: manual bursts from whole team hitting same endpoint trigger 429 — use personal dev keys. **Correlation ID** header (`X-Correlation-ID`) should be set in manual tests to trace server logs.

For GDPR, do not point Postman history sync at production PII without policy approval — Bruno local-only helps.

## Comparison / Trade-offs

| Tool | Strength | Weakness |
|------|----------|----------|
| curl | Universal, scriptable, CI | No built-in collection UI |
| Postman | Rich GUI, mocks, team sync | Cloud lock-in, drift |
| Bruno | Git-native, offline | Smaller community |

**Swagger UI:** great for discovery; poor for chained auth flows and custom edge cases. **Automated tests:** regression safety; manual tools: exploration speed.

## Key Takeaways

1. Manual HTTP tools **isolate the wire protocol** from app frameworks.
2. **curl** is mandatory for servers, scripts, and shareable repro steps.
3. **Postman** excels at GUI workflows, environments, and team sharing via platform.
4. **Bruno** stores requests as **files in Git** — better for review and compliance.
5. Always set **Content-Type** and **Authorization** explicitly (`0.3.11`).
6. Read **status and response headers**, not JSON body alone.
7. Save working requests as **living documentation** for QA and incidents.
