## Answers

### 1. [Recall]
**Answer:**
- **HEAD** — указатель на текущий commit (tip ветки).
- **Index (staging)** — snapshot следующего commit.
- **Working tree** — файлы в рабочем каталоге на диске.

---

### 2. [Recall]
**Answer:** Убирает файл из staging: **index** приводится к состоянию **HEAD**. Working tree **не меняется**.

**Why:** `--staged` = unstage operation.

---

### 3. [Understanding]
**Answer:** В **working tree как unstaged** изменения (index сброшен к HEAD~1, WT файлы физически остались с содержимым отменённого commit).

---

### 4. [Understanding]
**Answer:** `--soft` — index **не** сбрасывается (изменения остаются **staged**). `--mixed` — index сбрасывается к target commit (изменения **unstaged**).

---

### 5. [Understanding]
**Answer:** `revert` добавляет **новый forward commit** — teammates делают обычный `git pull` без force push. `reset --hard` откатывает branch pointer — требует **force push**, ломает history у тех, кто уже pull'нул.

---

### 6. [Understanding]
**Answer:** Из **index** (staging area). Если index совпадает с HEAD — файл возвращается к последнему commit для unstaged правок.

---

### 7. [Application]
**Answer:** `git restore --staged .env`

---

### 8. [Application]
**Answer:** `git reset --soft HEAD~1`

---

### 9. [Application]
**Answer:**
```bash
git revert a1b2c3d
git push origin main
```

---

### 10. [Analysis]
**Answer:**
```bash
git reflog
git reset --hard <hash-before-reset>   # или HEAD@{n}
```

---

### 11. [Analysis]
**Answer:** **Неверно.** Revert **не удаляет** commit — добавляет новый commit с обратным patch. Плохой commit остаётся в DAG; состояние кода откатывается.

---

### 12. [Application]
**Answer:** `git restore .` (или `git restore :/` в некоторых версиях — обычно `git restore .` из корня repo)

---

### 13. [Understanding]
**Answer:** `HEAD~1` — первый parent текущего commit (один commit назад). После `reset --hard HEAD~1` HEAD указывает на parent; index и WT соответствуют этому commit.

---

### 14. [Analysis]

| Mode | HEAD | Index | Working tree |
|------|------|-------|--------------|
| --soft | ✓ | — | — |
| --mixed | ✓ | ✓ | — |
| --hard | ✓ | ✓ | ✓ |

(✓ = changes/resets to target)

---

### 15. [Understanding]
**Answer:**
- **Rewrite:** чище history, но **скрывает** ошибку в audit trail; опасен для shared.
- **Revert:** **прозрачный** audit (видно ошибку и fix); безопасен для команды; history длиннее.

---

### 16. [Application]
**Answer:** Файл полностью совпадает с **HEAD** — и в index, и в working tree. Все локальные staged и unstaged правки в этом файле **потеряны**.

---

### 17. [Analysis]
**Answer:** Merge commit имеет **два parents**. Git не знает, какую mainline считать «правильной» для inverse patch — `-m 1` выбирает первого parent (обычно ветка, в которую мержили).

---

### 18. [Recall]
**Answer:** Reflog **локальный** (`.git/logs/`). Восстановить commit с **другой машины** через свой reflog **нельзя** — нужен hash с той машины, remote, или backup.

## Model answers for practical sections

### P1. Three-tree scenario

| Command | HEAD | Staged | Unstaged in app.ts |
|---------|------|--------|---------------------|
| `git restore app.ts` | C3 | есть | **нет** (WT ← index, unstaged потеряно) |
| `git restore --staged app.ts` | C3 | **нет** | есть |
| `git reset --soft HEAD~1` | C2 | **есть** (весь C3 snapshot) | есть |
| `git reset --hard HEAD~1` | C2 | **нет** | **потеряны** |

---

### P2. Choose the tool

1. `git restore <file>`
2. `git reset HEAD~1` (или `reset --mixed`)
3. `git revert <hash>`
4. `git reflog` then `git reset --hard <hash>`
