# Undo: restore, revert vs reset (soft/mixed/hard)

> Roadmap: `0.1.8` · Узел: `0.1` — Git: основы · Глубина: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Объяснить три дерева Git: **HEAD**, **index (staging)** и **working tree**.
- Использовать **`git restore`** для безопасного discard или unstage изменений файлов.
- Использовать **`git revert`** для отмены commit через **новый** commit на shared branches.
- Применять **`git reset --soft/--mixed/--hard`** и точно описывать, что при этом двигается.
- Восстанавливаться после ошибок через **`git reflog`**.
- Выбирать правильную undo-стратегию: **rewrite history** vs **add corrective commit**.

---

## Why This Matters

Каждый разработчик рано или поздно сталкивается с моментом, когда в Git оказалось не то, что задумывалось. В staging area случайно оказался `secrets.env`. Commit ушёл в `main` вместо feature branch. Нужно отменить merge прошлой недели на ветке, которую уже pull'нули три коллеги. Команды этого урока — способ восстановиться, но неправильная undo-команда может стоить часов работы или сломать shared history команды.

Разница между уверенным Middle и нервным Junior не в заучивании syntax. Она в понимании **что куда двигается** при каждой команде. У Git нет одной кнопки «undo». Есть несколько механизмов, которые в разговоре звучат похоже — «можешь отменить тот commit?» — но под капотом ведут себя принципиально по-разному. Одни двигают branch pointers назад и переписывают directed acyclic graph. Другие добавляют новые commits, исправляющие историю вперёд. Третьи трогают только файлы на диске, не двигая history. Ошибка на shared repository вынуждает painful force push и rebase у всей команды. Ошибка локально может навсегда уничтожить uncommitted edits, которые так и не стали Git objects.

Урок идёт глубоко, потому что undo operations стоят на пересечении three-tree model Git, reference mechanics и collaboration etiquette. Освоишь это — incidents станут recoverable stories, а не panicked Slack threads.

## Core Concepts

Git управляет проектом через три синхронизированных представления, которые разработчики часто смешивают в голове. Разделить их — foundation для каждого undo-решения.

Первое дерево — **HEAD**, текущий commit на ветке — последний snapshot, записанный в history. Когда говорят «последний commit», имеют в виду то, на что указывает HEAD, обычно через branch reference вроде `refs/heads/feature`. HEAD — authoritative «как Git считает, выглядит проект в committed form».

Второе дерево — **index**, staging area или cache. Здесь snapshot, который запишет следующий `git commit`. `git add` копирует content из working tree в index и создаёт blob objects для staged content. Index — flat structure в `.git/index`: paths, modes, blob hashes, metadata. Это черновик следующего commit.

Третье дерево — **working tree**, реальные файлы на диске, которые показывает editor. Его меняешь, когда пишешь код. Git сравнивает working tree с index — unstaged changes; index с HEAD — staged changes.

Обычный forward flow связывает все три дерева по цепочке. Редактируешь файл в working tree. `git add` копирует edits в index. `git commit` создаёт новый commit object из index и двигает branch reference вперёд — HEAD на новый commit. Undo commands ломают или разворачивают части этой цепочки — но не все ломают одни и те же части.

```
edit file.ts          →  working tree changes
git add file.ts       →  working tree → index
git commit            →  index → new commit, HEAD moves forward
```

Помимо трёх деревьев, undo philosophy делится на два лагеря, критичных для team projects. **Rewrite history** — двигаем branch pointers назад или меняем commits на месте, чтобы репозиторий выглядел так, будто certain commits не было. `git reset`, `git rebase` переписывают. **Add new truth** — existing commits остаются, append corrective commits. Так работает `git revert`. На shared branches, которые others уже pull'нули, rewrite заставляет всех reconcile divergent history. Forward correction через revert — normal `git pull`, undo приходит как ещё один commit.

Rule of thumb, который спасает карьеры: если кто-то мог pull'нуть commit, который хочешь отменить — prefer revert over reset.

Ещё один частый сценарий, который проясняет разницу между restore и reset: ты редактировал пять файлов, но понял, что один из них (`appsettings.json`) нужно вернуть к состоянию последнего commit, а остальные четыре оставить как есть. Здесь не нужен reset — он двинул бы всю ветку. Достаточно `git restore appsettings.json`, и только этот файл на диске станет как в index (или HEAD, если не был staged). Reset отвечает на другой вопрос: «я хочу, чтобы указатель ветки указывал на другой commit». Смешение этих двух намерений — главный источник ошибок у тех, кто учит Git только по шпаргалкам команд.

## Under the Hood

Branch — не контейнер commits, а movable pointer на один commit hash. `refs/heads/main` может указывать на `a1b2c3d`. HEAD обычно на branch name, не напрямую на commit: HEAD → `refs/heads/main` → `a1b2c3d`. `git commit` создаёт новый commit, parent — current tip, branch reference двигается вперёд. `git reset --hard` на older commit двигает branch reference назад. Новые commits ещё какое-то время exist как objects в `.git/objects`, но ни один branch на них не указывает — unreachable до garbage collection.

HEAD может указывать напрямую на commit в **detached HEAD** — урок `0.1.11`. При reset на normal branch двигается branch ref, на который косвенно указывает HEAD. Поэтому `git reset` влияет на history, а `git restore` — нет: restore manipulates index и working tree через checkout machinery без движения branch tip.

Index хранит больше, чем список paths. Каждая entry включает stage number для merges — stage zero для normal, stages 1–3 для conflicted paths. Для undo важно: `git add` пишет blob objects и обновляет index; `git reset --mixed` repopulates index из tree target commit; `git reset --hard` repopulates index и checkout files в working tree под match.

На object level при hard reset с `C3` на `C2`:

```
До:   main → C1 ─ C2 ─ C3
После reset --hard на C2:  main → C1 ─ C2    (C3 unreachable от main)
```

Commit `C3` не удаляется мгновенно. Остаётся в object database и reflog до expiration и GC. Recovery возможен.

Revert создаёт другую форму history. Если `C3` ввёл bug и ты его revert'ишь:

```
До:   main → C1 ─ C2 ─ C3
После revert:  main → C1 ─ C2 ─ C3 ─ C4
```

`C4` — новый commit, tree которого applies inverse patch `C3`. Git использует ту же merge machinery, что при conflict resolution — если revert не apply cleanly, потому что later commits трогали те же строки, resolve вручную как merge. Original mistake виден в history — compliance advantage или cosmetic annoyance, зависит от audit requirements команды.

**Reflog** — local append-only log движений references в `.git/logs/HEAD` и `.git/logs/refs/heads/<branch>`. Каждая строка: old hash, new hash, actor, timestamp, message (`commit`, `reset`, `checkout`). Reflog по умолчанию не push'ится на remote. Personal safety net, когда hard-reset слишком далеко и нужен hash, где HEAD был тридцать секунд назад. Entries expire — default ninety days — recovery не бесконечен, но покрывает essentially all «oops» в daily work.

Почему `--hard` destructive для uncommitted work? Независимо от branch pointers survive только content, дошедший до object database. Staged content имеет blob objects — `git add` их создал. Pure working-tree edits без stage никогда не hashed — hard reset overwrites files под target commit tree, edits gone без object для recovery. Эта asymmetry — главная причина pause перед `--hard`.

Полезно мысленно пройти полный цикл `git reset --mixed HEAD~1` по объектам. До команды ветка указывает на commit C3, index совпадает с деревом C3, на диске те же байты (плюс возможные unstaged правки поверх index). Git читает дерево commit C2 и перезаписывает index этим деревом — staged-состояние «откатывается» к C2. Указатель ветки перемещается на C2. Рабочая директория Git не трогает: файлы на диске остаются с содержимым из C3, но теперь Git показывает их как unstaged diff относительно C2. Именно поэтому mixed reset — любимый способ «раскоммитить, но не потерять код». При `--soft` index не перезаписывается: в staging area остаётся снимок C3, хотя HEAD уже на C2 — удобно, если нужно только поправить message или разбить один commit на два. При `--hard` Git после обновления index ещё и выписывает файлы на диск из дерева C2, затирая любые расхождения.

Revert на уровне объектов не двигает указатель назад. Git вычисляет patch, обратный изменениям выбранного commit, и пытается применить его поверх текущего tip. Если с момента «плохого» commit кто-то менял те же строки, возникают conflict markers — как при merge. После разрешения конфликтов и завершения revert в history появляется новый commit, а спорный остаётся видимым. Для аудита и compliance это плюс: видно и ошибку, и исправление. Для эстетики линейной history — минус; команды с жёстким «main только fast-forward без лишних revert-коммитов» всё равно чаще выбирают revert на shared main, потому что альтернатива — force push — дороже для всех.

Reflog стоит воспринимать как журнал движений указателей только на твоей машине. Когда коллега говорит «я сделал hard reset и потерял три commit», первый вопрос — делал ли он это до `git gc` и есть ли ещё записи в `git reflog`. Восстановление выглядит как `git reset --hard HEAD@{n}` на запись *до* ошибочного reset. Reflog не синхронизируется с remote: если ты пересоздал clone, старых записей там не будет. Срок хранения по умолчанию около 90 дней — для повседневных «ой» этого более чем достаточно.

## Syntax / Commands / API

Git 2.23 ввёл `git restore` как modern, focused команду для движения content между index и working tree без touch HEAD. Раньше `git checkout` overload'или и branch switch, и file restoration — common confusion. Restore separates file operations от branch operations.

Discard unstaged changes в working tree — restore file из index:

```bash
git restore Program.cs
```

Index unchanged; файл на диске match staged. Если ничего не staged — match HEAD. Unstage file — index entry обратно к HEAD, working tree edits остаются:

```bash
git restore --staged Program.cs
```

На диске edits на месте; Git убирает файл из draft следующего commit. Unstage и discard working tree одним шагом — полный return к last committed state:

```bash
git restore --staged --worktree Program.cs
```

Заменяет older pattern `git checkout HEAD -- Program.cs`. Restore never moves branch — feature: fix local file mistakes без touch history.

`git revert` создаёт новый commit, отменяющий указанный. Original остаётся в history.

```bash
git revert a1b2c3d
git revert --no-edit a1b2c3d
```

`--no-edit` — default revert message без editor. Revert merge commit — указать, какой parent mainline:

```bash
git revert -m 1 <merge-commit-hash>
```

`-m 1` — «revert к state первого parent», обычно branch, в который merge'или (e.g. `main`). Без этого Git не choose automatically и abort с error. После successful revert на shared branch — normal push, fix без force.

`git reset` moves HEAD и optionally index и working tree. Три mode:

```bash
git reset --soft  <commit>
git reset --mixed <commit>   # default
git reset --hard  <commit>
```

Soft — только HEAD. Index still содержит то, что было staged, включая snapshot из «удалённого» commit. Working tree untouched. Ideal: commit too early, re-stage иначе или amend content перед recommit. Mixed — HEAD и index под target commit. Files на disk как были; changes из undone commit — unstaged modifications. Самый частый «undo last commit but keep work». Hard — все три: HEAD, index, working tree match target exactly. Всё не в tree того commit — discarded из index и overwritten на disk.

Mixed reset, undo last commit:

```
ДО:
  HEAD  → C3 (bad commit)
  Index → same as C3
  WT    → edited files

ПОСЛЕ git reset HEAD~1  (--mixed):
  HEAD  → C2
  Index → matches C2  (changes из C3 — unstaged diff)
  WT    → unchanged на disk
```

Recovery через reflog:

```bash
git reflog
# f4e8b21 HEAD@{0}: reset: moving to HEAD~1
# a1b2c3d HEAD@{1}: commit: Add feature

git reset --hard HEAD@{1}
```

## Examples

Scenario A: `git add .` и в staging попал `secrets.env`. Unstage один file, edits оставить.

```bash
git restore --staged secrets.env
```

File на disk с changes; в next commit не попадёт. Добавь в `.gitignore` перед следующим add.

Scenario B: правил `src/temp.cs`, experiment failed — выбросить edits.

```bash
git restore src/temp.cs
```

Working tree match index; если не staged — match last commit.

Scenario C: commit на private feature branch, никто не pull'нул, commit premature — wrong message, wrong files.

```bash
git reset --soft HEAD~1
```

Changes остаются staged как в removed commit. Fix staging, message, commit again. Если нужны changes unstaged для selective re-add:

```bash
git reset HEAD~1
```

Mixed default; `--mixed` optional.

Scenario D: bad commit уже на `origin/main`. Colleagues могли pull. Не reset и force push.

```bash
git pull
git revert badcommit_hash
git push origin main
```

Team получает forward-moving undo. CI на revert как на любой commit. History показывает mistake и correction — acceptable в большинстве audit environments.

Scenario E: `git reset --hard HEAD~3` и сразу понял — reset too far.

```bash
git reflog
git reset --hard HEAD@{1}
```

Entry до mistaken reset. Recovery часто thirty seconds, если act до GC.

Scenario F: revert merge, который принёс broken feature в main.

```bash
git log --oneline -5
git revert -m 1 abc1234
```

Conflicts resolve, push. Feature branch exists; undone только integration в main.

## Common Mistakes & Anti-patterns

`git reset --hard` на shared branch + force push — classic team-breaking mistake. Переписываешь history, на которой others built. Им нужны fetch и reset или rebase local work. На published commits — revert, unless team explicitly coordinates rewrite — и тогда revert default для main.

Путать `restore` и `reset` — failed expectations или accidental history moves. **Restore** — files в index и working tree; **reset** — HEAD. Unstage only — `git restore --staged` clearer intent, чем `git reset HEAD file`, хотя older form works.

Думать, что revert deletes bad commit — surprise в `git log`. Revert adds inverse; original stays. By design для shared safety и audit.

Revert merge без `-m` fails или wrong results — Git needs mainline parent. Usually `-m 1`.

Ignore reflog после hard reset — unnecessary panic. Commits rarely instantly gone. Reflog first.

`--hard`, когда хватило `--mixed` или `--soft` — destroys work, которое meant to keep. Pause: discard working tree files или только move branch pointer?

## Production & Real-World Notes

На unpushed local feature branches soft и mixed reset — everyday tools перед pull request. Squashing WIP часто через interactive rebase, но soft reset к branch point — similar staging control проще для simple cases.

На protected main в GitHub/GitLab force push blocked. Даже local reset — remote rejects rewrite. Revert не только etiquette — может быть единственный mechanism.

Secrets в commit — reset insufficient, если push был. Secret в remote history, forks, CI logs. Rotate credential, `git filter-repo` или BFG если policy requires, revert introducing commit forward. Local reset не unpublish secrets.

CI на push treats revert normally. После revert breaking merge — monitor pipeline на revert commit, confirm main green, notify team.

Emergency force push — когда rewrite unavoidable — broadcast communication. Teammates `git fetch`, `git reset --hard origin/branch` без local commits, или rebase local work на rewritten tip. High coordination cost — почему revert default.

Undo only last commit message на unpushed — `git commit --amend` проще reset — урок `0.1.10`. Amend когда tree correct, metadata wrong.

Перед любым `reset --hard` полезно мысленно задать вопрос: «Есть ли правки только на диске, которые я ещё не добавил в index?» Если да — `--hard` уничтожит их без следа в object database. В сомнительных случаях сначала `git stash push -u`, потом reset — stash станет страховкой, которую можно вернуть через pop.

## Comparison / Trade-offs

Restore vs reset vs revert — три разных вопроса. Restore: «files или staging match something без change history». Reset: «branch назад к previous commit, optionally files с собой». Revert: «new commit отменяет past commit, history preserved».

Soft reset preserves staged — recommit. Mixed preserves working tree, unstages — re-organize next commit. Hard preserves nothing beyond target snapshot — только когда genuinely discard all local divergence.

Rewrite через reset и rebase — clean linear narrative «as if mistakes never happened». Aesthetic и иногда required перед merge polished feature. Hides errors от casual log readers — pro для external repos, con для compliance с full audit. Forward через revert — mistake и fix visible; lower collaboration cost.

Neither universally superior. Middle chooses based on: published commits?, audit needs visible corrections?, teammates depend on current tip? В сомнении на shared branch — default revert.

## Quick Reference

Discard unstaged edits: `git restore <file>`. Unstage: `git restore --staged <file>`. Undo last commit, keep staged: `git reset --soft HEAD~1`. Undo last, keep unstaged: `git reset HEAD~1`. Undo last, discard all: `git reset --hard HEAD~1`. Undo pushed safely: `git revert <hash>`. Lost commits: `git reflog`.

## Key Takeaways

- Три дерева: HEAD (last commit), index (staging), working tree (disk).
- `git restore` — index и working tree без движения branch.
- `git reset --soft/mixed/hard` — HEAD и optionally index и working tree.
- `git revert` — inverse commit; safe для shared pushed branches.
- `git reflog` — local reference movements; recovery после mistaken reset.
- Shared/pushed history: revert, не reset без team coordination.
- `--hard` destroys uncommitted work, never staged в objects.
