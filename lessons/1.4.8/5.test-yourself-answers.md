## Answers

### 1. [Recall]
**Answer:** `true` — продолжить на захваченном `SynchronizationContext` (если был); `false` — не захватывать, resume на thread pool / inline.
**Why:** Это имя параметра `ConfigureAwait(continueOnCapturedContext)`.

### 2. [Recall]
**Answer:** Только на **SynchronizationContext** (scheduling continuation). **ExecutionContext** может flow отдельно.
**Why:** `ConfigureAwait` не отключает `AsyncLocal` и logical call context полностью.

### 3. [Understanding]
**Answer:** При completion awaiter вызывает зарегистрированный callback → `MoveNext` state machine. `ConfigureAwait` определяет, через `SyncContext.Post` или напрямую на pool.
**Why:** Связь с `1.4.6`–`1.4.7`: await = pause state machine; resume = следующий `MoveNext`.

### 4. [Understanding]
**Answer:** Library не знает caller context (UI, test SyncContext). `false` избегает лишнего marshaling и снижает deadlock при sync-over-async у caller.
**Why:** Defensive API design для любого host.

### 5. [Understanding]
**Answer:** ASP.NET Core **не устанавливает** request `SynchronizationContext`; capture «пустой» — нет лишнего `Post`.
**Why:** См. `1.4.9` — ключевое отличие от Framework.

### 6. [Application]
**Answer:** **Deadlock** (классический сценарий).
**Why:** Continuation post на UI thread, заблокированный в `GetResult()`.

### 7. [Application]
**Answer:** **Небезопасно** — вероятно не UI thread.
**Why:** Нужен default capture или явный `Dispatcher.Invoke`.

### 8. [Analysis]
**Answer:** **Неверно** — capture per await.
**Common mistake:** Думают, что context «отключается на метод».

### 9. [Analysis]
**Answer:** **Не рекомендуется** для public library — завтра вызовут из WPF/test.
**Why:** API contract шире текущего deployment.

### 10. [Understanding]
**Answer:** **`true`** (equivalent `ConfigureAwait(true)` implicit).
**Why:** Compiler default для `await`.

### 11. [Application]
**Answer:** UI code, где после await нужен тот же UI thread без ручного `Dispatcher`.
**Why:** Explicit `true` редко; чаще просто `await` без `ConfigureAwait(false)`.

### 12. [Analysis]
**Answer:** **Не всегда** — помогает при SyncContext deadlock; не при pool starvation или других блокировках.
**Why:** Правильный fix — async all the way (`1.4.14`).
