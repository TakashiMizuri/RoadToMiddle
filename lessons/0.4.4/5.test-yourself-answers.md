## Answers

### 1. [Recall]
**Answer:** Примеры: no connection setup, no reliability/retransmit, no ordering guarantee, datagram (not stream) semantics — любые три.
**Why:** Core UDP vs TCP differences.

### 2. [Recall]
**Answer:** **53** (TCP тоже для large/truncated responses).
**Why:** IANA convention.

### 3. [Understanding]
**Answer:** Late retransmitted frame useless/worse than skip; TCP delay grows buffer → stall; UDP allows drop and continue timeline.
**Why:** Time-sensitive media trade-off.

### 4. [Understanding]
**Answer:** UDP preserves message boundaries (one send = one datagram unit); TCP merges bytes — app must frame (HTTP Content-Length etc.).
**Why:** Stream vs message model.

### 5. [Understanding]
**Answer:** UDP fast for small query/response one RTT; if response too large or **TC truncated**, client retries **TCP** for full answer.
**Why:** Hybrid DNS design — preview 0.4.5.

### 6. [Application]
**Answer:** a) TCP b) UDP c) UDP (often) d) TCP — correctness and stream APIs.
**Why:** Reliability requirement matrix.

### 7. [Application]
**Answer:** HTTP/3 uses **QUIC over UDP 443**; blocked UDP → no QUIC → fallback TCP HTTP/2 (`0.3.13`).
**Why:** Transport path selection.

### 8. [Analysis]
**Answer:** **C** неверно — UDP не guarantees order between datagrams.
**Why:** A, B, D correct.

### 9. [Analysis]
**Answer:** Примеры: **DNS (53)**, **NTP**, **DHCP**, **QUIC/HTTP/3 (443)**, **VoIP/WebRTC**.
**Why:** Legitimate UDP workloads.

### 10. [Understanding]
**Answer:** **Transport layer** (with TCP); below — **IP (internet)**; above — application (DNS, RTP, QUIC).
**Why:** Same layer as TCP, different semantics.

### 11. [Recall]
**Answer:** Send datagram without waiting ACK or establishing connection; no built-in retry if lost.
**Why:** Connectionless best-effort.

### 12. [Application]
**Answer:** **Yes**, order not guaranteed. App adds sequence numbers and reorder/discard logic if order matters.
**Why:** UDP does not serialize datagrams.
