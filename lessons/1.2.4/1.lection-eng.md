# Access Modifiers — public, private, protected, internal

> Roadmap: `1.2.4` · Node: `1.2` — C#: OOP · Depth: **understanding**

## Learning Objectives

After this lesson you will be able to:

- Apply **`public`**, **`private`**, **`protected`**, and **`internal`** to classes and members.
- Combine modifiers like **`protected internal`** and **`private protected`** at a basic level.
- Decide **minimum necessary visibility** for types and members.
- Relate access modifiers to **assembly boundaries** and **`InternalsVisibleTo`** (concept).
- Understand **`private`** as the default for class members and **`internal`** as default top-level type visibility.
- Preview how **`protected`** supports inheritance (`1.2.6`).

---

## Why This Matters

Encapsulation (`1.2.5`) is not only "use properties" — it is **who may touch what**. Access modifiers are the compiler-enforced fence. Public surface is a **contract**; everything else is implementation detail you may refactor freely.

Libraries expose small public APIs and keep helpers `internal`. Domain classes keep fields `private` and expose behavior through methods. Getting visibility wrong leaks invariants (public mutable state) or blocks valid extension (`everything private` in a base class meant for inheritance).

Middle developers read **`internal`** in framework code and know it means "same assembly (plus friends)," not "secret from runtime."

## Core Concepts

### The Visibility Spectrum

Access modifiers answer: **from where can this member be accessed?**

| Modifier | Same class | Derived class | Same assembly | Other assembly |
|----------|------------|---------------|---------------|----------------|
| `private` | Yes | No | No | No |
| `protected` | Yes | Yes | No* | No* |
| `internal` | Yes | No** | Yes | No |
| `protected internal` | Yes | Yes | Yes | Derived only*** |
| `private protected` | Yes | Yes (same assembly) | Yes | No |

\*Derived access works across assemblies if type is subclass.  
\*\*Unless also derived.  
\*\*\*`protected OR internal` — union, not intersection.

Default rules: class members are **`private`** if unspecified. Top-level types (not nested) are **`internal`** by default — visible within the **assembly** (`.dll`/`.exe`), not outside unless `public`.

### `public` — Universal Contract

**`public`** members and types are callable from **any code** referencing the assembly (if type is public). This is your **API surface** — breaking changes hurt consumers.

```csharp
public class ApiClient
{
    public Task<User> GetUserAsync(int id) { ... }
}
```

Expose **`public`** only what callers need. Implementation helpers should not be public.

### `private` — Implementation Detail

**`private`** restricts to the **declaring class** only — not even derived classes in another file:

```csharp
public class BankAccount
{
    private decimal _ledgerBalance;

    public void Deposit(decimal amount)
    {
        _ledgerBalance += amount;
    }
}
```

This is the default for fields and the right choice for storage (`1.2.2`). **`private`** methods extract helpers without expanding API.

### `protected` — Inheritance Hook

**`protected`** visible to the class and **derived classes**:

```csharp
public class Entity
{
    protected void MarkUpdated() => LastModifiedUtc = DateTime.UtcNow;
    public DateTime LastModifiedUtc { get; private set; }
}
```

Subclasses call `MarkUpdated()`; external callers cannot. **`protected`** is for **extension points**, not general API — overuse couples subclasses to base internals (`1.2.11` fragile base class).

### `internal` — Assembly-Local

**`internal`** types and members visible **inside the same assembly**:

```csharp
internal class SqlMigrationRunner { ... }

public class OrderService
{
    internal void RebuildCacheForTests() { ... }
}
```

Common for **implementation types** not meant for NuGet consumers. Unit test projects sometimes use **`InternalsVisibleTo`** attribute in `AssemblyInfo` to access `internal` members — production callers still blocked.

### Combining Modifiers

**`protected internal`**: accessible if **`protected` OR `internal`** — wider reach.

**`private protected`**: accessible only in **derived classes within the same assembly** — narrowest inheritance band.

Use combined forms rarely; prefer clear `private` + small public API.

### Classes and Nested Types

A **`public class`** can contain **`private nested`** helper types — pattern for builder implementation details:

```csharp
public class QueryBuilder
{
    private class Clause { ... }
}
```

Nested type visibility is relative to containing type.

### Files and Partial Classes

`internal` top-level class in one file is same assembly visibility as in another — **assembly** is the boundary, not folder (`1.2.18` partial classes share one type across files).

## Examples

**Example 1 — minimum public surface:**

```csharp
public class Invoice
{
    public decimal Total { get; private set; }
    private readonly List<Line> _lines = new();

    public void AddLine(string sku, decimal price) { ... }
}
```

Callers see `Total` and `AddLine`; line storage is private.

**Example 2 — internal repository implementation:**

```csharp
internal sealed class EfOrderRepository : IOrderRepository
{
    public async Task<Order?> FindAsync(Guid id) { ... }
}
```

Public interface `IOrderRepository` in domain; EF details stay assembly-local.

**Example 3 — protected for template method pattern (preview):**

```csharp
public abstract class Parser
{
    public Document Parse(string input)
    {
        var tokens = Tokenize(input);   // protected hook
        return Build(tokens);
    }
    protected abstract IReadOnlyList<Token> Tokenize(string input);
}
```

Derived parsers override `protected` hook; public `Parse` stays stable.

## Common Mistakes & Anti-patterns

**Everything `public`** — no encapsulation, wide coupling.

**`protected` fields** — subclasses directly mutate base state; prefer protected methods or controlled properties.

**Confusing `internal` with security** — it is compile-time visibility, not runtime protection (reflection can still reach).

**Testing only through public API when `InternalsVisibleTo` would isolate tests** — trade-off between black-box tests and testability.

## Production & Real-World Notes

NuGet packages: **`public`** API is semver-contract; **`internal`** helpers refactor freely.

ASP.NET Core projects: many app classes are `public` because framework reflects/controllers bind — still minimize public mutable state.

Friend assemblies for tests: `[assembly: InternalsVisibleTo("MyApp.Tests")]`.

---

## Comparison / Trade-offs

| Modifier | Use when |
|----------|----------|
| `private` | Default storage and helpers |
| `protected` | Controlled extension for inheritance |
| `internal` | Assembly implementation details |
| `public` | Stable consumer-facing contract |

Prefer **`private` + public methods** over **`protected` fields`**.

---

## Quick Reference

```
private          → this class only (default member)
protected        → this class + derived
internal         → same assembly (default top-level type)
public           → everywhere with reference
protected internal → protected OR internal
private protected  → derived in same assembly only
```

---

## Key Takeaways

- Access modifiers enforce **who can call what** at compile time.
- Default member visibility is **`private`**; default type is **`internal`**.
- **`public`** is a contract — keep it small.
- **`internal`** hides assembly implementation; not a security boundary.
- **`protected`** supports inheritance hooks — use deliberately.
- Minimum visibility reduces coupling and enables refactoring.
