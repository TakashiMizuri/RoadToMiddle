# Remote: origin, fetch, pull, push

> Roadmap: `0.2.6` · Node: `0.2` — Git: ветки и collaboration · Depth: практика

## Learning Objectives

После этой лекции ты сможешь:

- Определить **remote** как именованную ссылку на другой Git-репозиторий.
- Объяснить, почему **`origin`** — общепринятое имя remote по умолчанию.
- Различить **`git fetch`**, **`git pull`** и **`git push`** — что передаётся по сети и что меняется локально.
- Описать **remote-tracking branches** (`origin/main`) и их связь с локальными ветками.
- Проследить путь от локальных коммитов на feature branch до shared remote.
- Выбрать правильную команду, когда нужно **увидеть** удалённые изменения без слияния.

## Why This Matters

В узле `0.2` ты узнал, что ветки — это подвижные указатели на коммиты, что merge и rebase меняют связи между ними, и что конфликты возникают при расхождении историй. Всё это происходило **внутри одного репозитория** — папки `.git` на твоём ноутбуке. Для solo-экспериментов этого достаточно, но профессиональная fullstack-разработка — collaborative. ASP.NET Core API живёт на GitHub или GitLab; ветка коллеги с React — на том же сервере; CI собирает то, что есть на remote, а не то, что случайно осталось unstaged на твоей машине.

**Remote** — абстракция Git для «другой копии проекта». Это не магическое облако, а **URL плюс набор refs**, о которых твой локальный Git знает. При `git clone` Git создаёт remote с именем **`origin`**, указывающий на сервер, с которого клонировали. С этого момента большая часть collaboration vocabulary — «запушь ветку», «подтяни main», «сделай fetch перед rebase» — предполагает понимание трёх глаголов: **fetch**, **pull**, **push**.

Ошибки в этих командах приводят к реальным инцидентам. Разработчик, который всегда делает `git pull`, не читая вывод merge, может случайно создать merge commit, которого не хотел. Тот, кто `git push` без fetch, получает «non-fast-forward» — сообщение, осмысленное только если понимаешь, что remotes развиваются независимо. Тот, кто путает fetch с pull, может влить remote work, который никогда не просматривал. Middle-уровень — знать **что едет по сети** и **что меняется на диске** для каждой команды.

## Core Concepts

### Что такое Remote?

**Remote** — **прозвище другого Git-репозитория**, обычно на сетевом сервере. Локальный repo хранит URL remote и периодически синхронизирует знание о его ветках и тегах. Самое частое имя — **`origin`**, создаётся автоматически при `git clone`. Можно добавить другие remotes — например `upstream` при fork open-source проекта — но один `origin` покрывает типичный team workflow.

```bash
git remote -v
# origin  https://github.com/team/api.git (fetch)
# origin  https://github.com/team/api.git (push)
```

Важно: **`origin` — это не «сам GitHub»**, а **настроенный peer твоего проекта** на этом хосте. Удали конфигурацию remote — локальная history останется; пропадёт только закладка для связи с сервером.

### Локальные ветки vs Remote-Tracking Branches

В `0.2.1` ты узнал, что **branch** — указатель на commit. Локальные ветки вроде `feature/login` живут в `refs/heads/`. При **fetch** Git обновляет **remote-tracking branches** в `refs/remotes/` — например `origin/main`. Это **read-only снимки** того, где ветки были на remote при последнем успешном контакте. На `origin/main` не коммитят напрямую.

```
Локальный repo после fetch:

  refs/heads/main          → твой локальный main (можешь быть ahead/behind)
  refs/heads/feature/auth  → твоя feature work

  refs/remotes/origin/main         → где был main на origin при last fetch
  refs/remotes/origin/feature/auth → где была feature/auth на origin
```

Отсюда сообщения `git status`: **«Your branch is behind 'origin/main' by 3 commits»**. Git сравнивает локальный `main` с **кэшированной картинкой** remote `main`, а не с live network call каждый раз. Пока не сделаешь fetch, картинка устаревает.

### git fetch — скачать без интеграции

**`git fetch`** связывается с remote, скачивает недостающие commits и objects, **обновляет remote-tracking branches**. Он **не** меняет локальные ветки, working tree и `HEAD`. Fetch — **обновление карты** удалённого мира, оставаясь на месте.

```bash
git fetch origin
# Обновляет origin/main, origin/develop и т.д.
# Локальный main и файлы: без изменений
```

После fetch можно изучить:

```bash
git log main..origin/main --oneline   # коммиты на remote, которых нет локально
git log origin/main..main --oneline   # коммиты у тебя, которых нет на remote
git diff main origin/main             # diff на уровне файлов
```

Fetch — безопасный первый шаг перед merge, rebase или осторожным push. Команды на busy `main` часто делают **`git fetch` несколько раз в день** и интегрируют осознанно — привычка, которая окупается перед `0.2.7` upstream tracking и `pull --rebase`.

### git pull — fetch плюс интеграция

**`git pull`** — **составная команда**: **`git fetch`** плюс интеграция в **текущую ветку**. По умолчанию интеграция — **merge**, который может создать merge commit при расхождении. Pull **может** менять рабочую ветку и working tree — в отличие от одного fetch.

```bash
git checkout main
git pull origin main
# Эквивалент:
# git fetch origin main
# git merge origin/main
```

Ты уже знаешь из `0.2.2`, что может произойти: fast-forward если просто отстаёшь, merge commit если обе стороны добавили коммиты, или **конфликты** — решаются как в `0.2.5`. Pull удобен, но **непрозрачен**, если не знаешь, какая половина выполнилась; многие senior'ы предпочитают **`git fetch` + явный `merge` или `rebase`**. Rebase при pull — тема `0.2.7`.

### git push — опубликовать коммиты

**`git push`** загружает локальные коммиты на remote и просит сервер **сдвинуть указатель ветки**. Обычно пушат **локальную ветку** в **ветку на origin**:

```bash
git push origin feature/login
```

Remote принимает push, когда твои коммиты — **fast-forward** относительно того, что уже есть. Если кто-то запушил в ту же ветку после твоего last fetch, push **отклоняется** как non-fast-forward. Это защита shared history, не баг. Исправление: fetch, integrate (merge или rebase), затем push — никогда слепой `push --force` на shared branches.

Push **создаёт** remote branch, если её ещё нет (если policy сервера разрешает). После первого push другие разработчики увидят `origin/feature/login` после fetch.

### Конвенция origin

**`origin`** — не техническое требование Git, а **сильная конвенция** от `git clone`. Инструменты, туториалы и CI предполагают это имя. Переименовать можно (`git remote rename`), но это путает. При fork репозитория `origin` твоего clone — **твой fork**; исходный repo часто добавляют как **`upstream`** — паттерн при contribution в библиотеки, от которых зависит ASP.NET или React app.

## Under the Hood

При fetch или push Git использует **wire protocol** (HTTPS или SSH), чтобы договориться, что есть у другой стороны. Клиент отправляет известные **refs**; сервер отвечает **packfile** объектов, которых клиенту не хватает. Objects **content-addressed** (из `0.1.5`), дубликаты blobs между ветками передаются один раз. Fetch сохраняет объекты в `.git/objects` и пишет ref-файлы remote-tracking, например `.git/refs/remotes/origin/main`.

Push отправляет **объекты, которых нет на remote**, и запрос обновить ref — например «установить `refs/heads/feature/login` в SHA `abc123`». Сервер проверяет права и **non-fast-forward rules**. Hosting platforms добавляют branch protection: `main` может отклонять direct push и требовать pull requests (`0.2.8`).

`git pull` с merge вызывает ту же merge machinery, что `git merge origin/main` после fetch — включая conflict markers в working tree. `git push` не отправляет working tree и staging area — только **committed** history, достижимую от tip pushed branch.

## Syntax / Commands / API

```bash
# Remotes
git remote -v
git remote show origin
git remote add upstream https://github.com/org/original.git

# Fetch (безопасно — локальные ветки не двигаются)
git fetch                    # все remotes
git fetch origin             # один remote
git fetch origin main        # одна ветка

# Pull (fetch + интеграция в текущую ветку)
git pull                     # default remote + upstream branch если настроен
git pull origin main         # явно
git pull --ff-only           # fail вместо merge commit если не fast-forward

# Push
git push origin feature/x    # опубликовать ветку
git push -u origin feature/x # push + upstream (см. 0.2.7)
git push origin --delete old-branch

# Inspect без интеграции
git log HEAD..origin/main --oneline
git diff origin/main...HEAD   # изменения на твоей ветке с момента расхождения
```

## Examples

### Пример 1: Ежедневная синхронизация на main

Начинаешь день на `main`:

```bash
git checkout main
git fetch origin
git log --oneline main..origin/main    # что нового на сервере?
# Если есть коммиты:
git merge origin/main    # или: git pull origin main
```

Fetch-first позволяет **прочитать** входящие коммиты до merge. Если CI упал на remote `main`, увидишь это в log messages до изменения working tree.

### Пример 2: Опубликовать feature branch

После коммитов на локальной `feature/health`:

```bash
git push -u origin feature/health
```

На remote появилась ветка; teammates делают `git fetch` и видят `origin/feature/health`. Открываешь PR (`0.2.8`) с этой ветки. До push feature существовала только на твоей машине.

### Пример 3: Отклонение non-fast-forward

Ты и коллега оба коммитили в `feature/shared`:

```bash
git push origin feature/shared
# ! [rejected] ... non-fast-forward
```

Локальная картинка устарела. Восстановление:

```bash
git fetch origin
git log --oneline HEAD..origin/feature/shared   # коммиты коллеги
git merge origin/feature/shared                 # или rebase — trade-offs 0.2.4
# resolve conflicts если есть — 0.2.5
git push origin feature/shared
```

### Пример 4: fetch без паники

Перед рискованным rebase на `main`:

```bash
git fetch origin
git rebase origin/main
```

Если rebase пошёл не так, старый tip остаётся в reflog; rewritten history ещё не запушена.

## Common Mistakes & Anti-patterns

**Считать `git pull` безобидным sync** — pull может merge'ить и конфликтовать; всегда знай текущую ветку. Pull на `main`, когда хотел обновить feature branch — классическая ошибка.

**Никогда не делать fetch** — работаешь со stale `origin/*` refs; status «врёт» мягко, пока push не упадёт.

**Push без pull на shared branches** — циклы rejection или, хуже, ущерб от force-push.

**Путать remote-tracking branch с локальной** — checkout `origin/main` даёт **detached HEAD**; используй `git switch main` и merge/rebase от `origin/main`.

**Думать, что push загружает uncommitted work** — едут только commits; dirty tree остаётся локально.

**Использовать `origin` вместо `upstream`** — PR'ы могут уйти не в тот репозиторий после fork.

## Production & Real-World Notes

CI/CD триггерится на **remote** events — push в `main`, открытие pull request. Незапушенные коммиты pipeline не запускают. Branch protection на `main`/`develop` блокирует direct push; пушат feature branches и мержат через PR.

Крупные команды используют **fork workflow** (open source) или **shared repository** (большинство компаний). SSH keys или tokens аутентифицируют push; HTTPS с credential manager так же распространён на Windows.

**Partial clone** и **sparse checkout** оптимизируют огромные monorepo — семантика remotes та же. В **air-gapped** средах remotes на внутренних Git-серверах с идентичным fetch/push.

Следи за **отставанием**: `git fetch && git status` на long-lived branches предотвращает monster merges к релизу.

## Comparison / Trade-offs

| Команда | Сеть | Обновляет origin/* | Меняет локальную ветку / working tree |
|---------|------|--------------------|---------------------------------------|
| `fetch` | Да | Да | Нет |
| `pull` | Да | Да | Да (merge/rebase) |
| `push` | Да | Нет (обновляется сервер) | Нет |

**Fetch + ручной merge/rebase** vs **pull**: явный контроль vs one-liner. **HTTPS** vs **SSH**: credential prompts vs управление ключами — оба нормальны при consistent setup.

## Quick Reference

```
REMOTE        = прозвище (обычно origin) → URL другого repo
FETCH         = скачать objects + обновить origin/* ; локальная ветка не меняется
PULL          = fetch + merge в текущую ветку (по умолчанию)
PUSH          = загрузить commits; сдвинуть remote branch pointer

origin/main   = remote-tracking branch (last known remote state)
Local main    = твоя ветка; может расходиться с origin/main

Перед push на shared branch: fetch → integrate → push
Rejected push: fetch → merge/rebase → push (не --force на shared)
```

## Key Takeaways

1. **Remote** — именованный peer repository; **`origin`** — default от `clone`.
2. **Remote-tracking branches** (`origin/main`) кэшируют remote state после fetch.
3. **`git fetch`** скачивает history без merge в текущую ветку.
4. **`git pull`** = fetch + integrate; может merge'ить и конфликтовать.
5. **`git push`** публикует commits; сервер отклоняет non-fast-forward при отставании.
6. **Fetch before integrate** — безопасная professional habit.
7. Uncommitted work не едет через push — только commits.
