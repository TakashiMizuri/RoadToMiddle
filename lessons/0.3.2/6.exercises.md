# Exercises: URL Anatomy

> Complete after lectures and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Label the parts
**Goal:** Parse URLs mentally.

Label scheme, host, port (if explicit), path, query, fragment for:

1. `http://localhost:5000/health`
2. `https://api.example.com/v1/users?page=1#profile`
3. `/static/logo.png` (relative — what base do you need?)

**Success criteria:** Three correct decompositions; #3 notes need for base URL.

---

### E2. What hits the server?
**Goal:** Remember fragment exclusion.

For URL `https://httpbin.org/get?a=1#ignore`, write exact path+query the server receives in the request target (not full URL).

**Success criteria:** `/get?a=1` only; no `#ignore`.

---

## Medium

### M1. Build a query string
**Goal:** Correct encoding.

Build a URL path + query for search term `C# & .NET`, param name `q`, base `https://search.test/search`. Use JavaScript or C# UriBuilder.

**Success criteria:** Proper percent-encoding; no raw `&` or space in value.

---

### M2. Relative URL trap
**Goal:** Fix a real bug.

Page URL: `https://app.com/app/dashboard`. Broken code: `fetch('api/notifications')`. Expected API: `https://app.com/api/notifications`.

Give two fixes (different valid approaches).

**Success criteria:** Leading slash `/api/...` or absolute URL or configured axios baseURL.

---

## Hard (stretch)

### H1. Design discussion
**Goal:** Path vs query trade-offs.

Design URLs for: list orders (filters: status, date range), get single order, get order line items.

Write proposed paths + query usage in 5–8 sentences defending choices for a REST API `v1`.

**Success criteria:** Resource nouns in path; filters in query; single order by path id; no verbs in URL.
