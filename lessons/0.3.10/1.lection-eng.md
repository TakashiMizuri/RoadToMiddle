# REST Principles: Resources, Stateless, Uniform Interface

> Roadmap: `0.3.10` · Node: `0.3` — HTTP and the Web · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Explain **REST** as architectural style using HTTP, not a synonym for "JSON API."
- Identify **resources** as nouns addressed by URIs (`0.3.2`) and manipulated with HTTP methods (`0.3.4`).
- Describe **statelessness**: each request carries all context; server does not rely on client session memory between requests (except stored resource state).
- Apply the **uniform interface** constraints: identification of resources, manipulation through representations, self-descriptive messages, HATEOAS awareness.
- Relate REST constraints to status codes (`0.3.6`) and headers (`0.3.7`) you already studied.
- Recognize when an API is **RESTful** vs **REST-inspired** — and why both are common in industry.

---

## Why This Matters

Teams label every ASP.NET Core Web API "REST" because it returns JSON over HTTP. That shorthand hides a design philosophy from Roy Fielding's dissertation: use HTTP's strengths instead of tunneling custom RPC over POST. When you model endpoints as **resources** with meaningful methods, **cache** (`Cache-Control`, ETag from `0.3.7`) and **proxies** work with you. When you expose `/api/DoCheckout` as POST-only verbs, you re-invent semantics HTTP already provides — and lose uniform tooling.

You do not need fanatical HATEOAS links in every response to benefit from REST thinking. Middle developers still must decide: nouns vs verbs in URLs, idempotent updates (`0.3.5`), correct status codes, stateless JWT vs session cookie (`0.3.9`), and pagination patterns. This lesson gives the **conceptual frame** for API design in later phases (Problem Details, versioning, OpenAPI) without treating REST as religion.

## Core Concepts

### REST Is an Architectural Style, Not a Framework

**REST (Representational State Transfer)** defines **constraints** for scalable web systems. It is not a library, not "CRUD in ASP.NET," not mandatory hypermedia. HTTP is the primary protocol REST was derived from — but REST is **how you use** HTTP, not HTTP itself.

Richardson Maturity Model (Leonard Richardson) levels — useful shorthand:

| Level | Name | Characteristics |
|-------|------|-----------------|
| 0 | Swamp of POX | Single URI, POST everything |
| 1 | Resources | Multiple URIs / resources |
| 2 | HTTP Verbs | Methods + status codes |
| 3 | Hypermedia | HATEOAS links drive client |

Most production APIs sit at **level 2**: resources + verbs + status codes; hypermedia optional.

### Resources and URIs

A **resource** is anything identifiable you expose: user, order, product catalog, search result collection. Resources have **representations** — JSON, XML, HTML — transferred in HTTP bodies with `Content-Type` (`0.3.7`).

URIs **identify** resources (`0.3.2`):

```
GET /users/42
GET /orders?status=open
POST /users          → creates new user resource
```

**Nouns in paths, verbs in methods** — good default:

- ✅ `DELETE /orders/991`
- ❌ `POST /deleteOrder?id=991` (RPC smell)

Collections vs items:

```
GET /products        → collection
GET /products/7      → single item
POST /products       → add to collection
```

Nested resources sparingly — `/users/42/orders` OK when orders cannot exist without user context; avoid deep nesting `/a/1/b/2/c/3/d/4`.

Representations carry **state** of the resource at a point in time. Updating representation updates resource (via PUT/PATCH semantics from `0.3.4`).

### Stateless Interaction

**Stateless** means the server **does not store client conversation context** between requests — each request must contain everything needed to process it: auth (`Authorization` or `Cookie`), target URI, method, body, conditional headers.

Server **may** store **resource state** in database — that is not violation. Violation is: "remember that user clicked step 2 of wizard in server RAM keyed only by socket" without id in request.

Why statelessness scales: any app server can handle any request; load balancers need no sticky sessions for logic (cookies/JWT still carry identity **in the request itself**). Failure recovery simpler — no orphaned server-side wizard state.

Cookies (`0.3.9`) do not break statelessness if session id in cookie **is part of each request** and server looks up state from store — debate exists, but practical REST APIs treat this as stateless enough. **Server-side session store** vs **JWT self-contained** is engineering trade-off, not REST purity contest.

### Uniform Interface

Fielding's four uniform interface constraints (simplified for daily work):

**Identification of resources** — URIs name resources unambiguously.

**Manipulation through representations** — clients exchange representations (JSON DTOs); server maps to internal models. Client does not reach into server memory.

**Self-descriptive messages** — each message includes enough metadata to understand it: `Content-Type`, `Content-Length`, cache headers, status code meaning. Error format documented (Problem Details later).

**Hypermedia as the engine of application state (HATEOAS)** — responses include links (`rel`, `href`) telling client available transitions: `"cancel": { "href": "/orders/991/cancel", "method": "POST" }`. Clients discover actions without hard-coded URL lists. Full HATEOAS rare in mobile/SPA era; **partial links** (pagination `next`, `Location` on 201) are common.

Uniform interface enables ** intermediaries**: caches, gateways, CDNs understand GET safe/idempotent (`0.3.5`) and cache 200+ETag responses.

### Connecting to HTTP Machinery You Know

REST principles map directly to prior lessons:

- **Methods** (`0.3.4`) — uniform verbs on resources.
- **Safe/idempotent** (`0.3.5`) — GET/PUT/DELETE enable caching and retries.
- **Status codes** (`0.3.6`) — 201 Created, 204 No Content, 409 Conflict express resource outcomes.
- **Headers** (`0.3.7`) — Content negotiation, auth, caching.
- **HTTPS** (`0.3.8`) — transport security for representations in transit.
- **Cookies** (`0.3.9`) — one way to carry auth token each request while appearing "session-like."

REST is the **glue narrative** across node `0.3`.

### RESTful vs Practical APIs

Real teams compromise:

- **RPC endpoints** for complex actions: `POST /orders/991/cancel` — not pure REST (verb in path) but clear.
- **Query endpoints** for reports: `GET /reports/monthly` — collection of computed resource.
- **Batch/bulk** operations — awkward in strict REST; often pragmatic POST.
- **GraphQL/gRPC** — different styles for other trade-offs.

Middle developer judgment: apply REST constraints where they reduce coupling and leverage HTTP; break them consciously with documented reason.

## Under the Hood

### Why Statelessness Enables Scale

Consider sticky sessions storing cart only in server memory. Scale to ten servers — user hits server B, cart empty. Stateless fix: cart **resource** at `/carts/{id}` or client sends cart id each request; any server reads from Redis/DB.

Load balancer **round-robin** works when `Authorization: Bearer` or `Cookie: session=xyz` on **every** request re-establishes identity and authorization without affinity.

Affinity (sticky sessions) still used for WebSockets and some legacy — not required for REST JSON CRUD.

### Cacheability as a First-Class Benefit

GET on representation with `Cache-Control` and ETag — intermediary cache serves **304** (`0.3.6`). RPC POST tunnel `POST /api { "action": "getProduct", "id": 7 }` — not cacheable by standard HTTP semantics. Resource-oriented GET `/products/7` participates in web infrastructure.

## Syntax / Commands / API

### Resource-oriented route sketch (ASP.NET Core)

```csharp
app.MapGet("/books/{id}", GetBook);
app.MapPost("/books", CreateBook);
app.MapPut("/books/{id}", ReplaceBook);
app.MapPatch("/books/{id}", PatchBook);
app.MapDelete("/books/{id}", DeleteBook);
```

### Light HATEOAS (pagination)

```json
{
  "items": [ ... ],
  "links": {
    "next": "/books?page=3",
    "prev": "/books?page=1"
  }
}
```

## Examples

### Example 1: Order as resource

`GET /orders/991` returns representation. `PATCH` with `{ "status": "shipped" }` transitions state — **409** if invalid transition (`0.3.6`). Client sent JWT each time — stateless auth.

### Example 2: Violating statelessness

Server stores "current checkout step" in static dictionary keyed by IP only — second request from different pod fails. Fix: include `checkoutId` in URI or body.

### Example 3: Level 2 vs Level 0

Level 0: all calls `POST /api` with `{ "method": "GetUser", "id": 42 }`. Level 2: `GET /users/42` with **200** and cache headers for public profile data.

## Common Mistakes & Anti-patterns

**Verbs in URL** — `/getUser`, `/createOrder` duplicates HTTP method semantics.

**Ignoring idempotency** — retry POST create duplicates orders; use idempotency keys or PUT to known URI.

**200 for all outcomes** — breaks uniform interface self-descriptive messages via status.

**Confusing REST with "no server state ever"** — databases store resource state; stateless means **no per-client session in server memory without request identifier**.

**Mandatory HATEOAS dogma** — teams skip links; document OpenAPI instead — acceptable at level 2.

## Production & Real-World Notes

OpenAPI describes resources and schemas — de facto contract instead of HATEOAS. Versioning via URL prefix `/v1/users` or header — consistency matters. Public APIs document error model and pagination cursors.

ASP.NET Core controllers can be resource-oriented (`UsersController` with route `users/{id}`) without being perfectly RESTful.

## Comparison / Trade-offs

| Style | Strength | Weakness |
|-------|----------|----------|
| REST level 2 | HTTP tooling, caches, clear semantics | Awkward for complex actions |
| RPC-style JSON POST | Flexible actions | Loses cache, verb semantics |
| HATEOAS level 3 | Evolvable server-driven client | Heavy for SPAs with fixed screens |
| GraphQL | Client-shaped queries | Different caching, complexity |

| Constraint | Practical meaning |
|------------|-------------------|
| Resources | Nouns + URIs |
| Representations | JSON/HTML bodies |
| Stateless | Full context in each request |
| Uniform interface | Standard methods, status, headers |
| HATEOAS | Links for transitions (optional) |

## Key Takeaways

- REST = architectural **constraints**, not a brand name for Web API.
- Model **resources** with URIs; use HTTP **methods** and **status codes** intentionally.
- **Stateless** requests enable scale; session cookies/JWT are OK if identity travels each request.
- **Uniform interface** leverages HTTP infrastructure (caches, proxies).
- Level **2** APIs dominate industry; know HATEOAS but do not fetishize it.
- Prior `0.3.x` lessons are the mechanical foundation REST builds on.
