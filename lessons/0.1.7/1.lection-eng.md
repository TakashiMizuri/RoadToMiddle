# git show and git blame — Forensics

> Roadmap: `0.1.7` · Node: `0.1` — Git Basics · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Inspect commits, tags, and blobs with **`git show`**.
- Use **`git blame`** to find who last modified each line of a file.
- Apply **`-L`**, **`-C`**, and **`-M`** flags for precise forensic analysis.
- Run a practical **bug-introduced workflow** from symptom to responsible commit.
- Use **`git log -S`** (pickaxe) to find commits that added or removed a string.

---

## Why This Matters

Picture a production incident at two in the morning. Login stopped working after yesterday's deploy, and the stack trace points to line 847 in `AuthService.cs`. Someone asks the question every on-call engineer dreads: which commit introduced the bad null check? Without Git forensics, you are left guessing from memory, scrolling through merged pull requests, or asking in Slack and hoping someone remembers. With forensics, you can answer in minutes — often without leaving the terminal.

The tools in this lesson — `git show`, `git blame`, and pickaxe search via `git log -S` — are read-only queries against Git's history database. They do not rewrite history, they do not change your working tree, and they do not require network access once you have the repository locally. That makes them safe to run during an incident, during code review, or while debugging a regression on a colleague's branch.

It is worth stating explicitly what forensics is *not*. Blame is not a weapon for punishment culture. The name is unfortunate — in professional teams, `git blame` means "who last touched this line so I can ask them about intent," not "who do we fire." A line blamed to Alice might have been correct when she wrote it; the bug might come from a later merge, a changed API contract, or data that no longer matches assumptions. Forensics gives you a starting point for investigation, not a verdict.

## Core Concepts

Every commit in Git is a snapshot of your project at a moment in time, linked backward to its parent or parents. Under the hood, that snapshot is not stored as a full copy of every file. Instead, Git stores a **tree** object describing the directory structure, **blob** objects holding file contents addressed by hash, and metadata: author, committer, timestamps, and message. Parent pointers form the directed acyclic graph you know as history.

When you run forensic commands, you are querying this object database. `git show` answers "what is this object, and how does it differ from its parent?" `git blame` answers "for each line in this file at this revision, which commit last introduced that exact line content?" Both commands compute their answers at query time from stored diffs and objects — Git does not maintain a separate per-line audit log inside each commit.

Think of Git as a forensic database where every write is immutable. Once a commit exists, its contents do not change. New commits add new facts. That immutability is what makes historical investigation trustworthy: you are not reading a log file that someone might have edited; you are traversing a content-addressed graph where tampering would change hashes and break the chain.

The distinction between annotated and raw views matters in practice. `git show` on a commit gives you metadata plus a **patch** — the diff against the first parent. That is the human-readable story of what changed. `git blame` gives you a per-line map from content back to commits. `git log -S` searches across history for commits where the *count* of a string in the tree changed — a different question entirely, but one that pairs naturally with blame when you need to find when an identifier first appeared.

## Under the Hood

Understanding how `git blame` works helps you interpret its output correctly and know when to reach for extra flags. Git does not store authorship per line inside commit objects. When you run blame, Git starts from the file content at the requested revision (usually `HEAD`) and walks history backward. For each line, it finds the commit that last introduced that line's content by comparing blob diffs across revisions. The algorithm is related to `git annotate` and uses caching for performance on large histories, but conceptually it is reverse engineering from snapshots.

When you pass `-C` or `-M`, Git enables copy and move detection heuristics. Plain blame only looks at the history of the current file path. If someone moved a function from `OldService.cs` to `NewService.cs` in a refactor, plain blame on the new file might attribute lines to the refactor commit — the person who cut and pasted — rather than the original author. The `-C` flag looks for lines copied from other files in the same commit; `-C -C` searches more broadly across history. The `-M` flag detects lines moved or re-indented within a file or between files. These heuristics are not perfect, but they dramatically improve attribution after large refactors.

For `git show`, Git first resolves your argument through `rev-parse`. If you pass a commit hash or branch name, Git loads the commit object, finds its parent, and generates a diff. If you pass a tag, Git dereferences annotated tags to the underlying commit. If you use the `rev:path` syntax — for example `git show v2.0.0:appsettings.json` — Git treats `rev` as a tree snapshot and walks the path to a blob, printing raw file content without a patch. If you pass a raw blob hash, Git decompresses the stored object and prints it. The resolution path looks like this:

```
git show <argument>
        │
        ▼
   rev-parse
        │
        ├── commit? → load parent → generate patch + metadata
        ├── tag?    → dereference → commit
        ├── rev:path? → tree lookup → blob content
        └── blob hash? → decompress → print
```

Merge commits deserve a brief note. By default, `git show` on a merge commit produces a **combined diff** that can look confusing because it merges changes from multiple parents. For forensic work on merges, `-m` shows separate diffs against each parent. You will use that less often than single-parent commits, but it saves confusion when you need it.

Pickaxe search (`git log -S`) works differently from blame. It walks commits and compares the number of occurrences of a string before and after each commit. If the count changed, that commit is listed. This is ideal for "when was `ValidateToken` first added?" but terrible for "when was `return` added?" — common strings match almost every commit. Narrow the path with `-- src/` or a specific file to keep results usable.

## Syntax / Commands / API

The basic form of `git show` is simple: run it with no arguments to see the latest commit on your current branch, or pass any revision — commit hash, branch name, tag, or `rev~n` — to inspect that point in history. The output includes the commit header (hash, author, date, message) followed by the patch. Add `--no-patch` if you only want metadata without the diff. Add `--stat` for a compact summary of which files changed and how many lines were added or removed.

The `rev:path` syntax is one of the most underused tools in forensic work. Command `git show a1b2c3d:src/Program.cs` prints the contents of `Program.cs` as it existed in commit `a1b2c3d`, without checking out that commit and without modifying your working tree. You can compare two historical versions by running show twice with different revisions. This is invaluable when a bug exists in production at tag `v2.0.0` but is fixed on `main` — you read the old file directly from the tag.

For `git blame`, the default invocation annotates every line of a file. On a thousand-line file, that is noisy. The `-L` flag limits the range: `git blame -L 840,860 Program.cs` blames only lines 840 through 860. In incident response, you almost always know the line number from a stack trace — blame that line first, not the entire file.

Default blame output includes the commit hash (abbreviated), author name, timestamp, line number, and line content. The commit hash is your link to the next step: `git show <that-hash>` to read the full change in context.

Flags `-C`, `-C -C`, and `-M` control copy and move detection as described above. Flag `-w` tells blame to ignore revisions where only whitespace changed — essential after a formatting pull request that would otherwise make every line look newly touched by the formatter.

Pickaxe search uses the log command: `git log -S "NullReferenceException" --oneline -- src/` lists commits where that string's occurrence count changed under `src/`. Add `-p` to include the patch in the output. Related flag `-G` uses regex against diff hunks and can be noisier; `-S` is count-based and often better for identifiers and API strings.

## Examples

Walk through a complete forensic workflow. A service throws `NullReferenceException` at line 847 of `src/Services/AuthService.cs` after a release. Step one: confirm what is on that line and who last modified it.

```bash
git blame -L 847,847 -- src/Services/AuthService.cs
```

Suppose the output shows commit `f4e8b21` by Jane Doe on June 15. Step two: inspect that commit in full.

```bash
git show f4e8b21
```

Read the commit message, the full diff, and surrounding changes. Was the null check removed intentionally as part of a refactor? Did the commit touch related tests? Often the answer is obvious from the patch alone. If the commit message says "Remove redundant null guard — UserService now guarantees non-null," you know to investigate `UserService` next, not Jane's intent.

If blame points to a recent refactor commit but the logic looks ancient, the line may have been moved. Re-run with copy and move detection:

```bash
git blame -C -M -L 847,847 -- src/Services/AuthService.cs
```

Now blame might attribute the line to an older commit from before the file rename. Follow that hash with `git show` again. Optionally, if you suspect a particular expression was introduced long ago, use pickaxe:

```bash
git log -S "user.Profile.Settings" --oneline -- src/
```

The flow from symptom to root cause follows a consistent pattern: stack trace gives you a line number, blame gives you a commit, show gives you context, and pickaxe or `-C -M` fills gaps when refactors obscured the trail.

```
Bug report (line 847 NRE)
        │
        ▼
  git blame -L 847,847
        │
        ▼
  commit f4e8b21 ──► git show f4e8b21
        │
        ├── makes sense? → fix forward / revert
        │
        └── moved code? → git blame -C -M
                    │
                    └── older commit? → git log -S
```

For release forensics, tags combine naturally with show. Command `git show v2.0.0 --stat` summarizes everything that shipped in that release relative to its parent. Command `git show v2.0.0:appsettings.json` reads configuration as it existed at release time — useful when comparing environment-specific settings across versions without deploying old builds.

Before code review on legacy code, blaming a fragile block tells you who to ask for context:

```bash
git blame -L 200,250 src/Legacy/PaymentGateway.cs
git blame -w -L 200,250 src/Legacy/PaymentGateway.cs
```

The second command ignores whitespace-only commits, so you see who last changed meaningful logic, not who ran the formatter.

## Common Mistakes & Anti-patterns

The most common cultural mistake is treating blame output as proof of fault. Blame shows the last commit that modified a line's content, not the commit that introduced the bug. A line written correctly two years ago becomes "blamed" to that old commit even if the bug is caused by new data or a changed caller. Always interpret blame as a pointer, then use `git show` and your domain knowledge to build the full story.

After mass reformatting, plain blame misleads everyone. Every line appears touched by the formatting commit. Use `git blame -w` to skip whitespace-only changes, or blame the file at the commit before the format PR merged. Teams that care about accurate attribution sometimes run blame before merging large format changes.

Refactors without `-C` and `-M` point to the wrong author and the wrong commit. If your investigation says "this line was added yesterday" but the code reads like six-month-old logic, rerun with copy detection before concluding.

On merge commits, default `git show` output can look empty or confusing because combined diffs hide per-parent changes. Use `git show -m` when you need to inspect what each parent contributed.

Pickaxe on common strings floods you with useless results. Searching for `return`, `null`, or `{` matches nearly every commit in the repository. Always scope pickaxe searches to a directory or file, and prefer distinctive identifiers like method names or error message strings.

## Production & Real-World Notes

Modern IDEs — Visual Studio, Rider, Cursor, VS Code with GitLens — show inline blame annotations using the same underlying data as the CLI. The workflow remains: suspicious line, identify commit, read full change. CLI skills matter when you are SSH'd into a build agent, attached to a minimal container, or working in an environment without your IDE.

During incidents, pair blame with deployment timestamps and `git log --since` to narrow the commit range before blaming line by line. If deploy happened at 18:00 UTC, `git log --since="2026-06-14 18:00" --until="2026-06-15 18:00" --oneline` lists candidates; blame confirms which commit touched the failing line.

Large generated files — migrations, lock files, protobuf output — produce slow, meaningless blame. Do not blame ten thousand lines of generated JSON; blame the generator script or the commit that regenerated the file. Some teams mark generated paths in `.gitattributes` and exclude them from review tooling.

Author dates in blame output expose work patterns. Use that data for technical investigation, not surveillance. Healthy teams discuss code and intent openly; blame is a shortcut to the right conversation, not a scoreboard.

## Comparison / Trade-offs

`git blame` excels at answering "who last changed this line and when" but says nothing about why the change was made — the commit message and review thread supply that. `git show` tells the full story of one revision but one at a time; you follow the chain manually. Pickaxe search (`git log -S`) finds introduction or removal of a string across history but only detects count changes, not moves where count stays the same. Regex pickaxe (`git log -G`) matches patterns in diff hunks and can be noisier. For binary search through history when you have a good and bad commit but no line number, `git bisect` is the right tool — a topic for another lesson, but worth knowing blame is not the only forensic instrument.

## Quick Reference

Show a commit: `git show <hash>`. Read a file at a revision without checkout: `git show <rev>:path/to/file`. Blame a line range: `git blame -L 10,20 file.cs`. Follow moved or copied code: `git blame -C -M file.cs`. Ignore whitespace-only blame: `git blame -w file.cs`. Find when a string was added or removed: `git log -S "text" -p -- path/`.

---

## Key Takeaways

- `git show` displays commits, tags, and file snapshots via the `rev:path` syntax.
- `git blame` maps each line to the commit that last modified its content — computed from history, not stored per line.
- `-L` limits blame to a line range; `-C` and `-M` track copied and moved code across refactors.
- The standard forensic workflow is blame to get a commit hash, then show to read the full change.
- `git log -S` finds commits where a string's occurrence count changed — narrow with paths.
- Blame is an investigation tool; interpret results with context, not as automatic fault assignment.
