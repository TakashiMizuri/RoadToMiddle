# Git Essentials: init, clone, status, add, commit, log, diff

> Roadmap: `0.1.3` · Узел: `0.1` — Git: основы · Глубина: практика

## Learning Objectives

После этого урока ты сможешь:

- Создать новый repo через **`git init`** и скопировать существующий через **`git clone`**.
- Читать вывод **`git status`** (short, long, porcelain).
- Stage'ить изменения через **`git add`** (file, directory, `.`, `-A`, `-p`).
- Записывать snapshots через **`git commit`** (`-m`, `-a`) и писать полезные commit messages.
- Смотреть history через **`git log`** (`--oneline`, `--graph`, `--decorate`, `-p`, `-n`).
- Сравнивать состояния через **`git diff`** (unstaged, staged, between commits).
- Уверенно выполнять workflow **edit → add → commit → log → diff**.

---

## Why This Matters

В `0.1.2` ты изучил **three-tree model** как mental map: working directory, staging area, repository. Этот урок переводит карту в muscle memory. Семь команд — `init`, `clone`, `status`, `add`, `commit`, `log`, `diff` — есть в virtually каждой Git-сессии: от weekend API prototype до React feature в команде из сорока человек.

Pull requests, CI и code review предполагают базовую competence, которая звучит просто, но часто shaky на практике: видеть изменения, stage'ить только задуманное, commit'ить с message, за который future readers скажут спасибо, и смотреть history при regression. Remote `push`/`pull` строятся на local loop; branches предполагают понимание states. Даже internals `.git/` в `0.1.4` понятнее, когда ты видел, как команды меняют вывод `status`.

Считай эту главу heartbeat Git. Освой loop сначала; advanced tools — потом.

Если ты привык «сохранить файл в IDE и считать работу зафиксированной», Git потребует смены привычки. Save обновляет только working directory. History появляется только после осознанной цепочки: понять изменения (`status`, `diff`), выбрать, что войдёт в snapshot (`add`), проверить выбор (`diff --staged`), записать с понятным message (`commit`), при необходимости оглянуться назад (`log`). Эта дисциплина кажется лишней на первом проекте из пяти файлов, но на реальном fullstack-репозитории с десятками controllers, React components и config-файлами она отделяет аккуратную history от свалки «fix», «fix2», «final fix».

## Core Concepts

### Два способа получить repository

Каждая Git-история начинается с repository на диске — working tree плюс `.git`. Попасть туда можно двумя путями.

**`git init`** запускает **новый** проект локально. Git создаёт пустой skeleton repository в `.git/` в текущей папке (или указанном path). Remote ещё нет; history ещё нет. Все файлы проекта начинают как **untracked**, пока явно не сделаешь `git add`. Путь для greenfield: новый ASP.NET minimal API, scaffold React app, учебное упражнение.

**`git clone`** копирует **существующий** repository с URL — обычно GitHub, GitLab или Azure DevOps. Clone скачивает **полную history**, checkout default branch в working tree и настраивает remote `origin` на source. Используешь при входе в команду, open source или продолжении работы на другой машине.

Обе команды дают одну структуру: редактируемые файлы плюс `.git`. Clone дополнительно связывает с upstream с первого дня. После выбора entry point daily rhythm одинаков.

### Ежедневный workflow

Professional Git — не набор изолированных tricks, а repeating cycle, отражающий three-tree model из `0.1.2`. Редактируешь files в working directory. Смотришь state через `status` и content через `diff`. Stage'ишь осознанно через `add`. Review staged через `diff --staged`. Записываешь history через `commit`. Ориентируешься во времени через `log`. Remote sync позже; loop работает полностью offline.

```mermaid
flowchart LR
    A[Edit files] --> B[git status]
    B --> C[git diff]
    C --> D[git add]
    D --> E[git diff --staged]
    E --> F[git commit]
    F --> G[git log]
```

Диаграмма намеренно linear. Реальные сессии loop back — после `log` снова edit — но порядок внутри одного commit важен: не commit blindly до просмотра. Ниже — каждая команда с detail для confident use на real repository.

Обрати внимание: между `git diff` и `git add` ты принимаешь решение, *какие* изменения готов продать команде. Между `git diff --staged` и `git commit` — последний шанс отменить попадание лишнего в history. Профессионалы часто делают «status → diff → add → diff --staged → commit» почти машинально; у новичков ломается чаще всего середина — либо add всего подряд, либо commit без просмотра staged diff. Если выработаешь только одну привычку из этого урока, пусть это будет **`git diff --staged` перед каждым commit**.

## Under the Hood (Brief)

Команды не magic — они поддерживают три дерева, которые ты уже знаешь.

**`git init`** создаёт skeleton `.git/`: objects, refs, `HEAD`, config. **`git clone`** fetch'ит objects с remote, пишет в новый `.git`, checkout default branch, записывает `origin`. **`git add`** читает content из working tree, хеширует в blob objects, обновляет `.git/index`. **`git commit`** строит tree из index, commit object с metadata и parent, двигает branch ref. **`git log`** обходит commit graph назад от `HEAD`. **`git diff`** сравнивает blob content между двумя trees — working vs index, index vs `HEAD`, commit vs commit.

On-disk layout objects и refs — **`0.1.4`**. Пока команды — levers на conceptually понятной модели.

## Syntax / Commands / API

### `git init`

`git init` создаёт новый repository. По умолчанию — текущая директория; можно path для subdirectory. Modern Git позволяет явно назвать initial branch — сегодня conventional `main`:

```bash
mkdir my-app && cd my-app
git init
# Создаёт .git/

git init --initial-branch=main
git init -b main                 # короткая форма
```

После init каждый файл **untracked**. Git видит их в `status`, но не version'ит до `git add`. Ловушка: `git commit` сразу — «nothing to commit», новые файлы всё ещё untracked.

### `git clone`

`git clone` скачивает repository и полную history, checkout files для работы:

```bash
git clone https://github.com/user/repo.git
git clone https://github.com/user/repo.git my-folder
git clone --depth 1 https://github.com/user/huge-repo.git   # shallow — advanced
```

Clone неявно делает `git remote add origin <url>`. Commit локально сразу возможен; push требует credentials — позже. Shallow clone (`--depth 1`) меняет depth history на speed и disk; используй, понимая trade-off.

### `git status`

`git status` — dashboard: branch, staged, modified unstaged, untracked. Читай перед каждым commit.

```bash
git status              # long — подробный
git status -s           # short
git status --short
git status --porcelain  # machine-readable для scripts/tools
```

**Long format** группирует файлы в секции three-tree model. «On branch `main`» — где `HEAD`. «Changes to be committed» — **staged**, что запишет `git commit` сейчас. «Changes not staged for commit» — **modified** tracked, working tree ≠ index. «Untracked files» — paths вне version control.

**Short format** — одна строка на path с two-column code. ` M` — modified в working tree, не staged. `M ` — staged. `MM` — оба. `??` — untracked. Быстрое чтение short status становится second nature.

Пример:

```
$ git status -s
 M Program.cs
M  README.md
?? notes.txt
```

`Program.cs` — unstaged edits; `README.md` staged; `notes.txt` новый untracked.

### `git add`

`git add` переносит changes из working directory в staging area. Scope — разные decisions:

```bash
git add Program.cs           # один файл
git add src/                 # директория (recursive)
git add .                    # текущая директория и ниже
git add -A                   # все изменения во всём repo
git add --all
git add -u                   # только tracked (updates/deletes)
git add -p                   # patch mode — hunks интерактивно
git add -i                   # interactive menu
```

Разница `.` и `-A` кусает: `git add .` из subdirectory не stage'ит sibling folders. Dot — subtree вниз от cwd; `-A` — new, modified, deleted **по всему repository**. `-u` — tracked без новых untracked.

**Patch mode (`-p`)** делит changes на **hunks**. Bugfix — `y`, debug print — `n`, split — `s`, edit — `e`. Так unrelated edits в одном файле не попадают в один commit — atomic history.

### `git commit`

`git commit` записывает staged snapshot в repository history. Unstaged и untracked (без add) остаются снаружи.

```bash
git commit -m "Add user registration endpoint"
git commit -a -m "Fix validation typo"    # auto-stage tracked modified/deleted
git commit                                   # editor для message
git commit --amend                           # advanced — позже
```

`-a` удобен для мелких fix на **tracked** files, но **не** добавляет new files. Классическая confusion: создал `NewService.cs`, `commit -a` — файл untracked.

Commit messages — communication. Subject в **imperative mood** — «Add feature», не «Added feature». ~50 символов subject когда возможно. Body — **why**, не список filenames. Issue references — по policy команды.

Хорошо:

```
Fix null reference when order has no items

Orders without line items returned 500. Return empty array instead.
Fixes #128
```

Плохо: `fix`, `updated stuff`, `WIP`, `asdfasdf` — убивают value `git log` при bisect через полгода.

### `git log`

`git log` идёт по history назад от `HEAD`, newest first по умолчанию. Ответы на «what changed recently» и «when did this line appear».

```bash
git log
git log -n 5
git log --oneline
git log --graph
git log --decorate
git log --oneline --graph --decorate --all

git log -p                       # patch для каждого commit
git log -p -2
git log --stat

git log -- README.md
git log -S "functionName"        # pickaxe search
```

Daily: `--oneline --graph --decorate` на recent history. Archaeology: `-p` показывает diff каждого commit. Filter по path или pickaxe `-S` сужает noise в large repos.

Пример:

```
* a1b2c3d (HEAD -> main) Add JWT validation
* d4e5f6g Initial commit
```

### `git diff`

`git diff` сравнивает content между trees Git. Wrong variant — частая beginner mistake; выучи mapping.

```bash
git diff                         # working vs index (unstaged)
git diff --staged                # index vs HEAD (staged)
git diff --cached                # = --staged

git diff HEAD                    # working vs HEAD
git diff HEAD -- Program.cs

git diff abc123 def456
git diff abc123..def456
```

Edit без stage — `git diff` показывает, что попадёт при stage. После `git add` — `git diff --staged` = pre-flight commit. `git diff HEAD` — full picture против last commit.

Чтение unified diff:

```diff
--- a/Program.cs
+++ b/Program.cs
@@ -1,3 +1,4 @@
 unchanged line
-removed line
+added line
```

`-` removed, `+` added. `@@` — hunk header с line numbers.

## Examples

### Пример 1: Новый проект

С нуля — полный путь untracked → committed:

```bash
mkdir todo-api && cd todo-api
git init -b main

echo "# Todo API" > README.md
git status                    # untracked

git add README.md
git commit -m "Initial commit: add README"
git log --oneline
```

Pause на `status`: untracked ≠ staged. `add` перед `commit`. `log` подтверждает history.

### Пример 2: Clone и первое изменение

Existing project — другой origin story, local loop тот же:

```bash
git clone https://github.com/example/shop-api.git
cd shop-api
git status                    # clean

# edit OrderController.cs
git status                    # modified, unstaged
git diff

git add src/Orders/OrderController.cs
git diff --staged

git commit -m "Return 404 when order not found"
git log --oneline -3
```

Discipline: `diff` before add, `diff --staged` before commit — debug prints и secrets вне shared history.

### Пример 3: Partial staging с `-p`

Два logical change в одном файле — patch mode сохраняет atomic commits:

```bash
git add -p Program.cs
# y для bugfix hunk, n для debug print

git diff --staged
git commit -m "Fix divide-by-zero in calculator"
```

Debug print остаётся unstaged для отдельного commit или удаления.

### Пример 4: Сравнение commits

History — не только forward; сравнение past states для regressions:

```bash
git log --oneline
# f1a2b3c Add feature
# d4e5f6a Initial commit

git diff d4e5f6a f1a2b3c
git diff d4e5f6a..HEAD
```

Hashes из `log --oneline`; `A..B` — changes reachable from B since A.

Когда reviewer спрашивает «почему эта строка появилась», ты часто идёшь не в IDE, а в terminal: `git log -p -- path/to/file` или `git log -S "MethodName"` находит commit, который ввёл или удалил фрагмент. Связка `log` + `diff` между commits превращает history из абстрактного списка hashes в понятную историю решений — при условии, что commit messages были написаны для людей, а не как `fix`.

## Common Mistakes & Anti-patterns

**`git commit` без `git add` для new files** — они untracked. Nothing to commit при пустом stage; new files в status.

**`git commit -a` и new files** — `-a` только **tracked** modified/deleted.

**`git add .` из wrong directory** — только subtree ниже cwd. Root + `status` или `git add -A`.

**Meaningless messages** — боль для `log`, bisect, review. Message — часть artifact.

**Пропуск `git diff --staged`** — debug code и credentials в history. Staged diff — last defense до `.gitignore` (`0.1.6`).

**Путать `git diff` и `git diff --staged`** — unstaged edits vs will-be-committed — разные вопросы.

## Production & Real-World Notes

IDE и CI парсят **`git status --porcelain`** и **`git diff --name-only`** — stable для machines. Long status в terminal всё равно нужен.

**Conventional Commits** — `feat:`, `fix:`, `docs:` — например `fix(orders): return 404 when not found`. Changelogs, semantic release.

**GPG signing**, **commit-msg hooks**, `.gitmessage` — после basics, по policy.

Never commit **secrets** — `.env`, API keys. `.gitignore` + **`git diff --staged`**.

## Comparison / Trade-offs

Нет одного «best» invocation — habits под intent. **Specific paths** — default careful work. **`git add .`** — когда review всего scope. **`-A`** — repo-wide из любой subdirectory. **`-p`** — mixed concerns в одном файле. **`commit -a -m`** — tiny tracked fix. **`log --oneline -10`** — daily recall; **`log -p`** — deep investigation. Conscious choice лучше muscle memory «всегда add .».

## Key Takeaways

1. **`git init`** создаёт; **`git clone`** копирует с remote.
2. **`git status`** — dashboard; short codes и porcelain.
3. **`git add`** — scope: `.` vs `-A` vs paths; **`-p`** для hunks.
4. **`git commit -m`** — staged snapshot; **`-a`** только tracked.
5. **Messages** — imperative, subject, body = why.
6. **`git log`** — **`--oneline --graph --decorate`** для daily use.
7. **`git diff`** — правильные trees: unstaged / staged / commits.
