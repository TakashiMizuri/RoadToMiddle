## Why This Matters

В уроке `0.4.2` ты узнал, что TCP-соединение идентифицируется IP-адресом и номером порта, а **socket** привязывает процесс к порту на хосте. В `0.4.6` — как binding на `127.0.0.1` vs `0.0.0.0` определяет, принимает ли сервис подключения только с localhost или из сети. Эти уроки описывают, что происходит *внутри* машины, когда трафик уже пришёл. Здесь вопрос на слой выше: **какой трафик вообще разрешено доставить?**

**Firewall** — это привратник. Он inspect'ит пакеты — обычно по source IP, destination IP, destination port и protocol — и применяет policy: allow, deny или drop. В разработке всё часто крутится на одном ноутбуке без трения firewall. В production PostgreSQL на порту 5432 почти никогда не должен быть доступен из публичного интернета. ASP.NET Core API может слушать 5000 внутри, но наружу торчит только 443 через reverse proxy (`0.4.8`). Неправильно открытый порт — один из самых быстрых путей к breach: сканеры постоянно ищут открытые базы, Redis и admin-панели.

Middle-разработчику не нужно становиться network security engineer, но нужно понимать язык портов и firewall rules при debug deploy'ов, Docker Compose, cloud VM и review инфраструктурных PR. Когда говорят «security group allows 443 inbound» — ты должен точно знать, что это значит для React-приложения и API.

## Core Concepts

### Что firewall делает на самом деле

Firewall enforce'ит **policy на сетевой трафик**. В простейшем виде policy — упорядоченный список rules: «allow TCP 443 from anywhere», «allow TCP 22 from office IP range», «deny everything else». Когда пакет доходит до firewall — будь то Linux `iptables`/`nftables`, Windows Defender Firewall, AWS Security Group или hardware appliance — он match'ится с rules. Первое совпадение обычно побеждает; пакет forward'ится, reject'ится или silently drop'ится.

Firewall'ы работают на разных уровнях. **Stateless** rule может сказать «allow inbound TCP 443» без tracking, был ли connection инициирован изнутри. **Stateful** firewall помнит состояние connection: return traffic для outbound request автоматически разрешён, потому что это established session. Именно stateful inspection делает posture «allow outbound, block unsolicited inbound» дефолтом на большинстве серверов.

Критично: firewall контролирует **network reachability**, а не **authorization**. Открытый 443 пускает HTTPS к web server — но не доказывает, что caller может удалить user record. Authentication, authorization, validation и encryption на уровне приложения остаются твоей зоной ответственности. Считать «port closed = secure» — опасный shortcut.

### Ports как адреса сервисов

Port — 16-bit число (0–65535), multiplex'ирующее много сервисов на одном IP. 0–1023 — **well-known** (на Unix часто нужны root-привилегии). 1024–49151 — **registered**, 49152–65535 — **dynamic/ephemeral** (часто source port на клиенте).

Kestrel на `http://localhost:5000` — клиенты идут на port 5000. Nginx с TLS спереди — браузеры на 443 публичного IP; Nginx forward'ит внутрь на 5000 или 8080. **Одно приложение** может «жить» на разных портах в зависимости от точки в topology.

**Connection refused** — обычно никто не listen'ит на порту, или host firewall явно reject'ит. **Timeout** — пакет silently drop'нут (типично network firewall / SG без allow rule) — клиент retry'ит, пока не сдастся.

### Port 80 — HTTP

Port **80** — default для нешифрованного **HTTP**. Браузеры assume 80 для `http://example.com` без порта. В modern production 80 всё ещё важен, даже когда весь real traffic на HTTPS: лёгкий HTTP listener с **301/302 redirect** на HTTPS на 443. Let's Encrypt (HTTP-01 challenge) тоже требует временной reachability на 80.

С точки зрения security — sensitive data по plain HTTP на 80 в публичном интернете недопустимы: трафик readable и modifiable in transit. Закрыть 80 полностью можно, но редко; redirect-and-upgrade — usual compromise.

### Port 443 — HTTPS

Port **443** — default для **HTTPS** (HTTP over **TLS**). Сюда подключаются браузеры для React SPA или REST API по публичному URL. TLS даёт confidentiality и integrity on the wire; firewall просто permit'ит inbound TCP 443 к load balancer или web server.

В типичном fullstack deploy только 443 (и иногда 80 для redirect) open к `0.0.0.0/0`. Остальное — Kestrel internal port, PostgreSQL, Redis, queues — private network или localhost. Когда дойдёшь до CORS (`0.4.10`), remember: same-origin policy смотрит на scheme, host и port; `https://api.example.com` на 443 — другой origin, чем `http://api.example.com:5000`.

### Port 5432 — PostgreSQL

Port **5432** — default **PostgreSQL**. EF Core подключается через connection string: `Host=db.internal;Port=5432;Database=app;...`. В Docker Compose service `postgres` резолвится в container IP, 5432 — container port.

**Production rule of thumb:** 5432 reachable только с application servers (или bastion/VPN для admin), **никогда** из публичного интернета. Automated scans на open 5432 constant; weak passwords → ransomware за часы. Cloud providers кладут PostgreSQL в private subnet; SG allow'ит 5432 **только** от API tier SG, не от `0.0.0.0/0`.

В dev часто expose 5432 на localhost (`ports: "5432:5432"` в Compose) — map container port на loopback ноутбука; OK для разработки, если host firewall блокирует external access. Map 5432 на `0.0.0.0` на cloud VM — recurring incident pattern.

### Defense in Depth

Firewall rules — один слой **defense in depth**. Другие: TLS снаружи, strong auth, parameterized SQL, secrets не в git, patches, DB roles с minimal permissions. Закрытый 5432 не лечит SQL injection; SQL injection не заменяет закрытый 5432. Нужны оба.

**Principle of least privilege** для ports: expose minimum services minimum source IPs, необходимых для работы системы. React build — static files over 443; прямой доступ к 5432 не нужен. Только backend.

## Under the Hood

### Где живут rules в типичном cloud stack

AWS-style layout: React на CloudFront/S3 за HTTPS, ALB на public subnets с 443, ASP.NET Core на private subnets, RDS PostgreSQL на database subnets.

```
Internet → [SG: allow 443 in] → ALB (public)
                ↓
         [SG: allow app port from ALB only] → API servers (private)
                ↓
         [SG: allow 5432 from API SG only] → RDS (database subnet)
```

Каждый **security group** — stateful virtual firewall на ENI. Rules allow-only; implicit deny на остальное. **Network ACLs** — на границе subnet, могут explicit deny.

На Linux VM `iptables`/`nftables` могут further restrict после cloud SG. Docker добавляет `DOCKER-USER` chain; `-p 5432:5432` создаёт NAT rule — ещё одна причина понимать full path.

### Stateful vs Stateless Matching

Когда API server инициирует outbound к NuGet или payment API, response packets **related** к session. Stateful firewall помнит `(src, dst, src_port, dst_port, protocol)` и permit'ит return path.

Inbound attempts с random internet hosts на 5432 без matching outbound session → default deny → drop → client видит **timeout** или ICMP/RST в зависимости от config.

## Examples

**Development:** Compose `postgres:16` с `ports: ["5432:5432"]`, app `Host=localhost;Port=5432`.

**Production:** RDS private endpoint, SG 5432 only from API SG, connection string в secrets manager.

**Timeout vs refused:** `Connection timed out` к PostgreSQL → network policy (SG, routing). `Connection refused` → PostgreSQL не listen'ит или wrong interface (`0.4.6`).

## Common Mistakes & Anti-patterns

**«Временно» открыть PostgreSQL в интернет** — bots найдут за минуты. SSH tunnel, VPN, managed console.

**HTTPS на 443 ≠ encrypted database path** — ALB-to-API и API-to-PostgreSQL могут быть plain TCP в private VPC — acceptable с clear trust boundaries, но не end-to-end encryption.

**Wide port ranges «на всякий случай»** — лишняя attack surface.

**Docker port publish ≠ cloud firewall** — оба слоя проверять.

## Production & Real-World Notes

**Port matrices** в документации: service, port, protocol, source, destination. **IaC** для SG в git — review как app code. Pentest проверяет exposure 80/443 (и 22 с bastion).

## Comparison / Trade-offs

**Host firewall vs cloud SG:** SG на hypervisor edge; host firewall если compromised instance в том же VPC. **Deny vs drop:** reject быстрее fail; drop скрывает от reconnaissance. **Private DB subnet** без route к IGW — defense помимо SG alone.

## Key Takeaways

- Firewall — **какие пакеты доходят**, не **кто authorized** в app.
- **443** — public HTTPS; **5432** — только private network.
- **Least privilege** на ports и sources.
- **Refused vs timeout** сужает диагноз.
- Defense in depth: firewall + app security + TLS + secrets.
