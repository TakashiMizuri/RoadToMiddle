# SHA-1 Hashes, Commit Graph, and Parent Pointers

> Roadmap: `0.1.5` · Node: `0.1` — Git Basics · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Explain why Git uses **SHA-1** (and optionally **SHA-256**) hashes as object identifiers.
- Read a **commit object** and identify its fields: tree, parent(s), author, committer, message.
- Describe the commit history as a **Directed Acyclic Graph (DAG)**.
- Explain **parent pointers**, **merge commits** (two or more parents), and the **root commit** (no parent).
- Visualize history with `git log --graph`.
- Define **reachability** from HEAD and why it determines what Git keeps.
- Articulate how hashes provide **integrity** and **deduplication**.

---

## Why This Matters

In lesson `0.1.4` you opened `.git/` and learned that Git stores history as **objects** — blobs, trees, commits, and tags — addressed by cryptographic hashes. That lesson answered *what* sits on disk. This lesson answers *how those commits connect into history* and *why the hash of each commit matters beyond being a long ID*.

Without the graph model, Git feels like a tape recorder: commit, commit, commit, in a straight line. That illusion breaks the first time you merge a feature branch, recover work after a rebase, or wonder why cherry-picking "the same change" produces a commit with a different hash. Team workflows — merge commits vs squash merges, rebase vs merge, reading `git log --graph` in code review — all assume you see history as a **Directed Acyclic Graph (DAG)**, not a list.

Every Middle developer should be able to sketch a small graph on a whiteboard: circles for commits, arrows for parent pointers, a branch label on a tip, and an explanation of which commits `git log` will list from `HEAD`. This lesson builds that skill from the inside out. We start with why SHA-1 (or SHA-256) is the name of every object, read the anatomy of a commit, then follow parent links backward through merges and forks until reachability and integrity make intuitive sense.

## Core Concepts

### SHA-1 as Object Identity

Every Git object — whether it holds file bytes, a directory listing, a snapshot metadata block, or a tag message — is named by a hash of its **entire stored representation**. The formula is consistent: Git prefixes the payload with a type word, a space, the size, a null byte, then the content, and hashes the result. For a commit, the "content" includes the tree hash, parent hash lines, author and committer lines, and the message. For a blob, it is just the file bytes.

The default algorithm is **SHA-1**, producing forty hexadecimal characters like `a1b2c3d4e5f6789012345678901234567890abcd`. That string is not assigned by a server and not random. Two machines with the same commit body always compute the same hash. Change a single character in the commit message and the hash changes completely — because the commit object is different bytes.

This design bundles several properties developers rely on every day, even if they never articulate them. **Determinism** means reproducible identity across clones and CI machines. **Practical uniqueness** means collisions are vanishingly unlikely in normal use (cryptographers have demonstrated SHA-1 collision attacks in contrived settings; Git is gradually adding SHA-256 support for long-term safety). **Integrity** means if someone tampered with an object on disk, the stored hash would not match a recomputation, and `git fsck` would complain. **Deduplication** means identical file content across commits references one blob hash, so storage does not multiply with every commit that leaves a file unchanged.

Since **Git 2.29** (2020), you can initialize a new repository with SHA-256 instead:

```bash
git init --object-format=sha256
```

SHA-256 hashes are sixty-four hex characters. The object model, parent pointers, and graph structure are unchanged — only the hash function and length differ. Most existing repositories remain on SHA-1 for now; the concepts in this lesson apply to both.

### Reading a Commit Object

A commit is a text object you can inspect with `git cat-file -p <hash>`. Typical output looks like this:

```
tree 4b825dc642cb6eb9a060e54bf8d69288fbee4904
parent 9fceb02bedb2e9b327bd7467f1ca8a88630ef73d
author Alice <alice@example.com> 1710000000 +0000
committer Alice <alice@example.com> 1710000100 +0000

Add login feature
```

Read it line by line. The **tree** line is the hash of the root **tree** object — the complete directory snapshot at this commit. The **parent** line points backward to the previous commit. The **author** line records who wrote the changes and when (Unix timestamp plus timezone). The **committer** line records who created this commit object; it can differ from author after rebases or cherry-picks, when someone else reapplies patches. After a blank line comes the free-text **message**.

The commit's hash is computed over all of that header plus message. Amend the message and you create a new commit object with a new hash, even if the tree and parents are unchanged. Juniors sometimes think the hash is a random database key; it is a fingerprint of exact content.

Merge commits add multiple **parent** lines — one per merged line of history. The first parent is special in conventions and tooling: it is usually the branch you had checked out when you ran `git merge`. Commands like `git log --first-parent` follow only that line to show a simplified "mainline" story through noisy merge history.

### The Commit Graph Is a DAG

Individual commits are nodes. **Parent pointers** are directed edges from child to parent: each commit stores the hash(es) of its predecessor(s). Follow parents backward from any commit and you walk through time. Follow forward from the root and you see history branch and reunite.

Git history is a **Directed Acyclic Graph**. **Directed** means edges have a direction — child points to parent, not the reverse. **Acyclic** means you cannot start at a commit, follow parents, and return to the same commit. There are no loops in correct Git history. **Graph** means the shape is not always a single line: when two branches diverge, you get a fork; when they merge, you get a commit with two parents; when three branches merge at once (rare), you might see an octopus merge with three or more parents.

Picture a simple team scenario. `main` moves from C1 to C2. A feature branch adds C3 off C1. Merging feature into `main` creates C2 as a merge commit with parents pointing to the old `main` tip and to C3. Meanwhile a hotfix might live on another line. The ASCII sketch is crude but the topology is what matters:

```
        C3 ← feature tip
       /
  C1 ── C2 ← main tip (merge commit)
       \
        C4 ← hotfix branch (example)
```

Linear history — common on rebase-heavy teams — is simply a DAG with one parent per commit:

```
C1 ← C2 ← C3 ← HEAD
```

Each arrow is implemented as a `parent` field in the child commit. `git log` walks this graph from a starting point (usually `HEAD`) and orders commits for display. The ordering rules (topological, date-based, etc.) are configurable, but the underlying structure is always the DAG.

### Root Commits, Merge Commits, and Tips

The **root commit** is the first commit in a repository — or in a grafted or shallow history, the oldest commit you have locally. It has a tree, author, committer, and message, but **no parent line**. Every other commit has at least one parent.

A **merge commit** has **two or more parents**. It represents integrating two lines of work. Importantly, it still has **one tree** — one snapshot of the project after the merge resolution. It does not store two copies of every file; it stores two **parent links** and the merged result as a single tree. Juniors often misunderstand this and imagine merge commits double storage.

A **tip** is a commit with no children on a given branch — the newest commit on that line. The branch ref (for example `refs/heads/main`) points to that tip. When you read "branch `feature` is three commits ahead of `main`," you mean three commits along the DAG from `main`'s tip to `feature`'s tip without counting merge topology unless specified.

### Reachability: What History Means from HEAD

**HEAD** (usually through a branch ref) points to one commit. That commit is **reachable**. Every parent of a reachable commit is also reachable, recursively. The set of all reachable commits from `HEAD` is what `git log` shows by default — your **visible history**.

If you delete a branch ref, commits that were only reachable through that ref may become **unreachable** — no path from any ref or tag follows parent links to them. They still exist on disk for a while. **Reflog** may keep a branch name's old tip reachable for a grace period. Eventually `git gc` can prune unreachable objects. This is why deleting a branch does not instantly purge commits, and why recovery is sometimes possible if you act quickly.

Reachability is the hidden rule behind many operations. `git merge-base main feature` finds common ancestor commits reachable from both tips. `git branch --contains <hash>` asks which branch tips can reach that commit. Understanding reachability turns "where did my commits go?" into a graph question you can answer.

### Integrity Chains and Deduplication Across the Graph

Hashes do not only name objects in isolation. They **chain**. A commit hash depends on the tree hash, parent hash(es), and metadata. The tree hash depends on the hashes of its entries, which may be blobs or subtrees. A blob hash depends on file bytes. Alter any byte anywhere in that closure and at least one hash changes, breaking the link from child to parent. History becomes **tamper-evident** without a central authority — structurally similar to a blockchain without proof-of-work mining.

**Deduplication** works across the graph. If `Program.cs` is identical in fifty commits, all fifty trees may reference the same blob hash for that path. Storage grows when trees change, not every time you commit with an unchanged file. Teams with long histories and large assets benefit enormously from this property.

## Under the Hood

### How Git Computes a Commit Hash

When you run `git commit`, Git serializes the commit body: `tree` line, each `parent` line, `author`, `committer`, blank line, message. It prefixes that byte sequence with `commit <size>\0`, computes SHA-1 (or SHA-256), and writes the zlib-compressed object to `.git/objects/`. The hash printed as your new commit ID is that digest.

Because the serialized form includes parent hashes, each commit cryptographically incorporates its ancestors' identities. You cannot swap parents without changing the child hash. That is why rebasing — which replays commits onto a new parent — must create **new** commit objects even when file changes are identical.

### Walking the Graph in Commands

Git's core algorithms constantly walk the DAG. `git log` starts from a revision and follows parents (all of them at merges unless `--first-parent` is set). `git log main..feature` shows commits reachable from `feature` but not from `main` — useful for "what's in this PR?" `git merge-base` finds best common ancestors for merging. `git bisect` walks history to find a regression. The same parent-pointer machinery powers all of them.

The visualization flags you will use daily combine into one memorable command:

```bash
git log --oneline --graph --decorate --all
```

`--graph` draws ASCII branch and merge lines. `--oneline` compresses each commit to one line. `--decorate` shows ref names at tips. `--all` includes every branch, not only the current one. Run it on a repository with a merge and watch the DAG appear in the terminal — the abstract graph from this lesson made concrete.

To inspect parents directly:

```bash
git rev-parse HEAD^          # first parent
git rev-parse HEAD^2         # second parent (merge only)
git cat-file -p HEAD | grep parent
git rev-list --max-parents=0 HEAD   # find root commit(s)
```

The `^` notation is parent shorthand: `HEAD^` is first parent, `HEAD^2` is second on a merge. `HEAD~3` means three first-parent steps back — different from `HEAD^^^` when merges exist.

## Syntax / Commands / API

Beyond `git log --graph`, these commands support graph reasoning:

```bash
git merge-base main feature
git branch --contains <hash>
git fsck
```

`git merge-base` returns the common ancestor you would merge from. `git branch --contains` lists branches whose tips can reach the given commit. `git fsck` verifies object integrity across the entire store — the ultimate hash-chain health check.

## Examples

### Example 1: Linear History

Create two commits on one branch and graph them. The second commit's `parent` field will point to the first; the first has no parent.

```bash
git init && echo v1 > f.txt && git add f.txt && git commit -m "C1"
echo v2 >> f.txt && git add f.txt && git commit -m "C2"
git log --oneline --graph
# * def5678 C2
# * abc1234 C1
```

You have built the simplest DAG: a line of two nodes.

### Example 2: Merge Creates Two Parents

Branch, commit on the branch, merge back to `main`. Then inspect `HEAD`:

```bash
git checkout -b feature
echo feature >> f.txt && git add f.txt && git commit -m "Feature work"
git checkout main
git merge feature -m "Merge feature"
git cat-file -p HEAD | grep parent
# parent <main-old-tip>
# parent <feature-tip>
```

Two parent lines, one tree — the merge snapshot after Git combined the lines.

### Example 3: Same Patch, Different Hash

Cherry-pick and rebase copy changes but create **new** commits. The patch against a parent may look identical to an older commit elsewhere, but parent hashes, tree hashes, and timestamps differ — so the commit hash differs. That is why "replay this commit on `main`" is not "move this hash"; it is "create a new node with a new identity linked to a new parent."

## Common Mistakes & Anti-patterns

Believing commit hashes are random IDs leads to confusion when rebasing rewrites them. Thinking merge commits store two full copies of the project misses the tree/parent distinction — one tree, multiple parent links. Assuming deleted branches instantly delete commits ignores reachability and gc timing. Expecting author and committer to always match breaks down after rebase and cherry-pick, when committer reflects who ran the command. Treating history as necessarily linear makes team merge graphs look "messy" when they are actually correct DAGs. Confusing **tree hash** with **commit hash** causes errors when referring to snapshots vs historical nodes.

## Production & Real-World Notes

**Signed commits** add a cryptographic signature alongside the commit body; the content hash still covers the same fields. Hosting platforms offer **squash merge**, which creates one new commit on `main` instead of a merge commit — linear history at the cost of losing per-commit branch topology. **Rebase-first teams** prefer straight lines; **merge-first teams** preserve exact integration points. Large repositories often use `git log --first-parent` in dashboards to show a readable mainline. Security and compliance audits sometimes verify hash chains to detect tampered history.

---

## Comparison / Trade-offs

`git log` shows the full DAG from your starting revision — every parent path at merges. `git log --first-parent` simplifies to the line you would get if you only followed the first parent at each merge, which hides side-branch detail before the merge but reads like a single timeline. `git log --oneline --graph --all` is the wide-angle view for all branches at once, at the cost of verbose graphs in busy repositories.

SHA-1 remains the default forty-character hash for billions of existing objects. SHA-256 is the optional sixty-four-character future-proof alternative for new repositories concerned about long-term hash algorithm strength.

## Key Takeaways

1. Object ID = **hash of content** (SHA-1 default; SHA-256 optional).
2. Commits link backward via **parent** pointers forming a **DAG**.
3. **Root** = no parent; **merge** = multiple parents.
4. **HEAD** defines visible history via **reachability**.
5. Hashes chain commits for **integrity** and deduplicate unchanged **blobs/trees**.
6. `git log --graph` visualizes the DAG — essential for team workflows.
7. Rebase/cherry-pick create **new hashes** because parent/tree/metadata change.
