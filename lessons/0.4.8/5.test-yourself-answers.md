## Answers

### 1. [Recall]
**Answer:** **Клиента** (outbound от LAN к internet).

**Why:** Forward = client's representative to origins.

---

### 2. [Recall]
**Answer:** **Сервера** (inbound от internet к backends).

**Why:** Reverse = server's front door hiding internals.

---

### 3. [Recall]
**Answer:** **`X-Forwarded-For`**, **`X-Forwarded-Proto`** (also `Host`, `X-Request-Id`).

**Why:** Preserve client IP and original scheme behind proxy.

---

### 4. [Understanding]
**Answer:** **443** public — firewall/SG (`0.4.7`); **TLS termination** и routing на edge. Internal **8080** — plain HTTP в trusted network, проще certs и scale multiple instances за одним 443.

**Why:** Separation of edge and app tier.

---

### 5. [Understanding]
**Answer:** **L7** видит HTTP path и может route `/api` → pool A, `/` → pool B. **L4** видит только IP/ports — не различит paths на одном IP:443 без pass-through + separate listeners.

**Why:** L7 = application layer awareness.

---

### 6. [Understanding]
**Answer:** Client может **подделать** header; app решит, что connection secure / wrong IP для audit/rate limit. Trust только от **known proxy/LB IP** (`KnownProxies`).

**Why:** Headers не authenticated сами по себе.

---

### 7. [Application]
**Answer:** (1) Kestrel **down** или wrong port. (2) **Network** SG блокирует proxy→backend. (3) App **crash/timeout** на request. (4) Wrong **upstream** in nginx config.

**Why:** 502 = proxy didn't get valid upstream response.

---

### 8. [Application]
**Answer:** **`UseForwardedHeaders`** с **`XForwardedProto`** (и правильные KnownProxies/KnownNetworks).

**Why:** Kestrel видит HTTP от ALB, не знает external HTTPS без header.

---

### 9. [Analysis]
**Answer:** **Неверно.** Один **nginx** или **ALB** часто совмещает reverse proxy + load balancing. Роли логические, не обязательно отдельные boxes.

**Why:** Software/managed services combine functions.

---

### 10. [Analysis]
**Answer:** **Uneven load** (hot spot backend); **worse failover** (sessions tied to dead node); **unnecessary** для stateless JWT; **scale-out** less effective.

**Why:** Sticky only when server-side session affinity required.

---

### 11. [Recall]
**Answer:** Периодически **probe** backends (`/health`); **remove** failing from rotation.

**Why:** Avoid sending traffic to dead instances.

---

### 12. [Understanding]
**Answer:** **Forward:** сотрудник → office proxy → external sites; server видит proxy IP. **Reverse:** user → internet → ALB/nginx → ваш Kestrel; user не видит internal IPs.

**Why:** Opposite direction and beneficiary.

---

### 13. [Application]
**Answer:** **Typical:** TLS terminate на **CDN** (static) и/или **ALB**; internal Kestrel HTTP. (Accept: TLS at ALB only if CDN pass-through for API.)

**Why:** Edge termination pattern.

---

### 14. [Understanding]
**Answer:** Central **cert management**, **CPU offload**, единая точка для WAF/rate limit; backends упрощаются (HTTP internal). Trade-off: encrypt internal hop separately if compliance requires.

**Why:** Operational efficiency at edge.

---

### 15. [Analysis]
**Answer:** **`HTTP_PROXY`** — client (curl) использует **forward proxy** для исходящих. **`proxy_pass`** в nginx — **reverse proxy** config: nginx принимает inbound и forward'ит к backend.

**Why:** Env var = client-side; nginx inbound = server-side.
