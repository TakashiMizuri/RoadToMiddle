# Detached HEAD State: What It Is and How to Escape

> Roadmap: `0.1.11` · Node: `0.1` — Git Basics · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Define **HEAD** and explain what **detached HEAD** means at the ref level.
- Describe how you **enter** detached HEAD (checkout commit SHA, tag, remote branch without tracking).
- Explain why **new commits** in detached HEAD become **unreachable** when you leave without saving.
- Save work with **`git switch -c new-branch`** or **`git checkout -b`**.
- Return to a branch safely with **`git switch main`** or **`git checkout main`**.
- Recover "lost" commits using **`git reflog`**.

---

## Why This Matters

You run a command that feels innocent:

```bash
git checkout v1.0.0
# or
git checkout abc1234
```

Git responds with a warning that many developers skim past:

```
You are in 'detached HEAD' state...
```

The warning sounds alarming, so some people panic and think the repository is corrupted. Others ignore it entirely, make a few commits fixing something they found while browsing an old release, switch back to `main`, and only then discover that their work vanished from `git log`. Neither reaction is correct.

**Detached HEAD is not an error.** It is a deliberate, valid state that Git enters whenever you check out a specific commit rather than a branch name. Inspectors use it to read old code. Release engineers use it to verify what shipped in a tag. `git bisect` uses it while hunting regressions. CI systems spend most of their time in detached HEAD because they check out a specific SHA, not a developer's local branch. The state becomes dangerous only when you **commit new work** without attaching that work to a branch, then **leave** detached HEAD as if nothing happened.

Understanding HEAD at the plumbing level — what the `.git/HEAD` file contains, how refs keep commits alive, why dangling commits survive in the reflog — separates developers who fear Git from those who navigate history with confidence. This lesson goes deep because detached HEAD touches nearly every core Git concept: refs, commits, reachability, garbage collection, and recovery.

## Core Concepts

### What Is HEAD?

**HEAD** answers one question: *where am I working right now?* It is a pointer to your current checkout context. In the normal case — what Git documentation calls an **attached HEAD** — that pointer does not aim directly at a commit. Instead it aims at a **branch ref**, and the branch ref aims at a commit:

```
HEAD → refs/heads/main → commit C
```

You are "on branch `main`." When you create a new commit, Git advances the `main` ref to the new SHA. HEAD still points at `refs/heads/main`, which now points at the new tip. The branch name moves forward with your work. That is the contract you rely on every day: commits you make belong to the branch you think you are on.

### Detached HEAD Defined

**Detached HEAD** breaks that indirection. HEAD points **directly at a commit SHA**, with no branch name in between:

```
Normal (attached):   HEAD → refs/heads/main → commit C
Detached:            HEAD → commit abc1234   (no branch name)
```

There is no `refs/heads/*` entry tracking `abc1234` as its tip. You are still *at* a valid commit — your working tree matches that snapshot — but you are not *on* a branch. If you create new commits from this state, they extend a chain that only HEAD knows about:

```
Detached with new work:

  main → ... → C
                \
                 D → E (HEAD)    ← commits D and E exist, but no branch points here
```

Commit `C` is still where `main` points. Commits `D` and `E` are real objects in `.git/objects`, but after you run `git switch main`, HEAD moves back to `refs/heads/main → C`. Nothing references `D` or `E` anymore. They become **dangling commits** — reachable only through the reflog until garbage collection reclaims them. From the perspective of `git log main`, your "Emergency fix" never happened.

This is the core danger, stated plainly: **branches are the labels that keep commit chains alive.** Detached HEAD means you are reading history without a label. Commits you add are written to the object database but have no permanent name unless you create one.

### How You Enter Detached HEAD

Several everyday commands put you in this state, and recognizing them prevents surprises.

Checking out a raw commit SHA — `git checkout abc1234` or `git switch --detach abc1234` — always detaches. Checking out a **tag** does the same, because tags are refs that point at a fixed commit and do not advance when you commit: `git checkout v1.0.0` detaches at the tag's target. Moving relative to a branch tip without attaching also detaches: `git switch --detach main~3` checks out the commit three ancestors before `main`'s tip, but does not create or move `main`.

Checking out a **remote-tracking branch** directly — `git checkout origin/main` or `git switch origin/feature` without creating a local tracking branch — often detaches, depending on configuration. The habit of typing `git checkout origin/feature` to "just look at the remote version" is a common trap; you end up detached at the remote tip instead of on a local branch that tracks it.

**`git bisect`** deliberately enters detached HEAD while you test middle commits between known good and bad revisions. You should not commit fixes during bisect; you mark commits good or bad and let bisect move HEAD for you.

What does **not** detach you: `git switch feature-branch` or `git checkout feature-branch` when `feature-branch` is a local branch name. Those commands set `HEAD → refs/heads/feature-branch`.

## Under the Hood

### The HEAD File

Git stores HEAD as a small text file at `.git/HEAD`. Its contents tell you immediately whether you are attached or detached.

When attached to `main`:

```
ref: refs/heads/main
```

Git resolves HEAD by reading `refs/heads/main`, which contains a forty-character SHA, and that SHA is your current commit.

When detached:

```
abc1234567890abcdef1234567890abcdef123456
```

The commit hash is **literal** in the HEAD file — no `ref:` prefix. Git does not follow a branch; HEAD *is* the commit identifier. You can verify this yourself any time:

```bash
cat .git/HEAD
git status    # reports "HEAD detached at abc1234" or "HEAD detached at v1.0.0"
```

The `git status` message translates the plumbing into human language. Learning to read `.git/HEAD` directly builds the mental model that demystifies warnings.

### Reachability and Why Commits "Disappear"

Git keeps objects — commits, trees, blobs — as long as they are **reachable** from some ref. The main ref namespaces that anchor reachability:

```
refs/heads/*     branches
refs/tags/*      tags (usually immutable pointers to commits)
refs/remotes/*   remote-tracking branches
HEAD             while you are checked out
reflog entries   safety net with expiration
```

A commit is reachable if you can start at a ref and follow parent pointers until you hit that commit. While you are in detached HEAD at commit `E`, both `E` and its ancestors are reachable because HEAD itself counts. The moment you `git switch main`, HEAD becomes `ref: refs/heads/main → C`. Commits `D` and `E` are no longer on any path from any ref — unless the reflog still mentions them.

The **reflog** (`refs/logs/HEAD`) records every movement of HEAD with a timestamp and a description:

```bash
git reflog
# e123456 HEAD@{0}: checkout: moving from abc1234 to main
# f789012 HEAD@{1}: commit: hotfix on detached head
# abc1234 HEAD@{2}: checkout: moving from main to v1.0.0
```

Even "lost" commits remain recoverable for about **ninety days** by default (controlled by `gc.reflogExpire`). You can recreate a branch pointing at the lost tip:

```bash
git switch -c recovered-work f789012
# equivalent older form:
git branch recovered-work f789012 && git switch recovered-work
```

Now `refs/heads/recovered-work → f789012`, and the entire chain leading to `E` is permanently reachable again until you delete that branch.

### What Happens on `git switch main`

Walking through the steps clarifies why no prompt saves you automatically.

First, Git checks for uncommitted changes. If your working tree or index differs from the current commit, switch may refuse, stash, or carry changes depending on flags and file overlap. Assuming a clean tree, Git writes `ref: refs/heads/main` into `.git/HEAD`. It reads `refs/heads/main` to find commit `C`, then checks out `C`'s tree into your working directory. Commits `D` and `E`, if you made them while detached without creating a branch, have no named ref. They exist as objects and reflog entries only.

There is no dialog that says "You are leaving commits behind — create a branch?" Git assumes you know what you are doing. That design choice is why detached HEAD causes so many support tickets, and why **`git switch -c branch-name`** before your first commit off a tag is the professional habit.

### Tags Are Not Branches

A common confusion fuels detached HEAD accidents: developers treat tags like slow branches. A **branch** is a movable ref — new commits advance it. A **tag** (especially an annotated release tag) is a fixed bookmark on one commit. Checking out `v2.0.0` puts you at that release snapshot in detached state. If you commit, the tag stays on the old commit; your new work extends a orphan chain. Release hotfixes therefore start with **`git switch -c hotfix/2.0.1 v2.0.0`**, creating a branch at the tag before changing any files.

### Creating a Branch Saves the Line

While detached at commit `E` with new work you want to keep:

```bash
git switch -c fix-from-v1
```

Git creates `refs/heads/fix-from-v1` pointing at the current HEAD (`E`) and attaches HEAD to that ref. Commits `D` and `E` are now branch-reachable. You can push, open a PR, and merge normally. The older syntax `git checkout -b fix-from-v1` does the same thing; modern Git prefers `switch -c` for clarity.

```mermaid
gitGraph
  commit id: "A"
  commit id: "B"
  commit id: "C"
  branch main
  checkout main
  commit id: "D"
  commit id: "E"
```

In this diagram, if `main` still points at `C` while HEAD sits at `E` without a branch name, `D` and `E` hang off the side. Creating `fix-from-v1` at `E` gives that chain a home.

### Garbage Collection and the Limits of Recovery

Dangling commits are not deleted immediately. Git's **garbage collector** (`git gc`) runs periodically or on demand, traversing all reachable objects from refs and removing everything else. Until GC runs, orphaned commits from a detached HEAD session occupy disk space but remain intact. Reflog entries extend that window further: even after a commit becomes unreachable, the reflog may still hold its SHA with a human-readable message like `commit: Emergency fix on old tag`.

Recovery fails when too much time passes, when someone runs `git gc --prune=now` aggressively, or when the work happened on a machine whose reflog you no longer have — for example, if you committed on a CI runner without pushing. That is why the fix for detached HEAD is **procedural**, not technical: create a branch before you need recovery. Waiting until after you switch away turns a one-second `git switch -c` into an archaeology exercise with no guarantee of success.

You can inspect what Git would prune without doing it:

```bash
git fsck --unreachable    # list unreachable objects
git reflog expire --dry-run --all
```

These commands are rarely needed day to day, but they complete the mental model: detached commits are real objects subject to the same lifecycle rules as everything else in `.git/objects`.

### Symbolic HEAD and Future Checkouts

Modern Git also supports **symbolic refs** in some contexts, but your daily detached/attached distinction still flows through `.git/HEAD`. When Git checks out a branch, it writes the symbolic form `ref: refs/heads/branch-name`. When it detaches, it writes the raw SHA. Some porcelain commands (`git switch`, `git checkout`) translate user intent into these plumbing writes. Understanding that translation helps when a tool or IDE checks out "detached at origin/main" — underneath, it is the same literal SHA in HEAD, and the same rules about branching apply.

## Syntax / Commands / API

Enter detached state intentionally when inspecting history:

```bash
git switch --detach v1.0.0
git checkout abc1234
git switch --detach HEAD~2
```

Inspect where you are:

```bash
git status          # "HEAD detached at ..."
git branch          # current branch list; detached shows no branch checked out
git log --oneline -1
cat .git/HEAD       # raw attached vs detached indicator
```

Save work by naming the current tip:

```bash
git switch -c my-fix          # from current detached HEAD
git switch -c my-fix abc1234  # create branch at specific commit
```

Return without saving new commits (safe when you only read files):

```bash
git switch main
```

Recover after leaving detached commits behind:

```bash
git reflog
git switch -c rescue HEAD@{1}
git branch rescue f789012
```

## Examples

### Example 1: Safe inspection without new commits

You want to read `README.md` as it existed five commits ago:

```bash
git switch main
git switch --detach HEAD~5
cat README.md              # read old version
git switch main            # leave — no new commits were made, nothing to lose
```

No branch was needed because you did not produce new commits. Detached HEAD served as a temporary time machine.

### Example 2: The classic mistake

```bash
git checkout v2.0.0
echo "fix" >> app.txt
git commit -am "Emergency fix on old tag"   # commit E created while detached
git checkout main                           # HEAD leaves E behind
git log --oneline                           # "Emergency fix" is NOT on main
```

The fix exists in the object database but not on any branch you can see. Recovery:

```bash
git reflog
# locate the SHA for "Emergency fix"
git switch -c tag-hotfix <SHA>
```

Without reflog — for example on a fresh clone where you never committed — recovery may be impossible. That asymmetry is why teams treat detached commits as urgent until branched.

### Example 3: Correct hotfix workflow from a tag

Production runs `v2.0.0`. A bug must be patched from that exact snapshot:

```bash
git fetch origin --tags
git switch -c hotfix/from-v2 v2.0.0    # branch BEFORE editing
# make changes
git commit -am "Fix null reference in payment handler"
git push -u origin hotfix/from-v2
```

The branch preserves the commit chain from the first edit onward. Code review, CI, and merge proceed as with any feature branch.

## Common Mistakes & Anti-patterns

Ignoring the detached HEAD warning and committing anyway is the root cause of most "lost work" reports. Switching to `main` afterward feels like closing a tab — but unlike a browser, Git does not keep orphan commits visible in normal logs. Treating tags as branches leads to the same orphan chains; tags do not move forward. The habit of `git checkout origin/feature` for quick inspection creates detached state when `git switch feature` (after `git switch -c feature origin/feature` once) would attach properly.

Some developers create one commit while detached and assume merging `main` later will pick it up — it will not, because the commit is not on any branch reachable from `main`. Others run `git bisect`, fix the bug inline, commit the fix, and confuse bisect state with a proper branch — the fix belongs on a branch created after bisect finishes.

## Production & Real-World Notes

Release engineering almost always branches from tags: `git switch -c release/2.0.1 v2.0.0`. CI pipelines checkout specific SHAs or merge refs for pull requests — detached HEAD is the default in automation, not an anomaly. Document for your team: **detached ≠ broken repository**; **detached + unbranch'd commits + checkout elsewhere = silent data loss risk**.

Enterprise support patterns show that "I lost my commit" tickets resolve with **`git reflog`** in the vast majority of cases, provided the developer acts within the reflog retention window and has not run aggressive garbage collection. Teach juniors to run `git reflog` before panicking.

During `git bisect`, resist committing fixes. Record good/bad, finish bisect, then branch from the identified commit for the actual fix. Mixing bisect navigation with feature development compounds detached HEAD confusion.

When a colleague says they are "on origin/feature" ask whether they created a local tracking branch. If they ran `git checkout origin/feature` directly, they are detached at the remote tip — commits will not update `origin/feature` on push (push may fail or require creating a local branch anyway). The fix is `git switch -c feature origin/feature`, which creates `refs/heads/feature` tracking the remote. Small habit change, large reduction in detached HEAD incidents.

## Comparison / Trade-offs

When you only need to **browse old code**, detached checkout without commits is the lightest approach — enter, read, switch back. When you need to **fix code starting from a tag or old commit**, create a branch first with `-c`; the few extra keystrokes prevent reflog archaeology later. For **throwaway experiments** you might discard, detached HEAD without a branch is acceptable if you truly will not commit, or if you will commit and then reset away. For **finding regressions**, `git bisect` owns detached navigation; your job is to test, not to build permanent history until bisect completes.

## Quick Reference

Attached HEAD: `HEAD → branch ref → commit`. Detached HEAD: `HEAD → commit SHA` directly. Save work: `git switch -c branch-name`. Go back safely (no new commits): `git switch main`. Recover lost tips: `git reflog`, then `git switch -c rescue <SHA>`.

---

## Key Takeaways

- **HEAD** is your current position; normally it points to a **branch ref**, not directly to a commit.
- **Detached HEAD** means HEAD points **directly at a commit SHA** — common after checking out tags or raw SHAs.
- New commits in detached state **become unreachable** when you leave unless you attach them with **`git switch -c`**.
- **Tags do not advance**; they are not branches. Branch before committing off a tag.
- **`git reflog`** recovers recently orphaned commits for ~90 days locally.
- Detached HEAD is **normal for inspection and CI**; it is **dangerous for unbranch'd development**.
