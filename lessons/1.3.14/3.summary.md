# Extension methods — краткая выжимка

> `1.3.14` · ~10 мин на повторение

## Главная идея

**Extension method** — static method в static class с `this` у первого параметра. Вызов `x.F()` компилятор переписывает в `Extensions.F(x)`. Это **не** добавление метода в тип — только syntactic sugar и **ExtensionAttribute** в metadata для IDE.

## Ключевые пункты

- Синтаксис: `static class`, `static` method, `this T receiver`.
- **Instance method побеждает** над extension с той же сигнатурой.
- Extensions видны через **`using`** namespace static class (`System.Linq` для LINQ).
- **`this` может быть null** — NRE только если внутри разыменовать без проверки.
- LINQ: `Enumerable` / `Queryable` — extensions; query syntax → те же вызовы.
- IL: обычный static `call`; JIT может inline.
- Нет доступа к **private** members extended type.

## Самые важные примеры

```csharp
// Definition
public static bool IsNullOrEmpty(this string? s) =>
    string.IsNullOrEmpty(s);

// Call site lowering
name.IsNullOrEmpty();  // → StringExtensions.IsNullOrEmpty(name)

string? s = null;
s.IsNullOrEmpty();     // OK — не как s.Length
```

## «Под капотом» в одном абзаце

Roslyn резолвит extensions после instance candidates; помечает method `[ExtensionAttribute]`. Runtime — static dispatch, receiver первым аргументом. Ambiguity (два extension в scope) → CS0121; явный вызов `Class.Method(x)`.

## Частые ошибки

- Думать, что extension «на типе» в reflection как instance API.
- Забыть `using System.Linq`.
- Ожидать NRE на null receiver до входа в extension.
- Вешать тяжёлую domain logic на extensions.

## Мнемоника

> **Extension = static call в disguise; using brings it; instance always wins.**
