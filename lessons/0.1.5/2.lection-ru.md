# SHA-1 hashes, commit graph, parent pointers

> Roadmap: `0.1.5` · Узел: `0.1` — Git: основы · Глубина: понимание

## Learning Objectives

После этого урока ты сможешь:

- Объяснить, почему Git использует **SHA-1** (и опционально **SHA-256**) как идентификаторы objects.
- Прочитать **commit object** и назвать поля: tree, parent(s), author, committer, message.
- Описать history как **Directed Acyclic Graph (DAG)**.
- Объяснить **parent pointers**, **merge commits** (2+ parents), **root commit** (без parent).
- Визуализировать history через `git log --graph`.
- Определить **reachability** от HEAD и почему от этого зависит, что Git хранит.
- Объяснить, как hash'и дают **integrity** и **deduplication**.

## Why This Matters

В `0.1.4` ты заглянул в `.git/` и узнал, что Git хранит history как **objects** — blobs, trees, commits, tags — по криптографическим hash'ам. Тот урок ответил на вопрос *что лежит на диске*. Этот урок отвечает *как commit'ы соединяются в history* и *зачем hash каждого commit'а важнее, чем просто длинный ID*.

Без модели графа Git кажется магнитофонной лентой: commit, commit, commit по прямой. Иллюзия рушится при первом merge feature-ветки, после rebase или когда cherry-pick «того же изменения» даёт commit с другим hash. Team workflow — merge vs squash, rebase vs merge, чтение `git log --graph` в code review — предполагает, что ты видишь history как **DAG**, а не как список.

Middle-разработчик должен нарисовать на доске маленький граф: кружки — commit'ы, стрелки — parent pointers, метка ветки на tip, и объяснить, какие commit'ы покажет `git log` от `HEAD`. Мы соберём этот навык изнутри: зачем SHA-1 (или SHA-256) именует каждый object, анатомия commit'а, обход parent links через merge и fork, пока reachability и integrity не станут интуитивными.

## Core Concepts

### SHA-1 как identity объекта

Каждый Git object — байты файла, listing каталога, metadata snapshot или message тега — именуется hash'ем **всего** сохранённого представления. Формула едина: type, пробел, size, `\0`, content — и хеш. У commit в content входят tree hash, строки parent, author, committer и message. У blob — только байты файла.

По умолчанию алгоритм **SHA-1**, сорок hex-символов вроде `a1b2c3d4e5f6...`. Строка не с сервера и не случайна. Две машины с одним телом commit'а всегда получат один hash. Измени один символ в message — hash полностью другой, потому что bytes object'а другие.

В этом заложено то, на что разработчики опираются каждый день. **Determinism** — воспроизводимая identity на clone и в CI. **Практическая уникальность** — collision'ы в нормальной работе невероятны (у криптографов есть атаки на SHA-1 в искусственных условиях; Git постепенно добавляет SHA-256). **Integrity** — подмена bytes на диске даст несовпадение hash, `git fsck` заметит. **Deduplication** — одинаковый content файла в разных commit'ах → один blob hash, storage не множится с каждым commit'ом без изменений файла.

С **Git 2.29** (2020) можно инициализировать repo с SHA-256:

```bash
git init --object-format=sha256
```

Hash SHA-256 — шестьдесят четыре символа. Модель objects, parent pointers и граф та же — меняются функция и длина. Большинство существующих repo пока на SHA-1; концепции урока одинаковы.

### Чтение commit object

Commit — текстовый object, смотришь через `git cat-file -p <hash>`. Типичный вывод:

```
tree 4b825dc642cb6eb9a060e54bf8d69288fbee4904
parent 9fceb02bedb2e9b327bd7467f1ca8a88630ef73d
author Alice <alice@example.com> 1710000000 +0000
committer Alice <alice@example.com> 1710000100 +0000

Add login feature
```

Строка **tree** — hash root **tree**, полный snapshot каталогов в этом commit'е. **parent** — назад к предыдущему commit'у. **author** — кто написал изменения и когда (Unix timestamp + timezone). **committer** — кто создал этот commit object; после rebase/cherry-pick часто не совпадает с author. После пустой строки — **message**.

Hash commit'а считается по всему header + message. Amend message → новый commit object с новым hash, даже если tree и parent'ы те же. Новички думают, что hash — случайный ключ в БД; на деле это отпечаток exact content.

У merge commit несколько строк **parent** — по одной на влитую линию history. **First parent** особенный: обычно ветка, на которой ты был при `git merge`. `git log --first-parent` идёт только по ней — упрощённая «mainline» через шумные merge.

### Commit graph — это DAG

Отдельные commit'ы — узлы. **Parent pointers** — направленные рёбра от child к parent: commit хранит hash предшественника(ов). Идёшь по parent'ам назад — идёшь во времени. Смотришь от root вперёд — видишь ветвление и слияние.

History Git — **Directed Acyclic Graph**. **Directed** — рёбра имеют направление, child → parent. **Acyclic** — нельзя по parent'ам вернуться в тот же commit; петель в корректной history нет. **Graph** — не всегда одна линия: fork при расхождении веток, merge commit с двумя parent'ами при слиянии, редко octopus merge с тремя и более parent'ами.

Типичная картина команды: `main` от C1 к C2, feature добавляет C3 от C1, merge feature в `main` даёт C2 с parent'ами на старый tip `main` и на C3. Параллельно hotfix на другой линии:

```
        C3 ← tip feature
       /
  C1 ── C2 ← tip main (merge commit)
       \
        C4 ← hotfix (пример)
```

Линейная history — тот же DAG, но у каждого commit'а один parent:

```
C1 ← C2 ← C3 ← HEAD
```

Каждая стрелка — поле `parent` в дочернем commit'е. `git log` обходит граф от стартовой точки (обычно `HEAD`) и упорядочивает вывод. Правила сортировки настраиваются, структура всегда DAG.

### Root commit, merge commit и tip

**Root commit** — первый commit в repo (или самый старый при shallow/grafted history). Есть tree, author, committer, message, **нет parent**. Все остальные имеют минимум один parent.

**Merge commit** — **два или более parent'а**. Интеграция двух линий работы. Важно: **один tree** — один snapshot после merge resolution. Не две копии всех файлов, а два **parent link** и один результат в tree. Новички часто думают обратное.

**Tip** — commit без детей на данной ветке, самый новый на линии. Ref ветки (`refs/heads/main`) указывает на tip. «Ветка `feature` на три commit'а впереди `main`» — три узла по DAG от tip `main` до tip `feature`, без учёта merge topology, если не оговорено иначе.

### Reachability: что значит history от HEAD

**HEAD** (через branch ref) указывает на один commit. Он **reachable**. Каждый parent reachable commit'а тоже reachable, рекурсивно. Множество reachable от `HEAD` — то, что `git log` показывает по умолчанию — **visible history**.

Удалил ref ветки — commit'ы, reachable только через неё, могут стать **unreachable**: ни один ref/tag не ведёт parent-цепочкой к ним. На диске они ещё живут. **Reflog** может держать старый tip ветки какое-то время. Потом `git gc` может prune unreachable objects. Поэтому удаление ветки не мгновенно стирает commit'ы, и иногда recovery возможен.

Reachability — скрытое правило многих операций. `git merge-base main feature` — общие предки, reachable с обоих tip'ов. `git branch --contains <hash>` — какие tip'ы веток достигают commit. «Куда делись мои commit'ы?» — вопрос о графе, на который можно ответить.

### Цепочка integrity и deduplication в графе

Hash'и не только именуют objects по отдельности — они **связаны в цепочку**. Hash commit'а зависит от tree, parent(s), metadata. Hash tree — от hash записей (blobs, subtrees). Hash blob — от bytes. Измени байт где угодно в этом замыкании — сломается link child → parent. History **tamper-evident** без центрального органа — структурно похоже на blockchain без mining.

**Deduplication** работает по всему графу. Одинаковый `Program.cs` в пятидесяти commit'ах — один blob hash в trees. Storage растёт при изменении trees, не при каждом commit с неизменённым файлом.

## Under the Hood

### Как Git считает hash commit'а

При `git commit` Git сериализует body: `tree`, каждый `parent`, `author`, `committer`, пустая строка, message. Prefix `commit <size>\0`, SHA-1 (или SHA-256), zlib в `.git/objects/`. Напечатанный ID — этот digest.

В сериализованной форме есть hash parent'ов — каждый commit криптографически включает identity предков. Подменить parent без смены hash child нельзя. Поэтому rebase с новым parent обязан создавать **новые** commit objects, даже если diff файлов тот же.

### Обход графа в командах

Ядро Git постоянно ходит по DAG. `git log` от revision по parent'ам (все на merge, если нет `--first-parent`). `git log main..feature` — reachable из `feature`, но не из `main` («что в PR?»). `git merge-base` — общие предки для merge. `git bisect` — поиск регрессии. Одна и та же механика parent pointers.

Запоминаемая команда визуализации:

```bash
git log --oneline --graph --decorate --all
```

`--graph` — ASCII ветки и merge, `--oneline` — одна строка на commit, `--decorate` — имена ref на tip'ах, `--all` — все ветки. Запусти на repo с merge — абстрактный DAG из урока в терминале.

Родители напрямую:

```bash
git rev-parse HEAD^
git rev-parse HEAD^2
git cat-file -p HEAD | grep parent
git rev-list --max-parents=0 HEAD
```

`^` — parent: `HEAD^` first parent, `HEAD^2` second на merge. `HEAD~3` — три шага по first parent; на merge отличается от `HEAD^^^`.

## Syntax / Commands / API

Дополнительно к `git log --graph`:

```bash
git merge-base main feature
git branch --contains <hash>
git fsck
```

`git merge-base` — общий предок для merge. `git branch --contains` — ветки, чьи tip'ы достигают commit. `git fsck` — проверка integrity всего store.

## Examples

### Пример 1: линейная history

```bash
git init && echo v1 > f.txt && git add f.txt && git commit -m "C1"
echo v2 >> f.txt && git add f.txt && git commit -m "C2"
git log --oneline --graph
```

Второй commit ссылается parent'ом на первый; у первого нет parent — простейший DAG из двух узлов.

### Пример 2: merge — два parent'а

```bash
git checkout -b feature
echo feature >> f.txt && git add f.txt && git commit -m "Feature work"
git checkout main
git merge feature -m "Merge feature"
git cat-file -p HEAD | grep parent
```

Две строки parent, один tree — snapshot после слияния линий.

### Пример 3: тот же patch, другой hash

Cherry-pick и rebase копируют изменения, но создают **новые** commit'ы. Patch к parent может выглядеть как у старого commit'а, но parent hash, tree hash и timestamps другие — hash commit'а другой. «Перенести commit на `main`» — не «переместить этот hash», а «создать новый узел с новой identity и новым parent».

## Common Mistakes & Anti-patterns

«Hash случайный» — путаница при rebase. «Merge хранит две копии проекта» — на деле один tree и несколько parent link. «Удалил ветку — commit'ы сразу пропали» — игнорирование reachability и gc. «Author всегда = committer» — ломается после rebase/cherry-pick. «History всегда линейная» — merge-граф команды кажется «грязным», хотя DAG корректен. Путать **tree hash** и **commit hash** — разные objects.

## Production & Real-World Notes

**Signed commits** добавляют signature; content hash покрывает те же поля. **Squash merge** на хостинге — один commit на `main` вместо merge commit: линейность ценой потери topology ветки. **Rebase-команды** любят прямые линии; **merge-команды** сохраняют точки интеграции. В больших repo `git log --first-parent` для читаемой mainline. Аудиты иногда проверяют hash chain на tampering.

---

## Comparison / Trade-offs

`git log` — полный DAG от стартовой revision, все parent'ы на merge. `git log --first-parent` — только first parent на каждом merge, скрывает детали side-branch до merge, читается как одна линия. `git log --graph --all` — все ветки сразу, на шумных repo граф тяжёлый.

SHA-1 — default, сорок символов, миллиарды objects. SHA-256 — optional, шестьдесят четыре, для новых repo с прицелом на долгосрочную стойкость hash-алгоритма.

## Quick Reference

```
Commit: tree, parent(s), author, committer, message
Root:   без parent
Merge:  2+ parents
Graph:  DAG — directed, acyclic
Reachability: parent'ы от HEAD/refs
Hash:   integrity + deduplication + global ID
```

---

## Key Takeaways

1. Object ID = **hash content** (SHA-1 default; SHA-256 optional).
2. Commit'ы связаны **parent** pointers в **DAG**.
3. **Root** — без parent; **merge** — несколько parent'ов.
4. **HEAD** задаёт visible history через **reachability**.
5. Hash'и — **integrity** и deduplication **blob/tree**.
6. `git log --graph` — визуализация DAG.
7. Rebase/cherry-pick → **новые hash'и** (parent/tree/metadata меняются).
