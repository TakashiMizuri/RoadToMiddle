# git show и git blame — краткая выжимка

> `0.1.7` · ~8 мин на повторение

## Главная идея

Git — **forensic database**: каждый commit хранит snapshot + metadata. **`git show`** раскрывает один объект (commit, tag, файл на revision). **`git blame`** показывает, какой commit **последним** изменил каждую строку файла. Вместе они отвечают: *кто, когда и что именно* привнёс в код.

## Ключевые пункты

- `git show <hash>` — metadata + patch (diff к parent).
- `git show <rev>:path/file` — содержимое файла **без checkout** (snapshot на момент rev).
- `git blame file` — построчная привязка к commit + author + date.
- `git blame -L 10,20 file` — только строки 10–20 (экономит время на больших файлах).
- `-C` / `-M` — отслеживание **скопированного/перемещённого** кода после refactor.
- `git blame -w` — игнорировать commits, где менялся только whitespace.
- `git log -S "text"` — **pickaxe**: commits, где **количество** вхождений строки изменилось (появление/удаление).
- Forensic workflow: **blame строку → show commit → (опционально) log -S**.

## Самые важные примеры

**Найти автора строки 847:**
```bash
git blame -L 847,847 -- src/Services/AuthService.cs
git show <hash-from-blame>
```

**Файл на tag:**
```bash
git show v2.0.0:appsettings.json
```

**Когда добавили symbol:**
```bash
git log -S "ValidateToken" --oneline -- src/
```

## «Под капотом» в одном абзаце

Blame **не хранит** per-line данные в commits — алгоритм **вычисляет** их, идя назад по diffs и blobs. `git show` через `rev-parse` разрешает hash в commit/tag/blob/tree и либо строит patch к parent, либо выводит raw content (`rev:path`).

## Частые ошибки

- Считать blame «виноватостью» — это last modifier, не обязательно introducer бага.
- Blame после mass reformat без `-w`.
- Refactor без `-C -M` — blame указывает на move commit, не на оригинал.
- Pickaxe по слишком общей строке без `-- path/`.

## Мнемоника

> **Show** = «покажи этот snapshot». **Blame** = «кто последний трогал строку». **Pickaxe (-S)** = «когда эта строка впервые появилась/исчезла».
