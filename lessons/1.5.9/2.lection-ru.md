# Режимы GC: Workstation vs Server

> Roadmap: `1.5.9` · Node: `1.5` — C#: .NET Runtime и память · Depth: понимание

## Learning Objectives

После этого урока ты сможешь:

## Why This Matters

Garbage collector — не один застывший алгоритм, а **политический движок**, настроенный под роль машины. Ноутбук с WPF-редактором чувствителен к **отзывчивости UI**: длинные stop-the-world паузы ощущаются как зависание. Linux-VM с ASP.NET Core на 16 ядрах оптимизирует **совокупный throughput**: короткая пауза одного потока может быть приемлема, если остальные ядра продолжают обрабатывать запросы.

Неверный режим не ломает корректность, но **тихо ухудшает performance**. Типичные истории: API в production на **workstation GC** и скачки p99 под нагрузкой; или **server GC** в одноядерном контейнере — лишние GC-потоки конкурируют за один CPU. Middle-разработчик должен знать, какой режим активен, как его менять и какие симптомы указывают на mismatch.

Урок опирается на generations, LOH и mark-sweep из `1.5.6`–`1.5.8`. **Режим GC** определяет *сколько* коллекторов работает и *как* они чередуются с потоками приложения; поколения — *что* и *как часто* собирается.

## Core Concepts

### Два основных flavor

**.NET предлагает workstation и server GC** (плюс варианты вроде **DATAS GC** в .NET 8+ для очень нагруженных серверов):

**Workstation GC** рассчитан на **интерактивное приложение** на машине, где пользователь замечает паузы. Исторически — **короче паузы**, иногда ценой менее агрессивного параллелизма. В современном .NET обычно **один logical heap на процесс** (если не задан `GCHeapCount` или лимиты контейнера), с **background/concurrent** сборкой Gen2 для снижения блокировок.

**Server GC** рассчитан на **выделенный серверный процесс** на многопроцессорной машине. По умолчанию — **отдельный managed heap на logical processor**, параллельные **GC threads**, цель — **выше throughput** на allocation-heavy API. Пауза на отдельном потоке может быть **длиннее**, но суммарная пропускная способность процесса часто растёт.

Оба режима — generational tracing collectors; pauses не исчезают полностью. Разница в **параллелизме**, **layout heap'ов** и **дефолтном tuning** latency vs throughput.

### Как узнать активный режим

```csharp
Console.WriteLine($"Server GC: {GCSettings.IsServerGC}");
Console.WriteLine($"Latency mode: {GCSettings.LatencyMode}");
```

В опубликованном ASP.NET — **`runtimeconfig.json`**:

```json
{
  "runtimeOptions": {
    "configProperties": {
      "System.GC.Server": true
    }
  }
}
```

Переменные окружения: `DOTNET_gcServer=1` / `0`.

В `.csproj` ASP.NET Core часто:

```xml
<ServerGarbageCollection>true</ServerGarbageCollection>
```

Консольные приложения часто по умолчанию — workstation. **Измеряй** — defaults меняются между шаблонами и версиями SDK.

### Pause time vs throughput

При **blocking collection** потоки, которым нужен consistent heap view, могут **останавливаться** (детали зависят от поколения и фазы background GC). Workstation стремится к коротким паузам для UI. Server GC может принять более длинную координированную паузу, если **все ядра** быстрее заканчивают сборку и процесс возвращается к насыщенному throughput.

Для **ASP.NET Core** Gen2-пауза может задержать множество request threads — поэтому server GC — типичный default для multi-core серверов.

Для **desktop**, игр, real-time audio иногда включают **workstation** или **`GCLatencyMode.LowLatency`** / **`SustainedLowLatency`** на критических участках — ценой памяти или отложенной полной сборки.

### Background и concurrent collection

**Concurrent/background GC** позволяет части работы Gen2 идти **параллельно** с application threads, сокращая stop-the-world. В app code это обычно не переключается отдельно — часть runtime flavor и версии. В профайлере смотри concurrent mark threads; отличай **Gen0/1** (часто sub-ms — low ms) от **Gen2/LOH** (десятки ms и больше на больших heap).

### Контейнеры и облако

При **CPU limit** в Docker server GC threads могут **конкурировать** с app threads за доли ядра — иногда **workstation** или меньший **`GCHeapCount`** выигрывает. .NET учитывает **cgroup memory limits** (`DOTNET_GCHeapHardLimit`).

Правило: **GC под реальные vCPU и workload**. Multi-core API → server GC. CLI, sidecar, UI → workstation. Сомневаешься — **load test**.

## Under the Hood

У **server GC** каждый heap имеет **index**; объект остаётся на heap'е аллокации (без обычного cross-heap move для SOH), что даёт locality, но **imbalance** при неравномерной аллокации с одного потока.

**GC threads** — runtime threads, не thread pool. В server mode в debugger видны **`/.NET GC`**, **`GC Server #`**. Gen0/1 segments и LOH — per heap.

**`GCSettings.LatencyMode`**: `Batch`, `Interactive`, `LowLatency`, `SustainedLowLatency`, `NoGCRegion` — комбинируется с flavor; долгий `SustainedLowLatency` + server GC без мониторинга памяти опасен.

## Common Mistakes & Anti-patterns

**F5 ≠ production** — debug, runtimeconfig, env overrides.

**Server GC на одноядерном pod без теста.**

**Гонка за нулевыми паузами вместо снижения аллокаций.**

**Глобальный `LowLatency` без контроля памяти.**

## Comparison / Trade-offs

| Аспект | Workstation | Server |
|--------|-------------|--------|
| Цель | Interactive | Multi-core server |
| Heaps | Обычно один | По CPU (default) |
| Паузы | Короче на поток | Выше parallel throughput |
| Default | Console/UI | ASP.NET Core |

При жёстком p99 SLO — профилируй Gen2; pooling и меньше LOH часто важнее flip режима.

## Key Takeaways

- Workstation — **latency**; server — **throughput** на many-core.
- ASP.NET Core обычно **server GC** — проверяй runtime.
- Режим не заменяет **allocation discipline**.
- Малые CPU quotas в K8s — benchmark оба варианта.
- Валидируй **dotnet-counters** + load test.
