# Структура HTTP-сообщения: start line, headers, body

> Roadmap: `0.3.3` · Node: `0.3` — HTTP и Web · Depth: **глубоко**

## Learning Objectives

После этой лекции ты сможешь:

- Разложить HTTP **request** и **response** на start line, headers и optional body.
- Читать и писать raw HTTP/1.1 с корректными **CRLF** и grammar headers.
- Объяснить, как **Content-Length** и **chunked transfer encoding** ограничивают body.
- Отличить **имена headers** (case-insensitive) от values и знать стандартные headers.
- Описать, как Kestrel и browser парсят bytes в `HttpRequest` / `HttpResponse`.
- Различить **hop-by-hop** и **end-to-end** headers и роль proxies.
- Отлаживать malformed messages: нет blank line, wrong length, chunked errors.

---

## Why This Matters

В `0.3.1` — цикл request-response; в `0.3.2` — URL. Теперь «конверт»: **какие bytes идут по wire** после TCP/TLS. Frameworks скрывают это, но production требует raw view: `curl -v`, Fiddler, Wireshark, когда client шлёт неверный `Content-Length` и server ждёт body.

HTTP/1.1 — простой text; HTTP/2+ — framed binary. Grammar HTTP/1.1 нужна для debugging, mapping на HTTP/2 **pseudo-headers** (`:method`, `:path`), proxy к backend. Middle dev настраивает limits, streaming, gzip, CORS — всё это headers и body.

Сообщение без **blank line** после headers не парсится. Body без **Content-Length** или **chunked** — truncated read или security issues. Methods и status codes (`0.3.4`, `0.3.6`) сидят в start line; этот урок — chassis.

## Core Concepts

### Три части сообщения

HTTP/1.1 request или response:

1. **Start line** (одна строка).
2. **Header section** — zero or more lines, завершение пустой строкой.
3. **Body** (optional) — bytes, длина из headers/encoding.

```
start-line CRLF
header-name: header-value CRLF
...
CRLF
[message body]
```

**CRLF** = `\r\n`. Strict parsers ожидают `\r\n`; lone `\n` иногда «работает», но нарушает spec.

### Request start line

```
METHOD request-target HTTP-version
```

Пример:

```
GET /api/products?page=2 HTTP/1.1
```

**Method** — `0.3.4`. **Request-target** — path + query (`0.3.2`). **Version** — обычно `HTTP/1.1`.

Далее headers. Minimum HTTP/1.1: **Host**. Blank line. **GET** чаще без body; **POST**/`PUT`/`PATCH` — с body.

```
GET /health HTTP/1.1
Host: localhost:5000
Accept: application/json

```

Пустая строка после headers — сигнал parser'у: body или конец.

### Response start line

```
HTTP-version status-code reason-phrase
```

```
HTTP/1.1 200 OK
```

**Status code** — `0.3.6`. Reason phrase информационна; client смотрит на число. Headers, optional body.

```
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
Content-Length: 18

{"status":"ok"}
```

### Headers

`name: value`. Names **case-insensitive**. Values — одна line per header в HTTP/1.1.

Headers несут: **Content-Type**, **Content-Encoding** (gzip), **Cache-Control**, **ETag** (`0.3.7`), **Authorization**, **Connection**, custom (`X-Correlation-ID`).

ASP.NET Core: `IHeaderDictionary`; response headers в middleware/`Results`.

Duplicate names: `Set-Cookie` — несколько lines; некоторые объединяют comma-separated.

### Body: когда и сколько bytes

Не у каждого message есть body. **HEAD** — без body в response. **204**, **304** — без body.

Когда body есть, receiver должен знать конец:

**Content-Length: N** — ровно N bytes после blank line. Типично для JSON API.

**Transfer-Encoding: chunked** — chunks с hex size; terminator `0` chunk. Streaming, когда length unknown.

Не отправляй **оба** Content-Length и chunked на HTTP/1.1. HTTP/2 — DATA frames (`0.3.13`).

**Content-Type** интерпретирует bytes: `application/json`, `multipart/form-data` (`0.3.11`).

### Request vs response

Общая grammar headers + body. Request требует **Host**; response — status line вместо method line.

## Under the Hood

### State machine parser (HTTP/1.1)

Kestrel `HttpParser` типично:

1. Bytes до `\r\n` → start line.
2. Header lines до empty `\r\n`.
3. `Transfer-Encoding: chunked` → chunked decoder.
4. Иначе `Content-Length` → read N bytes.
5. Иначе body length 0 по rules.
6. Application handler; keep-alive → next message (`0.3.14`).

`Content-Length: 100`, client шлёт 50 → wait/timeout. 1000 при limit → **413 Payload Too Large** (`MaxRequestBodySize`).

### Chunked encoding

```
Transfer-Encoding: chunked

1a
{"id":1,"name":"Alice"
0

```

Chunk: **hex size** CRLF data CRLF. Final `0\r\n\r\n`. Streaming response без total length.

### Hop-by-hop vs end-to-end

**End-to-end** (`Authorization`, `Content-Type`) — до final server. **Hop-by-hop** (`Connection`, `Transfer-Encoding`) — один hop; proxy strip/consume. `Connection: close` — новый TCP каждый request.

### HTTP/2 mapping

HPACK compressed headers; **frames**. Pseudo-headers: `:method`, `:path`, `:status`. Body в DATA frames. Модель headers + body та же (`0.3.13`).

### Security

**Request smuggling** — conflicting Length/chunked на proxy. **Header injection** — CRLF в values. **Decompression bombs** — limit sizes.


```csharp
app.MapPost("/echo", async (HttpRequest req) =>
{
    using var reader = new StreamReader(req.Body);
    return Results.Text(await reader.ReadToEndAsync());
});
```

## Examples

**JSON POST** — start line + `Content-Type` + `Content-Length` + JSON body.

**204 No Content** — headers + blank line, zero body bytes.

**gzip** — `Content-Encoding: gzip`; body compressed binary.

## Common Mistakes & Anti-patterns

Нет **Host** в hand-crafted request → 400.

`\n` вместо **CRLF** — strict parsers fail.

Body без **Content-Type** awareness — parse errors.

Wrong **Content-Length** — breaks keep-alive next message.

Upload whole file в memory — stream `Request.Body` с limits.

---

## Production & Real-World Notes

Kestrel **limits**: MaxRequestBodySize, header count/line size — DoS protection.

Middleware order меняет headers/body (compression, exception handler).

**Problem Details** RFC 7807 для errors.

Не логировать **Authorization**/cookies в info logs.

---

## Comparison / Trade-offs

Content-Length — простой JSON; chunked — streaming unknown size.

HTTP/1.1 text vs HTTP/2 binary — readable vs multiplexing.

Large JWT в header vs session — header size limits.

## Quick Reference

```
REQUEST:  METHOD /path HTTP/1.1 \r\n + headers \r\n\r\n + [body]
RESPONSE: HTTP/1.1 CODE Reason \r\n + headers \r\n\r\n + [body]
```

Mandatory HTTP/1.1 request: **Host**.

---

## Key Takeaways

- Message = **start line + headers + blank line + optional body**.
- **CRLF**; empty line ends headers.
- Body via **Content-Length** or **chunked**.
- Header names case-insensitive; **Content-Type** — interpretation body.
- Parser = **state machine**; wrong length → hang/truncation.
- HTTP/2 — pseudo-headers + DATA frames, та же семантика.
