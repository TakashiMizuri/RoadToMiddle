# Proxy vs Reverse Proxy vs Load Balancer

> Roadmap: `0.4.8` · Node: `0.4` — Networking · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Define **forward proxy**, **reverse proxy**, and **load balancer** by where they sit in the traffic path and who they serve.
- Trace an HTTP request from browser through a reverse proxy to multiple Kestrel instances behind it.
- Explain why production ASP.NET Core apps typically listen on an internal port while the world sees port 443.
- Compare **Layer 4 (L4)** vs **Layer 7 (L7)** load balancing and when each is appropriate.
- Describe common headers (`X-Forwarded-For`, `X-Forwarded-Proto`) and why the backend must trust them carefully.
- Evaluate trade-offs between nginx, cloud ALB, and API gateways for a given architecture.

---

## Why This Matters

Lesson `0.4.7` established that only certain ports — usually 80 and 443 — should face the public internet, while your API process might listen on 5000 or 8080 inside a private network. Something must terminate TLS on 443, route requests to the correct internal service, and optionally spread load across multiple instances. That "something" is almost always a **reverse proxy** or **load balancer** — often both roles in one product.

Confusing these terms causes real bugs. A developer who thinks nginx is a "proxy" in the corporate VPN sense might misconfigure outbound traffic. A team that puts a reverse proxy in front of Kestrel but forgets `ForwardedHeaders` middleware will log wrong client IPs and generate broken HTTPS redirect URLs. A ops engineer who chooses an L4 balancer for path-based routing will wonder why `/api` and `/` cannot go to different backends.

As a Middle fullstack developer you will configure `launchSettings.json` locally, deploy behind IIS/nginx/ALB in production, and debug 502 Bad Gateway errors that sit squarely in this layer. This lesson gives you a mental model sharp enough to read architecture diagrams and nginx configs without guessing.

## Core Concepts

### Forward Proxy: The Client's Representative

A **forward proxy** (often simply "proxy") sits **close to the client** — historically on a corporate network — and acts **on behalf of clients** talking to the outside world. When your browser is configured to use `proxy.corp.example:8080`, HTTP requests go to the proxy first; the proxy forwards them to `https://github.com` and returns the response. The **origin server sees the proxy's IP**, not yours.

Forward proxies serve several purposes. **Policy enforcement** blocks forbidden sites or scans for malware. **Caching** stores copies of frequently requested external resources to save bandwidth. **Anonymity** (or at least IP hiding) applies when the proxy is the only visible client to remote servers. **Geo bypass** is a consumer use case for VPN-like proxies.

The client **must be configured** to use a forward proxy (browser settings, `HTTP_PROXY` environment variable). The origin server typically **does not know** it is behind a proxy unless headers like `Via` reveal it. Forward proxies are about **outbound** access control from an organization outward.

This is **not** what terminates HTTPS for your React app in production. When someone says "we need a proxy in front of the API," they almost always mean **reverse** proxy.

### Reverse Proxy: The Server's Front Door

A **reverse proxy** sits **close to the server** — in front of one or more backend applications — and acts **on behalf of servers** facing clients. The browser connects to `https://api.example.com` (the reverse proxy). The reverse proxy opens a **separate** connection to `http://10.0.1.5:8080` where Kestrel runs. The client **does not know** how many backends exist or which one handled the request.

Reverse proxies solve problems your application should not own. **TLS termination** decrypts HTTPS at the edge so Kestrel can speak plain HTTP on a trusted internal network — simpler certificate management and CPU offload. **Routing** sends `/api/*` to the API cluster and `/*` to static files or a CDN origin. **Compression** (gzip, brotli) reduces payload size. **Rate limiting** and **WAF** rules protect before traffic hits your code. **Connection pooling** to backends reduces handshake overhead.

In ASP.NET Core, when hosted behind a reverse proxy, enable forwarded headers so the app knows the original scheme and client IP:

```csharp
app.UseForwardedHeaders(new ForwardedHeadersOptions
{
    ForwardedHeaders = ForwardedHeaders.XForwardedFor | ForwardedHeaders.XForwardedProto
});
```

Without this, `Request.Scheme` might be `http` even though the user used HTTPS — breaking cookie security flags and redirect URLs.

### Load Balancer: Distributing Work

A **load balancer** distributes incoming requests across **multiple backend instances** to improve capacity, availability, and fault tolerance. When one API pod crashes, others keep serving; when traffic spikes, you add instances rather than melting one server.

Conceptually, a load balancer can be **dedicated hardware**, a **cloud managed service** (AWS ALB/NLB, Azure Load Balancer, GCP LB), or **software** (nginx upstream, HAProxy) — and many products combine **reverse proxy + load balancing** in one.

**Health checks** probe backends (`GET /health`) and stop sending traffic to failing nodes. **Session affinity (sticky sessions)** sends the same client to the same backend — sometimes needed for legacy in-memory session state, usually avoided in stateless APIs where JWT or Redis holds session data.

Load balancers operate at different OSI layers. **L4** balances based on IP and TCP/UDP port — fast, protocol-agnostic, no visibility into HTTP paths. **L7** understands HTTP: route `/orders` to service A and `/users` to service B, inspect cookies, insert headers. ASP.NET Core microservices behind an L7 ALB with path-based rules is a standard pattern.

### How They Relate — Not Mutually Exclusive

Real architectures stack roles:

```
Browser → [CDN] → [L7 Load Balancer / Reverse Proxy] → [App instance 1..N]
                              ↓
                    TLS terminate, route, balance
```

A single **nginx** can be reverse proxy (hides Kestrel), load balancer (`upstream` with multiple servers), and TLS terminator. An **AWS Application Load Balancer** is L7 load balancer + reverse proxy from the internet's perspective. A **forward proxy** in the corporate office is unrelated to this path unless employees' outbound npm installs go through it.

The diagnostic question: **Whose behalf?** Forward proxy = client's. Reverse proxy / LB = server's.

## Under the Hood

### Connection Flow: HTTPS to Three Kestrel Instances

Consider nginx as reverse proxy and load balancer:

```nginx
upstream api_backend {
    server 10.0.1.10:8080;
    server 10.0.1.11:8080;
    server 10.0.1.12:8080;
}

server {
    listen 443 ssl;
    server_name api.example.com;

    location / {
        proxy_pass http://api_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

**Step by step:**

1. Client completes **TLS handshake** with nginx (cert for `api.example.com`).
2. Client sends encrypted HTTP request inside TLS: `GET /orders HTTP/1.1`.
3. nginx decrypts, selects upstream server (round-robin by default), opens **new TCP connection** to e.g. `10.0.1.11:8080`.
4. nginx forwards HTTP (often plain text on private network): adds `X-Forwarded-For: client-ip`, `X-Forwarded-Proto: https`.
5. Kestrel processes request, returns response to nginx.
6. nginx encrypts response, sends to client, may reuse TLS session / keep-alive.

Two separate TCP connections: **client ↔ nginx** and **nginx ↔ Kestrel**. Latency adds one hop, but TLS and balancing buy operational wins.

### L4 vs L7 Decision Internals

**Network Load Balancer (L4)** sees packets: source/dest IP, ports, TCP flags. It might use **Direct Server Return** or NAT to forward. It cannot route based on `Host: api.example.com` vs `Host: www.example.com` on the same IP without SNI at TLS layer — NLB can pass through TLS without terminating.

**Application Load Balancer (L7)** terminates TLS (optional), parses HTTP, matches listener rules: host header, path pattern, query. It can inject headers, redirect HTTP→HTTPS, and attach AWS WAF. CPU cost is higher than L4; flexibility is much higher.

For a single monolithic ASP.NET app, L7 ALB with health checks on `/health` is the common cloud default. For extreme throughput on plain TCP (gaming, custom binary protocol), L4 wins.

### Trust Boundaries for Forwarded Headers

`X-Forwarded-For` is trivial to spoof if clients connect directly to Kestrel. Security model: **only the reverse proxy** may reach Kestrel; Kestrel **trusts** headers from that proxy's IP range via `KnownNetworks` / `KnownProxies` in `ForwardedHeadersOptions`. Misconfiguration — allowing the world to send fake `X-Forwarded-Proto: https` — breaks security assumptions.

## Syntax / Commands / API

### nginx reverse proxy (minimal)

```nginx
location /api/ {
    proxy_pass http://127.0.0.1:5000/;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection keep-alive;
}
```

### ASP.NET Core forwarded headers

```csharp
builder.Services.Configure<ForwardedHeadersOptions>(options =>
{
    options.ForwardedHeaders = ForwardedHeaders.XForwardedFor
        | ForwardedHeaders.XForwardedProto;
    options.KnownNetworks.Clear();
    options.KnownProxies.Add(IPAddress.Parse("10.0.0.1")); // your LB
});
// Must be early in pipeline:
app.UseForwardedHeaders();
```

### curl through proxy (forward proxy example)

```bash
curl -x http://proxy.corp:8080 https://example.com
```

### Health check endpoint (for LB)

```csharp
app.MapGet("/health", () => Results.Ok(new { status = "healthy" }));
```

## Examples

### Local Dev vs Production

**Local:** React `vite` on `:5173`, API on `:5000` — no reverse proxy; browser talks directly (CORS preview in `0.4.10`).

**Production:** User hits `https://app.example.com` → CloudFront or ALB on 443 → nginx or direct to Kestrel pods on `:8080` internal. Single public entry, horizontal scale behind it.

### 502 Bad Gateway

nginx logs `upstream timed out` while Kestrel is down or listening on wrong port. The reverse proxy could not get a valid response from backend. Fix the app or upstream config — not the client's browser.

## Common Mistakes & Anti-patterns

**Calling every intermediary a "proxy"** without forward vs reverse distinction — leads to wrong config files and security groups.

**Forgetting `UseForwardedHeaders`** — HTTPS links generated as `http://`, secure cookies mis-set, rate limiting by wrong IP.

**Sticky sessions by default** — masks uneven load; hides stateless design benefits. Prefer shared session store.

**TLS terminate at LB and also at Kestrel** — double encryption unnecessary unless compliance requires end-to-end encryption (mTLS between tiers).

**Trusting all `X-Forwarded-*` from any IP** — header spoofing vulnerability.

## Production & Real-World Notes

**Blue-green / rolling deploys:** LB drains connections from old version before removing instances. **Circuit breakers** at gateway level stop retry storms to failing services.

**WebSockets / SignalR:** L7 proxy must support connection upgrade (`Upgrade: websocket`); idle timeouts must exceed heartbeat interval.

**Observability:** Access logs at reverse proxy are the source of truth for external traffic patterns; correlate with app logs via request ID headers (`X-Request-Id`).

Kubernetes **Ingress** resource configures reverse proxy + LB rules declaratively; **Service** type LoadBalancer provisions cloud LB.

## Comparison / Trade-offs

| Component | Serves | Typical placement | Knows HTTP? |
|-----------|--------|-------------------|-------------|
| Forward proxy | Client | Corporate egress | Often yes |
| Reverse proxy | Server | DMZ / edge | Yes |
| L4 load balancer | Server | Edge or internal | No |
| L7 load balancer | Server | Edge | Yes |

**nginx on VM** vs **managed ALB:** nginx = full control, you patch and scale; ALB = managed HA, integrates with WAF/autoscaling, per-hour cost.

**Single reverse proxy** vs **API gateway:** gateway adds auth, throttling, request transformation, often service discovery — heavier than plain nginx for a monolith.

## Quick Reference

```
Forward:  Client → [Proxy] → Internet → Origin server
Reverse:  Client → Internet → [Reverse Proxy/LB] → Backend(s)
```

**Headers to know:** `X-Forwarded-For`, `X-Forwarded-Proto`, `Host`, `X-Request-Id`.

**502** — bad gateway (proxy couldn't reach healthy backend). **504** — gateway timeout.

---

## Key Takeaways

- **Forward proxy** = client-side outbound; **reverse proxy** = server-side front door.
- **Load balancer** spreads traffic across instances; often combined with reverse proxy.
- **L4** = IP/port; **L7** = HTTP-aware routing and TLS termination.
- Production **443 → reverse proxy → internal Kestrel port** is the standard pattern from `0.4.7`.
- Configure **forwarded headers** and **trust only your proxy IPs**.
- Read architecture diagrams by asking: who initiates, who terminates TLS, how many hops?
