## Answers

### 1. [Recall]
**Answer:** Workstation — typically **one managed heap** per process. Server GC — **one heap per logical processor** (by default).
**Why:** Parallel heaps allow parallel collection on multi-core servers.

### 2. [Recall]
**Answer:** **`GCSettings.IsServerGC`** (also `GC.GetGCMemoryInfo().HeapCount` hints at server layout).
**Why:** Runtime API reflects effective configuration after env + config merge.

### 3. [Recall]
**Answer:** **`.csproj`** — `<ServerGarbageCollection>true</ServerGarbageCollection>`; also **`runtimeconfig.json`** / template for publish output.
**Why:** MSBuild property flows into generated runtime config.

### 4. [Understanding]
**Answer:** Web APIs are **throughput-oriented** on **multi-core** servers; server GC uses **parallel heaps/threads** to sustain allocation-heavy request handling.
**Why:** Many concurrent thread-pool threads benefit from parallel collection vs single-heap workstation tuning.

### 5. [Understanding]
**Answer:** Server GC spins **multiple GC threads/heaps** that **contend** for fractional CPU; overhead may exceed benefit.
**Why:** GC threads need CPU to collect; on 250m core, workstation may be simpler.

### 6. [Understanding]
**Answer:** Portions of **Gen2** collection run **concurrently** with app threads, reducing **stop-the-world** duration.
**Why:** Mark phase can interleave; not all GC work blocks every allocator.

### 7. [Application]
**Answer:** **Environment variable** typically overrides for that launch context — effective mode depends on precedence rules; verify with `IsServerGC` at runtime.
**Why:** .NET merges config sources; don't assume without measuring.
**Common mistake:** Assuming csproj always wins over env.

### 8. [Application]
**Answer:** Check **GC mode** and **Gen2/LOH pause frequency**; profile **allocations** (strings, LINQ, LOH).
**Why:** Pauses come from collections triggered by allocation pressure, not mode alone.

### 9. [Analysis]
**Answer:** **False.** Single-core, interactive, or memory-constrained workloads may favor workstation or tuned heap count.
**Why:** Server GC targets parallel hardware throughput.

### 10. [Analysis]
**Answer:** **Yes — memory pressure risk.** Gen2 may be deferred → higher heap size, eventual large pause or OOM.
**Why:** Low latency modes trade collection for responsiveness temporarily.

### 11. [Recall]
**Answer:** Examples: **`gc-heap-size`**, **`gen-2-gc-count`**, **`% time in gc`**, **`alloc-rate`**.
**Why:** dotnet-counters System.Runtime provider exposes these.

### 12. [Application]
**Answer:** Examples: **single-core VM**, **desktop agent on user PC**, **short-lived CLI**, **latency-sensitive single-threaded** tool with proven workstation win in benchmarks.
**Why:** Must match hardware and SLO with measurement.
