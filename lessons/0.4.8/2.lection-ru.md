# Proxy vs Reverse Proxy vs Load Balancer

> Roadmap: `0.4.8` · Node: `0.4` — Сети · Depth: **глубоко**

## Learning Objectives

После этого урока ты сможешь:

- Определить **forward proxy**, **reverse proxy** и **load balancer** по месту в traffic path и по тому, **кого** они обслуживают.
- Проследить HTTP request от browser через reverse proxy к нескольким Kestrel instances.
- Объяснить, почему production ASP.NET Core слушает internal port, а мир видит 443.
- Сравнить **L4** vs **L7** load balancing и когда что уместно.
- Описать headers `X-Forwarded-For`, `X-Forwarded-Proto` и почему backend должен **осторожно** им доверять.
- Оценить trade-offs nginx, cloud ALB и API gateway для архитектуры.

## Why This Matters

Урок `0.4.7`: наружу открыты 80/443, API process слушает 5000/8080 внутри private network. Кто-то должен terminate TLS на 443, route requests к internal service и optionally spread load. Это **reverse proxy** или **load balancer** — часто обе роли в одном продукте.

Путаница терминов → real bugs. Dev думает nginx = corporate VPN proxy → misconfigure outbound. Team ставит reverse proxy, но забывает `ForwardedHeaders` → wrong client IPs и broken HTTPS redirects. Ops выбирает L4 для path-based routing → `/api` и `/` не разделить.

Middle fullstack настраивает `launchSettings.json` locally, deploy за IIS/nginx/ALB, debug'ит 502 Bad Gateway в этом слое. Этот урок даёт mental model для architecture diagrams и nginx configs.

## Core Concepts

### Forward Proxy: представитель клиента

**Forward proxy** («proxy») сидит **близко к client** — corporate network — и действует **от имени clients** к outside world. Browser с `proxy.corp.example:8080` шлёт requests на proxy; proxy forward'ит на `https://github.com`. **Origin server видит IP proxy**, не твой.

Цели: **policy** (block sites), **caching** external resources, **anonymity**/IP hiding, consumer **geo bypass**. Client **must be configured** (`HTTP_PROXY`, browser settings). Origin **не знает** о proxy, если headers не выдают.

Это **не** то, что terminate HTTPS для React app в production. «Proxy перед API» почти всегда = **reverse** proxy.

### Reverse Proxy: парадная дверь сервера

**Reverse proxy** **близко к server** — перед backend apps — действует **от имени servers** для clients. Browser → `https://api.example.com` (reverse proxy). Proxy открывает **отдельное** connection к `http://10.0.1.5:8080` (Kestrel). Client **не знает** сколько backends и кто обработал.

Reverse proxy решает то, что app не должна тащить: **TLS termination**, **routing** (`/api/*` vs static), **compression**, **rate limit/WAF**, **connection pooling** к backends.

ASP.NET Core за reverse proxy:

```csharp
app.UseForwardedHeaders(new ForwardedHeadersOptions
{
    ForwardedHeaders = ForwardedHeaders.XForwardedFor | ForwardedHeaders.XForwardedProto
});
```

Без этого `Request.Scheme` = `http` при user HTTPS → ломает cookies и redirects.

### Load Balancer: распределение нагрузки

**Load balancer** распределяет requests across **multiple instances** — capacity, availability, fault tolerance. Один pod упал — others serve; spike — scale out.

LB может быть hardware, **managed cloud** (ALB/NLB, Azure LB), **software** (nginx upstream, HAProxy). Многие продукты = **reverse proxy + LB**.

**Health checks** (`GET /health`) убирают failing nodes из rotation. **Sticky sessions** — same client → same backend; нужно редко (legacy in-memory session); в stateless API избегать (JWT, Redis).

**L4** — IP + TCP/UDP port, fast, protocol-agnostic, no HTTP paths. **L7** — HTTP: path routing, cookies, headers. Microservices за L7 ALB — standard.

### Как они связаны

```
Browser → [CDN] → [L7 LB / Reverse Proxy] → [App 1..N]
```

Один **nginx** = reverse proxy + LB (`upstream`) + TLS terminator. **AWS ALB** = L7 LB + reverse proxy снаружи. **Forward proxy** в office — про outbound npm, не про user traffic к вашему API.

Diagnostic: **Whose behalf?** Forward = client's. Reverse/LB = server's.

## Under the Hood

### Connection flow: HTTPS → три Kestrel

```nginx
upstream api_backend {
    server 10.0.1.10:8080;
    server 10.0.1.11:8080;
    server 10.0.1.12:8080;
}
server {
    listen 443 ssl;
    location / {
        proxy_pass http://api_backend;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

1. Client **TLS handshake** с nginx.
2. Encrypted `GET /orders` inside TLS.
3. nginx decrypts, pick upstream (round-robin), **new TCP** к `10.0.1.11:8080`.
4. Plain HTTP на private network + forwarded headers.
5. Kestrel responds → nginx encrypts → client.

**Два TCP connection:** client↔nginx и nginx↔Kestrel.

### L4 vs L7

**NLB (L4):** packets, IP/ports; может pass-through TLS (SNI). Не route по `/orders` vs `/users` без L7.

**ALB (L7):** optional TLS terminate, HTTP rules, WAF, redirects. Выше CPU cost, выше flexibility.

Monolith ASP.NET → L7 ALB + `/health` common. Extreme TCP throughput → L4.

### Trust boundaries

`X-Forwarded-For` spoof'ится, если client hits Kestrel directly. Model: **только reverse proxy** reaches Kestrel; trust headers **only from proxy IP** (`KnownProxies`). World + fake `X-Forwarded-Proto: https` = broken security.

## Syntax / Commands / API

```csharp
builder.Services.Configure<ForwardedHeadersOptions>(options =>
{
    options.ForwardedHeaders = ForwardedHeaders.XForwardedFor
        | ForwardedHeaders.XForwardedProto;
    options.KnownProxies.Add(IPAddress.Parse("10.0.0.1"));
});
app.UseForwardedHeaders(); // early in pipeline
```

```bash
curl -x http://proxy.corp:8080 https://example.com  # forward proxy
```

## Examples

**Local:** Vite `:5173`, API `:5000` — без reverse proxy (CORS в `0.4.10`).

**Production:** `https://app.example.com` → ALB:443 → Kestrel `:8080` internal.

**502:** nginx `upstream timed out` — backend down/wrong port.

## Common Mistakes & Anti-patterns

- «Proxy» без forward/reverse distinction.
- Забытый `UseForwardedHeaders`.
- Sticky sessions by default.
- Trust all `X-Forwarded-*` from any IP.

## Production & Real-World Notes

Blue-green: LB drains old instances. **WebSockets/SignalR:** `Upgrade` support, idle timeouts. **Ingress** в Kubernetes. Access logs на proxy = truth для external traffic.

## Comparison / Trade-offs

| | Serves | Knows HTTP? |
|---|--------|-------------|
| Forward proxy | Client | Often |
| Reverse proxy | Server | Yes |
| L4 LB | Server | No |
| L7 LB | Server | Yes |

nginx vs managed ALB: control vs managed HA. API gateway > nginx для auth/throttling/service mesh scenarios.

---

## Quick Reference

```
Forward:  Client → [Proxy] → Internet → Origin
Reverse:  Client → Internet → [Rev Proxy/LB] → Backend(s)
```

502 = bad gateway; 504 = gateway timeout.

---

## Key Takeaways

- **Forward** = client outbound; **reverse** = server front door.
- **LB** = multiple instances; often + reverse proxy.
- **443 → proxy → internal Kestrel** — pattern из `0.4.7`.
- **Forwarded headers** + trust only proxy IPs.
- **L4** IP/port; **L7** HTTP routing + TLS terminate.
