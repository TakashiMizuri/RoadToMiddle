# Answers: HTTP Status Codes

## Scoring

- **90–100%:** отлично — переходи к `0.3.7`
- **80–89%:** pass — повтори вопросы с ошибками
- **<80%:** вернись к `1.lection-eng.md`, `2.lection-ru.md`, секции Core Concepts и Under the Hood

## Answers

### 1. [Recall]
**Answer:** **4xx** (Client Error).

**Why:** Первая цифра 4 означает, что server считает request некорректным или неавторизованным с точки зрения client.

---

### 2. [Recall]
**Answer:** **201 Created** с header **Location**, указывающим URI нового resource.

**Why:** 201 семантически = resource born; Location — стандартный способ передать новый URI.

---

### 3. [Recall]
**Answer:** **301** — resource переехал на **новый URL** (redirect, permanent). **304** — **тот же URL**, representation **не изменилась**; conditional GET, **body нет**.

**Why:** 301 меняет canonical URI; 304 — revalidation без payload.

---

### 4. [Understanding]
**Answer:** Generic HTTP clients, caches и monitors ориентируются на **status line**, не на JSON. **200** трактуется как success → кэширование ошибок, ложные green metrics, отсутствие automatic retry/error handling.

**Why:** Status code — machine contract; body — application layer.

**Common mistake:** «Frontend всё равно парсит JSON» — ecosystem шире одного SPA.

---

### 5. [Understanding]
**Answer:** **401** — authentication missing/invalid («кто ты?»); client должен login/refresh token и retry. **403** — identity known, но action forbidden; retry с **теми же** credentials бесполезен.

**Why:** RFC semantics: 401 unauthenticated, 403 authenticated but denied.

---

### 6. [Understanding]
**Answer:** **304 Not Modified**, **response body отсутствует** (client использует cached copy).

**Why:** Conditional GET success without retransmitting representation.

---

### 7. [Understanding]
**Answer:** **409 Conflict** — optimistic concurrency / state conflict, не «not found» и не pure validation.

**Why:** Resource exists; concurrent updates clash — classic 409.

---

### 8. [Application]
**Answer:** **409 Conflict** (duplicate unique key / state conflict) или **422** (validation-style API). **409** предпочтительнее для unique constraint conflict; **422** если API treats duplicates as semantic validation.

**Why:** 400 слишком generic; команда должна быть consistent.

---

### 9. [Application]
**Answer:** **502 Bad Gateway** (proxy не получил valid upstream response).

**Why:** 500 — origin app; 502 — gateway layer; 503 — often origin says temporarily unavailable.

---

### 10. [Application]
**Answer:** **204 No Content** — semantic «success without body»; не 200 с empty body.

**Why:** 204 явно говорит clients/caches: body intentionally absent.

---

### 11. [Analysis]
**Answer:** **C** неверно — это описание **403**, не 401.

**Why:** 401 = auth problem; insufficient role after auth = 403.

---

### 12. [Analysis]
**Answer:** (1) **Pollutes 5xx error budget** и false on-call alerts. (2) Clients **retry 500** unnecessarily. (3) **Masks real bugs** среди validation noise. (4) Breaks **monitoring/SLO** separation 4xx vs 5xx.

**Why:** Validation — client fix (4xx), not server crash.

---

### 13. [Understanding]
**Answer:** **503 Service Unavailable** (temporary, node drained / not ready).

**Why:** Load balancers и orchestrators используют 503 для «не слать traffic сюда».

---

### 14. [Analysis]
**Answer:** **404** скрывает existence resource (security through obscurity) — attacker не знает, есть ли `/admin/reports`. **403** честнее для UX (logged-in user понимает «нет прав»), но ** confirms resource exists**.

**Why:** Threat model vs API clarity trade-off; document team policy.

---

## Model answers for practical sections

Create: **201** + `Location: /orders/{id}`.  
Get: **200** + JSON body.  
Delete: **204** (or **200** if API returns body — 204 preferred when no body).
