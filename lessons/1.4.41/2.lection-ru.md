# Сравнение sync-over-async fixes в legacy code

> Roadmap: `1.4.41` · Node: `1.4` — C# async · Depth: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Определить **sync-over-async** и **async-over-sync** и почему оба в legacy .NET.
- Сравнить fixes: **`await` refactor**, **`.GetResult()`**, **`Task.Run`**, **`IHostedService` bridge**.
- Ранжировать по **correctness**, **scalability**, **risk** в ASP.NET Core.
- Распознать **границы библиотек** без async signatures.
- Планировать **incremental migration** к async-all-the-way.
- Избегать **deadlock** с `SynchronizationContext` и `.Result`.

---

## Why This Matters

Legacy смешивает sync ADO.NET, SDK с `bool Save()`, UI callbacks и modern ASP.NET Core. Кто-то обернул `GetDataAsync().Result` в sync interface. Под нагрузкой Kestrel **blocked thread pool threads** → outages.

Middle не всегда greenfield. Наследуешь `ILegacyBillingClient.Charge(amount)` с blocking HTTP. Урок — **decision matrix**, не dogma.

## Core Concepts

### Определения

**Sync-over-async:** sync API **блокирует** на async — `.Result`, `.Wait()`, `.GetAwaiter().GetResult()`.

**Async-over-sync:** `async` оборачивает **sync blocking I/O** — fake async.

```csharp
await Task.Run(() => _legacySyncRepo.GetUser(id)); // async-over-sync
```

Sync-over-async блокирует thread на real async I/O (хуже). Async-over-sync жрёт pool без true async I/O.

### Fix 1: Async all-the-way (preferred)

Refactor chain на `async`/`await`. **Pros:** правильные threads, cancellation. **Cons:** ripple, breaking changes.

### Fix 2: Sync API остаётся — isolate

Для COM/old SDK — **bounded gate**, never from hot ASP.NET path if possible.

### Fix 3: Task.Run bridge

Quick unblock async controller. **Cons:** extra pool thread, sync I/O внутри всё ещё блокирует. **When:** short-term, low QPS.

### Fix 4: IHostedService для long sync init

Runtime path async; init отдельно.

### Fix 5: Dual API

`GetUserAsync` + `[Obsolete]` sync shim `GetUser` с `.GetResult()`.

### Deadlock vs starvation

UI/old ASP.NET: `.Result` + SyncContext → **deadlock**. ASP.NET Core: deadlock реже, **starvation** остаётся.

### Comparison

| Fix | Scalability | Effort |
|-----|-------------|--------|
| Async refactor | Best | High |
| Dual API | Good | Medium |
| Task.Run | Poor at scale | Low |
| .Result in request | Worst | None |

## Common Mistakes

- «В Core нет SyncContext — .Result OK» — starvation
- Task.Run вместо fix SDK
- Sync legacy на request path без bridge

---

## Quick Reference

```
Best: async refactor
Bridge: Task.Run (short, low QPS)
Never: .Result на hot request path
Dual API для migration
```

---

## Key Takeaways

- **Sync-over-async** блокирует; **async-over-sync** — fake async.
- **Preferred:** propagate async/await.
- **Task.Run** — tactical bridge.
- Core: меньше deadlock, не меньше starvation.
- **Dual API** — incremental migration.
