# Skip Links, Focus Visible, Color Contrast

> Roadmap: `0.5.12` · Node: `0.5` — HTML · Depth: понимание

## Learning Objectives

After this lesson you will be able to:

- Implement a **skip link** that moves keyboard focus to main content without affecting mouse-only users negatively.
- Style **focus indicators** with `:focus-visible` so keyboard users always see where focus is.
- Explain **WCAG contrast** requirements for text and non-text UI components at a practical level.
- Avoid common failures: removing outlines, low-contrast placeholder text, meaning by color alone.
- Connect skip links and focus rings to landmark structure (`<main id="main">`) from earlier HTML lessons.
- Verify contrast and focus with browser DevTools and automated checks.

## Why This Matters

Imagine tabbing through a enterprise dashboard: logo link, six nav items, search, language switcher, notification bell — before you reach the invoice table you came to edit. **Skip links** exist so keyboard and screen reader users can jump past repetitive chrome to **`#main`**. They are cheap to add, high impact, and expected on accessible sites. In `0.5.11` you managed focus in modals and custom widgets; skip links are the document-level counterpart — the first Tab stop on many government and SaaS apps.

**Focus visible** is the other half of the same story. Mouse users click targets; keyboard users depend on a clear **focus ring**. When CSS removes `outline: none` without a replacement, you fail **WCAG 2.4.7 Focus Visible** and frustrate anyone who tabs. Designers and developers negotiate focus styles that match brand **and** meet contrast minimums.

**Color contrast** determines whether text and icons remain readable for users with low vision, color deficiency, or screens in sunlight. It is measurable — not subjective — via contrast ratio formulas. Pairing contrast with focus and skip links completes a practical accessibility baseline before SEO metadata in `0.5.13`.

## Core Concepts

### Skip Links

A **skip link** is typically the first focusable element in the document — an anchor visually hidden until focused — targeting an id on main content:

```html
<body>
  <a class="skip-link" href="#main">Skip to main content</a>
  <header>...</header>
  <main id="main" tabindex="-1">...</main>
</body>
```

```css
.skip-link {
  position: absolute;
  left: -9999px;
  top: 0;
  z-index: 1000;
  padding: 0.5rem 1rem;
  background: #000;
  color: #fff;
}
.skip-link:focus {
  left: 0;
}
```

On **Activate** (Enter), the browser navigates to `#main` and moves **focus** to the target if it is focusable. Adding **`tabindex="-1"`** to `<main>` ensures the landmark receives programmatic focus after skip — otherwise focus might stay on the link while visual scroll jumps. Screen reader users hear they entered main content.

Skip links help **keyboard** users primarily; sighted mouse users rarely see them unless they Tab. Do not remove skip links because "nobody uses them" — analytics undercount keyboard-only workflows.

Multiple skip targets are valid on complex apps: "Skip to navigation", "Skip to search" — keep the first Tab stop the highest-value jump (usually main).

### Focus Visible

**`:focus`** matches any focused element. **`:focus-visible`** matches when the user agent decides focus should be ** visibly indicated** — typically keyboard Tab, not always mouse click (browser heuristics vary).

Recommended pattern:

```css
:focus {
  outline: none; /* only if replaced below */
}
:focus-visible {
  outline: 2px solid #005fcc;
  outline-offset: 2px;
}
```

Never leave interactive controls with **no** focus indicator. Custom buttons, links, and inputs need the same treatment. **Focus within** (`:focus-within`) helps composite fields — ring around label+input wrapper.

Modal focus traps from `0.5.11` still need visible focus **inside** the trap — users must see which button is active.

### Color Contrast Basics

WCAG defines **contrast ratio** between foreground and background (1:1 to 21:1). Common **Level AA** thresholds:

| Content | Minimum ratio |
|---------|----------------|
| Normal text (< 18pt or < 14pt bold) | **4.5:1** |
| Large text (≥ 18pt or ≥ 14pt bold) | **3:1** |
| UI components and graphical objects | **3:1** against adjacent colors |

**Level AAA** raises normal text to 7:1 — optional unless contract requires.

Contrast applies to **text**, **icons conveying information**, **form borders** essential to understanding, and **focus indicators** against adjacent background. Decorative graphics with no meaning are exempt.

Tools: Chrome DevTools **Accessibility pane** (contrast ratio), [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/), axe/Lighthouse.

**Do not rely on color alone** for state — success/error rows need text or icons with accessible names (`0.5.9` tables). Placeholder text often fails contrast and should not replace `<label>`.

### Bringing It Together

Landmarks structure the page: header/nav repeated, **`main`** holds unique content per route. Skip link + `id="main"` + `tabindex="-1"` wires keyboard entry. Inside main, `:focus-visible` styles keep traversal usable. Contrast on body text, buttons, and error messages keeps content readable. These three techniques are independent but complementary — fixing contrast does not replace skip links.

## Under the Hood

Browsers apply **:user preference** media queries: **`prefers-reduced-motion`** and high-contrast modes on some OSes alter defaults. **Forced colors** mode (Windows High Contrast) may override your colors — focus indicators must remain perceptible; avoid hard-coding only subtle box-shadow rings.

Focus visibility heuristics in Chromium: mouse click on button suppresses `:focus-visible` on some versions; keyboard Tab shows it. Test both input modalities.

Contrast calculation uses **relative luminance** of sRGB colors per WCAG formula — alpha compositing against background matters for text over images; semi-transparent gray on white often fails 4.5:1 unexpectedly.

Skip navigation uses **fragment navigation** + **focus management** — same primitives as in-page anchor links and SPA route focus moves.

## Common Mistakes & Anti-patterns

**No skip link on nav-heavy layouts** — long Tab sequences. Add one.

**Skip target without tabindex** — scroll without focus move confuses AT. Add `tabindex="-1"` to target landmark.

**outline: none everywhere** — WCAG failure. Replace with `:focus-visible`.

**Low-contrast disabled buttons** — if disabled state must be perceivable, maintain 3:1 for UI component boundary or provide text state.

**Color-only errors** — red border without message text fails multiple criteria.

## Production & Real-World Notes

Design tokens should include **focus ring** color meeting 3:1 against adjacent backgrounds. Document skip link in Storybook first Tab stop test.

Legal audits often test home + one form page: skip link, focus order, contrast on primary button and error text.

SPAs: re-insert skip link in layout shell; ensure `main` id stable across routes.

---

## Comparison / Trade-offs

**Visually hidden skip vs always visible** — hidden-until-focus reduces clutter; always visible helps discoverability — some sites combine.

**:focus vs :focus-visible** — `:focus-visible` reduces mouse click ring noise; still provide keyboard ring.

**AA vs AAA contrast** — AA default commercial bar; AAA for health/finance optional.

**Automated vs manual** — tools catch contrast math; skip link UX needs keyboard test.

## Key Takeaways

- **Skip links** — first Tab target jumping to **`#main`** with focusable landmark.
- **`tabindex="-1"` on main** — focus follows skip activation.
- **Never remove focus outline** without visible `:focus-visible` replacement.
- **Contrast ratios** are measurable — 4.5:1 normal text AA, 3:1 large/UI.
- **Color alone** cannot convey required state — pair with text/icons.
- DevTools and axe verify contrast; keyboard test verifies skip and focus.
