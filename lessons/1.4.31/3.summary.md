# UnobservedTaskException — краткая выжимка

> `1.4.31` · ~10 мин на повторение

## Главная идея

Faulted `Task`, который никто не await'ил и не прочитал через `.Exception`, при GC вызывает **`TaskScheduler.UnobservedTaskException`**. С .NET 4.5 процесс обычно не падает — нужен **startup handler** и дисциплина **await**.

## Ключевые пункты

- Observe: **await**, `.Exception` на completed faulted, sync wait (не в ASP.NET).
- Event регистрируют **один раз** в `Program.cs`.
- **`e.Exception.Flatten()`** — часто `AggregateException`.
- **`SetObserved()`** — пометить обработанным по политике runtime.
- Fire-and-forget (`1.4.30`) — главный источник unobserved.
- Handler не ловит ошибки **awaited** request pipeline — это middleware.

## Самые важные примеры

```csharp
TaskScheduler.UnobservedTaskException += (_, e) =>
{
    e.SetObserved();
    _logger.LogError(e.Exception, "Unobserved");
};
```

## «Под капотом» в одном абзаце

Faulted task хранит exception в holder до observe. GC финализирует неobserved task → callback в thread pool → event. Тайминг зависит от GC, поэтому баг «всплывает» позже запроса. Правильный consumer await'ит work в try/catch — exception observed сразу, event не нужен для этого task.

## Частые ошибки

- `SetObserved()` без логирования.
- Считать handler заменой очереди с await.
- Путать с `async void` unhandled path.

## Мнемоника

**«No await — no mercy for exceptions»**
