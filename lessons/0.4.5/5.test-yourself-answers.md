## Answers

### 1. [Recall]
**Answer:** **A** — IPv4 address(es) for name. **AAAA** — IPv6 addresses.
**Why:** Primary address records for connect.

### 2. [Recall]
**Answer:** Time (seconds) resolver/cache may treat answer as valid without re-querying authoritative.
**Why:** TTL drives propagation speed.

### 3. [Understanding]
**Answer:** **Stub** (OS/browser) sends query to configured **recursive** resolver; recursive walks DNS tree (or serves from cache) and returns final A/AAAA to stub.
**Why:** Split client simplicity vs global resolution.

### 4. [Understanding]
**Answer:** Old answer **cached** at recursive resolvers, ISP, OS/browser until **previous TTL** expires; not all caches refresh instantly on authoritative change.
**Why:** Distributed caching trade-off.

### 5. [Understanding]
**Answer:** DNS resolve host → TCP handshake to IP:443 → TLS handshake (SNI) → HTTP request bytes.
**Why:** Full stack order from 0.4.1/0.3.

### 6. [Application]
**Answer:** Resolver follows CNAME to `cdn.provider.net`, then **A/AAAA of target name** — IP там.
**Why:** CNAME not used directly for IP.

### 7. [Application]
**Answer:** **Retry same query over TCP** to get full response.
**Why:** UDP truncation fallback — 0.4.4.

### 8. [Analysis]
**Answer:** **A** неверно — apex CNAME often forbidden; use ALIAS/A.
**Why:** B, C, D correct.

### 9. [Analysis]
**Answer:** Bypasses **public recursive/authoritative DNS** for that name — **local hosts file** (stub-level override) before network DNS.
**Why:** Static mapping — ties 0.4.6.

### 10. [Understanding]
**Answer:** Same name, different answers by who asks — e.g. `db.corp.internal` → `10.0.1.5` on VPN resolver, NXDOMAIN on public 8.8.8.8.
**Why:** Internal vs external views.

### 11. [Recall]
**Answer:** **53** (UDP and TCP).
**Why:** IANA standard.

### 12. [Application]
**Answer:** Wait until **old TTL (86400) expired** everywhere so caches drop long-lived entries; then 60s TTL applies on new queries after cutover.
**Why:** DNS pre-staging pattern.

### 13. [Understanding]
**Answer:** IP changes on failover/CDN; hostname + DNS update; **TLS cert** and provider routing expect name; hard IP brittle.
**Why:** Operations + security.

### 14. [Application]
**Answer:** **DNS failure first** — no IP → no TCP connect attempted (or fails at name resolution stage in client).
**Why:** Ordering before 0.4.3.
