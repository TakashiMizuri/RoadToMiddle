## Why This Matters

Большая часть ASP.NET Core + React — **HTTP**: fetch JSON, forms. Затем product просит **live notifications**, collaborative editing, tick-by-tick prices. Polling каждую секунду жрёт bandwidth; faster HTTP/2 не даёт server **push arbitrary messages** без request. **WebSockets** upgrade HTTP в persistent **bidirectional** channel.

Wrong transport — operational pain: WebSockets через corporate proxy fail; SSE проще для **server→client only**; CDN/cache rules другие. Middle умеет сказать **when NOT WebSockets** — многим хватит SSE или refetch — и знает fallback ladder SignalR.

## Core Concepts

### HTTP request-response

HTTP **client-initiated**: server отвечает на request. «Новые данные» — **poll**, **long polling**, или push channel (SSE/WebSocket).

Polling прост, но N users × rate = load idle. Long polling — held connections, timeouts (`0.3.14`).

### WebSocket model

После **Upgrade** — **full-duplex**, low overhead frames, **persistent** до close. Text/binary frames; auth обычно на handshake (`wss://`, cookie, query token).

```
GET /chat HTTP/1.1
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: ...
Sec-WebSocket-Version: 13
→ 101 Switching Protocols → WebSocket frames
```

Когда **обе стороны** шлют часто: chat, games, trading UI, collaborative cursors.

### SSE — one-way alternative

**SSE** (`text/event-stream`): HTTP open, **server pushes** events; client **`EventSource`**, auto-reconnect. **Только server→client**; commands — обычный HTTP POST. Проще для notifications, progress, logs. Часто легче через proxies.

### Long polling — fallback

Server держит response до event/timeout; client reconnect. Works everywhere HTTP works. SignalR fallback transport.

### Когда хватит HTTP

CRUD REST + refetch on mutation. **TanStack Query** `staleTime` / `refetchOnWindowFocus`. **Webhooks** server-to-server. Status poll `GET /jobs/{id}`.

Не добавляй WebSockets «потому что real-time модно».

### ASP.NET Core и SignalR

**SignalR:** WebSockets, SSE, long polling — negotiates best. **Hub** + **Redis backplane** для multi-instance. Raw: **`HttpContext.WebSockets`**.

Scale: connections **sticky** к process — **session affinity** или backplane. Stateless REST scale проще.

### WebSockets vs HTTP/2 multiplexing

HTTP/2 — many **request-response streams**; WebSocket — **one long conversation** без HTTP headers per message. Разные задачи.

## Under the Hood

Client frames **masked** (RFC 6455). **Ping/pong** vs NAT idle (`0.3.14`). **`wss://`** — TLS. Proxies must support Upgrade. Validate **Origin** на handshake. Kestrel max message size.

## Examples

Scoreboard: SSE/WS push vs poll. Corporate proxy blocks WS → SignalR long poll. Order status: POST + SSE или poll — WS overkill без sub-second multi-job UI.

---

## Common Mistakes & Anti-patterns

WS для всего. No heartbeat. Auth без TLS. No backpressure. SignalR = only WS (нет). WS ties server connection limits.

---

## Production & Real-World Notes

Monitor concurrent WS, reconnect storms. Graceful shutdown. Redis backplane. Managed PubSub services. Mobile background → FCM/APNs.

---

## Comparison / Trade-offs

| Transport | Direction | Complexity | Use |
|-----------|-----------|------------|-----|
| HTTP poll | Client pull | Low | Rare updates |
| SSE | Server push | Medium | Feeds |
| WebSocket | Both ways | Higher | Chat, collab |
| REST | Request-response | Low | CRUD |

---

## Quick Reference

```
HTTP = client asks; WebSocket = 101 + duplex; SSE = server push one-way
Use WS: frequent both-way; SSE: push only; HTTP: CRUD + cache
Scale WS: sticky or Redis; wss:// in prod
```

---

## Key Takeaways

1. HTTP — client-driven; не для continuous push alone.
2. WebSocket — Upgrade → full-duplex.
3. SSE — server→client, проще.
4. Много UI live enough на HTTP/SSE.
5. WS scale ≠ stateless REST.
6. SignalR — transport negotiation.
7. wss://, heartbeats, reconnect, Origin check.
