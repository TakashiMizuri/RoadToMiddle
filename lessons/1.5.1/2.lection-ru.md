## Learning Objectives

## Why This Matters

В `1.1.6` ты узнал, что value types и reference types ведут себя по-разному — семантика копирования, stack frames для locals, heap для объектов. Тот урок описывал **где живут данные**, когда программа уже работает. Он не объяснял **кто** её запускает, **как** C# превращается в то, что выполняет CPU, и **почему** runtime может двигать объекты на heap, обеспечивать type safety и собирать garbage автоматически.

Каждый ASP.NET Core request, каждый EF Core query, каждый backend API — это **managed code**, который исполняет **CLR**. Когда в production появляются `BadImageFormatException`, `TypeLoadException` или необъяснимые JIT-задержки, разбор начинается с понимания этого pipeline — а не с угадывания синтаксиса.

Middle-разработчик держит в голове **compile time vs runtime**, **IL vs native code** и **что CLR гарантирует** (type safety, memory management, exception propagation) vs что остаётся твоей ответственностью (логические баги, leaks через static references, корректность P/Invoke).

## Core Concepts

### Что такое CLR?

**Common Language Runtime (CLR)** — execution engine .NET. Когда ты запускаешь .NET-приложение, CPU не «понимает C#» напрямую — он понимает machine instructions для твоей OS и архитектуры (x64, ARM64). C# — **high-level language**, компилируемый в промежуточную форму; CLR загружает её, верифицирует, компилирует в native code (через JIT — см. `1.5.2`) и управляет **памятью**, **потоками**, **исключениями** и **безопасностью**.

CLR — это **слой, похожий на мини-ОС для приложения**: планирует work на thread pool, выделяет и освобождает память, разрешает типы из metadata и предоставляет платформу (`System.Collections`, `System.Net.Http` и т.д.).

.NET **multi-language**: F#, VB.NET и другие языки target'ят один IL и формат metadata. CLR не интересует, кто написал IL — важно, что IL и metadata **valid and verifiable** (с оговорками для unsafe и interop).

### Managed vs Unmanaged Code

**Managed code** — код, выполнение которого контролирует CLR. Компилятор эмитит IL; CLR JIT-компилирует; **GC** CLR reclaim'ит unreachable объекты; исключения проходят через structured stack unwinding. Твои C# methods, ASP.NET middleware и EF queries — managed.

**Unmanaged (native) code** работает вне прямого контроля CLR — обычно C/C++ machine code через **P/Invoke**, **COM** или **C++/CLI**. CLR может *вызывать* unmanaged, но не GC-track'ит память, выделенную в native library, пока ты не обернёшь её (`SafeHandle`, `IDisposable`).

Граница важна в production: leak в native не проявится как GC pressure; crash в native может уронить весь process. Managed code жертвует частью raw performance ради **безопасности и продуктивности**.

### От C# Source к IL

При `dotnet build` компилятор **Roslyn** (`csc`) переводит source в **assembly**, содержащий:

1. **IL bytecode** — инструкции `ldarg.0`, `call`, `ret` в телах методов.
2. **Metadata** — таблицы типов, методов, полей, attributes (см. `1.5.3`).

IL **stack-based**: виртуальные инструкции push/pop на evaluation stack (это не тот native thread stack, где лежат stack frames из `1.5.4`). JIT мапит IL-операции на CPU registers и memory.

Пример — простой метод:

```csharp
public int Add(int a, int b) => a + b;
```

Упрощённый IL:

```il
.method public hidebysig instance int32 Add(int32 a, int32 b) cil managed
{
    ldarg.1
    ldarg.2
    add
    ret
}
```

IL редко пишут руками, но чтение проясняет **что runtime реально исполняет** до optimization. Инструменты: `ildasm`, ILSpy, dotPeek.

### Execution Pipeline

```
┌─────────────┐    Roslyn     ┌──────────────┐    Load      ┌─────────────┐
│  C# source  │ ────────────► │ Assembly     │ ───────────► │ CLR loader  │
│  (.cs)      │   compile     │ (.dll/.exe)  │              │ + verifier  │
└─────────────┘               │ IL + metadata│              └──────┬──────┘
                              └──────────────┘                     │
                                                                   │ JIT (1.5.2)
                                                                   ▼
                              ┌──────────────┐              ┌─────────────┐
                              │ Native code  │ ◄─────────── │ CPU         │
                              └──────────────┘              └─────────────┘
```

При **первом вызове** метода (или ahead-of-time в Native AOT) JIT переводит IL в native instructions в executable memory. Последующие вызовы переиспользуют скомпилированный код, пока не произойдёт tier recompilation (`1.5.2`).

### Managed Memory и связь с 1.1.6

CLR владеет **managed heap**, где живут reference type objects — как в `1.1.6`. **Stack frames** для вызовов методов — native структуры в cooperation с OS; locals value types и references сидят во frames, **тела** объектов — на heap при `new` class.

Поскольку CLR знает layout объектов из **metadata**, он может:

- Enforce type-safe доступ к полям.
- **Move** объекты при compacting GC (обновляя все references).
- Вставлять **write barriers** при generational promotion (`1.5.7`).

Unmanaged heap (malloc/free) так не работает — pointers могут dangling. Managed references **tracked**; GC находит live объекты через **roots** (`1.5.6`).

### Assemblies и Type System

**Assembly** — единица deployment: обычно один `.dll` или `.exe` с IL, metadata и опциональными resources. **Manifest** именует assembly, version и referenced assemblies. При load CLR resolve'ит references, загружает зависимости и строит in-memory **MethodTable** и runtime type structures (см. Under the Hood).

Strong naming, culture и `AssemblyLoadContext` влияют на load в plugin-сценариях — production команды ловят `FileLoadException` при mismatch версий.

## Under the Hood

### Компоненты CLR внутри Process

При старте ASP.NET Core **host** загружает **coreclr.dll** (Windows) или `libcoreclr.so` (Linux). Основные подсистемы:

- **Execution Engine (EE)** — dispatch, exceptions, stack walking.
- **Type Loader** — metadata → runtime types.
- **JIT** — IL → native (`1.5.2`).
- **GC** — generational collector (`1.5.6`–`1.5.8`).
- **Thread pool** — async continuations, `Task.Run`.

EE держит **MethodTable** на тип и **MethodDesc** на метод. Virtual calls — vtable; interface calls — stub caches после первого resolve.

### IL Verification и Type Safety

CLR **verifies** IL (если не `[SkipVerification]`): consistent stack depth, нет несовместимых store, нет произвольных memory writes. Поэтому нельзя взять address managed object и трактовать как raw byte array без `unsafe`/`Marshal` — verifier защищает invariants, на которых держится GC.

**Unsafe code** opt out из full verification — sharp tool для interop и hot paths, не default style.

### ECMA-335 и Portable IR

Формат IL и metadata стандартизован **ECMA-335**. Отсюда third-party языки и shared libraries. **R2R** и **Native AOT** precompile IL at build time — trade startup vs dynamic features.

IL как **portable bytecode** объясняет cross-platform: один IL на Windows и Linux; JIT эмитит **platform-specific** native code.

## Examples

### Inspect IL

```bash
dotnet build
dotnet tool install -g ilspycmd
ilspycmd MyApp.dll -p C#
```

Сравни decompile с source — `call` к `System.Console.WriteLine` resolve'ится через metadata token.

### Managed Exception

```csharp
void Main() {
    try { Thrower(); }
    catch (InvalidOperationException ex) { Console.WriteLine(ex.Message); }
}
void Thrower() => throw new InvalidOperationException("fail");
```

IL `throw`; EE walks stack frames (`1.5.4`), находит `catch` из metadata. Structured model vs C-style return codes.

## Common Mistakes & Anti-patterns

**«C# работает на CPU как C++».** C# работает **после** JIT (или AOT). Tuning учитывает tiered compilation и GC.

**«IL интерпретируется построчно».** Steady-state server — **JIT to native**, не classic interpreter loop.

**Путать IL evaluation stack и native stack.** Разные структуры; обе связаны с `1.5.4`, но не тождественны.

**Игнорировать assembly load failures.** Binding, RID packages, mix .NET Framework с .NET 8 — runtime metadata/deployment issues.

## Production & Real-World Notes

Self-contained и framework-dependent deployments загружают ту же CLR model. Container images pin **runtime version** — mismatch SDK vs base image даёт subtle bugs.

**Collectible AssemblyLoadContext** — unload plugins в long-lived processes.

**Diagnostics**: `dotnet-trace`, EventPipe, PerfView — JIT events, GC, tier promotion при latency spikes после cold start.

**Native AOT** — faster startup, меньше attack surface; reflection ограничен.

## Comparison / Trade-offs

| Aspect | Managed (.NET) | Native (C/C++) |
|--------|----------------|----------------|
| Memory safety | Verifier + GC | Manual; UB |
| Startup | JIT cost (unless AOT) | Immediate |
| Interop | P/Invoke overhead | Direct OS |
| Deployment | Runtime + assemblies | Single binary possible |

Managed — application logic, web APIs; native — deliberate edges.

## Key Takeaways

- **CLR** — load, verify, JIT, execute, GC, exceptions.
- **Managed** = IL под CLR; **unmanaged** = native вне GC.
- **C# → IL**, не напрямую CPU (кроме AOT).
- **Metadata** — type safety, GC layout, reflection.
- Memory model **`1.1.6`** реализует **CLR**.
- Unsafe ломает assumptions GC при misuse.
