# Exercises: W3C HTML Validation (Hands-on)

> Complete after lectures and passing `4.test-yourself.md`. Use [validator.w3.org](https://validator.w3.org/). Ask AI for hints, not full solutions.

## Easy

### E1. Clean Baseline
**Goal:** Confirm zero errors on valid skeleton.
**Steps:**
1. Copy minimal valid skeleton from `1.lection-eng.md` to `baseline.html`.
2. Upload to W3C validator (File Upload).
3. Enable **Show source**; save screenshot or copy "No errors" summary.
**Success criteria:** Document validates with **0 errors**.

### E2. Fix the List
**Goal:** Repair content model error.
**Steps:** Start from broken file:
```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <title>List fix</title>
</head>
<body>
  <nav>
    <ul>
      <a href="/">Home</a>
      <a href="/about">About</a>
    </ul>
  </nav>
</body>
</html>
```
Validate, read error, fix structure (semantic nav list), re-validate.
**Success criteria:** 0 errors; links inside proper list items.

## Medium

### M1. Error Hunt Bundle
**Goal:** Fix multiple intentional errors in one file.
**Steps:** Create `broken.html` containing **at least four** distinct error types from lecture:
- missing `title`
- duplicate `id`
- invalid custom attribute (no `data-`)
- unescaped `&` in text
Validate once to see all messages; fix iteratively until clean.
**Success criteria:** List four original error messages; final upload 0 errors.

### M2. From Invalid to Semantic Form
**Goal:** Connect validation with earlier HTML topics.
**Steps:** Invalid fragment:
```html
<div onclick="submit()">Go</div>
<input type="text" name="email">
```
Wrap in full document; replace div with `<button type="submit">`; associate label with input; move handler to JS `data-action` pattern if desired. Validate.
**Success criteria:** Valid HTML; button + label; no inline-only structural hacks that fail content model.

## Hard (stretch)

### H1. Rendered Output Check
**Goal:** Validate real template output.
**Steps:**
- If you have ASP.NET/Razor or static site: open any rendered page → **View Page Source** → save HTML → validate.
- OR save this intentionally messy "SSR output" and fix until valid:
```html
<!DOCTYPE html><html lang="en"><head><meta charset="UTF-8">
<title>Dashboard</title></head><body><main>
<div class="card"><button><a href="/delete">Delete</a></button></div>
</main></body></html>
```
Document each error and semantic fix.
**Success criteria:** Explain why nested interactive elements fail; final valid pattern (e.g. button OR link, not both nested).

### H2. CLI or CI Sketch (optional)
**Goal:** Plan automation.
**Steps:** Write 5-line pseudo CI step using `html5validator` or `vnu.jar` on `dist/**/*.html`. When would team run it (PR vs nightly)?
**Success criteria:** Fails build on validator errors; scope limited to static HTML artifacts.

## Hints (read only if stuck)

<details>
<summary>Hint E2</summary>
`<ul>` may only contain `<li>` (plus special elements). Wrap each `<a>` in `<li>`.
</details>

<details>
<summary>Hint M1 missing title</summary>
Add `<title>` inside `<head>`.
</details>

<details>
<summary>Hint H1 button/link</summary>
Use `<a class="btn">` styled as button OR `<button type="button">` with JS — not `<button><a>`.
</details>
