# Rebase — краткая выжимка

> `0.2.3` · ~10 мин на повторение

## Главная идея

**Rebase** replay'ит commit'ы feature на новый base (например latest `main`), создавая **новые commit objects** с новыми hash'ами. Graph feature становится linear; старые commit'ы unreachable с ветки. **Interactive rebase** (`-i`) редактирует список commit'ов перед push. **Golden rule:** не rebase history, которую другие уже используют.

## Ключевые пункты

- Rebase ≠ move — **новые parents → новые hashes** (`0.1.5`).
- `git rebase main` на feature; conflicts → fix → `--continue` или `--abort`.
- `-i`: pick, reword, squash/fixup, edit, drop, reorder.
- После rebase remote feature: **`git push --force-with-lease`**.
- Rebase + FF merge в `main` = linear main без merge bubble.
- Внутри — cherry-pick loop; reflog/`feature@{1}` — safety net.

## Самые важные примеры

```bash
git switch feature
git rebase origin/main
git push --force-with-lease

git rebase -i HEAD~3   # squash/reword перед PR
```

## «Под капотом» в одном абзаце

Git находит commit'ы на feature, не достижимые с нового base, снимает ref tip на base (M), для каждого commit вычисляет patch от parent, применяет на текущий HEAD, создаёт новый commit object. Parent chain меняется — hash меняется. Старые objects остаются до gc/reflog expiry. `--onto` ограничивает диапазон replay.

## Частые ошибки

- Rebase `main`/shared release.
- `git push --force` без lease.
- Interactive rebase уже pushed commit'ов без согласования команды.

## Мнемоника

> **Rebase** = «переписать черновик»; **merge** = «склеить две книги с закладкой на стыке». Черновик — свой; общую библиотеку не переписываем.
