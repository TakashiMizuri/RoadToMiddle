# AsyncLocal — Ambient Context in the Async Chain

> Roadmap: `1.4.32` · Node: `1.4` — C# async/await · Depth: понимание

## Learning Objectives

After this lesson you will be able to:

- Explain **`AsyncLocal<T>`** as **logical call context** that flows across `await` boundaries.
- Contrast it with **`ThreadLocal<T>`** and **`HttpContext`** for request-scoped data.
- Describe how the runtime **copies** values on async continuation (not shared mutable slots across unrelated branches).
- Name production uses: **correlation ID**, **culture**, **tenant ID**, **logging scope**.
- Recognize pitfalls: **memory leaks** via captured context, **wrong assumptions** about parallel branches.

---

## Why This Matters

You set `CultureInfo.CurrentCulture` on the request thread, then `await _db.SaveChangesAsync()` — after the continuation, culture may be wrong because the thread changed. You stash the user ID in a **static** field — parallel requests overwrite each other. **`AsyncLocal<T>`** is how .NET propagates **ambient** per-logical-operation state through async code without passing parameters through every method signature.

ASP.NET Core already uses this machinery: **`HttpContext`** is accessed via **`IHttpContextAccessor`**, backed by **`AsyncLocal<HttpContext>`**. Middle developers understand **`AsyncLocal`** when building **multi-tenant** logging, **OpenTelemetry** baggage, or custom middleware that must survive `await`.

## Core Concepts

### Logical Call Context vs Physical Thread

A single HTTP request is one **logical flow** that may hop across **many thread-pool threads** at each `await`. **`ThreadLocal<T>`** is wrong for request data — each hop might see a different value or empty slot.

**`AsyncLocal<T>`** stores data in the **execution context** that the runtime captures when scheduling continuations. When an `async` method awaits, the continuation restores the context — including `AsyncLocal` values — even on another thread.

### Basic Usage

```csharp
private static readonly AsyncLocal<string?> _correlation = new();

public static string? CorrelationId
{
    get => _correlation.Value;
    set => _correlation.Value = value;
}
```

Middleware sets it at request start:

```csharp
CorrelationId = context.TraceIdentifier;
try
{
    await _next(context);
}
finally
{
    CorrelationId = null; // avoid leaking to pool threads
}
```

Downstream code reads **`CorrelationId`** without parameter threading — useful for deep call stacks and third-party libraries you cannot change.

### Copy-on-Write Semantics

Assigning to **`AsyncLocal.Value`** creates a **logical copy** for the current execution context and **child contexts** spawned from it. Sibling parallel tasks started with **`Task.Run`** without flowing context may **not** see the same value unless you use **`ExecutionContext.Run`** or ensure context flow (default for `Task.Run` **does** capture execution context in many cases — behavior matters for diagnostics).

When two async branches diverge after setting different values, they do not corrupt each other — each continuation gets the value that was current when the branch was created. This is subtler than "one global variable" — read Microsoft docs when sharing **mutable reference types** through `AsyncLocal`.

### Relation to HttpContext

```csharp
// Simplified mental model
AsyncLocal<HttpContext?> → accessed via IHttpContextAccessor.HttpContext
```

Do not duplicate **`HttpContext`** in your own `AsyncLocal` unless you have a clear reason — use the accessor. For **non-HTTP** workers (`BackgroundService`), `HttpContext` is null; use explicit parameters or a different ambient key (tenant from message payload).

## Under the Hood

### ExecutionContext and Flow

`async` continuations run with **`ExecutionContext.Run`** (conceptually) to restore **LogicalCallContext** data. **`AsyncLocal`** values live in a dictionary-like structure attached to the context. **`SuppressFlow`** / **`RestoreFlow`** can disable copying for performance-sensitive library code — rare in app code.

### GC and Leaks

Leaving **`AsyncLocal`** set to large objects without clearing in **`finally`** can pin context on thread-pool threads reused for unrelated work — classic **ambient context leak**. Always **clear** at boundary (end of request, end of scope).

## Examples

### Tenant ID in Service Layer

```csharp
public class TenantContext
{
    private static readonly AsyncLocal<Guid?> _tenant = new();
    public static Guid? Current => _tenant.Value;
    public static IDisposable Use(Guid tenantId)
    {
        _tenant.Value = tenantId;
        return new DisposeAction(() => _tenant.Value = null);
    }
}
```

Middleware resolves tenant from host header, wraps pipeline in **`using (TenantContext.Use(id))`**.

### Logging Scope

Serilog **`LogContext.PushProperty`** uses similar ambient stacks. `AsyncLocal` is the primitive underneath patterns like **scoped logging enrichers**.

## Common Mistakes & Anti-patterns

Using **`AsyncLocal`** for **explicit business parameters** that every method should know — hides dependencies and complicates tests. Prefer constructor injection for required deps; ambient only for **cross-cutting** diagnostics.

Assuming **`Parallel.ForEachAsync`** child iterations inherit parent `AsyncLocal` without verifying — parallel branches have independent context copies.

Never clearing values — leaks across requests on same thread-pool thread.

Replacing **`CancellationToken`** propagation with ambient flags — cancellation must stay explicit.

---

## Production & Real-World Notes

**OpenTelemetry** `Activity` and baggage flow with execution context — align custom `AsyncLocal` with tracing IDs.

In **tests**, set ambient context in fixture **`using`** blocks; parallel test runs require isolation.

**Minimal APIs** and **filters** set correlation at edge; libraries read ambient — consistent pattern across monolith.

## Comparison / Trade-offs

| Mechanism | Flows across await | Right for request identity |
|-----------|-------------------|---------------------------|
| `AsyncLocal<T>` | Yes | Yes (cross-cutting) |
| `ThreadLocal<T>` | No | No |
| Method parameters | Yes | Yes (explicit deps) |
| `HttpContext` | Via accessor | Yes in HTTP only |

---

## Quick Reference

- **`AsyncLocal<T>`** = ambient per logical async flow.
- Set at boundary (middleware); **clear in `finally`**.
- Not a substitute for **DI** or **`CancellationToken`**.
- **`IHttpContextAccessor`** uses same idea for HTTP.
- Mind **parallel branches** and mutable reference types.

---

## Key Takeaways

- Physical threads change at `await`; **`AsyncLocal`** preserves logical context.
- Use for **correlation, tenant, culture** — not hidden required business args.
- **Clear** values to prevent leaks on pool threads.
- **`ThreadLocal`** fails for async server code.
- Understand **copy** semantics when branching parallelism.
