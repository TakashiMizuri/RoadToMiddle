## Answers

### 1. [Recall]
**Answer:** HEAD (GET returns body; HEAD metadata only).
**Why:** HEAD = GET semantics without body.

### 2. [Recall]
**Answer:** POST (typically to collection URI like `/users`).
**Why:** Server assigns Location/id; PUT when client sets URI.

### 3. [Recall]
**Answer:** **201 Created** + often **Location** header with new resource URI.
**Why:** Standard create success pattern.

### 4. [Understanding]
**Answer:** GET safe assumption — prefetch/crawler/CSRF trigger without user intent; GET may be cached. DELETE non-safe, not invoked by simple link prefetch same way; requires deliberate client with proper auth.
**Why:** Safe methods must not mutate state.

### 5. [Understanding]
**Answer:** PUT expects **full replacement** of resource at URI; PATCH **partial** update/merge.
**Why:** Different representation contracts.

### 6. [Understanding]
**Answer:** **CORS preflight** — browser checks server allows method/headers for cross-origin request before sending actual POST.
**Why:** Security for cross-origin non-simple requests.

### 7. [Understanding]
**Answer:** POST **not idempotent** — each request may create new resource. PUT **idempotent** — same replace state twice = same result.
**Why:** Core HTTP method properties (`0.3.5`).

### 8. [Application]
**Answer:** (a) GET (b) PUT (c) PATCH (d) DELETE.
**Why:** Standard CRUD mapping.

### 9. [Application]
**Answer:** **405 Method Not Allowed** with **Allow** header listing supported methods.
**Why:** Path exists but verb wrong.

### 10. [Analysis]
**Answer:** C неверно.
**Why:** OPTIONS also returns Allow for discovery, not only CORS.

### 11. [Analysis]
**Answer:** Complex/large filter in body; URL length limits; sensitive data not in query/logs; not cacheable search anyway.
**Why:** POST for process/search bodies is acceptable pattern.

### 12. [Understanding]
**Answer:** Lists methods server supports on resource — e.g. `Allow: GET, PUT, DELETE`.
**Why:** Client discovery + 405 guidance.

### 13. [Application]
**Answer:** e.g. `[HttpDelete("{id}")]` on controller route `api/items` or `app.MapDelete("/api/items/{id}", ...)`.
**Why:** Method-specific routing.

### 14. [Understanding]
**Answer:** **Cancel** is action/side effect on resource, not deletion of resource itself; POST command pattern `/actions/cancel` when cancel not idempotent or models workflow.
**Why:** RPC-style sub-resource POST vs DELETE entire entity.

### 15. [Recall]
**Answer:** GET, HEAD, OPTIONS (all safe by spec).
**Why:** Safe = intended no server state change.
