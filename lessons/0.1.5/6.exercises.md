## Easy

### E1. Read a commit object
**Goal:** Parse commit fields manually.

**Steps:**
1. In any repo with history, run `git log -1 --format=%H` â†’ full hash.
2. `git cat-file -p <hash>` â€” copy tree, parent(s), author, committer lines.
3. `git cat-file -t <tree-hash>` â€” confirm type `tree`.

**Success criteria:** You label each field and explain what tree hash represents.

### E2. Graph your history
**Goal:** Use `git log --graph`.

**Steps:**
1. Run `git log --oneline --graph --decorate -10` in a repo with at least one merge (or create one â€” see M1).
2. Mark which commit is HEAD and which lines show a merge.

**Success criteria:** You explain what `|\`, `|/`, `*` mean in the ASCII graph.

---

### E3. Parent navigation
**Goal:** Navigate the DAG with rev syntax.

**Steps:**
1. `git rev-parse HEAD` â†’ `git rev-parse HEAD^` â†’ `git rev-parse HEAD~3`
2. Write in one sentence: difference between `^` and `~`.

**Success criteria:** `^` = first parent; `~n` = n steps along first-parent chain.

## Medium

### M1. Build a small DAG
**Goal:** Create linear + branch + merge.

**Steps:**
1. New repo, 1 file, commit "A" (root).
2. Branch `feature`, commit "B" on feature.
3. On `main`, commit "C".
4. Merge feature into main with message "M".
5. Draw the DAG on paper; run `git log --graph --oneline --all`.
6. `git rev-parse HEAD^1` and `HEAD^2` â€” which commit is which?

**Success criteria:** Diagram matches log; first parent = C line, second = B.

---

### M2. Same content, different commit hash
**Goal:** See hash depends on more than file content.

**Steps:**
1. Create commit with message "version 1".
2. `git commit --amend -m "version 1 renamed"` (same tree, new message).
3. Compare `git rev-parse HEAD` before/after (use reflog for old hash).
4. Explain why tree might be identical but commit hash differs.

**Success criteria:** Written explanation mentions message in hash input.

---

### M3. Reachability experiment
**Goal:** Understand unreachable commits.

**Steps:**
1. Create branch `temp`, make 2 commits, note tip hash `T`.
2. `git checkout main` (do not merge temp).
3. `git branch -D temp`
4. `git log --oneline T` â€” works?
5. `git reflog` â€” find temp commits?
6. Optional: `git fsck --unreachable` after reflog expires (or read docs â€” don't wait days).

**Success criteria:** You explain role of reflog vs branch ref for reachability.

## Hard (stretch)

### H1. Find merge-base
**Goal:** Common ancestor in branched history.

**Steps:**
1. From M1 repo (or similar), run `git merge-base main feature` after merge (recreate branch tips if needed with labels).
2. Verify ancestor is on both histories: `git merge-base --is-ancestor <result> HEAD`.

**Success criteria:** You identify commit A (or correct common ancestor) and explain merge-base use in merges.

---

### H2. Hash algorithm awareness
**Goal:** SHA-1 vs SHA-256 (conceptual).

**Steps:**
1. Read `git config --get extensions.objectFormat` in your repo (empty = sha1).
2. Optional: create throwaway repo `git init --object-format=sha256` in `/tmp` or separate folder.
3. Compare hash length from `git rev-parse HEAD` (40 vs 64 chars).

**Success criteria:** One paragraph on why Git is transitioning and why it doesn't affect DAG concepts.
