# LINQ Performance — Avoid Multiple Enumeration

> Roadmap: `1.3.30` · Node: `1.3` — C#: Advanced Topics · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Define **deferred execution**, **multiple enumeration**, and **materialization** (`ToList`, `ToArray`).
- Detect code that enumerates the same **cold `IEnumerable`** twice and measure impact.
- Apply fixes: **materialize once**, **pass `IReadOnlyList`**, **`IQueryable` single round-trip**, **iterator fusion awareness**.
- Explain when **`Count()`** forces full scan vs **` ICollection.Count`** is O(1).
- Optimize LINQ pipelines for **EF Core** (single SQL, no client eval loops).
- Balance **memory vs repeated computation** with conscious trade-offs.

---

## Why This Matters

This is the #1 LINQ footgun in production:

```csharp
var query = LoadFromDatabase(); // IEnumerable, deferred
if (query.Any()) {
    var list = query.ToList(); // second full pass
    var count = query.Count();   // third pass — disaster
}
```

Each **`foreach`**, **`Count`**, **`ToList`**, **`Any`** on a **cold** sequence may re-run the entire pipeline — duplicate SQL queries, double file reads, doubled network calls. Juniors assume LINQ "runs once" like SQL; it **does not** until materialized.

Middle developers profile hot paths, materialize at **stable boundaries**, use **`IReadOnlyList<T>`** in method signatures when data is reused, and keep **`IQueryable`** composable until one **`ToListAsync`**. This lesson closes node **1.3** before **async** — where **`IAsyncEnumerable`** adds another layer of single-pass discipline.

## Core Concepts

### Deferred vs Immediate

| Kind | Examples | When work runs |
|------|----------|----------------|
| Deferred | `Where`, `Select`, `OrderBy`, custom `yield` | On enumeration |
| Immediate | `ToList`, `Count`, `First`, `Any`, `Sum` | Now (for that operator) |

**`var filtered = source.Where(x => x > 0)`** — no elements processed yet. **`foreach`** pulls one at a time through the chain (**iterator fusion** — nested `MoveNext` calls without intermediate lists for simple chains).

### Multiple Enumeration Problem

```csharp
IEnumerable<Order> GetOrders() => db.Orders.AsEnumerable(); // hypothetical cold

var orders = GetOrders();
Console.WriteLine(orders.Any());
foreach (var o in orders) { } // entire sequence again
```

If **`GetOrders()`** hits DB each enumeration → **2 round-trips**. Fix:

```csharp
var orders = GetOrders().ToList();
if (orders.Count > 0) { ... }
```

**`ToList()`** — single pass, O(n) memory, O(1) reuse.

### ICollection.Count vs Enumerable.Count()

```csharp
List<T> list = ...;
list.Count; // O(1) — property

IEnumerable<T> seq = list;
seq.Count(); // extension: if seq is ICollection<T>, O(1); else O(n) full scan
```

**`Count()`** extension method optimizes **`ICollection<T>`** / **`IReadOnlyCollection<T>`** — but **not** arbitrary deferred chain.

### Any + ToList Anti-pattern

```csharp
if (items.Any()) {
    Process(items.ToList());
}
```

Two passes. Prefer:

```csharp
var list = items.ToList();
if (list.Count > 0) Process(list);
```

Or **`if (items is ICollection<T> c && c.Count > 0)`** when already materialized.

### IQueryable — Single Round-Trip

```csharp
var q = db.Products.Where(p => p.IsActive);
var count = await q.CountAsync(); // SQL COUNT
var page = await q.OrderBy(p => p.Id).Take(20).ToListAsync(); // second SQL
```

Two queries — often OK (count + page). Worse:

```csharp
foreach (var id in ids) {
    var name = db.Products.Where(p => p.Id == id).Select(p => p.Name).First();
}
```

**N+1** — enumerate **`ids`** and query per id. Fix with **`Where(p => ids.Contains(p.Id))`** one query + dictionary.

### Captured Mutable Variables in Deferred LINQ

```csharp
int threshold = 10;
var q = source.Where(x => x > threshold);
threshold = 100;
foreach (var x in q) { } // uses 100 — deferred closure
```

Enumeration sees **current** variable value — surprise if changed before foreach.

### When NOT to Materialize

Large infinite or streaming sources — **never **`ToList`**** on whole stream. Use **single forward pass**:

```csharp
foreach (var item in stream.Where(...).Select(...))
    Process(item);
```

One enumeration, constant memory (if no buffering operators like **`OrderBy`** — which buffers all).

**`OrderBy`**, **`GroupBy`**, **`Reverse`** — **buffer entire sequence** (or large chunks) internally on first enumeration.

## Under the Hood

### Iterator Chaining Without Allocation

Simple **`Where`·`Select`** pipeline:

```
consumer MoveNext → Select MoveNext → Where MoveNext → source MoveNext
```

No intermediate **`List`** — **fusion**. Adding **`OrderBy`** breaks streaming — sort buffers keys.

### IEnumerable vs IReadOnlyList in APIs

Public method returning **`IEnumerable<T>`** promises lazy **or** one-shot — caller must not assume cache. Returning **`IReadOnlyList<T>`** documents **materialized, reusable** snapshot — clearer contract.

Some teams use **`IEnumerable` once** comment convention — better fix the type.

### EF Core Tracking and Re-query

**`AsNoTracking()`** query materialized once → DTO list. Reusing **`IQueryable`** with different operators → multiple SQL — fine if intentional. Accidental:

```csharp
IQueryable<Product> base = db.Products;
var a = base.Where(...).ToListAsync();
var b = base.Where(...).ToListAsync(); // two queries — OK
// Same IEnumerable from client eval — worse
```

### Memoization

**`.Memoize()`** (Rx) or custom cache-on-first-enumerate — rare in LOB apps; **`ToList`** is explicit and readable.

## Syntax / Commands / API

```csharp
// Materialize
.ToList(), .ToArray(), .ToDictionary(), .ToHashSet()
.ToListAsync(), .ToArrayAsync() // EF

// O(1) count when possible
((ICollection<T>)seq).Count
seq is IReadOnlyCollection<T> c ? c.Count : seq.Count()

// Single pass
foreach (...) { }
```

**Analyzers:** Roslyn analyzers / ReSharper warn on possible multiple enumeration.

## Examples

### API Handler — Materialize Once

```csharp
public async Task<IResult> Handle(Query q, CancellationToken ct)
{
    var rows = await db.Items
        .Where(i => i.TenantId == q.TenantId)
        .ToListAsync(ct);

    if (rows.Count == 0)
        return Results.NotFound();

    var total = rows.Sum(r => r.Amount); // in-memory, same list
    return Results.Ok(new { total, rows });
}
```

One SQL; reuse **`rows`**.

### Logging Pipeline Length

```csharp
// BAD:
var pipeline = Parse(file).Where(Valid).Select(Map);
_logger.LogInformation("Count {N}", pipeline.Count());
return pipeline.ToList();

// GOOD:
var list = Parse(file).Where(Valid).Select(Map).ToList();
_logger.LogInformation("Count {N}", list.Count);
return list;
```

### Streaming — Single Pass OK

```csharp
await foreach (var record in parser.ReadAsync(ct))
{
    if (!IsValid(record)) continue;
    await writer.WriteAsync(Map(record), ct);
}
```

No list — intentional single forward enumeration.

## Common Mistakes & Anti-patterns

**`Any()` then `foreach`** on same cold sequence.

**`Count()` for emptiness check** — use **`Any()`** if single pass and not materialized — but if you need both count and items, **`ToList` once**.

**Passing **`IEnumerable`** to multiple helpers** each enumerating.

**`OrderBy` on huge stream** — memory blow.

**Closure over changing locals** in deferred **`Where`**.

## Production & Real-World Notes

Use **`dotnet-trace`**, logging SQL count per request, or **`MiniProfiler`** to catch double queries.

Benchmark: **`ToList`** cost vs second DB round-trip — network usually dominates.

Code review checklist: every **`IEnumerable`** parameter — how many enumerations?

**`ImmutableArray`** / **`ReadOnlyMemory`** for hot reuse paths after materialize.

## Comparison / Trade-offs

| Situation | Strategy |
|-----------|----------|
| Reuse collection 2+ times | `ToList` / `ToArray` once |
| Stream millions, one pass | Single `foreach`, no `Count` first |
| Need count + page from DB | Two SQL OK; or window functions |
| Public API reuse | Return `IReadOnlyList` |
| Lazy composition | Keep `IEnumerable` but document single consumer |

---

## Quick Reference

- **Cold IEnumerable** = re-enumerate = re-run pipeline.
- **`ToList`** = materialize boundary.
- **`Count()`** on non-collection = O(n) scan.
- **`OrderBy`/`GroupBy`** buffer — not streaming.
- EF: prefer one composed query; avoid N+1 loops.

---

## Key Takeaways

- LINQ chains are **lazy** until materialized or immediate operator runs.
- **Multiple enumeration** duplicates I/O and CPU — materialize at boundaries.
- **`Any` + `ToList`** = two passes — combine.
- **`IReadOnlyList`** documents reusable data; **`IEnumerable`** often single-pass.
- **`OrderBy`** forces full buffer — plan memory accordingly.
- EF: count + enumerate intentionally — watch N+1.
