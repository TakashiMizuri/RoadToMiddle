# HttpClient pitfalls — краткая выжимка

> `1.4.26` · ~10 мин на повторение

## Главная идея

**HttpClient** нужно **переиспользовать** (точнее — shared handler pool). **`new HttpClient()` на каждый запрос** → socket/port exhaustion. Долгоживущий handler без rotation → **stale DNS**.

## Ключевые пункты

- Facade над **SocketsHttpHandler** + connection pool.
- **TIME_WAIT** после частых disconnect без keep-alive reuse.
- **PooledConnectionLifetime** — rotate connections и refresh DNS.
- **MaxConnectionsPerServer** — limit fan-out per host.
- **IHttpClientFactory** — managed lifetime (~2 min handler rotation).
- Не dispose **CreateClient()** как anti-pattern short client.
- Симптомы: SocketException, работает после restart, fails under load.

## Самые важные примеры

```csharp
// Anti-pattern
using var client = new HttpClient();

// ASP.NET
var client = _httpClientFactory.CreateClient("payments");
await client.GetAsync("/health", ct);
```

## «Под капотом» в одном абзаце

ConnectionPool держит TCP keep-alive к `(host, port)`. Новый handler каждый request не reuse pool → исчерпание локальных портов. DNS resolve при новых connections; stale handler = stale IP до rotation или restart.

## Частые ошибки

- «HttpClient IDisposable → using каждый раз».
- Static singleton без PooledConnectionLifetime в dynamic infra.
- Parallel fan-out + new client per iteration.

## Мнемоника

**One pool per downstream, not one client per call.**
