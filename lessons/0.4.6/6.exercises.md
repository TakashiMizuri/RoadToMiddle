# Exercises: localhost and Binding

> Complete after lectures and passing `4.test-yourself.md`.

## Easy

### E1. Bind or connect?
**Goal:** Separate listen vs client targets.

Label each as valid **bind**, valid **connect**, or **invalid**:

1. `127.0.0.1:5000` — Kestrel listen
2. `0.0.0.0:8080` — nginx listen
3. `0.0.0.0:8080` — browser URL
4. `203.0.113.10:443` — HttpClient connect

**Success criteria:** Four labels correct.

---

### E2. Who can connect?
**Goal:** Predict reachability.

Server binds `127.0.0.1:3000`. Circle who can TCP connect successfully:

- Same laptop browser
- Teammate PC on LAN
- Docker container on same host (bridge network, not host network)

**Success criteria:** Only same-host cases (with nuance for container networking).

## Medium

### M1. LAN dev setup
**Goal:** Practical Kestrel exposure.

Configure (describe commands/settings) ASP.NET Core so phone on Wi‑Fi reaches API. Include bind, URL to type on phone, one firewall note.

**Success criteria:** Non-loopback bind + machine LAN IP in client URL.

---

### M2. netstat interpretation
**Goal:** Read listen addresses.

Run `netstat -ano | findstr LISTENING`. Find one `0.0.0.0:...` and one `127.0.0.1:...` line. Explain difference for remote access.

**Success criteria:** Wildcard vs loopback-only explained.

## Hard (stretch)

### H1. Same-host proxy diagram
**Goal:** Production pattern.

ASCII diagram: Internet → nginx `0.0.0.0:443` → Kestrel `127.0.0.1:5000`. Label layers (`0.4.1`) for one HTTPS request.

**Success criteria:** TCP connect targets; loopback hop on same machine.
