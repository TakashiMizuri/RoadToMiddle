# LINQ Take, Skip, TakeWhile, SkipWhile

> Roadmap: `1.3.26` · Node: `1.3` — C#: Advanced Topics · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Slice sequences with **`Take`**, **`Skip`**, **`TakeWhile`**, and **`SkipWhile`** for pagination and windowing.
- Distinguish **fixed-count** slicing from **predicate-based** slicing and predict edge cases at boundaries.
- Compose **`Skip` + `Take`** for offset/limit pagination and know when **`OrderBy`** must precede them.
- Apply operators on **`IQueryable`** so EF translates to **`LIMIT`/`OFFSET`** (or equivalent).
- Avoid **unstable pagination**, **multiple enumeration**, and **`Skip(n)` on large n** performance traps.

---

## Why This Matters

Almost every list UI needs pagination: "show 20 items, page 3". **`Skip((page-1)*pageSize).Take(pageSize)`** is the idiomatic LINQ pattern. Log processing uses **`TakeWhile`** to read lines until a sentinel; stream parsing uses **`SkipWhile`** to skip headers. Unlike **`Where`**, **`TakeWhile`/`SkipWhile`** stop scanning the source once the condition flips — important for infinite or lazy sequences.

On databases, **`Skip/Take`** become **`OFFSET/FETCH`** — scanning and skipping millions of rows is expensive. **`TakeWhile`** on SQL may not translate cleanly. Knowing operator semantics prevents off-by-one page bugs and explains why cursor-based pagination replaces offset pagination at scale.

## Core Concepts

### Take and Skip — Fixed Windows

```csharp
var nums = Enumerable.Range(1, 10);
var page = nums.Skip(3).Take(4); // 4, 5, 6, 7
```

**`Skip(n)`** discards the first **n** elements and yields the rest. **`Take(n)`** yields at most **n** elements, then stops. Both are **deferred**; work happens per consumed element.

If **`n >= Count`**, **`Skip(n)`** yields empty. **`Take(n)`** on a short sequence yields all remaining elements — no exception.

**Pagination formula** (1-based page index):

```csharp
var page = 2, pageSize = 20;
var items = query
    .OrderBy(x => x.Id)   // stable order required!
    .Skip((page - 1) * pageSize)
    .Take(pageSize);
```

Without **`OrderBy`**, SQL and in-memory order may be undefined — pages "shuffle" between requests.

### TakeWhile — Prefix Until Condition Fails

```csharp
var nums = new[] { 2, 4, 6, 7, 8 };
var evens = nums.TakeWhile(n => n % 2 == 0); // 2, 4, 6 — stops BEFORE 7
```

**`TakeWhile(predicate)`** yields elements **while** predicate is **true** for each element. On the **first false**, enumeration **stops** — remaining source elements are **never read** (side effects in source after that point do not run).

**`TakeWhile(element, index)`** overload includes zero-based index in the predicate — useful for "first N lines" combined with other rules.

This differs from **`Where`**: **`Where`** filters all elements and walks the **entire** sequence; **`TakeWhile`** is a **prefix** operator with early termination.

### SkipWhile — Drop Prefix Until Condition Fails

```csharp
var lines = new[] { "", "", "Header", "Data1", "Data2" };
var data = lines.SkipWhile(l => string.IsNullOrWhiteSpace(l) || l == "Header");
// Data1, Data2
```

**`SkipWhile`** skips elements while predicate is true; when predicate becomes false, it yields **that element and all following** — including elements that would fail the predicate if evaluated again.

Example nuance:

```csharp
new[] { 1, 2, 3, 1 }.SkipWhile(x => x < 3); // 3, 1 — not just 3
```

After skip phase ends at `3`, the rest passes through unfiltered.

### Overloads With Index

```csharp
source.TakeWhile((item, index) => index < 5);
source.SkipWhile((item, index) => index < 2);
```

Index-based variants help when the rule is positional ("skip first two rows") without relying on **`Take(2)`** alone in composed pipelines.

## Syntax / Commands / API

```csharp
source.Take(count)
source.TakeWhile(predicate)
source.TakeWhile((item, index) => ...)
source.Skip(count)
source.SkipWhile(predicate)
source.SkipWhile((item, index) => ...)
```

**.NET 6+:** **`Take`/`Skip`** on **`IQueryable`** translate to SQL paging. For **`IEnumerable`**, negative count throws **`ArgumentOutOfRangeException`**.

## Examples

### API Pagination Endpoint

```csharp
public async Task<PageDto<OrderDto>> GetOrdersAsync(int page, int size, CancellationToken ct)
{
    var baseQuery = db.Orders
        .OrderByDescending(o => o.CreatedAt)
        .ThenBy(o => o.Id);

    var total = await baseQuery.CountAsync(ct);
    var items = await baseQuery
        .Skip((page - 1) * size)
        .Take(size)
        .Select(o => new OrderDto(o.Id, o.Total))
        .ToListAsync(ct);

    return new PageDto(items, total, page, size);
}
```

**`CountAsync`** on the filtered query without **`Skip/Take`** gives total pages. **`ThenBy(o => o.Id)`** breaks ties for stable sort.

### Read Until Sentinel

```csharp
IEnumerable<string> ReadUntilBlank(IEnumerable<string> lines) =>
    lines.TakeWhile(line => !string.IsNullOrWhiteSpace(line));
```

Stops at first blank line without reading the rest of the file — efficient for lazy line iterators.

### Skip Header Rows in CSV Stream

```csharp
var rows = ParseCsv(stream)
    .SkipWhile(row => row[0].StartsWith("#"))
    .TakeWhile(row => row[0] != "END");
```

Combines conditional prefix skip with bounded body take.

## Common Mistakes & Anti-patterns

**Pagination without `OrderBy`.** Non-deterministic pages.

**Using `Where` instead of `TakeWhile` for prefix.** Walks whole sequence; wrong semantics if you need early stop.

**Assuming `SkipWhile` filters like `Where` after the boundary.** Everything after first false is included.

**Large `Skip` on SQL.** `OFFSET 1000000` still scans — use keyset pagination for deep pages.

**Multiple enumeration** of cold **`IEnumerable`** between **`Count`** and **`Skip/Take`** — materialize or use **`IQueryable`** consistently.

---

## Production & Real-World Notes

Prefer **cursor / keyset pagination** (`WHERE id > @lastId ORDER BY id TAKE n`) over large offsets in high-traffic APIs. EF Core 8+ has improved translations; still profile.

For in-memory sequences, **`Skip/Take`** is O(n) to reach skip point — acceptable for small collections, not for repeated slicing on **`LinkedList`-like** custom iterators without indexing.

**`TakeWhile`/`SkipWhile`** on infinite streams (`while(true) yield`) are safe when predicates eventually fail or take bounded prefix — classic in reactive pipelines.

## Comparison / Trade-offs

| Need | Operator |
|------|----------|
| Page 3, 20 items | `Skip(40).Take(20)` after `OrderBy` |
| First N items | `Take(N)` |
| Longest prefix matching rule | `TakeWhile` |
| Drop header lines | `SkipWhile` |
| All matching items anywhere | `Where` (full scan) |

---

## Quick Reference

| Operator | Behavior | Stops early? |
|----------|----------|--------------|
| `Take(n)` | first n elements | after n |
| `Skip(n)` | drop first n | after window started |
| `TakeWhile(p)` | prefix while p true | on first false |
| `SkipWhile(p)` | skip while p true | on first false, yield rest |

---

## Key Takeaways

- **`Skip` + `Take`** = offset/limit; always **`OrderBy`** for pagination.
- **`TakeWhile`/`SkipWhile`** are prefix operators with early termination — not filters.
- **`SkipWhile`**: after boundary, **all** remaining elements pass through.
- SQL **`OFFSET`** cost grows with offset — keyset pagination at scale.
- Predicate vs count slicing solves different problems.
