# Code Review Etiquette — Giving and Receiving Feedback

> Roadmap: `0.2.10` · Node: `0.2` — Git: branches and collaboration · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Explain **why code review exists** beyond catching bugs — knowledge sharing, standards, ownership.
- Apply principles for **giving feedback**: specific, actionable, kind, prioritized.
- Apply principles for **receiving feedback**: assume good intent, clarify, avoid defensiveness.
- Distinguish **blocking** vs **non-blocking** (nit) comments.
- Use review comment patterns that work in async PR workflows (`0.2.8`).
- Recognize toxic and unproductive review behaviors and their cost to teams.

---

## Why This Matters

You can master remotes (`0.2.6`), upstream tracking (`0.2.7`), pull requests (`0.2.8`), and Conventional Commits (`0.2.9`) — and still fail as a teammate if code review becomes a battlefield. Review is the **human gate** in collaboration: the moment where quality, security, and shared understanding are negotiated before code touches `main` and reaches production.

For a Middle fullstack developer, review is bidirectional. You **receive** feedback on your ASP.NET controllers and React components; you **give** feedback on others' SQL queries, API contracts, and test coverage. Teams that review well ship faster over time — not because they catch every typo, but because they **spread context**, reduce bus factor, and align on patterns. Teams that review poorly accumulate resentment, rubber-stamp approvals, or endless style wars.

Etiquette is not politeness theater. It is **engineering communication** that keeps the PR workflow (`0.2.8`) moving: clear comments → faster resolution → fewer misunderstandings → safer merges. This lesson is conceptual (`понимание`), but the skills are practiced in every PR you touch.

## Core Concepts

### Why Teams Review Code

Code review addresses more than defect detection — though that matters. It **spreads knowledge** across the team: when you read a colleague's authentication middleware, you learn how your system handles tokens. It **enforces standards** — error handling, logging, EF query patterns — without relying on one tech lead's memory. It **improves design** before merge cost is paid: cheaper to rename a DTO in PR than after deploy. It **builds ownership**: shared codebase means shared responsibility, not siloed experts.

Research and industry experience consistently show review catches a subset of bugs; **tests and CI** catch others. Review's unique value is human judgment on readability, security smells, and fit with architecture — things linters miss.

### The Reviewer's Mindset

Effective reviewers start from **respect and curiosity**, not prosecution. The author spent hours on this diff; your job is to help it land safely, not to prove superiority.

**Review the code, not the person.** Say "this method allocates on every request" not "you always do this wrong." **Ask questions** when intent is unclear: "Could we use `AsNoTracking` here because this is read-only?" often opens better discussion than "Wrong EF usage." **Be specific** — point to line, suggest alternative or explain risk. **Prioritize**: blocking security issue before optional naming preference.

Block merges only for **real risk**: correctness, security, missing tests for critical paths, breaking API contracts without migration. Everything else can be **non-blocking nits** or follow-up tickets.

Time-box reviews. Long PRs (`0.2.8`) deserve split scope or async video walkthrough — not silent resentment.

### Giving Feedback — Practical Patterns

**Use tiers** (many teams encode in templates):

- **[blocking]** Must fix before merge — bug, security, data loss risk.
- **[suggestion]** Would improve code; author decides.
- **[nit]** Style preference; non-blocking.
- **[question]** Seek understanding; not necessarily change.
- **[praise]** Reinforce good patterns — reviews are not only criticism.

Example on ASP.NET PR:

> **[question]** Why is `SaveChangesAsync` called inside the loop? Could batching reduce round-trips?
>
> **[suggestion]** Consider returning `ProblemDetails` for validation errors — consistent with our API guidelines.
>
> **[nit]** `userId` → `userId` naming matches convention elsewhere; optional.

On React PR:

> **[blocking]** Stored XSS risk: `dangerouslySetInnerHTML` with unsanitized CMS content. Encode or sanitize server-side.
>
> **[praise]** Nice extraction of `useOrderSummary` hook — readable and testable.

**Actionable** means the author knows the next step. "This is confusing" is weak. "Extract validation into `OrderValidator` so controller stays thin" is strong.

Avoid **bike-shedding** — arguing longest over trivial style while missing SQL injection. Avoid **ghost approvals** — approving without reading because "they always get it right" — defeats the gate.

### Receiving Feedback — Practical Patterns

As author, **assume good intent** — reviewers want shippable code, not embarrassment. **Read all comments before replying** — batch understanding, avoid reactive piecemeal arguments. **Clarify** when needed: "Good catch — I'll fix" or "I chose X because Y; open to Z if you prefer."

**Do not take nits personally** — they should be labeled non-blocking; fix or push back politely. **Fix or explain** — every blocking comment needs resolution: new commit, reply with rationale, or request call if stuck.

Use **Conventional Commits** for follow-up: `fix: address review — sanitize CMS HTML`. Mark threads **resolved** in UI when done — signals completion to reviewer.

When you disagree, **argue with evidence**: benchmark, spec link, security doc — not rank or volume. Escalate to tech lead if deadlock on architecture.

**Thank reviewers** — especially for catching subtle bugs. Culture compounds.

### Blocking vs Non-Blocking

| Level | Merge impact | Examples |
|-------|--------------|----------|
| Blocking | Must resolve | Security flaw, wrong business logic, no tests on payment flow |
| Suggestion | Author choice | Alternative pattern, minor refactor |
| Nit | Optional | Variable name, comment wording |
| Question | Clarify only | Intent unclear |

Reviewers who mark everything blocking burn trust. Authors who ignore blocking comments burn `main`.

### Async PR Culture

Most review happens **asynchronously** in GitHub/GitLab comments, not meetings. Write for **future reader** — complete sentences, links to docs. Use **suggested changes** feature when trivial. Avoid sarcasm and ALL CAPS — tone disappears in text.

**Response time norms** vary; Middle developers aim for same-day acknowledgment on assigned reviews. Snoozing review duty blocks entire team's PR queue.

When feedback volume is high, **pair briefly** — five minutes on video can replace twenty comment round-trips.

## Under the Hood

Code review tools store comments as **metadata on diffs** — line anchors, threads, resolution state. They do not modify Git objects until you push commits. **CODEOWNERS** routes files to experts. **Required approvals** enforce minimum human gate before platform allows merge (`0.2.8`).

Some teams integrate **static analysis bots** (Sonar, CodeQL) — bot findings are feedback too; humans prioritize and override when false positives.

Review metrics (time-to-first-review, comments per PR) inform process improvement — but weaponizing metrics creates gaming; use carefully.

## Syntax / Commands / API

Review is mostly platform UI, but Git supports the loop:

```bash
# After review comments
git add .
git commit -m "fix(auth): address review — validate audience claim"
git push

# Optional: respond via CLI
gh pr comment 42 --body "Fixed in latest push — good catch on null check."
gh pr review 42 --approve --body "LGTM after null guard added."
```

**LGTM** ("Looks good to me") — approval shorthand; use only when you actually verified changes.

## Examples

### Example 1: Good vs bad blocking comment

**Bad:** "This is wrong."

**Good:** **[blocking]** Returning 200 with `{ success: false }` breaks our HTTP contract — clients expect 4xx for validation failure. Suggest `return ValidationProblem(ModelState)`.

Author can act immediately.

### Example 2: Author response

Reviewer: **[suggestion]** Cache `IOptions` snapshot in constructor instead of injecting `IOptionsMonitor` if values are static.

Author: "Valid — Monitor needed here because we hot-reload feature flags per `/docs/feature-flags`. No change; added comment in code."

Disagreement resolved with **documented rationale**.

### Example 3: Nit etiquette

Reviewer: **[nit]** Prefer `is not null` over `!= null` per C# team style guide.

Author fixes in 30 seconds or replies "Will align in follow-up PR to keep this diff focused." Both acceptable if nit labeled non-blocking.

### Example 4: Toxic pattern to avoid

"Did you even test this?" — accusatory, no actionable path. Replace with: **[blocking]** Submit button still enabled when form invalid — reproduce: leave email empty, click Submit, request fires. Expect disabled state per spec.

## Common Mistakes & Anti-patterns

**Reviewer bikeshedding** — endless style while logic unchecked.

**Author defensiveness** — "works on my machine" without addressing concern.

**Rubber-stamping** — approve in seconds on 500-line security PR.

**Drive-by scope creep** — "while you're here, refactor entire module."

**Public humiliation** — shaming in shared channel instead of PR thread.

**Ignoring bot + human feedback** — merging red CI with approval (`0.2.8`).

**Stale review** — commenting on outdated diff after author force-pushed without re-read.

## Production & Real-World Notes

Regulated environments may require **formal review records** — PR history is audit trail. Security-sensitive repos mandate **two approvals** on auth/crypto paths.

**On-call hotfixes** may use expedited review — still not zero review; post-merge review common with revert readiness.

Remote teams across timezones rely on **written clarity** — ambiguous comments delay releases by a day.

Senior engineers model tone — juniors copy behavior.

## Comparison / Trade-offs

| Style | Pros | Cons |
|-------|------|------|
| Strict blocking bar | High quality main | Slower merge, reviewer fatigue |
| Lightweight nits-only culture | Fast merges | Drift in standards |
| Pair review live | Rich context | Hard to scale async |

Balance: **block on risk**, **suggest on craft**, **nit optionally**.

## Quick Reference

```
WHY REVIEW: bugs + knowledge + standards + design + ownership

GIVING:
  - code not person | specific | actionable | prioritized
  - tiers: blocking | suggestion | nit | question | praise

RECEIVING:
  - assume good intent | read all first | fix or explain with evidence
  - conventional commits for follow-ups

BLOCK merge: correctness, security, critical untested paths
DO NOT BLOCK: pure style nits (unless team policy)

Async: complete sentences, no sarcasm, resolve threads when done
```

## Key Takeaways

1. Code review is **collaboration**, not gatekeeping theater.
2. **Specific, actionable, kind** feedback speeds PR workflow (`0.2.8`).
3. **Blocking** only for real risk; label nits as non-blocking.
4. Authors **fix or explain** — evidence over ego.
5. **Questions** often beat accusations for learning.
6. **Praise** good patterns — culture matters.
7. Etiquette scales async distributed teams.
