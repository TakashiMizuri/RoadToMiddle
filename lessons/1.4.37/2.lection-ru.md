## Why This Matters

Синхронная отладка интуитивна: call stack показывает, кто кого вызвал. **Async ломает иллюзию.** На паузе внутри `await db.QueryAsync()` stack часто показывает только **текущий continuation** — не controller, начавший request. **Deadlock** от `.Result` может показать заблокированные threads **без явного владельца**. **Fire-and-forget** `Task` падает на thread pool без связи с HTTP request.

Middle-разработчики тратят время на production-like баги: API timeout, зависший spinner, `AggregateException`. Без async-aware debugging — угадывание. С **Tasks window**, **Parallel Stacks** и **async stack trace** (.NET 6+ / VS 2022+) видно **какие tasks ждут**, **чего ждут**, **логическую цепочку причин**.

Урок про hands-on tooling — дополнение к теории `1.4.1`–`1.4.35`. Используешь еженедельно в ASP.NET + EF Core.

## Core Concepts

### Почему call stacks «врут» в async

Компилятор разбивает метод с `await` на **state machine**. После await выполнение может продолжиться на **другом thread pool thread**. Физический stack в точке resume не включает исходный synchronous caller линейно.

До enhanced diagnostics stack при exception выглядел так:

```
at TaskAwaiter.ThrowForNonSuccess(...)
at UserRepository.GetByIdAsync(...) line 42
```

**Controller action** отсутствовал — видно падение repository, не endpoint.

**.NET 6+** улучшил **async stack trace** для thrown exceptions. **VS 2022 17.4+** показывает **Async Stack Trace** в UI. **Tasks** нужен для **in-flight** waits без exception.

### Окно Tasks (Visual Studio)

При отладке: **Debug → Windows → Tasks**.

Список **Task**, о которых знает debugger:

| Информация | Значение |
|------------|----------|
| Id | Идентификатор Task |
| Status | Created, Waiting, Running, RanToCompletion, Faulted, ... |
| Method | Исходный async-метод |
| Location | Где выполняется логика |

**Сценарии:**

1. **Hang** — thread pool занят? Много `Waiting` на одном sync primitive.
2. **Never completes** — `WaitingForActivation` без перехода (нет `SetResult`, deadlock).
3. **Fire-and-forget** — faulted tasks видны, пока не собраны GC и отслеживаются debugger.

ПКМ → **Flag**. Комбинируй с break на `TaskScheduler.UnobservedTaskException`.

### Parallel Stacks и async

**Debug → Windows → Parallel Stacks** — threads и task-based views: какие tasks на каких threads, **blocked** связи. Request «висит», ASP.NET thread вернулся в Kestrel, но **логическая работа** на thread pool — Parallel Stacks + Tasks восстанавливают картину.

### Типичные сценарии

**Застряло на `await`:** Pause. Tasks — твой `Task`. Если `Waiting` — ищи **sync block** (`.Result` на другом thread). **Threads:** два thread, один на `Monitor`, другой держит lock в legacy sync code.

**Нет контекста exception:** **Tools → Options → Debugging → General → Enable .NET Async**. Повтори — dialog покажет async chain.

**async void handler:** Fault как unhandled на thread pool с shallow stack. Tasks может не показать как обычный `Task` — предпочитай `async Task` handlers.

### VS Code / dotnet

C# Dev Kit — ограниченный **Tasks** UI. Практика:

- **`dotnet-dump analyze`** — `clrthreads`, `syncblk`
- **Logging** с correlation IDs на каждом `await`
- **OpenTelemetry** spans

Для глубоких hangs в CI — Visual Studio или WinDbg; знай ограничение.

## Under the Hood

Debugger цепляется к **CLR thread pool** и **Task** через **ICorDebug** / **IAsyncDebug**. При смене state `Task` — события в Tasks window. **Stack trace enhancement** использует metadata в `Task` (.NET 5+ улучшения для `Exception.StackTrace` в async).

`ConfigureAwait(false)` **не** убирает task из Tasks — только **SynchronizationContext** resume.

**IAsyncStateMachine** — generated types; frames показывают **MoveNext** и `state: N`. `state: -1` — часто completed; положительные — await points.

## Examples

### Workflow «API never returns»

1. F5, breakpoint на `await _service.GetAsync()`.
2. Tasks — запомни Id `GetAsync`.
3. Step over await — не вернулось → Break All.
4. Tasks: тот же Id `Running`/`Waiting`?
5. Parallel Stacks: блок на SQL/network?
6. `Waiting` на `ManualResetEventSlim` — ищи `.Result`/`.Wait()` в chain.

### State machine frame

```
OrdersService.<ProcessOrderAsync>d__3.MoveNext() line 28
```

Строка 28 — `await` boundary. Считай `await` от начала метода — третий await вероятно hang point.

## Common Mistakes & Anti-patterns

**Смотреть только main thread** — ASP.NET request thread часто idle после первого `await`.

**Игнорировать Faulted в Tasks** — открой inner exception.

**Пустой Tasks ≠ нет async** — короткие tasks завершаются между breaks; tracepoints/logging.

**Release без PDB** — async stacks деградируют.

**Step into TaskAwaiter** — step **out** в user code.

## Production & Real-World Notes

Production hangs: **dotnet-trace**, dumps, **Application Insights**. Модель из Tasks переносится: **waiting task**, **ожидание**, **инициатор**.

**OpenTelemetry** `Activity` вокруг service calls — distributed traces как always-on Tasks window.

В code review: «Если этот `await` зависнет, что в Tasks?» — ловит sync-over-async.

## Key Takeaways

- Async дробит stacks; **Tasks** — logical task state.
- **Parallel Stacks** связывает thread pool с blocked tasks.
- **.NET 6+** улучшает exception async stacks.
- Hangs часто = **sync-over-async** — видно в Tasks/Threads.
- **async void** трудно отлаживать — избегай в app logic.
- Production: dumps/tracing; local: Tasks + breakpoints на await.
