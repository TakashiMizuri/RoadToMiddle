# Compare Sync-over-Async Fixes in Legacy Code

> Roadmap: `1.4.41` · Node: `1.4` — C# async · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Define **sync-over-async** and **async-over-sync** and why both appear in legacy .NET.
- Compare fixes: **`await` refactor**, **`GetAwaiter().GetResult()`**, **`Task.Run` offload**, **`IHostedService` bridge**.
- Rank solutions by **correctness**, **scalability**, **risk** in ASP.NET Core context.
- Recognize **library boundaries** where you cannot change signatures.
- Plan **incremental migration** from sync public API to async-all-the-way.
- Avoid **deadlock conditions** involving `SynchronizationContext` and `.Result`.

---

## Why This Matters

Legacy enterprise codebases mix eras: .NET Framework sync ADO.NET, third-party SDKs with only `bool Save()` methods, UI callbacks, and modern ASP.NET Core endpoints. Someone wrapped `GetDataAsync().Result` in a **sync repository interface** because the architect in 2015 said "async all the way is optional." Now Kestrel handles thousands of concurrent requests and **blocked thread pool threads** become outages.

Middle developers do not always greenfield. You inherit `ILegacyBillingClient.Charge(decimal amount)` that blocks on HTTP inside. This lesson is a **decision matrix** for fixes — not dogma that one hammer fits all — with production trade-offs.

## Core Concepts

### Definitions

**Sync-over-async:** Synchronous API **blocks** waiting on async work — `.Result`, `.Wait()`, `.GetAwaiter().GetResult()`.

```csharp
public User GetUser(int id) =>
    _repo.GetUserAsync(id).Result; // sync-over-async
```

**Async-over-sync:** `async` method that **wraps sync blocking I/O** — fake async.

```csharp
public async Task<User> GetUserAsync(int id) =>
    await Task.Run(() => _legacySyncRepo.GetUser(id)); // async-over-sync
```

Both harm scalability differently. Sync-over-async **blocks a thread** during real async I/O (worst). Async-over-sync **consumes two threads** (pool thread + blocked work) and still may block internally.

### Fix 1: Async All-the-Way (Preferred)

Refactor call chain to `async Task` / `await`:

```csharp
public async Task<User> GetUserAsync(int id, CancellationToken ct) =>
    await _repo.GetUserAsync(id, ct);
```

**Pros:** Correct thread usage, cancellation, composability.  
**Cons:** Ripple through interfaces — breaking change for public APIs.

**When:** You control the stack; greenfield modules; major version bump.

### Fix 2: Sync API Stays — Document and Isolate

If public contract must remain sync (COM interop, old SDK callback):

- **Isolate** blocking to dedicated **limited** thread pool or queue.
- **Never** call from ASP.NET request thread if avoidable.

```csharp
// Last resort — bounded parallelism
private static readonly SemaphoreSlim Gate = new(10);
public User GetUser(int id)
{
    Gate.Wait();
    try { return GetUserCoreAsync(id).GetAwaiter().GetResult(); }
    finally { Gate.Release(); }
}
```

Still blocks **some** thread — but caps blast radius. Document **"sync only for legacy callers."**

### Fix 3: Task.Run Offload (Async-over-Sync Bridge)

```csharp
public async Task<User> GetUserAsync(int id) =>
    await Task.Run(() => _legacy.GetUser(id), ct);
```

**Pros:** Quick unblock async controller; legacy sync code unchanged.  
**Cons:** Extra thread pool usage; sync I/O still blocks **a** thread; no true async I/O; bad under load.

**When:** Short-term bridge; CPU-bound legacy; low QPS internal tool.

**Not when:** High concurrency API — thread pool doubles pressure.

### Fix 4: Factory / Background Sync Bridge

Long sync initialization moved to **`IHostedService`**; runtime path stays async. Different from per-call `.Result` in request.

### Fix 5: Dual API Surface

```csharp
public interface IUserRepo
{
    Task<User> GetUserAsync(int id, CancellationToken ct);
}

// Obsolete sync shim for legacy only
[Obsolete("Use GetUserAsync")]
public User GetUser(int id) => GetUserAsync(id).GetAwaiter().GetResult();
```

New code uses async; compiler warns on sync. Gradual removal.

### Deadlock Mechanics (Why .Result Hurts)

Classic (UI / ASP.NET **old** with `HttpContext.Current`):

1. Thread A calls `.Result` on async method.
2. Async method needs to resume on **captured SyncContext**.
3. Thread A blocked — cannot process continuation.
4. **Deadlock.**

ASP.NET Core: **no request SyncContext** by default — many deadlocks **disappear** but **thread starvation** remains. Do not interpret "no deadlock" as "safe."

### Comparison Table

| Fix | Scalability | Effort | Risk |
|-----|-------------|--------|------|
| Async refactor | Best | High | Low if tested |
| Dual API + obsolete sync | Good long-term | Medium | Medium |
| Task.Run bridge | Poor at scale | Low | Medium |
| .Result in request | Worst | None | High |
| Bounded gate + .Result | Controlled harm | Medium | Medium |

## Under the Hood

`.Result` calls `Task.Wait` internally — blocks calling thread, escalates `AggregateException`, does not flow `ExecutionContext` optimizations as cleanly as `await`.

`Task.Run` queues work to thread pool — separate thread blocked on sync I/O inside delegate while caller awaits wrapper `Task` — **two pool interactions** per call.

**ConfigureAwait(false)** on internal async before `.Result` reduces UI deadlock, **not** starvation.

## Common Mistakes & Anti-patterns

**"ASP.NET Core has no SyncContext so .Result is fine"** — thread pool death still real.

**Task.Run everywhere** instead of fixing SDK usage — masks problem.

**Calling sync legacy from minimal API without bridge** — blocks request path.

**Removing sync shim too early** — breaks unknown consumers.

**Library exposes only `.Result` wrapper** publicly — pushes debt to all callers.

## Production & Real-World Notes

Measure **ThreadPool queue length**, **active threads**, **request latency** before/after migration. Use **feature flags** for new async endpoints parallel to old sync Windows service consumers.

For **third-party SDK** without async: vendor issue; isolate with dedicated microservice that speaks async outward.

Document in **ARCHITECTURE.md**: allowed patterns, forbidden in request path.


## Key Takeaways

- **Sync-over-async** blocks threads; **async-over-sync** wastes pool without real async I/O.
- **Preferred fix:** propagate `async`/`await` through stack.
- **Task.Run** — tactical bridge, not strategic architecture.
- ASP.NET Core avoids classic deadlock, **not** starvation.
- **Dual API** enables incremental migration.
- Rank fixes by **load profile** and **ownership** of code.
