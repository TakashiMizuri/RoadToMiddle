# Proxy vs Reverse Proxy vs Load Balancer — краткая выжимка

> `0.4.8` · ~10 мин на повторение

## Главная идея

**Forward proxy** — от имени **клиента** (corporate egress). **Reverse proxy** и **load balancer** — от имени **сервера** на edge: TLS на 443, routing, распределение по Kestrel instances. Production: `Browser → LB/reverse proxy → internal :8080`.

## Ключевые пункты

- Reverse proxy **скрывает** backends; client не знает count и IP instances.
- **TLS termination** на edge — Kestrel на private network часто plain HTTP.
- **L4 LB** — IP/port; **L7 LB** — HTTP paths, Host, cookies.
- **Health checks** убирают dead instances; **sticky sessions** — только если really needed.
- **`UseForwardedHeaders`** + **`KnownProxies`** — иначе wrong scheme/IP.
- **502** — proxy не получил valid response от backend.
- Forward и reverse — **разные** роли; nginx/ALB часто combine reverse + LB.

## Самые важные примеры

**Traffic path:**
```
Client --TLS:443--> nginx/ALB --HTTP:8080--> Kestrel (×N)
```

**Forwarded headers:**
```csharp
app.UseForwardedHeaders(/* XForwardedFor | XForwardedProto */);
```

## «Под капотом» в одном абзаце

Два TCP connection: client↔proxy (TLS decrypt) и proxy↔backend (often plain HTTP в VPC). L7 ALB parse HTTP и match rules до выбора target group. Round-robin на upstream — каждый request может hit другой pod; stateless API + JWT избегает sticky. Spoof `X-Forwarded-Proto` опасен, если Kestrel exposed напрямую — trust только от IP load balancer.

## Частые ошибки

- Называть всё «proxy» без forward/reverse.
- HTTPS redirects генерируются как `http://` без forwarded headers.
- L4 balancer для path-based routing.

## Мнемоника

> **Forward** = ты выходишь в интернет через посредника. **Reverse** = интернет заходит к тебе через посредника.
