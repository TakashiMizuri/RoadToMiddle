# Структура `.git/`: objects, refs, HEAD

> Roadmap: `0.1.4` · Узел: `0.1` — Git: основы · Глубина: **глубоко**

## Learning Objectives

После этого урока ты сможешь:

- Ориентироваться в каталоге `.git/` и объяснять назначение каждого ключевого компонента.
- Описать четыре типа Git-объектов: **blob**, **tree**, **commit**, **tag**.

- Объяснить **HEAD** как symbolic reference и что происходит при `git checkout`.

- Понимать **loose objects** vs **packfiles** на концептуальном уровне.

## Why This Matters

В уроке `0.1.3` ты изучил повседневные команды: `git init`, `git add`, `git commit`, `git log`. Это **интерфейс** — руль и педали. Каталог `.git/` — **машинное отделение**, механизм, который реально хранит history и делает возможными все команды.

Большинство разработчиков месяцами «водят» Git, не заглядывая в `.git/`. Это работает, пока что-то не ломается. Коллега сообщает о «detached HEAD», и ты не понимаешь, сохранена ли его работа. CI утверждает, что commit пропал. Кто-то случайно выполнил опасную команду и спрашивает, можно ли восстановить history. Репозиторий кажется повреждённым после плохого merge или shallow clone. В каждом случае лечение начинается с одного вопроса: *что именно Git хранит и где?*

Middle-разработчик не зубрит каждый файл в `.git/`. У него есть **ментальная модель**. History — не список diff'ов в таблице базы данных. Это **граф immutable objects**, связанных криптографическими hash'ами. Имена веток — не папки с файлами, а **лёгкие указатели**: крошечные текстовые файлы вида «tip ветки `main` — это commit `abc123`». `HEAD` — обычно ещё один маленький файл с ответом на вопрос «где я сейчас?». Когда эта модель укладывается в голове, `git reflog`, `git fsck`, merge conflicts, rebase и detached HEAD перестают быть магией.

Этот урок открывает капот. Мы пройдём по layout `.git/`, разберём четыре типа objects, увидим, как refs и HEAD стоят над object store, и проследим, что происходит на диске при stage и commit. К концу ты должен уметь объяснить модель хранения Git коллеге у доски — и знать, какой рычаг дёрнуть, когда что-то пошло не так.

## Core Concepts

### Каталог `.git/`: сам репозиторий

Когда ты выполняешь `git init` в папке проекта, Git создаёт скрытый каталог `.git/`. Этот каталог **и есть** репозиторий. Всё остальное — исходники, README, конфиги в IDE — живёт в **working tree**, то есть в checkout одного конкретного snapshot. Working tree изменяем и можно «выбросить»: если удалить все tracked-файлы снаружи, но `.git/` останется, весь проект можно восстановить из object database.

Аналогия простая. Working tree — черновик на столе. `.git/` — архив в подвале: каждая версия, которую ты когда-либо сохранял, с перекрёстными ссылками. `git checkout` копирует snapshot из архива на стол. `git add` и `git commit` решают, что с стола запечатывается в архив.

Свежий репозиторий после `git init` содержит скелет. Минимум: `HEAD` (текущая позиция), `config` (настройки, позже — remotes), `description` (чаще в bare repos), `hooks/` (скрипты на события вроде pre-commit), `info/` (в т.ч. global exclude), `objects/` (пока почти пустой, но здесь будет всё содержимое), `refs/` (указатели веток и тегов). После первого `git add` появляется `index` — staging area как бинарный файл на диске.

По мере работы добавляются файлы и папки. `refs/heads/main` будет хранить сорокасимвольный hash последнего commit на `main`. После `git fetch` появится `refs/remotes/origin/main` — где Git в последний раз видел remote-ветку. `logs/` хранит **reflog** — страховку, куда раньше указывали tip'ы веток. `COMMIT_EDITMSG`, `FETCH_HEAD`, `ORIG_HEAD` — служебные файлы на время операций.

Схема ниже не для заучивания наизусть, а для карты в голове. Обрати внимание на разделение: `objects/` — **содержимое**, `refs/` — **имена tip'ов commit'ов**, `HEAD` — над refs, говорит, какое имя (или сырой commit) сейчас checkout'нут.

```
my-project/
├── .git/                 ← РЕПОЗИТОРИЙ
│   ├── HEAD              ← «где я?»
│   ├── config            ← настройки, remotes
│   ├── hooks/            ← скрипты на события
│   ├── info/             ← global exclude
│   ├── objects/          ← ВСЁ содержимое (blob, tree, commit, tag)
│   ├── refs/             ← указатели веток/тегов
│   └── index             ← staging area (бинарный)
├── README.md             ← working tree
└── src/
```

### Content-addressable storage: идентичность = содержимое

Прежде чем назвать четыре типа objects, нужен принцип, который их объединяет. Git не выдаёт случайные serial number'ы. Он использует **content-addressable storage**: идентификатор object **есть** криптографический hash его содержимого.

В большинстве репозиториев сегодня это **SHA-1** — сорок hex-символов. (С Git 2.29+ для новых repo опционально **SHA-256**; layout и модель те же, меняется только длина hash.) При записи Git вычисляет `hash = SHA-1(type + " " + size + "\0" + content)`, сжимает zlib и пишет в путь из hash: первые два символа — подкаталог, остальные тридцать восемь — имя файла.

Последствия глубокие. **Determinism:** одинаковое содержимое → один hash на любой машине. **Integrity:** один испорченный бит — hash не сойдётся, Git увидит corruption. **Deduplication:** два файла с одинаковыми байтами — один blob. Storage растёт от **изменений**, а не от копий.

Hash — не метаданные «рядом» с content. Он **и есть** identity object. Когда commit указывает на parent hash, он ссылается на точный проверенный набор байтов. Эту цепочку мы развиваем в `0.1.5`.

### Четыре типа objects

Object database Git намеренно минимальна. Вся history — один из четырёх типов: **blob**, **tree**, **commit**, **tag**. У каждого та же схема content-addressing.

**Blob** — проще всего: сырые байты файла, без имени, без path, без permissions. `git add README.md` читает файл, оборачивает в blob header, хеширует, сохраняет. Второй файл с тем же content под другим именем в index ссылается на тот же blob hash. Два path — один object.

**Tree** — каталог в терминах Git. Плоский список записей: имя файла или подпапки, mode (обычный файл, executable, subdirectory), hash object. Tree для `src/` строится снизу, root tree ссылается на него. Snapshot commit'а — не zip файлов, а указатель на root tree, который разветвляется на trees и blobs.

**Commit** связывает snapshot с history и metadata: строка `tree`, ноль или больше `parent`, `author` и `committer` с timestamp, message после пустой строки. Commit **не хранит diff'ы** — только tree. `git diff` сравнивает trees и выводит разницу на лету. Snapshot'ы в storage, diff'ы по запросу — поэтому branching дёшев, а random access быстр.

**Tag** — именованная точка в history. **Lightweight tag** — файл в `refs/tags/` с hash commit'а, без tag object. **Annotated tag** (`git tag -a`) — полноценный **tag object** с message, tagger и указателем на commit. Релизы часто делают annotated — можно подписать и оставить заметки.

Концептуально blob выглядит так — type и size входят в hash:

```
blob object (концептуально):

  blob 6\0hello\n
        ↑   ↑
     header  content
```

Tree — listing директории на hash'ах:

```
tree object (концептуально):

  tree 72\0
  100644 README.md\0<blob-hash>
  100644 app.cs\0<blob-hash>
  040000 src\0<tree-hash-for-src/>
```

`100644` — обычный файл, `040000` — поддиректория (subtree). Commit над tree добавляет время: кто, когда, от каких parent'ов пришли сюда.

### Refs: человекочитаемые имена для commit'ов

Если objects только по hash'ам из сорока символов, навигация мучительна. **Refs** — маленькие **mutable** файлы: дружелюбное имя → hash commit'а.

Локальные ветки в `.git/refs/heads/`. Файл `.git/refs/heads/main` — одна строка `a1b2c3d4...` — **tip** ветки `main`. `git branch feature/login` создаёт `refs/heads/feature/login` на текущем commit. Файлы не копируются — только новый указатель.

Теги — `refs/tags/`. Remote-tracking ветки — `refs/remotes/<remote>/`, например `refs/remotes/origin/main` после fetch.

Главное — mutability. **Objects immutable:** записанный blob, tree или commit не меняется. **Refs mutable:** каждый новый commit на `main` обновляет `refs/heads/main`. Старые commit'ы остаются в objects; они просто не tip, пока на них указывает ref, tag или ancestry.

Удалить ветку — удалить файл на ~41 байт, не гигабайты history. Commit'ы могут жить до garbage collection, если ещё reachable.

### HEAD: где ты в графе

**HEAD** отвечает: на каком commit основан working tree и (обычно) на какой ветке ты стоишь. Хранится в `.git/HEAD`. Чаще это **symbolic ref** — не hash напрямую, а строка `ref: refs/heads/main`: «разрешай мою позицию через ref ветки `main`».

```
HEAD ──→ refs/heads/main ──→ commit abc1234
         (symbolic)           (40-char hash)
```

`git checkout feature` пишет в HEAD `ref: refs/heads/feature` и подменяет файлы working tree snapshot'ом tip'а этой ветки. `git commit` создаёт commit с parent = текущий HEAD, двигает ref ветки вперёд. HEAD по-прежнему symbolic на ту же ветку — но ref уже на новом hash.

**Detached HEAD** — исключение. HEAD содержит сырой hash commit'а, без `ref: refs/heads/...`. Ты на commit'е, не на ветке. Смотреть можно, но новые commit'ы не привязаны к имени ветки. Переключился на ветку — они могут стать unreachable (reflog поможет ненадолго). Понимание HEAD как файла превращает detached HEAD в диагностируемое состояние, а не загадочную ошибку.

## Under the Hood

### Как `git add` создаёт objects

Проследим `git add README.md` на уровне байтов — здесь working tree впервые встречается с object database.

Git читает байты файла. Собирает header: `blob`, пробел, размер, `\0`, content. Прогоняет через SHA-1. Hash — identity. Zlib-сжатие, запись в `.git/objects/ab/cdef...` — это **loose object**, по одному файлу на object.

Параллельно обновляется **index** (`.git/index`). Index — не Git object, а mutable staging. Для каждого staged path: path, blob hash, stat metadata, **stage number** для merge conflicts. Index — чертёж следующего commit: path → blob hash для сборки trees при commit.

```
Working tree          Index (.git/index)         Objects
─────────────         ──────────────────         ───────
README.md  ──add──→   README.md → blob:abc       abc = blob "..."
src/app.cs ──add──→   src/app.cs → blob:def      def = blob "..."
```

Неизменённые файлы могут не пересчитываться, если metadata совпадает с index — важная оптимизация в больших repo.

Проверить можно вручную:

```bash
git hash-object -w README.md
git ls-files -s
```

Первая команда показывает: blobs существуют без commit'ов. Вторая — как index ссылается на них по path.

### Как `git commit` создаёт objects

`git commit` превращает snapshot в history. Git читает index — каждый path и blob hash — и строит **tree objects** снизу вверх. Для `src/app.cs` blob известен; tree для `src/` листит `app.cs`. Root tree листит `README.md` и subtree `src/`. Каждый tree хешируется и сохраняется как object.

Дальше **commit object**: root tree hash, parent (кроме первого commit'а), author, committer, message — hash, запись. Обновление ref текущей ветки, например `refs/heads/main`. HEAD symbolic на эту ветку — менять HEAD не нужно, сдвинулся ref.

Второй commit меняет только `README.md`. До commit'а HEAD через `refs/heads/main` на commit A. Index — новый blob для README, тот же для `src/app.cs`. После — ref на commit B, parent A, tree с новым README и переиспользованным subtree `src/`. Commit A и его objects не тронуты. History append-only.

```
До commit:
  HEAD → refs/heads/main → commit A
  Index: README.md (blob v2), src/app.cs (blob v1)

После commit:
  HEAD → refs/heads/main → commit B
                              ├── parent: A
                              └── tree: T2
                                    ├── README.md → blob v2
                                    └── src/ → tree
                                          └── app.cs → blob v1

  commit A и его tree остаются — immutable history
```

Эта схема объясняет дешёвый branching (новый ref, общие objects), новые hash'и при rebase (новые parent'ы) и `git reset --soft` (двигается ref, objects на месте).

### Loose objects vs packfiles

Сначала каждый add/commit пишет **loose objects** в `.git/objects/` — один zlib-файл на object. Запись проста и быстра.

Со временем objects тысячи. Похожие blobs (версии одного файла) отличаются чуть-чуть. `git gc` упаковывает reachable objects в **packfiles** `.git/objects/pack/pack-*.pack` + `.idx`. Внутри pack — **delta-compression** относительно похожих objects.

Packs вручную почти не трогают. Они объясняют вывод `git count-objects -vH` и pack'и при fetch. **Loose — write path; packs — оптимизированное хранение после housekeeping.**

```
Loose:     .git/objects/e6/9de29bb2d1d6434b8b29ae775ad8c2e48c5391
Packed:    .git/objects/pack/pack-a1b2c3d4.pack
           .git/objects/pack/pack-a1b2c3d4.idx
```

### Связь с уроком 0.1.3 и следующими темами

Команды из `0.1.3` теперь можно читать буквально. `git status` сравнивает working tree, index и tip текущей ветки — три разных слоя, о которых мы говорили. `git diff` без флагов показывает разницу working tree vs index; `git diff --cached` — index vs последний commit. `git log` обходит commit objects, начиная с hash, на который указывает `HEAD` через цепочку refs. Когда в `0.1.5` ты увидишь parent pointers и DAG, вспомни: каждый из тех commit objects уже лежит в `.git/objects/`, а ветка — всего лишь ref-файл на tip.

Эта картина также объясняет, почему `git clone` копирует весь `.git/` (или его shallow-вариант): без object database и refs у тебя нет history, только файлы последнего checkout. Почему `git push` отправляет objects, которых нет на remote: имена commit'ов — hash'и content, их нельзя «переименовать» при передаче. Почему merge conflict оставляет маркеры в working tree, а не в objects: conflict — состояние index и файлов на столе; commit object появится только после того, как ты соберёшь согласованный snapshot и запустишь `git commit`.

### Инспекция objects вручную

Middle-разработчик должен знать plumbing-команды — теория становится фактом на диске.

```bash
cat .git/refs/heads/main
cat .git/HEAD
git cat-file -t abc1234
git cat-file -p abc1234
find .git/objects -type f
```

На учебном repo после пары commit'ов: `git cat-file -p HEAD` — commit; tree hash из вывода → `git ls-tree <tree-hash>` — от имени ветки до blob за минуты.

## Syntax / Commands / API

Команды ниже — для отладки и изучения internals; в обычном feature-work ими не пользуются ежедневно, но каждая привязана к модели выше.

| Команда | Назначение |
|---------|------------|
| `git init` | Создать skeleton `.git/` |
| `git hash-object -w <file>` | Создать blob вручную |
| `git cat-file -t <hash>` | Тип object |
| `git cat-file -p <hash>` | Содержимое object |
| `git ls-tree <tree-hash>` | Listing tree |
| `git rev-parse HEAD` | HEAD → commit hash |
| `git symbolic-ref HEAD` | На какую ветку указывает HEAD |
| `git count-objects -vH` | Loose vs packed статистика |
| `git gc` | Pack + optimize |

`git rev-parse HEAD` особенно полезен: сам разворачивает symbolic refs.

## Examples

### Пример 1: анатомия свежего repo

Пятиминутный эксперимент закрепляет урок лучше теории.

```bash
mkdir demo && cd demo
git init
echo "# Demo" > README.md
git add README.md
git commit -m "Initial commit"
```

`ls .git/` — `HEAD`, `config`, `objects`, `refs`, `index`, `logs`... `cat .git/HEAD` — `ref: refs/heads/main`. `cat .git/refs/heads/main` — полный hash. `git cat-file -p HEAD` — commit с tree и message. Ты прошёл refs → commit без GUI.

### Пример 2: один content — один blob

```bash
echo "shared" > file-a.txt
echo "shared" > file-b.txt
git add file-a.txt file-b.txt
```

`git ls-files -s` покажет один blob hash для обоих path — один loose object на диске.

### Пример 3: symbolic vs detached HEAD

```bash
git checkout main
cat .git/HEAD    # ref: refs/heads/main

git checkout abc1234
cat .git/HEAD    # abc1234... (без "ref:")
```

Detached — сигнал создать ветку перед commit'ами или работать read-only.

## Common Mistakes & Anti-patterns

Опаснее всего править или удалять файлы в `.git/objects/` руками — hash не сойдётся, `git fsck` найдёт corruption. Удалить весь `.git/` «чтобы починить merge» — уничтожить всю history; останется только working tree, часто без полного прошлого.

Думать, что ветка «содержит копии файлов» — ошибка: ветка — ref на один commit → tree → blobs. Путать **index** и object store — staging-баги: index mutable, objects permanent.

Ждать, что commit хранит diff — путаница с amend и rebase. Commit хранит **tree**. Commit'ить в detached HEAD без ветки — commit'ы станут unreachable при checkout в сторону; reflog не бесконечен.

## Production & Real-World Notes

В ежедневной работе ты редко открываешь `.git/` вручную, но модель из этого урока проявляется в типичных сценариях. Когда `git pull --rebase` «переписывает» commit'ы, на диске появляются новые commit objects с новыми hash'ами; старые остаются до gc, если на них ещё указывает reflog. Когда коллега говорит «ветка ушла вперёд на пять commit'ов», он имеет в виду, что ref-файл теперь указывает на commit, от которого пять шагов parent-цепочкой до старого tip. Когда IDE показывает «detached HEAD», она просто отражает содержимое `.git/HEAD` без префикса `ref:`.

На серверах — **bare repos** (`core.bare=true`): без working tree, содержимое `.git/` в корне. **Shallow clone** (`--depth N`) без большей части objects; часть операций требует полной history. **Partial clone** и **sparse checkout** — для monorepos.

При сомнениях в целостности — `git fsck`. **Reflog** в `logs/refs/heads/` — спасение после ошибочного reset. CI кэширует objects или делает shallow clone — trade-off скорость vs полная history.

## Comparison / Trade-offs

У Git несколько слоёв состояния с разной mutability.

| Концепция | Mutable? | Где хранится |
|-----------|----------|--------------|
| Working tree | Да | Каталог проекта |
| Index | Да | `.git/index` |
| Objects | Нет | `.git/objects/` |
| Branch refs | Да | `.git/refs/heads/` |
| HEAD | Да | `.git/HEAD` |

Loose objects быстрее при записи; packfiles экономят диск после `git gc`. За жизнь repo используются оба.

## Quick Reference

```
.git/
  HEAD              → symbolic ref → branch → commit
  index             → staging: paths → blob hashes
  objects/          → blob | tree | commit | tag
  refs/heads/       → local branches
  refs/tags/        → tags
  refs/remotes/     → remote-tracking
  config            → remotes, user, settings
```

Цепочка одного commit:

```
commit → tree → (blobs + subtrees) → file contents
       → parent commit(s)
```

## Key Takeaways

1. `.git/` **есть** репозиторий; working tree — checkout одного commit.
2. Четыре типа objects: blob, tree, commit, tag — все content-addressed.
3. **Refs** — именованные указатели; **HEAD** — обычно symbolic ref на ветку.
4. `git add` → **blobs** + **index**; `git commit` → **trees** + **commit** + движение ref.
5. Objects **immutable**; refs и HEAD **mutable**.
6. **Packfiles** сжимают objects после `git gc`.
7. Эта модель объясняет branching, merge, detached HEAD и recovery.
