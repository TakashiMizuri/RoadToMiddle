# Answers: Pull Request / Merge Request workflow

## Scoring

- 90–100%: отлично
- 80–89%: pass
- <80%: повтори лекции

---

## Answers

### 1. [Recall]
**Answer:** Одна и та же концепция — запрос слить одну ветку в другую с review; разные названия платформ.

**Why:** Функционально идентичны: source → target + discussion + automation.

---

### 2. [Recall]
**Answer:** **Source (head)** — ветка с твоими изменениями. **Target (base)** — ветка, **в которую** вливают (обычно `main`).

**Why:** PR всегда направлен source → target.

---

### 3. [Understanding]
**Answer:** Защита стабильности **shared** history, обязательный **review**, **CI** перед интеграцией; предотвращение случайного breakage production pipeline.

**Why:** `main` — trunk для всех; direct push обходит gates.

---

### 4. [Understanding]
**Answer:** PR **обновляется**: новые commits появляются в списке, diff пересчитывается, CI часто re-run.

**Why:** PR привязан к tip branch ref на remote.

---

### 5. [Understanding]
**Answer:** **Merge commit** сохраняет все commits PR + merge node на `main`. **Squash** сжимает все commits PR в **один** commit на `main`.

**Why:** Squash упрощает log main; теряется granular history на main (`0.2.2`).

---

### 6. [Application]
**Answer:** `git fetch origin` → `git rebase origin/main` (или merge по policy) → resolve conflicts (`0.2.5`) → `git push` / `--force-with-lease`.

**Why:** Конфликт = расхождение feature и target; чинится локально.

---

### 7. [Application]
**Answer:** (примеры) Свежий `main` (fetch/pull); ветка запушена с `-u`; CI локально green; осмысленный title/description; scope PR небольшой; linked ticket; self-review diff.

**Why:** Снижает round-trips и конфликты.

---

### 8. [Analysis]
**Answer:** **C** — неверно. При squash на `main` остаётся **один** commit (детали видны на странице PR, не в linear log main).

**Why:** A, B, D верны.

---

### 9. [Analysis]
**Answer:** Большой PR — **cognitive overload**, долгий review, выше риск пропустить баги. Вместо: **разбить** на несколько PR по логическим slices (API, UI, tests отдельно если возможно).

**Why:** Small batch review — industry best practice.

---

### 10. [Understanding]
**Answer:** Reviewer и QA могут **проверить** без угадывания; снижает время approval; документирует expected behavior.

**Why:** PR description — контракт для проверки изменения.

---

### 11. [Application]
**Answer:** Локально: `git switch main && git pull` и `git branch -d feature/...`. Remote: `git push origin --delete feature/...` (или кнопка в UI).

**Why:** Hygiene — не копить stale branches.

---

### 12. [Understanding]
**Answer:** Чтобы **уменьшить** конфликты позже, упростить review (diff vs актуальный main), ускорить merge; проактивный sync (`0.2.7`).

**Why:** Чем дольше PR открыт без sync, тем больше conflict debt.

---

### 13. [Recall]
**Answer:** Автоматически **закрывает** issue #42 при merge PR (GitHub keyword).

**Why:** Связь code ↔ ticket.

---

### 14. [Understanding]
**Answer:** (1) **Required CI checks** не green. (2) **Недостаточно approvals** / required reviewers не назначены. (3) Branch **not up to date** with target (policy). (4) PR в **draft**.

**Why:** Branch protection комбинирует условия.
