# Exercises: Amending Commits

> Complete after reading lectures and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Safe amend locally
**Goal:** Experience amend without push.

**Steps:**
1. Init repo, commit `readme.md`.
2. Edit `readme.md`, commit again with message `"second"`.
3. Realize you forgot to stage a new file `notes.txt`. Create it.
4. `git add notes.txt && git commit --amend --no-edit`
5. Run `git log --oneline` and `git show HEAD` — verify `notes.txt` is in the **same** commit as `"second"`.

**Success criteria:** One commit at tip contains both readme change and notes.txt; only two commits total in log.

---

### E2. Message amend
**Goal:** Change message only.

**Steps:**
1. Make a commit with message `"fix stuff"`.
2. `git commit --amend -m "Fix login validation error handling"`
3. Compare SHAs before/after (use `git reflog`).

**Success criteria:** You explain why SHA changed and old SHA appears in reflog.

---

## Medium

### M1. Amend vs new commit decision
**Goal:** Apply golden rule.

For each scenario, write **amend**, **new commit**, or **revert** and one sentence why:

1. Typo in message, feature branch, not pushed.
2. Forgot file, already merged to `main` yesterday.
3. Wrong delete in commit on your solo PR branch (not merged).
4. Bug shipped to production from tagged release.

**Success criteria:** 4/4 with valid reasoning referencing shared vs local history.

---

### M2. Observe non-fast-forward
**Goal:** See push rejection (simulated).

**Steps:**
1. Create bare remote: `mkdir ../remote.git && cd ../remote.git && git init --bare`
2. In project: add remote, push `main`.
3. Amend last commit locally.
4. Attempt `git push origin main` — observe rejection message.
5. **Do NOT force push.** Instead, create new commit fixing forward and push successfully.

**Success criteria:** You captured the error text and explained why forward commit worked.

## Hard (stretch)

### H1. Team policy document
**Goal:** Production thinking.

Write 5 bullet points for a team `CONTRIBUTING.md` section: **"When to amend vs when not to"**. Include: feature branches, PR squash merge, protected `main`, hotfix branches.

**Success criteria:** Mentions golden rule, force-push on PR branches, revert for production.
