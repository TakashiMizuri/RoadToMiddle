# Generics: Classes, Methods, and Constraints

> Roadmap: `1.3.1` · Node: `1.3` — C# advanced topics · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Explain **why generics exist** and how they differ from `object`-based collections.
- Declare **generic classes**, **methods**, and **interfaces** with type parameters.
- Apply **`where` constraints** (`class`, `struct`, `new()`, interface, base type) and predict compile-time errors.
- Distinguish **open types** from **closed constructed types** and read generic metadata.
- Describe how the **CLR implements generics** (reification vs erasure) and why value-type generics avoid boxing.
- Identify **common mistakes**: constraint over-specification, generic variance confusion (preview of `1.3.2`), and invalid `new()` on abstract types.

## Why This Matters

Before generics, reusable containers looked like `ArrayList` — every element stored as `object`. Reading an `int` required a cast and risked **`InvalidCastException` at runtime**. Generics move that check to **compile time**: `List<int>` cannot accidentally receive a `string`. You get **type safety**, **better IDE support**, and often **zero boxing** for value types.

Generics appear everywhere in modern C#: `List<T>`, `Dictionary<TKey, TValue>`, `Task<T>`, `IEnumerable<T>`, repository interfaces `IRepository<TEntity>`, and ASP.NET Core middleware that wraps `RequestDelegate`. Middle developers must not treat `<T>` as syntax sugar — they must understand **what the compiler generates**, **what constraints allow**, and **why `where T : class` changes nullable reference type analysis**.

Without this foundation, LINQ (`1.3.11`), dependency injection (registering `IOptions<T>`), and EF Core (`DbSet<T>`) remain opaque API calls instead of a coherent type system.

## Core Concepts

### The Problem Generics Solve

Imagine a stack that stores anything:

```csharp
public class ObjectStack
{
    private object[] _items = new object[8];
    private int _count;

    public void Push(object item) => _items[_count++] = item;
    public object Pop() => _items[--_count];
}
```

Pushing `42` **boxes** the `int` into a heap object. Popping requires `(int)Pop()` — wrong cast fails at runtime. The compiler cannot verify consistency between push and pop.

**Generics** parameterize types. `Stack<T>` is a **template**; `Stack<int>` and `Stack<string>` are **different closed types** at runtime in .NET:

```csharp
public class Stack<T>
{
    private T[] _items = new T[8];
    private int _count;

    public void Push(T item) => _items[_count++] = item;
    public T Pop() => _items[--_count];
}
```

`Stack<int>.Push(42)` stores the value directly in `T[]` — no boxing. `Pop()` returns `int` without cast.

### Type Parameters: Open vs Closed

`Stack<T>` is an **open generic type** — `T` is a placeholder. `Stack<int>` is a **closed constructed type** — all type arguments are concrete. You cannot instantiate an open type:

```csharp
// Stack<T> stack = new Stack<T>(); // compile error — T unknown
var stack = new Stack<int>();       // OK
```

Methods can be generic even inside non-generic classes:

```csharp
public static class Utils
{
    public static void Swap<T>(ref T a, ref T b)
    {
        (a, b) = (b, a);
    }
}
```

`Swap<int>` and `Swap<string>` share source code but the JIT compiles **specialized machine code** for reference types (often one shared code path) and **per value type** (separate native code for `int`, `double`, etc.).

### Generic Interfaces and Classes Together

Interfaces commonly use generics for composability:

```csharp
public interface IRepository<TEntity> where TEntity : class
{
    Task<TEntity?> GetByIdAsync(int id, CancellationToken ct = default);
    Task AddAsync(TEntity entity, CancellationToken ct = default);
}
```

Implementations close the type parameter:

```csharp
public class UserRepository : IRepository<User>
{
    // TEntity is User — compiler verifies method signatures match
}
```

Generic constraints on the interface propagate to implementers — you cannot implement `IRepository<User>` with a struct `User` if `where TEntity : class` is required.

### Constraints: Telling the Compiler What T Can Do

Without constraints, `T` is treated as **`object`** for members — you can only use `object` members on `T`. Constraints unlock APIs:

| Constraint | Meaning |
|---|---|
| `where T : struct` | `T` is non-nullable value type |
| `where T : class` | reference type (nullable in NRT context) |
| `where T : class?` | reference type, possibly nullable |
| `where T : notnull` | cannot be null (value or non-nullable ref) |
| `where T : unmanaged` | no managed fields — blittable for interop |
| `where T : new()` | parameterless constructor (for `new T()`) |
| `where T : SomeBase` | inherits or equals base |
| `where T : ISomeInterface` | implements interface |

Example combining constraints:

```csharp
public class Factory<T> where T : class, IEntity, new()
{
    public T CreateDefault() => new T();
}
```

`new()` requires a **public parameterless constructor**. Abstract classes fail this constraint at compile time even if they inherit `IEntity`.

**Multiple constraints** on one parameter are AND-ed. **Multiple type parameters** each have their own `where` clause:

```csharp
public static TOutput Convert<TInput, TOutput>(TInput input, Func<TInput, TOutput> map)
    where TInput : notnull
    where TOutput : struct
{
    return map(input);
}
```

### Generic Methods and Type Inference

The compiler often **infers** type arguments from arguments:

```csharp
int max = Utils.Max(3, 7); // infers Max<int>

public static T Max<T>(T a, T b) where T : IComparable<T>
    => a.CompareTo(b) >= 0 ? a : b;
```

When inference fails (no arguments tied to `T`), specify explicitly: `Utils.Max<double>(3, 7.5)`.

### Default Values and `default(T)`

For unconstrained `T`, `default(T)` is **`null`** for reference types and **zero-bit pattern** for value types. With `where T : struct`, `T?` (nullable value type) is available in modern C#.

## Under the Hood

### Reification in the CLR

.NET uses **reification**: generic type information survives to runtime. `typeof(List<int>)` and `typeof(List<string>)` are **different** `Type` objects. Contrast with Java's erasure (mostly compile-time only) — in C# you can reflect on `T` at runtime inside generic methods via `typeof(T)`.

At JIT time, the runtime generates code. **Reference type arguments** often share one implementation (all references are pointer-sized). **Each value type argument** gets its own JITted code — `List<int>` and `List<double>` use different array element sizes. This is why excessive generic value-type combinations can increase **native code size** (a trade-off for speed).

### IL and Metadata

A generic class compiles to IL with **type parameters** in metadata tokens. Call sites use **`callvirt`** / **`newobj`** on constructed types. Constraints are stored as metadata flags the compiler enforces; the runtime verifies some (e.g., inheritance).

Disassembling `Stack<int>.Push`:

- No boxing IL (`box`) for `int`.
- `ObjectStack.Push` with `int` would show `box int32` before `stelem.ref`.

### Boxing Avoidance and Performance

`List<int>` stores elements in `int[]` — contiguous, cache-friendly, no per-element heap objects. `ArrayList` with ints allocates a boxed object per element — more GC pressure. In hot paths (game loops, serialization buffers), this difference is measurable.

**Generic specialization** can also enable **devirtualization** and inlining when the JIT knows `T` is a struct implementing an interface (constrained generic interface calls).

### Limitations the Runtime Enforces

- Cannot use `T` as base class: `class Foo : T` — illegal.
- Cannot static-field `T` without generic class (each closed type gets its own static fields).
- **`typeof(T)`** works; **`new T()`** only with `new()` constraint.
- Enums as type parameters: treat as struct; no arithmetic on `T` without `where T : Enum` (.NET 7+) or interfaces.

## Syntax / Commands / API

```csharp
// Generic class
public class Pair<TFirst, TSecond> { ... }

// Generic method
public static void Print<T>(T value) => Console.WriteLine(value);

// Generic interface with constraint
public interface IComparable<T> where T : IComparable<T>
{
    int CompareTo(T other);
}

// Constraint clauses (after declaration)
public class Cache<TKey, TValue>
    where TKey : notnull
    where TValue : class, new()
{ }
```

Reflection:

```csharp
Type open = typeof(Dictionary<,>);
Type closed = typeof(Dictionary<string, int>);
Console.WriteLine(closed.IsGenericType);           // True
Console.WriteLine(closed.GetGenericTypeDefinition()); // Dictionary`2
```

## Examples

### Repository Pattern (Minimal)

```csharp
public interface IReadRepository<T> where T : class
{
    IReadOnlyList<T> GetAll();
}

public class EfReadRepository<T> : IReadRepository<T> where T : class
{
    private readonly DbContext _db;
    public EfReadRepository(DbContext db) => _db = db;

    public IReadOnlyList<T> GetAll() => _db.Set<T>().AsNoTracking().ToList();
}
```

`DbContext.Set<T>()` requires `T` to be an entity type — EF adds its own conventions; the generic constraint `class` matches reference-type entities.

### Generic Method with Inference

```csharp
public static class Guard
{
    public static T NotNull<T>(T? value, string paramName) where T : class
    {
        ArgumentNullException.ThrowIfNull(value, paramName);
        return value;
    }
}

User user = Guard.NotNull(maybeUser, nameof(maybeUser)); // T inferred as User
```

### Invalid: `new()` on No Constructor

```csharp
public abstract class Entity { }
// public class Bad<T> where T : Entity, new() { } // OK actually if T could be concrete
// But: new Bad<Entity>() // compile error — Entity has no public parameterless ctor usable for new()
```

## Common Mistakes & Anti-patterns

**Over-constraining `T`.** Adding `where T : class, new(), IEntity, IDisposable` when you only read data forces every consumer to satisfy all constraints and blocks struct projections. Start minimal; add constraints when the compiler demands a member.

**Using `object` or `dynamic` to escape generics.** Defeats compile-time checking. Prefer generic overloads or a non-generic interface slice (`IEnumerable` non-generic) only at boundaries.

**Assuming one static field across all `T`.** `GenericClass<T>.StaticCounter` is **per closed type** — `GenericClass<int>.StaticCounter` ≠ `GenericClass<string>.StaticCounter`.

**Confusing generic type parameters with method type parameters.** Shadowing: a method `void M<T>()` inside `class C<T>` has a **different** `T` unless you refer to the class parameter explicitly — rare but confusing in nested generics.

**Nullable reference types and `where T : class`.** Under `#nullable enable`, `T` inside the generic method may be treated as non-nullable reference type; `default(T)` is still `null` — flow analysis has limitations with unconstrained type parameters.

## Production & Real-World Notes

ASP.NET Core DI registers open generics: `services.AddScoped(typeof(IRepository<>), typeof(EfRepository<>))`. The container closes the type at resolution time when you inject `IRepository<Order>`.

Libraries expose generic extension points (`ILogger<T>`, `IOptions<T>`, `Channel<T>`) — category names in logs come from `typeof(T).Name`.

Performance-sensitive code (MemoryPack, System.Text.Json source generators) generates **type-specific serializers** per `T` to avoid reflection — generics at API level, specialized code at compile time.

Code review red flag: `List<object>` in internal business logic — usually means a missing generic abstraction.

## Comparison / Trade-offs

| Approach | Pros | Cons |
|---|---|---|
| **Generics** | Type-safe, often zero-boxing, composable | Larger JIT code for many value types; steeper signatures |
| **`object` + cast** | Simple, one runtime type | Runtime failures, boxing |
| **Inheritance (`StackInt`, `StackString`)** | Explicit | Code explosion, no inference |
| **`dynamic`** | Flexible | Runtime errors, no IntelliSense, slower |

Generics vs **interfaces without type params**: `IComparable` (non-generic, legacy) boxes value types; prefer `IComparable<T>`.

## Key Takeaways

- Generics provide **compile-time type safety** and **performance** by avoiding casts and boxing for value types.
- **Open types** (`List<T>`) vs **closed types** (`List<int>`) — only closed types are instantiated.
- **`where` constraints** unlock member access and are validated at compile time.
- .NET **reifies** generics — type info exists at runtime; value types get specialized JIT code.
- Static members in generic types are **per closed type**, not shared globally.
- DI and frameworks rely on **open generic registration** — understand closing type arguments at resolve time.
