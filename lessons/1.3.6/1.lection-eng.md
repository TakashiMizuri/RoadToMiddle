# Lambda Expressions: Expression Body vs Statement Body

> Roadmap: `1.3.6` · Node: `1.3` — C# advanced topics · Depth: практика

## Learning Objectives

After this lesson you will be able to:

- Write **lambda expressions** as **expression-bodied** and **statement-bodied** forms.
- Convert between **lambda**, **anonymous method** (legacy), and **local function**.
- Explain **closure** — captured variables, **display class**, lifetime implications (`1.3.5`).
- Use lambdas with **delegates**, **LINQ**, **events** — including **discard** and **explicit parameter types**.
- Apply **static lambda** (C# 9+) to avoid unintended captures.
- Avoid pitfalls: **loop closure**, **modified captured variable**, **async lambda** in wrong delegate type.

---

## Why This Matters

Lambdas are the default syntax for **`Where(x => x > 0)`**, **`Task.Run(() => ...)`**, **`button.Click += (s,e) => ...`**, and configuration callbacks in ASP.NET Core. Compact syntax hides machinery: compiler generates **private nested class** (display class) holding captured locals. Misunderstanding **closure in loop** (`for` + lambda) produces bugs that pass code review because syntax looks innocent.

Statement-bodied lambdas (`{ ... }`) enable multiple statements, local variables, and **`return`** — needed when expression form is insufficient. **Static lambdas** reduce allocations and accidental **`this`** capture in hot paths. This lesson completes the delegate chain from **`1.3.3`**–**`1.3.5`** with the syntax you will write most often.

## Core Concepts

### Syntax Forms

**Expression lambda** — single expression, implicit return:

```csharp
Func<int, int> square = x => x * x;
Func<int, int, int> add = (a, b) => a + b;
Action<string> log = msg => Console.WriteLine(msg);
```

**Statement lambda** — block body, explicit `return` if non-void:

```csharp
Func<int, int> factorial = n =>
{
    if (n <= 1) return 1;
    return n * factorial(n - 1); // recursive lambda — careful with self-reference
};
```

Zero parameters: **`() => ...`**. One parameter: **`x =>`** or **`(x) =>`** (parentheses required for explicit types: **`(int x) =>`**).

### Type Inference and Explicit Types

Usually compiler infers from **delegate target**:

```csharp
List<int> nums = new() { 1, 2, 3 };
var evens = nums.Where(x => x % 2 == 0);
```

When inference fails, cast delegate or specify types on parameters:

```csharp
Func<int, string> f = (int x) => x.ToString();
```

### Conversion to Delegate

Lambdas convert to **compatible delegate type** or **`Expression<Func<...>>`** (LINQ to Entities — deferred, not executed in memory):

```csharp
Expression<Func<User, bool>> expr = u => u.Age >= 18; // tree for EF
Func<User, bool> compiled = u => u.Age >= 18;         // IL for in-memory
```

**Method group** alternative: **`users.Where(users.Contains)`** when signature matches.

### Closures — Capturing Variables

Lambda **captures** outer local variables and **`this`** — lives as long as delegate instance:

```csharp
int threshold = 10;
Func<int, bool> above = x => x > threshold;
threshold = 20; // mutation visible when above invoked!
Console.WriteLine(above(15)); // False — uses current threshold
```

Compiler generates **display class** with fields for captures:

```csharp
// conceptual desugaring
class DisplayClass {
    public int threshold;
    public bool Lambda(int x) => x > threshold;
}
```

**Lifetime**: delegate keeps display class alive → keeps captured objects alive → **event leak** if subscriber captures **`this`** (`1.3.5`).

### Loop Closure Classic Bug

```csharp
var actions = new List<Action>();
for (int i = 0; i < 3; i++)
    actions.Add(() => Console.WriteLine(i));
foreach (var a in actions) a(); // prints 3, 3, 3 — not 0, 1, 2
```

**C# 5+** fixed **`foreach`** loop variable per iteration; **`for` loop** still shares one **`i`**. Fix:

```csharp
for (int i = 0; i < 3; i++)
{
    int copy = i;
    actions.Add(() => Console.WriteLine(copy));
}
```

Or use **`foreach`** with proper scoping.

### Static Lambda (C# 9+)

```csharp
Func<int, int, int> add = static (a, b) => a + b;
```

**`static`** — cannot capture locals or **`this`**; no display class allocation; useful in LINQ hot paths.

### `async` Lambda

```csharp
Func<Task<int>> work = async () =>
{
    await Task.Delay(100);
    return 42;
};
```

Use **`Func<Task>`** / **`Func<Task<T>>`**, not **`Action`** with **`async void`**. **`async () =>`** returning Task is correct.

### Discards and Unused Parameters

```csharp
button.Click += (_, _) => DoWork();
Array.ForEach(items, _ => count++); // invalid — need param name for body
```

Event handlers often **`(_, e)`** ignore sender.

## Under the Hood

Roslyn lowers lambda to:

1. **Static method** if no captures (or static lambda).
2. **Instance method on display class** if captures exist.

**Target** of delegate points to display class instance holding captured state.

**Expression trees** — lambda parsed into **`Expression` AST** instead of IL — EF Core translates to SQL.

**Local functions** (C# 7+) can capture like lambdas but support recursion and **`ref`** parameters cleanly:

```csharp
int LocalFactorial(int n) => n <= 1 ? 1 : n * LocalFactorial(n - 1);
```

## Syntax / Commands / API

```csharp
// Expression
x => x * 2
(x, y) => x + y
() => DateTime.UtcNow

// Statement
x => { var y = x * 2; return y; }

// Static (no capture)
static x => x * 2

// Explicit type
(int x) => x.ToString()

// Expression tree
Expression<Func<T, bool>> pred = t => t.Id > 0;
```

## Examples

### LINQ Filtering

```csharp
var active = users
    .Where(u => u.IsActive)
    .Select(u => u.Email)
    .ToList();
```

### Configuration Callback (ASP.NET style)

```csharp
builder.Services.AddOptions<MyOptions>()
    .Configure(o => o.Timeout = TimeSpan.FromSeconds(30));
```

### Statement Body with Validation

```csharp
Func<string, int> parse = s =>
{
    if (string.IsNullOrWhiteSpace(s))
        throw new ArgumentException("Empty", nameof(s));
    return int.Parse(s, CultureInfo.InvariantCulture);
};
```

### Avoiding Capture with Static

```csharp
var ids = items.Where(static item => item.IsValid).Select(static item => item.Id);
```

## Common Mistakes & Anti-patterns

**`for` loop closure** — all lambdas see final loop variable.

**Capturing large object** unnecessarily — hold delegate long-lived, object stays in memory.

**`async void` lambda** assigned to **`Action`**.

**Recursive expression lambda** without local function — self-reference limitations.

**Using lambda for `-=`** without storing reference (`1.3.3`).

**EF Core**: **`Expression<Func<>>`** required — plain **`Func`** may client-evaluate or fail translation.

## Production & Real-World Notes

**Roslyn analyzers** flag unnecessary captures; **static lambda** when possible.

**BenchmarkDotNet**: display class allocation measurable in tight loops.

**Testing**: replace **`Func<,>`** parameters with lambdas in unit tests — primary benefit of DI + Func.

**Minimal APIs**: route handlers **` () => Results.Ok()`** — lambda as entry point.

Prefer **local function** over statement lambda when recursion or **`ref`/`out`** needed.

## Comparison / Trade-offs

| Form | When |
|---|---|
| Expression lambda | Single expression, concise |
| Statement lambda | Multiple steps, try/catch |
| Local function | Recursion, reuse in method |
| Static lambda | No capture, performance |
| Method group | Existing named method |

## Quick Reference

| Syntax | Returns |
|---|---|
| `x => expr` | expr value |
| `x => { stmts; return v; }` | explicit return |
| `static x =>` | no captures |
| `async () =>` | Task (use Func<Task>) |

**Loop fix:** copy loop variable to inner local before lambda.

---

## Key Takeaways

- **Expression** vs **statement** lambda — one expression vs block + return.
- Lambdas convert to **delegates** or **expression trees**.
- **Closures** capture outer state — display class, lifetime extension.
- **`for` + lambda** — capture bug; copy variable or use foreach scoping.
- **`static` lambda** — no capture, less allocation.
- **`async`** lambdas need **`Func<Task>`**, not **`Action`**.
