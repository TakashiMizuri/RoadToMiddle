# Abstract classes vs Interfaces — decision guide

> Roadmap: `1.2.8` · Node: `1.2` — C# OOP · Depth: **глубоко**

## Learning Objectives

После этого урока ты сможешь:

- Назвать структурные отличия **`abstract class`** и **`interface`** в modern C#.
- Выбрать **abstract base**, когда нужны shared state, constructors, template methods.
- Выбрать **interface** для cross-cutting contracts и multiple inheritance of type.
- Комбинировать **class inheritance + interfaces** как в BCL и ASP.NET.
- Объяснить, почему C# — **multiple interfaces**, но **single base class**.
- Применять **decision checklist** для новых abstractions в production.

## Why This Matters

Уроки `1.2.6`–`1.2.7` — inheritance и polymorphism на **classes**. Архитектура разделяет два вопроса: **«Какое behavior обязано у каждой implementation?»** (contract) и **«Какую shared implementation/state централизовать?»** (reuse). **`interface`** — первое; **`abstract class`** — часто второе.

Неверный выбор → **`IBlob`** на 40 methods с copy-paste в implementers, или **`MegaBaseController`** hierarchies против composition. ASP.NET: **`Controller : ControllerBase`** (shared impl) + **`IActionFilter`**, **`IDisposable`** (contracts).

Middle-разработчик проектирует **`IRepository<T>`**, **`IEmailSender`**, **`IPipelineBehavior`** (interfaces) и **`Stream`**, **`DbContext`**, **`BackgroundService`** (abstract bases). Этот checklist — то, что senior применяет на design review.

## Core Concepts

### Abstract class

**`abstract class`** нельзя instantiate. Может содержать:

- **Abstract members** — без body; concrete derived must override.
- **Virtual members** — default + override.
- **Concrete members**, **fields**, **constructors**, **access modifiers**.

```csharp
public abstract class RepositoryBase<T> where T : class
{
    protected readonly DbContext Context;
    protected RepositoryBase(DbContext context) => Context = context;
    public abstract Task<T?> FindByIdAsync(Guid id, CancellationToken ct);
    public virtual IQueryable<T> Query() => Context.Set<T>();
}
```

**Single inheritance** — один base class. Abstract class = **is-a** с **partially defined** identity + shared wiring.

### Interface

**`interface`** — **contract**: signatures без forced common base. Modern C# — **default methods** (`1.2.10`), но роль — **capability**.

```csharp
public interface IEmailSender { Task SendAsync(...); }
public interface ITrackable { DateTime CreatedAt { get; } }
```

**Multiple interfaces** на одном class:

```csharp
public sealed class Invoice : Entity, IPrintable, IAuditable { }
```

State — в implementing class, не в interface (classic model).

### Сравнение

| Аспект | Abstract class | Interface |
|--------|----------------|-----------|
| Instantiate | Нет | Нет (сам interface) |
| Base limit | Один class | Много interfaces |
| Fields | Да | Нет instance fields (classic) |
| Constructors | Да | Нет instance ctors |
| Shared impl | Широко | Default methods (ограничено) |
| Смысл | is-a + shared DNA | can-do / capability |

### Когда abstract class

1. **Shared state** — **`_context`**, **`_logger`**, **`Id`**.
2. **Constructor enforcement** — invariants через base ctor.
3. **Template Method** — orchestration + virtual hooks.
4. **Protected helpers** — reuse без public noise.
5. **Single lineage** — **`Exception`**, **`Attribute`**.

### Когда interface

1. **Unrelated types** — **`IComparable`**, **`ILogger<T>`**.
2. **Multiple contracts** — disposable + cacheable + repository.
3. **DI/testing** — mock без production base.
4. **Avoid fragile base class** — нет forced ctor chain.
5. **Consumer owns hierarchy** — library **`IHandler`**, app **`HandlerBase`**.

### Комбинация

**`HandlerBase<TRequest> : IRequestHandler<...>`** — team abstract base; framework **`IRequestHandler`** для DI. **Interface для registration; abstract base для DRY.**

### Abstract vs concrete base

**`ControllerBase`** — concrete с **virtual** defaults. **`abstract class`**, когда **`new Base()`** always wrong — **`Shape`**, **`RepositoryBase`**.

## Under the Hood

**Abstract class** — normal object + vtable; **abstract** methods без RVA до concrete override.

**Interface** — **`InterfaceImpl`** на class; call через interface reference может идти через **interface map** (`1.2.9` explicit).

Multiple interfaces — без diamond class inheritance.

**API evolution:** новый **abstract** member ломает derived; новый interface member без default ломает implementers. Default interface methods (`1.2.10`) смягчают interfaces.

## Common Mistakes & Anti-patterns

**God interface**. **Abstract class с одним method** — скорее interface. **Inheritance только для reuse** (`1.2.11`). **Explicit interface без причины** (`1.2.9`).

## Production & Real-World Notes

Clean Architecture: interfaces в inner rings; infrastructure implements. **Small interfaces** (ISP).

---

## Quick Reference

Shared fields/ctors → **abstract class**. Capability/DI → **interface**. Оба — polymorphism.

**Вопрос:** *Shared state/ctor wiring у всех subtypes?* → abstract. *Только role?* → interface.

---

## Key Takeaways

- Abstract = single base + partial impl; interface = many contracts.
- CLR: vtable vs interface map.
- Production: **оба вместе**.
- Small interfaces; не deep abstract towers.
