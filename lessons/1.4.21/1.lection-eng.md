# TaskScheduler.FromCurrentSynchronizationContext

> Roadmap: `1.4.21` · Node: `1.4` — C# async · Depth: понимание

## Learning Objectives

After this lesson you will be able to:

- Explain **`SynchronizationContext`** and its role in **marshaling continuations** back to a specific thread.
- Describe **`TaskScheduler.FromCurrentSynchronizationContext()`** and when **`Task.Factory.StartNew`** uses it.
- Contrast **UI apps** (WinForms, WPF, MAUI) with **ASP.NET Core** (no request **`SynchronizationContext`**).
- Relate **`ConfigureAwait(false)`** to whether continuations capture context (**`1.4.8`**, **`1.4.9`**).
- Predict **deadlock** when UI thread **`.Result`** waits on async that posts back to UI.
- Choose correct patterns: **UI thread dispatch** vs **thread pool** in modern apps.

---

## Why This Matters

Before **`async/await`**, **`Task.Factory.StartNew`** with **`TaskScheduler.FromCurrentSynchronizationContext()`** was how background work **returned to the UI thread** to update controls. **`await`** automates this by **capturing** **`SynchronizationContext.Current`** (or **`TaskScheduler`**) at suspend and **posting** the continuation back.

If you **`await`** on a UI thread without **`ConfigureAwait(false)`**, continuations run on UI — correct for **`button.Text`**, wrong for CPU work that freezes the UI.

**ASP.NET Core** sets **`SynchronizationContext.Current` to `null`** — continuations default to **thread pool**. That's why **`ConfigureAwait(false)`** matters less in controllers but **critical in libraries** shared with UI or legacy ASP.NET.

Middle developers debugging **"works in API, deadlocks in WPF"** or mixing library code need this mental model — not to use **`FromCurrentSynchronizationContext`** everywhere, but to understand **where await resumes**.

## Core Concepts

### SynchronizationContext basics

**`SynchronizationContext`** abstracts **where a delegate runs**:

- **UI context** — post to UI message loop (single thread).
- **ASP.NET (legacy)** — **`HttpContext`**-aware context (System.Web).
- **Default** — **`null`** or thread pool.

**`Post(SendOrPostCallback, state)`** queues work; **`Send`** runs synchronously on target thread.

When you **`await`**, the awaiter asks: is there a captured context? If yes, schedule continuation via **`Post`**; if **`ConfigureAwait(false)`**, use thread pool.

### TaskScheduler.FromCurrentSynchronizationContext

**`TaskScheduler`** is lower-level scheduling abstraction. **`TaskScheduler.FromCurrentSynchronizationContext()`** returns a scheduler that queues tasks to **current `SynchronizationContext`**:

```csharp
var scheduler = TaskScheduler.FromCurrentSynchronizationContext();
Task.Factory.StartNew(() => UpdateUi(), CancellationToken.None,
    TaskCreationOptions.None, scheduler);
```

Pre-**async/await** pattern for **"run on UI after background work"**. Today prefer:

```csharp
await Task.Run(() => Compute());
label.Text = result; // resumes on UI if awaited on UI thread without ConfigureAwait(false)
```

**`StartNew`** with this scheduler is **legacy** — **`Task.Run`** ignores SyncContext; **`async/await`** replaces most uses (**`1.4.24`** why avoid **`StartNew`**).

### UI deadlock classic

```csharp
// WPF button handler — sync void
private void Button_Click(object sender, EventArgs e)
{
    string data = GetDataAsync().Result; // blocks UI thread
}

async Task<string> GetDataAsync()
{
    await Task.Delay(100); // continuation wants UI thread
    return "data";
}
```

UI thread blocked on **`.Result`**; continuation waits for UI thread → **deadlock**. Fix: **`async void`** handler with **`await`**, or **`ConfigureAwait(false)`** inside library (if no UI touch after).

### ASP.NET Core difference

**No `SynchronizationContext`** on request thread — **`await`** continuations run on **thread pool** (any thread). Deadlock from **UI pattern** rare; **sync-over-async starvation** remains (**`1.4.20`**).

Legacy **ASP.NET (System.Web)** had context — **`ConfigureAwait(false)`** in libraries prevented deadlocks when sync code blocked.

### ConfigureAwait connection

```csharp
await SomeLibraryMethodAsync().ConfigureAwait(false);
```

Library doesn't need UI; don't capture context — avoids deadlocks when caller blocks and reduces unnecessary marshaling.

In **ASP.NET Core app code**, **`ConfigureAwait(false)`** optional; in **shared libraries**, **default false** unless UI-specific API.

## Under the Hood

**`SynchronizationContext.Current`** stored in **`AsyncLocal`**-like flow for async — flows across **`await`** unless suppressed.

**`TaskScheduler.FromCurrentSynchronizationContext()`** wraps **`SynchronizationContext.Post`** into **`Task`** scheduling.

**`async/await`** compiler prefers **`SynchronizationContext`** over **`TaskScheduler`** for continuations when context exists.

## Syntax / API

| API | Role |
|-----|------|
| `SynchronizationContext.Current` | Captured context at await |
| `TaskScheduler.FromCurrentSynchronizationContext()` | Schedule to current context |
| `ConfigureAwait(false)` | Don't capture context |
| `Task.Run` | Thread pool — no context |
| `Dispatcher.Invoke` (WPF) | Explicit UI marshal |

## Examples

### WPF async event handler (correct)

```csharp
private async void Load_Click(object sender, RoutedEventArgs e)
{
    try
    {
        var data = await _api.FetchAsync().ConfigureAwait(true); // default — UI updates OK
        StatusLabel.Text = data;
    }
    catch (Exception ex)
    {
        MessageBox.Show(ex.Message);
    }
}
```

**`async void`** acceptable **only** for event handlers — with try/catch.

### Library method (correct)

```csharp
public async Task<string> FetchAsync()
{
    var json = await _http.GetStringAsync(url).ConfigureAwait(false);
    return Parse(json);
}
```

Safe for callers on UI or ASP.NET.

## Common Mistakes & Anti-patterns

**`.Result` on UI after async that captures UI** — deadlock.

**Heavy work after `await` on UI thread** — jank; offload with **`Task.Run`** + **`ConfigureAwait(true)`** only for UI update.

**Using `FromCurrentSynchronizationContext` in ASP.NET Core** expecting request thread affinity — **no single request thread**; use **`HttpContext`** accessor / **`AsyncLocal`**, not SyncContext.

**Assuming **`ConfigureAwait(false)`** in library fixes caller's **`.Result`** — caller still blocks a thread.

## Production & Real-World Notes

Shared **NuGet** libraries: **`ConfigureAwait(false)`** everywhere except UI helpers.

**Blazor Server** has **SynchronizationContext** — UI rules apply within circuits.

Test UI async with **stress click** — double-invoke handlers.

## Comparison / Trade-offs

| Environment | SyncContext | Default await resume |
|-------------|-------------|----------------------|
| WPF/WinForms | UI thread | UI (unless false) |
| ASP.NET Core | null | Thread pool |
| Legacy ASP.NET | HttpContext | Request context |
| Console | null | Thread pool |


## Key Takeaways

- **`SynchronizationContext`** controls **where `await` resumes**.
- **`TaskScheduler.FromCurrentSynchronizationContext`** — legacy; prefer **`async/await`**.
- **UI + `.Result`** → classic deadlock; **ASP.NET Core** → starvation instead.
- **`ConfigureAwait(false)`** in libraries prevents context capture.
- **ASP.NET Core** has **no** request SyncContext — don't rely on thread affinity.
- **`Task.Run`** always pool — different from context-aware await.
