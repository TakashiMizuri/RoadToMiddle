# HTTPS, TLS Handshake Basics, and Certificates

> Roadmap: `0.3.8` · Node: `0.3` — HTTP и Web · Depth: **глубоко**

## Learning Objectives

После этой лекции ты сможешь:

- Объяснить, почему **HTTPS** — это HTTP поверх **TLS**, а не отдельный протокол, заменяющий semantics из `0.3.3`.
- Описать фазы **TLS handshake**: TCP, ClientHello/ServerHello, certificate, key exchange, finished.
- Разобрать **X.509 certificates**: subject, issuer, validity, public key, chain of trust.
- Различить **symmetric** session keys и **asymmetric** keys в certificate — зачем оба.
- Распознать типичные failures: expired cert, hostname mismatch, untrusted CA, mixed content.
- Связать HTTPS с **Authorization**, **Cookie** `Secure` (`0.3.7`, `0.3.9`) и deployment (Kestrel, reverse proxy).

---

## Why This Matters

Methods (`0.3.4`), status codes (`0.3.6`), headers (`0.3.7`) предполагают передачу bytes между client и server. На публичном internet **plain HTTP exposes** их всем на пути: Wi‑Fi, ISP, compromised routers. Passwords в `Authorization: Basic`, tokens в `Cookie`, personal JSON — readable без encryption.

**HTTPS** оборачивает HTTP в **TLS**. URL: `https://`, port **443**. Для application code это тот же HTTP request-response — но transport **encrypts** и **integrity-protects** stream. Browsers помечают non-HTTPS login как insecure; modern APIs reject callbacks без TLS; **Secure** cookies ignored on HTTP.

Middle developer deploys ASP.NET Core за nginx/cloud LB, configures certificates (Let's Encrypt, Key Vault), debugs `NET::ERR_CERT_*`, объясняет **TLS termination at proxy** при HTTP к Kestrel на localhost. Handshake и certificate chain превращают browser errors в actionable fixes.

## Core Concepts

### HTTP vs HTTPS: Same Application Protocol, Different Transport

**HTTP** — messages: start line, headers, body. **HTTPS** = **HTTP over TLS over TCP**:

```
Application:   HTTP
Security:      TLS (encrypt + authenticate peer)
Transport:     TCP
```

После успешного TLS Kestrel читает decrypted HTTP как cleartext на port 80. `MapGet`, status codes, JSON — unchanged; меняется только wire format.

**TLS 1.2/1.3** today; SSL obsolete.

### Threat Model TLS Addresses

Без TLS: **eavesdrop**, **tamper**, **impersonate** server. TLS даёт:

- **Confidentiality** — encrypted payload.
- **Integrity** — AEAD detects tampering.
- **Authentication** — client verifies real server via **trusted CA** chain.

TLS **не заменяет** app auth (JWT, cookies) — защищает **channel**. Phishing с valid HTTPS на wrong domain возможен — user must verify URL.

### TLS Handshake Overview

Goal: cipher suite, authenticate server, derive **symmetric session keys**.

```
Client                                    Server
  | TCP connect (443)                         |
  |---------- ClientHello ------------------>|
  |<--------- ServerHello -------------------|
  |<--------- Certificate -------------------|
  |---------- ClientKeyExchange ----------->|
  |---------- Finished --------------------->|
  |<--------- Finished ----------------------|
  |======== Encrypted HTTP =================|
```

**TLS 1.3** — fewer round trips (1-RTT), **ECDHE** forward secrecy.

Application code не implements handshake — **Kestrel/OpenSSL**. Ты **configure certificates** и **minimum TLS version**.

### Symmetric vs Asymmetric

**Asymmetric** (RSA/ECDSA in cert): authentication, key agreement — slow.

**Symmetric** (AES-GCM, ChaCha20): bulk HTTP data — fast.

Hybrid model — industry standard.

### X.509 Certificates and Chain of Trust

Certificate binds **public key** to identity — **SAN** hostnames:

```
Subject: CN=api.example.com
Issuer: CN=Let's Encrypt R3
Valid: Not Before ... Not After ...
```

Browser trusts **root CAs** in trust store. Server sends leaf + **intermediate chain**. **Self-signed** — dev only.

Failures: **expired**, **wrong hostname**, **untrusted issuer**, **revoked** (OCSP).

**Let's Encrypt** — free DV via ACME.

### HTTPS in Development and Production

**Dev:** `dotnet dev-certs https --trust`. **Never** dev certs in production.

**Production:**

1. **TLS at reverse proxy** — cert on nginx, HTTP to Kestrel loopback.
2. **TLS at Kestrel** — encrypt to process.
3. **Cloud LB** — `X-Forwarded-Proto: https` — honor forwarded headers.

**HSTS** — force HTTPS in browser.

### Mixed Content and Secure Cookies

HTTPS page + HTTP scripts — **mixed content** blocked. **`Secure` cookies** (`0.3.9`) only over HTTPS.

## Under the Hood

### First `https://` Request

DNS → TCP → TLS handshake → cert validation → session keys → encrypted HTTP GET.

**SNI** — virtual hosting on one IP.

**Session resumption** — faster repeat visits.

### Certificate Pinning and mTLS (Awareness)

Pinning — mobile apps, brittle on rotation. **mTLS** — client cert, service-to-service.

### ASP.NET Core HTTPS

`UseHttpsRedirection()`, `UseHsts()`, `UseForwardedHeaders()` behind proxy — avoid redirect loops.

## Common Mistakes & Anti-patterns

HTTP login in production. Self-signed public site. Incomplete chain. `http://` links behind TLS. No expiry monitoring. TLS 1.0/1.1 enabled.

---

## Production & Real-World Notes

Automate renewal. Monitor expiry. TLS 1.2+ min. HSTS after all subdomains ready. Secrets in URLs still leak in logs.

## Comparison / Trade-offs

| Approach | Pros | Cons |
|----------|------|------|
| TLS at LB | Central certs | Possible HTTP internal hop |
| TLS at Kestrel | Encrypt to process | Per-instance certs |
| Let's Encrypt | Free, automated | DV only |

## Key Takeaways

- HTTPS encrypts transport; HTTP semantics same above TLS.
- Handshake + cert chain; symmetric keys for bulk data.
- Fix cert errors: expiry, hostname, chain, trust.
- Proxy termination + forwarded headers — common pattern.
- Secure cookies need HTTPS for users.
