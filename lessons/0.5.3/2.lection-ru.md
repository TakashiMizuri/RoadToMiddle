# Заголовки h1–h6, p, strong, em, mark

> Roadmap: `0.5.3` · Node: `0.5` — HTML · Depth: практика

## Learning Objectives

После этого урока ты сможешь:

- Строить логичную **иерархию заголовков** h1–h6 без пропусков уровней ради дизайна.
- Использовать **`<p>`** для абзацев вместо цепочек `<br>` и пустых div.
- Применять **`<strong>`** и **`<em>`** по смыслу, а не «жирный/курсив».
- Выделять релевантный текст через **`<mark>`** уместно.
- Объяснить, как заголовки формируют **outline** для assistive technology.
- Заменять presentational-разметку (`<b>`, `<i>`, `<font>`) на semantic-эквиваленты.

---

## Why This Matters

Landmarks из `0.5.2` говорят, *где* регионы страницы. **Заголовки** показывают, *как вложены идеи* внутри. Пользователь screen reader часто листает страницу списком headings (клавиша H в NVDA) быстрее, чем зрячий скроллит. Поисковики используют заголовки как сигнал структуры. Если каждая строка — `<div class="title">` или после `h1` сразу `h4`, потому что «так меньше шрифт», ты ломаешь эту навигацию.

Текстовая семантика не менее важна. **`<strong>`** — сильная важность или срочность; **`<em>`** — ударение при прочтении вслух. Браузер по умолчанию делает их bold и italic, но смысл в теге, не в CSS. Middle-разработчик не выбирает strong «чтобы пожирнее», если нужна только косметика.

Абзацы кажутся простыми, пока не увидишь `<br><br>` или `<p>&nbsp;</p>` для отступов. **`<p>`** — блочный абзац с предсказуемыми паузами для AT. **`mark`** подсвечивает текст, релевантный поиску или контексту — не заголовок и не ссылка.

Урок связывает структуру `0.5.2` с читаемым телом страницы — до форм `0.5.7` и ссылок `0.5.5`.

## Core Concepts

### Иерархия заголовков: один h1, вложенные уровни

Шесть уровней. **`h1`** — главный заголовок **sectioning content**, где он живёт: title страницы в `<main>` или title статьи в `<article>`. **`h2`** делит тему h1; **`h3`** — подраздел h2; до **`h6`**.

Правило: **не пропускай уровни ради стиля**. Нужен мелкий заголовок после h1 — пиши h2 и уменьши CSS, не прыгай на h4. Пропуски путают outline: после h1 ожидают h2, не h3 «из ниоткуда».

На одной странице clearest pattern — один видимый h1 в main. В сетке карточек у каждого `<article>` может быть свой h1 — согласуй правило в команде.

Заголовок описывает **следующий блок**, не чужой widget. «Введение» над intro — да. «Sidebar» над статьёй — нет.

### p: абзацы как блоки

**`<p>`** оборачивает абзац прозы. Блочный элемент: отступы; паузы у screen reader между абзацами.

Антипаттерны: несколько `<br>` вместо двух `<p>`; пустой `<p>` для spacing — margin в CSS; блочные элементы внутри `<p>` — парсер закроет p неявно.

Списки, заголовки, figure — **вне** paragraph.

### strong: важность

**`<strong>`** — **сильная важность**, серьёзность, срочность. «Warning: **do not** open…» — без strong смысл слабее.

Bold по умолчанию — побочный эффект. Critical constraints в legal/safety — strong. Не делай strong на каждом втором слове.

### em: ударение

**`<em>`** — **emphasis**, меняющий интонацию:

- "I *really* needed…" — em на really
- "I really **needed**…" — strong на needed

Em и strong могут вкладываться. Не используй em для названий книг (часто `<cite>`) или иностранных слов без `lang`.

### mark: подсветка для ссылки на контекст

**`<mark>`** — выделение **ради relevance** в другом контексте: hit поиска, шаг в инструкции. Default: жёлтый фон.

Не для «красивой подсветки» (CSS class). Не для syntax highlight в code ( `<code>` + highlighter). Пример: в сниппете результата поиска `<mark>charset</mark>`.

## Under the Hood

Заголовки в accessibility tree — role heading level 1–6. AT строит оглавление по **DOM order**, не visual order: CSS `order` не меняет порядок навигации по H.

strong/em → strong/emphasis semantics. mark → mark role. Misused headings — замечания axe/Lighthouse «heading order».

## Common Mistakes & Anti-patterns

Заголовки ради font-size. Пропуск h2. br soup. strong на всём подряд. mark как декор.

## Production & Real-World Notes
