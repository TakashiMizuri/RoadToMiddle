# Value Types vs Reference Types — Stack and Heap

> Roadmap: `1.1.6` · Node: `1.1` — C# Language Basics · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Distinguish **value types** and **reference types** by semantics, not just keywords.
- Explain **where data lives** in the CLR memory model: stack frames vs managed heap (with accurate nuance).
- Draw a **stack/heap diagram** for local variables, fields, and assignments.
- Predict **copy behavior**: assignment copies value vs copies reference.
- Connect **`1.1.3` primitives** (`int`, `bool`, `char`, `decimal`, etc.) to value-type storage.
- Explain why **`null`** applies to reference types and **nullable value types** (`int?`), not plain `int`.

---

## Why This Matters

In `1.1.3` you learned that `int` holds a whole number and `bool` holds `true` or `false`. In `1.1.5` you declared locals with `var` or explicit types. Those declarations feel like "a box named `count` holding 42." That mental model works locally, but it breaks the moment you pass variables to methods, store them in fields, or compare two `string` variables.

C# divides types into two fundamental categories with **different copying semantics** and **different memory stories**. Getting this wrong produces real bugs: mutating a shared object when you thought you made a copy, unexpected `NullReferenceException`, performance surprises when large structs copy on every assignment, and confusion about why `int` cannot be `null` but `int?` can.

Every later topic — classes, structs, generics, boxing in `1.1.7`, GC, async state machines — rests on this split. Middle developers do not memorize "struct = stack" as a slogan; they understand **what gets copied**, **what GC tracks**, and **where references live**.

## Core Concepts

### Two Categories, One Language

**Value types** store their data **directly** in the variable's memory location (conceptually "the variable *is* the value"). Built-in primitives from `1.1.3` — `int`, `long`, `float`, `double`, `decimal`, `bool`, `char` — are value types. Custom **`struct`** types are also value types.

**Reference types** store a **reference** (a pointer-like address) to an object whose **payload lives elsewhere** — on the **managed heap**. The variable "points to" the object; it is not the object itself. **`class`**, **`string`**, **`array`**, **`delegate`**, and **`interface`** implementations on classes are reference types.

The keyword you use (`struct` vs `class`) declares the category. The **semantics of assignment and copying** follow the category.

### Assignment: Copy the Bits vs Copy the Address

When you assign one **value type** variable to another, the **bits of the value are copied**:

```csharp
int a = 10;
int b = a;   // b gets its own copy of 10
b = 20;      // a is still 10
```

When you assign one **reference type** variable to another, the **reference is copied** — both variables point to the **same object** on the heap:

```csharp
var list1 = new List<int> { 1, 2 };
var list2 = list1;   // list2 points to the same List object
list2.Add(3);        // list1 also sees { 1, 2, 3 }
```

Changing `list2` mutates shared state unless you create a **new object** (deep copy, clone, immutable pattern).

**`string` is a reference type** but **immutable**: operations like concatenation create new string objects; you do not mutate existing string content in place (see `1.1.4`).

### Where Do Locals Live? Stack vs Heap (Accurate Model)

You will see diagrams labeling value types on the **stack** and reference type **objects** on the **heap**. That picture is **mostly right for local variables in typical methods**, but needs refinement:

**Stack (per thread):** Each method call creates a **stack frame** holding local variables and parameters. For value types, the **actual value** often lives directly in that frame. For reference types, the frame holds the **reference** (typically a pointer-sized slot), not the object body.

**Managed heap:** Reference type **objects** are allocated on the heap. The GC tracks them. Multiple references can aim at one object.

**Important nuance:** Value types can also live on the **heap** when they are **fields of a class**, elements of an array, boxed into `object`, or captured in closures. So "value type = stack" is an **oversimplification**. Correct rule: **reference type object data on heap; value type embedded where its container lives**.

```
Thread stack                         Managed heap
┌─────────────────────┐             ┌──────────────────────┐
│ Method Main()       │             │ List<int> object     │
│  count: 42 (int)    │             │  _items → array ─────┼──┐
│  listRef ───────────┼────────────►│                      │  │
│                     │             └──────────────────────┘  │
│ Method Foo()        │             ┌──────────────────────┐  │
│  price: 19.99m      │             │ int[] array          │◄─┘
│  nameRef ───────────┼────────────►│  [0]=1 [1]=2 ...     │
└─────────────────────┘             │  Customer struct in  │
                                    │  field slot (value)  │
                                    └──────────────────────┘
```

### Primitives from 1.1.3 on the Diagram

Each primitive value type holds data inline in its slot:

| Type | Category | Typical local storage |
|------|----------|----------------------|
| `int`, `long`, `byte` | Value | Inline in stack frame |
| `float`, `double` | Value | Inline |
| `decimal` | Value | Inline (128-bit struct internally) |
| `bool`, `char` | Value | Inline |
| `string` | Reference | Reference on stack → object on heap |

`decimal` is a **struct** (value type) wrapping a 128-bit representation — still copied by value on assignment.

### `null` and Nullable Value Types

A **reference type** variable can hold **`null`** — no object targeted:

```csharp
string? name = null;   // no string object
```

A plain **`int` cannot be null** — it always holds some `int` value (default `0`):

```csharp
int count = 0;         // not "empty" — zero is a valid value
```

When you need "missing int," use **`int?`** (nullable value type), shorthand for `Nullable<int>`:

```csharp
int? maybe = null;     // HasValue == false
```

Nullable value types are still **value types**: they wrap a `bool hasValue` flag plus the underlying `T` storage, copied by value.

### Default Values

Uninitialized locals must be assigned before use (definite assignment). Fields get defaults:

- Value types: **`0`**, **`false`**, **`'\0'`**, struct with all fields default.
- Reference types: **`null`**.

This explains why `int count;` as a field becomes `0`, but `string name;` as a field becomes `null` and throws if dereferenced without assignment.

### Passing to Methods: By Value by Default

C# passes arguments **by value** by default — meaning the **value of the argument expression** is copied into the parameter slot.

For value types, the **bits** copy — mutations inside the method do not affect the caller's variable.

For reference types, the **reference** copies — caller and callee share the **same object**; mutating the object through the parameter is visible to the caller. Replacing the parameter with a new object (`param = new List<int>()`) does **not** change the caller's variable — only the local reference slot.

Methods like `ref`, `out`, `in` change this story (`1.1.13`); for now, remember default pass-by-value of the reference.

## Under the Hood

### CLR View: Method Table, Objects, and Type Handles

The CLR treats **value types** and **reference types** differently in metadata and JIT code.

**Reference types** inherit (directly or indirectly) from **`System.Object`**. Each object on the heap carries a **method table pointer** (sync block, header) used for virtual dispatch and GC.

**Value types** inherit from `System.ValueType` → `Object` only through boxing; unboxed value types have **no object header** when stored inline. JIT generates code that copies the **sizeof(T)** bytes for assignments and parameter passing.

### Stack Frames and the Evaluation Stack

When JIT compiles a method, locals usually reside in the stack frame or registers. The IL instruction **`ldloc`** loads a value type local; **`ldloca`** loads its **address** (needed for `ref` parameters and mutating structs in place).

Reference locals store a **managed pointer** (GC-tracked reference) to the heap object. The GC must find these pointers during **root scanning** — stack roots, static fields, CPU registers.

### When Value Types Touch the Heap

Value types land on the heap in several cases:

1. **Field of a reference type class** — embedded in the object's layout.
2. **Boxing** — wrapping in `object` or interface (`1.1.7`).
3. **Array of value types** — `int[]` stores elements contiguously on the heap.
4. **Closure capture** — compiler may lift locals to a display class (heap).

So the Middle model is: **stack holds frames and references; heap holds objects and arrays; value types live inside whichever container owns them**.

### GC and Reference Types

The **garbage collector** reclaims heap objects unreachable from **GC roots**. Copying a reference does not create a new object — it adds another path to the same object. Value type copies do not trigger GC of the value itself (no object), unless boxing occurred.

Large **struct** copies can hurt performance without any GC — copying 256 bytes on every assignment is pure CPU cost.

## Syntax / API

```csharp
// Value types (primitives from 1.1.3)
int count = 42;
decimal price = 9.99m;
bool active = true;

// Reference types
string name = "Ada";
int[] numbers = { 1, 2, 3 };
var customer = new Customer();   // class

// Nullable value type
int? optional = null;

// struct — custom value type
public struct Point
{
    public int X;
    public int Y;
}
```

## Examples

**Example 1 — independent int copies:**

```csharp
void Demo()
{
    int x = 1;
    int y = x;
    y = 99;
    Console.WriteLine(x);   // 1
}
```

**Example 2 — shared reference:**

```csharp
void Demo()
{
    var a = new StringBuilder("hi");
    var b = a;
    b.Append("!");
    Console.WriteLine(a);   // "hi!" — same object mutated
}
```

**Example 3 — struct copied by value:**

```csharp
Point p1 = new() { X = 1, Y = 2 };
Point p2 = p1;
p2.X = 99;
Console.WriteLine(p1.X);    // 1 — struct copy
```

## Common Mistakes & Anti-patterns

**Slogan "structs always on stack."** They can live on heap as fields or in arrays.

**Assuming assignment always clones objects.** Reference assignment aliases; only value types copy data by default.

**Using `null` for "empty number" with `int`.** Use `int?` or a sentinel with clear meaning.

**Huge structs passed by value everywhere.** Prefer `in`/`ref readonly` (`1.1.13`) for large structs.

## Production & Real-World Notes

Domain modeling: IDs and small coordinates as **readonly struct**; entities with identity and lifetime as **class**. DTOs may be class (reference) or struct depending on immutability and size.

Profiling: unexpected allocations sometimes trace to **boxing** (`1.1.7`) or string concatenation, not reference assignment itself.

ASP.NET Core: per-request objects (DbContext, scoped services) are reference types on the heap; stack depth per request is bounded but heap allocation rate drives GC pauses.

## Comparison / Trade-offs

| | Value type | Reference type |
|---|------------|----------------|
| Assignment | Copies data | Copies reference |
| `null` | Only via `T?` | Allowed |
| Default | Zero/false | null |
| Inheritance | No (except interfaces) | Yes (single base class) |
| Typical use | Small immutable data | Identity, shared mutable state |

```
Local int a        → value in stack frame

Assignment ref     → alias same object
Assignment value   → independent copy of bits
```

## Key Takeaways

- **Value types copy data; reference types copy references** — the core semantic split.
- **Heap holds reference type objects**; **stack frames hold locals** — value inline, references as pointers.
- **Value types can live on the heap** as fields, array elements, boxed values.
- **`1.1.3` primitives are value types**; **`string` is reference type** (immutable).
- **`int?` is value type** representing optional `int`; plain `int` cannot be null.
- Default parameter passing **copies values/references**, not deep clones.
