## Answers

### 1. [Recall]
**Answer:** The SHA **changes completely** — amend creates a **new commit object** and moves the branch to it.

**Why:** Commits are content-addressed; any change produces a different hash.

---

### 2. [Recall]
**Answer:** **`--no-edit`** reuses the **existing commit message** while allowing new staged content to be included in the amended commit.

---

### 3. [Understanding]
**Answer:** The original commit object **still exists** unchanged in the object database until GC. Amend creates a **new** commit; the branch pointer moves. You don't mutate the old object — you replace the branch tip reference.

**Why:** Immutability is fundamental to Git's model.

---

### 4. [Understanding]
**Answer:** Local `main` now points to a **new commit** whose parent chain **does not include** the old tip from the remote's perspective — push is **not fast-forward**. Remote still has the pre-amend commit.

**Why:** Non-fast-forward means histories diverged.

---

### 5. [Understanding]
**Answer:** (Accept variations) **Do not rewrite commits on shared/public branches** that others have already fetched or built upon. Preserve history teammates rely on.

**Why:** Rewriting forces others to reconcile divergent histories (reset/rebase), risking lost work.

---

### 6. [Application]
**Answer:**
```bash
git add Dto.cs
git commit --amend --no-edit
```

**Why:** Stages file into amended commit; safe because unpushed and private branch.

---

### 7. [Application]
**Answer:** Create a **new commit** with the missing file, e.g. `git add Dto.cs && git commit -m "Add missing Dto.cs"`. Alternatively **`git revert`** if undoing the whole change — but for forgotten file, new commit is standard.

**Why:** Shared history must not be rewritten.

---

### 8. [Application]
**Answer:** **`git commit --amend -m "Corrected message"`** or **`git commit --amend`** (editor).

---

### 9. [Analysis]
**Answer:** **Generally yes** on solo PR feature branches — force-push with lease updates remote PR history; common workflow before merge. **Caveat:** if others checked out that branch, they must reset; `--force-with-lease` reduces accidental overwrite.

**Why:** PR branches are often treated as private until merged; team policy may vary.

---

### 10. [Analysis]
**Answer:** **C) is false** — `--force-with-lease` is safer than `--force` but **does not make amend safe on `main`**; rewriting shared main still breaks collaborators.

**Common mistake:** Thinking `--force-with-lease` fixes the social/technical problem of shared history rewrite.

---

### 11. [Understanding]
**Answer:** **`--amend`** rewrites/replaces the last commit (local rewrite). **`git revert`** creates a **new commit** that undoes a previous commit's changes — safe on shared branches because it **adds** history instead of rewriting.

---

### 12. [Understanding]
**Answer:** When you need to edit **more than the last commit** — e.g. squash last 3 commits, reword an older message, reorder commits before opening PR.

**Why:** `--amend` only affects `HEAD`.

---

### 13. [Analysis]
**Answer:** **No** — policy forbids force-push to `main`. Use **`git revert`** to undo the bad commit, or a **new forward commit** fixing the issue. Escalate if full revert needed.

**Why:** Compliance and team safety trump convenience of rewrite.

---

### 14. [Recall]
**Answer:** Any two:
- Last commit **never pushed**
- **Local-only** feature branch nobody else uses
- Cleanup on **personal PR branch** before first push / with team OK on force-push PR

---

### 15. [Application]
**Answer:**

| Situation | Action |
|-----------|--------|
| Local only, forgot file | **(1) `--amend`** |
| Pushed to shared `develop` | **(2) New commit** |
| Fix commit from 3 commits ago | **(4) `git rebase -i`** |
| Bad commit deployed to production | **(3) `git revert`** |
