# Partial classes — паттерн generated code

> Roadmap: `1.2.18` · Node: `1.2` — C# OOP · Depth: понимание

## Learning Objectives

После этой лекции ты сможешь:

- Объяснить, что делает ключевое слово **`partial`** на этапе компиляции.
- Безопасно разделять **class**, **struct**, **record** или **interface** по нескольким файлам.
- Отличать **ручную** partial-часть от **сгенерированной** tool-ом.
- Использовать **`partial` methods** с опциональной реализацией.
- Ориентироваться в **WinForms/WPF designer**, **EF Core**, **source generators**, scaffolding.
- Следовать правилам команды: какой файл редактировать и как пережить regen.

## Why This Matters

В solution лежат `MainForm.cs` и `MainForm.Designer.cs`, или `AppDbContext.cs` рядом с `AppDbContext.g.cs`. Имя класса одно, содержимое файлов разное: в одном — твои обработчики, в другом — `InitializeComponent` и поля контролов. Без **partial classes** designer при каждом сохранении затирал бы твои правки. Без понимания паттерна ты либо боишься трогать generated file, либо правишь то, что следующий build перезапишет.

Современный .NET — **наполовину руками, наполовину генерацией**. Source generators, EF, OpenAPI/gRPC clients, UI designers выпускают C#, который должен **скомпилироваться как один тип** вместе с твоим кодом. `partial` — шов: tool владеет одним файлом, ты — другим; компилятор сливает фрагменты в одно определение типа до semantic analysis.

Это не академия. Middle-разработчик правит `*.Designer.cs` вручную — следующее сохранение формы убивает fix. Другой кладёт бизнес-логику в generated gRPC — и удивляется overwrite при обновлении reference. Паттерн экономит часы и учит разделять **стабильный intent** и **повторяемый boilerplate**.

## Core Concepts

### Что означает partial

Модификатор **`partial`** на объявлении типа говорит компилятору: «Это **фрагмент** типа; другие фрагменты могут быть в других compilation units (обычно файлах).» Все фрагменты обязаны иметь **`partial`**, **то же имя типа**, совместимые модификаторы, тот же namespace и assembly.

На **compile time** компилятор C# **сливает** partial declarations в **один** тип. Члены всех частей объединяются. В итоговой assembly — **один** class, не вложенность и не наследование от себя.

```csharp
// Customer.cs
public partial class Customer
{
    public void SendInvoice() { }
}

// Customer.generated.cs
public partial class Customer
{
    public int Id { get; set; }
    public string Name { get; set; }
}
```

После merge у `Customer` есть `SendInvoice`, `Id`, `Name`. **private** из одной части видны в другой части **того же** типа — исключение из обычных правил visibility.

### Какие типы могут быть partial

**class**, **struct**, **record**, **interface**. Enum — нет. Вложенные типы — partial, если каждая часть повторяет nesting.

**Methods**, **properties**, **indexers**, **events** тоже могут быть **`partial`**: в одной части сигнатура (часто generated), в другой — **тело** (часто твой код). Hook для generators без правки generated body.

### Зачем делить файлы

**Организация для человека:** большие типы по concern — осторожно, не дроби чрезмерно.

**Владение tool-ом:** designer/generator **перезаписывает** свой файл. Твой файл не трогают. Контракт: **не редактировать `*.Designer.cs` / `*.g.cs` / auto-generated**, если команда не сказала иначе.

**Source generators:** emit `partial` на compile; ты дополняешь вторую половину в обычном `.cs`.

### Partial methods

**`partial method`**: объявление в одной части, **опциональная** реализация в другой:

```csharp
// Generated
public partial class Logger
{
    partial void OnLogWritten(string message);
    public void Write(string message) { OnLogWritten(message); }
}

// Твоя часть — опционально; без неё вызовы вырезаются
public partial class Logger
{
    partial void OnLogWritten(string message) =>
        Console.WriteLine($"Logged: {message}");
}
```

Упрощённо: return **`void`**, часто **`private`**, не **`virtual`**. Если **реализации нет** — компилятор **удаляет** call sites: нет runtime cost, не нужны пустые stub. Generators используют как **optional hooks**.

### Паттерн generated code

| Файл | Владелец | Содержимое |
|------|----------|------------|
| `Entity.cs` | Ты | Поведение, вычисляемые свойства |
| `Entity.Designer.cs` / `.g.cs` | Tool | Поля, wiring, metadata |

Комментарии `// <auto-generated>` — не трогать.

**Regen:** save в designer, `dotnet ef dbcontext scaffold`, build с source generator, update service reference.

**Кастомизация:**
1. Свой **`partial`** файл — методы/свойства (safest).
2. **Partial method** hooks.
3. **Наследование / wrapper** — старые tools.
4. Конфигурация вне generated — EF `OnModelCreating` в твоём partial `DbContext`.

### Partial interfaces и records

**`partial interface`** — generator добавляет члены к твоему interface; implementer обязан закрыть **все** части после merge.

**`partial record`** — то же merge semantics для behavior + generated serialization/helpers.

## Examples

### EF Core DbContext

```csharp
// AppDbContext.cs — ты
public partial class AppDbContext : DbContext
{
    protected override void OnModelCreating(ModelBuilder modelBuilder) =>
        modelBuilder.ApplyConfigurationsFromAssembly(typeof(AppDbContext).Assembly);
}

// AppDbContext.g.cs — illustrative
public partial class AppDbContext
{
    public DbSet<Order> Orders => Set<Order>();
}
```

### Designer (conceptual)

Designer: `InitializeComponent()`, `private Button btnOk;`. Ты: `btnOk_Click`. Save формы — только designer file.

## Common Mistakes & Anti-patterns

**Правки в auto-generated** — потеря при regen.

**Забыл `partial` на одном файле** — duplicate type / ошибка merge.

**Разный namespace/имя** — фрагменты не сольются.

**Бизнес-логика в Designer.cs** — работает до save designer.

**Partial ≠ runtime merge** — только compile time.

## Production & Real-World Notes

CI/analyzers могут ругаться на правки generated files.

**Git:** шум в designer diffs — норма.

Source generators: смотри `obj/generated` при отладке.

gRPC/OpenAPI: расширяй через **отдельный partial file**.

Code review: ручные правки в `#region auto generated` — red flag.

## Comparison / Trade-offs

| Подход | Плюсы | Минусы |
|--------|-------|--------|
| Partial split | Чёткая граница tool/human | Полный тип не в одном файле |
| Один файл | Видимость | Tools затирают правки |
| Наследование от generated | Non-partial tools | Хрупко |
| Composition | Без coupling | Boilerplate |

**Partial** — idiomatic .NET, когда tool и разработчик расширяют **одно** имя типа.

## Quick Reference

| Элемент | Правило |
|---------|---------|
| Keyword | `partial` на **каждом** фрагменте |
| Merge | Compile time → один тип |
| Generated | `*.Designer.cs`, `*.g.cs` — не редактировать |
| partial method | Без impl → вызовы удалены |
| Типы | class, struct, record, interface |

## Key Takeaways

- **`partial`** сливает фрагменты в **один** тип при компиляции.
- Tools владеют generated part; ты — своим partial file.
- **Partial methods** — опциональные hooks без cost, если не реализованы.
- Не правь auto-generated вручную — кастомизируй свой partial.
- EF, designers, source generators живут на этом паттерне.
- Имя, namespace, `partial` — must match на всех частях.
