## Answers

### 1. [Recall]
**Answer:** Статистику по потокам во время benchmark: completed work items, lock contention, thread pool-related counters (зависит от версии BDN).

**Why:** Помогает связать время выполнения с pressure на thread pool, не только с wall-clock.

---

### 2. [Recall]
**Answer:** В Debug JIT не оптимизирует код, tiering и inlining отличаются — timings не репрезентативны.

**Why:** BenchmarkDotNet документация требует Release для meaningful results.

---

### 3. [Understanding]
**Answer:** Оба ждут одну и ту же сеть/server delay; узкое место — RTT и remote processing, не способ ожидания на одном потоке.

**Why:** Выигрыш async — в **concurrency**, не в single-call latency.

**Common mistake:** Делать вывод «async быстрее» из одного вызова.

---

### 4. [Understanding]
**Answer:** Async I/O регистрирует операцию в kernel (IOCP); поток не blocked в receive, а continuation выполнится по завершении I/O через thread pool.

**Why:** Blocked thread = occupied worker; async wait = только state machine + socket state.

---

### 5. [Understanding]
**Answer:** До ~300 blocked workers (или рост queue); новые запросы ждут в очереди thread pool → рост latency, возможны timeouts/503; pool медленно добавит потоки.

**Why:** Sync-over-async удерживает worker на всё время downstream call.

---

### 6. [Application]
**Answer:** `BenchmarkDotNet`; `[MemoryDiagnoser]`, `[ThreadingDiagnoser]`; `[Benchmark]` methods; `[GlobalSetup]` для одного `HttpClient`; `dotnet run -c Release`.

**Why:** Стандартный minimal setup для HTTP + threading comparison.

---

### 7. [Application]
**Answer:** Локальный ASP.NET Minimal API с `Task.Delay` endpoint; фиксированный URL localhost; warmup; один shared `HttpClient`; controlled concurrency loop.

**Why:** Убирает WAN variance и внешние rate limits.

---

### 8. [Analysis]
**Answer:** **A** — неверно.

**Why:** Single-call latency часто равна; async выигрывает под concurrent load.

---

### 9. [Analysis]
**Answer:** `Task.Run` занимает **ещё один** thread pool thread на время sync HTTP — двойной расход; проблема масштабирования остаётся, только дороже.

**Why:** Band-aid, не fix; правильно — truly async `await`.

---

### 10. [Application]
**Answer:** `dotnet-counters` (threadpool metrics); load test k6/NBomber; sampling `ThreadPool.GetAvailableWorkerThreads` / queue length under load.

**Why:** Production-like validation beyond micro-benchmark.

## Model answers for practical sections

### P1
**Answer:** Class with `[Params] Concurrency`, shared `HttpClient`, `[Benchmark] async Task AsyncBatch()` → `Task.WhenAll(Enumerable.Range(0, Concurrency).Select(_ => client.GetAsync(...)))`; `[Benchmark] void SyncBatch()` → same but `.GetAwaiter().GetResult()` per task or `Parallel.For` with blocking — measure ops/sec or total time; optional `[ThreadingDiagnoser]`.

**Why:** Throughput at scale reveals thread pool cliff.
