# Code review etiquette — giving & receiving feedback

> Roadmap: `0.2.10` · Node: `0.2` — Git: ветки и collaboration · Depth: понимание

## Learning Objectives

После этой лекции ты сможешь:

- Объяснить, **зачем code review** — не только баги: knowledge sharing, standards, ownership.
- Применять принципы **дачи feedback**: specific, actionable, kind, prioritized.
- Применять принципы **приёма feedback**: assume good intent, clarify, без defensiveness.
- Различать **blocking** и **non-blocking** (nit) комментарии.
- Использовать паттерны комментариев в async PR workflow (`0.2.8`).
- Распознавать токсичное и контрпродуктивное поведение в review.

---

## Why This Matters

Можно освоить remotes (`0.2.6`), upstream (`0.2.7`), PR (`0.2.8`) и Conventional Commits (`0.2.9`) — и всё равно провалиться как teammate, если code review становится полем боя. Review — **human gate** в collaboration: момент, где quality, security и shared understanding согласуются до попадания кода в `main` и production.

Для Middle fullstack review **двунаправленный**. Ты **получаешь** feedback на ASP.NET controllers и React components; **даёшь** feedback на SQL, API contracts, tests. Команды с хорошим review ship faster со временем — не потому что ловят каждую опечатку, а потому что **распространяют context**, снижают bus factor, выравнивают patterns. Команды с плохим review копят resentment, rubber-stamp approvals или endless style wars.

Etiquette — не politeness theater. Это **engineering communication**, которая двигает PR workflow (`0.2.8`): ясные comments → быстрее resolution → меньше misunderstandings → safer merges.

## Core Concepts

### Зачем review

Больше чем defect detection. **Spreads knowledge** — читая middleware коллеги, учишься token handling. **Enforces standards** — error handling, logging, EF patterns. **Improves design** до merge cost. **Builds ownership** — shared codebase, shared responsibility.

Review ловит subset багов; **tests и CI** — другие. Уникальная ценность review — human judgment: readability, security smells, architecture fit.

### Mindset reviewer'а

Начинай с **respect и curiosity**. Author потратил часы; твоя задача — помочь land safely, не доказать превосходство.

**Review code, not person.** «This method allocates on every request», не «you always do this wrong». **Ask questions:** «Could we use `AsNoTracking` here?» лучше обвинения. **Be specific** — line, alternative, risk. **Prioritize**: security before naming nit.

Block только при **real risk**: correctness, security, missing tests на critical paths, breaking API без migration. Остальное — **non-blocking nits** или follow-up tickets.

### Дача feedback — паттерны

**Tiers** (многие команды в templates):

- **[blocking]** — must fix: bug, security, data loss.
- **[suggestion]** — улучшение; author решает.
- **[nit]** — style; non-blocking.
- **[question]** — понять intent.
- **[praise]** — reinforce good patterns.

ASP.NET example:

> **[question]** Why `SaveChangesAsync` inside loop?
>
> **[suggestion]** Return `ProblemDetails` for validation — API guidelines.
>
> **[nit]** Rename optional — matches convention.

React example:

> **[blocking]** XSS: `dangerouslySetInnerHTML` без sanitize.
>
> **[praise]** Nice `useOrderSummary` hook.

**Actionable** — author знает next step. «Confusing» слабо; «Extract `OrderValidator`» сильно.

Избегай **bike-shedding** и **ghost approvals**.

### Приём feedback

**Assume good intent.** **Read all comments before replying.** **Clarify:** «Fixed» или «Chose X because Y».

Не принимай nits лично — fix или polite pushback. **Fix or explain** blocking comments — новый commit, rationale, или call.

Follow-up commits: conventional messages (`0.2.9`). **Resolve threads** в UI.

При disagreement — **evidence**: benchmark, spec, security doc. Escalate к tech lead при architecture deadlock.

**Thank reviewers** — culture compounds.

### Blocking vs Non-Blocking

| Level | Merge | Примеры |
|-------|-------|---------|
| Blocking | Must fix | Security, wrong logic, payment без tests |
| Suggestion | Choice | Alternative pattern |
| Nit | Optional | Naming |
| Question | Clarify | Intent |

Всё blocking — burn trust. Игнор blocking — burn `main`.

### Async PR culture

Review в GitHub/GitLab comments. Пиши для **future reader** — полные предложения, links. **Suggested changes** для trivial fixes. Без sarcasm и CAPS.

Same-day acknowledgment на assigned reviews — Middle norm. High volume → **pair 5 min** вместо 20 comment rounds.

## Under the Hood

Comments — **metadata на diffs**; Git objects меняются только при push. **CODEOWNERS**, **required approvals** (`0.2.8`). Bots (Sonar, CodeQL) — тоже feedback; humans prioritize.

## Syntax / Commands / API

```bash
git commit -m "fix(auth): address review — validate audience claim"
git push
gh pr comment 42 --body "Fixed in latest push."
gh pr review 42 --approve --body "LGTM after null guard."
```

**LGTM** — только если реально проверил.

## Examples

### Good vs bad blocking

**Bad:** «This is wrong.»

**Good:** **[blocking]** 200 с `{ success: false }` ломает HTTP contract — suggest `ValidationProblem(ModelState)`.

### Disagreement resolved

Author объясняет `IOptionsMonitor` для hot-reload flags со ссылкой на docs — rationale documented, no change.

### Toxic → constructive

Вместо «Did you even test?» → **[blocking]** Reproduce steps + expected behavior.

## Common Mistakes & Anti-patterns

Bikeshedding, defensiveness, rubber-stamping, drive-by refactor demands, public humiliation, merge при red CI, stale comments после force-push.

---

## Production & Real-World Notes

Audit trail в regulated envs. Hotfix — expedited review, не zero. Remote teams — written clarity critical. Seniors model tone.

---

## Comparison / Trade-offs

Strict blocking — quality vs speed. Lightweight — speed vs drift. Balance: **block risk, suggest craft, nit optional**.

---

## Quick Reference

```
GIVING: code not person | specific | actionable | tiers
RECEIVING: good intent | read all | fix or explain + evidence
BLOCK: risk | DON'T BLOCK: optional style
```

---

## Key Takeaways

1. Review — **collaboration**, не gatekeeping theater.
2. **Specific, actionable, kind** ускоряет PR.
3. **Blocking** только для risk; nits labeled.
4. Authors **fix or explain** с evidence.
5. **Questions** > accusations.
6. **Praise** хорошие patterns.
7. Etiquette критичен для async distributed teams.
