# ValueTask — pooling и правило «не await'ить дважды»

> Roadmap: `1.4.15` · Node: `1.4` — C# async · Depth: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Объяснить, зачем существуют `ValueTask` и `ValueTask<T>` и когда они выигрывают у `Task`.
- Описать **pooling** реализаций `IValueTaskSource` и что сбрасывается между использованиями.
- Сформулировать правило **«не await'ить дважды»** и предсказать undefined behavior при нарушении.
- Выбирать между `Task`, `ValueTask` и синхронным return в API библиотек.
- Использовать `AsTask()` как безопасный escape hatch для потребителей.
- Распознавать production-баги из-за кеширования и повторного await `ValueTask`.

## Why This Matters

Каждый `await` на hot path, возвращающий `Task<T>`, часто аллоцирует **объект Task на heap** — даже когда операция завершилась синхронно (cache hit, чтение из памяти, быстрая валидация). В высоконагруженных ASP.NET Core endpoints и database drivers эти аллокации дают давление на GC и хвосты latency.

`ValueTask` появился в .NET Core 2.1 как **struct-based awaitable**, который может представлять либо **синхронный результат** (без аллокации), либо **обёрнутый `Task`** (когда операция реально асинхронна). **System.Text.Json**, **Entity Framework Core** и overload'ы **Stream.ReadAsync** используют `ValueTask`/`ValueTask<T>` именно по этой причине.

Цена строгая: pooled `ValueTask` может стать **невалидным после первого await**. Повторный await той же instance, хранение в поле и await позже после возврата source в pool — **undefined behavior**: тихая порча данных, неверные результаты или редкие краши. Middle-разработчик обязан знать этот контракт при **потреблении** и особенно при **авторстве** async API.

## Core Concepts

### Task: overhead reference type

`Task` и `Task<T>` — **reference types**. Когда async-метод приостанавливается, state machine часто завершается на **кешированном completed task** (`Task.CompletedTask`, `Task.FromResult`) на sync path — но многие ветки всё равно аллоцируют. Каждый `new TaskCompletionSource<T>()` или state machine, ушедшая в async mode, создаёт heap-объекты под контролем GC.

Для операций **async только иногда** — чтение из buffer (sync) vs network (async) — возврат `Task<T>` вынуждает либо always-async (расточительно), либо ручную возню с `TaskCompletionSource`.

### ValueTask как discriminated struct

`ValueTask` и `ValueTask<T>` — **struct'ы** с awaitable pattern (`GetAwaiter()` → `ValueTaskAwaiter`). Внутри хранится одно из:

1. **Синхронный результат** — значение `T` (`ValueTask<T>`) или void completion (`ValueTask`).
2. **Обёрнутый `Task` / `Task<T>`** — когда операция ушла в async.
3. **`IValueTaskSource` / `IValueTaskSource<T>`** — custom completion source, часто **pooled**.

Для caller'а `await valueTask` выглядит как `await task`. Awaiter проверяет sync completion и либо продолжает inline, либо регистрирует continuation на underlying task/source.

### Модель pooling

Когда библиотека использует **ручной `IValueTaskSource`** с object pool (типично в BCL и high-performance IO):

```
Caller вызывает ReadAsync() → pool берёт ManualResetValueTaskSourceCore
  → sync complete: ValueTask с результатом
  → async: continuation, по завершении — signal awaiter, RETURN TO POOL
```

После **первого завершённого await** `ValueTask` — **одноразовый билет**. Pooled source могут **reset и отдать другому caller'у**. Второй await на том же `ValueTask` может увидеть completion чужой операции — data race с точки зрения приложения.

**Правило:** каждый `ValueTask` — **single-consumer, single-use**. Не кешировать в поле, не передавать нескольким concurrent awaiter'ам, не `await` дважды.

### Когда ValueTask помогает

`ValueTask` хорош, когда:

- Метод **чаще всего** завершается синхронно (cached config, buffered read, fast validation).
- Ты контролируешь implementation и можешь использовать **`ManualResetValueTaskSourceCore<T>`** с pooling.
- Caller'ам не нужны multiple awaits, `.Result` или composition со stable `Task`.

Не помогает, когда операция **всегда async** — платишь за underlying `Task` и копирование struct без выигрыша.

### Безопасные паттерны потребления

**Верно — один await:**

```csharp
ValueTask<int> vt = stream.ReadAsync(buffer);
int bytes = await vt;
```

**Верно — конвертация при необходимости reuse:**

```csharp
ValueTask<int> vt = GetValue();
Task<int> task = vt.AsTask();
int a = await task;
int b = await task; // OK на Task
```

**Неверно — double await:**

```csharp
ValueTask<int> vt = GetValue();
int a = await vt;
int b = await vt; // UNDEFINED
```

## Under the Hood

### ManualResetValueTaskSourceCore

.NET даёт `ManualResetValueTaskSourceCore<TResult>` — struct, реализующий `IValueTaskSource<TResult>` без аллокации на операцию при pooling:

`ValueTask` хранит **version token** вместе с source. После reset version инкрементируется — stale awaiter'ы детектят invalidation (при корректном usage ты сюда не попадаешь).

При `await valueTask`:

1. `GetAwaiter()` → `ValueTaskAwaiter` / `ValueTaskAwaiter<T>`.
2. `IsCompleted` — если true, `GetResult()` синхронно.
3. Иначе `OnCompleted` на `Task` или `IValueTaskSource`.
4. После completion **consumer не должен трогать `ValueTask` снова**, если за ним pooled source.

`.AsTask()` **может аллоцировать** wrapper — цена безопасности.

## Syntax / API

| API | Назначение |
|-----|------------|
| `new ValueTask()` / `new ValueTask<T>(T result)` | Sync-completed awaitable |
| `new ValueTask(Task)` / `new ValueTask<T>(Task<T>)` | Обёртка Task |
| `ValueTask.FromCanceled()` / `FromException()` | Canceled/faulted sync paths |
| `valueTask.AsTask()` | Безопасный Task для reuse |
| `ManualResetValueTaskSourceCore<T>` | Pooled custom sources |

## Examples

### Кеш-сервис с fast path

```csharp
public ValueTask<User?> GetUserAsync(int id, CancellationToken ct)
{
    if (_cache.TryGetValue(id, out User? user))
        return new ValueTask<User?>(user);

    return new ValueTask<User?>(FetchFromDbAsync(id, ct));
}
```

Caller **await'ит один раз**. Cache hit избегает аллокации `Task`.

### AsTask() для WhenAll

```csharp
Task<int>[] tasks = items.Select(vt => vt.AsTask()).ToArray();
int[] results = await Task.WhenAll(tasks);
```

## Common Mistakes & Anti-patterns

**Double await «потому что в тестах работало»** — undefined behavior зависит от timing; в production под нагрузкой pool reuse проявится.

**`async ValueTask`** — при реальном `await` внутри часто аллоцируется state machine **и** backing `Task`. Используй `async Task` или manual `IValueTaskSource`.

**Хранение `ValueTask` в полях** — для deferred completion нужен `Task`.

**`.Result` на `ValueTask`** — sync-over-async плюс нарушение single-use.

## Production & Real-World Notes

Профилируй перед сменой public API на `ValueTask`. Часто **`AsTask()` на границах** — библиотека возвращает `ValueTask`, application layer конвертирует перед `Task.WhenAll`.

EF Core 5+ — многие API возвращают `ValueTask`; await inline в handler'ах, не храни.

## Comparison / Trade-offs

| | `Task` | `ValueTask` |
|---|--------|-------------|
| Multiple awaits | ✅ | ❌ (pooled) |
| Sync fast path | Часто alloc | Часто zero alloc |
| WhenAll | Native | `.AsTask()` |
| Public API default | ✅ | Только по метрикам |

## Key Takeaways

- `ValueTask` снижает аллокации при частом sync completion.
- Pooled source делает второй await **undefined**, не «плохим стилем».
- `.AsTask()` — для multiple awaits и composition.
- Авторам API: default `Task`; `ValueTask` — с profiling и документацией.
- `async ValueTask` редко оправдан.
