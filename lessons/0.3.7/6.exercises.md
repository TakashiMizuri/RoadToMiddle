# Exercises: HTTP Headers

> Complete after reading lectures and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Header direction
**Goal:** Request vs response.

Classify each as **request**, **response**, or **both**:

1. `Set-Cookie`
2. `If-None-Match`
3. `Cache-Control`
4. `ETag`
5. `Authorization`
6. `Content-Type`

**Success criteria:** 6/6 correct.

### E2. Fill the blanks
**Goal:** Correct header for intent.

| Intent | Header |
|--------|--------|
| Tell server response should be JSON | ? |
| Tell server POST body is JSON | ? |
| Send JWT access token | ? |
| Validator for conditional GET | ? (response) + ? (request) |

**Success criteria:** Accept, Content-Type, Authorization, ETag + If-None-Match.

## Medium

### M1. Cache policy design
**Goal:** Apply Cache-Control safely.

For each resource, write recommended `Cache-Control` (brief):

1. `/api/account/balance` (authenticated, sensitive)
2. `/assets/logo.png` (public, changes rarely)
3. `/assets/app.[hash].js` (immutable build output)
4. Public product catalog JSON (same for all users, updates hourly)

**Success criteria:** No `public` on (1); `immutable` or long max-age justified for (3); revalidation strategy for (4) if needed.

---

### M2. Inspect real headers
**Goal:** Read traces.

Use `curl -I` on any HTTPS site and a JSON API. Paste:

- One `Content-Type` response header
- Any `Cache-Control` or `ETag` if present
- Whether `Set-Cookie` appears

Write 3 sentences comparing static vs API responses.

**Success criteria:** Identifies caching differences.
