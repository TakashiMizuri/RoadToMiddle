# Default Interface Methods (C# 8+)

> Roadmap: `1.2.10` · Node: `1.2` — C# OOP · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Define **default interface methods (DIM)** as instance methods with bodies declared on an interface.
- Explain why C# added DIM — **interface evolution** without breaking existing implementers.
- Describe how a class **inherits**, **overrides**, or **explicitly implements** default methods.
- Recognize **limitations**: no instance fields on interfaces, diamond default conflicts, versioning cautions.
- Contrast DIM with **abstract class** default behavior and **extension methods**.
- Apply DIM sparingly in application code while reading framework interfaces that use them.

---

## Why This Matters

For decades C# interfaces were **pure contracts**: signatures only. Adding a new method to a shipped **`IRepository`** forced **every implementer** to compile again with a stub — or the change waited for a major breaking release. Java faced the same problem and added **default methods**; C# 8 (2019) followed with **default interface implementations**.

If you consume **.NET** libraries post–C# 8, you already depend on DIM indirectly — **`IAsyncDisposable`**, **`IEquatable<T>`** enhancements in BCL discussions, and community patterns for evolving plugin APIs. You rarely **author** DIM in CRUD apps, but you must understand **dispatch rules** when a class "does not implement" a method yet the call succeeds through the interface.

Misunderstanding DIM leads to surprise when **`((IMyService)obj).NewMethod()`** works but **`obj.NewMethod()`** does not (no public member), or when two inherited defaults **collide**. Middle developers participate in API design reviews where "add a default on the interface" is proposed instead of **abstract base** or **extension method** — you need trade-off vocabulary from this lesson.

## Core Concepts

### What Default Interface Methods Are

Before C# 8, this was illegal:

```csharp
public interface IGreeter
{
    string Greet(string name) => $"Hello, {name}"; // C# 8+
}
```

A **default interface method** provides an **implementation on the interface itself**. Any class implementing **`IGreeter`** **inherits** this behavior **without** declaring **`Greet`**, unless it chooses to override.

```csharp
public class SimpleGreeter : IGreeter { }

IGreeter g = new SimpleGreeter();
Console.WriteLine(g.Greet("Ada")); // Hello, Ada
```

**`SimpleGreeter`** has **no** public **`Greet`** method on the class surface. The call works because **`g`** is typed as **`IGreeter`** — dispatch uses interface mapping including default body.

This is **not** the same as **extension methods** (`1.3` topics): extensions are static, appear only when interface is in scope as extension namespace; DIM are **instance methods** on the interface type in metadata.

### Why the Feature Exists: Interface Evolution

Shipping **`IPlugin`** with **`void Init()`** only, then needing **`void Shutdown()`**:

**Without DIM:** add **`Shutdown`** to interface → **breaks** all implementers (CS0535 until they add method).

**With DIM:**

```csharp
public interface IPlugin
{
    void Init();
    void Shutdown() { /* safe no-op default */ }
}
```

Existing classes **compile unchanged**; new code may call **`Shutdown`** through **`IPlugin`**. Optional behavior ships as default; specialized plugins **override**:

```csharp
public class DatabasePlugin : IPlugin
{
    public void Init() { /* connect */ }
    public void Shutdown() { /* disconnect */ } // overrides default
}
```

Microsoft uses this pattern to evolve **interfaces in the BCL** with less breakage. Application teams should still treat **adding members** as serious API review — defaults reduce pain, not eliminate design cost.

### Overriding Defaults

A class overrides a default the same way as implicit interface implementation — **public member** matching signature:

```csharp
public interface IFormatter
{
    string Format(int value) => value.ToString();
}

public class HexFormatter : IFormatter
{
    public string Format(int value) => value.ToString("X");
}
```

**`HexFormatter`** public **`Format`** replaces default for **`IFormatter`** mapping. Call through **`IFormatter f = new HexFormatter()`** gets hex.

**Explicit implementation** also overrides default for that interface slot (`1.2.9`).

If class **does not** implement and **does not** override, compiler emits **InterfaceImpl** entry pointing to **default method** on interface — CLR executes interface body.

### What Interfaces Still Cannot Do

DIM **did not** turn interfaces into abstract classes:

- **No instance fields** in interfaces (C# 11 adds **static abstract** members for generics — advanced; still no instance fields).
- **No instance constructors** on interfaces.
- State in default methods must use **locals**, **static** members, or parameters — not per-object fields on interface.

Heavy shared state still belongs in **abstract class** (`1.2.8`) or **composition**.

Access modifiers on interface members expanded over C# versions (**`private`**, **`protected`**, **`internal`** interface methods) — defaults are usually **`public`** in contract sense.

### Diamond Problem and Multiple Inheritance of Implementation

A class implementing **`IA`** and **`IB`** where **both** define default **`void Log()`** with different bodies creates **conflict**. C# requires class to **override** and resolve:

```csharp
public interface IA { void Log() => Console.WriteLine("A"); }
public interface IB { void Log() => Console.WriteLine("B"); }

public class Worker : IA, IB
{
    public void Log() => Console.WriteLine("Resolved"); // must implement
}
```

Only **one** class hierarchy, but **multiple defaults** — language forces explicit resolution. Similar to explicit interface implementation for name collisions (`1.2.9`), but for **default bodies**.

Calling **`((IA)worker).Log()`** vs **`((IB)worker).Log()`** may still differ if class uses **explicit** per-interface implementations instead of single public **`Log`**.

### Default Methods vs Abstract Class vs Extension Methods

| Approach | Instance? | State on type? | Break implementers on add? |
|----------|-----------|----------------|----------------------------|
| DIM | Yes (via interface ref) | No instance fields | No (with default body) |
| Abstract class | Yes | Yes | Yes if new abstract |
| Extension method | Static syntax | No | No |

Choose **abstract base** when subtypes share **constructors and fields**. Choose **extension** for utility helpers not part of contract. Choose **DIM** when evolving **interface** in library where multiple unrelated classes already implement it and default behavior is acceptable.

## Under the Hood

Default methods live in **interface type metadata** with method bodies (ECMA-335 support for **.interfaceimpl`** and **MethodImpl** on interfaces). When class lacks implementation, **InterfaceImpl** references **default method def** on interface.

Virtual dispatch: **most specific** wins — class instance method > default interface method. **`object.GetType()`** on implementer unchanged.

Reflection: **`typeof(IPlugin).GetMethod("Shutdown")`** may show default; class may not declare method at all.

Performance: interface calls with defaults still **callvirt** through mapping — not free static extensions; JIT may devirtualize in sealed scenarios.

## Examples

### Example 1: Optional Hook

```csharp
public interface IAuditable
{
    DateTime CreatedAt { get; set; }
    void OnAudit() { } // optional hook — default empty
}

public class Order : IAuditable
{
    public DateTime CreatedAt { get; set; }
    // OnAudit not declared — default runs if called via IAuditable
}
```

### Example 2: Override When Needed

```csharp
public class SecureOrder : IAuditable
{
    public DateTime CreatedAt { get; set; }
    public void OnAudit() => SecurityLog.Write(CreatedAt);
}
```

### Example 3: Reading Framework-Style API

When documentation says "added in .NET N with default implementation," existing your **`MyHandler : IHandler`** may compile without change; new capability invoked only when cast to **`IHandler`** or when you opt in by adding override.

## Common Mistakes & Anti-patterns

**Treating DIM as place for shared state** — use abstract class or service.

**Assuming public access on class** — **`obj.DefaultMethod()`** fails if only default exists; need **`((I)obj).DefaultMethod()`**.

**Adding defaults carelessly to public API** — subtle behavior changes when defaults are non-trivial.

**Ignoring diamond conflicts** — compile errors until resolved; understand before multi-interface defaults.

**Replacing all base classes with interfaces + defaults** — loses ctor/field story.

## Production & Real-World Notes

Application code in typical ASP.NET projects **implements** interfaces with DIM more often than **defines** them. Library authors on NuGet consider DIM for **non-breaking additive APIs**.

Unit tests: mock frameworks (Moq) may require **Setup** for new default members depending on version — verify when upgrading interfaces.

Prefer **small defaults** (no-op, throw **`NotImplementedException`** for optional path) over complex logic in interface bodies — hard to test and evolve.

## Comparison / Trade-offs

**DIM vs breaking change:** DIM wins for backward compatibility; cost is complexity and diamond rules.

**DIM vs abstract class:** Abstract class when **one lineage + state**; DIM when **many existing implementers + optional additive behavior**.

**Versioning discipline:** Document whether new interface method has default; mark experimental APIs.

---

## Quick Reference

- **DIM** = method body on interface; C# 8+.
- Implementers **inherit default** without code until they override.
- Call through **interface reference** (or override on class as public member).
- **No instance fields** on interface; not a replacement for abstract class.
- **Multiple defaults** same signature → class must resolve.
- Use primarily for **library interface evolution**.

---

## Key Takeaways

- Default interface methods let interfaces ship **optional behavior** without breaking existing classes.
- Classes override defaults with normal implicit (or explicit) implementation.
- Defaults are invoked via **interface-typed** references when not overridden on class public API.
- Interfaces still lack instance state; heavy sharing stays in **abstract classes**.
- Diamond default conflicts require explicit class-level resolution.
- Application developers **consume** DIM more than author; framework evolution is the main motivation.
