# Наследование: base, override, sealed, hidden (new)

> Roadmap: `1.2.6` · Node: `1.2` — C# OOP · Depth: **глубоко**

## Learning Objectives

После этого урока ты сможешь:

- Объяснить **single inheritance** в C# и как derived class расширяет base class.
- Использовать **`base`** для вызова конструкторов и членов родителя.
- Различать **`override`** (полиморфная замена) и **`new`** (скрытие member).
- Применять **`sealed`** к методам и классам, чтобы «заморозить» цепочку наследования.
- Предсказывать, какой метод выполнится при одинаковой сигнатуре в base и derived.
- Распознавать, когда **`new`** — осознанное решение, а когда — ловушка сопровождения.

## Why This Matters

В уроках `1.2.1`–`1.2.5` ты строил классы как самодостаточные blueprint: поля за properties, constructors с инвариантами, access modifiers как границы. Реальные domain models редко ограничиваются одним классом. **`Invoice`** — это **`Document`**; **`CreditCardPayment`** — **`Payment`**. **Inheritance** моделирует отношение **is-a**: derived type **является частным случаем** base type и должен подставляться туда, где ожидается base — с оговорками, которые мы разберём здесь и в `1.2.7`.

C# допускает **single inheritance** для классов: только один прямой base class. Это убирает diamond problem из C++ multiple inheritance, но оставляет мощный механизм расширения. Язык даёт четыре связанных инструмента — **`base`**, **`override`**, **`sealed`**, **`new`** — которые в туториалах выглядят похоже, но на compile time и runtime ведут себя по-разному. Путаница **`override`** и **`new`** — один из самых частых источников багов «вызвался не тот метод» на code review. Путаница **`sealed`** с **`private`** — типы, которые кажутся расширяемыми, но ими не являются.

Middle-разработчик каждый день наследует framework base classes — **`ControllerBase`**, **`DbContext`**, **`Exception`**. Он переопределяет **`ToString`**, запечатывает performance-critical пути, иногда скрывает устаревший member base. Понимание **что compiler записывает в metadata** и **что runtime dispatch-ит** отделяет осознанный design от хрупкой случайности.

## Core Concepts

### Single inheritance и is-a

Каждый class в C# неявно наследует **`object`**, если не указано иное. Запись **`class Dog : Animal`** означает: **`Dog`** наследует instance fields (с учётом accessibility), methods, properties и events, которые base открывает derived коду. Derived **добавляет** members и **может заменить** поведение virtual members (подробно в `1.2.7`).

Inheritance — не «бесплатная» composition. Объект **`Dog`** **является** **`Animal`** при assignment:

```csharp
Animal pet = new Dog(); // legal — Dog is-a Animal
```

Это основа polymorphism. Но нарушение инвариантов base в derived constructor или overridden method ломает **Liskov Substitution Principle**: код, работавший с **`Animal`**, не должен ломаться при подстановке **`Dog`**. Inheritance связывает типы между релизами; используй его, когда specialization стабильна и shared behavior существенно — тему **`1.2.11`** развивает composition.

### Ключевое слово base

**`base`** встречается в двух местах: **constructor chaining** и **доступ к member**.

Когда выполняется constructor derived class, runtime сначала инициализирует base-часть объекта. Без явного **`: base(...)`** compiler вставляет вызов parameterless base constructor — и compilation падает, если у base нет доступного parameterless constructor.

```csharp
public class Animal
{
    protected string Name { get; }

    protected Animal(string name) => Name = name;
}

public class Dog : Animal
{
    public string Breed { get; }

    public Dog(string name, string breed) : base(name)
    {
        Breed = breed;
    }
}
```

**`base(name)`** запускает constructor **`Animal`** до присвоения **`Breed`**. Объект полностью constructed только после всей цепочки; не передавай **`this`** наружу из constructor — derived-половина может быть ещё не инициализирована.

В instance methods **`base.Method()`** вызывает **реализацию base**, даже если derived **override**-ит этот method. Полезно, когда override **оборачивает** или **расширяет** behavior, а не полностью его заменяет:

```csharp
public override string ToString() => $"[LOG] {base.ToString()}";
```

**`base`** разрешается **статически** к **immediate base type**. Он не «перепрыгивает» уровни и не dispatch-ится динамически — compiler точно знает, какой method вызвать.

### override: замена virtual behavior

**`override`** заменяет member, помеченный в base как **`virtual`**, **`abstract`** или **`override`**. Замена участвует в **runtime polymorphism**: reference типа base, указывающий на derived instance, вызывает **derived override**. Модель dispatch — тема `1.2.7`; пока запомни: **`override`** — **контракт** с base: та же signature, совместимые ожидания по behavior, metadata связывает override с **slot** virtual method table.

```csharp
public class Animal
{
    public virtual void Speak() => Console.WriteLine("...");
}

public class Dog : Animal
{
    public override void Speak() => Console.WriteLine("Woof");
}
```

Без **`virtual`** на base member **`override`** — compile error. Non-virtual methods override нельзя — только hide (следующий раздел). Ошибка **CS0506** защищает от polymorphism там, где его нет.

### new: hiding (неполиморфная замена)

Если base member **не virtual**, derived может объявить member с **той же name и signature** через **`new`**:

```csharp
public class Report
{
    public void Print() => Console.WriteLine("Base report");
}

public class DetailedReport : Report
{
    public new void Print() => Console.WriteLine("Detailed report");
}
```

**`new`** говорит compiler: «я **скрываю** inherited member, не override». Без **`new`** код компилируется, но с warning **CS0108** — hiding произошёл неявно. Явный **`new`** документирует intent.

Критическая разница в behavior:

```csharp
Report r = new DetailedReport();
r.Print(); // "Base report" — compile-time type Report

DetailedReport d = new DetailedReport();
d.Print(); // "Detailed report"
```

Hiding — **static dispatch** по **compile-time type** reference. **`override`** использовал бы **runtime type**. Поэтому **`new`** уместен только когда polymorphic behavior **намеренно не нужен** — часто сигнал, что лучше другое имя method или composition.

Properties и fields тоже можно hide; та же static resolution. Base member **не исчезает** из object — оба method существуют; доступный зависит от типа reference.

### sealed: остановка override или inheritance

**`sealed`** имеет два применения.

**Sealed method** — на **`override`**, запрещает дальнейший override в derived:

```csharp
public sealed override void Speak() => Console.WriteLine("Woof");
// RobotDog: CS0239 — нельзя override sealed Speak()
```

Authors framework seal-ят methods, когда дальнейшая customization ломает security или invariants — **`string.GetHashCode`** sealed в BCL.

**Sealed class** — никто не может наследовать:

```csharp
public sealed class PaymentResult { /* ... */ }
```

**`string`**, многие service types и performance-sensitive hierarchies. Ты жертвуешь extensibility ради ясности и возможностей JIT devirtualization.

**`sealed`** на class сильнее, чем seal каждого virtual member: **наследование типа** запрещено целиком. Non-sealed class со всеми sealed virtual methods всё ещё позволяет **subclassing** для новых members, но не для override этих methods.

## Under the Hood

### Object layout и inheritance

Instance derived class — **один object** на heap со **всеми instance fields** base и derived (base fields обычно идут раньше; точный layout — implementation detail CLR). Нет отдельных sub-object **`Animal`** и **`Dog`** — есть **`Dog`**, в памяти которого лежат поля **`Animal`**.

Method tables (**vtable** для reference types с virtual methods) — на **actual runtime type**. Когда **`Dog`** override-ит **`Speak`**, slot **`Speak`** в table **`Dog`** указывает на **`Dog.Speak`**. **`new`-hidden** methods **не заменяют** slot; добавляется отдельная запись в metadata derived, а base slot по-прежнему на base implementation.

### IL и dispatch

**`callvirt`** для virtual instance calls — resolution по runtime type **`this`**. Non-virtual — **`call`**, привязка на compile time.

При **`override`** metadata derived содержит **`.override`**, связь с base virtual slot. При **`new`** — **`hidebysig`**: для virtual dispatch derived method «не связан» с base slot.

Один раз посмотреть IL (ILDasm, ILSpy) на пару **`new`** vs **`override`** — polymorphism становится наглядным.

### Constructor chain и field initialization

Field initializers выполняются **до** тел constructors, **от base к derived**:

```
1. Animal field initializers
2. Animal constructor (через base chain)
3. Dog field initializers
4. Dog constructor
```

Если base constructor вызывает **virtual** method, overridden в derived, **derived override может выполниться на partially constructed object** — классический fragile base class. Production code часто избегает virtual calls из constructors.

## Syntax / Commands / API

| Механизм | Где | Эффект |
|----------|-----|--------|
| `: BaseType` | class | Single base inheritance |
| `: base(args)` | constructor | Вызов base constructor |
| `base.Member()` | method/property | Вызов реализации base |
| `virtual` | base member | Разрешает override |
| `override` | derived member | Полиморфная замена |
| `new` | derived member | Hide non-virtual base member |
| `sealed override` | method | Блок дальнейших override |
| `sealed class` | class | Блок inheritance |

## Examples

### Пример 1: Constructor chain с base

```csharp
public abstract class Entity
{
    public Guid Id { get; }
    protected Entity(Guid id) => Id = id;
}

public class Customer : Entity
{
    public string Email { get; }
    public Customer(Guid id, string email) : base(id)
    {
        Email = email ?? throw new ArgumentNullException(nameof(email));
    }
}
```

**`protected`** base constructor не даёт создать «голый» **`Entity`** снаружи, но позволяет derived инициализировать **`Id`**.

### Пример 2: override vs new

```csharp
Notifier n = new SmsNotifier();
n.Send("Hi");  // SMS — override, runtime dispatch
n.Log("Hi");   // Log: Hi — new hide, static dispatch на Notifier
```

Асимметрия удивляет junior один раз; после урока — нет.

## Common Mistakes & Anti-patterns

**`new` вместо `override`.** Polymorphic path вызывает base behavior. Fix: **`virtual`** + **`override`** или rename.

**Virtual calls из base constructors.** Derived override видит default fields. Отложи в **`Init`** или factory.

**Глубокие башни inheritance.** Три+ уровня — часто smell; см. `1.2.11`.

**Public `new` на framework base.** Callers с base reference не увидят API.

**Override без `base`, когда нужно extend.** Copy-paste base logic drift-ит — вызывай **`base`**.

## Production & Real-World Notes

**`ControllerBase`**, **`DbContext`**, **`Exception`** — осознанные hierarchies. При override **`Dispose`**: **`sealed public void Dispose()`** + **`protected virtual Dispose(bool)`** — controlled extensibility. Analyzers флагают **`new`** на inherited members (CA1060).

## Comparison / Trade-offs

**Inheritance vs composition (`1.2.11`).** Inheritance — is-a и shared implementation; composition — делегирование без coupling identity.

**override vs new.** **`override`** — virtual dispatch и LSP. **`new`** — отдельный method per compile-time type, без polymorphism.

**sealed class vs sealed methods.** Sealed class блокирует subclassing; sealed methods — только override chain.

## Key Takeaways

- Один base class; **`object`** — корень.
- **`base`** — static resolution parent members.
- **`override`** — runtime dispatch для virtual chain.
- **`new`** — hide, static dispatch по compile-time type.
- **`sealed`** — freeze override или inheritance.
- Constructors: base first; осторожно с virtual из base ctor.
