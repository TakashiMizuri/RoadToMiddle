# Exercises: Code review etiquette

> Complete after lectures and passing `4.test-yourself.md`. Lighter role-play and scenarios — no heavy Git required. Ask AI for feedback on your drafts, not full scripts.

## Easy

### E1. Label the comment
**Goal:** Classify feedback tier.

For each comment, write: **blocking | suggestion | nit | question | praise**

1. «SQL string concatenation with user input — use parameters.»
2. «Consider extracting this 80-line method into a service class.»
3. «Nice use of `ProblemDetails` — matches our API guide.»
4. «Why 30s timeout instead of 10s?»
5. «Extra blank line before return — optional.»

**Success criteria:** 5/5 labels with one-sentence why.

---

### E2. Rewrite toxic → constructive
**Goal:** Practice actionable tone.

Rewrite as professional PR comment:

| Toxic | Your version |
|-------|--------------|
| «This is a mess.» | |
| «Nobody writes code like this.» | |
| «Obviously wrong.» | |

**Success criteria:** Each references **behavior/code**, suggests fix or question, no personal attack.

---

## Medium

### M1. Author response drill
**Goal:** Fix or explain.

**Scenario A:** **[blocking]** Missing authorization on `DELETE /api/orders/{id}`.

**Scenario B:** **[nit]** Variable `data` → `orderSummary`.

Write short author replies (1–3 sentences each) including what you **do** next.

**Success criteria:** A commits to fix; B fixes or defers politely with reason.

---

### M2. Mini review checklist
**Goal:** Reviewer mindset for `0.2.8` PR.

Imagine PR: `feat(auth): add password reset email`.

Write **5 bullet checks** you would perform before approve (mix security, tests, UX, API).

**Success criteria:** At least one item per category; nothing about «general vibe» only.

---

### M3. Role-play thread (written)
**Goal:** Full async exchange.

**Setup:** Reviewer leaves:
> **[suggestion]** Return 404 instead of 200 with empty body when user not found.

Write:
1. Author reply (agree or disagree with evidence).
2. Follow-up reviewer message if author disagreed reasonably.
3. Final author resolution message.

**Success criteria:** Thread reaches clear outcome without sarcasm.

---

## Hard (stretch)

### H1. Facilitate disagreement
**Goal:** Middle-level escalation judgment.

**Scenario:** Author insists on raw SQL for performance; reviewer wants EF only — team standard says EF unless benchmark.

Write 4–6 sentences: how author and reviewer proceed **before** involving tech lead.

**Success criteria:** Mentions evidence (benchmark), team doc, possible compromise (EF + raw comment), escalation path.

---

## Submission checklist
