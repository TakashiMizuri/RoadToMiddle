# AsyncLocal — ambient-контекст в async-цепочке

> Roadmap: `1.4.32` · Node: `1.4` — C# async/await · Depth: понимание

## Learning Objectives

После урока ты сможешь:

- Объяснить **`AsyncLocal<T>`** как **logical call context**, текущий через `await`.
- Сравнить с **`ThreadLocal<T>`** и **`HttpContext`**.
- Описать **копирование** значений при continuation (не общий mutable слот между ветками).
- Назвать применения: **correlation ID**, **culture**, **tenant**, **logging scope**.
- Знать pitfalls: **утечки** контекста, **параллельные** ветки.

---

## Why This Matters

Выставили `CultureInfo` на потоке запроса, `await SaveChangesAsync()` — после continuation culture может сбиться: **поток сменился**. Положили user ID в **static** — параллельные запросы перетирают друг друга. **`AsyncLocal<T>`** переносит **ambient** состояние через async без параметра в каждой сигнатуре.

ASP.NET Core: **`IHttpContextAccessor`** опирается на **`AsyncLocal<HttpContext>`**. Middle понимает `AsyncLocal` при **multi-tenant** логах, **OpenTelemetry**, custom middleware после `await`.

## Core Concepts

### Logical flow vs физический поток

Один HTTP-запрос — **логический поток** через много thread-pool threads на каждом `await`. **`ThreadLocal`** для request data — неверно.

**`AsyncLocal`** живёт в **execution context**, который восстанавливается при continuation — даже на другом потоке.

### Базовое использование

```csharp
private static readonly AsyncLocal<string?> _correlation = new();

public static string? CorrelationId
{
    get => _correlation.Value;
    set => _correlation.Value = value;
}
```

Middleware:

```csharp
CorrelationId = context.TraceIdentifier;
try { await _next(context); }
finally { CorrelationId = null; }
```

Глубокий стек читает без параметров — для cross-cutting, не для обязательных business deps.

### Copy-on-write

Присвоение **`Value`** создаёт логическую копию для текущего и **дочерних** контекстов. Параллельные ветки после расхождения не портят друг друга. **Mutable reference types** в `AsyncLocal` — осторожно.

### HttpContext

Упрощённо: `AsyncLocal<HttpContext>` через accessor. В **BackgroundService** `HttpContext` null — параметры из сообщения, не ambient HTTP.

---

## Under the Hood

Continuations используют **ExecutionContext** — `AsyncLocal` в logical call context. **`SuppressFlow`** — редко в app code. Не очищенный `AsyncLocal` на pool thread → **leak** на чужой работе.

## Examples

### Tenant

```csharp
public static IDisposable Use(Guid tenantId)
{
    _tenant.Value = tenantId;
    return new DisposeAction(() => _tenant.Value = null);
}
```

### Logging

Serilog **`LogContext.PushProperty`** — похожий ambient stack.

---

## Common Mistakes & Anti-patterns

Скрывать обязательные business-параметры — тесты и явность страдают.

Думать, что parallel foreach всегда наследует parent без проверки.

Не **clear** в `finally`.

Заменять **`CancellationToken`** ambient-флагом.

---

## Quick Reference

- `AsyncLocal` = ambient на логический async flow.
- Set на границе; **clear в finally**.
- Не замена DI / cancellation.
- `IHttpContextAccessor` — тот же паттерн для HTTP.

---

## Key Takeaways

- Потоки меняются; **logical context** — нет.
- Correlation/tenant/culture — да; скрытые deps — нет.
- **Очищай** значения.
- **`ThreadLocal`** не для async server.
