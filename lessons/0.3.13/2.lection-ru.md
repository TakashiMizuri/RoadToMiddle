# HTTP/1.1 vs HTTP/2 vs HTTP/3 (QUIC) — обзор

> Roadmap: `0.3.13` · Node: `0.3` — HTTP и Web · Depth: понимание

## Learning Objectives

После этого урока ты сможешь:

- Кратко описать модель connections в **HTTP/1.1** и историческую проблему **head-of-line blocking**.
- Объяснить улучшения **HTTP/2**: binary framing, multiplexing, HPACK, server push (legacy).
- Описать **HTTP/3** поверх **QUIC (UDP)** и зачем multiplexing перенесён ниже transport layer.
- Узнавать negotiated version (DevTools, `curl -v`).
- Обсуждать **trade-offs** для Kestrel, reverse proxy, CDN без реализации protocol stacks.
- Связать выбор версии с **latency**, **TLS** и lossy networks.

---

## Why This Matters

Ты уже шлёшь requests через curl, но не всегда видишь **какую версию HTTP** они используют. От версии зависит performance под нагрузкой: SPA с десятками assets, API gateway, mobile на нестабильном Wi‑Fi. HTTP/1.1 требовал workarounds (domain sharding, bundling). HTTP/2 убрал wire-неэффективность, но **TCP HoL** при packet loss остался. HTTP/3 на **QUIC** изолирует streams при потере пакетов.

В application code HTTP/3 почти не выбирают — negotiate Kestrel, nginx, Cloudflare, browser. Middle всё равно нужен overview: почему HTTP/2 default на HTTPS, когда fallback на HTTP/1.1, почему «enable HTTP/2» не заменяет caching и good API design (`0.3.14`).

## Core Concepts

### HTTP/1.1 — text, sequential, keep-alive

**HTTP/1.1** — **plain text**: request line, headers, blank line, body. **`Connection: keep-alive`** переиспользует **TCP connection** для нескольких requests **последовательно** (pipelining редок и сломан на практике).

Browsers открывали **~6 parallel TCP connections** на host — connection churn и повторные TLS handshakes.

**HoL на уровне HTTP/1.1:** медленный request A блокирует B на **той же connection**. Parallel connections смягчают, но не убирают overhead.

Для JSON API HTTP/1.1 достаточен; больно при **многих мелких assets** — bundling во frontend.

### HTTP/2 — one connection, many streams

**HTTP/2** сохраняет semantic HTTP, меняет **framing**: binary **frames**, multiplexed **streams** на одном TCP. Response B может идти пока A ещё stream'ится.

**HPACK** сжимает headers. **Server push** — ограниченное adoption; чаще preload в HTML.

Negotiation: **TLS ALPN** (`h2`). Cleartext `h2c` — редко вне internal mesh.

**TCP HoL остаётся:** потеря TCP packet **stall все streams** на connection.

```
HTTP/1.1: Req A → Resp A → Req B → Resp B  (одна conn)
HTTP/2:   Stream 1 и 3 interleaved на одной conn
```

### HTTP/3 и QUIC

**HTTP/3** на **QUIC** (UDP, user-space stacks): encryption by default (TLS 1.3), **независимые streams при loss**, connection migration (Wi‑Fi ↔ LTE), осторожный **0-RTT** для idempotent ops.

Fallback: **HTTP/3 → HTTP/2 → HTTP/1.1** через **Alt-Svc** или DNS HTTPS records.

Для CRUD API в VPC HTTP/3 даёт gain mainly **edge-to-client**, не между microservices.

### Что не меняется

Methods, status codes, headers, `https://`, cookies, CORS, JSON — те же. Version — **transport optimization**, не новый API style.

## Under the Hood

**ALPN** в TLS: client `http/1.1`, `h2`; server выбирает. `curl -v` показывает accepted protocol.

HTTP/2: **flow control**, priority; proxy buffer ломает streaming. **gRPC** требует HTTP/2.

QUIC: **UDP 443**; firewall без UDP → fallback TCP. Load balancers должны speak QUIC или terminate на edge.

Kestrel: HTTP/1.1 + HTTP/2 + HTTP/3 на HTTPS где поддерживается.

```

## Examples

SPA: HTTP/1.1 — six conns; HTTP/2 — one conn, many streams. Mobile loss: HTTP/3 изолирует stream loss. CDN HTTP/3 → origin HTTP/2 — version меняется per hop.

---

## Common Mistakes & Anti-patterns

HTTP/2 не лечит slow SQL. Не отключать HTTP/1.1 без legacy clients. Server push ≠ cache headers. UDP firewall и HTTP/3. HTTP/2 ≠ WebSockets (`0.3.15`).

---

## Production & Real-World Notes

Edge terminates HTTP/3; origin часто HTTP/2/1.1. Logs с protocol version. HTTP/2 single conn → важность timeouts (`0.3.14`).

---

## Comparison / Trade-offs

| | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---|----------|--------|--------|
| Transport | TCP | TCP | UDP QUIC |
| Multiplexing | Parallel TCP | One TCP | QUIC streams |
| Loss HoL | Per conn | Whole TCP | Per-stream |
| JSON API | OK | OK | Marginal internal |

```

---

## Key Takeaways

1. HTTP/1.1 — text, HoL per connection.
2. HTTP/2 — multiplex on one TCP, HPACK.
3. TCP loss stalls all HTTP/2 streams → HTTP/3.
4. QUIC/UDP, per-stream isolation.
5. Version negotiate host/proxy/browser.
6. Protocol ≠ substitute for caching/design.
7. Observe with curl -v и DevTools.
