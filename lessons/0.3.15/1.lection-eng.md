# WebSockets vs HTTP — When to Use

> Roadmap: `0.3.15` · Node: `0.3` — HTTP and the Web · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Contrast **HTTP request-response** with **WebSocket full-duplex** messaging.
- Describe the **WebSocket handshake** (HTTP Upgrade) and what changes after it.
- Identify use cases where WebSockets excel (live dashboards, chat, gaming) vs where HTTP wins.
- Compare **WebSockets** to **SSE (Server-Sent Events)** and **long polling** at overview level.
- Recognize **scaling challenges** (sticky sessions, backplane) for WebSocket servers in ASP.NET Core.
- Choose between **REST + polling**, **SSE**, and **WebSockets** for a given product requirement.

---

## Why This Matters

Most of your ASP.NET Core and React work is **HTTP**: fetch JSON, render, submit forms. Then product asks for **live notifications**, collaborative editing, or tick-by-tick stock prices. Polling every second wastes bandwidth and adds latency; cramming everything into faster HTTP/2 (`0.3.13`) does not give the server a way to **push arbitrary messages** without a request. **WebSockets** upgrade an HTTP connection into a persistent, bidirectional channel — both sides send frames anytime.

Choosing wrong transport creates operational pain. WebSockets through corporate proxies sometimes fail; SSE is simpler for **server→client only** streams; HTTP caching and CDN rules do not apply the same way. Middle developers articulate **when not to use WebSockets** — many "real-time" features need only SSE or occasional fetch — and understand SignalR's fallback ladder.

## Core Concepts

### HTTP Request-Response Recap

HTTP is **client-initiated**: server sends data only in **response to a request** (except HTTP/2 push, largely deprecated). For "new data" the client polls (`setInterval` + fetch), uses **long polling** (hold request open until event), or opens a **push channel** (SSE/WebSocket).

Polling is simple but scales poorly: N users × poll rate = load even when idle. Long polling reduces empty responses but ties up connections and complicates timeouts (`0.3.14`).

### WebSocket Model

After **Upgrade handshake**, connection becomes **WebSocket**: **full-duplex**, **low overhead framing** (2–14 byte header typical), **persistent** until close. Either peer sends **text or binary** frames. No per-message HTTP headers like `Authorization` on every frame — auth usually happens at handshake (cookie, query token) or first application message.

Handshake starts as HTTP:

```
GET /chat HTTP/1.1
Host: api.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
```

Server **`101 Switching Protocols`** — then HTTP is done; WebSocket frames follow on same TCP connection.

Use when **both directions** need low-latency messages frequently: chat, multiplayer game input, live trading UI, collaborative cursors.

### Server-Sent Events (SSE) — One-Way Alternative

**SSE** (`text/event-stream`) keeps HTTP connection open; **server pushes** text events; client **`EventSource`** auto-reconnects. **Server→client only** — client still uses HTTP POST for commands. Simpler than WebSockets for **notifications feed**, **progress streams**, **live logs**. Works over standard HTTP/2, often easier through proxies.

Choose SSE when client does not need to send high-frequency messages on same channel.

### Long Polling — Compatibility Fallback

Client requests; server **holds response** until event or timeout, then client immediately reconnects. Works everywhere HTTP works. Higher latency and server connection count than WebSocket/SSE — legacy fallback (SignalR transports).

### When HTTP Alone Is Enough

- **CRUD REST** with occasional updates — cache + refetch on mutation.
- **TanStack Query** with **`staleTime`** and **`refetchOnWindowFocus`** — good enough for many admin panels.
- **Webhooks** — server-to-server push without browser WebSocket.
- **Batch jobs** — status via GET `/jobs/{id}` every few seconds.

Do not add WebSockets because "real-time sounds modern."

### ASP.NET Core and SignalR (Overview)

**SignalR** abstracts hubs over **WebSockets**, **SSE**, or **long polling** — negotiates best transport. **`Hub`** methods invoke client/server; **backplane** (Redis) broadcasts across server instances. Raw WebSockets: **`WebSocketMiddleware`** or **`HttpContext.WebSockets`** for full control.

Scaling WebSockets: connections are **sticky** to server process memory — load balancer **session affinity** or shared backplane required. Unlike stateless REST, horizontal scale needs design.

### WebSockets vs HTTP/2 Multiplexing

HTTP/2 multiplexes **independent request-response streams** on one connection — still request-response semantics. WebSocket is **one long-lived conversation** without re-establishing request context per message. Different problems: HTTP/2 optimizes many API calls; WebSocket optimizes **continuous event stream both ways**.

## Under the Hood

WebSocket **masking**: client frames XOR-masked (RFC 6455) — historical proxy safety. **Ping/pong** frames keep connection alive through NAT (`0.3.14` idle timeouts).

TLS: **`wss://`** — WebSocket over TLS, same port 443 often with ALPN still HTTP first for upgrade.

Proxies must support Upgrade; some strip it — fallback transports matter.

Browser **single origin** policy and cookies apply at handshake — CSRF on WebSocket upgrade is a known hardening topic (validate Origin header server-side).

Message size limits — Kestrel **`WebSocketOptions`** max receive size; huge payloads better chunked over HTTP download.

## Syntax / Commands / API

**Browser WebSocket (concept):**

```javascript
const ws = new WebSocket("wss://api.example.com/ws");
ws.onmessage = (e) => console.log(JSON.parse(e.data));
ws.send(JSON.stringify({ type: "ping" }));
```

**SSE client:**

```javascript
const es = new EventSource("/api/events");
es.onmessage = (e) => console.log(e.data);
```

**curl — WebSocket limited; test upgrade with:**

```bash
curl -i -N \
  -H "Connection: Upgrade" \
  -H "Upgrade: websocket" \
  -H "Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==" \
  -H "Sec-WebSocket-Version: 13" \
  https://echo.websocket.org/
```

**SignalR hub (sketch):**

```csharp
app.MapHub<ChatHub>("/hubs/chat");
```

## Examples

### Example 1: Live sports scoreboard

Scores update every few seconds for thousands of viewers. **SSE** or WebSocket from server push; polling every 1s crushes API. If viewers never send data on channel — **SSE** suffices. Comment chat beside scoreboard — **WebSocket** for bidirectional chat.

### Example 2: Dashboard behind corporate proxy

WebSocket blocked; SignalR falls back to **long polling** — works but higher latency. Product accepts or IT whitelists `wss://`.

### Example 3: Order status

User submits order via **HTTP POST**; progress via **SSE** stream or poll **`GET /orders/{id}`** every 2s until complete. WebSocket overkill unless sub-second updates across many concurrent jobs on same page.

## Common Mistakes & Anti-patterns

**WebSocket for everything** — operational cost (sticky sessions, reconnect logic, no CDN cache).

**No heartbeat** — NAT/firewall drops silent connections; implement ping/pong or client reconnect with backoff.

**Auth only in first message without TLS** — credentials leak; use **`wss://`** and validate at handshake.

**Ignoring backpressure** — fast producer floods slow client; drop, buffer with limits, or disconnect.

**Assuming SignalR = WebSockets only** — test fallback transports in staging.

**Same pool limits as REST** — long-lived WebSocket ties connection slot on server (`MaxConcurrentConnections`).

## Production & Real-World Notes

Monitor **concurrent WebSocket connections**, message rate, reconnect storms after deploy. **Graceful shutdown**: notify clients, drain hubs before process kill.

**Redis backplane** for multi-node SignalR. **Azure Web PubSub**, **Pusher**, managed services offload scale.

Rate limit at application layer — WebSocket bypasses typical per-request HTTP rate limits once connected.

For **mobile background**, WebSocket may suspend — push notifications (FCM/APNs) complement in-app live data.

## Comparison / Trade-offs

| Transport | Direction | HTTP compatible | Complexity | Typical use |
|-----------|-----------|-----------------|------------|-------------|
| HTTP poll | Client pull | Yes | Low | Rare updates |
| Long poll | Client pull (held) | Yes | Medium | Legacy fallback |
| SSE | Server push | Yes | Medium | Feeds, logs |
| WebSocket | Bidirectional | Upgrade | Higher | Chat, games, collab |
| HTTP/2 REST | Request-response | Yes | Low | CRUD APIs |

**WebSockets vs HTTP:** not replacement — complementary. Public REST stays HTTP; live channel adds WebSocket/SSE where justified.

## Quick Reference

```
HTTP        = request → response; server waits for client ask
WebSocket   = Upgrade 101; then bidirectional frames until close
SSE         = one HTTP stream; server → client events only

Use WebSocket: frequent both-way messages, low latency
Use SSE: server push only, simpler
Use HTTP: CRUD, cacheable resources, stateless scale

Scale WS: sticky LB or Redis backplane; wss:// always in prod
SignalR: WS + SSE + long poll fallback
```

## Key Takeaways

1. **HTTP** is client-driven request-response; not ideal for continuous server push.
2. **WebSocket** upgrades HTTP to **persistent full-duplex** messaging.
3. **SSE** covers **server→client** push with simpler HTTP semantics.
4. Many "live" UIs work with **HTTP + smart refetch** or SSE — WebSocket only when needed.
5. **Scaling WebSockets** requires sticky sessions or backplane — unlike stateless REST.
6. **SignalR** negotiates transport and simplifies hubs in ASP.NET Core.
7. Always **`wss://`**, heartbeats, reconnect backoff, and Origin validation in production.
