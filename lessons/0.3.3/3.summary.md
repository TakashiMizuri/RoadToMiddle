# Структура HTTP-сообщения — краткая выжимка

> `0.3.3` · ~10 мин на повторение

## Главная идея

HTTP/1.1 message = **start line** + **headers** + пустая строка + optional **body**. Длина body задаётся **Content-Length** или **Transfer-Encoding: chunked**, не «до закрытия TCP» произвольно.

## Ключевые пункты

- Request line: `METHOD /target HTTP/1.1`; response: `HTTP/1.1 200 OK`.
- Строки заканчиваются **CRLF** (`\r\n`); после headers — **blank line**.
- **Host** обязателен в HTTP/1.1 request.
- Header names **case-insensitive**; **Content-Type** — как читать body.
- **204**, **HEAD** — типично без body; POST/PUT — с body + length/chunked.
- **Hop-by-hop** headers (`Connection`, `Transfer-Encoding`) — для одного hop; proxy может strip.

## Самые важные примеры

```
POST /api/orders HTTP/1.1
Host: api.shop.com
Content-Type: application/json
Content-Length: 20

{"productId":7,"qty":2}
```

Chunked terminator: `0\r\n\r\n`.

## «Под капотом» в одном абзаце

Parser читает start line → headers до `\r\n\r\n` → по Content-Length читает N bytes или chunked loop. Kestrel limits защищают от huge headers/body. HTTP/2 упаковывает то же в pseudo-headers и DATA frames. Request smuggling — конфликт Length/chunked на proxy chain.

## Частые ошибки

- Нет Host / blank line в raw HTTP.
- Off-by-one Content-Length на keep-alive connection.
- Игнор charset для HTML; assume JSON без Content-Type check.
- Buffer entire upload в память без streaming.

## Мнемоника

> **Line, lines, blank, bytes** — start line, header lines, empty CRLF, then body length as declared.
