# traceroute / ping — Basic Diagnostics

> Roadmap: `0.4.12` · Node: `0.4` — Networking · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Use **ping** to measure reachability and round-trip time (RTT) to a host.
- Use **traceroute** (or `tracert` on Windows) to see the path packets take hop by hop.
- Interpret common outcomes: success, **100% packet loss**, high latency spikes, `* * *` hops.
- Connect diagnostic results to **latency** concepts from `0.4.11` and **firewall** drops from `0.4.7`.
- Choose ping vs traceroute vs `curl` for a given "is it the network?" question.
- Document findings in a format useful for ops tickets.

---

## Why This Matters

Production incidents often start with "the API is down" or "it's slow." Before you redeploy or restart PostgreSQL, you need to know whether the problem is DNS, routing, firewall, the server process, or application code. **ping** and **traceroute** are the first tools in that chain — available on every OS, no install required.

They do not solve everything: many hosts block ICMP ping; traceroute shows router path, not HTTP errors; a green ping does not prove port 443 works. But when a developer in Berlin cannot reach a staging server in Virginia, a traceroute showing where RTT jumps from 30 ms to 200 ms tells you the problem is geographic routing or a specific hop — not a bug in your React component.

This is the **practical capstone** of node `0.4`: you apply IP, TCP, DNS, latency, and firewall mental models with real commands. Next node **`0.5.1`** moves to HTML — the content layer your HTTP requests eventually render.

## Core Concepts

### What ping Does

**ping** sends **ICMP Echo Request** packets to a target IP or hostname and waits for **ICMP Echo Reply**. Output shows whether replies arrived and **RTT** per packet — directly measuring **latency** (`0.4.11`) at the IP layer.

```bash
ping -c 4 google.com
```

On Windows:

```powershell
ping -n 4 google.com
```

Typical lines: `64 bytes from ... time=23.4 ms`. Summary: `min/avg/max`, packet loss percentage. **0% loss** and stable RTT → host reachable at IP level (ICMP allowed). **100% loss** → host down, wrong IP, ICMP blocked by firewall, or network path broken.

Ping resolves hostname via **DNS** (`0.4.5`) first — if ping fails with "unknown host," fix DNS before blaming the network path.

Ping does **not** test TCP port 443 or your ASP.NET health endpoint. Use `curl` or `Test-NetConnection` for port-level checks after ping narrows the problem.

### What traceroute Does

**traceroute** discovers the **path** from your machine to destination by sending packets with increasing **TTL (Time To Live)**. Each router decrements TTL; when it hits zero, the router returns an ICMP "time exceeded" message — revealing that hop's address. Repeat until destination reached.

```bash
traceroute api.example.com
# macOS/Linux; may need sudo for some modes
```

Windows:

```powershell
tracert api.example.com
```

Output lists **hop number**, **router IP/hostname**, **RTT samples** (often three per hop). Early hops are local ISP; middle hops are backbone; final hops enter destination network.

**`* * *`** for a hop means no response — router may not respond to probes (common, not always failure). Sudden RTT jump at one hop suggests geographic distance or congestion there. Traceroute stops when destination ICMP/port response received (implementation-dependent: UDP, ICMP, or TCP on some systems).

### ICMP Blocking and False Negatives

Many production servers **block inbound ICMP** for security (`0.4.7`). ping fails while `curl https://api.example.com/health` succeeds. Conversely, ping to `8.8.8.8` works but your API port blocked — ping OK, app unreachable.

Interpret tools in context: **ping = IP reachability (if ICMP allowed)**; **curl = TCP + HTTP application path**.

### Relating to Prior Lessons

| Symptom | Tool | May indicate |
|---------|------|----------------|
| Unknown host | ping | DNS failure (`0.4.5`) |
| ping OK, curl timeout :443 | curl / nc | Firewall/SG (`0.4.7`) |
| High RTT only from remote office | ping compare | Latency/geography (`0.4.11`) |
| Path stops at hop 8 | traceroute | Routing/blackhole upstream |
| localhost works, remote fails | compare | Binding `0.0.0.0` vs firewall (`0.4.6`, `0.4.7`) |

## Under the Hood

### TTL and Hop Discovery

IP header **TTL** prevents infinite loops. traceroute sends probe with TTL=1 → first router responds; TTL=2 → second router; until destination. RTT at each step includes that segment's **latency**. This is not identical to your HTTP request path (different protocol, possible asymmetric routing) but correlates strongly for troubleshooting.

### Windows tracert vs Linux traceroute

**tracert** uses ICMP echo requests with incrementing TTL by default. **traceroute** on Linux often uses UDP to high ports or ICMP depending on flags. Firewalls treat these differently — same destination may show different hop visibility across tools.

## Syntax / Commands / API

### ping

| OS | Command | Notes |
|----|---------|-------|
| Linux/macOS | `ping -c 10 host` | Count limit |
| Windows | `ping -n 10 host` | Count limit |
| Windows | `ping -t host` | Until Ctrl+C |

### traceroute

| OS | Command |
|----|---------|
| Linux | `traceroute host` or `traceroute -I host` (ICMP) |
| macOS | `traceroute host` |
| Windows | `tracert host` |

### Port test (companion)

```bash
# Linux/macOS
nc -zv api.example.com 443
curl -v --connect-timeout 5 https://api.example.com/health
```

```powershell
Test-NetConnection api.example.com -Port 443
```

### Document a incident (template)

```
Target: api.staging.example.com
ping: 45ms avg, 0% loss (Berlin office)
tracert: RTT jump hop 9→10 (EU→US)
curl :443: connection timed out
Hypothesis: SG blocks 443 from non-VPN; not app regression
```

## Examples

### Scenario 1: DNS Broken

```bash
ping api.mycompany.local
# ping: cannot resolve api.mycompany.local: Unknown host
```

Fix DNS or `/etc/hosts` — no traceroute needed yet.

### Scenario 2: ICMP Blocked, App Fine

```bash
ping api.example.com
# 100% packet loss
curl -I https://api.example.com/health
# HTTP/2 200
```

Document: ICMP blocked by design; use HTTP health checks for monitoring.

### Scenario 3: Routing Latency

traceroute shows 25 ms for hops 1–4, then 180 ms from hop 5 — transatlantic link. Expected for US-East server from Asia; optimize with CDN/regional deploy (`0.4.9`, `0.4.11`), not micro-optimize SQL first.

## Common Mistakes & Anti-patterns

**"ping fails, server is down"** without curl/port test — ICMP may be blocked.

**Running traceroute once during glitch** — transient loss needs repeated samples.

**Ignoring asymmetric routing** — return path may differ; hop RTT is indicative, not contractual SLA.

**Using ping to debug CORS or 401** — application layer; use browser DevTools and curl with headers.

---

## Production & Real-World Notes

Monitoring uses **synthetic HTTP checks**, not ping, for public APIs. Internal networks may allow ICMP for NOC tools. **MTR** (`mtr`) combines ping + traceroute continuously — useful for intermittent loss.

Corporate VPN changes path — always note whether VPN on when comparing traceroutes.

## Comparison / Trade-offs

| Tool | Layer | Best for |
|------|-------|----------|
| ping | ICMP/IP | Basic reachability, RTT |
| traceroute | IP path | Where delay/loss appears |
| curl / nc | TCP/HTTP | Port open, app responding |
| nslookup | DNS | Name resolution |

## Quick Reference

- **ping** → RTT + packet loss (if ICMP allowed).
- **traceroute / tracert** → hop-by-hop path and per-hop latency.
- **`* * *`** → hop silent, not always failure.
- ping OK ≠ port 443 open; ping fail ≠ server down.
- Follow up with **curl** and **Test-NetConnection** for services.

## Key Takeaways

- ping measures **latency/reachability** at ICMP layer when permitted.
- traceroute locates **where** path slows or stops.
- Firewalls block ICMP and ports independently (`0.4.7`).
- Combine with DNS, curl, and latency vocabulary (`0.4.11`).
