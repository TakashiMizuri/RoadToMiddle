# IAsyncDisposable — краткая выжимка

> `1.4.35` · ~10 мин на повторение

## Главная идея

Ресурсы с async teardown (`DbContext`, streams, sockets) закрывают через **`IAsyncDisposable.DisposeAsync`** и **`await using`**, а не sync `Dispose` с блокировкой async work.

## Ключевые пункты

- **`await using`** вызывает `DisposeAsync` при выходе из scope (в т.ч. при exception).
- **`CreateAsyncScope()`** — async dispose scoped DI в background consumer.
- **`Dispose()` вызывающий `DisposeAsync().Wait()`** — anti-pattern (deadlock).
- `DisposeAsync` возвращает **`ValueTask`** — часто без аллокации.
- Dispose должен быть **idempotent**.
- Не **`_ = DisposeAsync()`** — await или observe faults.

## Самые важные примеры

```csharp
await using var scope = scopeFactory.CreateAsyncScope();
var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
await db.SaveChangesAsync(ct);
```

## «Под капотом» в одном абзаце

Компилятор генерирует finally с await DisposeAsync в async методе. DI scope при DisposeAsync обходит IAsyncDisposable/IDisposable сервисы в reverse order. Sync Dispose на типе с pending async I/O может блокировать pool thread или deadlock при неправильном wait.

## Частые ошибки

- `using` на IAsyncDisposable-only path.
- DbContext живёт дольше request scope.
- Fire-and-forget dispose.

## Мнемоника

**«Opened with await — closed with await using»**
