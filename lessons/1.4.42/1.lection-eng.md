## Why This Matters

Node `1.4` covered mechanics — state machines, `ConfigureAwait`, `ValueTask`, debugging, legacy fixes. A real ASP.NET solution has **hundreds of methods**. One `.Result` in a singleton or one `.ToList()` in a hot endpoint undermines the architecture. Teams need a **checklist** — like security review or accessibility pass — runnable before release and on every PR touching I/O.

This lesson synthesizes `1.4.1`–`1.4.41` into **actionable gates** for Middle developers owning a service codebase. After `1.4.42`, you move to CLR/runtime (`1.5.1`) with async patterns already operationalized in your web stack.

## Core Concepts

### Principle: Async at I/O Boundaries, All the Way Out

**Async all-the-way** does not mean "every method is `async`." It means: from **Kestrel** through **your code** to **database/network**, no synchronous blocking on I/O. CPU-bound work may use `Task.Run` deliberately — different decision (`1.4.x` parallel vs concurrent).

The checklist enforces: **if the callee has `*Async`, await it; never block with `.Result`.**

### Checklist Category 1: Endpoints

| Check | Pass criteria |
|-------|---------------|
| Controller actions return `Task` / `Task<T>` | No `void` except rare event patterns |
| Minimal APIs return `Task` / `ValueTask` | Await I/O |
| Accept `CancellationToken` | `HttpContext.RequestAborted` or parameter |
| No `.Result` / `.Wait()` in actions | Grep clean |
| No sync EF (`ToList()`, `SaveChanges()`) | Use `*Async` variants |
| No `Task.Run` for DB/network | Direct async APIs |

```csharp
// PASS
app.MapGet("/users/{id}", async (int id, IUserService svc, CancellationToken ct) =>
    await svc.GetAsync(id, ct));

// FAIL
app.MapGet("/users/{id}", (int id, IUserService svc) =>
    svc.GetAsync(id).Result);
```

### Checklist Category 2: Services and Repositories

| Check | Pass criteria |
|-------|---------------|
| I/O methods return `Task`/`Task<T>` | Consistent naming `*Async` |
| `CancellationToken` last parameter | Propagated to EF/HTTP |
| Interfaces async for I/O | Sync only for pure CPU/parsing |
| No hidden sync in properties | No `.Result` in getters |
| Scoped `DbContext` not shared parallel | One context per request per flow |

### Checklist Category 3: EF Core and Data Access

| Check | Pass criteria |
|-------|---------------|
| Queries use `ToListAsync`, `FirstOrDefaultAsync`, etc. | |
| `SaveChangesAsync` | |
| Raw SQL: `ExecuteSqlRawAsync` | |
| `await using` for connections if manual | |
| Projections in query, not sync iterate | Avoid materialize-then-map block |
| Connection resilience | Pool size configured; no per-query new connection storm |

### Checklist Category 4: HTTP Clients and External APIs

| Check | Pass criteria |
|-------|---------------|
| `IHttpClientFactory` | Not `new HttpClient()` per request |
| `GetAsync`, `PostAsync`, `SendAsync` awaited | |
| Polly/resilience policies async | |
| Timeouts via `CancellationTokenSource` linked to request token | |

### Checklist Category 5: Middleware and Filters

| Check | Pass criteria |
|-------|---------------|
| `InvokeAsync` returns `Task` | `await next(context)` |
| Action filters: `IAsyncActionFilter` | `await next()` |
| Exception middleware async-friendly | |
| Auth handlers async where I/O | |

### Checklist Category 6: Background and Fire-and-Forget

| Check | Pass criteria |
|-------|---------------|
| No `_ = DoAsync()` in request without tracking | Use `BackgroundTaskQueue` / channel (`1.4.34`) |
| `IHostedService` for startup init | Not ctor `.Wait()` |
| `UnobservedTaskException` logged | |
| Background work gets **its own scope** | `IServiceScopeFactory.CreateScope()` |

### Checklist Category 7: DI and Construction

| Check | Pass criteria |
|-------|---------------|
| No async ctor | `CreateAsync` / hosted warmup (`1.4.38`) |
| Singletons thread-safe | No mutable state without sync |
| No captive dependency async mismatch | Singleton not holding scoped `DbContext` |

### Checklist Category 8: Tooling and CI

| Check | Pass criteria |
|-------|---------------|
| Grep/CI rule: `.Result`, `.Wait(`, `.GetAwaiter().GetResult()` | Zero in `src/` except approved list |
| Analyzer: `AsyncFixer`, `VSTHRD` rules (optional) | Warnings as errors in CI |
| Load test baseline | Thread pool queue stable |
| OpenTelemetry spans on EF/HTTP | Verify async path in traces |

## Under the Hood

An **async-all-the-way** ASP.NET app keeps **thread pool utilization low** during I/O waits — Kestrel accepts more connections with fewer workers busy. Blocking calls **inflate** `ThreadPool` **queue length** and **completed work items** metric without increasing throughput.

**SyncContext** absence in ASP.NET Core means `ConfigureAwait(false)` in library code is optional for app layers — checklist does not require it in controllers.

**Minimal APIs** compile lambdas to async state machines same as controllers — same rules apply.

## Audit Workflow

1. **Automated grep** — `.Result`, `.Wait(`, `.GetAwaiter().GetResult()`, `.ToList()` on `IQueryable` in web project.
2. **Manual sample** — trace 3 critical endpoints on diagram (`1.4.40`).
3. **Load test** — 100 concurrent reads; watch thread pool counters.
4. **PR template** — checkbox: "No new sync-over-async."
5. **Allowlist** — document approved legacy shims with ticket IDs.

```bash
# Example ripgrep (adjust path)
rg "\.(Result|Wait\(|GetAwaiter\(\)\.GetResult\()" src/ --glob "*.cs"
rg "\.ToList\(\)" src/ --glob "*Controller*.cs"
```

## Common Mistakes & Anti-patterns

**Checkbox audit without grep** — misses new `.Result` in merge.

**Exempting "small" sync calls** — death by thousand cuts.

**async void in ASP.NET except IExceptionFilter** — use `async Task`.

**Returning `Task` without await** (accidental sync method returning task) — CA2012 / analyzer.

**Global `DbContext` singleton** — breaks async scoped model entirely.

## Production & Real-World Notes

**Readiness probe** should fail if hosted warmup incomplete — related to async init, not sync blocking.

**Kubernetes** CPU limits + thread pool blocking = cascading timeouts — checklist prevents class of incidents.

Re-run checklist on **major dependency upgrades** (EF, SDK) — new sync APIs appear.

Store checklist in repo `docs/async-checklist.md` — team contract.

## Comparison / Trade-offs

| Strictness | Benefit | Cost |
|------------|---------|------|
| Zero tolerance `.Result` | Max scalability | Legacy integration harder |
| Allowlisted shims | Pragmatic | Needs governance |
| Analyzers as warnings | Education | Noise until cleanup |
