# IAsyncDisposable — Async Cleanup

> Roadmap: `1.4.35` · Node: `1.4` — C# async/await · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Distinguish **`IDisposable`**, **`IAsyncDisposable`**, and when each applies.
- Implement **`await using`** and **`ConfigureAwait`** considerations in dispose paths.
- Use **`IAsyncDisposable`** with **`DbContext`**, streams, **`HttpClient`** wrappers, and **DI scopes**.
- Avoid **sync-over-async** in `Dispose()` — the classic `DisposeAsync().GetAwaiter().GetResult()` trap.
- Apply **`CreateAsyncScope()`** and host shutdown cleanup in background workers (`1.4.34`).

---

## Why This Matters

`DbContext`, `SqlConnection`, `FileStream` (with async APIs), gRPC channels, and custom resources may need **async flush/close** — sending a final buffer, committing with I/O, graceful protocol shutdown. Calling **sync `Dispose`** may block on pending I/O or **deadlock** if it waits on async work incorrectly.

C# 8+ **`IAsyncDisposable`** and **`await using`** let cleanup participate in the **async** world. ASP.NET Core 3+ and EF Core integrate async disposal in **`CreateAsyncScope()`**. Middle developers must dispose resources on the **same async path** they were used — especially in scoped services and queued background work.

## Core Concepts

### IDisposable vs IAsyncDisposable

**`IDisposable.Dispose()`** — synchronous, runs on calling thread, must complete quickly. Fine for releasing handles and pure memory.

**`IAsyncDisposable.DisposeAsync()`** — returns **`ValueTask`**, can **`await`** I/O during teardown.

A type may implement **both**. Rule of thumb:

- Used only synchronously, no pending async I/O → **`IDisposable`** may suffice.
- **`await`** used on the instance or async close required → prefer **`IAsyncDisposable`** (often both for backward compatibility).

### await using

```csharp
await using var stream = new FileStream(path, FileMode.Open, FileAccess.Read, FileShare.Read, 4096, useAsync: true);
await ProcessAsync(stream);
// DisposeAsync called at end of scope
```

Compiler expands to try/finally calling **`DisposeAsync()`**. For types implementing only **`IDisposable`**, `await using` still works via **sync dispose** wrapped as async (no real async benefit).

### DbContext and EF Core

```csharp
await using var scope = _scopeFactory.CreateAsyncScope();
var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
await db.SaveChangesAsync(ct);
// scope.DisposeAsync disposes DbContext — may flush async providers
```

**`CreateScope()`** + sync dispose still works but may block on async disposal paths internally — **`CreateAsyncScope()`** is preferred in async hosts.

### Do Not Sync-Block DisposeAsync

Anti-pattern in `IDisposable.Dispose()`:

```csharp
public void Dispose()
{
    DisposeAsync().AsTask().GetAwaiter().GetResult(); // deadlock risk
}
```

If you must expose sync dispose, document limitations or use **`IAsyncDisposable`** only and force callers to **`await using`**. Library types sometimes use **`SemaphoreSlim`** to guard double-dispose across sync/async paths — complex; prefer one clear pattern.

## Under the Hood

### DisposeAsync State Machine

`await using` on **`IAsyncDisposable`** generates async disposal in the **same state machine** as the method — disposal runs before leaving the block, even on exceptions — mirroring **`using`** guarantees.

### ValueTask in DisposeAsync

**`DisposeAsync`** returns **`ValueTask`** to avoid allocating when dispose completes synchronously (common for layered wrappers).

### DI Container Disposal

**`ServiceProviderScope`** / **`AsyncServiceScope`** disposes resolved **`IAsyncDisposable`** and **`IDisposable`** services in reverse construction order. Mixed lifetimes in one scope — all disposed at scope end.

### Finalizers

Async disposal does not replace **finalizers** for truly leaked references — still avoid relying on finalization; **`await using`** at scope boundaries is explicit.

## Syntax / Commands / API

| Construct | Behavior |
|-----------|----------|
| `await using var x = ...` | Calls `DisposeAsync` at scope exit |
| `await using (var x = ...)` | Block form |
| `IAsyncDisposable.DisposeAsync()` | Async cleanup |
| `CreateAsyncScope()` | Scope with async dispose |
| `ConfigureAwait(false)` in library dispose | Optional for library code |

## Examples

### Custom Async Resource

```csharp
public sealed class AsyncConnection : IAsyncDisposable
{
    private bool _disposed;

    public async ValueTask DisposeAsync()
    {
        if (_disposed) return;
        await SendGoodbyeAsync();
        await _socket.DisposeAsync();
        _disposed = true;
    }
}
```

Idempotent dispose — second call no-op.

### Background Queue Scope (`1.4.34`)

```csharp
await using var scope = _scopeFactory.CreateAsyncScope();
await work(scope.ServiceProvider, stoppingToken);
```

Ensures scoped **`IAsyncDisposable`** services tear down without sync block on consumer thread.

### IAsyncEnumerable + await using

```csharp
await foreach (var item in GetItemsAsync(ct))
{
    await using var lease = await RentAsync(item, ct);
    await ProcessAsync(lease, ct);
}
```

Nested async disposal per iteration — order: lease first, then enumerator disposal.

## Common Mistakes & Anti-patterns

**`using` instead of `await using`** on **`IAsyncDisposable`** — may call sync dispose path only or suppress async cleanup.

**Disposing singleton** `HttpClient`-like service per request — wrong lifetime; singleton disposes at app shutdown.

**Double dispose** races — guard with flag or **`Interlocked`**.

**Fire-and-forget `DisposeAsync()`** — `_ = resource.DisposeAsync()` — same observation issues as any unobserved task if dispose throws.

**Capturing disposed `DbContext`** after scope ends in queued work — scope disposal is the boundary.

---

## Production & Real-World Notes

On **host shutdown**, `IHost` disposes **`IAsyncDisposable`** hosted services — implement **`IAsyncDisposable`** on long-lived connections if async close matters.

**Tests**: `await using` in ` IAsyncLifetime` / test teardown — consistent async cleanup.

**Analyzers**: CA1849 (call async when possible) flags sync dispose on async-capable types.

For **streams**, `useAsync: true` + `await using` aligns open/read/close on async path.

---

## Comparison / Trade-offs

| Pattern | When | Risk |
|---------|------|------|
| `using` + `IDisposable` | Sync-only resources | Blocks on hidden async |
| `await using` + `IAsyncDisposable` | Async I/O resources | Caller must be async |
| `Dispose()` calls `DisposeAsync().Wait()` | Legacy interop | Deadlock |
| No explicit dispose | Short-lived GC types | Leaks handles |

## Quick Reference

- **`await using`** → **`DisposeAsync()`** at scope exit.
- **`CreateAsyncScope()`** in async ASP.NET workers.
- Don't **`.Result`** **`DisposeAsync`** in sync `Dispose`.
- EF **`DbContext`**: dispose scope, not singleton.
- Dispose **idempotent**; observe exceptions from dispose.
- Library dispose: consider **`ConfigureAwait(false)`**.

---

## Key Takeaways

- Async resources need **`IAsyncDisposable`**, not sync-blocked dispose.
- **`await using`** ties cleanup to async control flow like **`using`** for sync.
- **`CreateAsyncScope()`** pairs with background queue consumers.
- **Sync-over-async in Dispose** causes deadlocks — avoid.
- Scope boundary = dispose **`DbContext`** and scoped services.
- Unobserved **`DisposeAsync`** faults are still bugs — await them.
