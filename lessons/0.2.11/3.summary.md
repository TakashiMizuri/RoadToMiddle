# git cherry-pick и git revert для hotfix — краткая выжимка

> `0.2.11` · ~8 мин на повторение

## Главная идея

**Cherry-pick** копирует patch **одного** commit на текущую ветку (новый hash). **Revert** добавляет commit, **отменяющий** плохой shipped commit. На shared ветках hotfix = branch + pick или revert + push, **без force push**.

## Ключевые пункты

- **`git cherry-pick <hash>`** — новый commit с тем же diff; original остаётся на своей ветке.
- **Hotfix:** `main` → `hotfix/...` → fix или pick → PR → merge → tag → backport на `release/x`.
- **`git revert <hash>`** — безопасный rollback; bad commit виден в history.
- **Merge commit revert:** `git revert -m 1 <merge>`.
- **`-x`** — строка `(cherry picked from commit ...)` для audit.
- **`--abort` / `--continue`** — как при merge.
- **Cherry-pick vs merge:** один commit vs вся ветка.

## Самые важные примеры

**Hotfix с feature:**
```bash
git checkout -b hotfix/fix main
git cherry-pick -x f7a2b1c
```

**Rollback на main:**
```bash
git revert bad99 --no-edit
git push origin main
```

**Backport на LTS:**
```bash
git checkout release/2.2
git cherry-pick -x f7a2b1c
```

## Частые ошибки

- Force push вместо revert на `main`.
- Cherry-pick всей feature поштучно.
- Забытый backport на поддерживаемые release-ветки.
- Revert без учёта последующих commits на тех же строках.

## Мнемоника

> **Pick** = перенести **одну** правку. **Revert** = **новым** commit отменить старую. **Shared prod → revert, не reset.**
