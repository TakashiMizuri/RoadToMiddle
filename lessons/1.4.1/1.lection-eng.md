# Threads 101: Kernel Threads, Thread Pool, and Context Switch Cost

> Roadmap: `1.4.1` · Node: `1.4` — C# async ⚡ · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Distinguish a **kernel thread** (OS thread) from a **managed thread** and from a **`Task`**.
- Explain how the **.NET thread pool** schedules work and why it exists.
- Describe **context switch** cost and why creating threads is expensive.
- Relate thread count to **CPU cores** and **scalability** under load.
- Predict when blocking a thread pool thread causes **starvation**.
- Connect this foundation to why **`async/await`** exists (preview of `1.4.2`–`1.4.7`).

## Why This Matters

In `1.3.30` you learned that LINQ's lazy evaluation can cause **multiple enumeration** — wasted CPU cycles walking the same sequence twice. That is a *CPU efficiency* problem inside one thread. Node `1.4` opens a different dimension: **concurrency** — how your program uses **threads** while waiting on I/O or doing parallel CPU work.

Every ASP.NET Core request, every `HttpClient` call, every `DbContext` query eventually runs on **threads**. If you treat threads as free, you will build APIs that work on your laptop with ten users and collapse under production load. Middle developers must know what a thread *is*, what the **thread pool** does, and why **blocking** a pool thread is dangerous — before touching `async/await`.

This lesson is the floor. Without it, `Task.Run`, `async`, and "thread pool starvation" are magic words instead of engineering trade-offs.

## Core Concepts

### What Is a Thread?

A **thread** is the unit of execution the operating system schedules on a CPU core. Your process contains one or more threads; each has its own **stack** (typically 1 MB on Windows for default stack size), **instruction pointer**, and register state. The OS **kernel** creates and schedules these **kernel threads** (also **OS threads**). The CLR maps **managed threads** (`System.Threading.Thread`) onto kernel threads — usually one-to-one on modern .NET.

When you write `new Thread(() => { ... }).Start()`, you ask the CLR to create a managed thread, which asks the OS for a kernel thread. That is **expensive**: kernel data structures, stack allocation, scheduler registration. Creating thousands of threads per second is not viable for a web server handling thousands of concurrent requests.

**Parallelism** means multiple threads executing simultaneously on multiple cores. **Concurrency** means multiple tasks making progress — not necessarily at the same instant. One thread can serve many concurrent I/O operations if it does not block while waiting (the async model in later lessons). Confusing parallelism with concurrency leads to "I'll spawn a thread per request" designs that exhaust memory and the scheduler.

### The .NET Thread Pool

Instead of unbounded `new Thread()`, .NET provides a **thread pool**: a pool of worker threads the runtime creates, reuses, and grows under demand. When you call `Task.Run`, `ThreadPool.QueueUserWorkItem`, or when ASP.NET Core accepts an HTTP connection, work is typically queued to the **ThreadPool**.

The pool maintains a **global queue** and, since .NET 4, **local queues per worker thread** for work-stealing efficiency. A small number of threads start at process launch; the pool **injects** new threads when queued work exists and threads appear busy — but injection is **throttled** (roughly two new threads per 500 ms in classic behavior) to avoid overshooting.

Worker threads are **general-purpose**. ASP.NET Core, `Timer` callbacks, `Task.Run`, and many library APIs all compete for the same pool. That shared resource is why blocking one thread on I/O (synchronous `File.ReadAllText`, `.Result` on an incomplete task) hurts unrelated work — there are fewer idle workers for everyone else.

### Context Switch Cost

When the OS **preempts** thread A to run thread B, it performs a **context switch**: save A's registers and stack pointer, restore B's state, update scheduling metadata, possibly flush CPU caches. Context switches cost on the order of **microseconds to tens of microseconds** — small per switch, enormous at scale.

If you have 10,000 threads runnable on 8 cores, the OS spends significant time switching instead of doing useful work. High **ready-thread count** increases latency variance (tail latency spikes). Production systems target **enough threads to keep CPUs busy without massive oversubscription** — for I/O-heavy apps, that often means *fewer* blocked threads, not more (async model).

### Threads vs Tasks (Preview)

A **`Task`** represents asynchronous work; it is **not** a thread. A task may run on a thread pool thread, may complete without ever running user code on a thread (cached result), or may resume on a continuation after I/O. **`Task` is an abstraction; thread is an execution resource.** You will dissect `Task` in `1.4.3`; for now, remember: scheduling a task does not equal creating a thread.

## Under the Hood

### Kernel Scheduling and Quantums

On Windows, the **scheduler** assigns each runnable thread a **quantum** of CPU time on a core. When the quantum expires, or a higher-priority thread becomes ready, or the thread blocks (syscall for I/O, lock wait), the kernel switches context. **Blocking syscalls** remove the thread from the runnable set — the core can run something else. **Spin-waiting** keeps the thread runnable and wastes a core.

Linux uses **CFS** (Completely Fair Scheduler) with different mechanics but the same lesson: runnable threads compete for cores; blocking frees the core for others.

### Thread Pool Injection Algorithm (Conceptual)

The CLR thread pool watches **throughput**: completed work items per interval vs queue depth. If work backs up and completion rate stalls, it adds threads — slowly. Sudden bursts (10,000 `Task.Run` calls) queue work first; injection catches up over hundreds of milliseconds. **`ThreadPool.SetMinThreads`** can raise the floor (used carefully in some server apps) so more threads exist before the throttle kicks in — a tuning knob, not a default fix.

**I/O completion ports** (Windows) and similar mechanisms integrate async I/O completions with the pool: when an async read finishes, a pool thread (or the completing thread, depending on version and API) runs the continuation. That bridges OS async I/O to managed callbacks — groundwork for `1.4.6`.

### Stack Memory and Process Limits

Each thread's stack consumes virtual address space. A 64-bit process has huge virtual space, but **commit charge** and kernel handles still matter. Thread-per-request servers with 10k concurrent connections and 10k blocked threads can hit **memory pressure** and **handle limits** long before CPU saturates. The thread pool reuses a bounded set of stacks.

### `Thread` vs `ThreadPool` vs `Task`

| Mechanism | Creates new OS thread? | Typical use |
|-----------|------------------------|-------------|
| `new Thread()` | Yes (dedicated) | Long-lived background worker, custom STA thread |
| `ThreadPool.QueueUserWorkItem` | Reuses pool | Low-level queue |
| `Task.Run` | Queues to pool | CPU offload (see `1.4.4`) |
| `async/await` | No thread while awaiting I/O | I/O-bound scalability |

## Syntax / Commands / API

```csharp
// Explicit thread — avoid in app code except special cases
var t = new Thread(() => Console.WriteLine(Thread.CurrentThread.ManagedThreadId));
t.IsBackground = true; // don't keep process alive alone
t.Start();

// Thread pool — prefer Task.Run wrapper in modern code
ThreadPool.QueueUserWorkItem(_ => DoWork());

// Inspect pool thread
Console.WriteLine(Thread.CurrentThread.IsThreadPoolThread); // true on pool workers

// Tuning (use with measurement, not by default)
ThreadPool.GetMinThreads(out int minWorker, out int minIo);
ThreadPool.SetMinThreads(minWorker: 100, minIo: 100);
```

**Useful diagnostics:** `dotnet-counters` (`System.Runtime` → `threadpool-thread-count`, `threadpool-queue-length`), Process Explorer thread count, Application Insights thread pool metrics.

## Examples

### Example 1: One Thread, Sequential Work

```csharp
for (int i = 0; i < 5; i++)
{
    Console.WriteLine($"Work {i} on thread {Thread.CurrentThread.ManagedThreadId}");
}
```

All iterations run on the same thread (often the main thread). No parallelism — total time is sum of work.

### Example 2: Thread Pool Workers

```csharp
var tasks = Enumerable.Range(0, 8).Select(i => Task.Run(() =>
{
    Thread.Sleep(100); // simulates CPU or blocking — bad for I/O lesson later
    return i;
})).ToArray();
Task.WaitAll(tasks);
```

`Task.Run` queues eight work items. The pool runs them on available workers — you may see fewer than eight distinct thread IDs if workers are reused sequentially. This is **parallelism** bounded by pool size and cores.

### Example 3: Oversubscription Symptom

Queuing 50,000 long-running `Task.Run` jobs fills the global queue. Completion time stretches not only because of work volume but because injection throttling and context switches add overhead. Measure before assuming "more threads = faster."

## Common Mistakes & Anti-patterns

**Thread per request.** Legacy ASP.NET and naive socket servers created a thread per connection. Under load, thousands of blocked threads exhaust memory and scheduler capacity. Modern ASP.NET Core uses the thread pool plus async I/O.

**Blocking pool threads.** Synchronous `HttpClient.GetStringAsync().Result` or `Thread.Sleep` on a pool thread during a request holds a worker that could serve other requests. Symptom: app hangs under moderate concurrency with low CPU — **thread pool starvation** (`1.4.10`).

**Confusing `Task` with thread.** Starting 100 tasks does not mean 100 threads; the pool may use ~`Environment.ProcessorCount` workers for CPU-bound work.

**Creating `new Thread()` for short work.** Higher overhead than pool; harder to reason about in ASP.NET's shared environment.

## Production & Real-World Notes

ASP.NET Core Kestrel accepts connections and dispatches to the thread pool. Default **MinThreads** is often low; some teams raise it after seeing queue latency under burst traffic — always validate with load tests.

Container limits (Kubernetes CPU quota) interact with thread pool sizing: fewer effective cores → fewer useful parallel CPU workers.

Long-running **background services** should use `IHostedService` with dedicated logic, not unbounded `Task.Run` on the shared pool without `TaskCreationOptions.LongRunning` (which creates a dedicated thread — use sparingly).

Monitor **thread pool queue length** and **active threads** in production APM. Spikes correlate with sync-over-async and blocking calls.

## Comparison / Trade-offs

**Dedicated thread vs pool:** Dedicated threads give isolation and predictable stack but cost memory and OS resources. Pool threads amortize creation cost but are a **shared scarce resource**.

**More threads vs async I/O:** For **I/O-bound** work, async frees threads during waits (`1.4.2`). For **CPU-bound** work, threads (via pool) parallelize up to core count; beyond that, diminishing returns.

**Process-based vs thread-based concurrency:** `dotnet` microservices scale **horizontally** (more pods); within one process, thread pool + async maximizes throughput per instance.

## Key Takeaways

- Threads are **OS resources** with stack cost and scheduling overhead — not free.
- The **thread pool** reuses workers; ASP.NET and `Task.Run` depend on it.
- **Context switches** and **oversubscription** hurt scalability at high thread counts.
- **`Task` ≠ thread** — tasks are scheduled onto pool threads when needed.
- Blocking pool threads is a primary cause of production **hangs** under load.
