## Why This Matters

Before Flexbox, before React, before your ASP.NET Core API returns JSON — the browser receives **bytes** and must decide how to interpret them. The first signals in an HTML file are not visual; they are contractual. The **DOCTYPE** tells the browser which parsing rules to apply. The **`charset`** meta tag tells it how to decode those bytes into characters. The **`lang`** attribute tells assistive technology and search engines which human language the page uses.

These three lines look like ceremony when you use a framework template, but they are the foundation of every page you ship. An absent or legacy DOCTYPE can push browsers into **quirks mode**, where CSS box model and layout behave like IE6-era bugs. Middle developers do not treat the skeleton as boilerplate to ignore; they know *why* each piece exists so they can debug encoding issues in production, fix Lighthouse accessibility warnings, and review PRs that break document structure.

Node `0.5` builds the language of the web from the ground up. You already understand HTTP request-response (`0.3.1`): the server sends bytes, the browser parses them. HTML is the format those bytes often take. This lesson is the first page of that format — the envelope before the letter.

## Core Concepts

### HTML Documents Are Trees of Elements

An HTML document is a hierarchical tree. The root is the `<html>` element. It has two required children in practice: `<head>` for machine-oriented metadata and `<body>` for content humans see. Every other tag nests inside one of those branches. Browsers do not execute your intentions; they **parse** text into a DOM tree according to the HTML specification. If your markup is malformed, the parser applies **error recovery rules** — it will often still render something, but not necessarily what you meant.

That forgiving parser is a double-edged sword. Your page may "look fine" in Chrome while being structurally wrong, hurting accessibility tools, SEO crawlers, and future you when CSS selectors fail. Starting with a correct skeleton reduces silent breakage.

### DOCTYPE: Switching the Parser Into Standards Mode

The **DOCTYPE** is not an HTML element. It is a **declaration** — a special token at the very first line of the file, before `<html>`. In HTML5 it is short:

```html
<!DOCTYPE html>
```

Historically, DOCTYPE strings were long references to DTD URLs that told browsers which version of HTML you claimed to follow. HTML5 simplified this to a single, case-insensitive token that means: "parse this document using the modern HTML living standard."

Browsers maintain two major parsing modes: **no-quirks mode** (standards mode) and **quirks mode** (legacy compatibility). Quirks mode emulates old IE layout bugs — for example, width calculations that ignore padding differently. A correct HTML5 DOCTYPE triggers standards mode in all current browsers. Omitting DOCTYPE or using ancient declarations can leave quirks mode active, so your carefully written CSS from `0.6.x` behaves inconsistently.

The DOCTYPE does not appear in the DOM tree you manipulate with JavaScript. DevTools will not show it as an element. It is consumed at parse time and discarded — but its effect on layout persists for the life of the page.

### The Root Element and the lang Attribute

The `<html>` element wraps the entire document. It should include a **`lang`** attribute with a valid **BCP 47** language tag:

```html
<html lang="en">
```

or

```html
<html lang="ru">
```

Screen readers use `lang` to select pronunciation rules and voice profiles. When a page is mostly Russian but contains an English quote, you mark the quote with a nested `lang="en"` on a `<span>` or `<blockquote>` — not by changing the root unless the whole page switches language.

Search engines use `lang` as a signal for regional results. Translation tools and hyphenation engines rely on it. Browsers may offer to translate the page when `lang` does not match the user's locale. None of that works reliably if you omit `lang` and leave the default ambiguous.

For multilingual sites, one document usually equals one primary language. hreflang links (`0.5.13`) handle alternates across URLs; `lang` handles the current document.

### head: Metadata the User Does Not See (Mostly)

The `<head>` element collects **metadata**: data *about* the document, not the main content. Typical children include:

- `<meta charset="UTF-8">` — character encoding
- `<title>` — document title (shown in tab and search results; `0.5.13`)
- `<meta name="viewport" ...>` — mobile layout (`0.5.x` responsive topics)
- `<link rel="stylesheet" href="...">` — CSS
- `<script>` — JavaScript (prefer defer/async patterns in later lessons)

The head is parsed before the body renders. Critical metadata like charset should appear **early** so the browser does not mis-decode the first kilobytes of the file while waiting for a charset declaration buried at the bottom.

### UTF-8 and meta charset

**UTF-8** encodes virtually every character you will ever need — Latin, Cyrillic, CJK, emoji — in a variable-length byte sequence. It is the **default encoding of the web**. Your editor should save files as UTF-8; your server should send `Content-Type: text/html; charset=utf-8` in HTTP headers (`0.3.3`).

The HTML declaration mirrors that:

```html
<meta charset="UTF-8">
```

In HTML5 this form is preferred over older `<meta http-equiv="Content-Type" content="text/html; charset=UTF-8">`. Place it within the first **1024 bytes** of the document, ideally as the first child of `<head>`.

If charset is wrong or missing, the browser may guess — often incorrectly for Cyrillic or mixed-language content. You see replacement characters () or garbled text. Fixing encoding at the source (file save + meta + HTTP header aligned) beats CSS hacks every time.

### body: Where Visible Content Lives

Everything the user sees — headings, paragraphs, images, forms — belongs in `<body>`. The separation is semantic: head configures, body presents. Scripts can live in head or body; stylesheets usually link from head. Visible headings never belong in head.

## Under the Hood

When Kestrel or nginx serves `index.html`, the response body is a byte stream. The browser's HTML parser reads bytes, determines encoding (HTTP header wins over meta if both present — precedence rules matter in edge cases), converts to Unicode code points, tokenizes tags and text, builds the DOM, and only then applies CSS and runs scripts.

The DOCTYPE token is handled in a **pre-parse** step. If it matches the HTML5 trigger, the document is marked no-quirks. Then the tree builder creates an `html` element, attaches `head` and `body` as parsing continues. Meta charset, when encountered, can **restart** parsing from the beginning if the browser initially guessed wrong — another reason to put charset first in head.

View Source in DevTools shows your authored HTML; the **Elements** panel shows the **parsed** DOM after browser corrections (auto-inserted `<tbody>`, implied `<head>`, etc.). Comparing them teaches you what the parser fixed silently.

## Syntax / Commands / API

Minimal HTML5 document template:

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <title>My First Page</title>
</head>
<body>
  <p>Hello, world.</p>
</body>
</html>
```

Common `lang` values: `en`, `en-US`, `ru`, `de`, `zh-Hans`. Use the most specific tag you can justify.

Validate structure at [W3C Markup Validation Service](https://validator.w3.org/) — full workflow in `0.5.16`.

## Examples

### Example 1: Russian content with correct encoding

You are building a landing page for a Moscow team. The file is saved UTF-8 without BOM; the server sends UTF-8. The skeleton:

```html
<!DOCTYPE html>
<html lang="ru">
<head>
  <meta charset="UTF-8">
  <title>Команда разработки</title>
</head>
<body>
  <p>Добро пожаловать на наш сайт.</p>
</body>
</html>
```

If you copy this text into a file but save as Windows-1251, or omit charset while the server declares ISO-8859-1, Cyrillic breaks even though the HTML looks "correct" in your editor. Always align **file encoding**, **meta charset**, and **HTTP Content-Type**.

### Example 2: English page with a Russian phrase

Root stays `lang="en"`. The Russian name gets its own marker:

```html
<p>The project lead is <span lang="ru">Иван Петров</span>.</p>
```

Screen readers switch pronunciation for the span only.

## Common Mistakes & Anti-patterns

**Omitting DOCTYPE** — Page may enter quirks mode; subtle CSS bugs follow. Always include `<!DOCTYPE html>`.

**Charset declared too late** — Large `<head>` with many scripts before charset forces a decode guess. Put charset first.

**Using `<meta charset>` in body** — Invalid location; parser may ignore or relocate unpredictably.

**Wrong lang** — `lang="en"` on an all-Russian page hurts screen reader users and SEO. Match primary content language.

**Relying on `<html>` without DOCTYPE** — Some tutorials skip DOCTYPE in snippets; do not ship that to production.

## Production & Real-World Notes

Framework templates (Create React App, Vite, Razor `_Layout.cshtml`) inject DOCTYPE, charset, and lang for you. When you customize layouts, preserve them. In ASP.NET Core, set `<html lang="...">` from request culture if you localize. Ensure middleware sets charset on HTML responses.

CI pipelines sometimes run HTML validators on static exports. Lighthouse flags missing `lang` and document title. Encoding bugs often appear only in staging with real data containing non-ASCII names — test with Cyrillic or emoji early.

---

## Comparison / Trade-offs

HTML4 XHTML-style DOCTYPE with PUBLIC identifiers still validates in old docs; for new work HTML5 DOCTYPE is the only sensible choice. UTF-8 vs UTF-16: UTF-8 dominates the web — smaller for ASCII-heavy markup, compatible everywhere. Declaring charset only in HTTP headers works if you control the server exclusively; keeping `<meta charset>` helps local file opens and static hosting that omits charset in headers.

## Key Takeaways

- DOCTYPE is a parser switch, not a visible element; HTML5 uses `<!DOCTYPE html>`.
- `lang` on `<html>` helps screen readers, search engines, and translation tools.
- UTF-8 is the web default; declare it early with `<meta charset="UTF-8">`.
- Align file encoding, meta tag, and HTTP `Content-Type` charset.
- `<head>` configures; `<body>` displays — keep structure clean before styling.
