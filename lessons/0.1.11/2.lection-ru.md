# Detached HEAD: что это и как выйти

> Roadmap: `0.1.11` · Узел: `0.1` — Git: основы · Глубина: **глубоко**

## Цели урока

После этой лекции ты сможешь:

- Определить **HEAD** и объяснить **detached HEAD** на уровне refs.
- Описать, как **попадают** в detached HEAD (checkout SHA, tag, remote branch без tracking).
- Объяснить, почему **новые commits** без branch становятся **unreachable** при уходе.
- Сохранить работу через **`git switch -c`** или **`git checkout -b`**.
- Вернуться на branch: **`git switch main`**.
- Восстановить «потерянные» commits через **`git reflog`**.

---

## Зачем это нужно

Ты выполняешь команду, которая кажется безобидной:

```bash
git checkout v1.0.0
# или
git checkout abc1234
```

Git отвечает предупреждением, которое многие пролистывают:

```
You are in 'detached HEAD' state...
```

Звучит тревожно — кто-то думает, что repository повреждён. Другие игнорируют warning, делают несколько commits, правя что-то в старом release, возвращаются на `main` и только тогда обнаруживают: работы нет в `git log`. Обе реакции неверны.

**Detached HEAD — не ошибка.** Это преднамеренное, валидное состояние, в которое Git переходит, когда ты checkout'ишь конкретный commit, а не имя ветки. Inspectors читают старый код. Release engineers проверяют, что ушло в tag. **`git bisect`** ищет регрессии, временно detach'ая HEAD. CI-системы большую часть времени в detached HEAD — они checkout'ят конкретный SHA, а не local branch разработчика. Опасность начинается, когда ты **commit'ишь новую работу** без привязки к branch и **уходишь** из detached HEAD, будто ничего не было.

Понимание HEAD на plumbing-уровне — что лежит в `.git/HEAD`, как refs держат commits alive, почему dangling commits переживают в reflog — отделяет тех, кто боится Git, от тех, кто уверенно navigates history. Урок глубокий, потому что detached HEAD затрагивает почти все core-концепции Git: refs, commits, reachability, garbage collection, recovery.

Представь аналогию из библиотеки. **Branch** — закладка в книге с именем на обложке: «глава 5, мы здесь». Когда ты читаешь дальше и добавляешь заметки, закладка двигается вместе с тобой. **Detached HEAD** — ты открыл книгу на странице 87 по номеру, но закладки с именем нет. Ты можешь читать, можешь даже написать на полях — но если закрыть книгу и открыть по закладке «глава 5», твои записи на странице 87 нигде не числятся. В Git «записи на полях» — это commits; «закладка с именем» — branch ref.

## Основные концепции

### Что такое HEAD?

**HEAD** отвечает на один вопрос: *где я работаю прямо сейчас?* Это указатель на текущий checkout context. В нормальном случае — **attached HEAD** — указатель не смотрит напрямую на commit. Он смотрит на **branch ref**, а branch ref — на commit:

```
HEAD → refs/heads/main → commit C
```

Ты «на ветке `main`». Новый commit двигает ref `main` на новый SHA. HEAD по-прежнему указывает на `refs/heads/main`, который теперь на новом tip. Имя ветки двигается вместе с работой. На этом контракте держится ежедневная работа: commits принадлежат ветке, на которой ты думаешь, что сидишь.

Стоит подчеркнуть: HEAD — не branch. HEAD — указатель **на** branch (в attached состоянии) или **на** commit (в detached). Когда junior говорит «я на HEAD», технически это всегда true, но бессмысленно для диагностики. Полезный вопрос: «HEAD attached к какому ref?» — ответ `refs/heads/main` или «detached at abc1234» сразу проясняет ситуацию.

### Detached HEAD — определение

**Detached HEAD** ломает эту indirection. HEAD указывает **напрямую на commit SHA**, без имени ветки посередине:

```
Normal (attached):   HEAD → refs/heads/main → commit C
Detached:            HEAD → commit abc1234   (без имени ветки)
```

Нет записи `refs/heads/*`, для которой `abc1234` — tip. Ты всё ещё *на* валидном commit — working tree соответствует snapshot — но не *на* branch. Новые commits из этого состояния продлевают цепочку, о которой знает только HEAD:

```
Detached с новой работой:

  main → ... → C
                \
                 D → E (HEAD)    ← D и E существуют, но ни один branch сюда не указывает
```

Commit `C` — по-прежнему tip `main`. Commits `D` и `E` — реальные objects в `.git/objects`, но после `git switch main` HEAD возвращается к `refs/heads/main → C`. На `D` и `E` больше никто не ссылается. Они становятся **dangling commits** — reachable только через reflog, пока garbage collection не заберёт. С точки зрения `git log main` твой «Emergency fix» не случился.

Сформулируем опасность прямо: **branches — это labels, которые держат commit chains alive.** Detached HEAD — чтение history без label. Commits, которые ты добавляешь, пишутся в object database, но не имеют permanent name, пока ты его не создашь.

Git показывает warning не потому, что что-то сломалось, а потому что **контракт изменился**. В attached состоянии новые commits автоматически продлевают branch. В detached — нет. Warning — приглашение осознанно выбрать: только смотрю (без commit) или создам branch перед работой. Игнорировать его — значит принять риск silent loss.

### Как попадают в detached HEAD

Несколько повседневных команд ставят тебя в это состояние; узнавать их — значит не удивляться.

Checkout raw commit SHA — `git checkout abc1234` или `git switch --detach abc1234` — всегда detach'ит. Checkout **tag** — то же: tags указывают на фиксированный commit и не двигаются при commit'ах: `git checkout v1.0.0` detach'ит на target tag. Перемещение относительно tip branch без attach: `git switch --detach main~3` checkout'ит commit три ancestor'а назад от tip `main`, но не создаёт и не двигает `main`.

Checkout **remote-tracking branch** напрямую — `git checkout origin/main` или `git switch origin/feature` без local tracking branch — часто detach'ит. Привычка `git checkout origin/feature` «просто посмотреть remote version» — частая ловушка: detached на remote tip вместо local branch с tracking.

**`git bisect`** намеренно входит в detached HEAD, пока ты тестируешь средние commits между known good и bad. Fix'ы во время bisect commit'ить не нужно — отмечаешь good/bad, bisect двигает HEAD сам.

**Не detach'ит:** `git switch feature-branch`, когда `feature-branch` — local branch name. HEAD → `refs/heads/feature-branch`.

Ещё один частый путь — **GitHub Pull Request checkout** в IDE или CLI. Когда ты checkout'ишь PR merge commit или конкретный SHA из UI, ты часто оказываешься detached. Это нормально для review, но commit'ить правки прямо оттуда без создания branch — та же ловушка. Правильный flow: checkout PR → если нужны правки → `git switch -c review-fixes` → commit → push branch.

## Под капотом

### Файл HEAD

Git хранит HEAD как маленький текстовый файл `.git/HEAD`. Содержимое сразу говорит: attached или detached.

Attached к `main`:

```
ref: refs/heads/main
```

Git resolve'ит HEAD, читая `refs/heads/main` с forty-character SHA — это текущий commit.

Detached:

```
abc1234567890abcdef1234567890abcdef123456
```

Hash **буквально** в HEAD — без префикса `ref:`. Git не следует за branch; HEAD *есть* commit identifier. Проверка в любой момент:

```bash
cat .git/HEAD
git status    # "HEAD detached at abc1234" или "HEAD detached at v1.0.0"
```

`git status` переводит plumbing в человеческий язык. Чтение `.git/HEAD` напрямую строит mental model, который снимает mystique с warnings.

Когда Git checkout'ит commit, он обновляет три связанных артефакта: файл HEAD, working tree (файлы в проекте) и index (staging area). В detached состоянии HEAD указывает на старый snapshot — working tree отражает код **на тот момент**. Если ты правишь файлы и commit'ишь, новый commit строится **поверх** detached tip, но branch refs не двигаются. Именно поэтому `git log` на `main` не показывает твои правки: log следует по branch ref, а не по «где я когда-то был detached».

### Reachability и почему commits «исчезают»

Git хранит objects — commits, trees, blobs — пока они **reachable** от какого-то ref. Основные ref namespaces, которые anchor'ят reachability:

```
refs/heads/*     branches
refs/tags/*      tags (обычно immutable указатели на commits)
refs/remotes/*   remote-tracking branches
HEAD             пока ты checkout'нут
reflog entries   safety net с expiration
```

Commit reachable, если от ref можно дойти по parent pointers до него. Пока ты в detached HEAD на commit `E`, `E` и ancestors reachable — HEAD сам считается. В момент `git switch main` HEAD становится `ref: refs/heads/main → C`. Commits `D` и `E` больше не на пути от ref — если reflog их не упоминает.

**Reflog** (`refs/logs/HEAD`) записывает каждое движение HEAD с timestamp и описанием:

```bash
git reflog
# e123456 HEAD@{0}: checkout: moving from abc1234 to main
# f789012 HEAD@{1}: commit: hotfix on detached head
# abc1234 HEAD@{2}: checkout: moving from main to v1.0.0
```

«Потерянные» commits recoverable ~**девяносто дней** по умолчанию (`gc.reflogExpire`). Воссоздай branch на lost tip:

```bash
git switch -c recovered-work f789012
# старый синтаксис:
git branch recovered-work f789012 && git switch recovered-work
```

Теперь `refs/heads/recovered-work → f789012`, вся цепочка до `E` permanently reachable, пока не удалишь branch.

### Что происходит при `git switch main`

Пошагово — почему никакой prompt не спасает автоматически.

Сначала Git проверяет uncommitted changes. Working tree или index отличаются от текущего commit — switch может отказать, stash'нуть или перенести changes. При clean tree Git пишет `ref: refs/heads/main` в `.git/HEAD`. Читает `refs/heads/main` → commit `C`, checkout'ит tree `C` в working directory. Commits `D` и `E`, если ты делал их detached без branch, не имеют named ref. Они — objects и reflog entries.

Диалога «Ты оставляешь commits — создать branch?» нет. Git предполагает, что ты знаешь, что делаешь. Этот design choice — причина support tickets; **`git switch -c branch-name`** до первого commit off tag — professional habit.

### Tags — не branches

Частая путаница: tags как «медленные branches». **Branch** — movable ref, новые commits двигают его. **Tag** (особенно annotated release tag) — fixed bookmark на один commit. Checkout `v2.0.0` — snapshot release в detached state. Commit'ишь — tag остаётся на старом commit, новая работа — orphan chain. Release hotfixes начинают с **`git switch -c hotfix/2.0.1 v2.0.0`**: branch at tag до правок файлов.

Annotated tag хранит metadata (author, message, иногда GPG signature) и указывает на commit object. Lightweight tag — просто ref file с SHA. В обоих случаях tag **не двигается** при новых commits с detached HEAD. Production release process почти всегда: tag на `main` → deploy → hotfix branch от tag, не commit поверх tag in place.

### `git bisect` и временный detached state

**`git bisect start`** переводит repo в режим поиска commit'а, который внёс regression. Git checkout'ит middle commit между known good и known bad — ты в detached HEAD. Твоя задача: запустить tests, отметить `git bisect good` или `git bisect bad`, Git checkout'ит следующий middle commit. Цикл повторяется, пока не найдётся первый bad commit.

Во время bisect **не commit'ь fix**. Ты меняешь код для test, но результат test — только good/bad label. Fix делается **после** `git bisect reset`, на branch, созданном от найденного commit. Commit fix во время bisect создаёт orphan chain и ломает bisect state — классическая ошибка junior'ов при первом regression hunt.

### Branch сохраняет линию

Detached на commit `E` с работой, которую нужно keep:

```bash
git switch -c fix-from-v1
```

Git создаёт `refs/heads/fix-from-v1` на текущем HEAD (`E`) и attach'ит HEAD к этому ref. Commits `D` и `E` branch-reachable. Push, PR, merge — как обычно. Старый синтаксис `git checkout -b fix-from-v1` — то же; modern Git предпочитает `switch -c`.

```mermaid
gitGraph
  commit id: "A"
  commit id: "B"
  commit id: "C"
  branch main
  checkout main
  commit id: "D"
  commit id: "E"
```

Если `main` на `C`, а HEAD на `E` без branch name, `D` и `E` висят сбоку. `fix-from-v1` на `E` даёт цепочке home.

### Garbage collection и пределы recovery

Dangling commits не удаляются мгновенно. **Garbage collector** Git (`git gc`) запускается периодически или по demand, обходит все reachable objects от refs и удаляет остальное. До GC orphaned commits от detached session занимают disk space, но intact. Reflog entries продлевают окно: даже когда commit unreachable, reflog может хранить SHA с message вроде `commit: Emergency fix on old tag`.

Recovery fails, когда прошло слишком много времени, кто-то запустил `git gc --prune=now`, или работа была на машине без reflog — CI runner без push. Поэтому fix для detached HEAD **процедурный**: branch до recovery. Ожидание после switch away превращает секундный `git switch -c` в archaeology без guarantee.

Inspect без prune:

```bash
git fsck --unreachable
git reflog expire --dry-run --all
```

Редко нужны day to day, но завершают mental model: detached commits — real objects с тем же lifecycle, что всё в `.git/objects`.

### Symbolic HEAD и checkout commands

Daily detached/attached distinction идёт через `.git/HEAD`. Checkout branch — Git пишет `ref: refs/heads/branch-name`. Detach — raw SHA. Porcelain commands (`git switch`, `git checkout`) переводят user intent в эти plumbing writes. Когда IDE checkout'ит «detached at origin/main» — underneath тот же literal SHA в HEAD, те же rules про branching.

## Синтаксис

Detached intentionally — inspect history:

```bash
git switch --detach v1.0.0
git checkout abc1234
git switch --detach HEAD~2
```

Где ты:

```bash
git status
git branch
git log --oneline -1
cat .git/HEAD
```

Save work — назвать текущий tip:

```bash
git switch -c my-fix
git switch -c my-fix abc1234
```

Return без save (безопасно, если только читал):

```bash
git switch main
```

Recovery после ухода с detached commits:

```bash
git reflog
git switch -c rescue HEAD@{1}
git branch rescue f789012
```

## Примеры

### Пример 1: Безопасный inspect без новых commits

Прочитать `README.md` пять commits назад:

```bash
git switch main
git switch --detach HEAD~5
cat README.md
git switch main            # новых commits не было — нечего терять
```

Branch не нужен — новых commits не было. Detached HEAD как временная time machine.

### Пример 2: Классическая ошибка

```bash
git checkout v2.0.0
echo "fix" >> app.txt
git commit -am "Emergency fix on old tag"   # commit E в detached
git checkout main                           # HEAD оставил E
git log --oneline                           # fix не на main
```

Fix в object database, но не на branch, который виден. Recovery:

```bash
git reflog
git switch -c tag-hotfix <SHA>
```

Без reflog — fresh clone, не commit'ил — recovery может быть невозможен. Detached commits — urgent, пока не branch.

### Пример 3: Правильный hotfix от tag

Production на `v2.0.0`. Bug patch'ится с exact snapshot:

```bash
git fetch origin --tags
git switch -c hotfix/from-v2 v2.0.0    # branch ДО правок
git commit -am "Fix null reference in payment handler"
git push -u origin hotfix/from-v2
```

Branch сохраняет commit chain с первой правки. Review, CI, merge — как feature branch.

### Пример 4: Recovery через reflog step-by-step

Ты уже переключился на `main` и не видишь commit «Emergency fix». Не паникуй — последовательность такая:

```bash
git reflog | head -20
# ищешь строку с "Emergency fix" или checkout from tag
# допустим SHA f789012

git show f789012 --stat    # убедиться, что это нужный commit
git switch -c tag-hotfix f789012
git log --oneline -3       # fix снова виден
git push -u origin tag-hotfix
```

Если commit делался давно и reflog уже rotated — recovery может не сработать. Это не баг Git, а напоминание: branch cost nothing at commit time, reflog archaeology costs nerves and time.

### Пример 5: Detached во время code review

Review PR: checkout SHA merge commit для локального теста. Ты detached — это OK. Запустил тесты, нашёл баг, написал fix в working tree. **Перед commit:** `git switch -c pr-123-fix`. Теперь fix на branch, можно push и дополнить PR или открыть новый. Commit без branch → switch back to `main` → fix исчез из видимой history.

## Частые ошибки

Игнор warning и commit — корень «lost work». Switch на `main` ощущается как закрытие tab — но Git не держит orphan commits в normal logs. Tags как branches — те же orphan chains; tags не двигаются. `git checkout origin/feature` для inspect — detached; `git switch feature` (после `git switch -c feature origin/feature` один раз) — attach.

Один commit detached и ожидание, что merge `main` подхватит — не подхватит: commit не на branch, reachable от `main`. Bisect + inline fix + commit — fix на branch после bisect, не во время.

## Production notes

Release engineering branch'ит от tags: `git switch -c release/2.0.1 v2.0.0`. CI checkout'ит SHA или merge refs — detached default, не аномалия. Документируй для команды: **detached ≠ broken repo**; **detached + unbranch'd commits + checkout elsewhere = silent data loss**.

«Потерял commit» в support → **`git reflog`** в большинстве случаев, если в окне retention и не было aggressive GC. Bisect: не commit'ить fix'ы; good/bad, finish bisect, branch от identified commit.

Когда коллега говорит «сижу на origin/feature», уточни: создал ли local tracking branch. Если был `git checkout origin/feature` напрямую — detached на remote tip; commits не обновят `origin/feature` при push. Fix: `git switch -c feature origin/feature`. Малая смена habit — большое снижение detached HEAD incidents.

В onboarding полезно один раз открыть `.git/HEAD` в attached и detached состоянии и сравнить содержимое. Это занимает минуту и навсегда снимает ощущение, что Git «сломался». Параллельно покажи `git branch -a` и объясни разницу между `refs/heads/feature`, `refs/remotes/origin/feature` и detached HEAD на SHA remote tip.

## Сравнение подходов

**Browse old code** — detached без commits, read, switch back. **Fix от tag/old commit** — `-c` branch first. **Throwaway experiment** — detached OK если не commit или reset away. **Regression** — bisect owns navigation; permanent history после bisect.

Сводя в одну фразу для запоминания: detached HEAD — это checkout конкретного snapshot без имени ветки; безопасен для чтения, опасен для commit без `git switch -c`. Когда сомневаешься — `git status` и `cat .git/HEAD` за две секунды покажут, attached ты или нет. Middle-разработчик не боится warning — он знает, что делать до первого commit и после.

---

## Шпаргалка

Attached: `HEAD → branch ref → commit`. Detached: `HEAD → commit SHA`. Save: `git switch -c name`. Back (no new commits): `git switch main`. Recover: `git reflog` → `git switch -c rescue <SHA>`.

---

## Главное

- **HEAD** — текущая позиция; обычно **branch ref**, не commit напрямую.
- **Detached HEAD** — HEAD → **commit SHA**; часто после tags или raw SHA.
- Новые commits без branch **unreachable** при уходе — attach через **`git switch -c`**.
- **Tags не advance** — не branches. Branch before commit off tag.
- **`git reflog`** — recovery orphan commits ~90 дней locally.
- Detached **нормален для inspect и CI**; **опасен для development без branch**.
