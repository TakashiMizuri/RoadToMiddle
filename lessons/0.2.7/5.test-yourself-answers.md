## Answers

### 1. [Recall]
**Answer:** **Upstream** — default remote branch, с которым связана локальная ветка; обычно `origin/<same-name>`.

**Why:** Позволяет короткие `git pull`/`git push` и ahead/behind в status.

---

### 2. [Recall]
**Answer:** Пушит commits **и** записывает tracking config (`branch.<name>.remote` + `merge`) — связывает локальную ветку с remote branch.

**Why:** Без `-u` нужно каждый раз указывать ref или настроить `git branch -u`.

---

### 3. [Understanding]
**Answer:** Обычный pull делает **merge** (возможен merge commit). `pull --rebase` **перенакатывает** локальные commits поверх обновлённого upstream — history линейнее, без merge commit от этой операции.

**Why:** Оба могут вызвать конфликты; разница в форме графа (`0.2.2` vs `0.2.3`).

---

### 4. [Understanding]
**Answer:** **`git push`** (если upstream настроен) или `git push origin feature/api`.

**Why:** Ahead = локальные commits не на remote.

---

### 5. [Understanding]
**Answer:** Rebase **переписывает** commits (новые SHA). Remote всё ещё указывает на старые; push не fast-forward. Нужен **`git push --force-with-lease`** на своей feature (если policy разрешает).

**Why:** Golden rule public history (`0.2.4`, `0.1.10`) — force только на ветках, где ты единственный потребитель.

---

### 6. [Application]
**Answer:** `git branch -u origin/feature/billing` или `git push -u origin feature/billing`.

**Why:** Обе устанавливают upstream; второй также убедится, что remote актуален.

---

### 7. [Application]
**Answer:**
```bash
git fetch origin
git rebase origin/main
# resolve conflicts, git rebase --continue if needed
git push --force-with-lease   # если ветка уже была на remote
```

**Why:** Rebase feature на tip main даёт linear continuation без merge commit от main.

---

### 8. [Analysis]
**Answer:** **C** — неверно. Rebase при pull **может** конфликтовать на каждом replayed commit.

**Why:** A, B, D верны.

---

### 9. [Analysis]
**Answer:** При `pull.rebase true` каждый `git pull` **rebases** вместо merge — merge commits не создаются. Желаемо на **feature branches** для чистого log; неожиданно, если человек ожидал merge commits на shared ветках.

**Why:** Team policy должна быть явной в CONTRIBUTING.md.

---

### 10. [Understanding]
**Answer:** Файл **`.git/config`**, секция `[branch "feature/login"]`.

**Why:** Локальная конфигурация tracking, не на сервере.

---

### 11. [Application]
**Answer:** (1) **Интегрировать** remote изменения — обычно `git pull --rebase` или merge. (2) **Опубликовать** результат — `git push` (возможно `--force-with-lease` после rebase).

**Why:** Diverged = обе стороны добавили уникальные commits.

---

### 12. [Understanding]
**Answer:** Личная feature — переписывание затрагивает в основном тебя; на shared **main** другие и CI уже базируются на старых SHA — rewrite ломает clones и PR.

**Why:** Правило «не переписывать то, что другие стянули» (`0.2.4`).

---

### 13. [Recall]
**Answer:** `git rev-parse --abbrev-ref @{upstream}` или смотреть `git branch -vv`.

**Why:** `@{upstream}` — синтаксис для upstream текущей ветки.

---

### 14. [Application]
**Answer:** **`git rebase --abort`**

**Why:** Откатывает rebase, возвращая pre-rebase state.
