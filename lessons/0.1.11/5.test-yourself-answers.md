## Answers

### 1. [Recall]
**Answer:** HEAD points to a **branch reference** (e.g. `refs/heads/main`), which in turn points to a commit.

---

### 2. [Recall]
**Answer:** A **raw commit SHA** (40-character hex), **not** a `ref: refs/heads/...` line.

---

### 3. [Understanding]
**Answer:** HEAD points **directly to a commit**; it is **not** pointing to a **branch ref** that moves with new commits.

**Why:** Without branch ref, new commits have no persistent name.

---

### 4. [Understanding]
**Answer:** New commits exist only reachable from detached HEAD. Moving HEAD to `main` removes the only pointer (besides reflog). No branch or tag references them → **unreachable/dangling**.

**Why:** Git garbage collection removes unreachable objects; reflog retains SHAs temporarily.

---

### 5. [Understanding]
**Answer:** Reflog records previous HEAD positions including detached commits. You find the commit SHA and **`git switch -c branch SHA`** to restore.

**Why:** Reflog is local safety net independent of branch refs.

---

### 6. [Application]
**Answer:** **`git switch -c hotfix/3.1.1`** (or meaningful branch name) **before** committing.

**Why:** Attaches future commits to a branch ref from the start.

---

### 7. [Application]
**Answer:** **`git switch -c experiment`** (while still at tip with both commits) — creates branch at current HEAD including both commits.

**Alternative:** `git branch experiment && git switch experiment`

---

### 8. [Application]
**Answer:** Example:
```bash
git switch --detach HEAD~5
# read file
git switch main   # or previous branch name
```
No commits → nothing to lose.

---

### 9. [Analysis]
**Answer:** **False.** Detached HEAD is **valid and intentional** for inspecting history, tags, bisect. Not corruption.

---

### 10. [Analysis]
**Answer:** **A and C** cause detached HEAD. **B and D** attach HEAD to local branch refs.

---

### 11. [Analysis]
**Answer:** **`origin/main`** is a **remote-tracking ref**, not a local branch. Checkout/switch to it checks out the commit but may leave HEAD detached unless you create a local branch tracking it (e.g. `git switch -c main --track origin/main`).

**Common mistake:** Assuming remote-tracking refs behave like local branches for HEAD attachment.

---

### 12. [Understanding]
**Answer:**
- **No new commits:** Switching back only changes view — all commits still on branches; **nothing lost**.
- **With new commits:** Those commits lack branch ref after switch → **orphaned** unless recovered.

---

### 13. [Application]
**Answer:**
```bash
git reflog
# locate "lost fix" commit SHA
git switch -c recovered-fix <SHA>
# verify git log
# optional: git push -u origin recovered-fix
```

---

### 14. [Recall]
**Answer:** Any two: **`git bisect`**, inspecting **tags**, **CI checkout of SHAs**, **`git switch --detach`** for historical inspection.

---

### 15. [Analysis]
**Answer:** Second approach: first commit on detached tag **without branch** — easy to lose line when switching away. First approach creates **branch immediately** — commits always reachable.

**Why:** `-c` establishes `refs/heads/hotfix` before any work.

## Model answers for practical sections

### P1. HEAD file
**Answer:**
- `ref: refs/heads/develop` → **Not detached** (attached to branch)
- Raw hex SHA → **Detached**

---

### P2. Diagram
**Answer:** (Accept equivalent ASCII)

```
main:  A --- B --- C  (main points to C)

detached path:
        B --- D  (HEAD was here before switch)
```

After `git switch main` without `-c`: **`main` reaches A, B, C** — **not D**. B still reachable via `main`; **D orphaned** unless branch created or reflog recovery.
