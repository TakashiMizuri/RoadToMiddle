# Exercises: GC Modes — Workstation vs Server

> Complete after passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Inspect runtime GC mode
**Goal:** Console app prints `GCSettings.IsServerGC`, `LatencyMode`, and `GC.GetGCMemoryInfo().HeapCount`.
**Steps:** Create project; run Debug; publish and run published exe; compare.
**Success criteria:** Document which settings differ between F5 and publish (if any).

### E2. Toggle ServerGarbageCollection
**Goal:** Flip `<ServerGarbageCollection>` in csproj; rebuild publish; verify output changes.
**Success criteria:** Screenshot or log showing both true/false states.

## Medium

### M1. Load smoke test
**Goal:** Allocate ~100MB short-lived objects in loop under `ServerGarbageCollection=true` vs `false` (same machine).
**Steps:** Use `Stopwatch` + `GC.CollectionCount(2)`; optional `dotnet-counters` on PID.
**Success criteria:** Table: Gen2 count, elapsed, `IsServerGC` — one paragraph interpretation (not benchmark gospel).

### M2. runtimeconfig override
**Goal:** Add `runtimeconfig.template.json` with `"System.GC.Server": false` while csproj has true — observe precedence.
**Success criteria:** Explain effective mode after testing.

## Hard (stretch)

### H1. Container CPU limit experiment
**Goal:** Run same API in Docker with `cpus: "0.25"` vs `"2.0"`; compare latency under wrk/k6 with server GC.
**Success criteria:** Hypothesis + whether data supported it.

## Submission checklist

- [ ] At least one `IsServerGC` log from published artifact
- [ ] One csproj or runtimeconfig change documented
- [ ] Short note on workstation vs server for *your* machine
