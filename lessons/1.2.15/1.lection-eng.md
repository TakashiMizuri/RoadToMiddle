# readonly struct and ref struct Limitations

> Roadmap: `1.2.15` · Node: `1.2` — C# OOP · Depth: понимание

## Learning Objectives

After this lesson you will be able to:

- Explain what **`readonly struct`** guarantees and how the compiler enforces immutability.
- Describe **`ref struct`** stack-only restrictions and why they exist.
- Use **`Span<T>`** and **`ReadOnlySpan<T>`** safely as `ref struct` buffer views.
- Apply **`ref`**, **`in`**, **`out`**, and **`scoped`** parameters to avoid copies (`1.2.14`).
- List what **ref struct cannot do** (boxing, async, iterator fields, regular class fields).
- Recognize when **readonly record struct** (`1.2.13`) is the right API shape.

---

## Why This Matters

`1.2.14` showed structs copy on assignment and confuse developers when mutated through copies. **`readonly struct`** answers: "this value must not change after construction" — the compiler blocks instance mutations, enabling safer APIs and optimizations.

**`ref struct`** goes further: types like **`Span<T>`** must live only on the **stack** — they are views into memory (arrays, stackalloc buffers) without heap allocation. Modern high-performance .NET (`System.Text.Json`, `Kestrel` pipelines, `Memory<T>` slicing) relies on `Span<T>`. Violate ref struct rules and the compiler errors — preventing security holes (use-after-free style bugs if spans escaped to heap).

You will not implement custom `ref struct` daily, but you **will** call APIs taking `ReadOnlySpan<char>`, pass `stackalloc` buffers, and need to know why you cannot store `Span<T>` in a class field or `await` with span locals in older C# rules.

## Core Concepts

### readonly struct

Mark a struct **`readonly`** to promise instance state does not change after construction:

```csharp
public readonly struct Temperature
{
    public double Celsius { get; }
    public Temperature(double celsius) => Celsius = celsius;

    public Temperature ToFahrenheit()
        => new Temperature(Celsius * 9 / 5 + 32);
}
```

Rules:

- All **instance fields** must be `readonly`.
- Instance methods cannot assign to fields (except `init` in constructors).
- Compiler may avoid defensive copies when calling methods on readonly struct instances.

**`readonly record struct`** (C# 10) combines record synthesized equality with immutability:

```csharp
public readonly record struct Point(int X, int Y);
```

Mutating APIs return **new** instances (`with` or constructor) instead of changing `this`.

### Defensive Copies (Why readonly Helps)

For non-readonly struct, calling an instance method on a **readonly field** may copy the struct so the method cannot mutate the copy back into the field. **`readonly struct`** reduces these hidden copies — performance and clarity win.

### ref struct: Stack-Only Types

**`ref struct`** types cannot be:

- Boxed to `object` or `interface`
- Stored as instance fields of regular **classes** or **structs** (with narrow exceptions for `ref` fields in C# 11+)
- Captured into **async** or **iterator** state machines (historically; C# 13+ relaxes some `ref struct` in async with restrictions)
- Used as type parameters in generic classes unless type parameter is also `ref struct` (C# 12 `allows ref struct`)

```csharp
public ref struct StackOnlyBuffer
{
    public Span<byte> Data;
}
```

**Purpose:** represent **by-ref views** tied to stack lifetime — no GC heap promotion of the view itself.

### Span<T> and ReadOnlySpan<T>

BCL's flagship **ref struct**:

```csharp
Span<int> slice = array.AsSpan(2, 5);
ReadOnlySpan<char> text = "hello".AsSpan();
```

`Span<T>` is a **window** over contiguous memory (array, stackalloc, native pointer via `unsafe`). Slicing allocates **nothing** on heap — span is small struct holding ref + length.

```csharp
Span<byte> buffer = stackalloc byte[256];
buffer[0] = 1;
```

**Cannot:**

```csharp
public class Holder
{
    public Span<byte> Buffer;  // CS8345 — ref struct field in class
}
```

Use **`Memory<T>`** or **`byte[]`** for heap-stored long-lived buffers; use `Span` only in local processing scope.

### ref, in, out Parameters

From `1.2.14`:

| Modifier | Meaning |
|----------|---------|
| `ref` | Alias; must be definitely assigned |
| `out` | Alias for output; must be assigned before return |
| `in` | Readonly alias; avoids copy for large struct |
| `scoped` (C# 11) | ref cannot escape method scope — safety for spans |

```csharp
static int Parse(ReadOnlySpan<char> digits)
{
    int value = 0;
    foreach (var c in digits)
        value = value * 10 + (c - '0');
    return value;
}
```

### readonly vs const vs init

- **`const`** — compile-time, static
- **`readonly` field** — set in ctor, instance or static
- **`readonly struct`** — all instance state immutable
- **`init` accessor** — set at object construction time (`1.2.2`, `1.2.13`)

## Under the Hood

### Why ref struct Cannot Box

Boxing places a copy on the heap with object header — lifetime would outlive stack buffer `Span` points to → **dangling view**. CLR forbids boxing `ref struct` types.

### Span Internals (Conceptual)

`Span<T>` contains **`ref T _reference`** (or similar) + **`int _length`**. For array-backed span, reference points into array data. For `stackalloc`, points into stack frame — must not escape frame end.

### readonly struct JIT

JIT knows fields never change after ctor — enables enregistration and elision of copies when invoking methods on readonly instances.

## Examples

### Example 1: Immutable struct API

```csharp
public readonly struct Money
{
    public decimal Amount { get; }
    public string Currency { get; }

    public Money(decimal amount, string currency)
    {
        Amount = amount;
        Currency = currency;
    }

    public Money Add(decimal delta) => new(Amount + delta, Currency);
}
```

No `Amount += delta` mutating `this` — returns new value.

### Example 2: stackalloc + Span

```csharp
static string TrimUpper(ReadOnlySpan<char> input)
{
    Span<char> buffer = stackalloc char[input.Length];
    int j = 0;
    foreach (var c in input.Trim())
        buffer[j++] = char.ToUpperInvariant(c);
    return new string(buffer[..j]);
}
```

Stack buffer for intermediate chars — no `char[]` allocation if length small (watch `stackalloc` limits ~1 KB typical guidance for stack overflow safety).

### Example 3: Compiler Error (Learning)

```csharp
async Task BadAsync()
{
    Span<byte> s = stackalloc byte[10];
    await Task.Delay(1);
    // Error: cannot use ref struct across await boundary (rules evolve by language version)
}
```

Ref struct lifetime must not exceed stack frame incorrectly.

## Common Mistakes & Anti-patterns

**Storing `Span<T>` in instance field** — compile error or future bug if forced via unsafe.

**Mutable struct pretending to be readonly** — public setters defeat purpose.

**Large `stackalloc` in recursion** — stack overflow.

**Using `readonly struct` for huge blobs** — copying still expensive (`1.2.14`).

**Ignoring `scoped`** when passing spans into other methods — ref escape analysis failures.

## Production & Real-World Notes

**ASP.NET Core** uses `Span`/`ReadOnlySpan` in parsers and header processing internally. Application layer often stays with `string`/`ReadOnlyMemory<char>` unless profiling demands spans.

**Libraries** expose `ReadOnlySpan<char>` overloads alongside `string` for zero-allocation parsing (`int.TryParse(ReadOnlySpan<char>, ...)`).

**Analyzers** CA1815 suggests making structs readonly when possible.

## Comparison / Trade-offs

| Type | Immutability | Allocation | Restrictions |
|------|--------------|------------|--------------|
| `readonly struct` | Yes | Value copy | Size/copy cost |
| `struct` mutable | No | Value copy | foreach traps |
| `ref struct` | Often readonly usage | Stack view | No heap escape |
| `class` | Optional | Heap | Full OOP |

## Quick Reference

| Need | Use |
|------|-----|
| Immutable small value | `readonly struct` / `readonly record struct` |
| Slice array/stack without copy | `Span<T>` / `ReadOnlySpan<T>` |
| Long-lived buffer | `byte[]`, `Memory<T>` |
| Avoid large struct copy | `in` parameter |
| In-place mutation of caller struct | `ref` parameter |

---

## Key Takeaways

- **`readonly struct`** — compiler-enforced immutability; fewer defensive copies.
- **`ref struct`** — stack-only; **cannot box**; powers **`Span<T>`**.
- **Never store `Span<T>` in class fields** — use `Memory<T>` for heap lifetime.
- **`ref` / `in` / `out` / `scoped`** control aliasing and copies.
- **`readonly record struct`** — immutable value + record equality (`1.2.13`).
- Understand rules to use modern BCL APIs without lifetime bugs.
