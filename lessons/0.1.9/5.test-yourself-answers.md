## Answers

### 1. [Recall]
**Answer:** Stash saves **uncommitted changes** (modified tracked files in working tree and/or staging area) so you can get a **clean working tree** without committing WIP.

**Why:** Clean tree is required for safe branch switches and pulls without mixing unrelated work.

---

### 2. [Recall]
**Answer:** **`pop`** applies the stash and **removes** it from the stack. **`apply`** applies the stash but **keeps** it in `stash list`.

**Why:** Pop = one-time restore; apply = reusable copy.

**Common mistake:** Using pop when you might need the same stash on another branch вАФ use apply instead.

---

### 3. [Recall]
**Answer:** **`-u`** or **`--include-untracked`**. (Also **`-a` / `--all`** for ignored files, but `-u` is the standard answer.)

---

### 4. [Understanding]
**Answer:** Uncommitted changes can **block checkout** or **carry over** incorrectly. Stash moves changes off the working tree, so `switch` sees a clean state matching `HEAD`.

**Why:** Git protects you from losing work or polluting another branch with uncommitted diffs.

---

### 5. [Understanding]
**Answer:** **`stash@{0}`** is the **newest** (top of stack). **`stash@{2}`** is older вАФ third from top.

**Why:** Stash uses LIFO numbering; higher index = older entry.

---

### 6. [Understanding]
**Answer:** Git's object model (commits, trees, blobs) already handles snapshots and diffs efficiently. Reusing commits for stash avoids a separate storage system and enables `git stash show`, reflog, and merge semantics.

**Why:** Stash is implemented as merge commits with three parents (HEAD, index, working tree).

---

### 7. [Application]
**Answer:** **`git stash push -u -m "..."`** (or `git stash -u`).

**Why:** Without `-u`, untracked `notes.txt` remains and tree is not fully clean.

---

### 8. [Application]
**Answer:** **Often yes** вАФ if `pop` fails due to conflicts, Git may **keep the stash** so you don't lose it. After resolving, run `git stash drop` if entry still exists.

**Why:** Failed pop is designed to be safe; partial apply + conflict aborts removal.

**Common mistake:** Assuming pop always removes stash вАФ check `git stash list`.

---

### 9. [Application]
**Answer:** Use **`git stash apply`** (same stash on branch 1, switch, apply again on branch 2). When done: **`git stash drop stash@{n}`**.

**Why:** Apply does not consume the stash entry.

---

### 10. [Analysis]
**Answer:** **Stashes are local by default** вАФ `git stash` does not push to remote. Colleague confused stash with commits/branches.

**Why:** No `git push stash` workflow exists; must apply, commit, and push normally.

---

### 11. [Analysis]
**Answer:** **B) `git stash clear`** вАФ deletes **all** stashes irreversibly without per-entry confirmation.

**Why:** `drop` removes one; `clear` wipes entire WIP archive.

---

### 12. [Understanding]
**Answer:** **`-m`** adds a **custom message** visible in `git stash list`, making it easier to identify the right stash among many.

**Why:** Default message is auto-generated (`WIP on branch: ...`) вАФ less readable in busy stacks.

---

### 13. [Analysis]
**Answer:** **Untracked files remained** in the working directory вАФ they were never stashed. Switch may have succeeded if untracked files didn't conflict with target branch files.

**Why:** Default `stash push` only captures tracked modifications.

---

### 14. [Recall]
**Answer:** Any three of: **`list`**, **`show`**, **`drop`**, **`clear`**, **`branch`** (creates branch from stash).

---

### 15. [Application]
**Answer:** Example sequence:
```bash
git stash push -u -m "feature WIP"
git switch main
git pull origin main
# fix, commit, push hotfix
git switch feature/branch
git stash pop
```

**Why:** Stash вЖТ clean switch вЖТ work on main вЖТ return вЖТ restore WIP.
