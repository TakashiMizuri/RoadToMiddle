# Composition over Inheritance (композиция вместо наследования)

> Roadmap: `1.2.11` · Node: `1.2` — C# ООП · Depth: **глубоко**

## Learning Objectives

После этой лекции ты сможешь:

- Объяснить, **почему наследование не является механизмом повторного использования по умолчанию** в профессиональном C#.
- Различать отношения **«is-a»** (наследование) и **«has-a»** (композиция).
- Распознавать **fragile base class problem** и понимать, как `override` ломает предположения базового класса.
- Рефакторить иерархию наследования в **композицию + делегирование**, сохраняя полиморфизм там, где он нужен.
- Выбирать между **наследованием**, **композицией** и **интерфейсами** по практическому decision guide.
- Применять композицию в ASP.NET Core: **middleware pipeline**, **decorator-сервисы**, **strategy injection**.

## Why This Matters

В `1.2.6`–`1.2.7` ты изучил наследование и полиморфизм: `base`, `override`, virtual dispatch в runtime. Эти инструменты мощные — и опасные, когда их берут как первый ответ на любой вопрос про дизайн. Junior-кодовые базы часто разрастаются глубокими деревьями: `Animal` → `Mammal` → `Dog` → `GoldenRetriever`, или хуже — `BaseController` → `CrudController` → `ProductController`, куда в абстрактные родители протекает бизнес-логика.

Индустриальный мантра **«favor composition over inheritance»** не означает «никогда не наследуй». Она означает: **наследование связывает тебя с деталями реализации предков**; **композиция связывает с поведением, которое ты явно выбираешь**. Когда Microsoft поставляет ASP.NET Core, большая часть расширяемости — это композиция (inject `IMiddleware`, обёртка `HttpClient` через `DelegatingHandler`, списки `IAuthorizationHandler`). Наследование появляется там, где фреймворк действительно моделирует substitutability (`Exception`, `Stream`, `DbContext`).

Middle-разработчик должен уметь рефакторить хрупкую иерархию без паники, объяснить, *почему* `new` или `override` сломали production, и знать, когда sealed base class плюс interface — правильный trade-off.

## Core Concepts

### Наследование: что ты на самом деле покупаешь

Когда `class Dog : Animal`, компилятор и runtime устанавливают постоянную связь. `Dog` наследует поля, методы и контракты от `Animal`. Полиморфизм из `1.2.7` позволяет писать `Animal a = new Dog(); a.Speak();` и получать override собаки. Это **substitutability**: код, написанный для `Animal`, может работать с любым производным типом.

Цена — **жёсткая связность (tight coupling)**. `Dog` зависит от каждого `protected` члена, от семантики каждого virtual-метода и от каждого будущего изменения в `Animal`. Автор базового класса и автор каждого подкласса теперь в одной команде навсегда. Измени `Animal.Eat()` так, чтобы валидировать калории до проглатывания — и подкласс, переопределивший `Eat()` без `base.Eat()`, может тихо нарушить новый invariant. Сделай `protected` поле `private` — подклассы ломаются на compile time. Добавь новый abstract-метод — **каждый** concrete descendant должен реализовать его немедленно.

Наследование — это **compile-time контракт о целом семействе типов**. Это уместно, когда в домене действительно есть стабильная таксономия и общая реализация — но редко в application code, которая меняется каждый спринт.

### Композиция: сборка поведения из частей

**Composition** означает, что объект **имеет** другие объекты и **делегирует** им работу вместо наследования реализации. `EmailNotificationService` не наследует `SmsNotificationService`; он держит `INotificationSender` (или несколько) и вызывает их. Внешний объект контролирует, какие части существуют и как взаимодействуют.

```csharp
public sealed class OrderService
{
    private readonly IOrderRepository _repository;
    private readonly IPaymentGateway _payments;
    private readonly IEmailSender _email;

    public OrderService(
        IOrderRepository repository,
        IPaymentGateway payments,
        IEmailSender email)
    {
        _repository = repository;
        _payments = payments;
        _email = email;
    }

    public async Task PlaceOrderAsync(Order order, CancellationToken ct)
    {
        await _payments.ChargeAsync(order.Total, ct);
        await _repository.SaveAsync(order, ct);
        await _email.SendAsync(order.CustomerEmail, "Order placed", ct);
    }
}
```

`OrderService` — не «вид» repository или payment gateway. Он **использует** их. Можно подменить `IPaymentGateway` на test fake, не трогая цепочку наследования — потому что её нет. Так ежедневно выглядит ASP.NET Core DI: конструкторы получают capabilities; классы композируют поведение.

### «Is-a» vs «Has-a»

Классическая эвристика: наследование моделирует **is-a** (`GoldenRetriever` **is a** `Dog`), композиция — **has-a** (`Car` **has an** `Engine`). Эвристика ломается, когда люди форсируют is-a только ради reuse.

Спроси: **Было бы абсурдом в реальном домене подставить родителя вместо ребёнка везде?** Если `Manager` наследует `Employee` ради `Name` и `Id` — substitution часто работает. Если `Stack` наследует `List<T>` ради хранения — как в ранней Java — получаешь `stack.Push(1); stack.Insert(0, 2);`, уничтожающий семантику стека. Это **не** настоящий is-a; это наследование ради удобства.

Более острый тест из *Effective Java* (применим к C#): **Является ли подтип строгим подмножеством контракта поведения родителя, или добавляет ограничения?** Подтипы не должны ослаблять preconditions или усиливать postconditions так, что callers этого не ожидают. Если ребёнок запрещает то, что родитель разрешает — наследование неверный инструмент.

### Fragile Base Class Problem

**Fragile base class problem** называет конкретный режим отказа. Maintainer базового класса выпускает «безопасное» внутреннее изменение. Подклассы, написанные годы назад, предполагали старое поведение. Тесты библиотеки проходят; приложения ломаются в production.

Рассмотрим logging framework:

```csharp
public class Logger
{
    public virtual void Log(string message)
    {
        var formatted = Format(message);
        WriteToSink(formatted);
    }

    protected virtual string Format(string message) => $"[{DateTime.UtcNow:O}] {message}";
    protected void WriteToSink(string text) => Console.WriteLine(text);
}

public class JsonLogger : Logger
{
    protected override string Format(string message)
        => $"{{\"ts\":\"{DateTime.UtcNow:O}\",\"msg\":\"{message}\"}}";
}
```

Позже `Logger.Log` обновляют: вызывать `Validate(message)` до форматирования — но `JsonLogger` не знал про validation и по-прежнему override только `Format`. Или `Log` начинает батчить записи и ожидает, что `Format` чистый; `JsonLogger` делает side-effects. Базовый класс не может эволюционировать без аудита каждого подкласса в каждой consumer-сборке.

С композицией:

```csharp
public interface ILogFormatter { string Format(string message); }
public interface ILogSink { void Write(string text); }

public sealed class Logger
{
    private readonly ILogFormatter _formatter;
    private readonly ILogSink _sink;

    public Logger(ILogFormatter formatter, ILogSink sink)
    {
        _formatter = formatter;
        _sink = sink;
    }

    public void Log(string message)
    {
        var formatted = _formatter.Format(message);
        _sink.Write(formatted);
    }
}
```

`Logger` владеет алгоритмом. Меняешь formatters без subclassing. Новая validation живёт в `Logger.Log` один раз; нет скрытых overrides.

### Делегирование vs наследование для reuse

**Implementation inheritance** переиспользует код, сливая vtables и protected members. **Delegation** переиспользует код, держа экземпляр и пересылая вызовы.

**Decorator pattern** — композиция в disguise:

```csharp
public interface IOrderValidator
{
    ValidationResult Validate(Order order);
}

public sealed class OrderValidator : IOrderValidator
{
    public ValidationResult Validate(Order order)
    {
        if (order.Lines.Count == 0)
            return ValidationResult.Fail("Empty order");
        return ValidationResult.Ok();
    }
}

public sealed class FraudCheckingValidator : IOrderValidator
{
    private readonly IOrderValidator _inner;

    public FraudCheckingValidator(IOrderValidator inner) => _inner = inner;

    public ValidationResult Validate(Order order)
    {
        if (order.Total > 10_000 && order.IsNewCustomer)
            return ValidationResult.Fail("Manual review required");

        return _inner.Validate(order);
    }
}
```

Регистрация в DI: `services.AddSingleton<IOrderValidator>(sp => new FraudCheckingValidator(new OrderValidator()));`. Поведение сложено без иерархии `FraudCheckingValidator : OrderValidator`. Каждый decorator мал, тестируем и опционален.

### Когда наследование остаётся правильным

Наследование оправдано, когда:

1. **Framework extension points** требуют его (`ControllerBase`, `BackgroundService`, `DbContext`, `Exception`).
2. **Истинная substitutability**: любой код с базовым типом корректен с любым derived (`Liskov Substitution Principle`).
3. **Общий skeleton алгоритма** с контролируемыми точками расширения — **Template Method** — и команда владеет всей иерархией.

```csharp
public abstract class CsvExporterBase
{
    public async Task ExportAsync<T>(IEnumerable<T> rows, Stream output, CancellationToken ct)
    {
        await WriteHeaderAsync(output, ct);
        foreach (var row in rows)
            await WriteRowAsync(row, output, ct);
        await output.FlushAsync(ct);
    }

    protected abstract Task WriteHeaderAsync(Stream output, CancellationToken ct);
    protected abstract Task WriteRowAsync<T>(T row, Stream output, CancellationToken ct);
}
```

Здесь **последовательность** фиксирована; подклассы кастомизируют шаги. Риск сдержан, если base `abstract`, методов мало, иерархия мелкая и принадлежит одной команде.

**Interfaces** (`1.2.8`–`1.2.9`) развязывают контракт и реализацию. Предпочитай `INotificationSender` + несколько implementations вместо `abstract NotificationSender` с глубокими детьми, если Template Method не делится нетривиальным кодом.

## Under the Hood

### Как virtual dispatch подталкивает к неверной иерархии

Из `1.2.7`, `callvirt` на virtual-методе проходит method table в runtime. Наследование сливает эти таблицы вниз по цепочке. Каждый override — постоянный hook в dispatch path. IDE «go to implementation» при переименовании virtual-метода показывает каждый subclass — ripples через сборки.

Композиция заменяет одну vtable-индирекцию на **interface call** по private field — часто devirtualizable JIT, когда тип field sealed. Важнее: **граф зависимостей** становится явным в конструкторе: можно static-analyze, кто зависит от `IPaymentGateway`; наследование прячет зависимости в `protected` members.

### Liskov Substitution Principle (LSP)

**LSP:** если `S` — подтип `T`, объекты типа `T` должны быть заменяемы объектами типа `S` без нарушения корректности.

Нарушения тонкие:

```csharp
public class Rectangle
{
    public virtual int Width { get; set; }
    public virtual int Height { get; set; }
    public int Area => Width * Height;
}

public class Square : Rectangle
{
    public override int Width
    {
        set { base.Width = value; base.Height = value; }
    }

    public override int Height
    {
        set { base.Width = value; base.Height = value; }
    }
}
```

Функция `void Resize(Rectangle r) { r.Width = 5; r.Height = 10; }` ожидает area 50. Передай `Square` — area станет 100. Подтип изменил invariants, которые base обещал. **Fix:** не наследовать; отдельные типы или `IShape` без общих mutable width/height.

LSP — формальная причина, почему композиция выигрывает во многих иерархиях «для reuse»: композиция не обещает substitutability, которую нельзя выполнить.

### Memory и lifetime coupling

Экземпляр подкласса **is** экземпляр базового: один object header, поля base расположены перед derived. Подкласс не может отбросить состояние base. Композиция позволяет **менять implementations** в runtime и **меньшие** объекты, когда нужен только facet.

В DI scoped services композиция `DbContext` + `IClock` + `IEmailSender` держит каждый lifetime явным. Наследование `BaseServiceWithDbAndEmail` тянет к **god base classes**, куда каждая фича добавляет зависимости.

## Syntax / API

Новых ключевых слов C# нет — урок про design. Релевантные механизмы:

| Механизм | Роль в композиции |
|----------|-------------------|
| Constructor injection | Поставка composed dependencies |
| `private readonly` fields | Хранение collaborators |
| Interfaces | Контракты для частей |
| `sealed` на leaf classes | Запрет дальнейшего fragile extension |
| DelegatingHandler, middleware | Framework composition hooks |

**Шаги рефакторинга (наследование → композиция):**

1. Выдели **срезы поведения** в base class (validation, formatting, persistence).
2. Извлеки каждый срез в interface + default implementation.
3. Замени `protected virtual` hooks вызовами injected interfaces.
4. Удали пустые intermediate classes; оставь наследование только где LSP держится.

## Examples

### Пример 1: Иерархия птиц (классическая ошибка)

Design через наследование:

```csharp
public abstract class Bird
{
    public abstract void Fly();
}

public class Sparrow : Bird
{
    public override void Fly() => Console.WriteLine("Flying");
}

public class Penguin : Bird
{
    public override void Fly() => throw new NotSupportedException("Cannot fly");
}
```

Callers с `Bird` должны обрабатывать пингвинов, которые «взрываются». **Fix композицией:**

```csharp
public interface IFlyBehavior { void Fly(); }
public sealed class Flying : IFlyBehavior { public void Fly() => Console.WriteLine("Flying"); }
public sealed class Flightless : IFlyBehavior { public void Fly() => Console.WriteLine("Walks instead"); }

public sealed class Bird
{
    private readonly IFlyBehavior _flight;
    public Bird(IFlyBehavior flight) => _flight = flight;
    public void Move() => _flight.Fly();
}
```

`Bird` не subclass'ится ради способности летать; capability композируется.

### Пример 2: ASP.NET Core Middleware как композиция

Middleware — **составной pipeline**, не наследование:

```csharp
public sealed class CorrelationIdMiddleware
{
    private readonly RequestDelegate _next;

    public CorrelationIdMiddleware(RequestDelegate next) => _next = next;

    public async Task InvokeAsync(HttpContext context)
    {
        var id = context.Request.Headers["X-Correlation-ID"].FirstOrDefault()
                 ?? Guid.NewGuid().ToString("N");
        context.Response.Headers["X-Correlation-ID"] = id;
        await _next(context);
    }
}
```

Каждый middleware **has** next delegate; он не наследует `AuthenticationMiddleware`. Порядок настраивается в `Program.cs` — composition root.

## Common Mistakes & Anti-patterns

**Наследование только ради reuse.** Родитель экспонирует двадцать `public` методов; ребёнку нужны три. Наследование импортирует семнадцать лишних (`Stack : List<T>`).

**Глубокие иерархии.** Больше двух уровней concrete inheritance в application code — smell. Вариация поведения → interfaces + composition.

**Protected fields.** Подклассы привязаны к representation. Предпочитай `private` fields; `protected` accessors — только когда Template Method реально нужен — и тогда рассмотри strategy injection.

**Override без base.** Ломается при эволюции base (путаница `new` vs `override` из `1.2.6` усугубляет).

**God abstract base controllers.** `BaseApiController` с auth, logging, mapping и CRUD — фичи борются за `protected` helpers. Разнеси cross-cutting в filters (`IActionFilter`) и композируй сервисы.

## Production & Real-World Notes

Команды enforce **composition by default** через code review: «Почему здесь не interface?» Analyzers вроде Sonar флагают глубокие inheritance trees.

**Testing:** composed classes принимают mocks в конструкторах. Подклассы concrete bases часто требуют awkward partial mocks или только integration tests.

**Vertical slices** (`MediatR` handlers) favor один handler class, композирующий `DbContext` + domain services — не иерархии `BaseCreateHandler<T>`.

**Records и DTO** (`1.2.13`) редко наследуют; favor composition value objects.

При миграции legacy — **strangler**: извлекай interfaces из методов base по одному; inject; наконец `sealed` или удаление base.

## Comparison / Trade-offs

| Подход | Сила | Слабость | Типичное применение |
|--------|------|----------|---------------------|
| Inheritance | Общий алгоритм, framework hooks | Fragile base, LSP risk | `Exception`, `DbContext`, Template Method |
| Composition | Гибкость, тестируемость, явные deps | Больше boilerplate, wiring в DI | Application services, decorators |
| Interfaces | Слабая связность, multiple roles | Нет shared implementation | Контракты, strategy |

**Rule of thumb:** если сомневаешься, помечать класс `sealed`, потому что «кто-то захочет extend» — скорее нужен interface вместо `virtual` methods.

## Key Takeaways

- **Composition over inheritance** снижает coupling к деталям реализации предков.
- **Fragile base class:** изменения base ломают subclasses способами, которые compiler не ловит.
- **LSP** формализует, когда наследование семантически валидно.
- **Delegation + interfaces** заменяют большинство `protected virtual` extension points.
- **ASP.NET Core** extensibility (middleware, handlers, filters) — composition-first.
- Наследование остаётся верным для **framework types** и **истинного is-a** со стабильными контрактами.
- Рефакторинг: **extract interfaces** + **constructor injection**, не ещё один subclass.
