## Answers

### 1. [Recall]
**Answer:** Client.
**Why:** Client открывает соединение (типично) и отправляет первое HTTP-сообщение — request. Server слушает и отвечает.

### 2. [Recall]
**Answer:** Status line (код + reason), headers, optional body.
**Why:** Response = status + metadata (headers) + payload (body может отсутствовать, напр. 204).

### 3. [Recall]
**Answer:** Application layer (layer 7).
**Why:** HTTP не занимается маршрутизацией пакетов — это TCP/IP; HTTP описывает семантику запросов приложений.

### 4. [Understanding]
**Answer:** Протокол не хранит состояние между запросами; каждый request обрабатывается независимо. «Залогиненность» — application layer: cookie, JWT или session ID в **каждом** request.
**Why:** Stateless упрощает scaling — любой app server может обработать request с валидным token.
**Common mistake:** Путать TCP connection reuse с HTTP session state.

### 5. [Understanding]
**Answer:** Для React — **server** (принимает request, отвечает JSON). Для external API — **client** (инициирует outbound HTTP request).
**Why:** Роли определяются инициатором обмена, не названием процесса.

### 6. [Understanding]
**Answer:** HTML ссылается на JS/CSS/images; SPA после boot вызывает API; каждый asset — отдельный request-response.
**Why:** HTTP не «упаковывает» всё приложение в один response (кроме специальных случаев вроде SSR одной страницы).

### 7. [Understanding]
**Answer:** Пользователь видит ошибку/timeout на client. Server может завершить обработку и записать response в закрытое соединение — work wasted, возможны логи без доставки client'у.
**Why:** Client abort не обязательно отменяет server-side execution без cooperative cancellation.

### 8. [Application]
**Answer:** Например: (1) cookie с session ID, (2) `Authorization: Bearer <JWT>`, (3) server-side session store по ID из cookie.
**Why:** Любой механизм, повторяющий идентификатор state в каждом request.

### 9. [Application]
**Answer:** Вероятнее узкое место на server (или сеть до first byte) — долгая обработка до начала ответа; маленький body значит download быстрый.
**Why:** TTFB = time to first byte; большой Waiting при малом download указывает на server/network latency, не на размер payload.

### 10. [Analysis]
**Answer:** C неверно.
**Why:** Обычный HTTP/1.1 — server не push'ит произвольные данные без prior client request (исключения: WebSockets после upgrade, SSE и т.д.).

### 11. [Analysis]
**Answer:** HTTP не хранит login state в соединении как протокол. Повторный request быстрее может быть из cache или warm DB, но auth снова проверяется через cookie/JWT в headers; connection reuse (Keep-Alive) экономит TCP/TLS, не заменяет session mechanism.
**Common mistake:** Смешивать Keep-Alive с application session.

### 12. [Understanding]
**Answer:** HTTP client обычно **ждёт** response (или timeout) на тот же request. Message queue decouples во времени — producer не ждёт consumer synchronously.
**Why:** Request-response — immediate pairing; queues — async processing.

### 13. [Application]
**Answer:** curl — client; Kestrel — server. curl шлёт `GET /ping`; server pipeline возвращает 200 и body `pong`; curl печатает response.
**Why:** Один request → один response.

### 14. [Understanding]
**Answer:** Browser по-прежнему инициирует HTTP к одному host (proxy/LB); proxy выступает **server** для browser и **client** к Kestrel upstream — роли на каждом hop свои.
**Why:** Client-server на каждом участке цепочки сохраняется.

### 15. [Recall]
**Answer:** Time To First Byte — время от отправки request до первого byte response.
**Why:** Отражает server processing + network; большой TTFB = пользователь дольше ждёт начала ответа.
