# HttpClient Pitfalls — DNS and Socket Exhaustion

> Roadmap: `1.4.26` · Node: `1.4` — Async + ASP.NET preview · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Explain **socket exhaustion** from **`new HttpClient()` per request** and **`Dispose`** interaction with **`TIME_WAIT`**.
- Describe **DNS caching** behavior in **`SocketsHttpHandler`** and stale endpoint problems after failover.
- Choose correct **HttpClient lifetime** (singleton vs factory-managed) in **ASP.NET Core**.
- Identify symptoms in production: **`SocketException`**, **`AddressAlreadyInUse`**, intermittent **502** under load.
- Configure **`PooledConnectionLifetime`**, **`PooledConnectionIdleTimeout`**, and **`MaxConnectionsPerServer`**.
- Relate pitfalls to **`Parallel.ForEachAsync`** and microservice fan-out (`1.4.25`).

## Why This Matters

The most famous .NET footgun: **`using var client = new HttpClient()`** inside a method called per HTTP request. **`HttpClient` is IDisposable**, so developers assume short lifetime is correct — opposite of truth for **`HttpClient`**. Each instance may leave connections in **`TIME_WAIT`**; under load Windows/Linux exhaust **ephemeral ports** (~16k–28k range). Errors look random: **`Only one usage of each socket address`**, **`Unable to connect`**, success after restart.

Second pitfall: **DNS**. **`HttpClient`** caches DNS per handler instance. Long-lived singleton without **`PooledConnectionLifetime`** may stick to old IP after blue-green deploy or DNS TTL change — **mysterious partial outages**.

ASP.NET Core apps calling payment, auth, or inventory APIs need **one managed client per logical downstream** via **`IHttpClientFactory`** (`1.4.27`) — not raw **`new`**. Middle developers debug production outages with this mental model.

## Core Concepts

### HttpClient Lifetime Misconception

**`HttpClient`** is thin facade over **`HttpMessageHandler`** (default **`SocketsHttpHandler`**). Expensive state: **connection pool**, **DNS cache**, **SSL session tickets**. **`Dispose()`** on **`HttpClient`** disposes handler and **closes pool** — but if you create new client per request, you **never reuse connections** and accumulate closed sockets in **TIME_WAIT** (typically 30–120s).

Correct mental model:

- **One handler pool per logical downstream service** (or factory-managed rotation).
- Many **`GetAsync`** calls share pool — **keep-alive** reuses TCP connections.

Anti-pattern:

```csharp
public async Task<string> Bad()
{
    using var client = new HttpClient(); // DISPOSED every call
    return await client.GetStringAsync(url);
}
```

Better (interim, still not ideal in DI apps):

```csharp
private static readonly HttpClient Shared = new(); // process singleton
```

Best in ASP.NET: **`IHttpClientFactory.CreateClient("name")`**.

### Socket Exhaustion Mechanics

Each outbound TCP connection uses **local ephemeral port**. Rapid connect/disconnect without reuse fills **TIME_WAIT** slots. **`MaxConnectionsPerServer`** (default unlimited in many versions — effectively high) limits concurrent open connections per host; exhaustion is often **port reuse**, not connection count cap.

Symptoms under load test or **`Parallel.ForEachAsync`** without factory: success at low RPS, cascade failure at high RPS, recovery after minutes.

### DNS Stale Endpoints

**`SocketsHttpHandler`** resolves DNS when creating connections. Handler lifetime tied to **`HttpClient`** instance — **singleton HttpClient** = DNS cached for life of process until connection pool refresh.

**.NET Core 2.1+** **`IHttpClientFactory`** rotates handlers (default **2 minutes** **`HandlerLifetime`**) forcing periodic DNS refresh.

Manual config:

```csharp
new SocketsHttpHandler
{
    PooledConnectionLifetime = TimeSpan.FromMinutes(5), // rotate connections / DNS
    PooledConnectionIdleTimeout = TimeSpan.FromMinutes(2)
};
```

After Kubernetes service IP change without lifetime rotation — some pods hit old endpoints until restart.

### IDisposable Still Matters

Do **not** dispose **factory-created** clients from **`CreateClient()`** per call — factory owns handler lifetime. Dispose **custom** long-lived clients only on shutdown.

## Under the Hood

**`SocketsHttpHandler`** maintains **`ConnectionPool`** per `(host, port, schema)` key. HTTP/1.1 **keep-alive** returns connection to pool. **`HttpClient`** thread-safe for concurrent requests **when sharing one instance/handler**.

**`ServicePointManager`** (legacy **`HttpWebRequest`**) had **`ConnectionLeaseTimeout`** — different stack; modern **`HttpClient`** uses **`SocketsHttpHandler`** settings.

Linux **SNAT port exhaustion** in Azure/K8s — also mitigated by connection reuse and proper factory; may need **egress proxy** or more replicas.

## Syntax / Commands / API

```csharp
services.AddHttpClient("payments", client =>
{
    client.BaseAddress = new Uri("https://payments.internal/");
    client.Timeout = TimeSpan.FromSeconds(30);
})
.ConfigurePrimaryHttpMessageHandler(() => new SocketsHttpHandler
{
    PooledConnectionLifetime = TimeSpan.FromMinutes(5),
    MaxConnectionsPerServer = 20
});
```

- **`dotnet-counters`** — watch exceptions
- Logging **`HttpClient`** **`DelegatingHandler`**

## Common Mistakes & Anti-patterns

**new HttpClient() per request** — #1.

**Singleton HttpClient without PooledConnectionLifetime** — stale DNS.

**Disposing factory-created client** — breaks pooling assumptions if custom.

**Ignoring MaxConnectionsPerServer** when fan-out high.

**Sync `.Result`** combined with many clients — doubles pain (`1.4.22`).

---

## Production & Real-World Notes

Microsoft documentation explicitly warns: **HttpClient intended for reuse**. Align with **`IHttpClientFactory`**.

During incidents: check recent deploy (DNS), load spike (ports), new **`Parallel.ForEachAsync`** without bounds.

**HTTPS** + TLS resumption benefits from connection reuse — per-request clients lose performance too.

## Comparison / Trade-offs

| Approach | Connections | DNS refresh | ASP.NET |
|----------|-------------|-------------|---------|
| new per request | Worst | N/A | Never |
| static singleton | Good reuse | Stale risk | Legacy |
| IHttpClientFactory | Pooled | Handler rotation | **Recommended** |
| Custom handler lifetime | Tunable | Tunable | Advanced |

## Quick Reference

- Reuse **handler pool** — not new client per call.
- **`PooledConnectionLifetime`** — DNS + connection rotation.
- Symptoms: **SocketException**, TIME_WAIT spike.
- Factory **`CreateClient()`** — don't dispose per call pattern wrongly.
- Bound parallel I/O + shared factory.

---

## Key Takeaways

- **`HttpClient` dispose per request** → **socket exhaustion** under load.
- **Connection pooling** requires shared **handler** lifetime.
- **DNS** cached per handler — rotate with **PooledConnectionLifetime** / factory.
- **`IHttpClientFactory`** is the ASP.NET Core solution (`1.4.27`).
- Tune **MaxConnectionsPerServer** for fan-out.
- Combine with async I/O and bounded parallelism — not more clients.
