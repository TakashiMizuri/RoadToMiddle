# Tables: thead, tbody, th scope — Accessibility

> Roadmap: `0.5.9` · Node: `0.5` — HTML · Depth: практика

## Learning Objectives

After this lesson you will be able to:

- Structure tabular data with `<table>`, `<thead>`, `<tbody>`, `<tfoot>`, `<tr>`, `<th>`, and `<td>`.
- Choose when a table is appropriate versus CSS Grid or Flexbox for layout.
- Apply `scope="col"` and `scope="row"` so screen readers associate headers with data cells.
- Write a `<caption>` that describes the table purpose without duplicating every heading on the page.
- Mark complex tables with `headers` and `id` when row and column headers intersect.
- Validate table markup with browser DevTools and an accessibility checker.

---

## Why This Matters

Tables exist to relate **two or more dimensions of data**: product SKUs against warehouse locations, invoice line items against quantities and prices, sprint metrics against team members. When data is genuinely tabular, a `<table>` gives browsers and assistive technologies a predictable grid model. When developers misuse tables for page layout — a habit from the 1990s — they ship markup that is harder to maintain, harder to style responsively, and hostile to screen reader users who hear "table, 47 rows, 12 columns" before reaching the login form.

In `0.5.8` you learned how forms validate user input at the edge of the browser. Tables often sit beside or below forms: search results, admin dashboards, comparison matrices. A Middle developer shipping an ASP.NET Core + React product will render tables on both sides — server-generated Razor partials for reports, React components for interactive grids — but the **accessibility contract starts in HTML semantics**, not in a UI library. TanStack Table, AG Grid, and `<table>` in JSX all ultimately produce DOM nodes; if the underlying structure skips `<thead>` or mislabels headers, no amount of CSS fixes the experience for keyboard and screen reader users.

Accessible tables are also a legal and business requirement in many jurisdictions (WCAG 2.x, ADA, EAA). The fix is rarely exotic: correct structure, scoped headers, and a caption. The cost of skipping this is support tickets, audit findings, and lost trust.

## Core Concepts

### When a Table Is the Right Tool

Use `<table>` when each cell's meaning depends on **both** its row header and column header — or when the data model is inherently a matrix. Price comparison charts, calendar availability grids, and financial statements are classic examples. If you only need items in a row — a card gallery, a toolbar, a marketing hero — reach for Flexbox or Grid instead. Layout tables without semantic headers force assistive tech to announce dimensional structure where none exists conceptually.

That distinction matters in code review: "It looks like a table" is not enough; ask whether removing row/column relationships would lose meaning. If yes, use `<table>`. If no, use `<ul>`, `<dl>`, or CSS layout.

### Anatomy: thead, tbody, tfoot

A well-formed data table groups rows into sections:

- **`<thead>`** — header rows, typically one row of `<th>` cells naming columns.
- **`<tbody>`** — body rows of data, mostly `<td>` cells.
- **`<tfoot>`** — optional summary row (totals, averages); may appear in markup before `<tbody>` but renders after body in visual order per HTML spec.

Each row is a `<tr>`. Header cells are `<th>`; data cells are `<td>`. Browsers apply default bold centered styling to `<th>`, but **semantics** — not appearance — is why you choose `<th>` over `<td>` with bold CSS.

```html
<table>
  <caption>Q1 2026 sales by region (USD thousands)</caption>
  <thead>
    <tr>
      <th scope="col">Region</th>
      <th scope="col">January</th>
      <th scope="col">February</th>
      <th scope="col">March</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th scope="row">North</th>
      <td>120</td>
      <td>135</td>
      <td>142</td>
    </tr>
    <tr>
      <th scope="row">South</th>
      <td>98</td>
      <td>101</td>
      <td>115</td>
    </tr>
  </tbody>
  <tfoot>
    <tr>
      <th scope="row">Total</th>
      <td colspan="3">711</td>
    </tr>
  </tfoot>
</table>
```

The `<caption>` is the table's accessible name — announced when focus enters the table. Write it as a concise label ("Q1 2026 sales by region"), not a paragraph of marketing copy. One `<caption>` per table, placed immediately after opening `<table>`.

### scope: col and row

The `scope` attribute on `<th>` tells assistive technology whether that header labels **columns** or **rows**:

- **`scope="col"`** — header for all cells in its column below (until another header structure intervenes).
- **`scope="row"`** — header for all cells in its row to the right (in left-to-right locales).

For simple tables — one header row plus row labels in the first column — `scope` alone is sufficient and preferred over more complex patterns because it is easy to audit. Screen readers like NVDA and VoiceOver use scope to build a mental grid: "January, 135" instead of an orphan "135".

Do not put `scope` on `<td>`. Do not use `scope="colgroup"` or `scope="rowgroup"` unless you have grouped headers with `<colgroup>` / nested header rows; those are advanced cases covered in WCAG techniques.

### Complex tables: headers and id

When headers span multiple cells or the grid has nested categories — for example, quarters grouped under years — `scope` may be insufficient. The **`headers`** attribute on `<td>` (or `<th>`) lists `id` values of all applicable header cells:

```html
<th id="y2026" colspan="4">2026</th>
...
<th id="q1" headers="y2026">Q1</th>
<td headers="q1 y2026">42</td>
```

Use this when auditors flag ambiguous associations in multi-level headers. The maintenance cost is higher: every `id` must be unique on the page, and refactors break associations silently. Prefer simpler table designs in product UI when possible; reserve complex `headers`/`id` for static reports and compliance PDFs rendered as HTML.

### colspan, rowspan, and accessibility

Spanning attributes merge cells visually but do not remove the need for clear header relationships. A `colspan="3"` on a total cell still needs a row header ("Total") with `scope="row"`. Empty `<td>` placeholders for layout — merged blank corners — should still have headers referenced explicitly in complex grids. Never use colspan purely to align unrelated content.

## Under the Hood

Browsers construct an **internal table model** from your DOM: columns, rows, slots for header cells, and mapping from data cells to header cells. CSS `display: table` mimics layout but **does not** expose the same accessibility tree as a native `<table>` with `<th>`. That is why "div tables" fail audits even when they look identical.

The accessibility tree exposes `role="table"`, `row`, `columnheader`, `rowheader`, and `cell` with relationships derived from scope, headers/id, and section elements. Platform APIs (IAccessible2, UI Automation, NSAccessibility) consume this tree. ARIA `role="grid"` is a different beast — for interactive spreadsheets with roving tabindex — not a drop-in for static data tables.

When React re-renders tbody rows from JSON, the browser recomputes associations on each commit. Missing keys or conditional `<th>` confuse the same machinery humans rely on visually.

## Syntax / Commands / API

| Element / attribute | Purpose |
|---------------------|---------|
| `<table>` | Root of tabular data |
| `<caption>` | Accessible name / description |
| `<thead>`, `<tbody>`, `<tfoot>` | Row group sections |
| `<tr>` | Table row |
| `<th scope="col">` | Column header |
| `<th scope="row">` | Row header |
| `<td>` | Data cell |
| `colspan`, `rowspan` | Cell spanning |
| `headers="id1 id2"` | Explicit header association |
| `abbr` on `<th>` | Short label for long header (optional) |

Validation checklist in DevTools: Elements panel → Accessibility tree → verify columnheader/rowheader nodes and "Name" on caption.

## Examples

**Example 1 — Invoice lines.** E-commerce admin table: columns Item, Qty, Unit price, Line total; each row is a product. First row is `<thead>` with `scope="col"`. No row headers needed if the first column is product name in `<td>` — but if the first column is the primary identifier, use `<th scope="row">` for product name so sorting and screen readers treat it as a label.

**Example 2 — Schedule.** Rows = days, columns = time slots; intersecting cells = booked or free. Both scopes apply. A caption "Room B — weekly availability" orients users before they traverse dozens of cells.

**Example 3 — Anti-pattern.** Using `<table width="100%">` with a single `<tr>` of `<td>` columns for a three-column marketing layout. Replace with CSS Grid; run axe or Lighthouse — layout tables often trigger "table data missing headers" warnings.

## Common Mistakes & Anti-patterns

**Layout tables** — div-or-table used for columns; hurts AT and responsive design. **Fix:** CSS Grid/Flexbox.

**Bold `<td>` instead of `<th>`** — looks like a header, not announced as one. **Fix:** `<th scope="...">`.

**Missing `<caption>`** — users hear "table" with no context. **Fix:** one meaningful caption.

**Empty `<th>` for spacing** — decorative gaps should be CSS, not fake headers.

**Relying only on color** for cell state (red/green stock) without text or `aria-label` — fails color-blind users and WCAG. Pair color with text or icons with accessible names.

**Huge tables without structure** — thousands of `<tr>` in one `<tbody>` with no caption; pagination or `<thead>` sticky patterns help sighted users; caption and scopes help everyone.

## Production & Real-World Notes

Design systems often wrap `<table>` in components (`DataTable`, `TableHeader`, `TableRow`). Enforce **lint rules** or Storybook a11y tests so consumers cannot omit `<th scope>`. For React, avoid invalid DOM nesting (e.g. `<div>` inside `<table>` without `<tbody>` — browsers auto-insert tbody but fragments can surprise hydration).

Server-side Razor: `@foreach` rows inside `<tbody>`; keep header row in `<thead>` outside the loop. Export to Excel often preserves HTML table structure — good captions become worksheet titles in some converters.

Performance: very wide tables hurt mobile; consider horizontal scroll with `overflow-x: auto` on a wrapper **with** keyboard focusable region and visible focus ring — not `tabindex="0"` on every cell unless building a grid widget.

Legal: WCAG 2.2 Success Criterion **1.3.1 Info and Relationships** requires programmatic determination of relationships; tables are a frequent audit target.

## Comparison / Trade-offs

**Native `<table>` vs ARIA grid** — Static read-only data → `<table>`. Editable spreadsheet with cell selection → `role="grid"` pattern with keyboard spec. Mixing them confuses expectations.

**`<table>` vs CSS Grid for data** — Grid can visually mimic tables but lacks header association unless you manually set ARIA — more work, easier to break.

**Simple scope vs headers/id** — scope for 90% of app tables; headers/id when colspan/rowspan create ambiguous axes.

**Client-side sort** — reorder `<tr>` in DOM or re-fetch; preserve `<thead>` and scopes after sort so AT order matches visual order.

## Quick Reference

```
table
├── caption (one, first child)
├── thead → tr → th[scope=col]
├── tbody → tr → th[scope=row]? + td...
└── tfoot (optional totals)
```

| Pattern | Use |
|---------|-----|
| `scope="col"` | Column titles in header row |
| `scope="row"` | Row labels in first column |
| `headers` + `id` | Multi-level or spanned headers |
| `<caption>` | What the table summarizes |

## Key Takeaways

- Use `<table>` only for **tabular data**, not page layout.
- Structure with **thead / tbody / tfoot** and `<th>` vs `<td>` by meaning, not styling.
- **`scope="col"`** and **`scope="row"`** wire headers to cells for assistive tech.
- **`<caption>`** gives the table a name; do not skip it on data tables.
- Complex merged headers need **`headers`** and unique **`id`** values.
- Div-based "tables" lack the native accessibility mapping of semantic table markup.
- Test with keyboard, screen reader sample, and automated a11y tools before shipping.
