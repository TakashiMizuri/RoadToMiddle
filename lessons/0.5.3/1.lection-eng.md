# Headings h1–h6, p, strong, em, mark

> Roadmap: `0.5.3` · Node: `0.5` — HTML · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Build a logical **heading hierarchy** with `h1`–`h6` without skipping levels arbitrarily.
- Use **`<p>`** for paragraphs instead of `<br>` chains or empty divs.
- Apply **`<strong>`** and **`<em>`** for meaning, not bold/italic styling alone.
- Highlight search-relevant text with **`<mark>`** appropriately.
- Explain how headings create a **document outline** for assistive technology.
- Refactor presentational markup (`<b>`, `<i>`, `<font>`) to semantic equivalents.

---

## Why This Matters

Semantic landmarks from `0.5.2` tell users *where* regions are. **Headings** tell them *how ideas nest* inside those regions. A screen reader user often navigates by heading list (H key in NVDA) faster than sighted users scroll. Search engines weight heading text as structure signals. When every line is a `<div class="title">` or you skip from `h1` to `h4` because "it looked smaller," you break that navigation highway.

Text-level semantics matter equally. **`<strong>`** indicates importance or urgency; **`<em>`** stresses emphasis in sentence rhythm — not "make it bold/italic." Browsers default-style them bold and italic, but CSS could change appearance; meaning stays in the tag. Middle developers avoid `<b>`/`<i>` for content that carries importance unless purely decorative (HTML5 allows `<b>` for stylistic offset without extra importance — still prefer strong/em when meaning exists).

Paragraphs seem trivial until you see `<br><br>` soup or `<p>&nbsp;</p>` spacing hacks. **`<p>`** is a block-level paragraph with proper margin semantics and predictable screen reader pauses. **`mark`** highlights text referenced from search or context — yellow background by default — without pretending it is a heading or link.

This lesson connects structure (`0.5.2`) to readable body copy — the layer users actually read before forms (`0.5.7`) and links (`0.5.5`).

## Core Concepts

### Heading Hierarchy: One h1, Nested Levels

HTML provides six heading levels. **`h1`** is the top-level title of the **sectioning content** it sits in — often the page title inside `<main>`, or the article title inside `<article>`. **`h2`** subdivides that topic; **`h3`** subdivides an h2 section; and so on down to **`h6`**.

The classic rule: **do not skip levels for styling**. If you need a smaller visual title after `h1`, use `h2` and shrink it with CSS — do not jump to `h4` because the default font size matches your design. Skipped levels confuse outline navigation: users expect h3 under h2, not under h1 directly.

Multiple `h1` elements were debated in HTML5; in a single-page document one visible `h1` in `main` is the clearest pattern. Inside nested `article` elements, each article may have its own `h1` when articles are independent (card grid). Consistency within your project matters more than dogma — document the rule your team uses.

Headings must describe the **following content block**, not label unrelated widgets. "Introduction" above intro text — good. "Sidebar" above main article — bad.

### p: Paragraphs as First-Class Blocks

**`<p>`** wraps a paragraph of prose. Block-level: browsers add vertical spacing; screen readers insert pauses between paragraphs.

Anti-patterns:

- Multiple `<br>` to separate thoughts — use separate `<p>` elements.
- Empty `<p></p>` for vertical space — use CSS margin.
- Putting block elements inside `<p>` — parser will close `<p>` implicitly; invalid nesting causes DOM surprises.

One `<p>` per distinct paragraph. Lists, headings, figures live **outside** paragraphs.

### strong: Importance and Seriousness

**`<strong>`** means **strong importance**, seriousness, or urgency for its contents. Example: "Warning: **do not** open the panel while powered." The strong phrase changes interpretation if removed.

Default presentation is bold, but do not choose strong *because* you want bold. Choose it because meaning demands importance; use CSS if you only need visual weight on a span without semantic weight.

In legal or safety copy, strong marks critical constraints. In UI microcopy, use sparingly — everything bold becomes nothing bold.

### em: Stress Emphasis

**`<em>`** marks **emphasis** that shifts sentence meaning when spoken aloud. Compare:

- "I *really* needed that yesterday." (emphasis on really)
- "I really **needed** that yesterday." (strong importance on needed)

Em can nest inside strong and vice versa; order reflects which emphasis dominates when read.

Default style is italic. Screen readers may alter intonation. Do not use `<em>` for book titles (often `<cite>`) or foreign words alone (use `lang` on span).

### mark: Highlighted for Reference

**`<mark>`** highlights text **for reference purposes** due to relevance in another context — search keyword hit, quoted passage marker, active step in instructions. Browser default: yellow background.

Not for general highlighting decoration (use CSS class). Not for syntax highlighting in code blocks (use `<code>` + highlighter). Appropriate: user searched "charset" and you wrap matched word in results snippet with `<mark>charset</mark>`.

## Under the Hood

Headings contribute to the **accessibility tree** as heading role with level 1–6. AT builds a table of contents from DOM order, not visual order — CSS `order` or `position` does not change heading navigation sequence; DOM order does.

When you misuse headings for font sizing, Lighthouse and axe report "heading order" violations. Parser always creates heading elements if tags are valid; invalid `<p><div>` nesting may relocate nodes silently.

`strong` and `em` map to strong/emphasis semantics in accessibility APIs unless overridden by ARIA (rare). `mark` maps to mark role; some older AT ignored it — still correct for modern support and future-proofing.

## Syntax / Commands / API

```html
<main>
  <article>
    <h1>Deploying to Production</h1>
    <p>Before you ship, run the full test suite locally.</p>

    <section>
      <h2>Database migrations</h2>
      <p>Apply migrations with <strong>idempotent</strong> scripts only.</p>
      <p>You <em>must</em> back up before the first deploy.</p>
    </section>

    <section>
      <h2>Smoke tests</h2>
      <p>Search results highlighted: configure <mark>UTF-8</mark> in headers.</p>
    </section>
  </article>
</main>
```

Legacy vs semantic (content with meaning → prefer right column):

| Presentational | Semantic when meaning matters |
|----------------|------------------------------|
| `<b>` | `<strong>` |
| `<i>` | `<em>` |
| `<font size="5">` | heading level + CSS |

## Examples

### Example 1: Skipped levels fix

Bad:

```html
<h1>Product FAQ</h1>
<h4>Shipping</h4>
```

Fixed:

```html
<h1>Product FAQ</h1>
<h2>Shipping</h2>
```

Style h2 smaller with CSS if design requires.

### Example 2: strong vs em in one sentence

```html
<p>We <em>strongly</em> recommend that you <strong>never</strong> commit secrets.</p>
```

Em on "strongly" adds spoken stress; strong on "never" marks critical prohibition.

## Common Mistakes & Anti-patterns

**Headings for font size** — Use CSS `font-size`, not h4 pretending to be h2 visually.

**Skipping levels** — h1 → h3 breaks outline; fix hierarchy or use div+class if no semantic subsection.

**br soup instead of p** — Structural problem and bad for AT pauses.

**strong everywhere** — Dilutes importance; use for genuinely critical phrases.

**mark for decoration** — Use CSS background; reserve mark for referenced relevance.

## Production & Real-World Notes

CMS and WYSIWYG editors often emit `<b>`/`<i>`; sanitize to strong/em on save if policy requires. React Markdown renderers map `**` → strong, `*` → em — know the pipeline.

Design tokens for heading sizes decouple visual scale from h-level. Document heading rules in your component library (`Heading` component with level prop).

SEO: one clear h1 per view; avoid hiding keywords in invisible headings.

---

## Comparison / Trade-offs

`<b>`/`<i>` in HTML5 — allowed for stylistic offset without extra emphasis; use when text is bold/italic in original typography but emphasis meaning unchanged. When in doubt, strong/em. `<hgroup>` (rare) groups title + subtitle — subtitle often better as `<p class="subtitle">` under h1 for simpler outline.

---

## Key Takeaways

- Headings structure content for humans and AT — never skip levels for styling.
- One clear h1 pattern per page view inside main/article.
- p for paragraphs; avoid br chains and empty p spacing.
- strong = importance; em = spoken stress — not generic bold/italic.
- mark highlights search/reference relevance, not general design.
