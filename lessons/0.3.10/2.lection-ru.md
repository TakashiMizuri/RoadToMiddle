# REST Principles: Resources, Stateless, Uniform Interface

> Roadmap: `0.3.10` · Node: `0.3` — HTTP и Web · Depth: понимание

## Learning Objectives

После этой лекции ты сможешь:

- Объяснить **REST** как architectural style поверх HTTP, а не синоним «JSON API».
- Определять **resources** как nouns с URIs (`0.3.2`) и operations через HTTP methods (`0.3.4`).
- Описать **statelessness**: каждый request несёт весь context; server не хранит «разговорную» память между requests.
- Применять **uniform interface**: identification, representations, self-descriptive messages, awareness HATEOAS.
- Связать REST constraints со status codes (`0.3.6`) и headers (`0.3.7`).
- Отличать **RESTful** от **REST-inspired** API — оба common в industry.

---

## Why This Matters

Команды называют любой ASP.NET Web API «REST», если он returns JSON over HTTP. Это скрывает design philosophy Roy Fielding: использовать силу HTTP, а не tunnel custom RPC через POST. Когда endpoints — **resources** с meaningful methods, **cache** (`Cache-Control`, ETag) и **proxies** работают с тобой. Когда `/api/DoCheckout` — POST-only verb, ты re-invent semantics, которые HTTP уже даёт.

Не нужен fanatical HATEOAS в каждом response, чтобы benefit from REST thinking. Middle developer решает: nouns vs verbs в URLs, idempotent updates (`0.3.5`), correct status codes, stateless JWT vs session cookie (`0.3.9`), pagination. Эта лекция даёт **conceptual frame** для API design без REST as religion.

## Core Concepts

### REST — Architectural Style, Not Framework

**REST** defines **constraints** for scalable web systems. Не library, не «CRUD в ASP.NET». HTTP — primary protocol; REST — **how you use** HTTP.

**Richardson Maturity Model:**

| Level | Name | Characteristics |
|-------|------|-----------------|
| 0 | POX swamp | Single URI, POST everything |
| 1 | Resources | Multiple URIs |
| 2 | HTTP Verbs | Methods + status codes |
| 3 | Hypermedia | HATEOAS |

Большинство production APIs — **level 2**.

### Resources и URIs

**Resource** — anything identifiable: user, order, catalog. **Representations** — JSON/HTML в body с `Content-Type` (`0.3.7`).

```
GET /users/42
POST /users
DELETE /orders/991
```

**Nouns in paths, verbs in methods:**

- ✅ `DELETE /orders/991`
- ❌ `POST /deleteOrder?id=991`

Collections vs items: `GET /products`, `GET /products/7`, `POST /products`.

Nested resources умеренно — `/users/42/orders` когда orders не exist без user.

### Stateless Interaction

**Stateless** — server **не хранит client conversation context** между requests. Каждый request содержит всё нужное: auth (`Authorization`/`Cookie`), URI, method, body.

Server **may** store **resource state** в DB — это не violation. Violation — wizard step только в server RAM keyed by socket без id в request.

Statelessness **scales**: any app server handles any request; load balancers без sticky sessions для logic. Cookies/JWT OK если identity **в каждом request** — practical REST считает это достаточно stateless.

### Uniform Interface

**Identification of resources** — URIs.

**Manipulation through representations** — clients exchange JSON DTOs; server maps to models.

**Self-descriptive messages** — `Content-Type`, status meaning, cache headers, documented errors.

**HATEOAS** — links в response для transitions. Full HATEOAS rare; pagination `next`, `Location` on 201 — common.

Uniform interface помогает **intermediaries**: caches понимают safe GET (`0.3.5`) и ETag.

### Связь с уроками 0.3.x

- Methods (`0.3.4`), safe/idempotent (`0.3.5`)
- Status codes (`0.3.6`), headers (`0.3.7`)
- HTTPS (`0.3.8`), cookies (`0.3.9`)

REST — **glue narrative** узла `0.3`.

### RESTful vs Practical

Компромиссы: `POST /orders/991/cancel`, report endpoints, bulk POST, GraphQL/gRPC для other trade-offs. Break constraints **consciously** с documented reason.

## Under the Hood

### Statelessness и Scale

Cart только в server memory → scale breaks. Fix: cart **resource** `/carts/{id}` или cart id в каждом request; Redis/DB.

Round-robin без affinity когда auth в **every** request.

GET + Cache-Control + ETag → **304** через intermediaries. RPC `POST /api { action: getProduct }` — not cacheable стандартно.

## Examples

Order resource: PATCH status → **409** invalid. Stateless JWT each call.

Checkout step in static dict by IP — violates statelessness.

Level 0 vs Level 2: POST tunnel vs `GET /users/42`.

---

## Common Mistakes

Verbs in URL. Ignore idempotency. 200 for all. «REST = no DB state». HATEOAS dogma vs OpenAPI reality.

---

## Production & Real-World Notes

OpenAPI as contract. Versioning `/v1/`. Resource-oriented controllers без perfect REST.

## Comparison / Trade-offs

| Style | Strength | Weakness |
|-------|----------|----------|
| REST L2 | HTTP tooling, caches | Complex actions awkward |
| RPC POST | Flexible | Loses cache semantics |
| HATEOAS L3 | Evolvable client | Heavy for fixed SPA |

## Quick Reference

| Constraint | Meaning |
|------------|---------|
| Resources | Nouns + URIs |
| Representations | JSON bodies |
| Stateless | Full context per request |
| Uniform interface | Standard methods/status/headers |
| HATEOAS | Links (optional) |

---

## Key Takeaways

- REST = constraints, not brand.
- Resources + methods + status codes intentionally.
- Stateless requests scale; auth in each request OK.
- Level 2 dominates; know HATEOAS, don't fetishize.
- `0.3.1`–`0.3.9` — mechanical foundation.
