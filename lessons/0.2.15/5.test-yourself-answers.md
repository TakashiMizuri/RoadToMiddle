# Answers: Monorepo vs polyrepo

## Scoring

- 90–100%: отлично
- 80–89%: pass
- <80%: lection + summary

---

## Answers

### 1. [Recall]
**Answer:** **Один** Git repository содержит **несколько** projects/packages (API, web, libs и т.д.).
**Why:** Organizational model, not Git feature.

### 2. [Recall]
**Answer:** **Несколько** отдельных Git repositories, обычно по service/library/team boundary.
**Why:** Independent histories and releases.

### 3. [Understanding]
**Answer:** **Один PR/commit** может менять contract + consumer; CI/compilers ловят mismatch сразу.
**Why:** Atomic cross-layer change без cross-repo coordination.

### 4. [Understanding]
**Answer:** Примеры: каждый repo **свой semver tag**; deploy API без deploy web; CI red в одном repo не блокирует другой; teams автономны.
**Why:** Decoupled lifecycles — core polyrepo benefit.

### 5. [Understanding]
**Answer:** Иначе **каждый** commit запускает full build всех projects — медленно и дорого.
**Why:** Monorepo scale needs affected/path filters.

### 6. [Application]
**Answer:** Patch + tag в **library repo**; bump dependency + release в **API** и **web** repos (минимум 3 PR/releases).
**Why:** Polyrepo multiplies hotfix surface.

### 7. [Application]
**Answer:** **Monorepo** — малый team, tight coupling, atomic fullstack changes проще.
**Why:** Common startup/single-product pattern.

### 8. [Analysis]
**Answer:** **Не всегда** — microservices часто polyrepo, но можно monorepo с many services (Meta/Google style); граница deployment ≠ граница repo обязательно.
**Why:** Architecture vs VCS layout independent.

### 9. [Analysis]
**Answer:** Tag может **обещать** версию всего product, хотя web не менялся — confusion для consumers/ops; нужен per-package tag или changelog scope.
**Why:** Version semantics must match tag convention.

### 10. [Understanding]
**Answer:** First bad commit может быть в **unrelated package**; test без path scope даёт wrong culprit или flaky signal.
**Why:** Bisect spans whole graph — use targeted test script.

### 11. [Understanding]
**Answer:** **Polyrepo** — permissions per repository natural; monorepo needs path rules (enterprise features).
**Why:** Hosting model default.

### 12. [Application]
**Answer:** **Scoped history** — log только изменений API path в monorepo.
**Why:** Path filter in single repo.

### 13. [Comparison]
**Answer:** Monorepo: **cross-package** refactor one PR. Polyrepo: **isolated** blast radius per repo, teams don't step on each other.
**Why:** Coupling vs autonomy trade-off.

### 14. [Analysis]
**Answer:** **Worst of both worlds** — polyrepo overhead без независимых releases (tight coupling anyway).
**Why:** Lection anti-pattern.

### 15. [Recall]
**Answer:** **Submodules/subtrees**, **meta-repo** orchestration. Минус: **complexity**, drift, painful updates.
**Why:** Hybrid compromise costs.
