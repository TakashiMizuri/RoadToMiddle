# Answers: I/O-bound vs CPU-bound

## Scoring
- 90–100%: отлично · 80–89%: pass · <80%: повтори lection

## Answers

### 1. [Recall]
**Answer:** I/O-bound — время в ожидании внешних ресурсов. CPU-bound — время в вычислениях на CPU.
**Why:** Определяет выбор async vs parallel.

### 2. [Understanding]
**Answer:** Async освобождает threads в DB wait → больше concurrent queries на том же pool. BCrypt — pure CPU; async не убирает 100ms compute, только добавляет overhead.
**Common mistake:** «async = faster always».

### 3. [Understanding]
**Answer:** Async signature, но sync I/O внутри (часто через Task.Run) — pool thread всё ещё blocked на I/O; ложное чувство безопасности.
**Why:** Block moves, не исчезает.

### 4. [Application]
**Answer:** (a) CPU — Task.Run/Parallel bounded или background; (b) I/O — EF/Npgsql async; (c) CPU — sync или Task.Run, не fake async I/O.
**Why:** Classify by dominant wait.

### 5. [Application]
**Answer:** EF `Find` sync + Task.Run — fake async; блокирует pool thread; нужен `FirstOrDefaultAsync` или sync method honestly.
**Why:** No thread release during DB wait if sync API inside Run.

### 6. [Analysis]
**Answer:** Когда hot paths CPU-bound или fake async; pool still blocked; state machine overhead без I/O benefit.
**Why:** async helps I/O scalability, not magic.

### 7. [Understanding]
**Answer:** Один request — один thread blocked vs free не меняет network RTT. 200 sync — 200 blocked workers, queue/starvation; async — threads reuse, higher throughput, lower p99.
**Why:** Thread utilization under concurrency.

### 8. [Recall]
**Answer:** Throughput (и tail latency под load); не latency одного I/O hop.
**Why:** Wait time external.

### 9. [Analysis]
**Answer:** Нет для raw speed; cores saturated; нужен scale out, optimize algorithm, bounded parallel — не async на controller ради async.
**Why:** CPU ceiling.

### 10. [Application]
**Answer:** `await` DB; если validate heavy — `Task.Run` для CPU part или cache/precompute; не wrap DB in Task.Run.
**Why:** Separate classifications.
