# Exercises: ARIA, Keyboard, Focus

> Complete after lectures and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Replace div button
**Goal:** Native semantics.

Refactor:

```html
<div class="btn" onclick="save()">Save</div>
```

To accessible control without changing visual class more than needed.

**Success criteria:** Keyboard activatable; correct role; Tab reachable; no div-only click.

---

### E2. Disclosure widget
**Goal:** aria-expanded pattern.

Build HTML + minimal JS: button "Show details" toggles hidden paragraph; update `aria-expanded` and `aria-controls`.

Test Tab and Enter/Space only.

**Success criteria:** APG disclosure attributes; state syncs with visibility.

---

## Medium

### M1. Modal focus trap sketch
**Goal:** Focus lifecycle.

Implement or pseudocode modal open: focus moves to dialog title or first button; Tab cycles inside; Escape closes; focus returns to opener.

List three WCAG-related failures if skip restore focus.

**Success criteria:** Trap + restore documented; Escape handler; aria-modal or role=dialog.

---

### M2. Live region toast
**Goal:** aria-live without focus steal.

On form submit (mock), show "Profile updated" in `<div id="status" aria-live="polite">` without calling `.focus()` on status.

Verify with screen reader or DevTools a11y tree mutation note.

**Success criteria:** Message announced; focus stays on submit button or logical element.

---

## Hard (stretch)

### H1. Roving tabindex tablist
**Goal:** APG tabs keyboard.

Three tabs, one panel visible. Implement roving tabindex: arrow left/right moves tab focus; only one tab tabindex 0.

Reference W3C tabs pattern; note difference from incorrect `tabindex` 1,2,3 on tabs.

**Success criteria:** Arrow keys work; Tab enters panel; roles tab/tablist/tabpanel or equivalent native pattern comparison.
