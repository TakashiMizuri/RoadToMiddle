# Nullable Value Types: int?, DateTime?, and the ?? Operator

> Roadmap: `1.1.18` · Node: `1.1` — C# Language Basics · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Explain **`Nullable<T>`** (`T?` for value types) as a struct that wraps a value type with an **HasValue** flag.
- Distinguish **`null`** on nullable value types from **`default(T)`** on non-nullable value types.
- Use **`??`**, **`??=`**, and **`GetValueOrDefault`** to express defaults without exceptions.
- Predict how nullable value types interact with arithmetic, comparisons, and lifting.
- Separate **nullable value types** from **nullable reference types** (NRT) introduced in C# 8.

---

## Why This Matters

Value types — `int`, `DateTime`, `decimal`, `bool` — always hold a value. They cannot be **`null`**. That sounds safe until you model real data: an optional age on a form, an unset shipment date, a database column that allows **`NULL`**. Before nullable value types, teams abused **sentinel values** like `-1` for "unknown age" or **`DateTime.MinValue`** for "no date." Those sentinels collide with legitimate data and leak through calculations silently.

C# 2.0 introduced **`Nullable<T>`**, syntactic sugar **`T?`**, and later **`??`** made defaulting ergonomic. Entity Framework Core maps **`int?`** to nullable SQL columns naturally. API DTOs express optional numbers without lying about types. Understanding nullable **value** types is prerequisite for reading EF models, SQL **`NULL`** semantics, and LINQ translations — and for not confusing them with **nullable reference types**, a separate compile-time feature for **`string?`** and classes.

## Core Concepts

### What Nullable<T> Actually Is

**`Nullable<T>`** where **`T : struct`** is a **struct** with two fields conceptually: a **`bool HasValue`** and **`T Value`**. The shorthand **`int?`** is identical to **`Nullable<int>`**. You can assign either a concrete **`int`** or **`null`**, which sets **`HasValue = false`**.

```csharp
int? age = null;
Console.WriteLine(age.HasValue); // False

age = 25;
Console.WriteLine(age.Value);    // 25 — or use age!.Value if flow analysis unsure
```

Reading **`.Value`** when **`HasValue`** is **`false`** throws **`InvalidOperationException`**. That is why production code prefers **`GetValueOrDefault()`**, pattern matching, or **`??`** instead of blind **`.Value`**.

**`default(int?)`** is **`null`** — no value. **`default(int)`** is **`0`** — a real zero. The distinction matters when mapping from databases: **`NULL`** → **`null`**, **`0`** → **`0`**.

### Syntax: T? and Assignment

Any non-nullable value type can become nullable:

```csharp
int? count = 42;
DateTime? shippedAt = null;
decimal? discount = 0.15m;
bool? isActive = null; // tri-state bool: true, false, unknown
```

Assignment from **`T`** to **`T?`** is implicit widening. Assignment **`null`** clears the nullable. You cannot assign **`null`** to plain **`int`** without nullable context.

Boxing a nullable value type boxes the **underlying value** only when **`HasValue`** is true; **`null`** nullable boxes as **`null`** reference.

### The ?? Null-Coalescing Operator

**`??`** returns the left operand if it is not **`null`**; otherwise the right operand. For **`int?`**, "not null" means **`HasValue == true`**.

```csharp
int? input = null;
int effective = input ?? 0; // 0

DateTime? due = GetDueDate();
DateTime deadline = due ?? DateTime.UtcNow.AddDays(7);
```

The right side is **evaluated only if needed** — important when the default is expensive:

```csharp
int result = nullable ?? ComputeDefault(); // ComputeDefault not called if nullable has value
```

**`??`** chains: **`a ?? b ?? c`**. Operand types must be compatible; the compiler lifts types so **`int? ?? int`** yields **`int`**.

**`??=`** (C# 8) assigns only when left is **`null`**:

```csharp
int? cache = null;
cache ??= ExpensiveLookup(); // call once, then reuse
```

### GetValueOrDefault and Pattern Matching

**`GetValueOrDefault()`** returns **`Value`** if present, else **`default(T)`** — for **`int?`**, zero, which may or may not be what you want.

**`GetValueOrDefault(T defaultValue)`** supplies a custom fallback without **`??`**:

```csharp
int pages = pageCount.GetValueOrDefault(1);
```

Modern C# favors **pattern matching**:

```csharp
if (age is int years)
    Console.WriteLine(years);

// or
int years2 = age switch
{
    null => 0,
    var n => n
};
```

After **`is int x`** check, compiler knows **`HasValue`**.

### Lifting: Operations on Nullable Value Types

When operands are nullable, operators **lift**: if any operand has no value, result is **`null`** (for arithmetic); comparisons return **`bool?`** with **`null`** meaning "unknown."

```csharp
int? a = 5;
int? b = null;
int? sum = a + b;        // null
bool? eq = a == b;       // null, not false
```

This mirrors SQL **`NULL`** propagation in expressions — EF Core translates many lifted operations to SQL **`NULL`**. Do not assume **`(a == b)`** is **`false`** when either side missing; it may be **`null`**.

Explicit comparison to **`null`** uses **`is null`** / **`== null`** on the nullable itself, not on lifted equality.

### Nullable Value Types vs Nullable Reference Types

**Nullable value types** (`int?`) — runtime feature since C# 2; **`HasValue`** flag in a struct.

**Nullable reference types** (`string?`, **`#nullable enable`**) — compile-time annotations since C# 8; **`string?`** is still a reference that can be **`null`** at runtime; compiler warns on possible dereference.

```csharp
#nullable enable
string? name = null;  // NRT: warning if used without check
int? age = null;      // classic nullable value type
```

They share **`??`** syntax but solve different problems. **`int?`** adds a third state to value types; **`string?`** documents intent for references that were always nullable at runtime.

## Under the Hood

**`Nullable<T>`** is a struct defined in **`System`**. Layout: **`bool hasValue`** plus **`T value`**, with JIT optimizations eliding redundant checks in some patterns.

Calling **`.Value`** on empty nullable generates a throw path — the IL checks **`HasValue`** and calls **`ThrowHelper`**.

**`??`** compiles to conditional logic — not necessarily a method call — often similar to:

```csharp
// conceptual
temp = left;
result = temp.HasValue ? temp.GetValueOrDefault() : right;
```

For reference types, **`??`** is a null check; for **`Nullable<T>`**, **`HasValue`**.

Generic **`Nullable<T>`** constrains **`T : struct`**. Reference types use **`null`** without wrapping.

EF Core materializes DB **`NULL`** as **`null`** **`T?`**; **`NOT NULL`** columns map to non-nullable CLR types unless optional navigation rules differ.

## Syntax / Commands / API

```csharp
int? x = null;
bool has = x.HasValue;
int v = x.GetValueOrDefault(10);
int y = x ?? 10;
x ??= 10;

// Lifted
int? z = x + 5;

// Pattern
if (x is int n) { }

// NRT (separate feature)
#nullable enable
string? title = GetTitle();
```

| Expression | When HasValue false |
|------------|---------------------|
| `.Value` | throws |
| `.GetValueOrDefault()` | `default(T)` |
| `.GetValueOrDefault(d)` | `d` |
| `?? d` | `d` |

## Examples

### Example 1: Optional DTO field

```csharp
public record CreateProductRequest(string Name, decimal Price, int? Stock = null);

// Stock omitted in JSON → null, not confused with 0 units
```

Zero stock and unknown stock differ — impossible with plain **`int`** without sentinel.

### Example 2: Safe default for configuration

```csharp
int? configuredTimeout = configuration.GetValue<int?>("Request:TimeoutSeconds");
int timeout = configuredTimeout ?? 30;
```

Missing config key → **`null`** → 30 seconds; explicit **`0`** in config stays **`0`**, not "use default."

### Example 3: Tri-state filter

```csharp
bool? published = query.Published; // null = don't filter
var q = db.Posts.AsQueryable();
if (published is bool p)
    q = q.Where(x => x.IsPublished == p);
```

**`null`** means "parameter not provided"; **`true`/`false`** filter accordingly — common in search APIs.

## Common Mistakes & Anti-patterns

**Using `-1` or `MinValue` instead of `int?`/`DateTime?`** — Collides with valid data. Fix: nullable types.

**Blind `.Value`** — Throws when null. Fix: **`??`**, pattern matching, **`GetValueOrDefault`**.

**Treating lifted `false` like definite false** — **`null == null`** for **`int?`** is **`true`**; **`a == b`** when either null may be **`null` bool?**. Fix: explicit **`HasValue`** checks.

**Confusing NRT with `T?`** — **`string?`** is not **`Nullable<string>`** (illegal). Fix: separate mental models.

**`GetValueOrDefault()` when zero is valid data** — Fallback becomes ambiguous. Fix: explicit **`??`** with chosen default or stay nullable longer.

## Production & Real-World Notes

PostgreSQL **`NULL`** ≠ zero. EF queries must use **`.HasValue`** or **`!= null`** in expressions translatable to SQL. Client-side **`??`** after **`FirstOrDefault`** differs from SQL **`COALESCE`** placement — affects index use.

JSON **`null`** for missing optional field maps to **`null`** **`int?`**; omitting property may also mean null depending on serializer options.

Logging nullable: prefer **`age?.ToString() ?? "n/a"`** or structured logging with null-friendly templates.

Performance: **`Nullable<T>`** is cheap; no heap allocation unless boxed. Prefer nullable over **`string`** conversion for optional numbers.

## Comparison / Trade-offs

**`T?` vs sentinel value** — Nullable adds clarity and type honesty; sentinel avoids extra struct bit but rots over time.

**`??` vs `if (x.HasValue)`** — **`??`** concise for defaults; explicit branches when logic differs per branch.

**`bool?` vs enum tri-state** — **`bool?`** for unknown yes/no; enum when states are named (**`Unknown`, `Yes`, `No`**).

**Throw on null vs default** — Domain validation may require throw; read models often default. Choose per boundary.

## Quick Reference

```
int?  ≡ Nullable<int>     HasValue + Value
null  → HasValue false
??    → left if HasValue else right (right not evaluated if left ok)
??=   → assign default only when null
Lifted ops → null if any operand missing (arithmetic)
NRT string? ≠ Nullable<string> — compile-time warnings only
```

## Key Takeaways

1. **`T?`** wraps a struct with **`HasValue`**; **`null`** means absent value.
2. **`.Value`** throws when empty — prefer **`??`**, patterns, **`GetValueOrDefault`**.
3. **`??`** and **`??=`** express defaults lazily and readably.
4. **Lifted operators** propagate null like SQL — comparisons can yield **`bool?`**.
5. **`default(int?)`** is null; **`default(int)`** is zero — different semantics.
6. **Nullable value types** (runtime) ≠ **nullable reference types** (C# 8 annotations).
7. EF and SQL **`NULL`** map naturally to **`T?`** — avoid sentinel values in new code.
