# Task vs ValueTask: A Decision Tree

> Roadmap: `1.4.39` · Node: `1.4` — C# async · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Explain **allocation profile** of `Task` vs **pooled** `ValueTask` / `IValueTaskSource`.
- Apply a **decision tree** for API return types in libraries and application code.
- State when **`ValueTask` must not be awaited twice** or consumed after await.
- Recognize **`async ValueTask` methods** vs returning **synchronous result** wrapped in `ValueTask`.
- Compare **`ValueTask<T>`** with **`Task<T>`** for hot paths (caching, I/O often sync-complete).
- Avoid breaking changes when switching return types in public APIs.

---

## Why This Matters

`Task` is the default async currency in .NET — familiar, composable, safe to await multiple times (idempotent for completed tasks). **`ValueTask`** (C# 7+) was added for **performance**: when an operation **often completes synchronously**, returning a cached `Task` still allocates. `ValueTask` can wrap a **`T` result directly** or a **pooled** `IValueTaskSource` implementation — zero allocation on the sync path.

Misuse causes **subtle bugs**: awaiting the same `ValueTask` twice, storing `ValueTask` in a field and awaiting later after consumption. Library authors (EF Core, `Stream.ReadAsync`) use `ValueTask` on hot paths; application developers often **should stay on `Task`** unless profiling proves need.

Interviewers ask: "When `ValueTask`?" Middle answer: **decision tree + double-await rule**, not "always faster."

## Core Concepts

### Task: Reference Type, Always Safe (Mostly)

`Task` and `Task<T>` are **classes** — heap allocated (except `Task.CompletedTask` cached). Every `async Task` method that suspends allocates a state machine + often a `Task` object. **Await twice** on same `Task` instance: OK — completed tasks return cached result.

**Default choice** for public APIs, ASP.NET actions, business services — clarity and composability beat micro-allocations.

### ValueTask: Struct Wrapper

`ValueTask` / `ValueTask<T>` are **structs** wrapping one of:

1. **`T`** — operation completed synchronously (no `Task` alloc).
2. **`Task` / `Task<T>`** — fallback when async path taken.
3. **`IValueTaskSource` / `IValueTaskSource<T>`** — pooled custom async (advanced).

```csharp
public ValueTask<User?> GetCachedUserAsync(int id)
{
    if (_cache.TryGetValue(id, out var user))
        return new ValueTask<User?>(user); // sync path — cheap

    return new ValueTask<User?>(FetchFromDbAsync(id)); // wraps Task
}
```

### The Double-Await Rule

**Do not await a `ValueTask` more than once** unless documentation guarantees it wraps a `Task` (still discouraged). After await, the struct may be **consumed** — second await undefined behavior / exception.

**Do not store `ValueTask` in a field** and await later — pattern is for immediate consumption.

```csharp
// WRONG
_valueTask = GetAsync();
await _valueTask;
await _valueTask; // BUG

// RIGHT
var task = GetAsync().AsTask(); // materialize once if needed multiple times
```

Use **`.AsTask()`** when you need multiple awaits, `WhenAll`, or storage.

### async ValueTask Methods

```csharp
public async ValueTask<int> ComputeAsync() { ... }
```

Still generates state machine. Benefit appears when method **often returns without awaiting** (early sync return). If always suspends, little gain — stick with `Task`.

### Decision Tree

```
Start: choosing return type for async method
│
├─ Public API / most app code?
│   └─ YES → Task / Task<T>  ✓ default
│
├─ Hot path + profiling shows Task alloc pressure?
│   └─ NO → Task
│   └─ YES ↓
│
├─ Often completes synchronously (cache hit, buffered I/O)?
│   └─ NO → Task (ValueTask adds complexity for little gain)
│   └─ YES ↓
│
├─ Caller will await exactly once immediately?
│   └─ NO → Task (or AsTask() at boundary)
│   └─ YES ↓
│
├─ Will you implement IValueTaskSource pooling?
│   └─ Library/framework level → advanced ValueTask
│   └─ App code → simple ValueTask<T> sync returns OK
│
└─ Need composability (WhenAll, continuations stored)?
    └─ Task
```

### EF Core and Stream Examples

`DbContext.SaveChangesAsync` returns `Task` — high-level, not micro-hot per call in typical apps.

`Stream.ReadAsync(Memory<byte>)` returns `ValueTask<int>` — called in tight loops; sync completion common for buffered streams.

Follow framework patterns **inside** similar hot loops; do not blanket-convert controllers to `ValueTask`.

## Under the Hood

`ValueTask` struct holds a **union-like** reference to object or `T`. When await runs, `ValueTaskAwaiter` checks mode: if `T`, complete synchronously; if `Task`, delegate; if `IValueTaskSource`, call `GetResult` with token and possibly **return instance to pool** (e.g., `ArrayPool` backed implementations in `System.IO.Pipelines`).

**Pooling** requires correct `IValueTaskSource.GetStatus/OnCompleted/GetResult` implementation — easy to get wrong (double release, use-after-free). Reason library code owns pooling, not typical CRUD services.

`async ValueTask` methods use **`AsyncValueTaskMethodBuilder`** — can avoid `Task` allocation when method completes synchronously on first run.

## Common Mistakes & Anti-patterns

**Converting all methods to `ValueTask` "for perf"** without measurement — complexity cost.

**Caching `ValueTask` in `Task.WhenAll`** — must convert with `AsTask()`.

**Interface returns `ValueTask`, mock in tests awkward** — `Task` easier for test doubles.

**Breaking change** swapping `Task` → `ValueTask` on public API — binary and semantic contract change.

**Awaiting consumed `ValueTask` from cache** — if custom pool resets token.

---

## Production & Real-World Notes

Profile with **dotnet-counters**, **Allocation Tracker** in dotMemory, or **EventPipe** before changing signatures. ASP.NET Core pipeline itself uses `ValueTask` in places (`ReadAsync`, middleware delegates returning `ValueTask`).

Minimal APIs support `ValueTask` return from route handlers — fine for single await by framework.

Document team standard: **application layer `Task`**, **infrastructure hot paths `ValueTask`** after review.

## Comparison / Trade-offs

| | Task | ValueTask |
|---|------|-----------|
| Await twice | Safe (completed) | Unsafe |
| WhenAll | Native | Need AsTask() |
| Sync-fast path alloc | Task or Task<T> alloc | Often zero |
| Learning curve | Low | Medium-high |
| Public API default | Yes | Rare |

---

## Quick Reference

```
Default → Task<T>
Hot + often sync-complete + single await → ValueTask<T>
Multiple awaits / store / WhenAll → AsTask() or Task
Library tight loop + pooling → IValueTaskSource
Never await same ValueTask twice
```

---

## Key Takeaways

- **`Task`** — default; composable, safe, clear.
- **`ValueTask`** — perf when sync-complete path is common and awaited once.
- **Double-await `ValueTask`** — bug; use **`AsTask()`** to materialize.
- **Decision tree** beats blanket rules; profile first.
- **Pooling** — expert territory (`IValueTaskSource`).
- Public APIs: conservative with `ValueTask`.
