## Answers

### 1. [Recall]
**Answer:** **blob** — содержимое файла; **tree** — listing директории (имя → hash); **commit** — snapshot metadata + tree + parents; **tag** — annotated метка на object (обычно commit).

**Why:** Это единственные четыре типа objects в object database Git. Всё остальное (ветки, HEAD, index) — не objects.

---

### 2. [Recall]
**Answer:** `.git/refs/heads/<branch-name>`, например `.git/refs/heads/main`.

**Why:** Local branches — это ref-файлы с одним commit hash внутри.

---

### 3. [Recall]
**Answer:** Строка вида `ref: refs/heads/<branch-name>` — symbolic reference на текущую ветку.

**Why:** HEAD обычно не хранит hash напрямую, а указывает на branch ref, который уже указывает на commit.

---

### 4. [Understanding]
**Answer:** Git вычисляет hash (SHA-1) от header + content. Одинаковый content → одинаковый hash → один файл в `.git/objects/`. Второй файл с тем же content не создаёт новый object — index просто ссылается на существующий hash.

**Why:** Content-addressable storage: identity = content. Это deduplication «бесплатно».

**Common mistake:** Думать, что каждый path получает свой blob независимо от content.

---

### 5. [Understanding]
**Answer:** **Commit object** — immutable запись в `.git/objects/` с tree, parents, author, message. **Ref** (`main`) — mutable указатель (файл в `refs/heads/`) на tip commit. При новом commit создаётся **новый** commit object; ref **main** перезаписывается новым hash. Старый commit object остаётся.

**Why:** Objects не меняются; двигаются только refs.

---

### 6. [Understanding]
**Answer:** Symbolic ref — ref, который содержит не hash, а путь к другому ref (`ref: refs/heads/main`). HEAD → `refs/heads/main` → commit hash. Git «разрешает» цепочку при необходимости.

**Why:** Позволяет одному файлу HEAD переключаться между ветками без перезаписи hash commit'а в HEAD.

---

### 7. [Understanding]
**Answer:** Index — staging area: какие **paths** войдут в **следующий** commit и какие blob hash'и им соответствуют. Objects хранят content по hash, но не знают «это README в корне проекта». Index mutable и пересобирается при add/reset. Index — не object, потому что не content-addressed и не part of immutable history.

**Why:** Разделение: objects = history storage; index = mutable подготовка snapshot.

---

### 8. [Understanding]
**Answer:** (Примерный порядок)
1. Из index строятся tree objects (листья → корень).
2. Trees записываются в `.git/objects/`.
3. Создаётся commit object: tree hash, parent = текущий HEAD commit, author/committer/message.
4. Commit записывается в objects.
5. Обновляется ref текущей ветки (`refs/heads/main`) на новый commit hash.

**Why:** Commit — это новый object + движение branch tip, не изменение старых objects.

---

### 9. [Application]
**Answer:** **Один** уникальный blob object.

**Why:** Content `same\n` одинаков — hash одинаков. Два path в index, один blob в objects.

---

### 10. [Application]
**Answer:** **Detached HEAD** — commit'ы создаются, но ни одна branch ref не двигается. При `git checkout main` новые commit'ы могут стать unreachable (пока их не спасёт reflog или новая ветка).

**Why:** HEAD указывает на commit напрямую, не на `refs/heads/...`.

**Common mistake:** Думать, что commit «на main» автоматически — main ref не обновляется.

---

### 11. [Application]
**Answer:** `d832...` — hash **root tree object** для этого commit. Тип object: **tree**.

**Why:** Поле `tree` в commit указывает на snapshot файловой структуры.

---

### 12. [Analysis]
**Answer:** **A** неверно.

**Why:** Commit хранит указатель на **tree** (полный snapshot), не diff. Diff вычисляется между tree двух commit'ов.

---

### 13. [Analysis]
**Answer:** Если удалены нужные objects — `git log` может оборваться с ошибкой «missing object»; `git fsck` сообщит о missing/corrupt objects. Repo повреждён.

**Why:** History — граф objects; удаление ломает integrity и reachability.

---

### 14. [Analysis]
**Answer:** **Loose** — один object = один файл в `.git/objects/xx/...`, создаётся при add/commit. **Pack** — множество objects в `objects/pack/*.pack`, обычно после `git gc` или fetch с packed remote. Pack использует delta compression для экономии места.

**Why:** Write path оптимизирован на скорость (loose); gc оптимизирует хранение (pack).

---

### 15. [Recall]
**Answer:** (любые три из списка)
- `index` — staging area
- `config` — remotes, user.name, core settings
- `HEAD` — текущая ветка/commit
- `hooks/` — скрипты pre-commit и т.д.
- `logs/` — reflog
- `COMMIT_EDITMSG` — последнее commit message

**Why:** Полная картина repo включает не только objects и refs.

---

### 16. [Understanding]
**Answer:** **Lightweight tag** — файл в `refs/tags/v1.0` с commit hash (это ref, не object). **Annotated tag** — tag **object** в `.git/objects/` + ref `refs/tags/v1.0` указывает на tag object, который указывает на commit; содержит message, tagger, дату.

**Why:** Annotated tags — полноценные objects для release metadata; lightweight — просто имя → commit.

---

### 17. [Application]
**Answer:**
```
refs/heads/main → commit hash C
commit C → tree T (root)
tree T → entry "README.md" → blob hash B
blob B → bytes содержимого README.md
```

**Why:** Ref → commit → tree → blob → content.

---

## Model answers for practical sections

После `echo "hello" > README.md` и commit:

- `git hash-object README.md` → hash blob (с `\n` в конце): `ce013625030ba8dba906f756967f99e928fe4667` (если именно `hello\n`)
- `git rev-parse HEAD` → commit hash (зависит от author date; у каждого свой)
- `git cat-file -p HEAD` первая строка: `tree <hash>` — root tree

**Success:** ты увидел связь blob ↔ index ↔ commit ↔ ref.
