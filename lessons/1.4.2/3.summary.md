# I/O vs CPU — краткая выжимка

> `1.4.2` · ~10 мин

## Главная идея

**I/O-bound** — ждём disk/network/DB; **async** освобождает threads в wait. **CPU-bound** — считаем; нужен parallelism до cores, не «async magic». **Fake async** (`Task.Run` + sync Read) всё ещё блокирует pool.

## Ключевые пункты

- Sync I/O: thread blocked весь RTT.
- Async I/O: thread свободен → выше throughput при concurrency.
- Async не уменьшает latency одного hop и не ускоряет BCrypt.
- Классифицируй каждый шаг mixed workload отдельно.
- ASP.NET: true async end-to-end (HttpClient, EF, streams).

## Примеры

```csharp
await httpClient.GetAsync(url);           // I/O ✓
await Task.Run(() => Hash(pwd));          // CPU offload
await Task.Run(() => File.ReadAllText(p)); // ✗ fake async
```

## «Под капотом»

Blocking syscall снимает thread с runnable set. Overlapped I/O + completion callback → continuation на pool. Fake async только переносит block на другой worker.

## Частые ошибки

- `async` на CPU-only methods.
- `SaveChangesAsync` при sync driver mindset.
- Ожидание «async = faster» для одного CPU endpoint.

## Мнемоника

**Wait → await. Compute → parallel (bounded).**
