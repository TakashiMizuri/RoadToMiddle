# Answers: Generations Gen0/1/2

## Answers

### 1. [Recall]
**Answer:** Most objects die young; collecting young generations often is cheaper than full heap scans.
**Why:** Basis of generational GC.

### 2. [Recall]
**Answer:** Gen0 very often; Gen2 rarely and expensively (full collections).
**Why:** Survivors promoted upward.

### 3. [Understanding]
**Answer:** Objects surviving collection move Gen0→Gen1→Gen2.
**Why:** Older gens scanned less frequently.

### 4. [Understanding]
**Answer:** **Server GC** default for ASP.NET Core — per-CPU heaps, throughput-oriented.
**Why:** Web host configuration.

### 5. [Understanding]
**Answer:** Static roots live process lifetime → objects never die → accumulate in Gen2 → frequent expensive Gen2 considerations.
**Why:** Long-lived roots skip generational benefit.

### 6. [Application]
**Answer:** Allocate Gen0, unreachable after request → collected on next Gen0/ ephemeral GC — ideal.
**Why:** Short lifetime pattern.

### 7. [Application]
**Answer:** High **allocation rate** / many Gen0 collections — investigate allocations (`1.5.5`).
**Why:** Counter reflects collection frequency.

### 8. [Analysis]
**Answer:** Forces full GC disrupting generational tuning; hurts perf; callers should measure not force.
**Why:** GC self-tunes.

### 9. [Analysis]
**Answer:** **Gen2** immediately for LOH allocations (`1.5.8`).
**Why:** Large object rules.

### 10. [Application]
**Answer:** Track old→young references so Gen0 GC doesn't miss live young objects referenced only from Gen2.
**Why:** Write barrier + card marking.

### 11. [Understanding]
**Answer:** Gen2 work concurrent with app threads — shorter STW pauses on servers.
**Why:** Server GC feature.

### 12. [Application]
**Answer:** Storing every request object in static list — short-lived data promoted to Gen2 unnecessarily.
**Why:** Defeats hypothesis.

### 13. [Analysis]
**Answer:** Gen0/ephemeral usually shorter pauses; Gen2/full significantly longer ( workload dependent).
**Why:** Scope of collection differs.

### 14. [Understanding]
**Answer:** Tracked entities live until context disposed — Gen0 collect after request if no other roots; heavy graphs increase Gen0 survival/promotion.
**Why:** Per-request allocation volume.

### 15. [Application]
**Answer:** Correlate GC frequency and CPU time in GC with RPS/deploy changes — capacity tuning.
**Why:** Production observability.
