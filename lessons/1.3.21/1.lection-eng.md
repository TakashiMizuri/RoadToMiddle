# LINQ Join and GroupJoin — SQL-Like Joins in Memory

> Roadmap: `1.3.21` · Node: `1.3` — C#: Advanced Topics · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Distinguish **`Join`** (inner join) from **`GroupJoin`** (left join + grouping, SQL `LEFT OUTER JOIN` + subselect pattern).
- Translate SQL **`INNER JOIN`** / **`LEFT JOIN`** mental models to LINQ method and query syntax.
- Choose **`DefaultIfEmpty`** with **`GroupJoin`** to simulate left outer join flat results.
- Explain join algorithm costs on **`IEnumerable`** vs **`IQueryable`** (nested loops vs provider SQL).
- Avoid cartesian explosion and duplicate key pitfalls in join keys.
- Use **`join ... into`** (query syntax) equivalent to **`GroupJoin`**.

---

## Why This Matters

Relational data rarely lives in one table. Orders have customers, posts have tags, line items reference products. SQL expresses relationships with **`JOIN`**. LINQ brings the same compositional idea to objects and, through EF Core, to translated SQL. **`Join`** and **`GroupJoin`** are easy to confuse: **`Join`** drops non-matching rows like **`INNER JOIN`**; **`GroupJoin`** groups the right sequence per left key — the building block for **`LEFT JOIN`** patterns and hierarchical results (customer + all orders).

Junior code often reimplements joins with nested **`foreach`** and **`Dictionary`** manually — slower to write and easier to get wrong on null handling. Middle developers read LINQ join pipelines in repositories, write report queries, and debug EF SQL where a misplaced **`GroupJoin`** became a correlated subquery or cartesian product. Understanding keys, equality comparers, and deferred execution separates fluent LINQ from production incidents.

## Core Concepts

### Join: Inner Join on Key Equality

**`Enumerable.Join`** (and **`Queryable.Join`**) pairs elements from outer and inner sequences when keys match. Each match produces one output element via **`resultSelector`**.

```csharp
var query =
    customers.Join(
        orders,
        c => c.Id,
        o => o.CustomerId,
        (c, o) => new { c.Name, o.Total });
```

Semantically: **`FROM customers c INNER JOIN orders o ON c.Id = o.CustomerId`**. If a customer has zero orders, they disappear. If an order references a missing customer ID, that order has no pair — also excluded.

Keys must match by **`Equals`** — default equality comparer unless you pass **`IEqualityComparer`**. Reference types compare references unless overridden **`GetHashCode`/`Equals`**. Value types compare values. **Composite keys** use anonymous types or tuples:

```csharp
join on new { c.Region, c.Tier } equals new { o.Region, o.Tier }
```

Anonymous types in joins require **same property names and types** in C# query syntax — order of properties matters for tuple equality in method syntax with **`ValueTuple`**.

### GroupJoin: Group Right Side Per Left Key

**`GroupJoin`** takes outer + inner + key selectors + **`resultSelector(outer, innerGroup)`** where **`innerGroup`** is **`IEnumerable<TInner>`** of all inner rows matching the outer key.

```csharp
var query =
    customers.GroupJoin(
        orders,
        c => c.Id,
        o => o.CustomerId,
        (c, orderGroup) => new { c.Name, Orders = orderGroup });
```

Each customer appears **once**, even with zero orders — **`orderGroup`** may be empty. This mirrors **`LEFT JOIN`** followed by grouping inner rows per customer — or SQL nested collection pattern.

To **flatten** to left-outer join rows (customer + order or null), **`SelectMany`** with **`DefaultIfEmpty`** follows **`GroupJoin`** — classic LINQ recipe covered in next subtopic overlap; here is the shape:

```csharp
customers.GroupJoin(orders, c => c.Id, o => o.CustomerId, (c, g) => new { c, g })
    .SelectMany(x => x.g.DefaultIfEmpty(), (x, o) => new { x.c.Name, Order = o });
```

When **`o`** is null, you have customer with no order — **`LEFT OUTER JOIN`** flat shape.

### Query Syntax: join and join into

```csharp
var q =
    from c in customers
    join o in orders on c.Id equals o.CustomerId
    select new { c.Name, o.Total };

var q2 =
    from c in customers
    join o in orders on c.Id equals o.CustomerId into custOrders
    from o in custOrders.DefaultIfEmpty()
    select new { c.Name, OrderTotal = o != null ? o.Total : 0 };
```

**`join ... into custOrders`** is **`GroupJoin`**. Second **`from`** with **`DefaultIfEmpty`** flattens — left join. Query syntax **`equals`** requires **outer** key on left, **inner** key on right — reversed from method argument order but same semantics.

### Multiple Joins

Chain joins left-to-right — each result becomes outer for the next:

```csharp
from o in orders
join c in customers on o.CustomerId equals c.Id
join p in products on o.ProductId equals p.Id
select new { c.Name, p.Title, o.Qty };
```

With **`IQueryable`**, EF Core translates to SQL **`JOIN`** chain. With **`IEnumerable`**, each join is in-memory — often hash join built internally after materialization.

## Under the Hood

**`Enumerable.Join`** implementation (conceptually): build a **lookup** (hash map) of inner sequence keyed by inner key — O(m) — then iterate outer — O(n) — yielding matches — **O(n + m)** average if hash codes balanced. Naive nested loops would be O(n × m).

**Duplicate keys on inner:** **`Lookup`** stores **groups** of inner elements per key; **`Join`** emits **one output row per matching pair** — inner duplicate keys multiply rows (like SQL).

**Duplicate keys on outer:** each outer row matches its inner group independently.

**`GroupJoin`** builds the same lookup, but **`resultSelector`** receives **`IEnumerable`** per outer element without flattening — deferred until that inner sequence is enumerated.

**`IQueryable.Join`:** expression tree node **`Join`** translated by provider. EF Core emits SQL **`INNER JOIN`** with **`ON`** predicate. **`GroupJoin`** may become **`LEFT JOIN`** + subquery or client evaluation if translation fails — EF Core 3+ throws on client **`GroupJoin`** that cannot translate when strict.

**Deferred execution:** **`Join`**/` **`GroupJoin`** return sequences; SQL runs or memory join runs at enumeration / **`ToList()`**.

## Syntax / API

```csharp
// Inner join — method
outer.Join(inner, ok, ik, (o, i) => result);

// Group join — method
outer.GroupJoin(inner, ok, ik, (o, ig) => result);

// Left join pattern
outer.GroupJoin(...).SelectMany(x => x.g.DefaultIfEmpty(), ...);

// Comparer overload
outer.Join(inner, ok, ik, (o, i) => r, StringComparer.OrdinalIgnoreCase);
```

## Examples

### Inner join: orders with customer name

```csharp
var report = orders.Join(
    customers,
    o => o.CustomerId,
    c => c.Id,
    (o, c) => new OrderReportRow(c.Name, o.Id, o.Total));
```

Only orders with valid **`CustomerId`** appear. Orphan orders excluded — may be desired for integrity reports or not; use **`GroupJoin`** + **`DefaultIfEmpty`** from orders side if orders are outer.

### Hierarchical: customer with order list

```csharp
var dto = customers.GroupJoin(
    orders,
    c => c.Id,
    o => o.CustomerId,
    (c, ords) => new CustomerDto(
        c.Id,
        c.Name,
        ords.Select(o => o.Total).ToList()));
```

**`.ToList()`** inside selector **forces** order enumeration per customer — OK for in-memory; with **`IQueryable`**, may translate to SQL subquery or cause client eval — profile SQL.

### EF-style left join (concept)

```csharp
from c in db.Customers
join o in db.Orders on c.Id equals o.CustomerId into og
from o in og.DefaultIfEmpty()
select new { c.Name, Total = o == null ? 0 : o.Total };
```

Translated to **`LEFT JOIN`** when supported.

## Common Mistakes & Anti-patterns

**Swapping join keys** in query syntax — compile error if types mismatch; subtle logic bug in method syntax if you swap selectors.

**Using `Join` expecting left join behavior** — missing customers/orders silently dropped.

**Forgetting `DefaultIfEmpty`** after **`into`** — empty groups skip rows in flatten step incorrectly.

**Composite key property name mismatch** in anonymous types — join never matches.

**Joining on float/double** — equality unstable; join business keys (IDs), not computed floats.

**Multiple enumeration of `IQueryable` join** — multiple SQL round-trips; use **`ToList()`** once if needed twice.

## Production & Real-World Notes

Prefer **`IQueryable`** joins in EF for large datasets — let SQL join indexed columns. In-memory **`Join`** on two **`List`** of 100k rows loads both — acceptable in app memory; not for unbounded streams.

For **`Include`** vs **`Join`**: EF **`Include`** generates **`JOIN`** or split queries; explicit **`Join`** in LINQ gives projection control and avoids over-fetching columns.

Logging: **`ToQueryString()`** (EF5+) inspects join SQL. Missing indexes on **`CustomerId`** FK show up as seq scans in **`EXPLAIN`**.

## Comparison / Trade-offs

| Need | LINQ |
|---|---|
| Inner flat pairs | **`Join`** |
| Outer + collection per row | **`GroupJoin`** |
| Left flat pairs | **`GroupJoin` + SelectMany + DefaultIfEmpty** |
| SQL on server | **`IQueryable`** + translatable keys |
| Nested foreach clarity | Sometimes **`Join`** clearer; sometimes SQL raw |

**`Join`** vs manual **`Dictionary`**: **`Join`** is declarative; dictionary same Big-O with more boilerplate.

## Quick Reference

| SQL | LINQ |
|---|---|
| INNER JOIN | **`Join`** |
| LEFT JOIN (flat) | **`GroupJoin` + DefaultIfEmpty** |
| LEFT JOIN (nested) | **`GroupJoin`** only |
| `join into` | **`GroupJoin`** |

## Key Takeaways

- **`Join`** = inner join, unmatched rows dropped.
- **`GroupJoin`** = outer row + **`IEnumerable`** of matches; empty group still present.
- Left outer flat pattern needs **`DefaultIfEmpty`**.
- Keys need stable equality; composite keys via anonymous type/tuple.
- **`IEnumerable`** uses hash lookup; **`IQueryable`** translates to SQL JOIN.
- Duplicate inner keys multiply output rows in **`Join`**.
