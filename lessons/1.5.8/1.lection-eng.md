# Large Object Heap (LOH) and Fragmentation

> Roadmap: `1.5.8` · Node: `1.5` — .NET Runtime and Memory · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Define **LOH** (Large Object Heap) and the **~85,000 byte** allocation threshold.
- Explain why large objects go to **Gen2 immediately** and skip Gen0/Gen1.
- Describe **LOH fragmentation** — free holes, no compaction (historically; know modern changes).
- Distinguish **SOH compaction** vs **LOH non-compact** behavior and impact on memory.
- Identify workloads that stress LOH: **large arrays**, **MemoryStream** buffers, **string** builds.
- Apply mitigation: **ArrayPool**, **recyclable memory streams**, **chunking**, **POH** for pinning.

## Why This Matters

In `1.5.5`–`1.5.7` you studied SOH generational allocation. Objects **≥ 85,000 bytes** (contiguous allocation size — implementation default, subject to runtime) land on the **Large Object Heap**. LOH objects are **Gen2** immediately — collected only with **full GC**, expensive and less frequent.

Large **byte[]** for image processing, serialization buffers, or ORM materialization cause **fragmentation**: free memory exists but not contiguous — **`OutOfMemoryException`** while total free bytes seem sufficient. ASP.NET Core apps streaming multi-MB JSON hit LOH unless pooled.

Middle developers recognize LOH in **PerfView**, use **`RecyclableMemoryStreamManager`**, and understand **.NET 4.5.1+ LOH compaction** (optional, not default always-on).

## Core Concepts

### What Goes to LOH?

**Single allocation ≥ ~85,000 bytes** on 64-bit (85000 bytes threshold for arrays — `new byte[85000]` LOH; `84999` SOH Gen0). Applies to:

- Large **arrays** (`byte[]`, `object[]`, etc.).
- Very large **strings** ( contiguous char storage).
- Some internal buffers crossing threshold.

**Collections of small objects** don't use LOH as one block — each element separate unless one array backs them.

LOH is **separate managed heap segment** from SOH ephemeral generations.

### Gen2 and Collection Behavior

LOH allocations recorded as **Gen2** — no Gen0/Gen1 promotion path (`1.5.7`). LOH collected when **Gen2 GC** runs — less often, longer mark (`1.5.6`).

After Gen2 collection, **unreachable LOH objects freed** but memory may remain **fragmented** — gaps between surviving large blocks.

### Fragmentation Explained

```
LOH addresses: [Used 1MB][Free 500KB][Used 2MB][Free 300KB]
Need new 600KB contiguous → fail OOM though 800KB free total
```

**Compaction** on SOH moves objects to consolidate free space. **LOH historically did not compact** — fragmentation accumulates in long-running processes.

**.NET 4.5.1+**: **LOH compaction** available via `GCSettings.LargeObjectHeapCompactionMode` or `GC.Collect` with compaction — **expensive**; used sparingly (e.g., after large batch freed).

**.NET 5+ POH** — pinned allocations on **Pinned Object Heap** reduce SOH/LOH pinning fragmentation (related).

### Pinning Interaction

**Pinned** LOH objects (`GCHandleType.Pinned`) worsen fragmentation — can't move during compact even when compaction enabled for LOH. Prefer **POH** or **unpinned pooled buffers** when possible.

## Under the Hood

### Why 85KB Threshold?

Large objects expensive to copy during **compaction** — moving 1MB blocks costly. Placing them on LOH avoids copying on Gen0/Gen1 collections. Trade-off: fragmentation risk and Gen2-only reclamation.

### LOH Free List

Allocator searches **free list** for fitting block — **best fit** heuristics; fragmentation if size classes mismatch repeated alloc/free patterns of varying sizes.

### DATAS and Heap Hard Limits

Modern GC **DATAS** adjusts heap limits in containers — LOH counts toward total; OOM under cgroup limit still possible with fragmentation.

## Syntax / Commands / API

```csharp
// LOH allocation
var big = new byte[100_000]; // LOH

// Check generation
Console.WriteLine(GC.GetGeneration(big)); // 2

// Compaction mode (use carefully)
GCSettings.LargeObjectHeapCompactionMode = GCLargeObjectHeapCompactionMode.CompactOnce;
GC.Collect(); // triggers compact per mode

// Mitigation — ArrayPool (SOH chunks, not one LOH block)
var pool = ArrayPool<byte>.Shared;
var buffer = pool.Rent(100_000); // typically < LOH per rent size strategy
try { /* use */ }
finally { pool.Return(buffer); }
```

**RecyclableMemoryStream** (Microsoft.IO.RecyclableMemoryStream) — pools large buffers instead of LOH churn.

Diagnostics:

```bash
dotnet-counters --counters System.Runtime[loh-size,gc-heap-size]
# PerfView GCStats LOH allocations
```

## Examples

### MemoryStream Anti-pattern

```csharp
using var ms = new MemoryStream();
// Write 10MB — internal byte[] may LOH, discarded each request
```

Replace with pooled/recyclable stream — reuse buffers across requests.

### JSON Serialization

Deserializing large payload to `JsonDocument` or `byte[]` — LOH. Use streaming APIs (`Utf8JsonReader`) to process without single giant buffer when possible.

### String Concatenation in Loop

Many large intermediate strings — multiple LOH objects until GC. **`StringBuilder`** or join with known size reduces peaks.

## Common Mistakes & Anti-patterns

**Assuming free memory = can allocate** — fragmentation blocks large alloc.

**85000 vs 85001 off-by-one** in tests — know exact threshold behavior on your runtime.

**ArrayPool rent without Return** — leak + pool starvation.

**Forcing LOH compact every request** — catastrophic pause times.

**Single giant `List<T>` growth** — internal array resizes copy; final array may hit LOH.

---

## Production & Real-World Notes

**Long-running Windows Services / Linux daemons** — monitor LOH size trend over days; upward slope with flat traffic → leak or fragmentation.

**Image/file upload endpoints** — stream to disk/temp, don't buffer entire file in LOH `byte[]`.

**Entity Framework** — large result sets materialized to List — memory + LOH; pagination and projection.

**.NET GC tuning in containers** — `DOTNET_GCHeapHardLimit` family — coordinate with K8s limits.

## Comparison / Trade-offs

| Strategy | LOH impact | Complexity |
|----------|------------|------------|
| Single large array | High LOH | Simple |
| Chunked buffers | SOH chunks | Medium |
| ArrayPool | Reuse, smaller peaks | Must Return |
| Streaming API | Minimal large alloc | API design |
| LOH compact | Fixes fragmentation | Long pause |

## Key Takeaways

- **Large allocations** → **LOH**, **Gen2** immediately (`1.5.7`).
- **LOH GC** tied to full collections — expensive, infrequent.
- **Fragmentation** causes OOM despite free bytes — holes not contiguous.
- **Compaction** helps LOH but costly — prevention beats cure.
- **ArrayPool / recyclable streams** are production defaults for buffers.
- Completes **`1.1.6` heap story** with SOH vs LOH split.
