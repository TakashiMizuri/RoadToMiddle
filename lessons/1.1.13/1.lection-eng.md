# ref, out, in, params — Semantics

## Why This Matters

In lesson `1.1.12` you learned how methods accept parameters and return values. By default, C# passes arguments **by value**: the callee receives a copy of the bits that represent the argument. For small value types like `int`, copying is cheap. For large structs — a `Matrix4x4`, a custom `Point3D` with many fields — copying on every call adds up in hot loops. For reference types, the copy is only the pointer (eight bytes on 64-bit), but the callee still cannot replace the caller's variable unless you pass the **address** of that variable.

C# gives you four related tools — **`ref`**, **`out`**, **`in`**, and **`params`** — that change how arguments cross the method boundary. They look like small syntactic keywords, but they encode **contracts** between caller and callee. Misusing them produces compile errors (good) or subtle bugs (bad): reading an unassigned `out` parameter, mutating through `in`, or assuming `params` is allocation-free when it is not.

Middle developers encounter these modifiers daily in framework APIs: `int.TryParse(..., out int result)`, `Interlocked.CompareExchange(ref location, ...)`, `ReadOnlySpan<T>`-style APIs that take `in T` for large structs, and logging helpers with `params object[]`. Understanding semantics — not just memorizing signatures — lets you read BCL source confidently and design methods that express intent clearly.

## Core Concepts

### Pass by Value: The Default Mental Model

When you write `void Increment(int x)` and call `Increment(n)`, the runtime copies the value of `n` into `x`. Changes to `x` inside the method do not affect `n`. For reference types, the copy is the reference itself: both caller and callee point at the same object on the heap, so mutating **object state** (`list.Add(1)`) is visible to the caller, but reassigning the parameter (`list = new List<int>()`) is not.

This distinction — **mutating the object** versus **reassigning the variable** — is the foundation for understanding why `ref` exists. Without `ref`, a method cannot make the caller's variable refer to a different object or hold a different value type value.

### ref: Aliasing the Caller's Storage

The **`ref`** modifier means "pass an alias to the caller's variable." Both the parameter and the argument refer to the **same storage location**. If the callee assigns to the parameter, the caller sees the new value immediately.

```csharp
static void Swap(ref int a, ref int b)
{
    int temp = a;
    a = b;
    b = temp;
}

int x = 1, y = 2;
Swap(ref x, ref y);
// x == 2, y == 1
```

**Definite assignment rule for `ref`:** the argument must be a **definitely assigned** variable, field, or array element before the call. You cannot pass `ref` to an uninitialized local. You also cannot pass `ref` to a property or most expressions — only to **variables** (`ref` returns from `1.1.11` are the exception on the return side).

Use `ref` when the method must **read and write** the same storage the caller owns: swap helpers, interlocked operations, ref struct returns passed back in, or avoiding copy of a large struct when mutation is intended.

### out: Output Slots the Caller Need Not Initialize

**`out`** is also pass-by-reference, but the contract differs. The caller may pass an **unassigned** variable; the callee **must assign** the parameter before every return path (including exceptions — the compiler enforces assignment on normal returns). The callee should treat the incoming value as **uninitialized garbage** and not read it before writing.

```csharp
static bool TryDivide(int numerator, int denominator, out int quotient)
{
    if (denominator == 0)
    {
        quotient = 0; // still required for compiler; value is discarded on false
        return false;
    }
    quotient = numerator / denominator;
    return true;
}

if (TryDivide(10, 2, out int result))
    Console.WriteLine(result); // 5
```

Since C# 7, **`out` variables** can be declared inline in the argument list (`out int result`), which is idiomatic for `Try*` patterns.

**`ref` vs `out` at the call site:** both use `ref` at the IL level (byref), but the compiler tracks definite assignment differently. Choose **`out`** when the parameter exists only to **return** a value alongside a status flag or error code. Choose **`ref`** when the method **reads** the current value and may update it.

### in: Read-Only Aliasing for Efficiency

**`in`** passes by reference but forbids the callee from modifying the storage (except through explicit unsafe code). For **large readonly structs**, `in T` avoids copying while preventing accidental mutation — a compromise between performance and safety.

```csharp
readonly struct LargePayload
{
    public readonly double A, B, C, D, E, F;
    // ...
}

static double Sum(in LargePayload p) =>
    p.A + p.B + p.C + p.D + p.E + p.F;

var payload = new LargePayload { /* ... */ };
double total = Sum(in payload); // 'in' optional at call site for variables
```

At the call site, **`in` is optional** for variables: `Sum(payload)` implicitly passes `in payload`. For **`readonly struct`** parameters, the compiler may apply `in` automatically in some contexts to avoid defensive copies (a compiler optimization subject to change).

Use **`in`** for **read-only** access to large structs in performance-sensitive code. Do **not** use `in` for small structs like `Point` — copying four bytes is often cheaper than indirection and prevents other optimizations.

**Restriction:** `in` parameters cannot be modified, and you cannot pass them where a `ref` is expected. **`ref struct`** types (e.g., `Span<T>`) have additional rules — they cannot be stored on the heap or used as generic type parameters in many scenarios.

### params: Variable Argument Count, Fixed Signature

**`params`** lets the last parameter accept zero or more arguments of a specified array element type. The compiler collects extra arguments into a **single array** passed to the method.

```csharp
static int Sum(params int[] values)
{
    int total = 0;
    foreach (int v in values)
        total += v;
    return total;
}

Sum();           // values = empty array
Sum(1, 2, 3);    // values = new int[] { 1, 2, 3 }
Sum(new[] { 4, 5 }); // explicit array — no double wrapping
```

**Rules:** `params` must be the **last** parameter; only **one** `params` per method; the parameter type must be a **single-dimensional array**. Overloads without `params` may be preferred when the caller already has an array — passing `existingArray` directly avoids the compiler creating a new array for the call site in some cases, but when you write `Method(1, 2, 3)`, the compiler emits code that allocates `int[]`.

**`params ReadOnlySpan<T>`** (C# 13 / .NET 9+) reduces allocations for many call patterns by passing stack-friendly spans where supported — check your target framework before relying on it in library APIs.

## Under the Hood

### IL and the Managed Pointer

At the IL level, ordinary value parameters arrive on the evaluation stack by value. **`ref`**, **`out`**, and **`in`** parameters are marked **`byref`** in the method signature. The caller emits **`ldloca`** (load local address) or equivalent to pass a **managed pointer** to the callee's slot. Inside the callee, **`ldind`** / **`stind`** (indirect load/store) read and write through that pointer.

This is not C++ pointers: the GC tracks managed pointers and may relocate objects during compaction, so raw pointer arithmetic is forbidden in safe code. The runtime ensures that byref parameters remain valid for the duration of the call.

**`out`** and **`ref`** share the same IL calling convention; the difference is entirely **compile-time definite assignment analysis**. **`in`** adds **`readonly`** semantics in metadata (modreq on the parameter), which the JIT respects when optimizing.

### Definite Assignment Analysis

The C# compiler builds a **dataflow graph** for each method body. For **`out`** parameters, every control-flow path to `return` must include an assignment. For **`ref`** parameters, the compiler assumes the incoming value is valid for read. This is why switching a `TryParse`-style API from `out` to `ref` would incorrectly allow callers to pass uninitialized variables — the API contract changed.

### params and the Compiler

When you call `Log("a", "b", "c")` against `void Log(params string[] args)`, the compiler synthesizes:

```csharp
Log(new string[] { "a", "b", "c" });
```

That implies a **heap allocation** for the array (unless optimized away by the JIT in very specific scenarios — do not depend on elision). High-performance code paths sometimes offer overloads for `(T, T)` or `ReadOnlySpan<T>` instead of `params` to control allocations.

### ref Returns and ref locals (Connection to 1.1.11)

**`ref` returns** let a method return an alias to caller-owned storage (e.g., `ref T Item(int index)` on a struct buffer). **`ref` parameters** and **`ref` returns** compose: you can pass a `ref` return to another `ref` parameter. The same managed-pointer rules apply; lifetime must not extend beyond the underlying storage — returning `ref` to a local variable is a compile error because the local dies at method exit.

## Syntax / Commands / API

| Modifier | Caller must initialize? | Callee may read before assign? | Callee must assign before return? |
|----------|-------------------------|----------------------------------|-----------------------------------|
| (none)   | Yes                     | Yes                              | N/A                               |
| `ref`    | Yes                     | Yes                              | No                                |
| `out`    | No                      | No                               | Yes                               |
| `in`     | Yes                     | Yes (read-only)                  | No                                |

```csharp
// ref — read/write alias
void Update(ref int value) => value++;

// out — output only
bool TryGet(out int value) { value = 42; return true; }

// in — read-only alias
void Print(in DateTime dt) => Console.WriteLine(dt);

// params — variadic last parameter
void M(int x, params string[] rest) { }
```

## Examples

### Example 1: Try Pattern with out

The BCL uses `out` for operations that may fail without exceptions:

```csharp
if (int.TryParse(userInput, out int parsed))
    ApplyDiscount(parsed);
else
    ShowValidationError();
```

Here `parsed` is **definitely assigned** only inside the `if` block when `TryParse` returns `true`. The compiler's flow analysis understands the `Try*` pattern.

### Example 2: ref for In-Place Mutation

```csharp
static void Normalize(ref Vector3 v)
{
    float len = MathF.Sqrt(v.X * v.X + v.Y * v.Y + v.Z * v.Z);
    if (len > 0)
    {
        v.X /= len;
        v.Y /= len;
        v.Z /= len;
    }
}
```

Returning a new `Vector3` would work but adds copying; **`ref`** documents that the caller's instance is mutated in place.

### Example 3: in to Avoid Struct Copy

```csharp
static bool Equals(in Guid a, in Guid b) =>
    a == b; // reads only; no copy of 16-byte struct if passed by in
```

For tiny structs the benefit is negligible; for methods called millions of times per second on large readonly structs, **`in`** can matter.

## Common Mistakes & Anti-patterns

**Reading an `out` parameter before assignment** inside the callee is a logic bug; the compiler may not catch reads on all paths. Treat **`out`** as write-only until assigned.

**Using `ref` for everything "for performance"** often hurts readability and blocks inlining. Prefer return values or tuples unless aliasing is required.

**Assuming `params` is free** — each variadic call may allocate an array. In tight loops, pass `ReadOnlySpan<T>` or an explicit pre-allocated array.

**Passing `ref` to `readonly` fields** is restricted; immutability guarantees would break. Use patterns appropriate to your domain object.

**Confusing `in` with `const`** — `in` prevents the callee from reassigning the parameter slot; it does not deep-freeze mutable objects referenced by the struct.

## Production & Real-World Notes

Framework designers use **`out`** extensively for **`Try*`** methods to avoid exception cost on expected failure paths. **`ref`** appears in low-level threading (`Interlocked`), pooling, and high-performance collections. **`in`** is common in modern numerics and game math libraries shipping **`readonly struct`** types.

When exposing public APIs, **`out`** for optional outputs and **return tuples** `(bool success, T value)` are often clearer than multiple **`out`** parameters. **`params`** is convenient for logging and `string.Format`-style helpers but is a poor choice for core hot-path APIs without overloads.

Roslyn analyzers and nullable reference types interact with these modifiers: **`out`** parameters are considered **assigned** after the call when the method returns `true` in well-known patterns; custom Try methods may need `[NotNullWhen(true)]` attributes for flow analysis.

---

## Comparison / Trade-offs

**Return value vs `out`:** Returning `(bool, T)` or a dedicated result type is often more functional and composable with LINQ; **`out`** matches established BCL style and avoids allocating result wrappers for primitives.

**`ref` vs returning struct:** For small mutable structs, returning the updated copy is simple and avoids `ref` noise. For large structs or when the caller must update a specific slot in an array, **`ref`** wins.

**`in` vs pass by value:** Measure. JIT tier-1 optimizations sometimes make copies cheap. **`in`** shines for readonly structs above ~24–32 bytes called in loops — exact threshold is platform-dependent.

**`params` vs overloads:** Overloads `(T)`, `(T, T)`, `(T, T, T)` plus `ReadOnlySpan<T>` eliminate allocations at the cost of API surface area — a trade-off libraries like `MemoryExtensions` make explicitly.

## Key Takeaways

- C# defaults to **pass by value**; `ref`/`out`/`in` pass **managed pointers** (byref) with different assignment contracts.
- **`ref`** = read + write; argument must be initialized.
- **`out`** = write-only output; callee must assign; caller need not initialize.
- **`in`** = read-only reference; avoids copying large readonly structs.
- **`params`** collects trailing arguments into an array — often allocates.
- IL treats `ref` and `out` similarly; semantics are enforced by the compiler's definite assignment analysis.
- Choose modifiers for **API clarity** first, micro-optimization second.
