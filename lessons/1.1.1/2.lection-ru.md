## Why This Matters

Фаза 0 дала инструменты вокруг разработки — Git, HTTP, HTML. Фаза 1 начинается с языка и runtime для backend: **C# на .NET**. Прежде чем разбирать типы, память или async, нужен воспроизводимый способ создавать проекты, собирать и запускать их локально и в CI.

**.NET SDK** — не просто компилятор. Это toolchain: шаблоны проектов, NuGet, MSBuild, test runner, publish — всё через CLI **`dotnet`**. В командах не пишут `.csproj` с нуля каждый спринт: делают `dotnet new`, коммитят результат, а **`dotnet build`** вызывает MSBuild и даёт предсказуемый output. Понимание структуры проекта сейчас сэкономит часы, когда появятся ASP.NET Core, EF Core migrations и Docker с `COPY bin/Release/publish/`.

Если представлять «программу на C#» как один `.cs` и вызов `csc.exe`, будет трудно в реальных solution с десятками проектов и общим `Directory.Build.props`. Console template намеренно минимален — один project, одна точка входа — но уже показывает контракт любого .NET-приложения: **исходники → build → assembly → выполнение на runtime**.

## Core Concepts

### Что именно устанавливается

С [https://dotnet.microsoft.com/download](https://dotnet.microsoft.com/download) ставится пакет, в котором обычно есть:

- **SDK** — компиляторы (**Roslyn**), MSBuild, шаблоны, CLI. Нужен, чтобы **собирать** и **публиковать**.
- **Runtime** — CLR и базовые библиотеки. Нужен, чтобы **запускать** приложения. На сервере production часто только runtime.
- **ASP.NET Core Runtime** — хостинг веб-стека (Kestrel). Для console не обязателен, но входит в полный SDK.

После установки выполни `dotnet --info`. Увидишь версии SDK, runtime и **host** — исполняемый `dotnet`, который подбирает runtime под TFM приложения.

**Target framework moniker (TFM)**, например `net8.0`, объявляет, какой API и runtime ожидает проект. В `.csproj`: `<TargetFramework>net8.0</TargetFramework>`. Если собрали под `net8.0`, а на машине только .NET 6 — ошибка при запуске, не «тихая» поломка.

### CLI `dotnet` как оркестратор

Каждая команда делегирует работу MSBuild, NuGet или host:

- **`dotnet new`** — scaffold из шаблонов.
- **`dotnet restore`** — загрузка NuGet-зависимостей (часто неявно при build).
- **`dotnet build`** — компиляция в `bin/<Configuration>/<TFM>/`.
- **`dotnet run`** — build при необходимости + запуск.
- **`dotnet test`** — тесты (позже в roadmap).

CLI кроссплатформенный: те же команды на Windows, macOS, Linux — основа Docker и CI.

### Первое консольное приложение

```bash
dotnet new console -n HelloRoadmap -o HelloRoadmap
cd HelloRoadmap
dotnet run
```

`-n` — имя проекта (и default namespace). `-o` — папка. Без `-o` файлы создаются в текущей директории.

Ожидаемый вывод: `Hello, World!`. Его печатает шаблонный `Program.cs` с **top-level statements** (C# 9+): файл исполняемый без явного `Main`, хотя компилятор генерирует entry point.

Типичный **SDK-style** console project:

| Path | Роль |
|------|------|
| `HelloRoadmap.csproj` | Манифест — TFM, зависимости, настройки сборки |
| `Program.cs` | Точка входа |
| `obj/` | Промежуточные артефакты (не коммитить) |
| `bin/` | Собранный output (не коммитить) |

В modern project нет `packages.config` — зависимости через **`PackageReference`** в `.csproj`, кэш NuGet глобальный (`%USERPROFILE%\.nuget\packages`).

### Анатомия `.csproj`

Короткий файл благодаря импорту **`Microsoft.NET.Sdk`**:

```xml
<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net8.0</TargetFramework>
    <ImplicitUsings>enable</ImplicitUsings>
    <Nullable>enable</Nullable>
  </PropertyGroup>

</Project>
```

**`OutputType`**: `Exe` — console, `Library` — class library. **`ImplicitUsings`** добавляет общие `using` (`System`, `System.Linq`, …). **`Nullable`** включает nullable reference types — стандарт для нового кода.

Добавление пакета:

```bash
dotnet add package Newtonsoft.Json
```

Появится `<PackageReference … />`; restore — при следующем build.

### Build output и execution flow

При `dotnet build`:

1. MSBuild читает `.csproj` и SDK targets.
2. **Restore** подтягивает пакеты.
3. **Roslyn** компилирует `.cs` в IL → assembly (`HelloRoadmap.dll`; на Windows ещё host `.exe`).
4. Артефакты в `bin/Debug/net8.0/` (по умолчанию **Debug**).

`dotnet run` ≈ build + запуск `HelloRoadmap.dll`. Release: `dotnet run -c Release`.

**Entry point** — сгенерированный или явный `Main`. Top-level statements в `Program.cs` превращаются в `Main` во внутреннем классе компилятора.

### Solution vs project (preview)

**Project** (`.csproj`) — одна единица сборки. **Solution** (`.sln`) — группа projects (API, domain, tests). Скоро: `dotnet new sln`, `dotnet sln add`. Для `1.1.1` достаточно одного console project: **solution = контейнер workspace, project = unit сборки**.

## Under the Hood

### MSBuild и цепочка SDK imports

Строка `<Project Sdk="Microsoft.NET.Sdk">` разворачивается в imports из `{DOTNET_ROOT}/sdk/{version}/Sdks/.../Sdk.props` и `Sdk.targets`. Там сотни defaults: какие `.cs` компилировать, куда класть output, как работает `publish`.

Обычно imports не трогают — переопределяют через `<PropertyGroup>`. В monorepo часто **`Directory.Build.props`** с `<Nullable>enable</Nullable>` для всех projects.

### Pipeline компиляции Roslyn (упрощённо)

Исходный текст → **syntax trees** → **semantic model** → **IL emit** → assembly. Ошибки типов — на этапе compile. CLR **JIT**-компилирует IL в native code при запуске (кроме Native AOT — позже).

Runtime находит assembly с entry point, JIT-ит `Main`, выполняет. `Console.WriteLine` — из shared framework `System.Console.dll`, поставляется с runtime при framework-dependent deploy.

## Syntax / Commands / API

```bash
dotnet --info
dotnet --version

dotnet new console -n MyApp -o MyApp
dotnet new list
dotnet build
dotnet build -c Release
dotnet run
dotnet run -- arg1 arg2

dotnet clean
dotnet new globaljson --sdk-version 8.0.100
```

Шаблонный `Program.cs`:

```csharp
Console.WriteLine("Hello, World!");
```

Эквивалент с явным `Main`:

```csharp
namespace HelloRoadmap;

internal static class Program
{
    static void Main(string[] args)
    {
        Console.WriteLine("Hello, World!");
    }
}
```

## Common Mistakes & Anti-patterns

**Только runtime без SDK** — нет `dotnet new` / `dotnet build`. Разработчикам нужен **SDK**.

**Коммит `bin/` и `obj/`** — раздувает repo. Добавь в `.gitignore`.

**`dotnet run` не из папки project** — укажи `--project path/to/App.csproj` или `cd` в каталог с `.csproj`.

**Путать версию SDK и TFM** — SDK 8.0 может таргетить `net6.0`, `net7.0`, `net8.0`.

## Production & Real-World Notes

CI: `restore` → `build -c Release` → `test` → `publish -o ./publish`. **Publish** ≠ `bin/Debug`.

**`global.json`** фиксирует SDK для команды и CI. Docker: stage build на `sdk:8.0`, runtime image на `runtime:8.0` или `aspnet:8.0`.

## Comparison / Trade-offs

**Framework-dependent** — меньше размер, нужен runtime на сервере. **Self-contained** — runtime в bundle, больше размер. Console tools и microservices выбирают по ops.

**Top-level statements** vs явный `Main` — меньше boilerplate vs явная структура в libraries.

## Key Takeaways

- **SDK** = compiler + MSBuild + templates + CLI.
- **`dotnet new console`** → `.csproj` + `Program.cs`.
- **TFM** задаёт API/runtime; для FDD runtime должен быть на машине.
- Output в `bin/`; **`bin/`/`obj/`** не в Git.
- **`dotnet run`** = build + entry point.
- Top-level statements → compiler-generated `Main`.
