## Answers

### 1. [Recall]
**Answer:** `Content-Type` объявляет **media type** (MIME) тела сообщения — как интерпретировать bytes (JSON, form, multipart и т.д.), опционально с `charset`.
**Why:** Без этого получатель не знает, какой parser/deserializer применить.

### 2. [Recall]
**Answer:** **`application/x-www-form-urlencoded`**
**Why:** Это default `enctype` для HTML forms по спецификации.

### 3. [Understanding]
**Answer:** OAuth 2.0 (RFC 6749) **нормативно** требует token request body в form-urlencoded; JSON не описан как допустимый формат для token endpoint.
**Why:** Историческая совместимость с form POST и единообразие identity providers.
**Common mistake:** Отправлять JSON в Postman «потому что API везде JSON».

### 4. [Understanding]
**Answer:** Multipart передаёт file как **raw binary part** с отдельными headers; urlencoded — только text key=value, binary пришлось бы кодировать (Base64), что раздувает размер.
**Why:** Boundaries позволяют смешать text fields и binary streams в одном request.

### 5. [Understanding]
**Answer:** `[FromBody]` использует **JSON input formatter**; urlencoded body парсит **form value provider** / `[FromForm]`. Formatter не совпадает → model остаётся default/null.
**Why:** Binding pipeline выбирается по content type и attribute.

### 6. [Application]
**Answer:** **`Content-Type: application/json`**, параметр/action с **`[FromBody] CreateTaskDto`** (массив `tags` в JSON).
**Why:** Nested structure и REST convention; urlencoded для массивов неудобен и нестандартен.

### 7. [Application]
**Answer:** Form: **`enctype="multipart/form-data"`**; request: **`Content-Type: multipart/form-data; boundary=...`**
**Why:** Browser автоматически переключается на multipart при file input.

### 8. [Analysis]
**Answer:** **B** неверно — в urlencoded пробел может быть **`+`** или **`%20`**.
**Why:** `+` → space — правило application/x-www-form-urlencoded, не JSON.

### 9. [Analysis]
**Answer:** (1) Client шлёт media type, который endpoint **не поддерживает** (например XML, а API только JSON). (2) **`Content-Type` отсутствует** или неверный, server/formatters не находят match → 415.
**Why:** ASP.NET Core и многие frameworks явно возвращают 415 при unsupported formatter.

### 10. [Recall]
**Answer:** Client сообщает, **какие форматы response** он готов принять (например `application/json`).
**Why:** Content negotiation — server может выбрать serializer (JSON vs XML).

### 11. [Application]
**Answer:** `email=alex%40test.com&password=hello+world` или `password=hello%20world`
**Why:** `@` → `%40`; пробел → `+` или `%20` в urlencoded.

### 12. [Understanding]
**Answer:** Base64 ~+33% size; весь файл в memory при parse; нет streaming file part; логи и WAF тяжелее.
**Why:** Multipart designed для mixed field + binary upload.
