# Task.Run — offload CPU, не I/O

> Roadmap: `1.4.4` · Node: `1.4` — C# async ⚡ · Depth: **глубоко**

## Learning Objectives

- Назначение **`Task.Run`**: CPU-bound work в thread pool.
- Почему **`Task.Run` + sync I/O** — anti-pattern (fake async).
- **`Task.Run`** vs **`StartNew`** vs inline.
- **`TaskCreationOptions.LongRunning`** — когда редко.
- Bounded concurrency против **starvation**.
- **`await Task.Run`** без sync block request thread в queue wait.

---

## Why This Matters

`Task.Run` — один из самых misused API в ASP.NET. Оборачивают DB «чтобы async», добавляя overhead и **всё ещё blocking pool** на I/O внутри delegate.

Из `1.4.2`: I/O — true async; CPU — parallelism. **`Task.Run`** снимает CPU с request thread path через `await`, но delegate жжёт pool worker на CPU. Правильно — responsiveness pipeline; неправильно — double pool pressure.

---

## Core Concepts

### Что делает Task.Run

**`Task.Run`** ставит delegate в **thread pool**, возвращает Task. Не magic parallelism — scheduling.

```csharp
var hash = await Task.Run(() => BCrypt.HashPassword(pwd));
```

До старта delegate request может yield; во время BCrypt — worker 100% busy.

### Почему не для I/O

```csharp
await Task.Run(() => File.ReadAllText(path)); // WRONG
```

Pool thread blocked на весь read. Тот же blocked count + overhead. **`ReadAllTextAsync`**.

### Task.Run vs StartNew

**Prefer Task.Run.** StartNew — confusing defaults (`1.4.24`). **LongRunning** — dedicated thread для very long work; в web rare — **BackgroundService**.

### Bounding

Unbounded Task.Run × requests → starvation. **SemaphoreSlim**, **Parallel.ForEachAsync** с limit.

---

## Under the Hood

Queue → worker → completion → continuation. **ExecutionContext** flows (`AsyncLocal`).

Request уже на pool thread; **`await Task.Run`** continuation may be any worker; ASP.NET **HttpContext** flows.

Tiny work — inline sync, Task.Run overhead > benefit.

## Examples

### Пример 1: CPU offload на API

```csharp
public async Task<IActionResult> HashPassword(string pwd)
{
    var hash = await Task.Run(() => BCrypt.HashPassword(pwd));
    return Ok(hash);
}
```

Между queue и стартом delegate request thread может yield. Во время BCrypt worker занят CPU — это нормально для CPU-bound (`1.4.2`).

### Пример 2: Fake async для I/O

```csharp
await Task.Run(() => File.ReadAllText(path)); // WRONG
await File.ReadAllTextAsync(path);            // RIGHT
```

Первый вариант блокирует pool worker на весь disk wait — тот же starvation, плюс overhead scheduling.

### Пример 3: Legacy sync API

Нет async API у vendor — `Task.Run` + **SemaphoreSlim** для limit concurrency. Tactical debt: логировать, планировать замену.

## Common Mistakes & Anti-patterns

**Task.Run «because async controller»** — signature async, смысл sync; pool pressure без I/O benefit.

**Async lambda:** `Task.Run(async () => await Foo())` — стартует на pool; первый await в lambda может освободить worker; для sync I/O внутри всё равно wrong tool.

**Nested Task.Run** — умножает queue depth.

**WPF vs ASP.NET** — UI нужен Dispatcher после Run; в ASP.NET Core SyncContext нет (`1.4.9`).

---

## Production & Real-World Notes

Если median CPU offload < 1 ms — убери Task.Run (overhead > benefit).

Heavy batch CPU — **Channel** + **BackgroundService**, не per-request Run.

Растущий **threadpool-queue-length** при CPU 100% — unbounded offload; throttle.

---

## Comparison / Trade-offs

| Сценарий | Инструмент |
|----------|------------|
| DB/HTTP/File | Native async |
| Heavy CPU на request | Bounded Task.Run |
| Long batch | BackgroundService |
| Micro work | Sync inline |
