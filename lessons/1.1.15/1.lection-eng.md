# Collections Intro: List, Dictionary, HashSet

> Roadmap: `1.1.15` · Node: `1.1` — C# Language Basics · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Use **`List<T>`** for ordered, growable sequences with indexed access.
- Use **`Dictionary<TKey, TValue>`** for key-based lookup with expected O(1) average time.
- Use **`HashSet<T>`** for unique elements and set operations.
- Explain how each collection is backed internally at a high level.
- Choose the right collection for a given access pattern.
- Avoid common pitfalls: **`KeyNotFoundException`**, modifying collection during enumeration, wrong equality comparer.

---

## Why This Matters

Lesson `1.1.14` showed **arrays**: fixed length, simple, fast. Real applications need **dynamic size**, **fast lookup by key**, and **uniqueness** — "all tags for this post," "user id → profile," "distinct visitor ids today." The BCL generic collections in **`System.Collections.Generic`** are the default tools before you reach for specialized libraries.

`List<T>`, `Dictionary<TKey,TValue>`, and `HashSet<T>` appear in every layer: ASP.NET Core model binding lists, EF Core tracking keys, React API payloads deserialized to `List<Dto>`, deduplication before insert. Middle developers must know **operations and complexity**, not every method — but enough to avoid scanning a `List<T>` linearly when a `Dictionary` would be constant time on average.

These three types share patterns: **generics** (type-safe, no boxing for value types), **enumerators** for `foreach`, **capacity** growth strategies, and **custom equality** via `IEqualityComparer<T>`. Mastering them unlocks LINQ (`1.x` later) because LINQ operates on **`IEnumerable<T>`**, which all three implement.

## Core Concepts

### List<T>: Dynamic Array Semantics

**`List<T>`** wraps a **`T[]`** buffer with a **`Count`** of used elements. **`Add`** amortized O(1) — when full, capacity **doubles** (typical strategy), copies to new array.

```csharp
var ids = new List<int> { 10, 20, 30 };
ids.Add(40);
ids[0] = 15;           // index access — O(1)
int n = ids.Count;     // 4
bool has = ids.Contains(20); // O(n) linear scan
```

**When to use:** ordered collection, duplicates allowed, index by position matters, sequential iteration dominates.

**Key members:** `Add`, `AddRange`, `Insert`, `Remove`, `RemoveAt`, `Clear`, `Sort`, `Find`, `IndexOf`, `Capacity` (reserve with `new List<T>(capacity)` to reduce reallocations).

**Insert/remove in middle** — O(n) because elements shift. Heavy middle insertion → consider **`LinkedList<T>`** (rare in business code) or different structure.

### Dictionary<TKey, TValue>: Hash Table

**`Dictionary<TKey,TValue>`** maps keys to values using a **hash table**. Average **O(1)** lookup, insert, delete; worst case O(n) if many hash collisions.

```csharp
var scores = new Dictionary<string, int>
{
    ["alice"] = 100,
    ["bob"] = 85
};

scores["alice"] = 110;              // update
scores.TryGetValue("carol", out int s); // false, s = 0

if (scores.ContainsKey("bob"))
    Console.WriteLine(scores["bob"]);
```

**Indexer `dict[key]`** throws **`KeyNotFoundException`** if missing — use **`TryGetValue`** or **`ContainsKey`** when unsure. **`TryGetValue`** is preferred (one lookup vs two).

**Keys must be unique.** Adding duplicate key throws. **`null` keys** forbidden for reference types unless using special comparers (default: not allowed).

**Order:** Until .NET Core 3+ / .NET 5+, order was undefined. Modern **`Dictionary`** preserves insertion order for enumeration — do not rely on order for correctness unless documented.

**When to use:** lookup by stable key (id, email, code), caches, counts (with `int` values), graphs adjacency (key → list).

### HashSet<T>: Unique Unordered Bag

**`HashSet<T>`** stores **unique** elements using hashing — like **`Dictionary<TKey, TValue>`** keys without values.

```csharp
var seen = new HashSet<string> { "GET", "POST" };
seen.Add("GET");   // false — duplicate rejected
seen.Add("PUT");   // true

bool ok = seen.Contains("POST"); // O(1) average
```

**Set operations:** `UnionWith`, `IntersectWith`, `ExceptWith`, `IsSubsetOf`, etc.

**When to use:** deduplication, membership tests, set algebra ("tags user A and B share").

**No indexing** — no `set[i]`. Unordered — do not assume iteration order for logic.

### Choosing Among the Three

```
Need                          → Collection
────────────────────────────────────────────
Order + duplicates + index    → List<T>
Fast get/put by key           → Dictionary<K,V>
Unique items, membership      → HashSet<T>
```

Often you **combine**: `List<Order>` for display order + `Dictionary<int, Order>` for lookup by id built once.

### Equality and Keys

**Dictionary** and **HashSet** depend on **`GetHashCode()`** and **`Equals()`** for `TKey` / `T`. Value types like **`int`**, **`Guid`** work out of the box. **Custom classes** need consistent override or pass **`IEqualityComparer<T>`** to constructor.

**Mutable keys are dangerous:** if a key object's hash code changes while stored, the entry becomes lost — **use immutable keys** (string, int, record with init-only properties).

## Under the Hood

**`List<T>`:** internal `T[] _items`, `int _size`. Grow: allocate double capacity, `Array.Copy`. Exposes **`Capacity`** ≥ **`Count`**.

**`Dictionary<TKey,TValue>`:** buckets array + entries array; open addressing or chaining depending on version — conceptually **hash → bucket → entry**. Load factor triggers resize/rehash.

**`HashSet<T>`:** similar hash table without values; uses **`Dictionary<TKey, TValue>`**-like implementation internally in .NET.

All three throw **`InvalidOperationException`** if collection modified during **`foreach`** (except through enumerator's own remove patterns where supported — `List<T>` doesn't support that; use **`for` backwards** or **`RemoveAll`**).

## Syntax / Commands / API

```csharp
// List
List<T> list = new();
list.Add(item);
list.Capacity = 1000; // optional pre-size

// Dictionary
Dictionary<TKey, TValue> map = new();
map[key] = value;
map.TryGetValue(key, out var v);

// HashSet
HashSet<T> set = new(StringComparer.OrdinalIgnoreCase);
set.Add(item);
```

**Initializers:** collection expressions (C# 12+) `[1, 2, 3]` for lists; dictionary `[("a", 1)]` forms vary by version — classic `{ ["a"] = 1 }` still common.

## Examples

### Example 1: Build index from list

```csharp
static Dictionary<int, string> IndexById(IEnumerable<(int Id, string Name)> rows)
{
    var map = new Dictionary<int, string>();
    foreach (var (id, name) in rows)
        map[id] = name; // overwrites duplicate ids — document behavior
    return map;
}
```

One pass O(n) → later lookups O(1) average.

### Example 2: Distinct tags

```csharp
var tags = new List<string> { "c#", "dotnet", "c#", "api" };
var unique = new HashSet<string>(tags, StringComparer.OrdinalIgnoreCase);
// unique.Count == 3
```

### Example 3: Safe dictionary read

```csharp
if (!users.TryGetValue(userId, out User? user))
    return NotFound();

return Ok(user);
```

Avoid double lookup: don't `ContainsKey` then `[key]`.

## Common Mistakes & Anti-patterns

**`dict[key]` without check** → exception on miss. Prefer **`TryGetValue`**.

**Using `List.Contains` + indexer in hot loop** for many lookups — build **`Dictionary`** once.

**Mutating key object** after insert into **`Dictionary`** / **`HashSet`**.

**Modifying collection during `foreach`** — invalid operation exception.

**Assuming `HashSet` preserves order** for business rules.

**Wrong comparer** — `"A"` and `"a"` as different keys unless `OrdinalIgnoreCase`.

## Production & Real-World Notes

API design: expose **`IReadOnlyList<T>`** / **`IReadOnlyDictionary<K,V>`** from methods; mutate internally with **`List`** / **`Dictionary`**.

ASP.NET Core JSON deserializes JSON objects to **`Dictionary<string, JsonElement>`** or strong types; arrays to **`List<T>`**.

For thread-safe scenarios, **`ConcurrentDictionary`** — not in this lesson — replaces **`Dictionary`** under concurrency.

Pre-size collections when count known: **`new List<T>(expectedCount)`**, **`new Dictionary<K,V>(capacity)`** reduces reallocations/rehashes.

## Comparison / Trade-offs

| vs | List | Dictionary | HashSet |
|----|------|------------|---------|
| Lookup by index | Yes | No | No |
| Lookup by key | Manual | Yes | N/A |
| Duplicates | Yes | Keys unique | No |
| Order meaningful | Yes | Insertion (modern) | Unordered |

**Array vs List:** array fixed, slightly less overhead; list dynamic — default for unknown size.

**SortedDictionary vs Dictionary:** sorted keys O(log n) — use when need ordered keys without sorting each time.

## Quick Reference

```
List<T>              — Add, [i], Count, O(1) index, O(n) Contains
Dictionary<K,V>      — TryGetValue, [key], unique keys
HashSet<T>           — Add, Contains, unique, set ops
```

---

## Key Takeaways

- **`List<T>`** = dynamic array; index + order; duplicates OK.
- **`Dictionary<K,V>`** = hash map; **`TryGetValue`** for safe get.
- **`HashSet<T>`** = unique hash bag; membership and set ops.
- Pick structure by **access pattern**, not habit.
- **Immutable keys** for hash-based collections.
- Do not modify while enumerating without safe patterns.
