# I/O-Bound vs CPU-Bound: Why Async Helps Only I/O

> Roadmap: `1.4.2` · Node: `1.4` — C# async ⚡ · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Classify work as **I/O-bound** or **CPU-bound** and justify the label.
- Explain why **async/await** improves scalability for I/O but not raw CPU throughput.
- Describe what happens to a thread during **async I/O** vs **CPU computation**.
- Choose between **async I/O**, **`Task.Run`**, and **`Parallel`** for a given problem.
- Recognize **pseudo-async** (async wrapper over sync I/O) and its cost.
- Connect thread pool behavior from `1.4.1` to production API design.

## Why This Matters

In `1.4.1` you learned that threads are expensive and the **thread pool** is shared. The natural question is: how do we serve 5,000 concurrent HTTP clients without 5,000 blocked threads? The answer depends entirely on whether your bottleneck is **waiting** or **computing**.

Misclassifying work leads to classic mistakes: wrapping synchronous database calls in `async` methods without true async APIs (still blocks a pool thread), or using `async` for heavy image processing (adds overhead without speeding CPU). Middle developers must classify every hot path — controller, service, repository — as I/O-bound or CPU-bound before choosing async, `Task.Run`, or parallel algorithms.

## Core Concepts

### I/O-Bound Work

**I/O-bound** work spends most of its time **waiting** for external systems: network (HTTP, TCP), disk, database, message queue, GPU driver. The CPU is largely idle during the wait; the thread, if blocked synchronously, sits in the kernel wait queue doing nothing useful for your process.

Examples in a typical ASP.NET API: `await httpClient.GetAsync(...)`, `await dbContext.SaveChangesAsync()`, `await File.ReadAllBytesAsync(...)`. The operation duration is dominated by RTT, disk seek, or DB lock — not instruction throughput.

**Goal for I/O-bound:** maximize **throughput** (requests/sec) by **not tying up threads** while waiting. `async/await` registers a continuation when I/O completes; the thread returns to the pool to serve other requests (`1.4.6`–`1.4.7` cover how).

### CPU-Bound Work

**CPU-bound** work spends time **executing instructions**: hashing passwords, compressing JSON, ray tracing, complex LINQ on large in-memory collections, cryptography. Waiting is minimal; adding threads (up to core count) can increase **parallel throughput**; beyond cores, you gain contention, not speed.

Examples: `BCrypt.Verify`, `JsonSerializer.Serialize` on huge graphs, image resize in pure C#. **`async` does not make CPU work faster** — it may add state machine overhead. For CPU-bound work on a request thread, offload with **`Task.Run`** or **`Parallel.For`** to thread pool workers (`1.4.4`), or use a dedicated worker queue — but you cannot exceed physical cores meaningfully.

### The Utilization Picture

Imagine one thread timeline:

**Sync I/O:** `[CPU setup][======== WAIT ========][CPU resume]` — thread blocked entire wait.

**Async I/O:** `[CPU setup][thread free for others][continuation on pool thread]` — wait does not consume a running worker.

**CPU-bound:** `[======== COMPUTE ========]` — thread busy entire time; only solution is faster algorithm, more cores, or parallel split.

Under load, 1,000 sync I/O calls need ~1,000 blocked threads. 1,000 async I/O calls need far fewer threads active at once — only those momentarily executing continuations.

### Latency vs Throughput

**Async improves throughput** under concurrency for I/O; it does not reduce single-request latency of the I/O itself (network still takes 50 ms). One password hash still takes 100 ms CPU whether wrapped in `async` or not.

Confusing these leads to "I made it async so it should be faster" for a single CPU-heavy endpoint — false.

## Under the Hood

### Blocking vs Non-Blocking I/O at the OS

When you call synchronous `socket.Read`, the thread enters a **blocking syscall**; the kernel marks the thread not runnable until data arrives. With **overlapped/async I/O** (Windows IOCP, Linux `epoll`/`io_uring`), the operation starts and the thread returns immediately; completion is signaled later. .NET's `Socket.ReadAsync`, `FileStream.ReadAsync`, Npgsql async commands use this model — true async from hardware wait to continuation.

**Fake async:** `Task.Run(() => stream.Read(...))` moves blocking to another pool thread — does not reduce blocked threads system-wide; sometimes worse (extra scheduling).

### ASP.NET Core Request Thread Model

A request starts on a pool thread. Sync I/O holds it for the full wait. Async I/O releases it at `await` until completion callback. Kestrel can accept more connections than you have threads because waiting is not 1:1 with blocked workers.

### CPU Saturation

When all cores run at 100% CPU-bound work, adding async does not create capacity — you need **scale out** (more machines), **optimize algorithm**, or **reduce work**. Thread pool queue grows if you queue unbounded CPU tasks.

### Decision Flow

```
Is work waiting on external I/O?
  YES → true async API (await ReadAsync, EF async, HttpClient)
  NO  → CPU-bound
        Need parallel on server?
          YES → Task.Run / Parallel (bounded!) / dedicated workers
          NO  → stay sync on caller thread
```

## Syntax / Commands / API

```csharp
// I/O-bound — true async
await httpClient.GetStringAsync(url);
await dbContext.Users.ToListAsync();

// CPU-bound offload — not for I/O!
await Task.Run(() => BCrypt.HashPassword(password));

// Anti-pattern — fake async
public async Task<string> GetBadAsync()
{
    return await Task.Run(() => File.ReadAllText(path)); // still blocks A pool thread
}

// Better fake-async fix — expose sync honestly or use ReadAllTextAsync
public async Task<string> GetGoodAsync()
    => await File.ReadAllTextAsync(path);
```

## Examples

### Example 1: Sync vs Async HTTP Under Load

Two endpoints — one uses `httpClient.GetStringAsync`, one uses `.GetStringAsync().GetAwaiter().GetResult()`. Under 200 concurrent clients, sync version exhausts pool; async maintains lower latency p99. Same single-request network time; different **thread utilization**.

### Example 2: CPU Work on Request Thread

```csharp
[HttpGet("hash")]
public IActionResult Hash(string input)
{
    var hash = BCrypt.HashPassword(input); // blocks request thread 100ms CPU
    return Ok(hash);
}
```

Acceptable at low QPS; under load, blocks pool threads like I/O blocking. Offload:

```csharp
var hash = await Task.Run(() => BCrypt.HashPassword(input));
```

Still uses CPU; frees request thread only during queue wait — see `1.4.4` for nuances.

### Example 3: Mixed Workload

Service: fetch user from DB (I/O) + validate JWT signature (CPU). Pattern: `await` DB; `Task.Run` for crypto if heavy; or pre-compute. Classify each step separately.

## Common Mistakes & Anti-patterns

**`async` all CPU methods** — adds state machine cost, no I/O benefit; harder debugging.

**`Task.Run` for DB/network** — hides blocking on another pool thread; use provider's async API.

**Assuming `SaveChangesAsync` is magic** — must be true async driver; still can block if misused.

**Measuring only happy path** — sync works with 1 user; fails at 100 concurrent I/O.

## Production & Real-World Notes

Libraries: always prefer **`ReadAsync`/`WriteAsync`**, EF **`ToListAsync`**, **`ExecuteReaderAsync`**. Sync-over-async in NuGet dependencies is a common production incident source.

Container with **1 CPU limit**: parallel CPU offload beyond 1 worker rarely helps; async I/O still wins for concurrency.

**Polly**, retries, timeouts — compose with async; each retry is still I/O-bound pattern.

## Comparison / Trade-offs

| Work type | Best tool | Wrong tool |
|-----------|-----------|------------|
| HTTP/DB/File I/O | `await` true async | Sync, Task.Run wrapper |
| Heavy CPU on API | Bounded Task.Run / background queue | `async` without offload |
| Embarrassingly parallel CPU batch | `Parallel.ForEach` / TPL Dataflow | Thousands of Tasks |
| UI thread (WPF) | `await` + SyncContext | `Task.Run` for I/O then touch UI in continuation |

ASP.NET Core has **no** request SyncContext — continuations often resume on pool threads (`1.4.20`).

---

## Quick Reference

- **I/O-bound:** waiting >> computing → async frees threads.
- **CPU-bound:** computing >> waiting → parallelism, not async magic.
- **Fake async:** `Task.Run` + sync I/O — still blocks pool.
- **Throughput vs latency:** async helps former under load for I/O.

---

## Key Takeaways

- Classify every hot path: I/O or CPU.
- Async scales **concurrent I/O** by releasing threads during waits.
- Async does **not** speed pure CPU work.
- Use provider async APIs end-to-end.
