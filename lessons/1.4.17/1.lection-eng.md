## Why This Matters

Web apps constantly **produce work** (HTTP requests enqueue emails, webhooks, audit events) faster than **consumers** process it (SMTP, external APIs, DB writes). A naive **`List<T>` + lock`** grows until **OutOfMemoryException**. **`BlockingCollection`** blocks threads — poor fit for async ASP.NET Core.

**`System.Threading.Channels`** (.NET Core 3.0+) provides **async-friendly queues** optimized for single-producer/single-consumer and multi-producer scenarios. **`ChannelWriter.WriteAsync`** awaits when the bounded buffer is full — **backpressure** pushes slowness to producers instead of RAM.

You'll see channels in **background task queues**, **pipeline stages**, **rate-limited ingestion**, and **`1.4.34` BackgroundTaskQueue** in ASP.NET. Understanding channels separates "fire-and-forget `Task.Run`" hacks from structured throughput control.

## Core Concepts

### Producers, Consumers, and the Buffer

A **channel** is a thread-safe buffer between parties that may run at different speeds:

```
Producers → [ bounded buffer ] → Consumers
```

**`Channel<T>`** splits into **`Writer`** (producers) and **`Reader`** (consumers). Either side can be passed to different tasks — clear API boundary.

### Bounded vs Unbounded

**Unbounded** — `Channel.CreateUnbounded<T>()`: **`WriteAsync`** never waits for space (unless completed). Memory grows if consumers lag — use only when lag is impossible or volume is tiny.

**Bounded** — `Channel.CreateBounded<T>(capacity)`: when full, **`WriteAsync`** awaits until a reader frees a slot — **explicit backpressure**. Production default for request-sourced workloads.

```csharp
var options = new BoundedChannelOptions(100)
{
    FullMode = BoundedChannelFullMode.Wait
};
Channel<OrderEvent> channel = Channel.CreateBounded<OrderEvent>(options);
```

**`FullMode`** alternatives: **`DropWrite`**, **`DropOldest`**, **`Wait`** — choose based on whether losing events is acceptable (metrics) vs correctness (payments).

### Reading and Writing

**Writer:**

```csharp
await channel.Writer.WriteAsync(item, ct);
channel.Writer.TryWrite(item); // non-blocking; returns false if full/done
channel.Writer.Complete(error); // no more items; readers drain then complete
```

**Reader:**

```csharp
await foreach (var item in channel.Reader.ReadAllAsync(ct))
    await ProcessAsync(item);

// or manual loop:
while (await channel.Reader.WaitToReadAsync(ct))
{
    while (channel.Reader.TryRead(out var item))
        await HandleAsync(item);
}
```

**`ReadAllAsync`** yields until channel completed and empty — ergonomic consumer loop.

### Completion and Errors

Call **`Writer.Complete()`** when no more items will be written (shutdown signal). Optionally **`Complete(Exception)`** — pending readers may throw **`ChannelClosedException`**.

If a consumer fails permanently, coordinate cancellation via **`CancellationToken`** shared across producer and consumer tasks.

### Single vs Multiple Producers/Consumers

Channels support **MPMC** (multi-producer multi-consumer) with correct internal synchronization — still design clearly: often one writer task pool and **`N`** reader workers via **`Parallel.ForEachAsync`** or dedicated loops.

## Under the Hood

Channels use **lock-free techniques** and **`IValueTaskSource`** internally for efficient **`ReadAsync`/`WriteAsync`** — low allocation when operations complete synchronously (slot available immediately).

They are **in-process** only — not a distributed queue (Redis, RabbitMQ). For cross-service messaging, use message brokers; channels coordinate **inside one process**.

---

## Syntax / API

| API | Role |
|-----|------|
| `Channel.CreateBounded<T>(n)` | Fixed capacity + backpressure |
| `Channel.CreateUnbounded<T>()` | Grow until memory limit |
| `ChannelWriter.WriteAsync` | Async write; may wait if bounded full |
| `ChannelReader.ReadAsync` | Async read; waits if empty |
| `Reader.ReadAllAsync()` | `IAsyncEnumerable<T>` over channel |
| `Writer.TryComplete` / `Complete` | Signal end of production |
| `BoundedChannelFullMode` | Behavior when full |

## Examples

### Background work queue

```csharp
public class WorkQueue
{
    private readonly Channel<WorkItem> _channel =
        Channel.CreateBounded<WorkItem>(new BoundedChannelOptions(500)
        {
            FullMode = BoundedChannelFullMode.Wait
        });

    public ChannelWriter<WorkItem> Writer => _channel.Writer;

    public async Task RunConsumerAsync(CancellationToken ct)
    {
        await foreach (var work in _channel.Reader.ReadAllAsync(ct))
            await work.ExecuteAsync(ct);
    }
}
```

HTTP handler **`TryWrite`** or **`WriteAsync`** enqueues; hosted service runs **`RunConsumerAsync`**.

### Pipeline stage

Stage 1 reads HTTP → writes **`Channel<byte[]>`** chunks → Stage 2 parses → writes **`Channel<ParsedRow>`** → Stage 3 saves. Bounded channels prevent stage 1 from flooding memory if stage 3 is slow.

## Common Mistakes & Anti-patterns

**Unbounded channel for unbounded request rate** — memory bomb.

**Never calling `Writer.Complete()`** — consumers hang forever in **`ReadAllAsync`**.

**Fire-and-forget producer without handling full channel** — silent **`TryWrite` false** drops work.

**Using channel as cross-process queue** — won't work across machines.

**Replacing all DI with global static channel** — testability and lifetime issues.

## Production & Real-World Notes

Pair bounded channels with **metrics**: queue depth, write wait time, drop count if using **`DropOldest`**.

On app shutdown, **`Complete`** writer and **`await`** consumer drain with timeout — **`IHostApplicationLifetime`**.

For **ASP.NET Core**, prefer **`IHostedService`** consumer over naked **`Task.Run`**.

Related: **`1.4.34` BackgroundTaskQueue**, **`1.4.18` SemaphoreSlim** for limiting concurrent consumers rather than buffer size.

## Comparison / Trade-offs

| | `IAsyncEnumerable` | `Channel` |
|---|-------------------|-----------|
| Model | Pull iterator | Buffered queue |
| Producers | Typically one | Many |
| Backpressure | Implicit pull | Explicit bounded buffer |
| Decoupling | Tight consumer/producer timing | Loose |
| Distributed | No | No (in-process) |

## Quick Reference

```
Bounded = backpressure when full (prefer in web apps)
Writer.WriteAsync / Reader.ReadAllAsync
Writer.Complete() on shutdown
FullMode: Wait | DropWrite | DropOldest
In-process producer/consumer only
```

## Key Takeaways

- **Channels** decouple async producers and consumers with optional bounded buffers.
- **Bounded** channels apply **backpressure** via awaiting writes.
- Split **`ChannelWriter`** / **`ChannelReader`** for clear ownership.
- **`ReadAllAsync`** + **`await foreach`** is the idiomatic consumer loop.
- **`Complete()`** signals shutdown — don't forget on host stop.
- Not a distributed queue — use brokers for cross-service.
- Choose **`FullMode`** based on correctness vs telemetry tolerance.
