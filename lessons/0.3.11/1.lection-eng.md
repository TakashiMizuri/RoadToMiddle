# Content Types: application/json, form-urlencoded, multipart

## Why This Matters

In earlier HTTP lessons you learned that a request carries a method, URL, headers, and sometimes a **body**. The body is just bytes on the wire — meaningless without agreement on how to interpret them. That agreement is the **`Content-Type`** header (and, for responses, often mirrored by the client's **`Accept`** header). When your React app posts login credentials, when your ASP.NET Core controller receives a DTO, when Postman sends a file — each party must know whether those bytes are JSON, a flat key-value string, or a sequence of MIME parts separated by boundaries.

Choosing the wrong type causes subtle production bugs. A frontend that sends JSON while the API expects form-urlencoded will hit model binding failures or empty DTOs. An API that accepts multipart for a tiny JSON payload works but wastes parsing overhead and complicates logging. OAuth clients that POST tokens as JSON instead of form-urlencoded get rejected by spec-compliant servers. Middle-level developers do not memorize MIME strings — they understand **what each encoding represents** and **which layer parses it**.

## Core Concepts

### Content-Type as a Contract

Every HTTP message with a body should declare its media type in **`Content-Type`**. For requests, the server uses this to select a parser: JSON deserializer, form parser, or multipart reader. For responses, the client uses it to decide whether to call `response.json()`, read text, or treat bytes as a download.

The header can include a **`charset`**, typically `utf-8` for text-based types. Omitting charset for JSON is common because JSON is defined as UTF-8 (with optional BOM detection), but form types historically defaulted to ambiguous encodings — modern APIs always specify UTF-8 explicitly.

```
POST /api/users HTTP/1.1
Host: api.example.com
Content-Type: application/json; charset=utf-8
Content-Length: 47

{"email":"dev@example.com","name":"Alex"}
```

This contract is separate from **HTTP method**. You can send JSON with POST, PUT, or PATCH; the method says *what operation*, the content type says *how bytes are shaped*.

### application/json

**`application/json`** encodes data as a **JSON text document** — objects, arrays, strings, numbers, booleans, and `null`. It is the lingua franca of modern REST and GraphQL APIs because it maps cleanly to JavaScript objects and to C# classes via `System.Text.Json` or Newtonsoft.

JSON excels at **nested structure**: a user with an address object, a list of order lines, optional fields. It is human-readable in logs (with size limits in production), easy to validate with JSON Schema, and universally supported by HTTP clients. Downsides: no native binary blob inline (you Base64-encode or use separate upload endpoints), no streaming form fields without framing, and slightly larger than binary formats for numeric-heavy payloads.

ASP.NET Core with `[FromBody]` expects JSON by default when `Content-Type: application/json` is set. The input formatter deserializes into your DTO; validation attributes run afterward.

```json
{
  "title": "Fix login bug",
  "priority": 2,
  "tags": ["auth", "frontend"]
}
```

### application/x-www-form-urlencoded

**`application/x-www-form-urlencoded`** is the default encoding of **classic HTML forms** when `method="POST"` and no `enctype` override is set. The body is a single string of **`key=value` pairs joined by `&`**, with **percent-encoding** for reserved characters (space → `+` or `%20`, `&` → `%26`, non-ASCII → UTF-8 bytes then `%HH`).

```
username=alex%40example.com&password=Secret%21123&remember=true
```

This format is **flat**: one level of keys. Repeated keys or bracket notation (`items[0].id=1`) are conventions some servers understand, but there is no standard nested object model like JSON. It is compact for simple login and search forms and remains **required by OAuth 2.0** for token requests (`grant_type`, `client_id`, etc.) in the `application/x-www-form-urlencoded` body.

ASP.NET Core binds urlencoded bodies with `[FromForm]` on actions or parameters. `[FromBody]` does **not** read form-urlencoded data — a common integration mistake between SPA and legacy endpoints.

### multipart/form-data

**`multipart/form-data`** wraps each field (and each file) in a **separate MIME part**, separated by a **boundary** string declared in the header:

```
Content-Type: multipart/form-data; boundary=----WebKitFormBoundary7MA4YWxkTrZu0gW
```

Each part has its own mini-headers (`Content-Disposition: form-data; name="field"` or `filename="photo.jpg"`) and raw body bytes. Binary file content is **not** Base64-expanded inside JSON — it travels as-is within its part, which makes uploads efficient.

Browsers use multipart automatically when `<form enctype="multipart/form-data">` includes `<input type="file">`. APIs accept multipart for "create post with image" patterns. Parsing is heavier than urlencoded or JSON: the server must scan boundaries, enforce size limits per part, and guard against malicious filenames and content types.

```
------WebKitFormBoundary7MA4YWxkTrZu0gW
Content-Disposition: form-data; name="title"

Summer vacation
------WebKitFormBoundary7MA4YWxkTrZu0gW
Content-Disposition: form-data; name="photo"; filename="beach.jpg"
Content-Type: image/jpeg

<binary JPEG bytes>
------WebKitFormBoundary7MA4YWxkTrZu0gW--
```

The closing boundary ends with `--` to mark EOF.

### How the Three Types Relate

All three solve **"how do I put structured user input in an HTTP body?"** but with different trade-offs. JSON favors programmatic clients and rich structure. Urlencoded favors simple forms and OAuth compatibility. Multipart adds **file bytes** without encoding overhead. A single application often uses all three: JSON for `/api/*`, urlencoded for `/connect/token`, multipart for `/api/documents/upload`.

## Under the Hood

When Kestrel receives a request, it streams body bytes to middleware. For API controllers, **input formatters** run in order: the `Content-Type` is matched against supported formatters (`SystemTextJsonInputFormatter`, `JQueryFormValueProviderFactory` path for forms, etc.). Wrong or missing `Content-Type` may cause **415 Unsupported Media Type** or silent mis-binding if a formatter incorrectly claims the request.

Percent-encoding in urlencoded bodies uses `application/x-www-form-urlencoded` rules from the HTML spec: `+` decodes to space; `%XX` is hex byte values. JSON parsing is delegated to `Utf8JsonReader` — invalid JSON yields 400 with problem details if configured.

Multipart parsing is **streaming** in well-designed servers: parts can be written to disk or cloud storage above a size threshold instead of buffering entire files in RAM. Boundary strings must not appear in part content; clients generate random boundaries to minimize collision. **Sniffing** uploaded `Content-Type` inside a part is unreliable — validate magic bytes server-side for security.

On the response side, `Content-Type: application/json` triggers browser `fetch().json()` parsing; `text/html` renders as document. **MIME type** does not imply HTTP method or caching — those are separate headers (`Cache-Control`, `ETag`).

## Syntax / Commands / API

**Request headers (client sets):**

| Content-Type | Typical use |
|--------------|-------------|
| `application/json` | REST API bodies, webhooks |
| `application/x-www-form-urlencoded` | HTML forms, OAuth token POST |
| `multipart/form-data; boundary=...` | Forms with files |

**curl examples:**

```bash
# JSON
curl -X POST https://api.example.com/tasks \
  -H "Content-Type: application/json" \
  -d '{"title":"Review PR"}'

# Urlencoded
curl -X POST https://api.example.com/login \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "email=user%40example.com&password=secret"

# Multipart (curl -F sets type and boundary automatically)
curl -X POST https://api.example.com/upload \
  -F "title=Report" \
  -F "file=@./report.pdf"
```

**ASP.NET Core binding:**

```csharp
[HttpPost("login")]
public IActionResult Login([FromForm] LoginRequest request) { ... }

[HttpPost("items")]
public IActionResult Create([FromBody] CreateItemDto dto) { ... }

[HttpPost("upload")]
public IActionResult Upload([FromForm] IFormFile file, [FromForm] string title) { ... }
```

## Examples

### Example 1: SPA login to JSON API

A React app sends credentials as JSON because the API documents `[FromBody] LoginDto`. The fetch call sets `Content-Type: application/json` and `JSON.stringify`s the object. The server deserializes once; no form parser runs. If someone switches the frontend to urlencoded without updating the controller, `LoginDto` properties arrive null — the fix is align **both** sides on one type.

### Example 2: OAuth token endpoint

The OAuth 2.0 spec requires token requests as **form-urlencoded**, not JSON. A client POSTs `grant_type=authorization_code&code=...&redirect_uri=...`. IdentityServer and Azure AD enforce this. Sending JSON returns `400 invalid_request` — not because the data is wrong, but because the **media type contract** is wrong.

### Example 3: Profile photo upload

An endpoint accepts `multipart/form-data`: field `displayName` (text part) and `avatar` (file part). ASP.NET Core's `IFormFile` exposes stream, file name, and length. The controller validates extension and size, then stores to blob storage. Using JSON with Base64 for a 5 MB image bloats payload ~33% and parses entirely in memory — multipart streams the file part efficiently.

## Common Mistakes & Anti-patterns

**JSON body with `[FromForm]` or urlencoded with `[FromBody]`** — binding attributes and content type must match. The error looks like "validation failed on all fields" when fields are simply empty.

**Omitting `Content-Type` on POST with a body** — servers may reject the request or guess wrong. Always set it explicitly in API clients.

**Using multipart for pure JSON APIs** — unnecessary boundary overhead and harder log redaction.

**Trusting client-supplied file `Content-Type` in multipart** — attackers upload `shell.php` labeled `image/jpeg`. Validate content server-side.

**Assuming `+` and space are interchangeable in JSON** — they are urlencoded rules only; JSON strings use `\u` escapes or literal UTF-8.

## Production & Real-World Notes

API gateways and WAFs inspect `Content-Type` for policy rules. Rate limits may differ for upload endpoints. Structured logging often records content type but **redacts** bodies — multipart makes redaction harder; prefer dedicated upload URLs with short-lived tokens.

OpenAPI documents list **`requestBody.content`** schemas per media type — a single path may accept both `application/json` and `application/xml`; clients must send one the server lists. Versioning and breaking changes sometimes introduce `application/vnd.company.v2+json` — still JSON, with a vendor-specific MIME marker.

Large file uploads may bypass the app server entirely (**presigned S3 URLs**) with PUT and `Content-Type` set per object — multipart at the API layer is then only for small admin uploads.

## Comparison / Trade-offs

| Aspect | JSON | Urlencoded | Multipart |
|--------|------|------------|-----------|
| Structure | Nested objects/arrays | Flat keys | Flat fields + files |
| Binary files | Poor (Base64 in JSON) | Not suitable | Native binary parts |
| Browser default form | No (unless fetch) | Yes (default enctype) | With file inputs |
| Typical API style | REST/GraphQL | OAuth, legacy forms | Uploads |
| Parse cost | Low–medium | Low | Higher (boundaries) |
| Human log readability | Good | Good | Poor for binary parts |

**When in doubt for machine-to-machine APIs:** `application/json`. **When spec says form:** urlencoded. **When bytes are a file:** multipart or separate binary upload.

## Quick Reference

```
Content-Type     = how to interpret body bytes

application/json
  → JSON object/array; REST default; [FromBody] in ASP.NET Core

application/x-www-form-urlencoded
  → key=value&key2=value2; percent-encoding; HTML forms; OAuth tokens; [FromForm]

multipart/form-data; boundary=...
  → multiple parts; files + fields; [FromForm] + IFormFile

Client must send Content-Type matching server binding.
415 = server does not support that media type for this endpoint.
```

## Key Takeaways

1. **`Content-Type`** is the contract for interpreting request and response bodies.
2. **`application/json`** fits structured API data and nested objects; default for modern REST.
3. **`application/x-www-form-urlencoded`** is flat, form-style encoding; required for OAuth token POSTs.
4. **`multipart/form-data`** separates fields and files with boundaries; use for uploads.
5. ASP.NET Core **`[FromBody]`** vs **`[FromForm]`** must align with what the client sends.
6. Wrong content type causes empty models, 415 errors, or spec rejections — not always obvious validation messages.
7. Do not trust client file metadata; validate uploaded content on the server.
