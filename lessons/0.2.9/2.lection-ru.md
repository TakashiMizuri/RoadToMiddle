# Conventional Commits: feat, fix, chore, BREAKING

> Roadmap: `0.2.9` · Node: `0.2` — Git: ветки и collaboration · Depth: практика

## Learning Objectives

После этой лекции ты сможешь:

- Назвать формат **Conventional Commits**: `type(scope)!: description`.
- Использовать типы **`feat`**, **`fix`**, **`chore`**, **`docs`**, **`refactor`**, **`test`**, **`ci`**.
- Помечать **breaking changes** через `!` и/или footer `BREAKING CHANGE:`.
- Объяснить связь conventional messages с **changelogs**, **SemVer** и release automation.
- Писать **imperative**, краткие subject lines для squash PR titles.
- Избегать vague messages (`update`, `fix stuff`), ломающих automation.

---

## Why This Matters

Можно смержить идеальный код через безупречный PR (`0.2.8`) и всё равно навредить команде сообщениями `wip`, `fix`, `changes`. Через полгода `git log` должен отвечать: *что изменилось? это видел пользователь? сломали API?* **Conventional Commits** — лёгкая конвенция поверх обычных Git messages — делает history **machine-readable** и **human-scannable**.

Инструменты парсят `feat:` vs `fix:` для **changelogs**. **Semantic Release** бампит **MAJOR.MINOR.PATCH**: `feat` → minor, `fix` → patch, breaking → major. Даже без полной automation reviewers по списку commits в PR сразу видят intent. Squash-merge titles часто копируют convention: `feat(auth): add refresh token rotation (#102)`.

Для Middle fullstack, shipping ASP.NET и React, consistent messages связывают Git history с **release notes**, **версиями client SDK** и **postmortems**.

## Core Concepts

### Формат

```
<type>[optional scope][optional !]: <description>

[optional body]

[optional footer(s)]
```

**Type** — вид изменения (обязателен). **Scope** — подсистема (`api`, `auth`, `ui`). **!** — breaking indicator. **Description** — короткий imperative summary. **Body** — контекст, migration. **Footer** — `BREAKING CHANGE:`, `Fixes #42`.

```
feat(orders): add PATCH endpoint for partial updates
fix(auth): reject expired refresh tokens
chore(deps): bump EF Core to 8.0.4
```

### Основные типы

**`feat`** — новая capability; user-visible или новый API. **MINOR** в SemVer.

**`fix`** — исправление бага для users/production. **PATCH**.

**`chore`** — maintenance без изменения production behavior: deps, tooling.

**`docs`** — только документация.

**`refactor`** — структура без fix/feature behavior.

**`test`**, **`ci`**, **`perf`**, **`style`** — по смыслу названия.

### Breaking Changes

Ломает контракт для consumers — удалённый endpoint, переименованное поле JSON.

1. **`!`:** `feat(api)!: remove v1 orders endpoint`
2. **Footer:** `BREAKING CHANGE: ...`

→ **MAJOR** version bump. Координация backend/frontend обязательна.

### Правила subject

- **Imperative:** «add endpoint», не «added».
- **~50 символов** где возможно.
- Без `update`, `misc`, `fix issue`.
- **Один logical change per commit** (`0.1.2`).

### И PR workflow

Каждый commit в PR — conventional. При squash title отражает весь PR. `Fixes #42` в footer.

## Under the Hood

Conventional Commits — **parsing convention**, не фича Git. **commitlint**, **semantic-release** матчат regex заголовка. CI может **отклонять** non-conforming commits.

---

## Syntax / Commands / API

```bash
git commit -m "feat(health): expose liveness probe endpoint"
git commit -m "feat(api)!: rename UserDto" -m "BREAKING CHANGE: clients update mapping."
git commit --amend -m "fix(orders): correct EU tax"
git rebase -i origin/main   # reword плохих messages
```

---

## Examples

### ASP.NET feature

```
feat(notifications): add POST /api/notifications/preferences

Closes LINEAR-441
```

### Breaking

```
feat(users)!: return ISO-8601 dates only

BREAKING CHANGE: CreatedAt no longer Unix ms.
```

### Chore vs fix

`chore(ci): cache NuGet` — не `fix(ci)`, если pipeline не был сломан.

---

## Common Mistakes & Anti-patterns

Всё `fix`/`update`; past tense; скрытый breaking без `!`; смешанные unrelated changes в одном commit.

---

## Production & Real-World Notes

**semantic-release** на `main`; **commitlint** в hooks; **scopes** в monorepo per package.

---

## Comparison / Trade-offs

Conventional vs free-form: automation и clarity vs гибкость. Squash-only teams теряют step-by-step PR narrative на main.

---

## Quick Reference

```
type(scope)!: imperative description
feat → MINOR | fix → PATCH | breaking → MAJOR
chore, docs, test, refactor, ci
```

---

## Key Takeaways

1. Формат `type(scope)!: description`.
2. **`feat`**, **`fix`**, **`chore`** — базовый набор.
3. Breaking — `!` или `BREAKING CHANGE:`.
4. Imperative, specific subjects.
5. Связь с PR и SemVer (`0.2.8`).
