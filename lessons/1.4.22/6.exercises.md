# Exercises: Benchmark Sync vs Async HTTP

> Complete after reading `1.lection-eng.md`, `2.lection-ru.md` and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Hello BenchmarkDotNet
**Goal:** Run a minimal Release benchmark comparing async vs sync-over-async for one URL.
**Steps:**
1. Create console app; add BenchmarkDotNet.
2. Implement two `[Benchmark]` methods using shared `HttpClient`.
3. Run `dotnet run -c Release`; capture mean and allocated bytes.
**Success criteria:** Both benchmarks complete; you can explain why mean times are similar.

### E2. Thread pool snapshot
**Goal:** Observe available worker threads under concurrent sync blocking.
**Steps:**
1. Write a console app that fires 100 parallel operations.
2. Log `ThreadPool.GetAvailableWorkerThreads` before, during, and after.
3. Compare async `Task.WhenAll` vs sync `.GetResult()` per call.
**Success criteria:** Written observation (3–5 sentences) of the difference.

## Medium

### M1. Local delay server
**Goal:** Stable benchmark target without external network jitter.
**Steps:**
1. Create Minimal API with `GET /delay/{ms}` returning after `Task.Delay`.
2. Point E1 benchmarks at localhost.
3. Add `[Params(1, 25, 100)]` concurrency and measure total batch duration.
**Success criteria:** Table of concurrency vs elapsed ms for async and sync paths.

### M2. Diagnosers report
**Goal:** Produce a benchmark report with memory and threading diagnosers.
**Steps:**
1. Add `[MemoryDiagnoser]` and `[ThreadingDiagnoser]`.
2. Export results to markdown or HTML (BenchmarkDotNet exporters).
3. Highlight one metric that differs between sync and async at high concurrency.
**Success criteria:** Screenshot or pasted summary with your interpretation.

## Hard (stretch)

### H1. ASP.NET proxy load sketch
**Goal:** Connect micro-benchmark insight to a tiny web proxy.
**Steps:**
1. ASP.NET Core app with endpoint `/proxy` calling downstream `/delay/500`.
2. Implement two branches (feature flag): async await vs sync-over-async.
3. Use `wrk`, `bombardier`, or k6 with 50 concurrent connections for 30s each.
4. Compare p95 latency and thread pool counter if available.
**Success criteria:** Short write-up: which branch failed first and why.

## Submission checklist

- [ ] All runs in **Release**
- [ ] One shared `HttpClient` per process (not per request)
- [ ] Documented concurrency level
- [ ] No `.Result` in ASP.NET controller — only in controlled sync benchmark path

## Hints (read only if stuck)

<details>
<summary>Hint E1</summary>
Use `[GlobalSetup]` to create `HttpClient`. Baseline = async method. Sync benchmark wraps same `GetStringAsync` with `GetAwaiter().GetResult()`.
</details>

<details>
<summary>Hint M1</summary>
`[Benchmark]` returning `Task` is awaited by BDN. For sync batch at high concurrency, consider `[Benchmark(OperationsPerInvoke = N)]` or loop inside one benchmark method to avoid process overhead dominating.
</details>
