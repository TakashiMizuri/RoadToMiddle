# HTTPS, TLS Handshake Basics, and Certificates

> Roadmap: `0.3.8` · Node: `0.3` — HTTP and the Web · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Explain why **HTTPS** is HTTP over **TLS** — not a separate protocol replacing HTTP semantics from `0.3.3`.
- Outline the **TLS handshake** phases: TCP, ClientHello/ServerHello, certificate, key exchange, finished.
- Describe **X.509 certificates**: subject, issuer, validity, public key, chain of trust.
- Distinguish **symmetric** session keys vs **asymmetric** certificate keys and why both exist.
- Identify common failures: expired cert, hostname mismatch, untrusted CA, mixed content.
- Relate HTTPS to **Authorization**, **Cookie** `Secure` flag (`0.3.7`, `0.3.9`), and production deployment (Kestrel, reverse proxy).

---

## Why This Matters

Everything you learned about HTTP messages — methods (`0.3.4`), status codes (`0.3.6`), headers (`0.3.7`) — assumes bytes travel between client and server. On the public internet, **plain HTTP exposes those bytes** to anyone on the path: Wi‑Fi hotspots, ISP equipment, compromised routers. Passwords in `Authorization: Basic`, session tokens in `Cookie`, personal data in JSON bodies — all readable without encryption.

**HTTPS** wraps HTTP inside **TLS (Transport Layer Security)**. The URL scheme becomes `https://`; default port **443** instead of 80. To users and application code, it is still HTTP request-response — but the transport encrypts and integrity-protects the stream. Browsers mark non-HTTPS login forms as insecure; modern APIs reject callbacks without TLS; **Secure** cookies are ignored on HTTP.

Middle developers deploy ASP.NET Core behind nginx or cloud load balancers, configure certificates (Let's Encrypt, Azure Key Vault), debug `NET::ERR_CERT_AUTHORITY_INVALID`, and explain why **TLS terminates at the proxy** while HTTP continues to Kestrel on localhost. Understanding the handshake and certificate chain turns mysterious browser errors into actionable fixes — not cargo-cult "just renew cert."

## Core Concepts

### HTTP vs HTTPS: Same Application Protocol, Different Transport

**HTTP** defines messages: start line, headers, body. **HTTPS** = **HTTP over TLS over TCP**. TLS sits between TCP and HTTP:

```
Application:   HTTP (requests/responses)
Security:      TLS (encrypt + authenticate peer)
Transport:     TCP
Network:       IP
```

After TLS succeeds, Kestrel reads decrypted HTTP identically to cleartext HTTP on port 80. Your `MapGet`, status codes, and JSON serializers unchanged — only the wire format differs.

The **S** in HTTPS reflects historical **SSL** naming; today **TLS 1.2/1.3** is standard; SSL is obsolete and insecure.

### Threat Model TLS Addresses

Without TLS, attackers can **eavesdrop** (read traffic), **tamper** (modify requests/responses), and **impersonate** server (fake bank site). TLS provides:

- **Confidentiality** — encrypted payload; observers see only metadata (IP, SNI hostname in many deployments, timing, size patterns).
- **Integrity** — tampering detected via authenticated encryption (AEAD ciphers).
- **Authentication** — client verifies it talks to the **real server** via certificate chain to a **trusted CA**.

TLS does **not** replace application authentication (JWT, cookies) — it protects the **channel**. Phishing sites can still have valid HTTPS with wrong domain if user clicks evil link — user must verify URL.

### TLS Handshake Overview (Basics)

Goal: agree on cipher suite, authenticate server (and optionally client), derive **shared symmetric session keys** for bulk encryption.

Simplified **TLS 1.2** flow:

```
Client                                    Server
  | TCP connect (port 443)                    |
  |---------- ClientHello ------------------>|
  |   (TLS version, cipher suites, random)    |
  |<--------- ServerHello -------------------|
  |   (chosen cipher, random)                 |
  |<--------- Certificate -------------------|
  |   (server X.509 chain)                    |
  |<--------- ServerHelloDone (1.2) ----------|
  |---------- ClientKeyExchange ----------->|
  |   (premaster secret / key share)          |
  |---------- ChangeCipherSpec / Finished -->|
  |<--------- ChangeCipherSpec / Finished ---|
  |======== Encrypted HTTP begins ===========|
```

**TLS 1.3** reduces round trips (often **1-RTT** handshake, **0-RTT** resumption with caveats). ClientHello and ServerHello still negotiate version and ciphers; certificate still proves identity; key agreement uses **ECDHE** (forward secrecy — past sessions safe if long-term key stolen later).

You do not implement the handshake in application code — **OS/browser and Kestrel/OpenSSL** do. You **configure certificates** and **minimum TLS version**.

### Symmetric vs Asymmetric Cryptography in TLS

**Asymmetric** (RSA/ECDSA in certificate): slow, used for **authentication** and initial key agreement. Server proves it holds private key matching certificate public key.

**Symmetric** (AES-GCM, ChaCha20-Poly1305): fast, used for **bulk HTTP data** after handshake. Session keys derived via key exchange — new keys per session (or resumed session).

Why both? Public-key crypto on every HTTP byte would crush CPU; hybrid model is standard.

### X.509 Certificates and Chain of Trust

A **certificate** binds a **public key** to an identity — typically **DNS hostname(s)** in **Subject Alternative Name (SAN)**:

```
Subject: CN=api.example.com
SAN: DNS:api.example.com, DNS:www.example.com
Issuer: CN=Let's Encrypt R3
Valid: Not Before ... Not After ...
Public Key: (embedded)
Signature: CA signed this cert
```

Browser/OS trusts **root CAs** in trust store. Server sends **leaf cert** + **intermediate chain**. Client verifies signature chain to trusted root. **Self-signed** certs fail unless manually trusted (dev only).

Common validation failures:

- **Expired** — past Not After date.
- **Wrong hostname** — cert for `example.com`, site is `api.example.com` without SAN.
- **Untrusted issuer** — corporate MITM proxy or misconfigured chain.
- **Revoked** — OCSP/CRL check (browser dependent).

**Let's Encrypt** automates free DV (Domain Validation) certs via ACME protocol — standard for production web.

### HTTPS in Development and Production

**Development:** `dotnet dev-certs https --trust` creates local trusted cert; browser warns if not trusted. **Never** ship dev certs to production.

**Production patterns:**

1. **TLS at reverse proxy** (nginx, Azure App Gateway): proxy holds public cert, HTTP to Kestrel on loopback — common, simplifies cert renewal.
2. **TLS at Kestrel**: `ConfigureKestrel` with cert from store/Key Vault — end-to-end encryption to app process.
3. **Cloud load balancer** terminates TLS, forwards with `X-Forwarded-Proto: https` — app must honor forwarded headers for redirects and cookie Secure logic.

**HTTP Strict Transport Security (HSTS)** response header forces browsers to use HTTPS only for domain — prevents sslstrip downgrade.

### Mixed Content and Secure Cookies

Page loaded over HTTPS must not load **active mixed content** (HTTP scripts/XHR) — browsers block or warn. Passive images may load with warnings.

Cookies with **`Secure` attribute** (`0.3.9`) sent **only over HTTPS**. Session fixation and token theft drop sharply when entire site is HTTPS.

## Under the Hood

### What Happens on First `https://` Request

1. DNS resolves hostname → IP.
2. TCP **three-way handshake** to port 443.
3. TLS handshake (1–2 RTT depending on version/resumption).
4. Certificate validation against trust store + hostname.
5. Session keys established; **Application Data** records carry encrypted HTTP.
6. HTTP **GET /** inside TLS — same start line/headers as lesson `0.3.3`.

**SNI (Server Name Indication)** in ClientHello lets one IP host many HTTPS sites — virtual hosting for TLS.

**Session resumption** (tickets/session IDs in 1.2; PSK in 1.3) skips full handshake on repeat visits — faster TTFB.

### Certificate Pinning and mTLS (Awareness)

**Certificate pinning** (hard-coding expected public key) — mobile apps sometimes; brittle when cert rotates.

**Mutual TLS (mTLS)** — client also presents certificate; common service-to-service in zero-trust meshes — beyond basics but know the term.

### ASP.NET Core HTTPS Redirection

`app.UseHttpsRedirection()` redirects HTTP→HTTPS using configured port. Behind proxy, configure `ForwardedHeadersOptions` so app knows original scheme — otherwise redirect loops or wrong cookie Secure behavior.

Kestrel `ListenOptions.UseHttps()` loads cert from file or store. Cipher policy via system/OpenSSL configuration.

## Syntax / Commands / API

### Browser / openssl inspection

```bash
openssl s_client -connect api.example.com:443 -servername api.example.com
# Shows certificate chain, protocol, cipher
```

```bash
curl -vI https://api.example.com
# TLS handshake + HTTP headers
```

### ASP.NET Core (Program.cs patterns)

```csharp
builder.WebHost.ConfigureKestrel(options =>
{
    options.ListenAnyIP(443, listen => listen.UseHttps());
});

app.UseHttpsRedirection();
app.UseHsts(); // production
```

Behind proxy:

```csharp
app.UseForwardedHeaders(new ForwardedHeadersOptions
{
    ForwardedHeaders = ForwardedHeaders.XForwardedFor | ForwardedHeaders.XForwardedProto
});
```

## Examples

### Example 1: First visit to bank site

Browser validates Let's Encrypt chain, checks SAN matches `bank.example.com`, completes TLS 1.3 handshake, sends encrypted `GET /login`. Attacker on Wi‑Fi sees only TLS records — not password later posted in JSON body.

### Example 2: Expired certificate

Admin forgot renewal. Users see `NET::ERR_CERT_DATE_INVALID`. Fix: renew via ACME, deploy new cert to nginx, verify intermediate chain complete.

### Example 3: TLS termination at nginx

Client ↔ nginx (HTTPS, public cert). nginx ↔ Kestrel `http://127.0.0.1:5000` (HTTP). Kestrel logs show HTTP but external path encrypted. Configure forwarded proto for `UseHttpsRedirection` and audit logs showing external scheme.

## Common Mistakes & Anti-patterns

**HTTP login forms in production.** Credentials and cookies exposed.

**Self-signed cert on public site without user trust.** Abandonment and support tickets.

**Incomplete certificate chain on server.** Android/old clients fail while desktop works.

**TLS terminated but app generates `http://` links.** Mixed content, broken cookies.

**Ignoring cert expiry monitoring.** Predictable outage.

**TLS 1.0/1.1 enabled.** Deprecated, PCI/non-compliance.

---

## Production & Real-World Notes

Automate renewal (cert-manager on Kubernetes, Azure managed certs). Monitor expiry 30 days ahead. Use **TLS 1.2+** minimum, prefer **1.3**. Disable weak ciphers. **HSTS** with includeSubDomains after verifying all subdomains support HTTPS.

Secrets in URLs still leak via logs and Referer — HTTPS does not fix logging discipline.

Cloudflare "flexible SSL" (HTTPS client→Cloudflare, HTTP Cloudflare→origin) — know your encryption gap.

## Comparison / Trade-offs

| Approach | Pros | Cons |
|----------|------|------|
| TLS at load balancer | Central cert management | HTTP hop internal — trust network |
| TLS at Kestrel | Encrypt to process | Cert per instance complexity |
| Let's Encrypt DV | Free, automated | No org identity (OV/EV rare now) |
| TLS 1.3 | Faster handshake | Old legacy clients excluded |

## Key Takeaways

- HTTPS encrypts **transport**; HTTP semantics unchanged above TLS.
- Handshake authenticates server via **certificate chain**; bulk data uses **symmetric** keys.
- Cert failures: **expiry**, **hostname**, **untrusted CA**, **broken chain**.
- Production: automate renewal, TLS 1.2+, HSTS, correct **forwarded headers** behind proxy.
- **Secure** cookies and API tokens assume HTTPS end-to-end for users.
