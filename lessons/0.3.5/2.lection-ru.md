# Safe vs Idempotent Methods — матрица

> Roadmap: `0.3.5` · Node: `0.3` — HTTP и Web · Depth: **глубоко**

## Learning Objectives

- Построить и интерпретировать **матрицу safe × idempotent** для common methods.

## Why This Matters

В `0.3.4` — назначение каждого method. Два cross-cutting свойства — **safe** и **idempotent** — govern ecosystem: browsers, CDN, proxies, Polly retry. Это не синонимы. **Safe** = read-only intent — intermediaries могут replay. **Idempotent** = один или N одинаковых requests — same intended effect — client может retry после ambiguous network failure.

Путаница → incidents: retry POST doubles payments; CDN cache GET, который mutates inventory; LB replay non-idempotent request. Middle dev internalize matrix как contracts с infrastructure.

Связь со status codes (`0.3.6` — retry 503), `Cache-Control`, `Idempotency-Key`.

## Core Concepts

### Safe methods

**Definition (RFC 9110):** Method **safe**, если **intended** semantics read-only — request не меняет server state, relevant для user. Safe = information retrieval.

Safe: **GET, HEAD, OPTIONS, TRACE** (TRACE disabled в production).

**Nuance:** Safe — про **intent**, не guarantee что server не пишет logs. Business state не должен меняться. Safe GET не charge card и не delete row.

**Infrastructure:**

- Browser **prefetch**, link preview bots — GET без explicit click.
- **CDN** cache GET/HEAD для других users при allow headers.
- **Crawlers** follow GET links.

GET mutation → CSRF via `<img src>`, cache poison.

### Idempotent methods

**Definition:** **Intended effect** same whether once or many times with same request. Second DELETE не «удаляет сильнее»; second PUT same body — same state.

Idempotent: **GET, HEAD, PUT, DELETE, OPTIONS, TRACE**.

**Not by default:** **POST**, **PATCH** (sometimes idempotent in practice).

**Client retry:** Ambiguous timeout — для idempotent retry OK; для POST blind retry duplicates — нужен **Idempotency-Key**.

**Idempotent ≠ safe:** PUT/DELETE change state but retry-safe re duplication.

**Safe ⇒ idempotent:** All safe methods idempotent. Converse false.

### Матрица

| Method | Safe | Idempotent | Retry без keys | Shared cache |
|--------|------|------------|----------------|--------------|
| GET/HEAD/OPTIONS | Yes | Yes | Yes | GET/HEAD yes |
| POST | No | No | **No** | No |
| PUT/DELETE | No | Yes | Yes | No |
| PATCH | No | Often* | Caution | No |

*PATCH: absolute set idempotent; increment not.

```
                 IDEMPOTENT
              Yes        No
         ┌─────────┬─────────┐
   Yes   │ GET     │         │ SAFE
         │ HEAD    │         │
         │ OPTIONS │         │
         ├─────────┼─────────┤
   No    │ PUT     │ POST    │ NOT SAFE
         │ DELETE  │ PATCH   │
         └─────────┴─────────┘
```

### POST и application idempotency

HTTP: POST **not idempotent**. Payment APIs need retries:

1. Client **`Idempotency-Key`** (UUID).
2. Server store `(key → response)`.
3. Duplicate POST same key — same response без re-charge.

Application idempotency, не меняет HTTP classification POST.

### PUT/DELETE details

**PUT** twice same body → same resource state.

**DELETE** twice → 404/204 second time; resource absent — no double harm.

**GET** twice → two reads; state unchanged intended.

### PATCH edge cases

PATCH `loginCount+1` — **not idempotent**. PATCH `status: active` absolute — idempotent if already active.

Prefer absolute field replacement in PATCH.

### Intermediaries

Shared caches — GET/HEAD primarily. Proxies may retry idempotent on connection reset. LB retry policies often exclude POST.

## Under the Hood

Default **HttpClient** no auto-retry POST. Polly explicit policies.

GET endpoint with `SaveChanges()` each hit — breaks safe semantics.

GET **`If-None-Match`** → 304 — safe, idempotent (`0.3.7`).

Distributed: retried PUT converges; retried POST without key → duplicate rows. DB unique on idempotency key.

## Examples

**Safe violation:** `GET /cart/add?sku=99` — crawler adds to cart. Fix: POST.

**Idempotent retry:** PUT profile timeout → retry same PUT OK.

**POST duplicate:** timeout retry without key → two orders. Fix: Idempotency-Key.

---

## Common Mistakes & Anti-patterns

GET mutation «для admin» — CSRF. Blind POST retry. Assume PATCH idempotent. Cache POST responses. Confuse safe with public.

---

## Production & Real-World Notes

API guidelines document properties. Monitor duplicate orders + gateway timeout. Integration tests: DELETE twice; POST twice without key.

GraphQL mutations — POST at HTTP layer; idempotency inside schema.

---

## Comparison / Trade-offs

PUT idempotent create vs POST. At-least-once network vs exactly-once app dedup. POST `/search` — not safe, acceptable trade.

## Key Takeaways

- **Safe** — cache, prefetch, CSRF surface.
- **Idempotent** — retry after network failure.
- Safe ⇒ idempotent; PUT/DELETE idempotent not safe.
- POST + **Idempotency-Key** for retry-safe commands.
- PATCH — analyze semantics.
- Matrix = contract с ecosystem.
