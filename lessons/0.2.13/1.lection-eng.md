# Tags: Lightweight vs Annotated, Semver Releases

> Roadmap: `0.2.13` · Node: `0.2` — Git: Branches and Collaboration · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Distinguish **lightweight** and **annotated** tags and when to use each.
- Create tags with **`git tag`**, **`git tag -a`**, and push with **`git push origin <tag>`** or **`--tags`**.
- Explain **semver** (MAJOR.MINOR.PATCH) for release naming.
- Point **`git checkout`** / deploy pipelines at immutable release tags.
- List tags, show tag metadata, and **delete/move tags** safely on remotes.
- Connect tags to **hotfix** and **bisect** workflows from prior lessons.

---

## Why This Matters

Branches move; tags mark **specific commits** that humans and machines treat as releases. When ops asks "what exactly is in production v2.3.1?" the answer must be a commit hash — ideally one referenced by an **annotated tag** with release notes, signer, and date. CI/CD, Docker image labels, NuGet/npm package versions, and `git bisect good v2.3.0` all assume tags exist and mean something stable.

Lightweight tags are bookmarks. Annotated tags are release artifacts. Semver gives predictable communication: patch for fixes, minor for backward-compatible features, major for breaking changes — aligned with Conventional Commits from `0.2.9`. Middle developers create tags in hotfix merges (`0.2.11`), never force-move tags casually on shared remotes, and understand that a tag is just a ref pointing into the object graph from `0.1.4`.

## Core Concepts

### What Is a Git Tag?

A **tag** is a named pointer to a commit (usually). Unlike branch names, tags are **not** meant to advance with new work — they label a moment in history: `v1.0.0`, `release-2026-07-01`, `prod-deploy-4421`.

```
refs/tags/v2.3.1  →  commit abc1234...
refs/heads/main   →  commit def5678...  (moves forward)
```

When you deploy "v2.3.1," you checkout or build that tag's commit — reproducible builds, audit trails, rollback targets.

### Lightweight Tags

**Lightweight tags** are simple refs — essentially a branch that nobody moves, but stored under `refs/tags/`:

```bash
git tag v2.3.1
```

No extra metadata object — just `refs/tags/v2.3.1` → commit hash. Fast to create, fine for local temporary markers or personal bookmarks. **Weak for releases:** no tagger, date, message, or GPG signature stored in a tag object.

### Annotated Tags

**Annotated tags** are full Git objects (type `tag`) with their own hash, pointing to a commit plus metadata:

```bash
git tag -a v2.3.1 -m "Release 2.3.1: payment hotfix SEC-401"
```

Contains tagger name/email, date, message, optional **GPG signature**. `git show v2.3.1` displays tag object then commit — rich release record. **Production releases should use annotated tags** — industry default for semver releases, signed releases in security-conscious teams.

Signed tags:

```bash
git tag -s v2.3.1 -m "Signed release 2.3.1"
git tag -v v2.3.1   # verify signature
```

### Listing and Inspecting Tags

```bash
git tag                    # list all
git tag -l "v2.*"          # pattern
git show v2.3.1            # annotated: tag object + commit
git rev-parse v2.3.1       # commit hash only
git log --oneline v2.3.0..v2.3.1   # commits between releases
```

Tags can point to **any object** historically, but today almost always **commits**.

### Pushing Tags to Remote

Tags are **local until pushed**:

```bash
git push origin v2.3.1           # one tag
git push origin --tags             # all local tags (careful)
```

Deleting remote tag (rare, coordinated):

```bash
git push origin :refs/tags/v2.3.1
# or modern: git push origin --delete v2.3.1
```

**Never retag** a published semver on the same version number with different commit — consumers cache artifacts; use `v2.3.2` instead. Moving tags is like rewriting published history.

### Semver for Releases

**Semantic Versioning** (`MAJOR.MINOR.PATCH`):

- **MAJOR** — incompatible API/behavior changes (breaking).
- **MINOR** — backward-compatible functionality.
- **PATCH** — backward-compatible bug fixes.

Pre-release: `v2.4.0-rc.1`, `v2.4.0-beta.2`. Build metadata: `v2.4.0+build.442` (less common in Git tag names).

Map from Conventional Commits (`0.2.9`):

| Commit prefix | Typical bump |
|---------------|--------------|
| fix: | PATCH |
| feat: | MINOR |
| feat!: or BREAKING CHANGE | MAJOR |

Tag **after** merge to release branch or `main`, on the **exact commit** CI built and tested — not before.

Typical flow after hotfix merge (`0.2.11`):

```bash
git checkout main
git pull
git tag -a v2.3.2 -m "v2.3.2: revert login regression"
git push origin v2.3.2
```

## Under the Hood

**Lightweight tag** storage: single ref file `.git/refs/tags/v2.3.1` containing 40-char commit hash (or packed-refs entry).

**Annotated tag** storage: ref points to **tag object** hash; tag object stores:

- object type (commit)
- object hash (target commit)
- tagger, date, message
- optional PGP signature

```
refs/tags/v2.3.1 → tag object t1a2g3...
tag object t1a2g3 → commit abc123...
```

`git cat-file -p v2.3.1` shows tag object for annotated tags; lightweight shows commit directly.

**Peeling:** commands that need commit hash **peel** annotated tags — follow tag object to commit. `git rev-parse v2.3.1^{commit}` explicit peel.

**Fetch behavior:** `git fetch` does not fetch all tags by default in all configs; `git fetch --tags` or remote tag refs update local tags. Clone may get tags reachable from default branch depending on `remote.tagOpt`.

Tags participate in **ref namespace** like branches but conventionally immutable. `git checkout v2.3.1` → detached HEAD at tagged commit (`0.1.11`).

## Syntax / Commands / API

```bash
git tag                          # list
git tag v1.0.0                   # lightweight
git tag -a v1.0.0 -m "message"   # annotated
git tag -a v1.0.0 <commit>       # tag specific commit
git push origin v1.0.0
git push origin --tags
git tag -d v1.0.0                # delete local
git push origin --delete v1.0.0  # delete remote
git checkout v1.0.0              # detached at release
```

## Examples

### Example 1: Release after PR merge

```bash
git checkout main && git pull
# CI green on merge commit abc1234
git tag -a v1.4.0 -m "v1.4.0: user dashboard feature"
git push origin v1.4.0
```

Pipeline triggers on tag push pattern `v*`.

### Example 2: Tag old commit for bisect baseline

```bash
git tag -l "v2.*"
git bisect start main v2.4.0   # v2.4.0 is known good tag
```

### Example 3: Lightweight vs annotated inspection

```bash
git tag demo-light HEAD~1
git tag -a demo-annot -m "annotated demo" HEAD
git cat-file -t demo-light    # commit
git cat-file -t demo-annot    # tag
```

## Common Mistakes & Anti-patterns

**Lightweight tags for production semver** — missing audit metadata.

**Tagging before CI passes** — tag points at broken commit.

**Retagging v2.0.0 to different hash** — breaks trust; ship v2.0.1.

**`git push --tags` blindly** — pushes experimental local tags.

**Forgetting to push tag** — release docs say v2.3.1 but remote has no ref.

**Tag on wrong branch tip** — tagged feature branch instead of merge commit on `main`.

## Production & Real-World Notes

**Release automation:** GitHub Releases, GitLab Release from annotated tag; changelog from Conventional Commits. **Protected tags** rules on who can push `v*`. **Container images** tagged `myapp:2.3.1` matching Git tag.

**LTS lines:** `v2.3.x` patch tags on maintenance branch; `main` at `v3.0.0`. Cherry-pick hotfixes (`0.2.11`) then tag each patch.

**Monorepos** (`0.2.15`) may tag per package (`api/v1.2.0`) — team convention required.

## Comparison / Trade-offs

| Type | Metadata | Use case |
|------|----------|----------|
| Lightweight | None (ref only) | Local, temp |
| Annotated | Message, tagger, optional GPG | Releases |
| Branch | Moves with commits | Development |

Semver vs calver: semver for libraries/APIs; calver (`2026.07.03`) for continuous deploy apps — pick team standard.

## Key Takeaways

- Tags mark **immutable** release points; branches move.
- **Annotated tags** for real releases; lightweight for quick local markers.
- **Semver** communicates change impact; align with Conventional Commits.
- Push tags explicitly; **never retag** published semver.
- Tags are refs in `refs/tags/` — peeled to commits for build/bisect.
- Hotfix flow ends with **tag + push** on verified merge commit.
