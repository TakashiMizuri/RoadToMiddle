# Namespaces, `using`, file-scoped namespaces

> Roadmap: `1.1.2` · Node: `1.1` — C#: основы языка · Depth: **практика**

## Learning Objectives

После этого урока ты сможешь:

- Объяснить **namespaces** как логические контейнеры, предотвращающие коллизии имён типов.
- Разрешать имена через **fully qualified name (FQN)** и директивы **`using`**.
- Различать **`using` import**, **`using static`** и **`using alias`**.
- Применять **file-scoped namespaces** (C# 10+) и сравнивать с блочным `namespace { }`.
- Связать **default namespace** project с структурой папок и **`RootNamespace`**.
- Организовывать типы в multi-file project без ambiguity.

## Why This Matters

В `1.1.1` был однофайловый console app. В реальных solution — сотни типов в десятках assemblies: `OrderService`, `UserController`, `StringExtensions`. Без системы имён две команды могут объявить `class Customer` — compiler error или привязка не к тому типу.

**Namespaces** — основной способ группировки типов в C#. Они частично повторяют иерархию папок и границы продукта: `Contoso.Billing.Invoices`. Вместе с **`using`** call site остаётся читаемым: `List<Order>` вместо длинного FQN.

Middle-разработчик ежедневно рефакторит layout namespaces — перенос DTO в shared library, ambiguous reference после NuGet upgrade, выравнивание folder ↔ namespace. File-scoped namespaces убирают лишний indent; знание обоих стилей нужно для legacy и modern code.

## Core Concepts

### Namespace как префикс qualified name

```csharp
namespace Contoso.App.Services;

public class EmailSender { }
```

FQN: `Contoso.App.Services.EmailSender`.

Compiler использует namespaces для **disambiguation**. `Contoso.App.Models.Customer` и `Contoso.Crm.Models.Customer` — разные типы. На call site — FQN или `using`.

Физические папки **не обязаны** совпадать с namespace — это convention для людей, не требование compiler.

### Block-scoped vs file-scoped

**Классический (block-scoped):**

```csharp
namespace Contoso.App
{
    public class Program
    {
        public static void Main() { }
    }
}
```

**File-scoped (C# 10+):**

```csharp
namespace Contoso.App;

public class Program
{
    public static void Main() { }
}
```

Один file-scoped namespace на файл, до остальных members. Modern SDK templates используют file-scoped.

### Директива `using`

В начале файла (после file-scoped namespace):

```csharp
using System;
using System.Collections.Generic;

namespace MyApp;

class Demo
{
    static void Run()
    {
        var list = new List<int>();
        Console.WriteLine(list.Count);
    }
}
```

**Global usings** — в `GlobalUsings.cs` или через `<ImplicitUsings>enable</ImplicitUsings>` — для всего project.

### FQN когда нужна точность

Два imported namespace с одним short name → **ambiguity**. Решения:

1. FQN: `System.Text.Json.JsonSerializer`
2. **Alias:** `using SysJson = System.Text.Json.JsonSerializer;`
3. Убрать один `using`, qualify только с одной стороны

### `using static`

Импортирует **static members** типа:

```csharp
using static System.Console;
WriteLine("Без префикса Console");
```

В production — умеренно; иначе неясно, откуда `WriteLine`.

### Default namespace

`dotnet new console -n MyApp` задаёт root namespace от имени project:

```xml
<RootNamespace>MyApp</RootNamespace>
```

Rename project **не** переименовывает namespaces в существующих файлах автоматически.

### Вложенные namespaces

```csharp
namespace A.B.C { class T { } }
// эквивалентно вложенным block namespace A { B { C { ... } } }
```

Предпочтительнее dotted form: `namespace A.B.C;`

## Under the Hood

### Name binding при компиляции

Roslyn **semantic model** резолвит identifier → symbol. Для `List<int>`: local scopes → imported namespaces → global namespace.

Namespaces — **compile-time** организация; в IL metadata qualified names типов сохраняются для reflection.

### Assemblies vs namespaces

**Assembly** (`.dll`) — unit деплоя/сборки. **Namespace** — логика. Одна assembly — много namespaces; один namespace — типы в разных assemblies (`System.*`). `using` импортирует namespaces; **references** подключают assemblies.

## Examples

### Multi-file project

**Models/User.cs**

```csharp
namespace RoadmapApp.Models;

public record User(int Id, string Name);
```

**Services/Greeter.cs**

```csharp
using RoadmapApp.Models;

namespace RoadmapApp.Services;

public static class Greeter
{
    public static string Greet(User user) => $"Hello, {user.Name}!";
}
```

**Program.cs**

```csharp
using RoadmapApp.Models;
using RoadmapApp.Services;

var user = new User(1, "Ada");
Console.WriteLine(Greeter.Greet(user));
```

### Ambiguity с alias

```csharp
using TextJson = System.Text.Json;
using NewtonsoftJson = Newtonsoft.Json;

namespace SerializationDemo;

class Compare
{
    static string ToSystemJson(object o) =>
        TextJson.JsonSerializer.Serialize(o);
}
```

## Common Mistakes & Anti-patterns

- Namespace ≠ folder — не ошибка, но мешает navigation.
- Нет `using System.*` в C# — только явные namespaces.
- Два file-scoped namespace в одном файле — CS8954.
- **God namespace** — все типы в `MyApp` без feature split.

---

## Production & Real-World Notes

Convention: `Company.Product.Feature`; tests — `*.Tests`. Analyzers проверяют alignment folder/namespace. Public API libraries держат stable namespaces; breaking changes иногда через `V2` subnamespace.

---

## Comparison / Trade-offs

File-scoped — меньше indent, один namespace на файл. Block-scoped — редко несколько namespaces в файле. FQN — zero ambiguity, verbose. `using` — readable, риск ambiguity.

## Quick Reference

```csharp
namespace A.B;              // file-scoped
using X = Long.Type;        // alias
using static System.Math;
global using System;
```

---

## Key Takeaways

- **Namespace** = logical группа; **FQN** = полный путь к типу.
- **`using`** — short names; ambiguity → qualify или alias.
- **File-scoped** (`namespace X.Y;`) — весь файл, C# 10+.
- **Assembly** ≠ **namespace**.
- **`RootNamespace`** в `.csproj` — default для новых файлов.
