# Amending commits — краткая выжимка

> `0.1.10` · ~6 мин на повторение

## Главная идея

**`git commit --amend`** заменяет **последний commit** новым (новый SHA), чтобы исправить message или добавить забытые файлы **без лишнего commit**. Безопасно только для **local/unpushed** work. На **shared/pushed** history — опасно: требует force push и ломает clones коллег.

## Ключевые пункты

- Amend **не редактирует** commit — создаёт **новый** и двигает branch pointer.
- Старый commit становится **dangling**, но recoverable через **reflog** локально.
- **`git add` + `git commit --amend --no-edit`** — добавить файлы, message не менять.
- После push amend → **non-fast-forward reject** → соблазн **`git push --force`**.
- **Golden rule:** не переписывать commits, которые другие уже pull'нули.
- На shared branches — **новый commit** или **`git revert`**, не amend.
- Для commits старше HEAD — **interactive rebase** (отдельная тема).

## Самые важные примеры

**Безопасно (не push'или):**
```bash
git add Forgotten.cs
git commit --amend --no-edit
git push origin feature/x   # первый push — OK
```

**Опасно (уже на main):**
```bash
git commit --amend --no-edit
git push --force origin main   # ❌ ломает команду
```

**Правильная альтернатива на shared:**
```bash
git add Forgotten.cs
git commit -m "Add missing file to health check feature"
```

## Частые ошибки

- Force-push `main` после amend — классический team incident.
- Забыли `git add` — amend меняет только message.
- Amend вместо rebase для нескольких commits подряд.

## Мнемоника

> **Amend = переписать последний, если никто не видел.** Push'нул — **новый commit**, не amend.
