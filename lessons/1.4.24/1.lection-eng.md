# Task.Factory.StartNew — Why Avoid It, Use Task.Run

> Roadmap: `1.4.24` · Node: `1.4` — Async + ASP.NET preview · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Explain **Task.Factory.StartNew** overload resolution and default **TaskScheduler** / **TaskCreationOptions** behavior.
- Identify bugs from **StartNew + async lambda** returning **`Task<Task>`** without **`Unwrap`**.
- Choose **`Task.Run`** for CPU-bound offload to thread pool with predictable defaults.
- Describe when **LongRunning** and custom schedulers are legitimate for **StartNew**.
- Relate misuse of **StartNew** to **ASP.NET thread pool starvation** and hidden **SyncContext** capture.
- Refactor legacy **StartNew** call sites to **`Task.Run`** or **true async I/O**.

## Why This Matters

`Task.Factory.StartNew` appeared in .NET 4.0 as a flexible way to schedule work on the thread pool or a custom **TaskScheduler**. Over a decade of production incidents taught the community a simpler rule: **`Task.Run` for thread pool queueing**, **`StartNew` only when you explicitly need non-default scheduler or creation options**. StartNew's defaults surprise developers: **`TaskCreationOptions.None`** still schedules on **current TaskScheduler** if not overridden — which is often **not** the thread pool in UI or custom parallel code. Worse, **`StartNew(async () => await Something())`** returns **`Task<Task>`** because the `async` lambda returns `Task`, and StartNew wraps that as the task body result — callers await the outer task and think work is done when only the delegate **started**.

In ASP.NET Core, offloading CPU work with **`Task.Run`** inside a request is sometimes valid (image resize, parsing), but **`StartNew` with wrong options** has queued work on unexpected schedulers or completed outer tasks before inner async work finished — causing **race bugs** and **premature response commits**. Middle developers must recognize StartNew in legacy codebases and migrate safely.

## Core Concepts

### StartNew Overloads and Defaults

```csharp
Task.Factory.StartNew(() => CpuWork()); // many overloads exist
```

Key parameters:

- **`Action` / `Func<T>` / `Func<Task>`** — work item.
- **`CancellationToken`**
- **`TaskCreationOptions`** — `LongRunning`, `PreferFairness`, `AttachedToParent`, `RunContinuationsAsynchronously`, etc.
- **`TaskScheduler`** — **`TaskScheduler.Current`** by default in some overloads, **`Default`** (thread pool) in others — **read the exact overload**.

The classic pitfall overload:

```csharp
// DANGEROUS with async lambda:
Task<Task> outer = Task.Factory.StartNew(async () =>
{
    await HttpClient.GetStringAsync(url);
});
// outer.IsCompleted may be true before HTTP finishes!
```

Correct patterns:

```csharp
await Task.Factory.StartNew(
    () => HttpClient.GetStringAsync(url),
    CancellationToken.None,
    TaskCreationOptions.DenyChildAttach,
    TaskScheduler.Default).Unwrap();

// Preferred for thread pool:
await Task.Run(async () => await HttpClient.GetStringAsync(url));
```

**Task.Run** is implemented roughly as **StartNew with TaskScheduler.Default** and safer defaults for common cases — not identical internally but **intent-clear**.

### Task.Run — When and Why

```csharp
await Task.Run(() => HeavyCpuTransform(data), cancellationToken);
```

**Use Task.Run when:**

- CPU-bound work would block ASP.NET request thread unacceptably long.
- You need thread pool execution with **predictable** scheduling.
- You are **not** replacing async I/O — **never** Task.Run just to wrap sync HTTP.

**Do not use Task.Run when:**

- Work is already I/O-bound — use `await` on truly async API.
- You need **LongRunning** dedicated thread — rare; consider **`TaskCreationOptions.LongRunning`** with explicit scheduler (background services).

### LongRunning and Custom Schedulers

**`TaskCreationOptions.LongRunning`** hints the scheduler to create **dedicated thread** instead of pool thread — for **blocking** loops listening on legacy APIs. **`TaskScheduler.FromCurrentSynchronizationContext()`** (see `1.4.21`) pairs with StartNew to marshal back to UI thread — legitimate StartNew use case.

Parallel class and **TPL Dataflow** use custom schedulers — StartNew remains relevant **inside framework code**, not typical app controllers.

## Under the Hood

**Task.Run** queues work to the global thread pool queue via **`ThreadPool.QueueUserWorkItem`** path (simplified). **StartNew** creates a **Task** object with scheduler reference; **`task.ScheduleAndStart`** asks the scheduler to queue it.

**AttachedToParent** (legacy) tied child completion to parent — error-prone with async; modern code avoids.

**DenyChildAttach** prevents child tasks attaching — safer default in migrations.

When outer **`Task<Task>`** completes, only the **async state machine started** — inner task still running. **`Unwrap()`** flattens **`Task<Task>`** → **`Task`**. **`Task.Run` with async delegate** handles unwrap pattern internally for you.

## Syntax / Commands / API

| API | Default scheduler | Async lambda safe? |
|-----|-------------------|-------------------|
| `Task.Run(action)` | Thread pool | Yes with `async` lambda |
| `Task.Factory.StartNew(action)` | **Varies by overload** | **Risk Task<Task>** |
| `StartNew(..., LongRunning)` | Often dedicated thread | Still unwrap issues |

Migration recipe:

```csharp
// Before
Task.Factory.StartNew(() => DoCpu(), token);

// After
Task.Run(() => DoCpu(), token);
```

```csharp
// Before (broken async)
Task.Factory.StartNew(async () => await DoIo());

// After
await DoIo(); // if true async I/O
// OR
await Task.Run(async () => await DoIo()); // if must offload wrapper — rare
```

## Examples

### Example 1: Task<Task> surprise

```csharp
var t = Task.Factory.StartNew(async () => { await Task.Delay(1000); });
await t; // outer completes ~immediately
// HTTP/async work may still run unattached
```

Fix: **`Unwrap()`** or **`Task.Run`**.

### Example 2: CPU parse in ASP.NET

```csharp
public async Task<IActionResult> Upload(IFormFile file)
{
    await using var ms = new MemoryStream();
    await file.CopyToAsync(ms);
    var rows = await Task.Run(() => ParseCsv(ms.ToArray()), HttpContext.RequestAborted);
    return Ok(rows);
}
```

**ParseCsv** is CPU-bound; **`CopyToAsync`** is true async I/O — no Task.Run on I/O.

### Example 3: LongRunning listener (legitimate StartNew)

```csharp
Task.Factory.StartNew(
    () => BlockingLegacyLoop(),
    TaskCreationOptions.LongRunning);
```

Dedicated thread for legacy blocking API — not for HTTP.

## Common Mistakes & Anti-patterns

**StartNew + async lambda without Unwrap** — #1 bug.

**Assuming StartNew always uses thread pool** — check scheduler parameter.

**Task.Run for every method** — adds overhead; only CPU-bound sections.

**AttachedToParent in async graphs** — deadlocks and confusing completion.

**Replacing sync-over-async with StartNew** — still blocks a pool thread inside delegate.

## Production & Real-World Notes

Code analyzers and reviews often flag **`Task.Factory.StartNew`** — replace with **`Task.Run`** unless comment justifies scheduler. Roslyn analyzers (e.g. **CA1414**, community rules) target StartNew misuse.

In **ASP.NET Core**, prefer **`BackgroundService`** + **`Channel`** for long-running work over fire-and-forget **StartNew**. For request-scoped CPU bursts, **`Task.Run`** with **`RequestAborted`** token is acceptable.

Legacy .NET Framework apps with **`Parallel.ForEach` + StartNew** need careful audit during porting to Core.

## Comparison / Trade-offs

| | Task.Run | Task.Factory.StartNew |
|---|----------|------------------------|
| Intent | Queue CPU work to pool | Full scheduler control |
| Footguns | Fewer | Many (scheduler, Task<Task>) |
| Async lambda | Supported correctly | Needs Unwrap |
| LongRunning | No | Yes (explicit options) |
| Typical app usage | **Preferred** | **Legacy / special** |

## Quick Reference

- **Default rule:** `Task.Run` for thread pool offload.
- **Never:** `StartNew(async () => ...)` without **`Unwrap()`**.
- **I/O:** `await` async APIs — not Run/StartNew wrapper on sync HTTP.
- **LongRunning:** dedicated thread — blocking legacy only.
- **ASP.NET:** Task.Run + **`RequestAborted`** for CPU sections.

---

## Key Takeaways

- **StartNew** defaults and overloads confuse even senior developers — **`Task.Run`** is the safe default.
- **async lambda + StartNew** → **`Task<Task>`** unless **`Unwrap()`**.
- **Task.Run** does not make I/O async — use real async HTTP/EF.
- **LongRunning** and custom schedulers are valid **StartNew** scenarios — rare in web apps.
- Migrate legacy **StartNew** to **Task.Run** or remove unnecessary offload.
- ASP.NET: CPU-bound **`Task.Run`**, I/O-bound **`await`**, background work **`BackgroundService`**.
