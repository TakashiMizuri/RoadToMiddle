# Polly — Retry, Timeout, Circuit Breaker with HttpClient

> Roadmap: `1.4.28` · Node: `1.4` — Async + ASP.NET preview · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Explain **retry**, **timeout**, and **circuit breaker** policies for outbound HTTP and their **trade-offs**.
- Apply **Polly v8** **`ResiliencePipeline`** / **`AddStandardResilienceHandler`** to **`IHttpClientFactory`** clients.
- Configure **jittered exponential backoff**, **handled status codes**, and **circuit break duration**.
- Predict **retry storms** when combined with **`Parallel.ForEachAsync`** high parallelism (`1.4.25`).
- Distinguish **per-attempt timeout** vs **HttpClient.Timeout** vs **total request budget**.
- Monitor circuit state transitions and avoid **masking** permanent **4xx** errors with blind retries.

---

## Why This Matters

Downstream microservices fail — transient network blips, **503** during deploy, slow responses. Naive **`HttpClient`** calls fail fast to your users unless you add **resilience**. Blind retries amplify outages (**retry storm**). **Circuit breaker** stops hammering sick dependencies, giving them recovery time and failing fast on your side.

**Polly** is the de facto .NET resilience library; **.NET 8+** ships **`Microsoft.Extensions.Http.Resilience`** integrating Polly with **`IHttpClientFactory`** (`1.4.27`). ASP.NET Core apps expose SLAs — Middle developers choose **which errors retry**, **how many attempts**, **timeout per try**, and **when to open the circuit** — not copy-paste blog snippets.

Wrong policy: retry **400 Bad Request**, infinite retry on **500**, no timeout → cascading latency. Right policy: **idempotent GET** retry with cap, **breaker** after failure threshold, **correlation** in logs.

## Core Concepts

### Retry

**Retry** re-executes failed attempts for **transient** faults: **`HttpRequestException`**, **408**, **429**, **502**, **503**, **504** (team-specific list).

```csharp
// Polly v8 style (conceptual)
new ResiliencePipelineBuilder<HttpResponseMessage>()
    .AddRetry(new RetryStrategyOptions<HttpResponseMessage>
    {
        MaxRetryAttempts = 3,
        Delay = TimeSpan.FromSeconds(1),
        BackoffType = DelayBackoffType.Exponential,
        UseJitter = true,
        ShouldHandle = new PredicateBuilder<HttpResponseMessage>()
            .HandleResult(r => (int)r.StatusCode == 503)
    });
```

**Jitter** spreads retries across clients — avoids synchronized thundering herd after recovery.

**Only retry safe/idempotent operations** unless **`Idempotency-Key`** header on POST — default safe: **GET**, **PUT** with idempotency design.

### Timeout

Layers:

| Layer | Scope |
|-------|--------|
| **`HttpClient.Timeout`** | Entire operation including retries (default 100s!) |
| **Per-attempt timeout** (Polly) | Single try |
| **CancellationToken** (ASP.NET) | User disconnect / budget |

Set **`HttpClient.Timeout`** **infinite** or high when Polly per-try timeout controls — otherwise **`HttpClient`** wins first.

```csharp
.AddTimeout(TimeSpan.FromSeconds(5)); // per attempt in pipeline
```

### Circuit Breaker

**Closed** (normal) → failures exceed threshold → **Open** (fail fast without calling downstream) → after break duration → **Half-Open** (trial call) → **Closed** or **Open**.

```csharp
.AddCircuitBreaker(new CircuitBreakerStrategyOptions<HttpResponseMessage>
{
    FailureRatio = 0.5,
    MinimumThroughput = 10,
    BreakDuration = TimeSpan.FromSeconds(30),
    ShouldHandle = ...
});
```

Prevents wasted threads and protects downstream during outage. Return **503** or fallback cache to callers.

### Standard Resilience Handler (.NET 8+)

```csharp
builder.Services.AddHttpClient("catalog")
    .AddStandardResilienceHandler(options =>
    {
        options.Retry.MaxRetryAttempts = 3;
        options.TotalRequestTimeout.Timeout = TimeSpan.FromSeconds(30);
        options.CircuitBreaker.SamplingDuration = TimeSpan.FromSeconds(30);
    });
```

Registers opinionated defaults aligned with Microsoft guidance — customize per downstream criticality.

Legacy **`AddPolicyHandler`** from **`Microsoft.Extensions.Http.Polly`** — migrate to v8 pipelines for new code.

## Under the Hood

Polly v8 uses **`ResiliencePipeline`** composable strategies executed around delegate. HTTP integration wraps **`SendAsync`** in **`DelegatingHandler`**.

Each retry = new HTTP attempt through handler chain — **auth handler** runs again (ensure token refresh idempotent). **Connection pool** reuse from factory still applies.

**Circuit open** throws **`BrokenCircuitException`** quickly — log and metric **breaker state** for dashboards.

## Examples

### Example 1: Retry only 503

Avoid retry on **404** — permanent. Predicate on status code + **`HttpRequestException`**.

### Example 2: Retry storm math

100 concurrent **`ForEachAsync`** × 4 retries × downstream dying = **400+** calls/sec — combine **lower parallelism**, **breaker**, **retry cap**.

### Example 3: Fallback (advanced)

```csharp
.AddFallback(new FallbackStrategyOptions<HttpResponseMessage>
{
    FallbackAction = static args =>
        Outcome.FromResultAsTask(new HttpResponseMessage(HttpStatusCode.OK)
        {
            Content = new StringContent("{\"cached\":true}")
        })
});
```

Use sparingly — stale data trade-off.

## Common Mistakes & Anti-patterns

**Retry all exceptions** including **400**, **401**, **404**.

**No jitter** — synchronized retries.

**HttpClient.Timeout = 100s** with many retries — hangs request thread budget.

**High parallelism + aggressive retry** — kills downstream (`1.4.25`).

**Circuit breaker without observability** — ops cannot see open state.

**Non-idempotent POST retry** without idempotency key.

## Production & Real-World Notes

Define **SLI/SLO** per dependency — payments stricter than analytics. **Chaos testing** validates breaker. Align with **`429 Retry-After`** header when present.

**Link headers** and **ProblemDetails** from downstream — surface to caller, do not infinite retry.

Combine with **`CancellationToken`** from **`RequestAborted`** (`1.4.29`) — cancel retries when client disconnects.

## Comparison / Trade-offs

| Strategy | Helps | Hurts if misused |
|----------|-------|------------------|
| Retry | Transient blips | Amplifies load |
| Timeout | Bounds latency | Too short → false failures |
| Circuit breaker | Fail fast, recovery | Wrong threshold → flapping |
| Bulkhead | Isolation | Complexity |

---

## Quick Reference

- Retry: transient only, jitter, max attempts, idempotent ops.
- Timeout: per-attempt + total budget; align HttpClient.Timeout.
- Breaker: threshold, break duration, half-open probe.
- Factory: **AddStandardResilienceHandler** (.NET 8+).
- Parallelism × retries = load multiplier — cap both.

---

## Key Takeaways

- **Polly** adds **retry**, **timeout**, **circuit breaker** to outbound HTTP via handler pipeline.
- **.NET 8+**: **`AddStandardResilienceHandler`** on **`IHttpClientFactory`** clients.
- Retry **transient** faults only; **jitter** backoff; respect **idempotency**.
- **Circuit breaker** protects sick dependencies and your thread pool.
- **`HttpClient.Timeout`** interacts with Polly — configure consciously.
- **Retry storms** with high **`Parallel.ForEachAsync`** — tune parallelism + policies together.
- Log and metric breaker state; do not retry permanent **4xx**.
