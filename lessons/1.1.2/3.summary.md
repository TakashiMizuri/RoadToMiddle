# Namespaces и using — краткая выжимка

> `1.1.2` · ~8 мин на повторение

## Главная идея

**Namespace** — логический префикс типа (**FQN** = `A.B.TypeName`), предотвращает коллизии имён. **`using`** импортирует namespace для коротких имён на call site. **File-scoped namespace** (`namespace X.Y;`) задаёт namespace на весь файл без лишних `{}`.

## Ключевые пункты

- **Block-scoped** `namespace X { }` vs **file-scoped** `namespace X;` — один namespace на файл в modern style.
- Ambiguity при двух `Customer` → **FQN** или **`using Alias = ...`**.
- **`using static`** — static members типа в scope (осторожно с читаемостью).
- **`global using`** / **ImplicitUsings** — project-wide imports.
- **Assembly** (deploy unit) ≠ **namespace** (logical grouping).
- **`RootNamespace`** в `.csproj` — default namespace новых файлов.

## Самые важные примеры

```csharp
namespace RoadmapApp.Services;

using RoadmapApp.Models;

public static class Greeter
{
    public static string Greet(User u) => $"Hello, {u.Name}!";
}
```

```csharp
using SysJson = System.Text.Json.JsonSerializer;
```

## «Под капотом» в одном абзаце

Roslyn при компиляции ищет symbol для каждого identifier: locals → imported namespaces → global. Qualified names типов остаются в assembly metadata для reflection. `using` не «копирует» код — только влияет на name resolution на этапе compile.

## Частые ошибки

- Два одинаковых short name без qualify/alias.
- Два file-scoped namespace в одном файле.
- Ожидать `using System.*` как в Java.

## Мнемоника

> **Namespace** — адрес типа; **using** — «ярлык» для короткой записи; при конфликте — полный адрес (FQN) или псевдоним.
