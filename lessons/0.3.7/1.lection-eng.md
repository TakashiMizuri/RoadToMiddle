# HTTP Headers: Content-Type, Accept, Authorization, Cookie, Cache-Control, ETag

## Learning Objectives

After this lesson you will be able to:

- Explain headers as **metadata** accompanying the start line and body from `0.3.3`.
- Negotiate representation format with **Content-Type** (request/response) and **Accept** (content negotiation).

## Why This Matters

Status codes (`0.3.6`) answer *whether* a request succeeded. Headers answer *how to interpret* the message, *who is calling*, *how long to cache*, and *what format* the body uses. A JSON API that omits `Content-Type: application/json` still works until a strict client, proxy, or security scanner rejects it. A browser that sends `Accept: text/html` to an API may receive HTML error pages instead of JSON unless you configure content negotiation. Cache misconfiguration ships stale prices or leaks private data into shared CDN nodes.

Every fullstack developer sets headers daily ‚Äî in controllers, middleware, nginx, CloudFront, and frontend `fetch` calls. Middle-level work requires knowing not just header *names* but **directive semantics**, **security attributes**, and **interaction** with status codes and methods from `0.3.4`‚Äì`0.3.5`. This lesson builds that mental model for the six headers that appear in almost every production trace.

## Core Concepts

### Headers: Case-Insensitive Key‚ÄìValue Metadata

HTTP headers are `Name: value` lines after the start line. Names are case-insensitive (`Content-Type` = `content-type`). Values may carry parameters (`Content-Type: application/json; charset=utf-8`). Headers split roughly into **request headers** (client ‚Üí server), **response headers** (server ‚Üí client), and **entity headers** describing the body.

Some headers are **forbidden** or **restricted** in browser JavaScript (e.g. setting `Host` from fetch) ‚Äî the user agent controls them for security. Server frameworks expose others via attributes (`[FromHeader]`) or middleware.

### Content-Type: What the Body Is

**Content-Type** declares media type of the **message body** (when present). The server uses it on responses; clients use it on requests with bodies (POST/PUT/PATCH).

Common values:

- `application/json` ‚Äî JSON APIs (default in modern ASP.NET Core).
- `application/x-www-form-urlencoded` ‚Äî HTML form default (`key=value&key2=value2`).
- `multipart/form-data` ‚Äî file uploads with boundaries (see `0.3.11`).

If the client sends JSON but `Content-Type: text/plain`, model binding may fail or parse incorrectly ‚Üí **415 Unsupported Media Type** or **400**. ASP.NET Core `[Consumes("application/json")]` enforces expected types.

Charset parameter matters for text: `text/html; charset=utf-8` prevents mojibake. JSON should be UTF-8; always specify charset for text responses.

### Accept: What the Client Wants Back

**Accept** performs **content negotiation**: client lists preferred response media types, optionally with **q-values** (quality weights):

```
Accept: application/json, text/plain;q=0.9, */*;q=0.8
```

Server picks the best match it can produce. APIs often ignore Accept and always return JSON; MVC apps return HTML or JSON based on Accept and route (`Accept: application/json` ‚Üí Problem Details JSON on errors).

Mismatch: client Accepts only `application/xml`, server offers JSON ‚Üí **406 Not Acceptable** (rare in practice; many servers ignore Accept).

**Accept** vs **Content-Type**: Accept = desired **response** format; Content-Type = actual **request/response body** format. Beginners swap them constantly ‚Äî remember direction.

### Authorization: Proving Identity (Stateless Credential)

**Authorization** carries credentials **for this request** ‚Äî typically:

```
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
Authorization: Basic dXNlcjpwYXNz
```

**Bearer** ‚Äî OAuth 2.0 / JWT access tokens; server validates signature, expiry, audience. **Basic** ‚Äî base64(`user:pass`), only over HTTPS; legacy and internal tools.

Unlike cookies (session id stored by browser, auto-sent), Authorization is **explicit** ‚Äî SPAs and mobile apps prefer it. ASP.NET Core JWT Bearer middleware reads this header.

Do not log full Authorization values ‚Äî secrets in log aggregators are a compliance incident.

**401** responses may include `WWW-Authenticate` describing required scheme ‚Äî pairs with `0.3.6`.

### Cookie: Session State on the Wire (Request Header)

The **Cookie** **request** header sends stored name=value pairs previously set by **Set-Cookie** **response** headers (detailed in `0.3.9`):

```
Cookie: session_id=abc; theme=dark
```

Browsers attach matching cookies automatically per domain/path rules. APIs for SPAs often use **HttpOnly** session cookies or move tokens to Authorization header.

**Cookie** (request) vs **Set-Cookie** (response): client echoes vs server instructs storage. Security attributes (`Secure`, `HttpOnly`, `SameSite`) live on Set-Cookie ‚Äî next lesson.

Server reads `Cookie` for session authentication; ASP.NET Core cookie authentication middleware validates encrypted session cookie.

### Cache-Control: Directing Caches

**Cache-Control** is the primary modern caching directive (replacing much of legacy `Expires`). It appears on **requests and responses**.

Response examples:

- `Cache-Control: no-store` ‚Äî do not store anywhere (sensitive data, PCI).
- `Cache-Control: private, max-age=3600` ‚Äî browser may cache 1 hour; shared CDN should not.
- `Cache-Control: public, max-age=86400, immutable` ‚Äî CDN may cache static assets with hash in filename.

Request examples:

- `Cache-Control: no-cache` ‚Äî cache may store but **must revalidate** with origin before use.
- `Cache-Control: max-age=0` ‚Äî similar revalidation intent.

Misusing `public` on personalized `/api/me` responses leaks user data through shared caches ‚Äî catastrophic.

Caching connects to **304 Not Modified** (`0.3.6`): revalidation succeeds ‚Üí 304, no body.

### ETag: Entity Tag for Conditional Requests

**ETag** is an opaque validator for a **specific version** of a representation:

```
ETag: "33a64df55142582555b22e4d9586"
```

Strong ETag ‚Äî byte-identical body. Weak ETag (`W/"..."`) ‚Äî semantically equivalent but not byte-identical.

Client sends conditional headers on subsequent GET:

```
If-None-Match: "33a64df55142582555b22e4d9586"
```

If unchanged ‚Üí **304**; if changed ‚Üí **200** with new body and new ETag.

**If-Match** used with PUT for optimistic concurrency ‚Äî mismatch ‚Üí **412 Precondition Failed** or **409**.

ETags enable bandwidth savings and concurrency control without custom version query parameters.

## Under the Hood

### Header Processing Order in ASP.NET Core

```
Kestrel parses headers ‚Üí Middleware (CORS adds Access-Control-*) ‚Üí
Authentication reads Authorization/Cookie ‚Üí
Content negotiation (Accept, Content-Type) ‚Üí
Action ‚Üí Response headers (Cache-Control, ETag) added by filters or middleware
```

**Forwarded headers** (`X-Forwarded-For`, `X-Forwarded-Proto`) matter behind reverse proxies ‚Äî otherwise HTTPS and client IP are wrong for redirects and rate limiting.

Model binding reads `Content-Type` to choose input formatter. Output formatters set response `Content-Type`.

### How Browsers Build Request Headers

For same-origin navigation, the browser sets `Accept`, `Cookie`, `User-Agent`, and security headers automatically. `fetch(url, { headers: { Authorization: 'Bearer ...' } })` adds Authorization but **cannot** set forbidden headers. CORS preflight (`OPTIONS`) checks whether `Authorization` and custom headers are allowed ‚Äî preview of later security topics.

### Cache Layers

```
Browser cache ‚Üí CDN ‚Üí Reverse proxy ‚Üí Origin
```

Each layer interprets `Cache-Control`, `ETag`, `Vary`. **`Vary: Accept-Encoding`** prevents serving gzip cache to client expecting identity. **`Vary: Authorization`** prevents serving one user's cached `/api/me` to another ‚Äî essential if caching authenticated responses at all (usually avoid).

## Syntax / Commands / API

### curl examples

```bash
curl -H "Accept: application/json" https://api.example.com/items/1

curl -X POST https://api.example.com/items \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer TOKEN" \
  -d '{"name":"Widget"}'

curl -I https://cdn.example.com/app.main.js
# Cache-Control, ETag in response headers
```

Conditional GET:

```bash
curl -H 'If-None-Match: "abc123"' -i https://api.example.com/items/1
# 304 if unchanged
```

### fetch (browser / React)

```javascript
fetch('/api/items', {
  headers: {
    'Content-Type': 'application/json',
    Accept: 'application/json',
    Authorization: `Bearer ${token}`,
  },
});
```

### ASP.NET Core

```csharp
[HttpGet]
[ResponseCache(Duration = 60, Location = ResponseCacheLocation.Client)]
public IActionResult GetPublicCatalog() => Ok(catalog);

Response.Headers.ETag = $"\"{versionHash}\"";
Response.Headers.CacheControl = "private, no-store";
```

## Examples

### Example 1: JSON API call chain

Client POST with `Content-Type: application/json` and `Authorization: Bearer ...`. Server validates JWT, binds JSON body, returns `Content-Type: application/json` response with `Cache-Control: no-store` (personalized data).

### Example 2: Static asset caching

Build outputs `app.a1b2c3.js`. Server responds `Cache-Control: public, max-age=31536000, immutable` and strong ETag. Browser caches aggressively; filename hash changes on deploy ‚Äî no stale bundle.

### Example 3: Conditional GET

GET `/catalog` returns ETag `"v47"`. Client caches body. Later GET sends `If-None-Match: "v47"`. Catalog unchanged ‚Üí **304**, zero JSON bytes. Catalog updated ‚Üí **200**, new ETag `"v48"`.

## Common Mistakes & Anti-patterns

**Swapping Accept and Content-Type.** Accept = want; Content-Type = is.

**Missing charset on text/html.** Encoding bugs in production.

**Logging Authorization or Cookie headers.** Credential leak.

**Cache-Control: public on authenticated JSON.** Shared cache serves private data.

**Ignoring Vary with compressed or negotiated responses.** Wrong content served from cache.

**Sending JSON without Content-Type.** Silent binding failures or security bypass attempts.

## Production & Real-World Notes

API gateways strip or inject headers (correlation IDs as `X-Request-Id`). CDNs honor `Cache-Control` and `ETag`; purge on deploy. Teams document **required request headers** in OpenAPI. ASP.NET Core `AddResponseCaching` middleware respects conditions ‚Äî use carefully with auth.

Security headers (`Strict-Transport-Security`, `Content-Security-Policy`) are separate but equally critical ‚Äî covered in security phases.

## Comparison / Trade-offs

| Mechanism | Strength | Weakness |
|-----------|----------|----------|
| Authorization Bearer | Stateless, mobile-friendly | Token theft impact; short TTL + refresh |
| Cookie session | HttpOnly reduces XSS theft | CSRF requires SameSite/token patterns |
| ETag revalidation | Bandwidth savings | Storage/compute to generate ETags |
| aggressive max-age | Speed, lower origin load | Stale content risk |
| no-store | Privacy | No cache benefit |

## Quick Reference

| Header | Direction | Role |
|--------|-----------|------|
| Content-Type | Request/Response | Body media type |
| Accept | Request | Preferred response type |
| Authorization | Request | Credentials (Bearer/Basic) |
| Cookie | Request | Stored cookies sent back |
| Set-Cookie | Response | Instruct browser to store cookie (`0.3.9`) |
| Cache-Control | Both | Cache behavior directives |
| ETag | Response | Version validator |
| If-None-Match | Request | Conditional GET (ETag) |

## Key Takeaways

- Headers carry **metadata** ‚Äî format, auth, caching ‚Äî orthogonal to status codes.
- **Content-Type** describes body; **Accept** negotiates response format.
- **Authorization** = explicit per-request credentials; **Cookie** = browser-managed session state.
- **Cache-Control** + **ETag** drive caching and **304** revalidation.
- Misconfigured caching on private data is a **security** issue, not just performance.
- ASP.NET Core formatters and middleware map directly to these headers.
