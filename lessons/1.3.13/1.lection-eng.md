# Pattern Matching: switch, Property, List, Relational

> Roadmap: `1.3.13` · Node: `1.3` — C# Advanced Topics · Depth: practice

## Learning Objectives

After this lesson you will be able to:

- Replace long `if/else` chains with **switch expressions** and **switch statements** that use pattern matching.
- Match on **types**, **constants**, **properties**, and **relational conditions** in a single `case`.
- Use **list patterns** (`[a, b, ..]`) to destructure sequences at match time.
- Apply **`when` guards** and **`or`/`and`/`not` combinators** without losing readability.
- Choose between **exhaustiveness**, **discard `_`**, and **default arms** deliberately.
- Refactor legacy control flow into maintainable pattern-based code in real services.

---

## Why This Matters

Before C# 7, `switch` only worked on discrete constant values — mostly integers and enums. Business logic therefore accumulated as nested `if (x is Foo f && f.Status == Active)` or long `else if` ladders that duplicated type checks and property reads. The code was correct but fragile: add a new derived type and you might forget a branch; rename a property and three `if` blocks break silently.

Modern **pattern matching** treats `switch` as a **structural classifier**: you describe the *shape* of data (type, properties, collection length, numeric range) and bind variables in the same expression. The compiler can warn when you miss a case (`switch` exhaustiveness on enums with `-W` analyzers). LINQ and ASP.NET codebases use this daily — HTTP result mapping, domain event routing, parsing API payloads, validating DML commands.

Pattern matching is not syntactic sugar for its own sake. It encodes **decision tables** readably and keeps **binding** (`var x`) colocated with **conditions**, which reduces bugs and review fatigue.

## Core Concepts

### From Classic switch to Patterns

Classic switch compares a value to constants:

```csharp
switch (status)
{
    case OrderStatus.Pending: ...
    case OrderStatus.Shipped: ...
    default: ...
}
```

Pattern matching generalizes the `case` label to a **pattern** — a template that may match and **introduce variables**:

```csharp
var message = status switch
{
    OrderStatus.Pending => "Waiting",
    OrderStatus.Shipped => "On the way",
    _ => "Unknown"
};
```

The **switch expression** (`value switch { ... }`) returns a value; the **switch statement** (`switch (value) { case ...: break; }`) executes statements. Both share pattern syntax.

### Type Patterns and Declaration Patterns

A **type pattern** checks runtime type and binds:

```csharp
object input = GetPayload();
if (input is string s)
    Console.WriteLine(s.Length);
```

In switch:

```csharp
var result = input switch
{
    string s => $"text:{s}",
    int n => $"number:{n}",
    _ => "other"
};
```

**Declaration patterns** combine type + local name: `CaseType variable`. The variable is in scope only in that arm. This replaces `(input as string) != null` casts.

### Property Patterns

You can match on **nested shape** without manual null checks:

```csharp
var discount = order switch
{
    { Total: > 1000, Customer.IsVip: true } => 0.15m,
    { Total: > 1000 } => 0.10m,
    { Customer.IsVip: true } => 0.05m,
    _ => 0m
};
```

Each `{ Property: pattern }` is a **property pattern**. `Total: > 1000` is a **relational pattern** on the property value. The compiler lowers this to null-safe property access (short-circuiting if an intermediate is null).

Property patterns shine for **DTOs**, **records**, and **anonymous types** in tests — you assert structure, not sequential getters.

### Relational Patterns

Relational patterns compare to constants using `<`, `>`, `<=`, `>=`:

```csharp
var band = score switch
{
    >= 90 => "A",
    >= 80 => "B",
    >= 70 => "C",
    _ => "F"
};
```

Order matters: first matching arm wins (like traditional switch). This replaces chained `if (score >= 90) ... else if ...`.

Combine with type patterns:

```csharp
var label = value switch
{
    int n and >= 0 and < 100 => "valid percent",
    int n and < 0 => "negative",
    _ => "not a percent"
};
```

### Logical Patterns: and, or, not

C# 9+ adds **combinators**:

- `p1 and p2` — both must match (same input).
- `p1 or p2` — either matches.
- `not p` — negation.

Example:

```csharp
var kind = obj switch
{
    string s and ("" or " " or null) => "blank", // null handled via string? elsewhere
    string s => "text",
    int i and > 0 => "positive",
    not null => "something",
    _ => "null"
};
```

Use combinators sparingly — deeply nested `and`/`or` hurts readability. Prefer property patterns or extract a local function.

### List Patterns (C# 11)

**List patterns** match `IList`, arrays, `Span`-like countable sequences via pattern on elements:

```csharp
var summary = numbers switch
{
    [] => "empty",
    [var only] => $"single:{only}",
    [var first, var second] => $"pair:{first},{second}",
    [var head, .. var tail] => $"head:{head}, rest:{tail.Count}",
};
```

- `[]` — empty collection.
- `[a, b, ..]` — at least two elements; `..` is a **slice** / rest pattern.
- `[.., var last]` — capture last element.

List patterns work with **any type** that supports pattern via `Length`/`Count` and indexer (including arrays and `List<T>`). They are ideal for **routing on tuple-like data**, **command argument arity**, or **parse results** without indexing manually.

### when Guards

Add extra conditions with **`when`**:

```csharp
switch (order)
{
    case { Total: var t } when t > 10_000:
        RequireApproval();
        break;
    case { Total: var t }:
        AutoApprove();
        break;
}
```

Guards do not introduce bindings; they filter an already matched pattern. Overusing `when` often signals the pattern should be more precise.

### Discard and Exhaustiveness

`_` is the **discard pattern** — matches anything not caught earlier. For enums, enable analyzer **CS8509** (non-exhaustive switch) in SDK-style projects to catch missing cases at compile time.

Records with positional syntax enable **positional patterns**:

```csharp
public record Point(int X, int Y);

var quadrant = point switch
{
    ( > 0, > 0) => "Q1",
    ( < 0, > 0) => "Q2",
    ( < 0, < 0) => "Q3",
    ( > 0, < 0) => "Q4",
    (0, 0) => "origin",
    _ => "on axis"
};
```

Positional patterns are property patterns in disguise for records.

## Under the Hood

Pattern matching is compiled to **efficient decision trees**, not a chain of naive `is` checks. The Roslyn compiler builds a **pattern graph**, orders tests (often cheapest first: type checks before property fetches), and may reuse **temps** for repeated property access.

At IL level, a switch expression typically becomes **`switch` opcode** (if dense integer/enum) or a sequence of **conditional branches** with **`isinst`/`castclass`** for type patterns. Property patterns call getters; the compiler inserts **null checks** so you do not get `NullReferenceException` mid-pattern.

List patterns lower to **length checks** + **indexer access** + recursive matching on tail via slice. No LINQ is involved at runtime — zero allocation for typical array/list matching.

Switch expressions must be **exhaustive** or have a discard; the compiler enforces this for bool and some enum switches depending on settings.

## Syntax / Commands / API

| Construct | Example |
|-----------|---------|
| Switch expression | `x switch { 1 => "a", _ => "b" }` |
| Type pattern | `case string s:` |
| Property pattern | `case { Id: > 0, Name: not null }:` |
| Relational | `case >= 18 and <= 65:` |
| List pattern | `case [var h, ..]:` |
| Combinator | `string s and not ""` |
| Guard | `case int n when n % 2 == 0:` |

## Examples

### HTTP result mapping (ASP.NET style)

```csharp
IResult MapError(Exception ex) => ex switch
{
    ValidationException ve => Results.ValidationProblem(ve.Errors),
    NotFoundException nf => Results.NotFound(nf.Message),
    UnauthorizedAccessException => Results.Unauthorized(),
    _ => Results.Problem(ex.Message)
};
```

Each arm binds only what it needs; adding a new exception type is one line.

### Parsing heterogeneous JSON nodes

```csharp
string Describe(object? node) => node switch
{
    null => "null",
    bool b => b ? "true" : "false",
    string s => $"\"{s}\"",
    int or long or double => "number",
    IList list => $"array[{list.Count}]",
    _ => "object"
};
```

`int or long or double` uses **`or`** on type patterns.

### Command arity router

```csharp
bool TryParseCommand(string[] args, out Command cmd) =>
    args switch
    {
        ["create", var name] => Success(new CreateCommand(name), out cmd),
        ["delete", var id] when int.TryParse(id, out _) => Success(new DeleteCommand(id), out cmd),
        ["list"] => Success(ListCommand.Instance, out cmd),
        _ => Fail(out cmd)
    };
```

List patterns replace manual `args.Length` switches.

## Common Mistakes & Anti-patterns

**Wrong arm order with relational patterns.** `>= 0` before `>= 90` swallows all scores. Put **most specific** (highest threshold) first.

**Replacing every if with switch.** Simple two-branch logic stays clearer as `if`. Pattern matching wins at **3+ branches** or **structural** conditions.

**Side effects inside switch expressions.** Switch expressions should be **pure**; use switch **statements** for multi-step imperatives.

**Non-exhaustive enum switch** without `_` — runtime surprise when enum gets a new value from API deserialization.

**Overloaded list patterns** without `..` — `[a,b,c]` only matches exactly three elements; use `[a,b,..]` for "at least two."

## Production & Real-World Notes

Teams often adopt **switch expressions** in mapping layers (domain → DTO, exception → ProblemDetails). Analyzers + nullable context make property patterns safe on partial API models.

For **high-frequency** paths (e.g., parsing millions of messages/sec), prefer **span-based** parsers or dedicated state machines — pattern matching clarity beats micro-optimization unless profiling says otherwise.

Code review checklist: arm order documented? New enum values handled? Guards not duplicating pattern conditions?

## Comparison / Trade-offs

| Approach | Pros | Cons |
|----------|------|------|
| if/else chain | Familiar, arbitrary logic | Verbose, duplicated casts |
| Classic switch | Fast for ints/enums | No structure |
| Pattern switch | Readable structure, binding | Learning curve, arm order pitfalls |
| Polymorphism | Open extension (OOP) | Requires type hierarchy |

Use **virtual dispatch** when behavior belongs to types; use **pattern matching** when behavior is external (functional style, cross-cutting mappers).

## Quick Reference

```
value switch { pattern => result, _ => default }

Patterns: constant | type var | property { A: pat } | relational >= n
          list [a, b, ..] | and | or | not | discard _

Statement: switch (e) { case pat when cond: ... break; }
```

## Key Takeaways

- Pattern matching classifies **data by shape**, not just by scalar value.
- **Switch expressions** return values; keep them side-effect free.
- **Property** and **relational** patterns collapse null-check + compare boilerplate.
- **List patterns** express arity and head/tail decomposition cleanly.
- **Arm order** matters for relational and general first-match semantics.
- Compiler builds decision trees; list patterns avoid LINQ allocations.
- Prefer patterns for mappers; prefer polymorphism for intrinsic type behavior.
