## Why This Matters

На собеседованиях async-in-ctor проверяют понимание **инвариантов конструирования**, **threading** и **композиции Task** — не только синтаксис. Junior предлагает `public async MyService()` или `public async string Data => await Load();`. Оба illegal или опасны. Senior объясняет: CLR должен **синхронно** вернуть ссылку из `new`, и предлагает **`CreateAsync`**, **factory** или **`InitializeAsync`** из composition root.

В production нарушения дают **случайные null**, **deadlocks** (`.Wait()` в ctor), **unobserved exceptions** от `_ = LoadAsync()` в ctor. ASP.NET Core DI резолвит **синхронно** — нельзя `await` в стандартном constructor injection. Middle проектирует сервисы **sync-constructible** и **async-initializable** на правильной границе (`IHostedService`, endpoint).

## Core Concepts

### Почему конструктор не может быть async

Задача ctor: выделить объект, вызвать base chain, инициализировать поля, вернуть **валидную ссылку**. Вызывающий ожидает:

```csharp
var svc = new MyService(); // завершилось до следующей строки
```

`async` методы возвращают **`Task`**, не `MyService`. Синтаксиса `async constructor` нет. Если:

```csharp
public MyService() => InitializeAsync(); // fire-and-forget
```

получаешь **запущенный, но не awaited** `Task` — ctor возвращается до конца init. **Инвариант нарушен:** объект «готов», `_cache` пуст.

**Правило компилятора:** `async` только у методов с подходящим return type. Конструктор возвращает instance через специальный CLR path — несовместим с async state machine.

### Ловушка `.Wait()` / `.Result` в ctor

```csharp
public MyService(IApiClient api)
{
    _data = api.FetchConfigAsync().Result;
}
```

Проблемы: **deadlock** с `SynchronizationContext`, **thread pool starvation** при массовом resolve, **sync-over-async**.

Ответ на interview: «Иногда работает, под нагрузкой падает — wrong default».

### Async factory

```csharp
public static async Task<ConfigService> CreateAsync(IApiClient api, CancellationToken ct)
{
    var config = await api.FetchConfigAsync(ct);
    return new ConfigService(config);
}
```

Caller: `var svc = await ConfigService.CreateAsync(api, ct);` — **полная инициализация** до use. DI:

```csharp
builder.Services.AddSingleton(await ConfigService.CreateAsync(api, ct));
```

Только на **composition root** при build host — не per-request.

### IAsyncInitializer / InitializeAsync

Когда нужен parameterless ctor:

```csharp
public interface IAsyncInitializable
{
    Task InitializeAsync(CancellationToken ct);
}
```

Host после resolve:

```csharp
foreach (var init in app.Services.GetServices<IAsyncInitializable>())
    await init.InitializeAsync(ct);
```

**Риск:** consumer использует сервис до init без enforcement.

### Async property getters

C# **не** разрешает `async` accessors:

```csharp
// ILLEGAL
public async string Name => await _repo.GetNameAsync();
```

Properties — **`get`/`set`**, должны возвращаться **синхронно**. Interview trap: «сделай async property» — правильно: **`GetNameAsync()`** или sync property на **preloaded** data.

**`.Result` в getter** — хуже:

```csharp
public string Name => _name ??= _repo.GetNameAsync().Result;
```

Блокировка, deadlock, N blocking calls в списках.

### Half-initialized и races

```csharp
public MyService() { _ = WarmCacheAsync(); }
```

Два request до warm — **race**. Тесты single-threaded проходят, production flakes.

**readonly** поля нельзя присвоить в continuation после ctor без unsafe patterns — init до «публикации» объекта.

## Under the Hood

**Object allocation** (`newobj` IL) выполняет ctor **синхронно** на calling thread. GC видит ссылку после return ctor. Async state machine должна была бы **вернуться до завершения ctor** для await — нарушает контракт CLR.

**DI** (`Microsoft.Extensions.DependencyInjection`) — **синхронный** `Activator.CreateInstance`. Нет стандартного `await ResolveAsync<T>()`. Async init **вне** ctor или через factory delegate.

**Property getters** → `get_Name()` — синхронный return `T`. `async Task<T>` меняет контракт; дизайнеры отвергли async properties — `obj.Name` выглядит sync, но скрывает suspension.

## Examples

### DI-friendly

```csharp
public sealed class ReportService(IConfigCache cache)
{
    public async Task<Report> BuildAsync(CancellationToken ct) =>
        await GenerateAsync(cache.Current, ct);
}
```

`IHostedService` грузит cache до трафика.

### Interview one-liner

> «В ctor нельзя await — конструирование должно синхронно завершиться валидной ссылкой. Используй `CreateAsync`, hosted startup init или явные async methods — не async properties и не `.Wait()` в ctor.»

## Common Mistakes & Anti-patterns

- `InitializeAsync().Wait()` в ctor
- Async lazy property
- Ожидание, что DI сам вызовет `InitializeAsync`
- Двухфазный init без enforcement
- Тесты ctor с `.Result` — pass, prod deadlock

## Production & Real-World Notes

**IHostedService** — место async warmup в ASP.NET. **Readiness probe** — not ready до init.

Blazor/MAUI — `SynchronizationContext` усиливает опасность `.Wait()` в ctor.

## Key Takeaways

- Ctor завершается **синхронно** — нет `async ctor`.
- **Async property** illegal — используй **async methods**.
- **CreateAsync** и **hosted warmup** — production patterns.
- Fire-and-forget в ctor → **races**, **unobserved exceptions**.
- **`.Wait()` в ctor** — red flag на interview и в prod.
- Default DI — **sync** resolve.
