# LINQ Aggregate and Scan — Fold Operations

> Roadmap: `1.3.23` · Node: `1.3` — C#: Advanced Topics · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Explain **`Aggregate`** as a **left fold** over a sequence with optional seed and result selector.
- Contrast **`Sum`/`Max`/`Count`** built-ins with custom **`Aggregate`** when no dedicated operator exists.
- Implement **`Scan`** (prefix / running accumulation) — not in **`System.Linq`**, but common in libraries and interviews.
- Choose **seed vs no-seed** overloads and predict behavior on **empty sequences**.
- Understand **deferred vs immediate** — **`Aggregate`** always consumes the full sequence (immediate terminal operator).
- Apply folds in production: pipelines, immutable aggregation, and functional composition without mutable outer variables.

---

## Why This Matters

Many reductions do not have a named LINQ operator. Building a comma-separated string from names, merging a sequence of **`AppConfig`** patches into one object, computing a custom financial rollup, or finding the longest string in a stream — all are **folds**: combine elements pairwise using an accumulator function. **`Aggregate`** is the general-purpose fold in **`System.Linq`**.

**`Scan`** (also called **prefix sum**, **running total**, **accumulate**) emits **every intermediate accumulator** — `[1,2,3]` → running sums `[1,3,6]`. Standard LINQ lacks **`Scan`**, but you see it in **MoreLINQ**, **Rx**, and functional stacks; implementing it with **`yield`** deepens iterator + fold understanding. Middle developers use **`Aggregate`** in report code and misuse it when **`Sum`** suffices, or forget the **empty sequence** behavior of the no-seed overload (throws vs seed overload returns seed).

## Core Concepts

### Aggregate Without Seed

```csharp
var longest = names.Aggregate((acc, next) =>
    next.Length > acc.Length ? next : acc);
```

Overload **`Aggregate(Func<T,T,T>)`** uses **first element as initial accumulator**, then combines with second, third, … — **n−1** invocations for **n** elements.

**Empty sequence:** throws **`InvalidOperationException`** — "Sequence contains no elements" — same family as **`Max()`** on empty.

**Single element:** returns that element; accumulator function **never runs**.

This overload is only valid when input type equals accumulator type (**`T`** in, **`T`** out).

### Aggregate With Seed

```csharp
var total = numbers.Aggregate(0, (acc, n) => acc + n);
```

**Seed** is starting accumulator. **Empty sequence:** returns **seed** without calling accumulator — important for "zero items = 0 total" semantics.

```csharp
Enumerable.Empty<int>().Aggregate(0, (a, n) => a + n); // 0
```

Seed type **`TAccumulate`** may differ from element type **`TSource`** — enables building **`StringBuilder`**, DTO, or dictionary in the fold.

### Result Selector Overload

```csharp
var report = orders.Aggregate(
    seed: new StringBuilder(),
    func: (sb, o) => sb.AppendLine(o.Id.ToString()),
    resultSelector: sb => sb.ToString());
```

Third overload: seed + fold + **final projection** from accumulator to result type — avoids extra local after fold.

### Aggregate vs Built-in Operators

| Need | Prefer |
|---|---|
| Sum of ints | **`Sum()`** — clearer, optimized |
| Product of ints | **`Aggregate(1, (a,n) => a * n)`** |
| Max | **`Max()`** / **`MaxBy()`** |
| Join strings | **`string.Join`** or **`Aggregate`** with **`StringBuilder`** for large data |
| Custom rollup type | **`Aggregate`** with seed |

Built-ins may use SIMD or provider SQL translation (**`IQueryable.Sum`**); **`Aggregate`** with arbitrary lambda often **client-only** for EF.

### Scan — Running Fold (Extension Pattern)

**`Scan`** yields accumulator after **each** element:

```csharp
public static IEnumerable<TAcc> Scan<TSource, TAcc>(
    this IEnumerable<TSource> source,
    TAcc seed,
    Func<TAcc, TSource, TAcc> func)
{
    var acc = seed;
    yield return acc; // optional: some variants skip initial seed
    foreach (var item in source)
    {
        acc = func(acc, item);
        yield return acc;
    }
}
```

Input **`[1,2,3]`**, seed **`0`**, add → **`[0,1,3,6]`** if initial seed emitted, or **`[1,3,6]`** if scan starts after first combine — **library convention varies**; document your choice.

**MoreLINQ** **`Scan`** includes overloads matching functional expectations. **`System.Linq`** — roll your own or reference MoreLINQ in app code (know dependency trade-off).

**Use cases:** running balance charts, incremental averages, pipeline visibility (log state after each step), functional reactive programming.

### Scan vs Aggregate

| | **Aggregate** | **Scan** |
|---|---|---|
| Output | Single final value | Sequence of intermediates |
| Execution | Immediate, full consume | Deferred (iterator) |
| In System.Linq | Yes | No (custom/MoreLINQ) |
| Empty input | No-seed throws; seed returns seed | Typically yields seed only |

## Under the Hood

**`Enumerable.Aggregate`** is a **foreach** loop — no magic, no parallelism by default:

```csharp
TAccumulate acc = seed;
foreach (var item in source)
    acc = func(acc, item);
return acc;
```

No-seed version: **`using IEnumerator`**, read first, loop rest.

**Short-circuit:** none — always full sequence unless source itself stops early.

**`ParallelEnumerable.Aggregate`** exists for parallel folds with **associative** operations — different API, partition + combine phases; non-associative folds produce wrong results.

**`IQueryable`**: **`Aggregate`** rarely translates to SQL as generic fold — EF typically **client evaluates** or throws. Use SQL **`SUM`**, window functions **`SUM() OVER`**, or raw SQL for running totals — not C# **`Scan`** on **`IQueryable`**.

**Allocations:** string concatenation in **`Aggregate((a,s) => a + s)`** — O(n²) immutable string churn; use **`StringBuilder`** seed or **`string.Join`**.

## Syntax / API

```csharp
source.Aggregate(func);
source.Aggregate(seed, func);
source.Aggregate(seed, func, resultSelector);

// Scan (custom)
source.Scan(seed, (acc, x) => ...);
```

## Examples

### Product

```csharp
var product = numbers.Aggregate(1, (acc, n) => acc * n);
// Empty → 1 (multiplicative identity)
```

### Build lookup in one pass

```csharp
var byId = records.Aggregate(
    new Dictionary<int, Record>(),
    (dict, r) => { dict[r.Id] = r; return dict; });
```

Prefer **`ToDictionary`** when key/value selectors are simple — clearer and optimized.

### Running totals

```csharp
var running = balances.Scan(0m, (acc, tx) => acc + tx.Amount);
foreach (var rt in running)
    Console.WriteLine(rt);
```

### Pipeline of transformations (fold into state)

```csharp
var finalState = events.Aggregate(
    InitialState,
    (state, e) => state.Apply(e));
```

Event sourcing / reducer pattern — same fold idea as Redux **`reduce`**.

## Common Mistakes & Anti-patterns

**No-seed on possibly empty sequence** — throws; use seed overload.

**`Aggregate` for Sum** — noise; use **`Sum`**.

**String `acc + item` in loop** — quadratic allocations.

**Non-associative fold in PLINQ `Aggregate`** — incorrect parallel results.

**Expecting EF to translate arbitrary Aggregate** — client eval surprise.

**Scan with wrong empty semantics** — document whether first yielded value is seed-only.

**Mutating shared object outside fold** inside accumulator — breaks purity and parallel safety; return new accumulator or mutate local copy deliberately.

## Production & Real-World Notes

Dashboard **running metrics**: compute in SQL window functions for large history; **`Scan`** in memory for small collections or tests.

Configuration merge: **`Aggregate`** over ordered patch list — explicit ordering matters; fold is not commutative unless you design it to be.

Logging fold steps: **`Scan`** + **`Zip`** for audit trail.

Prefer **`Sum`**, **`Average`**, **`Count`**, **`Min`**, **`Max`**, **`ToDictionary`**, **`ToLookup`** when they express intent — reviewers flag "clever **`Aggregate`**."

## Comparison / Trade-offs

| Operation | Tool |
|---|---|
| Final sum | **`Sum()`** |
| Final only custom | **`Aggregate` + seed** |
| All partials | **`Scan`** |
| SQL running total | Window function / **`SUM OVER`**, not LINQ Scan on IQueryable |
| Immutable string build | **`StringBuilder`** seed |

Functional **fold left** = **`Aggregate`**; **scan** = map fold history to sequence.

## Quick Reference

```csharp
// throws if empty
xs.Aggregate((a,b) => ...);

// empty → seed
xs.Aggregate(seed, (a,x) => ...);

// + project
xs.Aggregate(seed, (a,x) => ..., a => result);

// Scan — custom / MoreLINQ
xs.Scan(seed, (a,x) => ...);
```

## Key Takeaways

- **`Aggregate`** = terminal fold; always enumerates entire sequence.
- **No-seed** uses first element; **empty throws**.
- **Seed overload** returns seed on empty — pick intentionally.
- **`Scan`** = deferred running accumulators; not in **`System.Linq`**.
- Use dedicated operators when they exist; avoid string **`+`** in folds.
- EF rarely translates generic **`Aggregate`**/**`Scan`** — push to SQL when data is large.
