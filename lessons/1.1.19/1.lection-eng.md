# const vs readonly vs static readonly

> Roadmap: `1.1.19` · Node: `1.1` — C# Language Basics · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- State precisely when a field is a **compile-time constant** vs a **runtime-initialized immutable** field.
- Choose **`const`**, instance **`readonly`**, or **`static readonly`** for configuration, defaults, and domain constants.
- Explain how **`const` inlining** affects versioning and cross-assembly updates without recompilation.
- Read IL/metadata to recognize **literal constants** vs **static init** vs **instance init**.
- Avoid confusing **`readonly` fields** with **`readonly struct`**, **`init` accessors**, and **`readonly` methods**.

---

## Why This Matters

Every codebase declares values that should not change: HTTP header names, default page sizes, mathematical constants, feature flag keys. C# offers three similar-looking tools — **`const`**, **`readonly`**, and **`static readonly`** — and juniors treat them as interchangeable. They are not. Pick wrong and you ship subtle production bugs: a **`const`** changed in a library does not update consuming assemblies until they **recompile**; a **`readonly`** instance field duplicated per object when one shared **`static readonly`** would suffice; a **`const`** attempted with **`DateTime.UtcNow`** fails at compile time while **`static readonly`** expresses the intent correctly.

Middle developers maintain shared class libraries, configure ASP.NET options, and review PRs where someone marks a computed value **`const`** via a hack. Understanding **when the compiler embeds values into IL** versus **when the CLR loads them from field metadata** is the difference between predictable deployments and "we changed the constant but production still shows the old number." This lesson goes under the hood — IL, metadata, initialization order — so you choose consciously, not by habit.

## Core Concepts

### const: Compile-Time Constants

A **`const`** field must be initialized with a **constant expression** evaluable at compile time. Permitted types are **`byte`**, **`sbyte`**, **`short`**, **`ushort`**, **`int`**, **`uint`**, **`long`**, **`ulong`**, **`char`**, **`float`**, **`double`**, **`decimal`**, **`bool`**, **`string`**, and **`null`** reference for reference types. No **`DateTime`**, no **`new object()`**, no method calls except **`string`** concatenation of other constants.

```csharp
public static class ApiConstants
{
    public const int DefaultPageSize = 20;
    public const string VersionHeader = "X-Api-Version";
}
```

**`const`** members are **implicitly static**. You access them as **`ApiConstants.DefaultPageSize`** without an instance. They live in metadata as **literal constants** associated with the field token. Importantly, when another assembly **uses** your **`const`**, the C# compiler **inlines the literal value** into the consuming assembly's IL — the consumer does not always read your field at runtime.

That inlining is the central trade-off: maximum performance (often **`ldc.i4`** loads a number directly) and **`switch`** case label eligibility, but **binary coupling**. If Library A defines **`public const int Max = 100`** and App B references it, B's IL may contain **`100`** literally. You ship Library A **`Max = 200`** without recompiling App B — B may still use **`100`** until rebuilt. **`static readonly`** avoids that embedding behavior for the consumer.

**`const`** cannot be assigned in constructors — there is nothing runtime to assign; value is fixed at compile time in each assembly that inlines it.

### readonly: Instance-Bound Immutability After Construction

An instance **`readonly`** field may be assigned **only** in its declaration or in **constructors** of the same struct/class (including instance field initializers that run before constructor body). After construction completes, assignment is forbidden.

```csharp
public sealed class Order
{
    private readonly Guid _id;
    private readonly DateTime _createdAt;

    public Order()
    {
        _id = Guid.NewGuid();
        _createdAt = DateTime.UtcNow;
    }
}
```

Each **`Order`** carries its own **`_id`** and **`_createdAt`**. Values are determined at **runtime** when **`new Order()`** executes — **`DateTime.UtcNow`** is legal here, illegal for **`const`**.

**`readonly`** on a **struct** field does not make the whole struct immutable if other fields remain mutable; it only protects that field from reassignment after init. **`readonly struct`** (C# 7.2+) is a different feature — deep immutability semantics for the struct type itself.

For classes, **`readonly`** fields are **per-instance** storage slots in the object layout on the heap (alongside other fields). JIT may still optimize reads but the value is loaded from the object, not inlined from metadata at compile time of external callers.

### static readonly: Type-Bound Runtime Constants

**`static readonly`** combines **one storage location per type** (AppDomain lifetime, before type unload edge cases) with **runtime initialization**. Assignment occurs in **static field initializer** or **static constructor** — `.cctor`.

```csharp
public static class AppMetadata
{
    public static readonly string EnvironmentName =
        Environment.GetEnvironmentVariable("DOTNET_ENVIRONMENT") ?? "Production";

    public static readonly DateTime ProcessStartedUtc = DateTime.UtcNow;
}
```

All callers read the **same field** from the type's static memory. Changing the field's value at runtime is still forbidden after initialization (without reflection). Consumers reference the **field token**; the compiler does **not** inline **`static readonly`** values from referenced assemblies into consumer IL the way it does **`const`**.

Use **`static readonly`** when:

- Value requires **runtime computation** (environment, file read, **`Guid.NewGuid()`** once per process).
- Value must update in consuming apps when you update **only** the defining assembly (no consumer recompile required for the numeric/string value to refresh on next deploy of that DLL).
- Value is **reference type** created once and shared — e.g. **`static readonly Regex Pattern = new(...)`** (often paired with **`RegexOptions.Compiled`** trade-offs).

Use **`const`** when:

- True symbolic compile-time literal never changes across deployments (**`Math`-like**).
- You need **`const`** in attributes (**`[Obsolete("Msg", error: true)]`** parameters must be constant).
- **`switch`** cases or array size patterns requiring compile-time constants.

Use instance **`readonly`** when:

- Value belongs to **each object** and is set once at creation (identity, injected dependency reference frozen at construct, immutable aggregate root id).

### static readonly vs const Side by Side

```csharp
public class Settings
{
    public const int ConstTimeout = 30;
    public static readonly int ReadonlyTimeout = 30;
}
```

Source looks identical; IL differs. **`ConstTimeout`** consumers may embed **`30`**. **`ReadonlyTimeout`** consumers emit **`ldsfld`** load static field from **`Settings`**.

| Feature | `const` | `readonly` (instance) | `static readonly` |
|---------|---------|----------------------|-------------------|
| Storage | Metadata literal | Per instance | Per type (static) |
| Init time | Compile time | Constructor / field init | Static ctor / static init |
| `DateTime.Now` | Illegal | Legal in ctor | Legal in static init |
| Cross-assembly update without consumer rebuild | **No** (inlined) | Field load | Field load |
| Implicitly static | Yes | No | Yes (static modifier) |
| Can be `struct` field | N/A (static) | Yes | Yes (static) |

### Initialization Order and Static Constructors

**`static readonly`** fields initialize before any static member access triggers **`.cctor`**. Order among static fields follows declaration order in the type unless circular dependencies force careful ordering. Instance **`readonly`** fields initialize before constructor body: field initializers run, then constructor assignments.

A classic pitfall: **`static readonly`** depending on another type's static init — **type initializer cycles** can throw **`TypeInitializationException`**. Keep static init simple; defer heavy work to lazy patterns if needed.

Instance **`readonly`** injected via constructor is the standard DI pattern for **immutable dependencies**:

```csharp
public sealed class OrderService
{
    private readonly IOrderRepository _repo;
    public OrderService(IOrderRepository repo) => _repo = repo;
}
```

The reference cannot be swapped after construct — thread-safe publication if object fully constructed before exposure.

## Under the Hood

### Metadata and IL for const

In IL, **`const`** values appear in the **`Constant`** table linked to field metadata. Consumer method bodies often contain immediate loads:

```csharp
// C# consumer
int x = ApiConstants.DefaultPageSize;

// Conceptual IL
ldc.i4 20
stloc.0
```

No **`ldsfld`** — the number **`20`** is burned into the consumer assembly. Reflector/dotPeek on **consumer** shows literal, not a field read from constants library.

Attributes require compile-time constants because attribute arguments are encoded in metadata at compile time — same mechanism.

### IL for static readonly

Consumer IL:

```
ldsfld int32 Settings::ReadonlyTimeout
stloc.0
```

At runtime CLR loads **`Settings`**, runs **`.cctor`** once (before first static field access), assigns field, subsequent loads read memory. Updating **`Settings.dll`** with a new default changes behavior when deployed — even if consumer EXE not recompiled — assuming consumer loads new DLL.

### IL for instance readonly

Object allocation **`newobj`** runs constructor chain; **`stfld`** stores **`readonly`** values into object fields. JIT enforces assignment only from constructors via C# compiler rules; runtime **does not** magically prevent **`stfld`** from illegal paths if you bypass C# with reflection — **`readonly`** is compile-time enforcement with limited runtime reflection escape hatches.

### JIT Inlining and Optimization

**`const`** locals fold at JIT compile time in the **same** assembly. Cross-assembly **`const`** already inlined at compile time of consumer. **`static readonly`** rarely inlined across assemblies because value not guaranteed constant at JIT time (could theoretically change with reflection before init in pathological cases — JIT treats as normal static field).

Performance difference is usually negligible except micro-benchmarks; the **deployment/versioning** story drives choice.

### readonly vs init-only properties

C# 9 **`init`** accessors allow object initializer syntax immutability without **`readonly` field** — property backed by compiler-generated field. **`record`** types use **`init`** for with-expressions. **`readonly field`** is lower-level; **`init` property** participates in **`with`** copying. Different tools for immutability stories — not identical IL.

## Syntax / Commands / API

```csharp
public const double Pi = 3.14159;
public readonly int _instanceId;
public static readonly Regex Email = new(@"^...", RegexOptions.Compiled);

// illegal:
// public const DateTime Epoch = DateTime.UnixEpoch;
// legal:
public static readonly DateTime Epoch = DateTime.UnixEpoch;
```

**Expression-bodied static readonly:**

```csharp
public static readonly string BaseUrl = BuildBaseUrl();
private static string BuildBaseUrl() => /* ... */;
```

**Primary constructor (C# 12) capturing parameters into readonly fields** — syntactic sugar generating **`readonly`** backing fields for **`class`** primary ctor parameters.

## Examples

### Example 1: Library constant versioning trap

Library v1:

```csharp
public const int MaxItems = 10;
```

App compiled against v1 inlines **`10`**. Library v2 changes to **`20`**, drop-in DLL replace **without** recompiling App — App still enforces **`10`** in methods that imported constant into IL. Fix: change to **`public static readonly int MaxItems = 20;`** or expose via configuration API.

### Example 2: Per-instance identity

```csharp
public sealed class DomainEvent
{
    public Guid EventId { get; }
    public DateTime OccurredUtc { get; }

    public DomainEvent()
    {
        EventId = Guid.NewGuid();
        OccurredUtc = DateTime.UtcNow;
    }
}
```

**`init`** or get-only auto-properties set in ctor — runtime values, not **`const`**.

### Example 3: Shared compiled regex

```csharp
public static partial class ValidationPatterns
{
    public static readonly Regex Email =
        new(@"^[^@\s]+@[^@\s]+\.[^@\s]+$", RegexOptions.Compiled | RegexOptions.CultureInvariant);
}
```

**`static readonly`** ensures single instance; **`const`** impossible for **`Regex`**.

## Common Mistakes & Anti-patterns

**`const` for values that might change between deployments** — Consumers stale until rebuild. Fix: **`static readonly`** or configuration.

**Using **`const`** with computed **`string`** from method** — Does not compile. Fix: **`static readonly`**.

**Instance **`readonly`** for process-wide singleton value** — Wastes per-object copies if duplicated. Fix: **`static readonly`**.

**Expecting **`readonly`** to deep-freeze object graph** — Reference **`readonly`**; referenced object may still mutate. Fix: immutable types or defensive copies.

**Confusing **`readonly struct`** with **`readonly` field**** — Struct copy semantics vs field assignment rules.

**Static ctor throwing** — Type permanently **`TypeInitializationException`**. Fix: minimal static init.

## Production & Real-World Notes

Shared **NuGet** libraries exposing **`public const`** for limits and timeouts create implicit contract: any change is **breaking for behavior** even if signature unchanged — document semver accordingly; prefer **`static readonly`** + docs or **`IOptions<T>`** for tunable values.

ASP.NET Core **`IOptions<T>`** replaces many static constants — values from **`appsettings`**, reloadable via **`IOptionsMonitor`**, neither **`const`** nor **`static readonly`** fits dynamic config.

Analyzers: **`CA1802`** suggests **`static readonly`** over **`const`** for non-literal scenarios (where applicable).

Security: **`static readonly`** **`byte[]` key** still visible in memory; not a substitute for secret stores — but better than **`const string`** secret inlined in every assembly layer.

Code review checklist: public constants in libraries → ask "should this inline into consumers?"

## Comparison / Trade-offs

**`const` vs `#define`** — C# favors **`const`**; no preprocessor constant pollution.

**`static readonly` vs lazy static property** — **`Lazy<T>`** defers cost; **`static readonly`** eager at first type access.

**Configuration hierarchy** — **`const`** (truly fixed) → **`static readonly`** (process-fixed) → **`IOptions`** (environment-specific) → remote config service.

**Immutable object patterns** — **`readonly` fields + ctor`** vs **`record`** with **`init`** — team style and **`with`** needs decide.

## Quick Reference

```
const            compile-time literal, inlined into consumers, implicitly static
readonly         set in ctor/field init, per instance, runtime value OK
static readonly  one per type, static ctor/init, runtime value OK, ldsfld consumers

Choose const     attributes, true forever literals, switch cases
Choose static ro shared runtime init, avoid cross-assembly inline stale
Choose readonly  per-object identity / frozen deps at construct

const change in library → consumers must recompile to see new value
static readonly change  → redeploy defining DLL often enough
```

## Key Takeaways

1. **`const`** is compile-time; values may be **inlined into referencing assemblies**.
2. **`readonly`** is **per-instance**, assigned in constructors; supports runtime expressions.
3. **`static readonly`** is **per-type**, runtime init, loaded via **static field** — better for updatable shared values.
4. **`const`** cannot use **`DateTime.Now`**, **`new`**, or most method calls.
5. Cross-assembly **`const`** changes without consumer rebuild → **stale values** — production footgun.
6. **`readonly struct`**, **`init`**, and **`readonly` field** solve different immutability problems.
7. Shared libraries: prefer **`static readonly`** or options for tunable "constants."
