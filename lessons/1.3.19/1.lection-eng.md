# Local Functions vs Private Methods

> Roadmap: `1.3.19` · Node: `1.3` — C#: Advanced Topics · Depth: understanding

## Learning Objectives

After this lesson you will be able to:

- Explain what a **local function** is and where it lives in the type system.
- Decide when a **local function** is clearer than a **private method**, and when it is not.
- Use **static local functions** to avoid accidental captures and allocations.
- Recognize how **local functions** interact with **generics**, **async/await**, and **iterator blocks**.
- Refactor duplicated validation logic without polluting the class surface area.

---

## Why This Matters

Every class grows helper methods. Soon you have fifteen `private` methods whose only caller is one public method — `ValidateOrderHeader`, `ParseSegment`, `TryNormalizeEmail` — scattered through the file, forcing readers to jump up and down to understand a single workflow. Local functions (C# 7.0+) let you define a function **inside** another member, keeping the helper physically next to the code that uses it.

That locality matters for readability, but local functions are not "free private methods with extra syntax." They can **capture** outer variables like lambdas, become **`static`** to forbid capture, appear inside **`async`** and **`iterator`** methods, and even be referenced before their textual declaration within the same outer method. Middle developers use them daily in ASP.NET Core action methods, EF query builders, and validation pipelines — and misuse them when a helper is reused across three endpoints and should have been a private method or a dedicated class.

Understanding the trade-off — **scope and capture vs reusability and testability** — keeps your types focused and your methods readable under review.

## Core Concepts

### What Is a Local Function?

A local function is a method declared **inside** the body of another method, constructor, property accessor, or local function itself. It is not a member of the class; it exists only as a nested function within the enclosing member's scope.

```csharp
public decimal CalculateShipping(Order order)
{
    decimal BaseRate() => order.TotalWeight * 0.5m;

    decimal ApplyDiscount(decimal rate)
    {
        if (order.IsPremiumCustomer)
            return rate * 0.9m;
        return rate;
    }

    var rate = BaseRate();
    return ApplyDiscount(rate);
}
```

`BaseRate` and `ApplyDiscount` are invisible outside `CalculateShipping`. You cannot call them from another method on the same class. That restriction is the point: they are implementation details of one workflow, not part of the type's API — even the private API.

Compared to a **private instance method**, a local function has no access to `this` unless the outer method is an instance member and the local function is non-static. Compared to a **lambda** assigned to a `Func<>` delegate, a local function is a real function with a name, supports **`ref`/`out` parameters**, and by default does not require delegate allocation when called directly (the compiler emits a nested function, not necessarily a closure delegate).

### Local Functions vs Private Methods

**Private methods** belong to the type. They appear in IntelliSense for the whole class, can be called from any member, and are natural targets for unit tests if you use `InternalsVisibleTo` or extract to a testable helper class. They are the right choice when the same logic is invoked from **multiple** methods, when the helper is **complex enough to deserve its own tests**, or when you want a **stable name** in stack traces across call sites.

**Local functions** belong to one enclosing member. Use them when logic is **single-use**, **tightly coupled to local variables**, or when splitting a long method improves readability without expanding the class's private method list. A 40-line action method with three nested validation steps often reads better with local functions than with three new private methods at the bottom of a 600-line controller.

The decision is not about performance by default — JIT often inlines small private methods and local functions similarly. It is about **structure and intent**: "this exists only here" vs "this is a reusable part of the class."

### Static Local Functions (C# 8+)

When a local function does not need to read or write captured variables from the outer scope, mark it **`static`**:

```csharp
public IReadOnlyList<string> SplitTags(string input)
{
    static IEnumerable<string> SplitCore(string text, char separator)
    {
        foreach (var part in text.Split(separator, StringSplitOptions.RemoveEmptyEntries))
            yield return part.Trim();
    }

    return SplitCore(input, ',').Distinct(StringComparer.OrdinalIgnoreCase).ToList();
}
```

**`static`** on a local function prevents accidental capture of `input` or `this` inside `SplitCore`, which avoids hidden closure state and makes the data flow explicit through parameters. Code analyzers often suggest **`static`** when possible. If you need outer variables, drop **`static`** and pass only what you need — prefer explicit parameters over silent capture when the helper is more than a few lines.

### Capture Semantics

A non-static local function can use variables from the enclosing scope — parameters, locals, even **`ref` locals** in modern C#. That is convenient for incremental parsing:

```csharp
public bool TryParseRange(string text, out int min, out int max)
{
    min = max = 0;
    var span = text.AsSpan();

    bool TryReadInt(ref ReadOnlySpan<char> s, out int value)
    {
        value = 0;
        var i = 0;
        while (i < s.Length && char.IsDigit(s[i]))
        {
            value = value * 10 + (s[i] - '0');
            i++;
        }
        if (i == 0) return false;
        s = s[i..];
        return true;
    }

    if (!TryReadInt(ref span, out min)) return false;
    if (span.Length == 0 || span[0] != '-') return false;
    span = span[1..];
    return TryReadInt(ref span, out max) && span.IsEmpty;
}
```

Here `TryReadInt` is specialized to the parsing workflow. A private method would need extra parameters or a small struct to carry parser state. Capture keeps the code linear. The cost: **`TryReadInt` cannot be tested in isolation** without invoking `TryParseRange`, and over-capture (reading half the outer method's locals) makes the helper harder to reason about.

### Local Functions vs Lambdas

Both can capture outer variables. Prefer a **local function** when you need **`ref`/`out`**, **`yield return`**, **`async`/`await`** with clear recursion, or **direct recursive calls** by name. Prefer a **lambda** when you pass behavior as a **delegate** to LINQ (`Select`, `Where`) or APIs expecting `Func<>` / `Action<>`.

```csharp
// Local function — natural recursion
int Fibonacci(int n)
{
    int Fib(int k) => k < 2 ? k : Fib(k - 1) + Fib(k - 2);
    return Fib(n);
}

// Lambda — passed as delegate
var doubled = numbers.Select(x => x * 2);
```

Local functions declared inside **`async`** methods can be **`async`** themselves. The state machine is generated per async local function, same as for async lambdas — do not assume "local" means lighter.

## Under the Hood

At compile time, Roslyn lowers local functions into **compiler-generated nested methods** or **structs** on the enclosing type (for generic outer methods, the nested helper may be generic too). They are **not** IL `locals`; they are real methods with mangled names like `<CalculateShipping>g__BaseRate|0_0`.

When a local function **captures** outer state, the compiler creates a **closure class** (same machinery as lambdas) holding captured fields, and the local function becomes an instance method on that closure. A **`static`** local function skips the closure — only arguments pass through.

Because local functions are nested in metadata scope conceptually but emitted as sibling methods, **stack traces** show the generated name, which can look noisy in logs. Private methods appear with the name you chose. For production diagnostics on hot paths, a well-named private method sometimes reads cleaner in APM tools.

Local functions support **declaration order flexibility** within one outer method: you can call a local function before its declaration in source. Private methods require forward declarations only in the sense of class order — C# allows calling private methods defined later in the class file.

## Syntax / API

```csharp
// Statement-bodied local function
void Outer()
{
    int Add(int a, int b)
    {
        return a + b;
    }
}

// Expression-bodied
void Outer() => void Local() => Console.WriteLine("hi");

// static + iterator
IEnumerable<int> Outer()
{
    static IEnumerable<int> Range(int n)
    {
        for (int i = 0; i < n; i++)
            yield return i;
    }
    return Range(10);
}

// async local function
async Task OuterAsync()
{
    async Task<int> DelayedDouble(int x)
    {
        await Task.Delay(10);
        return x * 2;
    }
    return await DelayedDouble(5);
}
```

Local functions **cannot** have access modifiers (`public`, `private`). They **cannot** be `virtual`. Attributes on local functions are limited compared to type members.

## Examples

### Validation pipeline in a minimal API handler

A handler validates input before calling a service. Local functions keep each rule named and ordered without adding four private methods to the class:

```csharp
app.MapPost("/orders", (CreateOrderDto dto, IOrderService svc) =>
{
    static bool IsValidSku(string sku) =>
        sku.Length is >= 3 and <= 20 && sku.All(char.IsLetterOrDigit);

    void Reject(string message) =>
        throw new BadHttpRequestException(message);

    if (string.IsNullOrWhiteSpace(dto.CustomerEmail))
        Reject("Email required.");
    if (!IsValidSku(dto.Sku))
        Reject("Invalid SKU.");

    return svc.CreateAsync(dto);
});
```

`IsValidSku` is **`static`** — pure function of its argument. `Reject` captures nothing if implemented with `throw` only; if it needed `dto` for logging, it would become non-static or take parameters explicitly.

### When to extract to a private method

If `IsValidSku` is reused in `MapPut`, `MapPatch`, and an import job, move it to **`private static bool IsValidSku(string sku)`** on a shared validator class. Local functions should not become copy-paste across members.

## Common Mistakes & Anti-patterns

**Turning every private method into a local function.** If two public methods need the same helper, a local function in one method hides duplication — the other method will reimplement or call a private method anyway. Duplication wins unless you promote the helper.

**Large local functions (50+ lines).** Nesting does not reduce complexity; it hides it. Extract a private method or a small dedicated type when the helper has its own conceptual boundary.

**Over-capture.** A local function that reads ten outer locals is harder to test and reuse than one with three explicit parameters. Prefer passing values in.

**Assuming static local functions are always faster.** They avoid closure allocation, but micro-optimization without profiling rarely matters next to I/O. Choose **`static`** for clarity of data flow first.

**Recursive local functions without static when capture is unnecessary.** Fibonacci-style recursion should use **`static`** local functions with parameters to avoid accidental capture of mutable outer state.

## Production & Real-World Notes

In ASP.NET Core, local functions appear frequently in **`Program.cs` minimal APIs**, filter factories, and complex **`MapGroup`** setup where extracting a class feels heavy. Teams often allow local functions in handlers up to ~15 lines each; beyond that, reviewers ask for **`private static`** helpers or a **`Validator`** class.

For **testability**, business rules that affect money, security, or compliance should live in injectable services or at least **`private static`** pure functions in testable static classes — not buried as local functions inside 200-line methods. Local functions are a readability tool, not an architecture layer.

Stack traces from generated local function names show up in **Application Insights** and **Sentry**. If an exception originates in `<Outer>g__Helper|0_0`, map it mentally to your source name `Helper`.

## Comparison / Trade-offs

| Criterion | Local function | Private method |
|---|---|---|
| Visibility | Single enclosing member | Whole class |
| Reuse across members | No | Yes |
| Unit test in isolation | Impractical | Possible (or via public API) |
| Capture outer locals | Yes (if non-static) | No — pass parameters |
| `ref`/`out` params | Yes | Yes |
| Class surface area | Unchanged | Grows |
| Best for | Single-use workflow steps | Shared class logic |

When a "local" helper needs **`override`** or is part of a **strategy** shared by subclasses, it must be a **`protected`**/`private` instance method — local functions cannot participate in inheritance.

## Quick Reference

| Need | Use |
|---|---|
| Helper used once in one method | Local function |
| Helper used in 2+ methods | Private method (or static class) |
| Pure helper, no capture | `static` local function |
| Passed to LINQ as delegate | Lambda |
| Recursive with clear name | Local function |
| Must unit test rule alone | Private static or service |

---

## Key Takeaways

- Local functions nest **inside a member**; they are not class members.
- Choose local functions for **single-use, localized** workflow clarity.
- Choose private methods for **reuse, testing, and stable diagnostics**.
- **`static` local functions** avoid closures — prefer when capture is not needed.
- Local functions share **closure machinery** with lambdas when they capture.
- Do not use local functions to avoid **design** — extract types when logic grows.
