# Undo: restore, revert vs reset (soft/mixed/hard)

> Roadmap: `0.1.8` · Node: `0.1` — Git Basics · Depth: **deep**

## Learning Objectives

After this lesson you will be able to:

- Explain the three Git trees: **HEAD**, **index (staging)**, and **working tree**.
- Use **`git restore`** to discard or re-stage file changes safely.
- Use **`git revert`** to undo a commit with a **new** commit on shared branches.
- Use **`git reset --soft/--mixed/--hard`** and state exactly what moves.
- Recover from mistakes with **`git reflog`**.
- Choose the right undo strategy: **rewrite history** vs **add corrective commit**.

---

## Why This Matters

Every developer eventually faces the stomach-dropping moment when the wrong thing lands in Git. You staged `secrets.env` by mistake. You committed directly to `main` instead of your feature branch. You need to undo last week's merge on a branch that three teammates already pulled. The commands in this lesson are how you recover — but the wrong undo command can cost you hours of work or break your team's shared history.

The difference between a confident Middle developer and a nervous Junior is not memorizing command syntax. It is knowing **what moves where** when you run each command. Git does not have a single "undo" button. It has several mechanisms that look similar in conversation — "can you undo that commit?" — but behave very differently under the hood. Some move branch pointers backward and rewrite the directed acyclic graph. Some add new commits that patch forward. Some only touch files on disk without moving history at all. Choosing wrong in a shared repository can force painful force pushes and rebases across the team. Choosing wrong locally can permanently destroy uncommitted edits that never became Git objects.

This lesson goes deep because undo operations sit at the intersection of Git's three-tree model, reference mechanics, and collaboration etiquette. Master this, and incidents become recoverable stories instead of panicked Slack threads.

## Core Concepts

Git manages your project through three synchronized representations that developers often conflate. Understanding them separately is the foundation for every undo decision.

The first tree is **HEAD**, which represents the current commit on your branch — the last snapshot that was recorded in history. When people say "the last commit," they mean what HEAD points to, usually through a branch reference like `refs/heads/feature`. HEAD is the authoritative "what Git thinks the project looks like in committed form."

The second tree is the **index**, also called the staging area or cache. It holds the snapshot that the next `git commit` will record. When you run `git add`, you copy content from your working tree into the index and create blob objects for staged content. The index is a flat structure in `.git/index` listing paths, modes, blob hashes, and metadata. It is the draft of your next commit.

The third tree is the **working tree** — the actual files on disk that your editor shows. This is what you modify when you type. Git compares the working tree against the index to show unstaged changes, and compares the index against HEAD to show staged changes.

The normal forward flow connects all three trees in sequence. You edit a file in the working tree. You run `git add` to copy those edits into the index. You run `git commit` to create a new commit object from the index and move the branch reference forward so HEAD points to the new commit. Undo commands break or reverse parts of this chain — but they do not all break the same parts.

```
edit file.ts          →  working tree changes
git add file.ts       →  working tree → index
git commit            →  index → new commit, HEAD moves forward
```

Beyond the three trees, undo philosophy splits into two camps that matter enormously on team projects. **Rewrite history** means moving branch pointers backward or changing commits in place so the repository looks as if certain commits never happened. Commands like `git reset` and `git rebase` rewrite. **Add new truth** means leaving existing commits intact and appending corrective commits. `git revert` works this way. On shared branches that others have already pulled, rewrite forces everyone to reconcile divergent history. Forward correction with revert lets teammates run a normal `git pull` and receive the undo as just another commit.

The rule of thumb that saves careers: if anyone might have pulled the commit you want to undo, prefer revert over reset.

## Under the Hood

A branch is not a container of commits — it is a movable pointer to a single commit hash. `refs/heads/main` might point to `a1b2c3d`. HEAD usually points to a branch name, not directly to a commit, so HEAD → `refs/heads/main` → `a1b2c3d`. When you run `git commit`, Git creates a new commit object whose parent is the current tip, then moves the branch reference forward. When you run `git reset --hard` to an older commit, Git moves that branch reference backward. The newer commits still exist as objects in `.git/objects` for a while, but no branch points to them — they become unreachable until garbage collection.

HEAD can also point directly to a commit in **detached HEAD** state — covered in lesson `0.1.11`. During reset on a normal branch, you move the branch ref that HEAD indirectly references. That is why `git reset` affects history and `git restore` does not: restore manipulates index and working tree entries through checkout machinery without moving the branch tip.

The index stores more than a list of paths. Each entry includes a stage number used during merges — stage zero for normal entries, stages one through three for conflicted paths. For undo purposes, what matters is that `git add` writes blob objects and updates index entries, `git reset --mixed` repopulates the index from a target commit's tree, and `git reset --hard` repopulates the index and then checks out files to the working tree to match.

Consider what happens at the object level when you reset hard from `C3` back to `C2`:

```
Before:  main → C1 ─ C2 ─ C3
After reset --hard to C2:  main → C1 ─ C2    (C3 unreachable from main)
```

Commit `C3` is not instantly deleted. It remains in the object database and in your reflog until expiration and garbage collection. That is why recovery is possible.

Revert creates an entirely different shape. If `C3` introduced a bug and you revert it:

```
Before:  main → C1 ─ C2 ─ C3
After revert:  main → C1 ─ C2 ─ C3 ─ C4
```

Here `C4` is a new commit whose tree applies the inverse patch of `C3`. Git uses the same merge machinery as resolving conflicts — if the revert cannot apply cleanly because later commits modified the same lines, you resolve conflicts manually like a merge. The original mistake remains visible in history, which is either a compliance advantage or a cosmetic annoyance depending on your team's audit requirements.

The **reflog** is a local append-only log of reference movements stored in `.git/logs/HEAD` and `.git/logs/refs/heads/<branch>`. Each line records old hash, new hash, actor, timestamp, and a message such as `commit`, `reset`, or `checkout`. Reflog is not pushed to remotes by default. It is your personal safety net when you hard-reset too far and need to find the hash of where HEAD was thirty seconds ago. Entries expire — default ninety days — so recovery is not infinite, but it covers essentially all "oops" moments in daily work.

Why is `--hard` destructive to uncommitted work? Only content that reached the object database survives independently of branch pointers. Staged content has blob objects because `git add` created them. Pure working-tree edits that were never staged were never hashed — reset hard overwrites files to match the target commit tree and those edits are gone with no object to recover. This asymmetry is the single most important reason to pause before typing `--hard`.

## Syntax / Commands / API

Git 2.23 introduced `git restore` as the modern, focused command for moving content between index and working tree without touching HEAD. Before that, developers overloaded `git checkout` for both branch switching and file restoration — a common source of confusion. Restore separates file operations from branch operations.

To discard unstaged changes in the working tree, restoring a file from the index:

```bash
git restore Program.cs
```

The index is unchanged; the file on disk now matches what is staged. If nothing was staged, it matches HEAD. To unstage a file — move the index entry back to match HEAD while keeping your working tree edits:

```bash
git restore --staged Program.cs
```

Your disk file still has your edits; Git just removes it from the next commit's draft. To both unstage and discard working tree changes in one step, returning the file fully to the last committed state:

```bash
git restore --staged --worktree Program.cs
```

This replaces the older pattern `git checkout HEAD -- Program.cs`. Restore never moves the branch. That constraint is a feature — you can fix local file mistakes without touching history.

`git revert` creates a new commit that undoes a specified commit. The original stays in history.

```bash
git revert a1b2c3d
git revert --no-edit a1b2c3d
```

Use `--no-edit` when you accept the default revert message and want to avoid opening an editor. Reverting a merge commit requires telling Git which parent represents the mainline:

```bash
git revert -m 1 <merge-commit-hash>
```

The `-m 1` flag means "revert to the state of the first parent," which is usually the branch you merged into (e.g., `main`). Without it, Git cannot automatically choose and will abort with an error. After a successful revert on a shared branch, a normal push propagates the fix without force.

`git reset` moves HEAD and optionally the index and working tree. Three modes control how far the movement propagates:

```bash
git reset --soft  <commit>
git reset --mixed <commit>   # default when mode omitted
git reset --hard  <commit>
```

Soft reset moves only HEAD. The index still contains whatever was staged before — including the snapshot from the commit you just "removed." Your working tree is untouched. This is ideal when you committed too early and want to re-stage differently or amend the content before recommitting. Mixed reset moves HEAD and resets the index to match the target commit. Your working tree files stay as they are on disk, but the changes that were in the undone commit now appear as unstaged modifications. This is the most common "undo last commit but keep my work" pattern. Hard reset moves all three: HEAD, index, and working tree match the target commit exactly. Anything not in that commit's tree is discarded from the index and overwritten on disk.

Visualizing mixed reset undoing the last commit:

```
BEFORE:
  HEAD  → C3 (bad commit)
  Index → same as C3
  WT    → your edited files

AFTER git reset HEAD~1  (--mixed):
  HEAD  → C2
  Index → matches C2  (C3's changes now appear as unstaged diff)
  WT    → unchanged on disk
```

Reflog recovery after a mistake:

```bash
git reflog
# f4e8b21 HEAD@{0}: reset: moving to HEAD~1
# a1b2c3d HEAD@{1}: commit: Add feature

git reset --hard HEAD@{1}
# or: git reset --hard a1b2c3d
```

## Examples

Scenario A: you ran `git add .` and accidentally staged `secrets.env`. You want to unstage that one file but keep editing it.

```bash
git restore --staged secrets.env
```

The file remains on disk with your changes; it simply will not be included in the next commit. Add it to `.gitignore` before your next add.

Scenario B: you edited `src/temp.cs` locally, decided the experiment failed, and want to throw away those edits.

```bash
git restore src/temp.cs
```

The working tree copy now matches the index. If you never staged, that means it matches the last commit.

Scenario C: you made a commit on your private feature branch that nobody has pulled, and the commit is premature — wrong message, wrong grouping of files.

```bash
git reset --soft HEAD~1
```

Your changes remain staged exactly as they were in the removed commit. Fix staging, write a better message, commit again. If you prefer changes unstaged so you can selectively re-add:

```bash
git reset HEAD~1
```

Mixed is the default; `--mixed` is optional.

Scenario D: the bad commit is already on `origin/main`. Teammates may have pulled it. Do not reset and force push.

```bash
git pull
git revert badcommit_hash
git push origin main
```

The team receives a forward-moving undo commit. CI runs on the revert like any other commit. History shows the mistake and the correction — acceptable in most audit environments.

Scenario E: you ran `git reset --hard HEAD~3` and immediately realized you reset too far.

```bash
git reflog
git reset --hard HEAD@{1}
```

Find the entry from before the mistaken reset. Recovery often takes thirty seconds if you act before garbage collection and before closing the terminal session where you still remember what happened.

Scenario F: reverting a merge that brought a broken feature into main.

```bash
git log --oneline -5
# identify merge commit hash
git revert -m 1 abc1234
```

Resolve any conflicts, commit completes, push. The feature branch still exists; only main's integration is undone.

## Common Mistakes & Anti-patterns

Running `git reset --hard` on a shared branch and force pushing is the classic team-breaking mistake. You rewrite history that others built upon. They must fetch and reset or rebase their local work. Use revert on published commits unless the team explicitly coordinates a history rewrite — and even then, prefer revert for anything on main.

Confusing `restore` and `reset` causes either failed expectations or accidental history moves. Remember: **restore** adjusts files in the index and working tree; **reset** moves HEAD. If you only wanted to unstage, `git restore --staged` expresses intent more clearly than `git reset HEAD file`, though the older form still works.

Assuming revert deletes the bad commit leads to surprise when `git log` still shows it. Revert adds an inverse commit; the original remains. That is by design for shared safety and audit trails.

Reverting a merge without `-m` fails or produces wrong results because Git needs to know which parent line to preserve. Always identify the mainline parent — usually `-m 1`.

Ignoring reflog after a hard reset triggers unnecessary panic. Commits are rarely instantly gone. Check reflog before assuming disaster.

Using `--hard` when `--mixed` or `--soft` would suffice destroys work you meant to keep. Pause and ask: do I need to discard working tree files, or just move the branch pointer?

## Production & Real-World Notes

On unpushed local feature branches, soft and mixed reset are everyday tools for cleaning up before opening a pull request. Squashing WIP commits often starts with interactive rebase, but soft reset to branch point achieves similar staging control with less ceremony for simple cases.

On protected main branches in GitHub or GitLab, force push is blocked. Even if you could reset locally, the remote rejects the rewrite. Revert is not just etiquette — it may be the only available mechanism.

When secrets are committed, reset alone is insufficient if the commit was ever pushed. The secret exists in remote history, forks, and CI logs. Rotate the credential, use `git filter-repo` or BFG for history cleaning if policy requires, and revert the introducing commit forward. Resetting locally does not unpublish secrets.

CI pipelines triggered on push treat revert commits normally. After reverting a breaking merge, monitor the pipeline on the revert commit to confirm main is green again before notifying the team.

Team coordination after an emergency force push — when rewrite was unavoidable — requires broadcast communication. Teammates run `git fetch` and either `git reset --hard origin/branch` if they had no local commits, or rebase their local work onto the rewritten tip. High coordination cost is why revert remains the default recommendation.

For undoing only the last commit message on an unpushed commit, `git commit --amend` is simpler than reset — covered in lesson `0.1.10`. Choose amend when the tree is correct and only metadata is wrong.

## Comparison / Trade-offs

Restore versus reset versus revert answers three different questions. Restore: "make my files or staging area match something without changing history." Reset: "move my branch backward to a previous commit, optionally taking my files with it." Revert: "create a new commit that undoes a past commit while preserving history."

Soft reset preserves staged state — best when recommitting. Mixed reset preserves working tree but unstages — best when re-organizing what goes into the next commit. Hard reset preserves nothing beyond the target snapshot — best only when you genuinely want to discard all local divergence.

Rewrite history through reset and rebase produces clean linear narratives that read as if mistakes never happened. That is aesthetically pleasing and sometimes required before merging a polished feature branch. It hides errors from casual log readers — a pro for external-facing repos, a con for compliance that requires full audit trails. Forward correction through revert preserves transparency: the mistake and fix are both visible. Collaboration cost is lower because pulls are fast-forward merges.

Neither approach is universally superior. Middle developers choose based on whether commits are published, whether audit requires visible corrections, and whether teammates depend on the current tip.

## Quick Reference

Discard unstaged file edits: `git restore <file>`. Unstage: `git restore --staged <file>`. Undo last commit keeping staged: `git reset --soft HEAD~1`. Undo last commit keeping unstaged: `git reset HEAD~1`. Undo last commit discarding all: `git reset --hard HEAD~1`. Undo pushed commit safely: `git revert <hash>`. Find lost commits: `git reflog`.

---

## Key Takeaways

- Three trees: HEAD (last commit), index (staging), working tree (disk files).
- `git restore` adjusts index and working tree without moving the branch.
- `git reset --soft/mixed/hard` moves HEAD and optionally index and working tree.
- `git revert` appends an inverse commit — safe for shared, pushed branches.
- `git reflog` records local reference movements — recovery after mistaken reset.
- Shared or pushed history: revert, do not reset without team coordination.
- `--hard` destroys uncommitted work that was never staged into objects.
