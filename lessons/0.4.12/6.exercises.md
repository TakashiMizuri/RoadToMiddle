# Exercises: ping and traceroute (Hands-on)

> Complete after lectures and passing `4.test-yourself.md`. Run commands on your machine. Public targets only unless you own the server.

## Easy

### E1. Baseline ping
**Goal:** Read ping output.

**Steps:**
1. Run `ping -c 4 1.1.1.1` (Linux/macOS) or `ping -n 4 1.1.1.1` (Windows).
2. Record: packets transmitted/received, **packet loss %**, **min/avg/max RTT**.

**Success criteria:** All four fields documented in your notes.

---

### E2. Hostname vs IP
**Goal:** Connect DNS to ping.

**Steps:**
1. `ping -c 2 one.one.one.one` (or `1.1.1.1`).
2. `ping -c 2 cloudflare.com`.
3. Compare RTT — similar?

**Success criteria:** One sentence: did DNS resolution affect RTT measurement (hint: only if resolution slow/failed)?

---

## Medium

### M1. Traceroute hop map
**Goal:** Interpret path.

**Steps:**
1. Run `traceroute -I 8.8.8.8` (Linux), `traceroute 8.8.8.8` (macOS), or `tracert 8.8.8.8` (Windows).
2. Identify **first 3 hops** (IP or hostname) and RTT.
3. Note any `* * *` line — does trace continue past it?

**Success criteria:** Table of 3 hops with RTT; one sentence on silent hop if present.

---

### M2. ping vs HTTP
**Goal:** ICMP vs TCP application check.

**Steps:**
1. `ping -c 2 google.com`
2. `curl -I -s -o /dev/null -w "HTTP %{http_code} time_total %{time_total}s\n" https://google.com`  
   Windows PowerShell: `curl.exe -I https://google.com`
3. Compare: ping works? HTTP works?

**Success criteria:** Both succeed; note curl measures **application** layer including TLS.

---

### M3. Port check (Windows or cross-platform)
**Goal:** Companion to ping.

**Steps:**
- Windows: `Test-NetConnection google.com -Port 443`
- Linux/macOS: `nc -zv google.com 443` or curl as above

**Success criteria:** Confirm **TcpTestSucceeded** or **succeeded** / HTTP response — port 443 reachable even if ICMP were blocked.

## Hard (stretch)

### H1. Incident write-up
**Goal:** Production-style triage doc.

**Scenario (simulated):** Users in Office A report slow API; Office B fine. You run from Office A:

```
ping api.company.com → avg 220 ms, 0% loss
tracert api.company.com → RTT jumps at hop 7 to 200 ms
curl https://api.company.com/health → HTTP 200, time 1.2s
```

Write 5–8 sentences: hypothesis, what ping/traceroute/curl each proved, one recommended next step (not code fix).

**Success criteria:** Distinguishes network latency vs app slowness; does not blame ping failure alone.

---

### H2. Compare two targets
**Goal:** Latency geography intuition.

Ping and traceroute (or tracert) to:
- A regional target near you (e.g. local ISP DNS or national site)
- `aws.amazon.com` or `google.com`

Compare avg RTT and hop count. Relate to `0.4.11`.

**Success criteria:** Short paragraph comparing RTT; mentions distance/routing.

## Hints (read only if stuck)

<details>
<summary>Hint E1 Windows</summary>
Use `ping -n 4` instead of `-c`. RTT shown as `time=XXms` per reply; summary at end.
</details>

<details>
<summary>Hint M1 permission</summary>
Some Linux systems need `sudo traceroute` or `-I` ICMP mode if UDP blocked.
</details>

<details>
<summary>Hint M2 Windows curl</summary>
Use `curl.exe` in PowerShell to avoid Invoke-WebRequest alias differences, or use `Invoke-WebRequest -Method Head -Uri https://google.com`.
</details>

---

## Safety note

Do not traceroute or port-scan hosts you do not own without permission. Use public resolvers (1.1.1.1, 8.8.8.8) and major public websites for practice.
