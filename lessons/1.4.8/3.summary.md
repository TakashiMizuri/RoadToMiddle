# ConfigureAwait(false) — краткая выжимка

> `1.4.8` · ~10 мин на повторение

## Главная идея

`ConfigureAwait(bool)` задаёт, **куда** после `await` resume'ится state machine (`MoveNext`): захватить `SynchronizationContext` (default `true`) или продолжить на thread pool (`false`). В **libraries** — почти всегда `false`; в **ASP.NET Core app** — обычно не нужен.

## Ключевые пункты

- Параметр = `continueOnCapturedContext`.
- `true` → `SynchronizationContext.Post` если context был при await.
- `false` → без capture; continuation на thread pool / inline.
- Не путают с `ExecutionContext` / `AsyncLocal` — только SyncContext.
- Каждый `await` решает capture **отдельно**.
- UI code: `false` перед update controls — опасно.
- Связь с `1.4.7`: awaiter регистрирует callback на `MoveNext`.

## Самые важные примеры

```csharp
// library
await client.GetStringAsync(url).ConfigureAwait(false);

// ASP.NET Core — OK без ConfigureAwait
var user = await _repo.GetByIdAsync(id);
```

## «Под капотом» в одном абзаце

Compiler генерирует `ConfiguredTaskAwaitable`; `UnsafeOnCompleted` при `false` не сохраняет `SynchronizationContext.Current`. При completion `TaskAwaiter` либо `Post` в context, либо queue на pool. State machine struct тот же — меняется только scheduling policy continuation после await.

## Частые ошибки

- `ConfigureAwait(false)` в UI перед controls.
- Один `false` на метод вместо на каждый await.
- Ожидание, что `false` лечит thread pool starvation.

## Мнемоника

**Library = false everywhere. UI = default. Core app = omit.**
