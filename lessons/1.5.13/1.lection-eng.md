# stackalloc — Stack Allocation

> Roadmap: `1.5.13` · Node: `1.5` — C#: .NET Runtime and Memory · Depth: понимание

## Learning Objectives

After this lesson you will be able to:

- Explain **`stackalloc`** as allocation from the **current stack frame** returning **`Span<T>`** (modern C#).
- Contrast stack allocation with **heap** (`new`, **`ArrayPool`**) for small, short-lived buffers.
- State default **stack size** limits and **stack overflow** risk from large or unbounded `stackalloc`.
- Apply **`stackalloc`** safely in parse/format hot paths without returning spans to callers.
- Recognize analyzer guidance (**CA2014**) against **`stackalloc` in loops**.

---

## Why This Matters

Not every buffer deserves a heap allocation. Parsing a 64-byte header, formatting a GUID, or building a short UTF-8 key for a dictionary lookup can use **stack memory** that disappears when the method returns — **zero GC pressure**. **`stackalloc`** is the idiomatic C# syntax, especially combined with **`Span<T>`** from **`1.5.12`**.

Used carelessly, **`stackalloc` in a loop** or **`stackalloc int[1_000_000]`** blows the stack — **`StackOverflowException`**, process crash, no catch. Middle developers choose **thresholds**: tiny buffers on stack, larger on **`ArrayPool`** (`1.5.14`), unbounded on heap with clear ownership.

## Core Concepts

### What stackalloc Does

```csharp
Span<int> numbers = stackalloc int[4] { 1, 2, 3, 4 };
Span<byte> buffer = stackalloc byte[128];
```

Memory comes from the **stack** of the current thread — same region as local variables and stack frames (`1.5.4`). When the method returns, the frame is popped — memory is **invalid** (do not expose **`Span`** pointing to it outside the method).

Historically **`stackalloc`** returned pointers (`unsafe`); since C# 7.2 with **`Span`**, safe usage is default in safe context.

### When to Use

**Good fits:**

- Small fixed upper bound (e.g., ≤ 256 bytes/elements)
- **Single method** lifetime
- Hot path micro-optimization after profiling

**Poor fits:**

- Size depends on **user input** without strict cap
- **Loops** allocating each iteration (accumulated stack usage in recursive/loop scenarios — CA2014)
- Data must outlive method — use **heap**, **`ArrayPool`**, or **`MemoryPool`**

### Stack Size Limits

Default **~1 MB** per thread on many runtimes (OS-dependent; **ASP.NET** thread pool threads same rule). Deep recursion + large `stackalloc` → overflow.

**Rule of thumb:** **`stackalloc byte[512]`** often fine; **`stackalloc byte[8192]`** — question it; **`stackalloc byte[100_000]`** — almost always wrong (use pool).

Optional: **`[SkipLocalsInit]`** (.NET 5+) skips zero-init for perf — only if you fully write buffer before read.

### Pattern with Span APIs

```csharp
static bool TryParseGuid(ReadOnlySpan<char> text, out Guid guid)
{
    Span<char> tmp = stackalloc char[36];
    // copy/normalize into tmp if needed
    return Guid.TryParse(tmp, out guid);
}
```

Combine with **`Utf8Formatter`**, **`Base64`** into stack buffer then compare.

### vs ArrayPool

| | stackalloc | ArrayPool |
|---|------------|-----------|
| Lifetime | Method | Until Return |
| GC | None | None while rented |
| Size | Small only | Large OK |
| Thread | Current stack | Shared pools |

## Under the Hood

Compiler emits **`localloc`** IL (or equivalent) — allocates block on stack, **untracked by GC**. Speed: pointer bump in stack pointer register.

**Zero-init** by default unless SkipLocalsInit.

Nested calls: each frame's stackalloc dies with that frame.

---

## Common Mistakes & Anti-patterns

**Return Span from stackalloc** — undefined behavior when caller uses it.

**stackalloc size from user input** — security/DoS vector.

**stackalloc in loop** — multiplies stack pressure.

**Replacing every `new byte[32]`** without measurement — premature optimization.

---

## Production & Real-World Notes

Framework code uses stackalloc internally with conservative bounds. Your API code: prefer **`ArrayPool.Shared.Rent`** for buffers **> 1 KB** or unknown size.

In **ASP.NET**, request thread stack is shared pattern — same limits apply.

## Quick Reference

```csharp
Span<byte> buf = stackalloc byte[256];
// use buf in method only
// never return buf
```

---

## Key Takeaways

- **stackalloc** = fast, GC-free, **method-scoped** buffer via **Span**.
- Respect **stack limits**; cap sizes.
- Never **escape** Span to callers.
- Large/unknown → **ArrayPool** or heap.
- Profile before optimizing.
