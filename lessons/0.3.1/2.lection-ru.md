# Client-Server и модель request-response

> Roadmap: `0.3.1` · Node: `0.3` — HTTP и Web · Depth: понимание

## Learning Objectives

После этой лекции ты сможешь:

- Объяснить модель **client-server** как разделение **ролей**, а не физическую топологию.
- Описать **цикл request-response** от действия пользователя до страницы или ответа API.
- Отделить **HTTP** как протокол прикладного уровня от транспорта TCP/IP ниже.
- Определить, кто инициирует обмен в типичных web- и API-сценариях.
- Рассуждать о **statelessness** на уровне HTTP и о последствиях для масштабирования.
- Сопоставить роли browser, mobile app и server в fullstack-стеке ASP.NET Core + React.
- Предсказать, что происходит, когда client отправил request, но response не получил.

## Why This Matters

Любое fullstack-приложение — React SPA с ASP.NET Core API, мобильное приложение с теми же endpoints, webhook-receiver — опирается на один фундаментальный паттерн: **одна сторона спрашивает, другая отвечает**. Паттерн настолько привычен, что кажется невидимым: нажал кнопку — данные появились. Отправил форму — сервер провалидировал и сохранил. Но когда что-то ломается — timeout, CORS, 502 Bad Gateway, «network request failed» в консоли — нужно видеть механизм, а не только UI.

**Client-server model** называет роли в этом механизме. **Client** — любое ПО, которое *инициирует* request, потому что ему нужно что-то: HTML, JSON, файл, решение об авторизации. **Server** — ПО, которое *слушает* эти запросы и *отвечает* представлением ресурса, ошибкой или redirect. Роли логические, не привязаны к одной машине. Твой ASP.NET Core API — server для `/api/orders`, но становится client, когда вызывает Stripe или SendGrid. Browser — client при загрузке React, а React остаётся client при вызове API через `fetch`.

Понимание request-response на этом уровне связывает весь узел `0.3`. URL (`0.3.2`) говорят client'у, *куда* слать request. Структура сообщения (`0.3.3`) — *какие байты* идут по сети. Methods (`0.3.4`) и safe/idempotent (`0.3.5`) — *какую операцию* просит client. Status codes (`0.3.6`) — *исход*. Без рамки client-server это разрозненные факты; с ней — связная история о том, как работает web.

## Core Concepts

### Роли, а не машины

«Server» часто ассоциируют с rack в дата-центре, «client» — с ноутбуком или телефоном. Аппаратная картина полезна, но неполна. **Client и server — роли в диалоге**, и одно устройство может быть и тем и другим. При `dotnet run` и открытии `https://localhost:5001` в Chrome ноутбук одновременно хостит server (Kestrel) и client (browser). Когда API обращается к PostgreSQL, процесс API — client, PostgreSQL — server — на уровне протокола БД, не HTTP.

Client определяется **инициативой**: client открывает соединение (в HTTP/1.1 типично) и шлёт первое сообщение. Server ждёт на известном адресе — IP + port, часто 443 для HTTPS — пока не придёт request. Server не push'ит произвольные данные random client'ам без prior request, кроме отдельных паттернов вроде WebSockets (`0.3.15`), которые всё равно начинаются с HTTP upgrade request от client.

В production один server process обслуживает много client'ов параллельно. Kestrel принимает TCP-соединения, парсит HTTP requests, прогоняет middleware pipeline и пишет responses. Тысячи client'ов могут делить один IP server'а; routing (path, Host header) и authentication различают, чьи данные в каждом response.

### Цикл request-response

В основе HTTP простой контракт: **один request → один response** (в классической модели). Цикл повторяется миллиарды раз в сутки.

Пользователь открывает страницу продукта в React app. Browser (client) может сначала запросить JS bundle с CDN (server) — один request-response. React монтируется, вызывает `GET /api/products/42`, controller (server) через EF Core отдаёт JSON с `200 OK` — ещё один цикл. Картинка — ещё один. Один экран часто запускает **много последовательных или параллельных** HTTP-обменов.

Каждый цикл имеет одну абстрактную форму:

```
Client                                    Server
  |                                          |
  | -------- TCP connection --------------> |
  | -------- HTTP Request -----------------> |
  |         (method, URL, headers, body)     |
  |                                          | process
  | <------- HTTP Response ---------------- |
  |         (status, headers, body)          |
```

Client собирает **request**: method (часто GET для чтения), target URL, headers вроде `Accept: application/json`, иногда body. Server парсит request, выполняет логику — routing, auth, БД — и собирает **response**: status code, headers вроде `Content-Type`, body (HTML, JSON, пусто при 204). Client интерпретирует response: рендер HTML, обновление state в React, toast с ошибкой.

На уровне HTTP-сообщений цикл **синхронный**: client шлёт один request и ждёт один response в этой паре (хотя client может pipeline или multiplex несколько обменов на одном connection в HTTP/2+). `async/await` в коде нужен не потому, что HTTP «асинхронный по природе», а потому что **ожидание I/O** без блокировки потоков эффективнее, пока response идёт по сети.

### HTTP поверх транспорта

HTTP — протокол **прикладного уровня** (layer 7 OSI). Он предполагает надёжный byte stream — почти всегда **TCP**, или **QUIC** для HTTP/3 (`0.3.13`). HTTP сам не маршрутизирует пакеты и не шифрует. **TLS** (HTTPS, `0.3.8`) оборачивает TCP и даёт шифрование и проверку сертификата до потока HTTP-сообщений.

При вводе URL до первого HTTP-байта происходит многое: DNS → IP (`0.4.x`), TCP handshake, TLS handshake. Потом client шлёт `GET / HTTP/1.1`. Fullstack-разработчик редко реализует эти слои, но stack объясняет latency: «медленный API» может быть DNS, TLS, server logic или БД — не обязательно C#.

### Stateless по умолчанию

HTTP спроектирован **stateless**: каждый request несёт всё, что server нужно для обработки. Server по протоколу не «помнит» прошлые requests от того же client. Встроенной session в протоколе нет.

Приложениям нужна непрерывность — login, корзина, настройки. **State симулируют поверх stateless HTTP**: cookies (`0.3.9`), JWT в `Authorization`, server-side session store по session ID. Протокол остаётся stateless; **приложение** добавляет state через идентификаторы в каждом request. Это trade-off для horizontal scaling: любой app server в pool может обработать request с валидным token.

Statelessness также значит, что **retry неоднозначен** для non-idempotent операций — отсюда semantics methods в `0.3.4`–`0.3.5`. Дублированный POST может создать два заказа; дублированный GET обычно безвреден.

### Типичное fullstack-сопоставление

| Компонент | Роль | Инициирует |
|-----------|------|------------|
| Browser с React | HTTP client | Страница, assets, API |
| React SPA (fetch) | HTTP client | Requests к backend |
| ASP.NET Core | HTTP server | Responses; client к БД и внешним API |
| PostgreSQL | Server (wire protocol) | — |
| CDN | HTTP server | Static assets |

React работает **в** browser, но **является** client для API. `ProductsController` — server для `/api/products` и client при `HttpClient` к внешнему сервису.

## Under the Hood

Для depth «понимание» достаточно обзора; детали сообщений — в `0.3.3`.

Kestrel буферизует bytes, пока не распарсит request line и headers (и body по `Content-Length` или chunked). Строит `HttpContext` и передаёт в middleware pipeline. Controller или minimal API delegate выполняются в конце. По завершении Kestrel сериализует status, headers и body обратно в TCP.

На стороне client browser stack (или `HttpClient`) делает обратное. Timeout'ы на нескольких уровнях: TCP, TLS, HTTP client, server request timeout. «Зависший» request — server ещё считает, потерянный пакет или client уже не читает response.

## Syntax / Commands / API

Наблюдение request-response через **curl** (полный tooling — `0.3.12`):

```bash
curl -v https://httpbin.org/get
```

`-v` печатает headers request и response. В DevTools вкладка **Network** — каждый request-response с waterfall: DNS, connect, TLS, wait (TTFB), receive.

Minimal endpoint в ASP.NET Core:

```csharp
app.MapGet("/hello", () => Results.Text("Hello, client"));
```

Каждый `GET /hello` — один request; строка — body response.

## Examples

**Пример 1 — загрузка страницы = много циклов.** `https://shop.example.com/products`: HTML → JS → `GET /api/products` → картинки. Параллельные connections (`0.3.14`) ускоряют, но каждая стрелка — request → response.

**Пример 2 — API как server и client.** `POST /api/orders` (server). Внутри `await _paymentClient.ChargeAsync(...)` — HTTPS POST в Stripe (client). Два вложенных цикла.

**Пример 3 — нет response.** Client timeout 5 с, server отвечает через 30 с. Client abort; user видит ошибку. Server может дописать response в уже закрытое соединение — wasted work.

## Common Mistakes & Anti-patterns

**Server = одна машина.** Load balancer, несколько Kestrel, отдельная БД — логическая роль распределена.

**Server «сам вызовет» client по plain HTTP.** Стандартный HTTP не push'ит на arbitrary IP. Нужны polling, SSE, WebSockets или push notifications.

**Stateless HTTP ≠ нет login session.** Session на application layer через cookies/tokens в **каждом** request.

**Backend часто HTTP client.** Outbound `HttpClient` — timeouts, retry с учётом idempotency, circuit breakers.

## Production & Real-World Notes

Load balancer терминирует TCP от client'ов и открывает новые к app servers. Reverse proxy (nginx, YARP) перед Kestrel.

**Health checks** — `GET /health` → `200` в deadline; иначе restart или вывод из pool.

**Correlation ID** (`X-Correlation-ID`) через client → API → downstream HTTP для трассировки в логах.

Timeout'ы на обеих сторонах: client abort и server timeout против hung handlers.

## Comparison / Trade-offs

**Client-server vs P2P.** Web стандартизировал client-server для caching и центрального контроля данных; CRUD API сюда ложится естественно.

**Request-response vs очереди.** HTTP — немедленный ответ или timeout; очереди — decoupling по времени. HTTP для instant feedback; очереди для фона и peak absorption.

**Thin vs thick client.** SPA переносит logic в browser; classic MVC рендерит HTML на server. HTTP request-response тот же; меняется число round-trip'ов и доля JSON vs HTML.

## Key Takeaways

- Client-server — **роли**, не железо.
- HTTP: **request → response** на один обмен.
- HTTP — **application layer**; TCP/TLS/DNS ниже.
- **Stateless** протокол; state через cookies, tokens, session ID.
- Одно действие пользователя — **много** HTTP-циклов.
- API — **server** для browser и **client** к БД и third-party.
- Сбой на любом этапе: DNS, connect, TLS, logic, timeout.
