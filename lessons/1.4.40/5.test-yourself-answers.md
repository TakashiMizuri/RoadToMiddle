## Answers

### 1. [Recall]
**Answer:** `RequestDelegate` → `Task` (`Task Invoke(HttpContext)`).
**Why:** Async middleware contract.

### 2. [Recall]
**Answer:** **Scoped** (per HTTP request).
**Why:** Registered `AddDbContext` scoped; disposed end of request.

### 3. [Understanding]
**Answer:** `await` на async ADO.NET регистрирует I/O completion; thread не ждёт socket — возвращается в pool.
**Why:** OS async I/O + continuation on complete.

### 4. [Understanding]
**Answer:** Connection pool — лимит **TCP connections** к DB; thread pool — **worker threads** для CPU/continuations. Исчерпание независимо.
**Why:** Много free threads + zero connections = pool timeout.

### 5. [Understanding]
**Answer:** `HttpContext.RequestAborted` — Kestrel сигналит при закрытии клиента; model binding injects `CancellationToken`.
**Why:** Cooperative cancellation chain.

### 6. [Application]
**Answer:** Kestrel → middleware chain → routing/controller → service → DbContext/Npgsql → PostgreSQL (и обратно serialization).
**Why:** Standard layered flow.

### 7. [Application]
**Answer:** В middleware, вызывающем `await next(context)`, или раньше если custom middleware async-read body.
**Why:** Pipeline async from first `next`.

### 8. [Analysis]
**Answer:** Блокирует thread pool thread на sync I/O внутри `Task.Run` — ложная async, хуже scalability, не настоящий async I/O path.
**Why:** Offload ≠ async I/O.

### 9. [Analysis]
**Answer:** DbContext not thread-safe — race/corruption; EF не поддерживает parallel ops на одном instance.
**Why:** Scoped = single-threaded request assumption.

### 10. [Understanding]
**Answer:** Да — нет request-affined SyncContext в ASP.NET Core.
**Why:** Thread pool queue any worker.

### 11. [Application]
**Answer:** SQL text, EXPLAIN, indexes, N+1 Includes, network to DB, pool wait time — не Kestrel first.
**Why:** Trace pinpoints EF span.

### 12. [Understanding]
**Answer:** Меньше work в Change Tracker на read path — быстрее map to DTO, меньше memory per request.
**Why:** Read-only flow optimization.
