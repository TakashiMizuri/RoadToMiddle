# Exception Handling in Async Code

## Learning Objectives

After this lesson you will be able to:

- Explain how exceptions propagate through **`await`** and async state machines.
- Use **`try/catch/finally`** correctly around **`await`** (including **`finally`** timing).
- Unwrap **`AggregateException`** from **`Task.Wait()`**, **`.Result`**, and **`Task.WhenAll`** failures.
- Handle **`Task.WhenAll`** when multiple tasks fault ‚ÄĒ which exception surfaces on **`await`**.
- Contrast **`async void`** exception behavior with **`async Task`** (**`1.4.13`**).
- Apply **`ExceptionDispatchInfo`**, **`throw;`**, and logging best practices in async pipelines.

---

## Why This Matters

Async code **looks** like sync **`try/catch`**, but failure modes differ: exceptions may sit on **`Task.Exception`**, wrap in **`AggregateException`**, or crash the process via **`async void`**. **`Task.WhenAll`** with three failing tasks exposes **one** exception on **`await`** while others hide in **`task.Exception`**.

Production incidents: "exception swallowed after await", "logged wrong stack trace after **`throw ex`**", "API returns 500 but only first inner error visible". Middle developers must know **where** the exception lives at each suspension point and how ASP.NET Core middleware observes unhandled exceptions from **`async Task`** actions.

## Core Concepts

### Exceptions and await

When an awaited task **faults**, the **`await`** throws the **first inner exception** (not wrapped in **`AggregateException`**) on the awaiting thread/context:

```csharp
try
{
    await Task.FromException(new InvalidOperationException("boom"));
}
catch (InvalidOperationException ex)
{
    // caught here ‚ÄĒ same as synchronous throw after resume
}
```

The compiler-generated state machine calls **`Task.GetAwaiter().GetResult()`**-equivalent logic that **unwraps** **`AggregateException`** for a single failure.

### try/catch/finally with await

**`finally`** runs when control leaves **`try`** ‚ÄĒ whether by completion, exception, or **`return`**. With **`await` in try`**, **`finally`** executes **after** the awaited operation completes and before the method returns to caller:

```csharp
try
{
    await SaveAsync();
}
finally
{
    await CleanupAsync(); // legal ‚ÄĒ async finally via compiler rewrite
}
```

If **`await SaveAsync()`** throws, **`catch`** runs (if matches), then **`finally`**. Order matches sync semantics extended across suspensions.

**Anti-pattern:** empty **`catch`** that swallows ‚ÄĒ faulted state hidden; caller thinks success.

### AggregateException ‚ÄĒ when it appears

**Direct await** ‚ÄĒ usually **no** **`AggregateException`**; unwrapped.

**Sync blocking:**

```csharp
task.Wait();           // throws AggregateException
var x = task.Result;   // throws AggregateException (wrapping)
```

**`Task.WhenAll`:**

```csharp
Task t1 = Task.FromException(new Exception("A"));
Task t2 = Task.FromException(new Exception("B"));
try
{
    await Task.WhenAll(t1, t2);
}
catch (Exception ex)
{
    // typically ex.Message == "A" ‚ÄĒ FIRST fault only
}
// t2.Exception still on t2 ‚ÄĒ inspect manually if needed
```

For **all** failures: after **`WhenAll`**, check each task:

```csharp
await Task.WhenAll(tasks);
foreach (var t in tasks.Where(t => t.IsFaulted))
    _logger.LogError(t.Exception, "Task failed");
```

Or **`Task.WhenAll`** + custom aggregation in .NET 6+ patterns.

### Exception filters and when clauses

Works with async:

```csharp
catch (HttpRequestException ex) when (ex.StatusCode == HttpStatusCode.NotFound)
{
    return NotFound();
}
```

Filter runs when exception **reaches** catch block after await unwrap.

### async void ‚ÄĒ recap

**`async void`** events: unhandled exception ‚Üí **`SynchronizationContext`** / crash ‚ÄĒ **not** observable by caller **`await`**. Never in business logic (**`1.4.13`**).

### Observing without await

Faulted task **ignored**:

```csharp
_ = FailingAsync(); // compiler warning CS4014 ‚ÄĒ fire-and-forget
// Exception may become UnobservedTaskException
```

Always **`await`** or attach **`.ContinueWith`** / **`SafeFireAndForget`** with logging in controlled scenarios.

## Under the Hood

On fault, task stores exception in **`Task.m_exception`** (implementation detail). Awaiter **`GetResult()`** throws **`ExceptionDispatchInfo`**-preserved stack when possible.

**`ExceptionDispatchInfo.Capture(ex).Throw()`** rethrows with original stack ‚ÄĒ use when marshaling exceptions across threads without **`await`**.

State machine **`SetException`** completes the builder task as faulted; awaiting caller receives throw at resume point ‚ÄĒ stack trace spans async boundary correctly if not re-wrapped with **`throw ex`**.

## Syntax / API

| Scenario | Behavior |
|----------|----------|
| `await faulted task` | Throws inner exception |
| `task.Result` / `Wait()` | `AggregateException` |
| `Task.WhenAll` + await | First fault thrown; others on tasks |
| `try/catch` around await | Standard flow across suspend |
| `throw;` | Preserves stack |
| `throw ex;` | Resets stack ‚ÄĒ avoid |

## Examples

### WhenAll with logging all failures

```csharp
async Task RunBatchAsync(IReadOnlyList<Func<Task>> jobs, CancellationToken ct)
{
    Task[] tasks = jobs.Select(j => j()).ToArray();
    try
    {
        await Task.WhenAll(tasks);
    }
    catch (Exception)
    {
        var failures = tasks.Where(t => t.IsFaulted).Select(t => t.Exception!).ToList();
        _logger.LogError("Batch failed: {Count} faults", failures.Count);
        throw; // or throw new AggregateException(failures);
    }
}
```

### ASP.NET Core action

```csharp
[HttpGet("{id}")]
public async Task<ActionResult<Order>> Get(int id)
{
    try
    {
        return await _service.GetAsync(id);
    }
    catch (OrderNotFoundException)
    {
        return NotFound();
    }
    // unhandled ‚Üí exception middleware ‚Üí ProblemDetails
}
```

Middleware sees exception **after** entire async action unwinds.

## Common Mistakes & Anti-patterns

**Assuming `WhenAll` throws `AggregateException` on await** ‚ÄĒ usually single exception.

**`catch (Exception) { return null; }`** after await ‚ÄĒ hides failures.

**`.Result` in library** ‚ÄĒ **`AggregateException`** surprise + deadlock risk.

**Rethrow with `throw ex`** ‚ÄĒ loses async stack frames.

**Catching `TaskCanceledException` too broadly** ‚ÄĒ masks real bugs vs expected cancel.

---

## Production & Real-World Notes

Use **global exception handler** + **ProblemDetails** ‚ÄĒ still log **correlation ID** at first catch if translating to 404/409.

**Polly** wraps exceptions ‚ÄĒ catch policy-specific types or base after retries exhausted.

Test **`CancellationToken`** paths separately from fault paths.

**Activity** / OpenTelemetry: record exception on span at boundary.

| Approach | Pros | Cons |
|----------|------|------|
| await + try/catch | Natural flow | Only one exception at a time from WhenAll |
| Task.WhenAll + inspect all | Full picture | More code |
| `.Result` | Sync hack | AggregateException, deadlock |


## Key Takeaways

- **`await`** unwraps task exceptions ‚ÄĒ usually no **`AggregateException`**.
- **`Wait()`/`.Result`** wrap in **`AggregateException`** ‚ÄĒ avoid in async apps.
- **`Task.WhenAll`**: first fault on **`await`**; inspect siblings for multi-failure.
- **`try/catch/finally`** semantics extend across suspension points.
- Preserve stacks with **`throw;`** and **`ExceptionDispatchInfo`**.
- **`async void`** and unobserved tasks are separate failure classes.
- Log and translate at boundaries; don't swallow after **`await`**.
