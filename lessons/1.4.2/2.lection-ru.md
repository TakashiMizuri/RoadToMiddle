# I/O-bound vs CPU-bound: почему async помогает только I/O

## Why This Matters

В `1.4.1` — threads дороги, **thread pool** общий. Как обслужить 5000 concurrent HTTP clients без 5000 blocked threads? Ответ зависит от того, bottleneck — **ожидание** или **вычисление**.

Ошибка классификации: sync DB в `async` method без true async API (всё ещё блокирует pool thread); `async` для тяжёлой обработки изображений (overhead без ускорения CPU). Middle-разработчик классифицирует каждый hot path — controller, service, repository — до выбора async, `Task.Run` или parallel algorithms.

## Core Concepts

### I/O-bound work

**I/O-bound** — большую часть времени **ждёт** внешние системы: network, disk, database, queue. CPU простаивает; thread при sync блокировке сидит в kernel wait queue.

Примеры: `await httpClient.GetAsync`, `await dbContext.SaveChangesAsync`, `await File.ReadAllBytesAsync`. Длительность — RTT, disk seek, DB lock, не instruction throughput.

**Цель для I/O:** максимизировать **throughput**, **не держа threads** в wait. `async/await` регистрирует continuation при completion I/O; thread возвращается в pool (`1.4.6`–`1.4.7`).

### CPU-bound work

**CPU-bound** — **выполнение инструкций**: BCrypt, JSON serialize huge graph, image resize, тяжёлый LINQ in-memory. Мало waiting; threads до числа cores дают parallel throughput; дальше — contention.

**`async` не ускоряет CPU work** — может добавить state machine overhead. Offload — **`Task.Run`**, **`Parallel.For`** (`1.4.4`), bounded.

### Utilization picture

**Sync I/O:** `[CPU setup][======== WAIT ========][CPU resume]` — thread blocked весь wait.

**Async I/O:** `[CPU setup][thread free][continuation]` — wait не consume running worker.

**CPU-bound:** `[======== COMPUTE ========]` — thread busy; нужны algorithm, cores, parallel split.

1000 sync I/O ≈ 1000 blocked threads. 1000 async I/O — far fewer active threads одновременно.

### Latency vs throughput

**Async улучшает throughput** при concurrency для I/O; не уменьшает latency одного network hop (50 ms остаётся 50 ms). Один BCrypt hash — 100 ms CPU и sync, и «async».

## Under the Hood

### Blocking vs non-blocking I/O

Sync `socket.Read` — **blocking syscall**; thread not runnable до data. **Overlapped/async I/O** (IOCP, epoll) — operation started, completion later. `ReadAsync`, Npgsql async — true async.

**Fake async:** `Task.Run(() => stream.Read(...))` — blocking на другом pool thread; не уменьшает blocked threads globally.

### ASP.NET Core request model

Request на pool thread. Sync I/O держит его. Async I/O release at `await`. Kestrel принимает больше connections, чем threads.

### CPU saturation

Все cores 100% CPU-bound — async не создаёт capacity; scale out, optimize, reduce work.

### Decision flow

```
Ждём external I/O? → true async API
Иначе CPU-bound → Task.Run/Parallel (bounded) или sync on caller
```

## Syntax / Commands / API

```csharp
await httpClient.GetStringAsync(url);
await Task.Run(() => BCrypt.HashPassword(password)); // CPU only

// Anti-pattern
public async Task<string> Bad() =>
    await Task.Run(() => File.ReadAllText(path));

public async Task<string> Good() =>
    await File.ReadAllTextAsync(path);
```

## Examples

Sync `.Result` vs `await GetStringAsync` под 200 clients — sync exhausts pool; async lower p99. Same network time; different thread utilization.

Mixed: DB I/O + JWT verify CPU — `await` DB; `Task.Run` для crypto если heavy.

---

## Common Mistakes & Anti-patterns

`async` на pure CPU — overhead без benefit. `Task.Run` для network/DB — fake async. Sync работает с 1 user; падает при 100 concurrent I/O.

---

## Production & Real-World Notes

True async end-to-end: ReadAsync, EF async, HttpClient. Sync-over-async в dependencies — incident source. 1 CPU container — parallel CPU rarely helps; async I/O still wins.

---

## Comparison / Trade-offs

| Work | Tool | Wrong |
|------|------|-------|
| HTTP/DB/File | await true async | Sync, Task.Run wrap |
| Heavy CPU on API | Bounded Task.Run | async без offload |
| Parallel batch CPU | Parallel.ForEach bounded | Thousands Tasks |

---

## Quick Reference

- I/O-bound → async frees threads.
- CPU-bound → parallelism, not async magic.
- Fake async → Task.Run + sync I/O.

---

## Key Takeaways

- Классифицируй hot path: I/O или CPU.
- Async scales concurrent I/O.
- Async не ускоряет pure CPU.
- Provider async APIs end-to-end.
- Далее: **`Task`** (`1.4.3`).
