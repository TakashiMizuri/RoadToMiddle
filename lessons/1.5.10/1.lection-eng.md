# IDisposable vs Finalizer — The Dispose Pattern

> Roadmap: `1.5.10` · Node: `1.5` — C#: .NET Runtime and Memory · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Distinguish **deterministic disposal** (`IDisposable`) from **non-deterministic finalization** and explain why GC does not replace `Dispose`.
- Implement the **standard dispose pattern** for sealed and unsealed types, including **`Dispose(bool disposing)`** and **`GC.SuppressFinalize(this)`**.
- Explain the **finalization queue**, **f-reachable queue**, and why finalizers extend object lifetime by at least one GC cycle.
- Decide when a **finalizer is justified** (safe handle wrapper) vs an anti-pattern (managed-only cleanup).
- Connect to **`1.3.16`** (`using`, `await using`, `IAsyncDisposable`) as the consumer side of the same contract.
- Diagnose **double dispose**, **disposed object use**, and **finalizer resurrection** edge cases.

## Why This Matters

Managed memory is reclaimed when objects become unreachable. **Unmanaged resources** — file descriptors, socket handles, database connections, `SafeHandle` internals — are **outside** the GC's primary model. If you open ten thousand files and drop references without `Dispose`, you may have "free" managed wrappers while the OS refuses new handles. Production incidents from missing disposal look like **connection pool exhaustion**, **EMFILE** errors, and **locked files** on Windows long before OutOfMemoryException.

`IDisposable` is the contract for **prompt, developer-controlled cleanup**. Finalizers exist as a **safety net** when callers forget — but they run **later**, on a **dedicated thread**, in **undefined order**, and they **promote** objects through extra GC phases. Middle developers implement libraries and services where the **full dispose pattern** is required for extensibility (`virtual Dispose(bool)`), not only `using var` on BCL types.

This lesson goes deeper than **`1.3.16`**, which covered everyday `using` and async disposal. Here we focus on **why** the pattern looks the way it does, **what the runtime does** with finalizers, and **how** to implement base/derived hierarchies correctly.

## Core Concepts

### Two Cleanup Channels

When an object wrapping a native handle becomes unreachable, two things *can* happen:

1. **`Dispose()`** — you call it (directly or via `using`). Runs **deterministically** on the calling thread, **immediately** when scope ends. Releases handle, clears state, optionally suppresses finalization.

2. **Finalizer** (`~TypeName()`) — the runtime registers the object at allocation if a finalizer exists. When unreachable, GC **promotes** it to the **finalization queue** instead of freeing memory immediately. A **finalizer thread** runs `~TypeName()` **later** — seconds or minutes under load, **once** (unless resurrected).

**GC never calls `Dispose` for you** unless you use a pattern that wraps it (`using`). Finalizers and `Dispose` are **orthogonal** unless you wire them together with `SuppressFinalize`.

For types holding **only managed** references (`List<T>`, `string`), you typically **do not** implement `IDisposable` unless you aggregate disposable children and own their lifetime policy.

### IDisposable Recap (See 1.3.16)

```csharp
public interface IDisposable
{
    void Dispose();
}
```

Consumers should write:

```csharp
using var stream = File.OpenRead(path);
// or
await using var conn = await OpenConnectionAsync();
```

**Idempotency** is mandatory: second `Dispose()` must not throw or double-free. Guard with `_disposed` flag.

### The Standard Dispose Pattern

For a class that **owns unmanaged resources** or **must support inheritance**, Microsoft documents the **dispose pattern**:

```csharp
public class ResourceHolder : IDisposable
{
    private IntPtr _handle;
    private bool _disposed;

    public void Dispose()
    {
        Dispose(disposing: true);
        GC.SuppressFinalize(this);
    }

    protected virtual void Dispose(bool disposing)
    {
        if (_disposed) return;
        _disposed = true;

        if (disposing)
        {
            // Release managed objects that implement IDisposable
            // e.g. _ownedStream?.Dispose();
        }

        // Release unmanaged resources — ALWAYS in this method
        if (_handle != IntPtr.Zero)
        {
            NativeMethods.CloseHandle(_handle);
            _handle = IntPtr.Zero;
        }
    }

    ~ResourceHolder()
    {
        Dispose(disposing: false);
    }
}
```

**`disposing == true`** — called from public `Dispose()`. Safe to touch other **managed** disposables (they might already be disposed if caller order is wrong — still guard).

**`disposing == false`** — called from **finalizer**. **Do not** reference other managed objects — they may already be finalized (undefined order). Only release **direct** unmanaged state.

**`GC.SuppressFinalize(this)`** in public `Dispose()` tells the runtime: "Do not finalize this object; unmanaged cleanup already done." Skipping this when a finalizer exists wastes a **finalizer queue** round-trip.

**Sealed types** with no unmanaged resources and no subclassing can use a **simplified pattern** — single `Dispose()` with `_disposed` guard, no finalizer — as in `1.3.16`.

### Finalizer: When and Why

Add a finalizer **only** if the type **directly** holds unmanaged resources and callers might forget `Dispose`. BCL examples: **`FileStream`**, **`SafeHandle`** derivatives.

**Do not** add finalizers for:

- Pure managed classes
- "Just in case" — it **hurts** performance (all instances pay registration cost until suppressed)
- Logging in finalizer — unreliable, may throw

Modern code prefers **`SafeHandle`** — the handle type owns the finalizer; your public class only calls `ReleaseHandle` via `Dispose`:

```csharp
private sealed class NativeHandle : SafeHandle
{
    public NativeHandle() : base(IntPtr.Zero, ownsHandle: true) { }
    public override bool IsInvalid => handle == IntPtr.Zero;
    protected override bool ReleaseHandle() => NativeMethods.Close(handle);
}
```

Your `ResourceHolder` disposes the `SafeHandle` in `Dispose(bool)` — often **no finalizer on the outer class**.

### Derived Classes

Derived type **must** override `Dispose(bool)`:

```csharp
public class Derived : ResourceHolder
{
    private Stream? _stream;

    protected override void Dispose(bool disposing)
    {
        if (disposing)
            _stream?.Dispose();

        base.Dispose(disposing); // base frees handle
    }
}
```

Public `Dispose()` stays **non-virtual**; **`Dispose(bool)` is protected virtual** — prevents callers from accidentally calling the bool overload and skipping `SuppressFinalize`.

Do **not** override finalizer in derived unless derived adds **its own** unmanaged fields not cleaned in base. Usually base finalizer calling `Dispose(false)` suffices via virtual dispatch... **Careful:** if derived overrides `Dispose(bool)` and base has no finalizer but derived adds unmanaged field, derived needs finalizer **or** SafeHandle.

### IAsyncDisposable

Async cleanup (`await using`) mirrors sync — see **`1.3.16`** and **`1.4.35`**. Pattern:

```csharp
public async ValueTask DisposeAsync()
{
    await DisposeAsyncCore().ConfigureAwait(false);
    Dispose(disposing: false); // if also IDisposable
    GC.SuppressFinalize(this);
}

protected virtual ValueTask DisposeAsyncCore() => ValueTask.CompletedTask;
```

Do not block async dispose with `.Wait()` on I/O.

## Under the Hood

### Finalization Pipeline

When GC sees an unreachable object **with a finalizer**:

1. Object is **not** collected immediately.
2. Reference copied to **finalization queue**; object **re-rooted** (reachable again from runtime).
3. Next GC may collect **after** finalizer runs and object again unreachable (unless resurrected).

**f-reachable queue** — internal structure finalizer thread drains. Under load, backlog grows → **memory stays elevated** until finalizers catch up.

**Finalizer thread** is **one** (per process historically; details vary). Long-running finalizers **block** other finalizations.

**`GC.SuppressFinalize(this)`** removes object from needing finalization path if not yet run.

### Why Finalizers Hurt Performance

- Allocation registers finalization entry (cost even if you always Dispose).
- Promoted objects survive **extra Gen0/Gen1** collections.
- Non-deterministic cleanup → handles live longer → pressure.

**Dispose + SuppressFinalize** is the fast path.

### Object resurrection

Inside `~TypeName()`, reassigning `this` to a static field makes object **reachable again** — **resurrection**. Rare, confusing, almost always a bug. After finalizer, next GC collects unless resurrected again.

### Thread safety

`Dispose()` may be called from multiple threads. Pattern uses `_disposed` flag; for concurrent dispose, **`Interlocked`** or lock — BCL disposables often document "not thread-safe" unless stated.

## Syntax / Commands / API

| API | Role |
|-----|------|
| `IDisposable.Dispose()` | Public deterministic cleanup |
| `protected virtual Dispose(bool)` | Extensibility hook |
| `~TypeName()` | Finalizer — last resort |
| `GC.SuppressFinalize(this)` | Skip finalizer after successful Dispose |
| `SafeHandle` | Encapsulate native handle + finalizer |
| `using` / `await using` | Compiler-generated try/finally |

Analyzer rules: **CA1063** (implement IDisposable correctly), **CA1816** (call SuppressFinalize).

## Examples

### Sealed Wrapper — No Finalizer Needed

```csharp
public sealed class TempFile : IDisposable
{
    private readonly string _path;
    private bool _disposed;

    public TempFile()
    {
        _path = Path.GetTempFileName();
    }

    public void Dispose()
    {
        if (_disposed) return;
        _disposed = true;
        if (File.Exists(_path))
            File.Delete(_path);
    }
}
```

Managed-only OS artifact (path string + file on disk) — **deterministic delete** via `Dispose`; no finalizer because no raw handle field and `File.Delete` from finalizer is fragile.

### SafeHandle-Based (Preferred for Native)

Outer class has **no finalizer**; `SafeHandle` ensures handle release if `Dispose` forgotten.

## Common Mistakes & Anti-patterns

**Finalizer on managed-only class** — adds cost, no benefit.

**Calling managed dispose from `Dispose(false)`** — other objects may be dead; throws or corrupts state.

**Public `virtual Dispose()`** — callers override wrong method; use protected `Dispose(bool)`.

**Throwing from finalizer** — swallowed or crash; never throw.

**Forgetting `SuppressFinalize`** — object still finalized after successful Dispose.

**Using disposed object** — guard public methods with `ObjectDisposedException.ThrowIf(_disposed, this)` (.NET 7+).

## Production & Real-World Notes

ASP.NET **scoped** services dispose at **request end** — `DbContext`, `IAsyncDisposable` repositories. **Singleton** must not hold per-request disposable state without factory/scoped resolution.

**HttpClient**: `IHttpClientFactory` manages handler lifetime — do not dispose singleton `HttpClient` from DI incorrectly.

Code review checklist: every `new FileStream` / `SqlConnection` has **ownership** (`using` or injected scoped lifetime). Library types exposing unmanaged resources implement **full pattern** + XML doc "call Dispose".

## Comparison / Trade-offs

| Approach | Pros | Cons |
|----------|------|------|
| `using` only on BCL types | Simple | No custom resource types |
| Sealed `IDisposable` | Clear | No inheritance |
| Full dispose pattern | Extensible, correct unmanaged | Boilerplate |
| Finalizer | Safety net | Slow, non-deterministic |
| `SafeHandle` | Centralized native cleanup | Learning curve |

Prefer **`SafeHandle`** over raw `IntPtr` + own finalizer for new native interop.

## Quick Reference

```
Dispose() → Dispose(true) → SuppressFinalize
~Type()   → Dispose(false) → unmanaged only
Derived   → override Dispose(bool), call base
Consumer  → using / await using (1.3.16)
```

## Key Takeaways

- **Dispose now**; **finalize later** — never rely on finalizers for normal cleanup.
- Full pattern: public `Dispose`, protected virtual `Dispose(bool)`, optional finalizer, **`SuppressFinalize`**.
- Finalizers **delay** collection and run on **finalizer thread** — expensive.
- Use **`SafeHandle`** for native handles; link consumer code to **`1.3.16`**.
- **`disposing` flag** separates managed (true) vs unmanaged-only (false) cleanup paths.
