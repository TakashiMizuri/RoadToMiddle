# Span, ReadOnlySpan, and Memory — Stack-Safe Slicing

> Roadmap: `1.5.12` · Node: `1.5` — C#: .NET Runtime and Memory · Depth: **глубоко**

## Learning Objectives

After this lesson you will be able to:

- Explain **`Span<T>`** and **`ReadOnlySpan<T>`** as **ref struct** views over contiguous memory without copying.
- Slice arrays, strings, and stack memory safely with **bounds-checked** `[start..end]` syntax.
- Distinguish **`Memory<T>`** / **`ReadOnlyMemory<T>`** for **heap-friendly** async and field storage vs **`Span`** stack-only rules.
- Use **`Span<T>`** APIs (`Parse`, `TryParse`, UTF-8 processing) to reduce allocations in hot paths.
- Understand **`ref struct`** restrictions (no boxing, no async, limited fields) and why they exist.
- Connect to **`1.5.13` stackalloc** and **`ArrayPool`** for buffer strategies.

---

## Why This Matters

Every `string.Substring` and `array.Skip().Take()` on a hot path allocates new objects and copies bytes. Parsing HTTP headers, JWT segments, CSV fields, or protobuf in ASP.NET Core at thousands of requests per second turns allocation rate into **Gen0 pressure**, **GC pauses**, and **cache misses**. **`Span<T>`** (.NET Core 2.1+) gives a **window** into existing memory — array segment, stack buffer, or native pointer — with **length** and **safety checks**, often **zero-copy**.

Middle developers read modern BCL APIs: **`MemoryExtensions.Split`**, **`Utf8Parser`**, **`SequenceReader`**, **`PipeReader`** — all Span-centric. Misusing **`Span`** across `await` boundaries causes compile errors; **`Memory<T>`** exists for that boundary. Knowing when to use each prevents both **allocations** and **stack overflow** or **use-after-free** bugs (rare in safe C#, but **`stackalloc` in loops** is a real pitfall — see `1.5.13`).

## Core Concepts

### Span as a View

**`Span<T>`** is a **ref struct** containing a **reference** (byref) to memory start and **length**. It does not own memory — **whoever allocated** the backing store must outlive the span.

```csharp
int[] data = { 10, 20, 30, 40 };
Span<int> slice = data.AsSpan(1, 2); // views [20, 30]
slice[0] = 99;
// data[1] is now 99 — same backing array
```

**`ReadOnlySpan<T>`** — same but **read-only** API (e.g., over **`string`** chars without mutating immutable string — mutation through span over string memory is blocked by type).

**Slice syntax:**

```csharp
ReadOnlySpan<char> text = "Hello, World".AsSpan();
var hello = text[..5];
var world = text[7..];
```

Bounds checked — no buffer overrun in safe code.

### ref struct Rules

**`Span<T>`** is **`ref struct`** — **stack-only** (with exceptions for **`ref fields`** in C# 12+ in specific cases):

- Cannot be **field** of regular class (except **`ref field`** rules)
- Cannot be boxed, put in **`object`**, **`IEnumerable`**, most generics
- Cannot **`await`** or use in **`async`** methods across suspension
- Cannot capture in **lambda** that escapes method (unless local function constraints met)

These rules keep the **byref** from outliving its backing memory — **stack frame pop** must not leave dangling Span.

### Memory<T> for Async and Fields

**`Memory<T>`** is a **struct** (not ref struct) holding **`object?`** manager + index + length — can live on **heap**, in **fields**, and **`await`** with **`Memory<T>`** overloads:

```csharp
ReadOnlyMemory<byte> payload = responseBody;
await stream.WriteAsync(payload, cancellationToken);
```

Convert: **`memory.Span`** (may copy if not array-backed — check **`MemoryMarshal.TryGetArray`**).

Use **`Span`** for synchronous hot loops; **`Memory`** when storage crosses **`await`** or DI fields.

### Common BCL Patterns

**Parsing without substring:**

```csharp
ReadOnlySpan<char> s = line.AsSpan();
if (int.TryParse(s, out int n)) { /* ... */ }
```

**UTF-8:**

```csharp
Span<byte> utf8 = stackalloc byte[128];
int written = Encoding.UTF8.GetBytes(text.AsSpan(), utf8);
```

**Split:**

```csharp
foreach (Range token in line.AsSpan().Split(' ')) { /* ... */ }
```

### stackalloc and Span

**`stackalloc byte[n]`** yields **`Span<byte>`** (C# 7.2+):

```csharp
Span<int> buf = stackalloc int[256];
```

Memory from **current stack frame** — fast, no GC; **limited size**; **do not** return span pointing to stackalloc buffer from method (frame gone → invalid). **`1.5.13`** covers stack limits.

### ArraySegment vs Span

**`ArraySegment<T>`** — older wrapper; **`AsSpan()`** preferred. Span unifies array, stack, unmanaged.

## Under the Hood

**`Span<T>`** is **`ref struct`** with **`ref T _reference`** and **`int _length`** (simplified). JIT elides bounds checks in proven-safe loops.

**Interior pointers** — Span can reference **middle** of array; GC must pin when passing to native code — **`fixed`** or **`GCHandle`**.

**`ReadOnlySpan<char>` over string** — special: references string data inline (UTF-16) without allocation.

**Memory** may use **`ArrayPool`** rented buffer internally in some APIs; **`IMemoryOwner<T>`** from **`MemoryPool`** for ownership.

## Syntax / Commands / API

| Type | Owns memory | async | Field on class |
|------|-------------|-------|----------------|
| `Span<T>` | No | No | ref field only |
| `ReadOnlySpan<T>` | No | No | ref field only |
| `Memory<T>` | No (manager) | Yes | Yes |
| `ReadOnlyMemory<T>` | No | Yes | Yes |

```csharp
array.AsSpan(start, length);
memory.Span; // sync access
MemoryMarshal.Cast<byte, int>(bytes);
```

## Examples

### Zero-copy trim

```csharp
static ReadOnlySpan<char> Trim(ReadOnlySpan<char> s)
{
    int start = 0, end = s.Length - 1;
    while (start <= end && char.IsWhiteSpace(s[start])) start++;
    while (end >= start && char.IsWhiteSpace(s[end])) end--;
    return s[start..(end + 1)];
}
```

No new string until you **`ToString()`** when needed.

### Pipe-friendly

ASP.NET Core **`PipeReader`** produces **`ReadOnlySequence<byte>`** — multi-segment; **`SequenceReader`** walks without merging buffers when possible.

## Common Mistakes & Anti-patterns

**Returning Span over stackalloc** from method — **use-after-free**.

**Huge stackalloc** — stack overflow.

**Span in async method** — compile error; use **Memory**.

**Assuming Span always zero-copy** — **`ToString()`**, **`ToArray()`** allocate.

**Mutating ReadOnlySpan<char> from string** — use **`ReadOnlySpan`**.

## Production & Real-World Notes

Kestrel and **`System.Text.Json`** use Span internally. Profile before micro-optimizing — clarity matters unless hot path proven.

**Analyzers** help: CA2014 do not use stackalloc in loops.

Libraries expose **`ReadOnlySpan<char>`** overloads alongside **`string`** for flexibility.

## Comparison / Trade-offs

| Approach | Allocations | Complexity |
|----------|-------------|------------|
| `Substring` | New string | Low |
| `Span` slice | None | Medium |
| `ArrayPool` + Span | Rent only | Medium |
| `Memory` + async | Minimal copy | Medium |

## Quick Reference

- **Span** = sync, stack-scoped, zero-copy view.
- **Memory** = async, storable.
- **ref struct** = lifetime tied to stack discipline.
- **AsSpan()** on arrays/strings.

---

## Key Takeaways

- **Span/Memory** slice without copying contiguous memory.
- **ref struct** restrictions enforce lifetime safety.
- Use **ReadOnlySpan** for parse/read; **Memory** for **await**.
- Pair with **stackalloc** / **ArrayPool** (`1.5.13`, `1.5.14`).
- Modern BCL is Span-first — learn the idioms.
