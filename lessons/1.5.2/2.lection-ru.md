# JIT-компиляция: Tier 0, Tier 1 и Tier 2

> Roadmap: `1.5.2` · Node: `1.5` — .NET Runtime и память · Depth: **глубоко**

## Learning Objectives

После урока ты сможешь:

- Объяснить **JIT compilation** как перевод IL в **native machine code** at runtime.
- Описать **tiered compilation**: **Tier 0**, **Tier 1**, **Tier 2**.
- Предсказать trade-offs **startup vs steady-state** от tiered behavior.
- Назвать, когда **R2R** и **Native AOT** обходят tiered JIT.
- Связать JIT с **CPU spikes**, **code memory**, **recompilation** в production.
- Объяснить, почему **first call** медленнее последующих.

## Why This Matters

В `1.5.1` ты проследил C# → IL → CLR. CPU не исполняет IL. **Just-In-Time (JIT) compiler** в CoreCLR переводит IL каждого метода в native instructions **когда нужно** — обычно при первом вызове. Cross-platform deployment (один IL, разные JIT backends) vs **latency на cold paths** и **CPU на compilation**.

ASP.NET Core платит **warm-up tax**: первые requests JIT'ят controllers, EF, JSON serializers. Horizontal scaling — warm-up на каждом pod. Middle-разработчик объясняет **P99 spikes после deploy**, настраивает **ReadyToRun**, не путает JIT time с медленной БД.

Tiers объясняют, почему benchmarks **must warmup** — Tier 0 может быть на порядки медленнее Tier 2.

## Core Concepts

### Что делает JIT

При первом resolve call target CLR **JIT** (`clrjit.dll`):

1. Читает IL и metadata метода.
2. Verification rules.
3. Internal IR (trees, SSA в optimizing tiers).
4. Emits **native code** в **executable memory**.
5. Patches call sites на native entry.

Code несёт **GC info**, **exception handlers**, **stack frame layout** — связь с `1.5.4` и GC roots `1.5.6`.

### Tiered Compilation

Historically один optimizing JIT — хороший peak, **slow first compile**. **Quick JIT (Tier 0)** — fast minimal-opt; **tier recompilation** — hot methods → full optimization.

**.NET Core 3+ / .NET 5+** — tiered by default:

| Tier | Compile speed | Code quality | Trigger |
|------|---------------|--------------|---------|
| **Tier 0** | Very fast | Lower throughput | First execution |
| **Tier 1** | Medium | Good | Call threshold |
| **Tier 2** | Slower | Best | Hot methods, OSR |

**Fast bootstrap → better → best** по call counts.

### Promotion

```
First call → Tier 0
       │ N calls (~30, configurable)
       ▼
Background reJIT → Tier 1/2 (inlining, loops, SIMD)
       └── Call sites updated transparently
```

**OSR** — replace hot loop mid-execution без exit метода.

Cold paths остаются Tier 0 — OK для error handling.

### Optimizing JIT

Higher tiers: **inlining**, **loop opts**, **devirtualization**, **register allocation**, **intrinsics** (`Vector<T>`, `BitOperations`).

Tier 0 — больше spills, меньше analysis — correct but slower.

### JIT и Memory

- **Code heap** — native bodies (не GC heap).
- **Temporary JIT heap** при compile.

**ReJIT** — CPU bumps. Твои allocations — managed heap (`1.5.5`).

## Under the Hood

### clrjit Pipeline

```
IL → importer → morph → flow graph → optimizations → emit (x64/ARM64)
```

**GC info** — где managed references на каждой instruction point. **EH clauses** — unwind data.

**PGO** (.NET 6+) — runtime instrumentation для inlining.

### Call Counting

Runtime increment'ит counters on entry; threshold → **background JIT thread** — не блокирует app threads на long compile.

`DOTNET_TieredCompilation`, `DOTNET_ReadyToRun` — tune осознанно.

### ReadyToRun и Native AOT

**R2R** — precompiled native в assembly; fallback JIT при mismatch.

**Native AOT** — compile at build, no clrjit at runtime; reflection limited.

## Common Mistakes & Anti-patterns

Benchmark без warmup. Disable tiering globally вместо R2R/AOT. Assume one JIT forever — tier promotion replaces code. **Generic explosion** — отдельные native bodies per `List<T>` instantiation.

---

## Production & Real-World Notes

Horizontal scaling multiplies JIT. Container CPU limits stretch warm-up. **dotnet-trace** Jit events. Server GC + JIT spikes в flame graphs.

---

## Comparison / Trade-offs

| Approach | Startup | Peak | Flexibility |
|----------|---------|------|-------------|
| Tiered JIT | Good | Excellent after warm | Full dynamic |
| R2R | Better | Good | Larger deploy |
| Native AOT | Best | Good | Limited reflection |

---

## Key Takeaways

- **JIT** — IL (`1.5.1`) → CPU.
- **Tiered** — startup (T0) vs throughput (T2).
- **Warmup** обязателен для perf и probes.
- **R2R/AOT** — alternatives when runtime JIT costly.
