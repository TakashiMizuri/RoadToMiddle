# Exercises: git show and git blame — Forensics

> Complete after reading `1.lection-eng.md`, `2.lection-ru.md` and passing `4.test-yourself.md`. Ask AI for hints, not full solutions.

## Easy

### E1. Inspect your last commit
**Goal:** Use `git show` fluently.

**Steps:**
1. Run `git show --stat` on `HEAD`.
2. Run `git show HEAD --no-patch` — what extra info appears vs step 1?
3. Pick one file from the commit; view its content at that revision: `git show HEAD:path/to/file`.
4. Write one sentence: what changed and why (from commit message).

**Success criteria:** Three commands executed; you can explain `rev:path` syntax.

---

### E2. Blame a line range
**Goal:** Targeted blame with `-L`.

**Steps:**
1. Open any source file with 50+ lines in your repo (or create a sample).
2. Run `git blame -L 1,30 <file>` and `git blame <file>` — compare speed/usefulness.
3. Pick one line; run `git show <hash>` for the hash shown on that line.
4. Record: author, date, and whether the change matches the line's purpose.

**Success criteria:** Full chain blame → show documented in notes.

---

## Medium

### M1. Mini incident investigation
**Goal:** Full forensic workflow.

**Scenario (use real repo or simulate):**
- A function `Divide(a, b)` exists; line with `return a / b` has no zero check.
- Pretend QA reported a divide-by-zero bug on that line.

**Steps:**
1. `git blame -L <n>,<n> -- <file>` on the division line.
2. `git show <commit>` — read full context.
3. `git log -S "Divide" --oneline -- <file or dir>` — when was the method introduced?
4. Write a 5-line incident note: symptom → line → commit → author → recommended fix type (not the fix itself).

**Success criteria:** Note references real hashes from your repo.

---

### M2. Compare file across two revisions
**Goal:** Historical file read without checkout.

**Steps:**
1. Choose a text file with at least 3 commits of history.
2. `git show HEAD:file` and `git show HEAD~2:file` — spot one difference verbally.
3. Optional: `git diff HEAD~2 HEAD -- file` to confirm.

**Success criteria:** You explain why `git show rev:file` is safer than checkout for quick peek.

---

## Hard (stretch)

### H1. Move detection with -C -M
**Goal:** See blame limitations and flags.

**Steps:**
1. Create `legacy/Helper.cs` with method `NormalizeEmail`, commit.
2. Copy method to `services/EmailService.cs`, delete from Helper (or rename file), commit.
3. `git blame services/EmailService.cs` — note uniform recent commit.
4. `git blame -C -M services/EmailService.cs` — compare first lines of `NormalizeEmail`.
5. Document: when plain blame lied, and what `-C -M` revealed.

**Success criteria:** Before/after blame snippet in notes; two-sentence explanation.

---

### H2. Pickaxe vs grep in history
**Goal:** Understand `-S` value.

**Steps:**
1. Pick a unique string that appears in exactly one commit in a file (add a `// MARKER-XYZ` in a commit if needed).
2. `git log -S "MARKER-XYZ" --oneline -- <file>`
3. `git log --grep="MARKER"` — compare results.
4. Explain when pickaxe is better than `--grep` (hint: code presence vs message text).

**Success criteria:** Written comparison table: pickaxe vs grep vs blame.

## Hints (read only if stuck)

<details>
<summary>Hint E1</summary>

`--no-patch` suppresses the diff hunk output but keeps author, date, and commit message — useful for log-style inspection.
</details>

<details>
<summary>Hint H1</summary>

You may need `-C -C` if a single `-C` does not trace across commits. Start with `git blame -C -M -L <start>,<end> file`.
</details>

<details>
<summary>Hint H2</summary>

`--grep` searches commit **messages** only. `-S` searches **content changes** in diffs (appearance/disappearance of a string).
</details>
