# Amending commits: когда можно, когда нельзя

> Roadmap: `0.1.10` · Узел: `0.1` — Git: основы · Глубина: понимание

## Цели урока

После этой лекции ты сможешь:

- Объяснить, что делает **`git commit --amend`** с последним commit.
- Определить, когда amend **безопасен** (local, unpushed).
- Определить, когда amend **опасен** (pushed / shared history).
- Сравнить **`--amend`** с созданием **нового commit** для исправления ошибок.
- Сформулировать **golden rule of public history**: не переписывать commits, которые другие уже pull'нули.
- Понимать, когда нужен **interactive rebase** (без глубокого погружения).

---

## Зачем это нужно

Знакомая ситуация: ты закончил фичу, выполнил `git commit -m "Add user login endpoint"`, ещё ничего не push'нул — сидишь на local branch, допиливаешь перед pull request. И тут замечаешь опечатку в message или понимаешь, что `appsettings.Development.json` так и не попал в staging area. Без amend инстинкт подсказывает сделать ещё один commit: для опечатки, потом для забытых файлов. В итоге в history три отдельных события там, где было одно логическое изменение.

Этот шум важнее, чем кажется на первый взгляд. Code review усложняется: ревьюеру приходится мысленно склеивать три commits в один intent. **`git bisect`**, инструмент поиска commit'а, который внёс баг, рассматривает каждый commit как потенциального виновника — три commits на одну фичу утраивают пространство поиска без всякой пользы. А когда через полгода кто-то читает `git log`, он видит «fix typo» и «add forgotten file» вместо чистой записи того, что реально попало в код.

**`git commit --amend`** существует именно для **последнего commit'а**: он позволяет влить исправления в только что созданный commit и сохранить linear history. Но amend — не carte blanche на переписывание history когда угодно. Тот же механизм, который наводит порядок локально, может сломать clone коллеги, инвалидировать открытые pull request'ы и вынудить опасный force-push, если применить его к commits, которые уже покинули твою машину. Понимание, когда amend — удобство, а когда — минное поле, — один из самых чётких маркеров перехода от junior-привычек к Git-грамотности уровня Middle.

## Основные концепции

### Что на самом деле делает `--amend`?

Слово «amend» намекает на редактирование на месте — будто Git открыл последний commit и поменял несколько байт. Эта mental model неверна, и она создаёт путаницу позже. Commits в Git **immutable**: SHA-1 hash — отпечаток всего содержимого: tree файлов, parent commit, author, committer, message. Измени хоть один символ в message — получишь совершенно другой hash.

При `git commit --amend` Git **не модифицирует** существующий commit. Он создаёт **новый commit object**, который заменяет старый с точки зрения текущей ветки. Новый commit может включать свежие staged changes (если перед amend был `git add`), обновлённый message (если передал `-m` или правил в editor), или и то и другое. Branch pointer — скажем `refs/heads/main` или `refs/heads/feature/login` — переезжает на новый SHA. Предыдущий commit, который ты «редактировал», теряет branch reference. Он ещё какое-то время лежит на диске, но на него указывает только reflog.

Визуально переход выглядит так:

```
До amend:
  main → A → B → C (HEAD)
                  ↑ забыли файл

После git add forgotten.txt && git commit --amend:
  main → A → B → C' (HEAD)   ← новый SHA, включает forgotten.txt
                  C orphaned (в reflog ещё есть)
```

Orphaned commit `C` не уничтожается мгновенно. Garbage collector заберёт его позже, но локально ты можешь найти его через `git reflog` примерно девяносто дней. Это safety net при ошибочном amend — но он **не помогает** коллегам, у которых `C` никогда не было в reflog, потому что они pull'нули другую history с remote.

Три сценария покрывают большинство повседневного использования. Только message: `git commit --amend -m "Correct message"`. Забытые файлы: сначала `git add`, потом `git commit --amend --no-edit` — флаг `--no-edit` сохраняет message, добавляя staged content в replacement commit. Message через editor: просто `git commit --amend` откроет editor с текущим текстом.

### Golden Rule of Public History

Каждая команда рано или поздно формулирует правило: **никогда не переписывай commits, на которых уже построена чужая работа.** «Public» history — любой commit, push'нутый в shared remote branch (`main`, `develop`, release branch), где коллеги, CI или deployment-системы считают эти SHA источником правды.

Причина механическая, не социальная. После amend pushed commit'а local branch указывает на `C'`, remote — на `C`. Histories **diverged**. Обычный `git push` отклоняется: Git не принимает non-fast-forward updates, которые выбросят remote commits. «Решение», к которому тянутся — `git push --force`, перезаписывающий remote branch. Тот, кто уже pull'нул `C`, имеет clone с history, не совпадающей с сервером. Следующий push может упасть; понадобится rebase или reset; open PR'ы покажут странные diff'ы; CI перезапустится на SHA, которых на remote больше нет.

На local branch, который никогда не push'или, amend почти всегда безопасен — ты единственный потребитель этой history. На feature branch, где работаешь только ты, amend безопасен до тех пор, пока кто-то другой не pull'нет — тогда та же проблема divergence. На `main` после merge — нет: новый commit или `git revert`. На release tag, уже задеплоенном в production, переписывание недопустимо: артефакт привязан к immutable SHA.

## Под капотом

При `git commit --amend` Git выполняет последовательность шагов, похожую на обычный commit, но с другим выбором parent. Вместо текущего `HEAD` как parent amend берёт **parent текущего `HEAD`**. По сути Git говорит: «Собери новый commit поверх того, что было до моего последнего commit'а, включи staged content и сделай это новым tip.»

Если файлы изменились, Git строит новый **tree object**. **Author date** может сохраниться от оригинала (зависит от версии Git и опций), **committer date** обновится на «сейчас» — это видно в `git log --format=fuller`. Branch ref (например `.git/refs/heads/main`) атомарно обновляется на новый SHA. Старый tip становится unreachable от любой ветки, но остаётся в `refs/logs/HEAD` и `refs/logs/refs/heads/main`.

History видна в reflog:

```bash
git reflog
# abc1234 HEAD@{0}: commit (amend): Add user login
# def5678 HEAD@{1}: commit: Add user login   ← старый commit, recoverable локально
```

Если `def5678` уже был push'нут на origin до amend, remote всё ещё ссылается на него. Local `main` — на `abc1234`. До force-push сосуществуют две «истины»: твоя машина говорит `abc1234`, сервер — `def5678`. Это и есть divergence в конкретных терминах.

Amend строго ограничен **`HEAD`** — commit'ом на tip текущей ветки. Commit три шага назад amend не тронет. Здесь нужен **interactive rebase** (`git rebase -i`), отдельная тема позже. Те же правила golden rule: переписывание pulled commits наносит тот же ущерб, только сразу по нескольким SHA.

## Синтаксис

```bash
# Только message
git commit --amend -m "New message"

# Добавить staged files, сохранить message
git add forgotten.cs
git commit --amend --no-edit

# Author/date (редко — исправление metadata)
git commit --amend --author="Name <email>" --no-edit
git commit --amend --date="2026-07-01T12:00:00"

# Message через editor
git commit --amend
```

Prerequisite для забытых файлов: **`git add` до `--amend`**. Staging говорит Git, какой content войдёт в replacement commit. Без staged changes amend меняет только message.

## Примеры

### Пример 1: Безопасный amend на unpushed work

Feature branch `feature/health` существует только на твоём laptop. Ты commit'ишь controller, но забываешь helper class:

```bash
git commit -m "Add health check endpoint"
git add HealthCheckExtensions.cs
git commit --amend --no-edit
git push origin feature/health       # первый push — remote не видел старый commit
```

Идеальный случай amend. Remote не получил неполный commit. При первом push сервер видит один корректный commit. Без force push, без divergence, без impact на коллег. Многие команды поощряют этот паттерн на feature branches перед PR.

### Пример 2: Опасный amend после push в shared branch

Та же ошибка, но ты уже push'нул в `main`:

```bash
git commit -m "Add health check"
git push origin main                 # shared — другие могли pull'нуть
git add forgotten.cs
git commit --amend --no-edit
git push origin main                 # REJECTED: non-fast-forward
git push --force origin main         # ⚠️ ОПАСНО — переписывает shared history
```

После amend коллеги с оригинальным commit'ом имеют `def5678` в history. Force-push заменяет remote на `abc1234`. Local `main` коллег всё ещё на `def5678`, которого на сервере больше нет. Им придётся `git fetch origin && git reset --hard origin/main` — с риском потерять local commits — или rebase на новую history. Высокая цена за один забытый файл.

Безопаснее после push — follow-up commit:

```bash
git add forgotten.cs
git commit -m "Add missing HealthCheckExtensions to health check commit"
```

Log чуть шумнее, но history согласована для всех. Force push не нужен.

### Пример 3: Выбор между amend и новым commit

Decision tree проще, чем кажется. Один вопрос: **мог ли кто-то построить работу на этом commit'е?** Нет — amend свободно. Да — новый commit или revert. На open PR, где работаешь только ты, amend + force-push PR branch (`git push --force-with-lease origin feature/health`) — норма на многих командах. `--force-with-lease` откажется push'ить, если кто-то обновил remote branch с твоего последнего fetch — небольшая страховка против bare `--force`.

## Частые ошибки

Самая разрушительная — amend на `main`/`develop` и force-push «потому что мелкий fix». Мелкий fix всё равно даёт новый SHA, а protected branches с CI, deployment hooks и clone'ами коллег умножают blast radius. Частая ошибка — `git commit --amend` без staging, ожидая подхват unstaged changes. Amend включает только index; unstaged modifications остаются в working directory.

Amend merge commits без понимания результата — confusing conflict state: у merge commit два parent'а, замена редко нужна. Amend чтобы скрыть ошибку от review подрывает доверие: ревьюер одобрил конкретный diff на конкретном SHA, amend меняет diff молча. Amend для squash трёх local commits перед PR — не тот инструмент; нужен interactive rebase с теми же правилами rewrite.

## Production notes

Команды часто сочетают amend с **squash merge**: разработчики amend'ят feature branch, platform мержит PR одним commit'ом в `main`. Rewrite только на private branch; `main` не видит force-pushed amendments.

Branch protection на `main` — блок force push, обязательные PR, CI — частично против history rewrite на shared lines. Signed commits: amend создаёт новый object, подпись перегенерируется. В regulated средах audit trail требует immutable SHA задеплоенных релизов; amend tagged release — compliance violation.

Когда ошибка уже published и amend нельзя, **`git revert`** создаёт commit, отменяющий предыдущий, не стирая его из history. Production-safe путь для «мы shipped broken code на `main`».

## Сравнение подходов

**`git commit --amend`** — чистая linear history, пока commit private. Цена — новый SHA и риск divergence при позднем amend. **Новый fixup commit** — шумнее, но безопасен на shared branch; честная запись «забыли и добавили позже». **`git revert`** — когда bad commit public и нужно отменить эффект, не притворяясь, что его не было. **Interactive rebase** — reshape нескольких local commits перед PR; те же rewrite mechanics, но не только `HEAD`.

Универсально «лучшего» нет. Middle-level judgment — подобрать инструмент под то, кто ещё зависит от меняемых commits.

---

## Шпаргалка

Typo, не push'или: `git commit --amend -m "..."`. Забыли файл, не push'или: `git add ... && git commit --amend --no-edit`. Ошибка на shared `main`: новый commit или `git revert`. Cleanup нескольких local commits: `git rebase -i` (позже). Уже push в ветку, которую используют другие: **не amend** — follow-up commit.

---

## Главное

- **`--amend` заменяет HEAD commit** новым object с новым SHA — не edit in place.
- **Безопасен local/unshared** — remote не видел старый commit.
- **Опасен после push в shared branches** — нужен force push, ломает clone'ы других.
- **Golden rule:** не переписывать public history, от которой зависят коллеги и CI.
- На shared — **новый commit** или **revert**; amend — для private work.
- **`--no-edit`** — добавить staged files без смены message.
- Commits старше последнего — **interactive rebase** с теми же осторожностями.
